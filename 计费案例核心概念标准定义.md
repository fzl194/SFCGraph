# 计费案例核心概念标准定义

> 范围: 基于计费场景业务图谱、计费统一知识库、计费特性知识、UPF/SMF 原始特性文档、命令文档以及 UDG 特性使用说明形成的标准定义。
> 目的: 解决 `业务域 / 场景 / 特性 / 子特性 / Task` 的边界模糊问题，为后续知识抽取和图谱落库提供一致判定标准。

---

## 1. 结论先行

这五个对象的边界，不应该靠“感觉像不像”来划，而应该按下面这条链判断：

```text
业务域
  -> 场景
  -> 配置方案
  -> 特性
  -> 子特性
  -> Task
  -> 命令
```

它们的本质区别是：

- `业务域`：长期存在的问题空间
- `场景`：该问题空间里反复出现的一类需求情境
- `特性`：产品定义、客户可识别、通常带特性编号的正式能力
- `子特性`：单个正式特性内部的稳定细分形态
- `Task`：最小可复用配置原子

其中最重要的三条判定规则是：

1. **有正式特性编号的，优先判定为 `Feature`，不是 `SubFeature`。**
2. **能独立回答“现场到底在解决什么问题”的，优先判定为 `Scenario`，不是 `Feature`。**
3. **已经下沉到“如何组织一段稳定配置动作链”的，优先判定为 `Task`，不是 `Feature`。**

## 3. 业务域 BusinessDomain

### 3.1 标准定义

`业务域` 是一类**长期存在、可持续积累、覆盖多个现网场景和多个产品能力的工程知识空间**。

它回答的是：

- 这一大类业务问题到底在研究什么
- 这一类问题下面会派生出哪些典型场景
- 这些场景通常会调用哪些特性栈和配置方法

它不是：

- 某个单独客户需求
- 某个单独套餐
- 某个单独特性
- 某段配置动作

### 3.2 计费案例中的判定

在现有材料里，`业务感知` 明确应判定为 `BusinessDomain`，不是场景。

原因：

1. [业务感知业务图谱.md](</D:/mywork/KnowledgeBase/SFCGraph/business-graph/计费场景/SKILL/knowledge/业务感知业务图谱.md>) 把它定义成一个覆盖计费、带宽控制、访问限制、本地分流等多条业务主线的稳定知识空间。
2. 它不是单一需求，而是一组更上位的业务处理能力。
3. 它下面天然包含多个场景入口，而不只是计费。

### 3.3 判断标准

满足下面三个条件，优先判定为 `BusinessDomain`：

1. 覆盖多个场景，而不是一个场景。
2. 覆盖多个特性，而不是一个特性。
3. 脱离某个单独方案后仍然成立。

### 3.4 计费案例中的标准例子

- `业务感知`：是业务域
- `计费场景`：不是业务域，是场景
- `内容计费基本功能`：不是业务域，是特性

## 4. 场景 NetworkScenario

### 4.1 标准定义

`场景` 是业务域内部**反复出现的一类稳定需求情境、问题类型或配置目标模式**。

它回答的是：

- 现场到底在解决什么问题
- 什么情况下会落入这一类问题
- 希望得到什么典型结果

它不是：

- 产品能力编号
- 某个命令链
- 某个配置对象
- 某次具体实施项目

### 4.2 计费案例中的判定

[01计费场景业务图谱.md](</D:/mywork/KnowledgeBase/SFCGraph/business-graph/计费场景/01计费场景业务图谱.md>) 中的 `NS-01 计费场景` 就是标准场景：

- `scenario_summary`：对不同业务流采用不同计费方式，结合默认计费、免费业务和配额动作完成计费闭环
- `judgment_basis`：需要按业务粒度差异化计费，或需要按流量/时长/事件统计使用量
- `typical_outcome`：专项业务单独计费、免费业务不计费、普通业务默认计费，配额耗尽后切换动作

这说明“场景”的关键不在于它用了哪些特性，而在于它描述了一类稳定问题和目标。

### 4.3 为什么计费不是业务域

因为从材料看，`计费` 只是 `业务感知` 下的一类稳定情境：

- 它有明确判断条件
- 它有典型结果
- 它会调用多种特性组合
- 它只是业务感知的一条主线，不覆盖全部业务感知问题

所以 `计费` 应判定为 `NetworkScenario`，不是 `BusinessDomain`。

### 4.4 判断标准

满足下面四个条件，优先判定为 `Scenario`：

1. 能独立描述一线问题情境。
2. 有清晰的进入条件或判断依据。
3. 有典型结果，而不是只有实现方式。
4. 通常需要多个特性或多个 task 共同支撑。

### 4.5 计费案例中的标准例子

- `计费场景`：是场景
- `内容计费`：通常不是场景，优先判定为特性
- `融合计费方式选择`：不是场景，是决策点或方案内分支

## 5. 特性 Feature

### 5.1 标准定义

`特性` 是产品定义的、客户可识别的、通常可销售或可控制的正式能力对象。

这是当前最清晰的一类对象，直接以产品文档为准。

[特性使用说明_42197352.md](</D:/mywork/KnowledgeBase/SFCGraph/output/UDG_Product_Documentation_CH_20.15.2/特性部署/特性指南/UDG特性使用说明/特性使用说明_42197352.md>) 已给出标准定义：

- 特性是产品包的主要卖点
- 每条特性都是满足客户特定商业价值诉求的端到端解决方案
- 特性是客户视角的概念
- 特性是可定价、可购买的高层次功能或功能集合

### 5.2 图谱中的判定规则

在当前体系里，优先采用以下硬规则：

**规则 1：有正式特性编号，就优先是 `Feature`。**

例如：

- `GWFD-020301 内容计费基本功能`
- `GWFD-020302 基于业务时长的计费`
- `GWFD-020303 基于业务流量的计费`
- `GWFD-020306 支持事件计费`
- `GWFD-020300 在线计费`
- `GWFD-010171 离线计费`
- `GWFD-010173 融合计费`
- `WSFD-109002 内容计费基本功能`
- `WSFD-011206 支持融合计费`

即使其中某些能力从语义上看像“某个更大能力的细分”，只要产品已经给了独立特性编号，就仍然应判定为 `Feature`。

### 5.3 为什么“时长计费/流量计费/事件计费”不是子特性

[cross-feature-analysis.md](</D:/mywork/KnowledgeBase/SFCGraph/business-graph/计费场景/feature-knowledge/cross-feature-analysis.md>) 明确指出：

- `GWFD-020302`、`GWFD-020303`、`GWFD-020306` 都是独立特性
- 它们各自有独立特性 ID、License、首次发布版本

因此它们在图谱里应当是 `Feature`，而不是 `SubFeature`。

### 5.4 判断标准

满足下面任意两条，优先判定为 `Feature`：

1. 有正式特性编号。
2. 有独立特性概述文档。
3. 有独立 License 或版本信息。
4. 能作为正式能力被业务方案引用。

## 6. 子特性 SubFeature

### 6.1 标准定义

`子特性` 是**单个正式特性内部**在特定维度上的稳定细分形态。

它不是“所有更细的东西都叫子特性”，而是必须同时满足：

1. 它属于某一个 `Feature` 内部；
2. 它能被稳定重复引用；
3. 它本身又没有被产品定义成独立特性编号。

### 6.2 为什么不能把所有细分能力都叫子特性

这是当前最容易混的地方。

如果一个细分能力已经有了独立特性编号，它就不应该再退回 `SubFeature`。  
也就是说：

- `内容计费` 和 `时长计费` 的关系，在语义上可能是“基础 + 增强”
- 但在产品定义上，它们是两个正式特性

图谱定义必须服从正式产品能力边界，而不是只服从概念相似度。

### 6.3 计费案例里更像子特性的东西

在当前计费材料中，更适合作为 `SubFeature` 的，是**单个特性内部的稳定实现形态**，例如：

- 在 `GWFD-020301 内容计费基本功能` 下：
  - 基于 URL 的内容识别
  - 基于 L3/L4 的内容识别
  - any 默认兜底计费
  - 异常流量缺省计费

这些在 [部署UPF_28493406.md](</D:/mywork/KnowledgeBase/SFCGraph/business-graph/SKILL/feature/UDG/UPF/GWFD-020301/部署UPF_28493406.md>) 的任务示例里都稳定出现，但它们没有独立正式特性编号，更适合作为 `SubFeature` 或 `Feature` 下的细分语义。

另外，多代际/多模式差异也可以形成 `SubFeature`，前提仍然是：**没有独立特性编号，但需要被单独引用和挂规则。**

### 6.4 判断标准

满足下面四条时，优先判定为 `SubFeature`：

1. 明确隶属于某一个 `Feature`。
2. 具备稳定细分维度，如协议、计费模式、业务类型、部署模式、代际。
3. 会被单独挂 task、规则、决策或方案引用。
4. 没有自己的正式特性编号。

### 6.5 反例

以下对象不要判成 `SubFeature`：

- `GWFD-020302 时长计费`
- `GWFD-020303 流量计费`
- `GWFD-020306 事件计费`
- `GWFD-010173 融合计费`

这些都应判定为 `Feature`。

## 7. Task

### 7.1 标准定义

`Task` 是**最小可复用配置原子**。

它回答的是：

- 为了实现某类能力或方案，需要完成哪一个稳定配置动作单元
- 这个动作单元需要什么输入
- 它产出什么结果
- 它通常由哪些命令按什么稳定顺序完成

它不是：

- 单条命令
- 单个参数
- 单个配置对象
- 一个完整特性
- 一个完整场景

### 7.3 Task 与命令的边界

`Task` 不是命令。

例如：

- `ADD URR` 是命令
- `配置计费动作链` 是 task

后者通常需要按稳定顺序调用多条命令：

```text
ADD URR
-> ADD URRGROUP
-> ADD PCCPOLICYGRP
```

因此，task 的粒度应高于命令，但低于特性。

### 7.4 Task 与特性的边界

`Task` 也不是特性。

例如：

- `GWFD-020301 内容计费基本功能` 是特性
- `配置计费动作链`、`配置过滤规则`、`配置 Rule` 是 task

同一个特性通常拆成多个 task；
同一个 task 也可能被多个特性或多个方案复用。

这正是 task 需要独立建模的原因。

### 7.5 判断标准

满足下面五条中的三条，优先判定为 `Task`：

1. 它表达的是一个稳定配置动作，而不是一个能力名称。
2. 它有明确输入和输出。
3. 它通常由一组固定命令链完成。
4. 它可以被多个方案或多个特性复用。
5. 它再往下拆就会直接落到命令层。

### 7.6 Task 的标准边界

推荐把 task 控制在这个粒度：

- 足够小，能复用
- 但又不小到退化成单条命令

例如：

**合理的 task：**

- 配置计费动作链
- 配置过滤规则
- 配置流过滤器与绑定
- 配置 Rule
- 配置 UserProfile 与 Rule 绑定
- 配置融合计费总开关
- 配置 CHF 交互触发条件

**不合理的 task：**

- `ADD URR` 本身
- `URRID=1111` 这种参数值设置

这些已经属于命令层或参数层。

## 8. 五类对象的边界总表

| 对象 | 核心问题 | 稳定性 | 是否有正式编号 | 是否直接落命令 | 计费案例典型例子 |
| --- | --- | --- | --- | --- | --- |
| `BusinessDomain` | 这一大类业务到底在研究什么 | 最高 | 否 | 否 | `业务感知` |
| `NetworkScenario` | 现场到底在解决什么问题 | 高 | 否 | 否 | `计费场景` |
| `Feature` | 产品正式提供了什么能力 | 高 | 通常有 | 否 | `GWFD-020301 内容计费基本功能` |
| `SubFeature` | 单个正式特性内部有哪些稳定细分形态 | 中高 | 无独立正式编号 | 否 | `URL 内容识别计费`、`默认 any 兜底计费` |
| `Task` | 为了实现能力，需要做哪一个可复用配置动作 | 高 | 无正式特性编号 | 间接落命令 | `配置计费动作链` |

## 9. 当前推荐的判定流程

遇到一个候选对象时，按下面顺序判断：

### 9.1 第一步：先问它是不是正式产品能力

如果有独立特性编号、独立特性文档、独立 License 或版本信息，优先判定为 `Feature`。

### 9.2 第二步：如果不是特性，再问它是不是单个正式特性内部的稳定细分

如果它明显隶属于一个特性，且需要被单独引用，但没有自己的特性编号，判定为 `SubFeature`。

### 9.3 第三步：如果也不是，再问它是不是现场问题情境

如果它能回答“什么情况下属于这类问题、目标是什么”，判定为 `Scenario`。

### 9.4 第四步：如果还不是，再问它是不是一个稳定配置动作单元

如果它有输入、输出、命令链、复用价值，判定为 `Task`。

### 9.5 第五步：剩下的才考虑命令、参数、配置对象

如果已经细到单条命令、参数、对象，就不再属于上面五类。

## 11. 最后一句话

在当前体系里，最稳的边界应当是：

- `业务域` 看问题空间
- `场景` 看需求情境
- `特性` 看正式产品能力编号
- `子特性` 看单个正式特性内部的稳定细分
- `Task` 看最小可复用配置动作

其中，**特性边界服从产品文档，Task 边界服从配置复用粒度，场景边界服从现网问题表达。**
