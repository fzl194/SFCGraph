# 特性层对象与关系定义 v1

> 定位：三层图谱中的特性图谱层，只定义产品能力世界的对象、关系和边界。
> 前提：命令层以 `CommandGraph/` 最新定义为准；任务层以 `ConfigTask/` 最新定义为准。本文不重复定义命令、参数、配置对象、任务字段。

---

## 1. 一句话定位

特性层回答：

- 产品正式提供了哪些能力
- 能力之间有什么依赖、互斥和协同
- 能力需要什么 License
- 能力通常拆成哪些 ConfigTask
- 能力级有什么规则、决策和编排

特性层不回答：

- 具体敲哪些 MML 命令
- 命令参数如何取值
- 配置对象之间如何引用
- task 内部命令顺序如何组织

这些全部交给任务层和命令层。

主链是：

```text
Feature / SubFeature
  -> ConfigTask
  -> MMLCommand / CommandParameter / ConfigObject
```

## 1A. ID 与版本治理

特性层 ID 规则对齐命令层和任务层，统一使用四段式实例键：

```text
{nf}@{version}@{ObjectType}@{local_name}
```

约定：

| 维度 | 约定 |
| --- | --- |
| 分层键 | `网元 × 版本`，如 `UDG@20.15.2`。每层是一份完整特性层资产集，跨版本/跨网元当下不去重合并，演进靠 diff。 |
| 实例键 | 节点对象统一 `网元@版本@对象类型@本地名`，对象类型进 ID 确保跨类型唯一。 |
| 本地名 | 优先使用源头稳定编号；没有稳定编号的对象才使用层内同类型顺序号。 |
| 跨层锚点 | Feature 使用正式特性编号作为 local_name；ConfigTask 仍通过任务层的 `task_logical_name` 做语义路由。 |

实例键示例：

| 对象 | ID 示例 | local_name 来源 |
| --- | --- | --- |
| `Feature` | `UDG@20.15.2@Feature@GWFD-020301` | 正式特性编号 |
| `SubFeature` | `UDG@20.15.2@SubFeature@GWFD-020301:URL识别计费` | 所属 Feature + 细分语义名 |
| `License` | `UDG@20.15.2@License@LKV3G5BCBC01` | License code |
| `FeatureRule` | `UDG@20.15.2@FeatureRule@00001` | 层内同类型顺序号 |
| `DecisionPoint` | `UDG@20.15.2@DecisionPoint@00001` | 对齐任务层 DecisionPoint 规则 |

关系不建成对象，不使用四段式对象实例键。边表按关系管理，稳定键由以下字段共同确定：

```text
source_id + relation_type + target_id
```

如需落 JSONL 表，可额外提供 `edge_id` 作为工程去重键，但它不是图谱对象 ID。

边表通用字段：

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `source_id` | string | 是 | 源节点四段式实例键 |
| `relation_type` | string | 是 | 关系类型，如 `requires_license` |
| `target_id` | string | 是 | 目标节点四段式实例键 |
| `nf` | string | 是 | 网元，如 `UDG` |
| `version` | string | 是 | 版本，如 `20.15.2` |
| `edge_id` | string | 否 | 工程去重键，建议由 `source_id + relation_type + target_id` 派生 |
| `source_evidence_ids` | string[] | 否 | 关系证据 |

## 2. 特性层保留对象

### 2.1 Feature

`Feature` 是产品正式能力本体。

只要满足下面任一强信号，优先建成 `Feature`：

- 有正式特性编号，如 `GWFD-*`、`WSFD-*`、`IPFD-*`
- 有独立特性文档
- 有独立 License 或版本发布边界
- 客户或工程交付可识别为一个正式能力

判定原则：

- 特性边界服从产品定义
- 有正式编号的能力不要降级为 `SubFeature`
- 即使语义上像某能力的细分，只要产品给了正式编号，也仍是 `Feature`

例子：

- `GWFD-020301 内容计费基本功能` 是 `Feature`
- `GWFD-020302 基于业务时长的计费` 是 `Feature`
- `GWFD-020303 基于业务流量的计费` 是 `Feature`
- `GWFD-020306 支持事件计费` 是 `Feature`
- `GWFD-020403 IPv4v6 双栈` 是 `Feature`
- `WSFD-106203 别名 APN` 是 `Feature`

`Feature` 第一版字段分三类：原始 md 字段、抽取/归一化字段、来源/运行上下文字段。

原始 md 字段（概述 md 固定 13 节，全部原文保留）：

| 字段 | 类型 | 必填 | 对应 md 节 | 说明 |
| --- | --- | --- | --- | --- |
| `applicable_nf_raw` | string | 是 | 适用NF | 原始 NF 列表 |
| `definition_raw` | string | 是 | 定义 | 原始定义全文 |
| `customer_value_raw` | string | 否 | 客户价值 | 原文（含受益方表格） |
| `application_scenario_raw` | string | 否 | 应用场景 | 原文 |
| `availability_raw` | string | 否 | 可获得性 | 原文（含 NF 表 + License 声明） |
| `feature_interaction_raw` | string | 否 | 与其他特性的交互 | 原文（依赖表） |
| `system_impact_raw` | string | 否 | 对系统的影响 | 原文 |
| `restrictions_raw` | string | 否 | 应用限制 | 原文 |
| `principle_raw` | string | 否 | 原理概述 | 原文 |
| `charging_raw` | string | 否 | 计费与话单 | 原文 |
| `spec_raw` | string | 否 | 特性规格 | 原文 |
| `standards_raw` | string | 否 | 遵循标准 | 原文 |
| `release_history_raw` | string | 否 | 发布历史 | 原文 |

抽取/归一化字段：

| 字段 | 类型 | 必填 | 来源 | 说明 |
| --- | --- | --- | --- | --- |
| `id` | string | 是 | 派生 | 四段式实例键 `{nf}@{version}@Feature@{feature_code}` |
| `feature_code` | string | 是 | xlsx/文件名 | 正式特性编号，如 `GWFD-020301` |
| `name` | string | 是 | xlsx/文件名 | 特性名 |
| `is_directory` | bool | 是 | xlsx | 目录特性 / 叶子特性 |
| `catalog_section` | string | 否 | xlsx section | 清单分区，如 `年费基本包-智能策略控制&计费基本包` |
| `parent_feature_code` | string | 否 | xlsx parent_feature_id | 目录父，作为 `catalog_parent` 表达，不当 `depends_on` |
| `applicable_nf` | string[] | 否 | `applicable_nf_raw` 归一化 | 归一化后 NF 数组，如 `[SGW-U, PGW-U, UPF]` |
| `nf_support_map` | string | 否 | xlsx 代际列 | 代际支持矩阵，如 `GGSN(2G&3G)=M; …` |
| `first_release_version` | string | 否 | `release_history_raw` 解析 | 首次发布版本，如 `20.0.0` |
| `standards` | object[] | 否 | `standards_raw` 解析 | 结构化标准数组 `[{category,number,name}]` |
| `feature_category` | enum | 否 | 推断 | `base`/`enhanced`/`integration`/`operations`/`protocol` |
| `config_relevance` | enum | 否 | 推断 | `required`/`indirect`/`none`/`ops_only` |

来源/运行上下文字段：

| 字段 | 类型 | 必填 | 来源 | 说明 |
| --- | --- | --- | --- | --- |
| `nf` | string | 是 | pipeline | 图谱分层网元，如 `UDG` |
| `version` | string | 是 | pipeline | 图谱分层版本，如 `20.15.2` |
| `source_path` | string | 是 | 文件系统 | 概述 md 相对路径 |
| `has_overview` | enum | 是 | 状态 | `yes`/`no_overview`/`empty`/`no_docs`/`file_missing` |

`feature_category` 与 `config_relevance` 口径（第一版推断，待 ConfigTask 接入后校准）：

| 枚举 | 含义 |
| --- | --- |
| `feature_category=base` | 基础能力：架构、协议底座，如云化分布式、SA-Basic |
| `feature_category=enhanced` | 增强能力：基础上的扩展，如内容计费、智能 Shaping |
| `feature_category=integration` | 集成能力：跨特性组合或第三方接入，如 SFIP |
| `feature_category=operations` | 运维能力：可靠性、扩缩容、自愈，如 SFFD-* 系列 |
| `feature_category=protocol` | 协议能力：协议接入、地址分配 |
| `config_relevance=required` | 需要配置：有明确激活/配置文档 |
| `config_relevance=indirect` | 间接配置：经其他特性配置触发 |
| `config_relevance=none` | 无需配置：协议底座或自动生效 |
| `config_relevance=ops_only` | 仅运维触发：License、可靠性类 |

`feature_category` 第一版以 xlsx `catalog_section` 为主锚点 + 定义/关键词辅助推断；`config_relevance` 结合激活文档存在性与"无需配置"声明推断。

关系不入 `Feature` 节点，单出边表（对齐 §1A）：

| 边表 | 来源 md 节 | 关系 |
| --- | --- | --- |
| `feature_depends_on.jsonl` | 与其他特性的交互 | `depends_on` |
| `feature_conflicts_with.jsonl` | 与其他特性的交互 | `conflicts_with`（带 `conflict_pair_id` 对称去重） |
| `feature_requires_license.jsonl` | 可获得性 | `requires_license`（`source_type=feature_overview`，License 节点复用 `licenses.jsonl`） |

弱语义依赖（`affects`/`interacts_with`/`supports`/`other`）不直接进核心边表，先进 `feature_relation_candidates.jsonl` 待审。

### 2.2 SubFeature

`SubFeature` 是单个正式 `Feature` 内部的稳定细分能力。

只有同时满足以下条件才建：

- 没有正式特性编号
- 隶属于某个 `Feature`
- 需要被方案、规则、task 或决策点单独引用
- 具有稳定语义边界

不满足这些条件的轻量差异，不建 `SubFeature`，放回 `Feature` 的差异维度或交给 `DecisionPoint`。

适合建 `SubFeature` 的例子：

- 内容计费里的 URL 识别计费
- 内容计费里的 L34 识别计费
- 内容计费里的 any 兜底计费
- 某个正式特性内部需要单独挂 task 的协议形态

不适合建 `SubFeature` 的例子：

- 只是参数枚举不同
- 只是 License 编号不同
- 只是同一配置骨架下的场景变体
- 已经有正式特性编号的能力

### 2.3 License

`License` 表达产品能力的授权控制项。

在特性层里，`License` 不是 License 文件，不是 LAC，不是 ESN，也不是运行状态；它对应产品文档中的 **License 控制项**。

因此第一版只保留两类数据：

| 数据 | 说明 | 输出 |
| --- | --- | --- |
| License 控制项表 | 每行一个 License 控制项 | `licenses.jsonl` |
| Feature-License 边表 | 每行一条 Feature 需要某 License 的关系 | `feature_requires_license.jsonl` |

暂不单独建立 License 控制项之间的关系表。控制项文档里的“相关控制项”先作为 `related_control_items_raw` 原文保留，不解释成关系。

它只回答：

- 哪个能力需要哪个 License
- 哪些 Feature 共享某个 License
- License 归属哪个产品范围和 NF
- License 是功能开关还是资源控制项
- License 的取值范围和默认值是什么

它不回答：

- `SET LICENSESWITCH` 怎么配置
- License 命令参数如何填写
- License 文件如何申请、校验和激活
- LAC、ESN、License 文件类型和运行状态如何建模

这些由任务层和命令层承接。

无 License 的能力不强制建伪 License 节点，可在 `requires_license` 关系上表达“无独立 License”。

`License` 第一版只定义一个节点表。字段分三类：原始 md 字段、抽取/归一化字段、来源/运行上下文字段。

原始 md 字段：

| 字段 | 类型 | 必填 | 对应 md 位置 | 说明 |
| --- | --- | --- | --- | --- |
| `source_heading_raw` | string | 是 | 小节标题 | 原始标题，如 `#### [82209822 LKV3G5BCBC01 内容计费基本功能]` |
| `applicable_nf_raw` | string | 否 | 表格行：`归属NF` / `适用NF` | 原始 NF 单元格值，如 `SGW-U、PGW-U、UPF`、`ALL`、空值 |
| `description_raw` | string | 否 | 表格行：`功能描述` | 原始功能描述 |
| `implementation_description_raw` | string | 否 | 表格行：`实现描述` | 原始实现描述；抽取时需容错 `实现描述HTTP头增强` 这类脏键 |
| `value_range_raw` | string | 否 | 表格行：`取值范围` | 原始取值范围 |
| `default_value_raw` | string | 否 | 表格行：`默认值` | 原始默认值 |
| `feature_refs_raw` | string | 否 | 表格行：`对应特性` | 原始对应特性单元格值，如 `GWFD-020301 内容计费基本功能`、`不涉及` |
| `related_control_items_raw` | string | 否 | 表格行：`相关控制项` | 原始相关控制项单元格值，第一版不解释成关系 |
| `application_scenario_raw` | string | 否 | 表格行：`应用场景` | 原始应用场景 |

抽取/归一化字段：

| 字段 | 类型 | 必填 | 来源 | 说明 |
| --- | --- | --- | --- | --- |
| `id` | string | 是 | 派生 | 四段式实例键，规则 `{nf}@{version}@License@{license_code}` |
| `control_item_id` | string | 是 | 从 `source_heading_raw` 解析 | License 控制项编号，如 `82209822` |
| `license_code` | string | 是 | 从 `source_heading_raw` 解析 | License 代码，如 `LKV3G5BCBC01` |
| `name` | string | 是 | 从 `source_heading_raw` 解析 | License 控制项名称 |
| `license_domain` | string | 是 | 从目录/文件名解析 | 控制项所属域，如 `UDG`、`VAS`、`SFIP`、`Service Fabric` |
| `control_item_type` | enum | 是 | 从文件名解析 | `function` 或 `resource` |
| `applicable_nf` | string[] | 否 | 从 `applicable_nf_raw` 归一化 | 归一化后的 NF 数组 |
| `feature_refs` | string[] | 否 | 从 `feature_refs_raw` 解析 | 解析出的特性编号，如 `GWFD-020301`、`SFFD-010004` |

来源/运行上下文字段：

| 字段 | 类型 | 必填 | 来源 | 说明 |
| --- | --- | --- | --- | --- |
| `nf` | string | 是 | pipeline 参数 | 图谱分层网元，如 `UDG` |
| `version` | string | 是 | pipeline 参数 | 图谱分层版本，如 `20.15.2` |
| `source_path` | string | 是 | 文件系统 | 来源 md 路径 |

字段和原始 md 的映射：

| 原始位置 | 原始字段 | schema 字段 | 说明 |
| --- | --- | --- | --- |
| 小节标题 | `#### [82209822 LKV3G5BCBC01 内容计费基本功能]` | `source_heading_raw` + `control_item_id` / `license_code` / `name` | 标题原文保留，同时拆三段 |
| 目录/文件名 | `UDG License控制项/资源控制项` | `license_domain` / `control_item_type` | `资源控制项` -> `resource`，`功能控制项` -> `function` |
| 表格 | `归属NF` / `适用NF` | `applicable_nf_raw` / `applicable_nf` | 先保留原文，再解析数组 |
| 表格 | `功能描述` | `description_raw` | 原文保留 |
| 表格 | `实现描述` | `implementation_description_raw` | 原文保留；抽取时需容错 `实现描述HTTP头增强` 这类脏键 |
| 表格 | `取值范围` | `value_range_raw` | 原文保留 |
| 表格 | `默认值` | `default_value_raw` | 原文保留 |
| 表格 | `对应特性` | `feature_refs_raw` / `feature_refs` | 先保留原文，再解析 `GWFD-*` / `SFFD-*` 等编号 |
| 表格 | `相关控制项` | `related_control_items_raw` | 第一版不解释成边 |
| 表格 | `应用场景` | `application_scenario_raw` | 原文保留 |

审查结论：

- 原始控制项表格稳定字段是 8 个：`归属NF/适用NF`、`功能描述`、`实现描述`、`取值范围`、`默认值`、`对应特性`、`相关控制项`、`应用场景`。
- `version` 已表达图谱分层版本，不再保留重复的 `product_version`。
- `license_domain` 和 `applicable_nf` 不是一回事：前者来自目录域，后者来自表格中的归属 NF。
- `source_heading_raw` 与 `control_item_id/license_code/name` 有信息重叠，但它是证据追溯字段，保留原文不参与业务建模。
- 每个 License 小节表格里的 8 行都有原文落点；抽取字段只从这些原文落点派生。

### 2.4 FeatureRule

`FeatureRule` 是能力层规则。

它约束的是能力本身、能力组合、能力边界、能力到 task 的使用方式。

适合归入 `FeatureRule` 的规则：

- Feature 之间的强依赖
- Feature 之间的互斥
- Feature 之间的协同关系
- License 组合约束
- UDG/UNC 或 C-U 协同约束
- 能力边界限制
- 纯描述性底座定位
- Feature 选择哪些 task 的规则
- 同一正式 Feature 内部的语义反转或侧别差异

不归入 `FeatureRule` 的规则：

- task 内跨命令绑定规则，归 `TaskRule`
- task 内参数固定/引用/条件出现，归 `TaskRule` 或 `DecisionPoint`
- 单命令语法、参数依赖、删除保护，归 `CommandRule`
- 业务方案选择原则，归 `BusinessRule` 或业务层 `DecisionPoint`

### 2.5 FeatureTaskOrder

`FeatureTaskOrder` 表达某个 `Feature` 或 `SubFeature` 下 task 的稳定展开顺序。

它不是独立节点对象，也不单独建边对象；第一版直接作为 `decomposes_to` 边表上的属性管理。

它只编排 `ConfigTask`，不编排命令。

适合表达：

- task A 通常先于 task B
- task B 依赖 task A 的输出
- 某个 task 是 fallback
- 某个 task 必须最后执行
- 某条 task 边受特性层决策影响

不适合表达：

- task 内命令顺序
- 命令之间参数传递
- 单命令执行前置

这些属于任务层或命令层。

### 2.6 DecisionPoint

`DecisionPoint` 是跨层通用对象，不是特性层独占对象。

当一个选择点发生在能力层，才挂在 `Feature / SubFeature` 上。

适合挂在特性层的决策：

- 选择哪个 SubFeature
- 选择哪组 ConfigTask
- 选择哪条能力分支
- 选择哪种 Feature 组合
- 选择 C/U 侧哪种协同模式

不适合挂在特性层的决策：

- 只影响某个 task 内参数值
- 只影响某个 task 内命令组合
- 只影响命令参数条件必选

这些下沉到任务层或命令层。

## 3. 特性层关系

### 3.1 has_subfeature

```text
Feature -> SubFeature
```

表示正式特性内部包含一个稳定细分能力。

使用条件：

- `SubFeature` 无正式特性编号
- 该细分能力需要被单独引用

禁止用它表达正式特性之间的父子关系。正式编号之间的关系应使用 `depends_on`、`cooperates_with` 或产品目录父子字段。

### 3.2 depends_on

```text
Feature / SubFeature -> Feature / SubFeature
```

表示没有目标能力，源能力不成立或无法完整工作。

适合表达：

- 能力前置
- 协议底座前置
- 控制面/用户面依赖
- License 能力链的功能依赖
- 增强特性依赖基础特性

例子：

```text
内容计费 -> SA-Basic
业务级 FUP -> SA-Basic
智能 Shaping -> BWM
L2TP U 侧执行 -> L2TP C 侧决策
```

### 3.3 conflicts_with

```text
Feature / SubFeature -> Feature / SubFeature
```

表示两个能力不能同时应用于同一范围或同一配置目标。

适合表达：

- 地址分配方式互斥
- 隧道方式互斥
- 同一控制目标上的机制冲突
- 产品明确限制不能并用

互斥是逻辑对称关系；存储时可以保留双向证据，也可以用同一 pair id 去重展示。

### 3.4 cooperates_with

```text
Feature / SubFeature -> Feature / SubFeature
```

表示两个能力需要协同，但不是严格前置依赖。

适合表达：

- UDG/UNC 配对使用
- C-U 分工协同
- 同一业务方案中经常组合出现
- 一个能力负责决策，另一个能力负责执行

如果缺少目标能力会导致源能力不可用，应使用 `depends_on`，不要使用 `cooperates_with`。

### 3.5 requires_license

```text
Feature / SubFeature -> License
```

表示能力受 License 控制项门控。

允许表达：

- 一个 Feature 需要一个 License
- 一个 Feature 需要多个 License
- 多个 Feature 共用一个 License
- C/U 侧 License 不对称
- 某个 Feature 无独立 License

License 开启动作本身不是特性层关系，交给 `ConfigTask`。

`requires_license` 建议字段如下：

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `source_id` | string | 是 | 源 Feature / SubFeature 四段式实例键 |
| `relation_type` | string | 是 | 固定为 `requires_license` |
| `target_id` | string | 是 | 目标 License 四段式实例键 |
| `nf` | string | 是 | 网元，如 `UDG` |
| `version` | string | 是 | 版本，如 `20.15.2` |
| `feature_code` | string | 是 | 正式特性编号冗余，便于检索，如 `GWFD-020301` |
| `license_code` | string | 是 | License 代码冗余，便于检索 |
| `control_item_id` | string | 否 | License 控制项编号 |
| `edge_id` | string | 否 | 工程去重键，由 `source_id + relation_type + target_id` 派生 |
| `source_type` | enum | 是 | 来源类型：`feature_overview`、`license_doc` 或 `merged` |
| `source_evidence_ids` | string[] | 否 | 证据文档路径 |
| `confidence` | enum | 否 | 抽取置信度 |

### 3.6 decomposes_to

```text
Feature / SubFeature -> ConfigTask
```

表示能力通常拆成哪些可复用配置任务。

使用原则：

- Feature 不直接连 MMLCommand
- Feature 不直接连 ConfigObject
- Feature 只通过 `decomposes_to` 把能力交给任务层
- 一个 ConfigTask 可以被多个 Feature 复用
- 纯描述性 Feature 可以没有 `decomposes_to`

如果需要表达 Feature 下多个 task 的展开顺序，直接放在 `decomposes_to` 边属性上：

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `source_id` | string | 是 | Feature / SubFeature 四段式实例键 |
| `relation_type` | string | 是 | 固定为 `decomposes_to` |
| `target_id` | string | 是 | ConfigTask 四段式实例键 |
| `task_order` | integer | 否 | 在该 Feature 下的 task 展开顺序 |
| `order_role` | enum | 否 | `normal`、`first`、`last`、`fallback` |
| `source_evidence_ids` | string[] | 否 | 证据路径 |

### 3.7 constrained_by

```text
Feature / SubFeature -> FeatureRule
```

表示能力受特性层规则约束。

如果规则只约束 task 内部命令协同，不应挂这里。

### 3.8 constrains_task

```text
FeatureRule -> ConfigTask
```

表示某条特性规则影响某个 task 的使用方式。

适合表达：

- 某 FeatureRule 决定是否选择某 task
- 某 FeatureRule 限制某 task 的适用范围
- 某 FeatureRule 要求 task 组合使用
- 某 FeatureRule 规定 task 的侧别使用方式

### 3.9 has_decision

```text
Feature / SubFeature -> DecisionPoint
```

表示某个决策发生在能力层。

典型场景：

- 选择 Feature 内部哪个 SubFeature
- 选择哪组 task
- 选择 UDG/UNC 哪种协同模式
- 选择是否进入某个增强能力链

### 3.10 uses_semantic_object

```text
Feature / SubFeature -> SemanticObject
```

表示能力承接某类业务语义对象。

它是业务语义到能力世界的桥，不是配置对象引用。

禁止把 `ConfigObject` 伪装成 `SemanticObject` 挂在特性层。

### 3.11 supported_by

```text
Feature / SubFeature / FeatureRule / License -> EvidenceSource
```

表示特性层对象由原始证据支撑。

证据可以来自：

- 特性概述
- 配置/部署/激活文档
- 跨特性分析
- 产品清单
- License 信息
- 设计修正记录

## 4. Feature 与 SubFeature 判定

### 4.1 优先建 Feature 的情况

满足以下情况优先建 `Feature`：

- 有正式特性编号
- 有独立发布记录
- 有独立 License
- 有独立特性概述或配置文档
- 产品清单中独立列项
- 工程交付中作为独立能力开通

### 4.2 优先建 SubFeature 的情况

满足以下情况才建 `SubFeature`：

- 无正式特性编号
- 属于某个 Feature 内部
- 不是一次性配置场景
- 需要单独挂 task、规则、决策或方案引用

### 4.3 留在 variant_dimensions 的情况

以下差异优先留在 `variant_dimensions`：

- 参数枚举差异
- 支持的协议类型列表
- 地址类型差异
- 网元角色差异但不需要单独挂 task
- 场景名称差异但配置骨架一致
- 文档中的示例场景

### 4.4 交给 DecisionPoint 的情况

以下差异优先交给 `DecisionPoint`：

- 选择不同 task 分支
- 同一 task 骨架下命令组合变化
- 同一 task 下参数值模式变化
- 是否跳过某个 task
- 是否进入某个增强链路

## 5. FeatureRule 归口

### 5.1 应进入 FeatureRule

| 规则类型 | 说明 |
| --- | --- |
| `dependency_rule` | 能力前置、能力链依赖 |
| `license_rule` | License 组合、License 侧别、License 不对称 |
| `task_selection_rule` | 特性选择哪些 task 或跳过哪些 task |
| `nf_cooperation_rule` | UDG/UNC/C-U 协同分工 |
| `consistency_rule` | 跨 Feature 或跨 NF 的一致性 |
| `restriction_rule` | 互斥、禁止组合、能力限制 |
| `scope_rule` | 能力适用范围、纯描述性底座定位 |
| `naming_rule` | 特性层命名、侧别、语义反转 |
| `validation_rule` | 能力级校验，不是单命令语法 |

### 5.2 不应进入 FeatureRule

| 规则 | 归口 |
| --- | --- |
| 命令参数条件必选 | CommandRule |
| 命令参数取值范围 | CommandRule |
| 命令删除保护 | CommandRule |
| task 内对象绑定必须同时出现 | TaskRule |
| task 内多命令引用一致性 | TaskRule |
| task 内参数固定值/引用值分类 | ConfigTask 参数绑定或 TaskRule |
| 业务方案选择规则 | BusinessRule / 业务层 DecisionPoint |

---

## 6. 禁止关系

| 禁止关系 | 原因 | 替代 |
| --- | --- | --- |
| `Feature -> MMLCommand` | 跳过任务层 | `Feature -> ConfigTask -> MMLCommand` |
| `Feature -> CommandParameter` | 参数属于命令层 | `DecisionPoint -> CommandParameter` 或 `ConfigTask.commands.parameters` |
| `Feature -> ConfigObject` | 配置对象属于命令层 | `Feature -> ConfigTask -> targets ConfigObject` |
| `FeatureRule -> CommandParameter` | 特性规则过度下沉 | `TaskRule` 或 `CommandRule` |
| `decomposes_to` 边属性直接编排 MMLCommand | Feature 只能编排 task，不能编排命令 | task 内命令顺序归任务层 |
| `SubFeature` 代替正式编号 Feature | 破坏产品能力边界 | 建 `Feature` |
| `Feature` 承载 task 内命令列表 | 与任务层重复 | `Feature decomposes_to ConfigTask` |
| `Feature` 承载配置对象层级 | 与命令层重复 | `ConfigObject` 层内关系 |

## 7. 和任务层、命令层的分工

### 7.1 特性层负责

- 正式能力边界
- 能力依赖
- License 门控
- 能力级规则
- Feature 到 ConfigTask 的分解
- Feature 内 task 编排
- 能力层决策

### 7.2 任务层负责

- 最小可复用配置动作
- task 的命令组
- task 内参数绑定分类
- task 内规则
- task 内决策
- task 内命令顺序

### 7.3 命令层负责

- MMLCommand
- CommandParameter
- ConfigObject
- CommandRule
- 参数依赖
- 对象关系
- 命令语法与执行约束

---

## 8. 当前 v1 结论

特性层的最小稳定定义是：

```text
Feature / SubFeature
  --depends_on / conflicts_with / cooperates_with--> Feature / SubFeature
  --requires_license--> License
  --constrained_by--> FeatureRule
  --decomposes_to--> ConfigTask
  --decomposes_to(task_order)--> ConfigTask
  --has_decision--> DecisionPoint
  --uses_semantic_object--> SemanticObject
  --supported_by--> EvidenceSource
```

其中：

- `Feature` 服从产品正式编号和能力边界
- `SubFeature` 只承接无正式编号但需独立引用的稳定细分能力
- `FeatureRule` 只承接能力级规则
- task 顺序作为 `decomposes_to` 边属性管理，不单独建边对象
- 命令、参数、配置对象不在特性层重复建模
