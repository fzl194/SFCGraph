## 1. 概述

### 1.1 特性定义（是什么）

为了解决日趋拥塞的网络带宽，针对拥塞小区的带宽控制，华为提出了支持**基于初始接入位置的策略控制**方案。

基于初始接入位置的策略控制方案，是指根据用户**激活时的位置信息（ULI, User Location Information）**，匹配对应的带宽控制策略，并由 GGSN-U/PGW-U/UPF 限制用户带宽。例如对拥塞小区进行流控和业务限制，限制整个小区的 P2P 总流量等。

**核心机制（产品文档原文）**：用户激活过程中，GGSN-C/PGW-C/SMF 获取 ULI 信元，并根据 ULI 信元分配/获取带宽管理策略：
- 若 SGSN/MME/AMF **未上报**位置信息，则从上下文中获取 SGSN/MME/AMF 的 **MCC 和 MNC** 作为初始接入位置信息（位置兜底机制）
- 组网中**有** PCRF/PCF（动态规则 & 预定义规则）：PCRF/PCF 根据 GGSN-C/PGW-C/SMF 携带的初始接入位置信息决策并下发带宽策略
- 组网中**无** PCRF/PCF：GGSN-C/PGW-C/SMF 根据初始接入位置信息决策并下发带宽策略（本地PCC）

用户访问业务过程中，GGSN-U/PGW-U/UPF 根据带宽管理策略，管控用户的带宽，防止资源紧张时某些业务过度占用资源。

## 3. 原理

### 3.1 实现原理：两种 PCC 模式（动态 PCC + 本地 PCC）

系统支持**动态 PCC**和**本地 PCC**两种管控模式，这是本特性的核心变体维度：

| 模式 | 决策方 | 接口 | 适用场景 | UNC 侧配置 |
|------|--------|------|----------|-----------|
| **动态 PCC** | PCRF/PCF | Gx（4G）/ N7（5G） | 组网中有 PCRF/PCF（动态规则 & 预定义规则） | 动态规则：仅 License；预定义规则：需配置位置信息 |
| **本地 PCC** | GGSN-C/PGW-C/SMF 本地 | PFCP | 组网中无 PCRF/PCF，或回滚到本地 PCC | 需配置位置信息 + 位置组 + 绑定链 |

**关键区分（预定义规则 vs 动态规则）**：
- **动态规则**：PCRF/PCF 决策并下发完整的带宽策略（含位置匹配），UNC 仅透传 ULI，**不需要 UNC 配置位置信息**
- **预定义规则**：PCRF/PCF 下发预定义规则名，UNC 根据本地配置的位置信息匹配并激活对应规则，**需要 UNC 配置位置信息**

> SourcePath: `实现原理_27915137.md` §动态PCC/本地PCC
> SourcePath: `激活基于初始接入位置的策略控制_27915138.md` §操作场景

### 3.2 ULI（User Location Information）位置信息

用户初始接入位置信息，由 SGSN/MME/AMF 在用户激活（PDP 一次上下文激活/Attach）时上报给 GGSN-C/PGW-C/SMF。位置类型按制式区分：

| 位置类型 | 适用制式 | 关键字段 | 说明 |
|----------|----------|----------|------|
| **CGI** | 2G/3G | MCC + MNC + LAC + CI | Cell Global Identifier，需规划 LAC + CI |
| **ECGI** | 4G | MCC + MNC + ECI | E-UTRAN Cell Global Identifier，需规划 ECI |
| **NCGI** | 5G | MCC + MNC + NCI | NR Cell Global Identifier，需规划 NCI |

**位置兜底机制**：如果 SGSN/MME/AMF **没有上报**位置信息，则从上下文中获取 SGSN/MME/AMF 的 **MCC 和 MNC** 作为初始接入位置信息。

> SourcePath: `特性概述_27915136.md` §原理概述
> SourcePath: `激活基于初始接入位置的策略控制_27915138.md` §表1

### 3.3 位置组（Location Group）与绑定链

将多条用户位置信息绑定到一个位置组（USRLOCATIONGRP），便于按区域（如 CBD 商圈、某簇小区）批量下发策略。一个位置组可包含多条 USRLOCATION 记录。

**位置组绑定链（4级绑定）**：

```
USRLOCATION（单个位置，CGI/ECGI/NCGI）
    │ ADD USRLOCATIONGRP（多条位置 → 一个位置组）
    ▼
USRLOCATIONGRP（位置组，如 locgrp_test）
    │ MOD/ADD UPBINDUPG（位置组 + 用户模板 → 用户模板组）
    ▼
UPBINDUPG（用户模板组绑定，USERPROFGNAME ↔ USERPROFILENAME ↔ LOCGROUPNAME）
    │ ADD APNUSRPROFG（用户模板组 → APN/DNN）
    ▼
APNUSRPROFG（APN 与用户模板组绑定）
    │ 用户激活上报 ULI
    ▼
USERPROFILE（用户模板，绑定带宽规则，规则来自 WSFD-211005）
    │
    ▼
带宽规则激活 → UNC 下发策略 → PGW-U/UPF 执行带宽限制
```

> SourcePath: `激活基于初始接入位置的策略控制_27915138.md` §操作步骤

### 3.4 灵活时间机制（分时段管控）

规则中可携带**时间范围**（如 9:00-11:00 忙时），PGW-C/GGSN-C/SMF 根据灵活时间的机制决定**何时**向 PGW-U/GGSN-U 下发带宽控制规则。实现分时段的带宽管控。

**示例**：9:00-11:00 忙时对 CBD 区域基于小区控制 P2P 业务的带宽。

### 3.5 关键信元与协议交互（Gx/N7/PFCP）

#### 3.5.1 Gx 接口信元（4G，GGSN-C/PGW-C ↔ PCRF）

| 接口 | 方向 | 关键信元（AVP） | 说明 |
|------|------|-----------------|------|
| Gx | GGSN-C/PGW-C → PCRF | **3GPP-User-Location-Info AVP**（CCR-Initial 消息） | 携带用户初始接入位置信息（CGI/ECGI） |
| Gx | PCRF → GGSN-C/PGW-C | 带宽控制策略（CCA-Initial 消息） | PCRF 根据位置匹配决策带宽策略并下发 |

#### 3.5.2 N7 接口（5G，SMF ↔ PCF）

与 Gx 接口类似（产品文档明确），SMF 通过 N7 接口与 PCF 交互，5G 场景下 ULI 信元为 NCGI。

#### 3.5.3 PFCP 接口（UNC → UDG，策略下发）

| 接口 | 方向 | 关键信元 | 说明 |
|------|------|----------|------|
| PFCP | PGW-C/GGSN-C/SMF → PGW-U/GGSN-U | **PFCP Session Establishment Request**（携带带宽控制策略） | UNC 将决策的带宽策略下发到 UDG 执行 |
| PFCP | PGW-U/GGSN-U → PGW-C/GGSN-C/SMF | PFCP Session Establishment Response | UDG 安装规则后返回响应 |

> SourcePath: `实现原理_27915137.md` §动态PCC 步骤1.b~1.e

### 3.6 业务流程（动态 PCC + 本地 PCC 双模式）

#### 3.6.1 动态 PCC 业务流程（Gx/N7 接口）

```
┌─────────────────────────────────────────────────────────────────┐
│ 1. 获取并安装基于初始接入位置的带宽控制策略                        │
│    a. 用户接入网络                                                │
│       MME/SGSN → PGW-C/GGSN-C：Create PDP Context Request /      │
│       Create Session Request                                     │
│    b. PGW-C/GGSN-C → PCRF：CCR-Initial 请求消息                   │
│       消息携带 3GPP-User-Location-Info AVP（关键信元）             │
│    c. PCRF 根据位置信息匹配用户所在服务区                          │
│       决策使用的带宽控制策略                                       │
│       PCRF → PGW-C/GGSN-C：CCA-Initial 消息（下发策略）           │
│    d. PGW-C/GGSN-C → PGW-U/GGSN-U：PFCP Session Establishment    │
│       Request（携带带宽控制策略）                                  │
│       ※ 规则携带时间范围时，PGW-C/GGSN-C 根据灵活时间机制         │
│         决定何时下发带宽控制规则（分时段管控）                     │
│    e. PGW-U/GGSN-U 安装规则 → PFCP Session Establishment         │
│       Response                                                   │
│    f. PGW-C/GGSN-C → MME/SGSN：Create PDP Context Response /     │
│       Create Session Response                                   │
└──────────────────────────────┬──────────────────────────────────┘
                               │
                               ▼
┌─────────────────────────────────────────────────────────────────┐
│ 2. 对上行报文进行带宽控制                                         │
│    a. 用户访问业务，向 PGW-U/GGSN-U 发送 Traffic Flow             │
│    b. PGW-U/GGSN-U 根据安装的带宽规则进行带宽控制                 │
│       - 超出允许值 → 缓存/丢弃超出报文                            │
│       - 未超出允许值 → 正常处理后转发                             │
└──────────────────────────────┬──────────────────────────────────┘
                               │
                               ▼
┌─────────────────────────────────────────────────────────────────┐
│ 3. 对下行报文进行带宽控制                                         │
│    a. 服务器向 PGW-U/GGSN-U 发送用户的业务报文                    │
│    b. PGW-U/GGSN-U 根据带宽规则控制                              │
│       - 超出允许值 → 缓存/丢弃                                    │

```

#### 3.6.2 本地 PCC 业务流程（PFCP 接口，无 PCRF/PCF）

```
┌─────────────────────────────────────────────────────────────────┐
│ 1. 获取并安装基于初始接入位置的带宽控制策略                        │
│    a. 用户接入网络                                                │
│       MME/SGSN → PGW-C/GGSN-C：Create PDP Context Request /      │
│       Create Session Request                                     │
│    b. PGW-C/GGSN-C 根据 3GPP-User-Location-Info AVP              │
│       【本地决策】使用的带宽策略（关键差异：无 PCRF/PCF 交互）     │
│       匹配 USRLOCATION/USRLOCATIONGRP/UPBINDUPG/APNUSRPROFG 链路 │
│       PGW-C/GGSN-C → PGW-U/GGSN-U：PFCP Session Establishment    │

```

### 3.7 访问限制场景中的角色（三轨体系下的定位）

在访问限制场景（NS-03）中，WSFD-211001 作为**基于位置的访问/带宽限制触发特性（UNC侧）**，属于 **轨道A（PCC体系）的位置条件维度**：

- **位置条件触发**：以用户激活时的初始接入位置（CGI/ECGI/NCGI）作为策略匹配条件
- **策略下发链路**：通过 GGSN-C/PGW-C/SMF → PCRF/PCF（动态PCC）或本地决策（本地PCC）→ PGW-U/UPF，激活对应的访问限制/带宽控制规则
- **分时段管控**：结合灵活时间机制（如 9:00-11:00 忙时），按时间+位置双维度触发限制

**如何触发访问限制动作**：

```
位置条件（USRLOCATION + USRLOCATIONGRP）
  ↓ 用户激活上报 ULI
匹配用户所在服务区/小区（CGI/ECGI/NCGI）
  ↓
绑定用户模板组（UPBINDUPG: USERPROFGNAME ↔ USERPROFILENAME ↔ LOCGROUPNAME）
  ↓
绑定APN/DNN（APNUSRPROFG）
  ↓
用户模板(USERPROFILE)绑定的规则被激活
  ↓
UNC(SMF/PGW-C) 下发策略 → PGW-U/UPF 执行访问限制动作
（动作类型由USERPROFILE绑定的规则定义：带宽限制、阻塞、重定向等，见WSFD-211005）
```

**与 UDG 侧重定向/阻塞特性的串联关系**：
- 本特性在 UNC(SMF/PGW-C) 侧完成位置匹配与策略决策
- 下发的策略由 PGW-U/UPF（UDG侧）执行
- 可与 UDG 侧重定向（GWFD-110281 用户Portal / GWFD-110282 WebProxy / GWFD-110284 HTTP智能重定向）、阻塞、ADC（GWFD-020357）等访问限制动作串联

## 6. 验证与调测

### 6.1 验证方法

#### 6.1.1 调测前提与目的

当运营商需要部署基于初始接入位置的策略控制功能，以实现对拥塞的小区进行带宽控制时，需对 UNC 的基于初始接入位置的策略控制功能进行调测，确保本功能可以正常使用。

> 适用：GGSN-C/PGW-C/SMF（UNC 控制面）

#### 6.1.2 调测数据准备

| 类别 | 参数名称 | 取值样例 | 获取方法 |
|------|---------|---------|---------|
| 用户信息 | 用户 IMSI 号（IMSI） | 460000123456789 | 测试终端自带 |
| 测试终端使用的 APN | APN 名称（APN） | apn-test | 已配置数据中获取 |

工具：测试终端、N4/Gx 镜像接口抓包工具

> SourcePath: `调测基于初始接入位置的策略控制_27915139.md` §必备事项

#### 6.1.3 调测执行步骤

**步骤1**：进入 "MML 命令行-UNC" 窗口。

**步骤2**：执行 `LST LICENSESWITCH` 查询 License 配置开关是否已打开。

```
LST LICENSESWITCH:LICITEM="LKV2PCIAL01";
```

预期输出（正常）：
```
-------------------------------
  License Item  =  LKV2PCIAL01
  Switch        =  ENABLE
-------------------------------
---    END
```

判断：
- SWITCH=ENABLE → 步骤3
- SWITCH=DISABLE → 执行 `SET LICENSESWITCH:LICITEM="LKV2PCIAL01",SWITCH=ENABLE;` 打开开关

**步骤3**：创建 N4 接口跟踪和 Gx 接口跟踪（此处以 Gx 接口为例，N7 接口调测步骤与 Gx 类似）。

**步骤4**：激活 `apn-test` APN 下热点小区的一个用户。

**步骤5**：根据 **Gx 接口跟踪消息**，观测 GGSN-C/PGW-C 是否将 ULI 信元传递给 PCRF。

预期跟踪行为：
- CCR-Initial 消息中携带 **3GPP-User-Location-Info AVP**（ULI 信元）
- ULI 内容与用户所在小区一致（CGI/ECGI/NCGI）

判断：
- ULI 信元已透传给 PCRF → 步骤6
- 未透传 → 参考 `激活基于初始接入位置的策略控制_27915138.md` 修改配置

**步骤6**：根据 **N4 接口跟踪消息**，观测 PGW-C/GGSN-C 是否将基于初始接入位置的策略传递给 PGW-U/GGSN-U。

预期跟踪行为：
- PFCP Session Establishment Request 消息携带带宽控制策略
- 策略内容与 PCRF 决策（动态PCC）或本地决策（本地PCC）一致

判断：
- 策略已下发到 PGW-U/GGSN-U → 步骤7
- 未下发 → 参考 `激活基于初始接入位置的策略控制_27915138.md` 修改配置

**步骤7**：在 PGW-U/GGSN-U 上验证策略是否生效。

验证方法（在 UDG 侧）：
- 用户上行业务带宽超出允许值时，报文被缓存/丢弃
- 用户下行业务带宽超出允许值时，报文被缓存/丢弃
- 带宽未超出允许值时，正常转发

判断：
- 策略生效（带宽被限制）→ 调测结束
- 策略未生效 → 步骤8

**步骤8**：联系华为技术支持。
- a. 在镜像接口或服务器上开启抓包工具，执行步骤3 和步骤4 并保存报文
- b. 执行 `EXP MML` 命令将当前配置数据导出为 MML 脚本文件并保存
- c. 收集并保存上述所有查询信息
- d. 收集归纳所有信息并联系华为技术支持解决
