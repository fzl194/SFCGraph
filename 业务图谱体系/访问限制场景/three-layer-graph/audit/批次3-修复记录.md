## 4. L4 · 03-task-layer.md DP 编号分层命名定案（MEDIUM）

### 4.1 问题回顾

批次 2 把 03 §6 的任务层 DecisionPoint 表补齐了 Schema §8.8 必选字段（owner_layer / owner_ref_type / owner_ref 等），但 DP 编号仍沿用 `DP-AC-01/02/03`，与 01 业务层 DP-AC-01~08 字面重名（虽 owner_layer=task vs business 语义不同）。批次 2 记录标记为"P3 大工作量，建议改 DP-AC-T01/T02/T03"。

### 4.2 定案决策：保留 + 强化 owner_layer 字段说明

采用 **Schema §8.8 `owner_layer` 字段分层区分** 方案（而非改编号为 DP-AC-T01/T02/T03），理由：

1. **Schema 合规优先**：`owner_layer=task` + `owner_ref_type=config_task` + `owner_ref=T-xxx/T-AC-xxx` 三字段联合唯一标识任务层 DP；业务层为 `owner_layer=business` + `owner_ref_type=network_scenario` + `owner_ref=NS-AC-01`。Schema 字段已提供权威区分机制，无需改 ID。Schema §8.8 设计 `owner_layer` 字段正是为支持"同一 decision_id 在不同层语义不同"的场景。
2. **跨文件引用一致**：05 §6 通过括号标注"（业务层）""（任务层）"消歧（如"DP-AC-01（任务层）POLICYTYPE 路由"），与本表 owner_layer=task 语义一致；00 §3.1 注明"任务层 DP（★ ID 与业务层复用，owner_layer=task 区分）"。
3. **稳定性**：保持 ID 稳定，避免批量重命名引起的引用断裂风险（01/05/00 均有 DP-AC-01~03 引用，改 ID 需同步多文件）。
4. **演进路径保留**：若未来三场景融合出现跨场景 DP 冲突，可再启用 `DP-AC-Txx` 前缀方案。

### 4.3 修复动作

将 03 §6 的"ID 复用说明（U-M-06，P3 待处理）"段落重写为"★ ID 分层命名说明（U-M-06，P5 批次 3 定案）"，明确：
- 采用 owner_layer 字段分层方案；
- 4 条理由（Schema 合规 / 跨文件引用一致 / 稳定性 / 演进路径）；
- 与 01 业务层 DP（owner_layer=business）的区分机制。

### 4.4 验证

- 03 §6 三条 DP 的 owner_layer 均为 `task`（批次 2 已补）；
