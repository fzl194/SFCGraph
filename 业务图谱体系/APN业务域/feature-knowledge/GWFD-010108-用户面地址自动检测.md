## 1. 概述

### 1.1 特性定义（是什么）

用户面地址自动检测是 **UDG 本地分配IP地址时**，UDG 通过向 PDN/DN 侧服务器发送探测消息，检查 UDG 到 PDN/DN 侧服务器之间**路由是否正常**的功能。UDG 支持 **PING、DNS、Tracert** 三种探测方式，在本地构造探测报文（ICMP / DNS 查询 / TTL 递增 ICMP），根据是否收到对端响应判断路由连通性。

> 来源：`GWFD-010108 用户面地址自动检测特性概述_04636902.md`（"定义"、"原理概述-探测方式"章节）[EV-FK-01]

### 1.2 适用NF（UDG/UNC网元）

| 涉及NF | 网元角色 | 支持版本 | 功能说明 |
|--------|---------|---------|---------|
| PGW-U / UPF | 用户面（UDG） | **UDG 20.2.0 及后续版本** | 构造探测报文（PING、DNS、Tracert），向 PDN/DN 侧指定目的 IP 地址发送探测消息；接收 PDN/DN 侧的响应消息，判断 UDG 到 PDN/DN 侧服务器路由的连通性 |

> 说明：本特性**仅适用于 UDG（PGW-U/UPF）用户面**，UNC（SMF/PGW-C）侧无对应特性。来源：特性概述"可获得性-涉及NF"章节 [EV-FK-01]

- 本特性支持在 **UDG 本地地址池分配 UE IP 地址的场景**下使用。但是当 UDG 上部署内置 NAT 功能后，由于 PGW-U/UPF 无法感知外部服务器 IP 地址公网私网 IP 类型，因此有可能出现用户面 IP 地址探测失败的情况。该情况可以联系华为技术支持工程师处理。

- 本特性**不支持 NAT 地址转换**。

## 2. 核心概念

### 2.1 关键术语

| 术语 | 全称 | 说明 |
|------|------|------|
| PING | Packet Internet Groper（因特网包探索器） | 测试网络端对端连接性的程序，通过给目的 IP 地址发送数据包，判断对方是否返回同样大小的数据包来确定两端路由连通性 |
| ICMP | Internet Control Message Protocol（因特网控制报文协议） | PING 与 Tracert 探测报文的承载协议（RFC 792） |
| DNS | Domain Name System（域名系统） | 提供域名与 IP 地址相互映射转换的分布式数据库；DNS 探测方式在 UDG 本地构造标准查询报文 |
| Tracert | 跟踪路由 | 路由跟踪诊断程序，通过发送不同 TTL 值的 ICMP 报文，根据中间路由器的"ICMP 已超时"响应定位故障位置 |
| TTL | Time To Live（生存时间） | Tracert 探测中递增的报文字段，路径上每个路由器转发前减 1，减为 0 时回应"ICMP 已超时" |
| 探测源 IP | Source IP of probe | 探测报文的源地址，从本地地址池网段范围内选取的空闲地址；若不指定，PING/DNS 从每个路由网段随机选一个，Tracert 随机选一个地址网段 |
| 探测目的 IP | Destination IP of probe | PDN/DN 侧的目的服务器 IP；DNS 探测方式必须为 DNS Server IP |
| 全量探测 | Whole Detection | 针对指定 SECTION 下所有路由网段或当前 POOL 内所有地址段下所有路由网段触发探测；**Tracert 不支持全量探测** |
| PDN/DN 侧 | Packet Data Network / Data Network side | 探测目标侧，即 Gi/SGi/N6 接口外联的外部网络 |

> 来源：特性概述"原理概述-探测方式"章节、激活文档"数据"表格 [EV-FK-01] [EV-FK-03]

### 2.2 对象模型

本特性不引入新的配置对象，复用 GWFD-010105 的 POOL/SECTION 对象作为探测目标范围。探测动作本身通过**运维命令（STR/STP/DSP）**触发，不产生持久化配置对象：

```
┌──────────────────────────────────────────────────────┐
│  探测触发（运维命令，瞬时态）                           │
│                                                      │
│   STR PDNROUTETST  ──→  启动一次探测任务               │
│        │                  （同一时间仅一个任务）        │
│        │                                              │
│        ├── IPPOOL      ← 复用 GWFD-010105 的 POOL     │
│        ├── SECTIONNUM  ← 复用 GWFD-010105 的 SECTION  │
│        ├── TESTMETHOD  ← PING / DNS / TRACERT         │
│        ├── SRCIPV4ADDR ← 地址池网段内空闲 IP（可选）   │
│        ├── DSTIPV4ADDR ← PDN/DN 侧服务器 IP           │
│        ├── DOMAINVALUE ← DNS 探测域名（DNS 专用）      │
│        └── WHOLEDETECT ← 全量探测开关                 │
│                                                      │
│   STP PDNROUTETST  ──→  强制终止当前探测              │
│                                                      │
│   DSP PDNTSTRESULT ──→  查询探测结果（成功/失败）      │
│        └── DISPLAYMODE ← SUCCESS / FAIL / 默认全部    │
└──────────────────────────────────────────────────────┘
                          │
                          ▼
┌──────────────────────────────────────────────────────┐
│  探测结果存储（瞬时态，最多 128 条失败记录）            │
│                                                      │
│   探测失败结果表（规格上限 128 个）                    │
│   ↑ 被 ALM/调测读取后即清除（非持久化配置）            │
└──────────────────────────────────────────────────────┘
```

> 来源：参考信息"命令"章节、激活文档"操作步骤/任务示例"、特性规格 [EV-FK-02] [EV-FK-03] [EV-FK-01]

### 2.3 在地址分配场景的角色

本特性是**地址分配链路的"调测/诊断"旁路工具**，不参与地址分配主流程，仅在 GWFD-010105 本地地址池分配已配置完成后，作为运维手段切入：

```
[GWFD-010105 主流程]
  C 面指示 → UDG 本地地址池分配 IP → 生成 WLR 下行路由 → OSPF 发布
                                        │
                                        │ 调测/故障定位时切入
                                        ▼
  [GWFD-010108 旁路探测] ★ 仅本地地址池场景可用
    1. 运维触发 STR PDNROUTETST（指定 IPPOOL + DSTIP + TESTMETHOD）
    2. UDG 本地构造探测报文（ICMP/DNS/TRACERT），源 IP 从地址池网段选
    3. 探测报文发往 PDN/DN 侧，UDG 剔除回响（不计费）
    4. DSP PDNTSTRESULT 查询连通性结果
    5. 失败 → 根据 Tracert 跳数定位故障路由器位置
```

> 来源：综合特性概述"原理概述"、§1.5 依赖关系 [EV-FK-01]

## 3. 原理与流程

### 3.1 实现原理：三种探测方式

#### 3.1.1 PING 探测方式（默认）

PING（Packet Internet Groper）通过给目的 IP 地址发送一个数据包，再判断对方是否返回一个同样大小的数据包来确定两端路由连通性。

- UDG 在本地构造一个**固定格式的 ICMP 请求报文**
- 期待收到对端 PDN/DN 侧服务器对应的 **ICMP 应答报文**
- 请求报文和应答报文的对应关系通过报文中的 **sequence 字段**识别
- **能收到对应的应答报文 → 判断路由连接正常；否则判断失败**

适用：默认探测方式，适用于未禁止 ICMP 的常规网络环境。

#### 3.1.2 DNS 探测方式（防火墙禁止 PING 时使用）

实际组网环境中，存在大多数**防火墙会禁止 PING 报文**的情况，此时 PING 探测方式会导致路由探测失败，可选择 DNS 探测方式。

- UDG 在本地构造一个**固定域名或指定域名的标准查询报文**
- 发送给配置的**目的 DNS Server IP 地址**（目的 IP 必须为 DNS Server）
- **能收到对应的响应报文 → 判断路由连接正常；否则判断失败**

适用：防火墙禁 ICMP 的组网环境。

#### 3.1.3 Tracert 探测方式（故障定位）

当网络中出现故障，前两种探测方式探测失败时，可进一步使用 Tracert 探测方式，**查看网络中出现故障的位置**，为故障诊断提供依据。

- 通过给目的 IP 发送**不同 TTL 值的 ICMP 报文**
- 路径上每个路由器在转发数据包前将 TTL 递减 1，减为 0 时给源头回应"**ICMP 已超时**"或"**目的端口不可达**"响应
- UDG 本地构造并发送一个 **TTL=1** 的探测数据包，收到中间第一跳路由器的"ICMP 已超时"响应
- 每次发送的探测数据包 **TTL 递增 1**，直到收到目的 IP 响应或 **TTL 达到最大值（30）**
- 探测报文指定的端口是一个在目的 IP 处没有应用的端口，目的 IP 收到探测报文时响应"**目的端口不可达（ICMP port unreachable）**"消息，表示 Tracert 探测结束
- 若中间路由有故障，故障路由器不回复响应，**根据 Tracert 探测上报的跳数可判断故障发生的位置**

适用：需要定位故障路由器位置的诊断场景。**不支持全量探测**。

> 来源：特性概述"原理概述-探测方式"章节 [EV-FK-01]

### 3.2 探测机制：源 IP 选取与全量探测

#### 3.2.1 探测源 IP 选取规则

UDG 根据本地配置（`STR PDNROUTETST`）信息（VPN、源 IP、目的 IP）构造探测报文：

- **探测源 IP 是本地地址池网段范围内一个空闲地址**
- **若未配置该参数（SRCIPV4ADDR）**：
  - **PING / DNS 探测方式**：UDG 从**每一个路由网段内随机选取一个 IP** 作为探测源 IP
  - **Tracert 探测方式**：由于探测时间稍长，UDG **随机选取一个地址网段的 IP** 作为探测源 IP，**不遍历所有的地址网段**；因此针对 Tracert 探测方式可以配置指定的探测源 IP 地址

> **重要提醒（文档原文）**：探测源 IP 地址建议选取地址网段中的**空闲地址**，否则会导致在线用户被去激活。

#### 3.2.2 全量探测（WHOLEDETECT）

若需检查整个网络的路由连通性，可进行全量探测：

- **指定地址段号（SECTIONNUM）进行全量探测**：针对**此地址段下的所有路由网段**触发探测
- **不指定地址段号进行全量探测**：针对**当前 POOL 内所有地址段下的所有路由网段**触发探测
- **Tracert 探测方式不支持全量探测**

#### 3.2.3 探测目的 IP

- 探测目的 IP 地址是 **PDN/DN 侧的目的服务器 IP 地址**
- 对于 **DNS 探测方式，探测目的 IP 地址必须是 DNS Server IP 地址**

> 来源：特性概述"原理概述-探测机制"章节、激活文档"数据-SRCIPV4ADDR"字段说明 [EV-FK-01] [EV-FK-03]

### 3.3 业务流程（端到端探测）

```
┌─────────────────────────────────────────────────────────────┐
│ 0. 前置：GWFD-010105 已配置本地地址池（POOL/SECTION）         │
│    （本特性仅本地地址池场景可用，外部地址池不可用）             │
└──────────────────────────────┬──────────────────────────────┘
                               │
                               ▼
┌─────────────────────────────────────────────────────────────┐
│ 1. 运维触发探测                                               │
│    STR PDNROUTETST: IPPOOL=..., IPVERSION=...,                │
│                     SRCIPV4ADDR=...(可选), DSTIPV4ADDR=...,   │
│                     TESTMETHOD=PING/DNS/TRACERT,              │
│                     DOMAINVALUE=...(DNS专用),                 │
│                     WHOLEDETECT=ENABLE/DISABLE,               │
│                     SECTIONNUM=...(全量专用)                  │
└──────────────────────────────┬──────────────────────────────┘
                               │
                               ▼
┌─────────────────────────────────────────────────────────────┐
│ 2. 源 IP 选取                                                 │
│    ├── 配置了 SRCIPV4ADDR → 直接使用                          │
│    └── 未配置 SRCIPV4ADDR：                                   │
│         ├── PING/DNS → 从每个路由网段随机选一个 IP             │
│         └── TRACERT → 随机选一个地址网段的 IP（不遍历）        │
└──────────────────────────────┬──────────────────────────────┘
                               │
                               ▼
┌─────────────────────────────────────────────────────────────┐
│ 3. UDG 本地构造探测报文                                       │
│    ├── PING    → ICMP Echo Request（带 sequence）             │
│    ├── DNS     → DNS 标准查询报文（DOMAINVALUE 指定域名）      │
│    └── TRACERT → ICMP 报文 TTL=1，逐次 TTL+1，最大 30         │
└──────────────────────────────┬──────────────────────────────┘
                               │ Gi/SGi/N6 接口
                               ▼
┌─────────────────────────────────────────────────────────────┐
│ 4. 探测报文发往 PDN/DN 侧目的服务器（DSTIPV4ADDR）            │
│    └── UDG 自身剔除构造的探测报文（不计费、不产生话单）        │
└──────────────────────────────┬──────────────────────────────┘
                               │
                               ▼
┌─────────────────────────────────────────────────────────────┐
│ 5. 响应接收与判定（每路由网段最长探测时间 2 秒）               │
│    ├── PING    → 收到 ICMP Echo Reply → 连接正常              │
│    ├── DNS     → 收到 DNS Response → 连接正常                 │
│    └── TRACERT → 收到"目的端口不可达" → 探测结束              │
│                 中间路由故障 → 故障路由器不响应，按跳数定位    │
└──────────────────────────────┬──────────────────────────────┘
                               │
                               ▼
┌─────────────────────────────────────────────────────────────┐
│ 6. 探测结果存储（最多 128 条失败记录）                        │
│    └── 可被 DSP PDNTSTRESULT 查询（DISPLAYMODE=SUCCESS/FAIL） │
└──────────────────────────────┬──────────────────────────────┘
                               │
                               ▼
┌─────────────────────────────────────────────────────────────┐
│ 7. 异常处理                                                   │
│    └── STP PDNROUTETST → 强制终止当前探测（误操作/超时）       │
│        注：TRACERT 无全部响应场景约需 450 秒返回结果           │
└─────────────────────────────────────────────────────────────┘
```

> 来源：综合特性概述"原理概述"、激活文档"操作步骤"、特性规格 [EV-FK-01] [EV-FK-03]

### 3.4 协议交互

本特性为 UDG 本地运维功能，**不涉及与 UNC（SMF/PGW-C）的控制面信令交互**，仅在 UDG 用户面本地完成：

## 4. 配置规则

### 4.1 激活步骤

本特性**无独立激活配置**（无 License、无 ADD 类配置命令）。激活即"运维触发"：通过 STR/STP/DSP 三条运维命令完成探测任务。前提是 GWFD-010105 本地地址池已配置完成。

```
前置：GWFD-010105 已完成本地地址池配置（ADD POOL / ADD SECTION）

步骤1：（可选）查询可用地址池与地址段
  ├── LST POOL              → 确认 IPPOOL 名称
  └── LST SECTION / LST POOLGROUP → 确认 SECTIONNUM

步骤2：（可选）查询当前路由表（用于探测前后对比）
  ├── DSP ROUTE             → IPv4 路由表（防止探测影响业务正确路由处理）
  └── DSP ROUTE6            → IPv6 路由表

步骤3：启动 PDN/DN 侧路由探测（三选一）
  ├── PING 方式    → STR PDNROUTETST: ..., TESTMETHOD=PING
  ├── DNS 方式     → STR PDNROUTETST: ..., TESTMETHOD=DNS, DOMAINVALUE=...
  └── TRACERT 方式 → STR PDNROUTETST: ..., TESTMETHOD=TRACERT（不支持全量）

步骤4：查询探测结果
  └── DSP PDNTSTRESULT: DISPLAYMODE=SUCCESS / FAIL /（默认全部）

步骤5：（异常处理）强制终止探测
  └── STP PDNROUTETST       → 误操作或探测超时时使用
```

> 来源：激活文档"操作步骤"、特性概述"原理概述-探测机制"（建议 DSP ROUTE 对比） [EV-FK-03] [EV-FK-01]

### 4.5 约束条件

| 约束类型 | 约束内容 |
|---------|---------|
| **仅本地地址池场景** | 本特性仅在 UDG 采用**本地地址池分配 IP 地址**的场景下使用 |
| **外部地址池不支持** | 不支持在**外部地址池**（SMF、RADIUS 分配地址、静态地址）场景下使用 |
| **NAT 不支持** | 本特性不支持 NAT 地址转换 |
| **内置 NAT 探测失败风险** | UDG 部署内置 NAT 后，PGW-U/UPF 无法感知外部服务器 IP 公网私网类型，可能出现探测失败（联系华为技术支持） |
| **探测任务串行** | 同一时间**只能执行一个探测命令**，同时执行多个时后执行的会失败 |
| **单 APN 串行** | 同一时间只支持一个 APN 下的探测，未完成当前探测时不允许再下发 |
| **源 IP 占用风险** | 探测源 IP 建议选空闲地址；全量探测且网段 IP 全被占用时，可能导致**极少数（几个或几十个）在线用户被去激活**，谨慎使用 |
| **Tracert 耗时长** | TRACERT 在未回全部探测响应的场景下，约需 **450 秒**左右才返回结果 |
| **全量探测时长** | 全量探测时探测时长成倍增加，探测任务最多探测 **5 分钟**；未探测到的子段可用指定 SECTION 的全量探测 |
| **全量探测规格** | Tracert 探测方式不支持全量探测 |
| **失败记录上限** | 支持记录的最大探测失败结果个数为 **128 个** |
| **单网段探测时长** | 每个路由网段最长探测时间 **2 秒** |
| **互斥：静态地址路由冗余** | 不支持与 GWFD-010107 同时应用 |
| **互斥：Routing Behind MS** | 不支持与 GWFD-110910（License LKV3G5RBMS01）同时应用 |
| **互斥：IPv6 Prefix Delegation** | 不支持与 GWFD-020406（License LKV3G5P6PD01）同时应用 |
| **互斥：L2TP VPN** | 不支持与 GWFD-020412（License LKV3G5L2TP01）同时应用 |

## 5. 配置案例

### 5.1 典型场景：UDG 到 PDN/DN 侧服务器路由连通性调测

**场景描述**：调测 UDG 设备上用户业务访问是否正常，需要检查 UDG 到 PDN/DN 侧服务器之间的路由连通性。使用本地地址池 `testpool`，目的服务器 IP `10.1.1.1`，分别演示 PING、DNS、Tracert 三种探测方式。

**前提**：GWFD-010105 本地地址池已配置（`ADD POOL: POOLNAME="testpool", POOLTYPE=LOCAL, ...` 已完成）。

**MML 命令序列（原样保留文档示例）**：

```
=== 任务一：以 PING 探测方式启动 PDN/DN 侧路由探测功能 ===

// 全量探测（不指定 SECTIONNUM，遍历 POOL 内所有地址段所有路由网段）
STR PDNROUTETST: IPPOOL="testpool", IPVERSION=IPV4, DSTIPV4ADDR="10.1.1.1", TESTMETHOD=PING, WHOLEDETECT=ENABLE, SECTIONNUM=0;

// 非全量探测（指定 SRCIPV4ADDR，仅探测单一源 IP）
STR PDNROUTETST: IPPOOL="testpool", IPVERSION=IPV4, SRCIPV4ADDR="10.10.1.1", DSTIPV4ADDR="10.1.1.1", TESTMETHOD=PING;

=== 任务二：以 DNS 探测方式启动 PDN/DN 侧路由探测功能 ===

// 全量探测
STR PDNROUTETST: IPPOOL="testpool", IPVERSION=IPV4, DSTIPV4ADDR="10.1.1.1", TESTMETHOD=DNS, DOMAINVALUE="www.example.com", WHOLEDETECT=ENABLE, SECTIONNUM=0;

// 非全量探测
STR PDNROUTETST: IPPOOL="testpool", IPVERSION=IPV4, DSTIPV4ADDR="10.1.1.1", TESTMETHOD=DNS, DOMAINVALUE="www.example.com";

=== 任务三：以 Tracert 探测方式启动 PDN/DN 侧路由探测功能（不支持全量探测）===

STR PDNROUTETST: IPPOOL="testpool", IPVERSION=IPV4, DSTIPV4ADDR="10.1.1.1", TESTMETHOD=TRACERT;

=== 查询 PDN/DN 侧路由探测结果 ===

// 所有的探测结果，包括探测成功的结果和探测失败的结果
DSP PDNTSTRESULT:;

// 所有探测成功的结果
DSP PDNTSTRESULT:DISPLAYMODE=SUCCESS;

// 所有探测失败的结果
DSP PDNTSTRESULT:DISPLAYMODE=FAIL;
```

> 来源：激活文档"任务示例-脚本"（MML 原样保留）[EV-FK-03]

### 5.2 场景变体对照

| 变体 | TESTMETHOD | 全量探测 | 必填参数 | 典型用途 |
|------|-----------|---------|---------|---------|
| PING 全量 | PING | ENABLE | IPPOOL, DSTIPV4ADDR | 常规路由连通性批量检查 |
| PING 单点 | PING | DISABLE（默认） | IPPOOL, SRCIPV4ADDR, DSTIPV4ADDR | 针对特定源 IP 的连通性检查 |
| DNS 全量 | DNS | ENABLE | IPPOOL, DSTIPV4ADDR（DNS Server）, DOMAINVALUE | 防火墙禁 ICMP 环境批量检查 |
| DNS 单点 | DNS | DISABLE | IPPOOL, DSTIPV4ADDR（DNS Server）, DOMAINVALUE | 防火墙禁 ICMP 环境单点检查 |
| Tracert | TRACERT | **不支持** | IPPOOL, DSTIPV4ADDR（, 可选 SRCIPV4ADDR） | 故障定位（按跳数定位故障路由器） |

### 6.4 常见问题与排查

| 问题现象 | 可能原因 | 排查方法 |
|---------|---------|---------|
| 探测命令执行失败 | 同一时间已有探测任务在执行（任务串行） | 等待当前探测完成；或 `STP PDNROUTETST` 终止后重试 |
| 探测命令执行失败 | 同一 APN 下已有未完成的探测 | 等待当前 APN 探测完成 |
| 在线用户被去激活 | 探测源 IP 选用了在线用户 IP（未选空闲 IP） | 重新规划空闲 SRCIPV4ADDR；全量探测时若网段全占用属预期风险 |
| PING 探测全部失败 | 网络中防火墙禁止 ICMP 报文 | 改用 DNS 探测方式（TESTMETHOD=DNS） |
| Tracert 长时间无响应 | TRACERT 无全部响应场景约需 450 秒 | 等待至 450 秒；或 `STP PDNROUTETST` 终止 |
| 全量探测未覆盖所有子段 | 探测任务最多探测 5 分钟，时长不足 | 对未探测到的子段使用指定 SECTIONNUM 的全量探测 |
| 探测失败但路由实际正常 | UDG 部署内置 NAT，无法感知公网私网 IP 类型 | 联系华为技术支持 |
| 探测失败（外部地址池场景） | 本特性不支持外部地址池（SMF/RADIUS/静态） | 确认地址分配方式为本地地址池（GWFD-010105） |
| 探测失败（与互斥特性同时应用） | 与 GWFD-010107/110910/020406/020412 同时应用 | 确认未与互斥特性同时启用 |
| 探测影响业务路由 | 探测前后路由变化未对比 | 探测前后用 `DSP ROUTE`/`DSP ROUTE6` 对比路由表 |

> 来源：综合特性概述"应用限制"、"原理概述"说明、激活文档"操作步骤"说明整理 [EV-FK-01] [EV-FK-03]
