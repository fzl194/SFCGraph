## 1. 概述

### 1.1 特性定义（是什么）

UDG 支持两种地址分配方式：**外部地址分配**和**用户面地址分配**，可以为UE提供三种类型的地址：IPv4地址、IPv6地址、双栈地址（IPv4和IPv6地址）。

- **外部地址分配**：指 PGW-C/SMF 为UE分配IP地址后将UE的IP地址通知 UDG，由 UDG 完成路由发布的功能。
- **用户面地址分配**：指 UDG 根据本地地址池为UE分配IP地址并完成路由的发布功能。

> 来源：`output/UDG_Product_Documentation_CH_20.15.2/特性部署/特性指南/UDG特性指南/基本接入功能/GWFD-010104 地址分配方式/GWFD-010104 地址分配方式特性概述_67531788.md`（"定义"章节）

> 来源：`激活地址分配方式_71653959.md`（"必备事项-前提条件"）

### 2.2 通用激活步骤骨架（外部地址分配方式）

```
步骤1：（可选）配置白名单检测功能
  ├── ADD VPNINST           → 配置VPN实例（用户面侧）
  ├── ADD POOLGROUP         → 配置地址池组
  ├── ADD POOL              → 配置 EXTERNAL 类型地址池（★区别于010105的LOCAL）
  │     └── CHECKIPVALID=ENABLE（可选） → 开启静态地址白名单检测
  ├── ADD SECTION           → 配置地址段（IPv4/IPv6）
  ├── ADD POOLBINDGROUP     → 绑定地址池到地址池组
  ├── ADD APN               → 配置 APN 实例（绑定 VPN）
  ├── SET APNADDRESSATTR    → 配置指定APN下支持使用主机地址接入（HOSTROUTEIP=ENABLE）
  └── ADD POOLGRPMAP        → 配置地址池组与APN的映射关系

步骤2：配置手机下行路由（路由发布核心）
  ├── ADD L3VPNINST         → 创建 L3VPN 实例
  ├── ADD VPNINSTAF         → 设置 VPN 实例地址族
  ├── ADD OSPF              → 启动 OSPF 进程（指定 VPN 与 Router ID）
  ├── ADD OSPFAREA          → 添加 OSPF 区域
  ├── ADD OSPFNETWORK       → 配置运行 OSPF 协议的接口
  └── ADD OSPFIMPORTROUTE   → 引入 WLR 路由（PROTOCOL=wlr）
```

> 来源：`激活地址分配方式_71653959.md`（"操作步骤"章节）

## 3. 原理

### 3.1 实现原理：总体地址分配流程

UE地址分配总体流程（9步）：

1. UPF/PGW-U 向 SMF/PGW-C 上报是否用户面本地分配地址
2. UE 向 SMF/PGW-C 申请分配IP地址
3. SMF/PGW-C 向 HSS/UDM 获取用户签约数据
4. SMF/PGW-C 根据用户签约数据和本地策略指示 UPF/PGW-U 具体的地址分配方式
5. UPF/PGW-U 为UE选择用户面地址分配或者外部地址分配方式
6. PGW-U/UPF 根据具体的地址分配方式完成UE地址分配，并在PDU会话建立响应中返回给 SMF/PGW-C
7. SMF/PGW-C 向UE返回分配的UE地址
8. UE 获取IP地址
9. UE 在获取IP地址后便可成功发起数据业务

> 来源：`GWFD-010104 地址分配方式特性概述_67531788.md`（"原理概述"章节，图1 UE地址分配流程）

### 3.2 用户面地址分配能力上报（ADUP标识位）

在偶联消息交互中，PGW-U/UPF 会将用户面地址分配能力上报给 PGW-C/SMF。PGW-U/UPF 在偶联建立/响应消息（PFCP Association Setup Request/Response）或偶联更新请求/响应（PFCP Association Update Request/Response）消息中会携带 **UP Function Flag** 信元指示 PGW-U/UPF 是否具备用户面地址分配能力。UP Function Flag 信元中的 **ADUP 标志位**标识 PGW-U/UPF 支持用户面地址分配方式。该标志位会**恒定设为 1**，标识 PGW-U/UPF 支持用户面地址分配方式。

> 来源：`实现原理/地址分配流程_73558472.md`（"用户面地址分配能力上报"章节）

### 3.3 地址分配方式选择（CHV4/CHV6/V4/V6 真值表）

PGW-C/SMF 通过会话建立请求消息（PFCP Session Establishment Request）中的 **UE IP Address** 信元指示 PGW-U/UPF 具体的地址分配方式。PGW-U/UPF 根据 UE IP Address 信元中的 **CHV4、CHV6、V4、V6** 标志位判断采用外部地址分配或用户面地址分配方式。

- **CHV6 和 CHV4**：表示 U 面分配的地址
- **V4 和 V6**：表示 C 面分配的地址

#### 3.3.1 地址分配方式选择真值表（标准协议 BIT391=0）

| CHV4 | CHV6 | V4位 | V6位 | IPv4 address | IPv6 address | PGW-U/UPF地址分配处理方式 | 地址分配方式 |
|------|------|------|------|--------------|--------------|--------------------------|-------------|
| 0 | 0 | 1 | 0 | 携带IPv4地址 | - | 取该IPv4地址为UE的IP地址 | **外部地址分配** |
| 0 | 0 | 0 | 1 | - | 携带IPv6地址 | 取该IPv6地址为UE的IP地址 | **外部地址分配** |
| 0 | 0 | 1 | 1 | 携带IPv4地址 | 携带IPv6地址 | 取IPv4地址和IPv6地址为UE的IP地址 | **外部地址分配** |
| 1 | 0 | 0 | 0 | - | - | 根据本地地址池为UE分配IPv4地址 | **用户面地址分配** |
| 0 | 1 | 0 | 0 | - | - | 根据本地地址池为UE分配IPv6地址 | **用户面地址分配** |
| 1 | 1 | 0 | 0 | - | - | 根据本地地址池为UE分配IPv4和IPv6地址 | **用户面地址分配** |
| 1 | 0 | 0 | 1 | - | 携带IPv6地址 | 取该IPv6地址为IPv6 + 本地分配IPv4 | **外部+用户面（混合）** |
| 0 | 1 | 1 | 0 | 携带IPv4地址 | - | 取该IPv4地址为IPv4 + 本地分配IPv6 | **外部+用户面（混合）** |

> 说明1：对于表中未给出的场景，PGW-U/UPF 均以激活失败处理。
> 说明2：当地址分配方式为用户面地址分配时，PFCP Session Establishment Request 消息中携带 UE IP Address Pool Identity 信元时，PGW-U/UPF 会解析信元中的 Framed-Pool 或者 Framed-IPv6-Pool，根据地址池名称从指定地址池中为UE分配地址；当地址池名称是一个 `*` 的字符串，则使用普通本地地址池为UE分配地址。
> 说明3：当设置软参 **BIT391=1** 时，PGW-U/UPF 根据 UE IP Address 信元中的 Spare 作为扩展 UPA，通过 Spare、V4、V6 标志位判断采用外部地址分配或用户面地址分配方式（华为私有扩展）。

> 来源：`实现原理/地址分配流程_73558472.md`（"地址分配方式选择"章节，表1）

### 3.4 两种分配方式的功能点对比

#### 3.4.1 外部地址分配方式（本特性独立承担的核心配置）

| 功能点 | 说明 |
|-------|------|
| 功能点1：路由发布 | PGW-U/UPF 的主要功能。将生成 WLR（Wireless Route）路由，并通过动态路由协议（如OSPF）发布到骨干网上。当 ADD POOL 中的 POOLTYPE 设置为 EXTERNAL，该地址池即为 EXTERNAL 类型地址池 |
| 功能点2：UE地址白名单检测 | 当 ADD POOL 中的 CHECKIPVALID 设置为 ENABLE 时，PGW-U/UPF 将检查 PGW-C/SMF 下发的UE地址是否在配置的 EXTERNAL 类型的 POOL 内。如果不在，返回会话建立失败响应给 PGW-C/SMF |
| 功能点3：UE地址冲突检测 | 当 PGW-C/SMF 下发UE地址但未下发掩码时，如果检测到相同VPN属性下的UE地址冲突，则根据软参 BYTE44 配置决定去活原用户或者拒绝新用户接入 |

#### 3.4.2 用户面地址分配方式（由 GWFD-010105 详细承载）

| 功能点 | 说明 |
|-------|------|
| 功能点1：UE地址分配 | PGW-U/UPF 在承载上下文激活/PDU会话建立阶段直接从本地地址池中为UE分配IP地址，同时完成路由发布功能 |
| 功能点2：UE地址上报 | PGW-U/UPF 为UE分配IP地址后，将通过会话响应消息中的 Created PDR 或 Created Traffic Endpoint 信元将UE的IP地址上报给 PGW-C/SMF。本地IPv4地址池不分配主机号为全0和全1的地址给UE |
| 功能点3：路由发布 | 同外部地址分配，生成 WLR 路由并通过动态路由协议发布。**详细细节请查看 GWFD-010105 用户面地址分配**（文档原文明确引用） |
| 功能点4：RADIUS下发地址池名称 | SMF/PGW-C 从 RADIUS 获取地址池名称并下发到 PGW-U/UPF，命中则为UE分配IP地址并生成 WLR 路由；未命中则UE分配地址失败，用户退网 |

> 来源：`实现原理/地址分配流程_73558472.md`（"地址分配流程-外部地址分配方式"与"用户面地址分配方式"章节）

### 3.5 用户下行路由生成与发布

#### 3.5.1 用户下行路由定义

用户下行路由指外部 PDN/DN 到 UE 的下行路由，用于下行报文（PDN/DN 服务器发送给UE的报文）的回程寻址。其作用是 PDN/DN Server 和路径中的路由设备能将UE下行报文发送给正确的 UPF，UPF 接收并处理用户下行报文，然后转发给对应的基站，由基站发送给UE。

#### 3.5.2 用户地址来源（两种）

| 地址来源 | POOLTYPE | 说明 |
|---------|----------|------|
| 本地分配地址 | LOCAL | UPF 给UE分配IP地址（动态IP，同一用户不同时刻开机分配到的IP可能不同） |
| 外部分配地址 | EXTERNAL | 其他NF给UE分配IP地址，或采用 UDM 上的UE静态签约IP地址（同一用户如签约IP未变更，分配到的IP不变） |

> 不管采用哪种地址来源，UE IP地址均由 UPF 对外发布。

#### 3.5.3 UPF 对外发布方式（三种）

| 发布方式 | 说明 | 是否需提供UE IP地址段 | 是否需提供isu-pod子接口IP |
|---------|------|---------------------|-------------------------|
| 静态路由 | PDN/DN Server、UPF与PDN/DN之间的路由设备上配置到UE IP地址段的静态路由或默认路由 | 是 | 是 |
| OSPF动态路由 | UPF和DC-GW之间建立OSPF邻居，将wlr类型的路由发布出去 | 否 | 否 |
| BGP动态路由 | UPF和DC-GW之间建立BGP邻居，将wlr类型的路由发布出去 | 否 | 是（当采用静态路由实现BGP本端IP与BGP PEER IP互通时） |

#### 3.5.4 本地分配地址的路由生成过程

UPF 上配置 LOCAL 类型地址池（POOL_LOCAL）后，会自动生成两类路由：

| 路由类型 | 含义 | 何时生成 | Route State |
|---------|------|---------|-------------|
| **wlr_bh**（黑洞路由） | 地址段整体黑洞路由，用于对外发布 | 用户发起PDU会话前即存在 | Active Adv（可发布） |
| **wlr_ud**（用户路由） | 单用户主机路由，/32 或带掩码 | 用户PDU会话建立后生成 | Active NoAdv（不发布，本地有效） |

示例（UE IP段 10.1.1.0~10.1.1.255）：
```
// 会话建立前
VRF_N6   10.1.1.0    24   wlr_bh   ...  Active Adv    // 黑洞路由，对外发布

// 会话建立后（用户分到 10.1.1.32/27）
VRF_N6   10.1.1.0    24   wlr_bh   ...  Active Adv    // 黑洞路由仍存在
VRF_N6   10.1.1.32   27   wlr_ud   ...  Active NoAdv  // 用户路由，本地有效
```

OSPF/BGP 引入 wlr 类型的路由时，将 wlr_bh 路由的 Route Prefix 和 Route Mask Length 发布给 DC-GW。

#### 3.5.5 外部分配地址的路由生成过程

UPF 上配置 EXTERNAL 类型地址池（POOL_EXTERNAL）后，仅生成 wlr_ud 用户路由（不生成 wlr_bh 黑洞路由）：

```
// 会话建立前
VRF_N6   10.1.2.0    27   wlr_ud   ...  Active Adv    // 用户路由，可发布

// 会话建立后（UPF获取其他NF分配的IP或UE签约IP）
VRF_N6   10.1.2.0    27   wlr_ud   ...  Active Adv    // 用户路由，可发布
```

> 关键差异：本地分配生成 wlr_bh + wlr_ud 两类路由；外部分配仅生成 wlr_ud 一类路由。
> 来源：`实现原理/用户下行路由生成与发布_96489749.md`（全文）

### 3.6 协议交互

| 接口 | 交互网元 | 关键信元 | 说明 |
|------|---------|---------|------|
| N4（5G）/ Sxa / Sxb（4G） | SMF/PGW-C ↔ UPF/PGW-U | PFCP Association Setup Request/Response（UP Function Flag, ADUP 标志位） | U面能力上报（恒定为1） |
| N4 / Sxa / Sxb | SMF/PGW-C → UPF/PGW-U | PFCP Session Establishment Request（UE IP Address, CHV4/CHV6/V4/V6） | C面指示U面分配方式及类型 |
| N4 / Sxa / Sxb | UPF/PGW-U → SMF/PGW-C | PFCP Session Establishment Response（Created PDR / Created Traffic Endpoint） | U面回传分配的UE IP（用户面分配方式下） |
| Gi/SGi/N6 | UPF/PGW-U ↔ DN/DC-GW | OSPF/BGP 路由发布（wlr类型） | 用户下行路由发布 |

> 来源：`实现原理/地址分配流程_73558472.md`、`实现原理/用户下行路由生成与发布_96489749.md`

## 5. 配置案例

### 5.1 典型场景：外部地址分配方式（本特性独立承担的核心场景）

**场景描述**：UE接入核心网，UDG 采用**外部地址分配方式**为UE分配IP地址。即 PGW-C/SMF 为UE分配IP地址后通知 UDG，由 UDG 完成路由发布和白名单检测。

**MML命令序列（原样保留，来自激活文档任务示例）**：

```
//配置外部地址分配时的白名单检测功能。

ADD VPNINST:VPNINSTANCE="vpn1";

ADD POOLGROUP: POOLGRPNAME="poolgroup1";

ADD POOL:POOLNAME="testpool",POOLTYPE=EXTERNAL,HASVPN=ENABLE,VPNINSTANCE="vpn1";

ADD SECTION:POOLNAME="testpool",SECTIONNUM=1,IPVERSION=IPV4,V4STARTIP="10.10.1.1",V4ENDIP="10.10.1.10";

ADD POOLBINDGROUP:POOLGROUPNAME="poolgroup1", POOLNAME="testpool";

ADD APN:APN="apn-test",HASVPN=ENABLE,VPNINSTANCE="vpn1";

SET APNADDRESSATTR: APN="apn-test", HOSTROUTEIP=ENABLE;

ADD POOLGRPMAP: MAPPINGNAME="mapping1", APN="apn-test", POOLGROUPNAME="poolgroup1";

//配置手机下行路由。

ADD L3VPNINST:VRFNAME="vpn1";

ADD VPNINSTAF:VRFNAME="vpn1",AFTYPE=ipv4uni;

ADD OSPF:PROCID=100,VRFNAME="vpn1",SCHEMAROUID="10.10.10.1";

ADD OSPFAREA:PROCID=100,AREAID="0.0.0.0";

ADD OSPFNETWORK:PROCID=100,AREAID="0.0.0.0", IPADDRESS="10.13.21.0", WILDCARDMASK="0.0.0.255";

ADD OSPFIMPORTROUTE:PROCID=100,TOPOID=0,PROTOCOL=wlr;
```

> 来源：`激活地址分配方式_71653959.md`（"任务示例-脚本"）

### 5.2 场景变体对照（★本特性 vs GWFD-010105）

| 变体 | 本特性 GWFD-010104 | GWFD-010105 用户面地址分配 |
|------|-------------------|---------------------------|
| 承载的分配方式 | **外部地址分配**（独立承担）+ 用户面地址分配（总览，细节交010105） | 用户面地址分配（4种子方式：基于APN/DNN、基于位置、基于SMF、基于RADIUS下发地址池名） |
| ADD POOL POOLTYPE | **EXTERNAL** | LOCAL |
| CHECKIPVALID白名单检测 | ENABLE（核心功能，仅EXTERNAL有效） | 不适用 |
| SET APNADDRESSATTR主要参数 | HOSTROUTEIP=ENABLE | SUPPORTIPV4/SUPPORTIPV6/IGNOREV4POOLID/IGNOREV6POOLID |
| 路由生成 | 仅 wlr_ud（用户路由） | wlr_bh（黑洞路由）+ wlr_ud（用户路由） |
| 地址分配规则 | 无需 SET IPALLOCRULE（地址由C面分配） | 需 SET IPALLOCRULE / SET APNIPALLOCRULE（三级优先级） |
| SMF相关配置 | 无 | 基于 SMF / SMF+APN 子方式需 ADD CPNODEID + SET IPALLOCBYSMFGLBSW |
| License | 无 | 无 |
| 应用限制 | 无（文档声明） | C面必须为华为配套设备（PFCP私有扩展） |

## 6. 验证与调测

### 6.1 验证方法（外部地址分配方式）

#### 6.1.1 调测前提与目的

用户入网时采用**外部地址分配方式**激活。调测目的：确保用户可以正常接入。

> 适用：PGW-U、UPF

#### 6.1.2 调测数据准备

| 类别 | 参数名称 | 取值样例 | 获取方法 |
|------|---------|---------|---------|
| 用户信息查询 | 用户IMSI号（IMSI） | 460000123456789 | 测试终端自带 |
| 测试终端使用的APN | APN名称（APN） | apn-test | 已配置数据中获取（终端接入运营商自身IP网络中配置的APN实例名） |

工具：测试终端、OM Portal

#### 6.1.3 调测执行步骤

**步骤1**：测试终端使用"apn-test"APN发起接入网络请求。

**步骤2**：执行 `DSP SESSIONINFO` 命令，通过测试终端的IMSI查看测试终端是否成功激活。

```
DSP SESSIONINFO:QUERYTYPE=IMSI,IMSI="460000123456789";
```

预期输出（外部地址分配方式的标志）：
```
......
-------------------------------
                                 IMSI  =  460000123456789
                                     ......              
                             PDP Type  =  IPv4 
                    IPv4 Address type  =  EXTERNAL ALLOC IP ADDRESS
                     IPv4 PDP address  =  10.10.10.1         
                                     ......    
                             APN name  =  apn-test
                                     ......       
(Number of results = 1)
---    END
```

判断：
- 测试终端激活成功，且 **"IPv4 Address type" 的值为 "EXTERNAL ALLOC IP ADDRESS"**，且使用的"IPv4 PDP address"与规划值一致 → 测试终端接入成功，调测结束
- 如果测试终端激活成功，但使用的"IPv4 Address type"或"IPv4 PDP address"与规划值不一致，或测试终端激活失败 → 执行步骤3

> ★关键差异：外部地址分配（010104）期望值为 `EXTERNAL ALLOC IP ADDRESS`；用户面地址分配（010105）期望值为 `UPF ALLOC IP ADDRESS`。

**步骤3**：收集信息并寻求技术支持。
- a. 在OM Portal上创建用户跟踪任务并保存报文
- b. 执行 `EXP MML` 命令将当前配置数据导出为MML脚本文件并保存
- c. 收集并保存上述所有查询信息
- d. 收集归纳所有信息并联系华为技术支持解决

> 来源：`调测地址分配方式_71653960.md`（"操作步骤"章节）

### 6.4 常见问题与排查

| 问题现象 | 可能原因 | 排查方法 |
|---------|---------|---------|
| 用户激活失败（外部地址分配） | UE地址未通过白名单检测（CHECKIPVALID=ENABLE但地址不在EXTERNAL POOL内） | LST POOL 确认 POOLTYPE=EXTERNAL 与地址段范围；核对 PGW-C/SMF 下发的UE地址 |
| 用户激活成功但地址类型不符 | 实际为用户面地址分配（UPF ALLOC）而非外部地址分配 | DSP SESSIONINFO 查 IPv4 Address type，EXTERNAL ALLOC 方为外部地址分配；UPF ALLOC 为本地分配（需检查 POOLTYPE 配置） |
| 下行报文无法回程 | WLR路由未发布 | DSP ROUTE 查 wlr_bh/wlr_ud 路由；检查 ADD OSPFIMPORTROUTE PROTOCOL=wlr（不能只选 WLR_SP 或 WLR_UD） |
| OSPF邻居建立失败 | Router ID冲突或区域不匹配 | 检查 SCHEMAROUID 全网唯一性；LST OSPFAREA 核对 AREAID |
| 地址冲突误判 | C面下发UE地址未下发掩码，触发 BYTE44 冲突检测 | 根据 BYTE44 配置决定去活原用户或拒绝新用户；调整 BYTE44 策略 |
| EXTERNAL POOL 配置后仍报地址不在池 | POOLTYPE误配为LOCAL | LST POOL 确认 Pool Type 为 POOL_EXTERNAL；重新 ADD POOL |
| 真值表未列场景导致激活失败 | PFCP BIT组合异常 | 检查 C面下发消息；对照真值表确认 BIT 组合合法 |

## 9. Stage 3 横向分析预留问题

> 供 Stage 3 横向分析阶段验证，本阶段基于单特性精读无法定论。

1. **010104 与 010105 的特性边界是否清晰**：010104 承载外部地址分配，010105 承载用户面地址分配，两者在 ADD POOL POOLTYPE（EXTERNAL vs LOCAL）上明确分离。但 SET APNADDRESSATTR 命令在两特性中均有使用（010104 用 HOSTROUTEIP，010105 用 SUPPORTIPV4/V6/IGNOREV4/V6POOLID），需确认是否为同一命令的不同参数用法。
2. **010108 的真实定位**：需读取 GWFD-010104 文档后确认其与 010104/010105 的关系（本特性文档未提及 010108）。
3. **010104 与 UNC 侧 WSFD-010502/WSFD-010504 的对应关系**：控制面侧的地址分配方式如何与本特性对应，需 Stage 3 跨产品线分析。
4. **外部地址分配的 C 面配置**：本特性仅描述 UDG 侧（白名单检测+路由发布），C 面（PGW-C/SMF）如何分配IP并下发，需参考 UNC 侧文档。
