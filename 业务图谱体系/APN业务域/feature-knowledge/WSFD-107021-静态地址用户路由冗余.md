## 1. 概述

### 1.1 特性定义（是什么）

当 GGSN/PGW-C/SMF 上配置了静态 IP 地址网段与 UPF 的绑定关系后，静态 IP 地址用户、Radius 分配 IP 地址用户接入网络时，基于配置的网段绑定关系为用户选择指定的 UPF。**如果此 UPF 故障，则静态 IP 地址网段内的用户业务就会异常**。

本特性是指 **GGSN/PGW-C/SMF 支持针对任意静态 IP 地址网段可以绑定两个 UPF**：一个 UPF 作为主用，另一个 UPF 作为备用。**主用 UPF 故障时，为静态 IP 地址网段内的用户选择到备用的 UPF**，防止用户业务异常。

本特性中的 UPF 是指包含了 SGW-U、PGW-U 的 UPF（即用户面网元统称）。

> 来源：`WSFD-107021 静态地址用户路由冗余特性概述_76652606.md`（"定义"章节）

## 2. 核心概念

### 2.1 关键术语

| 术语 | 全称 | 说明 |
|------|------|------|
| 静态地址用户 | Static Address User | HLR/HSS/UDM 静态签约分配 IP 的用户，以及 Radius 分配 IP 的用户（本特性覆盖的两种地址用户） |
| 主用 UPF | Primary UPF | 静态 IP 网段绑定的高优先级 UPF（PRIORITY=0），正常时承载业务 |
| 备用 UPF | Standby UPF | 静态 IP 网段绑定的低优先级 UPF（PRIORITY=1），主用故障时承载业务 |
| Full-Mesh 组网 | Full-Mesh Networking | 5G 场景下任意 SMF 可以纳管所有 UPF 的组网方式，是冗余切换的前提 |
| UPF 选择优先级 | UPF Selection Priority | ADD UPFBINDGRP 的 PRIORITY 参数，0 为高优先级（主用），1 为低优先级（备用） |
| PFCP 心跳检测 | PFCP Heartbeat Detection | SMF 通过 N4 接口心跳检测感知 UPF 故障的机制 |
| Gi/N6 接口 | Gi/N6 Interface | UPF 到外部 PDN/DN 的接口；故障期间选择到备用 UPF 的用户通过备用 UPF 的 Gi/N6 接口访问外部网络 |
| GRE Tunnel | GRE 隧道 | 主备 UDG 间建立的隧道，主用 UDG 恢复后用于将备用 UDG 上激活用户的业务报文转发回主用 UDG（由 UDG 侧 GWFD-010107 配置） |

> 来源：综合 `WSFD-107021 静态地址用户路由冗余特性概述_76652606.md`（"定义/原理概述"章节）及 UDG 侧 `GWFD-010107 静态地址用户路由冗余特性概述_19612410.md`（"定义/原理概述"章节）

```
│   │ = INHERIT      │    └─────────────┬────────────────────┘    │
│   │                │                  │ ADD UPFBINDGRP × 2       │
│   │ ①UPF_Inst_1    │                  │  （★主备绑定关键命令）    │
│   │ ②UPF_Inst_2    │         ┌────────┴────────┐                │
```

核心对象说明：

| 对象 | 作用 | 本特性角色 |
|------|------|-----------|
| ADDRPOOL | 静态地址池（POOLTYPE=UDM） | 定义静态 IP 网段来源（与 WSFD-010502 复用） |
| SECTION | 地址段（V4STARTIP/V4ENDIP） | 定义静态 IP 网段范围（如 10.0.0.0~10.1.2.3） |
| ADDRPOOLGRP | 地址池组（POOLGRPTYPE=UDM） | 组织地址池 |
| POOLBINDGRP | 地址池↔地址池组绑定 | 关联池到组 |
| UPNODE | UPF 节点（ADDRALLOCMODE=INHERIT） | 注册主备两个 UPF 实例 |
| ADDRUPGROUP | UPF 组（UPFGRPTYPE=UDM） | 一个组容纳主备两个 UPF（★本特性关键） |
| **UPFBINDGRP** | UPF↔UPF 组绑定（含 PRIORITY） | **★ 本特性核心命令：PRIORITY=0 主用，PRIORITY=1 备用** |
| POOLGRPMAP | 地址池组↔UPF 组映射 | 将静态网段映射到主备 UPF 组 |

### 2.3 在地址分配场景的角色

本特性是**地址分配域的可靠性增强特性**，位于"静态 IP 段→UPF 绑定"链路的冗余保护层：

```
静态IP用户/Radius用户接入
    ↓
[WSFD-010502] 基于静态IP段选择 UPF（无冗余：单 UPF 绑定）
    ↓
    ├── 正常场景：选择主用 UPF 建立会话
    │
    ↓ ★ WSFD-107021 冗余增强介入点
    │
[WSFD-107021] 主用 UPF 故障时，切换到备用 UPF（C 面决策）
    ↓
    ├── 主用故障：选择备用 UPF，业务经备用 UPF Gi/N6 接口
    │
    ↓ N4 PFCP 指示
    │
[GWFD-010107] UDG 侧通过 GRE 隧道实现实际报文冗余转发（U 面执行）
    ├── 故障期：业务报文不经 GRE 隧道，直经备用 UPF
    └── 主用恢复：备用 UPF 上用户报文经 GRE 隧道转发回主用 UPF
```

具体角色说明：

1. **决策层（C 面）**：UNC 基于优先级策略选择主备 UPF，感知 UPF 故障（PFCP 心跳）
2. **保护对象**：静态 IP 地址网段内的用户（静态签约用户 + Radius 分配 IP 用户）
3. **触发条件**：主用 UPF 故障（通过 PFCP 心跳检测感知）
4. **与静态地址分配（WSFD-010502 UDM 子方式）的协同**：本特性复用 WSFD-010502 的"静态 IP 段→UPF 绑定"配置体系（ADDRPOOL/SECTION/ADDRPOOLGRP/POOLBINDGRP/POOLGRPMAP 完全一致），区别仅在于 **UPF 组内绑定两个 UPF 并指定优先级**（WSFD-010502 场景为单 UPF 绑定）

## 3. 原理与流程

### 3.1 实现原理（三场景切换机制）

本特性的核心是基于 **UPF 优先级 + PFCP 心跳检测** 的主备切换机制：

```
┌─────────────────────────────────────────────────────────────────┐
│ 场景一：主用 UPF-1 正常                                          │
│                                                                 │
│  静态IP用户接入                                                  │
│      ↓                                                          │
│  UNC 基于静态IP段绑定关系 + 优先级 → 选择高优先级 UPF-1          │
│      ↓                                                          │
│  业务报文路径：UE → 基站 → PE → UPF-1 → PE → PDN/DN             │
│  （通过 UPF-1 的 Gi/N6 接口访问外部网络）                       │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│ 场景二：主用 UPF-1 故障                                          │
│                                                                 │
│  UNC 通过 PFCP 心跳检测感知 UPF-1 故障                           │
│      ↓                                                          │
│  静态IP用户接入（或已有用户重新激活）                             │
│      ↓                                                          │
│  UNC 选择低优先级备用 UPF-2 承载业务                             │
│      ↓                                                          │
│  业务报文路径：UE → 基站 → PE → UPF-2 → PE → PDN/DN             │
│  （通过 UPF-2 的 Gi/N6 接口访问外部网络）                       │
│                                                                 │
│  ★ 故障期间：业务报文不经过 GRE 隧道                            │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│ 场景三：主用 UPF-1 故障恢复                                      │
│                                                                 │
│  UPF-1 恢复，由于 UPF-1 优先级较高                               │
│      ↓                                                          │
│  新上线用户 → 直接选择 UPF-1                                     │
│                                                                 │
│  已在 UPF-2 上的用户：                                           │
│      ├── 上行报文：UE → 基站 → PE → UPF-2                        │
│      └── 下行报文：PDN/DN → PE → UPF-1 ──GRE隧道──→ UPF-2 → UE  │
│          （★ GRE 隧道转发由 UDG 侧 GWFD-010107 实现）            │
└─────────────────────────────────────────────────────────────────┘
```

> 来源：`WSFD-107021 静态地址用户路由冗余特性概述_76652606.md`（"原理概述"章节）、UDG 侧 `主用PGW-U_UPF故障场景_19644452.md`

### 3.3 下行路由发布机制（UDG 侧协同）

主备 UPF 切换时，下行路由的发布与撤销由 UDG 侧负责（与 GWFD-010107 协同）：

| UPF 状态 | 下行路由发布 | 下行数据流向 |
|---------|-------------|-------------|
| UPF-1 正常 | UPF-1 和 UPF-2 同时向 PDN 侧发布 UE 静态签约 IP 网段（OSPF/BGP） | 选路到 UPF-1（高优先级） |
| UPF-1 故障 | **UPF-1 不对外发布下行路由**，UPF-2 继续发布 | 选路到 UPF-2 |
| UPF-1 恢复 | UPF-1 恢复发布；UPF-2 上已有用户的下行报文经 GRE 隧道转发到 UPF-1 | PDN→UPF-1→GRE隧道→UPF-2→UE（已在 UPF-2 的用户） |

> 来源：UDG 侧 `主用PGW-U_UPF故障场景_19644452.md`（"说明"段落）

### 3.4 C-U 对称冗余保护机制（★ 重点）

本特性与 UDG 侧 GWFD-010107 构成 **C-U 对称冗余保护对**，两侧职责明确分工：

```
┌──────────────────────────────────────────────────────────────────┐
│ UNC（C 面：GGSN/PGW-C/SMF） - WSFD-107021                        │
│ 冗余决策方：选择主用还是备用 UPF                                  │
│                                                                  │
│   ┌─────────────────────────────────────────────────────┐        │
│   │ ① 配置静态IP段→主备UPF组绑定（PRIORITY=0/1）         │        │
│   │ ② PFCP 心跳检测感知 UPF 故障                         │        │
│   │ ③ 主用故障 → 决策选择备用 UPF                        │        │
│   │ ④ 通过 N4 PFCP 指示 UPF 建立会话                     │        │
│   └─────────────────────────────────────────────────────┘        │
└──────────────────────────┬───────────────────────────────────────┘
                           │
                           │ N4 PFCP（会话建立指示）
                           ▼
┌──────────────────────────────────────────────────────────────────┐
│ UDG（U 面：PGW-U/UPF） - GWFD-010107                              │
│ 冗余执行方：通过 GRE 隧道实现实际报文冗余转发                      │
│                                                                  │
│   ┌─────────────────────────────────────────────────────┐        │
│   │ ① 主备 UDG 间建立 GRE Tunnel（SET REDUNDUSER 开关） │        │
│   │ ② 主用故障：备用 UDG 承载业务（不经 GRE）            │        │
│   │ ③ 主用恢复：备用 UDG 上用户下行报文经 GRE 转发回主用 │        │
│   │ ④ OSPF/BGP 发布/撤销下行路由（网段级）              │        │
│   └─────────────────────────────────────────────────────┘        │
│                                                                  │
│ 数据一致性约束：                                                  │
│   UNC 静态段 ⊆ UDG 地址池（GWFD-010104）                          │
└──────────────────────────────────────────────────────────────────┘
```

**C-U 对称性的关键证据**：

| 对照维度 | WSFD-107021（UNC/C面） | GWFD-010107（UDG/U面） | 对称性 |
|---------|------------------------|------------------------|--------|
| 首次发布版本 | v01 / UNC 20.5.0 | v01 / UDG 20.5.0 | **完全对称**（同版本首发） |
| 网元角色 | GGSN/PGW-C/SMF（控制面） | PGW-U/UPF（用户面） | C-U 对称 |

| 决策/执行 | 决策方（选主备 UPF） | 执行方（GRE 隧道转发） | 职责分工 |
| 触发机制 | PFCP 心跳检测 | 路由发布/撤销 + GRE 隧道 | 互补 |
| 系统影响 | 无（文档声明） | 有（GRE 转发导致性能下降） | 影响在 U 面 |


## 4. 配置规则

### 4.1 激活步骤

本特性无独立 License。激活即"配置使能"：在 WSFD-010502 静态 IP 段→UPF 绑定配置基础上，增加主备 UPF 绑定。

**通用激活步骤骨架（10步）**：

```
步骤1：创建本地静态地址地址池（POOLTYPE=UDM, CHECKIPVALID=Enable, HASVPN=Enable）
  └── ADD ADDRPOOL

步骤2（可选）：开启静态地址用户白名单检查
  └── ADD POOLBINDAPN     → 绑定地址池与APN

步骤3：配置本地地址池里的地址段
  └── ADD SECTION          → V4STARTIP/V4ENDIP，最多64个地址段

步骤4：配置地址池组（POOLGRPTYPE=UDM）
  └── ADD ADDRPOOLGRP

步骤5：配置地址池与地址池组绑定
  └── ADD POOLBINDGRP

步骤6：配置静态地址用户使用的【主用、备用】UPF（ADDRALLOCMODE=INHERIT）
  └── ADD UPNODE × 2       → ★ 本特性关键：配置 2 个 UPF 实例

步骤7：配置静态地址用户使用的 UPF 组（UPFGRPTYPE=UDM）
  └── ADD ADDRUPGROUP

步骤8：配置【主、备】UPF 与 UPF 组绑定（含 PRIORITY）
  └── ADD UPFBINDGRP × 2   → ★ 本特性核心：PRIORITY=0(主) + PRIORITY=1(备)

步骤9：配置地址池组与 UPF 组绑定
  └── ADD POOLGRPMAP
```

> 来源：`激活静态地址用户路由冗余_76652607.md`（"操作步骤"章节）

## 5. 配置案例

### 5.1 典型场景：静态 IP 网段→主备 UPF 绑定（企业园区冗余保护）

**场景描述**：某企业用户接入网络时，为用户分配的 IP 地址采用静态 IP 地址分配方式。配置 UPF 名称为 UPF_Instance_1 的 UPF1 为**主用 UPF**，配置 UPF 名称为 UPF_Instance_2 的 UPF2 为**备用 UPF**，当主用 UPF 故障时，可以为用户选择到备用 UPF 进行业务，提高业务可靠性。

UPF_Instance_1 和 UPF_Instance_2 上已经配置包含网段 **10.0.0.0~10.1.2.3** 的地址池信息，需要配置静态 IP 地址段 10.0.0.0~10.1.2.3 到 UPF_Instance_1 和 UPF_Instance_2 的关系：在主用 UPF 正常时，为用户选择到主用 UPF；在主用 UPF 故障时，为用户选择到备用 UPF。

**MML命令序列（原样保留）**：

```
//创建本地地址池。
ADD ADDRPOOL: POOLNAME="pool01", IPVERSION=IPv4, POOLTYPE=UDM, CHECKIPVALID=Enable, HASVPN=Enable, VPNINSTANCE="VRF-A";

//配置地址池与APN的绑定关系。
ADD POOLBINDAPN:APN="huawei.com",POOLNAME="pool01";

//配置本地地址池里的IPv4地址段。
ADD SECTION: POOLNAME="pool01", SECTIONNUM=1, IPVERSION=IPv4, V4STARTIP="10.0.0.0", V4ENDIP="10.1.2.3";

//配置地址池组名称。
ADD ADDRPOOLGRP: POOLGRPNAME="poolgroup1", POOLGRPTYPE=UDM;

//配置地址池与地址池组绑定。
ADD POOLBINDGRP: POOLGRPNAME="poolgroup1", POOLNAME="pool01";

//配置静态地址用户使用的主、备UPF。
ADD UPNODE: NFINSTANCENAME="UPF_Instance_1", ADDRALLOCMODE=INHERIT;
ADD UPNODE: NFINSTANCENAME="UPF_Instance_2", ADDRALLOCMODE=INHERIT;

//配置静态地址用户使用的UPF组。
ADD ADDRUPGROUP: UPFGRPNAME="upfgroup1", UPFGRPTYPE=UDM;

//配置主、备UPF与UPF组绑定。
ADD UPFBINDGRP: UPFGRPNAME="upfgroup1", UPFID="UPF_Instance_1", PRIORITY=0;
ADD UPFBINDGRP: UPFGRPNAME="upfgroup1", UPFID="UPF_Instance_2", PRIORITY=1;

//配置地址池组与UPF组绑定。
ADD POOLGRPMAP: MAPPINGNAME="mapping01", POOLGRPNAME="poolgroup1", UPFGRPNAME="upfgroup1";
```

> 来源：`激活静态地址用户路由冗余_76652607.md`（"任务示例"章节，原样保留 MML）

### 5.2 场景变体对照（与 WSFD-010502 单 UPF 绑定对照）

| 变体 | 核心差异 | 关键命令/参数 | 与本特性关系 |
|------|---------|--------------|-------------|
| WSFD-010502 单 UPF 绑定 | 1 个 UPFBINDGRP，无 PRIORITY | ADD UPFBINDGRP × 1 | 父场景（无冗余） |
| **WSFD-107021 主备 UPF 绑定**（本特性） | 2 个 UPFBINDGRP，PRIORITY=0/1 | ADD UPFBINDGRP × 2（PRIORITY=0 + PRIORITY=1） | 冗余增强 |
| 主用 UPF 故障切换 | UNC 自动选择备用 UPF | 无额外配置（PFCP 心跳触发） | 故障场景 |
| 主用 UPF 恢复回切 | 新用户回主用；老用户经 GRE 隧道 | 无额外配置（UDG 侧 GWFD-010107 GRE 隧道） | 恢复场景 |

> 说明：本特性的 MML 脚本与 WSFD-010502 静态 IP 段→UPF 绑定脚本**高度相似**，差异仅在于：①ADDRPOOL 增加 HASVPN/VPSINSTANCE；②UPNODE 配置 2 个；③UPFBINDGRP 执行 2 次并指定 PRIORITY。

### 6.5 常见问题与排查

| 问题现象 | 可能原因 | 排查方法 |
|---------|---------|---------|
| 主备切换不生效 | UPFBINDGRP 未配置 PRIORITY 或只配置了 1 个 UPF | LST POOLGRPMAP / 检查 UPFBINDGRP 是否配置 2 次（PRIORITY=0 + PRIORITY=1） |
| 主用故障后用户未切到备用 | UNC 未感知到 UPF 故障（PFCP 心跳异常） | 检查 N4 接口连通性；检查 PFCP 心跳配置 |
| 用户激活失败 | UNC 静态段未包含在 UDG 地址池内（C-U 数据不一致） | 核对 UNC 静态段 ⊆ UDG（GWFD-010104）地址池 |
| 用户激活失败 | ADDRPOOL 配置但未配 SECTION | 补配 ADD SECTION |
| VPN 绑定不一致 | ADDRPOOL 的 HASVPN/VPNINSTANCE 与 UPF 侧不一致 | 核对两侧 VPN 配置，保持一致 |
| 地址池地址段超限 | 一个池超过 64 个地址段 | 拆分多个地址池 |
| UPF 节点未找到 | NFINSTANCENAME 未通过 ADD PNFPROFILE 配置 | 先配置 ADD PNFPROFILE，再 ADD UPNODE |
| 主用恢复后老用户业务异常 | UDG 侧 GWFD-010107 GRE 隧道未配置或故障 | 检查 UDG 侧 SET REDUNDUSER 开关、ADD GRETUNNEL 配置 |
