## 1. 概述

### 1.1 特性定义（是什么）

鉴权功能实现对**用户的身份识别和鉴权**，以及对用户业务请求有效性的判断，使合法用户能使用网络提供的业务。鉴权过程和移动性管理相关过程**关联发生**（伴随附着/注册/RAU/TAU/服务请求等流程触发）。本特性覆盖 **2G&3G、4G、5G 三代**，分别由 UNC 的 **SGSN、MME、AMF** 承担鉴权管理功能，与 HLR/HSS/UDM（含 ARPF）以及 AUSF（5G）协同完成**接入层双向鉴权与密钥协商**（AKA，Authentication and Key Agreement）。

**★与 WSFD-011305（Radius鉴权接入）的本质区别**：本特性是**底层接入鉴权**——对 **SIM/USIM 卡身份（IMSI/SUPI）** 进行鉴权，使用 AKA 系列协议（GSM AKA / UMTS AKA / EPS AKA / 5G AKA / EAP-AKA'），由 UE 与 HLR/HSS/UDM 交互完成；WSFD-011305 是**应用层/APN 级业务鉴权**——对**用户名密码**进行鉴权，使用 PAP/CHAP via Radius 或本地匹配，由 UE/UNC 与 AAA Server 交互完成。两者鉴权对象、鉴权协议、鉴权时机、鉴权网元完全不同，属并列的两套鉴权体系。

> 来源：`鉴权功能（2G&3G）/特性概述_85223131.md`（"定义"章节）、`鉴权功能（4G）/特性概述_70014629.md`（"定义"章节）、`鉴权功能（5G）/特性概述_60375189.md`（"定义"章节）

## 2. 核心概念

### 2.1 关键术语

| 术语 | 全称 | 说明 |
|------|------|------|
| **AKA** | Authentication and Key Agreement（鉴权与密钥协商） | 鉴权和密钥协商流程的统称，包含安全认证与密钥同步协商两个作用 |
| **GSM AKA** | GSM 鉴权 | 基于 SIM 卡和 SGSN 中的鉴权**三元组**（RAND、SRES、Kc）进行；使用 GEA 加密算法 |
| **UMTS AKA** | UMTS 鉴权 | 基于用户的 USIM 卡和 SGSN 中的鉴权**五元组**（RAND、XRES、AUTN、CK、IK）进行；使用 UEA 加密算法；较 GSM 增加**一致性检查**和**终端对网络的鉴权**功能 |
| **EPS AKA** | EPS 鉴权 | 4G 鉴权，基于 EPS 鉴权**四元组**（RAND、AUTN、XRES、K_ASME）进行；MME 扮演 ASME（Access Security Management Entity）角色 |
| **5G AKA** | 5G 鉴权 | 基于 EPS AKA 演变，**增加归属网络鉴权流程以防欺诈攻击**；鉴权向量为四元组（RAND、AUTN、XRES*、K_AUSF）；AMF 完成服务网络鉴权，AUSF 完成归属网络鉴权 |
| **EAP-AKA'** | EAP-AKA Prime | 基于 USIM 的 EAP 认证方式；鉴权向量为**五元组**（RAND、AUTN、XRES、CK'、IK'）；由 AUSF 完成网络侧鉴权 |
| RAND | Random Challenge | 网络提供给 UE 的不可预知随机数；UE 用 RAND 计算鉴权响应参数及安全保密参数；GSM/UMTS/EPS 中长度 16 字节 |
| AUTN | Authentication Token | 网络提供给 UE 的参数，用于**UE 对网络进行鉴权**（双向鉴权中的"用户鉴权网络"方向）；UMTS/EPS 中长度 17 字节 |
| SRES | Signed Response | GSM 鉴权期望响应参数（由 MS 根据 RAND 和 Kc 计算） |
| XRES / XRES* | Expected Response | 期望响应参数；XRES 用于 GSM/UMTS/EPS/EAP-AKA' 网络侧鉴权比对；**XRES\*** 由 UDM 根据 XRES、RAND 和服务网络名称推衍，仅用于 5G AKA 归属网络鉴权 |
| RES / RES* | Response | UE 实际计算的响应参数；RES* 发送给 AMF，AMF 计算 HRES* 与存储的 HXRES* 比对完成服务网络鉴权 |
| CK | Cipher Key | 加密密钥；作为用户信息（语音及用户传输的其它内容）的加密参数；由 UDM 和 UE 根据 RAND 和 K 计算；UMTS 长度 16 字节 |
| IK | Integrity Key | 完整性密钥；作为网络与终端信令的完整性保护参数；UMTS 长度 16 字节 |
| CK' / IK' | - | EAP-AKA' 算法中，由 UDM 根据 CK 和 IK 计算得出，作为鉴权向量的一部分发送给 AUSF |
| K | Key | 用户**根密钥**；用户开户后得到，存储在 UDM/HLR/HSS 和 SIM/USIM 卡中；所有衍生密钥的源头 |
| K_ASME | EPS 根密钥 | 根据 CK/IK 以及 ASME（MME）的 PLMN ID 推演得到；长度 32 字节；E-UTRAN 接入下 MME 扮演 ASME |
| K_AUSF | 归属网络 AUSF 密钥 | 5G 密钥架构中归属网络层密钥；EAP-AKA' 中由 AUSF/UE 根据 CK'/IK' 计算；5G AKA 中由 UDM/UE 根据 CK/IK 计算 |
| K_SEAF | 锚点密钥 | 5G 锚点密钥（SEAF 是 AMF 内的功能组件）；由 UE 和 AUSF 根据 K_AUSF 计算得出，通过 AUSF 发送给 AMF；3GPP 与非 3GPP 切换时共享，无需重新认证 |
| K_AMF | 服务网络 AMF 密钥 | UE 和 AMF 根据 K_SEAF 计算得出，用于推衍后续 NAS 和空口密钥、非 3GPP 接入密钥 |
| KSI_ASME / ngKSI | Key Set Identifier | NAS 密钥集标识；4G 中为 KSI_ASME，5G 中为 ngKSI；用于识别 K_AMF 和鉴权成功后产生的安全上下文 |
| ABBA | - | 5G 鉴权信元，用于实现安全特性的降级保护 |
| SUPI / SUCI | Subscription Permanent/Concealed Identifier | 5G 用户永久/隐藏标识；当 AMF 判断 UE 已有合法 5G-GUTI 时消息包含 SUPI，否则包含 SUCI（加密） |
| 5G-GUTI | 5G Globally Unique Temporary UE Identity | 5G 全局唯一临时 UE 标识 |
| ASME | Access Security Management Entity | 接入安全管理实体；E-UTRAN 接入模式下 MME 扮演 ASME 角色，从 HSS 接收顶层密钥 K_ASME |
| ARPF | Authentication Credential Repository and Processing Function | UDM 内的功能组件；负责存储根密钥并生成鉴权向量（5G HE AV） |
| SEAF | Security Anchor Function | AMF 内的功能组件；5G 锚点密钥 K_SEAF 的处理点 |
| 5G HE AV | 5G Home Environment Authentication Vector | UDM 生成的归属环境鉴权向量，包含 RAND、AUTN、XRES*、K_AUSF |
| 5G SE AV | 5G Serving Environment Authentication Vector | AUSF 生成发送给 AMF 的服务环境鉴权向量，包含 RAND、AUTN、HXRES*（移除 K_SEAF） |

> 来源：`鉴权功能（5G）/实现原理（适用于5G）_66260580.md`（密钥架构）、`鉴权功能（4G）/特性概述_70014629.md`（EPS 四元组定义）、`鉴权功能（4G）/实现原理（适用于4G）_70014630.md`（K_ASME/ASME）、`鉴权功能（2G&3G）/特性概述_85223131.md`（GSM 三元组/UMTS 五元组定义）、`鉴权功能（5G）/特性概述_60375189.md`（5G 认证方法对照表）

### 2.3 5G 密钥推导链（★5G 鉴权核心架构）

5G 网络安全密钥架构（自顶向下逐层推衍）：

```
根密钥 K（存储在 UDM/ARPF 和 SIM/USIM 卡）
   │
   │ UDM 根据 RAND 和 K 计算
   ▼
CK、IK（加密密钥、完整性密钥）
   │
   ├──→ (5G AKA) UDM 和 UE 根据 CK、IK 计算 K_AUSF
   └──→ (EAP-AKA') UDM 根据 CK、IK 计算 CK'、IK' → AUSF 和 UE 根据 CK'、IK' 计算 K_AUSF
   │
   ▼
K_AUSF（归属网络 AUSF 密钥）
   │
   │ UE 和 AUSF 根据 K_AUSF 计算
   ▼
K_SEAF（锚点密钥，AUSF 发送给 AMF）
   │
   │ UE 和 AMF 根据 K_SEAF 计算
   ▼
K_AMF（服务网络 AMF 密钥）
   │
   ├──→ K_NASint（NAS 信令完整性保护）、K_NASenc（NAS 信令加密性保护）
   ├──→ K_N3IWF（非 3GPP 接入密钥，非 3GPP 接入间不互转）
   ├──→ K_gNB（NG-RAN 密钥）
   │       └──→ K_UPenc/K_UPint（用户面加密/完整性）
   │            K_RRCint/K_RRCenc（RRC 信令加密/完整性）
   └──→ NH（Next Hop，AMF 发送给 gNodeB，与 K_gNB 一起推衍后续密钥）
```

## 3. 原理与流程

### 3.1 ★5G AKA 鉴权机制（★重点）

#### 3.1.1 5G AKA 核心特点

5G AKA 基于 EPS AKA 演变，**增加了归属网络鉴权流程以防欺诈攻击**。这是 5G 相对 4G 在鉴权上的关键安全增强。

- **鉴权网元分工**：AMF 完成**服务网络鉴权**；AUSF 完成**归属网络鉴权**
- **鉴权向量**：四元组，包含 RAND、AUTN、XRES\*、K_AUSF
- **限制**：5G AKA **不支持请求多个鉴权向量，也不支持预获取鉴权向量**（与 4G 预取鉴权集功能形成对比）

> 来源：`鉴权功能（5G）/特性概述_60375189.md`（5G 认证方法对照表）

#### 3.1.2 5G AKA 信令流程（11步，★归属网络鉴权确认是核心增强）

```
1. [内部] AUSF 选择流程（初始注册中选择 AUSF 进行鉴权）
       │
2. AMF ──Nausf_UEAuthentication_CreateUeAuthContext Request──→ AUSF
       │   关键信元：supiOrSuci（UE 的 SUPI 或 SUCI）、servingNetworkName
       │   机制：AMF 判断 UE 已有合法 5G-GUTI → 消息含 SUPI；否则含 SUCI
       ▼
3. AUSF ──Nudm_UEAuthentication_Get Request──→ UDM
       │   机制：AUSF 通过 servingNetworkName 检查 AMF 是否有资格使用该服务网络
       ▼
4. UDM ──Nudm_UEAuthentication_Get Response──→ AUSF
       │   ★关键：UDM 先将 SUCI 解密为 SUPI → 根据 SUPI 获取用户真身份
       │   ★关键：根据用户签约信息（MOD AMDATA）选择鉴权方式
       │   ★生成 5G HE AV：RAND、AUTN、XRES*、K_AUSF
       ▼
5. AUSF ──Nausf_UEAuthentication_CreateUeAuthContext Response──→ AMF
       │   关键信元：authType（"5G_AKA"/"EAP_AKA_PRIME"/"EAP_TLS"）
       │   ★AUSF 存储 XRES*、SUPI/SUCI、K_AUSF
       │   ★AUSF 通过 XRES*、K_AUSF 计算 HXRES*、K_SEAF，生成 5G AV
       │   ★AUSF 移除 K_SEAF，生成 5G SE AV（RAND、AUTN、HXRES*）发送给 AMF
       ▼
6. AMF ──Authentication request──→ UE
       │   关键信元：ngKSI（NAS密钥集标识）、ABBA、RAND、AUTN
       │   机制：UE 鉴权失败则返回 Authentication failure 并明确原因
       ▼
7. UE ──Authentication response──→ AMF
       │   关键信元：Authentication response parameter（RES*）
       │   ★机制：UE 根据 AUTN 鉴权网络 → 计算鉴权响应 RES*、K_AUSF、K_SEAF
       │   ★服务网络鉴权（在 AMF）：AMF 通过 RES* 计算 HRES*，与 HXRES* 比对
       │     - HRES* 和 HXRES* 不一致 → AMF 认为服务网络鉴权失败
       │     - UE 初始 NAS 使用 SUCI → AMF 发 Authentication reject 拒绝鉴权
       │     - UE 初始 NAS 使用 5G GUTI → AMF 重新获取 SUCI 并再次尝试鉴权
       ▼
8. AMF ──Nausf_UEAuthentication_UeAuthConfirm Request──→ AUSF
       │   关键信元：resStar（UE 提供的 RES*）
       │   ★触发：AMF 判断服务网络鉴权成功后，将 RES* 发送给 AUSF
       ▼
9. AUSF ──Nausf_UEAuthentication_UeAuthConfirm Response──→ AMF  ★归属网络鉴权确认
       │   关键信元：authResult（AUTHENTICATION_SUCCESS）、supi、kseaf
       │   ★机制：AUSF 比对收到的 RES* 和存储的 XRES*
       │     - 一致 → 归属网络鉴权成功，消息含 K_SEAF，AMF 推衍 K_AMF
       │     - 不一致 → 归属网络鉴权失败，AMF 发 Authentication reject 或重新身份识别
       ▼
10. AUSF ──Nudm_UEAuthentication_ResultConfirmation Request──→ UDM
       │   机制：通知 UDM 鉴权结果（含 SUPI、鉴权时间、认证方法、服务网络名称）
       ▼
11. UDM ──Nudm_UEAuthentication_ResultConfirmation Response──→ AUSF
           机制：UDM 存储鉴权结果并响应
```

> 来源：`鉴权功能（5G）/鉴权流程（适用于5G）/5G AKA鉴权_80005829.md`（11 步信令流程，逐字保留）

#### 3.1.3 5G AKA 双层鉴权机制（★防欺诈核心）

5G AKA 实现了**服务网络鉴权 + 归属网络鉴权**双层确认：

| 鉴权层 | 执行网元 | 鉴权比对 | 失败处理 |
|--------|---------|---------|---------|
| **服务网络鉴权** | AMF | AMF 计算 HRES*（从 RES*）与 HXRES*（从 5G SE AV）比对 | 不一致则鉴权失败；SUCI 场景发 reject，GUTI 场景重新获取 SUCI |
| **归属网络鉴权**（★5G新增） | AUSF | AUSF 比对收到的 RES* 与存储的 XRES* | 不一致则鉴权失败，发 reject 或重新身份识别 |

> 来源：`5G AKA鉴权_80005829.md`（步骤 7、9 的"相关机制"章节）

### 3.2 ★EAP-AKA' 鉴权机制（★重点）

#### 3.2.1 EAP-AKA' 核心特点

EAP-AKA' 是基于 USIM 的 EAP 认证方式，5G 中用于非 3GPP 接入或运营商策略选择的场景。与 5G AKA 不同，EAP-AKA' **由 AUSF 独立完成网络侧鉴权**（无需 AMF 做服务网络鉴权）。

- **鉴权网元**：AUSF
- **鉴权向量**：五元组，包含 RAND、AUTN、XRES、CK'、IK'

> 来源：`鉴权功能（5G）/特性概述_60375189.md`（5G 认证方法对照表）

#### 3.2.2 EAP-AKA' 信令流程（12步）

```
1. [内部] AUSF 选择流程
2. AMF ──Nausf_UEAuthentication_CreateUeAuthContext Request──→ AUSF
3. AUSF ──Nudm_UEAuthentication_Get Request──→ UDM
4. UDM ──Nudm_UEAuthentication_Get Response──→ AUSF
       ★生成 EAP-AKA' AV：RAND、AUTN、XRES、CK'、IK'
5. AUSF ──Nausf_UEAuthentication_CreateUeAuthContext Response──→ AMF
       ★AUSF 将 EAP-Request/AKA'-Challenge 发送给 AMF，携带 RAND、AUTN
6. AMF ──Authentication request──→ UE
       关键信元：ngKSI、ABBA、RAND、AUTN、★EAP message（传送 EAP 消息）
       机制：UE 鉴权失败返回 Authentication failure；失败原因 "ngKSI already in use" 时核心网可重新选 ngKSI 并发起新鉴权
7. UE ──Authentication response──→ AMF
       关键信元：★EAP message
8. AMF ──Nausf_UEAuthentication_EapAuthMethod Request──→ AUSF
       关键信元：eapPayload（EAP 消息透传给 AUSF）
9. AUSF ──Nausf_UEAuthentication_EapAuthMethod Response──→ AMF
       关键信元：eapPayload、kseaf、authResult（AUTHENTICATION_SUCCESS）
       ★消息含 EAP Success 和 K_SEAF；AUSF 鉴权初始收到 SUCI 则 Response 含 SUPI
10. AMF ──Authentication result──→ UE
       关键信元：ngKSI、ABBA
11. AUSF ──Nudm_UEAuthentication_ResultConfirmation Request──→ UDM
12. UDM ──Nudm_UEAuthentication_ResultConfirmation Response──→ AUSF
```

> 来源：`鉴权功能（5G）/鉴权流程（适用于5G）/EAP AKA'鉴权_79926121.md`（12 步信令流程）

#### 3.2.3 5G AKA vs. EAP-AKA' 差异对照

| 维度 | 5G AKA | EAP-AKA' |
|------|--------|----------|
| 演进来源 | 基于 EPS AKA 演变 | 基于 USIM 的 EAP 认证方式 |
| 鉴权网元 | AMF（服务网络）+ AUSF（归属网络）双层 | AUSF（独立完成网络侧鉴权） |
| 鉴权向量 | 四元组（RAND、AUTN、XRES\*、K_AUSF） | 五元组（RAND、AUTN、XRES、CK'、IK'） |
| AMF 比对 | 计算 HRES* vs HXRES*（服务网络鉴权） | 不做服务网络鉴权，透传 EAP 消息 |
| 多向量/预获取 | **不支持** | 文档未明确限制 |
| 归属网络鉴权确认 | AUSF 比对 RES* vs XRES* | AUSF 在 EAP 会话中完成 |
| NAS 信令携带 | Authentication response parameter（RES*） | EAP message 信元 |
| N12 接口确认消息 | Nausf_UEAuthentication_UeAuthConfirm | Nausf_UEAuthentication_EapAuthMethod |

### 3.4 4G EPS AKA 鉴权机制

#### 3.4.1 EPS 鉴权原理

用户接入网络时，如需鉴权，MME 选取一组未使用过的鉴权四元组向量向 UE 发起鉴权请求（携带 RAND、AUTN、KSI_ASME）。USIM 根据 AUTN 对网络进行鉴权，成功则 UE 计算 RES 返回 MME。MME 用 XRES 与 RES 比对，相同则鉴权成功。成功后 UE 计算并存储 K_ASME 用于后续加密及完整性保护。

EPS 鉴权四元组（RAND、AUTN、XRES、K_ASME）由 MME 向 HSS 请求获取。

> 来源：`鉴权功能（4G）/实现原理（适用于4G）_70014630.md`（"EPS鉴权原理"章节）

#### 3.4.2 4G EPS AKA 流程（2阶段：取鉴权集 + EPS AKA）

**阶段1：MME 到 HSS 取鉴权数据**

```
1. MME ──Authentication Information Request──→ HSS
       携带：IMSI、Serving Network's Identity（SN id）、NT（Network Type）
2. HSS ──Authentication Information Answer──→ MME
       返回：一组或多组 EPS 鉴权向量（RAND、AUTN、XRES、K_ASME）
       机制：请求多组则编序后传送
```

**阶段2：EPS 用户鉴权流程（EPS AKA）**

```
1. MME ──User Authentication Request──→ ME/USIM
       携带：RAND、AUTN、KSI_ASME（从 HSS 获取的多组中取一组）
2. ME/USIM ──User Authentication Response/Failure──→ MME
       机制：
       - USIM 根据 AUTN 对网络鉴权，失败则返回 User Authentication Failure 并指明原因
       - 成功则 UE 根据 RAND 计算 RES 返回 MME
       - MME 用 XRES 和 RES 比对：相同→鉴权成功；不同→发 User Authentication Reject
       - MME 鉴权成功后 UE 计算 K_ASME 用于后续加密及完整性保护
       - 若 MME 鉴权失败且 UE 流程消息携带 GUTI，可发身份识别流程请求 IMSI 决定是否二次鉴权；若携带 IMSI 则直接发 Reject
```

> 来源：`鉴权功能（4G）/实现原理（适用于4G）_70014630.md`（"EPS鉴权流程"章节）

#### 3.4.3 ★4G 预取鉴权集功能（4G 特有）

出于安全考虑，MME 中保存的每组鉴权集只能使用一次，**MME 中最多保存 5 组鉴权集**。当鉴权集用完后需向 HSS 请求新的鉴权集。

UNC 提供**预取鉴权集**功能：在用户附着、TAU 等过程中使用最后一组 MME 保存的鉴权集时，MME **主动向 HSS 请求新的鉴权集**。因新鉴权集流程与正在进行的附着/TAU 携带信息无关，故用户附着/TAU 过程无需等待新鉴权集流程结束，提高了处理速度，改善用户体验。

> 来源：`鉴权功能（4G）/实现原理（适用于4G）_70014630.md`（"预取鉴权集"章节）

### 3.5 2G&3G GSM/UMTS 鉴权机制

#### 3.5.1 鉴权功能内容

鉴权功能主要包括：
- 允许网络对手机提供的身份标识进行检查
- 提供参数供手机计算新的加密密钥（UMTS 和 GSM 鉴权）
- 提供参数供手机计算新的 UMTS 一致性密钥（**仅 UMTS 鉴权支持**）
- 允许手机来鉴权网络（**仅 USIM 卡支持**）

UNC 还支持灵活设置：
- **预取鉴权集功能**：参数 PREGETAUTHSETS 控制（用到最后一组鉴权集时是否预先去 HLR 取）
- **基于 IMSI 号段设置不同鉴权参数**：根据 MS 的 IMSI 号段在配置的鉴权参数表匹配；匹配到则采用基于 IMSI 号段的策略，否则采用 GBAUTHCIPH/IUAUTHCIPH 表中的策略

> 来源：`鉴权功能（2G&3G）/特性概述_85223131.md`（"鉴权功能"章节）

#### 3.5.2 GSM 鉴权 vs. UMTS 鉴权

| 维度 | GSM 鉴权 | UMTS 鉴权 |
|------|---------|-----------|
| 卡类型 | SIM 卡 | USIM 卡 |
| 鉴权向量 | 三元组（RAND、SRES、Kc） | 五元组（RAND、XRES、AUTN、CK、IK） |
| 加密算法 | GEA（GPRS 加密算法） | UEA（UMTS 加密算法） |
| 终端鉴权网络 | **不支持** | **支持**（USIM 验证 AUTN） |
| 一致性检查 | **不支持** | **支持**（IK 一致性密钥） |

**关键增强**：与 GSM 鉴权流程相比，UMTS 鉴权增加了**一致性检查**功能和**终端对网络的鉴权**功能，使 UMTS 网络安全特性进一步增强。

> 来源：`鉴权功能（2G&3G）/特性概述_85223131.md`（"鉴权类型"章节）

#### 3.5.3 2G&3G 鉴权流程

```
1. 网络侧发起鉴权前，若 SGSN 内无鉴权参数三元组或五元组，先发起到 HLR 取鉴权集
2. HLR ──响应消息──→ SGSN
       返回鉴权参数三元组或五元组
       机制：HLR 可能通过多个响应消息返回；返回的鉴权集组数可能少于请求组数（UNC 均可识别支持）
3. SGSN ──鉴权请求──→ MS
       GSM 鉴权携带：RAND、CKSN（Ciphering Key Sequence Number）、Ciphering Algorithm
       UMTS 鉴权携带：RAND、AUTN、KSI（Key Set Identifier）
4. MS ──响应──→ SGSN
       机制：
       - USIM 卡验证 AUTN（终端对网络鉴权）
       - 接受后，MS 根据 RAND 计算密钥（GSM→Kc；UMTS→CK、IK）和期望响应（GSM→SRES；UMTS→XRES）
       - MS 返回 SRES/XRES 给 SGSN
       - SGSN 比对 MS 计算的 SRES/XRES 与本地保存的值，相同则鉴权通过
```

> 来源：`鉴权功能（2G&3G）/实现原理（适用于2G&3G）_92631697.md`（鉴权流程 4 步）

### 3.6 ★5G 与 4G 网络鉴权功能对比

| 维度 | 4G | 5G |
|------|----|----|
| **认证框架** | 3GPP：UE、MME、HSS；非 3GPP：UE、ePDG(untrusted)、TGW、HSGW(trusted)、AAA、HSS | UE、AMF、AUSF、UDM |
| **认证方法** | 3GPP：EPS-AKA；非 3GPP：EAP-AKA、EAP-AKA' | EAP-AKA'、5G AKA（**增加归属网络鉴权确认流程**） |
| **安全锚点** | 3GPP：MME；非 3GPP：3GPP AAA server；**3GPP 和非 3GPP 切换需重新认证** | **AMF**；**3GPP 和非 3GPP 切换无需重新认证，共享锚点密钥** |

> 来源：`鉴权功能（5G）/特性概述_60375189.md`（"5G与4G网络鉴权功能对比"章节）

### 3.7 ★与 WSFD-011305（Radius鉴权接入）的层次关系（★重点结论）

| 维度 | **WSFD-010301 鉴权功能（本特性）** | **WSFD-011305 Radius鉴权接入** |
|------|-----------------------------------|-------------------------------|
| **鉴权层次** | **底层接入鉴权**（网络接入层） | **应用层/APN 级业务鉴权**（业务接入层） |
| **鉴权对象** | **SIM/USIM 卡身份**（IMSI/SUPI/SUCI） | **用户名密码**（PAP/CHAP 或本地匹配） |
| **鉴权协议** | **AKA 系列**（GSM AKA / UMTS AKA / EPS AKA / 5G AKA / EAP-AKA'） | **PAP / CHAP via Radius** 或 UNC 本地匹配 |
| **鉴权网元** | UE ↔ SGSN/MME/AMF ↔ HLR/HSS/UDM(ARPF)/AUSF | UE/UNC ↔ AAA Server（Radius）或 UNC 本地 |
| **鉴权时机** | **移动性管理流程**（附着/注册/RAU/TAU/服务请求）伴随触发 | **PDP上下文/缺省承载/PDU会话创建**时根据 ACCESSMODE 决策 |
| **配置对象** | NGUSRSECPARA / S1USRSECPARA / GBAUTHCIPH / IUAUTHCIPH | SET APNAUTHATTR（APN 级，4种 ACCESSMODE） |
| **License** | 无 | 无 |
| **与 AAA Server 关系** | **不涉及** AAA Server | TRANS_AUTH/NON_TRANS 方式**强依赖** AAA Server（WSFD-011306） |
| **是否相互依赖** | **不依赖** WSFD-011305（本特性三代概述均声明"不涉及与其他特性交互"） | **不依赖** WSFD-010301（011305 概述仅声明依赖 WSFD-011306） |

**层次关系核心结论**：

- **两套独立并列的鉴权体系**：WSFD-010301 是接入层身份鉴权（SIM 卡 + AKA），WSFD-011305 是业务层用户鉴权（账密 + Radius）。两者在文档层面**互不引用**，各自独立声明"不涉及与其他特性的交互"（010301）或"仅依赖 WSFD-011306"（011305）。
- **执行时序**：底层接入鉴权（010301）通常**先于** APN 业务鉴权（011305）发生——UE 先通过 AKA 完成网络接入（注册/附着），再在 PDP上下文/承载/PDU会话创建时根据 APN 的 ACCESSMODE 触发业务鉴权。
- **互不感知**：底层接入鉴权成功是 UE 接入网络的前提，但 APN 业务鉴权决策并不读取底层 AKA 结果；反之亦然。两者通过"UE 接入网络"这一共同状态隐式协同，无直接信令或数据耦合。

> 来源：本特性三代特性概述"与其他特性的交互"章节（均声明"不涉及"）；WSFD-011305 特性概述"与其他特性的交互"章节（仅声明依赖 WSFD-011306）；关系结论基于特性语义与配置对象差异推导

### 3.8 ★与 WSFD-010400（用户数据管理）的协同

| 协同维度 | 说明 |
|---------|------|
| **鉴权数据来源** | 本特性（WSFD-010301）依赖 HLR/HSS/UDM 提供鉴权向量（三元组/四元组/五元组/5G HE AV）。HLR/HSS/UDM 中的用户签约数据（含根密钥 K、鉴权算法、签约的鉴权方式）由 WSFD-010400 用户数据管理特性维护。 |
| **5G 鉴权方式选择** | 5G AKA 鉴权流程步骤4 明确："UDM 根据 SUPI 获取用户真身份，根据**用户签约信息（MOD AMDATA）**选择鉴权方式"。这里的 AMDATA（鉴权数据）即由 WSFD-010400 管理。 |
| **鉴权结果存储** | 5G AKA 流程步骤10-11：AUSF 通过 Nudm_UEAuthentication_ResultConfirmation 通知 UDM 鉴权结果（含 SUPI、鉴权时间、认证方法、服务网络名称），UDM 存储鉴权结果。此 UDM 鉴权结果存储由 WSFD-010400 支持。 |
| **特性文档引用** | 本特性文档**未直接引用** WSFD-010400（本特性三代概述均声明"不涉及与其他特性的交互"）。两者协同基于 UDM/HLR/HSS 作为共同的数据存储网元，属隐式数据依赖，非显式特性依赖。 |

> 来源：`5G AKA鉴权_80005829.md`（步骤4 MOD AMDATA、步骤10-11 UDM 存储鉴权结果）；WSFD-010400 文档清单（同属安全管理功能，管理 UDM/UDR 用户签约数据）

## 5. 配置案例

### 5.1 典型场景一：5G 服务请求流程强制鉴权

**场景描述**：配置所有用户在服务请求流程中需要强制鉴权。系统默认对初始注册、移动性 Inter 注册、空闲态 EPS 到 5GS 注册流程开启鉴权；本场景额外对服务请求流程开启强制鉴权。

**MML命令序列（原样保留）**：

```
//配置所有用户在服务请求流程中需要鉴权。
ADD NGUSRSECPARA: SUBRANGE=ALL_USER, AUTHEVENT=SERVICE_REQUEST, AUTHPERIOD=24;
```

> 来源：`激活鉴权功能（适用于5G）_66260458.md`（"任务示例"完整脚本）

### 5.2 典型场景二：4G 打开鉴权加密开关

**场景描述**：打开 MME 鉴权加密开关，配置用户范围为 DEFAULT（缺省），安全策略为 AUTHANDPROTECTED，加密算法为 EEA2-1，完整性算法为 EIA2-1。

**MML命令序列（原样保留）**：

```
//打开鉴权加密开关
MOD S1USRSECPARA:USRRANGE=DEFAULT,SECPLC=AUTHANDPROTECTED,SUPTCIPHAGTH=EEA2-1,SUPTINTAGTH=EIA2-1;
```

> 来源：`激活鉴权功能（适用于4G）_62847962.md`（"任务示例"完整脚本）

### 5.3 典型场景三：2G&3G 在 INTER 位置更新流程鉴权

**场景描述**：增加所有用户的鉴权配置，在 INTER 位置更新流程需要鉴权。需分别在 GSM 网络（Gb 模式）和 UMTS 网络（Iu 模式）设置。

**MML命令序列（原样保留）**：

```
// 脚本1：在GSM网络中设置。
//增加所有用户的鉴权配置，在INTER位置更新流程进行鉴权。
MOD GBAUTHCIPH: SUBRANGE=ALL_USER, SECPLC=AUTHANDPROTECTED, AUTHEVENT=INTER_RAU-1;

// 脚本2：在UMTS网络中设置。
//增加所有用户的鉴权配置，在INTER位置更新流程进行鉴权。
MOD IUAUTHCIPH: SUBRANGE=ALL_USER, SECPLC=AUTHANDPROTECTED, AUTHEVENT=INTER_RAU-1;
```

> 来源：`激活鉴权功能（适用于2G&3G）_82934116.md`（"任务示例"完整脚本，含 GSM 和 UMTS 两套）

### 6.5 常见问题与排查

| 问题现象 | 可能原因 | 排查方法 | 适用代际 |
|---------|---------|---------|---------|
| 用户附着失败 | 鉴权加密开关未打开（SECPLC 未设为 AUTHANDPROTECTED） | LST S1USRSECPARA 核对 SECPLC；MOD S1USRSECPARA 修正 | 4G |
| 用户附着失败 | HSS 中未签约 EPS 业务或鉴权向量生成失败 | 检查 HSS 用户签约数据；核查取鉴权集流程 | 4G |
| 用户附着失败 | HLR 中未签约 GPRS 业务 | 检查 HLR 用户签约数据 | 2/3G |
| USIM 用户 WCDMA 附着失败 | UMTS 鉴权参数未配置（IUAUTHCIPH） | LST IUAUTHCIPH 核对；MOD IUAUTHCIPH 修正 | 2/3G |
| SIM 用户 GPRS 附着失败 | GSM 鉴权参数未配置（GBAUTHCIPH） | LST GBAUTHCIPH 核对；MOD GBAUTHCIPH 修正 | 2/3G |
| Intra RAU 2G切3G 失败 | USIM 卡故障回复 Auth Failure | SET SOFTPARAOFBIT 设置 DWORD6 BIT6=1（仅发加密） | 2/3G |
| 5G 服务请求未鉴权 | AUTHEVENT 未配置 SERVICE_REQUEST | LST NGUSRSECPARA 核对；ADD NGUSRSECPARA 补充 | 5G |
| 5G 鉴权失败（MAC 错误） | AUTN 验证失败（密钥不匹配或卡故障） | 检查 USIM 卡；核查 UDM 鉴权向量；监控 1929467379 指标 | 5G |
| 5G 鉴权失败（同步失败） | 鉴权向量同步失败 | 核查 UDM/ARPF；监控 1929467381 指标 | 5G |
| 5G 服务网络鉴权失败 | HRES* 与 HXRES* 不一致 | 核查 AMF-AUSF 交互；SUCI 场景会 reject，GUTI 场景重新获取 SUCI | 5G |
| 5G 归属网络鉴权失败 | RES* 与 XRES* 不一致 | 核查 AUSF 比对逻辑；发 reject 或重新身份识别 | 5G |
| 切换流程时延过大 | 开启了强制鉴权 | 网络规划时不建议对切换流程开启强制鉴权 | 5G |

> 来源：综合三代调测文档（调测步骤）、参考信息（告警/软参/测量指标）、特性概述（应用限制/对系统影响）、5G 鉴权策略（切换不建议强制鉴权）
