## 1. 概述

### 1.1 特性定义（是什么）

终端二次鉴权（UPF 转接）功能主要应用在 **5G 场景**下，由于 **UPF 下沉**，当 **DN-AAA 部署到企业或园区**时，SMF 与 DN-AAA 之间不能直接互通，需要经过 **UPF 转接**。此场景下 SMF 和 UPF 间建立设备级的 **N4 隧道（N4 GTP-U 隧道）**，用来转接 SMF 与 DN-AAA 之间二次鉴权的消息。二次鉴权消息对 UPF 而言为普通的数据报文。

> 来源：`WSFD-108007 终端二次鉴权特性概述_73984008.md`（"定义"章节）[EV-FK-01]

### 1.2 适用 NF（UDG/UNC 网元）

| 涉及 NF | 网元角色 | 支持版本 | 功能说明 |
|---------|---------|---------|---------|
| UPF / SMF | 用户面（UPF）+ 控制面（SMF） | UNC 20.8.0 及之后版本 | UPF 通过 N4 GTP-U 隧道转发 SMF 与 DN-AAA 之间二次鉴权的消息 |
| AAA Server | 外部 DN-AAA 鉴权服务器（企业/园区） | 无特殊需求 | 实现对用户的二次鉴权（部署在企业/园区内部） |

### 1.8 应用场景

UPF 下沉，DN-AAA 部署到企业或园区，SMF 与 DN-AAA 之间不能直接互通，**需要 UPF 转发** SMF 与 DN-AAA 之间二次鉴权的消息。

> 来源：`WSFD-108007 终端二次鉴权特性概述_73984008.md`（"应用场景"章节）[EV-FK-01]

### 1.9 对系统的影响

如果 UPF 到 DN-AAA 抄送服务器的转发路径故障或者抄送服务器故障时，**SMF 无法感知抄送服务器的状态**。

> 来源：`WSFD-108007 终端二次鉴权特性概述_73984008.md`（"对系统的影响"章节）[EV-FK-01]

### 1.10 应用限制

- 当启用终端二次鉴权功能后，**SMF 不支持 EAP 鉴权方式，只支持 PAP、CHAP 鉴权**；
- SMF 与 DN-AAA 之间的二次鉴权消息**只支持 Radius 协议，不支持 Diameter 协议**。
- 一个 SMF 网元上部署的所有业务中，**直连 AAA 和经 UPF 中转 AAA 场景中的 Radius Server IP 不可以相同**，否则会造成直连 AAA 场景业务受损。

> 来源：`WSFD-108007 终端二次鉴权特性概述_73984008.md`（"应用限制"章节）[EV-FK-01]

## 2. 原理

### 2.1 ★二次鉴权（DN-AAA，企业 DN 场景）机制（核心）

#### 2.1.1 二次鉴权经 UPF 转接组网方案

UPF 下沉到边缘，连接企业、园区网络。企业、园区部署 DN-AAA，与 SMF 间不能直接互通。SMF 和 DN-AAA 之间的二次鉴权消息**经 UPF 转接**。

```
┌─────────────────────────────────────────────────────────────────────┐
│ 5G 组网（UPF 下沉 + 企业/园区 DN-AAA）                                │
│                                                                     │
│   ┌─────────┐    ┌──────────────────┐    ┌─────────────────────┐    │
│   │   UE    │    │       SMF        │    │   DN-AAA Server     │    │
│   │ (终端)  │    │   (UNC 控制面)   │    │ (企业/园区鉴权AAA)  │    │
│   └────┬────┘    └────────┬─────────┘    └──────────┬──────────┘    │
│        │                  │                         │               │
│        │  PDU 会话建立    │                         │               │
│        │─────────────────▶│                         │               │
│        │                  │  二次鉴权 Radius 消息   │               │
│        │                  │  (需经 UPF 转接)        │               │
│        │                  └──────────┬──────────────┘               │
│        │                             │                              │
│        │                  ┌──────────▼──────────────┐               │
│        │                  │          UPF            │               │
│        │                  │  (下沉边缘，N4 GTP-U    │               │
│        │                  │   隧道转发 Radius 消息) │               │
│        │                  └──────────┬──────────────┘               │
│        │                             │  转发 IP 报文（Radius）      │
│        │                             └─────────────────────────────▶│
│        │                                                           │
│   说明：SMF 与 DN-AAA 不能直接互通，由 UPF 在 N4 GTP-U 隧道上       │
│   承载 Radius 鉴权消息（对 UPF 而言为普通数据报文）                  │
└─────────────────────────────────────────────────────────────────────┘
```

> 来源：`WSFD-108007 终端二次鉴权特性概述_73984008.md`（"原理概述/二次鉴权经 UPF 转接组网方案"章节及图1）[EV-FK-01]

#### 2.1.2 二次鉴权消息经 UPF 转接流程（4 步）

```
步骤1：SMF 选择 UPF 创建 Radius 消息专用的 N4 会话
  └── SMF → UPF：PFCP Session Establishment Request 消息

步骤2：UPF 识别专用 N4 会话
  └── UPF 根据 PDR/FAR 中携带的 source interface、destination interface
      及各自对应的 3gpp interface type，识别此会话为创建 N4 GTP-U 隧道的
      专用 N4 会话

步骤3：SMF → DN-AAA 方向（解封装转发）
  └── UPF 收到 SMF 发送的报文后，解析 GTP-U 头的信息，
      匹配到 SMF 下发的 PDR，使用对应的 FAR 进行报文转发，
      剥离 GTP-U 信息，直接转发 IP 报文（即 Radius 消息）给 DN-AAA

步骤4：DN-AAA → SMF 方向（加封装回送）
  └── UPF 收到 DN-AAA 发送的 Radius 消息后，匹配 SMF 下发的 PDR，
      使用对应的 FAR，对收到的 IP 报文进行 GTP-U 加封装，
      转发 GTP-U 报文给 SMF，完成消息转发
```

> 来源：`WSFD-108007 终端二次鉴权特性概述_73984008.md`（"原理概述/二次鉴权消息经 UPF 转接流程"章节及图2，原文 1~4 步）[EV-FK-01]

#### 2.1.3 终端二次鉴权功能要点（4 项）

| # | 功能要点 | 说明 |
|---|---------|------|
| 1 | 对接多个企业/园区 | UPF 可能同时对接多个企业、园区，UNC 同时支持**不少于 6000 个 UPF 转接会话** |
| 2 | 支持重选 UPF 转发 | N4-U 会话建立后，如 UPF 故障，SMF 会选择另外的 UPF 发送二次鉴权消息 |
| 3 | 支持重选 DN-AAA | 当 DN-AAA Server 故障，SMF 发送二次鉴权消息无响应，SMF 会选择另外的 DN-AAA 发送二次鉴权消息 |
| 4 | 边缘和中心 UPF 主备容灾 | 边缘 UPF 和中心 UPF 之间进行容灾，正常情况下用户通过边缘 UPF 访问园区业务；当边缘 UPF 故障时用户通过中心 UPF 访问园区业务 |

> 来源：`WSFD-108007 终端二次鉴权特性概述_73984008.md`（"原理概述/终端二次鉴权功能"章节 1~4 条）[EV-FK-01]

### 2.2 ★与 WSFD-011305（APN Radius 鉴权）/ WSFD-011306（Radius 功能）的关系（一次鉴权 vs 二次鉴权）

| 关联特性 | 关系类型 | 鉴权时机 | 鉴权对象 / AAA | 文档依据 |
|---------|---------|---------|---------------|---------|
| **WSFD-011305 Radius鉴权接入** | **强依赖（一次鉴权）** | PDU 会话建立时的一次鉴权（接入时） | 接入侧 AAA（运营商/ISP/企业 AAA，地址空间属于运营商或 ISP） | 本特性"与其他特性的交互"明确声明：**使用终端二次鉴权功能前要激活 Radius 鉴权接入功能** [EV-FK-01]；激活文档必备事项要求完成 Radius 鉴权接入配置 [EV-FK-03] |
| **WSFD-011306 Radius功能** | **强依赖（Radius 协议支撑）** | 一次鉴权与二次鉴权共用 | 提供 Radius Server 组、AAA Server 数据、PAP/CHAP 协议 | 激活文档必备事项要求完成 Radius 功能配置 [EV-FK-03]；本特性限制"只支持 PAP/CHAP、Radius 协议"由 Radius 功能承载 [EV-FK-01] |
| **WSFD-108007 本特性** | 二次鉴权（增强） | 会话建立后访问企业 DN 时的二次鉴权（访问 DN 时） | **DN-AAA（部署在企业/园区内部）** | 本特性定义章节 [EV-FK-01] |

**核心结论（一次 vs 二次鉴权）**：

- **一次鉴权（WSFD-011305）**：用户在 **PDU 会话建立时**进行的接入鉴权，鉴权对象是 UE 接入运营商网络的合法性，AAA 侧是接入侧 AAA（可能部署在运营商/ISP/企业网）。
- **二次鉴权（WSFD-108007）**：用户**访问特定企业数据网络（DN）时**进行的补充鉴权，鉴权对象是 UE 访问企业/园区 DN 的授权，AAA 侧是 **DN-AAA（部署在企业/园区内部）**。
- **强依赖链路**：WSFD-108007 文档明确声明**使用终端二次鉴权功能前要激活 Radius 鉴权接入功能（WSFD-011305）**，且激活文档前置条件进一步要求 **Radius 功能（WSFD-011306）** 配置完成。即三者构成 **"Radius 功能 → Radius 鉴权接入 → 终端二次鉴权"** 的强依赖链。
- **鉴权协议共用**：一次鉴权与二次鉴权都通过 Radius 协议承载（PAP/CHAP），但**二次鉴权不支持 EAP、不支持 Diameter**（本特性应用限制）。

> 来源：`WSFD-108007 终端二次鉴权特性概述_73984008.md`（交互、定义、应用限制章节）[EV-FK-01]、`激活终端二次鉴权_24743509.md`（必备事项）[EV-FK-03]

### 2.3 ★与接入方式（企业专网 IPSec/GRE）的场景关联

终端二次鉴权（DN-AAA 经 UPF 转接）典型应用于**企业专网接入**场景，与 APN 业务域接入方式类特性的关联如下：

| 接入方式特性 | 场景关联 | 关系说明 |
|------------|---------|---------|
| IPFD-016000 IPSec 功能（UNC/SMF）| 企业专网安全接入（加密隧道） | 企业分支/园区通过 IPSec 隧道接入 5G 核心，DN-AAA 部署在企业内网，SMF 与 DN-AAA 不能直连，由 UPF 经 N4 GTP-U 转接 Radius 二次鉴权消息。IPSec 保障的是 UE-到-企业网的承载层安全，二次鉴权保障的是访问企业 DN 的应用层授权 |
| IPFD-015002 GRE（UDG+UNC）| 企业专网轻量封装接入（不加密） | GRE 提供企业 DN 的隧道封装；DN-AAA 部署在企业侧时，二次鉴权 Radius 消息经 UPF 转接，GRE 与二次鉴权为不同层面的机制（GRE 承载层 vs. Radius 鉴权层） |
| 企业 DN 场景共性 | MEC 解决方案归类 | 本特性归类于 **MEC 解决方案**（多接入边缘计算），与 UPF 下沉、企业园区本地化部署的 MEC 组网方案天然契合 |

**核心场景关联结论**：

- **UPF 下沉 + 企业专网** 是触发本特性的典型组网：当企业通过 IPSec/GRE 等接入方式构建专网，并将 DN-AAA 部署在企业内部时，SMF 与 DN-AAA 物理上不可直达，必须经 UPF 转接。
- **接入方式（IPSec/GRE）解决"数据面如何通"**，**终端二次鉴权解决"访问企业 DN 谁被授权"**，二者在企业专网场景中互补。
- 本特性文档未直接引用 IPFD-015002/IPFD-016000，关系基于 **UPF 下沉 + 企业/园区 DN-AAA 部署**的场景语义推断（Stage 3 横向分析时进一步验证）。

> 来源：`WSFD-108007 终端二次鉴权特性概述_73984008.md`（定义、应用场景、原理章节）[EV-FK-01]；接入方式关联基于特性归类（MEC 解决方案）与场景语义

### 4.3 配置顺序强约束（文档明确）

| 约束 | 说明 |
|------|------|
| **UPF 侧 VPN 前置** | 配置"中转 UPF 与 Radius 服务器、客户端的绑定关系"之前需要**先在 UPF 上执行 ADD NETWORKINSTVPNMAP 命令**。否则 SMF 尝试建立 N4-U 会话失败，需延迟一段时间（最大 15 分钟）后才会再自动重试。 |
| **UPFRDSSVR 先于 UPFRDSCLIENTIP** | 需先配置 ADD UPFRDSSVR，再配置 ADD UPFRDSCLIENTIP，否则 **AAA 计费业务无法正常进行**。 |
| **UPFRDSCLIENTIP 必须最后执行** | ADD UPFRDSCLIENTIP 执行后 SMF 会**立即触发建链**，需要其他配置都配置完成后最后执行；若中间有其它命令修改，需要先执行 RMV UPFRDSCLIENTIP 再重新执行 ADD UPFRDSCLIENTIP，否则 SMF 建立 N4-U 会话失败（延迟最大 15 分钟自动重试）。 |
| **UPLISTNAME 小写建议** | UPLIST4RDS、UPFRDSCLIENTIP、UPFRDSSVR 命令配置中的 UPLISTNAME 参数**建议统一用小写**。 |

### 4.5 约束条件

| 约束类型 | 约束内容 |
|---------|---------|
| **鉴权方式限制** | 启用本特性后，**SMF 不支持 EAP 鉴权方式，只支持 PAP、CHAP 鉴权** |
| **协议限制** | SMF 与 DN-AAA 之间的二次鉴权消息**只支持 Radius 协议，不支持 Diameter 协议** |
| **Radius Server IP 唯一性** | 一个 SMF 网元上部署的所有业务中，**直连 AAA 和经 UPF 中转 AAA 场景中的 Radius Server IP 不可以相同**，否则会造成直连 AAA 场景业务受损 |
| **UPF VPN 前置** | UPF 侧必须先执行 ADD NETWORKINSTVPNMAP，否则 N4-U 会话建立失败 |
| **配置顺序** | UPFRDSSVR 必须先于 UPFRDSCLIENTIP；UPFRDSCLIENTIP 必须最后执行 |
| **强依赖** | 使用本特性前必须已激活 WSFD-011305 Radius 鉴权接入（及 WSFD-011306 Radius 功能） |

## 5. 配置案例

### 5.1 典型场景：配置 UPF 中转 Radius 鉴权消息（企业 DN-AAA 场景）

**场景描述**：UPF 下沉到企业/园区边缘，DN-AAA（192.168.10.2）部署在企业内部，SMF（192.168.0.20）与 DN-AAA 不能直接互通，需配置 UPF 中转 Radius 鉴权消息，完成终端二次鉴权。

**MML 命令序列（原样保留激活文档任务示例脚本）**：

```
//增加与UPF对接的本端GTP-U地址。

ADD CPGTPUADDR:IPVERSION=IPV4,IPV4ADDR="192.168.0.20";

//向RADIUS服务器使用的UPF列表中增加UPF。

ADD UPLIST4RDS: UPLISTNAME="uplist1", UPINSTANCEID="up1";

//可选：增加RADIUS服务器的UPF列表中指定UPF的功能属性

ADD RDSUPFCTRL: UPLISTNAME="uplist1", UPINSTANCEID="up1",PREFERENCE=1,LOCKED=ENABLE;

//增加中转UPF与Radius客户端的绑定关系。

ADD UPFRDSSVR: SERVERTYPE=AUTHENTICATION, IPVERSION=IPV4, SERVERIPV4="192.168.10.2", UPLISTNAME="uplist1";

ADD UPFRDSCLIENTIP: CLIENTYPE=AUTHENTICATION, IPVERSION=IPV4, UPLISTNAME="uplist1", VPNINSTANCE="vpnInstance1", CLIENTIPV4="192.168.10.1";
```

> 来源：`激活终端二次鉴权_24743509.md`（"任务示例/脚本"，原文完整保留）[EV-FK-03]

### 5.2 场景变体对照

| 变体 | 场景说明 | 核心配置差异 |
|------|---------|-------------|
| 单企业 DN-AAA | UPF 对接单一企业/园区 DN-AAA | 基础配置（本案例），1 个 UPLIST + 1 个 UPFRDSSVR + 1 个 UPFRDSCLIENTIP |
| 多企业 DN-AAA | UPF 同时对接多个企业/园区（≥6000 会话） | 配置多个 UPLISTNAME，多组 UPFRDSSVR/UPFRDSCLIENTIP，VPNINSTANCE 区分 |
| DN-AAA 计费 + 鉴权 | 同一 DN-AAA 兼做鉴权与计费 | SERVERTYPE/CLIENTYPE 分别配置 AUTHENTICATION 与 ACCOUNTING（或 ACCT_AND_AUTH） |
| 多 UPF 容灾 | 边缘 UPF 故障时中心 UPF 接管 | UPLIST4RDS 增加多 UPINSTANCEID；RDSUPFCTRL 配置 PREFERENCE 优先级 |
| DN-AAA 主备 | DN-AAA Server 主备容灾 | 同一 UPLISTNAME 下配置多个 SERVERIPV4（主备 DN-AAA），依赖"支持重选 DN-AAA"机制 |

> 来源：基于 `WSFD-108007 终端二次鉴权特性概述_73984008.md`（功能要点 1~4：多企业对接、重选 UPF、重选 DN-AAA、边缘中心主备）推导 [EV-FK-01]

### 6.4 常见问题与排查

| 问题现象 | 可能原因 | 排查方法 |
|---------|---------|---------|
| N4-U 会话建立失败，延迟最大 15 分钟自动重试 | UPF 侧未先执行 ADD NETWORKINSTVPNMAP 配置 VPN | 在 UPF 侧补配 ADD NETWORKINSTVPNMAP；重新执行 UPFRDSCLIENTIP（先 RMV 再 ADD） |
| AAA 计费业务无法正常进行 | 未先配置 ADD UPFRDSSVR 就配置了 ADD UPFRDSCLIENTIP | 补配 ADD UPFRDSSVR，再 RMV + ADD UPFRDSCLIENTIP |
| N4-U 会话建立后Radius消息不通 | 中间命令修改后未重新触发建链 | 先 RMV UPFRDSCLIENTIP，再重新 ADD UPFRDSCLIENTIP |
| 二次鉴权鉴权方式异常 | 启用本特性后仍用 EAP 鉴权 | 改用 PAP 或 CHAP（本特性不支持 EAP） |
| 直连 AAA 场景业务受损 | 直连 AAA 与中转 AAA 的 Radius Server IP 相同 | 重新规划 IP，确保两类场景 Radius Server IP 不同 |
| UPF 故障后二次鉴权失败 | 单 UPF 部署，未配多 UPF 容灾 | 配置多 UPINSTANCEID + RDSUPFCTRL 优先级，启用重选 UPF 转发 |
| DN-AAA 故障二次鉴权无响应 | 单 DN-AAA 部署，未配主备 | 配置多 SERVERIPV4（主备 DN-AAA），启用重选 DN-AAA 机制 |
| Radius 客户端 IP 与规划不一致 | CLIENTIPV4 或 VPNINSTANCE 配置错误 | 重新执行激活配置流程；DSP RDSPFCPSESSINFO 核对 |
| Diameter 协议鉴权失败 | 本特性不支持 Diameter | 改用 Radius 协议 DN-AAA |

> 来源：综合 `WSFD-108007 终端二次鉴权特性概述_73984008.md`（应用限制、功能要点）[EV-FK-01]、`激活终端二次鉴权_24743509.md`（配置顺序约束）[EV-FK-03]、`调测终端二次鉴权_74143640.md`（预期/非预期处理）[EV-FK-04]

### 8.2 ★一次鉴权 vs 二次鉴权 核心结论（文档依据汇总）

| 维度 | WSFD-011305 一次鉴权（Radius 鉴权接入） | WSFD-108007 二次鉴权（终端二次鉴权） | 文档依据 |
|------|----------------------------------------|--------------------------------------|---------|
| 鉴权时机 | PDU 会话建立时（接入时） | 会话建立后访问企业 DN 时 | 两特性"定义"章节 |
| 鉴权对象 | UE 接入运营商网络的合法性 | UE 访问企业/园区 DN 的授权 | 两特性"定义"章节 |
| AAA 侧 | 接入侧 AAA（运营商/ISP/企业） | **DN-AAA（部署在企业/园区内部）** | 本特性"定义"+"应用场景" |
| 物理路径 | SMF ↔ AAA Server（可直连或经网络） | SMF ↔ UPF ↔ DN-AAA（**经 UPF 转接**，N4 GTP-U 隧道） | 本特性"定义"+"原理" |
| 协议 | PAP/CHAP via Radius（不支持 EAP 仅限二次） | PAP/CHAP via Radius（**不支持 EAP、不支持 Diameter**） | 本特性"应用限制" |
| License | 无 | **82200DSF LKV2SECAA01** | 两特性"可获得性" |
| 强依赖 | WSFD-011306 Radius 功能（NON_TRANS/TRANS_AUTH 方式） | **WSFD-011305 Radius 鉴权接入 + WSFD-011306 Radius 功能** | 本特性"与其他特性的交互"+ 激活文档必备事项 |

**核心结论**：

- **一次鉴权是接入鉴权**（WSFD-011305），决定 UE 能否接入运营商网络；**二次鉴权是 DN 访问鉴权**（WSFD-108007），决定 UE 能否访问特定企业/园区 DN。
- **二者非互斥而是级联**：企业 DN 场景下，UE 先通过一次鉴权接入网络，再在访问企业 DN 时触发二次鉴权到 DN-AAA。
- **本特性强依赖 WSFD-011305**（文档明确声明），且通过激活文档必备事项进一步依赖 WSFD-011306，构成 **Radius 功能 → Radius 鉴权接入 → 终端二次鉴权** 的强依赖链。
- **UPF 下沉 + 企业 DN-AAA 部署**是触发本特性"经 UPF 转接"机制的唯一场景根因（SMF 与 DN-AAA 物理不可直达）。
