# 批次5：APN 业务域格式对齐修复（对齐访问限制场景 graph_parser）

> **修复日期**：2026-06-23
> **修复目标**：解决前端 graph_parser 无法解析 APN 业务域 Task→Command 边的问题
> **根因**：APN 的 Task 编号 `T-APN-xxx` 不被 graph_parser 识别（graph_parser 按纯数字 Task 编号解析），且 `command_refs` 在批次4被错误对象化为 `CMD-xxx(命令名)` 格式（应保持纯文本命令名，graph_parser 通过 command_name 匹配 04 的 MMLCommand）
> **参照标杆**：`业务图谱体系/访问限制场景/three-layer-graph/`（前端可解析的成功格式）

---

## §1 根因诊断（对比实证）

| 维度 | 访问限制（前端可解析） | APN（修复前，不解析） | 差异 |
|------|----------------------|---------------------|------|
| Task 编号 | `T-001` 纯数字（generic）/ `T-AC-101` 场景前缀（独有） | `T-APN-001` 全部带 APN 前缀 | ★ graph_parser 不认 `T-APN-xxx`，只认纯数字或场景短前缀（AC/BW/CH） |
| `command_refs` | 纯文本命令名：`ADD FILTER`, `ADD RULE` | 批次4对象化为 `CMD-UDG-014(ADD POOL)` | ★ graph_parser 靠 command_name 匹配，非 CMD-xxx 对象 ID |
| 05 §4.1 invokes | `T-001 任务名 \| invokes \| ADD FILTER (CMD-UDG-005) \| 说明`（命令名主，CMD 辅助） | `T-APN-001 \| CMD-UDG-004, CMD-UDG-005 \| 边数`（CMD 主，无命令名） | ★ 缺命令名，graph_parser 无法匹配 |

**核心结论**：graph_parser 的解析逻辑是「Task 编号纯数字 + command_refs 纯文本命令名 → 匹配 04 的 MMLCommand.command_name 建立 Task→Command 边」。APN 的 `T-APN-xxx` 编号和对象化 `command_refs` 双重偏离，导致边建立失败。

## §5 影响与后续

### 5.1 端到端边恢复

修复后，graph_parser 应能正确建立 APN 业务域的 Task→Command 边：
- Task 编号纯数字 → graph_parser 识别 ConfigTask 节点
- command_refs 纯文本命令名 → 匹配 04 的 MMLCommand.command_name → 建立 invokes 边
- 预计恢复 183 条 Task→Command 边（60 Task 有命令，T-109 空设计）

### 5.2 跨场景合并注意

APN 域 Task 现用纯数字（T-001~T-708），与访问限制（T-001~T-008 generic + T-AC-101~109）、带宽（T-001~T-008 + T-101~T-603）、计费（T-001~T-008 + T-101~T-311）在各自文件独立解析不冲突。若未来四场景合并为统一图谱，需为各场景 Task 加场景前缀（如 T-APN-001 / T-AC-001）避免节点 ID 冲突——但当前各场景图谱文件独立，graph_parser 按文件解析，无需合并前缀。

### 5.3 批次4 对象化决策复盘

批次4 将 command_refs 对象化为 CMD-xxx 的初衷是「精确对象引用、零悬挂」，但偏离了 graph_parser 的实际解析机制（command_name 匹配）。批次5 回归访问限制实证格式，证明「纯文本命令名 + graph_parser command_name 匹配」才是正确路径。CMD-xxx 对象 ID 保留在 04 层和 05 §4.1 的括号辅助位，兼顾精确性与可解析性。
