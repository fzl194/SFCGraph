# GWFD-110101 SA-Basic 知识文档

> 聚焦带宽控制场景的业务感知基础特性

---

## 1. 概述

### 1.1 特性定义（是什么）

SA-Basic（Service Awareness Basic）是业务感知（SA）的基本功能，主要包括三四层解析、协议识别和七层解析。UDG支持对用户数据报文进行业务感知，判断用户业务类型，并根据规划对这些业务进行多样化处理，如计费、执行重定向、头增强，或识别Host、是否开启了Tethering等。

在带宽控制场景中，SA-Basic是整个业务感知带宽控制链路的起点和基础：它负责对用户报文进行逐层解析和识别，将识别结果（协议类型、URL、业务类别等）提供给后续规则匹配引擎，由规则匹配引擎触发BWM（Bandwidth Management）/Shaping/Policing等带宽控制动作。

### 1.2 适用NF（UDG/UNC网元）

| 涉及NF | 网元角色 | 支持版本 | 功能说明 |
|--------|---------|---------|---------|
| SGW-U / PGW-U / UPF | 用户面（UDG） | UDG 20.0.0及后续版本 | 对报文进行识别和解析，根据解析结果匹配规则，对不同业务进行不同的计费和动作处理（含带宽控制）；自动连接内网升级服务器检查SA特征库版本更新及下载 |
| SGW-C / PGW-C / SMF | 控制面（UNC） | UNC 20.0.0及后续版本 | 下发动态规则给UDG；基于本地策略或PCF/PCRF下发的规则生成PDR |

### 1.6 客户价值

| 受益方 | 受益描述 |
|-------|---------|
| 运营商 | 对用户业务进行有效的控制和精细化的管理，实现业务的增值，提高用户满意度。在带宽控制场景中，通过SA识别特定业务（如视频、P2P），对该业务实施差异化的带宽控制策略 |
| 用户 | 享受更优质的服务，如提升用户上网速度，保证VIP用户带宽。带宽控制可保障高优先级业务的带宽体验 |

### 1.7 对系统的影响

- 开启SA-Basic特性后，匹配到规则的用户业务流的所有报文都需要进行业务感知，系统处理负荷将增加，报文转发性能和吞吐量将下降
- 如果增加UDG支持SA解析的最大URL长度（SAURLLEN参数），会增加系统消耗，对系统性能有影响，具体影响程度依赖于具体话务模型
- 详细性能影响需要基于流量模型进行评估

### 1.8 特性规格与标准

- 本特性无特殊规格
- 本特性不涉及标准约束
- 本特性不涉及计费与话单
- 本特性不涉及与其他特性的交互（但SA-Basic是其他高级SA特性的基础）

## 2. 核心概念

### 2.1 关键术语

| 术语 | 全称 | 说明 |
|------|------|------|
| SA | Service Awareness（业务感知） | 高效地解析和识别网络上的各种应用，完成用户数据报文的解析、协议识别和内容识别，获得URL等有价值的信息 |
| 三四层解析 | L3/L4 Parsing | 三层（网络层）解析IP地址，四层（传输层）解析协议端口号，获取报文五元组信息 |
| 协议识别 | Protocol Identification | 通过知名端口和特征字识别报文所属的应用层协议 |
| 七层解析 | L7 Parsing | 对识别出的协议报文进行深度解析，提取URL、Method、响应码等关键字信息 |
| Rule | 规则 | 业务规则，由"条件"和"动作"两个维度定义 |
| FlowFilter | 流过滤器 | 定义三四层匹配条件（IP、端口、协议类型） |
| L7Filter | 七层过滤器 | 定义七层匹配条件（URL、Method等） |
| PCC | Policy and Charging Control | 策略和计费控制框架 |
| PDR | Packet Detection Rule | 报文检测规则，由UNC基于本地策略或PCF/PCRF下发的规则生成 |
| UserProfile | 用户模板 | 将规则绑定到用户的容器 |
| CategoryProp | 分类属性 | 用于带宽控制的分类属性对象 |
| BwmUserGroup | 带宽管理用户组 | 用于组级带宽控制的用户分组 |
| URR/URRGroup | Usage Reporting Rule/Group | 使用率上报规则/规则组 |
| Tethering | 热点共享 | 通过TTL值识别用户是否开启热点共享 |
| 前缀URL | Prefix URL | 代理服务器场景中，报文中代理服务器的域名信息 |

| 对象 | 作用 | 带宽控制场景角色 |

| CategoryProp | 分类属性 | 带宽控制核心对象，定义带宽控制分类和动作 |
| BwmUserGroup | 带宽管理用户组 | 用于组级带宽控制 |

### 2.3 在带宽控制场景的角色

SA-Basic在带宽控制场景中扮演"感知入口"的角色，是整个带宽控制链路的第一环：

```
用户报文到达UDG
    ↓
[SA-Basic] 三四层解析（IP/端口/协议）
    ↓
[SA-Basic] 协议识别（知名端口/特征字）
    ↓
[SA-Basic] 七层解析（URL/Method/响应码）
    ↓
[规则匹配] 匹配Rule条件
    ↓  ← SA-Basic的工作到此结束
[动作执行] 触发带宽控制动作
    ├── BWM（带宽管理）：限速/保障
    ├── Shaping（流量整形）：平滑流量
    └── Policing（策略管制）：丢弃/降级
```

具体角色说明：

1. **业务识别**：SA-Basic负责将"无差别的IP报文流"转化为"有业务标签的报文流"，识别出是HTTP视频、P2P下载、即时通信等具体业务类型
2. **属性提取**：解析出URL、Method等七层信息，供规则匹配使用
3. **规则触发**：SA-Basic的解析结果作为规则匹配的输入，当匹配到特定业务时，触发带宽控制动作（如对视频流量限速、对P2P降级）
4. **基础依赖**：后续的"基于业务感知的带宽控制"（GWFD-110311）、"基于业务的Shaping"（GWFD-020354）等高级特性都依赖SA-Basic提供的业务识别能力

## 3. 原理与流程

### 3.1 实现原理

#### 3.1.1 解析与识别机制

SA-Basic对用户数据报文的处理按照固定顺序进行，依次为三四层解析、协议识别和七层解析：

```
用户数据报文
    │
    ▼
┌─────────────────────────────────────────┐
│ 第一步：三四层解析                        │
│                                         │
│  三层（网络层）：源/目的IP地址            │
│  四层（传输层）：协议类型、源/目的端口号   │
│  输出：五元组信息                         │
└───────────────┬─────────────────────────┘
                │
                ▼
┌─────────────────────────────────────────┐
│ 第二步：协议识别                          │
│                                         │
│  方式1：知名端口识别（优先）              │
│    → 快速、准确率高                       │
│  方式2：特征字识别（兜底）                │
│    → 按预置特征库识别，返回7层协议和子协议 │
│  输出：协议类型（HTTP/HTTPS/DNS/FTP等）   │
└───────────────┬─────────────────────────┘
                │
                ▼
┌─────────────────────────────────────────┐
│ 第三步：七层解析                          │
│                                         │
│  对识别出的协议报文进行深度解析           │
│  提取：URL、Method、响应码等关键字信息    │
│  特殊：前缀URL检测与剥离                  │
│  输出：七层关键信息                       │
└───────────────┬─────────────────────────┘
                │
                ▼
        规则匹配引擎
```

#### 3.1.2 三四层解析详解

- **三层（网络层）**：处理数据包的IP路由，可监控IP报文的源/目的IP地址
- **四层（传输层）**：为两台主机上的应用程序提供端到端服务，不同应用层协议在传输层用不同端口号定义

三四层可解析的报文信息：

| 解析项 | 说明 |
|--------|------|
| 源IP地址 | 报文发送方的IP地址 |
| 目的IP地址 | 报文接收方的IP地址 |
| IP协议类型 | TCP / UDP / ICMP 等 |
| 源端口号范围 | 起始端口号 ~ 结束端口号 |
| 目的端口号范围 | 起始端口号 ~ 结束端口号 |

#### 3.1.3 协议识别详解

**知名端口协议识别**（优先方式）：

对于标准协议类型，使用标准或运营商定义/推荐的端口进行识别。通过知名端口识别的准确率很高，适合特征字识别不明显或识别流程复杂的协议。

常见知名端口映射表：

| 协议 | 知名端口 | 说明 |
|------|---------|------|
| HTTP | 80 / 8080 | Web浏览 |
| WAP1.X | 9200-9201 | WAP协议 |
| IMAP4 | 143 | 邮件接收 |
| DNS | 53 | 域名解析 |
| FTP | 20 / 21 | 文件传输（20为数据端口，21为控制端口） |
| SMTP | 25 | 邮件发送 |
| POP3 | 110 | 邮件接收 |
| RTSP | 554 | 流媒体 |
| TFTP | 69 | 服务器侦听端口，数据面端口为服务器随机确定 |

识别策略：
1. 优先使用知名端口识别，可加快协议识别处理
2. 若知名端口不能识别报文协议，则通过特征字完成协议识别
3. 可通过软参 BIT414 控制选择知名端口识别还是协议识别组件识别的承载层协议
4. 可通过软参 BIT469 控制知名端口识别成功后是否还需通过特征库再识别

**特征字协议识别**（兜底方式）：

- 按照预置的特征库进行识别
- 返回7层协议和子协议结果

#### 3.1.4 七层解析详解

UDG识别出报文协议类型后，对每一个请求和响应解析出相应的关键字信息：

| 解析项 | 说明 |
|--------|------|
| URL | 统一资源定位符，如 `http://www.example.com/video/stream` |
| Method | HTTP方法，如GET/POST/CONNECT等 |
| 响应码 | HTTP响应状态码 |

**URL长度配置**：
- 通过 `SET SRVCOMMONPARA` 命令的 `SAURLLEN` 参数配置系统支持解析的最大URL长度
- 取值：511 或 1023 字节
- 若URL长度超过配置的规格，将进行截断处理，可能导致业务策略匹配结果不准确
- 增大URL长度会增加系统消耗，对系统性能有影响

**前缀URL处理**：

当用户访问的业务需要与代理服务器交互时，业务报文中可能同时包含前缀URL和真实URL：

| 概念 | 说明 | 示例 |
|------|------|------|
| 前缀URL | 代理服务器的域名信息 | `http://www.example.com` |
| 真实URL | 用户实际访问的目标 | `http://www.example2.com` |
| 完整URL | SA解析得到的URL | `http://www.example.com/http://www.example2.com` |

处理流程：
1. UDG对业务报文进行前缀URL的检测和匹配
2. 若匹配成功，从SA解析得到的URL中剥离掉前缀URL，采用其后的真实URL进行策略匹配
3. 若不匹配，按照常规URL进行策略匹配

前缀URL规格：

| 规格项 | 限制值 |
|--------|--------|
| 前缀URL组数量 | 最多100个 |
| 每组前缀URL数量 | 最多10个 |
| 前缀URL总数 | 最多1000个 |

#### 3.1.5 规则匹配机制

UDG从UNC获取用户所属PDR（Packet Detection Rule），PDR由UNC基于本地策略或PCF/PCRF下发的规则生成。

规则匹配过程：
1. 用户访问业务时，UDG基于PDR中携带的规则信息进行匹配
2. 匹配过程按三四层解析、协议识别、七层解析的顺序逐层进行
3. 直到规则匹配成功后，执行相应的策略动作（如带宽控制）
4. 匹配结果可用于触发计费、执行重定向、头增强，或识别Host、Tethering等

### 3.2 业务流程（端到端SA识别到动作执行）

端到端的业务感知控制流程：

```
┌─────────────────────────────────────────────────────────────────┐
│ 1. 策略生成                                                      │
│    PCRF/PCF 根据用户签约信息、业务信息或用户状态信息进行决策       │
│    生成控制策略（含QoS及计费策略）                                │
└──────────────────────────────┬──────────────────────────────────┘
                               │ N7接口 / Gx接口
                               ▼
┌─────────────────────────────────────────────────────────────────┐
│ 2. 策略下发                                                      │
│    PCRF/PCF 通过 N7 接口将 QoS 及计费策略下发给                   │
│    SGW-C / PGW-C / SMF（UNC 控制面）                             │
└──────────────────────────────┬──────────────────────────────────┘
                               │ N4接口 / Sx接口
                               ▼
┌─────────────────────────────────────────────────────────────────┐
│ 3. 策略转发                                                      │
│    SGW-C / PGW-C / SMF 通过 N4 接口将策略下发给                   │
│    SGW-U / PGW-U / UPF（UDG 用户面）                             │
│    下发形式：PDR（Packet Detection Rule）                        │
└──────────────────────────────┬──────────────────────────────────┘
                               │
                               ▼
┌─────────────────────────────────────────────────────────────────┐
│ 4. 报文处理（SA-Basic核心）                                      │
│    用户数据报文到达 UDG，按以下顺序处理：                          │
│                                                                 │
│    4a. 三四层解析                                                │
│        → 解析源/目的IP、协议类型、源/目的端口                     │
│                                                                 │
│    4b. 协议识别                                                  │
│        → 知名端口识别（优先）→ 特征字识别（兜底）                 │
│        → 输出：7层协议和子协议                                   │
│                                                                 │
│    4c. 七层解析                                                  │
│        → 解析URL、Method、响应码                                 │
│        → 前缀URL检测与剥离                                       │
│        → 输出：七层关键信息                                       │
└──────────────────────────────┬──────────────────────────────────┘
                               │
                               ▼
┌─────────────────────────────────────────────────────────────────┐
│ 5. 规则匹配                                                      │
│    基于解析结果（三四层信息 + 协议类型 + 七层信息）               │
│    与 PDR 中携带的规则信息逐层匹配                                │
│    直到规则匹配成功                                              │
└──────────────────────────────┬──────────────────────────────────┘
                               │
                               ▼
┌─────────────────────────────────────────────────────────────────┐
│ 6. 策略执行                                                      │
│    根据匹配的规则执行相应动作：                                   │
│    - 计费控制（URR/URRGroup）                                    │
│    - PCC控制（PCCActionProp/PCCPolicyGrp）                       │
│    - 带宽控制（CategoryProp/BwmUserGroup）★带宽控制场景          │
│    - 重定向、头增强等                                             │
│    - QoS信息传递给无线、核心网共同进行端到端资源管理               │
└─────────────────────────────────────────────────────────────────┘
```

### 3.3 协议交互

SA-Basic涉及的接口交互：

| 接口 | 交互网元 | 消息类型 | 说明 |
|------|---------|---------|------|
| N7（5G）/ Gx（2/3/4G） | PCRF/PCF ↔ SGW-C/PGW-C/SMF | 策略下发 | PCRF/PCF生成控制策略并下发给控制面 |
| N4（5G）/ Sx | SGW-C/PGW-C/SMF ↔ SGW-U/PGW-U/UPF | PDR下发 | 控制面将策略转化为PDR下发给用户面 |
| 内部 | UDG内部处理 | 报文解析 | 三四层解析→协议识别→七层解析→规则匹配 |

| 约束类型 | 约束内容 |
|---------|---------|
| HTTP Method匹配限制（WAP1.x） | 当协议类型为WAP1.x时，UDG仅支持将GET、POST和CONNECT方法作为method的匹配条件 |
| HTTP Method匹配限制（HTTP） | 当协议类型为HTTP时，UDG支持将GET、POST、CONNECT、PUT、DELETE、OPTIONS、HEAD、TRACE方法作为method的匹配条件 |
| PUT/DELETE/OPTIONS/HEAD/TRACE策略限制 | 当将HTTP协议的PUT、DELETE、OPTIONS、HEAD和TRACE方法作为匹配条件时，支持的策略为除URL过滤、ADD EXTENDEDFILTER命令作为过滤条件的动作以外的其他策略，如计费 |

| 性能影响 | 开启SA-Basic后，所有匹配到规则的报文都需要进行业务感知，系统处理负荷增加，报文转发性能和吞吐量下降 |

## 5. 配置案例

### 5.1 典型场景：SA识别业务后触发带宽控制

**场景描述**：用户访问视频网站时，UDG通过SA-Basic识别出视频业务流量，匹配规则后触发带宽控制动作（如限速到10Mbps），实现基于业务感知的带宽管理。

**配置步骤和MML命令序列**：

```
=== 步骤1：加载协议特征库和解析数据库 ===

LOD SIGNATUREDB: FILENAME="signature_db_v2.0.dat"
  -- 加载协议特征库，使特征字协议识别可用

LOD PARSERDB: FILENAME="parser_db_v2.0.dat"
  -- 加载协议解析数据库，使七层解析可用

=== 步骤2：配置知名端口 ===

ADD WELLKNOWNPORT: PROTNAME="HTTP", PORT=80
  -- 配置HTTP知名端口为80，加速HTTP协议识别

=== 步骤3：配置三四层匹配条件 ===

ADD FLOWFILTER: FLTID=1, PROTTYPE=TCP, DSTPORT=80
  -- 创建流过滤器1：匹配TCP协议、目的端口80的HTTP流量

ADD FLTBINDFLOWF: FLTID=1, FLOWID=1, ACTION=DENY
  -- 绑定流过滤器1的过滤器关系

ADD PROTBINDFLOWF: FLTID=1, PROTNAME="HTTP"
  -- 将HTTP协议绑定到流过滤器1

=== 步骤4：配置七层匹配条件 ===

ADD L7FILTER: L7FLTID=1, PROTNAME="HTTP", URL="*video*.example.com*"
  -- 创建七层过滤器1：匹配包含video的URL

=== 步骤5：配置带宽控制动作 ===

ADD CATEGORYPROP: CATID=1, DIR=BOTH, CIR=10240, PIR=10240
  -- 创建分类属性1：双向限速10Mbps（CIR=PIR=10240kbps）
  -- 这是带宽控制的核心配置

=== 步骤6：配置业务规则 ===

ADD RULE: RULEID=1001, L7FLTID=1, CATID=1
  -- 创建规则1001：当匹配七层过滤器1（视频URL）时，
  -- 执行分类属性1（限速10Mbps）

=== 步骤7：配置用户模板并绑定规则 ===

ADD USERPROFILE: UPID=1, UPNAME="video_bwm_profile"
  -- 创建用户模板1

ADD RULEBINDING: UPID=1, RULEID=1001, PRIORITY=100
  -- 将规则1001绑定到用户模板1，优先级100

=== 步骤8：配置业务公共参数 ===

SET SRVCOMMONPARA: SAURLLEN=1023
  -- 配置URL最大解析长度为1023字节（支持更精确的URL匹配）

=== 步骤9：配置CP-UP一致性策略 ===

SET CUINCONSPOLICY: BYPASSSW=DISABLE
  -- 使用默认策略：配置不一致时UPF回复Failed Rule ID
```

**场景变体**：

| 变体 | 场景说明 | 核心差异 |
|------|---------|---------|
| 代理服务器场景 | 用户通过代理访问视频网站 | 需额外配置 `ADD PREFIXURLGRP` 和 `ADD PREURLGBINDUP` 处理前缀URL |
| 多业务分级带宽 | 不同业务分配不同带宽 | 配置多个CategoryProp和Rule，分别设置不同的CIR/PIR |
| 基于用户组带宽 | VIP用户组享受更高带宽 | 配置BwmUserGroup，关联UserProfile到不同用户组 |
| 时段带宽控制 | 高峰期限速、闲时不限速 | 结合时间段策略（需GWFD-110311特性支持） |

### 6.2 常见问题与排查

| 问题现象 | 可能原因 | 排查方法 |
|---------|---------|---------|
| SA特征库自动升级失败 | 内网升级服务器不可达或特征库文件损坏 | 检查 ALM-81011 告警；检查升级服务器连通性 |
| 协议解析数据库加载失败 | PARSERDB文件缺失或版本不匹配 | 检查 ALM-81030 告警；重新执行 LOD PARSERDB |
| 业务识别不准确 | URL长度截断导致策略匹配失败 | 检查 SAURLLEN 参数；适当增大URL长度（注意性能影响） |
| 规则匹配失败 | CP-UP配置不一致 | 检查 ALM-81054 告警；使用 SET CUINCONSPOLICY 调整处理策略 |
| 代理服务器场景URL匹配失败 | 前缀URL未正确配置或剥离 | 检查 PREFIXURLGRP 和 PREURLGBINDUP 配置 |
| 带宽控制未生效 | CategoryProp未正确关联到Rule | 检查 ADD RULE 中的 CATID 参数；检查 RULEBINDING |
| WAP1.x Method匹配失败 | WAP1.x仅支持GET/POST/CONNECT | 确认Method类型；不支持的方法无法匹配 |
| 性能下降明显 | SA-Basic对每条匹配报文进行解析 | 评估流量模型；优化规则减少不必要的SA处理 |
| 知名端口识别不生效 | 知名端口未配置或BIT414/BIT469设置问题 | 检查 WELLKNOWNPORT 配置；检查 BIT414、BIT469 软参 |
| HTTP头域解析异常 | 报文头域不完整 | 检查 BIT1195（增强匹配）、BIT1197（SA失败处理）软参 |

### 7.1 接口与信元

| 接口 | 涉及网元 | 关键信元 | 说明 |
|------|---------|---------|------|

- UNC基于本地策略或PCF/PCRF下发的规则生成PDR
- PDR中携带规则信息（Rule、UserProfile等）
- UDG基于PDR进行规则匹配

| 参数 | 说明 |

| Rule | 业务规则 |
| UserProfile | 用户模板 |
| QosProp | QoS属性 |
| BwmUserGroup | 带宽管理用户组 |
| CbbID（URRID） | 计费桶ID（URR ID） |
