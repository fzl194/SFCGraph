# GWFD-020353 基于累计流量的策略控制 知识文档

## 概述

### 特性定义（FUP基础版，2_3_4G场景）

基于累计流量的策略控制是针对用户在一段时间内的**会话级累计流量**进行策略控制。当会话级累计流量达到预先设置的阈值时，改变计费和控制策略，如：改变计费的费率、阻塞用户业务或降低业务速率。

该特性是 **FUP（Fair Usage Policy，公平使用策略）** 的基础版本实现。其核心思路是：对用户的整个会话流量进行累计监控（不区分业务类型），当用户在指定周期内的累计流量达到限额后，执行降速、阻塞或改费率等策略动作，保障网络资源公平使用。

典型应用场景：
- 限制某类用户在固定周期内总的使用量，例如普通用户在一周内使用总量最多10GB。
- FUP场景：用户包月配额10GB，随着用户使用量增加，逐步降低用户带宽，直至阻塞业务。

> PS网络智能化业务部署或改造需要评估现有网络的业务模型、资费策略、网络资源占用情况。具体部署或改造时，请联系华为公司当地办事处获取专门的系统集成服务支持。

## 核心概念

### 关键术语（FUP、累计流量、监控周期、阈值动作）

| 术语 | 说明 |
|------|------|
| **FUP（Fair Usage Policy）** | 公平使用策略。运营商通过限制用户某周期内的使用量，在超出限额时降低速率或提高费率，达到网络公平使用原则，使访问相同业务的用户得到同样的网络体验 |
| **会话级累计流量** | 用户在一段时间内整个PDU会话/承载的流量累计值。区别于业务级累计流量（GWFD-110312），本特性以**整个会话**为粒度进行统计，不区分业务类型 |
| **流量阈值（volumeThreshold）** | PCRF/PCF下发的会话级流量上限。当PGW-U/UPF检测到累计流量达到该阈值时，触发上报 |
| **监控周期** | 运营商配置的流量累计统计周期，例如一周、一个月。周期内累计流量未超限则正常使用，超限则触发策略变更 |
| **阈值动作** | 累计流量达到阈值后执行的动作。包括：改变计费费率、降低业务速率（通过更新authSessAmbr）、阻塞用户业务等 |
| **UMID（umId）** | 使用监控策略数据标识，在一个PDU会话内唯一标识一个监控策略 |
| **URR（Usage Reporting Rule）** | 使用量上报规则。PGW-U/UPF侧的流量统计和上报规则实体 |
| **Usage Monitoring Control（UMC）** | 使用监控控制。通过suppFeat字段在特性协商阶段确定是否支持该功能 |
| **Volume Quota** | 流量配额。当流量统计达到配额时触发Reporting Trigger |

### 对象模型（监控对象、阈值、动作）

本特性的监控对象、阈值和动作在协议层面通过以下信元层级描述：

```
SmPolicyDecision（PCRF/PCF下发的策略决策）
  ├── sessRules（会话级规则）
  │     └── refUmData → 引用 umDecs 中的 umId
  ├── umDecs（使用监控决策数据）
  │     ├── umId（监控策略标识）
  │     ├── volumeThreshold（总流量阈值）
  │     ├── volumeThresholdUplink（上行流量门限）
  │     └── volumeThresholdDownlink（下行流量门限）
  └── policyCtrlReqTrigger（策略控制请求触发条件，如US_RE）
```

在PGW-U/UPF侧，通过PFCP接口映射为：

```
PFCP Session（N4会话）
  └── Create URR（使用量上报规则）
        ├── URR ID（唯一标识）
        ├── Measurement Method（计费方式：流量/时长/事件）
        ├── Reporting Trigger（上报触发条件）
        └── Volume Threshold（流量阈值）
```

各对象职责：

| 对象/信元 | 所在网元 | 职责 |
|-----------|----------|------|
| **sessRules** | PCRF/PCF | 定义会话级规则，通过refUmData引用umDecs |
| **umDecs** | PCRF/PCF | 定义使用监控数据，包括流量阈值（总/上行/下行） |
| **policyCtrlReqTrigger** | PCRF/PCF | 标识达到流量阈值后上报的触发条件 |
| **URR** | PGW-U/UPF | 使用量上报规则，定义统计方式和上报触发条件 |
| **Volume Threshold** | PGW-U/UPF | 流量阈值，当累计流量达到该值时触发PFCP Session Report |
| **authSessAmbr** | PCRF/PCF | 授权的会话聚合最大比特率，用于更新上下行带宽 |

### 与GWFD-110312（业务级FUP）的差异

| 对比维度 | **GWFD-020353 基于累计流量的策略控制** | GWFD-110312 基于业务累计流量的策略控制 |
|----------|---------------------------------------|---------------------------------------|
| 流量统计粒度 | **会话级**（用户整个会话流量累计，不区分业务） | **业务级**（特定业务类型的流量累计） |
| 业务识别依赖 | **不依赖SA**（Service Awareness） | **依赖SA**，需先识别业务类型 |
| 监控对象 | 整个PDU会话/承载的全部流量 | 仅匹配特定业务流的流量（如P2P、HTTP等） |
| URR绑定方式 | URR**关联到所有Create PDRs**（因为sessRule对整个会话有效） | URR只绑定在**指定业务流对应的PDR**上 |
| 策略定义位置 | **sessRules**（SessionRule）中定义，通过refUmData引用umDecs | **PccRule**中定义，通过refUmData引用umDecs |
| 典型场景 | 用户总流量超限后整体降速 | 某类业务流量超限后对该类业务降速，不影响其他业务 |
| License项 | 82200AFM LKV3G5PCBT01 | 82209776 LKV3G5FPBS01 |
| 首次发布版本 | 20.1.0 | 20.0.0 |
| 文档数量 | 2个（特性概述、实现原理） | 3个（特性概述、实现原理、激活配置） |

> 定位：GWFD-020353是FUP的基础版本（会话级），GWFD-110312是其增强版本（业务级）。两者可独立部署，也可组合使用。

## 原理与流程

### 实现原理

基于累计流量的策略控制基本原理为：

```
PCRF/PCF对用户业务进行授权，下发会话流量上报阈值、计费规则和QoS规则给PGW-C/SMF
    → PGW-C/SMF将策略转发给PGW-U/UPF（生成URR规则，关联到所有PDR）
        → PGW-U/UPF根据规则对用户业务进行控制，基于会话统计流量
            → PGW-U/UPF检测到累计流量达到阈值时，向PGW-C/SMF上报流量
                → PGW-C/SMF将流量转发给PCRF/PCF
                    → PCRF/PCF对流量进行累计，进行策略决策并下发新策略
                        → PGW-C/SMF将新策略下发给PGW-U/UPF执行（降速/阻塞/改费率）
```

核心机制：

- **sessRule（会话级规则）对整个会话有效**：由于sessRule不绑定特定业务流，PGW-C/SMF生成的URR需要**关联到所有Create PDRs**，即统计整个会话的全部流量。
- **URR ID与UMID映射**：PGW-C/SMF建立URR ID和UMID之间的映射关系，用于在N4接口和Npcf接口之间传递流量上报信息。
- **阈值递进机制**：PCRF/PCF可以多轮下发阈值。第一轮阈值耗尽后下发新阈值，用户继续使用；新阈值再次耗尽后，PCRF/PCF不再下发阈值，改为下发QoS规则进行降速控制。

### 业务流程

#### 会话级累计流量策略控制流程（完整）

```
步骤1:  UE发起PDU Session Establishment流程

步骤2:  AMF进行PGW-C/SMF选择
        → 向PGW-C/SMF发送Nsmf_PDUSession_CreateSMContext消息

步骤3:  PGW-C/SMF向PCRF/PCF发送Npcf_SMPolicyControl_Create Request
        → suppFeat字段指示是否支持Usage Monitoring Control功能

步骤4:  PCRF/PCF从UDR获取策略签约数据

步骤5:  PCRF/PCF进行特性协商，生成会话管理策略（含FUP相关策略）
        → 回复Npcf_SMPolicyControl_Create Response
        → 消息携带SmPolicyDecision信元组
        → 包含会话级流量阈值（sessRules.refUmData → umDecs.umId → volumeThreshold）

步骤6:  PGW-C/SMF根据sessRule和umDecs等信元信息生成N4接口URR规则
        → 由于sessRule对整个会话有效，URR关联到所有Create PDRs
        → 建立URR ID和UMID之间的映射

步骤7:  PGW-C/SMF向PGW-U/UPF发送PFCP Session Establishment Request
        → 携带Create URR信元组，指示流量阈值相关信息
        → 含URR ID、Measurement Method、Reporting Trigger、Volume Threshold

步骤8:  PGW-C/SMF收到PFCP Session Establishment Response，PFCP会话建立成功

步骤9:  用户访问数据业务，PGW-U/UPF检测到流量阈值耗尽
        → 发送PFCP Session Report Request通知PGW-C/SMF
        → 携带Usage Report信元组（含URR ID、Usage Report Trigger=VOLTH、Volume Measurement）

步骤10: PGW-C/SMF根据URR ID和UMID映射，向PCRF/PCF通知流量阈值耗尽

步骤11: PGW-C/SMF发送Npcf_SMPolicyControl_Update Request通知PCRF/PCF
        → 携带accuUsageReport信元组（含refUmId、volUsage）

步骤12: PCRF/PCF进行策略决策，决定下发新的阈值
        → PGW-C/SMF响应Npcf_SMPolicyControl_Update Response下发新阈值

步骤13: PGW-C/SMF检查是否有新的流量阈值
        → 如果有，则更新相应的URR规则

步骤14: PGW-C/SMF发送PFCP Session Modification Request
        → 携带Update URR信元组（含URR ID、Measurement Method=VOLUM、Reporting Trigger=VOLTH、Volume Threshold=新阈值）

步骤15: 用户继续访问数据业务，PGW-U/UPF检测到流量阈值再次耗尽
        → 发送PFCP Session Report Request通知PGW-C/SMF

步骤16: PGW-C/SMF再次发送Npcf_SMPolicyControl_Update Request通知PCRF/PCF流量阈值耗尽

步骤17: PCRF/PCF进行策略决策，不再下发流量阈值
        → 更新QoS规则，对用户进行带宽控制
        → 示例：上行带宽降为1 Mbit/s，下行带宽降为2 Mbit/s
        → Npcf_SMPolicyControl_Update Response携带authSessAmbr信元组更新QoS规则

步骤18: PGW-C/SMF检查无新流量阈值下发
        → 通知PGW-U/UPF不再上报流量

步骤19: PGW-C/SMF发送PFCP Session Modification Request
        → 通知PGW-U/UPF更新业务规则
```

#### 会话释放时累计流量上报流程

会话释放时，由于URR规则处于激活状态，PGW-U/UPF会在会话删除响应中上报累计使用流量。

```
步骤1-7:  与会话建立流程相同（PDU Session Establishment）

步骤8:  用户发起PDU会话释放流程

步骤9:  AMF向PGW-C/SMF发送Nsmf_PDUSession_UpdateSMContext消息

步骤10: PGW-C/SMF执行PDU会话释放流程
        → 向PGW-U/UPF发送PFCP Session Delete Request释放N4会话

步骤11: 由于URR规则处于激活状态
        → PGW-U/UPF在PFCP Session Delete Response中携带Usage Report信元组
        → Usage Report Trigger = TERMR（终止PFCP Session上报流量，删除会话策略）
        → Volume Measurement = 实际使用的累计流量（且未达到阈值）

步骤12: PGW-C/SMF向PCRF/PCF转发会话释放和流量上报

步骤13: PCRF/PCF回复Npcf_SMPolicyControl_Delete Response给PGW-C/SMF
```

### 协议交互（Gx）

本特性在5G场景下使用N7接口（Npcf_SMPolicyControl），在2/3/4G场景下使用Gx接口（等价的策略控制消息）。关键协议交互信元如下：

#### SmPolicyDecision 关键信元

| 主信元 | 子信元 | 信元解释 |
|--------|--------|----------|
| **sessRules** | - | SessionRule。会话级规则定义 |
| sessRules | **refUmData** | 对使用监控数据决策的引用。会话级累计流量在SessionRule中定义，通过umID引用 |
| **umDecs** | - | DecisionData。使用监控决策数据 |
| umDecs | **umId** | 标识使用监控策略数据在一个PDU会话 |
| umDecs | **volumeThreshold** | 总流量阈值 |
| umDecs | **volumeThresholdUplink** | 上行流量门限 |
| umDecs | **volumeThresholdDownlink** | 下行流量门限 |
| **policyCtrlReqTrigger** | - | 标识达到流量阈值后上报的触发条件，如US_RE |
| **suppFeat** | - | 标识特性协商结果，如UMC |

#### PFCP Create URR 关键子信元

| 子信元 | 信元解释 |
|--------|----------|
| **URR ID** | 唯一标识一个Create URR |
| **Measurement Method** | 标识计费方式，包括流量计费（VOLUM）、时长计费（DURAT）或事件计费（EVENT） |
| **Reporting Trigger** | 标识计费上报的触发条件，包含Periodic Reporting（定期上报）、Volume Quota（流量配额）、Time Quota（时间配额）、Linked Usage Reporting（关联URR）等 |
| **Volume Threshold** | 总流量阈值 |

#### PFCP Usage Report 关键子信元

| 子信元 | 信元解释 |
|--------|----------|
| **URR ID** | 标识上报的URR |
| **Usage Report Trigger** | 标识URR的Trigger。VOLTH=流量阈值耗尽；TERMR=终止PFCP Session上报 |
| **Volume Measurement** | 标识流量使用情况，包含总流量、上行流量和下行流量 |

#### accuUsageReport 关键子信元

| 子信元 | 信元解释 |
|--------|----------|
| **refUmId** | UMID |
| **volUsage** | 累计流量信息，从N4 Usage Report中获得 |

#### Update URR 关键子信元（阈值更新时）

| 子信元 | 信元解释 |
|--------|----------|
| **URR ID** | 标识上报的URR |
| **Measurement Method** | 标识计费方式，信元值是VOLUM |
| **Reporting Trigger** | 标识计费上报的触发条件，信元值是VOLTH |
| **Volume Threshold** | 新的流量阈值 |

#### authSessAmbr（阈值耗尽后降速时）

| 子信元 | 信元解释 |
|--------|----------|
| **uplink** | 授权上行最大比特率（如1 Mbit/s） |
| **downlink** | 授权下行最大比特率（如2 Mbit/s） |

### 约束条件

1. **PCC基本功能依赖**：必须先完成PCC基本功能（GWFD-020351）的激活配置。UDG需要通过PCC框架从PGW-C/SMF获取URR规则。
2. **系统性能影响**：启用该特性需要UDG与PGW-C/SMF不断进行交互，同时需要动态处理PGW-C/SMF下发的策略，对系统性能影响较大。规划时需考虑启用该特性的用户比例。
3. **性能评估**：详细性能影响需要基于流量模型进行评估，请联系华为技术支持获取服务。
4. **应用限制**：本特性无应用限制（原始文档明确说明）。
5. **计费与话单**：本特性本身不涉及计费与话单（计费规则由PCRF/PCF下发的计费策略决定）。
6. **特性规格**：本特性无特殊规格（原始文档明确说明）。

## 配置案例

### 典型场景：用户月流量达到阈值后降速（含完整MML）

**场景描述**：运营商为用户提供包月套餐，配额10GB。用户在一个计费周期内的累计流量达到10GB后，将用户上下行带宽降低（上行1 Mbit/s、下行2 Mbit/s），保障网络公平使用。

**流程说明**：

本场景的策略控制流程由PCRF/PCF主导决策，UDG侧执行流量统计和策略执行。

```
阶段1: PDU会话建立时
  → PCRF/PCF下发SmPolicyDecision，包含：
     - sessRules.refUmData → umDecs.umId
     - umDecs.volumeThreshold = 第一段阈值（如5GB）
  → PGW-C/SMF生成URR规则，关联到所有PDR
  → PGW-U/UPF按URR规则统计全部会话流量

阶段2: 第一段阈值（5GB）耗尽
  → PGW-U/UPF发送PFCP Session Report（Usage Report Trigger=VOLTH）
  → PCRF/PCF下发新阈值（如10GB）
  → 用户继续正常使用

阶段3: 第二段阈值（10GB）耗尽
  → PGW-U/UPF再次发送PFCP Session Report
  → PCRF/PCF不再下发新阈值
  → PCRF/PCF下发authSessAmbr：上行1 Mbit/s、下行2 Mbit/s
  → PGW-U/UPF执行降速

阶段4: 会话释放时
  → PGW-U/UPF在PFCP Session Delete Response中携带最终Usage Report
  → Usage Report Trigger=TERMR
  → 上报累计流量（未达到当前阈值的部分）
```

**UDG侧MML配置**：

UDG侧需要完成License激活和PCC基本功能配置。以下为最小配置：

```mml
// ====== 步骤1：打开本特性的License开关 ======
SET LICENSESWITCH:LICITEM="LKV3G5PCBT01",SWITCH=ENABLE;

// ====== 步骤2：确认PCC基本功能已激活 ======
// 参见 GWFD-020351 PCC基本功能 的完整激活配置
// 包括：PCC全局配置、与PGW-C/SMF的PFCP对接配置等
```

> 本特性不需要在UDG侧单独配置累计流量阈值。URR规则中的Volume Threshold由PGW-C/SMF在PFCP Session Establishment Request中动态下发。UDG侧只需确保PCC基本功能正常工作、License已激活。

**策略效果**：

| 阶段 | 累计流量 | 用户带宽 | 触发动作 |
|------|----------|----------|----------|
| 阶段1 | 0 ~ 5GB | 正常（如上行100M、下行500M） | PCRF/PCF下发第一段阈值5GB |
| 阶段2 | 5GB ~ 10GB | 正常 | PCRF/PCF下发第二段阈值10GB |
| 阶段3 | >10GB | **降速（上行1Mbit/s、下行2Mbit/s）** | PCRF/PCF不再下发阈值，下发authSessAmbr |
| 会话释放 | 结算 | - | 上报最终累计流量 |

## 验证与调测

本特性的验证主要关注以下几个方面：

1. **License激活验证**：确认License开关已打开，特性可用。通过查询License状态确认 `LKV3G5PCBT01` 已激活。

2. **PCC基本功能验证**：确认PCC基本功能正常工作，PGW-U/UPF与PGW-C/SMF之间的PFCP会话建立成功。

3. **URR规则下发验证**：确认在PDU会话建立时，PGW-C/SMF正确下发了Create URR信元组（含Volume Threshold）到PGW-U/UPF。

4. **流量统计验证**：确认PGW-U/UPF正确统计整个会话流量，包括总流量、上行流量和下行流量。

5. **阈值触发上报验证**：确认当累计流量达到Volume Threshold时，PGW-U/UPF正确触发PFCP Session Report（Usage Report Trigger=VOLTH）。

6. **降速策略执行验证**：确认PCRF/PCF下发的authSessAmbr在PGW-U/UPF侧正确执行，用户带宽按预期降低。

> 原始文档中未提供独立的调测章节。实际调测需结合PCC基本功能（GWFD-020351）的调测方法，以及PCRF/PCF侧的策略下发验证。重点关注URR ID与UMID映射是否正确建立、Volume Threshold是否正确下发。
