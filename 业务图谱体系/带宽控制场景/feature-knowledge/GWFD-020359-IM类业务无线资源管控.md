# GWFD-020359 IM类业务无线资源管控 - 特性知识文档

> **场景归属**: 业务感知 → 带宽控制子场景（辅助特性：IM类业务无线资源管控）
> **产品**: UDG (UPF网元)
> **场景定位**: 识别IM类业务 → 上报无线侧/PCRF → 触发无线资源管控/带宽调整
> **辅助特性说明**: 针对IM业务"小包频繁、保活心跳"特征的专门管控

## 1. 特性概述

### 1.1 基本信息

| 属性 | 值 |
|------|-----|
| 特性ID | GWFD-020359 |
| 特性名称 | IM类业务无线资源管控 |
| 产品 | UDG |
| 适用NF（网元角色） | PGW-U |
| 首次发布版本 | 20.3.0 |
| 当前文档版本 | 20.15.2 |

| 遵循标准 | 3GPP TS 29.060（Gn/Gp接口GTP协议） |

### 1.2 核心能力一句话描述

在无线资源有限的情况下，UDG通过SA（Service Awareness）功能识别出IM类业务，并将业务类型映射为IP报文的DSCP值，传递给BSC/RNC无线侧，由无线侧基于DSCP值对IM类业务进行无线资源调度优化。

### 1.3 在带宽控制场景中的定位

本特性属于带宽控制场景的**辅助特性**，定位为"IM类业务专用管控"：

- **核心关注点**：不是核心网侧的限速或带宽整形（那是PCC/BWM的职责），而是通过DSCP标记将IM业务身份传递到**无线侧**（BSC/RNC），由无线侧进行调度优先级调整。
- **与通用带宽控制的差异**：通用带宽控制在UPF/GW上执行限速/调度，本特性是在UPF上做识别和标记，实际的资源管控动作发生在BSC/RNC无线侧。
- **带宽管控联动价值**：识别IM业务后，同一套SA识别能力可被PCC规则复用，触发核心网侧的带宽管控（如限速、专载创建），形成"无线侧 + 核心网侧"双重管控。

### 1.4 IM类业务特点

IM（Instant Messaging，即时通讯）类业务的共性特征，是本特性存在的根本原因：

| 业务特征 | 说明 | 对无线资源的影响 |
|----------|------|------------------|
| 小包频繁 | 单次交互数据量小（文字消息、状态同步），但发送频率高 | 频繁占用PDCH信道，信令开销大 |
| 保活心跳 | 为维持长连接，周期性发送心跳包（Keep-Alive） | 无数据传输时也不能释放空口资源 |
| 长时间在线 | 用户全天挂机，即使不主动通信也保持连接 | 持续占用无线信道，挤占语音等高优先级业务资源 |

典型IM业务示例：QQ、MSN、Fetion（飞信）、微信、WhatsApp等。

### 1.5 与SA、ADC、BWM的协作关系

| 协作特性 | 关系类型 | 说明 |
|----------|----------|------|
| SA相关特性（SA-Basic 82209749、SA-IM 82209763） | **依赖** | UDG基于SA执行业务检测，开通本特性必须同时开启SA功能。SA-Basic是所有SA协议解析的基础，必须开启。 |

## 2. 核心概念

### 2.1 IM类业务（即时通讯）

IM类业务是指QQ、MSN、Fetion、微信、WhatsApp等即时通讯应用。本特性针对的是这些业务的**小包、保活心跳、长时间在线**特征，而非业务的具体功能内容。

在本特性的DSCP映射表中，已定义的IM/相关业务类型包括：

| DSCP值 | 业务类型 |
|--------|----------|
| 12 | QQ |
| 14 | MSN |
| 18 | Fetion（飞信） |
| 20 | WAP2.0 |
| 22 | Default（默认） |
| 26 | HTTP |
| 28 | WAP1.x |
| 30 | FTP |
| 34 | MMS |
| 36 | RTSP |
| 38 | PING |
| 46 | TCP建链报文 |

其中DSCP=12（QQ）、DSCP=14（MSN）、DSCP=18（Fetion）是典型的IM类业务映射值。

### 2.2 业务特征：小包、保活心跳、长时间在线

这三者是IM业务占用无线资源的根因：

- **小包**：每个数据包很小，但需要完整的无线信道分配流程，信令开销/有效载荷比极高。
- **保活心跳**：应用层为维持TCP长连接，每隔几十秒到几分钟发送一次心跳包，导致无线侧即使无用户数据也要保持RRC连接和PDCH分配。
- **长时间在线**：用户不主动退出IM应用，连接全天存在，无线侧资源被持续占用。

**运营商痛点**：IM业务报文心跳频繁、长期在线，导致无数据传输时也不能释放空口资源，进而影响语音业务体验。语音业务在高并发IM场景下体验下降。

### 2.3 无线资源管控（RRC、调度优先级、DRX）

本特性最终落地的无线侧管控动作（发生在BSC/RNC上）：

- **减少下行分配的无线PDCH数量**：BSC/RNC基于DSCP值识别IM类业务后，减少该业务下行方向的PDCH信道分配，节省无线资源。
- **调度优先级调整**：将IM类业务的综合调度优先级降低，让语音等高优先级业务优先获得无线资源。
- **DRX（Discontinuous Reception）优化**：针对保活心跳特征，优化非连续接收参数，减少空闲监听开销。

### 2.4 UPF上报机制（上报给基站/PCRF）

本特性采用**DSCP方式**进行业务信息传递，这是华为私有实现方式：

- **上报路径**：UDG（PGW-U）→ SGSN（透传） → BSC/RNC（解析DSCP并执行管控）
- **上报载体**：IP报文的ToS（Type of Service）域，DSCP是ToS域的低6位（0~5bit）
- **上报内容**：业务类型标识，通过DSCP值编码

注意：本特性的"上报"不是通过N2/N4信令通知基站，而是通过**修改用户数据报文的DSCP值**，让报文本身携带业务标识，经SGSN透传到达BSC/RNC。这与5G SA架构下的N2/N4信令通知机制不同——本特性当前仅支持2/3G用户接入。

PCRF的角色：PCRF根据用户信息下发IM类业务优化预定义规则（通过PCC架构），UDG基于该规则执行Remark动作修改DSCP值。PCRF是规则的决策源，但最终的DSCP标记和无线侧管控不经过PCRF中转。

### 2.5 与SA识别的关系

SA（Service Awareness，业务感知）是本特性的识别能力底座：

- SA-Basic（82209749）是所有SA协议解析的基础，提供协议特征库加载和报文解析能力。
- SA-IM（82209763）是专门针对IM类业务的SA识别能力。
- 本特性通过SA的七层协议识别能力，识别出报文属于哪种IM协议（如qq_im），然后匹配PCRF下发的预定义规则，执行Remark动作修改DSCP值。

**关键约束**：由于IM协议类型不断更新和增加，为了保证IM类业务网络优化效果，需要定期升级UDG的协议特征库（通过LOD SIGNATUREDB命令）。

## 3. 实现原理与流程

### 3.1 DSCP值的定义

DSCP（Differentiated Services Code Point）是IPv4报文头中ToS（Type of Service）域的低6位（0~5bit）。UDG将IM业务类型映射为DSCP值，带至BSC/RNC侧，由无线侧做相关资源优化处理。

完整DSCP映射表见第2.1节。

### 3.2 DSCP方式的实现过程（端到端流程）

```
┌─────────┐   ①预定义规则    ┌─────────┐   ②传递规则     ┌─────────┐
│  PCRF   │ ──────────────→ │ PGW-C   │ ─────────────→ │ UDG/    │
│ (决策源) │                  │(UNC)    │                 │ PGW-U   │
└─────────┘                  └─────────┘                 └────┬────┘
                                                              │
                                            ③SA识别IM业务     │
                                              匹配预定义规则   │
                                              Remark修改DSCP   │
                                                              │
                                                              ▼
┌──────────┐    ⑤DSCP识别      ┌──────────┐   ④携带DSCP报文  ┌──────────┐
│ BSC/RNC  │ ◀──────────────  │  SGSN    │ ◀────────────── │  UDG/    │
│ (无线管控) │  调整无线资源     │ (透传)    │                  │  PGW-U   │
└──────────┘                  └──────────┘                  └──────────┘
```

**流程详解**：

1. **PCRF下发规则**：PCRF根据用户信息下发IM类业务优化预定义规则。该规则定义了识别到IM业务后应执行的Remark动作（修改DSCP值）。
2. **PGW-C传递规则**：PGW-C（UNC网元，20.3.0及后续版本）将PCRF下发的IM类业务优化预定义规则传递给UDG（PGW-U）。
3. **UDG识别+标记**：PGW-U（UDG）通过SA功能识别出IM业务类型，基于识别结果匹配PCRF下发的预定义规则，并基于预定义规则定义的Remark动作，修改IM业务IP报文的DSCP值。这个DSCP值标识此用户报文是IM类业务报文。
4. **报文传递**：PGW-U（UDG）将修改了DSCP值的数据报文传递给SGSN，SGSN将此报文透传给BSC/RNC。
5. **无线侧管控**：BSC/RNC收到用户报文后，基于报文的DSCP值识别出IM类业务报文，并按照对应的调度处理方式调整无线资源分配（如减少PDCH分配、降低调度优先级）。

### 3.3 上报无线侧的机制

本特性的"上报"本质是**通过数据报文本身的DSCP字段携带业务标识**：

- **不是信令上报**：不通过N2/N4/S1等信令接口通知基站，而是修改每个用户下行数据报文的DSCP值。
- **逐包标记**：每个匹配到IM业务规则的下行报文都会被修改DSCP值，BSC/RNC逐包解析。
- **仅下行优化**：由于是先通过UDG识别出IM协议后再传递给BSC/RNC进行控制，所以只能针对用户**下行**数据报文的无线信道资源进行优化，对上行无线信道资源不作改变。
- **透传路径**：UDG → SGSN（透传，无特殊要求） → BSC/RNC（解析DSCP并执行）。

### 3.4 上报PCRF的机制

PCRF在本特性中的角色是**规则下发方**，不是被上报方：

- PCRF根据用户信息（如签约信息、套餐）决策是否对该用户启用IM类业务优化。
- PCRF通过PCC架构下发"IM类业务优化预定义规则"给PGW-C，PGW-C再传递给UDG。
- UDG执行该规则（Remark动作），修改DSCP值。
- 流程上不存在"UDG上报识别结果给PCRF"的步骤；PCRF是决策前置完成的。

### 3.5 触发无线资源管控的流程

无线侧管控由BSC/RNC执行，触发条件是收到DSCP值非默认值的下行报文：

1. BSC/RNC解析报文的DSCP值（ToS域低6位）。
2. 根据DSCP映射表判断业务类型（如DSCP=12为QQ）。
3. 按照BSC/RNC上预先配置的调度策略，对该业务执行无线资源调度调整。
4. 典型动作：减少下行PDCH分配数量、降低综合调度优先级。

### 3.6 触发带宽管控的流程

本特性本身不直接触发核心网侧的带宽限速/整形。若需要核心网侧带宽管控联动，需要额外配置PCC规则：

- UDG的SA识别能力可被多个Rule复用：同一个识别到qq_im的流过滤器，既可绑定REMARK_FPI规则（本特性），也可绑定其他策略类型的规则（如限速规则）。
- PCRF可在同一PCC规则中同时下发Remark动作和Rating Group/Charging规则，实现"标记 + 计费"联动。

### 3.7 完整端到端流程总结

| 步骤 | 执行网元 | 动作 |
|------|----------|------|
| 1 | PCRF | 根据用户信息下发IM类业务优化预定义规则 |
| 2 | PGW-C (UNC) | 将预定义规则传递给UDG |
| 3 | UDG (PGW-U) | SA识别IM业务类型 → 匹配预定义规则 → Remark修改DSCP值 |
| 4 | UDG (PGW-U) | 将含DSCP标记的报文发给SGSN |
| 5 | SGSN | 透传报文给BSC/RNC |
| 6 | BSC/RNC | 解析DSCP值 → 识别IM业务 → 调整无线资源调度 |

### 4.5 禁用Alias Marking（DSCP方式必需）

**MOD USERPROFILE** — 禁用Alias Marking功能：

DSCP方式下，必须禁用Alias Marking功能。因为该功能使能时，报文ToS域的值不会被改写，会导致DSCP标记无法生效。

| 参数 | 取值样例 | 说明 |
|------|----------|------|
| USERPROFILENAME | up_test | 用户模板名称 |
| ALIASMARKING | DISABLE（不使能） | Alias Marking使能标记，DSCP方式必须设为不使能 |

### 4.6 与PCC规则的联动

本特性的规则（ADD RULE，POLICYTYPE=REMARK_FPI）是UDG本地配置的业务匹配规则。PCRF下发的预定义规则通过PCC架构到达UDG后，UDG将SA识别结果与预定义规则匹配，执行Remark动作。

联动要点：
- PCRF下发的PCC规则定义了"识别到IM业务后执行Remark"的策略。
- UDG本地的ADD RULE配置实现了具体的Remark动作参数（如REMARKCLASS=AF12）。
- PCC基本功能（GWFD-020351）必须先完成激活，本特性才能正常工作。

## 5. 配置案例

### 5.1 案例1：识别QQ业务后通过DSCP标记触发无线侧资源管控

**场景描述**：在GPRS/UMTS网络中，运营商希望识别QQ业务，将QQ业务类型通过DSCP值传递给BSC/RNC，由无线侧减少QQ业务的下行PDCH分配，提升语音业务体验。

**前提条件**：
- 已完成PCC基本功能（GWFD-020351）激活。
- 已加载License（LKV3G5ITSR01）。
- UDG和BSC/RNC均为华为设备（DSCP方式是华为私有实现）。
- 仅针对2/3G用户接入。

**完整命令序列**：

```mml
// 步骤1：打开本特性的License配置开关
SET LICENSESWITCH: LICITEM="LKV3G5ITSR01", SWITCH=ENABLE;

// 步骤2：配置三四层过滤条件（IM识别使用协议识别，三四层配置为ANY）
ADD FILTER: FILTERNAME="filter_IM_any", L34PROTTYPE=STRING, L34PROTOCOL=ANY;
ADD FLOWFILTER: FLOWFILTERNAME="flowfilter_IM_01";
ADD FLTBINDFLOWF: FLOWFILTERNAME="flowfilter_IM_01", FILTERNAME="filter_IM_any";
SET REFRESHSRV: REFRESHTYPE=ALL;

// 步骤3：加载协议特征库和解析库（如已加载可跳过）
LOD SIGNATUREDB: LOADMODE=LATEST;
LOD PARSERDB: LOADMODE=LATEST;

// 步骤4：配置七层过滤条件（绑定qq_im协议识别QQ业务）
ADD PROTBINDFLOWF: FLOWFILTERNAME="flowfilter_IM_01", PROTOCOLNAME="qq_im";
// 系统自动在60s后将L7Filter置为生效，如需立即生效执行：
// SET REFRESHSRV: REFRESHTYPE=ALL;

// 步骤5：配置IM类业务规则（将QQ业务映射为DSCP值AF12）
ADD RULE: RULENAME="rule_IM_01", POLICYTYPE=REMARK_FPI,
    FILTERINGMODE=FLOWFILTER,
    FLOWFILTERNAME="flowfilter_IM_01",
    REMARKFPISEL=REMARK, REMARKCFGTYPE=CLASS, REMARKCLASS=AF12, PRIORITY=20;

// 步骤6：配置规则与用户模板的绑定关系
ADD USERPROFILE: USERPROFILENAME="up_test";
ADD RULEBINDING: USERPROFILENAME="up_test", RULENAME="rule_IM_01";

// 步骤7（DSCP方式必需）：禁用Alias Marking功能
MOD USERPROFILE: USERPROFILENAME="up_test", ALIASMARKING=DISABLE;
```

**效果说明**：
配置完成后，当QQ用户的下行数据报文经过UDG时：
1. SA识别出报文为qq_im协议。
2. 匹配rule_IM_01规则。
3. 执行Remark动作，将报文DSCP值修改为AF12（DSCP=12）。
4. 报文经SGSN透传到达BSC/RNC。
5. BSC/RNC解析DSCP=12，识别为QQ业务，减少下行PDCH分配，降低调度优先级。

### 5.2 案例2：同时识别多种IM业务（QQ + MSN）分别标记不同DSCP

**场景描述**：运营商希望同时管控QQ和MSN两种IM业务，分别标记不同的DSCP值，使BSC/RNC能够区分不同IM业务类型进行差异化调度。

**完整命令序列**：

```mml
// 步骤1：License开关（同案例1，如已开启可跳过）
SET LICENSESWITCH: LICITEM="LKV3G5ITSR01", SWITCH=ENABLE;

// 步骤2：配置三四层过滤条件（QQ和MSN各一个流过滤器）
// QQ的过滤条件
ADD FILTER: FILTERNAME="filter_QQ_any", L34PROTTYPE=STRING, L34PROTOCOL=ANY;
ADD FLOWFILTER: FLOWFILTERNAME="flowfilter_QQ_01";
ADD FLTBINDFLOWF: FLOWFILTERNAME="flowfilter_QQ_01", FILTERNAME="filter_QQ_any";

// MSN的过滤条件
ADD FILTER: FILTERNAME="filter_MSN_any", L34PROTTYPE=STRING, L34PROTOCOL=ANY;
ADD FLOWFILTER: FLOWFILTERNAME="flowfilter_MSN_01";
ADD FLTBINDFLOWF: FLOWFILTERNAME="flowfilter_MSN_01", FILTERNAME="filter_MSN_any";

SET REFRESHSRV: REFRESHTYPE=ALL;

// 步骤3：加载特征库（如已加载可跳过）
LOD SIGNATUREDB: LOADMODE=LATEST;
LOD PARSERDB: LOADMODE=LATEST;

// 步骤4：配置七层过滤条件（分别绑定qq_im和msn协议）
ADD PROTBINDFLOWF: FLOWFILTERNAME="flowfilter_QQ_01", PROTOCOLNAME="qq_im";
ADD PROTBINDFLOWF: FLOWFILTERNAME="flowfilter_MSN_01", PROTOCOLNAME="msn";

// 步骤5：配置业务规则（QQ→AF12/DSCP=12，MSN→AF13/DSCP=14）
ADD RULE: RULENAME="rule_QQ_01", POLICYTYPE=REMARK_FPI,
    FILTERINGMODE=FLOWFILTER,
    FLOWFILTERNAME="flowfilter_QQ_01",
    REMARKFPISEL=REMARK, REMARKCFGTYPE=CLASS, REMARKCLASS=AF12, PRIORITY=20;

ADD RULE: RULENAME="rule_MSN_01", POLICYTYPE=REMARK_FPI,
    FILTERINGMODE=FLOWFILTER,
    FLOWFILTERNAME="flowfilter_MSN_01",
    REMARKFPISEL=REMARK, REMARKCFGTYPE=CLASS, REMARKCLASS=AF13, PRIORITY=21;

// 步骤6：配置用户模板和规则绑定（同一UserProfile可绑定多个Rule）
ADD USERPROFILE: USERPROFILENAME="up_test";
ADD RULEBINDING: USERPROFILENAME="up_test", RULENAME="rule_QQ_01";
ADD RULEBINDING: USERPROFILENAME="up_test", RULENAME="rule_MSN_01";

// 步骤7：禁用Alias Marking
MOD USERPROFILE: USERPROFILENAME="up_test", ALIASMARKING=DISABLE;
```

**效果说明**：
- QQ报文DSCP标记为AF12（值12），BSC/RNC识别为QQ业务。
- MSN报文DSCP标记为AF13（值14），BSC/RNC识别为MSN业务。
- BSC/RNC可根据不同DSCP值对不同IM业务执行差异化调度策略。

> 注意：案例2中MSN的PROTOCOLNAME（如"msn"）和REMARKCLASS（AF13）为根据DSCP映射表推断的参考配置，实际部署时需根据特征库中实际的协议名称和ADD RULE命令支持的REMARKCLASS取值进行确认。

### 5.3 使用流过滤器组的替代配置方式

当运营商希望基于流过滤器组（而非单个流过滤器）设置过滤条件时，可使用ADD FLOWFILTERGRP命令配置流过滤器组：

- ADD RULE的FILTERINGMODE参数设置为FLOWFILTERGRP。
- 使用ADD FLOWFILTERGRP命令创建流过滤器组，将多个流过滤器加入同一组。
- FLOWFILTERNAME参数替换为流过滤器组名称。

这种方式适用于需要对多种IM业务应用相同Remark策略的场景。

## 6. 验证与调测

### 6.1 调测前提条件

- 已完成激活IM类业务无线资源管控（第5节配置案例）。
- Gn/Gp镜像接口上已安装第三方抓包工具。
- 准备测试终端和测试用IMSI号、APN。

### 6.2 调测数据准备

| 参数 | 取值样例 | 来源 |
|------|----------|------|
| 用户IMSI号（IMSI） | 460000123456789 | 测试终端自带 |
| APN名称（APN） | apn-test | 与对端协商，取自激活配置 |

### 6.3 调测步骤（完整9步流程）

#### 步骤1：验证License开关

执行 LST LICENSESWITCH 查询License是否允许使用本特性：

```mml
LST LICENSESWITCH:LICITEM="LKV3G5ITSR01";
```

预期输出：
```
         License项  =  LKV3G5ITSR01
       License名称  =  IM类业务无线资源管控
              开关  =  打开
```

- 开关为"打开" → 继续步骤2。
- 开关为"关闭" → 执行 SET LICENSESWITCH 打开开关。

#### 步骤2：启动抓包

打开UDG PDN侧/接入侧镜像接口上的抓包工具，准备抓取测试终端的出入报文。

#### 步骤3：测试终端接入网络

测试终端使用配置的APN（如apn-test）接入网络。

- 成功接入 → 继续步骤4。
- 无法接入 → 调测UDG的接入功能。

#### 步骤4：验证IM业务识别和DSCP标记

测试终端访问QQ业务，检查跟踪报文中的ToS值是否为AF12（DSCP=12）。

- ToS值为AF12 → 说明UDG的SA识别及Remark生效，**功能正常，调测完成**。
- ToS值不是AF12 → 执行步骤5继续排查。

> 如果存在多个IM类业务，请逐个业务进行调测。

#### 步骤5：检查Alias Marking是否关闭

执行 LST USERPROFILE 查看Alias Marking功能是否关闭：

```mml
LST USERPROFILE:USERPROFILENAME="up_test";
```

预期输出：
```
                         用户模板名称  =  up_test
                Alias Marking使能标记  =  不使能
```

- 不使能 → 继续步骤6。
- 使能 → 执行 MOD USERPROFILE 禁用Alias Marking，然后返回步骤3重新调测。

> **关键排查点**：DSCP方式下，Alias Marking使能时，报文ToS域的值不会被改写，是DSCP标记不生效的常见原因。

#### 步骤6：验证特征库加载状态

执行 DSP SIGNATUREDB 查询协议特征库加载状态，确认UDG可以识别报文的七层协议类型：

```mml
DSP SIGNATUREDB:;
```

预期输出：
```

   特征库版本号  =  01.0002.1063.01
```

- 加载完成 → 继续步骤7。
- 未正常加载 → 重新执行 LOD SIGNATUREDB 加载特征库，然后返回步骤3。

#### 步骤7：验证用户会话信息
