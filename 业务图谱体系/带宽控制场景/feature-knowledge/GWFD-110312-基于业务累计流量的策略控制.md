# GWFD-110312 基于业务累计流量的策略控制 知识文档

## 概述

### 特性定义

基于业务累计流量的策略控制是运营商针对用户访问的业务在一段时间内的**业务累计流量**进行策略控制。当某类业务的累计流量达到预先设置的阈值时，改变计费和控制策略，例如改变计费费率、阻塞用户业务或降低业务速率。

该特性是 **FUP（Fair Usage Policy，公平使用策略）** 在业务级别的实现。其核心思路是：对特定业务类型（如P2P）的流量进行累计监控，当用户的该类业务流量在指定周期内达到限额后，执行降速、重定向、改变费率等策略动作，保障网络资源公平使用。

典型应用场景：
- 限制某类业务在固定周期内的总使用量，例如P2P业务在一周内使用总量最多50MB。
- 包月用户随着特定业务使用量增加，逐步降低用户业务带宽，直至阻塞业务。

### 适用NF

| NF | 产品角色 | 职责 |
|----|----------|------|
| **PGW-U / UPF** | 用户面（UDG） | 根据PGW-C/SMF传递的PCC策略，负责业务数据流检测、策略执行（带宽控制、计费执行等） |
| PGW-C / SMF | 控制面（UNC） | 向PCRF/PCF上报用户业务的流量；将PCRF/PCF下发的策略转发给PGW-U/UPF |
| PCRF / PCF | 策略控制 | 向PGW-C/SMF下发流量阈值；累计PGW-C/SMF上报的业务流量；下发计费规则和QoS规则 |

> 本知识文档聚焦 UDG（PGW-U/UPF）侧的配置和执行。

## 核心概念

### 关键术语

| 术语 | 说明 |
|------|------|
| **FUP（Fair Usage Policy）** | 公平使用策略。运营商通过限制用户某类业务在周期内的使用量，在超出限额时降低速率或提高费率，达到网络公平使用原则 |
| **业务累计流量** | 用户访问特定业务在一段时间内的流量累计值。区别于用户总流量累计，这里以**业务类型**为粒度进行统计 |
| **流量阈值（volumeThreshold）** | PCRF/PCF下发的业务级流量上限。当PGW-U/UPF检测到累计流量达到该阈值时，触发上报 |
| **UMID（umId）** | 使用监控策略数据标识，在一个PDU会话内唯一标识一个监控策略 |
| **URR（Usage Reporting Rule）** | 使用量上报规则。PGW-U/UPF侧的流量统计和上报规则实体 |
| **Usage Monitoring Control（UMC）** | 使用监控控制。通过suppFeat字段在特性协商阶段确定是否支持该功能 |
| **预定义规则（Predefined Rule）** | 预先在PGW-C/SMF和PGW-U/UPF配置好的PCC规则和URR，PCRF/PCF通过引用规则名来激活 |
| **降速** | 当业务累计流量达到阈值后，通过更新QoS规则（如authSessAmbr）降低上下行带宽 |
| **重定向/阻塞** | 策略动作的一种，将用户业务流重定向到特定页面或直接阻塞 |

### 对象模型

本特性在UDG侧涉及以下配置对象层级：

```
UserProfile（用户模板）
  └── RuleBinding（规则绑定）
        └── Rule（业务策略规则）
              ├── FlowFilter（流过滤器）── Filter（过滤器，L3/L4过滤条件）
              └── PCCPOLICYGRP（PCC策略组）
                    └── URRGROUP（URR组）
                          ├── UPURRNAME1/DOWNURRNAME1 → URR（在线计费上报规则）
                          └── UPURRNAME2/DOWNURRNAME2 → URR（累计流量监控上报规则）
```

各对象职责：

| 对象 | 职责 | 说明 |
|------|------|------|
| **Filter** | L3/L4报文过滤条件 | 定义协议类型、端口号等匹配条件 |
| **FlowFilter** | 流过滤器 | 组织多个Filter，作为规则匹配的入口 |
| **URR** | 使用量上报规则 | 定义上报模式（ONLINE/MONITORINGKEY）、流量配额、统计类型 |
| **URRGROUP** | URR组 | 绑定上行和下行方向的URR，上下行各可绑定多组URR |
| **PCCPOLICYGRP** | PCC策略组 | 绑定URRGROUP，是PCC策略的容器 |
| **Rule** | 业务策略规则 | 将FlowFilter（匹配条件）和PCCPOLICYGRP（策略动作）关联 |
| **UserProfile** | 用户模板 | 规则绑定的用户级容器，实际匹配用户流量 |

### 与GWFD-020353基础版FUP的差异

| 对比维度 | GWFD-020353 基于累计流量的策略控制 | **GWFD-110312 基于业务累计流量的策略控制** |
|----------|-------------------------------------|---------------------------------------------|
| 流量统计粒度 | **用户级**（用户全部流量累计） | **业务级**（特定业务类型的流量累计） |
| 业务识别依赖 | 不依赖SA | **依赖SA（Service Awareness）特性**先识别业务类型 |
| 监控对象 | 整个PDU会话/承载的流量 | 仅匹配特定业务流的流量（如P2P、HTTP等） |
| URR绑定方式 | URR绑定在所有PDR上 | URR**只绑定在指定业务流对应的PDR上** |
| 典型场景 | 用户总流量超限后降速 | 某类业务流量超限后对该类业务降速，不影响其他业务 |
| 协议层级 | L3/L4 | L3/L4和**L7（通过预定义规则）** |

## 原理与流程

### 实现原理

基于业务累计流量的策略控制基本原理为：

```
PCRF/PCF下发策略（含业务流量阈值）
    → PGW-C/SMF转发策略给PGW-U/UPF（生成URR规则，绑定到指定业务PDR）
        → PGW-U/UPF检测用户数据流，累计特定业务流量
            → 达到阈值后，PGW-U/UPF上报流量给PGW-C/SMF
                → PGW-C/SMF上报给PCRF/PCF
                    → PCRF/PCF策略决策，下发新的QoS/计费规则
                        → PGW-U/UPF执行新策略（降速/阻塞/改费率）
```

核心机制分为两个层级：

**L3/L4层业务级累计流量控制（动态PCC规则方式）：**
- PCRF/PCF通过PccRule中的 `refUmData` 引用 `umDecs` 中的 `umId`，将业务级流量阈值绑定到特定PCC规则。
- PCC规则中的 `flowInfos` 定义了业务流描述（SDF Filter），限制URR只统计匹配该业务流的流量。
- PGW-C/SMF根据PccRule生成N4接口URR规则，绑定到对应PDR。

**L7层业务级累计流量控制（预定义规则方式）：**
- 使用预定义PCC规则（Predefined Rule），PGW-C/SMF和PGW-U/UPF预先配置好规则内容。
- PCRF/PCF只需下发 `pccRuleId`（和可选的 `refCondData`），`umDecs` 中定义流量阈值但 `umId` 与 `pccRule` 无绑定关系。
- PGW-C/SMF根据预定义规则名读取本地配置，获得预定义URR ID和UMID的映射关系。
- URR ID最高位bit区分预定义URR和动态URR：高位bit为1代表预定义URR ID。

### 业务流程

#### L3/L4层业务级累计流量策略控制流程

```
步骤1: UE发起PDU Session Establishment流程
步骤2: AMF进行PGW-C/SMF选择，发送Nsmf_PDUSession_CreateSMContext
步骤3: PGW-C/SMF向PCRF/PCF发送Npcf_SMPolicyControl_Create Request
       → suppFeat字段指示是否支持Usage Monitoring Control
步骤4: PCRF/PCF从UDR获取策略签约数据
步骤5: PCRF/PCF进行特性协商和策略决策
       → 回复Npcf_SMPolicyControl_Create Response（含FUP策略）
       → 下发业务级流量阈值（PccRule.refUmData → umDecs.umId → volumeThreshold）
步骤6: PGW-C/SMF根据PccRule和umDecs生成N4接口URR规则
       → URR只绑定在指定业务流对应的PDR上
       → 建立URR ID和UMID之间的映射
步骤7: PGW-C/SMF向PGW-U/UPF发送PFCP Session Establishment Request（Create PDR）
       → SDF Filter中的Flow Description从PccRule中的flowInfos提取
步骤8: PDU会话建立成功
步骤9: 用户访问指定业务，PGW-U/UPF检测流量阈值耗尽
       → 发送PFCP Session Report Request（携带Usage Report）
步骤10: PGW-C/SMF根据URR ID和UMID映射关系，通知PCRF/PCF流量阈值耗尽
步骤11: PGW-C/SMF发送Npcf_SMPolicyControl_Update Request通知PCRF/PCF
步骤12: PCRF/PCF策略决策，不再下发流量阈值，更新QoS规则
       → Npcf_SMPolicyControl_Update Response携带authSessAmbr更新带宽
       → 示例：上行带宽降为1 Mbit/s，下行带宽降为2 Mbit/s
步骤13: PGW-C/SMF检查无新流量阈值下发，通知PGW-U/UPF不再上报流量
步骤14: PGW-C/SMF发送PFCP Session Modification Request，更新业务规则
```

#### L7层业务级累计流量策略控制流程

```
前置: PGW-C/SMF、PGW-U/UPF、PCRF/PCF均已配置预定义规则
      - PGW-C/SMF预定义PCC规则名与PCRF/PCF一致
      - PGW-C/SMF预定义URR ID和umID对应关系，umID与PCRF/PCF一致
      - PGW-C/SMF预定义规则名与PGW-U/UPF一致
      - PGW-C/SMF预定义URR ID与PGW-U/UPF一致

步骤1-6: 与L3/L4流程类似，但PCRF/PCF下发的是预定义规则名
         → pccRules中只需定义pccRuleId和refCondData（可选）
         → umDecs中定义流量阈值，但umId和pccRule无绑定关系
步骤7: PGW-C/SMF根据预定义规则名读取本地配置
       → 获得预定义URR ID和UMID映射
       → 生成预定义规则PDR和预定义URR
步骤8: PGW-C/SMF向PGW-U/UPF发送PFCP Session Establishment Request
       → Create PDR包含Activate Predefined Rule和URR ID
       → URR ID高位bit为1表示预定义URR
步骤9-15: 后续阈值检测、上报、策略更新流程与L3/L4一致
```

### 协议交互

SmPolicyDecision关键信元（L3/L4动态方式）：

| 主信元 | 子信元 | 信元解释 |
|--------|--------|----------|
| PccRule | pccRuleId | PCC规则标识 |
| PccRule | flowInfos | 业务流描述（定义URR统计范围） |
| PccRule | **refUmData** | 对使用监控数据决策的引用，业务级流量阈值在PCCRule中定义，通过umID引用 |
| umDecs | umId | 标识使用监控策略数据在一个PDU会话 |
| umDecs | **volumeThreshold** | 总流量阈值 |
| policyCtrlReqTriggers | - | 达到流量阈值后上报的触发条件，如US_RE |

### 参数说明

#### ADD URR 关键参数

| 参数 | 说明 | 取值示例 |
|------|------|----------|

### 约束条件

1. **URRID一致性**：UDG和UNC两端的URR标识取值必须一致。
2. **SA依赖**：必须先开启所有SA特性以识别业务类型，否则无法区分业务流量。
3. **PCC基本功能依赖**：必须先完成PCC基本功能激活配置。
4. **UE Category限制**：对于UE Category用户，要求PCRF/PCF下发的业务级流量阈值不低于1GB，否则影响业务流量累计准确性。
5. **性能影响**：该特性需要UDG、PGW-C/SMF和PCRF/PCF之间不断交互，同时需动态处理策略，对系统性能影响较大，规划时需考虑启用该特性的用户比例。
6. **刷新顺序**：`SET REFRESHSRV` 必须在Filter配置完成后执行，使配置生效。
7. **计费与话单**：本特性本身不涉及计费与话单（计费规则由PCRF/PCF下发的计费策略决定）。

## 配置案例

### 典型场景：累计流量达到阈值后降速或重定向

**场景描述**：对访问TCP端口80（HTTP业务）的用户，实现在线流量计费和累计流量上报功能。当用户的该类业务累计流量达到PCRF/PCF配置的阈值后，由PCRF/PCF下发新的QoS规则进行降速控制。

**完整MML配置脚本**：

```mml
// ====== 步骤1：打开本特性的License配置开关 ======
SET LICENSESWITCH:LICITEM="LKV3G5FPBS01",SWITCH=ENABLE;

// ====== 步骤2：配置三四层过滤条件 ======

// 2.1 配置过滤器：匹配TCP协议，服务器端口80
ADD FILTER:FILTERNAME="filter_red",L34PROTTYPE=STRING,L34PROTOCOL=TCP,SVRSTARTPORT=80,SVRENDPORT=80;

// 2.2 配置流过滤器
ADD FLOWFILTER:FLOWFILTERNAME="fg_red";

// 2.3 绑定过滤器到流过滤器
ADD FLTBINDFLOWF:FLOWFILTERNAME="fg_red",FILTERNAME="filter_red";

// 2.4 刷新使Filter生效
SET REFRESHSRV:REFRESHTYPE=USERPROFILE;

// ====== 步骤3：配置使用量上报规则信息 ======

// 3.1 配置在线计费URR（USAGERPTMODE=ONLINE）
ADD URR:URRNAME="urr_online",URRID=1000,USAGERPTMODE=ONLINE,DFTQUOTAVOLUME=4294967295,ONLMETERINGTYPE=VOLUME;

// 3.2 配置累计流量监控URR（USAGERPTMODE=MONITORINGKEY）
ADD URR:URRNAME="urr_mk",URRID=1002,USAGERPTMODE=MONITORINGKEY;

// 3.3 配置URR组：上下行各绑定两个URR
ADD URRGROUP:URRGROUPNAME="cp_red",UPURRNAME1="urr_online",DOWNURRNAME1="urr_online",UPURRNAME2="urr_mk",DOWNURRNAME2="urr_mk";

// ====== 步骤4：配置PCC策略组 ======
// 当需要计费时配置URRGROUPNAME参数
ADD PCCPOLICYGRP:PCCPOLICYGRPNM="cg_red",URRGROUPNAME="cp_red",URRGROUPNAME="urrgroup1";

// ====== 步骤5：配置业务策略规则 ======
// 策略类型为PCC，关联流过滤器和PCC策略组
ADD RULE:RULENAME="rule_red",POLICYTYPE=PCC,FILTERINGMODE=FLOWFILTER,FLOWFILTERNAME="fg_red",POLICYNAME="cg_red";

// ====== 步骤6：配置业务策略组合 ======

// 6.1 创建用户模板
ADD USERPROFILE:USERPROFILENAME="up_test";

// 6.2 绑定规则到用户模板
ADD RULEBINDING:USERPROFILENAME="up_test",RULENAME="rule_red";
```

**配置说明**：

| 配置项 | 值 | 说明 |
|--------|-----|------|
| 过滤条件 | TCP端口80 | 匹配HTTP业务流量 |
| 在线计费URR | urr_online (ID=1000) | ONLINE模式，默认配额约4GB，按流量统计 |
| 累计流量URR | urr_mk (ID=1002) | MONITORINGKEY模式，用于FUP流量监控 |
| URR组 | cp_red | 绑定上下行两个URR |
| PCC策略组 | cg_red | 绑定URR组cp_red和urrgroup1 |
| 业务规则 | rule_red | FLOWFILTER模式，关联fg_red和cg_red |
| 用户模板 | up_test | 绑定rule_red |

**策略执行说明**：
- 本配置在UDG侧完成业务流识别和流量统计上报的基础配置。
- 实际的流量阈值（volumeThreshold）和降速QoS规则由PCRF/PCF在PDU会话建立时动态下发。
- 当业务累计流量达到PCRF/PCF下发的阈值后，PCRF/PCF通过更新authSessAmbr来改变上下行带宽（如上行1Mbit/s、下行2Mbit/s）。

## 验证与调测

本特性的验证主要关注以下几个方面：

1. **License激活验证**：确认License开关已打开，特性可用。
2. **Filter匹配验证**：确认三四层过滤条件正确匹配目标业务流量。
3. **URR上报验证**：确认PGW-U/UPF正确统计业务流量并在达到阈值时触发上报。
4. **策略执行验证**：确认PCRF/PCF下发的降速/重定向策略在PGW-U/UPF侧正确执行。

> 原始文档中未提供独立的调测章节。实际调测需结合PCC基本功能的调测方法，以及PCRF/PCF侧的策略下发验证。UDG和UNC两端的URRID必须一致，这是常见配置错误点。
