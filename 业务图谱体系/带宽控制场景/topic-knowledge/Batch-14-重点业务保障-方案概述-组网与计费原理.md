# Batch-14: 重点业务保障 - 方案概述、组网与计费原理

---

## 1. 概述

本批文档来自业务专题"5G Core 重点业务保障解决方案"，涵盖该解决方案的顶层设计全貌。9篇文档按内容可分为三大板块：

- **方案定位与场景划分**（概述2篇）：定义典型场景（全华为组网）与异厂商PCF场景（现网异厂商PCF + 华为智能PCF叠加部署）两大模式，明确两者的本质区别。
- **组网与接口架构**（2篇）：分别描述两种场景下的网元拓扑、接口协议栈、遵循的3GPP标准，以及端到端信令链路。
- **核心实现原理**（3篇）：综合判断逻辑（6类场景的决策矩阵）、重点业务保障管理总览（GBR与Non-GBR两种保障类型的条件差异）、计费原理（独立计费与归一化计费两种方案）。
- **工程约束**（1篇）：版本配套要求，列出SMF、UPF、PCF、NRF、NWDAF（UDC/UDN）、RAN-OAM、BOSS等全部涉及网元的最低版本和功能描述。

这些文档构成了重点业务保障解决方案的"骨架"：从业务方案定位出发，经组网架构落地，到核心决策逻辑和计费衔接，最后以版本兼容性收束，为实际部署提供了完整的参考框架。

## 2. 核心知识点

### 2.1 两种部署场景的本质差异

重点业务保障解决方案分为**典型场景**和**异厂商PCF场景**两种模式，差异核心在于PCF的角色分配：

| 维度 | 典型场景 | 异厂商PCF场景 |
|------|----------|--------------|
| PCF部署 | 现网仅部署华为智能PCF，同时承担PCC业务和智能数据分析业务 | 现网已有异厂商PCF（执行PCC业务），需额外部署一对华为智能PCF（仅做智能业务分析，不执行PCC） |
| N7会话 | 单一N7会话（SMF<->华为PCF） | 双N7会话：第一条SMF<->异厂商PCF（PCC策略），第二条SMF<->智能PCF（智能分析订阅） |
| 智能UPF选择 | 由现有流程处理 | SMF根据异厂商PCF下发的智能PCC Rule选择智能UPF，再发起第二条N7连接 |
| 订阅发起方 | 华为PCF通过N23向NWDAF订阅 | 智能PCF通过N23向NWDAF订阅，携带异厂商PCF ID |
| 保障建议接收方 | 华为PCF（同一PCF既订阅又接收保障建议） | 异厂商PCF（NWDAF通过N5向异厂商PCF发起保障建议，而非智能PCF） |
| 异厂商PCF ID变更处理 | 不涉及 | SMF感知变化后通知智能PCF，智能PCF先取消旧订阅再重新下发 |
| SMF最低版本 | UNC 20.12.2 | UNC 20.15.0（需支持创建第二条N7会话） |

**关键结论**：异厂商PCF场景下，智能PCF只负责"订阅分析"通道，实际保障执行（专载创建/终止）仍由异厂商PCF通过标准N5接口完成。这意味着保障决策与保障执行被拆分到不同的PCF实体上。

**架构设计深意**：异厂商PCF场景的设计哲学是"不侵入现网PCC"。现网异厂商PCF已经承载了全部PCC业务逻辑，直接修改其行为风险极高。通过引入智能PCF作为"分析代理"，将智能数据分析能力（N23订阅）与现网PCC策略执行能力（N5专载创建）解耦，既实现了智能保障，又避免了对现网PCF的侵入式改造。智能PCF本质上是一个"信令中继+分析适配器"。

**双N7会话的技术含义**：SMF为同一用户维护两条N7会话，第一条连接异厂商PCF完成标准PCC流程（会话建立、策略获取），第二条连接智能PCF完成智能分析订阅。这两条N7会话在同一SMF实例中并存，但逻辑隔离。SMF需具备区分两条N7会话用途的能力，这是UNC 20.15.0版本的核心增量功能。

### 2.2 端到端业务保障流程（7步典型 vs 13步异厂商）

**典型场景流程（7步）**：

1. BOSS向PCF下发保障套餐 -> 用户激活时PCF识别保障用户 -> 通过NRF寻址NWDAF -> N23下发保障订阅
2. NWDAF基于SMF ID寻址SMF -> Nsmf下发质差分析订阅
3. SMF转译为N4消息发送给UPF
4. UPF持续进行业务体验质量分析 -> 带宽低于阈值时通过Nupf上报质差事件
5. NWDAF（UDN）通过FTP/SFTP周期性（15分钟）从RAN-OAM获取无线性能指标 -> UDC通过HTTP接口从UDN获取小区容量和PRB利用率
6. NWDAF综合分析是否可保障 -> N5向PCF发起保障建议
7. PCF决策是否创建专载 -> 将结果通知NWDAF

**异厂商PCF场景流程（13步）**：

- 前3步（接入与双N7建立）：AMF/MME根据DNN/APN选择支持双N7的SMF -> SMF与异厂商PCF交互获取智能PCC Rule -> SMF选择智能UPF并发起第二条N7到智能PCF
- 第4步（订阅建立）：智能PCF通过N23向NWDAF订阅
- 第5~8步：与典型场景步骤2~5基本一致（订阅下发 -> 质差分析 -> 综合判断）
- 第9~13步：NWDAF通过N5向**异厂商PCF**（非智能PCF）发起保障建议 -> 异厂商PCF决策专载创建/终止

**差异要点**：异厂商场景多了"双N7建立"和"智能UPF选择"两个前置步骤，保障建议的接收对象从智能PCF变成了异厂商PCF。

**流程时序对比**：

| 阶段 | 典型场景 | 异厂商PCF场景 |
|------|----------|--------------|
| 用户接入 | 标准AMF/SMF选择 | AMF/MME需根据DNN/APN选择支持双N7的SMF |
| PCC策略获取 | PCF向SMF下发标准PCC规则 | 异厂商PCF向SMF下发标准PCC规则 + 智能PCC Rule |
| 分析订阅建立 | 华为PCF -> N23 -> NWDAF | 智能PCF -> N23 -> NWDAF（携带异厂商PCF ID） |
| 质差分析链路 | NWDAF -> Nsmf -> SMF -> N4 -> UPF | 完全一致 |
| 保障建议通道 | NWDAF -> N5 -> 华为PCF | NWDAF -> N5 -> 异厂商PCF |
| 专载创建 | 华为PCF决策并执行 | 异厂商PCF决策并执行（标准N5流程） |
| 订阅维护 | 无需特殊处理 | 异厂商PCF ID变更 -> SMF通知智能PCF -> 取消旧订阅 -> 重新订阅 |

**关键风险点**：异厂商场景中，如果异厂商PCF不支持标准N5接口的PolicyAuthorization服务中的某些扩展信元，可能导致保障建议中的部分信息（如medComponents媒体部件）无法被异厂商PCF正确解析，需要在部署前验证异厂商PCF的N5接口兼容性。

### 2.3 综合判断决策矩阵

NWDAF收到UPF质差消息后，依据6类场景进行综合判断。这是重点业务保障的"大脑"：

| 场景编号 | 场景大类 | 触发条件 | NWDAF处理 |
|----------|----------|----------|-----------|
| 场景1 | 应用质差（首次） | UPF首次检测到用户质差并上报 | 判断4项条件全满足后，通过Npcf_PolicyAuthorization_Create向PCF发起保障 |
| 场景2 | 质差业务流变更 | 同一subAppId下新增非首条主流质差 | 未触发保障：合并判断；已触发：Update消息，新增流mirBwUl/Dl置0（GBR不叠加） |
| 场景3 | 质差业务流变更 | 同一appId下新subAppId主流质差 | 未触发：合并判断；已触发：新subAppId独占GBR，Update携带新GBR/MBR值 |
| 场景4 | 质差业务流变更 | 新appId主流质差（跨应用） | 未触发：合并判断；已触发：新subAppId独占GBR，Update携带新GBR/MBR值 |
| 场景5a | 业务流结束 | subAppStatus=TERMINATED，全部流结束 | 未触发：不处理；已触发：按配置决定延迟释放（SET QOSGUARTIMER），不延迟则Delete通知PCF |
| 场景5b | 业务流部分结束 | flowStatus有TERMINATED但subAppStatus=RUNNING | 未触发：不处理；已触发：Update删除流（fStatus=REMOVED），不回收GBR资源 |
| 场景6 | 用户位置变更 | UPF上报小区变更 | 未触发：判断新小区是否满足条件；已触发：满足则刷新配额继续保障，不满足则停止保障 |

**场景1的4项保障判断条件**（GBR保障）：
1. 小区GBR配额足够
2. 小区不拥塞
3. 无会话级阻塞（不在会话级保障失败定时器保护内）
4. 无小区级阻塞（不在小区级保障失败定时器保护内）

**Non-GBR保障时**：条件1（小区GBR配额）不做判断，仅检查条件2~4。

**媒体部件分配规则**：
- 每条主流分配独立的medComponents媒体部件
- 同一subAppId内新增流：mirBwUl/mirBwDl置0，不叠加GBR
- 不同subAppId/appId新增主流：独占GBR，携带各自的GBR/MBR值

**阻塞定时器机制解析**：

阻塞定时器是防止"保障乒乓效应"的关键保护机制。当一次保障尝试失败时（如小区资源不足导致PCF拒绝保障建议），NWDAF会启动对应级别的阻塞定时器：

- **会话级阻塞**：特定用户的某次会话保障失败后，在该会话的阻塞定时器保护期内，不再对该会话发起保障。保护期结束后，如果质差仍然存在，可以重新尝试。
- **小区级阻塞**：某小区的保障失败后，在该小区的阻塞定时器保护期内，不再对该小区下的任何用户发起保障。这防止了在资源持续紧张的小区反复发起注定失败的保障请求。

阻塞定时器的时长需要根据现网实际情况调优：过短会导致频繁重试浪费信令资源，过长会导致用户质差恢复延迟。

**流变更场景的业务逻辑**：

场景2/3/4的差异在于"新增质差流"与"已有质差流"的层级关系：

- **场景2**（同一subAppId内）：如云游戏业务中同一子应用有多条主流。新增流不需要额外GBR资源，mirBw置0表示该流不额外限制带宽，复用已有的GBR配额。
- **场景3**（同一appId不同subAppId）：如同一视频应用中切换到不同子应用。新subAppId需要独占GBR，即分配新的GBR/MBR值，与原有subAppId的GBR独立。
- **场景4**（不同appId）：如从视频直播切换到云游戏。跨应用的新主流质差，处理逻辑与场景3一致，新subAppId独占GBR。

这说明GBR资源是按subAppId维度分配的，而非按appId或按用户维度。一个用户可能同时有多个subAppId的GBR保障，每个subAppId独立占用一份GBR资源。

### 2.4 GBR保障 vs Non-GBR保障

通过UDC的 **ADD APPGROUP** 命令"保障类型"参数配置：

| 维度 | GBR保障 | Non-GBR保障 |
|------|---------|-------------|
| 判断条件 | 小区GBR配额 + 小区拥塞 + 会话级阻塞 + 小区级阻塞（4项全满足） | 小区拥塞 + 会话级阻塞 + 小区级阻塞（3项满足即可） |
| 资源类型 | 保证比特速率，独占无线资源 | 非保证比特速率，不独占资源但提升优先级 |
| 版本要求 | UDC 20.12.2起支持 | UDC 20.13.2起支持Non-GBR专载保障 |

**异厂商PCF场景约束**：UDC不支持从UDN获取小区容量和PRB利用率，因此：
- 不支持将"小区拥塞"作为判断条件
- 可在UDC本地配置小区容量，通过本地配置替代判断

### 2.5 计费原理：两种方案

重点业务保障引入专载后，保障流量如何计费是一个关键问题：

**方案一：保障流量独立计费**

| 要素 | 说明 |
|------|------|
| 适用条件 | N5接口支持X-AF-Charging-Identifier计费标识；运营商对保障业务有独立计费诉求 |
| 实现原理 | NWDAF在N5消息中携带X-AF-Charging-Identifier -> PCF用它替换原始RG生成动态规则 -> SMF通过N4下发到UPF -> UPF基于知识库识别保障业务流量并独立计费 |
| 风险提示 | 若现网采用基于IP/URL白名单识别应用（而非知识库），与NWDAF传递计费标识的方式存在计费差异，追求精准计费的运营商不建议选择 |

**方案二：保障流量归一化计费**

| 要素 | 说明 |
|------|------|
| 适用条件 | 运营商对保障业务无独立计费诉求；PCF下发动态规则不携带ChargingData信元 |
| 实现原理 | NWDAF在N5消息中不携带X-AF-Charging-Identifier -> PCF下发动态规则不携带ChargingData -> SMF根据软参构造无效URR ID下发给UPF -> UPF对携带无效URR ID的PDR执行原有计费流程 |
| 版本要求 | 若SMF和UPF为华为设备，需分别升级到UNC 20.13.2.36 / UDG 20.13.2.10或后续版本 |

**方案对比**：方案一实现"保障流量单独计费"（可能是优惠或特殊费率），方案二实现"保障流量按原规则计费"（用户无感知）。方案二通过"构造无效URR ID"这一机制让保障专载流量回归原有计费路径，是一个精巧的工程设计。

**计费方案的深层逻辑**：

保障专载（dedicated bearer）创建后，UPF上会有专门匹配保障业务流的PDR规则。正常情况下，每个PDR关联一个URR（Usage Reporting Rule），UPF按URR定义的规则统计流量并上报给SMF/CHF进行计费。

方案一的核心是让保障流量走一条全新的计费路径：NWDAF通过X-AF-Charging-Identifier指定新的费率标识，PCF据此生成带有新RG（Rating Group）的动态规则，最终在UPF上为保障流量创建独立的URR，实现差异化计费。

方案二的核心是让保障流量"隐身"于原有计费路径：SMF构造一个"无效URR ID"（可以被UPF识别为不触发计费上报的ID），UPF遇到该URR ID时不产生新的计费记录，流量回归原有PDR-URR关联路径，按用户原有套餐计费。这意味着保障专载虽然创建了独立的PDR来匹配保障流量，但计费层面保持透明。

**方案选择的关键考量**：

1. **识别方式差异**：UPF采用知识库识别应用时，方案一的计费标识能精确匹配保障流量；若采用IP/URL白名单识别，知识库与白名单的识别范围可能不一致，导致计费偏差。这是因为NWDAF侧基于知识库判定质差，而UPF计费侧可能基于白名单匹配流量，两者覆盖的业务流范围可能不同。

2. **异厂商PCF兼容性**：方案一要求N5接口支持X-AF-Charging-Identifier，异厂商PCF可能不支持该扩展信元。如果现网为异厂商PCF，方案一的可行性取决于异厂商PCF的接口能力。

3. **版本敏感度**：方案二对华为SMF/UPF版本有明确要求（UNC 20.13.2.36 / UDG 20.13.2.10），因为"构造无效URR ID"这一功能是华为私有实现，依赖特定版本的软参支持。

### 2.6 版本配套全景

重点业务保障是跨网元协同方案，对版本有严格的最低要求：

| 网元 | 最低版本 | 关键能力增量 |
|------|----------|-------------|
| SMF | UNC 20.12.2 | 用户实时位置上报、Nsmf接口解析、N4转译 |
| SMF | UNC 20.13.0 | 根据PCF下发智能规则选择智能UPF |
| SMF | UNC 20.15.0 | 异厂商PCF场景下创建第二条N7会话（异厂商场景必需） |
| UPF | UDG 20.12.2 | 业务主流筛选、质差分析、Nupf质差上报 |
| PCF | UPCF 20.13.0 | N23订阅、N5接收保障建议 |
| NRF | USC 23.1.2.8 或 UNC 20.13.2 | NF服务发现 |
| NWDAF(UDC) | UDC 20.12.2 | 核心保障功能（N23/Nsmf/Nupf/N5/Nudn全接口） |
| NWDAF(UDC) | UDC 20.13.0 | 跨区域漫游、智能码率识别 |
| NWDAF(UDC) | UDC 20.13.2 | 跨制式切换保障、Non-GBR专载保障 |
| NWDAF(UDC) | UDC 20.15.0 | NRF注册携带NwdafInfoList、3GPP标准N5接口发起保障建议 |
| NWDAF(UDN) | CloudUDN V100R024C10SPC100 | 小区容量/PRB利用率预测、Nudn接口双向交互 |
| RAN-OAM | 第三方设备 | FTP/SFTP提供无线PRB统计指标 |
| BOSS | 第三方设备 | 质差套餐签约 |

**版本演进脉络**：UDC 20.12.2是基础版本（核心保障闭环），20.13.0扩展了漫游和码率识别，20.13.2增加了Non-GBR和跨制式，20.15.0完善了NRF注册和标准N5接口。版本升级路径清晰体现了功能的增量演进。

**版本兼容性矩阵要点**：

在实际部署中，版本配套不仅仅是"最低版本"问题，还涉及**功能组合可用性**：

| 功能组合 | SMF要求 | UPF要求 | UDC要求 | PCF要求 |
|----------|---------|---------|---------|---------|
| 基础GBR保障（典型场景） | UNC 20.12.2 | UDG 20.12.2 | UDC 20.12.2 | UPCF 20.13.0 |
| Non-GBR保障 | UNC 20.12.2 | UDG 20.12.2 | UDC 20.13.2 | UPCF 20.13.0 |
| 跨制式切换保障 | UNC 20.12.2 | UDG 20.12.2 | UDC 20.13.2 | UPCF 20.13.0 |
| 跨区域漫游保障 | UNC 20.12.2 | UDG 20.12.2 | UDC 20.13.0 | UPCF 20.13.0 |
| 异厂商PCF场景 | UNC 20.15.0 | UDG 20.12.2 | UDC 20.15.0 | 异厂商（N7+N5） |
| 归一化计费 | UNC 20.13.2.36 | UDG 20.13.2.10 | UDC 20.12.2 | UPCF 20.13.0 |
| 3GPP标准N5发起保障 | UNC 20.12.2 | UDG 20.12.2 | UDC 20.15.0 | UPCF 20.13.0 |

**关键洞察**：SMF的版本要求变化最频繁（20.12.2 -> 20.13.0 -> 20.15.0），因为SMF是策略转译的核心枢纽，新增功能（智能UPF选择、双N7会话）都需要SMF配合。UPF版本要求最稳定（20.12.2即可满足基础需求），因为UPF的功能定位（质差检测+策略执行）相对固定。

### 2.8 保障建议消息族

综合判断场景中涉及三类Npcf_PolicyAuthorization消息：

| 消息 | 触发时机 | 关键信元 |
|------|----------|----------|
| Create_Request | 场景1：首次满足保障条件 | 完整保障信息（GBR/MBR、medComponents、mirBwUl/Dl等） |

## 3. 配置调测要点

### 3.1 关键命令与参数

| 命令 | 网元 | 参数 | 说明 |
|------|------|------|------|
| ADD APPGROUP | UDC | 保障类型 | 配置应用组进行GBR还是Non-GBR保障 |
| SET QOSGUARTIMER | UDC | N5会话延迟删除时长(秒) | 控制质差结束后是否延迟释放专载，避免频繁创建/删除 |

### 3.2 版本配套硬约束

- **异厂商PCF场景**：SMF必须 >= UNC 20.15.0（双N7会话能力），UDC必须 >= UDC 20.15.0（标准N5接口）
- **Non-GBR保障**：UDC必须 >= UDC 20.13.2
- **归一化计费方案二**：SMF >= UNC 20.13.2.36，UPF >= UDG 20.13.2.10
- **NRF**：USC 23.1.2.8 或 UNC 20.13.2（二选一）
- **UDN**：CloudUDN V100R024C10SPC100

### 3.3 组网要素

- **典型场景**：BOSS -> PCF(华为) -> NWDAF(UDC) -> SMF -> UPF；UDN <- SFTP/FTP <- RAN-OAM；UDC <-> UDN via Nudn
- **异厂商场景**：额外引入智能PCF，SMF需维护双N7会话；异厂商PCF ID变更时需触发订阅重建
- **无线数据采集**：UDN以15分钟为周期从RAN-OAM采集PRB指标，UDC通过HTTP接口从UDN获取预测结果

### 3.4 异厂商PCF场景的关键部署约束

异厂商PCF场景的部署复杂度显著高于典型场景，需要特别注意以下约束：

**约束1：UDC无法获取实时无线数据**

异厂商PCF场景下，UDC不支持从UDN获取小区容量和PRB利用率。这意味着：
- 无法使用"小区拥塞"作为保障判断条件
- 替代方案是在UDC本地配置小区容量（静态值），但静态值无法反映实时负载变化
- 保障决策的准确性受影响，可能出现"基于过时容量信息发起保障"的情况
- 建议在异厂商场景中优先使用Non-GBR保障（不依赖GBR配额判断）

**约束2：异厂商PCF的N5接口兼容性**

保障建议通过标准N5接口（Npcf_PolicyAuthorization服务）发送给异厂商PCF。需要验证：
- 异厂商PCF是否支持3GPP TS 29.514定义的PolicyAuthorization_Create/Update/Delete服务
- 保障建议中的medComponents、mirBwUl/Dl、GBR/MBR等信元是否能被异厂商PCF正确解析
- 异厂商PCF对专载创建的支持程度（是否支持动态创建GBR专载）

**约束3：双N7会话的SMF能力**

SMF需要同时维护与异厂商PCF和智能PCF的两条N7会话：
- 必须正确区分两条N7会话的用途（PCC vs 分析订阅）
- 异厂商PCF ID变更时，SMF需要感知并通知智能PCF触发订阅重建
- 如果SMF重启或会话重建，需要确保两条N7会话的一致性

**约束4：智能PCF部署为一对（主备）**

原文指出"部署一对用于智能业务的智能PCF"，说明智能PCF需要主备冗余。这对NRF寻址和故障倒换提出了要求：
- NRF需要正确管理智能PCF的注册和发现
- 主备倒换时，NWDAF侧的N23订阅需要无缝迁移
- 智能PCF故障不应影响现网PCC业务（因为PCC由异厂商PCF独立处理）

### 3.6 计费方案选择决策树

```
运营商是否对保障业务有独立计费诉求？
├── 是 -> N5是否支持X-AF-Charging-Identifier？
│        ├── 是 -> 现网是否基于IP/URL白名单识别应用？
│        │        ├── 是 -> 计费差异风险，不建议方案一
│        │        └── 否 -> 方案一（独立计费）
│        └── 否 -> 不满足方案一前提，考虑方案二
└── 否 -> PCF动态规则是否携带ChargingData信元？
         ├── 保障业务不携带 -> 方案二（归一化计费），需升级SMF/UPF版本
         └── 携带 -> 不满足方案二前提，需定制方案
```

## 4. 与带宽控制的关系

### 4.1 组网架构是带宽控制的物理基础

重点业务保障解决方案的组网架构（PCF-NWDAF-SMF-UPF-UDN-RAN-OAM）是带宽控制场景的完整基础设施。带宽控制的核心操作（策略下发、专载创建、码率调整）都依赖这套组网链路。

- **PCF**：带宽策略决策中心，接收NWDAF建议后决定是否创建GBR/Non-GBR专载
- **SMF**：策略转译节点，将PCF决策通过N4接口传递给UPF
- **UPF**：策略执行节点，执行码率限制、业务识别、质差检测
- **NWDAF(UDC)**：智能分析中枢，综合判断是否需要带宽保障
- **NWDAF(UDN)**：无线数据底座，提供小区容量和PRB利用率预测

### 4.2 综合判断是带宽决策的高级形态

带宽控制场景中的码率策略（如FUP门限触发降速、带宽管理模板限速）通常是**规则驱动**的：满足条件A就执行动作B。

而重点业务保障的综合判断是**多维度动态决策**：
- 不仅看业务质量（UPF质差检测），还看无线资源（小区GBR配额、PRB利用率）
- 不仅看当前状态，还看历史失败（会话级/小区级阻塞定时器保护）
- 不仅做创建决策，还做变更（新增流、小区切换）和释放决策
- 支持GBR（独占资源）和Non-GBR（提升优先级）两种资源保障等级

这种多维度综合判断代表了带宽控制从"静态规则匹配"向"动态智能决策"的演进方向。

**与带宽控制各子场景的交叉点**：

| 带宽控制子场景 | 与重点业务保障的交叉点 |
|----------------|----------------------|
| PCC策略控制 | 重点业务保障通过N5接口向PCF发起保障建议，最终以PCC规则形式下发，与PCC策略控制共享PCF-SMF-UPF链路 |
| 带宽管理（BWM） | GBR保障创建的专载携带GBR/MBR参数，直接控制保障流量的上下行码率，与带宽管理的码率模板体系互补 |
| FUP门限控制 | FUP降速后可能触发质差，质差又触发重点业务保障。两者形成"降速->质差->保障"的级联关系 |
| QoS承载映射 | GBR/Non-GBR保障最终通过QoS Flow和专载实现，与QoS承载映射的5QI体系直接关联 |
| ADC应用识别 | UPF使用特征库/知识库识别应用是重点业务保障的前提，与ADC应用识别能力直接关联 |
| 流量整形（Shaping） | 保障专载的mirBwUl/mirBwDl参数本质是码率限制，与流量整形机制共享底层实现 |

### 4.3 计费原理直接关联计费子场景

计费方案二（归一化计费）中"SMF构造无效URR ID下发给UPF，UPF对携带无效URR ID的PDR执行原有计费流程"这一机制，直接关联计费场景中的URR（Usage Reporting Rule）体系。理解这一机制需要掌握：
- URR ID的构造规则和无效ID的定义
- PDR与URR的关联关系
- 计费方案一中的X-AF-Charging-Identifier如何替换RG生成动态规则
- ChargingData信元在计费链路中的作用

### 4.4 版本配套约束影响带宽控制特性可用性

带宽控制场景中的各特性（如Non-GBR专载保障、跨制式切换保障、标准N5接口）都受版本约束。理解版本配套要求有助于判断特定带宽控制能力是否在现网可用，以及升级路径如何规划。

## 5. 关键术语

| 术语 | 全称/含义 |
|------|-----------|
| NWDAF | Network Data Analytics Function，网络数据分析功能，本方案中由UDC（分析决策）和UDN（数据底座）组成 |
| UDC | 统一数据中心，作为NWDAF的分析决策实体 |
| UDN | 统一数据网络，作为NWDAF的数据底座，提供小区容量和PRB利用率预测 |
| PCF | Policy Control Function，策略控制功能 |
| 智能PCF | 华为部署的仅用于智能业务分析的PCF，不执行PCC业务 |
| 异厂商PCF | 现网已有的非华为PCF，执行PCC业务，在异厂商场景中接收保障建议 |
| GBR | Guaranteed Bit Rate，保证比特速率，独占无线资源的保障方式 |
| Non-GBR | 非保证比特速率，不独占资源但提升QoS优先级 |
| PRB | Physical Resource Block，物理资源块，无线资源分配的最小单位 |
| GBR配额 | 小区可分配给GBR保障业务的资源上限 |
| mirBwUl/Dl | Maximum IP Bit Rate Uplink/Downlink，上下行最大IP带宽 |
| medComponents | 媒体部件，每条质差主流对应一个独立的媒体部件 |
| subAppId | 子应用ID，标识同一应用下的不同业务流 |
| appId | 应用ID，标识不同应用（如视频直播、云游戏） |
| X-AF-Charging-Identifier | AF（Application Function）计费标识，用于独立计费方案 |
| ChargingData | 计费数据信元，PCF下发动态规则时携带的计费信息 |
| URR | Usage Reporting Rule，使用量上报规则，计费体系核心信元 |
| PDR | Packet Detection Rule，报文检测规则 |
| RG | Rating Group，费率组 |
| SRR | Session Report Request，PFCP会话报告请求 |
| N23接口 | PCF <-> NWDAF，订阅QoS保障分析 |
| N5接口 | NWDAF <-> PCF，发起保障建议（PolicyAuthorization服务） |
| Nsmf接口 | NWDAF <-> SMF，发送QOS_ANA事件订阅 |
| Nupf接口 | UPF <-> NWDAF，通知业务体验监测信息 |
| Nudn接口 | UDC <-> UDN，获取小区容量和PRB利用率预测 |
| 会话级阻塞 | 会话级保障失败定时器保护期内，该会话不能再发起保障 |
| 小区级阻塞 | 小区级保障失败定时器保护期内，该小区不能再发起保障 |
| 智能PCC Rule | 异厂商PCF在N7消息中向SMF下发的规则，用于触发第二条N7建立和智能UPF选择 |
