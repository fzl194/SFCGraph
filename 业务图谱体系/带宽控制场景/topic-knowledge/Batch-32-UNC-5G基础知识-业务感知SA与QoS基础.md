## 2. 核心知识点

### K-001: 业务感知（SA）的定义

业务感知（Service Awareness，简称SA），是指在用户会话过程中，对用户的数据报文进行解析，从而区分出用户使用的不同业务，并根据业务策略对业务数据进行转发/控制的过程。

- 与5G SA（Standalone）组网缩写相同但含义完全不同，须区分。
- 核心价值：让核心网"看到"用户在做什么业务，从而对报文进行差异化处理（分配带宽、配置时延、禁止访问等）。
- 类比快递分拣：核心网像分拣中心，根据数据包的"包裹信息"（报文特征）识别目的地（业务类型），并根据"配送政策"（策略）送达不同路径。

**证据**: 何为业务感知——What？_86009700.md

### K-002: 业务感知的三大应用方向（Why）

业务感知的本质是对流量的"经营"，三大典型应用方向：

| 应用方向 | 说明 | 典型场景 |
|----------|------|----------|
| **计费精细化** | 基于时间段、位置、用户类别、业务实现策略控制 | 免流量（特定APP不计费）、高峰时段限速、特定位置免费体验推广 |
| **带宽控制** | 对P2P、FTP等协议的带宽控制，保证网络资源均衡分配 | 防止下载占用大量带宽影响其他用户；资源充沛时保证签约P2P业务的服务质量 |
| **安全防护** | 过滤、管理UPF承载的数据面流量 | 避免来自不安全数据中心或恶意攻击；防止无恶意误操作影响网络性能和正常业务 |

**对运营商**：高效控制和利用带宽资源，制定灵活策略，保障和提升收益。
**对用户**：细分业务并提供个性化服务，提升使用体验。

**证据**: 使用业务感知——Why？_32687383.md

### K-003: 业务感知的端到端流程（7步全景）

业务感知的完整流程分为7个步骤：

| 步骤 | 动作 | 说明 |
|------|------|------|
| Step1 | PCF下发规则 | PCF下发规则经SMF"翻译"后到达UPF，存放于UPF规则库中。规则有多个，多种类型 |
| Step2 | 用户数据到达UPF | 用户接入完成、开始会话后，数据报文经基站到达UPF |
| Step3 | 解析与识别 | UPF对报文进行解析与识别，获取报文特征。细分为三四层解析、协议识别、七层解析 |
| Step4 | 规则匹配 | UPF将报文特征与规则库中的规则（过滤条件+策略+优先级+其他参数）匹配 |
| Step5 | 执行策略 | 成功匹配的报文执行对应策略：计费、阻塞/重定向、带宽控制、头增强等 |
| Step6 | 数据转发 | 用户数据从UPF继续转发到网络/业务服务器 |
| Step7 | 用量上报 | UPF定期向SMF上报使用情况（计费套餐用量、使用时长和具体时间等） |

下行报文的业务感知方式与上行基本一致。

**证据**: 业务感知概览_32946283.md

### K-004: 业务感知的三大核心概念——过滤条件、策略、规则

#### 过滤条件（业务感知的"手"）

过滤条件是规则的一部分，用于"抓住流量"——将用户流量特征与规则匹配：

- **三四层过滤**：以网络层/传输层协议、端口号、IP地址作为过滤条件。
- **七层过滤**：以应用层URL、User-Agent、method等七层关键字段作为过滤条件。
- **协议/协议组**：以应用层协议（HTTP、RTSP等）或协议组合作为过滤条件。
- **流过滤器**：以上三种过滤条件的组合，用于实现过滤逻辑。一条规则只能绑定一个流过滤器，但可包含多个过滤条件。

#### 策略（业务感知的"腿"）

策略代表业务感知需要实现的内容：

| 策略类型 | 含义 |
|----------|------|
| **PCC** | 计费与策略控制，配置PCC策略实现计费和策略控制（禁止访问、重定向等） |
| **BWM** | 带宽管理，配置带宽管理分类策略，分类结果用于带宽管理的策略匹配 |
| **HEADEN** | 头增强，基于头增强策略向HTTP/RTSP请求报文头域中插入字段 |
| **WEBPROXY** | Web Proxy，配置WebProxy的IPFarm对象名称，设置服务器地址池 |
| **SMARTREDIRECT** | 智能重定向，配置Captive Portal的IPFarm名称、HTTP智能重定向名称或DNS重写动作 |

一条规则只能绑定一条策略。按绑定策略类型可分为PCC规则、BWM规则、HEADEN规则等。

#### 规则（业务感知的"身体"）

规则是完成业务感知的最小单元，绑定过滤条件、策略、规则优先级和其他参数。按协议定义分为三类：

| 规则类型 | 规划位置 | 启用方式 | 特点 |
|----------|----------|----------|------|
| **预定义规则** | UPF和SMF | 已部署但未启用，需PCF通过Npcf消息通知SMF和UPF启用 | 必不可少的基本家具，能覆盖大多数使用需求 |
| **动态规则** | PCF | PCF通过Npcf消息直接下发后启用 | 灵活但功能有限，对UPF/SMF无感知，**无法包含七层过滤条件** |
| **本地规则** | UPF和SMF | SMF找不到PCF时直接下发 | 应急场景使用，无PCF时保证业务相对正常进行 |

**证据**: 相关概念_86169670.md

### K-005: 解析与识别的三层体系

UPF对报文的解析识别分为三大类（依次深入）：

1. **三四层解析**（浅度报文检测）：
   - 第三层（网络层）：监控IP报文的源/目的IP地址。
   - 第四层（传输层）：检测报文段的协议端口号（不同应用层协议用不同端口号定义）。
   - 三四层解析是后续协议识别和七层解析的基础。

2. **协议/协议组识别**：
   - 对用户报文的协议或协议组进行识别。
   - 只有识别出协议后，才能进行后续七层解析。

3. **七层解析**（深度报文业务感知）：
   - 在三四层解析和协议识别后，对报文进行精确的业务感知。
   - 获取七层协议类型或目的URL等有价值信息。
   - 对每个请求和响应解析出关键字信息：URL、Method、响应码。

**证据**: 业务感知流程_85850060.md — 解析与识别章节

### K-006: 规则匹配的三大原则

华为产品中，规则匹配遵循三个原则：

1. **优先级匹配**：所有报文根据规则的优先级，逐条匹配规则。
2. **全条件匹配**：报文只有符合一条规则所有的过滤条件，才算成功匹配这条规则。
3. **类型独立匹配**：不同类型的规则会同时匹配，且每种类型的规则至多只会成功匹配一条。

匹配有两个维度：
- **类型间独立**：PCC规则、BWM规则、HEADEN规则等各类型之间的匹配互相独立。
- **类型内优先**：同类型规则按优先级先后匹配，直到有一条匹配成功或全部失败。

例如：UPF部署16条4种类型规则，一条HTTP协议、访问10.11.12.13的数据流可以同时匹配到PCC规则4 + BWM规则7 + HEADEN规则10。

**证据**: 业务感知流程_85850060.md — 规则匹配与执行章节

### K-007: 规则下发机制

规则下发涉及SMF与UPF之间的会话建立/修改、SMF与PCF之间的会话建立/修改：

- **用户接入时**：PCF根据用户信息下发需要启用的预定义规则或动态规则。
- **用户会话过程中**：UPF上报的会话信息经SMF处理到达PCF，PCF根据信息判断是否需要下发/更新/删除规则。新下发的规则经SMF到达UPF。
- **无PCF场景**：SMF直接下发需要启用的本地规则到UPF。

策略变更请求也可能导致规则在会话过程中下发新规则，发起方包括：
- UE（业务变更需要更新QoS）
- SMF（UDM识别到本地配置策略修改）
- PCF（本端配置策略修改）
- 基站（现有QoS不能满足业务要求）

这些变更通过信令交互汇总到SMF，由SMF向PCF发送Npcf_SMPolicyControl_Update Request消息。

**证据**: 业务感知流程_85850060.md — 规则下发章节

### K-008: 5G QoS数据通路——QoS Flow

QoS Flow是5G中端到端QoS控制的最小粒度：

- 相同通路上的所有数据流获得相同的QoS保障（调度策略、缓冲队列管理等）。
- 不同的QoS保障需要不同的通路来提供。
- QFI（QoS Flow ID）是PDU Session中QoS差异化的最细粒度标识，在一个PDU会话中必须唯一。

4G vs 5G数据通路对比：

| 对比项 | 4G | 5G |
|--------|-----|-----|
| 数据通路 | EPS Bearer | QoS Flow |
| 通路标识 | EBI | QFI |
| 通路分类 | 默认承载 / 专有承载 | 默认QoS Flow / 专有QoS Flow |
| 底层承载 | 各段一一对应 | 一个PDU会话共用一个NG-U Tunnel；Radio Bearer与QoS Flow为一对多映射 |

关键演进：QoS Flow使CN和RAN实现功能解耦，5G CN可适配各种接入技术（如固网接入），由RAN自主决策空口如何承载QoS Flow。

**证据**: QoS_42268275.md — 数据通路章节

### K-009: 从业务到通路的映射机制

5G使用两级映射机制（4G为一级映射）：

- **第一级映射**：数据包 → QoS Flow。使用包过滤器集（PFS）与业务数据包匹配。UE侧使用QoS Rule（N QoS Rule ↔ 1 QoS Flow），UPF侧使用PDR（N PDR ↔ 1 QoS Flow）。
- **第二级映射**：QoS Flow → Radio Bearer。空口侧由RAN负责，一个Radio Bearer可承载多个QoS Flow。

为区分NG-U隧道中传输的包属于哪个QoS Flow，UE和UPF需要在数据包封装头上添加QFI标识。

4G/5G QoS映射差异：

| 对比项 | 4G | 5G |
|--------|-----|-----|
| 映射机制 | 一级映射：数据包→EPS Bearer | 两级映射：数据包→QoS Flow→Radio Bearer |
| 特征 | 对称性：上下行处理一致 | 非对称性：上下行处理不一致 |
| UL执行网元 | UE | UE |
| DL执行网元 | P-GW | UPF、RAN |
| UL关联工具 | UL TFT | QoS Rule + 空口映射规则 |
| DL关联工具 | DL TFT | PDR + 空口映射规则 |

**证据**: QoS_42268275.md — 从业务到通路章节

### K-010: 5G QoS参数体系

5G QoS参数从三个维度评价，并增加可选参数：

#### 基本参数（对应4G，有继承和增强）

| 维度 | 参数 | 说明 | 4G→5G变化 |
|------|------|------|-----------|
| **有没有路** | ARP（分配保留优先级） | 标识业务获取通路的能力，控制QoS Flow建立/修改优先级。包含优先级、抢占能力、被抢占属性 | 无变化 |
| **路有多宽** | xBR（GBR/Non-GBR） | GBR通路：GFBR/MFBR（对应4G的GBR/MBR）；Non-GBR通路：Session-AMBR（对应4G的APN-AMBR）+ UE-AMBR | GBR/MBR更名为GFBR/MFBR；APN-AMBR更名/细化为Session-AMBR |
| **怎样算好** | 5QI | 业务质量索引，代表资源类型、优先级、可靠性、丢包率等参数取值集合 | 对应4G QCI；新增delay critical GBR资源类型；新增标准5QI值69、79~85；新增Averaging Window和Maximum Data Burst Volume参数 |

#### 5G新增可选参数

| 参数 | 适用对象 | 作用 | 应用场景 |
|------|----------|------|----------|
| RQA | Non-GBR QoS Flow | 指示支持Reflective QoS机制 | 网页浏览等不需保证带宽的业务 |
| Notification Control | GBR QoS Flow | RAN不能满足GFBR时通知SMF，发起QoS Flow修改/移除；条件改善后恢复 | 可根据QoS变化改变速率的业务，如视频直播 |
| Maximum Packet Loss Rate | GBR QoS Flow | 指示空口侧能接受的最大丢包率 | 仅用于语音业务 |

**证据**: QoS_42268275.md — QoS参数章节

### K-011: QoS控制机制

PCF/SMF是QoS控制的关键NF：PCF拥有QoS最终决策权，SMF根据PCF指示进行QoS Flow控制，UPF/RAN/UE执行用户面动作。

#### 两种控制机制

| 控制类型 | 适用对象 | 原理 |
|----------|----------|------|
| **信令控制** | 所有QoS Flow | SMF在PDU会话建立/修改/用户面激活时向UPF/RAN/UE发送消息，携带QoS Flow信息（QFI、PFS、QoS参数等） |
| **Reflective QoS** | Non-GBR QoS Flow | UE根据下行数据包自行推演上行QoS Rule，无需信令通知。减少新应用部署时的PDU会话修改信令 |

#### QoS参数决策流程

- **默认QoS Flow**：SMF从UDM获取QoS → 根据本地配置得到协商值 → 上报PCF。PCF可覆盖（以PCF下发为准）。
- **专有QoS Flow**：PCF直接下发QoS参数（网络侧发起），或UE请求但最终以PCF下发为准（UE发起）。

#### Reflective QoS工作机制

1. UE在注册/会话建立时上报Reflective QoS能力。
2. SMF确定激活时，发送包含RQI的SDF QoS控制信息给UPF，发送包含RQA的QoS profile给AN。
3. UPF在下行数据包隧道包头打上RQI标记。
4. AN将RQI和QFI通过空口传给UE。
5. UE根据收到的数据生成上行QoS Rule，启动RQ定时器。定时器超时则删除推导出的QoS Rule。

**证据**: QoS_42268275.md — QoS如何控制章节

### K-012: PCC静态规则结语——带宽精细化管理的方向

随着人们对视频清晰度、流畅度要求的不断提高，网络拥塞的可能性不断增加。在带宽未满足畅所欲用时，如何通过精细化管理，实现用户、内容提供商、运营商的"皆大欢喜"，是必须考虑的问题。

业务感知在实际应用中看似简单（免流量、重定向、超额限速），但随着5G普及、流量越来越便宜，如何精细化"经营"流量的话题将越来越受关注。技术应用于网络发展相互促进。

**证据**: 结语_33031965.md（PCC静态规则系列）、结语_32789113.md（SA系列）

## 3. 配置调测要点

### SA识别配置要素

规划业务感知规则需要确定业务的逻辑和目的：

1. **确定过滤逻辑**：对什么样的用户（用户类型）浏览的什么内容（过滤条件）需要启用何种类型（策略类型和具体策略）的内容感知。
2. **配置过滤条件**：根据业务需要配置三四层过滤（IP、端口）、协议过滤（HTTP、RTSP等）、七层过滤（URL、method等），组合为流过滤器。
3. **设定策略**：根据业务目标选择PCC（计费）、BWM（带宽控制）、HEADEN（头增强）、WEBPROXY、SMARTREDIRECT等策略类型。
4. **设定优先级**：不同业务的过滤条件可能有交集，需要不同的业务优先级以区分命中交集时属于哪个业务。

### QoS参数配置要素

| 配置层 | 参数 | 决策来源 |
|--------|------|----------|
| QoS Flow级 | 5QI | PCF决策（默认Flow可SMF本地决策，PCF覆盖） |
| QoS Flow级 | ARP（优先级/抢占/被抢占） | PCF决策 |
| GBR Flow | GFBR/MFBR | PCF直接下发 |
| GBR Flow | Notification Control | PCF决策 |
| Non-GBR Flow | RQA（是否启用Reflective QoS） | SMF发起 |
| 会话级 | Session-AMBR | SMF从UDM获取，PCF可覆盖 |
| UE级 | UE-AMBR | AMF从UDM获取，发送给RAN |

### 典型套餐规则规划示例

以文中的业务套餐为例，规则规划映射：

| 业务编号 | 过滤条件 | 策略类型 | 策略 | 优先级 |
|----------|----------|----------|------|--------|
| 1 | www.huawei.com | PCC | 1元/GB | 中 |
| 2 | RTSP协议+IP地址段 | PCC | 免费 | 高 |
| 3 | www.example.com | PCC | 阻塞 | 最高 |
| 4 | 任意数据流 | PCC | 2元/GB | 低 |
| 5 | 任意数据流 | PCC | 免费 | 最低 |
| 6 | 任意数据流 | BWM | 限速1Mbps | 一般 |
| 7 | www.vmall.com | HEADEN | 插入MSISDN号 | 一般 |

注意：业务5的"套餐耗尽"和业务6的"流量超50G"条件属于PCF是否下发规则的判断条件，需在PCF上配置，不在规则本身体现。

## 4. 与带宽控制的关系

### SA是带宽控制场景的入口

业务感知是整个策略控制的入口机制：

- **识别层**：通过三四层解析、协议识别、七层解析，UPF识别出用户报文属于什么业务。
- **匹配层**：将识别出的报文特征与规则库中的BWM规则（及其他类型规则）匹配。
- **执行层**：对成功匹配的报文执行带宽控制策略（限速、分类、调度等）。

没有SA的识别和匹配能力，带宽控制策略就无法精确地作用于特定业务。

### 三大套餐与带宽控制场景的直接对应

业务感知的三大应用方向与带宽控制场景中的三大套餐直接对应：

| 套餐 | SA应用方向 | 核心策略类型 | 与带宽控制的关系 |
|------|-----------|-------------|----------------|
| 套餐1 计费场景 | 计费精细化 | PCC | 间接：通过计费规则控制用户行为 |
| **套餐2 带宽控制场景** | **带宽控制** | **BWM** | **直接：BWM规则绑定带宽管理分类策略，实现限速、FUP、差异化调度** |
| 套餐3 访问限制场景 | 安全防护 | PCC（阻塞/重定向）+ SMARTREDIRECT | 间接：控制访问行为而非带宽 |

### QoS是带宽控制的执行框架

QoS Flow为带宽控制提供了执行承载：

- **BWM策略结果→QoS Flow映射**：BWM规则匹配后产生的分类结果，最终需要映射到具有不同QoS参数的QoS Flow上执行。
- **GBR保障**：对需要保证带宽的业务（如视频直播），通过GBR QoS Flow的GFBR参数确保最低带宽。
- **Non-GBR限速**：对需要限速的业务（如FUP超额限速），通过Non-GBR QoS Flow的Session-AMBR参数或BWM限速策略实现。
- **差异化调度**：不同QoS Flow的5QI和ARP参数决定了空口调度的优先级和资源保障程度。

### 完整闭环

```
SA识别业务 → 规则匹配(BWM/PCC/HEADEN) → 策略执行(分类/限速) → QoS Flow承载(5QI/ARP/xBR) → 空口调度保障
```

这个闭环正是带宽控制场景中"识别→决策→执行→保障"的技术基础。

| **BWM** | Bandwidth Management，带宽管理 |
| **HEADEN** | Head Enhancement，头增强 |
| **WEBPROXY** | Web Proxy，Web代理 |
| **SMARTREDIRECT** | 智能重定向 |
| **PCF** | Policy Control Function，策略控制功能 |
| **SMF** | Session Management Function，会话管理功能 |
| **UPF** | User Plane Function，用户面功能 |
| **QoS** | Quality of Service，服务质量 |
| **QoS Flow** | 5G数据通路，端到端QoS控制最小粒度 |
| **QFI** | QoS Flow ID，QoS Flow标识 |
| **5QI** | 5G QoS Identifier，5G QoS索引（对应4G QCI） |
| **ARP** | Allocation and Retention Priority，分配保留优先级 |
| **GBR** | Guaranteed Bit Rate，保证比特率 |
| **Non-GBR** | 非保证比特率 |
| **GFBR** | Guaranteed Flow Bit Rate，保证流比特率（对应4G GBR） |
| **MFBR** | Maximum Flow Bit Rate，最大流比特率（对应4G MBR） |

| **PFS** | Packet Filter Set，包过滤器集（对应4G PF） |

| **QoS Rule** | UE侧的QoS映射规则（对应4G UL TFT） |
| **TFT** | Traffic Flow Template，业务流模板（4G的QoS映射工具） |
| **NG-U Tunnel** | N3接口隧道，一个PDU会话共用一个 |
| **DRB** | Data Radio Bearer，数据无线承载 |
| **eMBB** | Enhanced Mobile Broadband，增强移动宽带 |
| **uRLLC** | Ultra-Reliable Low Latency Communications，超可靠低时延通信 |
| **mMTC** | Massive Machine Type Communications，海量机器类通信 |
| **MSISDN** | Mobile Station International Subscriber Directory Number，移动用户国际号码 |
| **RQI** | Reflective QoS Indication，反射QoS指示 |
