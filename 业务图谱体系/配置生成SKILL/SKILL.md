# 配置生成 SOP

> **通用配置生成 SKILL**：本 SOP 与业务域/子场景解耦，业务域和子场景作为数据动态加载。
> ★**设计原则**：SKILL.md 只承载 SOP 流程，不绑定任何具体业务域/子场景。业务域、子场景的扩展只更新 `common/业务域注册表.md` 和 `knowledge/{业务域}/{子场景}/` 目录，**本文件永不修改**。
> ★**面向目标态**：知识当前以文件形式加载（`knowledge/{业务域}/{子场景}/...`），将来迁移到后端 API 接口时，只需把加载入口从路径换为接口调用，SOP 主体不动。

---

## 1. 定位

Agent 根据用户的需求描述和现网配置脚本，经过**业务域与子场景识别、需求理解、图谱匹配、参数收集、配置生成与全量核查**，输出可直接执行的 MML 配置脚本。

本 SOP 是**业务域无关**的通用配置生成工作流：
- **业务域**（如业务感知、APN 接入、…）和**子场景**（如计费、带宽控制、访问限制、…）由 `common/业务域注册表.md` 动态定义，本 SOP 不写死任何域/场景
- 新增业务域或子场景 = 注册表加一行 + 建对应知识目录，**本 SOP 不改**

## 3. 整体流程

```
Phase 0a: 业务域识别（读业务域注册表，关键词路由到 {业务域}）
Phase 0b: 子场景识别（在域内，关键词路由到 {子场景}）
  输出: 锁定 {业务域}/{子场景}，加载对应知识模块
  ↓
Phase 1: 需求理解与图谱匹配
  输入: 需求 + 现网配置脚本路径(必须)
  输出: 方案上下文 + 决策点答案 + 现网对象清单
  阻塞: 未提供现网脚本 → 停止要求提供
  ↓
Phase 2: 方案确认【GATE-1 STOP】→ 产出 xxx方案.md
  ↓
Phase 3: 参数收集 → 产出 xxx规划数据表.md
  ↓
Phase 4: 规划数据确认【GATE-2 STOP】
  ↓
Phase 5: 配置生成（图谱规则作内置约束）→ 每网元 MML 文件
  ↓
Phase 6: AI MML 核查（直接调 coremaster configcheck 接口）
  ↓
Phase 6.5: 核查确认【GATE-3 STOP】
  ↓
Phase 7: 输出交付
```

### 3.1 用户交互规则

专业平实、表格展示、追问说明原因与影响；3 个 GATE 必须 STOP 等用户确认。用户中途可"暂停/重新开始/换业务域/换子场景/到此为止"。

## 4. Phase 0: 业务域与子场景识别

### 4.0 强制加载

```
必须读取：
1. knowledge/common/业务域注册表.md   — 全部业务域+子场景注册数据（路由依据）
```

### 4.1 Phase 0a 业务域识别

1. 从用户需求提取关键词，按注册表的"业务域识别关键词"匹配业务域
2. 多域命中按主诉求判断；不确定则向用户确认
3. 未命中 → 提示用户补充属于哪个业务域
4. 锁定 `{业务域}`

### 4.2 Phase 0b 子场景识别

1. 在锁定的业务域内，按该域下子场景的"子场景识别关键词"匹配
2. 多场景命中按主诉求判断（如某域下"计费+限速"按主诉求定）
3. 未命中 → 提示用户补充是该域下的哪个子场景
4. 锁定 `{子场景}`

### 4.3 输出

```
已识别：业务域={业务域名}，子场景={子场景名}
将加载知识模块：knowledge/{业务域}/{子场景}/
请提供现网配置脚本以进入需求分析（Phase 1）。
```

## 5. Phase 1: 需求理解与图谱匹配

⚠ 完成后必须 STOP 等用户在 Phase 2 确认。禁止在本阶段生成任何 MML 命令。

### 5.0 强制加载（Phase 0 锁定 `{业务域}/{子场景}` 后）

> ⚠ **执行顺序硬约束**：必须**先完成本节全部图谱加载，再进行 §5.1 现网脚本处理**。
> **禁止先搜索/Read 现网脚本再加载图谱**——不加载图谱就不知道该解析哪些命令、对象如何归类、方案如何匹配，会导致盲目搜索和错误归类。
> 顺序：Phase 0 识别子场景 → **§5.0 加载图谱** → §5.1 现网脚本存在性验证 → §5.2 现网解析（Bash grep）。

```
必须读取：
1. knowledge/{业务域}/{子场景}/three-layer-graph/00-overview.md   — 子场景全景
2. knowledge/{业务域}/{子场景}/three-layer-graph/01-business-graph.md — 场景/方案/决策点/规则
3. knowledge/{业务域}/{子场景}/three-layer-graph/02-feature-graph.md — 特性依赖/License/特性规则（理解方案能力组成）
4. knowledge/{业务域}/{子场景}/three-layer-graph/04-command-graph.md — 命令/参数/对象（现网解析+差异分析必需）
5. knowledge/{业务域}/{子场景}/ref-phase/ref-phase1-*.md          — 现网解析参考（如有）
6. knowledge/common/通用配置规则.md                               — 通用约束

按需：
7. knowledge/{业务域}/{子场景}/three-layer-graph/05-cross-layer-mapping.md — 跨层映射（差异分析）
8. knowledge/{业务域}/{子场景}/three-layer-graph/03-task-layer.md — 任务编排（任务清单引用时）
```

### 5.1 前置条件检查

- **【GATE-A】现网脚本路径**：未提供 → STOP 要求提供相关网元侧配置导出文件
- **【GATE-B】脚本存在性**：用 **Glob 或 Bash `ls`** 验证每个文件存在且可读，**严禁对现网脚本用 Read 工具**（数万至数十万行，Read 会溢出且无意义），否则 STOP

### 5.2 步骤

1. **场景识别**：匹配业务图谱的 NetworkScenario
2. **网元与决策点解析**：提取该子场景的 DecisionPoint 答案，缺失必须追问，追问时引用图谱选项含义
3. **现网配置解析**：**必须用 Bash 工具调 `grep` 命令**按命令类型分批搜索现网脚本（**每次只搜一个命令类型**，严禁合并；**严禁对现网脚本用 Read 工具**——数万至数十万行，Read 必溢出）。
   - ⚠ **用 Bash grep，不用 Grep 工具**：现网脚本是用户提供的大配置文件（.txt/.cfg/.mml），用 **Bash 工具执行 grep 命令**最直接可控（如 `grep -E '^(ADD|SET|MOD|RMV) +URR' 现网脚本.txt`，支持 -c/-A/-B/输出重定向）；Grep 工具基于 ripgrep、为搜工作区代码文件设计，不适合现网大配置文件。
   - 命令清单见该子场景 `04-command-graph.md` 的 MMLCommand 表（§5.0 已加载，据其确定要 grep 搜哪些命令）。分网元侧提取对象
4. **差异分析**：对每个 ConfigObject 判断 复用/修改/新建/删除

> 现网脚本数万至数十万行，严禁 Read 全量读取，必须用 Bash grep 按命令类型分批搜索（通用配置规则 §2）。

### 5.3 输出

```markdown
## 匹配结果
**业务域/子场景**: {域名}/{场景名}
**方案**: {CS-xx 方案名}
**决策点**: {该场景 DP 答案}
**现网分析**: {现有对象清单 + 可复用对象}
**任务编排**: {按序 Task，引用 03-task-layer.md ConfigTask ID}
```

## 6. Phase 2: 方案确认【GATE-1】

**【GATE-1】方案确认审核点。**

展示 Phase 1 匹配结果，**产出 `xxx方案.md`**（方案文件，便于追溯）。展示后 STOP：

```
请确认以上匹配结果（已生成 xxx方案.md）。确认后进入参数收集。
```

- 要求修正 → 回 Phase 1 重新匹配
- 换方案 → 展示其他 CS-xx
- **用户确认前不得执行 Phase 3，不得生成任何配置命令。**

## 7. Phase 3: 参数收集 → 规划数据表

⚠ 必须 GATE-1 通过。完成后 STOP 等 Phase 4。禁止生成 MML。

### 7.0 强制加载

```
必须读取：
1. knowledge/{业务域}/{子场景}/three-layer-graph/04-command-graph.md — CommandParameter 定义
2. knowledge/{业务域}/{子场景}/ref-phase/ref-phase3-*.md            — 参数推导规则
3. knowledge/{业务域}/{子场景}/kb/（按需章节）                        — 参数语义/特殊规则
按需（多网元协同时）：
4. 该场景跨网元协同约束知识
```

### 7.1 步骤

1. **推断参数**：从需求、现网命名规律、图谱方案示例、kb 默认值推断；枚举值必须从 `04-command-graph.md` CommandParameter 获取
2. **产出 `xxx规划数据表.md`**（结构化，按网元 × 命令 × 参数）：

```markdown
## 规划数据表 — {方案名}
### 网元: {网元类型1}
| 命令 | 参数 | 值 | 来源 | 状态 |
|------|------|-----|------|------|
| ADD xxx | param | value | 推断/现网/用户提供 | 已知/待确认/待提供 |
### 网元: {网元类型2}
| ... |
### 全局参数（多网元共用，必须一致）
| 参数 | 网元1侧 | 网元2侧 | 来源 | 状态 |
```

3. **优先级分析**（如该场景涉及有序规则/对象，必须独立执行并用户确认）：
   - 是否涉及优先级、以及优先级机制（如业务感知的 RULE.PRIORITY、带宽的 BWMRULEPRI 等）由该场景 `04-command-graph.md` / 域规则定义
   - 涉及时：用 Bash grep 提取现网相关优先级值，排序，将用户"第N优先"映射到位置，**用户确认后才能生成带优先级的命令**
   - 通用原则：优先级数值的语义（越小越高 / 越大越高）由场景规则明确，禁凭记忆

### 7.2 输出

展示规划数据表（标注"待提供/待确认"）+ 优先级分析，要求用户补充。

## 9. Phase 5: 配置生成（图谱规则作内置约束）

⚠ 必须 GATE-2 通过。无例外。

### 9.0 强制加载

```
必须读取：
1. knowledge/{业务域}/{子场景}/three-layer-graph/04-command-graph.md — MMLCommand+CommandParameter+CommandRule
2. knowledge/{业务域}/{子场景}/three-layer-graph/03-task-layer.md   — 任务编排/TaskCommandOrderEdge(排序依据)+TaskRule
3. knowledge/{业务域}/{子场景}/three-layer-graph/01-business-graph.md — BusinessRule(生成内置约束)
4. knowledge/{业务域}/{子场景}/ref-phase/ref-phase5-*.md            — 命令模板/排序规则
5. knowledge/{业务域}/{子场景}/kb/（按需章节）                       — 场景特殊配置规则
6. knowledge/common/通用配置规则.md                                  — 通用约束
```

### 9.1 步骤

1. **按规划数据表生成命令**：每网元一个 MML 文件，参数值从规划数据表取
2. **按依赖顺序排列**：具体顺序由该子场景 `03-task-layer.md` 的 TaskCommandOrderEdge 定义（通用原则：底层对象先于上层、被引用对象先于引用者、绑定/刷新等收尾动作最后；具体链路由场景图谱决定，不由本 SOP 写死）
3. **内置图谱规则约束**（生成时即遵守）：BusinessRule(`01`) + CommandRule(`04`) + TaskRule(`03`) + 通用配置规则
4. **场景特有配置规则**：加载 `ref-phase5` 和 `kb` 对应章节（该子场景的专属配置逻辑）

### 9.2 输出

每网元 MML 文件 + 询问是否核查：
```
配置文件已生成。是否调用 AI MML 核查能力验证脚本正确性？
```

## 13. Agent 执行约束

1. **禁止跳过任何 Phase**：Phase 0-7 严格顺序，禁止跳过 GATE-1/2/3
2. **GATE 必须 STOP**：Phase 2/4/6.5 展示后立即停止，等用户下一条消息；确认前不得执行下一 Phase
3. **现网配置为必须**：未验证现网脚本存在前不得进入 Phase 1 业务逻辑
4. **现网脚本处理三约束**：① **先图谱后现网**——Phase 1 必须先完成 §5.0 图谱加载才能处理现网脚本，禁止先搜现网再加载图谱；② **严禁 Read，用 Bash grep**——现网脚本数万至数十万行，必须用 **Bash 工具调 grep 命令**分批搜索（**不用 Grep 工具**、不用 Read），存在性验证只用 Glob/ls；③ **分批**——每次 grep 只查一个命令类型，严禁合并
5. **严格依赖顺序**：按各场景 `03-task-layer.md` 的 TaskCommandOrderEdge 排列命令（底层对象先于上层、被引用对象先于引用者；具体顺序由场景图谱定义）
6. **追问而非猜测**：缺失决策信息或必填参数必须追问
7. **动网保守**：现网对象修改必须有用户确认
8. **知识库优先**：场景 kb 特殊规则 > 图谱约束 > 通用规则
9. **命名遵循现网**：新对象命名沿用现网规律
10. **跨网元一致**：多网元共用参数必须一致
11. **图谱为业务准确性保障**：业务事实从图谱/知识库动态加载，禁止凭记忆填写
12. **强制文件加载**：每 Phase 标注的必须读取文件，业务逻辑前先 Read
13. **优先级必须用户确认**：若场景涉及有序规则（如业务感知 RULE.PRIORITY / 带宽 BWMRULEPRI），提取现网相关优先级值分析后 STOP 确认；优先级数值语义由场景规则定义，禁凭记忆
14. **业务域/子场景隔离**：跨域/跨场景组合需求，主路由+次要作协同约束；配置分场景输出
15. **参数枚举禁凭记忆**：必须从 `04-command-graph.md` CommandParameter 获取
