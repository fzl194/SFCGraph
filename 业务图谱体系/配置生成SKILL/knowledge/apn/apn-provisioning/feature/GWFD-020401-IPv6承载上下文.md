## 1. 概述

### 1.1 特性定义（是什么）

UDG 支持承载于骨干网的 IPv6 PDP上下文 / EPS承载 / PDU Session / QoS Flow 的**激活、去激活和更新**。UDG 能够为用户创建 IPv6 类型的 PDP上下文 / EPS承载 / PDU Session / QoS Flow，IPv6 类型的数据包与 IPv4 类型的数据包一样，在接口上透明传输；目前 UDG 支持基本的 IPv6 业务接入功能：**支持用户面 IPv6 承载**。

> 来源：`output/UDG_Product_Documentation_CH_20.15.2/特性部署/特性指南/UDG特性指南/IPv6功能/GWFD-020401 IPv6承载上下文/GWFD-020401 IPv6承载上下文特性概述_38276175.md`（"定义"、"原理概述"章节）

## 2. 核心概念

### 2.1 关键术语

| 术语 | 全称 | 说明 |
|------|------|------|
| IPv6承载上下文 | IPv6 Bearer Context | IPv6 类型的 PDP上下文/EPS承载/PDU Session/QoS Flow，承载于骨干网 |
| PDP Type | Packet Data Protocol Type | 上下文协议类型，取值 IPv4 / IPv6 / IPv4v6（本特性聚焦 IPv6，与 GWFD-020403 聚焦 IPv4v6 互补） |
| PDP上下文 | PDP Context | 2G/3G 分组数据上下文（IPv6 承载在此层创建） |
| EPS承载 | EPS Bearer | 4G 演进分组系统承载（IPv6 承载在此层创建） |
| PDU Session | 协议数据单元会话 | 5G PDU会话（IPv6 承载在此层创建） |
| QoS Flow | 服务质量流 | 5G QoS Flow（IPv6 承载在此层创建） |
| Router Solicitation (RS) | 路由器请求 | UE 发往 UDG 的请求消息，用于请求 IPv6 路由信息（可选，UDG 不依赖） |
| Router Advertisement (RA) | 路由器通告 | UDG 发往 UE 的通告消息，给 IPv6 用户通告路由、IP 地址前缀等信息（★与 GWFD-020403 共享此机制） |
| WLR | White Label Route / UDG 内部用户路由 | UDG 内部生成的用户下行路由（WLR_SP/WLR_UD），需通过 OSPFv3 IMPORTROUTE 发布到骨干网 |
| OSPFv3 | OSPF version 3 | IPv6 专用的 OSPF 路由协议（与 IPv4 侧 OSPF 对称，用于 IPv6 下行路由发布） |
| 地址族类型 | Address Family Type | VPN 实例的地址族，IPv6 场景必须配置 AFTYPE=ipv6uni |

### 2.2 对象模型（IPv6承载上下文配置体系）

IPv6 承载上下文的配置架构基于"承载建立 + IPv6地址族激活 + IPv6下行路由发布"的三层模型：

```
                      ┌──────────────────────────────┐
                      │   License 使能层（前置开关）   │
                      │   SET LICENSESWITCH           │
                      │   LICITEM=LKV3G5V6PB01        │
                      └───────────────┬──────────────┘
                                      │
                                      ▼
        ┌─────────────────────────────────────────────────┐
        │  Layer 1: IPv6 承载建立（依赖地址分配基础设施）   │
        │  ─ 由 GWFD-010105 本地池或其他方式分配 IPv6 地址 │
        │  ─ UDG 创建 IPv6 类型的 PDP上下文/EPS承载/PDU会话 │
        │  ─ RA 通告机制（与 GWFD-020403 共享）            │
        └─────────────────────┬───────────────────────────┘
                              │
                ┌─────────────┴─────────────┐
                ▼                           ▼
   ┌────────────────────────┐   ┌────────────────────────┐
   │ Layer 2a: IPv6 地址族   │   │ Layer 2b: 路由策略      │
   │ ADD VPNINST (逻辑口VPN) │   │ ADD ROUTEPOLICY         │
   │ ADD L3VPNINST           │   │ ADD ROUTEPOLICYNODE     │
   │ ADD VPNINSTAF           │   │   POLICYNAME=policy1    │
   │   AFTYPE=ipv6uni ★      │   │   NODESEQUENCE=10       │
   │ (IPv6地址族激活)         │   │   MATCHMODE=permit      │
   └───────────┬────────────┘   │ ADD MATCHROUTETYPE      │
               │                │   ROUTETYPE=nssaExternal1│
               │                └─────────────┬──────────┘
               │                              │
               └──────────────┬───────────────┘
                              │
                              ▼
              ┌──────────────────────────────┐
              │  Layer 3: IPv6 下行路由发布    │
              │  ADD OSPFV3 (进程)            │
              │    PROCID=100                 │
              │    VRFNAME=VPN_Internet       │
              │    ROUTERID=10.10.10.1        │
              │  ADD OSPFV3AREA               │
              │    AREAID=0.0.0.1             │
              │  ADD OSPFV3IMPORTROUTE ★      │
              │    PROTOCOL=wlr               │
              │    引入 WLR 路由到骨干网        │
              └──────────────────────────────┘
```

核心配置对象说明：

| 对象 | 作用 | IPv6 承载上下文场景角色 |
|------|------|------------|
| VPNINST（逻辑口） | 逻辑口 VPN 实例 | 用户面逻辑接口归属的 VPN，必须与外联口 VPN 一致 |
| L3VPNINST | L3 VPN 实例 | 外联口的 VPN 实例（IPv6 承载的下一条出口 VPN） |
| VPNINSTAF | VPN 实例地址族 | **AFTYPE=ipv6uni 激活 IPv6 地址族（IPv6 承载必备）** |
| ROUTEPOLICY | 路由策略 | 控制 IPv6 用户路由发布到骨干网时的过滤规则 |
| ROUTEPOLICYNODE | 路由策略节点 | 路由策略的匹配节点（permit/deny） |
| MATCHROUTETYPE | 路由类型匹配 | 按路由类型匹配（如 nssaExternal1） |
| OSPFV3 | OSPFv3 进程 | **IPv6 侧下行路由发布主进程（与 IPv4 侧 OSPF 对称）** |
| OSPFV3AREA | OSPFv3 区域 | OSPFv3 区域配置 |
| OSPFV3IMPORTROUTE | OSPFv3 引入路由 | **引入 WLR（用户内部路由）发布到骨干网（IPv6 下行路由可达性关键）** |

### 2.3 在 APN 地址分配场景的角色

GWFD-020401 在 APN 地址分配场景中扮演"**IPv6 承载基础设施层**"的角色，是所有 IPv6 相关地址分配特性的共同前置依赖（详见 §7.1）：

```
┌──────────────────────────────────────────────────────────────┐
│ Layer A: IPv6 承载上下文基础设施层（GWFD-020401）            │
│   License LKV3G5V6PB01                                        │
│   - 支持 IPv6 PDP上下文/EPS承载/PDU Session/QoS Flow 的       │
│     激活、去激活、更新                                          │
│   - 用户面 IPv6 承载（透明转发）                              │
│   - RA 通告机制（与 GWFD-020403 共享描述）                    │
│   - IPv6 下行路由发布（OSPFv3 引入 WLR）                      │
└──────────────────────────┬───────────────────────────────────┘
                           │ 被以下 IPv6 高阶特性依赖
                           ▼
┌──────────────────────────────────────────────────────────────┐
│ Layer B: IPv6 高阶特性（依赖 GWFD-020401）                   │
│   ├─ GWFD-020403 IPv4v6 双栈接入（文档无显式依赖，但语义依赖）│
│   ├─ GWFD-020406 IPv6 Prefix Delegation（★显式依赖，见 §7.2）│
│   └─ GWFD-010105 用户面地址分配（IPv6 地址类型落地，见 §7.3） │
└──────────────────────────────────────────────────────────────┘
```

具体角色：

1. **承载生命周期管理**：GWFD-020401 定义并支持 IPv6 PDP上下文/EPS承载/PDU Session/QoS Flow 的**激活、去激活、更新**三个生命周期动作。GWFD-010105 等地址分配特性仅负责"分配地址"，不涉及承载本身的建立/释放。
2. **IPv6 透明转发能力**：定义 IPv6 类型的数据包在接口上透明传输的能力（GWFD-010105 文档未单独定义此能力）。
3. **IPv6 下行路由发布**：本特性激活文档以 OSPFv3 + IMPORTROUTE WLR 为核心，专门解决 IPv6 用户下行路由发布到骨干网的问题。这是 GWFD-010105 的"用户下行路由生成与发布"在 IPv6 侧的具体实现。
4. **共享 RA 机制**：本特性的 RA 通告机制描述与 GWFD-020403 高度一致（同一文档模板），表明 IPv6 单栈与 IPv4v6 双栈共享相同的 RA 通告基础设施。

## 3. 原理与流程

### 3.1 实现原理：IPv6 承载建立与RA通告

#### 3.1.1 IPv6 承载上下文创建机制

承载建立完成后，UDG 能够为用户创建 **IPv6 类型**的 PDP上下文 / EPS承载 / PDU Session / QoS Flow。在 IPv6 承载上下文中：

- IPv6 类型的数据包与 IPv4 类型的数据包**一样，在接口上透明传输**
- UDG 可对上下行的 IPv6 报文进行转发

> 文档原文：UDG 能够为用户创建IPv6类型的 PDP上下文/EPS承载 /PDU Session/QoS Flow，IPv6类型的数据包与IPv4类型的数据包一样，在接口上透明传输。目前，UDG 支持基本的IPv6业务接入功能：支持用户面IPv6承载。

> 来源：`GWFD-020401 IPv6承载上下文特性概述_38276175.md`（"原理概述"章节）

#### 3.1.2 IPv6 地址通告机制（Router Advertisement）

承载建立完成后，UE 可能会向 UDG 发送 **Router Solicitation** 消息，获取 IPv6 地址信息。关键行为：

- **无论是否收到 Router Solicitation 消息**，只要用户的 **PDP Type** 是 **IPv6** 或者 **IPv4v6**，UDG 都会向 UE 发送 **Router Advertisement** 消息
- RA 消息用于给 IPv6 用户通告路由、IP 地址前缀等信息

> 文档原文：承载建立完成后，UE可能会向 UDG 发送Router Solicitation消息，获取IPv6地址信息。无论是否收到Router Solicitation消息，只要用户的PDP Type是IPv6或者IPv4v6，UDG都会向UE发送Router Advertisement消息，给IPv6用户通告路由、IP地址前缀等信息。

> ★ 关键观察：此段 RA 机制描述与 GWFD-020403 §3.1.2 完全一致（逐字相同），表明 GWFD-020401 与 GWFD-020403 共享同一套 RA 通告基础设施。

> 来源：`GWFD-020401 IPv6承载上下文特性概述_38276175.md`（"原理概述"章节）

#### 3.1.3 IPv6 下行路由发布机制（OSPFv3 + WLR）

IPv6 用户接入后，UDG 内部生成用户下行路由（WLR，含 WLR_SP 和 WLR_UD 两类）。本特性通过 OSPFv3 的 IMPORTROUTE 功能将 WLR 路由发布到骨干网，使骨干网侧设备能够将下行报文路由到正确的 UDG。

关键约束：

- **采用路由策略方式发布路由时，不能只选择发布 WLR_SP 或者 WLR_UD 路由，必须选择 WLR 方式发布所有路由**（文档操作步骤9"说明"原文）

> 来源：`激活IPv6承载上下文_38276176.md`（"操作步骤"步骤9"说明"）

### 3.2 业务流程（IPv6 单栈端到端接入）

基于 GWFD-020401 文档（承载建立、RA 通告、路由发布）与 GWFD-010105（地址池分配）综合，IPv6 单栈用户接入流程如下：

```
┌─────────────────────────────────────────────────────────────────┐
│ 1. 承载建立（PDU会话/PDP上下文建立请求携带 PDP Type = IPv6）      │
│    UE 通过 SMF/SGSN/MME 发起会话建立，请求 IPv6 接入              │
└──────────────────────────────┬──────────────────────────────────┘
                               │
                               ▼
┌─────────────────────────────────────────────────────────────────┐
│ 2. C面决策 + PFCP 地址类型协商                                    │
│    SGW-C/PGW-C/SMF 通过 PFCP 指示 UDG 分配 IPv6（CHV6=1）         │
│    （详见 GWFD-010105 §3.4 真值表 IPv6 行）                       │
└──────────────────────────────┬──────────────────────────────────┘
                               │
                               ▼
┌─────────────────────────────────────────────────────────────────┐
│ 3. UDG IPv6 地址分配（依赖 GWFD-010105 地址池体系）               │
│    UDG 根据 APN→POOLGROUP 映射，从 IPv6 池分配一个 IPv6 地址      │
└──────────────────────────────┬──────────────────────────────────┘
                               │
                               ▼
┌─────────────────────────────────────────────────────────────────┐
│ 4. IPv6 路由通告（RA 机制，★本特性核心）                         │
│    无论是否收到 UE 的 RS，只要 PDP Type = IPv6：                  │
│    UDG 主动向 UE 发送 Router Advertisement 消息                  │
│    通告 IPv6 路由、IP 地址前缀                                    │
└──────────────────────────────┬──────────────────────────────────┘
                               │
                               ▼
┌─────────────────────────────────────────────────────────────────┐
│ 5. IPv6 承载上下文建立完成                                        │
│    UDG 创建 IPv6 类型的 PDP上下文/EPS承载/PDU Session/QoS Flow    │
│    （GWFD-020401 核心：激活、去激活、更新生命周期）               │
└──────────────────────────────┬──────────────────────────────────┘
                               │
                               ▼
┌─────────────────────────────────────────────────────────────────┐
│ 6. IPv6 下行路由发布（OSPFv3 引入 WLR，★本特性激活文档核心）     │
│    UDG 内部生成 WLR 用户路由                                     │
│    通过 ADD OSPFV3IMPORTROUTE PROTOCOL=wlr 发布到骨干网          │
│    骨干网侧设备可路由下行 IPv6 报文到本 UDG                       │
└──────────────────────────────┬──────────────────────────────────┘
                               │
                               ▼
┌─────────────────────────────────────────────────────────────────┐
│ 7. IPv6 数据双向传输                                              │
│    UE 后续可进行 IPv6 报文的收发                                  │
│    UDG 在接口上透明传输 IPv6 报文至 PDN/DN                        │
└─────────────────────────────────────────────────────────────────┘
```

> 说明：步骤2/3 的 PFCP 信元协商与地址池分配细节见 GWFD-010105；步骤4 的 RA 机制是 GWFD-020401 与 GWFD-020403 共享描述；步骤5/6 是 GWFD-020401 的独有贡献（承载生命周期管理 + IPv6 下行路由发布）。

> 来源：`GWFD-020401 IPv6承载上下文特性概述_38276175.md`（"原理概述"章节）；`激活IPv6承载上下文_38276176.md`（"操作步骤"）

### 3.3 协议交互

| 接口 | 交互网元 | 消息类型 | 说明 |
|------|---------|---------|------|
| N4（5G）/ Sxa / Sxb（4G） | SMF/PGW-C ↔ UPF/PGW-U | PFCP Session Establishment Request/Response | C面指示 IPv6 分配（CHV6=1）；U面回传分配的 IPv6 地址 |
| Uu / 空口（用户面） | UDG → UE | **Router Advertisement** | UDG 向 IPv6 用户通告 IPv6 路由与前缀（★本特性核心机制，与 GWFD-020403 共享） |
| Uu / 空口（用户面） | UE → UDG | Router Solicitation（可选） | UE 请求 IPv6 路由信息（UDG 不依赖此消息也会主动发 RA） |
| Gi/SGi/N6 | UPF/PGW-U ↔ DN | IPv6 报文透明传输 | IPv6 数据转发 |
| 骨干网（OSPFv3） | UDG ↔ 骨干网路由器 | OSPFv3 LSA（引入 WLR 路由） | **IPv6 用户下行路由发布到骨干网（GWFD-020401 独有）** |

> 来源：`GWFD-020401 IPv6承载上下文特性概述_38276175.md`（"原理概述"章节）；`激活IPv6承载上下文_38276176.md`（OSPFv3 引入 WLR 流程）

## 4. 配置规则

### 4.1 激活步骤

IPv6 承载上下文特性的激活和配置流程：

```
步骤1：打开本特性的 License 配置开关
  └── SET LICENSESWITCH:LICITEM="LKV3G5V6PB01",SWITCH=ENABLE;

步骤2：配置逻辑口的 VPN 实例
  └── ADD VPNINST:VPNINSTANCE="VPN_Internet"
      （★逻辑接口的VPN一定要与其对应的外联口的VPN相同）

步骤3：配置外联口的 VPN 实例（含 IPv6 地址族）
  ├── ADD L3VPNINST:VRFNAME="VPN_Internet"
  └── ADD VPNINSTAF:VRFNAME="VPN_Internet",AFTYPE=ipv6uni ★

步骤4：配置路由策略
  └── ADD ROUTEPOLICY:POLICYNAME="policy1"

步骤5：添加路由策略节点
  └── ADD ROUTEPOLICYNODE:POLICYNAME="policy1",NODESEQUENCE=10,MATCHMODE=permit

步骤6：添加匹配路由类型
  └── ADD MATCHROUTETYPE:POLICYNAME="policy1",NODESEQUENCE=10,ROUTETYPE=nssaExternal1

步骤7：启动 OSPFv3 进程
  └── ADD OSPFV3:PROCID=100,ROUTERID="10.10.10.1",VRFNAME="VPN_Internet"

步骤8：增加 OSPFv3 区域
  └── ADD OSPFV3AREA:PROCID=100,AREAID="0.0.0.1"

步骤9：引入 WLR 路由，将路由发布到骨干网上
  └── ADD OSPFV3IMPORTROUTE:PROCID=100,TOPOID=0,PROTOCOL=wlr,ROUPOLINAME="policy1"
      （★不能只选 WLR_SP 或 WLR_UD，必须选 WLR 发布所有路由）
```

> 来源：`激活IPv6承载上下文_38276176.md`（"操作步骤"、"任务示例"章节）

## 5. 配置案例

### 5.1 典型场景：配置 MS/UE 下行路由（IPv6 承载上下文）

**场景描述**：IPv6 用户接入网络时，需要在 UDG 上配置手机下行路由，确保用户能访问数据业务。本案例演示完整的 IPv6 承载上下文激活流程（License + VPN + IPv6 地址族 + 路由策略 + OSPFv3 + WLR 引入）。

**配置步骤和 MML 命令序列（原样保留文档脚本）**：

```
//打开本特性的License配置开关。

SET LICENSESWITCH:LICITEM="LKV3G5V6PB01",SWITCH=ENABLE;

//配置VPN实例。

ADD VPNINST:VPNINSTANCE="VPN_Internet";

//配置VPN实例。

ADD L3VPNINST:VRFNAME="VPN_Internet";
ADD VPNINSTAF:VRFNAME="VPN_Internet",AFTYPE=ipv6uni;

//配置路由策略。

ADD ROUTEPOLICY:POLICYNAME="policy1";

//添加路由策略节点。

ADD ROUTEPOLICYNODE:POLICYNAME="policy1",NODESEQUENCE=10,MATCHMODE=permit;

//添加匹配路由类型。

ADD MATCHROUTETYPE:POLICYNAME="policy1",NODESEQUENCE=10,ROUTETYPE=nssaExternal1;

//启动 OSPF v3进程。

ADD OSPFV3:PROCID=100,ROUTERID="10.10.10.1",VRFNAME="VPN_Internet";

//增加 OSPF v3区域。

ADD OSPFV3AREA:PROCID=100,AREAID="0.0.0.1";

//引入WLR路由，将路由发布到骨干网上。

ADD OSPFV3IMPORTROUTE:PROCID=100,TOPOID=0,PROTOCOL=wlr,ROUPOLINAME="policy1";
```

> 来源：`激活IPv6承载上下文_38276176.md`（"任务示例-脚本"章节，原样保留）

**脚本要点解读**：

| 配置片段 | IPv6 承载语义 |
|---------|---------|
| `SET LICENSESWITCH:LICITEM="LKV3G5V6PB01"` | 打开 IPv6 承载上下文能力使能开关（前置必配） |
| `ADD VPNINST:VPNINSTANCE="VPN_Internet"` | 逻辑口 VPN 实例（必须与外联口 VPN 一致） |
| `ADD L3VPNINST + ADD VPNINSTAF AFTYPE=ipv6uni` | 外联口 VPN + **IPv6 地址族激活**（IPv6 承载必备） |
| `ADD ROUTEPOLICY + ROUTEPOLICYNODE + MATCHROUTETYPE` | 三层路由策略配置（nssaExternal1 匹配） |
| `ADD OSPFV3 PROCID=100 ROUTERID=10.10.10.1 VRFNAME=VPN_Internet` | IPv6 侧 OSPFv3 进程（Router ID 全网唯一） |
| `ADD OSPFV3AREA PROCID=100 AREAID=0.0.0.1` | OSPFv3 区域 |
| `ADD OSPFV3IMPORTROUTE PROTOCOL=wlr` | **★ 核心：引入 WLR（用户内部路由）发布到骨干网** |

**★ 与 GWFD-020403 激活脚本的差异（关键对比）**：

| 维度 | GWFD-020401（IPv6 承载上下文） | GWFD-020403（IPv4v6 双栈接入） |
|------|------------------------------|------------------------------|
| License | LKV3G5V6PB01 | LKV3G5VDSA01 |
| 地址池配置 | **不含 ADD POOL/SECTION/POOLGROUP/POOLGRPMAP** | 含完整双池配置 |
| APN 配置 | **不含 ADD APN** | 含 ADD APN（双 VPN） |
| VPN 地址族 | **含 ADD VPNINSTAF AFTYPE=ipv6uni** | 无（由 APN 的 HASVPNIPV6 承担） |
| 路由发布 | **含 OSPFv3 + IMPORTROUTE WLR**（重点） | 无（参考信息列出但激活脚本未含） |
| 核心关注点 | IPv6 承载基础设施 + 下行路由发布 | 双栈能力使能 + 双栈对称配置 |

→ **结论**：GWFD-020401 激活脚本不含地址池与 APN 配置，证实本特性是"承载基础设施"，地址分配由 GWFD-010105 等承担；GWFD-020403 激活脚本不含 OSPFv3 路由发布，证实双栈特性不解决下行路由问题，下行路由由本特性承担。两者职责互补。

### 5.2 场景变体

| 变体 | 核心差异 | 关键参数变化 |
|------|---------|------------|
| IPv6 + IPv4v6 双栈联合部署 | 同时激活 GWFD-020401 + GWFD-020403 | 两个 License 都加载；先激活 020401（承载基础），再激活 020403（双栈使能） |
| IPv6 + IPv6 PD 联合部署 | 同时激活 GWFD-020401 + GWFD-020406 | 两个 License 都加载；GWFD-020406 显式依赖 GWFD-020401（见 §7.2） |
| 不同路由类型匹配 | 路由策略匹配其他类型 | `ADD MATCHROUTETYPE ... ROUTETYPE=<其他类型>`（如 external1、external2） |
| 不同 OSPFv3 区域规划 | 多区域组网 | `ADD OSPFV3AREA ... AREAID=<其他值>` |
| BGP/MPLS IPv6 VPN 替代 OSPFv3 | 下行路由通过 BGP/MPLS VPN 发布 | 见 GWFD-020411 MPLS VPN（BGP/MPLS IPv6 VPN 子特性） |

## 6. 验证与调测

### 6.1 验证方法

#### 6.1.1 调测前提与目的

在部署网络时，需要通过本地 IPv6 地址分配方式给 MS/UE 分配下行路由。此时，需要对 UDG 的 IPv6 承载上下文进行调测。适用于 SGW-U、PGW-U、UPF。

#### 6.1.2 调测数据准备

| 类别 | 参数名称 | 取值样例 | 获取方法 |
|------|---------|---------|---------|
| 用户信息查询 | 用户IMSI（IMSI） | 460000123456789 | 测试终端 |
| 测试APN | APN | apn-test | 已配置数据中获取 |

工具：测试终端、OM Portal

> 该操作无需准备数据。

#### 6.1.3 调测执行步骤

**步骤1**：测试终端使用 "apn-test" APN 接入网络，用户 IP 地址类型为 IPv6。

**步骤2**：执行 `DSP SESSIONINFO` 命令，通过 IMSI 查看测试终端激活后为用户分配的 IPv6 地址。

```
DSP SESSIONINFO:QUERYTYPE=IMSI,IMSI="460000123456789";
```

预期输出（原样保留文档示例）：
```
......

 -------------------------------
                                 IMSI  =  460000123456789
                                     ......              
                             PDP Type  =  IPv6 
                     IPv6 PDP address  =  fc00:1:403:1:0:0:0:0         
                                     ......    
                             APN name  =  apn-test
                                     ......       
 (Number of results = 1)
---    END
```

判断规则：
- 如果 **"IPv6 PDP address"** 与用户激活时携带的 IPv6 地址一致，且 **"PDP Type"** 为 IPv6 → 调测结束
- 如果 "IPv6 PDP address" 与用户激活时携带的 IPv6 地址不一致，或 "PDP Type" 不为 IPv6 → 执行步骤3

**步骤3**：收集信息并寻求技术支持。
- a. 执行 `EXP MML` 命令将当前配置数据导出为 MML 脚本文件并保存
- b. 查看并收集对端设备配置及接口状态信息
- c. 收集归纳所有信息并联系华为技术支持解决

> 来源：`调测IPv6承载上下文_38276177.md`（"操作步骤"章节）

### 6.2 验证要点（IPv6 单栈二元组判断）

与双栈（GWFD-020403 三元组）不同，IPv6 单栈调测需**同时验证两个字段**：

| 验证字段 | 期望值 | 判读 |
|---------|-------|------|
| PDP Type | **IPv6** | 必须为 IPv6（非 IPv4 也非 IPv4v6），证明 IPv6 承载创建成功 |
| IPv6 PDP address | 与用户激活时携带的 IPv6 地址一致 | IPv6 地址在规划段内且与签约/分配一致 |

> ★ 对比：GWFD-020403 验证三元组（PDP Type + IPv4 PDP address + IPv6 PDP address），GWFD-020401 验证二元组（PDP Type + IPv6 PDP address）。两者的 PDP Type 期望值不同（IPv4v6 vs IPv6）。

### 6.3 常见问题与排查

| 问题现象 | 可能原因 | 排查方法 |
|---------|---------|---------|
| 用户激活失败，PDP Type 不为 IPv6 | License 未加载或 SET LICENSESWITCH 未使能 | LST LICENSESWITCH 确认 LKV3G5V6PB01=ENABLE；检查 License 文件加载状态 |
| PDP Type=IPv6 但 IPv6 PDP address 为空或不在段内 | 地址池未配置（GWFD-010105 未激活）；IPv6 VPN 未绑定 | LST POOL / LST SECTION 确认 IPv6 池；LST APN 确认 HASVPNIPV6；确认已完成 GWFD-010105 配置 |
| IPv6 地址分配成功但下行报文不通 | OSPFv3 未启动；WLR 路由未发布 | LST OSPFV3 确认进程状态；LST OSPFV3IMPORTROUTE 确认 PROTOCOL=wlr 已配置 |
| 下行路由只发布部分用户（WLR_SP 或 WLR_UD） | OSPFV3IMPORTROUTE 未使用 wlr 而使用了 wlr_sp/wlr_ud | 修改为 PROTOCOL=wlr 发布所有路由（文档强制约束） |
| 逻辑口报文查询路由失败 | 逻辑口 VPN 与外联口 VPN 不一致 | LST VPNINST 核对 VPNINSTANCE 与 LST L3VPNINST 的 VRFNAME 一致 |
| IPv6 地址族未激活 | 未配置 ADD VPNINSTAF AFTYPE=ipv6uni | LST VPNINSTAF 确认 ipv6uni 地址族存在 |
| OSPFv3 邻居无法建立 | Router ID 冲突 | 检查 ROUTERID 全网唯一性 |
| 用户激活成功但无法访问 IPv6 业务 | RA 通告异常，UE 未获得 IPv6 路由 | 抓包确认 UDG 是否下发 Router Advertisement；检查 OSPFv3 路由发布 |
| CPU 占用率持续上升 | IPv6 承载数过多 | 评估话务模型；扩容 UDG 资源 |
| 弱覆盖场景掉话率上升 | eNodeB 无法下发 RA 给终端 | 参考 UNC 侧 IPv6 前缀代理（GWFD-020406）5GC 业务流程相关排查 |

### 7.2 ★ GWFD-020401 与 GWFD-020406 的关系（文档显式依赖）

**结论：GWFD-020406（IPv6 Prefix Delegation）显式依赖 GWFD-020401（IPv6 承载上下文）。GWFD-020401 是 IPv6 PD 的前置条件。**

#### 7.2.1 文档依据（GWFD-020406 原文）

GWFD-020406 的"与其他特性的交互"章节明确列出：

> | 依赖 | GWFD-020401 IPv6承载上下文 | 82209828 IPv6承载上下文 | UDG 需要建立 IPv6 承载上下文，因此需开启 IPv6 承载上下文特性。 |

> 来源：`GWFD-020406 IPv6 Prefix Delegation特性概述_79370029.md`（"与其他特性的交互"章节）

#### 7.2.2 依赖关系定位

| 维度 | GWFD-020401（IPv6 承载上下文） | GWFD-020406（IPv6 PD） |
|------|------------------------------|----------------------|
| 核心职责 | **承载基础设施**：IPv6 承载的激活/去激活/更新、透明转发、下行路由发布 | **前缀代理**：UE 作为无线移动路由器，将 IPv6 前缀下发给 LAN 侧多终端 |
| License要求 | 82209828 LKV3G5V6PB01 | 82200CKF LKV3G5P6PD01 |
| 发布版本 | UDG 20.0.0 | UDG 20.5.0（晚于 020401） |
| IPv6 前缀长度 | 不涉及（由 010105 地址池决定，64 位为主） | **49~63**（< 64，PD 专用） |
| 文档显式依赖 | 无（向上不显式声明） | **显式依赖 GWFD-020401** |

#### 7.2.3 关系图

```
GWFD-020401 IPv6 承载上下文（License LKV3G5V6PB01，承载基础设施层）
   │
   │  提供 IPv6 承载基础设施：
   │  - IPv6 PDP上下文/EPS承载/PDU Session/QoS Flow 激活/去激活/更新
   │  - 用户面 IPv6 承载（透明转发）
   │  - RA 通告机制
   │  - IPv6 下行路由发布（OSPFv3 + WLR）
   │
   ▼ 被依赖（GWFD-020406 文档显式声明）
   │
GWFD-020406 IPv6 Prefix Delegation（License LKV3G5P6PD01，前缀代理层）
   │
   │  在 020401 基础上叠加：
   │  - UE 作为无线移动路由器
   │  - IPv6 前缀长度 < 64（49~63）
   │  - IPv6D flag + Prefix Delegation Bits field
   │  - 为 LAN 侧多终端下发独立 IPv6 地址前缀
   │
   ▼
完整的 IPv6 移动路由器场景（家庭/小企业多终端接入）
```

#### 7.2.4 实践配置意义

部署 IPv6 Prefix Delegation 时，需**同时激活两个特性**：
1. 先激活 GWFD-020401（License LKV3G5V6PB01 + OSPFv3 路由发布基础设施）
2. 再激活 GWFD-020406（License LKV3G5P6PD01 + PD 专用地址池配置，前缀 < 64）

GWFD-020406 的 PD 能力建立在 GWFD-020401 的 IPv6 承载之上；没有 020401 的承载基础设施，PD 的前缀代理无法落地为真正的用户承载。

### 7.3 ★ GWFD-020401 与 GWFD-020403 / GWFD-010105 的关系（共享 RA 基础设施）

#### 7.3.1 与 GWFD-020403 的关系：IPv6 单栈 vs IPv4v6 双栈

**结论：GWFD-020401（IPv6 承载上下文）与 GWFD-020403（IPv4v6 双栈接入）共享同一套 RA 通告基础设施描述，是 IPv6 单栈与 IPv4v6 双栈的并列特性，非父子关系。**

**文档依据**：

1. **RA 机制描述逐字相同**：GWFD-020401 §原理概述 与 GWFD-020403 §原理概述 的 RA 段落完全一致（"无论是否收到 Router Solicitation 消息，只要用户的 PDP Type 是 IPv6 或者 IPv4v6，UDG 都会向 UE 发送 Router Advertisement 消息..."）。
2. **PDP Type 范围重叠**：两者都覆盖 PDP Type = IPv6 和 IPv4v6。
3. **License 不同**：020401 为 LKV3G5V6PB01，020403 为 LKV3G5VDSA01。
4. **激活脚本互补**：020401 含 OSPFv3 路由发布不含地址池；020403 含地址池不含路由发布。

**实践配置意义**：
- 仅 IPv6 单栈：激活 GWFD-020401 + GWFD-010105（地址池）
- IPv4v6 双栈：激活 GWFD-020401 + GWFD-020403 + GWFD-010105（建议同时激活 020401 提供下行路由发布能力，因 020403 不解决下行路由）

#### 7.3.2 与 GWFD-010105 的关系：承载 vs 地址池

**结论：GWFD-020401（IPv6 承载上下文）与 GWFD-010105（用户面地址分配）是"承载生命周期 / 地址池分配"的协同关系，非父子关系。**

**文档依据**：

1. **GWFD-020401 激活脚本不含 ADD POOL/SECTION/POOLGROUP/POOLGRPMAP**，证明地址分配由其他特性承担。
2. **GWFD-010105 明确支持 IPv6 地址类型**："用户面地址分配是指 UDG 本地为 UE 分配地址，可提供三种类型的 IP 地址：IPv4 地址、IPv6 地址、IPv4v6 双栈地址。"
3. **GWFD-020401 调测依赖 IPv6 地址分配**：DSP SESSIONINFO 显示的 "IPv6 PDP address" 来自 GWFD-010105 的地址池分配。

**职责分工**：

| 职责 | GWFD-020401（承载） | GWFD-010105（地址池） |
|------|---------------------|---------------------|
| IPv6 地址分配 | **不负责**（激活脚本无 ADD POOL） | **负责**（提供 POOL/SECTION 体系） |
| 承载生命周期（激活/去激活/更新） | **负责** | 不涉及 |
| RA 通告机制 | **负责**（描述） | 不涉及 |
| IPv6 下行路由发布（OSPFv3 + WLR） | **负责** | 部分涉及（用户下行路由生成与发布，IPv4 侧为主） |
