## 1. 概述

### 1.1 特性定义（是什么）

**接入控制是 UDG 基于用户协商的带宽使用情况，对用户接入网络进行限制**。即当在线用户已用带宽过大、超出协商的最大带宽时，UDG 根据协商的最大带宽和保证带宽对流进行流控，实现基于用户的带宽控制功能。

控制方式包含两种：
- **CAR（Committed Access Rate）带宽控制**：当上下行数据报文超出用户协商带宽时，报文被丢弃；未超出则转发。
- **Shaping 流量整形**：使用缓存队列（GTS 队列）和令牌桶完成，超带宽报文缓存而非立即丢弃，缓满才丢。

> 来源：`GWFD-010151 接入控制特性概述_69675200.md`（"定义"、"应用场景"、"原理概述"章节）

### 1.2 适用NF（UDG/UNC网元）

| 涉及NF | 网元角色 | 支持版本 | 功能说明 |
|--------|---------|---------|---------|
| SGW-U / PGW-U / UPF | 用户面（UDG） | UDG 20.0.0及后续版本 | 当用户已使用的带宽超出协商的带宽资源时，根据接入限制策略（CAR/Shaping）对带宽进行控制 |

## 2. 核心概念

### 2.1 关键术语

| 术语 | 全称 | 说明 |
|------|------|------|
| 接入控制 | Access Control | UDG 基于用户协商的带宽使用情况，对用户接入网络进行限制（注意：本特性的"接入控制"特指带宽维度，非鉴权维度的接入权限控制） |
| CAR | Committed Access Rate（承诺接入速率） | 一种带宽控制方式：超出协商带宽的报文直接丢弃，未超出则转发 |
| Shaping | Traffic Shaping（流量整形） | 一种带宽控制方式：使用 GTS 队列 + 令牌桶，超带宽报文缓存而非立即丢弃 |
| GTS | General Traffic Shaping | 通用流量整形队列，Shaping 方式的核心数据结构 |
| 令牌桶 | Token Bucket | 按"用户已协商的带宽速率"向桶中放置令牌；报文发送需消耗令牌 |
| 协商带宽 | Negotiated Bandwidth | C 面（SGW-C/PGW-C/SMF）与周边网元协商后下发给 U 面的用户最大/保证带宽 |
| APN | Access Point Name | 接入点名称，本特性的控制方式绑定在 APN 粒度 |
| APNQOSATTR | APN QoS Attribute | APN 的 QoS 属性配置对象，承载上下行 CAR/Shaping 开关 |
| ARD | Access Restriction Data | （对比概念）控制面侧 WSFD-106003 用于接入限制的签约数据，与 U 面带宽维度的接入控制**机制完全不同** |

### 2.2 对象模型

本特性在 UDG 侧的配置对象极其精简，核心为 **APN 实例 + APNQOSATTR 属性**二元结构：

```
┌─────────────────────────────────────────────────────┐
│ APN 实例（ADD APN）                                   │
│   APN = "apn"                                        │
│   └── APNQOSATTR 属性（SET APNQOSATTR）              │
│         ├── CARSHAPESWUL（上行开关 ENABLE/DISABLE）   │
│         ├── CARSHAPEUL （上行方式 CAR / SHAPE）       │
│         ├── CARSHAPESWDL（下行开关 ENABLE/DISABLE）   │
│         └── CARSHAPEDL （下行方式 CAR / SHAPE）       │
└─────────────────────────────────────────────────────┘
```

| 对象 | 作用 | 接入控制场景角色 |
|------|------|----------------|
| APN | APN/DNN 实例 | 承载接入控制策略的挂载点（控制方式绑定在 APN 下） |
| APNQOSATTR | APN 的 QoS 属性 | 接入控制核心配置对象，承载上下行 CAR/Shaping 开关 |

> 说明：文档未提及其他独立的"接入控制对象"（如独立的 ACL、RateLimit 对象等）。控制策略完全封装在 APNQOSATTR 中。

> 来源：`激活接入控制_69675201.md`（"操作步骤"）、`GWFD-010151 接入控制参考信息_75667084.md`（"命令"清单：APN管理 + APNQOSATTR管理两组命令）

### 2.3 在 APN 业务域的角色

GWFD-010151 在 APN 业务域中扮演 **"带宽维度的接入闸门"** 角色，与会话管理、地址分配等共同构成用户接入链路：

```
UE 发起 PDU 会话建立
    ↓
[GWFD-010101 会话管理] 建立 PDU 会话
    ↓
[GWFD-010105 用户面地址分配] 分配 UE IP
    ↓
[C 面协商] SMF/PGW-C 与周边网元协商用户带宽资源 → 下发至 U 面
    ↓
[GWFD-010151 接入控制] ★ 按协商带宽对用户流量执行 CAR/Shaping 流控
    ↓
用户业务流量按带宽约束转发
```

具体角色说明：
1. **带宽执行点**：C 面决策"协商带宽"，U 面（GWFD-010151）执行"超出即流控"
2. **用户级保护**：防止单用户过度占用带宽影响其他在线用户
3. **APN 粒度配置**：控制策略绑定在 APN 下，可针对不同 APN 差异化配置

## 3. 原理与流程

### 3.1 实现原理：两种接入控制方式

在 EPC/5GC 网络中 UDG 支持对超出用户协商带宽资源的报文进行接入控制，包含两种方式：

#### 3.1.1 CAR 带宽控制（直接丢弃）

```
用户上下行数据报文到达 UDG
    │
    ▼
┌─────────────────────────────────────────┐
│ 判断：报文是否超出用户协商带宽？          │
└───────────────┬─────────────────────────┘
                │
       ┌────────┴────────┐
       ▼                 ▼
   未超出              超出
       │                 │
       ▼                 ▼
   转发报文           丢弃报文
```

特点：**实现简单、无延迟、无抖动**，但超带宽报文立即丢失。

#### 3.1.2 Shaping 流量整形（缓存 + 令牌桶）

Shaping 流量整形使用 **GTS 队列 + 令牌桶** 协同完成，处理过程如下：

```
用户上下行数据报文到达 UDG
    │
    ▼
┌─────────────────────────────────────────┐
│ 判断：GTS 队列是否存在？                  │
└───────────────┬─────────────────────────┘
                │
       ┌────────┴────────┐
       ▼                 ▼
   不存在              存在
       │                 │
       ▼                 ▼
┌──────────────┐    ┌──────────────────┐
│ 令牌桶处理    │    │ 报文直接进入GTS队列│
│ 按协商带宽速率│    │ 等待周期性调度转发 │
│ 向桶中放令牌  │    └──────────────────┘
└──────┬───────┘
       │
       ▼
┌─────────────────────────────────────┐
│ 判断：令牌桶是否有足够令牌发送报文？    │
└───────────────┬─────────────────────┘
                │
       ┌────────┴────────┐
       ▼                 ▼
     有令牌              无令牌
       │                 │
       ▼                 ▼
   报文继续发送       产生 GTS 队列
   令牌相应减少       报文缓存入队列
                      GTS 周期性取报文与令牌比较
                      直至令牌不足 或 队列发送完毕

                      若 GTS 队列已满 → 报文丢弃
```

特点：**平滑流量、减少突发丢包**，但会引入缓存延迟。

**令牌桶机制要点**（文档原文）：
- 令牌桶按"用户已协商的带宽速率"要求向桶中放置令牌
- 报文被发送的同时，令牌做相应的减少
- 令牌充足 → 报文直接发送
- 令牌不足 → 产生 GTS 队列，报文缓存

> 来源：`GWFD-010151 接入控制特性概述_69675200.md`（"原理概述-系统实现"章节、图1 流量整形过程描述）

### 3.2 业务流程（端到端带宽协商到接入控制执行）

```
┌─────────────────────────────────────────────────────────────────┐
│ 1. 策略与带宽协商（C 面主导）                                     │
│    SGW-C / PGW-C / SMF（UNC 控制面）与周边网元协商用户报文带宽资源 │
│    生成用户协商的最大带宽 / 保证带宽信息                           │
└──────────────────────────────┬──────────────────────────────────┘
                               │ N4 / Sxa / Sxb 接口
                               ▼
┌─────────────────────────────────────────────────────────────────┐
│ 2. 协商带宽下发至 U 面                                            │
│    C 面通过 PFCP 会话消息将协商带宽参数下发至 SGW-U/PGW-U/UPF      │
└──────────────────────────────┬──────────────────────────────────┘
                               │
                               ▼
┌─────────────────────────────────────────────────────────────────┐
│ 3. U 面接入控制策略配置（管理员静态配置，本特性核心）              │
│    管理员在 UDG 上通过 MML 配置：                                  │
│    a. ADD APN         → 创建 APN 实例（控制策略挂载点）            │
│    b. SET APNQOSATTR  → 绑定上下行 CAR/Shaping 开关到 APN          │
└──────────────────────────────┬──────────────────────────────────┘
                               │
                               ▼
┌─────────────────────────────────────────────────────────────────┐
│ 4. 用户业务流量到达 U 面                                          │
│    用户在线使用业务，上下行数据报文持续到达 UDG                     │
└──────────────────────────────┬──────────────────────────────────┘
                               │
                               ▼
┌─────────────────────────────────────────────────────────────────┐
│ 5. 实时带宽检测与流控执行（本特性核心动作）                        │
│    UDG 比较"用户已用带宽"与"协商带宽"：                            │
│    - 配置为 CAR：超出 → 丢弃；未超出 → 转发                       │
│    - 配置为 Shaping：超出 → GTS队列缓存 + 令牌桶调度；缓满 → 丢弃  │
│    按上下行方向独立配置、独立执行                                  │
└─────────────────────────────────────────────────────────────────┘
```

> 来源：综合 `GWFD-010151 接入控制特性概述_69675200.md`（涉及NF角色、原理概述）与 `激活接入控制_69675201.md`（配置步骤）

本特性无独立 License，激活即"配置使能"。激活流程极简（2步）：

```

  └── ADD APN : APN=APN名称

  └── SET APNQOSATTR : APN=APN名称,
                        CARSHAPESWUL=ENABLE,        ← 上行使能开关
```

## 5. 配置案例

### 5.1 典型场景：上下行采用 CAR 方式对接入带宽进行控制

**场景描述**：操作员在 UDG 上进行数据配置，当在线用户使用带宽超过协商的带宽资源时，上下行采用 CAR 方式对带宽进行控制，即对超出的带宽数据包进行丢弃。

**适用网元**：SGW-U、PGW-U、UPF

**MML命令序列（原样保留文档脚本）**：

```
//增加APN示例。
ADD APN:APN="apn";

//配置上下行带宽控制方式。
SET APNQOSATTR:APN="apn",CARSHAPESWUL=ENABLE,CARSHAPEUL=CAR,CARSHAPESWDL=ENABLE,CARSHAPEDL=CAR;
```

> 来源：`激活接入控制_69675201.md`（"任务示例"章节"脚本"）

### 5.2 场景变体对照

| 变体 | 上行（CARSHAPEUL） | 下行（CARSHAPEDL） | 适用场景 |
|------|-------------------|-------------------|---------|
| 上下行均 CAR（文档示例） | CAR | CAR | 严格带宽约束，超限即丢，无延迟 |
| 上下行均 Shaping | SHAPE | SHAPE | 平滑流量，容忍突发，可接受少量延迟 |
| 上行 CAR + 下行 Shaping | CAR | SHAPE | 上行严格限制、下行容忍突发（如视频下行场景） |
| 上行 Shaping + 下行 CAR | SHAPE | CAR | 上行容忍突发、下行严格限制（如上行大文件上传场景） |
| 单向使能 | ENABLE+CAR/SHAPE | DISABLE | 仅在一个方向上执行接入控制 |

> 说明：除"上下行均 CAR"为文档明示外，其余变体系基于"上下行独立配置、CAR/SHAPE 二选一"的参数语义推导，文档未给出对应脚本样例。Stage 3 横向分析可结合带宽控制场景（GWFD-110311）的 CategoryProp 配置进一步验证。

### 7.2 ★与 WSFD-106003 用户接入控制功能（SMF/控制面侧）的 C-U 关系

任务要求重点核对 GWFD-010151（UDG/UPF 侧）与 WSFD-106003（任务标注为"SMF侧"）的 C-U 对称/分工关系。经核对同域 WSFD-106003 产品文档，**两者并非对称的 C-U 分工关系，而是机制完全不同的两类"接入控制"**，需在图谱中明确区分：

| 对比维度 | GWFD-010151 接入控制（UDG/U 面侧） | WSFD-106003 用户接入控制功能（UNC/C 面侧） |
|---------|-----------------------------------|------------------------------------------|
| 网元角色 | SGW-U / PGW-U / UPF（用户面） | **SGSN / MME**（注：文档实际适用 NF 为 SGSN/MME，并非 SMF；任务描述"SMF侧"与文档不符） |
| 控制维度 | **带宽维度**（基于协商带宽的流控） | **接入权限维度**（基于用户属性的允许/拒绝接入） |
| 控制依据 | 用户已用带宽 vs 协商带宽 | 签约 ARD / 签约 APN / 卡类型（SIM/USIM） |
| 控制方式 | CAR 丢弃 / Shaping 缓存整形 | 直接拒绝接入（返回拒绝原因值）或允许接入 |
| 控制时机 | 用户在线使用业务期间，持续流控 | 用户附着/接入网络时，一次性判定 |
| 控制对象 | 用户流量（报文级） | 用户接入请求（会话级） |
| License 要求 | 无 | 需 License 82206571 LKV2ARD02 用户接入控制功能 |
| 配置对象 | APN + APNQOSATTR（CAR/Shaping 开关） | GBARD/IUARD/S1ARD（接入限制参数） |
| 遵循标准 | 3GPP TS 23.214 / 29.244 / 33.210 | 3GPP TS 29.002 / 29.272 |
| 是否对称 | ❌ **非对称**：两者不是同一控制目标在 C/U 面的分工，而是两类独立机制 | ❌ **非对称** |

**关键发现**：
1. **GWFD-010151 与 WSFD-106003 不是 C-U 对称关系**。它们一个在 U 面做带宽流控，一个在 C 面做接入权限判定，机制、对象、依据、License、标准均不同。
2. **任务描述的"SMF侧"与 WSFD-106003 文档实际适用 NF 不符**：WSFD-106003 文档明确适用 NF 为 **SGSN/MME**（UNC 作为 SGSN/MME 启动本特性），而非 SMF。WSFD-106003 系列文档分两支：适用于 SGSN/MME、适用于 AMF——均为控制面接入权限控制，均非"SMF侧带宽对称项"。
3. **真正的 U 面带宽接入控制在 C 面的对称项应为 C 面的带宽协商功能**（由 SGW-C/PGW-C/SMF 与周边网元协商用户带宽并下发），而非 WSFD-106003。文档中 C 面角色仅以"SGW-C/PGW-C/SMF：与周边网元协商用户报文的带宽资源"一笔带过，**未指明对应的 UNC 侧独立特性 ID**。Stage 3 需在 UNC 侧查找"带宽协商/会话 AMBR 协商/QoS 协商"类特性以补全对称关系。

> 来源：`GWFD-010151 接入控制特性概述_69675200.md`（全篇）、`WSFD-106003 用户接入控制功能（适用于SGSN、MME）特性概述_68358176.md`（全篇对比）
