## 1. 概述

### 1.1 特性定义（是什么）

本特性支持在 **PDP上下文激活流程 / PDN连接建立流程 / PDU会话建立流程** 中为 UE 同时分配 **IPv4 和 IPv6 两种类型的地址**，使 UE 后续可以使用 IPv4v6 地址进行数据传输，为用户提供 IPv4v6 双栈接入服务。

UNC 在一次上下文激活中为用户创建一个上下文，该上下文**同时分配两个 IP 地址**：一个 IPv4 地址 + 一个 IPv6 地址。这是与单栈（IPv4 / IPv6）激活的本质差异——一个会话承载两种协议栈。

> 来源：`output/UNC 20.15.2 产品文档(裸机容器) 05/网络部署/特性部署/UNC特性指南/IPv6功能/WSFD-104002 IPv4v6双栈接入/IPv4v6双栈接入（2G_3G_4G）/特性概述_48043389.md`（"定义"章节）、`IPv4v6双栈接入（5G）/特性概述_48043373.md`（"定义"章节）、`特性概述_48043389.md`（"双栈的地址分配方式"章节）

## 2. 原理

### 2.1 双栈技术基础

双栈技术是 IPv4 向 IPv6 过渡的有效技术。双栈指网络中的节点**同时支持 IPv4 和 IPv6 协议栈**，源节点根据目的节点的不同选用不同的协议栈，而网络设备根据报文的协议类型选择不同的协议栈进行处理和转发。双栈节点与 IPv4 节点通讯时使用 IPv4 协议栈，与 IPv6 节点通讯时使用 IPv6 协议栈。

> 来源：`IPv4v6双栈接入（5G）/特性概述_48043373.md`（"原理概述/双栈地址分配方式"章节）

### 2.2 双栈地址分配方式（两种决策源）

UNC 为用户创建一个上下文，此上下文中**同时分配两个 IP 地址**（一个 IPv4 + 一个 IPv6）。根据分配地址的来源，分两种决策方式：

| 决策方式 | 地址来源 | UNC 角色 |
|---------|---------|---------|
| **UDM 静态分配** | UE 在 HSS/UDM 注册分组业务时确定的 IPv4v6 地址（静态地址） | 上下文激活过程中直接在信元中携带 IPv4v6 静态地址，通过激活请求消息发送给 PGW-C / GGSN / SMF |
| **SMF 动态分配** | SMF 本地地址池 | SMF 在 PDU 会话建立阶段为该 PDU 会话分配 IPv4v6 双栈地址 |

> 来源：`特性概述_48043389.md`（"双栈的地址分配方式"章节）、`IPv4v6双栈接入（5G）/特性概述_48043373.md`（"原理概述/双栈地址分配方式"章节）

### 2.3 IPv4v6 双栈控制策略（2/3/4G，两阶段决策）

#### 2.3.1 阶段一：签约数据修改控制判断策略

UNC 收到激活请求后，按以下决策链判断用户的 PDP/PDN Type 是否应纠正为 IPv4v6：

```
1. 查看HLR或HSS中的签约数据是否支持双栈或IPv4v6双栈类型？
   ├─ 是 → 跳到步骤5（保存签约数据）
   └─ 否 → 步骤2
2. 查看地址是否为静态IP？
   ├─ 是 → 跳到步骤5（静态IP不修改）
   └─ 否 → 步骤3
3. 根据(IMSI号段、PDP TYPE、APN)三元组查询 LST SMSUBDATA，
   查看是否配置了将签约数据"PDP Type"修改为"IPv4v6"的纠正规则？
   ├─ 是 → 步骤4
   └─ 否 → 步骤5
4. 按规则修改签约数据（执行纠正）
5. 保存签约数据
```

> 来源：`特性概述_48043389.md`（"原理概述/IPv4v6双栈控制策略 - 签约数据修改控制判断策略"章节）

#### 2.3.2 阶段二：IPv4v6 双栈启动控制判断策略

完成签约数据匹配后，进一步决策是否实际启动双栈：

```
1. 查询签约数据匹配结果是否为IPv4v6双栈？
   ├─ 是 → 步骤2
   └─ 否 → 步骤5（直接保存PDP类型到PDP上下文）
2. 读取 LST SMFUNC，查询全局双栈开关(DUALFLAG)是否开启？
   ├─ 是 → 步骤4（启动双栈）
   └─ 否 → 步骤3
3. 不启动双栈，按 SET SMFUNC 配置项决定优选V4还是V6
4. GnGp SGSN/MME 启动IPv4v6双栈
5. GnGp SGSN/MME 将PDP类型保存到PDP上下文
```

> 来源：`特性概述_48043389.md`（"原理概述/IPv4v6双栈控制策略 - IPv4v6双栈启动控制判断策略"章节）

### 2.4 PDP 上下文激活涉及的 IPv4v6 流程（2/3G）

```
1. 通过签约数据修改控制策略，使该用户可以支持IPv4v6业务
2. UE → GnGp SGSN：Activate PDP Context请求
   （若PDP address为空 = 动态地址；UE请求IPv4v6双栈 → 启动双栈控制策略）
3. PDP Type确定为IPv4v6双栈后：
   GnGp SGSN → GGSN：Create PDP Context Request
     携带：双栈标志 Dual Address Bearer Flag=1 + PDP Type=IPv4v6
   GGSN → GnGp SGSN：Create PDP Context Response
     携带：End User Address（含IPv4v6类型）
   ★ GGSN可接受或拒绝双栈请求：
     - 拒绝时双栈被纠正为单栈，直接下发单栈地址，原因值：
       · "New PDP type due to network preference"（GGSN选了与UE不一致的PDP Type）
       · "New PDP type due to single address bearer only"（GGSN支持IPv4/IPv6单栈但不支持双栈）
4. GnGp SGSN → RNC：RAB Assignment Request
   （若IPv4v6双栈，消息携带 PDP Type Information Extension 字段）
5. IPv6场景下：GGSN 生成 IPv6 路由广播（前缀）经 SGSN 发送到 MS
```

> 来源：`特性概述_48043389.md`（"原理概述/PDP上下文激活涉及的IPv4v6流程"章节）

### 2.5 PDN 连接涉及的 IPv4v6 流程（4G）

```
1. 通过签约数据修改控制策略，使该用户可以支持IPv4v6业务
2. UE → MME：PDN连接请求
   （MME选PGW-C，建默认承载；UE请求IPv4v6双栈 → 启动双栈控制策略）
3. PDN Type确定为IPv4v6双栈后：
   MME → SGW-C → PGW-C：Create Session Request
     携带：双栈标识 Dual Address Bearer Flag=1 + PDN Type=IPv4v6
   PGW-C → SGW-C → MME：Create Session Response
     透传，携带：PDN Type + PDN Address
   ★ PGW-C可接受或拒绝双栈请求，拒绝时双栈纠正为单栈，原因值同2/3G
4. MME → eNodeB：Setup Bearer Request
   含Nas消息 Activate Default EPS Bearer Request（若双栈，Nas消息携带双栈信息）
5. eNodeB → MME：Activate default EPS bearer context accept
6. SGW-C ← MME：Modify Bearer Request (EPS Bearer Identity)
   建立S1-U缺省承载，建立S1-U与S5/S8-U缺省承载映射
7. SGW-C → MME：Modify Bearer Response (EPS Bearer Identity)
8. IPv6场景下：PGW-C 生成 IPv6 路由广播（前缀）经 MME 发送到 UE
```

> 来源：`特性概述_48043389.md`（"原理概述/PDN连接涉及的IPv4v6流程"章节）

### 2.6 PDU 会话建立流程中的 IPv4v6 双栈地址分配（5G）

```
1. UE → AMF：PDU Session Establishment Request
   携带：Requested PDU Session Type 信元（可选，取值 IPv4/IPv6/IPv4v6）
2. AMF 选SMF → SMF：Nsmf_PDUSession_CreateSMContext Request
   传递：Requested PDU Session Type
3. SMF ↔ UDM：获取签约数据
   - Allowed PDU Session Type(s)（允许的PDU会话类型）
   - Default PDU Session Type（缺省PDU会话类型）
   - 静态IP地址（可为IPv4v6双栈类型）
   ★ 若UE请求类型 与 Allowed或Default 均不一致 → SMF拒绝建会话，
     经 AMF Downlink NAS Transport 将拒绝结果传给UE
4. SMF 选UPF → 将分配好的IP地址发给UPF
5. SMF → AMF：Namf_Communication_N1N2MessageTransfer
   已分配IP地址经 AMF、(R)AN 传递给UE
6. IPv6场景下：SMF 生成 IPv6 路由广播（前缀）
   经 N4 会话和 UPF 发送到UE
```

> 来源：`IPv4v6双栈接入（5G）/特性概述_48043373.md`（"原理概述/业务流程"章节）

### 2.8 双栈拒绝降级机制（重要）

当 PGW-C / GGSN 拒绝 UE 的双栈请求时，**双栈会被纠正为单栈**，直接下发单栈地址给 UE，并携带以下原因值之一：

| 原因值 | 含义 |
|--------|------|
| New PDP type due to network preference | GGSN/PGW-C 选择了一个与 UE 请求不一致的 PDP Type |
| New PDP type due to single address bearer only | UE 请求双栈，GGSN/PGW-C 支持 IPv4 和 IPv6 单栈，但不支持双栈 |

> 来源：`特性概述_48043389.md`（"原理概述/PDP上下文激活"和"PDN连接"章节的双栈拒绝描述）

### 2.9 其它业务涉及的 IPv4v6 流程

- **RAU（Inter/Intra SGSN RAU, 2/3G）**：旧侧 PDP 上下文含 IPv4+IPv6 地址；新侧重建承载时启动双栈控制策略
- **TAU（2/3G）**：PDP 上下文含 IPv4+IPv6 地址
- **修改 PDP 上下文（2/3G）**：GGSN 触发的修改流程可能修改 PDP 地址，SGSN 保存 Update PDP Context Request 中 End User Address 里新增的 PDP Type 及 PDP Address
- **Service Request（2/3G）**：重建 RAB，若 IPv4v6 双栈则 RAB Assignment Request 携带 PDP Type Information Extension
- **Serving RNS Relocation（2/3G）**：旧侧 Forward Relocation Request 含 IPv4+IPv6；新侧重建承载启动双栈控制策略
- **Handover（2/3G + 4G）**：旧侧 Forward Relocation Request / Context Response 含 IPv4+IPv6；新侧重建承载启动双栈控制策略
- **RAU / TAU / Handover（4G）**：覆盖 Inter S1↔Iu/Gb、Inter S1↔S1 多种场景，逻辑同 2/3G（旧侧含双栈地址，新侧重建启动双栈控制策略）

> 来源：`特性概述_48043389.md`（"原理概述/其它业务涉及的IPv4v6流程"章节，2/3G + 4G）

### 6.7 常见问题与排查

| 问题现象 | 可能原因 | 排查方法 |
|---------|---------|---------|
| 双栈激活失败，被纠正为单栈 | PGW-C/GGSN 支持单栈但不支持双栈 | 检查 UE 收到的原因值："New PDP type due to single address bearer only" → 升级 PGW-C/GGSN 支持双栈 |
| 双栈激活失败，PDP Type 与请求不一致 | PGW-C/GGSN 因网络偏好选了与 UE 不一致的 PDP Type | 检查原因值："New PDP type due to network preference" → 核对网络偏好配置 |
| 全局双栈未启动（2/3/4G） | SET SMFUNC DUALFLAG=NO | 执行 LST SMFUNC 核对 DUALFLAG；SET SMFUNC DUALFLAG=YES |
| 签约数据未纠正为 IPv4v6（2/3/4G） | HLR/HSS 不支持 IPv4v6 签约且未配 SMSUBDATA | 执行 LST SMSUBDATA 核对纠正规则；ADD SMSUBDATA 配置 IPv4→IPv4v6 纠正 |
| Dual Address Bearer Flag 未传递（2/3G） | GTP-C V0/V1 兼容性配置中 Common Flags 未携带 | 执行 LST GTPCCMPT 核对；MOD GTPCCMPT CMFLG=YES |
| 5G 双栈依赖失败 | 未激活 WSFD-104001 IPv6 承载上下文 | 确认已加载 82209965/82209968 License 并激活 IPv6 承载上下文 |
| 实际 PDN 类型不为 IPv4v6 | License 开关未打开 | LST LICENSESWITCH 核对；SET LICENSESWITCH SWITCH=ENABLE |
| 5G 会话被 SMF 拒绝 | UE 请求类型与 Allowed/Default PDU Session Type 均不一致 | 核对 UDM 签约的 Allowed/Default PDU Session Type 是否包含 IPv4v6 |
| 协议互通失败（2/3G） | GGSN/SGSN 或 RNC Iu 口版本不满足 | 升级到 3GPP TS 29.060 R9 2010Q3+ / TS 25.413 R9 2010Q2+ |
| 离线计费话单 IP 字段缺失（2/3/4G） | GCDR/EGCDR 只支持一个 IP 字段 | 已知限制（非故障），双栈用户话单只能记录一个 IP |

### 8.2 与 GWFD-020403（UDG 侧）的 C-U 协同

| # | 维度 | GWFD-020403（UDG/U面） | WSFD-104002（UNC/C面） | 协同/差异 |
|---|------|------------------------|------------------------|----------|
| 1 | 角色定位 | U面执行方：按C面指示建立双栈承载、转发双栈报文 | C面决策方：决定双栈类型、分配双栈地址、生成IPv6路由广播 | **C-U 协同分工**：决策在C面，执行在U面 |
| 2 | 决策核心 | 无决策权（按C面 Dual Address Bearer Flag + 双栈地址执行） | **双栈类型决策权**（签约匹配→纠正→全局开关→地址分配→拒绝降级） | 决策权统一在C面 |
| 3 | IPv6 路由广播生成 | U面不生成（透传） | **C面生成**（GGSN/PGW-C/SMF 生成 IPv6 前缀，经 U面透传给UE） | 差异：IPv6 前缀生成在C面 |
| 4 | License | UDG 侧 License | 4 个 C 面 License 控制项分散 | 差异：License 独立 |
| 5 | 拒绝降级决策 | U面无降级决策权 | C面 PGW-C/GGSN 决策降级（双栈→单栈 + 原因值） | 决策权在C面 |
| 6 | 文档数量 | 5 篇（UDG侧） | 12 篇（代际拆分更细） | 差异：C面文档更丰富 |

### 8.3 与 WSFD-010502（地址分配方式）的关系

| # | 维度 | WSFD-010502（地址来源决策） | WSFD-104002（双栈类型决策） | 关系 |
|---|------|----------------------------|---------------------------|------|
| 1 | 决策维度 | 决定"地址从哪里来"（UDM/Radius/SMF本地池/DHCP 4种来源） | 决定"是IPv4v6双栈还是单栈"（双栈类型决策） | **正交组合**：两个决策维度独立，共同决定一次激活的最终地址 |
| 2 | 组合示例 | UDM 静态分配 + IPv4v6 双栈 = UDM 签约的静态 IPv4v6 地址 | - | 本特性 §2.2 "UDM静态分配"方式即为此组合 |
| 3 | 组合示例 | SMF 本地池分配 + IPv4v6 双栈 = SMF 从本地池动态分配 IPv4+IPv6 | - | 本特性 §2.2 "SMF动态分配"方式即为此组合 |
| 4 | License | 无 License | 4 个 License 控制项 | **差异**：同域特性 License 要求不同 |
| 5 | MML 命令体系 | ADD ADDRPOOL/SECTION/ADDRPOOLGRP/POOLBINDGRP/UPNODE/ADDRUPGROUP/UPFBINDGRP/POOLGRPMAP（地址池+UPF组体系） | SET SMFUNC/ADD SMSUBDATA/SET LICENSESWITCH（开关+签约纠正+License体系） | **差异**：两特性命令体系完全独立，无重叠 |
| 6 | 静态地址场景协同 | WSFD-010502 定义"UDM静态分配"方式（HLR/HSS/UDM签约静态IP） | WSFD-104002 定义"UDM静态IPv4v6"（HLR/HSS/UDM签约静态IPv4v6双栈） | **协同**：本特性的静态IPv4v6场景复用 WSFD-010502 的静态分配签约数据机制 |
| 7 | 配置树关系 | 同属"地址分配"大类 | 同属"地址分配"大类 | 配置树兄弟节点（推断 WSFD-010502 为父，WSFD-104002 为子） |
