## 1. 概述

### 1.1 特性定义（是什么）

会话管理功能是 UDG 的基本功能，是用户与外部 DN/PDN（Data Network / Packet Data Network）建立连接、进行数据业务的基础。UDG 的会话管理功能包括网络侧发起的会话管理和 UE 发起的会话管理，覆盖 2G/3G（PDP 上下文）、4G（EPS 承载）、5G（PDU 会话）三种接入制式。

华为 UDG 支持 3GPP 标准定义的 GERAN/UTRAN、E-UTRAN、5G 无线网络接入，分别实现 GPRS/UMTS、LTE、5G 网络用户的接入。UDG 在会话管理中作为**用户面网元**，配合控制面（SGSN/MME、SGW-C/PGW-C、SMF）完成 PDP 上下文激活/去激活/更新、EPS 缺省/专有承载激活/去激活/更新、PDU 会话建立/修改/释放等流程，为 UE 提供到外部数据网络的用户面数据转发与连接通路。

> 来源：`GWFD-010101 会话管理（5G）特性概述_73618343.md`（"定义"章节）、`GWFD-010101 会话管理（4G）特性概述_68633827.md`（"定义"章节）、`GWFD-010101 会话管理（2_3G）特性概述_61678540.md`（"定义"章节）

## 2. 激活

> 本特性**无独立激活步骤**。三份特性概述均明确声明"本特性无需配置即可使用"。会话管理作为 UDG 基本功能，随 UDG 网元用户面接口（N4/Sxa/Sxb）与控制面的偶联建立而自动具备能力。实际"激活一个用户会话"的动作发生在用户接入时的 PDU/PDP 建立流程中，由控制面通过 PFCP/GTP-C 信令触发，而非通过 MML 配置命令开启特性开关。

| 触发方式 | 用户接入时由控制面下发 PFCP/GTP-C 会话建立请求，UDG 被动建立用户面会话上下文 |

## 3. 原理

### 3.1 会话管理业务流程总览（按接入制式 + 流程类型 + 发起方）

| 接入制式 | 流程类型 | 发起方 | 业务场景 |
|---------|---------|--------|---------|
| 5G | PDU 会话建立 | UE | UE 需要与外部网络进行业务交互（如开机注册后建立默认会话）；UE 从 4G EPS 切换到 5GS；UE 从非3GPP切换到3GPP接入 |
| 5G | PDU 会话修改 | UE | UE 能力变更、QoS 参数修改（如打开新 APP 触发新 QoS Flow） |
| 5G | PDU 会话修改 | AN(RAN) | (R)AN 识别现有 QoS 不能满足业务要求时发起 |
| 5G | PDU 会话修改 | PCF | PCF 识别配置策略有修改，指示 SMF 更新会话策略 |
| 5G | PDU 会话修改 | UDM | UDM 识别策略有修改，通知 SMF 更新订阅的会话数据 |
| 5G | PDU 会话释放 | UE | UE 不再需要相关业务时发起 |
| 5G | PDU 会话释放 | AMF | UE 与 AMF 的会话状态不匹配，或 UE 的网络切片不可用时，AMF 发起 |
| 5G | PDU 会话释放 | SMF | SMF 识别本地配置策略刷新时发起 |
| 5G | PDU 会话释放 | PCF | PCF 根据本地策略通过 update 消息通知 SMF 需释放的 PDU 会话 |
| 4G | 缺省承载激活 | UE | 附着流程的一部分或单独流程；MME 进行 SGW-C/PGW-C 选择，SGW-C/PGW-C 完成 IP 地址分配和权限检查 |
| 4G | 专有承载激活 | 网络侧 | 在 UE 和 EPC 之间建立使用特定 QoS 和 TFT 的承载上下文 |
| 4G | 承载去激活 | MME/SGW-C/PGW-C | 去激活一个或多个 EPS 承载；最后一个缺省承载只能在去附着流程中删除 |
| 4G | 承载更新 | UE/HSS/PCRF/SGW-C/PGW-C | 修改 EPS 承载上下文参数（QoS、TFT） |
| 2/3G | PDP 上下文激活（一次/二次） | MS | MS 附着后发起，建立 MS 与 PDN 之间的虚拟数据通道 |
| 2/3G | PDP 上下文去激活 | MS/UDG | 释放 PDP 上下文，MS 的 IP 地址被释放 |
| 2/3G | PDP 上下文更新 | MS/RNC/PCRF/UDG | 修改已激活 PDP 上下文的参数（QoS、TFT） |

> 来源：5G 特性概述"原理概述"章节、4G 特性概述"会话管理流程介绍（表2）"章节、2/3G 特性概述"会话管理流程介绍（表1）"章节

### 3.2 5G PDU 会话建立流程（UDG/UPF 侧视角，★重点）

5G 核心网支持 PDU 连接业务，即在 UE 和 DNN 所标识的数据网络之间提供 PDU 交换服务。PDU 会话的创建、修改、释放通过 UE 和 SMF 之间的 NAS 信令交换实现。PDU 会话建立流程包括 UE 的 IP 地址分配、UPF 选择、PCC 策略获取以及 UE 接入外部网络的权限检查等。

| PFCP Session Establishment Request（SMF→UPF） | CP F-SEID（SMF 会话标识）；Create PDR（PDR ID, PDI, Source interface, Local F-TEID, SDF Filter, UE IP Address, UE IP Address Pool Identity）；Create FAR（FAR ID, Apply Action, Forwarding Parameters, Destination interface）；Create URR；Create QER（QER ID, Gate Status, QFI）；Create BAR | 接收 SMF 下发的流检测/转发/统计/QoS 执行/缓存规则，建立用户面会话上下文 |
| PFCP Session Establishment Response（UPF→SMF） | Cause；UP F-SEID（UPF 会话标识）；UE IP Address（UPF 本地地址池分配时回传）；Local F-TEID（UPF 侧 N3 隧道端点） | 分配 UPF 侧会话标识、N3 隧道端点；若负责地址分配则回传 UE IP |
| PFCP Session Modification Request（SMF→UPF） | Update FAR（下行 FAR）；Update Forwarding Parameters；Outer header creation（AN 侧 TEID+IP） | 将(R)AN 侧隧道信息写入下行 FAR，打通下行数据转发通路 |
| PFCP Session Modification Response（UPF→SMF） | Cause | 确认下行隧道更新 |

> 来源：`PDU会话建立_22418374.md`（步骤16/17/27/28 关键信元）

### 3.3 5G PDU 会话修改流程（4类发起方 × UPF 视角）

| 发起方 | 触发场景 | UPF 涉及的 N4 交互 |
|--------|---------|-------------------|
| UE | UE 能力变更、QoS 参数修改（新业务触发新 QoS Flow） | 步骤11/12：SMF→UPF PFCP Session Modification Request（Update PDR/FAR/QER），UPF→SMF Response（文档标注：QoS 参数修改涉及 UPF 侧修改时触发） |
| AN(RAN) | (R)AN 识别现有 QoS 不能满足业务要求 | 同上：QoS 变更时 SMF 与 UPF 交互 PFCP Session Modification |
| PCF | PCF 识别配置策略修改，指示 SMF 更新会话策略 | 同上；文档明确 Remove PDR / Remove FAR / Remove QER（删除规则） |
| UDM | UDM 识别策略修改，通知 SMF 更新订阅会话数据 | QoS 改变需通知 (R)AN/UE 时经 SMF→UPF PFCP Modification |

**PDU 会话修改流程共性**：UE 发起修改的 9 步核心流程为 UE→AMF（PDU session modification request）→SMF→PCF（Npcf_SMPolicyControl_Update）→SMF→AMF→(R)AN→UE（PDU session modification command）；(R)AN→AMF→SMF（PDU SESSION RESOURCE MODIFY RESPONSE）；SMF→UPF（PFCP Session Modification）；UPF→SMF（Response）；SMF→AMF→UE（PDU session modification command complete）。

> 来源：`PDU会话修改（UE发起）_22578338.md`、`PDU会话修改（AN发起）_72654737.md`、`PDU会话修改（PCF发起）_22895266.md`、`PDU会话修改（UDM发起）_22735962.md`（"业务场景"与"信令流程"章节）

### 3.4 5G PDU 会话释放流程（4类发起方 × UPF 视角）

| 发起方 | 触发场景 | UPF 涉及的 N4 交互 |
|--------|---------|-------------------|
| UE | UE 不再需要相关业务时发起会话释放 | 步骤3/4：SMF→UPF PFCP Session Deletion Request，UPF→SMF Response（丢弃该 PDU 会话相关数据包，释放 N4 会话相关的所有隧道资源和上下文） |
| AMF | UE 与 AMF 的会话状态不匹配，或 UE 的网络切片不可用时 | SMF→UPF PFCP Session Deletion（释放 SM 上下文前触发） |
| SMF | SMF 识别本地配置策略刷新时发起 | 步骤1/2：SMF→UPF PFCP Session Deletion Request，UPF→SMF Response（含 Usage Report 若已开通 URR） |
| PCF | PCF 根据本地策略通过 update 通知 SMF 需释放的 PDU 会话 | SMF→UPF PFCP Session Deletion |

**PDU 会话释放流程共性**：核心动作是 SMF→UPF **PFCP Session Deletion**（区别于修改的 Modification），UPF 收到后丢弃该 PDU 会话相关的数据包并释放所有 N4 隧道资源和上下文。若已开通 URR，响应消息中携带 Usage Report。释放后 SMF 向 PCF 发送 Npcf_SMPolicyControl_Delete 释放 SM 策略关联，向 AMF 发送 Nsmf_PDUSession_NotifySMContextStatus 上报 SM 上下文状态（RELEASED）。

> 来源：`PDU会话释放（UE发起）_72815297.md`、`PDU会话释放（AMF发起）_72735009.md`、`PDU会话释放（SMF发起）_22418378.md`、`PDU会话释放（PCF发起）_72934785.md`（"业务场景"与"信令流程"章节）

### 3.5 4G EPS 承载会话管理原理

#### 3.5.1 基本概念

| 概念 | 描述 |
|------|------|
| PDN 连接 | EPS 通过 E-UTRAN 和 EPC，为 UE 提供一条 UE 与 PDN 之间的 IP 连接 |
| EPS 承载 | 用于唯一标识基于 GTP 协议的 UE 和 P-GW 之间的共同 QoS 业务流 |
| 缺省承载 | UE 建立 PDN 连接时建立，在该 PDN 连接过程中不释放，提供永久性连接；缺省承载删除则 PDN 连接释放 |
| 专有承载 | 除缺省承载外连接到相同 PDN 的承载，根据不同 QoS 需求建立（如视频、网页） |

**承载关系约束**：
- APN 与 PDN 一一对应；一个终端可通过多个 APN 接入不同 PDN
- 同一 APN 下可建立多个 PDN 连接
- 同一 PDN 连接必须且只能建立一个缺省承载，但可建立多个专有承载
- 由于无线侧限制，**一个终端最多只能建立 11 个承载**

> 来源：`GWFD-010101 会话管理（4G）特性概述_68633827.md`（"基本概念"与表1）

#### 3.5.2 4G 会话管理流程（4类）

| 流程 | 发起方 | 核心动作 | UPF(PGW-U/SGW-U) 侧 |
|------|--------|---------|---------------------|
| 缺省承载激活 | UE（附着的一部分或单独流程） | MME 进行 SGW-C/PGW-C 选择；SGW-C/PGW-C 完成 IP 地址分配、UE 接入 ISP/企业网权限检查 | 通过 Sxb 接口接收 SGW-C/PGW-C 下发的会话建立指示，建立用户面承载上下文 |
| 专有承载激活 | 网络侧 | 在 UE 和 EPC 之间建立使用特定 QoS 和 TFT 的承载上下文 | 通过 Sxb 接收新承载建立指示 |
| 承载去激活 | MME 或 SGW-C/PGW-C | 去激活一个或多个 EPS 承载；最后一个缺省承载只能在去附着流程中删除；去激活过程中 UE IP 地址被释放，MME/SGW-C/PGW-C 承载上下文被删除 | 通过 Sxb 接收承载释放指示，释放用户面资源 |
| 承载更新 | UE/HSS/PCRF/SGW-C/PGW-C | 修改 EPS 承载上下文参数（QoS、TFT）；UE 可请求分配/修改特定 QoS，或请求新的 GBR 承载 | 通过 Sxb 接收更新后的 QoS/TFT |

> 说明：TFT（Traffic Flow Template）是一组数据包过滤器的集合，UE 使用 TFT 决定业务数据流和无线承载的匹配关系。GBR 主要体现为带宽管理。
> 来源：`GWFD-010101 会话管理（4G）特性概述_68633827.md`（"会话管理流程介绍-表2"）

### 3.6 2/3G PDP 上下文会话管理原理

#### 3.6.1 2/3G 会话管理流程（3类）

| 流程 | 发起方 | 核心动作 | UPF(PGW-U) 侧 |
|------|--------|---------|---------------|
| PDP 上下文激活 | MS（一次激活 / 二次激活） | MS 附着后发起，在 MS 和 PDN 之间建立虚拟数据通道；网络侧根据用户请求及签约数据完成 IP 地址分配、QoS 参数协商、MS 接入 ISP/企业网权限检查 | PGW-U 通过 Gn/Gp 接口与 PGW-C 配合，建立 PDP 上下文对应的用户面 |
| PDP 上下文去激活 | MS 或 UDG | 释放 PDP 上下文；MS 的 IP 地址被释放；UDG 中的 PDP 上下文被删除 | 释放用户面资源 |
| PDP 上下文更新 | MS/RNC/PCRF/UDG | 修改已激活 PDP 上下文参数（QoS、TFT） | 更新用户面 QoS |

> 说明：2/3G 特性首次发布版本为 **20.3.2**（晚于 4G/5G 的 20.0.0）；PGW-U 通过 **Gn 接口和 RNC 互通**。
> 来源：`GWFD-010101 会话管理（2_3G）特性概述_61678540.md`（"会话管理流程介绍-表1"、"涉及NF-PGW-U"）

### 3.7 协议交互（★N4/PFCP 接口会话交互）

| 接口 | 交互网元 | 关键消息 | 说明 |
|------|---------|---------|------|

| N2（5G） | (R)AN ↔ AMF | PDU SESSION RESOURCE SETUP/MODIFY/RELEASE COMMAND/RESPONSE | N2 SM 信令（含 CN/AN 隧道信息、QoS Profile、QFI） |

## 5. 配置案例

> 本特性**无独立配置案例**（无 MML）。以下提供基于 PFCP 信令的会话建立**信令流程样例**，展示 UPF 在 PDU 会话建立中的关键消息内容（非 MML 配置，保留原始信令描述）。

### 5.1 典型场景一：5G PDU 会话建立（SMF 分配 IP，UPF 作为用户面转发）

**场景描述**：UE 开机注册后发起 PDU 会话建立，请求 IPv4 类型。SMF 从 UDM 获取签约、从 PCF 获取策略，决策由 SMF 分配 UE IP（或由 UPF 本地分配，详见 GWFD-010105），通过 N4 接口向 UPF 下发 PDR/FAR/URR/QER 规则，UPF 分配 N3 隧道端点并回传，最终 SMF 将 AN 侧隧道信息更新到 UPF 打通下行通路。

**关键信令消息序列（UPF 视角，原样保留文档描述）**：

```
//阶段C：SMF ↔ UPF N4 会话建立
16. SMF → UPF: PFCP Session Establishment Request
    关键信元：
      CP F-SEID                          //SMF 侧会话唯一标识
      Create PDR:
        PDR ID                            //PFCP 会话内 PDR 标识
        PDI                               //匹配的报文
        Source interface (access/core)    //access=上行PDR(N3), core=下行PDR(N6)
        Local F-TEID                      //SMF 分配的 N3 隧道端点
        SDF Filter                        //业务数据流识别分类
        UE IP Address                     //SMF 分配的 UE IP（或指示 UPF 分配）
        UE IP Address Pool Identity       //UPF 分配时指示地址池名
      Create FAR:
        FAR ID
        Apply Action                      //转发/丢弃/缓存
        Forwarding Parameters
        Destination interface (core/access)
      Create URR                          //统计上报规则
      Create QER:
        QER ID
        Gate Status                       //上下行允许转发或丢弃
        QoS Flow Identifier (QFI)
      Create BAR                          //缓存动作规则

17. UPF → SMF: PFCP Session Establishment Response
    关键信元：
      Cause                               //接受或拒绝
      UP F-SEID                           //UPF 侧会话唯一标识
      UE IP Address                       //UPF 本地分配时回传 UE IP
      Local F-TEID                        //UPF 侧 N3 隧道端点（TEIDU）

//阶段E：AN 隧道通知
27. SMF → UPF: PFCP Session Modification Request
    关键信元：
      Update FAR                          //更新下行 FAR
      Update Forwarding Parameters
      Outer header creation               //AN 侧 TEID + IP 地址

28. UPF → SMF: PFCP Session Modification Response
    关键信元：
      Cause
```

> 来源：`PDU会话建立_22418374.md`（步骤16/17/27/28 原始描述）

### 5.2 典型场景二：5G PDU 会话释放（SMF 发起，策略刷新）

**场景描述**：SMF 识别本地配置策略刷新时发起 PDU 会话释放。

**关键信令消息序列（UPF 视角）**：

```
1. SMF → UPF: PFCP Session Deletion Request
   关键信元：
     message type = pfcp-session-deletion-request

2. UPF → SMF: PFCP Session Deletion Response
   关键信元：
     Cause = request-accepted

```

### 5.3 场景变体对照（UPF 侧 N4 交互对比）

| 流程类型 | 发起方 | UPF 侧 N4 消息 | UPF 核心动作 |
|---------|--------|---------------|-------------|
| PDU 会话建立 | UE | PFCP Session Establishment（接收） + PFCP Session Modification（接收 AN 隧道） | 分配 UP F-SEID、N3 TEIDU；可选分配 UE IP；建立 PDR/FAR/URR/QER；下行 FAR 写入 AN 隧道 |

## 6. 验证与调测

> 本特性**无独立调测文档**（16 个源文件均为特性概述与实现原理流程文档，无"调测"文档）。会话级调测通常通过查询用户上下文、跟踪 N4 信令、统计性能指标进行。

### 6.1 验证方法

#### 6.1.1 调测前提与目的

- 调测前提：UDG 已完成 N4/Sxa/Sxb 接口配置并与控制面偶联成功；用户已签约
- 调测目的：验证用户 PDU/PDP 会话能正常建立、修改、释放；UPF 侧会话上下文与控制面一致

#### 6.1.2 调测执行步骤（推荐方法）

| 步骤 | 操作 | 预期结果 |
|------|------|---------|
| 1 | 测试终端发起 PDU 会话建立（5G）/附着+缺省承载建立（4G）/PDP 激活（2/3G） | 终端激活成功，获得 IP 地址 |
| 2 | 在 UDG 上查询用户会话上下文（如 `DSP SESSIONINFO`，命令归属 GWFD-010105） | 显示用户 IMSI、PDP Type、UE IP、APN name、IPv4 Address type 等会话信息 |
| 3 | 在 O&M 上跟踪 N4 接口信令 | 观察到 PFCP Session Establishment/Modification/Deletion 消息交互正常 |
| 4 | 触发会话修改（如新业务 QoS Flow） | UPF 收到 PFCP Session Modification，QER 更新成功 |
| 5 | 触发会话释放（UE 关机或网络侧释放） | UPF 收到 PFCP Session Deletion，资源释放，回带 Usage Report（若开通 URR） |

### 6.4 常见问题与排查

| 问题现象 | 可能原因 | 排查方法 |
|---------|---------|---------|
| PDU 会话建立失败（N4 PFCP Establishment 失败） | UPF 资源不足；PDR/FAR/QER 规则配置错误；地址分配失败（若 UPF 负责分配） | 跟踪 N4 信令，查看 PFCP Session Establishment Response 的 Cause；查地址池占用（DSP POOLUSAGE，归属 GWFD-010105） |
| 下行数据不通（会话已建立） | AN 隧道未更新到 UPF 下行 FAR（步骤27未完成） | 跟踪 PFCP Session Modification，确认 Outer header creation 已下发 |
| 会话修改后 QoS 不生效 | QER 未更新或 Gate Status 为 closed | 跟踪 PFCP Session Modification，检查 Update QER |
| 会话释放后资源未回收 | PFCP Session Deletion 未送达或 UPF 处理失败 | 跟踪 PFCP Session Deletion Response 的 Cause；查 UPF 会话上下文是否清理 |
| 会话状态不匹配（UE 与 AMF） | 触发 AMF 发起释放的场景 | 参考 PDU 会话释放（AMF发起）流程排查 |
| 承载数超限（4G/2-3G） | 单用户承载已达 11 | 查用户承载计数，限制新承载建立 |
| PDU 会话类型协商失败 | UE 请求的 PDU type 与 UDM 签约/PCF 授权不一致 | 查 PCF Npcf_SMPolicyControl_Create Response 的 ipv4Index/ipv6Index；查 UDM 签约 dnnConfigurations |
