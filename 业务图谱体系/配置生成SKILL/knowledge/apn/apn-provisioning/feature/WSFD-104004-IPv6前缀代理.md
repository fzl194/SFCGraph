## 1. 概述

### 1.1 特性定义（是什么）

IPv6 Prefix Delegation（IPv6 PD，IPv6 前缀代理）应用于 **移动 VPN 网络**，是一种通过一台无线设备（MS/UE，作为无线移动路由器 / Wireless Router）将**多台终端设备**接入网络并与网络侧设备进行双向数据业务的技术方案。MS/UE 从 UNC（GGSN-C / SGW-C / PGW-C / SMF）获取一段 **IPv6 delegated-prefix（IPv6 代理前缀，长度通常为 /60 或 /64 之外的更短前缀）**，再由 MS/UE 将此前缀划分为多个 64 位前缀，下发给其下挂的终端设备，使每台下挂终端获得独立的 IPv6 前缀。

**与 IPv6 承载上下文（WSFD-104001）的本质差异**：
- WSFD-104001 为 **UE 本身** 分配 IPv6 地址/前缀（/64，UE 自己用）
- WSFD-104004 为 **UE 下挂的网络** 分配更长的 IPv6 delegated-prefix（UE 作路由器为下挂终端再分配）

> 来源：`特性概述_76459525.md`（"定义"+"原理概述"章节）

### 1.2 适用NF（UDG/UNC网元）

本特性为纯控制面（UNC）特性，仅涉及 **会话侧 NF**（GGSN-C / SGW-C / PGW-C / SMF）——不含接入侧（AMF/MME/SGSN），因为 PD 的决策与协商发生在会话建立之后、由 GGSN-C/PGW-C/SMF 主导：

| 涉及NF | 网元角色 | 接入代际 | 支持版本 | 功能说明 |
|--------|---------|---------|---------|---------|
| GGSN-C | 会话侧（UNC，2/3G） | GPRS/UMTS | UNC 20.5.0及后续版本 | 为 MS 分配 IPv6 delegated-prefix；与 AAA Server 交互获取 PD 网段；经 PGW-U 透传 DHCPv6/ICMPv6 协商 |
| SGW-C / PGW-C | 会话侧（UNC，4G） | EPC | UNC 20.5.0及后续版本 | 为 UE 分配 IPv6 delegated-prefix；经 PGW-U 透传 DHCPv6/ICMPv6 协商 |
| SMF | 会话侧（UNC，5G） | 5GC | UNC 20.5.0及后续版本 | 为 UE 分配 IPv6 delegated-prefix；经 UPF 透传 DHCPv6/ICMPv6 协商；通过 PFCP Session Report 与 UPF 交互 PD 协商信息 |
| Wireless Router（MS/UE） | 终端（路由器角色） | 全代际 | 无特殊要求 | 作为无线移动路由器，承担下挂终端上下行数据转发；从 IPv6 delegated-prefix 段中为下挂终端分配 64 位前缀 |
| SGW-U / PGW-U / UPF | 用户面（UDG） | 全代际 | 无特殊要求 | 使用 IPv6 PD 前缀唯一识别用户；对 IPv6 PD 用户报文转发、计费和策略控制；透传 ICMPv6 RS/RA 与 DHCPv6 报文（实际用户面执行见 GWFD-020406） |
| AAA Server | 外部鉴权/地址服务器 | 全代际 | 无特殊要求 | 对 MS/UE 接入鉴权；Radius 方式下为 MS/UE 分配 IPv6 地址前缀 + IPv6 delegated-prefix |

> ★ 适用NF 表中文档原文仅列"GGSN-C、SGW-C、PGW-C、SMF"四个 NF，**未列接入侧 NF（AMF/MME/SGSN）**——这是本特性与 WSFD-104001/WSFD-104002（均含接入侧 NF）的显著差异，体现了"PD 决策是会话侧职责"的产品架构定位。

## 2. 原理

### 2.1 IPv6 PD 的两种前缀分配方式（★核心决策维度）

UNC（GGSN-C/SGW-C/PGW-C/SMF）为用户提供 **两种** IPv6 delegated-prefix 的分配方式：

| 分配方式 | 前缀来源 | 协商路径 | 应用限制 |
|---------|---------|---------|---------|
| **Radius 方式** | UNC 向 Radius 服务器申请 IPv6 delegated-prefix，再通过 **DHCPv6 方式**将前缀分配给 MS/UE | UNC ↔ AAA Server（Radius）→ MS/UE（DHCPv6） | 同 APN 下 PD 前缀长度可不同（灵活） |
| **本地地址池方式** | UNC 通过本地配置的本地地址池为用户分配 IPv6 delegated-prefix，再通过 **DHCPv6 方式**将前缀分配给 MS/UE | UNC 本地 ADDRPOOL → MS/UE（DHCPv6） | 同 APN 下 PD 前缀长度必须相同（严格） |

> ★ **关键说明（文档原文）**："IPv6 Prefix Delegation 功能**不支持从 DHCP Server 获取并分配** IPv6 delegated-prefix"——即 DHCP 在本特性中仅作为 **向 MS/UE 下发前缀的载体协议**，前缀的"源"只能是 Radius 或本地地址池，不能是 DHCP Server。

> 来源：`特性概述_76459525.md`（"原理概述/系统实现"章节）

### 2.2 PD 协商的协议交互序列（跨代际共性）

PD 协商涉及三类协议交互，顺序固定：

```
阶段1：会话建立 + IPv6 地址前缀分配（含 PD 标记）
  ┌─────────────────────────────────────────────────────────┐
  │ UNC（GGSN-C/PGW-C/SMF） ← AAA Server：鉴权响应            │
  │   ★ AAA 携带 IPv6 Address, 置位 IPv6D,                  │
  │     通过 IPv6 Prefix Delegation Bits field 指示前缀长度   │
  │ UNC（GGSN-C/PGW-C/SMF） → PGW-U/UPF：PFCP Session Estab  │
  │   ★ UNC 携带 IPv6 Address, 置位 IPv6D, 指示前缀长度       │
  │ PGW-U/UPF → UNC：PFCP Session Estab Response            │
  │   ★ UPF 同样支持为 PD 终端分配 IPv6 地址并置位 IPv6D      │
  └─────────────────────────────────────────────────────────┘
                          ↓
阶段2：ICMPv6 协商（UE 获取 UE 本地用的 IPv6 前缀）
  ┌─────────────────────────────────────────────────────────┐
  │ UE/MS → PGW-U/UPF：ICMPv6 Router Solicitation (RS)        │
  │   （发起 IPv6 地址自动配置协商）                          │
  │ PGW-U/UPF → UNC：PFCP Session Report Request             │
  │   （上报 IPv6 地址自动协商请求）                          │
  │ UNC → PGW-U/UPF：PFCP Session Report Response            │
  │ PGW-U/UPF → UE/MS：ICMPv6 Router Advertisement (RA)      │
  │   ★ 携带分配给 UE 的 IPv6 地址前缀                       │
  │   ★ 同时 O-flag 置位为 1，表示后续可通过 DHCPv6 获取其他参数│
  └─────────────────────────────────────────────────────────┘
                          ↓
阶段3：DHCPv6 PD 流程（UE 获取 delegated-prefix，下挂网络用）
  ┌─────────────────────────────────────────────────────────┐
  │ UE/MS → PGW-U/UPF：DHCPv6 Solicit                        │
  │   ★ 携带 IA_PD option + Rapid Commit option +            │
  │     Prefix Exclude Option                                │
  │ PGW-U/UPF ↔ UNC：PFCP Session Report（双向，上报+响应）   │
  │ [完整流程] PGW-U/UPF → UE：DHCPv6 Advertise              │
  │ [完整流程] UE → PGW-U/UPF：DHCPv6 Request                │
  │ [完整流程] PGW-U/UPF ↔ UNC：PFCP Session Report          │
  │ PGW-U/UPF → UE/MS：DHCPv6 Reply                          │
  │   ★ 携带为其分配的 IPv6 delegated-prefix +                │
  │     Prefix Exclude Option                                │
  │ UE/MS → 下挂终端：UE 从 IPv6 delegated-prefix 段中        │
  │   为下挂终端分配 64 位前缀                                │
  └─────────────────────────────────────────────────────────┘
```

> ★ **IPv6D 标记位** 是 PD 的关键协议位：AAA → UNC → UPF 三方均需置位 IPv6D，并通过 IPv6 Prefix Delegation Bits field 指示前缀长度。这是 PD 用户区别于普通 IPv6 用户的协议标识。

> ★ **快速流程 vs 完整流程**：阶段3 中步骤 16-19（DHCPv6 Advertise / Request）在 **快速流程（Rapid Commit）** 中被省略，仅保留 Solicit → Reply 两步。Solicit 消息中的 **Rapid Commit option** 是快速流程的触发条件。

> ★ **PFCP Session Report 双向交互** 是 C-U 协同的核心通道：UPF 收到 UE 的 RS / DHCPv6 报文后，不自行处理，而是通过 PFCP Session Report Request 上报给 UNC；UNC 决策后通过 Response 返回；UPF 再将响应报文透传给 UE。这体现了 PD 决策在 C 面、报文转发在 U 面的分工。

> 来源：`5GC业务流程_59334341.md`、`EPC业务流程_59374325.md`、`GPRS_UMTS业务流程_14374504.md`（"业务流程图"+"图说明"）

### 2.3 PD 业务流程的代际差异（5GC vs EPC vs GPRS/UMTS）

三代际流程结构高度一致（均含 3 阶段 21-22 步），差异主要在网元角色和消息类型：

| 流程要素 | 5GC | EPC | GPRS/UMTS |
|---------|-----|-----|-----------|
| **会话建立消息** | Nsmf_PDUSession_CreateSMContext | Create Session Request（MME→SGW→PGW-C） | Create PDP Context Request（SGSN→PGW-C） |
| **UNC 角色** | SMF | PGW-C | PGW-C（GPRS/UMTS 中 GGSN-C 通过 PGW-C 体现） |
| **U 面网元** | UPF | PGW-U | PGW-U |
| **C-U 接口协议** | PFCP | PFCP | PFCP |
| **终端称呼** | UE | UE | MS |
| **会话/承载类型** | PDU 会话 | PDN 连接 | PDP 上下文 |
| **总步骤数** | 21 步（快速流程）/ 25 步（完整流程） | 22 步（完整流程） | 21 步（完整流程） |
| **PD 协商起始节点** | SMF（步骤 3-4 与 AAA 交互） | PGW-C（步骤 4-5 与 AAA 交互） | PGW-C（步骤 3-4 与 AAA 交互） |

> ★ **代际共性远大于差异**：PD 的协议交互序列、IPv6D 置位逻辑、DHCPv6 协商选项（IA_PD + Rapid Commit + Prefix Exclude）、PFCP Session Report 通道 —— 在三代际中完全一致。差异仅在承载类型名称和网元角色名。

> ★ **GPRS/UMTS 文档说明原文**："无线终端（MS）发起激活请求"——用 MS 而非 UE，且未提 RNC（仅提 SGSN 与 PGW-C），体现 2/3G 的接入网无关性（PD 在 SGSN 之后的会话侧发生）。

> 来源：综合三套代际业务流程文档

### 2.4 IPv6D 协议位的端到端传递（AAA → UNC → UPF）

IPv6D 置位逻辑是 PD 特性区别于普通 IPv6 用户的协议级标识，跨三个网元一致性传递：

| 网元 | 消息 | IPv6D 置位动作 | 标识含义 |
|------|------|---------------|---------|
| **AAA Server**（P1） | 鉴权响应 → UNC | 携带 IPv6 Address，**置位 IPv6D**，通过 IPv6 Prefix Delegation Bits field 指示前缀长度 | AAA 为 PD 终端分配 IPv6 地址并标识为 PD 用户 |
| **UNC**（GGSN-C/PGW-C/SMF）（P2） | PFCP Session Establishment Request → UPF | 携带 IPv6 Address，**置位 IPv6D**，通过 IPv6 Prefix Delegation Bits field 指示前缀长度 | UNC 将 PD 标识传递给 U 面，使 U 面按 PD 用户处理 |
| **UPF / PGW-U**（P3） | PFCP Session Establishment Response → UNC | 携带 IPv6 Address，**置位 IPv6D**，通过 IPv6 Prefix Delegation Bits field 指示前缀长度 | UPF 确认 PD 标识，后续按 PD 用户进行转发、计费、策略控制 |

> ★ P1/P2/P3 的三网元一致性置位是 PD 功能正常工作的协议前提。任一网元未置位 IPv6D，将导致 PD 用户被当作普通 IPv6 用户处理，无法触发后续 DHCPv6 PD 协商。

> 来源：`5GC业务流程_59334341.md`（P1/P2/P3 说明）、`EPC业务流程_59374325.md`（P1/P2/P3 说明）、`GPRS_UMTS业务流程_14374504.md`（P1/P2/P3 说明）

### 2.5 PD 与 IPv6 承载上下文（WSFD-104001）的能力分工

PD 流程的"阶段 1"实际**复用并扩展**了 WSFD-104001 提供的 IPv6 承载建立能力：

| 能力 | WSFD-104001（IPv6 承载上下文） | WSFD-104004（IPv6 前缀代理） |
|------|------------------------------|----------------------------|
| 承载建立 | 决策"是否建立 IPv6 承载" | 不决策承载，复用 WSFD-104001 建立的承载 |
| 地址分配 | 为 UE 本身分配 IPv6 地址/前缀（/64） | 为 UE 下挂网络分配 IPv6 delegated-prefix |
| 路由广播生成 | GGSN/PGW-C/SMF 生成 IPv6 RA（前缀） | **不生成 RA**，由 ICMPv6 RS/RA 触发上报后透传 |
| DHCPv6 | 不涉及 | **核心协议**，通过 DHCPv6 IA_PD 携带 delegated-prefix |
| AAA 鉴权 | 不强制（可选 UDM 静态） | **强制** Radius 鉴权获取 PD 网段 |
| IPv6D 置位 | 不涉及 | **核心标识位**，AAA→UNC→UPF 三方置位 |

> ★ WSFD-104004 的"阶段 1"（会话建立）本质是 WSFD-104001 流程的 PD 增强——在原有 IPv6 承载建立基础上叠加 IPv6D 置位与 AAA PD 网段获取。这就是 WSFD-104004 强依赖 WSFD-104001 的技术根因。

> 来源：综合 WSFD-104004 三代际流程 + WSFD-104001 知识文档 §2.5（5G PDU 会话建立 17 步流程）

### 2.6 C-U 协同分工（与 UDG 侧 GWFD-020406 对称）

```
┌─────────────────────────────────────────────────────────────────┐
│ UNC（C 面）- WSFD-104004                                        │
│ IPv6 PD 决策方与 delegated-prefix 分配方                        │
│                                                                 │
│   核心动作：                                                     │
│   ① 接收 AAA 鉴权响应中的 IPv6 delegated-prefix                  │
│      （IPv6D 置位 + PD Bits field 指示长度）                     │
│   ② 将 IPv6D 标识经 PFCP Session Estab 传给 UPF                  │
│   ③ 处理 UPF 经 PFCP Session Report 上报的                      │
│      ICMPv6 RS / DHCPv6 Solicit 报文                            │
│   ④ 决策 UE 本地用 IPv6 前缀（RA 携带）                          │
│   ⑤ 决策 UE 下挂网络用 delegated-prefix（DHCPv6 Reply 携带）     │
│   ⑥ 触发 UPF 将 RA / DHCPv6 Reply 透传给 UE                     │
│                                                                 │
│   决策结果通过信元/PFCP 携带：                                    │
│     - IPv6D 置位标识（PFCP Session Estab/Report）                │
│     - IPv6 Prefix Delegation Bits field（前缀长度）              │
│     - DHCPv6 IA_PD option（delegated-prefix 内容）              │
│     - RA O-flag=1（指示后续 DHCPv6）                            │
└────────────────────────────┬────────────────────────────────────┘
                             │
                             │ PFCP（Session Estab + Session Report 双向）
                             │ + ICMPv6 RS/RA + DHCPv6 透传
                             ▼
┌─────────────────────────────────────────────────────────────────┐
│ UDG（U 面）- GWFD-020406                                        │
│ IPv6 PD 执行方：标识 PD 用户 + 透传 PD 协商报文                  │
│                                                                 │
│   - 按 C 面下发的 IPv6D 标识识别 PD 用户                         │
│   - 使用 IPv6 PD 前缀唯一识别用户                                │
│   - 透传 UE ↔ UNC 之间的 ICMPv6 RS/RA 报文                      │
│     （UE RS → UPF → PFCP Report → UNC；UNC 决策 → PFCP Resp    │
│      → UPF → RA → UE）                                          │
│   - 透传 UE ↔ UNC 之间的 DHCPv6 报文                            │
│     （Solicit/Advertise/Request/Reply 经 PFCP Report 通道上报）  │
│   - 对 IPv6 PD 用户的数据报文进行转发、计费和策略控制            │
│   - 不参与 PD 决策，不自行生成 delegated-prefix                  │
└─────────────────────────────────────────────────────────────────┘
```

> ★ **对称性说明**：WSFD-104004（UNC/C 面）与 GWFD-020406（UDG/U 面）构成 PD 功能的 C-U 分工——C 面做"PD 决策 + 前缀分配"，U 面做"PD 用户识别 + 报文透传 + 转发计费"。这与 WSFD-104001 ↔ GWFD-020401（IPv6 承载 C-U 分工）形成**对称架构**。

> 来源：综合 WSFD-104004 三代际流程文档（C 面动作）+ `特性概述_76459525.md`（U 面职责描述）

## 3. 激活

### 3.1 激活通用前置条件

- 已阅读 `WSFD-104004 IPv6前缀代理特性概述`
- 已完成 **激活 RADIUS 功能**（强依赖 WSFD-011306）
- 已完成加载 License
- AAA Server 上已配置好分配给终端设备使用的 IP 地址段
- 已配置好 VPN 实例、APN、本地地址池（如使用 Local 分配方式）

### 3.2 激活步骤

激活流程共 5 个主步骤（含 License 开关 + 3 个配置块 + 1 个 APN 属性块）：

```
步骤1：进入 "MML命令行-UNC" 窗口

步骤2：打开本特性的 License 开关
  └── SET LICENSESWITCH:LICITEM="LKV3W9V6PD11",SWITCH=ENABLE;

步骤3：配置 IPv6 VPN（3 个子步骤）
  3a. 创建 VPN 实例
      └── ADD VPNINST: VPNINSTANCE="vpn1";
  3b. 使能 IPv6 地址族
      ├── ADD L3VPNINST: VRFNAME="vpn1";
      └── ADD VPNINSTAF: VRFNAME="vpn1", AFTYPE=ipv6uni;
  3c. 配置路由标识符
      └── MOD VPNINSTAF: VRFNAME="vpn1", AFTYPE=ipv6uni, VRFRD="65512:2";

步骤4：配置分配 IPv6 代理前缀（地址池）
  4a. 配置 IPv6 地址前缀名 + IPv6 前缀关联的 VPN 实例
      └── ADD ADDRPOOL: POOLNAME="pool01", POOLTYPE=Local,
                        IPVERSION=IPv6, HASVPN=Enable,
                        VPNINSTANCE="vpn1";
  ★ 备注1：如果 APN 使能了 L2TP 功能，则无需开启 IPv6 PD 功能
  ★ 备注2：在功能开启前，需要在 AAA Server 上配置好分配给
            终端设备使用的 IP 地址段

步骤5：配置 APN 下的地址分配属性（3 个子步骤）
  5a. 指定 APN 名称
      └── ADD APN: APN="huawei.com";
  5b. 配置地址分配属性（本地 Local 或 Radius）
      └── SET APNADDRESSATTR: APN="huawei.com", IPV6ALLOCTYPE=LOCAL;
  5c. 配置 APN 所属的 IPv6 APN（VPN 绑定）
      └── MOD APN: APN="huawei.com", HASVPNIPV6=ENABLE,
                   VPNINSTANCEIPV6="vpn1";
```

> ★ 与 WSFD-104001（仅 License 开关，零配置）相比，本特性激活步骤显著更复杂——需配置 VPN 实例、地址池、APN 地址分配属性三个配置块，是地址分配域中激活最重的特性之一。

> ★ **适用 NF 统一**：激活文档明确声明"适用于 GGSN-C、SGW-C、PGW-C、SMF"——四个会话侧 NF 共享同一套激活流程，无代际差异化的激活命令。

> 来源：`激活IPv6前缀代理_76459527.md`（"操作场景"+"必备事项"+"操作步骤"+"任务示例"章节）

## 5. 配置案例

### 5.1 典型场景：APN huawei.com 使能 IPv6 PD（本地地址池方式）

**场景描述**：使能 APN "huawei.com" 的 IPv6 Prefix Delegation 功能，配置**本地地址池**分配 IPv6 代理前缀，绑定的 VPN 实例为 "vpn1"。适用于家庭或小型企业用户通过 UE（作为无线移动路由器）下挂多台终端接入网络的场景。

**MML 命令序列（原样保留）**：

```
//打开本特性的License配置开关。

SET LICENSESWITCH:LICITEM="LKV3W9V6PD11",SWITCH=ENABLE;

//创建IPv6 VPN实例。

ADD VPNINST
: VPNINSTANCE="vpn1";
ADD L3VPNINST: VRFNAME="vpn1";
ADD VPNINSTAF: VRFNAME="vpn1", AFTYPE=ipv6uni;
MOD VPNINSTAF: VRFNAME="vpn1", AFTYPE=ipv6uni, VRFRD="65512:2";

//配置本地地址池分配IPv6代理前缀。

ADD ADDRPOOL
: POOLNAME="pool01", POOLTYPE=Local, IPVERSION=IPv6, HASVPN=Enable, VPNINSTANCE="vpn1";

//配置APN下的IP地址分配属性。

ADD APN
: APN="huawei.com";

SET APNADDRESSATTR
: APN="huawei.com", IPV6ALLOCTYPE=LOCAL;

MOD APN
: APN="huawei.com", HASVPNIPV6=ENABLE, VPNINSTANCEIPV6="vpn1";
```

> 来源：`激活IPv6前缀代理_76459527.md`（"任务示例/脚本"）

### 5.2 场景变体对照

| 变体 | 分配方式 | 核心差异 | 关键命令变化 |
|------|---------|---------|------------|
| 本地地址池 PD（基础场景） | LOCAL | 前缀来自 UNC 本地地址池，同 APN 前缀长度必须相同 | `SET APNADDRESSATTR: IPV6ALLOCTYPE=LOCAL` |
| Radius 动态 PD | RADIUS | 前缀来自 AAA Server 动态下发，同 APN 前缀长度可不同 | `SET APNADDRESSATTR: IPV6ALLOCTYPE=RADIUS` + 需在 UNC 配置 AAA 规划的 PD Radius 地址池 |
| 双栈 + PD 叠加 | LOCAL 或 RADIUS | UE 同时获得 IPv4 地址 + IPv6 前缀 + IPv6 delegated-prefix | 必须先激活 WSFD-104002（双栈）+ WSFD-104001（IPv6 承载）+ 本特性，三特性叠加 |
| 5GC SMF PD | LOCAL 或 RADIUS | UNC 角色=SMF，承载=PDU 会话，U 面=UPF | 激活命令完全相同（跨代际统一） |
| EPC PGW-C PD | LOCAL 或 RADIUS | UNC 角色=PGW-C，承载=PDN 连接，U 面=PGW-U | 激活命令完全相同 |
| GPRS/UMTS GGSN-C PD | LOCAL 或 RADIUS | UNC 角色=GGSN-C（文档中常以 PGW-C 体现），承载=PDP 上下文 | 激活命令完全相同 |

### 6.6 常见问题与排查

| 问题现象 | 可能原因 | 排查方法 |
|---------|---------|---------|
| DHCPv6 Reply 未携带 IPv6 代理前缀 | License 开关未打开 | LST LICENSESWITCH 核对 LKV3W9V6PD11；SET LICENSESWITCH SWITCH=ENABLE |
| DHCPv6 Reply 未携带 IPv6 代理前缀 | 未激活 WSFD-104001 IPv6 承载上下文（强依赖） | 核对 LKV2IPV6AM01/LKV2IPV6SM01/LKV2IPV601 是否已激活 |
| DHCPv6 Reply 未携带 IPv6 代理前缀 | 未配置 WSFD-011306 Radius 功能（强依赖） | 核对 Radius 功能配置；AAA Server 是否可达 |
| DHCPv6 Reply 未携带 IPv6 代理前缀 | APN 的 IPV6 分配方式未配置或配置错误 | LST APNADDRESSATTR 核对 IPV6ALLOCTYPE（LOCAL/RADIUS） |
| DHCPv6 Reply 未携带 IPv6 代理前缀 | 本地地址池配置错误（IPVERSION/POOLTYPE） | LST ADDRPOOL 核对 IPVERSION=IPV6、POOLTYPE=Local |
| DHCPv6 Reply 未携带 IPv6 代理前缀 | Radius 方式下 AAA 未动态下发 PD 前缀 | 核对 AAA Server 是否在 Radius 地址池中规划并下发 PD 网段 |
| 双栈 + PD 叠加场景 PD 失败 | 未激活 WSFD-104002 IPv4v6 双栈接入 | 核对双栈特性 License 与激活状态 |
| Local 方式下多用户前缀长度冲突 | 同 APN 下 Local 地址池前缀长度不一致 | 统一同 APN 下 Local 地址池前缀长度（应用限制 1） |
| 掉话率上升（弱覆盖场景） | UE 无法通过 UPF 透传 RS 报文（eNodeB 不下发） | 已知系统影响（§1.9），需评估弱覆盖场景部署 |
| L2TP APN 场景 PD 不生效 | L2TP APN 无需开启 PD | 已知排除约束（激活文档步骤4备注1） |
