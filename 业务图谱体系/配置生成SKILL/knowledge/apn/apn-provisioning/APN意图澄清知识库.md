# APN意图澄清知识库

> 本文档用于意图澄清阶段，提供场景化问答引导，帮助用户将模糊的业务需求转化为确定的配置方案。
>
> **路径说明**：本文档中所有路径均相对于SKILL.md所在目录。
>
> **原理说明**：各配置选项的详细技术原理见 `references/00-配置原理知识.md`，SKILL执行时会根据需要加载相应章节。

## 核心原则

- **先理解，后提问**：从用户描述中提取业务场景，而不是直接问配置选项
- **场景化引导**：用业务语言提问，而非技术术语
- **渐进式澄清**：每轮只问一个问题，逐层深入
- **提供上下文**：解释每个选项的含义和影响
- **灵活变通**：根据用户意图的不同，灵活变通提问，不要每次只根据当前文档，多结合`references/00-配置原理知识.md`知识引导用户。允许多层嵌套加载，因此`references/00-配置原理知识.md`中也会标明其他细节详见其他文档

## 业务场景分类

### 场景1：企业专网/工业互联网

**典型用户**：工厂、园区、企业总部
**典型设备**：工控设备、传感器、PLC、办公终端
**业务需求**：

- 设备需要访问企业内网服务器
- 数据需要在公网上传输，但需要安全保护
- 通常需要固定IP地址

**引导策略**：先确认是否需要固定IP，再确认安全传输需求，最后确认鉴权方式

### 场景2：物联网（IOT）

**典型用户**：电力、水务、燃气、农业
**典型设备**：智能电表、水表、燃气表、农业传感器
**业务需求**：

- 大量设备上报数据到云平台
- 通常不需要访问内网，只需要上报
- 地址通常自动获取

**引导策略**：通常使用动态地址+透明接入，重点确认是否需要按区域管理

### 场景3：通用Internet访问

**典型用户**：普通移动用户、CPE终端
**典型设备**：手机、随身WiFi、家庭网关
**业务需求**：

- 访问Internet
- 地址自动获取
- 通常不需要二次鉴权

**引导策略**：通常使用UPF分配+透明接入+双栈，重点确认地址类型需求

### 场景4：IMS语音业务

**典型用户**：运营商VoLTE/VoNR业务
**典型设备**：支持语音的5G/4G终端
**业务需求**：

- IMS注册和呼叫
- 需要特殊的接入控制

**引导策略**：通常使用SMF动态分配+透明接入+双栈

### 一、地址分配决策

> 详见 `references/00-配置原理知识.md` 第1章 地址分配原理

#### 1.1 地址分配方式概览

| 分配方式        | 分配主体             | 地址类型       | 适用场景                                 |
| --------------- | -------------------- | -------------- | ---------------------------------------- |
| **UPF分配**     | UDG（UPF）本地地址池 | IPv4/IPv6/双栈 | 普通移动用户、区域化管理、按业务类型分配 |
| **SMF分配**     | UNC（SMF）本地地址池 | IPv4/IPv6/双栈 | SMF精确控制场景、UPF不支持地址分配时     |
| **UDM分配**     | 签约数据             | 静态地址       | 企业专网、固定IP需求、工业模组           |
| **RADIUS分配**  | AAA服务器            | 动态/静态      | AAA统一管控、企业园区鉴权+地址绑定       |
| **DHCP/DHCPv6** | 外部DHCP服务器       | IPv4/IPv6      | 传统企业DHCP基础设施、地址资源节省       |
| **LNS分配**     | L2TP LNS服务器       | IPv4/IPv6/双栈 | L2TP VPN远程办公、隧道接入企业内网       |

#### 1.2 意图澄清问题设计

**基础问题**：

```
设备的IP地址是怎么分配的？
① 在开户时就分配好固定的，每次都用这个地址（→ UDM分配）
② 每次连接时从网络自动获取一个新地址（→ 动态分配）
③ 由AAA服务器控制分配（→ RADIUS分配）
④ 企业已有DHCP服务器，由DHCP分配（→ DHCP分配）
⑤ 通过L2TP隧道，由企业LNS服务器分配（→ LNS分配）
```

**澄清问题（针对动态分配）**：

```
动态分配时，IP地址是从哪里来的？
① 从UPF的地址池自动分配（按业务类型/APN分配）
② 从UPF的地址池自动分配（按UPF设备分配）
③ 从UPF的地址池自动分配（按位置区域分配）
④ 从SMF的地址池自动分配
```

**位置区追问**：

```
是否需要按用户所在位置分配不同的IP地址？
（例如：不同城市的用户分配不同网段的地址）
① 是的，需要按位置分配（→ 基于位置区分配）
② 不需要，统一从同一个地址池分配
```

**AAA控制追问**：

```
IP地址是否需要由企业的AAA服务器统一控制分配？
① 是的，AAA服务器决定分配哪个地址（→ RADIUS分配）
② 不需要，网络自动分配即可
```

#### 1.3 详细决策矩阵

| 用户业务需求                     | 推荐分配方式               | 配置树路径                       | 技术依据                                   |
| -------------------------------- | -------------------------- | -------------------------------- | ------------------------------------------ |
| 企业设备需要固定IP，便于识别管理 | UDM静态分配                | 地址分配信息:UDM分配             | IP地址在**HSS/UDM签约时确定**[00-1.4]      |
| 普通移动用户上网，动态地址       | UPF动态分配（基于SMF）     | 地址分配信息:UPF分配:基于SMF     | UPF**本地地址池动态分配**[00-1.2]          |
| 物联网设备，按APN业务类型管理    | UPF动态分配（基于APN/DNN） | 地址分配信息:UPF分配:基于APN/DNN | 按APN/DNN绑定地址池分配[00-1.2]            |
| 需要按位置区域分配地址           | UPF分配（基于位置区）      | 地址分配信息:UPF分配:基于位置区  | 根据用户LAC/TAC位置分配地址[00-1.2.1]      |
| SMF需要精确控制地址分配          | SMF分配                    | 地址分配信息:SMF分配:*           | SMF**本地地址池分配**[00-1.3]              |
| AAA服务器统一管控分配            | RADIUS分配                 | 地址分配信息:Radius分配          | 鉴权过程中从**AAA服务器获取地址**[00-1.5]  |
| 企业已有DHCP基础设施             | DHCP/DHCPv6分配            | 地址分配信息:DHCP分配            | UNC作为DHCP Client**代理用户申请**[00-1.6] |
| L2TP VPN远程办公                 | LNS分配                    | 地址分配信息:LNS分配             | LNS服务器通过**PPP协商分配**[00-1.7]       |

#### 1.4 UPF分配三种子场景对比

| 维度            | 基于APN/DNN    | 基于SMF                       | 基于位置区              |
| --------------- | -------------- | ----------------------------- | ----------------------- |
| **映射对象**    | APN/DNN标识    | SMF Node ID                   | LAC/TAC位置区           |
| **适用场景**    | 按业务类型分配 | 按UPF选择分配（CU Full Mesh） | 按区域分配              |
| **配置复杂度**  | 中             | 低                            | 高（需License）         |
| **移动性影响**  | 无             | 无                            | 用户移动**需重新分配**  |
| **全局开关**    | 无             | `SET IPALLOCBYSMFGLBSW`       | `SET IPALLOCBYLOCGLBSW` |
| **License要求** | 否             | 否                            | 是（LKV3G5LBAA01）      |

#### 1.5 SMF分配三种子场景对比

| 维度         | 基于APN/DNN       | 基于UPF                 | 基于位置区              |
| ------------ | ----------------- | ----------------------- | ----------------------- |
| **映射对象** | APN/DNN           | UPF实例                 | TAC/N2TAC位置区         |
| **适用场景** | 按业务类型选择UPF | 按UPF选择UPF            | 按区域选择UPF           |
| **全局开关** | 无                | `SET IPALLOCBYUPFGLBSW` | `SET IPALLOCBYLOCGLBSW` |

#### 1.6 地址分配方式原理总结

| 分配方式       | 原理特点                                           | 地址来源            |
| -------------- | -------------------------------------------------- | ------------------- |
| **UPF分配**    | UPF用户面本地为UE分配地址，地址可动态复用          | UPF本地地址池       |
| **SMF分配**    | SMF控制面从本地地址池分配，去激活后地址可延迟释放  | SMF本地地址池       |
| **UDM分配**    | UE使用IP地址在HSS/UDM签约时确定的静态IP地址        | UDM签约数据（静态） |
| **RADIUS分配** | PDU会话激活阶段，用户鉴权过程中从AAA服务器获取地址 | AAA服务器下发       |
| **DHCP分配**   | SMF作为DHCP Client代理用户向DHCP Server申请地址    | 外部DHCP服务器      |
| **LNS分配**    | 通过L2TP隧道与LNS服务器交互，PPP协商过程中分配地址 | LNS服务器（PPP）    |

### 二、接入方式决策

> 详见 `references/00-配置原理知识.md` 第3章 接入隧道原理

#### 2.1 意图澄清问题设计

**基础问题**：

```
这些设备的数据要通过Internet传到企业内网吗？
① 是的，需要通过公网传输
② 不走公网，走专线连接

如果需要通过公网传输，数据需要加密保护吗？
① 需要加密（重要数据，安全要求高）→ IPSec隧道
② 轻度加密或不需要（性能优先）→ GRE隧道
③ 不需要，直接通过NAT转换即可 → VPN
```

#### 2.2 详细决策矩阵

| 业务场景     | 网络环境         | 推荐接入方式 | 配置树路径         | 技术依据                                |
| ------------ | ---------------- | ------------ | ------------------ | --------------------------------------- |
| 企业专网     | 公网传输，需加密 | IPSec隧道    | 接入方式信息:IPSec | **IPSec**提供数据加密和认证[00-3.2]     |
| 企业专网     | 公网传输，轻量级 | GRE隧道      | 接入方式信息:GRE   | **GRE**仅封装不加密，性能开销低[00-3.1] |
| 企业专网     | 专线连接         | 直连（VPN）  | 接入方式信息:VPN   | 专线物理隔离，无需隧道封装              |
| 物联网       | 公网传输         | 直连（NAT）  | 接入方式信息:VPN   | 设备仅上报数据，通过NAT转换访问Internet |
| 通用Internet | -                | 直连（VPN）  | 接入方式信息:VPN   | 直接访问Internet，无需隧道              |

#### 2.3 隧道技术对比

| 特性           | GRE            | IPSec                |
| -------------- | -------------- | -------------------- |
| **加密**       | 无             | ESP提供加密          |
| **性能开销**   | 低             | 较高（加密处理）     |
| **安全性**     | 无（仅封装）   | 高（加密+认证）      |
| **适用场景**   | 轻量级隧道需求 | 需数据加密的安全传输 |
| **配置复杂度** | 简单           | 较复杂（证书/IKE）   |
| **规格限制**   | 最大4k个隧道   | -                    |
| **嵌套限制**   | 最多两层嵌套   | -                    |

### 三、鉴权方式决策

> 详见 `references/00-配置原理知识.md` 第2章 鉴权接入原理

#### 3.1 意图澄清问题设计

**基础问题**：

```
用户接入网络时需要输入用户名密码吗？
① 不需要，SIM卡合法即可直接接入 → 透明接入
② 需要输入企业统一分配的用户名密码 → 不透明接入
③ 不需要输入，但需要网络统一鉴权 → 透明鉴权
④ 没有AAA服务器，简单本地鉴权即可 → 本地鉴权
```

#### 3.2 详细决策矩阵

| 用户回答              | 鉴权方式   | 配置树路径   | Radius鉴权接入   | Radius功能 | 技术依据                                 |
| --------------------- | ---------- | ------------ | ---------------- | ---------- | ---------------------------------------- |
| SIM卡合法直接接入     | 透明接入   | 鉴权计费信息 | ✅ TRANS_NON_AUTH | ❌ 不配置   | HSS/UDM已完成鉴权[00-2.1]                |
| 需要AAA统一用户名密码 | 透明鉴权   | 鉴权计费信息 | ✅ TRANS_AUTH     | ✅ 配置     | 使用**网络侧配置的用户名和密码**[00-2.1] |
| 企业AAA二次鉴权       | 不透明接入 | 鉴权计费信息 | ✅ NON_TRANS      | ✅ 配置     | 用户通过PCO携带用户名密码[00-2.1]        |
| 本地简单鉴权          | 本地鉴权   | 鉴权计费信息 | ✅ LOC_AUTH       | ❌ 不配置   | UNC**本地配置用户名密码**[00-2.1]        |

#### 3.3 鉴权方式原理对比

| 鉴权方式       | 是否需要Radius服务器 | 用户是否输入 | 适用场景                           |
| -------------- | -------------------- | ------------ | ---------------------------------- |
| **透明接入**   | 否                   | 否           | 运营商直接提供业务，如Internet访问 |
| **透明鉴权**   | 是                   | 否           | 运营商统一鉴权，用户无感知但可计费 |
| **不透明接入** | 是                   | 是           | 企业APN，ISP/企业对用户权限有要求  |
| **本地鉴权**   | 否                   | 是           | 无AAA服务器时的简单鉴权            |

#### 3.4 Radius功能与鉴权方式关联

| 鉴权方式   | 是否需要Radius功能 |
| ---------- | ------------------ |
| 透明接入   | 否                 |
| 透明鉴权   | 是                 |
| 不透明接入 | 是                 |
| 本地鉴权   | 否                 |

> **注意**：PPP用户暂不支持本地鉴权功能[00-2.1]

### 四、地址类型决策

> 详见 `references/00-配置原理知识.md` 第1.1章 地址分配方式概览

#### 4.1 意图澄清问题设计

**基础问题**：

```
您的设备支持哪种IP地址？
① 只支持IPv4
② 只支持IPv6
③ 都支持（双栈）
④ 不确定

您的网络侧支持哪种地址类型？（可多选）
① 只支持IPv4
② 只支持IPv6
③ 都支持（双栈）
```

#### 4.2 地址类型决策矩阵

| 终端支持  | 网络支持 | 推荐类型       | 说明               |
| --------- | -------- | -------------- | ------------------ |
| IPv4 only | IPv4     | IPv4           | 兼容性最优         |
| IPv6 only | IPv6     | IPv6           | 兼容性最优         |
| 双栈终端  | 双栈     | IPv4v6（双栈） | 灵活性和兼容性最佳 |
| IPv4 only | 双栈     | IPv4           | 避免v6不可用问题   |
| 未知      | IPv4     | IPv4           | 兼容性优先         |
| 未知      | 双栈     | IPv4v6         | 灵活性优先         |

## 常见场景方案速查

| 业务场景                 | 典型描述                   | 地址分配              | 鉴权       | 地址类型 | 推荐接入方式 | 依据说明                                          |
| ------------------------ | -------------------------- | --------------------- | ---------- | -------- | ------------ | ------------------------------------------------- |
| **工厂工控设备访问内网** | 设备需固定IP，访问企业内网 | UDM静态分配           | 不透明接入 | IPv4     | IPSec隧道    | 企业设备需固定IP，公网传输需加密，企业AAA二次鉴权 |
| **智慧农业传感器上报**   | 大量设备上报数据到云平台   | UPF动态（基于APN）    | 透明接入   | IPv4     | VPN(NAT)     | 设备仅上报数据，动态地址，SIM卡合法即可接入       |
| **家庭CPE宽带**          | 普通Internet访问           | UPF动态（基于SMF）    | 透明接入   | IPv4v6   | VPN(NAT)     | 普通Internet访问，双栈满足IPv4/IPv6终端           |
| **VoLTE语音业务**        | IMS注册和呼叫              | SMF动态分配           | 透明接入   | IPv4v6   | VPN          | IMS语音需SMF精确寻址，双栈支持                    |
| **运营商热点覆盖**       | 不同区域热点按位置管理     | UPF动态（基于位置区） | 透明接入   | IPv4     | VPN          | 按区域分配地址，便于计费和区域化管理              |
| **企业AAA二次鉴权**      | 企业AAA服务器统一管控      | RADIUS分配            | 不透明接入 | IPv4     | VPN或GRE     | AAA服务器控制鉴权和地址分配，支持企业安全策略     |
| **传统企业DHCP迁移**     | 企业已有DHCP基础设施       | DHCP分配              | 透明接入   | IPv4     | VPN          | 复用企业现有DHCP服务器，无需新建地址池            |
| **企业L2TP VPN**         | 远程办公，隧道接入内网     | LNS分配               | 不透明接入 | IPv4v6   | L2TP隧道     | L2TP隧道+LNS验证保证访问安全，地址由LNS分配       |
| **区域化运营管理**       | 不同城市分配不同网段       | UPF动态（基于位置区） | 透明接入   | IPv4     | VPN          | 按位置分配地址，支持区域化运营和精准营销          |
