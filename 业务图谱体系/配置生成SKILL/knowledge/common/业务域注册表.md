## 4. Phase 0 两层路由规则

### 4.1 Phase 0a 业务域识别
1. 从用户需求提取关键词，按 §1 业务域注册表"识别关键词"匹配
2. 多域命中按主诉求判断；不确定则向用户确认
3. 未命中 → 提示"未匹配到业务域，请补充属于哪个业务域（业务感知 / APN接入 / …）"
4. 锁定 `{业务域}`（如 business-awareness）

### 4.2 Phase 0b 子场景识别
1. 在锁定业务域的子场景表（§2 / §3 / …）中按"识别关键词"匹配
2. 多场景命中按主诉求判断（如业务感知域下"计费+限速"→ 主诉求计费，限速属计费 FUP）
3. 未命中 → 提示用户补充是该域下的哪个子场景
4. 锁定 `{子场景}`（如 charging）

### 4.3 跨域/跨场景组合
一条业务流可命中多域/多场景规则（如业务感知计费 + 带宽叠加，或业务感知 + APN 接入条件）。处理：主域/主场景路由，次要作协同约束；配置按场景分别输出。

## 5. 子场景知识模块标准结构

每个 `knowledge/{业务域}/{子场景}/` 必须含：

```
{子场景}/
├── three-layer-graph/   (00-overview ~ 06-evidence，7 文件；★Phase 1 入口用 00-overview.md)
├── ref-phase/           (ref-phase1 场景识别 / ref-phase3 参数收集 / ref-phase5 配置生成 / ref-phase6 核查规则)
├── kb/                  (配置参数库：命令/参数/约束/特殊规则)
└── feature/             (特性知识 + cross-feature-analysis)
```

> 不同业务域的对象体系可能差异较大（如 APN 的地址池/隧道对象 vs 业务感知的 PCC/URR 对象），各域独立建模，共享对象见各域子场景表说明。
