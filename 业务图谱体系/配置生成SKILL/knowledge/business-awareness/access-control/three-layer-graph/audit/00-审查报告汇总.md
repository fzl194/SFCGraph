### 2.2 核心结论

图谱**骨架设计可信、四轨架构表达清晰、Schema §13 禁止关系全部合规、EV ID 双层命名机制（06 §0 别名表）是一项好的工程实践**，但存在 **3 类系统性缺陷**：

1. **★ FTOE Task 编号系统性错位**（最严重）：F02 §6 的 25 条 FTOE 用 `T-AC-201~504` 编号，而 F03 只定义 `T-AC-101~109`，两者含义完全错位（如 F02 的 `T-AC-102`=配置ADC规则，但 F03 的 `T-AC-102`=配置头增强对象）。这是端到端链路的最大断裂点，F00/F05 的链路示例反而引用了 F03 的编号（T-AC-101~109），形成"03 与 00/05 一致、02 孤立"的三方分裂。
2. **证据 ID `EV-CA-D1~D4` 未注册**：F03 的所有 TaskCommandOrderEdge（22 个 EV-CA-D1、8 个 EV-CA-D2、31 个 EV-CA-D3、6 个 EV-CA-D4）和 F05 链路示例引用了 `EV-CA-D1~D4`（合计 70 处），但 F06 未将它们注册为独立 EvidenceSource（只在 F03 §0.4 描述为"附录 D 的 4 类端到端流程"）。严格按 Schema §8.11 应当注册。
3. **CS-AC-07 引用了未实例化的 Feature**：F01 §2.7 / F05 §1 的 CS-AC-07 uses_feature 列出 `WSFD-106003 / WSFD-105003 / WSFD-106005 / WSFD-105006` 4 个 Feature，但 F02 仅实例化 15 个 Feature（UDG 11 + UNC 4），这 4 个 UNC 特性未在 F02 建模，导致跨层引用断裂。

## 3. HIGH 级问题清单（必须修，共 9 项）

### U-H-01 ★ FTOE Task 编号系统性错位（最严重跨层断裂）

- **位置**：F02 §6.1~§6.5（FTOE-AC-001~025，25 条编排边）vs F03 §1~§4（ConfigTask 定义）
- **问题描述**：
  - F02 §6 的 FTOE 使用 `T-AC-201~204 / T-AC-301~303 / T-AC-401~405 / T-AC-501~504` 共 **22 个 Task ID** 作为 `from_task_ref` / `to_task_ref`。
  - 但 F03 只实例化 `T-AC-101~109` 共 9 个 Task，**完全没有 T-AC-201~504**。
  - 更严重的是，F02 与 F03 对相同编号的 Task 含义完全不同：
    | Task ID | F02 §6 Task 说明 | F03 Task 定义 |
    |---------|----------------|--------------|
    | T-AC-101 | 配置三四层过滤（FILTER+FLOWFILTER+FLTBINDFLOWF） | **配置 ADC 检测参数与规则** |
    | T-AC-102 | 配置 ADC 规则（ADCPARA + RULE:POLICYTYPE=ADC） | **配置头增强对象**（HEADEN，跨4特性） |
    | T-AC-103 | 配置计费属性（URR/URRGROUP/PCCPOLICYGRP） | **配置 HTTP 智能重定向** |
    | T-AC-301~504 | EXTENDEDFILTER / SMARTHTTPREDIR / ICAP / USRLOCATION 等 | **不存在** |
  - F00 §5 链路示例和 F05 §7 端到端链路引用的是 F03 的编号（T-AC-101~109），形成"**03 与 00/05 一致、02 孤立**"的三方分裂。
- **Schema 依据**：§9.7 `FeatureTaskOrderEdge.from_task_ref` / `to_task_ref` 必须指向真实存在的 ConfigTask；§7 端到端链路四条路径必须成立。
- **严重级**：**HIGH**（端到端链路在 F02 处完全断裂，影响所有依赖 FTOE 的下游应用如配置生成、影响分析）
- **修复建议**（P3，大工作量）：
  - 方案 A（推荐）：统一 F02 §6 FTOE 的 `from_task_ref` / `to_task_ref` 到 F03 已定义的 `T-AC-101~109`，并重新映射 FTOE-AC-001~025 的链路逻辑。例：GWFD-020357 ADC 链应改为 `T-AC-007 → T-AC-008 → T-AC-101(ADC) → T-AC-107(PCC动作组) → T-003(PCC规则) → T-004(用户模板) → T-006(刷新)`。
  - 方案 B：在 F03 中补充 T-AC-201~504 的 Task 定义（但这会与 T-AC-101~109 含义冲突，不推荐）。
  - 同步修正 F02 §6 各 FTOE 子节的 "Task 说明" 文字，确保与 F03 定义一致。
  - 修复后需重新核查 F05 §3 decomposes_to 表与 F02 §6 的一致性。

## 7. 已知问题专项（EV ID 命名异类、双轨/四轨表述）的统一建议

### 7.1 EV ID 命名异类统一建议

- **现状**：
  - 01 / 02 / 03 使用 `EV-FK-AC-NN`（序号形式，NN=01~15）。
  - 04 使用 `EV-FK-AC-XXXXXX`（特性编号形式，如 EV-FK-AC-110471）。

- **影响评估**：
  - 当前可追溯性已通过别名表保证（06 §6.2 核查通过）。
  - 但跨场景合并时（计费用 EV-FK-NN、带宽带用 EV-FK-NN、访问限制用 EV-FK-AC-NN + EV-FK-AC-XXXXXX），命名空间易冲突。
- **统一建议**（P2 快捷 / P3 大工作量）：
