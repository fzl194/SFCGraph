## 1. SA 识别前置（业务感知域规则 §5）

访问限制场景**所有 L7 动作特性都直接依赖 SA-Basic**（GWFD-110101）的协议解析能力，SA 是整个场景的数据基础。

### 1.1 SA 依赖矩阵（来自 cross-feature-analysis §2.1）

| 特性 | SA 依赖说明 |
|------|-----------|
| ADC（GWFD-020357） | 基于 SA 执行 L7 业务检测 |
| HTTP 头增强（GWFD-110261） | 需要对 HTTP 报文进行解析 |
| RTSP 头增强（GWFD-110262） | 需要对 RTSP 报文进行解析 |
| HTTPS 头增强（GWFD-110263） | 需要对 HTTPS 报文进行解析 |
| Portal（GWFD-110281） | SA-Basic + SA-Web Browsing + SA-Mobile |
| WebProxy（GWFD-110282） | SA 解析获得 Web Proxy 动作和重定向地址 |
| DNS 纠错（GWFD-110283） | SA 解析获得 URL + SA-Network Administration 解析 DNS |
| HTTP 智能重定向（GWFD-110284） | SA 解析和规则匹配（SA-Web Browsing） |
| 头防欺诈（GWFD-110401） | 识别和解析 HTTP 报文（SA-Web Browsing） |
| URL 过滤（GWFD-110471） | SA 解析获得 URL（SA-Web Browsing + SA-Mobile + HTTP2.0 Host 识别） |

### 1.2 SA 数据层次

```
SA 特征库（由 LOD SIGNATUREDB 加载）
    │
    ▼
SA-Basic 引擎（GWFD-110101，License: 82209749 LKV3G5SABS01）
    ├── L3/L4 解析（IP/端口/协议） → FILTER 匹配
    ├── 协议识别（知名端口 + 签名匹配） → PROTBINDFLOWF 绑定
    └── L7 解析（URL / Method / Response Code / SNI） → L7FILTER / EXTENDEDFILTER 匹配
    │
    ▼
业务标识输出（SVC/APP、URL、Method、UserAgent、Content-Type）
    │
    ▼
供 ADC / 头增强族 / 重定向族 / URL 过滤 / 防欺诈 等特性使用
```

### 1.3 SA 命令（T-008，generic Task）

```mml
LOD SIGNATUREDB: ...;  (加载 SA 特征库，ADC/头增强族/Portal/DNS纠错/防欺诈/URL过滤共用)
LOD PARSERDB: ...;  (加载协议解析库，头防欺诈显式依赖)
```

## 3. 规则匹配机制

### 3.1 类型独立原则（BR-AC-02）

不同类型规则（PCC/BWM/HEADEN/WEBPROXY/SMARTREDIRECT/ADC）匹配**独立进行**：
- 每种类型至多一条生效
- 不同类型可叠加执行（如阻塞 + 限速 + 插头同时命中一条报文）
- 这是访问限制可与计费、带宽控制并存的基础

### 3.2 一条规则绑定一条策略（BR-AC-03）

- 规则与策略是 **1:1 关系**
- 策略与动作类型是 **1:1 关系**
- 若要同一流量同时执行阻塞和重定向，必须通过**两条不同优先级的规则**分别承载

### 3.3 优先级规则（业务感知域规则 §3）

- **PRIORITY 数字越小，优先级越高**（固定规则，不可更改）
- "第 N 优先" = 第 N 高优先级 = PRIORITY 数值第 N 小
- 分析前必须用 Grep 提取现网**所有 RULE 的 PRIORITY**（含非本场景 RULE），按数值排序后定位

### 3.4 ★ 双轨并存时的优先级考量（CR-AC-14）

- 轨道 A（PCC 体系）与轨道 B（URL 过滤体系）可并存于同一用户业务流
- 双轨 RULE 独立匹配：PCC 类型 RULE 与 URL 过滤 RULE 用 POLICYTYPE=PCC 区分（但动作体系不同）
- WebProxy 必须配两条 RULE（POLICYTYPE=WEBPROXY + POLICYTYPE=PCC），RULENAME 不能同名（CR-AC-01）
- 优先级冲突时执行结果取决于配置优先级和规则匹配顺序（CR-AC-14），需现网验证

## 4. REFRESHSRV 时序约束（BR-AC-05, TR-AC-01, CR-AC-12/13）

### 4.1 必须最后执行（业务感知域规则 §1.1）

- UDG 侧策略变更（FILTER/FLOWFILTER/L7FILTER/EXTENDEDFILTER/RULE 等）后必须执行 `SET REFRESHSRV:REFRESHTYPE=ALL;`
- 执行后约 **60 秒**（PROTBINDFLOWF 定时器）策略才完全下发到转发面
- **SMF/UNC 侧无 REFRESHSRV**（SMF 侧策略即时生效）

### 4.2 REFRESHTYPE 取值

| 取值 | 说明 |
|------|------|
| ALL | 全量刷新 |
| USERPROFILE | 用户模板级刷新（粒度较细） |

### 4.3 60 秒窗口期约束（CR-AC-13）

SET REFRESHSRV 后约 60 秒内（PROTBINDFLOWF 定时器）：
- 策略尚未完全下发到转发面
- 此窗口期内修改 FILTER/L7FILTER 可能导致策略不一致
- **REFRESHSRV 必须是最后执行的命令**

## 5. 跨网元一致性（业务感知域规则 §2）

涉及 PCRF/PCF + UNC/SMF + UDG/UPF 多网元时：

### 5.1 三网元预定义规则一致

- RULENAME / FLOWFILTERNAME / ADCPARA 在 PCF、SMF、UPF 三处必须完全一致
- 定向业务/ADC 必须用预定义规则（因 PCF 无 L7 识别能力）

### 5.2 双产品共用参数一致（UPF ↔ SMF）

- URRID、RG、USAGERPTMODE、RULENAME、USERPROFILENAME

### 5.3 POLICYNAME 跨策略类型不冲突

- 同一产品内 PCC/QOS/BWM/CHARGING/HEADEN/WEBPROXY/SMARTREDIRECT/ADC 等不同 POLICYTYPE 的 RULENAME 不能相同（CR-AC-01）

### 5.4 访问限制特有三处一致

- URL 过滤 CATEGORYID：UDG + ICAP Server + PCRF/PCF
- 位置信息 CGI/ECGI/NCGI：UNC + RAN + PCRF/PCF

## 6. 通用过滤链（业务感知域规则 §1.2，T-001/T-002）

访问限制场景的标准过滤链：

```
ADD FILTER (定义三四层过滤条件)
    │
    ▼
ADD FLOWFILTER (定义流过滤器)
    │
    ├── ADD FLTBINDFLOWF (绑定 Filter 到 FlowFilter)
    │
    ▼
[可选] ADD L7FILTER → ADD PROTBINDFLOWF (七层扩展)
    │
    ▼
[可选] ADD EXTENDEDFILTER + ADD ERRORCODE (重定向族多维匹配)
    │
    ▼
SET REFRESHSRV:REFRESHTYPE=ALL (策略刷新生效，约 60 秒，必须最后)
    │
    ▼
ADD RULE:FLOWFILTERNAME=... (规则引用)
```

> 使用此过滤链的特性：**全部 11 个独有特性**（轨道 A 全部 + 轨道 B URL 过滤触发层共用三四层匹配链）。

## 7. OR 条件：FLOWFILTERGRP vs 多 RULE

| 方案 | 适用场景 |
|------|---------|
| FLOWFILTERGRP | 多个条件执行**完全相同**的动作 |
| 多 RULE | 需要不同优先级或后续可能分化 |

**默认**：使用 FLOWFILTERGRP（更简洁）。
