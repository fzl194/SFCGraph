# GWFD-110302 基于上下行解耦的视频承载信令控制 - 特性知识文档

> **场景归属**: 业务感知 → 带宽控制子场景（辅助特性：视频上下行带宽解耦）
> **产品**: UDG (UPF网元)
> **场景定位**: 视频业务上下行带宽解耦 → 独立控制上下行专用承载 → 带宽优化
> **辅助特性说明**: 针对视频业务上下行不对称特点，解耦承载实现精细带宽控制

---

## 1. 特性概述

### 1.1 基本信息

| 项目 | 内容 |
|------|------|
| 特性ID | GWFD-110302 |
| 特性名称 | 基于上下行解耦的视频承载信令控制 |
| 所属产品 | UDG |
| 适用NF | PGW-U、UPF |
| 首次发布版本 | 20.8.2 |
| License控制项 | 82200EBQ LKV3G5SCUD01 基于上下行解耦的视频承载信令控制 |
| 依赖NF | PGW-C/SMF（UNC 20.2.0及后续版本）、PCRF/PCF |

### 1.2 核心能力一句话描述

UDG 通过业务感知功能识别出特定的视频业务，对其业务的**下行链路**建立专有承载/专有QoS Flow，下行报文走专有承载，上行报文仍走缺省承载，从而实现视频业务上下行带宽的解耦控制。

### 1.3 在带宽控制场景中的定位

本特性属于带宽控制场景中的**辅助特性**，定位为"视频上下行带宽解耦"。

- 视频业务（直播、多流视频）典型流量模型：**下行高带宽、上行低带宽**。
- 传统上下行绑定承载：为下行视频流量建立高带宽专载时，上行也绑定在同一个专载上，造成上行空口资源浪费。
- 本特性解决方案：**只为下行建立专有承载**，上行继续使用缺省承载，实现上下行带宽独立控制、节省上行空口资源。

### 1.4 与 GWFD-020358（业务触发的QoS保证）的关系

GWFD-020358（业务触发的QoS保证）是通用的"业务触发建立专有承载"特性，建立的专载上下行绑定。

本特性 GWFD-110302 可视为 GWFD-020358 的**视频增强版**：
- 继承了 GWFD-020358 的业务识别 → 触发专载建立的完整链路。
- 新增 `DECOUPLINGSW`（上下行解耦开关）参数，在 `ADD QOSPROP` 中配置为 `ENABLE`。
- 开启后，建立的专有承载只承载下行报文，上行报文仍走原有承载/QoS Flow。

### 1.5 视频业务上下行不对称的应用场景

满足用户对视频业务差异化保障需求，通过为下行链路建立专有承载/专有QoS Flow对特定的视频业务进行业务体验保障，典型场景包括：
- **直播业务**：下行视频流大，上行仅有少量心跳/控制信令。
- **多流视频业务**：同时存在多条下行视频流，上行交互流量极小。
- **点播/短视频**：下行为主，上行仅含请求报文。

在这些场景下，为下行视频流量分配高QoS专载、为上行保留缺省承载，可在保障视频体验的同时节省上行空口资源。

## 2. 核心概念

### 2.1 上下行解耦（Uplink/Downlink Decoupling）

"上下行解耦"指：为同一业务流分别建立方向性不同的承载，使得**上行报文**和**下行报文**分别走不同的承载/QoS Flow。

在本特性中，上下行解耦的含义是：
- **下行**：视频业务的下行报文走新建的**专有承载/专有QoS Flow**（高QoS保障）。
- **上行**：视频业务的上行报文继续走**缺省承载/缺省QoS Flow**（不额外占用高QoS资源）。

这与传统的"上下行绑定专载"形成对比：传统方式中，一旦为某业务建立专载，上下行报文都走该专载。

### 2.2 视频业务的上下行不对称性

视频业务（尤其直播、点播、短视频）的流量模型具有显著的方向性不对称：
- 下行流量：视频流数据，带宽需求高（可达数 Mbps 至数十 Mbps）。
- 上行流量：请求报文、心跳、ACK、少量控制信令，带宽需求极低（通常 < 100 Kbps）。

如果为下行视频流量建立高带宽专载且上下行绑定，上行空口会预留与下行同等的高QoS资源，形成上行资源浪费。上下行解耦正是为解决这一不对称性而设计。

### 2.3 上行专用承载 vs 下行专用承载

| 承载类型 | 方向性 | 在本特性中的使用 | QoS 保障 |
|----------|--------|------------------|----------|
| 缺省承载/缺省QoS Flow | 上下行 | 视频业务**上行报文**继续走此承载 | 缺省QoS |
| 专有承载/专有QoS Flow | 本特性中仅承载下行 | 视频业务**下行报文**走此承载 | 高QoS（如QCI/5QI、ARP差异化） |

无线侧基于专有承载/专有QoS Flow的QoS参数（如 QCI、ARP 等），提供差异化转发调度策略，对下行视频业务提供差异化体验保障。

### 2.4 TFT（业务流模板，方向性）

在4G网络中，UE/PGW-U 通过与过滤器关联的 **UL/DL TFT（Traffic Flow Template）**，将上/下行业务报文映射到对应的承载上去。

- TFT 是方向性的：UL TFT 匹配上行报文，DL TFT 匹配下行报文。
- 5G 中与 TFT 类似的概念分别为：UE 上的 **QoS Rule** 和 UPF 上的 **PDR（Packet Description Rule / Packet Detection Rule）**。

本特性利用 TFT/PDR 的方向性：新建专载只下发 DL TFT（下行方向），使下行报文映射到专载，上行报文因无新的 UL TFT 匹配而继续走缺省承载。

### 2.5 与传统上下行绑定承载的差异

| 对比维度 | 传统上下行绑定专载（GWFD-020358） | 上下行解耦专载（GWFD-110302） |
|----------|-----------------------------------|-------------------------------|
| 专载承载方向 | 上下行均走专载 | 仅下行走专载，上行走缺省 |
| 上行QoS | 与下行同等高QoS | 缺省QoS |
| 上行空口资源 | 占用高QoS资源（浪费） | 不额外占用 |
| 触发开关 | 无需 DECOUPLINGSW | QOSPROP 中 DECOUPLINGSW=ENABLE |
| 适用业务 | 通用业务 | 视频类上下行不对称业务 |
| 验证方法 | 上下行TEIDU相同 | 上下行TEIDU不同 |

## 3. 实现原理与流程

### 3.1 PCC 模式预定义规则机制

本特性基于 **PCC 模式预定义规则** 工作：

1. 预先在 UDG 上配置规则内容（Filter、FlowFilter、Rule、URR、QOSPROP、PCCPOLICYGRP 等）。
2. 规则名称由 PGW-C/SMF 下发给 UDG。
3. UDG 根据接收的规则名称，匹配本地配置的具体规则。
4. UDG 从 PGW-C/SMF 获取用户所属预定义 PDR（Packet Detection Rule），预定义 PDR 由 PGW-C/SMF 基于本地策略或 PCRF/PCF 下发的规则生成。

### 3.2 业务感知识别

UDG 通过**业务感知功能（SA-Basic）**对业务报文进行内容解析：

- **三四层信息解析**：获得五元组（源IP、目的IP、源端口、目的端口、协议类型）。
- **七层信息解析**：获取 URL 等应用层信息。

将解析得到的关键信息和 PDR 进行比较，相符合即匹配成功。

### 3.3 触发上下行解耦专载建立的机制

当用户业务匹配到预定义规则，且规则中绑定了 **QoS-URR（Usage Reporting Rule，使用量上报规则，USAGERPTMODE=QOS）** 和 **QOSPROP（QoS Property，QoS属性，DECOUPLINGSW=ENABLE）** 时：

1. UDG 触发 **QoS 上报**。
2. UDG 向 PGW-C/SMF 发送 **PFCP Session Report Request** 消息，携带 Usage Report 信元组，包含 QoS 类型的 URR ID，Usage Report Trigger 为 **start**。
3. PGW-C/SMF 收到后，建立基于**下行链路**的专有承载/专有QoS Flow。
4. PGW-C/SMF 向 PGW-U/UPF 回复 **PFCP Session Report Response**，Create PDR 信元组中携带 URR ID（应与 Request 中的 URR ID 一致）。
5. 新建的专载只承载下行报文；上行报文继续走缺省承载。

### 3.4 完整端到端流程

```text
┌──────────┐   1.用户接入     ┌──────────┐                    ┌──────────┐
│   UE     │ ───────────────→ │ PGW-U/UPF│                    │ PGW-C/SMF│
│          │                  │  (UDG)   │                    │          │
└──────────┘                  └──────────┘                    └──────────┘
     │                              │                               │
     │   2.发起视频业务流量         │                               │
     │ ───────────────────────────→ │                               │
     │                              │                               │
     │                  3.SA识别视频业务(三四层+七层匹配)          │
     │                              │   匹配预定义PDR               │
     │                              │   规则含 QoS-URR + QOSPROP    │
     │                              │   (DECOUPLINGSW=ENABLE)       │
     │                              │                               │
     │                              │ 4.PFCP Session Report Request │
     │                              │   (URR ID, Trigger=start)     │
     │                              │ ────────────────────────────→ │
     │                              │                               │
     │                              │        5.建立下行专载/QoS Flow│
     │                              │        (仅DL TFT/PDR)         │
     │                              │                               │
     │                              │ ←──────────────────────────── │
     │                              │ 6.PFCP Session Report Response│
     │                              │   (Create PDR, URR ID)        │
     │                              │                               │
     │  7.下行视频报文 → 专载转发    │                               │
     │ ←─────────────────────────── │   (下行走专载,高QoS)          │
     │                              │                               │
     │  8.上行报文 → 缺省承载转发   │                               │
     │ ───────────────────────────→ │   (上行走缺省,不占专载资源)   │
     │                              │                               │
```

**流程要点**：
- 步骤3：业务感知识别是触发条件，必须先配置好 Filter/FlowFilter/Protocol 绑定。
- 步骤4：UDG 主动发起 QoS 上报，这是上下行解耦专载建立的信令起点。
- 步骤5：PGW-C/SMF 只创建下行方向的专载（DL TFT），不创建上行 UL TFT。
- 步骤7-8：解耦效果 —— 上下行走不同承载，TEIDU 不同。

### 4.2 上下行解耦相关核心配置参数

本特性相对于通用 QoS 保证（GWFD-020358）的**关键差异参数**：

| 命令 | 参数 | 取值 | 说明 |
|------|------|------|------|
| ADD URR | USAGERPTMODE | **QOS** | 固定取值，必须为 QoS 类型上报 |
| ADD QOSPROP | DECOUPLINGSW | **ENABLE** | **上下行解耦核心开关**，必须配置为 ENABLE 才能启用解耦 |
| ADD QOSPROP | QOSURRNAME | 引用URR名称 | 绑定 QoS 类型 URR，用于触发 QoS 事件上报 |

**配置约束**：
- `DECOUPLINGSW=ENABLE` 是启用本特性的必要条件。如果该参数不设为 ENABLE，则建立的专载仍为传统的上下行绑定专载。
- `USAGERPTMODE=QOS` 是固定取值，不能改为其他模式。只有 QoS 类型的 URR 上报才能触发上下行解耦专载建立。

### 4.3 视频业务识别规则配置

视频业务识别依赖 SA-Basic（GWFD-110101）的七层协议解析能力，通过以下配置链实现：

```text
Filter（三四层过滤）
    ↓ 绑定
FlowFilter（流过滤器）
    ↓ 绑定七层协议
PROTBINDFLOWF → PROTOCOLNAME="example_video"（视频协议）
    ↓ 被Rule引用
Rule（业务规则，POLICYTYPE=PCC）
    ↓ 绑定UserProfile
UserProfile → 用户
```

**三四层过滤参数**：

| 参数 | 取值样例 | 说明 |
|------|----------|------|
| FILTERNAME | filter_test | 过滤器名称 |
| L34PROTTYPE | STRING | 三四层IPv4协议输入类型 |
| L34PROTOCOL | ANY | 三四层协议类型，ANY表示不限定 |

**七层协议绑定参数**：

| 参数 | 取值样例 | 说明 |
|------|----------|------|
| FLOWFILTERNAME | flowfilter_test1 | 流过滤器名称 |
| PROTOCOLNAME | example_video | 七层视频协议名称（需在SA签名库中预定义） |

### 4.4 与 PCC 规则、专用承载的联动

本特性配置链路的完整绑定关系：

```text
UserProfile (up_test)
  └── RULEBINDING ──→ Rule (rule_test)
                        ├── FILTERINGMODE = FLOWFILTER
                        ├── FLOWFILTERNAME = flowfilter_test1
                        │     ├── FLTBINDFLOWF ──→ Filter (filter_test, 三四层)
                        │     └── PROTBINDFLOWF ──→ example_video (七层协议)
                        ├── POLICYTYPE = PCC
                        └── POLICYNAME = ppg1 (PCC策略组)
                              ├── QOSPROPNAME = qos-property1
                              │     ├── QOSURRNAME = urr_1 (USAGERPTMODE=QOS)
                              │     └── DECOUPLINGSW = ENABLE  ★ 解耦开关
                              └── URRGROUPNAME = urrgp_test (计费用，可选)
```

**联动要点**：
- Rule 的 `POLICYTYPE` 必须为 `PCC`。
- Rule 的 `POLICYNAME` 指向 PCC 策略组，策略组中的 QOSPROP 必须开启 `DECOUPLINGSW=ENABLE`。
- QOSPROP 中的 `QOSURRNAME` 必须引用 `USAGERPTMODE=QOS` 的 URR。
- `SET REFRESHSRV` 必须在 FlowFilter 相关配置完成后执行，否则新配置不生效。

## 5. 配置案例

### 案例 1：视频业务下行高带宽专载 + 上行缺省承载（标准解耦场景）

**场景描述**：运营商希望为 example_video 协议的视频业务提供下行差异化QoS保障，上行保持缺省承载，实现上下行解耦。

**数据规划**：

| 参数 | 取值 |
|------|------|
| 三四层过滤器名称 | filter_test |
| 流过滤器名称 | flowfilter_test1 |
| 七层协议名称 | example_video |
| URR名称 | urr_1 |
| URR标识 | 1 |
| QoS属性名称 | qos-property1 |
| PCC策略组名称 | ppg1 |
| URR组名称 | urrgp_test |
| 业务规则名称 | rule_test |
| 用户模板名称 | up_test |
| 规则优先级 | 10 |

**完整命令序列**：

```mml
// 步骤1：打开本特性的License配置开关
SET LICENSESWITCH:LICITEM="LKV3G5SCUD01",SWITCH=ENABLE;

// 步骤2：配置三四层过滤条件
ADD FILTER:FILTERNAME="filter_test",L34PROTTYPE=STRING,L34PROTOCOL=ANY;

// 步骤3：配置流过滤器，绑定三四层、七层过滤条件
ADD FLOWFILTER:FLOWFILTERNAME="flowfilter_test1";

// 步骤3a：绑定三四层过滤条件到流过滤器
ADD FLTBINDFLOWF:FLOWFILTERNAME="flowfilter_test1",FILTERNAME="filter_test";

// 步骤3b：绑定七层过滤条件到流过滤器
ADD PROTBINDFLOWF:FLOWFILTERNAME="flowfilter_test1",PROTOCOLNAME="example_video";

// 步骤3c：刷新使新配置的Filter生效
SET REFRESHSRV:REFRESHTYPE=USERPROFILE;

// 步骤4：配置QoS属性，打开上下行解耦开关
// 步骤4a：配置QoS类型URR
ADD URR:URRNAME="urr_1",URRID=1,USAGERPTMODE=QOS;

// 步骤4b：配置QoS属性，DECOUPLINGSW=ENABLE 是解耦核心参数
ADD QOSPROP:QOSPROPNAME="qos-property1",QOSURRNAME="urr_1",DECOUPLINGSW=ENABLE;

// 步骤5：配置PCC策略组（需要计费时配置URRGROUPNAME）
ADD PCCPOLICYGRP:PCCPOLICYGRPNM="ppg1",QOSPROPNAME="qos-property1",URRGROUPNAME="urrgp_test";

// 步骤6：配置业务规则
ADD RULE:RULENAME="rule_test",POLICYTYPE=PCC,FILTERINGMODE=FLOWFILTER,FLOWFILTERNAME="flowfilter_test1",POLICYNAME="ppg1",PRIORITY=10;

// 步骤7：配置业务策略组合
// 步骤7a：创建用户模板
ADD USERPROFILE:USERPROFILENAME="up_test";

// 步骤7b：绑定规则到用户模板
ADD RULEBINDING:USERPROFILENAME="up_test",RULENAME="rule_test";
```

**配置效果**：
- 当用户 up_test 的视频流量（匹配 example_video 协议）经过 UDG 时，SA 识别到视频业务。
- 规则 rule_test 匹配成功，触发 PCC 策略组 ppg1。
- QOSPROP 中 DECOUPLINGSW=ENABLE，UDG 发起 QoS 上报（PFCP Session Report Request，URR ID=1，Trigger=start）。
- PGW-C/SMF 建立下行专载（仅含 DL TFT）。
- 视频下行报文走高QoS专载，上行报文走缺省承载。

### 案例 2：基于流过滤器组的视频解耦配置（多协议场景）

**场景描述**：运营商有多个视频协议（video_proto_a、video_proto_b）需要触发解耦，使用流过滤器组统一管理。

**配置方式**：将步骤6中的 `FILTERINGMODE` 改为 `FLOWFILTERGRP`，并额外使用 `ADD FLOWFILTERGRP` 命令配置流过滤器组。

```mml
// 使用流过滤器组时的Rule配置
ADD RULE:RULENAME="rule_test_multi",POLICYTYPE=PCC,FILTERINGMODE=FLOWFILTERGRP,FLOWFILTERGRPNAME="ffg_video",POLICYNAME="ppg1",PRIORITY=10;
```

> 注意：流过滤器组需通过 `ADD FLOWFILTERGRP` 命令预先配置，将多个流过滤器组合在一起。其余 QoS 属性、PCC 策略组配置与案例 1 完全一致。

## 6. 验证与调测

### 6.1 调测前置条件

| 类别 | 参数 | 取值样例 | 说明 |
|------|------|----------|------|
| 用户跟踪任务 | IMSI | 460000123456789 | 测试终端IMSI |
| APN | APN名称 | apn1 | 测试终端使用的APN |
| 业务触发协议 | PROTOCOLNAME | example_video | 七层过滤规则中配置的视频协议 |

**工具**：测试终端、OM Portal。

### 6.2 License 开关验证

```mml
LST LICENSESWITCH:LICITEM="LKV3G5SCUD01";
```

- `SWITCH` 为 `ENABLE`：License已打开，继续后续验证。
- `SWITCH` 为 `DISABLE`：执行 `SET LICENSESWITCH:LICITEM="LKV3G5SCUD01",SWITCH=ENABLE;` 打开开关。

### 6.3 接入与基本业务验证

1. 测试终端使用 apn1 接入网络 → 成功接入后继续。
2. 测试终端浏览 Web 业务（如访问 www.huawei.com）→ 能正常访问后继续。

### 6.4 解耦专载建立验证

触发视频业务（匹配 example_video 协议）后，检查信令消息：

**步骤A：检查 PFCP Session Report Request（UDG → PGW-C/SMF）**

- 消息中应携带 Usage Report 信元组。
- Usage Report 中应包含 QoS 类型的 URR ID。
- Usage Report Trigger 应为 **start**。

- 如果 UDG 正确上报 → 解耦功能生效，继续步骤B。
- 如果未上报或信元不全 → 功能未生效，转入配置排查（6.6节）。

**步骤B：检查 PFCP Session Report Response（PGW-C/SMF → UDG）**

- Create PDR 信元组中携带的 URR ID 应与 Request 中的 URR ID **一致**。
- 一致 → 继续步骤C。
- 不一致 → 执行 `DSP SESSIONINFO` 排查。

**步骤C：查询用户上下文**

```mml
DSP SESSIONINFO:QUERYTYPE=IMSI,IMSI="460000123456789";
```

- 检查测试终端是否建立了专有承载/专有QoS Flow。
- 检查使用的 APN name 是否与规划值一致。

### 6.5 解耦效果验证（TEIDU 对比）

**核心验证方法**：查看基于 IMSI 的用户跟踪报文，对比上行报文和下行报文中 **GTP 头中的 TEIDU**：

- **TEIDU 不一致** → 上下行走不同承载 → **解耦功能正常**，调测结束。
- **TEIDU 一致** → 上下行走同一承载 → 解耦未生效，转入排查。

### 6.6 常见问题排查

**排查路径**：按配置链从外到内逐层验证。

**排查1：UserProfile 绑定规则**

```mml
LST RULEBINDING:USERPROFILENAME="up_test";
```

预期输出：
```
用户模板名称  =  up_test
    规则名称  =  rule_test
    策略类型  =  PCC
```

**排查2：Rule 配置**

```mml
LST RULE:RULENAME="rule_test";
```

预期输出应包含：
```
规则名称 = rule_test
策略类型 = PCC
流过滤器名称 = flowfilter_test1
PCC策略组名称 = ppg1
```

**排查3：PCC 策略组与 QoS 属性（★ 解耦开关关键检查）**

```mml
LST PCCPOLICYGRP:PCCPOLICYGRPNM="ppg1";
```

预期输出：
```
PCC策略组名称 = ppg1
Qos属性名称   = qos-property1
```

```mml
LST QOSPROP:QOSPROPNAME="qos-property1";
```

预期输出（关键）：
```
Qos属性名称            = qos-property1
QoS使用量上报规则名称  = urr_1
上下行解耦开关         = 使能（开启）   ★ 必须为"使能（开启）"
```

- 如果 `上下行解耦开关` 显示为"未使能"或"关闭" → 检查 `ADD QOSPROP` 是否配置了 `DECOUPLINGSW=ENABLE`。

```mml
LST URR:URRNAME="urr_1";
```

预期输出：
```
使用量上报规则名称 = urr_1
URR标识            = 1
使用量上报模式     = QOS   ★ 必须为 QOS
```

**排查4：过滤条件验证**

```mml
LST FLOWFILTER:FLOWFILTERNAME="flowfilter_test1";
LST FILTER:FILTERNAME="filter_test";
LST PROTBINDFLOWF:FLOWFILTERNAME="flowfilter_test1";
```

预期输出：
```
流过滤器名称  = flowfilter_test1
协议名称      = example_video
```

- 如果 Rule 绑定的流过滤器名称、PCC策略组名称与规划值不一致 → 参考激活文档重新配置。
- 如果三四层/七层过滤信息与规划值不一致 → 重新配置后重新执行步骤5（触发视频业务验证）。

**排查5：如果上述配置均正确但解耦仍未生效**

1. 在 OM Portal 上创建用户跟踪任务，重新执行接入并保存报文。
2. 执行 `EXP MML` 命令导出当前配置数据为 MML 脚本文件并保存。
3. 收集所有查询信息和跟踪报文。
4. 联系华为技术支持。

### 7.5 对系统的影响

- **性能影响**：需要打开业务感知功能进行视频业务识别，对系统转发性能有影响。
- **信令开销**：UDG 需主动触发专载建立，增加信令消息处理开销，CPU 占用率相应上升。

### 7.6 应用限制

- **并发限制**：为避免7层业务触发专载导致的信令风暴，只有**并发链路流小于8个协议或协议组**才适合配置为基于流触发专载建立。
- **协议限制**：本特性**只支持 HTTP1.x 协议，不支持 HTTP2.0 协议**。
- **加密限制**：本特性**不支持加密场景基于 host 的计费和控制**。

### 7.7 特性规格

| 规格名称 | 规格指标 |
|----------|----------|
| 每个用户支持的最大专有承载个数 | 10 |
| 每个用户支持的最大专有QoS Flow个数 | 63 |

### 7.8 计费与话单

本特性**不涉及计费与话单**。但 PCC 策略组中的 `URRGROUPNAME` 参数仍可配置（用于计费联动），不影响解耦功能本身。

### 7.9 告警与软参

本特性**无相关告警**，**无相关软参**。
