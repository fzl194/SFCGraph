# GWFD-110331 基于业务流标识的无线资源优化 - 特性知识文档

> **场景归属**: 业务感知 → 带宽控制子场景（辅助特性：无线资源优化）
> **产品**: UDG (UPF网元)
> **场景定位**: 业务流标识 → 上报无线侧 → 基站调度优化 → 间接提升有效带宽
> **辅助特性说明**: 通过业务标识触发无线侧调度优化，间接影响带宽体验

### 1.2 核心能力一句话描述

UDG开启FPI（Flow Priority Index）差异化控制开关后，将FPI信息映射到业务数据报文的DSCP字段或GTP-U扩展头传递到无线侧（RAN），无线侧根据该标识进行业务流的QoS差异化调度，从而对不同业务实施不同策略控制。

### 1.3 在带宽控制场景中的定位

本特性属于**带宽控制场景的辅助特性**，走的是"无线侧优化路径"：

- **不直接做用户面带宽限速**：本特性不在UPF上执行门控、限速、整形等带宽控制动作。
- **通过无线侧调度间接影响有效带宽**：UPF在报文中标记业务流标识（FPI/DSCP），基站据此调整调度优先级和资源块分配，使高价值业务在空口资源紧张时获得更多调度机会。
- **有效带宽提升机制**：当空口资源繁忙时，高优先级业务被优先调度，其实际可用带宽（有效带宽）得到保障；低优先级业务让出资源，有效带宽相应降低。

### 1.4 与SA、BWM、ADC的协作关系

| 协作对象 | 关系 | 说明 |
|----------|------|------|
| **SA-Basic (GWFD-110101)** | 强依赖 | UDG基于SA执行业务识别，本特性开通必须同时开启SA功能。SA识别业务后，才能匹配FPI规则并标记业务流标识。 |
| **PCC基本功能 (GWFD-020351)** | 组网基础 | 本特性与PCC基本功能特性组网相同，复用PCC动作属性、PCC策略组等配置框架。调测前需完成PCC基本功能调测。 |
| **BWM（带宽管理）** | 互补 | BWM在UPF侧直接做限速/整形，本特性在无线侧做调度优化。两者可并行部署，形成"核心网限速+无线侧优先保障"的立体化带宽控制策略。 |
| **ADC（应用检测控制）** | 业务识别支撑 | ADC/SA检测出具体业务后，可触发不同的FPI规则，实现"按业务类型分配无线调度优先级"。 |

## 2. 核心概念

### 2.1 FPI（Flow Priority Index）业务流标识

FPI是本特性的核心概念，是一个0~255的整数值，用于标识业务流的优先级：

- **FPI值含义**：代表业务流在无线侧的调度优先级。FPI值越大，通常对应越高的调度优先级队列。
- **FPI值与队列映射**：FPI值在RAN上映射到固定的7个调度队列（队列0~队列6），该映射关系在RAN上是固定的，不能通过命令修改。但每个队列的调度权重值可以通过命令修改。

| FPI值范围 | 队列号 | 默认调度权重 |
|-----------|--------|-------------|
| 0~7、55~255 | 队列0 | 10 |
| 8~15 | 队列1 | 1 |
| 16~23 | 队列2 | 10 |
| 24~31 | 队列3 | 10 |
| 32~39 | 队列4 | 10 |
| 40~47 | 队列5 | 10 |
| 48~55 | 队列6 | 10 |

注意：队列1（FPI值8~15）的默认调度权重为1，明显低于其他队列的10，意味着该范围内FPI值的业务在空口资源紧张时获得的调度机会最少。这是一个需要特别注意的规划要点。

### 2.2 FPI传递方式（两种方案）

本特性支持两种将FPI信息传递到无线侧的方案：

#### 方案一：DSCP方案（版本1，20.6.0起支持）

- **传递机制**：UDG将匹配到的规则中的FPI信息映射到业务数据报文的**内层IP包的DSCP字段**（DiffServ Code Point），通过修改下行报文的DSCP值传递给无线侧。
- **无线侧行为**：无线侧根据DSCP字段的取值进行资源调度。
- **性能影响**：对IP的DSCP包头做映射，对UPF性能的影响可以忽略。
- **适用场景**：对性能敏感、不需要GTP-U扩展头的场景。

#### 方案二：GTP-U扩展头方案（版本2，20.15.0起支持）

- **传递机制**：UDG将FPI值映射到用户下行报文的**GTP-U扩展头**部分，通过华为自定义的GTP-U扩展头信元传递FPI信息。
- **无线侧行为**：无线侧根据GTP-U扩展头中的FPI值进行差异化调度。
- **能力协商**：需要UDG和RAN侧进行能力协商，确认双方均为华为设备且支持该功能。
- **性能影响**：需要增加对GTP-U扩展头的处理，会增加UPF的CPU资源消耗。扩展头最长12字节，会影响网络中MTU（Maximum Transfer Unit）的规划。
- **适用场景**：需要更精确的FPI传递、且整网支持GTP-U扩展头的场景。

### 2.3 FPI策略来源（GTP-U方案专属）

在GTP-U方案中，FPI值有两种来源：

| 策略来源参数（FPIPOLOCYORIGIN） | 取值 | 说明 |
|-------------------------------|------|------|
| **INTERNAL（本地规则）** | INTERNAL | UDG将本地配置的FPI值映射到GTP-U扩展头。业务数据流匹配UDG本地配置的带有FPI信息的规则后，将规则中的FPI值写入GTP-U扩展头。 |
| **EXTERNAL（报文DSCP）** | EXTERNAL | UDG将用户下行数据报文中已有的DSCP值映射到GTP-U扩展头。适用于上游网络已做好DSCP标记的场景。 |

### 2.4 无线资源调度机制

无线侧（RAN）接收到携带FPI信息的报文后，进行差异化调度：

- **调度优先级**：不同FPI值对应不同队列，高优先级队列的业务数据报文优先获得空口资源。
- **资源块分配**：当空口资源繁忙时，保障高优先级业务优先调度；网络闲时，空口资源充分利用，满足所有业务的流量需求。
- **收敛特性**：当信道条件相同、网络忙时空口资源受限、S1/N3接口业务数据足够多时，不同业务速率之间的比值收敛于预先配置的对应业务权重的比值。
- **实际网络表现**：由于各种业务抢占带宽能力不同，某些业务S1/N3接口业务数据不足时，RAN在调度完高优先级业务后会用剩余资源调度低优先级业务，实际速率比值可能不严格收敛于权重比值。但与不开启该特性相比，始终能够提升高优先级业务体验。

### 2.5 GTP-U扩展头信元格式

华为定义的GTP-U扩展头信元格式包含以下字段：

| 字段 | 说明 |
|------|------|
| Length | 该信元的长度（octets），包括Length、Type和Content，遵循3GPP标准 |
| Next Sub Extension Header Type | 指示Sub Extension Header Content中包含的首个子信元类型 |
| Sub Extension Header Content | 华为扩展头信元所携带的内容，可以包含多个子信元 |
| Next Extension Header Type | 下一个扩展头的类型，遵循3GPP标准 |

子信元类型定义：

| Sub Extension Header Type（十进制） | 方向 | 消息内容 |
|-------------------------------------|------|----------|
| 0 | 上行/下行 | 指示没有更多的子信元 |
| 1 | 上行 | RAN节点能力信息，包括版本信息（Version） |
| 4 | 上行 | RAN与CN之间的心跳维护消息，只在上行报文中携带 |
| 5~7 | - | Reserved（保留） |
| 8 | 下行 | GW能力信息，包括版本信息（Version） |
| 9 | 下行 | 业务优先级标识FPI |
| 10~15 | - | Reserved（保留） |

关键子信元说明：
- **Type=1（上行）**：RAN节点发送能力信息，告知UDG自己是华为设备且支持本功能。
- **Type=8（下行）**：UDG发送GW能力信息，告知RAN自己是华为设备且支持本功能。
- **Type=9（下行）**：携带业务优先级标识FPI值，这是本功能的核心传递信元。
- **Type=4（上行）**：心跳维护，用于实时检测RAN侧能力的变化情况。

### 2.6 能力协商过程

在GTP-U方案中，用户激活后，UDG和RAN侧需要进行能力协商：

1. 用户发起激活请求流程。
2. 用户激活成功后，无线侧与UDG通过数据报文的GTP-U扩展头进行能力协商，通知彼此是否为华为设备且支持GTP-U扩展头方式的业务流标识的无线资源优化功能。
3. RNC/eNodeB将RAN节点能力信息（华为设备标识）填充到首个上行GTP-U扩展头中（Type=1），通知UDG RAN侧的能力。
4. UDG收到RAN侧发送的携带RAN侧能力的上行GTP-U报文后，决定是否将自身能力通过GTP-U扩展头（Type=8）通知RAN侧。
5. 能力协商完成后，UDG支持**实时检测RAN侧能力的变化情况**，便于实时调整对数据报文的处理策略。

## 3. 实现原理与流程

### 3.1 业务流标识标记机制

UDG在本地的业务规则中配置FPI值。当用户业务数据流匹配到带有FPI信息的规则时，UDG根据配置的传递方式（DSCP或GTP-U扩展头）将FPI信息写入下行报文。

标记流程的关键环节：

1. **SA业务识别**：SA（Service Awareness）功能识别用户正在访问的业务类型。
2. **规则匹配**：识别出的业务数据流匹配UDG本地配置的业务规则（ADD RULE配置）。
3. **FPI值读取**：从匹配的规则中读取配置的FPI值（FPIVALUE参数）。
4. **标识写入**：根据SET FPIFUNC配置的传递方式，将FPI值写入报文的DSCP字段或GTP-U扩展头。

### 3.2 标识上报流程

#### 3.2.1 DSCP方案端到端流程

```
[用户终端] → 业务访问 → [UPF/UDG]
                              │
                    1. SA识别业务
                    2. 匹配FPI规则
                    3. 读取FPI值
                    4. 将FPI值映射到内层IP包DSCP字段
                              │
                              ↓
                         [基站/RAN]
                              │
                    5. 解析DSCP字段
                    6. 根据DSCP值进行QoS调度
                              │
                              ↓
                         [空口资源分配]
```

详细步骤：
1. 用户发起标准的内容计费用户激活流程。
2. 用户进行业务访问时，业务数据流匹配到UDG本地配置的带有FPI信息的规则。
3. PGW-U/UPF将匹配到的规则中的FPI信息映射到业务数据报文的DSCP部分。
4. PGW-U/UPF将刷新了DSCP部分的数据报文传递给无线侧。
5. 无线侧根据数据报文中DSCP部分的取值对业务数据报文进行调度。

#### 3.2.2 GTP-U扩展头方案端到端流程

```
[用户终端] → 激活请求 → [UPF/UDG] ←→ [基站/RAN]
                                    │
                          能力协商（GTP-U扩展头）
                          确认双方均为华为设备
                                    │
[用户终端] → 业务访问 → [UPF/UDG]
                          │
                1. SA识别业务
                2. SET FPIFUNC参数处理
                   ├─ FPIPOLOCYORIGIN=INTERNAL：
                   │   匹配本地FPI规则 → FPI值写入GTP-U扩展头
                   └─ FPIPOLOCYORIGIN=EXTERNAL：
                       读取报文DSCP值 → 映射到GTP-U扩展头
                          │
                          ↓
                     [基站/RAN]
                          │
                3. 解析GTP-U扩展头中的FPI值
                4. 根据FPI值进行差异化调度
                          │
                          ↓
                     [空口资源分配]
```

详细步骤：
1. 用户发起激活请求流程。
2. 用户激活成功后，无线侧与UDG通过数据报文的GTP-U扩展头进行能力协商，通知彼此是否为华为设备，且支持GTP-U扩展头方式的业务流标识的无线资源优化功能。
3. 用户进行业务访问时，UDG根据本地配置的SET FPIFUNC参数进行处理：
   - 当FPIPOLOCYORIGIN设置为INTERNAL时：用户业务数据流匹配UDG本地配置的带有FPI信息的规则，将规则中的FPI值映射到用户下行报文GTP-U扩展头部分，然后将刷新GTP-U扩展头的报文传递给无线侧。
   - 当FPIPOLOCYORIGIN设置为EXTERNAL时：UDG将用户下行数据报文中的DSCP值映射到报文GTP-U扩展头部分，然后将刷新GTP-U扩展头的报文传递给无线侧。
4. 无线侧接收到报文后，根据报文GTP-U扩展头中的业务标识FPI值对报文进行差异化调度。

### 3.3 基站调度优化机制

无线侧基于FPI值的调度优化遵循以下原则：

| 场景 | 调度行为 | 业务影响 |
|------|----------|----------|
| 空口资源充足（网络闲时） | 所有队列业务均被调度，空口资源充分利用 | 所有业务流量需求尽量满足，FPI差异不明显 |
| 空口资源紧张（网络忙时） | 高优先级队列业务优先调度 | 高FPI值业务获得更多空口资源，有效带宽提升 |
| 业务数据不足（S1/N3接口） | 高优先级调度完后，剩余资源调度低优先级 | 速率比值不完全收敛于权重比，但高优先级始终受益 |

调度权重说明：
- 队列的调度权重值可以通过RAN侧命令修改（不是在UDG上修改）。
- FPI值与队列号的映射关系在RAN上是固定的，不能通过命令修改。
- 默认配置下，队列1（FPI 8~15）权重仅为1，其余队列权重为10。在规划FPI值时，需注意避免将高价值业务误配到队列1。

### 3.4 对系统的影响

**性能影响程度**：低

| 方案 | 性能影响 | 网络规划影响 |
|------|----------|-------------|
| GTP-U方案 | 需要增加对GTP-U扩展头的处理，会增加UDG的CPU资源消耗，具体消耗程度需根据当前话务模型评估 | RAN侧和核心网侧之间传输携带GTP-U扩展头的数据报文（最长12字节），会影响网络中MTU的规划，需要整网规划MTU值 |
| DSCP方案 | 对IP的DSCP包头做映射，对性能的影响可以忽略 | 无额外MTU影响 |

**系统容量**：在同等话务模型下，容量不变。

### 4.3 FPI功能配置（核心命令）

**命令**: `SET FPIFUNC`

该命令用于配置FPI差异化控制功能，是本特性最核心的配置命令。

| 参数 | 参数名称 | 取值样例 | 说明 |
|------|----------|----------|------|
| FPISWITCH | FPI使能开关 | ENABLE | 用于配置是否开启FPI差异化控制功能。开启本功能后，要求无线侧同时也支持该功能。 |
| FPITRANSMETHOD | FPI传递类型 | DSCP | 用于设置FPI的传递方式。可选DSCP（映射到报文DSCP字段）或GTP-U扩展头方式。 |

```
// DSCP方案
SET FPIFUNC:FPISWITCH=ENABLE,FPITRANSMETHOD=DSCP;
```

**关键约束**：开启FPI功能后，要求无线侧同时支持该功能。本特性属于解决方案特性，要求无线侧网元支持基于业务流标识的无线资源优化功能。

### 4.4 业务过滤器配置

#### 4.4.1 配置过滤器（ADD FILTER）

| 参数 | 参数名称 | 取值样例 | 说明 |

| FILTERNAME | 过滤器名字 | filter_test | UDG通过将用户上下行报文匹配Filter，实现三四层报文的分析，从而过滤出会触发内容感知的数据报文。 |
| L34PROTTYPE | 三四层IPv4协议输入类型 | STRING | 协议输入方式 |
| L34PROTOCOL | 三四层协议类型 | TCP | 匹配的协议类型 |

```
ADD FILTER:FILTERNAME="filter_test",L34PROTTYPE=STRING,L34PROTOCOL=TCP;
```

#### 4.4.2 配置流过滤器（ADD FLOWFILTER）

| 参数 | 参数名称 | 取值样例 | 说明 |
|------|----------|----------|------|
| FLOWFILTERNAME | 流过滤器名称 | ff_test | 用于绑定Filter的FlowFilter |

```
ADD FLOWFILTER:FLOWFILTERNAME="ff_test";
```

#### 4.4.3 流过滤器绑定过滤器（ADD FLTBINDFLOWF）

```
ADD FLTBINDFLOWF:FLOWFILTERNAME="ff_test",FILTERNAME="filter_test";
```

#### 4.4.4 刷新使配置生效（SET REFRESHSRV）

```
SET REFRESHSRV:REFRESHTYPE=ALL;
```

新增或修改Filter后，必须执行SET REFRESHSRV使新配置的Filter生效。

### 4.5 PCC动作属性配置

**命令**: `ADD PCCACTIONPROP`

| 参数 | 参数名称 | 取值样例 | 说明 |
|------|----------|----------|------|
| PCCACTPROPNAME | PCC动作属性名称 | pap_test | PCC动作属性的名称 |
| UPINITDNGATE | 上行发起下行门控 | PASS | **关键参数**：FPI规则携带在下行数据报文中，因此下行数据需要配置为直接通过（PASS）。上下行动作使用ADD PCCACTIONPROP命令中配置的动作。 |

```
ADD PCCACTIONPROP:PCCACTPROPNAME="pap_test",UPINITDNGATE=PASS;
```

**关键约束**：UPINITDNGATE必须配置为PASS，因为FPI信息是通过下行数据报文传递给无线侧的，如果下行报文被门控（丢弃），FPI信息将无法到达无线侧。

| PCCACTPROPNAME | PCC动作属性名称 | pap_test | 分类组绑定的动作属性，需从已配置数据中获取 |

ADD PCCPOLICYGRP:PCCPOLICYGRPNM="ppg_test",PCCACTPROPNAME="pap_test",URRGROUPNAME="urrgp_test";

### 4.7 业务规则配置（含FPI规则）

**命令**: `ADD RULE`

本特性需要配置两类规则：

#### 4.7.1 PCC策略规则（用于绑定PCC策略组）

| 参数 | 参数名称 | 取值样例 | 说明 |
|------|----------|----------|------|
| RULENAME | 规则名称 | rule_test1 | 业务规则名称 |
| POLICYTYPE | 策略类型 | PCC | 标准PCC策略类型 |
| FILTERINGMODE | 流过滤器或流过滤器组选择 | FLOWFILTER | 选择使用流过滤器模式 |
| FLOWFILTERNAME | 流过滤器名称 | ff_test | 规则绑定的流过滤器 |
| PRIORITY | 优先级 | 1 | 规则匹配优先级 |
| POLICYNAME | 策略名称 | ppg_test | 规则绑定的PCC策略组名称 |

```
ADD RULE:RULENAME="rule_test1",POLICYTYPE=PCC,FILTERINGMODE=FLOWFILTER,FLOWFILTERNAME="ff_test",PRIORITY=1,POLICYNAME="ppg_test";
```

#### 4.7.2 FPI重标记规则（核心：配置FPI值）

| 参数 | 参数名称 | 取值样例 | 说明 |
|------|----------|----------|------|
| RULENAME | 规则名称 | rule_test2 | 业务规则名称 |
| POLICYTYPE | 策略类型 | REMARK_FPI | **FPI重标记策略类型** |
| FILTERINGMODE | 流过滤器或流过滤器组选择 | FLOWFILTER | 选择使用流过滤器模式 |
| FLOWFILTERNAME | 流过滤器名称 | ff_test | 规则绑定的流过滤器 |
| PRIORITY | 优先级 | 2 | 规则匹配优先级 |
| REMARKFPISEL | 重标记或FPI选择 | FPI | 选择FPI |
| FPIVALUE | FPI值 | 63 | **核心参数**：配置的业务流标识值，全网规划 |

```
ADD RULE:RULENAME="rule_test2",POLICYTYPE=REMARK_FPI,FILTERINGMODE=FLOWFILTER,FLOWFILTERNAME="ff_test",PRIORITY=2,REMARKFPISEL=FPI,FPIVALUE=63;
```

**FPI值规划要点**：
- FPIVALUE的取值范围为0~255。
- 不同FPI值范围对应不同RAN调度队列（见第2.1节表格）。
- 高价值业务建议配置在队列6（FPI值48~55）或队列0的高位段（FPI值55~255）。
- 低价值或可降级业务可配置在队列1（FPI值8~15，默认权重仅1）。
- 需全网统一规划，确保FPI值在整网语义一致。

#### 4.7.3 使用流过滤器组模式（可选）

如果运营商希望基于流过滤器组设置过滤条件：

1. 配置流过滤器组：
```
ADD FLOWFILTERGRP:...
```
2. 配置规则时，FILTERINGMODE选择FLOWFILTERGRP：
```
ADD RULE:...,FILTERINGMODE=FLOWFILTERGRP,...;
```

### 4.8 与PCC规则的联动关系

本特性的FPI规则与标准PCC规则的关系：

| 维度 | PCC规则 | FPI规则 |
|------|---------|---------|
| POLICYTYPE | PCC | REMARK_FPI |
| 主要功能 | 绑定PCC策略组，执行门控/计费等动作 | 标记业务流标识FPI值，传递给无线侧 |
| 策略名称 | POLICYNAME指向PCC策略组 | 无POLICYNAME，使用REMARKFPISEL+FPIVALUE |
| 优先级 | 通过PRIORITY参数排序 | 通过PRIORITY参数排序 |
| 联动 | PCC规则和FPI规则通常配置在同一流过滤器上，按优先级依次匹配 | FPI规则可以与PCC规则并存，互不冲突 |

典型联动配置模式：同一流过滤器上配置两条规则——一条PCC规则（绑策略组，做门控/计费）+ 一条REMARK_FPI规则（标记FPI值，传递无线侧）。

## 5. 配置案例

### 5.1 案例一：DSCP方案基础配置

**场景描述**：在UDG上使能基于业务流标识的无线资源优化功能，FPI信息以DSCP方式向RAN侧传递。为TCP业务流配置FPI值63（对应队列0，默认权重10），实现无线侧对该业务流的优先调度保障。

**配置前提**：
- 已完成SA-Basic（GWFD-110101）功能激活
- 已完成PCC基本功能（GWFD-020351）调测
- 已配置URR组（如需计费）
- 无线侧支持基于业务流标识的无线资源优化功能

**完整命令序列**：

```mml
// 步骤1：打开License开关
SET LICENSESWITCH:LICITEM="LKV3G5WOFR01",SWITCH=ENABLE;

// 步骤2：配置FPI使能开关，选择DSCP传递方式
SET FPIFUNC:FPISWITCH=ENABLE,FPITRANSMETHOD=DSCP;

// 步骤3：配置三四层过滤规则
// 3.1 配置过滤器（匹配TCP协议）
ADD FILTER:FILTERNAME="filter_test",L34PROTTYPE=STRING,L34PROTOCOL=TCP;
// 3.2 配置流过滤器
ADD FLOWFILTER:FLOWFILTERNAME="ff_test";
// 3.3 流过滤器绑定过滤器
ADD FLTBINDFLOWF:FLOWFILTERNAME="ff_test",FILTERNAME="filter_test";
// 3.4 刷新使配置生效
SET REFRESHSRV:REFRESHTYPE=ALL;

// 步骤4：配置PCC动作属性（下行门控必须为PASS）
ADD PCCACTIONPROP:PCCACTPROPNAME="pap_test",UPINITDNGATE=PASS;

// 步骤5：配置PCC策略组（当需要计费时配置URRGROUPNAME）
ADD PCCPOLICYGRP:PCCPOLICYGRPNM="ppg_test",PCCACTPROPNAME="pap_test",URRGROUPNAME="urrgp_test";

// 步骤6：配置业务规则
// 6.1 PCC策略规则（绑定PCC策略组，做门控和计费）
ADD RULE:RULENAME="rule_test1",POLICYTYPE=PCC,FILTERINGMODE=FLOWFILTER,FLOWFILTERNAME="ff_test",PRIORITY=1,POLICYNAME="ppg_test";
// 6.2 FPI重标记规则（标记FPI值为63，映射到DSCP传递给无线侧）
ADD RULE:RULENAME="rule_test2",POLICYTYPE=REMARK_FPI,FILTERINGMODE=FLOWFILTER,FLOWFILTERNAME="ff_test",PRIORITY=2,REMARKFPISEL=FPI,FPIVALUE=63;
```

**配置验证点**：
1. 执行`LST LICENSESWITCH:LICITEM="LKV3G5WOFR01";`确认Switch为ENABLE。
2. 确认SET FPIFUNC配置成功，FPISWITCH=ENABLE，FPITRANSMETHOD=DSCP。
3. 确认ADD PCCACTIONPROP中UPINITDNGATE=PASS。
4. 确认ADD RULE中POLICYTYPE=REMARK_FPI的规则FPIVALUE=63。
5. 抓包验证下行报文DSCP字段是否携带正确的FPI值。

### 5.2 案例二：多业务FPI差异化调度配置

**场景描述**：运营商需要为三类业务提供差异化无线调度保障：
- **视频流媒体业务**（高优先级）：FPI=56（队列0高位段，默认权重10），保障视频体验
- **网页浏览业务**（中优先级）：FPI=48（队列6，默认权重10），正常保障
- **后台同步业务**（低优先级）：FPI=10（队列1，默认权重1），让出空口资源

> 注意：以下命令仅展示核心差异部分（过滤器、规则配置），License和FPIFUNC配置同案例一。

```mml
// 配置三个过滤器（假设通过不同端口或IP匹配不同业务）
ADD FILTER:FILTERNAME="filter_video",L34PROTTYPE=STRING,L34PROTOCOL=TCP;
ADD FILTER:FILTERNAME="filter_web",L34PROTTYPE=STRING,L34PROTOCOL=TCP;
ADD FILTER:FILTERNAME="filter_bg",L34PROTTYPE=STRING,L34PROTOCOL=TCP;

// 配置三个流过滤器
ADD FLOWFILTER:FLOWFILTERNAME="ff_video";
ADD FLOWFILTER:FLOWFILTERNAME="ff_web";
ADD FLOWFILTER:FLOWFILTERNAME="ff_bg";

// 绑定过滤器
ADD FLTBINDFLOWF:FLOWFILTERNAME="ff_video",FILTERNAME="filter_video";
ADD FLTBINDFLOWF:FLOWFILTERNAME="ff_web",FILTERNAME="filter_web";
ADD FLTBINDFLOWF:FLOWFILTERNAME="ff_bg",FILTERNAME="filter_bg";

// 刷新使配置生效
SET REFRESHSRV:REFRESHTYPE=ALL;

// 配置PCC动作属性（所有业务下行门控均为PASS）
ADD PCCACTIONPROP:PCCACTPROPNAME="pap_video",UPINITDNGATE=PASS;
ADD PCCACTIONPROP:PCCACTPROPNAME="pap_web",UPINITDNGATE=PASS;
ADD PCCACTIONPROP:PCCACTPROPNAME="pap_bg",UPINITDNGATE=PASS;

// 配置PCC策略组
ADD PCCPOLICYGRP:PCCPOLICYGRPNM="ppg_video",PCCACTPROPNAME="pap_video";
ADD PCCPOLICYGRP:PCCPOLICYGRPNM="ppg_web",PCCACTPROPNAME="pap_web";
ADD PCCPOLICYGRP:PCCPOLICYGRPNM="ppg_bg",PCCACTPROPNAME="pap_bg";

// 配置业务规则——视频流媒体（高优先级，FPI=56）
ADD RULE:RULENAME="rule_video_pcc",POLICYTYPE=PCC,FILTERINGMODE=FLOWFILTER,FLOWFILTERNAME="ff_video",PRIORITY=1,POLICYNAME="ppg_video";
ADD RULE:RULENAME="rule_video_fpi",POLICYTYPE=REMARK_FPI,FILTERINGMODE=FLOWFILTER,FLOWFILTERNAME="ff_video",PRIORITY=2,REMARKFPISEL=FPI,FPIVALUE=56;

// 配置业务规则——网页浏览（中优先级，FPI=48）
ADD RULE:RULENAME="rule_web_pcc",POLICYTYPE=PCC,FILTERINGMODE=FLOWFILTER,FLOWFILTERNAME="ff_web",PRIORITY=1,POLICYNAME="ppg_web";
ADD RULE:RULENAME="rule_web_fpi",POLICYTYPE=REMARK_FPI,FILTERINGMODE=FLOWFILTER,FLOWFILTERNAME="ff_web",PRIORITY=2,REMARKFPISEL=FPI,FPIVALUE=48;

// 配置业务规则——后台同步（低优先级，FPI=10，队列1权重仅1）
ADD RULE:RULENAME="rule_bg_pcc",POLICYTYPE=PCC,FILTERINGMODE=FLOWFILTER,FLOWFILTERNAME="ff_bg",PRIORITY=1,POLICYNAME="ppg_bg";
ADD RULE:RULENAME="rule_bg_fpi",POLICYTYPE=REMARK_FPI,FILTERINGMODE=FLOWFILTER,FLOWFILTERNAME="ff_bg",PRIORITY=2,REMARKFPISEL=FPI,FPIVALUE=10;
```

**调度效果预期**：
- 空口资源紧张时：视频业务和网页业务（权重10）获得约10倍于后台同步业务（权重1）的调度机会。
- 空口资源充足时：三类业务均能获得所需带宽。
- 视频和网页业务的有效带宽体验在网络繁忙时得到保障。

### 6.5 常见问题排查

| 问题现象 | 可能原因 | 排查方法 | 解决方案 |
|----------|----------|----------|----------|

| RAN | 无特殊需求 | 根据数据报文的FPI信息进行资源调度 |
| PGW-U/UPF | UDG 20.15.0及后续版本 | PGW-U/UPF作为策略中心，本地生成FPI规则。将FPI信息映射到业务数据报文的DSCP部分或GTP-U扩展头传递到RAN侧。 |

### 7.5 发布历史

| 特性版本 | 发布版本 | 发布说明 |
|----------|----------|----------|
| 2 | 20.15.0 | 支持能力协商，支持将FPI值映射到GTP-U扩展头。新增GTP-U扩展头传递方式、INTERNAL/EXTERNAL策略来源选择、UDG与RAN侧能力协商机制。 |
| 1 | 20.6.0 | 首次发布。支持DSCP方案，将FPI值映射到报文DSCP字段传递给无线侧。 |

### 7.6 应用限制

1. 本特性属于**解决方案特性**，要求无线侧网元支持基于业务流标识的无线资源优化功能。
2. GTP-U方案需整网支持，需要整网规划MTU值（GTP-U扩展头最长12字节）。
3. FPI功能开启后，要求无线侧同时支持该功能，否则FPI信息无法被正确解析和调度。
4. FPI值与队列号映射在RAN上固定，不能通过命令修改；队列调度权重值可在RAN侧通过命令修改。
