### 1.2 核心能力

基于业务感知的带宽控制（BWM, Bandwidth Management）特性，是UNC侧（控制面）通过PCC策略机制，将带宽控制规则下发给UPF/PGW-U/GGSN-U（用户面），由用户面网元执行业务感知（SA, Service Awareness）识别和带宽管理。

核心能力一句话描述：**当用户面网元通过SA识别出特定业务（如P2P、VoIP）后，按照UNC下发的BWM规则对该业务流量进行上行/下行带宽控制，保护带宽资源不被恶意占用。**

### 1.3 在带宽控制场景中的定位（★核心）

WSFD-211005是带宽控制子场景的**★核心特性**，原因如下：

1. **SA触发BWM的完整链路入口**：本特性定义了从SA业务识别到BWM带宽执行的触发机制。在UNC侧，通过配置POLICYTYPE=BWM的规则（ADD RULE），将带宽控制策略绑定到UserProfile，再通过N4接口下发到UPF执行。
2. **控制面决策中枢**：UNC作为控制面，负责接收PCRF/PCF下发的动态BWM规则或使用本地静态BWM规则，决策"对哪些业务、做什么带宽控制"，然后通过N4接口传递给UPF。
3. **与其他带宽控制特性的协同基础**：本特性提供的Rule→UserProfile→UserProfileGroup→APN绑定体系，是其他带宽控制特性（如GWFD-110311 UDG侧执行、GWFD-020354 Shaping整形等）的控制面基础。

## 2. 核心概念

### 2.1 SA（Service Awareness）业务感知

SA是用户面网元（UPF/PGW-U/GGSN-U）对用户数据流进行深度包检测（DPI）的能力。通过预置的业务特征库，用户面网元能够识别出流量属于哪类业务（如P2P、VoIP、视频流媒体、网络游戏等）。

- **SA识别的对象**：SVC（Service，业务大类）和APP（Application，具体应用）。例如，SVC="P2P"下可包含BitTorrent、eMule等多个APP。
- **SA识别机制**：基于协议特征、端口特征、行为特征、流量模式等多维度匹配。
- **SA在BWM中的角色**：SA识别出业务后，将业务标识传递给BWM模块，BWM根据预设规则对匹配的业务流执行带宽控制。

### 2.2 BWM（Bandwidth Management）带宽管理

BWM是用户面网元根据控制面下发的策略，对特定业务流量进行速率控制的能力。BWM包含两种执行模式：

| 模式 | 英文 | 机制 | 特点 |
|------|------|------|------|
| 整形 | Shaping | 令牌桶平滑，缓冲超出速率的报文 | 平滑输出，可能增加延迟，不丢包 |
| 监管 | Policing | 令牌桶直接丢弃超出速率的报文 | 立即生效，可能丢包，不增加延迟 |

### 2.3 CIR/PIR/CBS/PBS 带宽参数

| 参数 | 全称 | 含义 | 说明 |
|------|------|------|------|
| CIR | Committed Information Rate | 承诺信息速率 | 保证的最低带宽。低于此速率的流量总是被放行。 |
| PIR | Peak Information Rate | 峰值信息速率 | 允许的最大带宽。超出此速率的流量被限制。 |
| CBS | Committed Burst Size | 承诺突发大小 | 允许的承诺速率下的突发数据量。通常按 CIR × 时间间隔 计算。 |
| PBS | Peak Burst Size | 峰值突发大小 | 允许的峰值速率下的突发数据量。通常按 PIR × 时间间隔 计算。 |

带宽执行逻辑（双速令牌桶）：
- 流量 ≤ CIR：放行（绿色）
- CIR < 流量 ≤ PIR：根据策略放行或限制（黄色）
- 流量 > PIR：丢弃或降级（红色）

### 2.4 Shaping vs Policing 对比

| 维度 | Shaping（整形） | Policing（监管） |
|------|-----------------|------------------|
| 处理方式 | 缓冲超额报文，延迟发送 | 直接丢弃超额报文 |
| 报文丢失 | 低（缓冲满才丢） | 高（立即丢弃） |
| 延迟影响 | 增加延迟（排队等待） | 不增加延迟 |
| 抖动 | 可能产生抖动 | 无额外抖动 |
| TCP重传 | 较少（平滑降速） | 较多（丢包触发重传） |
| 适用场景 | 需要平滑限速的场景（如视频保障） | 需要硬限制的场景（如P2P限速） |
| 资源消耗 | 较高（需要缓冲队列） | 较低（无需缓冲） |

### 2.5 CAR（Committed Access Rate）

CAR是基于令牌桶的速率限制机制。在BWM中，CAR定义了对特定业务流的速率限制参数集（包含CIR、CBS、PIR、PBS等）。每条BWM规则关联一个CAR参数集来执行具体的速率控制。

### 2.6 PCC规则与BWM策略的关系

在UNC侧，带宽控制规则是PCC规则的一种特殊类型，通过策略类型（POLICYTYPE）字段区分：

| POLICYTYPE取值 | 含义 | 说明 |
|----------------|------|------|
| BWM | 带宽管理策略 | 用于基于业务感知的带宽控制 |
| CHARGING | 计费策略 | 用于内容计费、流量计费等 |
| 其他策略类型 | 其他PCC策略 | 视具体特性而定 |

当POLICYTYPE=BWM时，该规则被识别为带宽控制规则，通过PCC体系下发到用户面后，由UPF执行SA识别和带宽管理。

### 2.7 关键配置对象

| 对象 | MML命令 | 作用 | 说明 |
|------|---------|------|------|
| Rule（规则） | ADD RULE | 定义BWM策略规则 | POLICYTYPE=BWM标识为带宽控制规则 |
| UserProfile（用户模板） | ADD USERPROFILE | 规则容器，绑定多个规则 | UPTYPE=PCC |
| RuleBinding（规则绑定） | ADD RULEBINDING | 将Rule绑定到UserProfile | 支持指定策略类型绑定 |
| UserProfileGroup（用户模板组） | ADD USRPROFGROUP | UserProfile的容器 | 用于APN级绑定 |
| UPBindUPG（用户模板绑定） | ADD UPBINDUPG | 将UserProfile绑定到UserProfileGroup | 支持DEFAULT等绑定类型 |
| APNUserProfileG（APN用户模板组绑定） | ADD APNUSRPROFG | 将UserProfileGroup绑定到DNN/APN | 实现APN级策略下发 |

### 2.8 BWM执行层级

用户面网元的BWM可以在多个层级执行带宽控制：

| 层级 | 说明 | 典型场景 |
|------|------|----------|
| 整机级 | 对整机所有流量进行总带宽限制 | 保护网元总带宽 |
| 用户组级 | 对特定用户组进行带宽限制 | 基于套餐的带宽控制 |
| 用户级 | 对单个用户进行带宽限制 | 单用户限速 |
| 会话级 | 对单个PDU会话/PDP上下文限制 | 会话级AMBR |
| 业务流级 | 对特定业务流（SDF）限制 | 基于业务感知的精细控制 |

本特性（WSFD-211005）主要关注**业务流级**的带宽控制，即基于SA识别出的特定业务（如P2P、VoIP）进行带宽管理。

## 3. 实现原理与流程

### 3.1 总体架构

基于业务感知的带宽控制采用CU分离架构：

```
┌─────────────────────────────────────────────────────────────────────────┐
│                         控制面 (UNC)                                     │
│                                                                         │
│  PCRF/PCF ──(Gx/N7)──► SMF/PGW-C/GGSN-C                                │
│                           │                                             │
│                           │ 1. 接收/配置BWM规则                          │
│                           │ 2. 绑定到UserProfile                        │
│                           │ 3. 绑定到UserProfileGroup                   │
│                           │ 4. 绑定到APN/DNN                            │
│                           │                                             │
│                           ▼ (N4接口)                                    │
│                                                                         │
├─────────────────────────────────────────────────────────────────────────┤
│                         用户面 (UDG)                                     │
│                                                                         │
│  UPF/PGW-U/GGSN-U                                                      │
│     │                                                                   │
│     │ 1. 接收N4下发的BWM规则                                            │
│     │ 2. 对用户流量执行SA识别（DPI）                                     │
│     │ 3. 匹配BWM规则（业务→规则）                                        │
│     │ 4. 执行Shaping/Policing                                           │
│     │                                                                   │
│     ▼                                                                   │
│  用户数据流（受控）                                                      │
└─────────────────────────────────────────────────────────────────────────┘
```

### 3.2 SA识别业务的机制

SA（业务感知）是用户面网元的核心能力，通过以下机制识别业务：

1. **DPI（深度包检测）**：解析报文载荷特征，匹配业务特征库中的签名规则。
2. **特征库**：预置的业务特征库包含各类业务的协议特征、端口特征、行为特征等模式。
3. **业务分类**：识别结果分为SVC（业务大类）和APP（具体应用）两层。例如SVC="P2P"，APP="BitTorrent"。
4. **动态更新**：特征库可在线更新，以适应新业务和新协议。

SA识别在用户面完成，控制面（UNC）不直接参与DPI检测，只负责将"识别到某业务后应执行什么带宽策略"的规则下发给用户面。

### 3.3 BWM规则匹配流程

```
用户数据包到达UPF
    │
    ▼
SA识别模块（DPI检测）
    │
    ├──► 识别出 SVC/APP（如 P2P/BitTorrent）
    │         │
    │         ▼
    │    BWM规则匹配引擎
    │         │
    │         ├── 匹配条件：SVC/APP标识 + 用户信息 + APN/DNN
    │         │
    │         ├── 匹配命中 ──► 执行对应BWM规则
    │         │                    │
    │         │                    ├── Shaping模式：令牌桶平滑，缓冲超额报文
    │         │                    └── Policing模式：令牌桶监管，丢弃超额报文
    │         │
    │         └── 未匹配 ──► 按默认策略处理（通常不限速）
    │
    └──► 未识别业务 ──► 按默认策略处理
```

### 3.4 UNC侧（SMF控制面）的完整流程

#### 3.4.1 动态PCC场景（有PCRF/PCF）

当部署了PCRF/PCF时，带宽控制策略由PCRF/PCF决策：

```
步骤1：PCRF/PCF通过Gx（2G/3G/4G）或N7（5G）接口下发预定义BWM规则名给SMF/PGW-C
         │
         ├── 预定义规则：PCRF/PCF仅发送规则名（如"rule_bwm_p2p"）
         │   SMF和UPF两侧均需预配置同名的规则定义
         │
         └── 预定义规则组：PCRF/PCF发送UserProfile名
             SMF和UPF两侧均需预配置同名的UserProfile

步骤2：SMF/PGW-C接收预定义规则名，在本地Rule/UserProfile配置中查找对应定义
         │
         └── 本地ADD RULE配置的POLICYTYPE=BWM规则即为预定义规则的本地定义

步骤3：SMF/PGW-C通过N4接口将BWM规则传递给UPF/PGW-U/GGSN-U
         │
         └── N4接口消息中携带规则名、策略类型（BWM）、优先级等信息

步骤4：UPF/PGW-U/GGSN-U接收N4消息，安装BWM规则到本地策略执行引擎
         │
         └── 后续用户面数据流触发SA识别和BWM执行
```

#### 3.4.2 本地PCC场景（无PCRF/PCF或PCRF/PCF故障）

当未部署PCRF/PCF，或PCRF/PCF故障时，SMF/PGW-C使用本地静态规则：

```
步骤1：SMF/PGW-C使用本地配置的ADD RULE（POLICYTYPE=BWM）作为本地静态规则

步骤2：通过UserProfile → UserProfileGroup → APN/DNN绑定体系，确定哪些用户应用哪些BWM规则

步骤3：SMF/PGW-C通过N4接口将本地BWM规则传递给UPF/PGW-U/GGSN-U

步骤4：UPF接收并执行，流程同动态PCC场景
```

#### 3.4.3 两种场景的决策路径

```
                    ┌─ 存在PCRF/PCF ──► 动态PCC路径
                    │                    │
用户激活/策略更新 ──┤                    ├── PCRF/PCF下发预定义规则名
                    │                    ├── SMF本地匹配Rule定义
                    │                    └── SMF通过N4下发到UPF
                    │
                    └─ 无PCRF/PCF或故障 ──► 本地PCC路径
                                         │
                                         ├── SMF使用本地配置的Rule
                                         ├── 通过UserProfile绑定体系确定规则
                                         └── SMF通过N4下发到UPF
```

### 3.5 控制面与用户面的协作

| 阶段 | 控制面（UNC/SMF） | 用户面（UDG/UPF） |
|------|-------------------|-------------------|
| 策略准备 | 配置ADD RULE(POLICYTYPE=BWM)、ADD USERPROFILE、绑定关系 | 配置BWMSERVICE、BWMPROFILE等用户面BWM参数 |
| 用户激活 | 接收用户激活请求，确定适用的UserProfile和规则 | 建立用户会话，等待N4规则下发 |
| 策略下发 | 通过N4接口将BWM规则下发给UPF | 接收N4消息，安装BWM规则 |
| 业务识别 | 不参与（由用户面完成） | SA模块执行DPI，识别业务类型 |
| 带宽执行 | 不参与（由用户面完成） | 按BWM规则执行Shaping/Policing |
| 策略更新 | PCRF/PCF触发或本地触发更新，通过N4通知UPF | 接收更新，调整BWM执行参数 |
| 用户去激活 | 删除N4规则，通知UPF清理 | 清理用户BWM规则和统计 |

### 3.6 端到端完整流程

以下描述一个完整的端到端流程，从用户激活到带宽控制生效：

```
1. 前置配置阶段
   a. UNC侧：ADD RULE(RULENAME="bwm_p2p", POLICYTYPE=BWM, PRIORITY=10)
   b. UNC侧：ADD USERPROFILE(USERPROFILENAME="up_bwm", UPTYPE=PCC)
   c. UNC侧：ADD RULEBINDING(USERPROFILENAME="up_bwm", RULENAME="bwm_p2p")
   d. UNC侧：ADD USRPROFGROUP(USERPROFGNAME="upg_bwm")
   e. UNC侧：ADD UPBINDUPG(USERPROFGNAME="upg_bwm", UPBINDTYPE=DEFAULT, USERPROFILENAME="up_bwm")
   f. UNC侧：ADD APNUSRPROFG(APN="internet", USERPROFGNAME="upg_bwm")
   g. UDG侧：配置对应的BWMSERVICE、BWMPROFILE（CIR/PIR/Shaping/Policing参数）

2. 用户激活阶段
   a. UE发起PDU会话建立请求（APN="internet"）
   b. SMF接收请求，查找APN="internet"绑定的UserProfileGroup="upg_bwm"
   c. SMF确定UserProfile="up_bwm"，其中包含BWM规则"bwm_p2p"

3. 策略交互阶段（动态PCC场景）
   a. SMF通过N7/Gx接口向PCRF/PCF发送会话建立请求
   b. PCRF/PCF返回策略决策，可能包含预定义BWM规则名
   c. SMF将PCRF/PCF决策与本地Rule配置匹配

4. N4规则下发阶段
   a. SMF通过N4接口向UPF下发PFCP会话建立请求
   b. N4消息中携带BWM规则（PDR/FAR/QER/URR等IE）
   c. UPF接收并安装规则到本地策略执行引擎

5. SA识别阶段（用户面执行）
   a. 用户开始使用业务（如启动BitTorrent下载）
   b. UPF的SA模块对数据流执行DPI检测
   c. SA识别出业务为 P2P/BitTorrent

6. BWM执行阶段（用户面执行）
   a. BWM规则匹配引擎查找匹配规则（业务标识→规则）
   b. 命中"bwm_p2p"规则，执行对应的Shaping/Policing
   c. 按CIR/PIR参数控制上下行速率

7. 策略更新阶段（可选）
   a. PCRF/PCF可通过RAR/N7通知更新策略
   b. SMF通过N4通知UPF更新BWM规则
   c. UPF调整执行参数

8. 用户去激活阶段
   a. UE发起PDU会话释放
   b. SMF通过N4通知UPF删除规则
   c. UPF清理BWM规则和统计数据
```

### 4.2 ADD RULE 命令详解

ADD RULE是定义BWM规则的核心命令。

| 参数 | 参数名 | 类型 | 必填 | 约束/取值 | 说明 |
|------|--------|------|------|-----------|------|
| RULENAME | 规则名称 | String | 是 | 存在多条数据时，该参数+策略类型不能完全相同 | BWM规则的唯一标识 |
| POLICYTYPE | 策略类型 | Enum | 是 | **BWM**（带宽管理） | 标识此规则为带宽控制规则 |
| PRIORITY | 全局优先级 | Integer | 是 | 值越小，优先级越高 | 规则冲突时的优先级判断 |

配置示例：
```
ADD RULE: RULENAME="rule_test", POLICYTYPE=BWM, PRIORITY=10;
```

关键约束：
- RULENAME + POLICYTYPE 组合必须唯一。同名规则可以存在不同策略类型（如一个BWM、一个CHARGING）。
- PRIORITY值越小优先级越高。当多条规则同时匹配时，优先级高的规则先生效。
- 用于动态PCC时：PCRF/PCF下发预定义规则名，SMF根据本地ADD RULE配置匹配。RULENAME必须与PCRF/PCF侧规划的规则名一致。
- 用于本地PCC时：直接使用本地ADD RULE配置的规则。

### 4.3 ADD USERPROFILE 命令详解

UserProfile是规则的容器，一个UserProfile可以绑定多个Rule。

| 参数 | 参数名 | 类型 | 必填 | 约束/取值 | 说明 |
|------|--------|------|------|-----------|------|
| USERPROFILENAME | 用户模板名称 | String | 是 | 全网规划，需唯一 | UserProfile的唯一标识 |
| UPTYPE | 用户模板类型 | Enum | 是 | **PCC**（固定取值） | 本特性固定使用PCC类型 |

配置示例：
```
ADD USERPROFILE: USERPROFILENAME="up_test", UPTYPE=PCC;
```

### 4.4 ADD RULEBINDING 命令详解

将Rule绑定到UserProfile，实现规则组合。

| 参数 | 参数名 | 类型 | 必填 | 约束/取值 | 说明 |
|------|--------|------|------|-----------|------|
| USERPROFILENAME | 用户模板名称 | String | 是 | 必须已通过ADD USERPROFILE创建 | 目标UserProfile |
| RULENAME | 规则名称 | String | 是 | 必须已通过ADD RULE创建 | 要绑定的规则 |
| POLICYTYPE | 策略类型 | Enum | 否 | BWM/CHARGING等 | 不指定时绑定同名规则的所有策略类型 |

配置示例：
```
ADD RULEBINDING: USERPROFILENAME="up_test", RULENAME="rule_test";
```

关键说明：
- 如果不指定策略类型（POLICYTYPE），则把规则名称相同的所有策略类型的规则绑定到用户模板上。
- 建议在BWM场景中明确指定POLICYTYPE=BWM，避免误绑定其他策略类型的规则。

### 4.5 ADD USRPROFGROUP 命令详解

创建UserProfileGroup，作为UserProfile的容器。

| 参数 | 参数名 | 类型 | 必填 | 约束/取值 | 说明 |
|------|--------|------|------|-----------|------|
| USERPROFGNAME | 用户模板组名称 | String | 是 | 本端规划，需唯一 | UserProfileGroup的唯一标识 |

配置示例：
```
ADD USRPROFGROUP: USERPROFGNAME="upg_test";
```

### 4.6 ADD UPBINDUPG 命令详解

将UserProfile绑定到UserProfileGroup。

| 参数 | 参数名 | 类型 | 必填 | 约束/取值 | 说明 |
|------|--------|------|------|-----------|------|
| USERPROFGNAME | 用户模板组名称 | String | 是 | 必须已通过ADD USRPROFGROUP创建 | 目标UserProfileGroup |
| UPBINDTYPE | 用户模板绑定类型 | Enum | 是 | DEFAULT等 | 绑定类型 |
| USERPROFILENAME | 用户模板名称 | String | 是 | 必须已通过ADD USERPROFILE创建 | 要绑定的UserProfile |

配置示例：
```
ADD UPBINDUPG: USERPROFGNAME="upg_test", UPBINDTYPE=DEFAULT, USERPROFILENAME="up_test";
```

### 4.7 ADD APNUSRPROFG 命令详解

将UserProfileGroup绑定到DNN/APN，实现APN级策略下发。

| 参数 | 参数名 | 类型 | 必填 | 约束/取值 | 说明 |
|------|--------|------|------|-----------|------|
| APN | APN/DNN名称 | String | 是 | 全网规划 | 目标APN/DNN |
| USERPROFGNAME | 用户模板组名称 | String | 是 | 必须已通过ADD USRPROFGROUP创建 | 要绑定的UserProfileGroup |

配置示例：
```
ADD APNUSRPROFG: APN="huawei.com", USERPROFGNAME="upg_test";
```

### 4.8 配置顺序

完整的配置顺序如下：

```
步骤1：License配置
  LST LICENSESWITCH（检查License开关状态）
  SET LICENSESWITCH（打开"82200CQU LKV3TCBSA01 基于业务感知的带宽控制"开关）

步骤2：前置依赖检查
  确认WSFD-109101 PCC基本功能已激活
  确认N4接口配置完成

步骤3：配置BWM规则
  ADD RULE: RULENAME="规则名", POLICYTYPE=BWM, PRIORITY=优先级;

步骤4：配置UserProfile并绑定规则
  ADD USERPROFILE: USERPROFILENAME="模板名", UPTYPE=PCC;
  ADD RULEBINDING: USERPROFILENAME="模板名", RULENAME="规则名";

步骤5：配置UserProfileGroup并绑定UserProfile
  ADD USRPROFGROUP: USERPROFGNAME="模板组名";
  ADD UPBINDUPG: USERPROFGNAME="模板组名", UPBINDTYPE=DEFAULT, USERPROFILENAME="模板名";

步骤6：绑定到APN/DNN
  ADD APNUSRPROFG: APN="DNN名", USERPROFGNAME="模板组名";
```

### 4.9 关键参数说明

#### 4.9.1 POLICYTYPE=BWM 的含义

POLICYTYPE=BWM是本特性的关键标识。当ADD RULE的POLICYTYPE设置为BWM时：
- 该规则被标记为带宽管理策略规则。
- PCRF/PCF可通过预定义规则名引用该规则（动态PCC场景）。
- SMF可直接安装该规则到UserProfile（本地PCC场景）。
- N4接口下发时，携带BWM策略标识，UPF据此执行带宽管理。

#### 4.9.2 PRIORITY 优先级机制

- 值越小，优先级越高。
- 当多个规则同时匹配同一业务流时，优先级高的规则先生效。
- 动态规则与预定义规则优先级相同时，动态规则优先。
- 规划建议：不同业务类型的BWM规则应设置不同的优先级，避免冲突。

#### 4.9.3 UPBINDTYPE 绑定类型

| 取值 | 含义 | 说明 |
|------|------|------|
| DEFAULT | 默认绑定 | 该UserProfile作为UserProfileGroup的默认模板 |

### 4.10 UDG侧（用户面）配置要点

BWM的实际执行参数（CIR/PIR/CBS/PBS/Shaping/Policing模式）在UDG侧配置。UNC侧的ADD RULE定义的是控制面策略逻辑（规则名、策略类型、优先级），UDG侧的BWMSERVICE和BWMPROFILE定义的是执行参数。

| UDG侧配置 | 作用 | 对应UNC侧 |
|-----------|------|-----------|
| BWMSERVICE | 定义BWM服务实例 | 对应UNC侧的BWM规则 |
| BWMPROFILE | 定义带宽参数（CIR/PIR等） | UNC侧不直接配置，通过N4下发 |
| SVC/APP定义 | 定义业务感知规则 | UNC侧不参与，由用户面DPI完成 |

> 注意：UNC侧ADD RULE的RULENAME必须与UDG侧预定义规则名称一致（动态PCC场景），或与PCRF/PCF规划一致。两侧规则名不匹配将导致策略下发失败。

## 5. 配置案例

### 案例1：P2P业务带宽控制（基础场景）

**业务场景描述**：
运营商需要对P2P业务（如BitTorrent）进行带宽控制，限制其上下行速率，避免P2P流量占用大量网络带宽资源，确保其他业务正常使用。P2P业务的上行限制为10Mbps，下行限制为50Mbps。

**前提条件**：
- License "82200CQU LKV3TCBSA01 基于业务感知的带宽控制" 已加载并使能
- WSFD-109101 PCC基本功能已激活
- UDG侧已完成SA特征库加载和BWMPROFILE配置（CIR/PIR参数）
- N4接口配置完成

**配置步骤（UNC侧命令序列）**：

```
// 步骤1：检查并使能License
LST LICENSESWITCH:LICITEM="LKV3TCBSA01";
// 如未使能，执行：
SET LICENSESWITCH:LICITEM="LKV3TCBSA01",SWITCH=ENABLE;

// 步骤2：配置P2P业务的BWM规则
// 规则名"bwm_p2p"需与UDG侧BWMSERVICE配置和PCRF/PCF规划一致
ADD RULE: RULENAME="bwm_p2p", POLICYTYPE=BWM, PRIORITY=10;

// 步骤3：配置UserProfile并绑定BWM规则
ADD USERPROFILE: USERPROFILENAME="up_bwm_p2p", UPTYPE=PCC;
ADD RULEBINDING: USERPROFILENAME="up_bwm_p2p", RULENAME="bwm_p2p", POLICYTYPE=BWM;

// 步骤4：配置UserProfileGroup并绑定UserProfile
ADD USRPROFGROUP: USERPROFGNAME="upg_bwm_p2p";
ADD UPBINDUPG: USERPROFGNAME="upg_bwm_p2p", UPBINDTYPE=DEFAULT, USERPROFILENAME="up_bwm_p2p";

// 步骤5：绑定到DNN
ADD APNUSRPROFG: APN="internet", USERPROFGNAME="upg_bwm_p2p";
```

**关键参数取值说明**：
- RULENAME="bwm_p2p"：规则名需全网一致（UNC/UDG/PCRF侧）
- POLICYTYPE=BWM：标识为带宽管理策略
- PRIORITY=10：优先级，根据实际规划设置
- APN="internet"：目标DNN，所有使用该DNN的用户将应用此BWM策略

**验证方式**：
1. 执行 LST RULE 确认规则配置正确
2. 测试终端使用"internet" DNN激活
3. 执行 DSP PDUSESSION 确认会话激活成功
4. 检查N4接口跟踪报文，确认BWM规则已传递给UPF
5. 在UDG侧验证SA识别和BWM执行效果

### 案例2：VoIP业务带宽保障（多规则场景）

**业务场景描述**：
运营商需要对VoIP业务进行带宽保障，同时对P2P业务进行限速。VoIP业务优先级高于P2P业务。同时配置两种业务的BWM规则，通过一个UserProfile统一管理。

**配置步骤**：

```
// 步骤1：License检查（同案例1，略）

// 步骤2：配置VoIP业务的BWM规则（高优先级）
ADD RULE: RULENAME="bwm_voip", POLICYTYPE=BWM, PRIORITY=5;

// 步骤3：配置P2P业务的BWM规则（低优先级）
ADD RULE: RULENAME="bwm_p2p", POLICYTYPE=BWM, PRIORITY=10;

// 步骤4：配置UserProfile，绑定两个BWM规则
ADD USERPROFILE: USERPROFILENAME="up_bwm_mixed", UPTYPE=PCC;
ADD RULEBINDING: USERPROFILENAME="up_bwm_mixed", RULENAME="bwm_voip", POLICYTYPE=BWM;
ADD RULEBINDING: USERPROFILENAME="up_bwm_mixed", RULENAME="bwm_p2p", POLICYTYPE=BWM;

// 步骤5：配置UserProfileGroup并绑定
ADD USRPROFGROUP: USERPROFGNAME="upg_bwm_mixed";
ADD UPBINDUPG: USERPROFGNAME="upg_bwm_mixed", UPBINDTYPE=DEFAULT, USERPROFILENAME="up_bwm_mixed";

// 步骤6：绑定到DNN
ADD APNUSRPROFG: APN="internet", USERPROFGNAME="upg_bwm_mixed";
```

**关键参数取值说明**：
- bwm_voip PRIORITY=5：VoIP优先级更高（值更小），保障VoIP业务带宽
- bwm_p2p PRIORITY=10：P2P优先级较低，在带宽紧张时优先限制P2P
- 一个UserProfile绑定多个Rule：实现多业务统一策略管理

**验证方式**：
1. LST RULE 查询确认两条BWM规则均已配置
2. VoIP测试：发起VoIP通话，验证通话质量不受P2P流量影响
3. P2P测试：启动P2P下载，验证上下行速率被限制在规划范围内
4. N4跟踪：确认两条规则均已通过N4接口下发到UPF

### 案例3：按APN区分的带宽控制策略

**业务场景描述**：
运营商有不同的DNN/APN提供不同等级服务。"premium" DNN的用户享受更高的带宽保障，"standard" DNN的用户使用标准的P2P限速策略。通过不同的UserProfileGroup绑定到不同APN实现差异化策略。

**配置步骤**：

```
// 步骤1：配置BWM规则
// Premium用户的增强带宽保障规则
ADD RULE: RULENAME="bwm_premium", POLICYTYPE=BWM, PRIORITY=5;
// Standard用户的P2P限速规则
ADD RULE: RULENAME="bwm_p2p", POLICYTYPE=BWM, PRIORITY=10;

// 步骤2：配置Premium UserProfile
ADD USERPROFILE: USERPROFILENAME="up_premium", UPTYPE=PCC;
ADD RULEBINDING: USERPROFILENAME="up_premium", RULENAME="bwm_premium", POLICYTYPE=BWM;

// 步骤3：配置Standard UserProfile
ADD USERPROFILE: USERPROFILENAME="up_standard", UPTYPE=PCC;
ADD RULEBINDING: USERPROFILENAME="up_standard", RULENAME="bwm_p2p", POLICYTYPE=BWM;

// 步骤4：配置Premium UserProfileGroup并绑定到premium DNN
ADD USRPROFGROUP: USERPROFGNAME="upg_premium";
ADD UPBINDUPG: USERPROFGNAME="upg_premium", UPBINDTYPE=DEFAULT, USERPROFILENAME="up_premium";
ADD APNUSRPROFG: APN="premium", USERPROFGNAME="upg_premium";

// 步骤5：配置Standard UserProfileGroup并绑定到standard DNN
ADD USRPROFGROUP: USERPROFGNAME="upg_standard";
ADD UPBINDUPG: USERPROFGNAME="upg_standard", UPBINDTYPE=DEFAULT, USERPROFILENAME="up_standard";
ADD APNUSRPROFG: APN="standard", USERPROFGNAME="upg_standard";
```

**关键参数取值说明**：
- 不同APN绑定不同UserProfileGroup，实现按DNN差异化策略
- Premium用户使用更高优先级的BWM规则（PRIORITY=5）
- Standard用户使用标准P2P限速规则（PRIORITY=10）

**验证方式**：
1. 测试终端使用"premium" DNN激活，验证增强带宽保障
2. 测试终端使用"standard" DNN激活，验证P2P限速效果
3. 对比两个DNN下用户的实际带宽体验差异

### 案例4：动态PCC场景下的BWM配置（有PCRF/PCF）

**业务场景描述**：
运营商部署了PCRF/PCF，由PCRF/PCF动态决策何时对哪些用户启用BWM策略。PCRF/PCF通过N7/Gx接口下发预定义规则名，SMF根据本地配置匹配并执行。

**配置步骤**：

```
// 步骤1：与PCRF/PCF规划一致的预定义BWM规则
// 规则名必须与PCRF/PCF侧配置的预定义规则名完全一致
ADD RULE: RULENAME="predefined_bwm_p2p", POLICYTYPE=BWM, PRIORITY=10;

// 步骤2：配置预定义规则组（UserProfile）
// UserProfile名也需与PCRF/PCF侧规划一致
ADD USERPROFILE: USERPROFILENAME="predefined_up_bwm", UPTYPE=PCC;
ADD RULEBINDING: USERPROFILENAME="predefined_up_bwm", RULENAME="predefined_bwm_p2p", POLICYTYPE=BWM;

// 步骤3：配置UserProfileGroup并绑定到DNN
ADD USRPROFGROUP: USERPROFGNAME="upg_predefined_bwm";
ADD UPBINDUPG: USERPROFGNAME="upg_predefined_bwm", UPBINDTYPE=DEFAULT, USERPROFILENAME="predefined_up_bwm";
ADD APNUSRPROFG: APN="internet", USERPROFGNAME="upg_predefined_bwm";

// 后续由PCRF/PCF动态下发规则激活/去激活指令
// SMF自动将PCRF/PCF决策通过N4接口传递给UPF
```

**关键说明**：
- 动态PCC场景下，规则名和UserProfile名必须与PCRF/PCF侧规划完全一致。
- PCRF/PCF负责决策"何时对哪个用户启用哪个BWM规则"。
- SMF负责匹配本地配置并通过N4下发到UPF。
- UPF负责执行SA识别和BWM带宽控制。
- 规则激活/去激活由PCRF/PCF通过RAR（Gx）或N7通知触发，SMF自动响应。

```
步骤5：验证会话激活
  执行 DSP PDUSESSION 命令，通过IMSI查看测试终端是否激活成功
  ├── 激活成功 → 继续步骤6
  └── 激活失败 → 联系华为技术支持工程师调测基本会话接入功能

步骤6：验证N4规则下发
  检查N4接口跟踪报文，确认指定BWM规则已传递给UPF
  ├── 规则已传递 → 调测结束（控制面侧验证完成）
  └── 规则未传递 → 继续步骤7

步骤7：排查规则配置
  执行 LST RULE 命令查询带宽控制规则的名称是否与UPF/PCF上的配置一致
  ├── 一致 → 继续步骤8
  └── 不一致 → 修改为与规划值一致后重新从步骤4执行

步骤8：联系华为技术支持
  1. 重新执行上述步骤并保存报文
  2. 执行 EXP MML 命令将当前配置数据导出为MML脚本文件并保存
  3. 收集并保存上述所有查询信息
  4. 收集归纳所有信息并联系华为技术支持解决
```

### 6.3 SA识别验证

SA识别在用户面（UPF/PGW-U/GGSN-U）完成，验证方法需在UDG侧操作：

1. **UDG侧SA统计查询**：查询特定SVC/APP的识别命中次数和流量统计
2. **UDG侧SA跟踪**：对特定用户启用SA识别跟踪，查看识别到的业务列表
3. **端到端验证**：使用已知业务类型的应用（如启动BitTorrent客户端），在UDG侧确认SA正确识别为P2P业务

### 6.4 BWM执行验证

BWM执行在用户面完成，验证方法：

1. **UDG侧BWM统计查询**：查询BWM规则的命中次数、限速/整形丢弃报文数、通过报文数
2. **UDG侧流量监控**：实时监控特定用户的上下行流量速率，确认限速效果
3. **端到端验证**：
   - 启动目标业务（如P2P下载）
   - 在UDG侧确认SA识别成功
   - 在UDG侧确认BWM规则命中
   - 使用流量测试工具验证上下行速率被限制在预期范围
4. **Shaping效果验证**：观察流量曲线是否平滑（整形模式）
5. **Policing效果验证**：观察是否有报文丢弃（监管模式）

### 6.5 常见问题与排查思路

#### 问题1：BWM规则未通过N4下发到UPF

**排查步骤**：
1. 执行 LST RULE 确认规则配置存在且POLICYTYPE=BWM
2. 确认RULENAME与PCRF/PCF或本地规划一致
3. 确认Rule已通过ADD RULEBINDING绑定到UserProfile
4. 确认UserProfile已通过ADD UPBINDUPG绑定到UserProfileGroup
5. 确认UserProfileGroup已通过ADD APNUSRPROFG绑定到用户使用的APN/DNN
6. 检查N4接口连接状态是否正常
7. 检查License开关是否使能

#### 问题2：用户激活失败

**排查步骤**：
1. 执行 DSP PDUSESSION 查看会话状态和失败原因
2. 检查APN/DNN配置是否存在（LST APN）
3. 检查PCC基本功能是否正常（WSFD-109101）
4. 检查N4接口连通性
5. 查看用户跟踪任务的N4/Gx/N7消息，定位失败环节

#### 问题3：SA未识别出目标业务

**排查思路**：
- SA识别在用户面（UDG侧）完成，UNC侧无法直接排查
- 需在UDG侧检查SA特征库是否已加载且为最新版本
- 确认目标业务的协议特征已被特征库覆盖
- 使用UDG侧SA跟踪工具验证识别过程

#### 问题4：BWM限速效果不明显

**排查思路**：
- BWM执行在用户面（UDG侧）完成
- 需在UDG侧检查BWMPROFILE的CIR/PIR参数设置
- 确认Shaping/Policing模式选择正确
- 检查BWM执行层级是否正确（整机/用户组/用户/会话/业务流）
- 排查是否存在更高优先级的规则覆盖了BWM规则

#### 问题5：动态PCC场景下PCRF/PCF下发的规则名不匹配

**排查步骤**：
1. 检查PCRF/PCF下发的预定义规则名
2. 执行 LST RULE 确认UNC侧本地配置的规则名
3. 两者必须完全一致（区分大小写）
4. 如不一致，修改UNC侧ADD RULE的RULENAME与PCRF/PCF规划一致

### 7.7 应用限制

1. **业务范围限制**：基于业务感知的带宽控制特性可以对P2P和VoIP协议报文进行带宽控制。是否能够对其他协议报文进行带宽控制需要联系营销支持部确认。
2. **License数量限制**：支持的带宽控制受License申请的数量限制。
3. **系统性能影响**：启用该功能需要GGSN-C/PGW-C/SMF与PCRF/PCF不断进行交互，同时需要动态处理PCRF/PCF下发的策略，对系统性能影响比较大。规划时需考虑启用该特性的用户比例。详细性能影响需要基于流量模型进行评估，请联系华为技术支持获取服务。
4. **法律合规**：流量经营类业务功能，实际部署时需在遵循当地法律允许的目的和范围内提供服务，以保证终端用户的通信自由和业务知情权。

### 7.8 对系统的影响

- **信令交互增加**：SMF/PGW-C与PCRF/PCF之间的Gx/N7信令交互频次增加。
- **N4消息开销**：BWM规则通过N4接口下发，增加N4消息大小和处理开销。
- **用户面执行开销**：UPF执行SA识别和BWM需要额外的CPU和内存资源。
- **规划建议**：启用该特性时需评估用户比例和流量模型，避免对系统整体性能造成过大影响。

### 7.9 计费和话单

本特性不涉及计费与话单。带宽控制是策略执行功能，不产生独立的计费行为。如果同时配置了计费策略（POLICYTYPE=CHARGING），计费行为由计费特性独立处理。

### 7.10 特性规格

本特性无特殊规格限制。具体规格能力受系统整体规格约束（如UserProfile数量、规则数量等，参见WSFD-109101 PCC基本功能的规格限制）。

### 7.11 告警信息

本特性无相关告警。

### 7.12 软参信息

本特性无相关软参。

### 7.13 测量指标

本特性无相关测量指标。带宽控制的执行统计在用户面（UDG侧）完成。
