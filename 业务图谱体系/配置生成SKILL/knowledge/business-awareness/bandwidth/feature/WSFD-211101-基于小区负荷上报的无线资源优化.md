# WSFD-211101 基于小区负荷上报的无线资源优化 - 特性知识文档

> **场景归属**: 业务感知 → 带宽控制子场景（辅助特性：小区负荷触发带宽调整，UNC侧）
> **产品**: UNC (SMF网元)
> **场景定位**: SMF接收小区负荷 → PCRF决策 → 动态调整带宽策略
> **对应UDG特性**: GWFD-110332 基于小区负荷上报的无线资源优化

---

## 1. 特性概述

### 1.1 基本信息

| 项目 | 内容 |
|------|------|
| 特性ID | WSFD-211101 |
| 特性名称 | 基于小区负荷上报的无线资源优化 |
| 所属产品 | UNC 20.7.0及后续版本 |
| 涉及网元 | GGSN-C / PGW-C（控制面），PGW-U（用户面，属UDG） |
| 适用NF | GGSN、PGW-C |
| 首次发布 | UNC 20.7.0 |
| License控制项 | 82209457 LKV3W9WOCR11 基于小区负荷上报的无线资源优化 |

### 1.2 核心能力一句话描述

当无线侧小区负荷状态发生变化时，PGW-U将小区负荷信息上报给GGSN-C/PGW-C（UNC侧），由GGSN-C/PGW-C通过Gx接口上报给PCRF，PCRF根据负荷信息和用户签约信息进行策略决策，动态下发新策略对指定用户或业务进行带宽控制，达到解除/减缓小区拥塞的目的。

### 1.3 在带宽控制场景中的定位

本特性是带宽控制场景中的**辅助特性**，核心定位是：

- **动态负荷触发器**：小区负荷（Cell Load level）变化作为动态带宽策略调整的触发条件
- **实时感知**：实时感知无线侧拥塞状态，将触发事件传递给策略决策链路
- **策略联动**：与BWM（基于业务感知的带宽控制，WSFD-211005）协作，PCRF决策后下发的带宽策略可能涉及业务感知识别（当业务无法通过三四层特征识别时需依赖BWM）

### 1.4 与GWFD-110332（UDG侧）的对应关系

| 维度 | WSFD-211101（UNC侧，本文档） | GWFD-110332（UDG侧） |
|------|------------------------------|----------------------|
| 网元 | GGSN-C/PGW-C（控制面） | PGW-U（用户面） |
| 核心职责 | 接收PGW-U上报的负荷信息 → 转发至PCRF → 接收PCRF策略 → 下发策略给PGW-U | 与RAN侧进行能力协商 → 检测小区负荷变化 → 上报负荷信息给PGW-C → 执行策略对业务进行带宽控制 |
| 接口角色 | Gx接口（与PCRF）、N4/PFCP接口（与PGW-U） | GTP-U扩展头（与RAN侧）、PFCP接口（与PGW-C） |
| 能力协商 | 参与trigger安装 | 与无线侧通过GTP-U扩展头进行能力协商 |
| 配置焦点 | APN级拥塞上报使能、License开关 | PGW-U侧负荷上报功能使能 |

两个特性是端到端解决方案的两面：GWFD-110332负责用户面的负荷感知与策略执行，WSFD-211101负责控制面的信息传递与策略路由。

### 1.5 与BWM、FUP的协作关系

- **与WSFD-211005（基于业务感知的带宽控制）的交互**：若PCRF下发的策略需要对特定业务进行限制，且该业务无法通过三四层特征识别时，需要部署BWM特性。本特性提供负荷触发条件，BWM提供业务识别能力，两者配合实现"拥塞时限制特定业务带宽"的策略。
- **与WSFD-109101（PCC基本功能）的依赖关系**：本特性强依赖PCC基本功能，CELL-CONGESTION-CHANGE trigger通过Gx接口下发，PCC策略通过N4接口传递。开通本特性必须同时开启PCC基本功能。

## 2. 核心概念

### 2.1 小区负荷（Cell Load）

小区负荷是无线侧UTRAN/eNodeB对当前小区无线资源使用状态的量化度量，存在不同的负荷等级（Cell Load level）。当小区出现拥塞时，负荷状态会发生变更，触发上报流程。源文档中明确区分了两种拥塞状态：

- **Congestion**（拥塞）：小区资源紧张但仍可服务
- **Overload**（过载）：小区资源极度紧张

### 2.2 负荷上报机制（UNC侧视角）

UNC侧（GGSN-C/PGW-C）在负荷上报链路中承担**中继和转发**角色，上报路径为：

```
UTRAN/eNodeB --(GTP-U扩展头)--> PGW-U --(PFCP消息)--> GGSN-C/PGW-C --(Gx CCR消息)--> PCRF
```

UNC侧的关键动作：
1. 接收PGW-U通过PFCP消息上报的小区负荷状态变更信息
2. 将小区负荷信息和用户实时位置通过Gx接口的CCR-U消息上报给PCRF
3. 接收PCRF下发的新策略并安装
4. 将新策略通过N4/PFCP接口下发给PGW-U执行

### 2.3 负荷阈值与触发条件

负荷上报的触发不依赖固定阈值配置，而是基于**小区负荷状态变更事件**。无线侧检测到用户的Cell Load level发生变化时即触发上报。关键触发器为：

- **CELL-CONGESTION-CHANGE Trigger**（事件编号1003）：由PCRF下发给PGW-C，再传递给PGW-U。PGW-U安装该trigger后，当检测到小区负荷状态变化时触发上报流程。

### 2.4 动态带宽策略调整

PCRF收到CCR-U消息中的小区负荷信息和用户位置信息后，综合用户签约信息进行策略决策，可能下发的策略包括：

- 对低优先级用户进行带宽限制
- 对特定业务（如P2P）进行限制
- 将有限资源向高价值用户和高价值业务倾斜
- 拥塞解除后恢复正常带宽策略

### 2.5 N7/N4接口的负荷信息传递

在5G SA架构中（SMF/UPF对应PGW-C/PGW-U的演进），负荷信息传递的接口映射关系：

| 4G/EPC术语 | 5G SA术语 | 本特性文档使用术语 | 职责 |
|------------|-----------|-------------------|------|
| Gx接口 | N7接口 | Gx接口 | PCRF/PCF ↔ PGW-C/SMF之间的策略控制接口，传递trigger、策略规则 |
| Sx/PFCP接口 | N4接口 | N4/PFCP接口 | PGW-C/SMF ↔ PGW-U/UPF之间的接口，传递trigger、PCC规则、负荷信息 |
| GTP-U扩展头 | GTP-U扩展头 | GTP-U扩展头 | RAN ↔ PGW-U/UPF之间的用户面扩展，携带小区负荷信息 |

## 3. 实现原理与流程

### 3.1 组网要求

本特性属于**端到端解决方案特性**，要求：

- 端到端网元都需要是HUAWEI设备
- 无线侧网元（UTRAN和eNodeB）支持基于小区负荷上报的无线资源优化功能
- SGSN/SGW能透传扩展的GTP-U上下行报文
- 用户为PCC本地用户（不支持拜访用户和漫游用户）
- PCRF支持小区负荷上报功能

组网结构与基本PCC架构相同，但要求无线侧和核心网侧均支持本特性。

### 3.2 能力协商过程

能力协商用于确认UTRAN/eNodeB和PGW-U都是华为设备，且都部署了基于小区负荷上报的无线资源优化功能。

**步骤1：Trigger下发（PCRF → PGW-C → PGW-U）**

1. PCC用户激活时，PCRF将CELL-CONGESTION-CHANGE trigger下发给PGW-C
2. PGW-C将CELL-CONGESTION-CHANGE trigger传递给PGW-U
3. PCRF也可以选择在任意CCA消息或RAR消息中下发该trigger
4. PGW-C和PGW-U都需要安装该trigger

**步骤2：能力协商（RAN ↔ PGW-U）**

1. PCC用户激活后，无线侧与PGW-U通过GTP-U扩展头进行能力协商

> 注意：能力协商过程的详细实现请参见UDG产品手册中"GWFD-110332 基于小区负荷上报的无线资源优化"。

### 3.3 小区负荷上报与策略更新过程

以下为完整的小区负荷上报及策略调整端到端流程：

```
┌─────────────┐     ┌─────────┐     ┌──────────────┐     ┌─────────────┐     ┌──────┐
│ UTRAN/eNodeB│     │  PGW-U  │     │ GGSN-C/PGW-C │     │    PCRF     │     │ SGSN │
│  (RAN侧)    │     │ (UDG)   │     │   (UNC)      │     │             │     │/SGW-U│
└──────┬──────┘     └────┬────┘     └──────┬───────┘     └──────┬──────┘     └──┬───┘
       │                 │                 │                    │               │
       │  ① 小区负荷状态变化                                               │
       │  通过GTP-U扩展头                                                  │
       │  上报用户位置+负荷信息  ────────→ │                               │
       │                 │  (经SGSN/SGW-U透传)                              │
       │                 │                 │                    │               │
       │                 │  ② PFCP消息     │                    │               │
       │                 │  传递负荷信息    │                    │               │
       │                 │  +CELL-CONGESTION-CHANGE              │               │
       │                 │  -CHANGE trigger  │                   │               │
       │                 ├────────────────→│                    │               │
       │                 │                 │  ③ Gx CCR-U消息    │               │
       │                 │                 │  上报用户位置+负荷  │               │
       │                 │                 │  +3GPP-User-       │               │
       │                 │                 │   Location-Info    │               │
       │                 │                 ├───────────────────→│               │
       │                 │                 │                    │ ④策略决策      │
       │                 │                 │                    │ 综合负荷+签约  │
       │                 │                 │                    │               │
       │                 │                 │  ⑤ 下发新策略      │               │
       │                 │                 │  (CCA/RAR)         │               │
       │                 │                 │←───────────────────┤               │
       │                 │                 │                    │               │
       │                 │  ⑥ 安装策略后   │                    │               │
       │                 │  下发PCC规则    │                    │               │
       │                 │  给PGW-U        │                    │               │
       │                 │←────────────────┤                    │               │
       │                 │                 │                    │               │
       │                 │  ⑦ 根据策略对   │                    │               │
       │                 │  低优先级用户/  │                    │               │
       │                 │  指定业务进行   │                    │               │
       │                 │  带宽控制       │                    │               │
       │                 │                 │                    │               │
```

**详细步骤说明：**

1. **小区负荷上报（RAN → PGW-U）**：无线侧和PGW-U都具有小区负荷上报能力的情况下，无线侧检测到用户小区负荷状态（Cell Load level）发生变化，则通过数据报文中的GTP-U扩展头将用户位置信息及小区负荷信息上报给PGW-U。此信息经SGSN/SGW-U透传。

2. **负荷信息上报（PGW-U → PGW-C）**：PGW-U检测到小区负荷状态发生变化，且已安装CELL-CONGESTION-CHANGE trigger触发器，则通过PFCP消息传递给PGW-C。

3. **负荷信息上报至PCRF（PGW-C → PCRF）**：PGW-C通过Gx接口的CCR-U消息将用户位置信息（3GPP-User-Location-Info）及小区负荷状态（CELL-CONGESTION-CHANGE trigger）上报给PCRF。

4. **PCRF策略决策**：PCRF综合上报的小区负荷信息、用户实时位置和用户签约信息进行策略决策。

5. **策略下发（PCRF → PGW-C）**：PCRF下发新的策略给PGW-C（通过CCA或RAR消息）。

6. **策略下发（PGW-C → PGW-U）**：PGW-C安装规则后再通过N4/PFCP接口下发给PGW-U。

7. **策略执行（PGW-U）**：PGW-U根据更新后的策略对低优先级用户或指定业务进行带宽控制。

### 3.4 关键消息和信元

| 接口 | 消息/信元 | 说明 |
|------|-----------|------|
| Gx（PGW-C ↔ PCRF） | CCR-U | PGW-C向PCRF上报的信用控制请求消息，携带CELL-CONGESTION-CHANGE(1003) Trigger和3GPP-User-Location-Info |
| Gx（PCRF → PGW-C） | CCA / RAR | PCRF向PGW-C下发策略和trigger的消息，CCA-Initial时即可下发CELL-CONGESTION-CHANGE trigger |
| Gx（PGW-C → PGW-C） | CELL-CONGESTION-CHANGE(1003) Trigger | 小区拥塞变化事件触发器，由PCRF下发，PGW-C和PGW-U均需安装 |
| Gx | ULI AVP（User-Location-Info） | 用户位置信息，在CCR-U消息中携带。软参DWORD519 Bit14控制是否产生2个ULI AVP |
| N4/PFCP（PGW-U ↔ PGW-C） | PFCP消息 | 负荷信息从PGW-U传递到PGW-C的载体 |
| GTP-U扩展头（RAN ↔ PGW-U） | GTP-U Extension Header | 携带用户位置信息及小区负荷信息的数据报文扩展头 |

### 4.4 负荷阈值与触发策略联动配置

本特性的负荷触发策略联动不在UNC侧直接配置阈值，而是通过以下方式实现：

1. **UNC侧（本特性范围）**：通过SET APNREPORTATTR使能APN级拥塞上报，使PGW-C能够正确传递负荷信息
2. **PCRF侧（非本特性配置范围）**：
   - PCRF上规划策略时需要增加下发小区负荷状态变化上报trigger事件的配置
   - PCRF规划基于小区负荷状态的策略（不同负荷等级对应不同的带宽控制策略）
3. **PGW-U侧（非本特性配置范围）**：
   - PGW-U上需要使能基于小区负荷上报的无线资源优化功能
4. **RAN侧（非本特性配置范围）**：
   - RAN侧需要打开小区拥塞状态上报开关

### 4.5 软件参数

| 软参 | 位置 | 作用 |
|------|------|------|
| DWORD519 Bit14 | PCC策略控制管理 > 软件参数（SMF_GGSN_SGW-C_PGW-C） | 控制在开启本特性后，当小区拥塞上报位置时，Gx的CCR-U中是否会有2个ULI AVP |

### 4.6 配置前置条件

执行本特性配置前，必须满足：

1. 已阅读并理解WSFD-211101特性文档
2. 已阅读并理解WSFD-109101 PCC基本功能（2G/3G/4G，基于Gx接口）特性文档
3. 已完成激活PCC基本功能
4. 已完成加载License（82209457 LKV3W9WOCR11）

### 4.7 对系统的影响

- RAN侧和核心网之间传输携带GTP-U扩展头的IP报文，可能会影响网络中MTU值
- GGSN-C/PGW-C需要实时向PCRF上报小区负荷变化事件，有信令交互，会增加GGSN-C/PGW-C的CPU资源消耗和信令开销，具体消耗程度需要根据当前话务模型进行评估

## 5. 配置案例

### 5.1 案例1：高负荷触发限速 - APN级小区拥塞上报

**场景描述：**

通过APN"apn1"接入的小区存在拥塞，且无法扩容。运营商需要部署基于小区负荷上报的无线资源优化功能，通过限制低优先级用户和特定业务（如P2P）来解除/减缓小区拥塞，将有限资源向高价值用户倾斜。

**端到端配置序列：**

**步骤1：打开License配置开关（UNC侧）**

```
// 打开本特性的License配置开关
SET LICENSESWITCH:LICITEM="LKV3W9WOCR11",SWITCH=ENABLE;
```

**步骤2：使能APN的小区拥塞上报功能（UNC侧）**

```
// 使能APN"apn1"支持基于小区负荷上报的无线资源优化功能
SET APNREPORTATTR: APN="apn1", CONGESTIONRPT=ENABLE;
```

**步骤3：验证配置（UNC侧）**

```
// 查询License开关是否打开
LST LICENSESWITCH:LICITEM="LKV3W9WOCR11";
// 预期结果：SWITCH = ENABLE

// 查询APN的拥塞上报是否使能
LST APNREPORTATTR:APN="apn1";
// 预期结果：拥塞控制 = 使能
```

**步骤4：PCRF侧配置（非MML，需PCRF侧工程师配合）**

- 配置下发CELL-CONGESTION-CHANGE trigger事件的规则
- 规划基于小区负荷状态的PCC策略（如：Congestion状态时限制低优先级用户带宽至1Mbps）
- 配置策略更新规则（如：拥塞解除后恢复正常带宽策略）

**步骤5：PGW-U侧配置（UDG侧，参见GWFD-110332）**

- 使能PGW-U的基于小区负荷上报的无线资源优化功能
- 配置预定义规则（RULENAME需与PCRF下发的规则名称一致）

**步骤6：RAN侧配置（需RAN侧工程师配合）**

- 打开小区拥塞状态上报开关
- 确保支持与核心网侧进行能力协商

**效果验证：**

```
// 在OM Portal创建用户跟踪任务，跟踪IMSI=460000123456789的用户
// 测试终端使用apn1接入网络
// 增加接入用户数和流量，使小区拥塞状态变为Congestion或Overload

// 检查用户跟踪结果：
// 1. PCRF在CCA-Initial是否下发CELL-CONGESTION-CHANGE(1003) Trigger给PGW-C → 应为是
// 2. PGW-C是否下发CELL-CONGESTION-CHANGE(1003) Trigger给PGW-U → 应为是
// 3. 拥塞发生后是否有CCR-U消息，且携带CELL-CONGESTION-CHANGE(1003) Trigger和3GPP-User-Location-Info → 应为是
// 4. PCRF是否下发新策略（如限速规则） → 应为是
// 5. PGW-U是否执行限速 → 需在PGW-U侧验证
```

### 5.2 案例2：多APN小区拥塞上报部署

**场景描述：**

运营商有多个APN（apn_internet、apn_iot、apn_voip），其中apn_internet和apn_iot所在小区可能出现拥塞，需要对这两个APN使能小区负荷上报功能，而apn_voip因需保障语音质量不进行拥塞限制。

**配置序列：**

```
// 步骤1：打开License开关（仅一次）
SET LICENSESWITCH:LICITEM="LKV3W9WOCR11",SWITCH=ENABLE;

// 步骤2：对internet APN使能拥塞上报
SET APNREPORTATTR: APN="apn_internet", CONGESTIONRPT=ENABLE;

// 步骤3：对IoT APN使能拥塞上报
SET APNREPORTATTR: APN="apn_iot", CONGESTIONRPT=ENABLE;

// 步骤4：VoIP APN保持默认（不使能），或显式去使能
SET APNREPORTATTR: APN="apn_voip", CONGESTIONRPT=DISABLE;

// 步骤5：验证所有APN配置
LST APNREPORTATTR:APN="apn_internet";
// 预期：拥塞控制 = 使能
LST APNREPORTATTR:APN="apn_iot";
// 预期：拥塞控制 = 使能
LST APNREPORTATTR:APN="apn_voip";
// 预期：拥塞控制 = 去使能（或未配置）
```

**策略联动说明：**

- PCRF侧需为apn_internet和apn_iot分别规划拥塞策略
- 拥塞时可对apn_internet的低优先级用户限速，对apn_iot的MTC业务降低优先级
- apn_voip的语音业务不受小区拥塞上报影响，始终保障带宽

## 6. 验证与调测

### 6.1 调测前置条件

执行调测前必须完成：

1. 已完成激活基于小区负荷上报的无线资源优化（UNC侧配置）
2. 已完成调测PCC基本功能
3. PGW-U上已配置基于小区负荷上报的无线资源优化功能
4. 在RAN侧已配置小区拥塞状态上报
5. PCRF上已配置下发小区负荷状态变化上报的trigger事件

### 6.2 调测所需数据与工具

| 类别 | 参数 | 取值样例 | 说明 |
|------|------|----------|------|
| 用户跟踪 | IMSI | 460000123456789 | 测试终端自带 |
| 测试APN | APN名称 | apn1 | 已通过ADD APN命令配置，可用LST APN查询 |

**工具：** 测试终端、OM Portal

### 6.3 调测操作步骤

**步骤1：验证License开关状态**

```
LST LICENSESWITCH:LICITEM="LKV3W9WOCR11";
```

- 如果SWITCH为ENABLE → 继续步骤2
- 如果SWITCH为DISABLE → 执行 `SET LICENSESWITCH:LICITEM="LKV3W9WOCR11",SWITCH=ENABLE;` 打开开关

**步骤2：创建用户跟踪任务**

在OM Portal上创建用户跟踪任务，跟踪测试终端的IMSI。

**步骤3：测试终端接入网络**

测试终端使用"apn1" APN接入网络：
- 成功接入 → 继续步骤4
- 无法接入 → 调测接入功能

**步骤4：检查PCRF的Trigger下发**

根据用户跟踪任务，检查PCRF的CCA-Initial是否下发CELL-CONGESTION-CHANGE(1003) Trigger给GGSN-C/PGW-C：
- 是 → 继续步骤5
- 否 → 联系PCRF侧工程师，确保PCRF侧配置正确后重新执行步骤3

> PCRF可以在任何CCA消息或RAR消息中下发CELL-CONGESTION-CHANGE Trigger给GGSN-C/PGW-C，不限于CCA-Initial。

**步骤5：检查PGW-C的Trigger下发**

根据用户跟踪任务，检查GGSN-C/PGW-C是否下发CELL-CONGESTION-CHANGE(1003) Trigger给PGW-U：
- 是 → 继续步骤6
- 否 → 执行步骤11（联系华为技术支持）

**步骤6：检查预定义规则一致性**

根据用户跟踪任务，检查PCRF、GGSN-C/PGW-C下发的预定义规则名称是否与PGW-U上的RULENAME一致：
- 一致 → 继续步骤7
- 不一致 → 将GGSN-C/PGW-C、PCRF和PGW-U上的预定义规则名称修改为相同后重新执行步骤7

**步骤7：触发小区拥塞**

增加接入用户数和用户访问的流量，使无线侧小区拥塞状态发生变更（变为Congestion或Overload）。

**步骤8：检查CCR-U消息**

根据用户跟踪任务，检查是否有CCR-U消息，且该消息中携带CELL-CONGESTION-CHANGE(1003) Trigger和3GPP-User-Location-Info：
- 有 → **调测成功，功能正常**
- 没有 → 继续步骤9

**步骤9：检查PGW-U上报**

根据用户跟踪任务，检查PGW-U是否向GGSN-C/PGW-C发送了CELL-CONGESTION-CHANGE(1003) Trigger和3GPP-User-Location-Info：
- 是 → 继续步骤10
- 否 → 联系PGW-U和RAN侧工程师，确保PGW-U和RAN侧配置正确后重新执行步骤7

**步骤10：检查APN拥塞控制配置**

```
LST APNREPORTATTR:APN="apn1";
```

检查指定APN下的"拥塞控制"是否为"使能"：
- 是 → 继续步骤11（联系华为技术支持）
- 否 → 执行 `SET APNREPORTATTR: APN="apn1", CONGESTIONRPT=ENABLE;` 使能后重新执行步骤3

**步骤11：联系华为技术支持**

当以上步骤无法解决问题时：
1. 重新执行上述步骤并保存报文
2. 执行EXP MML命令导出当前配置数据为MML脚本文件并保存
3. 收集并保存上述所有查询信息
4. 收集归纳所有信息并联系华为技术支持解决

### 6.4 常见问题排查

| 问题现象 | 可能原因 | 排查方法 |
|----------|----------|----------|
| PCRF未下发CELL-CONGESTION-CHANGE Trigger | PCRF侧未配置trigger事件 | 联系PCRF侧工程师确认配置 |
| PGW-C收到trigger但未传递给PGW-U | PGW-C配置异常或PGW-U未安装trigger | 检查用户跟踪中N4接口消息 |
| PGW-U未上报负荷信息 | PGW-U或RAN侧能力协商失败 | 联系PGW-U和RAN侧工程师确认配置和协商结果 |
| CCR-U消息中无3GPP-User-Location-Info | 用户位置信息传递链路异常 | 检查PGW-U到PGW-C的PFCP消息是否携带位置信息 |
| CCR-U消息中ULI AVP数量异常 | 软参DWORD519 Bit14配置 | 检查DWORD519 Bit14是否按需配置 |
| APN级拥塞控制未使能 | SET APNREPORTATTR未配置或配置为DISABLE | 执行LST APNREPORTATTR查询，必要时重新配置 |
| 预定义规则名称不一致 | PCRF/PGW-C/PGW-U三侧RULENAME不同 | 统一三侧预定义规则名称 |
| 测试终端无法接入网络 | 接入功能异常 | 调测接入功能（非本特性范围） |

- 本特性属于解决方案特性，端到端网元都需要是HUAWEI设备
- 要求无线侧网元（UTRAN和eNodeB）支持基于小区负荷上报的无线资源优化功能
- SGSN/SGW能透传扩展的GTP-U上下行报文
- 要求用户为PCC本地用户（不支持拜访用户和漫游用户）
- PCRF支持小区负荷上报功能
