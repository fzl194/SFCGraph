# WSFD-109108 基于接入点策略控制 - 特性知识文档

> **场景归属**: 业务感知 → 带宽控制子场景（辅助特性：接入点级策略控制）
> **产品**: UNC (PGW-C网元)
> **场景定位**: 按WiFi接入点维度（UE Local IP + Port）感知用户位置 → 差异化策略控制
> **辅助特性说明**: 接入点级位置感知作为差异化带宽/QoS策略的下发依据

---

## 1. 特性概述

### 1.1 基本信息

| 属性 | 值 |
|------|-----|
| **特性ID** | WSFD-109108 |
| **特性名称** | 基于接入点策略控制 |
| **产品** | UNC |
| **适用NF** | PGW-C |
| **支持版本** | UNC 20.5.0及后续版本 |
| **首发版本** | 20.5.0（首次发布） |
| **License控制项** | 82209475 LKV3WPWULI11 基于接入点策略控制 |

### 1.2 核心能力

当非可信接入（Untrusted Non-3GPP）用户的IP或Port发生变更时，UNC将用户新的IP或Port上报给PCRF/PCF/CG/Radius Server，PCRF/PCF/CG/Radius Server以此为依据判断用户位置，进行差异化处理。

**一句话描述**：通过感知WiFi用户接入点位置变化（UE Local IP + UE UDP Port），向PCRF/PCF等策略网元上报位置变化事件，触发基于位置的差异化策略控制。

### 1.3 在带宽控制场景中的定位

本特性属于带宽控制场景的**辅助特性**，其核心价值在于提供用户位置维度的策略输入：

- **位置感知维度**：通过WiFi用户的IP地址和Port信息标识用户所在的物理位置（接入点），为上层策略网元（PCRF/PCF）提供位置信息基础。
- **策略触发机制**：当用户位置变化时触发Event-Trigger上报，PCRF/PCF据此下发新的QoS和带宽策略，实现基于位置的差异化带宽控制。
- **与BWM协作**：接入点位置信息可以作为BWM（带宽管理）的输入条件之一。不同接入点（如企业区域WiFi、公共区域WiFi）的用户可以匹配不同的带宽策略模板。
- **与FUP协作**：接入点位置感知不直接参与FUP流量累计，但PCRF可以结合位置和累计流量信息下发综合策略。
- **与PCC协作**：本特性**强依赖**PCC基本功能（WSFD-109101），通过Gx接口向PCRF上报Event-Trigger。

### 1.4 APN（2/3/4G）vs DNN（5G）说明

本特性中"接入点"的含义与传统APN/DNN不同，需要明确区分：

| 概念 | 适用网络 | 含义 | 与本特性的关系 |
|------|----------|------|----------------|
| **APN** | 2/3/4G | Access Point Name，接入点名称，标识用户接入的外部数据网 | 传统APN维度策略控制不在本特性范围内 |
| **DNN** | 5G | Data Network Name，数据网络名称，APN的5G演进 | 传统DNN维度策略控制不在本特性范围内 |
| **UE Local IP + Port** | Non-3GPP（WiFi） | 用户在WiFi网络中的本地IP地址和UDP端口号 | **本特性核心**：通过此组合标识用户接入点位置 |

本特性的"接入点"特指**非可信非3GPP接入场景下，WiFi用户所在的网络位置**，通过UE Local IP和UE UDP Port的组合来标识。此位置信息与用户签约的APN/DNN不同，它反映的是用户在WiFi网络中的物理/逻辑位置。

### 1.5 与BWM、FUP、PCC的协作关系

```
                    ┌─────────────┐
                    │   PCRF/PCF   │
                    │  (策略决策)  │
                    └──────┬───────┘
                           │ 下发差异化策略
                           ▼
┌─────────────┐    Event-Trigger(43)    ┌─────────────┐
│  PGW-C/UNC  │◄────────────────────────│  位置变化    │
│  (本特性)    │  上报UE Local IP+Port  │  感知        │
└──────┬──────┘                         └─────────────┘
       │
       ├──→ BWM: PCRF根据位置下发不同带宽策略
       ├──→ FUP: PCRF结合位置和流量信息下发策略
       ├──→ PCC: 通过Gx接口上报位置，依赖PCC基本功能
       └──→ 计费: CG/Radius根据位置做差异化计费
```

## 2. 核心概念

### 2.1 非可信Non-3GPP接入（Untrusted Non-3GPP Access）

非可信Non-3GPP接入是指用户通过不受运营商信任的非3GPP网络（如公共WiFi）接入EPC核心网。在此场景下：

- **ePDG**（Evolved Packet Data Gateway）作为非可信Non-3GPP接入的安全网关，为UE提供信令和数据业务通道。
- UE通过WiFi网络连接到ePDG，ePDG再连接到PGW-C和3GPP AAA Server。
- PGW-C通过S2b接口与ePDG互通，通过S6b接口与3GPP AAA Server互通。

### 2.2 UE Local IP（用户本地IP地址）

UE Local IP是WiFi用户在WiFi网络中的本地IP地址。UNC通过ePDG发送的用户激活、更新消息中获取此信息。

- 该IP地址反映用户在WiFi网络中的网络位置。
- 当用户在WiFi网络中移动（如从一个AP漫游到另一个AP）时，UE Local IP可能发生变化。
- UE Local IP是策略网元判断用户位置的依据之一。

### 2.3 UE UDP Port（用户UDP端口）

UE UDP Port是WiFi用户在建立到ePDG的IPSec隧道时使用的UDP端口号。

- 与UE Local IP共同标识用户在WiFi网络中的精确位置。
- 当NAT设备为用户分配新的IP/Port映射时，UE UDP Port会发生变化。

### 2.4 接入点级策略（基于位置的策略控制）

接入点级策略控制的核心逻辑：

1. **位置标识**：UNC通过UE Local IP + UE UDP Port的组合唯一标识WiFi用户所在的接入点位置。
2. **变化检测**：当用户的IP或Port发生变化时，UNC检测到位置变化。
3. **事件上报**：UNC向PCRF/PCF上报位置变化事件（Event-Trigger = UE_LOCAL_IP_ADDRESS_CHANGE(43)）。
4. **策略决策**：PCRF/PCF根据新的位置信息，决定是否下发新的QoS、带宽、计费等策略。
5. **策略执行**：PGW-C/PGW-U执行PCRF/PCF下发的新策略。

### 2.5 Event-Trigger信元

Event-Trigger是Gx接口（PGW-C与PCRF之间）中的信元，用于PGW-C向PCRF上报触发事件。

- **UE_LOCAL_IP_ADDRESS_CHANGE(43)**：本特性的核心事件，取值为43，表示用户的本地IP地址发生变化。
- 当UNC检测到WiFi用户的IP或Port变化时，向PCRF上报此Event-Trigger，并携带最新的IP和Port信息。
- PCRF收到此事件后，可据此下发策略更新消息（如RAR，Re-Auth-Request），触发策略变更。

### 2.6 与会话级策略的关系

| 维度 | 接入点级策略（本特性） | 会话级策略（PCC基本功能） |
|------|------------------------|---------------------------|
| **触发条件** | UE Local IP/Port变化 | 会话建立、修改、删除 |
| **策略粒度** | 按接入点位置维度 | 按会话维度 |
| **策略来源** | PCRF基于位置决策 | PCRF基于签约/规则决策 |
| **依赖关系** | 依赖PCC基本功能 | 独立运作 |
| **上报接口** | Gx（Event-Trigger 43） | Gx（标准PCC流程） |

接入点级策略是在会话级策略之上的增强：会话建立时按标准PCC流程下发策略，会话存续期间当用户位置变化时，通过本特性触发策略更新。

## 3. 实现原理与流程

### 3.1 组网架构

基于接入点策略控制特性的组网架构涉及以下网元：

| 网元 | 角色 | 支持版本要求 |
|------|------|-------------|
| **UE** | WiFi终端，支持WiFi网络接入功能 | 无特殊要求 |
| **ePDG** | 连接3GPP AAA Server、PGW-C和PGW-U，为UE提供信令和数据业务通道 | 无特殊要求 |
| **PGW-C** | 通过S2b接口与ePDG互通，通过S6b接口与3GPP AAA Server互通，为本特性的执行主体 | UNC 20.5.0及后续版本 |
| **PGW-U** | 根据PGW-C下发的策略处理数据报文 | 无特殊要求 |
| **PCRF/PCF** | 根据签约信息或运营商策略，下发位置更新订阅消息，基于上报的位置信息更新策略 | 无特殊要求 |
| **CG** | 对PGW-C发送的话单进行存储、合并、标准化预处理，可基于WiFi用户位置做差异化计费 | 无特殊要求 |
| **Radius Server** | 接收鉴权/计费抄送消息，可基于WiFi用户位置做差异化处理 | 无特殊要求 |

组网连接关系：

```
                        └──────┬───────┘
                               │ S6b
                        ┌──────┴───────┐
          WiFi网络      │    ePDG       │

     │   UE     │──────────────┤ S2b

              │   CG     │ │ BS   │ │Radius Server │
              └──────────┘ └──────┘ └──────────────┘
```

### 3.2 位置信息获取流程

UNC获取WiFi用户位置信息的流程：

1. **UE通过WiFi接入ePDG**：UE在WiFi网络中获取Local IP地址和UDP Port，建立到ePDG的IPSec隧道。
2. **ePDG转发激活消息**：ePDG在向PGW-C发送用户激活消息时，携带UE Local IP和UE UDP Port信息。
3. **PGW-C提取位置信息**：PGW-C从ePDG的消息中解析出UE Local IP和UE UDP Port，作为用户位置标识。
4. **位置信息存储**：PGW-C在会话上下文中保存当前的位置信息，用于后续变化检测。

### 3.3 位置变化检测与上报流程

当WiFi用户的IP或Port发生变化时的处理流程：

```
UE位置变化(IP/Port变更)
        │
        ▼
┌───────────────────┐
│ ePDG发送更新消息   │
│ 给PGW-C           │
│ (携带新IP+Port)   │
└────────┬──────────┘
         │
         ▼
┌───────────────────┐
│ PGW-C检测到       │
│ IP或Port变化      │
└────────┬──────────┘
         │
         ▼
┌───────────────────────────────────────────┐
│ PGW-C向PCRF上报Event-Trigger               │
│ 取值: UE_LOCAL_IP_ADDRESS_CHANGE(43)       │
│ 携带: 最新的用户IP和Port信息               │
│ 接口: Gx                                  │
└────────┬──────────────────────────────────┘
         │
         ▼
┌───────────────────┐
│ PCRF根据新位置     │
│ 下发差异化策略     │
│ (QoS/带宽/计费)   │
└────────┬──────────┘
         │
         ▼
┌───────────────────┐
│ PGW-C执行新策略    │
│ PGW-U执行报文处理  │
└───────────────────┘
```

### 3.4 完整端到端流程

从用户接入到位置变化策略更新的完整端到端流程：

**阶段一：用户初始接入**

1. UE通过WiFi网络获取Local IP和UDP Port。
2. UE建立到ePDG的IPSec隧道。
3. ePDG通过S2b接口向PGW-C发送会话建立请求，携带UE Local IP和UE UDP Port。
4. PGW-C通过Gx接口向PCRF发起CCR-Initial，携带用户签约信息和初始位置信息。
5. PCRF下发初始策略（QoS、带宽、计费规则等）。
6. PGW-C建立会话，PGW-U安装转发规则。
7. 用户数据通路建立完成。

**阶段二：位置变化触发策略更新**

1. UE在WiFi网络中移动或NAT重映射，导致Local IP或UDP Port变更。
2. ePDG检测到变化，通过S2b接口向PGW-C发送更新消息，携带新的UE Local IP和UE UDP Port。
3. PGW-C检测到IP或Port变化，通过Gx接口向PCRF上报Event-Trigger，取值为UE_LOCAL_IP_ADDRESS_CHANGE(43)，携带新的IP和Port。
4. PCRF根据新位置信息进行策略决策：
   - 如果新位置对应不同的策略区域，下发新的QoS和带宽策略。
   - 如果新位置不影响策略，不下发策略更新。
5. PGW-C执行PCRF下发的新策略，更新PGW-U的转发规则。

**阶段三：位置信息同步到计费和Radius**

1. **离线话单**：UNC在WiFi用户产生离线话单时，话单扩展字段中携带用户当前的IP和Port信息。当WiFi用户的IP或Port变化时，不会触发生成新容器或话单，不涉及计费和话单结构变更。
2. **Radius消息**：
   - WiFi用户激活时：UNC向Radius Server发送Radius Accounting Start消息，携带用户IP和Port信息。
   - WiFi用户去活时：UNC向Radius Server发送Radius Accounting Stop消息。
   - WiFi用户IP或Port变化时：UNC通过Radius Accounting Update消息通知Radius Server。

### 3.5 带宽控制视角的流程分析

从带宽控制场景来看，本特性的核心价值在于提供位置维度的策略触发：

| 流程节点 | 带宽控制关联 | 说明 |
|----------|-------------|------|
| 位置变化检测 | 触发条件 | 用户从一个WiFi区域移动到另一个区域 |
| Event-Trigger上报 | 策略请求 | 请求PCRF基于新位置重新评估带宽策略 |
| PCRF策略决策 | 带宽分配 | 不同位置对应不同带宽模板（如企业区域保障、公共区域限速） |
| PGW-C执行 | 策略生效 | 新的QoS/限速规则在PGW-U生效 |

### 4.5 策略绑定配置说明

本特性的策略配置不在PGW-C侧通过MML命令完成，而是通过以下机制实现：

1. **PCRF侧配置**：PCRF配置位置变化订阅（Event-Trigger UE_LOCAL_IP_ADDRESS_CHANGE(43)），以及不同位置对应的策略规则。
2. **PGW-C侧执行**：PGW-C被动响应位置变化，上报事件，执行PCRF下发的策略。
3. **CG侧配置**：CG需要支持解析处理话单中的WiFi用户IP和Port扩展字段。
4. **Radius Server侧配置**：Radius Server需要支持接收和处理携带位置信息的Radius消息。

### 4.6 应用限制

- 如果开启PGW-CDR中记录WiFi用户的位置信息功能，CG需要支持解析处理记录的WiFi用户IP和Port字段。
- 本特性仅适用于非可信Non-3GPP（WiFi）接入场景，不适用于可信Non-3GPP接入或3GPP接入。

| 类别 | 参数名称 | 取值样例 | 说明 |

- 已完成PCC基本功能（WSFD-109101）配置。

### 5.2 案例：多区域差异化带宽控制（应用场景推演）

**场景描述**：
运营商部署多个WiFi热点区域（企业园区、商场、公共交通），通过本特性实现基于WiFi接入位置的差异化带宽控制：

| WiFi区域 | 用户位置特征 | PCRF策略 | 带宽效果 |
|----------|-------------|----------|----------|
| 企业园区AP | UE Local IP属于企业园区IP段 | 高优先级QoS、保障带宽 | 上下行保障50Mbps |
| 公共商场AP | UE Local IP属于商场IP段 | 中等QoS、共享带宽 | 上下行限速10Mbps |
| 公共交通AP | UE Local IP属于交通IP段 | 低优先级QoS、限速 | 上下行限速2Mbps |

**策略执行流程**：

1. **初始接入**：用户在企业园区接入WiFi，PGW-C向PCRF上报初始位置（IP1+Port1）。
2. **PCRF决策**：PCRF识别IP1属于企业园区IP段，下发高保障带宽策略。
3. **位置变化**：用户移动到商场WiFi，IP变为IP2+Port2。
4. **事件上报**：PGW-C检测到IP/Port变化，上报Event-Trigger(43)。
5. **策略切换**：PCRF识别IP2属于商场IP段，下发中等限速策略。
6. **策略生效**：PGW-C/PGW-U执行新的带宽策略，用户带宽从保障50Mbps切换为限速10Mbps。

**注意**：此案例中的具体带宽策略配置在PCRF侧完成，PGW-C仅负责位置信息上报和策略执行。本特性提供的是位置感知能力，实际带宽策略由PCRF决策。

## 6. 验证与调测

### 6.1 调测目标

验证当WiFi用户的IP或Port发生变更时，PGW-C能够正确向PCRF上报位置变化事件（Event-Trigger = UE_LOCAL_IP_ADDRESS_CHANGE(43)），携带正确的IP和Port信息。

### 6.2 调测前置检查清单

| 检查项 | 验证方法 | 预期结果 |
|--------|----------|----------|
| License开关 | 执行 `LST LICENSESWITCH` | 本特性对应SWITCH为ENABLE |
| PCC基本功能 | 参考WSFD-109101调测 | PCC基本功能正常运作 |
| APN配置 | 执行 `LST APN` | 测试APN（如huawei.com）已配置 |
| 用户签约 | 确认测试IMSI已签约 | 用户可正常接入 |
| 跟踪任务 | 创建跟踪任务（N4/Gx/N7） | 跟踪任务创建成功 |

### 6.3 调测步骤（标准流程）

按照源文档调测指南的标准流程执行：

1. **进入MML命令行**：进入"MML命令行-UNC"窗口。

2. **检查License开关**：执行 `LST LICENSESWITCH`，确认本特性License开关为ENABLE。如果为DISABLE，执行 `SET LICENSESWITCH` 打开。

3. **创建跟踪任务**：创建用户跟踪任务，消息类型选择**N4**、**Gx**和**N7**。

4. **用户接入**：测试终端使用APN"huawei.com"发起接入网络请求。

5. **验证激活**：执行 `DSP PDUSESSION`，通过测试终端IMSI查看是否激活成功。
   - 激活成功且"用户IP地址"与规划值一致：记录用户IP（IP1），继续下一步。
   - 激活失败：调测基本会话接入功能。

6. **触发位置变化**：用户关机，然后再开机。

7. **验证Event-Trigger**：根据Gx/N7接口跟踪报文，检查PGW-C是否向PCRF上报Event-Trigger信元，取值为**UE_LOCAL_IP_ADDRESS_CHANGE(43)**，携带用户IP和Port信息。
   - 是：调测结束，特性工作正常。
   - 否：执行故障排查。

### 6.5 系统影响评估

在调测和运行过程中，需要关注以下系统影响：

| 影响项 | 说明 | 监控建议 |
|--------|------|----------|
| CPU占用率 | 当PCRF或Radius Server订阅了WiFi用户位置变化的trigger时，CPU占用率会随着WiFi用户位置变化的次数而升高 | 监控CPU利用率，关注大规模WiFi漫游场景 |
| 话单扩展 | PGW-CDR中通过扩展字段记录WiFi用户的位置信息（用户IP和PORT） | 确认CG支持解析处理扩展字段 |
| 计费一致性 | WiFi用户的IP或Port变化时，不会触发生成新容器或话单，不涉及计费和话单变更 | 无特殊计费风险 |

### 6.6 常见问题排查表

| 问题现象 | 可能原因 | 排查方法 | 解决方案 |
|----------|----------|----------|----------|
| Event-Trigger未上报 | License未打开 | 执行 `LST LICENSESWITCH` | 执行 `SET LICENSESWITCH` 打开 |
| Event-Trigger未上报 | PCC基本功能未配置或异常 | 检查WSFD-109101 PCC基本功能 | 重新调测PCC基本功能 |
| Event-Trigger未上报 | 用户接入类型非Non-3GPP | 检查用户是否通过WiFi/ePDG接入 | 确认用户通过非可信Non-3GPP接入 |
| Event-Trigger未上报 | IP/Port实际未变化 | 对比跟踪报文中的IP/Port值 | 确认用户确实发生了位置变化 |
| CG解析话单失败 | CG不支持扩展字段 | 检查CG版本和配置 | 升级CG或配置支持WiFi用户IP和Port字段解析 |
| CPU占用率异常升高 | 大规模WiFi用户漫游 | 监控CPU利用率 | 评估PCRF订阅trigger的粒度，优化策略 |
