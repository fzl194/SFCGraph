# 业务感知域配置规则（3 子场景共享）

> 业务感知域（计费 / 带宽控制 / 访问限制）3 子场景共享的配置规则。
> ★**仅适用于业务感知域**，其他业务域（如 APN）不适用。
> 各子场景独有规则在各自 `ref-phase/` 和 `three-layer-graph/` 的 Rule 对象（BR/FR/TR/CR）。
> Phase 5 配置生成时作内置约束（业务感知域内）。

---

## 1. 执行时序

### 1.1 REFRESHSRV 必须最后执行（UDG/UPF 侧）
- 所有策略变更完成后必须执行 `SET REFRESHSRV`
- 执行后约 **60 秒**（PROTBINDFLOWF 定时器）策略才完全下发到转发面
- 违反影响：配置命令成功但策略未生效，验证失败
- **SMF/UNC 侧无 REFRESHSRV**（SMF 侧策略即时生效）

### 1.2 配置链依赖顺序（业务感知对象体系）
```
底层过滤(FILTER/L7FILTER)
  → 流过滤器(FLOWFILTER → FLTBINDFLOWF → PROTBINDFLOWF)
  → 策略动作（场景相关）：
       计费 URR/PCCACTIONPROP · 带宽 BWMCONTROLLER · 访问限制 HEADEN/SMARTHTTPREDIR/CFTEMPLATE
  → 策略组合（场景相关）：
       URRGROUP/PCCPOLICYGRP · BWMUSERGROUP
  → 规则汇聚(RULE)
  → 容器(USERPROFILE，仅新建时)
  → 绑定(RULEBINDING / URRGRPBINDING / UPBINDUPG / APNUSRPROFG)
  → 刷新(SET REFRESHSRV，最后，仅 UPF 侧)
```

---

## 2. 跨网元一致性（业务感知特有）

涉及 PCRF/PCF + UNC/SMF + UDG/UPF 多网元时：

- **三网元预定义规则一致**：RULENAME / FLOWFILTERNAME / ADCPARA 在 PCF、SMF、UPF 三处必须完全一致（定向业务/ADC 必须用预定义规则，因 PCF 无 L7 识别能力）
- **双产品共用参数一致**（UPF ↔ SMF）：URRID、RG、USAGERPTMODE、RULENAME、USERPROFILENAME
- **POLICYNAME 跨策略类型不冲突**：同一产品内 PCC/QOS/BWM/CHARGING 等不同 POLICYTYPE 的 RULENAME 不能相同

---

## 3. 优先级规则（RULE PRIORITY）

- **PRIORITY 数字越小，优先级越高**（固定规则，不可更改）
- "第N优先" = 第N高优先级 = PRIORITY 数值第N小
- 第N优先的数值必须 **大于第(N-1)优先、小于第(N+1)优先**
- 分析前必须用 Grep 提取现网**所有 RULE 的 PRIORITY**（含非本场景 RULE），按数值排序后定位

---

## 4. 策略链独立原则

- 每条业务的策略链（计费 URR 三件套 / 带宽 BWM 控制链 / 访问限制动作链）**彼此独立，不可跨业务复用**
- 默认/兜底规则单独建（低 PRIORITY，或 DFTURRGRPNAME 等机制）

---

## 5. SA 识别前置

- 业务感知所有策略匹配依赖 SA 识别（SA-Basic + 各协议 SA 子特性）
- L7 动作（ADC / URL 过滤 / 头增强 / 重定向）依赖对应协议 SA 特性开启
- 动态规则不支持七层特征 → 七层匹配（URL/ADC）必须用预定义/本地规则

---

## 6. 各子场景独有的补充规则

- **计费**：见 `charging/ref-phase/` 与 `charging/kb/`（计费三件套、融合计费三联前置、配额耗尽动作）
- **带宽控制**：见 `bandwidth/ref-phase/` 与 `bandwidth/kb/`（CAR/Shaping 不可同对象叠加、FUP 降速优先级覆盖、GBR 资源预留）
- **访问限制**：见 `access-control/ref-phase/` 与 `access-control/kb/`（PCC 动作双轨道：POLICYTYPE 隐式五子轨 + URL 过滤 CFTEMPLATE.ACTION 显式；PERMIT 唯一性；重定向族协议层差异）
