# GWFD-020301 内容计费基本功能 知识文档

## 概述

### 定义

内容计费（Content-based Charging），即基于业务的计费，是一种通过包过滤和分析技术识别出用户上下行数据的业务类型，并针对不同的业务实施不同费率的计费方法。

内容计费可通过以下三种计费系统实现：
- 在线计费系统（OCS）
- 离线计费系统（CG/BS）
- 融合计费系统（OCS + CG/BS）

### 适用NF

SGW-U、PGW-U、UPF

### 涉及网元与角色

| 涉及NF | 支持版本 | 功能说明 |
|--------|----------|----------|
| SGW-C/PGW-C/SMF（UNC） | UNC 20.0.0及后续版本 | 将UDG的业务统计结果上报给计费系统 |
| SGW-U/PGW-U/UPF（UDG） | UDG 20.0.0及后续版本 | 对用户业务进行识别和解析；统计不同业务的流量、时长计费信息，并上报UNC |
| CG | 无特殊要求 | 对SGW-C/PGW-C发送来的话单进行存储、合并、标准化预处理 |
| OCS | 无特殊要求 | 进行用户业务配额管理 |

## 核心概念

### 功能逻辑

内容计费的核心逻辑：UDG根据SA功能对业务流进行解析，并进行计费规则匹配，匹配成功的业务会标记对应URR ID，最终由CG、OCS或CHF进行计费。

简单来说：对不同业务类型使用不同计费策略，达到差异化计费。

### 核心配置对象（计费三件套）

内容计费的配置围绕三个核心对象展开：

| 配置对象 | MML命令 | 作用 | 配置层级 |
|----------|---------|------|----------|
| URR（使用量上报规则） | ADD URR | 定义一条费率规则，包含费率标识、计费模式、统计类型 | 基础费率定义 |
| URRGROUP（使用量上报规则组） | ADD URRGROUP | 将在线URR和离线URR组合在一起，支持上下行分别配置 | 费率组合 |
| PCCPOLICYGRP（PCC策略组） | ADD PCCPOLICYGRP | 将URRGROUP绑定为PCC策略组，供RULE引用 | 策略绑定 |

三件套关系：`URR -> URRGROUP -> PCCPOLICYGRP`，每条业务独立配置一组三件套。

### 匹配规则配置对象

| 配置对象 | MML命令 | 作用 |
|----------|---------|------|
| FILTER（过滤器） | ADD FILTER / ADD FILTERIPV6 | 定义三四层过滤条件（IP、端口、协议） |
| L7FILTER（七层过滤器） | ADD L7FILTER | 定义七层过滤条件（URL、Host等） |
| FLOWFILTER（流过滤器） | ADD FLOWFILTER | 聚合容器，将三四层和七层过滤条件绑定在一起 |
| FLTBINDFLOWF（过滤器绑定） | ADD FLTBINDFLOWF | 将三四层FILTER绑定到FLOWFILTER |
| PROTBINDFLOWF（协议绑定） | ADD PROTBINDFLOWF | 将七层协议和L7FILTER绑定到FLOWFILTER |
| RULE（规则） | ADD RULE | 将FLOWFILTER和PCCPOLICYGRP绑定，定义匹配优先级 |
| USERPROFILE（用户模板） | ADD USERPROFILE | 规则组容器，承载多条RULE |
| RULEBINDING（规则绑定） | ADD RULEBINDING | 将RULE绑定到USERPROFILE |

### 依赖特性

| 交互类型 | 相关特性 | 控制项名称 | 交互说明 |
|----------|----------|------------|----------|

## 原理与流程

### 内容解析（SA功能）

内容解析是实现内容计费的基础。UDG借助SA功能对通过的业务报文进行特征分析：

- **三四层解析**：获得五元组信息（源IP、目的IP、源端口、目的端口、协议类型）
- **七层解析**：获取URL等应用层信息

详细SA功能参见GWFD-110101 SA-Basic。

### 计费规则匹配

计费规则由UNC通过N4接口的Create PDR（Packet Detection Rule）信元组下发。

#### Create PDR关键子信元

| 子信元 | 信元解释 |
|--------|----------|
| PDR ID | 唯一标识一个Create PDR |
| Precedence | 标识PDR优先级 |
| PDI（Packet Detection Information） | 匹配用户报文的动态规则，包含Source Interface、Local F-TEID、Network Instance、UE IP address等 |
| Activate Predefined Rules | 标识下发的预定义规则或规则组，用于激活本地配置的User Profile或Rule |

#### 匹配流程

UDG会匹配所有PDR，包括：
- **动态PDR**：PDR中未携带Activate Predefined Rules子信元
- **预定义PDR**：PDR中携带了Activate Predefined Rules子信元

匹配成功的规则中，**优先级最高的规则**被UDG使用。

#### 计费ID确定

- 若匹配成功的是**动态PDR**：使用此动态PDR里的URR ID计费，此URR ID关联到UNC中配置的RG+SID。
- 若匹配成功的是**预定义PDR**：使用此预定义PDR里的URR ID + 匹配中的预定义Rule里的URR ID计费，此URR ID关联到UNC中配置的RG+SID。

### User Profile下Rule规则匹配流程

1. 一个User Profile下包含一组Rule。
2. 每个Rule有不同的优先级，匹配时优先使用**高优先级**的Rule。
3. 高优先级匹配失败时，再使用低优先级的Rule。
4. 每个Rule可以配置多种过滤条件，只有当**所有过滤条件均匹配成功**时，这个Rule才算匹配成功。

#### 过滤条件组合

Rule支持的过滤条件组合：

| 组合方式 | 说明 |
|----------|------|
| Filter | 仅三四层过滤 |
| Filter + Protocol | 三四层 + 协议 |
| Filter + Protocol + L7Filter | 三四层 + 协议 + 七层（最完整） |
| Protocol | 仅协议 |
| Protocol + L7Filter | 协议 + 七层 |

### 可选功能

#### 重定向计费

UDG上的重定向应用场景包括：
- IP重定向（三四层重定向）
- 用户Portal重定向（七层URL重定向）
- URL重定向（七层URL重定向）
- 在线计费FUI重定向（七层URL重定向）

**三四层重定向**：通过设置MS/UE和目的服务器间的数据流费率RG进行计费。

**七层重定向**：UDG构造返回给MS/UE的HTTP 302重定向报文，以及TCP RST拆链报文。

重定向报文的计费方式：
- HTTP 302报文和TCP RST报文：根据SET URRGRPBINDING命令的"重定向URR组名称"参数设置的URR ID进行计费。如果没有配置URR，则使用对应的业务流的URR。
- 重定向到Portal页面的报文：根据ADD RULE设置的到Portal页面数据流的URR ID进行计费。
- **在线计费用户余额耗尽FUI重定向**：重定向报文免费通过。

重要约束：如果FUI重定向同时配置了在线重定向费率和离线重定向费率，存在在线重定向费率配额不足导致只有离线重定向费率计费的情况，造成在线和离线不一致。**推荐重定向费率只配置离线费率。**

#### 关联URL计费

通过ADD L7FILTER命令的"Referer关联计费开关"参数，控制从源页面链接出去的浏览业务按照源页面的计费费率进行内容计费。将源页面访问流量和链接到其他网站的页面访问流量统计到相同的内容计费实例中。

#### FTP内容计费

如果需要对FTP流量进行单独计费，需要在ADD PROTBINDFLOWF命令中配置"协议名称"参数为FTP。**不能只是根据知名端口进行过滤**，因为FTP协议的信令面与数据面分开，数据面使用的端口是随机的。

## 配置规则

### PCF/SMF/UPF配置一致性要求

内容计费配置需要在PCF、SMF、UPF网元上统一进行，保证网元间配置的一致性：

| 配置项 | 一致性要求 |
|--------|------------|
| URRID | SMF和UPF上的配置需要保持一致 |
| USAGERPTMODE | SMF和UPF上的配置需要保持一致 |
| OFFMETERINGTYPE | SMF和UPF上的配置需要保持一致 |
| ONLMETERINGTYPE | SMF和UPF上的配置需要保持一致 |
| RULENAME | SMF和UPF上的配置需要保持一致 |
| RULE下的POLICYTYPE | SMF和UPF上的配置需要保持一致 |
| RULE下的POLICYNAME | SMF和UPF上的配置需要保持一致 |
| USERPROFILENAME | PCF、SMF、UPF间名称一致，不一致时PCF下发的策略无法生效 |
| RULEBINDING中的RULENAME | SMF和UPF上需要保持一致，不一致策略无法按预期生效 |
| RULEBINDING中的POLICYTYPE | SMF和UPF上需要保持一致，不一致策略无法按预期生效 |

### 配置流程（11步）

| 步骤 | 命令 | 说明 |
|------|------|------|
| 1 | ADD URR | 配置使用量上报规则（费率定义） |
| 2 | ADD URRGROUP | 配置使用量上报规则组（费率组合） |
| 3 | ADD PCCPOLICYGRP | 配置PCC策略组，绑定URR组 |
| 4 | ADD FILTER / ADD FILTERIPV6 | 配置三四层过滤条件 |
| 5 | ADD L7FILTER | 配置七层过滤条件 |
| 6 | ADD FLOWFILTER | 配置流过滤器（聚合容器） |
| 7 | ADD FLTBINDFLOWF | 将三四层过滤条件绑定到流过滤器 |
| 8 | ADD PROTBINDFLOWF | 将七层过滤条件和协议绑定到流过滤器 |
| 9 | ADD RULE | 添加规则，绑定流过滤器和PCC策略组 |
| 10 | ADD USERPROFILE | 添加用户模板 |
| 11 | ADD RULEBINDING | 将规则绑定到用户模板 |
| 12 | SET URRGRPBINDING | 配置缺省费率 |
| 13 | ADD SPECURRGRPLIST | 配置防欺诈URR列表（可选） |
| 14 | SET REFRESHSRV | 将新配置的Filter置为生效 |

### URR配置规则

#### 关键参数

| 参数 | 说明 | 约束 |
|------|------|------|
| URRNAME | 使用量上报规则名称 | 全网规划 |
| URRID | URR标识 | SMF和UPF上必须一致 |
| USAGERPTMODE | 使用量上报模式 | OFFLINE（离线）/ ONLINE（在线） |
| OFFMETERINGTYPE | 离线计费统计类型 | VOLUME（按流量）/ DURATION（按时长）/ FREE（免费）等 |
| ONLMETERINGTYPE | 在线计费统计类型 | VOLUME（按流量）/ DURATION（按时长）/ FREE（免费）等 |

#### 计费模式与统计类型

每个业务通常需要配置两条URR：
- 一条USAGERPTMODE=OFFLINE，配OFFMETERINGTYPE
- 一条USAGERPTMODE=ONLINE，配ONLMETERINGTYPE

### URRGROUP配置规则

#### 关键参数

| 参数 | 说明 | 约束 |
|------|------|------|
| URRGROUPNAME | 使用量上报规则组名称 | 本端规划 |
| UPURRNAME1 | 上行发起URR名称1（通常是离线URR） | 已通过ADD URR配置 |
| DOWNURRNAME1 | 下行发起URR名称1（通常是离线URR） | 已通过ADD URR配置 |
| UPURRNAME2 | 上行发起URR名称2（通常是在线URR） | 已通过ADD URR配置 |
| DOWNURRNAME2 | 下行发起URR名称2（通常是在线URR） | 已通过ADD URR配置 |

#### 核心约束

1. **URRGROUP下必须同时配置在线URR和离线URR**。只配置一种计费模式且计费模式和用户的计费模式不匹配时无法计费。

2. **融合计费特殊场景**：当SMF使用融合计费时，SMF上的SET USRPROFCHARGE、SET APNCHARGECTRL、ADD CHARGEMETHOD命令的"RGAPPLIED"配置为"DEFAULT"时，**URRGROUP下不能同时配置离线和在线URR**，否则会导致计费冲突。

3. **UPURRNAME1/2和DOWNURRNAME1/2只是编号**，没有优先级语义。1和2用于区分不同计费模式的URR（如1=离线，2=在线）。

### PCCPOLICYGRP配置规则

| 参数 | 说明 | 约束 |
|------|------|------|
| PCCPOLICYGRPNM | PCC策略组名称 | 本端规划 |
| URRGROUPNAME | URR组名称 | 已通过ADD URRGROUP配置 |
| SIGURRGRPNAME | 信令URR组名称（可选） | 已通过ADD URRGROUP配置，用于异常场景 |

### FILTER配置规则

#### IPv4过滤器（ADD FILTER）

| 参数 | 说明 | 取值样例 |
|------|------|----------|
| FILTERNAME | 过滤器名字 | L34_dns |
| L34PROTTYPE | 三四层IPv4协议输入类型 | STRING |
| L34PROTOCOL | 三四层协议类型 | ANY / TCP / UDP等 |
| MSSTARTPORT | 手机起始端口号 | 0 |
| MSENDPORT | 手机结束端口号 | 65535 |
| SVRSTARTPORT | 服务器起始端口号 | 53 |

约束：MSSTARTPORT <= MSENDPORT，SVRSTARTPORT <= SVRENDPORT。

#### IPv6过滤器（ADD FILTERIPV6）

| 参数 | 说明 | 取值样例 |
|------|------|----------|
| FILTERNAME | 过滤器名字 | ipv6_ims |
| L34PROTTYPEV6 | 三四层IPv6协议输入类型 | STRING |
| L34PROTOCOL | 三四层协议类型 | ANY |
| SVRIPMODE | 服务器IP地址配置模式 | IP |
| MSSTARTPORT | 手机起始端口号 | 0 |
| MSENDPORT | 手机结束端口号 | 65535 |
| SVRSTARTPORT | 服务器起始端口号 | 0 |
| SVRENDPORT | 服务器结束端口号 | 65535 |
| SVRIPV6 | 服务器IPv6地址 | fc00:8028:5002:10D::181 |
| SVRIPV6MASKTYPE | 服务器IPv6地址掩码类型 | LENGTHTYPE |
| SVRIPV6MASKLEN | 服务器IPv6地址掩码长度 | 128 |

### L7FILTER配置规则

| 参数 | 说明 | 约束 |
|------|------|------|
| L7FILTERNAME | 七层过滤器名称 | 本端规划 |
| SUBL7FLTNAME | 子七层过滤器名称 | 如dns、http等 |
| URLTYPE | URL类型 | URL |
| URL | URL地址 | 如www.huawei.com |

重要约束：
- 防欺诈功能开启时，RULE下绑定URL类型的7层filter时**必须同时绑定三四层filter**。
- 防欺诈未开启时，RULE下可仅绑定URL类型的7层filter。

### FLOWFILTER配置规则

| 参数 | 说明 | 约束 |
|------|------|------|
| FLOWFILTERNAME | 流过滤器名称 | 本端规划 |

**核心约束**：FLOWFILTER必须绑定过滤条件（通过FLTBINDFLOWF或PROTBINDFLOWF），否则该FLOWFILTER匹配失败，对应的Rule为无效配置。

### FLTBINDFLOWF配置规则

| 参数 | 说明 |
|------|------|
| FLOWFILTERNAME | 流过滤器名称 |
| FILTERNAME | 三四层过滤器名称 |

作用：将三四层过滤条件绑定到流过滤器。

### PROTBINDFLOWF配置规则

| 参数 | 说明 |
|------|------|
| FLOWFILTERNAME | 流过滤器名称 |
| PROTOCOLNAME | 协议名称（如dns、FTP） |
| L7FILTERNAME | 七层过滤器名称 |

作用：将七层过滤条件和协议绑定到流过滤器。

注意：FTP内容计费时，PROTOCOLNAME必须配置为FTP，不能依赖端口匹配。

### RULE配置规则

| 参数 | 说明 | 约束 |
|------|------|------|
| RULENAME | 规则名称 | SMF和UPF上必须一致 |
| POLICYTYPE | 策略类型 | PCC |
| FILTERINGMODE | 流过滤器或流过滤器组选择 | FLOWFILTER |
| FLOWFILTERNAME | 流过滤器名称 | 已通过ADD FLOWFILTER配置 |
| POLICYNAME | 策略名称 | 已通过ADD PCCPOLICYGRP配置 |
| PRIORITY | 优先级 | 数字越小优先级越高 |

#### 优先级配置规则

1. **数字越小优先级越高**。
2. **只绑定三四层filter的RULE的优先级应该低于绑定7层filter的RULE的优先级**。
3. 过滤条件为any的RULE，优先级设置为最低（如65000）。

### USERPROFILE配置规则

| 参数 | 说明 | 约束 |
|------|------|------|
| USERPROFILENAME | 用户模板名称 | PCF、SMF、UPF间名称一致 |

不一致时PCF下发的策略无法生效。

### RULEBINDING配置规则

| 参数 | 说明 | 约束 |
|------|------|------|
| USERPROFILENAME | 用户模板名称 | 已通过ADD USERPROFILE配置 |
| RULENAME | 规则名称 | 已通过ADD RULE配置 |
| POLICYTYPE | 策略类型 | 与RULE中的POLICYTYPE一致 |

SMF、UPF上USERPROFILE下绑定的RULENAME、POLICYTYPE需要保持一致，不一致策略无法按照预期生效。

### SET URRGRPBINDING（缺省费率）

| 参数 | 说明 | 约束 |
|------|------|------|
| USERPROFILENAME | 用户模板名称 | 已配置 |
| DFTURRGRPNAME | 缺省URR组名称 | 已通过ADD URRGROUP配置 |
| DFTSIGURRGNAME | 缺省信令URR组名称 | 已通过ADD URRGROUP配置，用于异常场景 |

核心约束：
- 每个USERPROFILE都必须绑定缺省费率URR组。
- USERPROFILE需要将默认业务和默认信令的计费属性都配置完整，不配置时按照RULE没有匹配到费率的报文无法计费。

### SET REFRESHSRV（刷新过滤器）

| 参数 | 说明 |
|------|------|
| REFRESHTYPE | 刷新类型，ALL表示刷新所有 |

**硬约束**：不执行SET REFRESHSRV命令，FILTER的配置变更不会生效。必须是配置流程的最后一步。

### ADD SPECURRGRPLIST（防欺诈URR列表）

| 参数 | 说明 |
|------|------|
| URRGROUPNAME | URR组名称 |

用途：对有欺诈风险的费率，将费率加入防欺诈费率列表。

## 配置案例

### 典型场景说明

根据以下4个场景，配置User Profile和对应规则：

| 场景 | 业务描述 | 匹配方式 |
|------|----------|----------|
| 场景1 | 访问www.huawei.com网站 | L34 + L7（URL匹配） |
| 场景2 | 访问IMS业务 | L34 IPv6（服务器IP匹配） |
| 场景3 | 除上述两种情况外的基础业务（any） | L34 any to any |
| 场景4 | 未匹配中过滤条件的异常场景 | 信令URR组（免费） |

### 完整MML配置脚本

#### 步骤1：配置URR

```
// 场景1：www.huawei.com网站的URR
ADD URR:URRNAME="URR_URL_offline", URRID=12500, USAGERPTMODE=OFFLINE, OFFMETERINGTYPE=VOLUME;
```

## 验证与调测

### 对系统的影响

- 三四层（TCP/IP层）解析处理对系统吞吐量影响较小。
- 七层（应用层）解析处理对系统吞吐量影响较大。
- 需要进行三四层或七层解析的报文越多，系统吞吐量下降越明显。

### 应用限制

1. UDG支持协议特征库定期在线升级，设备的协议解析能力会不断变化。
2. 内容计费特性**不包含对加密协议报文的解析功能**，例如SSL、FTPS报文。
3. 当网络中部署iCache并采用旁路方案时，数据流会重定向到iCache，导致UDG无法获取原业务数据包的URL和目的IP地址，内容计费不准确。
