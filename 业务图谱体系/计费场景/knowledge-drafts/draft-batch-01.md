## 一、计费架构与演进

### K01: 5G计费系统演进路线
> 来源: 计费解决方案_85721346

**原理知识**

4G及NSA网络中：
- 离线计费：收集用户计费信息生成话单，通过Ga接口访问CG
- 在线计费：收集计费信息及信用控制，通过Gy接口访问OCS

5G SA网络中：
- CHF（Charging Function）融合CG的CGF（计费网关功能）和OCS的ABMF（账户结余管理）+RF（批价功能）
- Nchf接口融合Ga口和Gy口
- **计费本质不变**：搜集流量、时长等信息上报，并对用户进行信用控制

融合计费的优势：
- 配合5GC的SBA架构，利于IT系统云化
- CHF既做配额管理又生成话单，数据归一
- 避免离线与在线计费分离时结果可能不一致的问题

### K02: 5G NSA场景计费选择
> 来源: 计费解决方案_85721346

**方案设计知识**

NSA场景下两种选择：
1. 对2/3/4/5G流量统一计费（**推荐**）
2. 对5G流量单独计费（由gNodeB统计，通过控制面上报CG）

单独5G计费的限制：
- 不支持在线计费
- 不支持基于内容的计费
- 推荐统一计费的原因：基于内容的差异化计费是主流

### K03: 5GC计费架构中的网元角色
> 来源: 计费解决方案_85721346

**原理知识**

| 网元 | 计费相关功能 |
|------|------------|
| SMF | 处理CHF下发的计费参数；计费事件时搜集用量并上报CHF；对在线计费进行配额监控；通过N4向UPF下发配额和计费参数；处理UPF配额用量上报 |
| PCF | 向SMF提供计费策略 |
| CHF | 在线计费：批价、扣费、配额及计费参数下发；离线计费：计费参数下发；处理SMF上报的配额用量并生成CHF-CDR |
| UPF | 按SMF下发的计费参数和配额对用户用量进行计量；满足条件后向SMF上报用量 |

### K04: Nchf服务化接口操作
> 来源: 计费解决方案_85721346

**协议知识**

| 服务操作 | 功能 | 发起方 | 对应融合计费消息 | 对应4G消息 |
|---------|------|--------|----------------|-----------|
| Nchf_ConvergedCharging_Create | 计费会话建立 | SMF | Charging Data Request/Response [Initial] | CCR-I/CCA-I |
| Nchf_ConvergedCharging_Update | 计费会话更新 | SMF | Charging Data Request/Response [Update] | CCR-U/CCA-U |
| Nchf_ConvergedCharging_Release | 计费会话结束 | SMF | Charging Data Request/Response [Termination] | CCR-T/CCA-T |
| Nchf_ConvergedCharging_Notify | 重授权/去活通知 | CHF | Charging Notify Request/Response | RAR/RAA |

### K05: 融合计费端到端消息流程
> 来源: 计费解决方案_85721346

**方案设计知识**

1. PDU会话创建 → SMF向PCF申请计费规则 → PCF返回**计费粒度及计费方式**
2. SMF按优先级**选择对应的CHF**
3. SMF向CHF申请配额 → CHF返回配额及**计费事件**
4. SMF向UPF下发计费规则、计费事件和计费参数
5. 用户访问业务 → UPF解析报文 → 按规则统计用量 → 满足条件时UPF向SMF上报
6. SMF上报到CHF并申请新配额 → 循环直至去活或余额不足

## 二、计费粒度与计费方式

### K06: 计费粒度分类
> 来源: 计费解决方案_85721346

**原理知识**

| 计费粒度 | 说明 | 别名 |
|---------|------|------|
| PDU会话计费 | 不区分业务，所有业务统一费率 | **普通计费** |
| 业务流计费 | 基于业务差别计费，不同业务不同费率(Rating Group) | **内容计费** |
| QoS flow计费 | 基于QoS流计费，不同QoS流不同费率 | **漫游计费** |

### K07: 离线计费与在线计费的本质区别
> 来源: 计费解决方案_85721346

**原理知识**

- **在线计费**：计费会话创建时，SMF**需到CHF申请业务的配额和相关计费参数**
- **离线计费**：计费会话创建时，SMF**只需到CHF申请相关计费参数，无需申请配额**

同一PDU会话中可同时存在在线和离线计费的数据流（融合计费场景）。

现阶段运营商基本按用户区分：部分用户只做在线，部分只做离线。
后续可能按业务区分：正常业务在线计费，免费业务离线计费（仅用于话单核对）。

### K08: CHF选择优先级
> 来源: 计费解决方案_85721346

**配置知识**

CHF选择优先级从高到低：
1. SMF从PCF下发的FQDN选择CHF
2. SMF根据UDM获取的用户签约CC（Charging Characteristics）选择本地配置的CHF
3. SMF通过NRF发现CHF
4. SMF根据本地配置的CC选择本地配置的CHF

## 三、计费事件（Trigger Condition）

### K09: 计费事件分类体系
> 来源: 计费解决方案_85721346

**原理知识**

**按粒度区分：**
- **PDU会话级（Session级）**：对所有RG生效
- **业务级（RG级）**：仅对指定RG生效

**按上报方式区分：**
- **立即上报（Immediate）**：事件发生时立即收集用量并向CHF上报
- **延时上报（Deferred）**：事件发生时收集用量暂时缓存，在下次立即上报事件时一起上报

**配置优先级：**
- SMF本地可配置计费事件是否使能及上报方式（作为缺省值）
- CHF可向SMF下发计费事件，下发的优先级高于本地配置
- CHF可修改部分计费事件的使能状态和上报方式

**5G vs 2/3/4G差异：**
- 2/3/4G中计费事件大多是立即上报
- 5G融合计费中一些事件的上报方式可修改，运营商可灵活制定，减少信令冲击

**示例：**

| 计费Trigger | 级别 | 缺省类别 | CHF可变更类别 | CHF可使能/去使能 |
|------------|------|---------|-------------|----------------|
| 用户位置变更 | Session/RG | 延迟上报 | 是 | 是 |
| 流量阈值到达 | RG | 立即上报 | 否 | 是 |

## 四、计费可靠性

### K10: CHF主备部署与故障处理
> 来源: 计费解决方案_85721346

**方案设计知识**

- SMF感知主CHF故障 → 切换到备用CHF → 主恢复后可回切
- **主备均故障**时的处理方案：
  - **方案一（推荐建网初期）**：业务放通，SMF本地缓存话单，CHF恢复后重发。前提：SMF和CHF均为华为且CHF短时间故障。非华为或长时间故障可能导致计费损失。
  - **方案二**：去激活已在线用户，新用户无法激活。影响用户体验。

### K11: 4/5G互操作计费
> 来源: 计费解决方案_85721346

**方案设计知识**

基于N26接口的互操作：
- PGW-C与SMF必须共部署
- 使用N40接口对接CHF
- 共部署时PGW-C采用与SMF一致的融合计费机制和接口

5G SA用户的计费场景（均通过SMF+CHF）：
- 在5G NR覆盖下
- 回落到4G覆盖下
- 在4G附着场景下

## 五、业务感知(SA)原理

### K12: 业务感知定义
> 来源: 何为业务感知——What？_86009700

**原理知识**

业务感知（Service Awareness，SA，与5G SA组网无关）：
在用户会话过程中，对数据报文进行解析，区分不同业务，并根据业务策略对业务数据进行转发/控制的过程。

应用场景举例：
- 下载视频 → 分配较多带宽
- 游戏 → 配置较低时延与丢包
- 网页浏览 → 预留少量带宽
- 不安全IP → 禁止访问

### K13: 业务感知的核心流程（7步）
> 来源: 业务感知概览_32946283

**原理知识**

1. **规则下发**：PCF下发规则 → SMF"翻译" → UPF存放于规则库
2. **数据到达**：用户报文经基站到达UPF
3. **解析与识别**：UPF解析报文特征（三四层/协议/七层）
4. **规则匹配**：报文特征与规则库匹配
5. **策略执行**：执行计费/阻塞/重定向/带宽控制/头增强等
6. **数据转发**：报文从UPF继续转发
7. **用量上报**：UPF定期向SMF上报用量（计费用量、时长等）

### K14: 过滤条件体系
> 来源: 相关概念_86169670

**原理知识**

| 过滤条件类型 | 说明 |
|------------|------|
| 三四层过滤 | 网络层/传输层协议、端口号、IP地址 |
| 七层过滤 | URL、User-Agent、method等应用层关键字段 |
| 协议/协议组 | HTTP、RTSP等应用层协议或协议组合 |
| 流过滤器 | 以上三种的组合，实现过滤逻辑 |

**关键规则：**
- 每条规则只能绑定一个流过滤器
- 流过滤器是其他过滤条件的集合
- 报文必须符合规则下**所有**过滤条件才算匹配成功

### K15: 策略类型分类
> 来源: 相关概念_86169670

**原理知识**

| 策略类型 | 含义 | 典型用途 |
|---------|------|---------|
| PCC | 计费与策略控制 | 计费、禁止访问、重定向 |
| BWM | 带宽管理 | 带宽分类、限速 |
| HEADEN | 头增强 | HTTP/RTSP报文头插入字段 |
| WEBPROXY | Web代理 | 服务器地址池设置 |
| SMARTREDIRECT | 智能重定向 | CaptivePortal、DNS重写 |

**关键规则：**
- 一条规则只能绑定一条策略
- 可按绑定的策略类型分类规则（PCC规则、BWM规则等）

### K16: 规则类型分类（动态/预定义/本地）
> 来源: 相关概念_86169670, 何为PCC静态规则——What？_86169704

**原理知识**

| 规则类型 | 规划位置 | 策略决策者 | 触发方式 | 七层支持 |
|---------|---------|-----------|---------|---------|
| 动态规则 | PCF | PCF | PCF根据用户条件动态下发 | **不支持** |
| 预定义规则 | UPF+SMF+PCF | PCF | PCF下发规则名称激活 | **支持** |
| 本地规则 | UPF+SMF | SMF | SMF找不到PCF时下发 | **支持** |

**详细对比：**
- 动态规则：PCF下发业务流特征及管理动作，SMF/UPF不需特殊配置，PCF可随时增删
- 预定义规则：三网元需定义相同规则标识，UPF定义流特征和动作，PCF通过标识激活
- 本地规则：UPF/SMF定义相同标识，SMF下发激活。SMF不支持实时更新已有规则组

**使用场景：**
- 有PCF → 动态规则+预定义规则
- 无PCF或PCF故障 → 本地规则
- 需要七层匹配 → 必须用预定义规则或本地规则（动态规则不支持七层）

### K17: 规则匹配原则
> 来源: 业务感知流程_85850060

**原理知识**

1. 所有报文按规则优先级逐条匹配
2. 报文必须符合一条规则**所有**过滤条件才算成功匹配
3. **不同类型**的规则会同时匹配，每种类型至多匹配一条

匹配维度：
- 不同类型规则间独立匹配（PCC/BWM/HEADEN各自匹配）
- 同类型规则间按优先级先后匹配，直到有一条匹配成功或全部失败

### K18: UPF支持的流动作组合
> 来源: 何为PCC静态规则——What？_86169704

**原理知识**

UPF支持的"流"动作：计费、带宽管理、头增强、QoS等。
- 支持单个动作（仅计费）
- 支持多个动作组合（计费+FUP）

## 六、PCC静态规则配置

### K19: 静态规则的关键一致性约束
> 来源: PCC静态规则原理——How？_86009740

**配置知识（隐性规则）**

对于同一个预定义规则：
1. PCF、SMF和UPF上**规则标识名称**必须一致
2. SMF和UPF上的**URR ID**必须一致

这是配置正确性的前提条件。

### K20: 预定义规则安装流程
> 来源: PCC静态规则原理——How？_86009740

**协议知识**

1. 用户开机 → PDU会话创建请求
2. SMF选择PCF → 建立SM策略偶联 → 获取SM策略
3. PCF决策 → 向SMF下发**预定义规则名称**（如Prerule01、Prerule02等）+ 动态规则 + Triggers
4. SMF选择UPF → 会话建立请求（携带预定义规则名称）
5. UPF安装预定义规则及动态规则 → 返回响应

UPF成功安装后：
- 对接收到的报文进行业务识别
- 判断是否满足规则中的过滤条件
- 满足后按照规则关联的动作处理

### K21: 预定义规则中的Usage Monitoring机制
> 来源: PCC静态规则原理——How？_86009740

**方案设计知识**

在PCF侧：
- 向SMF下发UmData（Usage Monitoring Data），用于监控业务使用量
- 根据累计流量决定下发/删除规则

在SMF侧：
- 将UmData转换成MK（Monitoring Key）URR传递给UPF
- 达到上报阈值时将UPF上报的业务流量转发给PCF

在UPF侧：
- 对匹配的业务流量统计到对应的流量统计容器（URR）中
- 达到上报阈值时向SMF上报

**方案设计示例（A视频专项流量包）：**
- 套餐1（视频专项）：优先计入30G A视频流量，URL+七层匹配，PCC策略
- 套餐2（视频FUP）：每天2000M以内100Mbps，超过后25Mbps，需规则动态替换
- 套餐3（通用流量）：三四层any+七层any，PCC策略
- 套餐4（通用FUP）：20G以内50Mbps，超过后1Mbps，需规则动态替换

### K22: 流量触发的规则动态替换
> 来源: PCC静态规则原理——How？_86009740

**方案设计知识**

当用户流量累计达到阈值时，PCF会动态更新规则：
1. PCF更新规则（删除旧规则，安装新规则）
2. SMF/UPF删除旧规则并安装新规则
3. 新规则生效后，后续报文按新策略处理

**示例：**
- A视频流量累计达2000M → 删除Prerule02(100Mbps) → 安装Prerule03(25Mbps)
- 通用流量累计达20G → 删除Prerule05(50Mbps) → 安装Prerule06(1Mbps)

### K23: 动态规则不支持七层的根本原因
> 来源: 使用PCC静态规则——Why？_32946325

**原理知识**

PCF作为策略控制实体，不处理用户数据报文，因此不具备七层识别能力。
UPF作为业务感知及策略执行的实体，支持通过协议特征库/解析库识别业务。

对于需要七层识别（如A视频业务的HTTP+URL特征）的场景，**必须使用预定义规则或本地规则**。
