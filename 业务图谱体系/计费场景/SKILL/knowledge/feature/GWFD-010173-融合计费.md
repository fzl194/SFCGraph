# GWFD-010173 融合计费 知识文档

## 概述

### 特性定义

融合计费（Converged Charging）是5G网络引入的统一计费架构。其核心含义是：**同一个N40（Nchf）会话可以同时支持在线RG（Rating Group）和离线RG**，即带配额管理的计费（相当于在线计费）与不带配额管理的计费（相当于离线计费）在同一接口上完成。

2/3/4G网络中，在线计费通过Gy接口与OCS交互，离线计费通过Ga接口与CG交互，两套接口独立。5G网络中，PGW-U/UPF收集用户计费信息（时长、流量、事件等），上报到PGW-C/SMF；PGW-C/SMF与CHF（Charging Function）之间通过统一的Nchf接口和融合计费服务进行实时计费和后计费，从而简化网络、降低消息数量。

### 适用网元

- SGW-U、PGW-U、UPF（用户面）
- SMF/PGW-C（控制面，UNC侧）
- CHF（计费功能）
- PCF（策略控制）

### 版本要求

| 涉及NF | 支持版本 | 功能说明 |
|--------|----------|----------|
| SMF | UNC 20.0.0及后续版本 | 指示UPF收集并上报计费信息，处理计费事件并向CHF上报 |
| PGW-U/UPF | UDG 20.0.0及后续版本 | 收集用户资源使用信息，上报给SMF |
| CHF | 无特殊要求 | 配额管理、处理计费信息并生成话单 |
| PCF | 无特殊要求 | 向SMF下发计费规则 |

### License

本特性无需获得License许可即可获得服务。但调测时需确认内容计费基本功能对应的License开关（LKV3G5BC01）已打开。

### 客户价值

- 运营商：支持服务化架构和接口，降低接口耦合度；在线/离线统一接口，简化网络
- 用户：可以选择更为多样化的资费方案

## 核心概念

### 融合计费模式

融合计费同时支持两种计费模式：

1. **带配额管理的计费（在线计费）**：CHF按照用户账户情况授权可用资源（配额），SMF执行并上报。当账户余额不足时，CHF不再授权资源，SMF实时停止用户业务。
2. **不带配额管理的计费（离线计费）**：SMF仅在PDU会话/业务使用过程中及结束后收集计费信息并上报给CHF，不会实时影响业务使用。

### RG（Rating Group）

RG即费率组/费率标识。在融合计费架构中：

- RG由SMF侧通过 `SET USRPROFCHARGE`、`SET APNCHARGECTRL`、`ADD CHARGEMETHOD` 命令的 **RGAPPLIED** 参数决定其应用方式
- RGAPPLIED有三种取值：
  - **ONLINERGONLY**：仅使用在线RG
  - **OFFLINERGONLY**：仅使用离线RG
  - **DEFAULT**：默认模式，由SMF根据用户签约决定使用在线或离线

### URR与融合模式

在融合计费模式下，**每个业务需要创建2个URR**：

| URR | 用途 | USAGERPTMODE | 计费统计类型参数 |
|-----|------|--------------|-----------------|
| URR_xxx_offline | 离线计费 | OFFLINE | OFFMETERINGTYPE |
| URR_xxx_online | 在线计费 | ONLINE | ONLMETERINGTYPE |

这两个URR同时配置在同一个URRGROUP中：
- UPURRNAME1/DOWNURRNAME1：绑定离线URR
- UPURRNAME2/DOWNURRNAME2：绑定在线URR

### 关键规则：RGAPPLIED与URRGROUP配置约束

| RGAPPLIED取值 | URRGROUP配置要求 | 说明 |
|---------------|-----------------|------|
| ONLINERGONLY | URRGROUP中只配在线URR | 仅在线计费 |
| OFFLINERGONLY | URRGROUP中只配离线URR | 仅离线计费 |
| DEFAULT | URRGROUP中**只能配一种模式**（不能同时配离线和在线URR） | 否则会导致计费冲突 |

> **重要约束**：当SMF使用融合计费且RGAPPLIED=DEFAULT时，URRGROUP下**不能**同时配置离线和在线URR，否则会导致计费冲突。这与RGAPPLIED为其他值时的要求（必须同时配置两种URR）正好相反。

## 原理与流程

### 融合计费端到端流程

```
用户终端 ──[1]PDU会话建立──> SMF ──[2]Charging Data Request[Initial]──> CHF
                                    携带: Charging ID, 切片信息, CC, Multiple Unit Usage

                            CHF ──[3]Charging Data Response[Initial]──> SMF
                                    携带: 计费Trigger, Multiple Unit Information

                            SMF ──[4]PFCP Session Establishment Request──> UPF
                                    携带: Create URR信元组

                            UPF ──[5]PFCP Session Establishment Response──> SMF

用户终端 <──────[6]PDU会话建立成功──────────────────────────────────>

用户终端 ──[7]访问业务──> UPF（到达阈值/配额时触发上报）
                            UPF ──[7]PFCP Session Report Request──> SMF
                                    携带: Usage Report信元组

                            SMF ──[8]PFCP Session Report Response──> UPF

                            SMF ──[9]Charging Data Request[Update]──> CHF
                            CHF ──[10]Charging Data Response[Update]──> SMF
```

### Create URR 主要子信元

| 子信元 | 说明 |
|--------|------|
| URR ID | 唯一标识一个Create URR |
| Measurement Method | 计费方式：流量计费、时长计费或事件计费 |
| Reporting Triggers | 上报触发条件：Periodic Reporting、Volume Quota、Time Quota、Volume Threshold、Time Threshold、Linked Usage Reporting等 |
| Volume Quota | 流量配额 |
| Time Quota | 时间配额 |
| Volume Threshold | 流量阈值（总流量、上行、下行） |
| Time Threshold | 时间阈值 |
| Linked URR ID | 关联URR（A link B时，B上报时也上报A） |

### Usage Report 主要子信元

| 子信元 | 说明 |
|--------|------|
| URR ID | 标识上报的URR |
| UR-SEQN | 唯一标识Usage Report |
| Start Time | 起始时间 |
| End Time | 结束时间 |
| Volume Measurement | 流量使用情况（总流量、上行、下行） |
| Duration Measurement | 时长使用情况（ISTM=0：从首包到业务终止；ISTM=1：从首次获授权配额到去激活/终止） |
| Time of First Packet | 首包时间 |
| Time of Last Packet | 尾包时间 |

### 配额超时与业务阻塞机制

当达到阈值或配额耗尽时，UPF发送PFCP Session Report Request申请新配额，并启动定时器：

- **定时器时长** = T3RESPONSE x N3REQUEST + 4秒
- T3RESPONSE 和 N3REQUEST 由 `SET UPPFCPPATH` 命令配置
- 定时器超时未收到配额更新消息时，UPF**阻塞业务**

> 注意：对于一个业务，只能有一种计费类型（离线或在线）。计费类型全局规划，UPF可通过 `ADD URR` 本地配置。

## 配置规则

### 配置架构

内容计费配置需要在PCF、SMF、UPF三个网元上统一进行，保证配置一致性：

```
PCF ──策略下发──> SMF ──PFCP/N4下发──> UPF
（计费规则）     （费率映射/URR创建）   （URR上报/流量统计）
```

### 配置流程（5步）

```
步骤1: 配置业务费率（URR + URRGROUP）
步骤2: 配置PCC策略组（PCCPOLICYGRP，绑定URRGROUP）
步骤3: 配置过滤规则（FILTER + FLOWFILTER + 绑定）
步骤4: 配置业务规则（RULE + USERPROFILE + RULEBINDING）
步骤5: 配置缺省费率（SET URRGRPBINDING）+ 刷新生效（SET REFRESHSRV）
```

### 详细配置步骤

#### 步骤1：配置URR和URRGROUP

每个业务配置2个URR（1个离线 + 1个在线），然后组装到1个URRGROUP中。

**URR配置要点**：
- URRID、USAGERPTMODE、OFFMETERINGTYPE/ONLMETERINGTYPE 在SMF和UPF上必须保持一致
- 离线URR：USAGERPTMODE=OFFLINE，统计类型通过OFFMETERINGTYPE设置
- 在线URR：USAGERPTMODE=ONLINE，统计类型通过ONLMETERINGTYPE设置
- 统计类型：VOLUME（按流量）、DURATION（按时长）、EVENT（按事件）、FREE（免费）

**URRGROUP配置要点**：
- UPURRNAME1/DOWNURRNAME1：上下行触发的离线URR
- UPURRNAME2/DOWNURRNAME2：上下行触发的在线URR
- 融合模式下必须同时配置在线和离线URR（RGAPPLIED != DEFAULT时）
- RGAPPLIED=DEFAULT时只能配一种模式

#### 步骤2：配置PCC策略组

```
ADD PCCPOLICYGRP: PCCPOLICYGRPNM="<策略组名>", URRGROUPNAME="<URR组名>";
```

PCCPOLICYGRP将PCC动作属性与URR组绑定。

#### 步骤3：配置过滤规则

过滤规则分三四层和七层：

- **三四层过滤**：ADD FILTER（IPv4）或 ADD FILTERIPV6（IPv6）
  - 关键参数：L34PROTOCOL（协议类型）、MSSTARTPORT/MSENDPORT（手机端口范围）、SVRSTARTPORT/SVRENDPORT（服务器端口范围）
- **七层过滤**：ADD L7FILTER
  - 关键参数：URLTYPE（URL类型）、URL（匹配的URL）
- **流过滤器**：ADD FLOWFILTER
  - 必须绑定过滤条件，否则匹配失败，对应Rule为无效配置
- **绑定关系**：
  - ADD FLTBINDFLOWF：将三四层过滤器绑定到流过滤器
  - ADD PROTBINDFLOWF：将七层过滤器绑定到流过滤器

#### 步骤4：配置规则和用户模板

```
ADD RULE: RULENAME="<规则名>", POLICYTYPE=PCC, FILTERINGMODE=FLOWFILTER,
         FLOWFILTERNAME="<流过滤器名>", POLICYNAME="<PCC策略组名>", PRIORITY=<优先级>;
```

**优先级规则**：
- 数字越小优先级越高
- 只绑定三四层filter的RULE优先级应**低于**绑定7层filter的RULE
- any过滤条件的RULE优先级设为最低（如65000）

**用户模板**：
- ADD USERPROFILE 创建模板
- ADD RULEBINDING 将规则绑定到模板
- USERPROFILE名称在PCF、SMF、UPF间必须一致，否则策略无法生效
- RULENAME、POLICYTYPE在SMF和UPF上也必须一致

#### 步骤5：配置缺省费率和刷新

```
SET URRGRPBINDING: USERPROFILENAME="<模板名>", DFTURRGRPNAME="<缺省URR组>",
                   DFTSIGURRGNAME="<缺省信令URR组>";
```

- 每个USERPROFILE必须绑定缺省费率URR组
- DFTURRGRPNAME：默认业务费率
- DFTSIGURRGNAME：默认信令费率（异常场景）
- **必须执行 SET REFRESHSRV:REFRESHTYPE=ALL** 使Filter配置生效，否则不生效

### 防欺诈配置

```
ADD SPECURRGRPLIST: URRGROUPNAME="<有欺诈风险的费率组名>";
```

将有欺诈风险的费率加入防欺诈费率列表。开启防欺诈时，RULE下绑定URL类型的7层filter必须同时绑定三四层filter。

### 关键一致性约束

| 配置项 | 约束 |
|--------|------|
| URRID | SMF和UPF必须一致 |
| USAGERPTMODE | SMF和UPF必须一致 |
| OFFMETERINGTYPE/ONLMETERINGTYPE | SMF和UPF必须一致 |
| RULENAME | SMF和UPF必须一致 |
| RULE下的POLICYTYPE | SMF和UPF必须一致 |
| RULE下的POLICYNAME | SMF和UPF必须一致 |
| USERPROFILENAME | PCF、SMF、UPF三方必须一致 |
| RULEBINDING中的RULENAME | SMF和UPF必须一致 |
| RULEBINDING中的POLICYTYPE | SMF和UPF必须一致 |

## 配置案例

### 案例概述

配置一个包含4种场景的融合计费方案：

| 场景 | 业务描述 | 过滤条件 | 计费模式 | 优先级 |
|------|---------|----------|---------|--------|
| 场景1 | 访问www.huawei.com | 七层URL + 三四层dns | VOLUME流量计费 | 100 |
| 场景2 | IMS语音业务 | IPv6三四层（服务器IPv6地址） | VOLUME流量计费 | 110 |
| 场景3 | 基础业务（any） | 三四层any（IPv4+IPv6） | VOLUME流量计费 | 65000/65010 |
| 场景4 | 异常流量（未匹配规则） | 信令URR组 | FREE免费 | - |

### 完整配置脚本

#### 1. 配置URR

```
// 场景1：www.huawei.com网站
ADD URR:URRNAME="URR_URL_offline", URRID=12500, USAGERPTMODE=OFFLINE, OFFMETERINGTYPE=VOLUME;
ADD URR:URRNAME="URR_URL_online",  URRID=2300,   USAGERPTMODE=ONLINE,  ONLMETERINGTYPE=VOLUME;

// 场景2：IMS语音业务
ADD URR:URRNAME="URR_IMS_offline", URRID=13500, USAGERPTMODE=OFFLINE, OFFMETERINGTYPE=VOLUME;
ADD URR:URRNAME="URR_IMS_online",  URRID=3300,   USAGERPTMODE=ONLINE,  ONLMETERINGTYPE=VOLUME;

// 场景3：基础业务（any）
ADD URR:URRNAME="URR_any_offline", URRID=11500, USAGERPTMODE=OFFLINE, OFFMETERINGTYPE=VOLUME;
ADD URR:URRNAME="URR_any_online",  URRID=1300,   USAGERPTMODE=ONLINE,  ONLMETERINGTYPE=VOLUME;

// 场景4：异常流量（免费）
ADD URR:URRNAME="URR_abnormal_offline", URRID=14500, USAGERPTMODE=OFFLINE, OFFMETERINGTYPE=FREE;
ADD URR:URRNAME="URR_abnormal_online",  URRID=4300,   USAGERPTMODE=ONLINE,  ONLMETERINGTYPE=FREE;
```

**命名规律**：`URR_{业务}_{模式}`，每业务2个URR（offline + online），共8个URR。

#### 2. 配置URRGROUP

```
ADD URRGROUP:URRGROUPNAME="UrrGp_URL", UPURRNAME1="URR_URL_offline", DOWNURRNAME1="URR_URL_offline",
              UPURRNAME2="URR_URL_online", DOWNURRNAME2="URR_URL_online";

ADD URRGROUP:URRGROUPNAME="UrrGp_IMS", UPURRNAME1="URR_IMS_offline", DOWNURRNAME1="URR_IMS_offline",
              UPURRNAME2="URR_IMS_online", DOWNURRNAME2="URR_IMS_online";

ADD URRGROUP:URRGROUPNAME="UrrGp_any", UPURRNAME1="URR_any_offline", DOWNURRNAME1="URR_any_offline",
              UPURRNAME2="URR_any_online", DOWNURRNAME2="URR_any_online";

ADD URRGROUP:URRGROUPNAME="UrrGp_abnormal", UPURRNAME1="URR_abnormal_offline", DOWNURRNAME1="URR_abnormal_offline",
              UPURRNAME2="URR_abnormal_online", DOWNURRNAME2="URR_abnormal_online";
```

**规律**：每个URRGROUP绑定1个离线URR（UP/DOWNURRNAME1）+ 1个在线URR（UP/DOWNURRNAME2），上行和下行使用相同URR。

#### 3. 配置PCC策略组

```
ADD PCCPOLICYGRP:PCCPOLICYGRPNM="PPG_URL", URRGROUPNAME="UrrGp_URL";
ADD PCCPOLICYGRP:PCCPOLICYGRPNM="PPG_IMS", URRGROUPNAME="UrrGp_IMS";
ADD PCCPOLICYGRP:PCCPOLICYGRPNM="PPG_any", URRGROUPNAME="UrrGp_any", SIGURRGRPNAME="UrrGp_abnormal";
```

**要点**：
- PPG_any 绑定了 SIGURRGRPNAME="UrrGp_abnormal"，用于异常场景的信令计费
- 其他PCC策略组只绑定URRGROUP，不配信令URR组

#### 4. 配置防欺诈列表

```
ADD SPECURRGRPLIST:URRGROUPNAME="UrrGp_URL";
```

将有欺诈风险的费率（URL业务）加入防欺诈列表。

#### 5. 配置过滤规则

```
// 场景1：www.huawei.com（七层URL + 三四层dns）
ADD FILTER: FILTERNAME="L34_dns", L34PROTTYPE=STRING, L34PROTOCOL=ANY,
            MSSTARTPORT=0, MSENDPORT=65535, SVRSTARTPORT=0, SVRENDPORT=65535;
ADD L7FILTER: L7FILTERNAME="L7_URL", SUBL7FLTNAME="dns", URLTYPE=URL, URL="www.huawei.com";
ADD FLOWFILTER: FLOWFILTERNAME="ff_L7";
ADD FLTBINDFLOWF: FLOWFILTERNAME="ff_L7", FILTERNAME="L34_dns";
ADD PROTBINDFLOWF: FLOWFILTERNAME="ff_L7", PROTOCOLNAME="dns", L7FILTERNAME="L7_URL";

// 场景2：IMS语音（IPv6三四层）
ADD FILTERIPV6:FILTERNAME="ipv6_ims", L34PROTTYPEV6=STRING, L34PROTOCOL=ANY,
               SVRIPMODE=IP, MSSTARTPORT=0, MSENDPORT=65535, SVRSTARTPORT=0, SVRENDPORT=65535,
               SVRIPV6="fc00:8028:5002:10D::181", SVRIPV6MASKTYPE=LENGTHTYPE, SVRIPV6MASKLEN=128;
ADD FLOWFILTER: FLOWFILTERNAME="ff_IMS";
ADD FLTBINDFLOWF: FLOWFILTERNAME="ff_IMS", FILTERNAME="ipv6_ims";

// 场景3：基础业务any（IPv4 + IPv6）
ADD FILTER: FILTERNAME="L34_any", L34PROTTYPE=STRING, L34PROTOCOL=ANY,
            MSSTARTPORT=0, MSENDPORT=65535, SVRSTARTPORT=0, SVRENDPORT=65535;
ADD FILTERIPV6:FILTERNAME="L34_any_ipv6", L34PROTTYPEV6=STRING, L34PROTOCOL=ANY,
               MSSTARTPORT=0, MSENDPORT=65535, SVRSTARTPORT=0, SVRENDPORT=65535;
ADD FLOWFILTER: FLOWFILTERNAME="ff_L34any";
ADD FLOWFILTER: FLOWFILTERNAME="ff_L34any_IPv6";
ADD FLTBINDFLOWF: FLOWFILTERNAME="ff_L34any", FILTERNAME="L34_any";
ADD FLTBINDFLOWF: FLOWFILTERNAME="ff_L34any_IPv6", FILTERNAME="L34_any_ipv6";
```

#### 6. 配置规则

```
ADD RULE: RULENAME="Rule001_dns", POLICYTYPE=PCC, FILTERINGMODE=FLOWFILTER,
          FLOWFILTERNAME="ff_L7", POLICYNAME="PPG_URL", PRIORITY=100;
ADD RULE: RULENAME="Rule002_IMS", POLICYTYPE=PCC, FILTERINGMODE=FLOWFILTER,
          FLOWFILTERNAME="ff_IMS", POLICYNAME="PPG_IMS", PRIORITY=110;
ADD RULE: RULENAME="Rule003_any", POLICYTYPE=PCC, FILTERINGMODE=FLOWFILTER,
          FLOWFILTERNAME="ff_L34any", POLICYNAME="PPG_any", PRIORITY=65000;
ADD RULE: RULENAME="Rule003_any_IPv6", POLICYTYPE=PCC, FILTERINGMODE=FLOWFILTER,
          FLOWFILTERNAME="ff_L34any_IPv6", POLICYNAME="PPG_any", PRIORITY=65010;
```

**优先级分析**：
- 100：七层URL匹配（最精确，优先级最高）
- 110：IPv6三四层IMS匹配
- 65000/65010：any匹配（最低优先级，兜底）

#### 7. 配置用户模板和绑定

```
ADD USERPROFILE: USERPROFILENAME="userprofile001";
ADD USERPROFILE: USERPROFILENAME="userprofile002";

ADD RULEBINDING: USERPROFILENAME="userprofile001", RULENAME="Rule001_dns", POLICYTYPE=PCC;
ADD RULEBINDING: USERPROFILENAME="userprofile001", RULENAME="Rule003_any", POLICYTYPE=PCC;
ADD RULEBINDING: USERPROFILENAME="userprofile001", RULENAME="Rule003_any_IPv6", POLICYTYPE=PCC;
ADD RULEBINDING: USERPROFILENAME="userprofile002", RULENAME="Rule002_IMS";
```

**设计**：
- userprofile001：匹配URL业务 + any兜底（支持IPv4和IPv6）
- userprofile002：仅匹配IMS语音业务

#### 8. 配置缺省费率

```
SET URRGRPBINDING:USERPROFILENAME="userprofile001", DFTURRGRPNAME="UrrGp_any", DFTSIGURRGNAME="UrrGp_abnormal";
SET URRGRPBINDING:USERPROFILENAME="userprofile002", DFTURRGRPNAME="UrrGp_IMS", DFTSIGURRGNAME="UrrGp_abnormal";
```

**要点**：
- userprofile001 的默认业务费率是any组，默认信令费率是abnormal组
- userprofile002 的默认业务费率是IMS组，默认信令费率也是abnormal组
- 两个USERPROFILE都共享同一个信令异常费率组

#### 9. 刷新生效

```
SET REFRESHSRV: REFRESHTYPE=ALL;
```

> **必须执行**。不执行此命令，FILTER配置变更不会生效。

### 配置三件套对应关系

| 业务 | URR（2个） | URRGROUP（1个） | PCCPOLICYGRP（1个） |
|------|-----------|----------------|-------------------|
| URL网站 | URR_URL_offline + URR_URL_online | UrrGp_URL | PPG_URL |
| IMS语音 | URR_IMS_offline + URR_IMS_online | UrrGp_IMS | PPG_IMS |
| 基础业务any | URR_any_offline + URR_any_online | UrrGp_any | PPG_any |
| 异常流量 | URR_abnormal_offline + URR_abnormal_online | UrrGp_abnormal | （作为SIGURRGRP使用） |

## 验证与调测

### 调测前提

1. 已完成License加载
2. 已阅读GWFD-010173融合计费特性文档

### 调测步骤

#### 步骤1：检查License开关

```
LST LICENSESWITCH:LICITEM="LKV3G5BC01";
```

- SWITCH=ENABLE：继续下一步
- SWITCH=DISABLE：执行 `SET LICENSESWITCH` 打开

#### 步骤2：创建N4接口跟踪任务

在OM Portal上创建N4接口跟踪任务。

#### 步骤3：终端接入

测试终端使用配置的DNN（如apn-test）接入网络。

#### 步骤4：检查PFCP Session Modification Request

查看SMF给UPF发送的PFCP Session Modification Request消息中：
- Create URR信元组的Reporting Triggers
- Volume Threshold、Time Threshold等信元
- 确认下发的触发条件和配额管理信息

#### 步骤5：测试新业务访问

测试终端访问未下发配额的新业务：
- UPF应发送PFCP Session Report Request，携带新URR ID
- 查看SMF是否回复PFCP Session Report Response

**正常流程**：SMF回复并下发新配额 -> 调测完成

**异常排查**：

| 现象 | 排查动作 |
|------|---------|
| SMF未下发配额 | 检查是否有ALM-81017 PFCP路径断告警，检查N4链路 |
| 存在告警且N4异常 | 根据告警处理，消除后重新测试 |
| 无告警但未下发 | 收集N4报文和配置数据，联系技术支持 |
| 业务被阻塞 | 执行SET URRFAILACTION命令配置RETRYFAILACT=CONTINUE放通 |

### 阻塞业务的应急放通

当配额超时导致业务被阻塞时，可通过以下命令临时放通：

```
SET URRFAILACTION: RETRYFAILACT=CONTINUE;
```
