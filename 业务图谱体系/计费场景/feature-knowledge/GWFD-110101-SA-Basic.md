# GWFD-110101 SA-Basic 知识文档

## 概述

### 定义

SA-Basic是SA（Service Awareness，业务感知）的基本功能，主要包括三四层解析、协议识别和七层解析。UDG支持对报文进行业务感知，判断用户业务，并根据规划对这些业务进行多样化处理，如计费、执行重定向、头增强，或识别Host、是否开启了Tethering等。

### 适用NF

| 涉及NF | 支持版本 | 功能说明 |
|--------|---------|---------|
| SGW-U/PGW-U/UPF | UDG 20.0.0及后续版本 | 对报文进行识别和解析，根据解析结果匹配规则，对不同业务进行不同的计费和动作处理；自动连接内网升级服务器检查SA特征库版本更新及下载 |
| SGW-C/PGW-C/SMF | UNC 20.0.0及后续版本 | 下发动态规则给UDG |

- 详细性能影响需要基于流量模型进行评估

## 核心概念

### 业务感知（SA）

SA（Service Awareness）是高效地解析和识别网络上的各种应用。业务感知完成用户数据报文的解析、协议识别和内容识别，获得目的URL等有价值的信息，为业务解析和安全防护等特性提供依据。

### 三四层解析

- **三层（网络层）**：处理数据包的IP路由，可监控IP报文的源/目的IP地址
- **四层（传输层）**：为两台主机上的应用程序提供端到端服务，不同应用层协议在传输层用不同端口号定义，可检测报文段的协议端口号

三四层可解析的报文信息：
- 源IP地址
- 目的IP地址
- IP协议类型（TCP/UDP/ICMP等）
- 源端口号范围（起始端口号~结束端口号）
- 目的端口号范围（起始端口号~结束端口号）

### 协议识别

通过知名端口和特征字识别协议，完成精确的协议识别功能：
- **知名端口协议识别**：对于标准协议类型，使用标准或运营商定义/推荐的端口进行协议识别
- **特征字协议识别**：按照预置的特征库进行识别，返回7层协议和子协议结果

### 七层解析

UDG识别出报文协议类型后，对每一个请求和响应解析出相应的关键字信息，如URL、Method、响应码。

### 规则（Rule）

一个业务可以有一到多条策略，策略是规则（Rule）的集合。规则从两个维度定义：
- **条件**：在特定的条件下（如匹配特定的过滤器）
- **动作**：执行特定的动作（如计费、重定向、带宽控制等）

### 配置架构核心对象

配置架构涉及以下核心对象及其关系：

| 对象 | 说明 |
|------|------|
| Rule | 业务规则，包含条件和动作两个维度 |
| FlowFilter | 流过滤器，定义三四层匹配条件 |
| L7Filter | 七层过滤器，定义七层匹配条件（如URL、Method等） |
| UserProfile | 用户模板，绑定规则到用户 |
| PCCActionProp | PCC动作属性 |
| PCCPolicyGrp | PCC策略组 |
| URR/URRGroup | 使用率上报规则/规则组，用于计费控制 |
| CategoryProp | 分类属性，用于带宽控制 |

## 原理与流程

### 业务感知全景流程

用户接入网络时，业务感知的控制过程如下：

1. **策略生成**：PCRF/PCF根据用户签约信息、业务信息或用户状态信息等进行决策，生成控制策略
2. **策略下发**：PCRF/PCF通过N7接口将QoS及计费策略下发给SGW-C/PGW-C/SMF
3. **策略转发**：SGW-C/PGW-C/SMF通过N4接口将这些策略再下发给SGW-U/PGW-U/UPF
4. **报文处理**：SGW-U/PGW-U/UPF对用户数据报文的处理按照以下顺序进行：
   - 三四层解析
   - 协议识别
   - 七层解析
5. **规则匹配**：对用户报文进行匹配，直到规则匹配成功
6. **策略执行**：根据匹配的规则将QoS信息传递给无线、核心网共同进行端到端资源管理

### 解析与识别处理顺序

SA-Basic对用户数据报文的处理按照固定顺序进行：

```
三四层解析 → 协议识别 → 七层解析 → 规则匹配 → 策略执行
```

#### 三四层解析

- 三层为网络层，处理数据包的IP路由，可监控IP报文的源/目的IP地址
- 四层为传输层，为两台主机上的应用程序提供端到端服务
- UDG支持三四层信息的过滤解析，可感知报文的五元组信息

#### 协议识别

知名端口和特征字识别协议功能可以完成精确的协议识别：

**知名端口协议识别**：
- 对于许多标准的协议类型，使用标准或运营商定义或推荐的端口进行协议识别
- 通过知名端口进行识别的准确率很高，而通过特征字识别不明显或识别流程复杂的协议适合使用知名端口
- 常见知名端口映射：

| 协议 | 知名端口 |
|------|---------|
| HTTP | 80/8080 |
| WAP1.X | 9200-9201 |
| IMAP4 | 143 |
| DNS | 53 |
| FTP | 20/21 |
| SMTP | 25 |
| POP3 | 110 |
| RTSP | 554 |
| TFTP | 69（服务器侦听端口，数据面端口为服务器随机确定） |

- 在UDG上配置知名端口，通过知名端口直接识别出协议，可以加快协议的识别处理
- 若配置的知名端口不能识别报文协议，则通过特征字完成协议识别，根据协议识别结果完成后续解析

**特征字协议识别**：
- 按照预置的特征库进行识别
- 返回7层协议和子协议结果

#### 七层解析

- UDG识别出报文协议类型后，对每一个请求和响应解析出相应的关键字信息，如URL、Method、响应码
- 支持URL长度配置：通过 `SET SRVCOMMONPARA` 命令的 `SAURLLEN` 参数配置系统支持解析的最大URL长度为511或1023字节
- 若URL长度超过命令配置的规格将进行截断处理，可能导致业务策略匹配结果不准确

#### 前缀URL处理

若用户访问的业务需要与代理服务器进行交互，业务报文中可能同时包含前缀URL和真实URL：
- 前缀URL表示代理服务器的域名信息
- UDG对业务报文进行前缀URL的检测和匹配
- 若匹配成功，则从SA解析得到的URL中剥离掉前缀URL，采用其后的真实URL进行策略匹配
- 否则，将按照常规URL进行策略匹配

**前缀URL示例**：
- SA解析得到的URL为 `http://www.example.com/http://www.example2.com`
- 其中 `http://www.example.com` 表示前缀URL
- `http://www.example2.com` 表示真实URL
- 需要检测和剥离掉前缀URL，采用真实URL进行策略匹配

**前缀URL规格**：
- 支持配置100个前缀URL组
- 每个前缀URL组支持配置10个前缀URL
- 可配置的前缀URL总数为1000个

### 规则匹配原理

UDG从UNC获取用户所属PDR（Packet Detection Rule），PDR由UNC基于本地策略或PCF/PCRF下发的规则生成。

规则匹配过程：
1. 用户访问业务时，UDG基于PDR中携带的规则信息进行匹配
2. 匹配过程按三四层解析、协议识别、七层解析的顺序逐层进行
3. 直到规则匹配成功后执行相应的策略动作

### UNC和UDG配置映射

SA配置在UNC和UDG上需要保持一致的参数：
- APN/VPN
- FlowFilter（5G映射为Gx接口appid）
- ServiceProp
- Rule
- UserProfile
- QosProp
- BwmUserGroup
- CbbID（URRID）

**配置一致性检查**：
- UDG每隔30分钟对控制面、用户面的配置进行扫描
- 如果配置不一致，则会产生 **ALM-81054 CP和UP关键配置不一致（风筝S）** 告警
- 后续扫描如果没有出现配置不一致的问题，则告警恢复

**配置不一致处理策略**（通过 `SET CUINCONSPOLICY` 的 `BYPASSSW` 参数控制）：

| BYPASSSW取值 | 处理策略 |
|-------------|---------|
| DISABLE（默认） | 对于配置相关的Create和Modify消息，UPF回复Failed Rule ID，UPF与SMF交互失败；对于Delete消息，UPF会忽略。激活场景中如果UPF与SMF交互失败则激活失败；更新场景中如果UPF与SMF交互失败则继续使用上一次的策略 |
| ENABLE | UPF激活失败时如果有安装成功的PDR，则不回复Failed Rule ID，即不通知SMF。激活时如果某个PDR因为配置不一致导致安装失败，则忽略此PDR；如果某个URR安装失败，则忽略PDR中与此URR的关联。只要有能够安装成功的PDR，则按照已安装成功的规则进行业务，同时增加标记标识此用户为CU不一致受影响用户，并向SMF回复激活成功 |

## 配置规则

### 应用限制

1. **HTTP Method匹配限制**：
   - 当协议类型为WAP1.x时，UDG仅支持将GET、POST和CONNECT方法作为method的匹配条件
   - 当协议类型为HTTP时，UDG支持将GET、POST、CONNECT、PUT、DELETE、OPTIONS、HEAD、TRACE方法作为method的匹配条件

2. **PUT/DELETE/OPTIONS/HEAD/TRACE方法的策略限制**：
   - 当在UDG上将HTTP协议的PUT、DELETE、OPTIONS、HEAD和TRACE方法作为method的匹配条件时
   - 支持将执行的策略配置为除URL过滤、ADD EXTENDEDFILTER命令作为过滤条件的动作以外的其他策略，如计费

3. **控制面与用户面配合**：
   - 业务感知功能建议控制面和用户面配合，共同完成配置
   - 控制面、用户面要求合一规划Rule、User Profile、URR/CbbID等参数
   - 在用户使用服务时，控制面也可能会动态下发PCC、业务感知参数
   - 当多套参数重复时（如SMF和UPF配置了同名User Profile，但这个User Profile下绑定的Rule不完全相同），UPF会优先使用控制面下发的配置

## 配置案例

### 应用场景

SA-Basic特性主要应用于以下场景：

#### 场景1：基于时间段、位置、用户类别的策略控制

- **时间段控制**：当网络使用高峰期时，一些业务会匹配到带宽控制策略，通过限速优化网络，缓解整体网络拥塞情况
- **位置控制**：当用户处于球场、电影院等人口密度较大的地点，对用户数据流量进行分流，达到保障网络稳定性的目的
- **用户类别控制**：根据用户MSISDN/GPSI号段，对号段应用不同的策略，区分计费并达到差异化体验

#### 场景2：感知报文属性

- **业务识别**：识别用户具体业务，如访问的网址、用户行为（下载文件、观看视频或直播等），并基于不同的业务制定不同的策略
- **Tethering检测**：通过TTL值，识别用户有无开热点并有其他用户通过该热点接入（即Tethering），并对Tethering行为进行控制或计费
- **异常流量防护**：判断用户是否存在异常下行流量，有效防止下行流量攻击，维护用户和运营商的权益

### 配置架构说明

业务感知的配置遵循"条件 + 动作"的规则模型：

```
业务需求描述
    ↓ 分解对照
条件维度（匹配条件）：
  - 三四层条件：FlowFilter（IP、端口、协议类型）
  - 协议条件：知名端口、特征字识别
  - 七层条件：L7Filter（URL、Method、响应码）
    +
动作维度（策略动作）：
  - 计费控制：URR/URRGroup
  - PCC控制：PCCActionProp/PCCPolicyGrp
  - 带宽控制：CategoryProp
  - 其他：重定向、头增强等
    ↓ 映射
UDG具体配置实现
```
