# 三层图谱对象与关系设计

> 这篇文档回答一个问题:三层图谱到底长什么样、怎么用。
>
> 为了讲清楚,我们用一个真实的业务诉求——"给视频业务做差异化计费"——从头到尾穿一遍。穿的过程中,你会自然看到三层图谱里有哪些对象、对象之间怎么分层、对象之间是什么关系。
>
> 严格依据:根目录本体定义、根目录 schema、`计费场景/three-layer-graph/` 实例文档。

---

## 1. 从一个现场问题说起

假设现场来了这样一个需求:

> 客户要给视频业务做差异化计费:看视频按流量扣费,配额用完后不直接断网,而是重定向到一个提醒页面。在线计费模式,UPF 和 SMF 都要配。

一个有经验的工程师接到这个需求,脑子里会快速过几件事:

1. **这是什么业务问题?** —— 这是计费场景,要做差异化计费,配额耗尽要重定向。
2. **靠什么产品能力实现?** —— 要用内容计费、在线计费这些正式特性;视频网站靠 URL 识别,属于内容计费里的一个子能力。
3. **配置分几步做?** —— 先规划识别条件,再配计费动作链,再配规则,最后配用户档案绑定。
4. **每一步敲哪些命令?** —— 动作链是 ADD URR → URRGROUP → PCCPOLICYGRP;规则是 ADD RULE;绑定是 ADD USERPROFILE → RULEBINDING → SET URRGRPBINDING → REFRESHSRV。

这四类知识,现在分散在业务文档、特性手册、命令手册里,彼此不连通。工程师要在脑子里把它们拼起来。

**三层图谱就是把这四类知识缝成一张端到端可追溯的图。** 上面这四个问题,分别对应图的四段:

```text
业务图谱     ←  这是什么业务问题、选什么方案
  ↓
特性图谱     ←  靠什么正式能力实现
  ↓
任务原子层   ←  配置分哪几个可复用动作
  ↓
命令图谱     ←  每个动作敲哪些命令、操作哪些对象
```

对外叫"三层",建模时是四段加一个证据层。中间多出来的"任务原子层"是个解耦器:特性不直接落命令,先拆成可复用配置动作;证据层独立在外,记录所有知识的原始来源。

下面就用视频差异化计费这个例子,一层一层往下穿。穿的过程中,每一层有哪些对象、对象之间是什么关系,会自然出现。

## 2. 业务图谱:这是什么问题、选什么方案

### 2.1 业务层的骨架对象

工程师第一个判断:这是"业务感知"这个大类下的"计费场景"。于是图谱里点亮三个对象:

```text
业务感知  ─contains→  计费场景  ─instantiated_as→  差异化计费组合方案
(BusinessDomain)       (NetworkScenario)        (ConfigurationSolution)
```

这三个对象是业务图谱的骨架:

- **BusinessDomain(业务域)**:一类长期存在的工程知识空间。这里就是"业务感知"。
- **NetworkScenario(场景)**:业务域里反复出现的一类稳定需求情境。这里就是"计费场景"。
- **ConfigurationSolution(配置方案)**:某个场景下形成的一套稳定方案闭包。这里就是"差异化计费组合方案"。

> 这三者的边界怎么分?一句话:**能独立回答"现场到底在解决什么问题"的,优先是 Scenario;已经具体到"怎么组织一套方案"的,是 Solution。** 业务域是 Scenario 上面那层讨论边界。

### 2.2 方案上有决策、有规则、有语义桥梁

光有骨架不够。差异化计费方案要做很多选择,这些选择会决定后面用哪些特性、走哪些命令。这些选择在图谱里叫 **DecisionPoint(决策点)**:

- 网元范围选哪个? → UPF+SMF / UPF only / SMF only (DP-00)
- 计费方式选哪个? → 离线 / 在线 / 融合 (DP-01)
- 配额耗尽怎么办? → BLOCK / REDIRECT / FORWARD (DP-02)
- 按什么层次匹配? → L34 / L7 URL / L7 协议 / 混合 (DP-03)

视频例子这里:DP-00=UPF+SMF、DP-01=在线、DP-02=REDIRECT、DP-03=L7 URL。

方案还要受一些规则约束,这些规则叫 **BusinessRule(业务规则)**,比如:

- URRID 必须全局唯一
- 默认 URR 组必须配置
- SMF 和 UPF 的 RG 值要一致
- REFRESHSRV 必须最后执行

业务层还有一种特殊对象叫 **SemanticObject(语义对象)**,它是业务和技术之间的桥。业务说"配额""过滤条件""计费方式",这些词不直接是命令,也不直接是特性,要先翻成语义对象,再由特性和命令去承接:

- BA.FilterCondition(过滤条件语义)
- BA.Charging(计费语义)
- BA.Quota(配额语义)
- BA.Rule(规则语义)
- BA.Priority(优先级语义)
- BA.Binding(绑定语义)

### 2.3 业务层往下连到哪里

方案定了,就要往下连到正式特性。这个连接关系叫 **uses_feature**:

```text
差异化计费组合方案
  ├─ uses_feature → GWFD-110101 SA-Basic           (识别底座)
  ├─ uses_feature → GWFD-020351 PCC基本功能         (执行骨架)
  ├─ uses_feature → GWFD-020301 内容计费基本功能     (计费能力底座)
  ├─ uses_feature → GWFD-020300 在线计费            (计费方式,由 DP-01 选定)
  └─ uses_feature → WSFD-011206 融合计费(SMF 侧)
```

注意:业务图谱**只保留 uses_feature 这条边**,不在业务层展开特性详情——那是特性图谱的事。

到这里,业务图谱的全貌就有了:

```text
BusinessDomain ─contains→ NetworkScenario ─instantiated_as→ ConfigurationSolution
（业务域）                 （场景）                             （配置方案）
                                                     │
                                     ┌───────────────┼───────────────┐
                                     ↓               ↓               ↓
                              has_decision    constrained_by   uses_semantic_object
                              (决策点)         (业务规则)        (语义桥梁)
                                                     │
                                                     ↓
                                              uses_feature → 特性图谱
```

## 3. 特性图谱:靠什么正式能力实现

顺着 uses_feature 走下来,进入特性图谱。

### 3.1 正式特性和子特性

计费场景下涉及的正式特性共 14 个(UDG/UPF 侧 9 个,UNC/SMF 侧 5 个)。视频差异化计费这个例子,主要点亮这几个:

```text
GWFD-020301 内容计费基本功能   ← 计费能力底座
  ├─ has_subfeature → SF-...-URL       基于 URL 识别计费   (视频网站走这条)
  ├─ has_subfeature → SF-...-L34       基于 L34 识别计费
  ├─ has_subfeature → SF-...-ANY       默认 any 兜底
  └─ has_subfeature → SF-...-ABNORMAL  异常流量缺省

GWFD-020300 在线计费           ← 计费方式,由 DP-01 选定
GWFD-110101 SA-Basic           ← 识别底座
GWFD-020351 PCC基本功能        ← 执行骨架
```

这里有两个关键对象:

- **Feature(特性)**:产品正式定义的能力,有独立特性编号、独立文档、独立 License。`GWFD-020301` 就是 Feature。
- **SubFeature(子特性)**:单个 Feature 内部的稳定细分形态,没有独立特性编号。`SF-...-URL`(URL识别计费)就是 SubFeature,因为它没有自己的特性编号,只是 GWFD-020301 内部的一种识别方式。它们之间的关系叫 **has_subfeature**。

> 判定规则很重要:**特性边界服从产品定义**。只要产品给了正式特性编号,优先判为 Feature。所以"时长计费""流量计费""事件计费""融合计费"虽然语义上像内容计费的细分,但产品都给了独立编号(GWFD-020302/020303/020306/010173),它们都是 Feature,不是 SubFeature。

### 3.2 特性之间的依赖和许可

特性之间不是孤立的。有依赖关系 **depends_on**:

```text
GWFD-020301 ─depends_on→ GWFD-110101   (内容计费依赖 SA-Basic 底座)
GWFD-020300 ─depends_on→ GWFD-020351   (在线计费需要 PCC 骨架)
GWFD-010173 ─depends_on→ GWFD-020301   (融合计费依赖内容计费底座)
WSFD-109002 ─depends_on→ WSFD-109101   (控制面内容计费依赖 PCC 规则框架)
WSFD-011206 ─depends_on→ WSFD-109002   (控制面融合计费依赖内容计费编排)
```

每个特性还可能需要购买 License,关系叫 **requires_license**:

```text
GWFD-110101 ─requires_license→ LKV3G5SABS01
GWFD-020351 ─requires_license→ LKV3G5PCCB01
GWFD-020301 ─requires_license→ LKV3G5BCBC01
GWFD-020302 ─requires_license→ LKV3G5TBCS01   (时长计费)
GWFD-020303 ─requires_license→ LKV3G5VBCS01   (流量计费)
WSFD-109101 ─requires_license→ LKV3W9SPCC11
WSFD-109002 ─requires_license→ LKV3W9BCC12
```

特性层还有自己的规则 **FeatureRule(特性规则)**,比业务规则更靠近能力本体:

- 跨网元名称一致性(USERPROFILENAME/RULENAME 在 PCF/SMF/UPF 必须一致)
- 跨网元 URR 参数一致性
- 默认 any 兜底规则优先级最低
- L34 规则优先级低于 L7 规则
- FLOWFILTER 必须绑定过滤条件
- 融合计费 RGAPPLIED 约束

### 3.3 特性怎么往下连

特性不直接落命令,先拆成可复用配置动作。这个"拆"的关系叫 **decomposes_to**。以 GWFD-020301 为例:

```text
GWFD-020301 内容计费基本功能
  ├─ decomposes_to → 规划识别条件
  ├─ decomposes_to → 规划 Rule 与优先级
  ├─ decomposes_to → 配置七层过滤条件       (子特性 URL 下沉到这里)
  ├─ decomposes_to → 配置流过滤器与绑定
  ├─ decomposes_to → 配置计费动作链
  ├─ decomposes_to → 配置 Rule
  ├─ decomposes_to → 配置 UserProfile 与 Rule 绑定
  └─ decomposes_to → 验证配置链逐层回查
```

视频例子因为走 URL 子特性,所以"配置七层过滤条件"这个动作被选中。

## 4. 任务原子层:配置分哪几个可复用动作

顺着 decomposes_to 下来,进入任务原子层。

这一层的核心对象只有一个:**ConfigTask(配置任务)**——最小可复用配置原子。它回答:为了实现某个能力,需要做哪个稳定配置动作、要什么输入、产出什么结果、由哪些命令完成。

> Task 不是产品官方对象,是图谱为了抽取"配置复用原子"自己定义的。它的边界服从一条规则:**最小可复用配置动作单元,不能再大到变成一个方案,也不能再小到退化成单条命令。**

计费场景共抽出 18 个 ConfigTask,分三类:

- **规划类**:规划识别条件、规划 Rule 与优先级、规划计费对象与费率标识、规划生效范围……
- **执行类**:配置 IP 地址列表、配置三四层过滤条件、配置七层过滤条件、配置流过滤器、配置计费动作链、配置 Rule、配置 UserProfile 与绑定、配置缺省费率、配置融合计费总开关、配置 CHF 交互触发条件……
- **验证类**:验证 License 开关、验证配置链逐层回查、验证 PFCP 上报

一个 task 可以被多个特性复用。比如"配置计费动作链"同时被 GWFD-020301 和 GWFD-020302/020303/020306 引用——因为它做的是同一件事:配 URR → URRGROUP → PCCPOLICYGRP。

### 4.1 task 之间的顺序

特性内部的 task 不是随便排的,有稳定顺序。这个顺序在图谱里用一种专门的边表达,叫 **FeatureTaskOrderEdge(特性-任务编排边)**:

```text
[GWFD-020301 内部]
规划识别条件 ─precedes→ 配置流过滤器
配置流过滤器 ─depends_on→ 配置 Rule         (传递 FLOWFILTER 名称)
配置计费动作链 ─depends_on→ 配置 Rule        (传递 PCCPOLICYGRP 名称)
配置 Rule ─precedes→ 配置 UserProfile 绑定
配置 UserProfile 绑定 ─precedes→ 验证配置链
```

视频例子走的就是这条链:URL 子特性 → 配置七层过滤条件 → 配置流过滤器 → 配置计费动作链 → 配置 Rule → 配置 UserProfile 绑定。

任务层还有自己的规则 **TaskRule(任务规则)**,约束 task 的输入输出、复用边界、校验条件。任务层的 **DecisionPoint** 用来决定:选哪些命令组合、走哪条参数值模式、进入哪个下游 task 分支。

## 5. 命令图谱:每步敲哪些命令、操作哪些对象

顺着 invokes 下来,进入命令图谱。

### 5.1 命令、参数、配置对象

每个 task 通过 **invokes** 边调用一组 MML 命令。以视频例子用到的几个 task 为例:

```text
配置计费动作链
  ├─ invokes → ADD URR
  ├─ invokes → ADD URRGROUP
  └─ invokes → ADD PCCPOLICYGRP

配置流过滤器与绑定
  ├─ invokes → ADD FLOWFILTER
  ├─ invokes → ADD FLTBINDFLOWF
  └─ invokes → ADD PROTBINDFLOWF

配置 Rule
  └─ invokes → ADD RULE

配置 UserProfile 与 Rule 绑定
  ├─ invokes → ADD USERPROFILE
  ├─ invokes → ADD RULEBINDING
  ├─ invokes → SET URRGRPBINDING
  └─ invokes → SET REFRESHSRV
```

这一层有四个核心对象:

- **MMLCommand(命令)**:一条正式 MML 命令。计费场景共 35 条(UPF×13 + SMF×22)。
- **CommandParameter(参数)**:命令参数,是否必选、取值范围、枚举、是否条件必选。
- **ConfigObject(配置对象)**:命令创建/修改/绑定/引用的稳定对象。共 30 个(UPF×10 + SMF×20),如 URR、URRGROUP、PCCPOLICYGRP、FLOWFILTER、RULE、USERPROFILE 等。
- **CommandRule(命令规则)**:命令层约束。

命令和对象之间的关系有两条:

- 每条命令通过 **operates_on** 操作一个主对象(如 ADD URR ─operates_on→ OBJ URR)
- 跨对象引用通过参数 **references** 表达(如 RULE.POLICYNAME ─references→ OBJ PCCPOLICYGRP)

对象之间也有关系边:

```text
URRGROUP ─contains→ URR
PCCPOLICYGRP ─refers_to→ URRGROUP
FLOWFILTER ─contains→ FILTER / L7FILTER
RULE ─refers_to→ FLOWFILTER / PCCPOLICYGRP
USERPROFILE ─refers_to→ RULE / URRGROUP
APNUSRPROFG ─refers_to→ APN / USRPROFGROUP
UPBINDUPG ─refers_to→ USERPROFILE
SELECTCHFGBYCC ─refers_to→ TNFGRP            (主/备 CHF)
```

**CommandRule(命令规则)** 是命令层的硬约束,例如:

- URRID 必须全局唯一
- FLOWFILTER 必须绑定过滤条件,否则无效
- RULE 必须引用已存在的 FLOWFILTER 和 PCCPOLICYGRP
- Filter 变更后必须 SET REFRESHSRV
- SET REFRESHSRV 后 30 秒内禁止改 Filter
- 某些对象被绑定后不能直接删除
- SMF 与 UPF 的 URR 关键参数必须一致

### 5.2 命令之间的顺序

单个 task 内部的命令也有稳定顺序。这个顺序用另一种专门的边表达,叫 **TaskCommandOrderEdge(任务-命令编排边)**:

```text
[配置计费动作链 内部]
ADD URR ─precedes→ ADD URRGROUP          (传递 URRNAME / URRID)
ADD URRGROUP ─precedes→ ADD PCCPOLICYGRP  (传递 URRGROUPNAME)

[配置流过滤器 内部]
ADD FLOWFILTER ─precedes→ ADD FLTBINDFLOWF   (传递 FLOWFILTERNAME)
ADD FLOWFILTER ─precedes→ ADD PROTBINDFLOWF  (条件 DP-03)

[配置 UserProfile 绑定 内部]
ADD USERPROFILE ─precedes→ ADD RULEBINDING      (传递 USERPROFILENAME)
ADD USERPROFILE ─precedes→ SET URRGRPBINDING
SET URRGRPBINDING ─must_be_last→ SET REFRESHSRV (Filter/UserProfile 全部完成才刷新)
```

> 注意:三层图谱里**只有两种编排边**——FeatureTaskOrderEdge(特性层,管 task 顺序)和 TaskCommandOrderEdge(任务层,管命令顺序)。方案级的顺序不单独建模,回到 DecisionPoint 和 BusinessRule 表达。

## 6. 把视频例子完整走一遍

现在把前面四层串起来,看视频差异化计费这条诉求怎么从业务一直落到命令:

```text
[业务图谱]
业务感知 ─contains→ 计费场景 ─instantiated_as→ 差异化计费组合方案
                                              │
                    ┌─────────────────────────┤
                    ↓                         ↓
              has_decision              uses_semantic_object
              DP-00 = UPF+SMF           BA.FilterCondition (视频网站 URL)
              DP-01 = 在线计费           BA.Charging (在线计费语义)
              DP-02 = REDIRECT           BA.Quota (配额与耗尽动作)
              DP-03 = L7 URL 匹配
                    │
                    ↓
              uses_feature
                    ↓
[特性图谱]
GWFD-020301 内容计费基本功能 ─requires_license→ LKV3G5BCBC01
  └─ has_subfeature → SF-...-URL (视频网站走 URL 识别)
        │
        ↓ decomposes_to
[任务原子层]   (FeatureTaskOrderEdge 编排)
配置七层过滤条件 ─precedes→ 配置流过滤器 ─depends_on→ 配置 Rule
        ↓                                        ↓
配置计费动作链 ─depends_on→ 配置 Rule ─precedes→ 配置 UserProfile 绑定
        │
        ↓ invokes
[命令图谱]   (TaskCommandOrderEdge 编排)
配置计费动作链:  ADD URR → ADD URRGROUP → ADD PCCPOLICYGRP
配置流过滤器:    ADD FLOWFILTER → ADD FLTBINDFLOWF → ADD PROTBINDFLOWF
配置 Rule:       ADD RULE
配置绑定:        ADD USERPROFILE → ADD RULEBINDING → SET URRGRPBINDING → SET REFRESHSRV(最后)
```

走完这条链,一条业务诉求就被翻译成了一组有顺序、有对象、有参数、有规则约束的命令实例。

整个过程中,每一层只回答自己那类问题:

| 层 | 回答什么 | 视频例子里的答案 |
|---|---|---|
| 业务层 | 这是什么问题、选什么方案 | 计费场景、差异化计费、在线、重定向、URL 匹配 |
| 特性层 | 靠什么正式能力 | 内容计费基本功能 + 在线计费 + URL 子特性 |
| 任务层 | 分哪几个可复用动作 | 配七层过滤 → 配流过滤器 → 配动作链 → 配 Rule → 配绑定 |
| 命令层 | 每步敲哪些命令 | ADD URR/URRGROUP/PCCPOLICYGRP、ADD FLOWFILTER/...、ADD RULE、ADD USERPROFILE/.../REFRESHSRV |

## 7. 回头看:三层图谱里到底有哪些对象和关系

走完例子,现在可以正式总结三层图谱的对象和关系了。能看懂前面六节的话,这张表只是个速查。

### 7.1 对象层级总表

从上到下,三层图谱(含任务原子层和证据层)的对象分五组:

| 层 | 对象 | 一句话角色 |
|---|---|---|
| 业务层 | BusinessDomain | 一类长期存在的工程知识空间 |
| | NetworkScenario | 业务域内反复出现的稳定需求情境 |
| | ConfigurationSolution | 某场景下的稳定方案闭包 |
| | BusinessRule | 业务层规则(场景/方案选择、范围、原则、诊断) |
| | DecisionPoint | 会引起结构分叉的稳定选择点(跨层复用) |
| | SemanticObject | 业务到能力的语义桥梁(跨层共享) |
| 特性层 | Feature | 产品正式定义的能力(有特性编号、License、版本) |
| | SubFeature | Feature 内部稳定细分形态(无独立编号) |
| | License | 跨层共享的能力许可 |
| | FeatureRule | 特性层规则(依赖、子特性选择、跨NF一致性) |
| 任务层 | ConfigTask | 最小可复用配置原子 |
| | TaskRule | task 层规则(输入输出、复用边界、校验) |
| 命令层 | MMLCommand | 一条正式 MML 命令 |
| | CommandParameter | 命令参数 |
| | ConfigObject | 命令操作的稳定配置对象 |
| | CommandRule | 命令层规则(语法、参数依赖、对象引用、运行时) |
| 证据层 | EvidenceSource | 统一证据本体,承接 md/excel/pdf,跨层共享 |

> Scope(作用范围)和 Participant(参与方)保留语义,但作为宿主对象的子对象,不独立成一级本体。

### 7.2 主链

```text
BusinessDomain → NetworkScenario → ConfigurationSolution
  → Feature / SubFeature → ConfigTask → MMLCommand → CommandParameter / ConfigObject
```

还有一条语义贯穿的副链:业务不直接跳到底层对象,中间经过语义对象这层桥。

```text
Scenario / Solution → SemanticObject → Feature / Task → ConfigObject
```

### 7.3 关系分五类

**归属关系(谁属于谁):**

- BusinessDomain ─contains→ NetworkScenario
- Feature ─has_subfeature→ SubFeature
- ConfigObject ─contains→ ConfigObject

**依赖关系(没有前者,后者不成立):**

- Feature ─depends_on→ Feature
- Feature ─requires_license→ License
- ConfigObject ─depends_on→ ConfigObject

**分解关系(上层向下展开):**

- ConfigurationSolution ─uses_feature→ Feature
- ConfigurationSolution ─uses_task→ ConfigTask
- Feature / SubFeature ─decomposes_to→ ConfigTask
- ConfigTask ─invokes→ MMLCommand
- MMLCommand ─operates_on→ ConfigObject
- CommandParameter ─references→ ConfigObject

**规则与决策约束(横向治理):**

- 各层对象 ─constrained_by→ 对应层 Rule
- Scenario / Solution / Task ─has_decision→ DecisionPoint
- Scenario / Solution / Feature ─uses_semantic_object→ SemanticObject

**编排关系(顺序、依赖、兜底)——唯一两类以边独立建模的顺序结构:**

- FeatureTaskOrderEdge:Feature → Task 顺序(宿主是 Feature / SubFeature)
- TaskCommandOrderEdge:Task → Command 顺序(宿主是 ConfigTask)

## 8. 三层图谱的设计边界

- **业务图谱**定义问题空间、场景和方案
- **特性图谱**定义正式能力和能力细分
- **任务原子层**定义最小可复用配置动作
- **命令图谱**定义命令、参数、配置对象和命令级约束
- **证据层**为所有对象和关系提供原始来源

三条最重要的边界判定规则:

1. **特性边界服从产品定义**——有正式特性编号的优先判为 Feature,不是 SubFeature。
2. **Task 边界服从配置复用粒度**——最小可复用配置原子,不能再大到变成方案,也不能再小到退化成单条命令。
3. **顺序边界服从所在宿主层级**——FeatureTaskOrderEdge 归特性层,TaskCommandOrderEdge 归任务层;方案级顺序回到 DecisionPoint 和 BusinessRule 表达。
