# 三层图谱本体标准定义

> 目标: 给出云核心网配置数字孪生的三层图谱本体定义，只描述对象本体、分类原则和关系主链，不细化到字段级 schema。
> 范围: 基于当前计费案例、产品特性文档、命令文档和既有收敛讨论形成。

---

## 1. 本体目标

三层图谱不是文档分类，也不是命令索引。

它的目标是把云核心网配置知识收敛成一套稳定本体，回答三类问题：

1. 业务上为什么要这样配
2. 产品能力上靠什么来配
3. 底层命令上最终怎么配

为了把这三类问题分开，图谱采用下面这条主链：

```text
业务域
  -> 场景
  -> 配置方案
  -> 特性 / 子特性
  -> 配置任务
  -> 命令
  -> 参数 / 配置对象
```

这条链不是执行流程，而是知识本体的层级展开。

---

## 2. 三层图谱的基本划分

如果只按对外口径说“三层”，建议理解为：

### 2.1 业务图谱

回答：

- 现场到底在解决什么问题
- 什么条件下属于这一类问题
- 一般采用什么配置方案
- 方案需要哪些能力、哪些配置任务

### 2.2 特性图谱

回答：

- 产品正式提供了什么能力
- 这些能力之间有什么依赖关系
- 单个能力内部有哪些稳定细分形态
- 一个能力通常拆成哪些配置任务

### 2.3 命令图谱

回答：

- 命令是什么
- 参数是什么
- 配置对象是什么
- 命令级约束是什么

### 2.4 共享层

虽然对外叫三层，但建模时有两个共享层必须单独承认：

1. `ConfigTask`
   - 它位于特性和命令之间
   - 是最小可复用配置原子
   - 既服务业务方案，也服务特性展开

2. `EvidenceSource`
   - 它位于三层之外
   - 统一承接原始 md、excel、pdf、txt 等证据

所以完整本体更准确地说是：

```text
业务图谱
特性图谱
任务原子层
命令图谱
证据层
```

## 3. 本体分类原则

### 3.1 先按“对象在知识世界里属于哪一层”分类

不按文档来源分，也不按实现顺序分。

例如：

- 来自业务文档的不一定属于业务层
- 来自特性文档的不一定属于特性层
- 来自命令文档的也不一定都要做成节点

### 3.2 稳定对象做本体，弱对象不强行实体化

稳定可复用、能长期积累、可跨方案引用的对象，才进入主本体。

只在某次方案上下文里成立、没有独立生命周期、只是某条边的包装对象，不轻易单独建模。

### 3.3 特性边界服从产品定义

只要有正式特性编号、独立特性文档、独立 License 或版本信息，优先判定为 `Feature`。

### 3.4 Task 边界服从配置复用粒度

`Task` 不是产品官方对象，而是图谱为了抽取配置复用原子而定义的对象。

它的边界由“是否是最小可复用配置动作单元”来决定。

### 3.5 关系优先回到边，不滥造中间节点

例如：

- `FeatureDependency` 更适合是一条 `depends_on` 边
- `FeatureLicense` 更适合是一条 `requires_license` 边
- `ConfigObject` 之间的包含、依赖、引用，也优先用边表达

---

## 4. 业务图谱本体定义

业务图谱是“业务组织图”，不是“命令执行图”。

它表达的是一类稳定业务问题如何被识别、分类、收敛成方案，并向下连接到能力和配置任务。

### 4.1 BusinessDomain

`BusinessDomain` 表示一类长期存在的工程知识空间。

它定义讨论边界，回答：

- 这一大类业务问题到底是什么
- 下面会派生出哪些典型场景

在计费案例里：

- `业务感知` 是 `BusinessDomain`

### 4.2 NetworkScenario

`NetworkScenario` 表示业务域内部反复出现的一类稳定需求情境或问题模式。

它回答：

- 什么情况下属于该场景
- 这个场景要达成什么典型结果

在计费案例里：

- `计费场景` 是 `NetworkScenario`

### 4.3 ConfigurationSolution

`ConfigurationSolution` 表示某个场景下形成的一套稳定方案闭包。

它回答：

- 这个场景一般怎么组织落地
- 会用哪些能力
- 会用哪些 task
- 有哪些关键决策分支

它高于特性，低于场景。

### 4.4 BusinessRule

`BusinessRule` 表示业务层的规则对象。

它约束：

- 场景选择
- 方案选择
- 作用范围
- 设计原则
- 诊断与验收判断

### 4.5 DecisionPoint

`DecisionPoint` 表示会引起后续结构分叉的稳定选择点。

在业务层，它通常决定：

- 选哪类方案
- 选哪些特性
- 选哪些 task

例如计费案例里的：

- 在线 / 离线 / 融合计费选择
- L34 / L7 识别路径选择
- 是否进入 SMF-CHF 交互链

### 4.6 SemanticObject

`SemanticObject` 是业务层向下连接能力和配置的语义桥梁。

它的作用是：

- 先把业务问题翻成语义对象
- 再由特性、task、命令去承接

例如计费里的：

- 计费规则
- 默认费率
- 业务识别条件
- 配额动作
- 融合计费触发条件

### 4.7 Scope 和 Participant

`Scope` 和 `Participant` 保留语义，但不作为一级本体。

它们是宿主对象的子对象：

- `Scope` 表示方案、特性或 task 的作用范围
- `Participant` 表示涉及的稳定参与方及职责

---

## 5. 特性图谱本体定义

特性图谱表达的是产品能力世界。

它的核心不是“怎么敲命令”，而是“哪类正式能力在支撑业务场景”。

### 5.1 Feature

`Feature` 是产品正式定义的能力本体。

它通常具备：

- 正式特性编号
- 独立特性文档
- 独立 License 或版本信息
- 客户可识别的能力边界

在计费案例里：

- `GWFD-020301 内容计费基本功能`
- `GWFD-020302 基于业务时长的计费`
- `GWFD-020303 基于业务流量的计费`
- `GWFD-020306 支持事件计费`
- `GWFD-020300 在线计费`
- `GWFD-010171 离线计费`
- `GWFD-010173 融合计费`

这些都应判定为 `Feature`，而不是 `SubFeature`。

### 5.2 SubFeature

`SubFeature` 表示单个正式特性内部的稳定细分形态。

它必须同时满足：

1. 隶属于某个 Feature
2. 需要被单独引用
3. 能单独挂 task、规则或决策
4. 没有自己的正式特性编号

例如在内容计费里，更像 `SubFeature` 的是：

- URL 内容识别计费
- L34 内容识别计费
- 默认 any 兜底计费
- 异常流量缺省计费

### 5.3 License

`License` 是跨层共享本体，但在特性层最常见。

它表达：

- 某个正式特性是否需要购买或打开能力许可
- 哪些特性共享或依赖某个 License

### 5.4 FeatureRule

`FeatureRule` 是特性层规则。

它约束：

- 特性依赖关系
- 子特性选择
- task 选择
- 命名一致性
- 跨 NF 一致性
- 特性级限制条件

它比业务规则更靠近能力本体，但还没有下沉到命令语法层。

### 5.5 FeatureTaskOrderEdge

`FeatureTaskOrderEdge` 表示某个 `Feature` 或 `SubFeature` 下，task 之间的稳定编排关系。

它表达的是：

- 先做什么 task
- 后做什么 task
- 哪个 task 依赖另一个 task
- 哪个 task 是兜底或最后执行

这类顺序是“特性本身通常怎么展开”的顺序，不是方案实例顺序。

---

## 6. 任务原子层本体定义

任务原子层的作用是把“特性能力”与“底层命令”解耦。

### 6.1 ConfigTask

`ConfigTask` 是最小可复用配置原子。

它回答：

- 为了实现某个能力，需要做哪个稳定配置动作
- 这个动作需要什么输入
- 它产出什么结果
- 它通常由哪些命令完成

在计费案例里，典型 `ConfigTask` 包括：

- 规划识别条件
- 规划 Rule 与优先级
- 规划计费对象与费率标识
- 配置过滤规则
- 配置流过滤器与绑定
- 配置计费动作链
- 配置 Rule
- 配置 UserProfile 与 Rule 绑定
- 配置缺省费率
- 配置融合计费总开关
- 配置 CHF 交互触发条件

### 6.2 TaskRule

`TaskRule` 是 task 层规则。

它约束：

- task 的输入输出依赖
- task 的复用边界
- task 的命名约束
- task 的适用条件
- task 的校验条件

它比 `FeatureRule` 更接近执行组织，但仍然不是命令语法规则。

### 6.3 DecisionPoint

在 task 层，`DecisionPoint` 用来决定：

- 选哪些命令组合
- 走哪条参数值模式
- 进入哪个下游 task 分支

因此，task 的差异优先通过 `DecisionPoint`、`TaskRule` 和关系边来表达。

### 6.4 TaskCommandOrderEdge

`TaskCommandOrderEdge` 表示单个 `ConfigTask` 内部的稳定命令顺序。

它表达的是：

- 哪条命令先执行
- 哪条命令依赖上一条命令
- 哪条命令必须最后执行
- 上下文如何在命令间传递

例如在“配置计费动作链”这个 task 内部：

```text
ADD URR
  -> ADD URRGROUP
  -> ADD PCCPOLICYGRP
```

这类顺序是“task 本体内部的稳定执行结构”。

## 7. 命令图谱本体定义

命令图谱表达的是 MML 本体世界。

### 7.1 MMLCommand

`MMLCommand` 是命令本体。

它表示：

- 一条正式 MML 命令
- 命令动作类型
- 命令操作对象

例如：

- `ADD URR`
- `ADD URRGROUP`
- `ADD PCCPOLICYGRP`
- `ADD RULE`
- `SET URRGRPBINDING`

### 7.2 CommandParameter

`CommandParameter` 是命令参数本体。

它表示：

- 参数名称
- 参数是否必选
- 参数取值范围或枚举
- 参数是否条件必选

它是命令语义的一部分，不应上提到业务层或特性层。

### 7.3 ConfigObject

`ConfigObject` 是稳定配置对象本体。

它表示：

- 命令创建、修改、绑定、引用的对象
- 对象之间的包含、依赖、引用、冲突关系

例如：

- `URR`
- `URRGROUP`
- `PCCPOLICYGRP`
- `FILTER`
- `L7FILTER`
- `FLOWFILTER`
- `RULE`
- `USERPROFILE`

### 7.4 CommandRule

`CommandRule` 是命令层规则。

它约束：

- 命令语法
- 参数依赖
- 参数互斥
- 对象引用
- 删除前置条件
- 运行时合法性

例如：

- `URRID` 必须全局唯一
- `FLOWFILTER` 必须绑定过滤条件，否则无效
- `RULE` 必须引用已存在的 `FLOWFILTER` 和 `PCCPOLICYGRP`
- 某些对象被绑定后不能直接删除

---

## 8. 证据层本体定义

### 8.1 EvidenceSource

`EvidenceSource` 是统一证据本体。

它独立于业务、特性、task、命令四层之外。

它的作用是：

- 记录这些本体和关系是从哪份原始材料抽出来的
- 统一承接 md、excel、pdf、txt 等证据
- 让不同层都能复用同一份原始来源

## 9. 三层之间的关系主链

### 9.1 业务到特性

```text
BusinessDomain
  -> NetworkScenario
  -> ConfigurationSolution
  -> Feature / SubFeature
```

含义：

- 业务域里有多个场景
- 一个场景会收敛成一个或多个配置方案
- 一个方案依赖若干正式特性或子特性

### 9.2 特性到任务

```text
Feature / SubFeature
  -> ConfigTask
```

含义：

- 正式能力不是直接落命令
- 先拆成可复用配置原子

### 9.3 任务到命令

```text
ConfigTask
  -> MMLCommand
  -> CommandParameter / ConfigObject
```

含义：

- task 最终靠命令完成
- 命令再落到参数和配置对象

### 9.4 语义对象的贯穿

```text
Scenario / Solution
  -> SemanticObject
  -> Feature / Task
  -> ConfigObject
```

含义：

- 业务不会直接跳到底层命令对象
- 中间先经过语义对象这层桥

## 10. 关系分类标准

当前本体中的关系，建议分成四类理解。

### 10.1 归属关系

表示“谁属于谁”：

- `BusinessDomain contains NetworkScenario`
- `Feature has_subfeature SubFeature`

### 10.2 依赖关系

表示“没有前者，后者不成立”：

- `Feature depends_on Feature`
- `ConfigObject depends_on ConfigObject`

### 10.3 分解关系

表示“上层对象向下展开”：

- `ConfigurationSolution uses_feature Feature`
- `ConfigurationSolution uses_task ConfigTask`
- `Feature decomposes_to ConfigTask`
- `ConfigTask invokes MMLCommand`

### 10.4 编排关系

表示“顺序、依赖、兜底、最后执行”等结构化组织关系：

- `FeatureTaskOrderEdge`
- `TaskCommandOrderEdge`

## 11. 计费案例中的标准落位

为了把定义落实，计费案例可按下面方式理解：

### 11.1 业务图谱

- `业务感知` -> `BusinessDomain`
- `计费场景` -> `NetworkScenario`
- `差异化计费组合方案` -> `ConfigurationSolution`

### 11.2 特性图谱

- `GWFD-020301 内容计费基本功能` -> `Feature`
- `GWFD-020302 基于业务时长的计费` -> `Feature`
- `GWFD-020303 基于业务流量的计费` -> `Feature`
- `GWFD-020306 支持事件计费` -> `Feature`
- `URL 内容识别计费` -> `SubFeature`
- `默认 any 兜底计费` -> `SubFeature`

### 11.3 任务原子层

- `配置计费动作链` -> `ConfigTask`
- `配置过滤规则` -> `ConfigTask`
- `配置流过滤器与绑定` -> `ConfigTask`
- `配置 Rule` -> `ConfigTask`
- `配置 UserProfile 与 Rule 绑定` -> `ConfigTask`

### 11.4 命令图谱

- `ADD URR` -> `MMLCommand`
- `ADD URRGROUP` -> `MMLCommand`
- `ADD PCCPOLICYGRP` -> `MMLCommand`
- `ADD RULE` -> `MMLCommand`
- `SET URRGRPBINDING` -> `MMLCommand`

## 12. 一句话收敛

三层图谱本体的最终边界应当是：

- 业务图谱定义问题空间、场景和方案
- 特性图谱定义正式能力和能力细分
- 任务原子层定义最小可复用配置动作
- 命令图谱定义命令、参数、配置对象和命令级约束
- 证据层为所有对象和关系提供原始来源

其中，**特性边界服从产品定义，Task 边界服从配置复用粒度，顺序边界服从所在宿主层级。**
