## Batch 1：回答（9 特性 11 篇）

### Q1："获取方法"列有 4 种值

| 值 | 含义 | 频率 | 对应参数类型（初步）|
|---|---|---|---|
| **本端规划** | 本地命名/开关/策略 | 最多（50%+）| variable / planned_naming + planned_local_policy |
| **全网规划** | 跨网元/对端协调的标识/ID/地址 | 次多 | variable / planned_user_data |
| **已配置数据中获取** | 引用前序命令产物的外键 | 较多 | reference |
| **固定取值** | 协议固定枚举/License | 较少（~6 次）| fixed |

**关键发现**：同一参数跨文档标注不一致！USAGERPTMODE 在在线计费标"全网规划"，在 URL 过滤标"固定取值"。→ **"获取方法"列有主观性，不能直接信任做分类依据**。建议用"参数枚举值是否封闭"做客观判据。

### Q2：同一 task 跨特性参数差异

- **ADD RULE.POLICYTYPE**：跨 3 篇（在线/离线/URL过滤）**全部=PCC，无差异**。→ 真固参（跨场景不变）。
- **ADD URR.USAGERPTMODE**：在线计费=ONLINE、离线计费=OFFLINE、URL过滤=OFFLINE(固定) → **跨场景有变化 = decision_driven**（但单特性内看着像固参/全网规划）。
- **ADD APN**：不同特性的参数集不同（双栈有 IPv6 VPN 字段，URL过滤只有 APN 名）→ **同一命令在不同 task 里的参数集不同**。
- **SET LICENSESWITCH.LICITEM**：每特性不同 License 编号 → variable/planned。

### Q3：文档结构 = 4 段（非 5 段），操作流程是可选的

**所有 11 篇文档只有 4 个 H2 段**：操作场景 / 必备事项（含前提条件+数据表）/ 操作步骤 / 任务示例。

**没有独立的"操作流程"段**！——流程信息隐含在"操作步骤"段的编号列表中。（注：内容计费 GWFD-020301 部署UPF 是例外，它有操作流程段——**部署类文档有，配置/激活类文档没有**。）

### Q4：配置变体 = 文件级拆分（1:N），不是表内分场景

**没有任何文档用"表1/2/3/4 分场景"**（只有内容计费部署UPF 用多张数据表分场景，但那是部署文档的特殊形态）。

**变体的主要形式**：一个 feature_id 下多份 md 文件，拆分维度 3 种：
- 组网/路由方案（GWFD-010155：4 份 = OSPFv3/VPN+OSPF/VPN缺省/静态+BFD）
- 网元角色（GWFD-020161：SMF 版 + UPF 版）
- 依赖阶段（GWFD-110471：ICAP 互通前置 + URL 过滤主功能）

"可选"步骤和条件逻辑（"> 说明"块）也常见。

### Q5：TaskRule 出现在 5 个位置，归 4 类语义

| 位置 | 频率 | 典型约束 |
|---|---|---|
| 数据表"说明"列 | **最高频** | "X 与 Y 保持一致"、"已通过 Z 命令配置" |
| 操作步骤"> 说明"块 | 高频 | "不能同时使用"、"如果...则"、"可选" |
| 前提条件段 | 中频 | "必须先完成 X"、"已参考 Y" |
| 数据表"取值样例"列 | 低频 | "必须配置为 DISABLE" |
| 操作场景段 | 低频 | "适用于 PGW-U"（网元适用性）|

4 类语义：**一致性约束** / **前置依赖** / **互斥** / **条件分支**。可用关键词规则匹配（"保持一致"/"必须"/"不能同时"/"如果"/"可选"/"否则"）。

## Batch 2：回答（10 篇新文档）

### Q1（续）："获取方法"列新增 2 种值，总计 6 种

| 值 | 含义 | 典型参数 |
|---|---|---|
| 本端规划 | 本地命名/策略/地址/阈值 | RULENAME, CARSHAPEUL, IFIPADDR, CIR |
| 全网规划 | 跨网元一致的 ID/地址 | URRID, ROUTERID, ACLSOURCEIP |
| **与对端协商** ★新 | 必须双方一致的算法/协议参数 | ESPAUTHALGO, DHGROUP, AUTHMETHOD |
| **对端规划** ★新 | 对端主导的取值 | SVRIPMODE, SVRIPV6（取决于对端配置）|
| 已配置数据中获取 | 引用前序命令产物 | UPURRNAME1, URRGROUPNAME |
| 固定取值 | 协议固定枚举 | USAGERPTMODE=OFFLINE（URL过滤中）|

### Q6："本端规划"实际混了 4 种，应该拆

| 子类型 | 含义 | 约束差异 | 例子 |
|---|---|---|---|
| **local_name** | 工程师自定义命名 | 自由文本，schema 验唯一性 | RULENAME, URRNAME, FLOWFILTERNAME |
| **local_enum_choice** | 从枚举里选一个 | 受 enum 集约束 | CARSHAPEUL=CAR/SHAPE, POLICYMODE=Isakmp |
| **site_address** | 现网实际地址/IP | 网管/规划系统提供 | IFIPADDR, SVRIPV6, LoopBack |
| **local_threshold** | 数值阈值/计时 | 受 min/max 约束 | CIR=2000000, PRIORITY=100, SADURATION=3600 |

**结论**：`variable_source` 应从原来的 `planned` 拆成这 4 子类（或至少拆"命名 vs 非命名"）。

### Q7：参数表 ≠ 命令全集（确认）

- IPSec：参数表只列 IPsec/IKE 协议参数，**整个 VNRS 侧（接口/VPN/路由/GRE）缺失**，但脚本里有。
- 多文档漏 License 开关和 REFRESHSRV（参数表没、脚本有）。
- **提取命令参数必须合并"参数表 + 脚本"**，不能只信参数表。

### Q8：依赖关系全靠隐式

10 篇无一显式声明步骤依赖。依赖通过两种隐式机制：
1. 编号顺序（步骤 N 引用步骤 M<N 的对象名）
2. "获取方法=已配置数据中获取"（间接依赖信号）

**依赖图必须自动推导**（参数引用 + 对象名同名匹配），文档不提供显式依赖。

### Q9：METERINGTYPE 决策点 — 文档复用 bug

**严重发现**：GWFD-020302(时长计费)/020306(事件计费) 的部署UPF 文档**复用了 GWFD-020301(流量计费) 的模板**，METERINGTYPE 全部=VOLUME，与特性语义矛盾（时长应=DURATION，事件应=EVENT）。

→ 计量方式（VOLUME/DURATION/EVENT）**是真实决策点**，但 UDG 文档对这个决策点的覆盖是缺失的。Phase B 聚合时需标注"文档样本不区分计量方式"。

### Q2（续）：POLICYTYPE 多值性确认

- GWFD-020351 PCC：POLICYTYPE=**PCC**（与计费一致）
- GWFD-110281 用户Portal：同一 USERPROFILE 下绑定了 **POLICYTYPE=SMARTREDIRECT** 和 **POLICYTYPE=PCC** 两种 RULE

→ **POLICYTYPE 是真正的多值决策点**（PCC / SMARTREDIRECT / ADC / ...），不是固参。单特性内看着固定，跨特性聚合后才能发现。

### Q5（续）：TaskRule 新位置 — 操作流程段 > 说明 blockquote

IPSec 的"双配原则"（VNRS 微服务和 IPsec 微服务必须一对一镜像配置）出现在**操作流程段的 `> 说明` blockquote** 中——这是 Batch 1 的 5 个位置未覆盖的第 6 个位置。

### Q11：参数分类模型——10 种类型已覆盖，无新类型

6 种"获取方法"值（本端规划/全网规划/与对端协商/对端规划/已配置数据中获取/固定取值）+ "本端规划"内 4 子类（local_name/local_enum_choice/site_address/local_threshold）= **10 种**，Batch 3 全部落入，未发现新类型。

但发现边界值"参见某文档"（GWFD-020482 引用外部 LLD 文档）——属于"外部参考"，不入参数分类。

### Q14：隐式时序约束——7 类推导不了

同名匹配可推导大多数依赖，但 7 类隐式时序/约束推导不了：

| 类型 | 含义 | 例子 |
|---|---|---|
| **license_first** | License 开关必须最先 | SET LICENSESWITCH（8/10 篇首步）|
| **refresh_with_delay** | 带延时的刷新后置 | SET REFRESHSRV（60s 后生效 L7Filter）|
| **external_prerequisite** | 外部前置 | LOD SIGNATUREDB（L7 特性必须先加载签名库）|
| **runtime_mutex** | 运行时互斥 | 地址检测"同一时间只能一个探测命令" |
| **variant_mutex** | 变体互斥 | "Tracert 不支持全量探测" |
| **change_side_effect** | 变更前置 | ROUTEENHANCEDSW 切换前需下线用户 |
| **post_verify** | 后置验证 | LST IPFARMSERVER 验证至少一个 server up |

### 新模式发现

1. **INHERIT 枚举值**（GWFD-110202 HTTP2.0回落）：表达跨层级继承（rule > APN > global），需扩展参数取值模型。
2. **License-only 特性**（GWFD-020422 Direct Tunnel）：整篇仅 1 条 SET LICENSESWITCH，无参数表——"该操作无需准备数据"。需 schema 支持"空参数表"。
3. **外部委托步骤**（GWFD-020422）：步骤指向 RNC/UNC 产品手册，不给 MML 命令。
4. **复杂度方差极大**：GWFD-110332（小区负荷）仅 2 参数 vs GWFD-020352（IPv6 SA）20+ 参数 5 张表。
5. **LOD SIGNATUREDB**：IPv6 SA / QoS七层都需要——L7 特性族的通用前置。

## Batch 4：回答（10 篇，验证 Q15-Q18）

### Q15：INHERIT 不普遍
仅 Batch 3 的 HTTP2.0回落 有显式 INHERIT。其他特性通过 SCOPE/RANGE 参数隐式表达层级继承（BWM 的 RANGE=SYSTEM）。**INHERIT 不是通用模式**，是特定特性的设计。

### Q16：SET LICENSESWITCH 首步 — 70% 出现，30% 不出现
- 7/10 篇首步出现。
- 不出现的 3 类：**基础功能**（地址分配/路径管理，无 License 门槛）、**License-only**（端到端跟踪，只有这 1 步）、**疑似遗漏**（关联URL核减：脚本有但步骤没列）。
- **提取规则**：以**脚本为权威源**，步骤缺时从脚本反补。

### Q17：60s 延时 — L7Filter 专属
- 60s 延时**只对 L7Filter 配置出现**（PROTBINDFLOWF 后）。其他类型刷新（USERPROFILE）无延时说明。
- REFRESHTYPE 与配置类型绑定：三四层=ALL，用户模板/规则=USERPROFILE。
- 延时值统一 60s，无其他数值。

### Q18：3 类"极简"特性
1. **License-only**（GWFD-020451）：仅 SET LICENSESWITCH，"无需准备数据"。
2. **单参数开关**（GWFD-020308）：1 个 ADD + 1 个参数（APPLAYERVOLUME=ENABLE）。
3. **差异配置**（GWFD-020307）："详细配置参考部署UPF，此处仅差异"——依赖基线特性。

## 最终综合归纳（4 轮 × 10 篇 = 40 篇 UDG 文档实证）

### 一、参数分类模型（实证 6 类）

| 类别 | 本质 | 约束 | 例子 | 对应 binding |
|---|---|---|---|---|
| **IDENTIFIER（标识）** | 工程师自定义命名 | 唯一性 | RULENAME, URRNAME, FLOWFILTERNAME | variable |
| **REFERENCE（引用）** | 引用已配置对象 | 指向已存在对象 | UPURRNAME1, URRGROUPNAME, FLOWFILTERNAME(in RULE) | reference |
| **ENUM（枚举）** | 固定取值集合 | enum 集合 | POLICYTYPE=PCC/ADC/SMARTREDIRECT, CARSHAPEUL=CAR/SHAPE | fixed 或 variable（视跨场景是否变化）|
| **NUMERIC（数值）** | 数值，可能带公式约束 | min/max 或公式 | CIR, CBS, PRIORITY, SADURATION | variable |
| **ADDRESS（地址/字符串）** | 现网 IP/URL/APN | 格式 + 全网协调 | SVRIPV6, URL, APN, ROUTERID | variable |
| **SWITCH（开关）** | ENABLE/DISABLE 二值 | 二值 | APPLAYERVOLUME, IPV4MATCH, ROUTEENHANCEDSW | fixed 或 variable |

> **关键认知**：fixed/variable/reference 是**行为维度**（值怎么定），这 6 类是**数据维度**（值是什么）。两者正交。一个 ENUM 类参数可能在一个场景里 fixed（POLICYTYPE=PCC），跨场景后 variable/decision_driven（PCC vs ADC vs SMARTREDIRECT）。**只有 Phase B 跨场景聚合才能判定**。

> "本端规划"标注有主观性（同一参数跨文档标注不一致），**不能直接信任做分类依据**。应以"枚举值是否封闭 + 跨场景是否变化"为客观判据。

### 二、Task 模式（6 种）

| 模式 | 特征 | 例子 |
|---|---|---|
| **License-only** | 仅 SET LICENSESWITCH，无参数表 | GWFD-020451 端到端跟踪 |
| **单参数开关** | 1-2 个 ADD/SET，极简 | GWFD-020308 7层流量统计 |
| **差异配置** | "参考基线，仅列差异" | GWFD-020307 TCP重传（依赖 020301 部署UPF）|
| **标准新建** | LICENSE + 多 ADD + REFRESHSRV | GWFD-020301 内容计费部署UPF |
| **协议/规则库加载** | 含 LOD 加载步骤 | GWFD-020359 IM管控, GWFD-110321 OTT规则库 |
| **系统参数调整** | 纯 SET 全局参数，无 ADD 对象 | GWFD-010102 路径管理 |

### 三、文档结构（稳定 4 段 + 操作流程可选）

- **所有文档统一 4 段**：操作场景 / 必备事项（含前提条件+数据表）/ 操作步骤 / 任务示例。
- **操作流程段**：仅**部署类**文档有（内容计费/IPSec），配置/激活类**没有**。无操作流程时，task 边界靠 Agent（决策 B）。
- **配置变体 = 文件级**（1 特性 N 份 md），拆分维度 3 种：组网方案/网元角色/依赖阶段。**不是表内分场景**。

### 四、TaskRule（实证归纳）

**位置**（6 个）：数据表说明列（最高频）> 操作步骤说明块 > 前提条件段 > 操作流程说明blockquote > 取值样例列 > 操作场景段。

**语义类型**（10 类）：
1. **一致性约束**（"X 与 Y 保持一致"）— 数据表说明列
2. **前置依赖**（"必须先配置 X"）— 前提条件 + 数据表
3. **互斥**（"不能同时使用"）— 操作步骤说明块
4. **条件分支**（"如果...则 / 可选"）— 操作步骤说明块
5. **license_required**（是否需要 License 首步）— 跨特性差异
6. **refresh_after_l7filter_60s**（L7Filter 60s 延时 + REFRESHSRV 立即生效）— 说明块
7. **refreshtype_binding**（REFRESHTYPE 与配置类型绑定 ALL/USERPROFILE）— 跨特性一致
8. **param_formula_constraint**（参数间数学约束 CIR=0.5*PIR）— 数据表
9. **enum_no_subset**（枚举不可子集化 WLR 不能拆 WLR_SP/WLR_UD）— 说明块
10. **idempotent_skip**（已加载则跳过 LOD SIGNATUREDB）— 步骤说明

**推导不了的隐式时序**（7 类）：license_first / refresh_with_delay / external_prerequisite / runtime_mutex / variant_mutex / change_side_effect / post_verify。

### 五、对 Phase A/B 设计的关键影响

1. **参数表 ≠ 命令全集** → Phase A 抽参数必须**合并参数表 + 脚本**，以脚本为权威源（步骤遗漏时从脚本反补）。
2. **"获取方法"有主观性** → 不能直接做分类依据；用"枚举封闭性 + 跨场景变化"做客观判据（Phase B 才能判定）。
3. **依赖图靠隐式推导** → "已配置数据中获取" + 对象名同名匹配，但 7 类隐式时序约束推导不了，需 TaskRule。
4. **跨场景聚合是核心**（Phase B）→ 单特性内的"fixed"可能跨场景变"decision_driven"（POLICYTYPE: PCC vs ADC vs SMARTREDIRECT）。这是 Phase A→B 的核心增值。
5. **极简特性需特殊处理** → License-only / 差异配置 / 单参数开关 不走标准 task 拆分流程。
6. **文档质量隐患** → 标注口径不一致（同参数跨文档"获取方法"不同）、文档复用 bug（020302/020306 复用 020301）、步骤遗漏（脚本有但步骤没）。

### 六、参数分类最终建议

binding_type 三值不变（**fixed / variable / reference**），但 variable_source 细化为**实证 5 类**：

| variable_source | 含义 | 对应"获取方法" |
|---|---|---|
| `identifier` | 工程师自定义命名 | 本端规划（名称类）|
| `site_data` | 现网地址/IP/ID/阈值 | 全网规划 + 本端规划（地址/数值类）|
| `enum_choice` | 从枚举集选一个 | 本端规划（枚举选择类）+ 对端协商 |
| `peer_negotiated` | 必须与对端一致 | 与对端协商 / 对端规划 |
| `decision_driven` | 由决策点决定 | 跨场景变化的"看似 fixed"|

> 与之前模型的差异：把原来的 `planned` 拆成了 `identifier` + `site_data` + `enum_choice`；新增 `peer_negotiated`（对端协商）；`decision_driven` 不再仅指单特性内的决策点，而是**跨特性聚合后才能发现的真实决策点**。
