## 1. 业务图谱层

### 1.1 业务域 → 场景 → 方案

| 对象类型 | ID | 名称 | 一句话说明 |
|---|---|---|---|
| BusinessDomain | service-awareness | 业务感知 | 对用户报文解析,区分不同业务,实现策略控制和计费控制 |
| NetworkScenario | NS-01 | 计费场景 | 对不同业务流采用不同计费方式,结合默认计费、免费业务和配额动作完成计费闭环 |
| ConfigurationSolution | DS-01 | 差异化计费组合方案(含配额分支) | 过滤条件识别 + Rule 优先级裁决 + PCC/URR 计费链 + 默认兜底 + 配额耗尽动作切换 |

**主链**:业务感知 ─`contains`→ 计费场景 ─`instantiated_as`→ 差异化计费组合方案

### 1.2 决策点(4 个)

### 1.3 业务规则(11 个)

| ID | 规则名 | 一句话约束 |
|---|---|---|
| BR-01 | SA 基础 License 开启断言 | SA-Basic + 内容计费 + PCC 三项 License 必须 ENABLE |
| BR-02 | 配置链逐层一致性校验 | RULEBINDING→RULE→PCCPOLICYGRP→URRGROUP→URR→FLOWFILTER→FILTER 各层名称绑定一致 |
| BR-03 | PFCP Usage Report 一致性 | Usage Report 中 URR ID 与配置一致 |
| BR-04 | CP/UP URR 一致性诊断 | URR 名称、RG 值、URRID 在 SMF 和 UPF 侧必须一致 |
| BR-05 | URRID 全局唯一 | URRID 在所有 URR 中唯一 |
| BR-06 | 默认 URR 组必须配置 | DFTURRGRPNAME 和 DFTSIGURRGNAME 必须同时配置 |
| BR-07 | RG 值跨侧一致性 | UDG.ADD URR.RG = UNC.ADD URR.RG |
| BR-08 | REFRESHSRV 时序约束 | 必须在所有 ADD/SET 完成后执行;执行后 30s 内不允许修改 Filter |
| BR-09 | PROTBINDFLOWF 协议匹配约束 | PROTOCOLNAME 必须与目标网站实际协议一致 |
| BR-10 | SMF 融合计费三联前置约束 | CHGMODE=NchfMode、CHARGECTRL 使能、CHFINIT=SENDREQ |
| BR-11 | SMF/UPF 跨网元一致性矩阵 | USERPROFILENAME/RULENAME/POLICYTYPE/URRID/RG 等两侧一致 |

**归属**:DS-01 `constrained_by` BR-01 ~ BR-11

## 3. 任务原子层

### 3.1 配置任务(18 个)

| Task ID | 名称 | 阶段 | 一句话目标 |
|---|---|---|---|
| T-PLAN-001 | 规划生效范围 | plan | 确定 UserProfile、DNN/APN 与默认兜底范围 |

| T-PLAN-003 | 规划 Rule 与优先级 | plan | 决定多业务重叠时的 Rule 层次和优先级 |
| T-PLAN-004 | 规划计费对象与费率标识 | plan | 规划 URR→URRGROUP→PCCPOLICYGRP 及 RG/SID/URRID |
| T-PLAN-005 | 规划配额耗尽动作 | plan | 规划 BLOCK / REDIRECT / FORWARD |
| T-EXEC-001 | 配置 IP 地址列表 | execute | 配置 IPLIST 供特定 IP 匹配 |
| T-EXEC-002 | 配置三四层过滤条件 | execute | 配置 L34 过滤条件 |
| T-EXEC-003 | 配置七层过滤条件 | execute | 配置 L7 URL/Host 过滤条件 |
| T-EXEC-004 | 配置流过滤器与绑定 | execute | 组合 FILTER/L7FILTER 并绑定到 FLOWFILTER |
| T-EXEC-005 | 配置计费动作链 | execute | 创建 URR→URRGROUP→PCCPOLICYGRP |
| T-EXEC-008 | 配置 Rule | execute | 绑定 FLOWFILTER 与 PCCPOLICYGRP 并设置优先级 |
| T-EXEC-010 | 配置 UserProfile 与 Rule 绑定 | execute | 创建 USERPROFILE,绑定 RULE 和默认计费组 |
| T-EXEC-011 | 核查 SMF 系统级计费前置 | execute | 核查 PCC 开关、计费接口模式、CHF 交互前置 |
| T-EXEC-012 | 配置 SMF 到 CHF 的 N40 接入与选择链 | execute | 配置 HTTPLEGRP/SBIAPLE/TNFGRP/SELECTCHFGBYCC |
| T-EXEC-013 | 配置 SMF 模板、Trigger 与异常处理 | execute | 配置 CCT/SELECTCCTBYCC/PDUTRIGGER/RGTRIGGER |

| T-VERIFY-002 | 验证配置链逐层回查 | verify | 逐层核查配置链对象与绑定 |
| T-VERIFY-003 | 验证 PFCP 会话上报与计费流量 | verify | 核查 PFCP Session Report 和计费流量上报 |

> 复用说明:T-EXEC-005(配置计费动作链)同时被 GWFD-020301 和 GWFD-020302/020303/020306 引用;T-EXEC-010 同时被 GWFD-020301 和 WSFD-109002 引用。一个 task 可被多个特性复用。

## 5. 图谱结构总览

把上面四层的对象和关系拼起来,计费场景构建出来的图谱长这样:

```text
业务感知(业务域)
  └─ 计费场景(场景)
       └─ 差异化计费组合方案(方案)
            │
            ├─ 4 个决策点 ──── 决定走哪条分支
            ├─ 11 条业务规则 ── 治理方案级约束
            ├─ 10 个语义对象 ── 业务到技术的语义桥
            │
            └─ uses_feature ──→ 14 个正式特性(UDG×9 + UNC×5)
                                    │
                                    ├─ 4 个子特性(均在内容计费下)
                                    ├─ 9 个 License
                                    ├─ 10 条特性规则
                                    ├─ depends_on ── 特性间依赖
                                    │
                                    └─ decomposes_to ──→ 18 个配置任务
                                                          │
                                                          ├─ 10 条特性-任务编排
                                                          │
                                                          └─ invokes ──→ 35 条 MML 命令(UDG×13 + UNC×22)
                                                                          │
                                                                          ├─ operates_on → 30 个配置对象(UDG×10 + UNC×20)
                                                                          ├─ 19 条命令规则
                                                                          └─ 21 条任务-命令编排
```

**跨层连接关系汇总:**

| 关系 | 起点 | → 终点 | 作用 |
|---|---|---|---|
| `contains` | 业务域 | 场景 | 业务域包含场景 |
| `instantiated_as` | 场景 | 方案 | 场景收敛为方案 |
| `has_decision` | 场景 / 方案 | 决策点 | 决定走哪条分支 |
| `constrained_by` | 场景 / 方案 / 特性 / 任务 | 对应层规则 | 横向治理约束 |
| `uses_semantic_object` | 场景 / 方案 / 特性 | 语义对象 | 业务到技术的语义桥 |
| `uses_feature` | 方案 | 正式特性 | 方案用到哪些能力 |
| `uses_task` | 方案 | 配置任务 | 方案通过哪些动作落地 |
| `has_subfeature` | 特性 | 子特性 | 特性内部细分 |
| `depends_on` | 特性 | 特性 | 特性间依赖 |
| `requires_license` | 特性 | License | 能力许可 |
| `decomposes_to` | 特性 / 子特性 | 配置任务 | 能力拆成可复用动作 |
| `invokes` | 配置任务 | MML 命令 | 动作调用哪些命令 |
| `operates_on` | MML 命令 | 配置对象 | 命令操作哪个对象 |
| FeatureTaskOrderEdge | 特性 | (任务间顺序) | 特性内 task 编排 |
| TaskCommandOrderEdge | 任务 | (命令间顺序) | task 内命令编排 |

这张图就是计费场景的完整知识图谱:从"为什么配"(业务层)到"靠什么配"(特性层)到"做哪些动作"(任务层)到"敲哪些命令"(命令层),每一步都有对象、有关系、有规则、有原始证据支撑。
