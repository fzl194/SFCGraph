# 业务图谱挖掘工作流指南

> 本文档抽象自"计费场景"知识挖掘全流程（555篇文档→12份草稿→统一知识库→业务图谱），供复用到其他子场景。
> 只需替换主题名称和目标目录，即可按相同工作流执行。

---

## 0. 前置输入（用户须提供）

开始之前，用户需明确以下信息：

| 项目 | 说明 | 计费场景示例 |
|------|------|-------------|
| **主题名称** | 本次挖掘的子场景名称 | 计费场景 |
| **目标目录** | `business-graph/{主题}/` | `business-graph/计费场景/` |
| **所属业务域** | 对应BusinessDomain | 业务感知 (service-awareness) |
| **关键词列表** | 用于文档筛选的核心关键词 | charging, URR, PCCPOLICYGRP, 计费, RG, 融合计费 |
| **已有基础（可选）** | 现有知识库或图谱文件 | SKILL/knowledge/计费知识库.md |

## 1. 产品文档入口路径（固定知识）

### 1.1 UDG 产品文档路径

| 层级 | 路径 | 说明 |
|------|------|------|
| 一望5G | `5G基础知识/一望5G/` | 基础概念层：SA专题、计费、PCC、QoS、分流 |
| 特性部署指南 | UDG特性部署指南 | 配置命令参考 |
| 命令参考 | UDG命令参考手册 | MML命令详细参数 |

### 1.2 UNC 产品文档路径

| 层级 | 路径 | 说明 |
|------|------|------|
| 一望5G | `5G基础知识/一望5G/` | 与UDG对称，保留UNC版本 |
| 业务专题 | UNC业务专题/ | 解决方案专题文档（最大量） |
| 协议流程 | UNC协议流程/ | 接口信令、会话流程 |
| 初始配置 | UNC初始配置/ | 开局配置 |
| 网络运维 | UNC网络运维/ | 告警、信令跟踪 |
| 知识问答 | UNC知识问答/ | Q&A |
| 术语 | UNC术语/ | 术语条目 |

### 1.3 去重规则

1. **一望5G层**：UDG和UNC文件ID相同、内容一致 → 保留UNC版本
2. **知识问答/术语**：同上，保留UNC版本
3. **业务专题/初始配置/协议流程/网络运维**：仅UNC有，无需去重

## 2. Stage 1: 文档筛选

**目标**：从产品文档全集中筛选出与主题相关的文件，形成可执行的阅读清单。

### 2.1 关键词扫描

按用户提供的关键词列表，在各层级路径中扫描md文件：

```
1. 遍历一望5G目录，扫描标题和正文是否包含关键词
2. 遍历业务专题目录，按子分类扫描
3. 遍历协议流程目录，按接口类型扫描
4. 遍历初始配置、网络运维、知识问答、术语
```

### 2.2 分批次组织

将筛选结果按知识层级分批次，每个批次写入独立的清单文件：

| Batch | 典型内容 | 预估文件数 |
|-------|---------|-----------|
| batch-01 | 一望5G基础概念 | 10-20 |
| batch-02 | 补充基础（如QoS、PCC相关参数） | 5-10 |
| batch-03 | 业务专题核心文档 | 100-300（需拆分） |
| batch-04 | E2E方案设计与策略 | 50-100 |
| batch-05 | 协议流程与信令 | 100-250 |
| batch-06 | 初始配置与运维 | 10-30 |
| batch-07 | 知识问答与术语 | 5-10 |

**大批次拆分规则**：单个batch超过50篇时，按子主题拆分为 `batch-XX-01`, `batch-XX-02`, ...。

### 2.3 产出

```
{目标目录}/
  {主题}-doc-list/
    batch-01-*.md          # 各批次清单（含UNC路径）
    final-{主题}-doc-list.md  # 汇总清单
    verify-report.md       # 校验报告
```

**校验报告内容**：
- 各分类覆盖率（直接相关/弱相关/遗漏）
- 遗漏分析：扫描补充的文件及理由
- 核心文件数统计

### 2.4 质量门禁

- [ ] 所有层级路径均已扫描
- [ ] 去重规则已执行
- [ ] 遗漏分析完成，无高价值遗漏
- [ ] 汇总清单中文件数与各批次合计一致

---

## 3. Stage 2: 分批阅读与知识提取

**目标**：逐批次阅读原始md文件，提取结构化知识条目。

### 3.1 知识条目格式

每条知识按以下结构记录，确保可追溯：

```markdown
### K{编号}: {主题}
> 来源: {UNC相对路径或文件ID}

**摘要**: 一句话概括
**详细内容**: 具体描述（方案设计/原理/配置/协议/运维）
**适用条件**: 何时适用（可选）
**关联知识**: 与其他知识条目的关系（可选）
```

### 3.2 编号规则

- 格式：`K{NNN}`，如 K001, K002, ...
- 每个batch内独立编号，融合时统一重编号
- 跨batch引用时使用 `K{batch编号}.{条目编号}` 临时标记

### 3.3 大批次阅读策略

当单个batch文件数 > 50时：

1. **按子主题拆分为Part**，每Part 20-40篇
2. **每Part产出一份草稿文件**：`draft-batch-{XX}-{YY}.md`
3. **Part内顺序**：先读原理/架构类 → 再读配置/方案类 → 最后读运维/故障类

### 3.4 产出

```
{目标目录}/
  knowledge-drafts/
    draft-batch-01.md
    draft-batch-02.md
    draft-batch-03-01.md    # 大批次拆分
    draft-batch-03-02.md
    ...
    draft-batch-07.md
```

### 3.5 质量门禁

- [ ] 每条知识标注了来源文件
- [ ] 知识范围覆盖：方案设计、原理、配置、协议、运维、故障案例
- [ ] 无大量重复条目（相似知识需合并）
- [ ] 所有batch均已处理

---

## 4. Stage 3: 知识融合

**目标**：将多份知识草稿融合为结构化的统一知识库。

### 4.1 融合规则

1. **去重**：同一知识点在多个batch出现时，保留最详细的版本，补充其他batch的补充信息
2. **归并**：相关知识点按主题归并到同一章节
3. **统一编号**：全库统一 K001-K{MAX} 编号
4. **保留溯源**：融合后的条目保留所有来源文件ID

### 4.2 分Part策略

当草稿总条目数 > 100 时，分Part融合：

| Part | 融合范围 | 典型主题 |
|------|---------|---------|
| Part 1 | batch-01 + batch-02 + 现有知识库 | 基础概念、架构、术语 |
| Part 2 | batch-03 系列 | 核心业务配置 |
| Part 3 | batch-04 + batch-05 | 方案设计、协议流程 |
| Part 4 | batch-06 + batch-07 | 运维、故障案例 |

### 4.3 质量门禁

- [ ] 全库无重复条目
- [ ] 编号连续无缺漏
- [ ] 每条知识至少1个来源标注
- [ ] 分类标签完整

---

## 5. Stage 4: 统一知识库

**目标**：产出最终的结构化统一知识库文件。

### 5.1 文件格式

```markdown
# {主题}统一知识库

> 融合日期: YYYY-MM-DD
> 来源: N份知识草稿，原始来源X篇产品文档
> 编号范围: K001-K{MAX}
> 分类标签: [原理] [配置] [方案设计] [隐性规则] [协议] [运维] [故障案例]

## 章节索引

| 章 | 主题 | 编号段 | 条目数 |
|----|------|--------|--------|
| 一 | xxx | K001-K012 | 12 |
| ... | ... | ... | ... |

---

## 第一章：{主题}

### K001: {标题} [标签]
> 来源: K01, K03

{知识内容}
```

### 5.2 章节组织原则

1. **从宏观到微观**：架构 → 原理 → 配置 → 方案 → 运维
2. **按知识类型分章**：同一类知识归入同一章
3. **每章编号段连续**：便于引用和定位

### 5.3 产出

```
{目标目录}/
  {主题}统一知识库.md
```

### 5.4 质量门禁

- [ ] 章节索引与实际内容一致
- [ ] 编号段无重叠无缺漏
- [ ] 分类标签使用规范集合中的标签
- [ ] 总条目数与各草稿去重后一致

## 6. Stage 5: 业务图谱实例化

**目标**：基于统一知识库，按Schema定义实例化业务图谱。

### 6.1 Schema引用

> **不嵌入Schema全文**，按需读取：[business-graph-schema-final.md](business-graph-schema-final.md)

Schema定义了12种对象类型和多种关系类型。实例化时需覆盖所有相关类型。

### 6.2 实例化5步骤

#### Step 1: 业务层主链

按层级依次实例化：

1. **BusinessDomain** — 继承根节点（通常已存在）
2. **NetworkScenario** — 定义本次子场景
   - 关键字段：`judgment_basis`（什么情况下属于该场景）、`typical_outcome`
3. **DeliverySolution** — 场景收敛为方案闭包
   - 关键字段：`core_mechanism_combo`（核心机制组合）
4. **EngineeringTask** — 拆分为 plan/execute/verify 三阶段任务
   - 每个任务需定义：`input_artifacts`, `output_artifacts`, `command`, `config_object`

#### Step 2: 支撑层

5. **Participant** — 方案涉及的参与方（网元、外部系统）
6. **Scope** — 策略的生效范围
7. **DecisionPoint** — 会导致后续分支的决策点
   - 每个决策点定义：`option_set`（选项集）、`impact_summary`（分支影响）
8. **ValidationPlan** — 验收检查模板
9. **BusinessRule** — 约束/校验/诊断规则

#### Step 3: 桥接层

10. **DomainSemanticObject** — 领域语义对象（防止直接坠落为命令图）
11. **Feature** — 产品特性（含License依赖）
12. **ConfigObject** — 配置对象链（含依赖顺序）

#### Step 4: 关系实例化

定义对象间的关系边：

| 关系类型 | 起点→终点 | 说明 |
|---------|----------|------|
| contains | Domain→Scenario | 域包含场景 |
| instantiated_as | Scenario→Solution | 场景实例化为方案 |
| uses_task | Solution→Task | 方案使用任务 |
| involves | Solution→Participant | 方案涉及参与方 |
| applies_to | Solution→Scope | 方案的生效范围 |
| has_decision | Scenario/Solution→DP | 决策点 |
| validated_by | Solution→ValidationPlan | 验收计划 |
| governed_by | Solution→BusinessRule | 业务规则约束 |
| uses_semantic_object | Scenario→DSO | 使用语义对象 |
| requires_capability | Solution→Feature | 需要特性能力 |
| realized_by_config | Solution→ConfigObject | 通过配置对象实现 |

#### Step 5: 决策驱动扩展（E1-E5 + A1-A2）

在基础关系之上，补充决策驱动的精细化关系：

| 扩展 | 类型 | 说明 |
|------|------|------|
| E1 | DecisionPoint → ConfigObject (parameter_binding) | 决策选项如何映射到配置参数 |
| E2 | DecisionPoint → EngineeringTask (gates) | 决策选项门控哪些任务(active/skip) |
| E3 | ConfigObject → ConfigObject (composes) | 配置对象间的组合关系与基数 |
| E4 | Feature → ConfigObject (differentiates) | 特性如何差异化配置参数 |
| E5 | EngineeringTask → EngineeringTask (output_propagates_to) | 任务间的名称/实体传播 |
| A1 | EngineeringTask.output_cardinality | 执行任务的输出数量公式 |
| A2 | ConfigObject naming/priority_pattern | 命名模式和优先级模式 |

### 6.3 产出

```
{目标目录}/
  0{序号}{主题}业务图谱.md
```

### 6.4 质量门禁

- [ ] 12种Schema类型均已覆盖（无实例的类型显式说明"本次不涉及"）
- [ ] 每个实例的必选字段均已填写
- [ ] 所有关系边均有业务解释
- [ ] DecisionPoint的option_set与实际配置参数一致
- [ ] ConfigObject依赖顺序正确（从底到顶）
- [ ] 与统一知识库的交叉引用表完整

## 7. 全局质量门禁

### 完成度检查

| 阶段 | 产出 | 检查项 |
|------|------|--------|
| Stage 1 | 文档清单 + 校验报告 | 覆盖率 ≥ 95%，无高价值遗漏 |
| Stage 2 | 知识草稿 (N份) | 每条标注来源，无大量重复 |
| Stage 3 | 融合草稿 (可选分Part) | 去重、归并、统一编号 |
| Stage 4 | 统一知识库 | 章节索引一致，编号连续 |
| Stage 5 | 业务图谱 | 12种Schema覆盖，关系完整 |
