# Batch-08: HTTP 头防欺诈（防欺诈族的安全增强）

> 批次 08 | 主题：头防欺诈与头增强的耦合关系 + 防欺诈族横向 | 来源：GWFD-110401 feature-knowledge + UDG防欺诈功能专题 | 特性数 1（+防欺诈族上下文） | 核心度 ★★★（HEADEN 轨道的安全增强层）

---

## 1. 概述

HTTP 头防欺诈（GWFD-110401）是访问限制场景中**最特殊的安全增强特性**：它不独立产生访问限制动作，而是**内嵌于头增强（HEADEN）动作的前置检测层**，确保头增强插入的字段正确，防止用户使用欺诈的头增强信息进行业务。

**防欺诈族上下文**（来自 UDG 防欺诈功能专题）：UDG 提供三类防欺诈能力，本批次聚焦其中的 HTTP 头防欺诈：

| 防欺诈类型 | 子方案 | 与访问限制场景的关系 |
|-----------|--------|---------------------|
| **DNS 计费防欺诈** | DNS端口防欺诈、DNS tunnel防欺诈、启发式防欺诈 | 计费场景相关，访问限制场景不直接涉及 |
| **HTTP 计费防欺诈** | Server IP校验、防欺诈业务规则数据库 | 计费场景相关，访问限制场景不直接涉及 |
| **HTTP 头防欺诈** | 头增强前置检测（**本批次重点**） | **直接相关**：保障头增强正确性 |

---

## 2. 核心知识点（KP）

### KP-1 头防欺诈的核心机制（来自 GWFD-110401 §1.1, §3.2）

头防欺诈是**头增强动作的前置检测层**，在 HTTP 头增强执行**之前**增加"判断插入字段是否正确"的动作：

```
UDG 收到用户 HTTP 请求报文
    ↓ 匹配到头增强规则
获取本地预先配置的增强字段信息（ADD HEADEN）
    ↓
判断用户 HTTP 报文头中是否已存在该字段？
    ├─ 不存在 → 执行正常的 HTTP 头增强动作（插入字段）
    │
    └─ 存在 → 检查字段取值是否正确
              ├─ 取值正确 → 继续业务（不重复插入）
              ├─ 取值不正确 → UDG 将字段修改为正确值（**纠正欺诈**）
              └─ 存在多个该字段 → 删除冗余字段，保留一个正确字段（**清理冗余**）
```

**三种处理结果对应三种业务含义**：
1. **正常插入**：用户报文未携带该字段，标准头增强流程
2. **纠正欺诈**：用户伪造了字段值，UDG 强制纠正为真实值
3. **清理冗余**：报文含多个相同字段（可能是浏览器自动插入），统一保留一个正确值

---

### KP-2 头防欺诈的两种触发方式（来自 GWFD-110401 §3.1）

| 触发方式 | 配置要求 | RULE POLICYTYPE |
|---------|----------|----------------|
| **基于三四层特征触发** | `ADD FILTER`（L3/L4 五元组），不配置 L7FILTER/PROTBINDFLOWF | HEADEN |
| **基于七层特征触发** | `ADD L7FILTER`（URL + METHODTYPE=GET/POST）+ `ADD PROTBINDFLOWF` | HEADEN |

**关键约束**：即使匹配到了规则，对**非 HTTP 协议**的业务流，UDG **不会**进行 HTTP 头防欺诈处理。

---

### KP-3 头防欺诈的两个关键开关（来自 GWFD-110401 §4.3）

头防欺诈的开关**全部内嵌于 `ADD HEADEN` 命令**，无独立配置对象：

| 参数 | 取值 | 含义 | 默认 |
|------|------|------|------|
| **ANTIFRAUD** | ENABLE/DISABLE | **使能 HTTP 头防欺诈**（本特性核心开关） | DISABLE |
| **GRAYLIST** | ENABLE/DISABLE | **灰名单模式**：只处理防欺诈，不插入头增强字段 | DISABLE |

**灰名单模式的特殊用途**：
- `GRAYLIST=ENABLE` + `ANTIFRAUD=ENABLE`：UDG 只检测并纠正错误字段，**不插入**头增强字段
- 适用于"只进行防欺诈而不需要头增强"的场景（如纯安全审计）

---

### KP-4 头防欺诈的配置对象（来自 GWFD-110401 §4.1）

头防欺诈**不引入新的配置对象**，全部复用头增强族的对象：

| 对象 | 命令 | 头防欺诈用途 |
|------|------|-------------|
| `FILTER` | ADD FILTER | 三四层匹配（基于三四层触发时） |
| `FLOWFILTER`/`FLTBINDFLOWF` | ADD FLOWFILTER / ADD FLTBINDFLOWF | 流过滤器链 |
| `L7FILTER` | ADD L7FILTER | 七层匹配（URL + METHODTYPE=GET/POST，基于七层触发时） |
| `PROTBINDFLOWF` | ADD PROTBINDFLOWF | 协议绑定（http + L7FILTER） |
| `WELLKNOWNPORT` | ADD WELLKNOWNPORT | 知名端口（识别 HTTP 协议端口 80/8080） |
| `SIGNATUREDB` | LOD SIGNATUREDB | 协议特征库 |
| `PARSERDB` | LOD PARSERDB | 协议解析数据库 |
| **`HEADEN`** | **ADD HEADEN** | **头增强对象，含 ANTIFRAUD 和 GRAYLIST 开关** |
| `RULE` | ADD RULE | 业务规则（POLICYTYPE=**HEADEN**） |
| `USERPROFILE`/`RULEBINDING` | ADD USERPROFILE / ADD RULEBINDING | 用户模板与规则绑定 |

**核心观察**：防欺诈**完全寄生**于头增强配置链路，没有独立 MML 命令。

---

### KP-5 头防欺诈的应用限制（来自 GWFD-110401 §6.2）

1. **报文类型限制**：只对 HTTP/WAP2.0 的 Post（上传）、下载、Get（浏览请求）报文有效，且仅对**上行报文**处理有意义
2. **头域类型限制**：只支持对**非标准 HTTP 头域**的增强字段防欺诈；如果头增强字段配置为 **HTTP 标准头域名**（如 `host`、`accept`），可能无法防欺诈
3. **协议限制**：只支持 HTTP1.x，不支持 HTTP2.0
4. **加密场景**：不支持加密场景下基于 host 的计费和控制

## 3. 关键发现（跨特性横向归纳）

### 发现-1 头防欺诈与头增强的强耦合关系（本批次核心）

头防欺诈是访问限制场景中**强耦合依赖关系最显著**的特性：

| 维度 | 头防欺诈（GWFD-110401） | 头增强（GWFD-110261） |
|------|------------------------|----------------------|
| **License 依赖** | **强依赖**：启用防欺诈必须开启头增强（双开 82209786 + 82209777） | 独立 |
| **配置载体** | **共用 HEADEN 对象**：ANTIFRAUD/GRAYLIST 内嵌于 ADD HEADEN 命令 | HEADEN 对象本身 |
| **执行顺序** | **前置检测**：防欺诈检测 → 字段纠正/冗余清理 → 头增强插入 | 后置执行 |
| **触发条件** | **共用过滤链**：共用 FILTER/L7FILTER/FLOWFILTER | 同上 |
| **RULE POLICYTYPE** | **共用 HEADEN** | HEADEN |
| **协议支持** | 仅 HTTP（不支持 RTSP） | HTTP/HTTPS/RTSP |

**对图谱建模的关键启示**：
1. 头防欺诈应建模为**头增强对象的 variant/属性**，而非独立特性
2. License 关系：`110401 constrained_by 110261`（启用防欺诈必须先启用头增强）
3. 执行时序：`110401 (前置检测) → 110261 (执行插入)`，灰名单模式下 110261 被跳过

---

### 发现-2 防欺诈族的三个层次（来自防欺诈功能专题）

UDG 防欺诈能力按"**检测维度**"分为三层：

```
┌──────────────────────────────────────────────────┐
│ Layer 1: DNS 计费防欺诈                            │
│   - DNS端口防欺诈：协议一致性深度解析              │
│   - DNS tunnel防欺诈：type=10/16 字段识别         │
│   - 启发式防欺诈：DNS 报文频率/比例统计           │
│   目标：阻止伪装 DNS 报文的免费上网               │
├──────────────────────────────────────────────────┤
│ Layer 2: HTTP 计费防欺诈                           │
│   - Server IP校验：Server IP 与 Host 域名匹配     │
│   - 防欺诈业务规则数据库：七层信息特征匹配        │
│   目标：阻止伪造 Host/URL 骗取免费费率            │
├──────────────────────────────────────────────────┤
│ Layer 3: HTTP 头防欺诈（本批次重点）               │
│   - 头增强前置检测：检查已有字段正确性            │
│   目标：阻止伪造 MSISDN/IMSI 头增强字段           │
└──────────────────────────────────────────────────┘
```

**访问限制场景的关联度**：
- Layer 1（DNS 计费防欺诈）：主要服务于计费场景，访问限制场景间接相关（通过 DNS纠错特性可能涉及）
- Layer 2（HTTP 计费防欺诈）：主要服务于计费场景
- **Layer 3（HTTP 头防欺诈）：直接服务于访问限制场景**——保障头增强插入的 MSISDN/IMSI 等字段正确，使后续的访问控制（Portal 认证、重定向授权）基于真实用户信息

---

### 发现-3 头防欺诈在访问限制链路中的位置

头防欺诈位于头增强动作的**前置安全层**，整个访问限制链路如下：

```
ADC/SA 解析（GWFD-020357）→ 识别 HTTP 报文、提取 Method/URL
    ↓
规则匹配（RULE POLICYTYPE=HEADEN）
    ↓
【头防欺诈检测 GWFD-110401】 ← 检查已有字段是否欺诈
    ├─ 不存在字段：继续头增强流程
    ├─ 字段正确：跳过头增强
    ├─ 字段错误：纠正后继续（或灰名单模式停止）
    └─ 多个字段：清理冗余后继续
    ↓
【头增强插入 GWFD-110261/262/263】 ← 插入 MSISDN/IMSI 等
    ↓
业务服务器获取增强字段
    ↓
【后续访问限制动作】
    ├─ Portal 认证（GWFD-110281）：基于 MSISDN 判断是否签约
    ├─ HTTP 智能重定向（GWFD-110284）：REDIRAPPENDINFO 携带 MSISDN
    └─ URL 过滤（GWFD-110471）：基于真实用户身份做细粒度控制
```

**关键启示**：头防欺诈保障的是**整条访问限制链路的可信基础**——若头增强字段被伪造，后续所有基于用户身份的访问控制都会失效。

---

### 发现-4 头防欺诈的协议覆盖盲区

头防欺诈**仅支持 HTTP/HTTPS**，RTSP 不支持（来自头增强差异表）：

| 协议 | 头防欺诈支持 | 启示 |
|------|-------------|------|
| HTTP | ✓ 支持 | 主要场景 |
| HTTPS | ✓ 支持 | 加密场景仍可检测（因 HTTPS 头增强在 SSL Extension 内） |
| **RTSP** | **✗ 不支持** | 流媒体场景的头增强字段无法防欺诈，存在安全盲区 |

**安全建议**：RTSP 头增强部署时，应配合其他安全机制（如 Streaming Server 自身的认证）弥补防欺诈盲区。

---

### 发现-5 头防欺诈映射到访问限制三种动作

| 访问限制动作 | 头防欺诈如何贡献 |
|-------------|----------------|
| **HEADEN（头增强）** | **间接核心**：不直接产生头增强，但保障头增强插入的字段正确性 |
| **DISCARD（阻塞）** | 不直接阻塞；但灰名单模式下若检测到欺诈字段会纠正而非阻塞 |
| **REDIRECT（重定向）** | 不直接重定向；但保障的重定向携带字段（如 REDIRAPPENDINFO 的 IMSI）真实性 |

**关键观察**：头防欺诈**不产生显式访问限制动作**，它是**安全前置层**，价值在于"让其他动作基于真实信息执行"。

## 4. 配置对象/命令复用清单

### 头防欺诈独有配置对象
- **无独立对象**：全部复用头增强族的 HEADEN 对象

### 头防欺诈的关键参数（内嵌于 ADD HEADEN）
- `ANTIFRAUD=ENABLE`（核心开关）
- `GRAYLIST=ENABLE`（灰名单模式，可选）

### 头防欺诈用到的辅助命令（非独有）
- `ADD WELLKNOWNPORT`（识别 HTTP 协议端口）
- `LOD SIGNATUREDB` / `LOD PARSERDB`（协议特征库与解析库）

### 与头增强族共用的配置对象（完全共用）
- `FILTER` → `FLOWFILTER` → `FLTBINDFLOWF`
- `L7FILTER` → `PROTBINDFLOWF`
- `HEADEN`（含 ANTIFRAUD/GRAYLIST 参数）
- `RULE`（POLICYTYPE=HEADEN） → `USERPROFILE` → `RULEBINDING`

- 强依赖头增强：`82209777 LKV3G5HTHE01`（必须同时开启）

## 6. 后续待办（供 P3 横向分析综合）

1. **防欺诈族整合**：DNS 计费防欺诈、HTTP 计费防欺诈虽然主要服务计费场景，但其检测能力（如 Server IP 校验）可能与访问限制场景的 URL 过滤、HTTP 重定向存在重叠或互补，P3 横向分析时需评估
2. **灰名单模式实际部署率**：现网中 GRAYLIST=ENABLE 的场景占比，需调研（是纯安全审计还是与正常头增强并存）
3. **RTSP 防欺诈盲区缓解**：RTSP 头增强不支持防欺诈，是否有 Streaming Server 侧的补偿机制
