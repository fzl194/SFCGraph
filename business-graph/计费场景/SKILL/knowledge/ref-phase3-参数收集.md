# Phase 3 参考文件：参数收集

## ⚠ 阶段执行约束（必须遵守）

**本文件仅用于 Phase 3。执行本文件内容前必须满足以下前置条件，否则 STOP：**

1. **用户必须已确认方案**：必须已通过 GATE-3（用户在 Phase 2 明确确认了匹配结果）。如果用户未确认方案，**STOP**，回到 Phase 2 等待用户确认。
2. **本阶段完成后必须 STOP**：执行完 §1~§4 后，**必须将参数表展示给用户并停止执行**，等待用户在 Phase 4（GATE-4）确认。不得自动进入配置生成。
3. **禁止在本阶段生成任何 MML 配置命令**：本阶段只收集和推断参数，不生成配置。

---

> 本文件定义 Phase 3 的 pipeline 要求、输出模板和操作规则。
> 业务知识（参数语义、默认值、推导规则）由 Agent 从图谱和知识库动态加载。

---

## 1. Pipeline 步骤

### Step 1: 从需求和现网中推断参数

**操作**：尽可能从以下来源自动推断参数值：
- 用户需求描述（业务名、匹配条件、计费方式等）
- 现网配置（命名规律、已有参数值）
- 业务图谱中的方案示例
- 知识库中的默认值

**必须加载**：
- `kb/08-PFCP-N4接口与URR.md` — URRID、RG、计量方式等参数语义
- `kb/13-计费三件套配置.md` — 三件套参数推导规则
- `04-command-graph.md` — 读取 CommandParameter 定义，确认参数枚举值和约束

**推断规则**（现网命名规律）：
- 如果现网 FILTER 命名为 `filter_xxx`，则新 FILTER 建议用相同前缀
- 如果现网 URR 命名为 `urr_xxx`，则新 URR 建议用相同前缀
- 优先级参考现网已有 RULE 的 PRIORITY 分布，保持合理间距

### Step 2: 列出参数表

**操作**：对每条业务按模板展示参数，标注所有"待提供"和"待确认"项。

**必须加载**：
- `04-command-graph.md` — 读取 ConfigObject 的关键参数列表，构建参数表
- `01-business-graph.md` — 读取 SemanticObject 定义，确认两侧参数差异

### Step 3: 全局参数

**操作**：列出不属于单条业务的全局参数（UserProfile、默认URR组、SMF系统级等）。

**必须加载**（当 DP-CH-02 包含 SMF 时）：
- `kb/18-SMF侧对象体系与协同约束.md` — SMF侧参数差异和协同要求

## 3. 优先级分析流程

> **核心规则**：**数字越小，优先级越高**。不可更改。

### 步骤

1. 从现网配置中提取所有相关 RULE 的 PRIORITY 值
2. 分析现网优先级分布规律（现网数据是最权威判断依据）
3. 根据用户给出的业务优先级列表，计算新规则 PRIORITY（间距取10的倍数）
4. 输出分析表，与用户确认

### 输出格式

```
## 优先级分析

**规则**：数字越小优先级越高（固定规则）

**现网 PRIORITY 分布**：
| RULE 名称 | PRIORITY | 优先级排名 |
|-----------|----------|-----------|
| 现网rule1 | 21 | 第1（最高） |
| 当前新增rule | 待定 | 第N（插入位置） |

**请确认以上优先级分析是否正确。**
```

### 禁止事项

- **禁止**自行假设优先级顺序，必须先从现网提取数据
- **禁止**生成带 PRIORITY 的 RULE 命令，直到完成分析并获得用户确认
- **禁止**在用户未确认前生成任何配置命令

## 4. 注意事项

- 参数表中每个字段都应标注"来源"（需求/现网/推断/图谱/知识库）和"状态"（已知/待确认/待提供）
- 两侧共用参数（URRID、RG、USAGERPTMODE 等）必须标注"**必须与UPF一致**"
- SMF侧参数差异需要从`kb/18-SMF侧对象体系与协同约束.md`确认，不要凭记忆填写
- 遇到不确定的参数语义，加载 `04-command-graph.md` 中对应的 CommandParameter 定义确认
