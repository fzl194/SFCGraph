# GWFD-020306 支持事件计费 知识文档

## 概述

事件计费（Event Based Charging）是依据用户使用某种数据业务的**次数**进行计费的一种计费方式，与基于流量、时长计费并列，是一种新维度的计费方式。

**典型场景**：用户访问HTTP业务，按访问URL次数收费；彩信收发按次数收费。

**适用NF**：PGW-U、UPF

**支持版本**：UDG 20.5.0及后续版本

**License要求**：`82200CKE LKV3G5EBCS01 支持事件计费`

**发布历史**：
| 版本 | 发布版本 | 说明 |
|------|----------|------|
| 01 | 20.5.0 | 首次发布 |
| 02 | 20.9.0 | 支持ECUR事件计费 |
| 03 | 20.10.0 | 支持BIT419时间戳记录开关 |

## 核心概念

### 事件计费与流量/时长计费的核心差异

| 维度 | 流量计费 | 时长计费 | 事件计费 |
|------|----------|----------|----------|
| 统计量 | 字节数（上下行） | 秒数 | 业务触发次数 |
| URR参数 | OFFMETERINGTYPE=VOLUME / ONLMETERINGTYPE=VOLUME | OFFMETERINGTYPE=DURATION / ONLMETERINGTYPE=DURATION | OFFMETERINGTYPE=EVENT / ONLMETERINGTYPE=EVENT |
| 配额单位 | 字节 | 秒 | 次数（serviceSpecificUnits） |
| 在线计费模式 | SCUR | SCUR | **仅SCUR** |
| 阈值/配额触发 | Volume Quota/Threshold | Time Quota/Threshold | Event Quota/Threshold |

### 事件计费支持的业务类型

UDG支持SCUR（Session Charging with Unit Reservation）配额预留方式的会话计费。无论是流量、时长还是事件计费，统一使用同一个在线计费会话来承载。

| 业务类型 | 说明 | 依赖SA特性 |
|----------|------|------------|
| HTTP业务 | 基于HTTP Get/Post请求次数计费。用户打开一个主页可能触发多个Get/Post请求，每个请求都作为独立事件计费 | SA-Web Browsing |
| WAP业务 | 基于WAP业务访问次数计费 | SA-Mobile |
| MMS业务 | 基于彩信使用次数计费。可区分收/发彩信，可区分单发/群发，群发可按接收者数量统计 | SA-Mobile |

**关键细节**：
- HTTP协议中connect报文用于建立安全隧道（MS/UE->proxy->server），不是真正的业务访问，**不触发事件计费**。
- HTTP下载的断点续传/多线程下载/多源下载，会当作**多个事件**计费。

### 三种计费模式下的事件计费

| 模式 | 对接网元 | 话单生成方 |
|------|----------|------------|
| 融合计费（Nchf/N40） | CHF | CHF |
| 在线计费（Gy） | OCS | OCS |
| 离线计费（Ga） | CG | PGW-C（通过Ga接口发送到CG） |

### 事件计费三种计费点

| 计费点 | 判定条件 | 配额管理 | 配置位置 |
|--------|----------|----------|----------|
| REQUEST | 收到用户业务请求报文即计费 | 直接占用配额 | ADD PCCPOLICYGRP |
| RESPONSE | 收到服务器成功响应报文后计费 | 预占用->正式占用/返还 | ADD PCCPOLICYGRP |
| FINISH | 收到服务器完整响应报文后计费 | 预占用->正式占用/返还 | ADD PCCPOLICYGRP |

**REQUEST与RESPONSE/FINISH的核心差异**：
- REQUEST：直接占用配额，无需预占用/返还机制。
- RESPONSE/FINISH：收到请求时**预占用**配额；识别到成功响应后**正式占用**；未识别到响应则**返还**预占配额。

## 原理与流程

### 融合计费（Nchf/N40）事件计费流程

#### 关键信元

| 信元方向 | 信元路径 | 含义 |
|----------|----------|------|
| CHF下发配额 | Charging Data Response > MultipleUnitInformation > grantedUnit > serviceSpecificUnits | 事件计费配额 |
| CHF下发阈值 | Charging Data Response > MultipleUnitInformation > unitQuotaThreshold | 事件计费阈值 |
| SMF申请配额 | Charging Data Request > MultipleUnitUsage > RequestedUnit > serviceSpecificUnits | 申请事件计费配额 |
| SMF上报用量 | Charging Data Request > MultipleUnitUsage > UsedUnitContainer > serviceSpecificUnits | 上报事件计费用量 |

#### 流程阶段

**阶段一：建立计费会话**

1. 用户发起PDU会话建立请求。
2. PGW-C/SMF向CHF发送Charging Data Request[Initial]，可携带预申请的事件计费配额（serviceSpecificUnits）。
3. CHF返回Charging Data Response[Initial]，携带事件计费配额和阈值。

   **CP/UP使能状态不一致处理**：
   - SMF使能事件计费，UPF不使能：UPF上报告警ALM-81054，按未开启事件计费处理。
   - SMF不使能事件计费，UPF使能：SMF上报告警ALM-81026。
   - SMF使能，CHF未下发事件配额：SMF异常去激活用户。
   - SMF未使能，CHF下发事件配额：SMF忽略。

4. SMF通过PFCP Session Establishment Request下发Create URR信元组：

| 子信元 | 说明 |
|--------|------|
| URR ID | 唯一标识一个URR |
| Measurement Method | EVENT字段=1标识支持事件计费。整个PDU会话过程中某RG是否使能事件计费**不会发生变化** |
| Reporting Triggers | 包含Periodic Reporting、Volume Quota、Event Quota、Volume Threshold、Event Threshold、Linked Usage Reporting等 |
| Event Quota | 事件配额 |
| Event Threshold | 事件阈值 |
| Linked URR ID | 关联URR上报 |

5. UPF返回响应，会话建立成功。

**阶段二：业务触发申请配额**

6. 用户访问业务，UPF识别需事件计费（由ADD URR的ONLMETERINGTYPE和SMF的Measurement Method决定），判断RG无可用事件配额时，向SMF申请新业务配额（Start Trigger）。
7. SMF将本地配置配额+UPF申请配额之和通过serviceSpecificUnits发给CHF。
8. SMF收到CHF下发的配额和阈值，通过PFCP Session Modification下发给UPF。

**阶段三：配额/阈值耗尽，用量上报和配额申请**

9. UPF监控事件计费配额用量，达到阈值或配额耗尽时向SMF上报用量并申请配额。上报信元：

| 子信元 | 说明 |
|--------|------|
| URR ID | 标识上报的URR |
| UR-SEQN | 唯一标识URR的Usage Report |
| EVETH | 事件计费阈值到触发上报 |
| EVEQU | 事件计费配额耗尽触发上报 |
| Event Time Stamp | 事件计费时间戳 |

10. SMF向CHF发起Charging Data Request[Update]，上报用量并申请新配额。
11. SMF收到CHF下发新配额，通过PFCP Session Modification下发给UPF。

**阶段四：计费会话去激活**

12. SMF向UPF发起用户去激活，获取最后的事件计费用量。
13. SMF向CHF发送Charging Data Request Release上报去活用量。

**定时器机制**：达到阈值或配额耗尽时，UPF发送PFCP Session Report Request申请配额并启动定时器（T3RESPONSE * N3REQUEST + 4秒），超时未收到配额更新则**阻塞业务**。

### 在线计费（Gy）事件计费

#### SCUR模式

在线计费SCUR模式的事件计费流程与融合计费一致，区别在于对接网元和信元不同：

| 关键差异点 | 融合计费 | 在线计费 |
|------------|----------|----------|
| 对接网元 | CHF | OCS |
| 下发配额信元 | CCA > MSCC > GSU > CC-Service-Specific-Units | CDR Response > MUI > grantedUnit > serviceSpecificUnits |
| 申请配额信元 | CCR > MSCC > RSU > CC-Service-Specific-Units | CDR Request > MUU > RequestedUnit > serviceSpecificUnits |
| 上报用量信元 | CCR > MSCC > USU > CC-Service-Specific-Units + Event-Charging-TimeStamp | CDR Request > MUU > UsedUnitContainer > serviceSpecificUnits |

#### 标准ECUR模式（仅在线计费支持）

ECUR（Event Charging with Unit Reservation）即配额预留方式的事件计费，由PGW-C实现事件计费会话管理及配额申请/返还/上报，PGW-U不需要感知事件计费模式是SCUR还是ECUR。

**与SCUR的关键区别**：
- SCUR：在传统会话计费中承载事件计费，不单独创建会话。
- ECUR：为事件计费单独建立会话，通过CCR-I/CCR-T管理会话生命周期。

**ECUR流程**：

| 阶段 | 流程 |
|------|------|
| 新业务 | UPF检测到事件 -> PGW-C发送CCR-I建立ECUR会话并申请配额（RSU = EVENTRSUVALUE + UPF请求事件数） -> OCS下发配额（不支持UQT，忽略时长/流量配额） -> 下发给UPF |
| 配额耗尽 | UPF上报配额耗尽 -> PGW-C发送CCR-T终结ECUR会话 -> PGW-C下发START Trigger，允许再次触发 |
| 配额不足 | UPF检测配额不足时上报已用量并申请新配额 -> PGW-C发送CCR-T终结当前ECUR会话 -> 收到CCA-T后发送CCR-I重新建立ECUR会话 -> OCS下发配额 -> 下发给UPF |
| QHT超时 | UPF上报QHT超时 -> PGW-C发送CCR-T终结ECUR会话 |
| VT超时 | PGW-C检测VT超时 -> 向UPF查用量并下发FAR(Drop) -> UPF阻塞业务并删除URR -> PGW-C发送CCR-T终结ECUR会话 -> PGW-C解除阻塞并下发START Trigger |

#### 彩信ECUR事件计费

当彩信事件计费要求每条彩信费用不同时，需为每条彩信单独建立ECUR会话。

**前提条件**：
1. 事件计费方式配置为标准ECUR。
2. PGW-C和PGW-U均开启软参BIT1250。

**彩信ECUR流程特点**：
- 每个被叫号码添加到一个MSCC中，每个MSCC中RSU{CC-Service-Specific-Units=1}。
- 忽略EVENTRSUVALUE，只按UPF请求的事件次数申请配额。
- Service-Information下新增MMS-Information信元组。
- OCS返回配额不等于请求配额时**去活用户**。

**被叫号码规格**：

| 被叫数量 | Service-Context-Id | MSCC方式 |
|----------|-------------------|----------|
| <=20 | mms_ggsn@huawei.com | 每个MSCC一个号码 |
| 21~75 | batchmms_ggsn@huawei.com | 只携带一个MSCC，CC-Service-Specific-Units=被叫用户数 |

- 被叫号码规格最大75个。
- 超规格时PGW-C上报超规格被叫号码个数给OCS，由OCS触发短信提醒。
- OCS处理群发彩信需向MNP查询被叫号码属性（串行查询），处理延迟较长。

### 离线计费（Ga）事件计费

离线计费通过话单将计费信息上报给BS。

**话单要求**：事件计费采用的话单类型必须是**R7以上的eG-CDR或PGW-CDR**。

**事件计费信息位置**：话单业务容器（List of Service Data）中的`eventBasedChargingInformation`字段。

**字段内容**：
- `numberOfEvents`：事件发生次数。
- `eventTimeStamps`：每个事件对应的时间戳列表。

**话单示例**（关键部分）：
```
listOfServiceData = {{
  ratingGroup:1007
  chargingRuleBaseName:amevent-http
  ...
  eventBasedChargingInformation
  {
    numberOfEvents:5
    eventTimeStamps{
      2009-05-31 17:09:27+00:00
      2009-05-31 17:09:28+00:00
      2009-05-31 17:09:36+00:00
      2009-05-31 17:09:36+00:00
      2009-05-31 17:09:42+00:00
    }
  }
}}
```

| GWFD-010173 融合计费 | 融合计费场景必须开启 |
| GWFD-020300 在线计费 | 使用OCS计费时必须开启 |
| GWFD-010171 离线计费 | 使用CG计费时必须开启 |

### 事件计费核心配置命令

#### ADD URR（使用率上报规则）

事件计费的关键参数：

| 参数 | 事件计费取值 | 说明 |
|------|-------------|------|
| USAGERPTMODE | ONLINE / OFFLINE | 在线/离线计费模式 |
| ONLMETERINGTYPE | **EVENT** | 在线计费统计类型设为EVENT |
| OFFMETERINGTYPE | **EVENT** | 离线计费统计类型设为EVENT |
| EVENTRSUVALUE | 数值 | 预申请事件配额值（ECUR模式下使用） |

**配置一致性要求**：URRID、USAGERPTMODE、ONLMETERINGTYPE/OFFMETERINGTYPE在SMF和UPF上的配置必须保持一致。

#### ADD PCCPOLICYGRP（PCC策略组）

事件计费点在此命令中配置：
- 配置事件计费点：REQUEST / RESPONSE / FINISH

#### 配置流程（10步）

```
1. ADD URR          - 配置URR（含EVENT类型）
2. ADD URRGROUP     - 配置URR组（绑定在线/离线URR）
3. ADD PCCPOLICYGRP - 配置PCC策略组（绑定URR组、配置事件计费点）
4. ADD FILTER       - 配置三四层过滤条件
5. ADD L7FILTER     - 配置七层过滤条件
6. ADD FLOWFILTER   - 配置流过滤器
7. ADD FLTBINDFLOWF / ADD PROTBINDFLOWF - 绑定过滤条件到流过滤器
8. ADD RULE         - 配置规则（绑定流过滤器和PCC策略组）
9. ADD USERPROFILE + ADD RULEBINDING - 配置用户模板并绑定规则
10. SET REFRESHSRV  - 刷新使配置生效
```

### CP/UP配置一致性要求

| 维度 | 要求 |
|------|------|
| URR ID | SMF和UPF一致 |
| USAGERPTMODE | SMF和UPF一致 |
| ONLMETERINGTYPE / OFFMETERINGTYPE | SMF和UPF一致 |
| RULENAME | SMF和UPF一致 |
| RULE下的POLICYTYPE、POLICYNAME | SMF和UPF一致 |
| USERPROFILENAME | PCF、SMF、UPF间一致 |
| USERPROFILE下绑定的RULENAME、POLICYTYPE | SMF和UPF一致 |

### 特殊约束

| 约束 | 说明 |
|------|------|
| 仅SCUR模式 | 事件计费只支持SCUR在线计费功能（ECUR仅在线计费额外支持） |
| 仅HTTP1.x | 不支持HTTP2.0协议 |
| 不支持加密场景基于host的计费 | 加密场景下无法解析host进行事件计费 |
| 不支持PCC动态规则 | 不支持对PCC动态规则进行事件计费 |
| 不支持Default Quota | 事件计费不支持Default Quota功能 |
| 免费RG不支持 | 免费RG（FREE）不支持使能事件计费 |
| 配额预占用 | RESPONSE/FINISH计费点时，事件发生即预留配额，成功次数+预留次数>=门限或配额时即申请新配额 |
| RG使能不可变 | 整个PDU会话过程中某RG是否使能事件计费不会发生变化 |

### 配额申请为0的场景

当事件计费的申请配额数为0时，SMF也会将serviceSpecificUnits带给CHF/OCS，以向CHF/OCS表明**该RG需要进行事件计费**。

## 配置案例

### 案例：HTTP URL事件计费配置

以下配置实现对访问www.huawei.com的HTTP业务进行事件计费（以流量计费为主，事件计费为辅的场景参考部署文档，事件计费需将ONLMETERINGTYPE设为EVENT）。

**事件计费URR配置示例**：

```mml
// 离线事件计费URR
ADD URR:URRNAME="urr_event_offline", URRID=22000, USAGERPTMODE=OFFLINE, OFFMETERINGTYPE=EVENT;

// 在线事件计费URR
ADD URR:URRNAME="urr_event_online", URRID=2400, USAGERPTMODE=ONLINE, ONLMETERINGTYPE=EVENT;

// URR组
ADD URRGROUP:URRGROUPNAME="UrrGp_event",
  UPURRNAME1="urr_event_offline", DOWNURRNAME1="urr_event_offline",
  UPURRNAME2="urr_event_online", DOWNURRNAME2="urr_event_online";

// PCC策略组（配置事件计费点，如REQUEST/RESPONSE/FINISH）
ADD PCCPOLICYGRP:PCCPOLICYGRPNM="PPG_event", URRGROUPNAME="UrrGp_event";

// 七层过滤器（URL匹配）
ADD L7FILTER:L7FILTERNAME="L7_event_url", SUBL7FLTNAME="http", URLTYPE=URL, URL="www.huawei.com";

// 流过滤器
ADD FLOWFILTER:FLOWFILTERNAME="ff_event";
ADD FLTBINDFLOWF:FLOWFILTERNAME="ff_event", FILTERNAME="L34_any";
ADD PROTBINDFLOWF:FLOWFILTERNAME="ff_event", PROTOCOLNAME="http", L7FILTERNAME="L7_event_url";

// 规则
ADD RULE:RULENAME="Rule_event", POLICYTYPE=PCC, FILTERINGMODE=FLOWFILTER,
  FLOWFILTERNAME="ff_event", POLICYNAME="PPG_event", PRIORITY=100;

// 用户模板与绑定
ADD USERPROFILE:USERPROFILENAME="userprofile_event";
ADD RULEBINDING:USERPROFILENAME="userprofile_event", RULENAME="Rule_event", POLICYTYPE=PCC;

// 刷新生效
SET REFRESHSRV:REFRESHTYPE=ALL;
```

## 验证与调测

### 离线计费验证

查看话单中`eventBasedChargingInformation`字段：
- 确认`numberOfEvents`值正确。
- 确认`eventTimeStamps`与实际业务访问时间一致。
- 确认话单类型为R7以上的eG-CDR或PGW-CDR。

### 在线/融合计费验证

1. 检查PFCP Session Report中Event Time Stamp信元是否正确。
2. 检查EVETH（阈值触发）和EVEQU（配额耗尽触发）是否正常上报。
3. 确认配额申请和下发流程正常。

### CP/UP一致性检查

- 检查告警ALM-81054（CP和UP关键配置不一致）。
- 检查告警ALM-81026（接口信元不一致）。

### 特性规格

| 规格名称 | 规格指标 |
|----------|----------|
| 离线计费用户每个Usage Report信元支持的事件计费时间戳个数 | 100 |

### 重要说明

- 事件计费支持和时长、流量计费**同时进行**，先满足的计费Trigger会上报所有维度配额的用量。
- PFCP定时器超时时间 = T3RESPONSE * N3REQUEST + 4秒（由SET UPPFCPPATH命令配置）。
- 融合计费场景下，当SMF使用DEFAULT RG策略时，URRGROUP下不能同时配置RG相同的离线和在线URR。
- 本特性只支持HTTP1.x，不支持HTTP2.0；不支持加密场景基于host的计费和控制；不支持PCC动态规则事件计费。
