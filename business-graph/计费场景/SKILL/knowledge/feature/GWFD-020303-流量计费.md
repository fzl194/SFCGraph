# GWFD-020303 基于业务流量的计费 知识文档

## 概述

### 特性定义

基于业务流量的计费是在业务识别解析的基础上，按照访问业务的**流量（字节数）**进行计费。UDG 负责统计不同业务的上下行流量访问量，并上报给控制面（UNC/SMF），由计费系统根据流量数据向终端用户收费。

### 适用NF

- **用户面**：SGW-U、PGW-U、UPF（UDG 侧）
- **控制面**：SGW-C、PGW-C、SMF（UNC 侧）
- **计费系统**：CG、OCS、CHF

### 版本信息

| 涉及NF | 支持版本 | 功能说明 |
|--------|----------|----------|
| SGW-C/PGW-C/SMF | UNC 20.0.0及后续版本 | 将UDG的业务统计结果上报给计费系统 |
| SGW-U/PGW-U/UPF | UDG 20.0.0及后续版本 | 对用户业务进行识别和解析，统计不同业务的访问量并上报UNC |
| CG | 无特殊要求 | 对在线计费用户进行鉴权、下发流量配额，基于业务类型对用户进行信用控制 |
| OCS | 无特殊要求 | 合并用户离线计费话单，发送给计费中心 |
| CHF | 无特殊要求 | 离线计费：生成话单；在线计费：鉴权、下发配额、信用控制 |

特性首次发布版本为 20.0.0。

### License

License 控制项：**82209824 LKV3G5VBCS01 基于业务流量的计费**。必须获取 License 许可后才能使用本特性。

### 与内容计费的关系

基于业务流量的计费是**内容计费基本功能（GWFD-020301）的增强形式**。使用基于业务流量的计费前，需要先开启内容计费基本功能（License：82209822）。同时，内容计费需要先识别出业务类型，因此还依赖 **SA-Basic（GWFD-110101）** 特性（License：82209749）。

**依赖链**：SA-Basic（业务识别） -> 内容计费基本功能 -> 基于业务流量的计费

## 核心概念

### 流量计费 vs 时长计费 vs 事件计费

三种计费方式的核心区别在于 URR 的 METERINGTYPE 参数取值：

| 计费方式 | METERINGTYPE 参数 | 统计对象 | 参数位置 |
|----------|-------------------|----------|----------|
| **流量计费** | `VOLUME` | 业务上下行字节数（datavolumeFBCUplink/Downlink） | 离线：OFFMETERINGTYPE=VOLUME；在线：ONLMETERINGTYPE=VOLUME |
| **时长计费** | `DURATION` | 业务持续时间（秒） | 离线：OFFMETERINGTYPE=DURATION；在线：ONLMETERINGTYPE=DURATION |
| **免费** | `FREE` | 不统计（不计费） | 仅用于兜底/异常场景 |

### METERINGTYPE=VOLUME 的含义

当 URR 的 OFFMETERINGTYPE 或 ONLMETERINGTYPE 设置为 `VOLUME` 时，UDG 按照匹配到的业务报文的**字节数**进行累计统计，区分上行流量和下行流量：

- **上行流量（Uplink）**：终端发送方向
- **下行流量（Downlink）**：终端接收方向
- **总流量（Total）**：上行 + 下行

### 业务流量统计机制

UDG 支持统计业务的上下行流量。统计粒度为**每个费率组（RG）或费率组+业务ID（RG+SID）组合**，即每组业务独立统计流量。

## 原理与流程

### 离线计费流量统计原理

1. UDG 识别业务类型并按 RG/RG+SID 组合累计上下行流量。
2. UNC 在话单中通过**业务容器（List of Service Data）**记录：
   - `datavolumeFBCUplink`：业务上行流量
   - `datavolumeFBCDownlink`：业务下行流量
3. CG 合并用户离线计费话单，发送给计费中心，最终生成基于业务流量的计费话单。
4. 是否根据流量向终端用户收费由**计费中心**决定。

### 在线计费流量统计原理

1. OCS 在 CCA 消息中通过信用控制实例的 **MSCC > GSU > CC-Total-Octets AVP** 下发流量配额。
2. UDG 统计业务消耗的流量，UNC 通过 CCR 消息上报：
   - **MSCC > USU > CC-Input-Octets AVP**：业务消耗的上行流量
   - **MSCC > USU > CC-Output-Octets AVP**：业务消耗的下行流量
   - **MSCC > USU > CC-Total-Octets AVP**：业务消耗的总流量
3. 每组业务通过 RG 或 RG+SID 组合标识。
4. 是否根据流量向终端用户收费由 **OCS** 决定。

### 融合计费流量统计原理

**带配额管理（在线计费）**：
- CHF 下发业务的流量配额：`Charging Data Request > MultipleUnitInformation > grantedUnit > totalVolume/uplinkVolume/downlinkVolume`
- SMF 上报业务的流量用量：`Charging Data Request > MultipleUnitUsage > UsedUnitContainer > totalVolume/uplinkVolume/downlinkVolume`

**不带配额管理（离线计费）**：
- SMF 通过 `Charging Data Request > MultipleUnitUsage > UsedUnitContainer > totalVolume/uplinkVolume/downlinkVolume` 将业务的计费信息上报给计费系统。

### 流量配额管理与上报机制

- **在线计费和融合计费带配额管理**时，OCS/CHF 下发流量配额，UDG 按配额扣减，配额用完前主动上报用量请求新配额。
- **离线计费和融合计费不带配额管理**时，没有配额下发机制，话单中直接记录流量数据。
- 话单中默认会记录用户级流量，是否根据流量向终端用户收费由计费中心/OCS决定。

## 配置规则

### 配置前提

1. 已完成加载 License（82209824 基于业务流量的计费、82209822 内容计费基本功能、82209749 SA-Basic）。
2. 已阅读融合计费原理。
3. PCF、SMF、UPF 网元上统一进行配置，保证网元间配置一致。

### SMF/UPF 配置一致性硬规则

以下参数在 SMF 和 UPF 上的配置**必须保持一致**，不一致将导致计费异常：

| 参数 | 命令 | 说明 |
|------|------|------|
| URRID | ADD URR | URR 标识必须一致 |
| USAGERPTMODE | ADD URR | 使用量上报模式（OFFLINE/ONLINE）必须一致 |
| OFFMETERINGTYPE | ADD URR | 离线计费统计类型必须一致 |
| ONLMETERINGTYPE | ADD URR | 在线计费统计类型必须一致 |
| RULENAME | ADD RULE | 规则名称必须一致 |
| POLICYTYPE | ADD RULE | 策略类型必须一致 |
| POLICYNAME | ADD RULE | 策略名称（即PCC策略组名）必须一致 |
| USERPROFILENAME | ADD USERPROFILE | 用户模板名称必须在PCF/SMF/UPF间一致 |
| RULEBINDING 中的 RULENAME/POLICYTYPE | ADD RULEBINDING | 绑定的规则名和策略类型必须一致 |

### 配置步骤（12步流程）

```
步骤1: ADD URR        -- 配置使用量上报规则（离线/在线各一条，METERINGTYPE=VOLUME）
步骤2: ADD URRGROUP   -- 配置URR组（绑定离线URR和在线URR）
步骤3: ADD PCCPOLICYGRP -- 配置PCC策略组（绑定URR组）
步骤4: ADD FILTER / ADD FILTERIPV6 -- 配置三四层过滤条件
步骤5: ADD L7FILTER   -- 配置七层过滤条件（可选）
步骤6: ADD FLOWFILTER -- 配置流过滤器
步骤7: ADD FLTBINDFLOWF -- 绑定三四层过滤条件到流过滤器
步骤8: ADD PROTBINDFLOWF -- 绑定七层过滤条件到流过滤器（可选）
步骤9: ADD RULE       -- 配置规则（绑定流过滤器和PCC策略组）
步骤10: ADD USERPROFILE -- 配置用户模板
步骤11: ADD RULEBINDING -- 将规则绑定到用户模板
步骤12: SET URRGRPBINDING -- 配置缺省URR组
步骤13: SET SPECTRAFURRGRP -- 配置全局缺省URR组
步骤14: ADD SPECURRGRPLIST -- 配置防欺诈URR列表（可选）
步骤15: SET REFRESHSRV -- 刷新Filter使配置生效
```

### 流量计费URR关键参数

**ADD URR 命令中流量计费的关键参数**：

| 参数 | 流量计费取值 | 说明 |
|------|-------------|------|
| URRNAME | 自定义命名 | 使用量上报规则名称 |
| URRID | 全网规划整数 | URR 标识，SMF/UPF 必须一致 |
| USAGERPTMODE | OFFLINE 或 ONLINE | OFFLINE=离线计费，ONLINE=在线计费 |
| OFFMETERINGTYPE | **VOLUME** | 离线计费统计类型，VOLUME=按流量计费 |
| ONLMETERINGTYPE | **VOLUME** | 在线计费统计类型，VOLUME=按流量计费 |

### URRGROUP 关键约束

- **URRGROUP 下必须同时配置在线 URR 和离线 URR**。只配置一种计费模式且计费模式和用户的计费模式不匹配时无法计费。
- UPURRNAME1/DOWNURRNAME1：第1组 URR（通常为离线），上下行分别指定。
- UPURRNAME2/DOWNURRNAME2：第2组 URR（通常为在线），上下行分别指定。

**融合计费特殊约束**：当 SMF 使用融合计费且 `SET USRPROFCHARGE`/`SET APNCHARGECTRL`/`ADD CHARGEMETHOD` 命令的 RGAPPLIED 配置为 DEFAULT 时，**URRGROUP 下不能同时配置离线和在线 URR**，否则会导致计费冲突。

### 缺省URR组与全局缺省URR组

- **SET URRGRPBINDING**：为 USERPROFILE 配置缺省业务 URR 组（DFTURRGRPNAME）和缺省信令 URR 组（DFTSIGURRGNAME）。不配置时，按照 RULE 没有匹配到 URR 的报文无法计费。
- **SET SPECTRAFURRGRP**：配置全局缺省 URR 组。当 SET URRGRPBINDING 中的缺省 URR 组未配置时，可以继承全局缺省 URR 组配置。
- GLBDFTURRGSW=ENABLE 和 GLBDFTSIGURRGSW=ENABLE 表示继承全局缺省 URR 组配置开关打开。

### 特殊流量处理

特殊流量是指当某些业务报文丢失，未能完成七层精确匹配时，已经转发的流量。例如在线计费用户下线时，业务流只有信令报文没有业务报文，已经转发的信令报文的流量称为特殊流量。

- 通过 `SET SPECTRAFURRGRP` 配置全局缺省 URR 组来处理特殊流量。
- 当 BIT1232 软参值设置为 1 时，特殊流量通过该规则组进行计费。
- 对于没有配置任何 URR 的业务场景，也通过该 URR 组进行计费。

### SET REFRESHSRV 硬规则

**不执行 SET REFRESHSRV 命令，FILTER 的配置变更不会生效。** 必须在所有 Filter 配置完成后执行 `SET REFRESHSRV: REFRESHTYPE=ALL`。

### 防欺诈功能

防欺诈功能开启时，RULE 下绑定 URL 类型的七层 filter 时**必须同时绑定三四层 filter**。防欺诈未开启时，RULE 下可仅绑定 URL 类型的七层 filter。

通过 `ADD SPECURRGRPLIST` 将有欺诈风险的费率组加入防欺诈费率列表。

### 规则优先级规则

- 数字越小优先级越高。
- **只绑定三四层 filter 的 RULE 的优先级应该低于绑定七层 filter 的 RULE 的优先级**。
- 过滤条件为 any 的 RULE，优先级设置为最低（如 65000）。

### FLOWFILTER 硬规则

FLOWFILTER **必须绑定过滤条件**，否则该 FLOWFILTER 匹配失败，对应的 Rule 为无效配置。

## 配置案例

### 场景总览

配置 User Profile（userprofile001），绑定三个不同的规则，再加一个异常场景规则：

| 场景 | 业务 | 过滤方式 | 规则名 | 优先级 | PCC策略组 |
|------|------|----------|--------|--------|-----------|
| 场景1 | www.huawei.com 网站业务 | 三四层 + 七层(URL) | Rule001_url | 100 | PPG_URL |
| 场景2 | IMS 语音业务 | 三四层(IPv6) | Rule002_IMS | 110 | PPG_IMS |
| 场景3 | 其他基础业务(any) | 三四层(any) | Rule003_any / Rule003_any_IPv6 | 65000/65010 | PPG_any |
| 场景4 | 未匹配异常流量 | 不匹配 | 兜底（通过DFTURRGRPNAME和DFTSIGURRGNAME） | - | PPG_any（SIGURRGRPNAME=UrrGp_abnormal） |

### 完整MML脚本

```mml
// ========== 步骤1: 配置URR（流量计费OFFMETERINGTYPE/ONLMETERINGTYPE=VOLUME） ==========

// 场景1: www.huawei.com 网站业务
ADD URR:URRNAME="urr_offline_URL", URRID=12500, USAGERPTMODE=OFFLINE, OFFMETERINGTYPE=VOLUME;
ADD URR:URRNAME="urr_online_URL", URRID=2300, USAGERPTMODE=ONLINE, ONLMETERINGTYPE=VOLUME;

// 场景2: IMS 语音业务
ADD URR:URRNAME="urr_offline_IMS", URRID=13500, USAGERPTMODE=OFFLINE, OFFMETERINGTYPE=VOLUME;
ADD URR:URRNAME="urr_online_IMS", URRID=3300, USAGERPTMODE=ONLINE, ONLMETERINGTYPE=VOLUME;

// 场景3: 基础业务(any)
ADD URR:URRNAME="urr_offline_any", URRID=11500, USAGERPTMODE=OFFLINE, OFFMETERINGTYPE=VOLUME;
ADD URR:URRNAME="urr_online_any", URRID=1300, USAGERPTMODE=ONLINE, ONLMETERINGTYPE=VOLUME;

// 场景4: 异常场景（不计费，METERINGTYPE=FREE）
ADD URR:URRNAME="URR_abnormal_offline", URRID=14500, USAGERPTMODE=OFFLINE, OFFMETERINGTYPE=FREE;
ADD URR:URRNAME="URR_abnormal_online", URRID=4300, USAGERPTMODE=ONLINE, ONLMETERINGTYPE=FREE;

// ========== 步骤2: 配置URRGROUP（每组绑定离线+在线URR） ==========

ADD URRGROUP:URRGROUPNAME="UrrGp_URL", UPURRNAME1="urr_offline_URL", DOWNURRNAME1="urr_offline_URL", UPURRNAME2="urr_online_URL", DOWNURRNAME2="urr_online_URL";
ADD URRGROUP:URRGROUPNAME="UrrGp_IMS", UPURRNAME1="urr_offline_IMS", DOWNURRNAME1="urr_offline_IMS", UPURRNAME2="urr_online_IMS", DOWNURRNAME2="urr_online_IMS";
ADD URRGROUP:URRGROUPNAME="UrrGp_any", UPURRNAME1="urr_offline_any", DOWNURRNAME1="urr_offline_any", UPURRNAME2="urr_online_any", DOWNURRNAME2="urr_online_any";
ADD URRGROUP:URRGROUPNAME="UrrGp_abnormal", UPURRNAME1="URR_abnormal_offline", DOWNURRNAME1="URR_abnormal_offline", UPURRNAME2="URR_abnormal_online", DOWNURRNAME2="URR_abnormal_online";

// ========== 步骤3: 配置PCC业务策略组 ==========

ADD PCCPOLICYGRP:PCCPOLICYGRPNM="PPG_URL", URRGROUPNAME="UrrGp_URL";
ADD PCCPOLICYGRP:PCCPOLICYGRPNM="PPG_IMS", URRGROUPNAME="UrrGp_IMS";
// 基础业务策略组，同时绑定信令URR组用于异常场景
ADD PCCPOLICYGRP:PCCPOLICYGRPNM="PPG_any", URRGROUPNAME="UrrGp_any", SIGURRGRPNAME="UrrGp_abnormal";

// ========== 步骤4-8: 配置过滤规则 ==========

// 场景1: 三四层 + 七层URL过滤器
ADD FILTER: FILTERNAME="L34_any", L34PROTTYPE=STRING, L34PROTOCOL=ANY, MSSTARTPORT=0, MSENDPORT=65535, SVRSTARTPORT=0, SVRENDPORT=65535;
ADD L7FILTER: L7FILTERNAME="L7_URL", SUBL7FLTNAME="http", URLTYPE=URL, URL="www.huawei.com";
ADD FLOWFILTER: FLOWFILTERNAME="ff_L7";
ADD FLTBINDFLOWF: FLOWFILTERNAME="ff_L7", FILTERNAME="L34_any";
ADD PROTBINDFLOWF: FLOWFILTERNAME="ff_L7", PROTOCOLNAME="http", L7FILTERNAME="L7_URL";

// 场景2: IPv6三四层过滤器（IMS语音，指定服务器IPv6地址）
ADD FILTERIPV6:FILTERNAME="ipv6_ims", L34PROTTYPEV6=STRING, L34PROTOCOL=ANY, SVRIPMODE=IP, MSSTARTPORT=0, MSENDPORT=65535, SVRSTARTPORT=0, SVRENDPORT=65535, SVRIPV6="fc00:8028:5002:10D::181", SVRIPV6MASKTYPE=LENGTHTYPE, SVRIPV6MASKLEN=128;
ADD FLOWFILTER: FLOWFILTERNAME="ff_IMS";
ADD FLTBINDFLOWF: FLOWFILTERNAME="ff_IMS", FILTERNAME="ipv6_ims";

// 场景3: any基础业务（IPv4 + IPv6各一套）
ADD FILTER: FILTERNAME="L34_any", L34PROTTYPE=STRING, L34PROTOCOL=ANY, MSSTARTPORT=0, MSENDPORT=65535, SVRSTARTPORT=0, SVRENDPORT=65535;
ADD FILTERIPV6:FILTERNAME="L34_any_ipv6", L34PROTTYPEV6=STRING, L34PROTOCOL=ANY, MSSTARTPORT=0, MSENDPORT=65535, SVRSTARTPORT=0, SVRENDPORT=65535;
ADD FLOWFILTER: FLOWFILTERNAME="ff_L34any";
ADD FLOWFILTER: FLOWFILTERNAME="ff_L34any_IPv6";
ADD FLTBINDFLOWF: FLOWFILTERNAME="ff_L34any", FILTERNAME="L34_any";
ADD FLTBINDFLOWF: FLOWFILTERNAME="ff_L34any_IPv6", FILTERNAME="L34_any_ipv6";

// ========== 步骤9: 配置规则 ==========

ADD RULE: RULENAME="Rule001_url", POLICYTYPE=PCC, FILTERINGMODE=FLOWFILTER, FLOWFILTERNAME="ff_L7", POLICYNAME="PPG_URL", PRIORITY=100;
ADD RULE: RULENAME="Rule002_IMS", POLICYTYPE=PCC, FILTERINGMODE=FLOWFILTER, FLOWFILTERNAME="ff_IMS", POLICYNAME="PPG_IMS", PRIORITY=110;
ADD RULE: RULENAME="Rule003_any", POLICYTYPE=PCC, FILTERINGMODE=FLOWFILTER, FLOWFILTERNAME="ff_L34any", POLICYNAME="PPG_any", PRIORITY=65000;
ADD RULE: RULENAME="Rule003_any_IPv6", POLICYTYPE=PCC, FILTERINGMODE=FLOWFILTER, FLOWFILTERNAME="ff_L34any_IPv6", POLICYNAME="PPG_any", PRIORITY=65010;

// ========== 步骤10-11: 配置User Profile和规则绑定 ==========

ADD USERPROFILE: USERPROFILENAME="userprofile001";
ADD USERPROFILE: USERPROFILENAME="userprofile002";
ADD RULEBINDING: USERPROFILENAME="userprofile001", RULENAME="Rule001_url", POLICYTYPE=PCC;
ADD RULEBINDING: USERPROFILENAME="userprofile001", RULENAME="Rule003_any", POLICYTYPE=PCC;
ADD RULEBINDING: USERPROFILENAME="userprofile001", RULENAME="Rule003_any_IPv6", POLICYTYPE=PCC;
ADD RULEBINDING: USERPROFILENAME="userprofile002", RULENAME="Rule002_IMS";

// ========== 步骤12-13: 配置缺省URR组和全局缺省URR组 ==========

SET URRGRPBINDING:USERPROFILENAME="userprofile001", DFTURRGRPNAME="UrrGp_any", DFTSIGURRGNAME="UrrGp_abnormal", GLBDFTURRGSW=ENABLE, GLBDFTSIGURRGSW=ENABLE;
SET SPECTRAFURRGRP: URRGROUPNAME="UrrGp_abnormal";

// ========== 步骤14: 配置防欺诈URR列表（可选） ==========

ADD SPECURRGRPLIST:URRGROUPNAME="UrrGp_URL";

// ========== 步骤15: 刷新Filter使配置生效 ==========

SET REFRESHSRV: REFRESHTYPE=ALL;
```

### 配置三件套总结

每个业务场景的流量计费配置遵循"三件套"模式：

| 层级 | 命令 | 流量计费关键参数 |
|------|------|-----------------|
| URR | ADD URR | OFFMETERINGTYPE=VOLUME 或 ONLMETERINGTYPE=VOLUME |
| URRGROUP | ADD URRGROUP | UPURRNAME1/DOWNURRNAME1（离线）+ UPURRNAME2/DOWNURRNAME2（在线） |
| PCC策略组 | ADD PCCPOLICYGRP | URRGROUPNAME 绑定 URR 组 |

## 验证与调测

### 验证方法

文档未提供独立的验证章节。结合配置实践，验证要点包括：

1. **配置一致性检查**：确认 SMF 和 UPF 上的 URRID、USAGERPTMODE、OFFMETERINGTYPE/ONLMETERINGTYPE 参数完全一致。
2. **规则生效验证**：确认执行了 `SET REFRESHSRV: REFRESHTYPE=ALL`，否则 Filter 配置不会生效。
3. **话单/流量验证**：
   - 离线计费：检查话单中 datavolumeFBCUplink 和 datavolumeFBCDownlink 字段是否有值。
   - 在线计费：检查 CCR/CCA 消息中 CC-Total-Octets/CC-Input-Octets/CC-Output-Octets AVP 是否正确。
   - 融合计费：检查 Charging Data Request 中 totalVolume/uplinkVolume/downlinkVolume 信元。
4. **缺省URR组验证**：确认 USERPROFILE 的 DFTURRGRPNAME 和 DFTSIGURRGNAME 已配置，未匹配的报文能走缺省计费。

### 常见问题

| 问题 | 可能原因 | 解决方案 |
|------|----------|----------|
| 无法计费 | URRGROUP 只配了一种计费模式（离线或在线），与用户计费模式不匹配 | URRGROUP 下必须同时配置在线和离线 URR |
| 计费冲突 | 融合计费场景下 URRGROUP 同时配了离线和在线 URR，且 RGAPPLIED=DEFAULT | 融合计费+RGAPPLIED=DEFAULT 时 URRGROUP 不能同时配离线和在线 URR |
| Filter 不生效 | 未执行 SET REFRESHSRV | 配置完成后必须执行 SET REFRESHSRV: REFRESHTYPE=ALL |
| RULE 无效 | FLOWFILTER 未绑定过滤条件 | FLOWFILTER 必须通过 FLTBINDFLOWF 绑定过滤条件 |
| PCF 策略无法生效 | USERPROFILE 名称在 PCF/SMF/UPF 间不一致 | 统一规划 USERPROFILE 名称，保证三端一致 |
| 部分报文无法计费 | USERPROFILE 未配置缺省 URR 组 | 配置 SET URRGRPBINDING 和 SET SPECTRAFURRGRP |
| URL 类七层规则欺诈 | 防欺诈开启时只绑了七层 filter 未绑三四层 | 防欺诈开启时 RULE 必须同时绑定三四层和七层 filter |
