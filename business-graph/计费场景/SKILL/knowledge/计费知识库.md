## 1. 计费特性体系与依赖

计费能力是一个分层特性栈，Agent 在配置前应确认 License 和前置特性已开启。

```
SA-Basic (GWFD-110101, License: LKV3G5SABS01)
  └→ 内容计费基本功能 (GWFD-020301, License: LKV3G5BCBC01)
       ├→ 基于业务流量的计费 (GWFD-020303, License: LKV3G5VBCS01)
       ├→ 基于业务时长的计费 (GWFD-020302, License: LKV3G5TBCS01)
       └→ 支持事件计费 (GWFD-020306, License: LKV3G5EBCS01)

离线计费 (GWFD-010171) / 在线计费 (GWFD-020300) / 融合计费 (GWFD-010173) 三选一或组合

PCC基本功能 (GWFD-020351, License: LKV3G5PCCB01) — 独立于内容计费，但共用RULE框架
```

**Agent 规则**：生成计费配置前，确认 `LST LICENSESWITCH` 中以下开关已 ENABLE：
- `LKV3G5SABS01` (SA-Basic)
- `LKV3G5BCBC01` (内容计费)
- `LKV3G5PCCB01` (PCC基本功能) — 如果使用静态规则

(→ feature/UDG/UPF/GWFD-020301/GWFD-020301 内容计费基本功能特性概述_66863837.md)

## 2. 计费配置对象体系

> **命令变体说明**：本知识库聚焦 ADD（新建）场景的参数指导。
> 现网修改用 SET/MOD，删除用 RMV，查询用 LST/DSP。
> 各命令的完整参数约束见 `SKILL/command/{UDG,UNC}/` 目录。

### 2.1 完整依赖链（从底到顶）

```
IPLIST ──────────────┐
                      ├→ FILTER ──→ FLTBINDFLOWF ──→ FLOWFILTER ──→ RULE
L7FILTER ──→ PROTBINDFLOWF ──────────────────────┘         │
                                                            │
URR ──→ URRGROUP ──→ PCCPOLICYGRP ─────────────────────────┘
                                                           │
USERPROFILE ←── RULEBINDING(绑定RULE) ←────────────────────┘
           ←── SET URRGRPBINDING(绑定默认计费组)
           ──→ SET REFRESHSRV(刷新生效)
```

### 2.2 对象分类：共享 vs 每业务独立

| 分类 | 对象 | 说明 |
|------|------|------|
| 底层共享 | IPLIST, FILTER, L7FILTER | 可被多个 FLOWFILTER 引用 |
| 中间共享 | FLOWFILTER, FLOWFILTERGRP, FLTBINDFLOWF, PROTBINDFLOWF | 可被多个 RULE 引用 |
| **每业务独立** | **URR, URRGROUP, PCCPOLICYGRP** | **每条业务需一套独立的三件套** |
| 汇聚层 | RULE | 每条业务一个（或一组 OR 条件一个） |
| 顶层共享 | USERPROFILE | 一个套餐一个 |

### 2.3 对象生命周期

| 动作 | 命令 | 场景 |
|------|------|------|
| 新建 | ADD | 对象不存在时 |
| 修改 | SET / MOD | 对象存在但参数需变更时 |
| 删除 | RMV | 先解绑上层，再删对象 |
| 查询 | LST / DSP | 验证配置 |

(→ SKILL/业务图谱.md 11.3节 T-EXEC-001~010)

## 3. 计费动作链详解

计费场景的核心是三层嵌套：**URR → URRGROUP → PCCPOLICYGRP**。

### 3.1 URR（使用量上报规则）

定义"如何计费"的基本单元。

**必填参数**：
| 参数 | 含义 | 取值 | Agent规则 |
|------|------|------|----------|
| URRNAME | URR名称 | 字符串1-31 | 建议与业务关联命名，如 `urr_carrier_l7` |
| URRID | URR标识 | 整数0-2147483646 | **所有URR内唯一**，建议从1000开始分配 |
| USAGERPTMODE | 上报模式 | OFFLINE / ONLINE / MONITORINGKEY / QOS | 计费场景选 OFFLINE 或 ONLINE |

**关键选填参数**：
| 参数 | 含义 | 前提 | 取值 |
|------|------|------|------|
| OFFMETERINGTYPE | 离线计费统计类型 | USAGERPTMODE=OFFLINE | VOLUME / TIME / EVENT / **FREE** / VOLUME_TIME / EVENT_VOLUME / EVENT_TIME / EVENT_VOLUME_TIME |
| ONLMETERINGTYPE | 在线计费统计类型 | USAGERPTMODE=ONLINE | 同上 |
| RG | 计费组(Rating Group) | OFFLINE 或 ONLINE | 整数0-4294967294，**需与SMF侧一致** |
| SID | 业务标识 | OFFLINE 或 ONLINE | 整数0-4294967294 |
| DFTQUOTAVOLUME | 默认流量配额 | ONLINE | 在线计费时配额耗尽前使用 |

**Agent 决策规则**：
- 用户说"离线计费" → `USAGERPTMODE=OFFLINE`
- 用户说"在线计费" → `USAGERPTMODE=ONLINE`
- 用户说"按流量计费" → `OFFMETERINGTYPE=VOLUME`（默认值，可省略）
- 用户说"按时长计费" → `OFFMETERINGTYPE=TIME`
- 用户说"免费业务" → `OFFMETERINGTYPE=FREE`
- 用户说"RG100" → `RG=100`
- **URRID 与 RG 独立**，但运维建议保持一致映射关系
- **RG 值需与 UNC/SMF 侧计费配置一致**

(→ command/UDG/ADD URR.md)

### 3.2 URRGROUP（使用量上报规则组）

组合上下行 URR，指定哪个 URR 处理上行报文、哪个处理下行报文。

**必填参数**：URRGROUPNAME

**关键选填参数**：UPURRNAME1-3, DOWNURRNAME1-3（至少设置一个）

> **编号含义**：UPURRNAME1/2/3 和 DOWNURRNAME1/2/3 只是编号，不代表优先级。
> 当同一方向配多个 URR 时（如在线+离线），它们同时生效。

**模式1：简单计费**（最常见）— 上下行用同一个 URR
```
ADD URRGROUP:URRGROUPNAME="urrg_carrier",UPURRNAME1="urr_carrier",DOWNURRNAME1="urr_carrier";
```

**模式2：融合计费** — 同一业务同时在线+离线
```
ADD URRGROUP:URRGROUPNAME="urrg_conv",UPURRNAME1="urr_offline",UPURRNAME2="urr_online",
  DOWNURRNAME1="urr_offline",DOWNURRNAME2="urr_online";
```

**模式3：上下行差异化** — 上行免费、下行计费
```
ADD URR:URRNAME="urr_up_free",URRID=2000,USAGERPTMODE=OFFLINE,OFFMETERINGTYPE=FREE;
ADD URR:URRNAME="urr_down_vol",URRID=2001,USAGERPTMODE=OFFLINE,OFFMETERINGTYPE=VOLUME,RG=500;
ADD URRGROUP:URRGROUPNAME="urrg_asym",UPURRNAME1="urr_up_free",DOWNURRNAME1="urr_down_vol";
```

**Agent 规则**：简单计费场景默认上下行使用同一个 URR。只有用户明确要求以下场景时才用不同 URR：
- 上下行差异化计费（模式3）
- 融合计费同时需要在线+离线（模式2）

(→ command/UDG/ADD URRGROUP.md)

### 3.3 PCCPOLICYGRP（PCC策略组）

将计费链封装为 RULE 可引用的策略组。

**必填参数**：PCCPOLICYGRPNM

**关键选填参数**：
| 参数 | 含义 | 场景 |
|------|------|------|
| URRGROUPNAME | 计费属性 | **计费场景必填** |
| PCCACTPROPNAME | PCC动作属性 | 需要门控/阻塞时填 |
| SIGNALASSOCIATE | 信令关联计费 | 默认ENABLE |

**计费模式**：
```
ADD PCCPOLICYGRP:PCCPOLICYGRPNM="pccg_carrier",URRGROUPNAME="urrg_carrier";
```

**计费+门控模式**：
```
ADD PCCPOLICYGRP:PCCPOLICYGRPNM="pccg_xxx",URRGROUPNAME="urrg_xxx",PCCACTPROPNAME="pccact_xxx";
```

**Agent 规则**：纯计费场景只填 URRGROUPNAME。只有同时需要门控/阻塞时才填 PCCACTPROPNAME。

(→ command/UDG/ADD PCCPOLICYGRP.md)

### 3.4 三件套映射表

每条业务需要一个完整的三件套，互不共享：

| 业务 | URR | URRGROUP | PCCPOLICYGRP |
|------|-----|----------|-------------|
| 业务1(华为网站) | urrA, ID=1000, RG=100 | urrgA(urrA↑↓) | pccgA(urrgA) |
| 业务2(免费RTSP) | urrB, ID=2000, FREE | urrgB(urrB↑↓) | pccgB(urrgB) |
| 兜底业务 | urrD, ID=3000, RG=300 | urrgD(urrD↑↓) | pccgD(urrgD) |

## 4. RULE 配置

RULE 是匹配条件（FLOWFILTER）与策略动作（PCCPOLICYGRP）的汇聚点。

**计费场景必填参数**：
| 参数 | 含义 | 取值 |
|------|------|------|
| RULENAME | 规则名称 | 字符串1-63 |
| POLICYTYPE | 策略类型 | **PCC**（计费场景固定） |
| FILTERINGMODE | 过滤模式 | FLOWFILTER 或 FLOWFILTERGRP |
| FLOWFILTERNAME / FLWFLTRGRPNAME | 过滤条件 | 条件必选 |
| PRIORITY | 优先级 | 整数1-4294967295，**越小越高** |
| POLICYNAME | 策略名称 | PCCPOLICYGRP 的名称 |

**Agent 规则**：
- 计费场景 `POLICYTYPE=PCC`
- `POLICYNAME` 对应 `PCCPOLICYGRP.PCCPOLICYGRPNM`
- PRIORITY 建议按10的倍数分配（便于后续插入新规则）
- 特定业务 PRIORITY < 兜底业务 PRIORITY

**OR 条件**：多个匹配条件执行**完全相同**的策略动作时，使用 FLOWFILTERGRP：
```
ADD RULE:RULENAME="rule_carrier",POLICYTYPE=PCC,FILTERINGMODE=FLOWFILTERGRP,
  FLWFLTRGRPNAME="ffgrp_carrier",PRIORITY=100,POLICYNAME="pccg_carrier";
```

(→ command/UDG/ADD RULE.md)

## 5. UserProfile 与兜底机制

### 5.1 UserProfile

**必填参数**：USERPROFILENAME

UserProfile 是策略容器，所有 RULE 通过 RULEBINDING 绑定到它。

```
ADD USERPROFILE:USERPROFILENAME="up_charging";
```

**注意**：UPSPECTYPE（DEFAULT/SPECIFICATION_LIMITED）决定了能绑定什么类型的 RULE，默认 DEFAULT。

(→ command/UDG/ADD USERPROFILE.md)

### 5.2 兜底计费：两种机制

UserProfile 有两种兜底计费机制，**应同时配置以保持一致**：

**机制1：DFTURRGRPNAME（默认URR组）**
```
SET URRGRPBINDING:USERPROFILENAME="up_charging",DFTURRGRPNAME="urrg_default",
  DFTSIGURRGNAME="urrg_default";
```
- 当流量未命中任何 RULE 时使用
- **必须配置**，否则未命中流量不计费

**机制2：显式兜底 RULE**
```
ADD FILTER:FILTERNAME="filter_default",L34PROTTYPE=STRING,L34PROTOCOL=ANY;
ADD FLOWFILTER:FLOWFILTERNAME="ff_default";
ADD FLTBINDFLOWF:FLOWFILTERNAME="ff_default",FILTERNAME="filter_default";
ADD RULE:RULENAME="rule_default",POLICYTYPE=PCC,FILTERINGMODE=FLOWFILTER,
  FLOWFILTERNAME="ff_default",PRIORITY=9000,POLICYNAME="pccg_default";
ADD RULEBINDING:USERPROFILENAME="up_charging",POLICYTYPE=PCC,RULENAME="rule_default";
```
- FILTER=ANY 匹配所有流量
- PRIORITY 设为最高数值（最低优先级）
- 更灵活，可同时指定计费策略和动作

**Agent 规则**：
- **两种机制都应配置**，DFTURRGRPNAME 和兜底 RULE 指向同一个 URRGROUP
- DFTSIGURRGNAME 与 DFTURRGRPNAME 保持一致

(→ command/UDG/SET URRGRPBINDING.md, → SKILL/业务图谱.md 13.2节)

## 6. URRID vs RG vs SID

这三个参数是 Agent 最容易混淆的：

| 参数 | 所属命令 | 含义 | 关系 |
|------|---------|------|------|
| URRID | ADD URR | PFCP Session Report 中的使用量上报规则标识 | 所有URR内唯一 |
| RG | ADD URR | Rating Group，3GPP定义的费率分组标识 | CHF/CG 使用此值匹配费率策略 |
| SID | ADD URR | Service ID，3GPP定义的业务标识 | 区分同一RG下不同业务 |

**Agent 规则**：
- 用户说"RG100" → `RG=100`
- URRID 建议从1000开始分配，可与RG保持映射关系但不是必须
- **RG 值必须与 SMF 侧配置一致**，否则报表与实际计费不一致
- SID 用于更细粒度的业务区分，通常不是必须配置

(→ command/UDG/ADD URR.md, → SKILL/配置生成SOP.md 13.1节)

## 7. 常见配置模式

### 7.1 离线计费标准模式

```
// 1. 创建URR
ADD URR:URRNAME="urr_biz1",URRID=1000,USAGERPTMODE=OFFLINE;
// 2. 创建URRGROUP
ADD URRGROUP:URRGROUPNAME="urrg_biz1",UPURRNAME1="urr_biz1",DOWNURRNAME1="urr_biz1";
// 3. 创建PCCPOLICYGRP
ADD PCCPOLICYGRP:PCCPOLICYGRPNM="pccg_biz1",URRGROUPNAME="urrg_biz1";
// 4. 创建RULE（绑定过滤条件和策略）
ADD RULE:RULENAME="rule_biz1",POLICYTYPE=PCC,FILTERINGMODE=FLOWFILTER,
  FLOWFILTERNAME="ff_biz1",PRIORITY=100,POLICYNAME="pccg_biz1";
// 5. 绑定到UserProfile
ADD RULEBINDING:USERPROFILENAME="up_charging",POLICYTYPE=PCC,RULENAME="rule_biz1";
```

### 7.2 免费业务模式

```
ADD URR:URRNAME="urr_free",URRID=4000,USAGERPTMODE=OFFLINE,OFFMETERINGTYPE=FREE;
ADD URRGROUP:URRGROUPNAME="urrg_free",UPURRNAME1="urr_free",DOWNURRNAME1="urr_free";
ADD PCCPOLICYGRP:PCCPOLICYGRPNM="pccg_free",URRGROUPNAME="urrg_free";
```

### 7.3 指定RG的差异化计费模式

```
ADD URR:URRNAME="urr_carrier",URRID=100,USAGERPTMODE=OFFLINE,RG=100;
ADD URRGROUP:URRGROUPNAME="urrg_carrier",UPURRNAME1="urr_carrier",DOWNURRNAME1="urr_carrier";
ADD PCCPOLICYGRP:PCCPOLICYGRPNM="pccg_carrier",URRGROUPNAME="urrg_carrier";
```

### 7.4 OR条件合并计费模式

当多个条件执行完全相同的计费策略时，使用 FLOWFILTERGRP：
```
// 创建两个FLOWFILTER，各自匹配不同条件
ADD FLOWFILTER:FLOWFILTERNAME="ff_url_carrier";
ADD FLOWFILTER:FLOWFILTERNAME="ff_prot_im";
// 合并为OR组
ADD FLOWFILTERGRP:FLWFLTRGRPNAME="ffgrp_biz",FLOWFILTERNAME="ff_url_carrier";
ADD FLOWFILTERGRP:FLWFLTRGRPNAME="ffgrp_biz",FLOWFILTERNAME="ff_prot_im";
// 一条RULE引用组
ADD RULE:RULENAME="rule_biz",POLICYTYPE=PCC,FILTERINGMODE=FLOWFILTERGRP,
  FLWFLTRGRPNAME="ffgrp_biz",PRIORITY=100,POLICYNAME="pccg_biz";
```

### 7.5 兜底规则完整模式

```
// 兜底FILTER（匹配所有流量）
ADD FILTER:FILTERNAME="filter_default",L34PROTTYPE=STRING,L34PROTOCOL=ANY;
ADD FLOWFILTER:FLOWFILTERNAME="ff_default";
ADD FLTBINDFLOWF:FLOWFILTERNAME="ff_default",FILTERNAME="filter_default";
// 兜底计费三件套
ADD URR:URRNAME="urr_default",URRID=100,USAGERPTMODE=OFFLINE,RG=100;
ADD URRGROUP:URRGROUPNAME="urrg_default",UPURRNAME1="urr_default",DOWNURRNAME1="urr_default";
ADD PCCPOLICYGRP:PCCPOLICYGRPNM="pccg_default",URRGROUPNAME="urrg_default";
// 兜底RULE（最低优先级）
ADD RULE:RULENAME="rule_default",POLICYTYPE=PCC,FILTERINGMODE=FLOWFILTER,
  FLOWFILTERNAME="ff_default",PRIORITY=9000,POLICYNAME="pccg_default";
// UserProfile + 双重兜底
ADD USERPROFILE:USERPROFILENAME="up_charging";
SET URRGRPBINDING:USERPROFILENAME="up_charging",DFTURRGRPNAME="urrg_default",
  DFTSIGURRGNAME="urrg_default";
ADD RULEBINDING:USERPROFILENAME="up_charging",POLICYTYPE=PCC,RULENAME="rule_default";
```

### 7.6 上下行差异化计费模式

```
// 上行免费URR
ADD URR:URRNAME="urr_up_free",URRID=2000,USAGERPTMODE=OFFLINE,OFFMETERINGTYPE=FREE;
// 下行按流量计费URR
ADD URR:URRNAME="urr_down_vol",URRID=2001,USAGERPTMODE=OFFLINE,OFFMETERINGTYPE=VOLUME,RG=500;
// 上下行差异化URRGROUP
ADD URRGROUP:URRGROUPNAME="urrg_asym",UPURRNAME1="urr_up_free",DOWNURRNAME1="urr_down_vol";
ADD PCCPOLICYGRP:PCCPOLICYGRPNM="pccg_asym",URRGROUPNAME="urrg_asym";
```

### 7.7 融合计费模式（在线+离线）

> 来源: `5G Core 计费解决方案/配置融合计费费率标识_90679537.md`

```
// 在线计费URR（配额控制）
ADD URR:URRNAME="urr_online",URRID=1001,USAGERPTMODE=ONLINE,ONLMETERINGTYPE=VOLUME,ONLINERG=300;
// 离线计费URR
ADD URR:URRNAME="urr_offline",URRID=1000,USAGERPTMODE=OFFLINE,OFFMETERINGTYPE=VOLUME,RG=300;
// 同一URRGROUP内同时引用两个URR（1号位=ONLINE，2号位=OFFLINE）
ADD URRGROUP:URRGROUPNAME="urrg_conv",UPURRNAME1="urr_online",DOWNURRNAME1="urr_online",
  UPURRNAME2="urr_offline",DOWNURRNAME2="urr_offline";
ADD PCCPOLICYGRP:PCCPOLICYGRPNM="pccg_conv",URRGROUPNAME="urrg_conv";
```

**Agent 注意**：
- 融合计费使用**两个独立URR**（一个ONLINE + 一个OFFLINE），放入同一个URRGROUP
- SMF 侧还需配置 CCT 模板（见§16）
- SMF 侧 URRGROUP 的 1号位放 ONLINE URR，2号位放 OFFLINE URR

## 8. 命令执行顺序

严格按依赖链从底到顶，REFRESHSRV 必须最后：

```
1. ADD IPLIST              // 底层（如需IP地址列表）
2. ADD FILTER              // L3/L4过滤条件
3. ADD L7FILTER            // L7过滤条件
4. ADD FLOWFILTER          // 流过滤器（必须在PROTBINDFLOWF之前创建！）
5. ADD FLTBINDFLOWF        // FILTER→FLOWFILTER绑定
6. ADD PROTBINDFLOWF       // 协议+L7→FLOWFILTER绑定（60s后生效）
7. ADD FLOWFILTERGRP       // OR条件组合（可选）
8. ADD URR                 // 计费规则
9. ADD URRGROUP            // 计费规则组
10. ADD PCCPOLICYGRP       // PCC策略组
11. ADD RULE               // 汇聚规则
12. ADD USERPROFILE        // 用户模板
13. SET URRGRPBINDING      // 默认计费组绑定
14. ADD RULEBINDING        // 规则绑定
15. SET REFRESHSRV:REFRESHTYPE=ALL;  // 刷新生效（必须最后，立即生效但30s内勿操作Filter）
```

**依赖关系强调**：
- **FLOWFILTER 必须在 PROTBINDFLOWF 之前创建**（PROTBINDFLOWF 引用 FLOWFILTERNAME）
- **FILTER 必须在 FLTBINDFLOWF 之前创建**（FLTBINDFLOWF 引用 FILTERNAME）
- **三件套必须按 URR → URRGROUP → PCCPOLICYGRP 顺序创建**
- **RULEBINDING 必须在 RULE 和 USERPROFILE 都创建之后执行**

**时序注意**：
- PROTBINDFLOWF 执行后 **60s** 生效
- REFRESHSRV 执行后 **30s** 内不允许修改 Filter 和被绑定的 IPLIST
- 建议在 REFRESHSRV 后等待 **60s** 再进行验证

## 9. 配置生效时机（关键！）

计费配置涉及多个"非即时生效"操作，Agent 必须在脚本注释中提醒运维人员。

### 9.1 REFRESHSRV 刷新机制

**高危命令**，用于将 Filter/FilterGroup/Acl/AclNode 的配置变更置为生效。

| REFRESHTYPE | 刷新范围 | 适用场景 |
|-------------|---------|---------|
| **ALL** | Filter + FilterGroup + Acl + AclNode | 最安全，推荐默认使用 |
| **USERPROFILE** | 仅 Filter + FilterGroup | 只改了计费匹配规则时 |
| **ACL** | 仅 Acl + AclNode | 只改了 ACL 时 |

**时序约束**：
- 命令执行后**立即生效**（Filter/FilterGroup 部分）
- ACL/AclNode 修改后需 **30s 后** 才生效
- 执行后 **30s 内** 不允许修改 Filter 和被绑定的 IPLIST
- 新增/修改 Filter 和 FilterGroup 后**必须** 执行 REFRESHSRV 才能生效
- 删除 Filter 后也**必须** 执行才能停止匹配

**Agent 规则**：
- 计费场景通常使用 `REFRESHTYPE=ALL` 或 `REFRESHTYPE=USERPROFILE`
- 脚本最后一条命令必须是 `SET REFRESHSRV`
- 在脚本注释中标注"执行后30秒内勿操作Filter"

(→ command/UDG/SET REFRESHSRV.md)

### 9.2 PROTBINDFLOWF 60秒延迟生效

`ADD PROTBINDFLOWF` 执行后 **60s 才生效**，比 REFRESHSRV 还慢。

**影响**：
- 绑定协议后 1 分钟内相关流量不会按新规则匹配
- 单个 FlowFilter 最多绑定 32 个 Protocol
- 最大记录数 100000，超过 90% 上报告警

**Agent 规则**：
- 如果配置涉及 PROTBINDFLOWF（L7协议匹配），在脚本注释中标注"协议绑定60秒后生效"
- 如需验证，建议在 REFRESHSRV 执行 60s 后再查

(→ command/UDG/ADD PROTBINDFLOWF.md)

## 10. 常见陷阱

### 10.1 URRID 不唯一
- URRID 在**所有URR** 内必须唯一（包括不同 USAGERPTMODE 的）
- 建议从1000开始分配，每个业务递增

### 10.2 RG 与 SMF 不一致
- UDG 侧的 RG 值必须与 SMF 侧计费配置一致
- 否则报表上报的 RG 与实际计费的 RG 不一致

### 10.3 缺少兜底计费
- 如果不配 DFTURRGRPNAME，未命中任何 RULE 的流量**不会计费**
- 必须同时配置 DFTURRGRPNAME 和 DFTSIGURRGNAME

### 10.4 REFRESHSRV 时机错误
- 所有 ADD/SET 命令完成后才能执行 REFRESHSRV
- 过早刷新会导致部分配置不生效
- ACL 修改后即使执行了 REFRESHSRV 也要等 30s

### 10.5 URRGROUP 至少绑定一个URR
- UPURRNAME1-3 和 DOWNURRNAME1-3 中至少设置一个
- 否则 URRGROUP 为空，无法执行计费

### 10.6 免费业务的 OFFMETERINGTYPE
- 免费业务需设 `OFFMETERINGTYPE=FREE`
- 仍需创建完整的 URR+URRGROUP+PCCPOLICYGRP 三件套
- FREE 不是不创建对象，而是创建一个计量类型为FREE的对象

### 10.7 RULE 的 POLICYTYPE 与 RULEBINDING 必须一致
- `ADD RULE:POLICYTYPE=PCC` → `ADD RULEBINDING:POLICYTYPE=PCC`
- 不一致会导致绑定失败

### 10.8 PROTBINDFLOWF 协议与网站协议不匹配
- `PROTOCOLNAME` 必须与目标网站实际协议一致
- 例如：目标是 https 网站，但配置了 `PROTOCOLNAME="http"`，则匹配失败
- 需要同时配 http 和 https：两条 `ADD PROTBINDFLOWF`

### 11.2 UNC URR 特有参数

UNC 的 ADD URR 比 UDG 多出以下关键参数：

**复合类型参数**（决定上报内容组合）：

| 参数 | 含义 | 取值 |
|------|------|------|
| OFFCOMPOUNDTYPE | 离线计费复合类型 | ONLYRG / ONLYSID / UPDOWNSID / RGSID |

**分离的上下行 Service ID**：

|------|------|

| 参数 | 含义 | 单位 |

| TIMERSUVALUE | 时间配额 | 秒 |
| VOLUMERSUVALUE | 流量配额 | 字节 |
| EVENTRSUVALUE | 事件配额 | 次 |

**Agent 规则**：
- UDG 配置只需关注 RG/SID/USAGERPTMODE/METERINGTYPE

(→ command/UNC/ADD URR.md)

- 生成 UDG 计费配置前，提醒用户确认 SMF 侧已 `SET PCCFUNC:HOMEPCCSWITCH=ENABLE`

### 11.5 离线计费 CG 连接（系统级前置配置）

| 命令 | 含义 | 说明 |
|------|------|------|
| ADD CG | 添加CG服务器 | 配置CG的IP和端口 |
| ADD CGGROUP | 添加CG组 | 可包含多个CG实现负载均衡 |

(→ command/UDG/ADD CG.md, → command/UNC/ADD CG.md)

### 11.6 PCC 故障处理（SET PCCFAILACTION）

当 PCC 策略执行失败时（如 CHF 不可达），需要定义故障处理策略。

**关键策略**：
| 策略 | 含义 |
|------|------|
| TERMINATE | 终止会话，停止流量 |
| CONTINUE | 继续服务，使用默认计费 |
| REDIRECT | 重定向到指定页面 |

**Agent 规则**：
- 在线计费场景建议配置 PCCFAILACTION 避免业务中断

- 属于系统级配置，提醒用户确认

(→ command/UNC/SET PCCFAILACTION.md)

### 11.7 UDG vs UNC 职责分工

| 层面 | UDG (UPF) | UNC (SMF) |
|------|-----------|-----------|
| **核心职责** | 业务识别、流量统计、数据面上报 | 计费策略决策、与CHF/OCS交互 |
| **URR 侧重点** | 匹配规则、计量类型、RG/SID | 复合类型、配额控制、CCT模板 |
| **PCC 控制** | RULE+PCCPOLICYGRP 执行 | SET PCCFUNC 全局开关 + 策略分配 |
| **配置顺序** | UDG 先配（数据面先就绪） | SMF 后配或同步（控制面引用数据面） |
| **License** | UDG 侧独立 License | SMF 侧独立 License |
| **REFRESHSRV** | 需要执行 | SMF 侧无此命令 |

### 11.8 SMF-UPF 一致性约束

Agent 必须在输出中提醒以下一致性要求：

| 约束项 | 说明 | 不一致的后果 |
|--------|------|-------------|
| RG 值 | UDG.ADD URR.RG = UNC.ADD URR.RG | 报表与实际计费不一致 |
| URRID | UDG.ADD URR.URRID = UNC.ADD URR.URRID | PFCP Session Report 无法关联 |
| URRGROUP 组成 | 上下行 URR 分配一致 | 计费统计不完整 |
| PCCPOLICYGRP 引用 | URRGROUP 引用一致 | 策略执行与预期不符 |

(→ feature/UNC/SMF/WSFD-011201/, → command/UNC/ADD URR.md)

## 12. L34层 vs L7层计费的匹配差异

用户需求中的"访问某网站"或"协议组为X"会影响匹配层次的配置：

用户需求中的"访问某网站"或"协议组为X"会影响匹配层次的配置：

| 用户描述 | 匹配层次 | FILTER配置 | PROTBINDFLOWF | L7FILTER |
|---------|---------|-----------|---------------|----------|
| "any to any" | L34 only | L34PROTOCOL=ANY | 不需要 | 不需要 |
| "访问www.carrier.com" | L7 URL | L34PROTOCOL=ANY | PROTOCOLNAME=http+https | URLTYPE=URL, URL=www.carrier.com/* |
| "协议组为IM" | L7 协议 | L34PROTOCOL=ANY | 逐协议绑定（whatsapp,wechat...） | 不需要（纯协议识别） |
| "特定IP地址" | L34 | SVRIPMODE=IPLIST | 不需要 | 不需要 |
| "TCP端口范围" | L34 | MSSTARTPORT+MSENDPORT | 不需要 | 不需要 |

**Agent 决策树**：
1. 用户说"所有流量"/"any" → FILTER(L34PROTOCOL=ANY)，不配 PROTBINDFLOWF
2. 用户说"访问某网站" → FILTER(ANY) + L7FILTER(URL) + PROTBINDFLOWF(http+https)
3. 用户说"某协议" → FILTER(ANY) + PROTBINDFLOWF(协议名)，无L7FILTER
4. 用户说"某协议组" → FILTER(ANY) + PROTBINDFLOWF(逐个协议)，**必须追问用户协议组内具体有哪些协议**
5. 用户说"特定IP" → FILTER(TCP, SVRIPMODE=IPLIST) + ADD IPLIST

**协议查询**：系统支持的协议列表可通过工程命令 `smctrldsp protocol-list` 查询。单个 FlowFilter 最多绑定 32 个协议。

(→ SKILL/配置生成SOP.md 13.3-13.4节, → command/UDG/ADD PROTBINDFLOWF.md)

## 14. SMF 侧计费属性 CC（Charging Characteristics）

CC 是3GPP定义的用户签约付费属性，SMF根据CC值选择CHF和CCT模板。

**CC标准取值**：
| 值 | 含义 |
|-----|------|
| 0x0100 | 热计费（Hot Billing） |
| 0x0200 | 统一费率（Flat Rate） |
| 0x0400 | 预付费（Prepaid） |
| 0x0800 | 普通计费（Normal Billing） |
| 自定义 | 用户定义值 |

**CC来源优先级**：RADIUS Server下发 > UDM签约 > SMF本地配置

**SMF本地CC配置粒度（优先级从高到低）**：
| 粒度 | 命令链 | 说明 |
|------|--------|------|
| UserProfile | `ADD CHARGECHAR` + `SET USRPROFCHARGE` | 每UserProfile绑一个CC |
| DNN | `ADD CHARGECHAR` + `SET APNCHARGECTRL` | 每DNN绑一个CC |
| 全局 | `SET GLBCHARGECHAR` | 全局默认CC |

**ADD CHARGECHAR 关键参数**：
| 参数 | 含义 | 说明 |
|------|------|------|
| CCNAME | CC名称 | 字符串标识 |
| HOME | 本地用户CC | 16进制值如 0x0800 |
| ROAM | 漫游用户CC | 16进制值 |
| VISIT | 拜访用户CC | 16进制值 |
| HOMESGSN | 本地用户使用UDM签约CC | ENABLE=使用UDM, DISABLE=使用本地 |
| ROAMSGSN | 漫游用户使用UDM签约CC | 同上 |
| VISITSGSN | 拜访用户使用UDM签约CC | 同上 |

**配置示例**：
```
// 创建CC并绑定到UserProfile
ADD CHARGECHAR:CCNAME="cc_normal",HOME="0x0800",ROAM="0x0800",VISIT="0x0800",
  HOMESGSN=DISABLE,ROAMSGSN=DISABLE,VISITSGSN=DISABLE;
ADD USERPROFILE:USERPROFILENAME="up-test",UPTYPE=PCC;
SET USRPROFCHARGE:USRPROFNAME="up-test",CCNAME="cc_normal";
```

**Agent 规则**：
- CC配置需在CHF选择和CCT模板之前完成
- 当 HOMESGSN=ENABLE 时使用UDM下发的CC，忽略本地配置
- 每个UserProfile/DNN只能绑定一个CC

(→ feature/UNC/SMF/WSFD-011206/配置计费属性CC_90776700.md, → WSFD-011201/CC_91845211.md)

## 15. SMF 侧 CHF 选择机制

融合计费(N40)时，SMF需要选择CHF（Charging Function）。7级优先级从高到低：

| 优先级 | 选择方式 | 命令 | 适用场景 |
|--------|---------|------|---------|
| 1 | 基于IMSI选择 | `ADD SELCHFGBYIMSI` | 拨测场景，指定IMSI到指定CHF |
| 2 | 基于IMSI号段 | `ADD SELCHFGBIMSISEG` | 多CHF负载均衡 |
| 3 | 基于PCF下发FQDN | `ADD TNFINS`（配置FQDN） | PCF指定CHF |
| 4 | 基于UDM签约CC | 自动（UDM下发CC） | 无需本地配置 |
| 5 | 标准化服务发现(NRF) | 无需配置 | NRF自动发现 |
| 6 | 基于本地CC选择 | `ADD SELECTCHFGBYCC` | CC→CHF组映射 |
| 7（最低） | 全局默认CHF组 | `SET GLBDFTCHFGROUP` | 兜底选择 |

**CHF组配置命令链**：
```
// 1. 创建CHF实例
ADD TNFINS:TNFINSINDEX=1,TNFTYPE=CHF,SCHEMA=http;
// 2. 配置CHF IP
ADD TNFINSIP:TNFINSINDEX=1,IPTYPE=IPV4,IPV4ADDR="192.168.94.2",PORT=3220;
// 3. 创建CHF组
ADD TNFGRP:TNFGRPINDEX=1,TNFTYPE=CHF,TNFGRPNAME="ChfGroup1";
// 4. 实例绑定到组
ADD TNFBINDGRP:TNFGRPINDEX=1,TNFINSINDEX=1;
```

**基于CC选择CHF**：
```
ADD SELECTCHFGBYCC:CCTYPE=VALUE,CCVALUE="0x0800",MASK="0xffff",
  PRIORITY=2,PRIMARYCHFGRP="ChfGroup1",SECONDARYCHFGRP="ChfGroup2";
```

**Agent 规则**：
- 现网最常用的是级别6（基于CC选择）+ 级别7（全局默认兜底）
- 主备CHF组必须都配置（PRIMARYCHFGRP + SECONDARYCHFGRP）
- PRIORITY数值越小优先级越高

(→ feature/UNC/SMF/WSFD-011206/配置CHF选择方式_92091086.md)

## 16. SMF 侧 CCT 融合计费模板

CCT（Converged Charging Template）包含融合计费参数（门限、触发条件等）。

**模板粒度优先级（从高到低）**：UserProfile > DNN > CC > 全局(global)

**ADD CCT 关键参数**：
| 参数 | 含义 | 单位 | 说明 |
|------|------|------|------|
| QHT | 配额空闲时间门限 | 秒 | 数据停止传送后触发上报 |
| VQT | 流量阈值触发百分比 | % | 剩余配额占比低于此值触发 |
| TQT | 时间阈值触发百分比 | % | 同上 |
| VT | 在线配额有效时长 | 分 | 超时后强制上报并申请新配额 |
| MAXNUMOFCCC | 计费条件改变阈值 | 次 | 改变N次后触发Update |
| PDUVOLUMELIMIT | PDU级流量阈值 | MB | PDU会话级触发 |
| PDUTIMELIMIT | PDU级时长阈值 | 分 | PDU会话级触发 |
| RGVOLUMELIMIT | RG级流量阈值 | MB | 费率组级触发 |
| RGTIMELIMIT | RG级时长阈值 | 分 | 费率组级触发 |
| FUATERMINATE | 最终配额动作 | — | TERM_SERVICE(阻塞)/其他 |
| MAXSVCCONTAINER | 最大业务容器数 | 个 | N40消息中最大容器数 |

**配置示例**：
```
// 创建CCT模板
ADD CCT:CCTMPLTNAME="cct_test",QHT=0,VQT=20,TQT=20,VT=30,
  MAXNUMOFCCC=3,PDUVOLUMELIMIT=500,PDUTIMELIMIT=30,
  RGVOLUMELIMIT=500,RGTIMELIMIT=30,FUATERMINATE=TERM_SERVICE;

// 绑定到CC
ADD SELECTCCTBYCC:CCTYPE=VALUE,CCVALUE="0x0800",CCTMPLTNAME="cct_test";

// 绑定到DNN
SET APNCHARGECTRL:APN="apn-test",CCTEMPLATE="cct_test";

// 绑定到UserProfile
SET USRPROFCHARGE:USRPROFNAME="up-test",CCTEMPLATE="cct_test";
```

**Agent 规则**：
- 修改全局模板用 `MOD CCT:CCTMPLTNAME="global"`
- 确定粒度后所有CCT参数应统一使用该粒度
- 与DCC模板（在线计费）的区别：CCT用于N40融合计费，DCC用于Gy在线计费

## 17. SMF 侧计费触发条件（Trigger）

计费Trigger决定SMF何时向CHF发送Charging Data Request消息。

**Trigger分类**：
- **Session级**：与PDU会话关联，适用整个会话
- **RG级**：与PDU会话内的RG关联，适用指定费率组

**上报方式**：
- **立即上报**：Trigger发生时立即发送Charging Data Request
- **延迟上报**：记录但不立即发送，在下次消息中携带

**SMF本地配置命令**：
| 命令 | 级别 | 说明 |
|------|------|------|
| `ADD PDUTRIGGER` | Session | 配置PDU会话级触发条件 |
| `ADD RGTRIGGER` | RG | 配置费率组级触发条件 |

**关键Trigger事件**：

| 分类 | 事件 | 级别 | 缺省上报方式 |
|------|------|------|------------|
| 会话建立 | Start of PDU Session | Session | 立即 |
| 计费条件变更 | Charging condition change | — | 立即 |
| **会话级门限** | PDU会话时间/流量/事件门限 | Session | 立即 |
| **RG级门限** | 费率组时间/流量/事件门限 | RG | 延迟 |
| 配额管理 | 配额耗尽/阈值/有效时间超时 | RG | 立即 |
| QoS/位置变更 | QoS change / User Location change | Session/RG | 延迟 |
| 会话结束 | End of PDU Session | Session | 立即 |

**优先级规则**：
1. CHF下发的Trigger > SMF本地配置
2. Session级和RG级冲突时，Session级优先
3. `SET CNVRGDCHGPARA` 的 `IGNORETRIGGER` 参数可忽略CHF下发的指定Trigger

**CCT模板中的Trigger门限**：
- `PDUVOLUMELIMIT`/`PDUTIMELIMIT` → Session级流量/时长门限
- `RGVOLUMELIMIT`/`RGTIMELIMIT` → RG级流量/时长门限
- `MAXNUMOFCCC` → 计费条件改变次数门限

**Agent 规则**：
- 大部分Trigger有合理缺省值，一般不需手动配置 PDUTRIGGER/RGTRIGGER
- CCT模板中的门限参数（PDUVOLUMELIMIT等）是更常用的配置方式
- 如果CHF下发的Trigger不符合预期，可用 IGNORETRIGGER 参数忽略

(→ feature/UNC/SMF/WSFD-011206/计费事件触发条件_90776688.md)

## 18. SMF 侧在线计费 DCC 模板（Gy接口）

在线计费（Gy接口）使用 DCCTEMPLATE（DCC模板），与融合计费的 CCT 模板是两套独立体系。

**DCC模板 vs CCT模板**：
| 维度 | DCC模板 | CCT模板 |
|------|---------|---------|
| 计费接口 | Gy（在线计费） | N40（融合计费） |
| 对端 | OCS | CHF |
| 模板命令 | `ADD DCCTEMPLATE` | `ADD CCT` |
| 全局模板名 | dcc-global | global |

**ADD DCCTEMPLATE 关键参数**：
| 参数 | 含义 | 单位 | 说明 |
|------|------|------|------|
| QHT | 配额空闲时间门限 | 秒 | 建议≥60或不使用(QHT=0) |
| QCT | 配额空耗时间门限 | 秒 | QCT=0表示连续统计时长 |
| VQT | 流量阈值触发百分比 | % | 剩余配额低于此值触发CCR |
| TQT | 时间阈值触发百分比 | % | 建议≥20 |
| VALIDTIME | 配额有效时长 | 分 | 超时后触发CCR |
| CCRINITRGNUM | CCR-I消息携带RG数 | 个 | 需与SET CTXSTARTRATING配合 |
| CCFH | 故障处理方式 | — | CONTINUE/TERMINATE/RETRY_AND_TERM |
| SERVICETRIGGER | 新业务触发CCR | — | ENABLE/DISABLE |
| RATTRIGGER | RAT改变触发CCR | — | ENABLE/DISABLE |
| SGSNTRIGGER | SGSN改变触发CCR | — | ENABLE/DISABLE |
| QOSTRIGGER | QoS改变触发CCR | — | ENABLE/DISABLE |
| ULITRIGGER | 位置改变触发CCR | — | ENABLE/DISABLE |

**DCC模板绑定粒度**（优先级从高到低）：UserProfile > APN/DNN > CC > 全局

**配置命令链**：
```
// 1. 创建DCC模板
ADD DCCTEMPLATE:DCCTMPLTNAME="dcc-test",CCFH=CONTINUE,QHT=0,VQT=0,TQT=20,
  VALIDTIME=30,SERVICETRIGGER=ENABLE,RATTRIGGER=ENABLE;
// 2. 用户激活时与OCS交互
SET OCSINIT:DCCTMPLTNAME="dcc-test",OCSINIT=ENABLE;
// 3. 绑定到APN
SET APNCHARGECTRL:APN="apn-test",DCCTEMPLATE="dcc-test";
// 或绑定到UserProfile
SET USRPROFCHARGE:USRPROFNAME="up-test",DCCTEMPLATE="dcc-test";
```

**在线计费URR特殊参数（SMF侧）**：
```
ADD URR:URRNAME="onlineURR",URRID=1100,USAGERPTMODE=ONLINE,
  ONLCOMPOUNDTYPE=ONLYRG,ONLINERG=100,ONLMETERINGTYPE=TIME;
```
- `ONLCOMPOUNDTYPE=ONLYRG`：在线计费仅上报RG
- `ONLINERG`：在线计费费率组（对应UDG侧的RG）
- `ONLMETERINGTYPE`：在线计量类型

**SET CTXSTARTRATING**：配置用户激活时CCR-I携带的RG
```
SET CTXSTARTRATING:USERPROFILENAME="up-test",CTXRURRGRPNAME1="urrgroup1";
```

**Agent 规则**：
- 在线计费（Gy）和融合计费（N40）是两套独立模板体系
- DCC模板的 CCFH=CONTINUE 时 Tx超时后转离线计费
- VQT/TQT建议≥20，避免CCR消息过于频繁

(→ feature/UNC/SMF/WSFD-109001/配置在线计费模板_95923574.md, → 配置CCR触发场景_95923545.md)

## 19. SMF 侧在线计费异常处理

在线计费会话中需要配置各种异常场景的处理动作。

**异常处理层次**：
| 层次 | 命令 | 说明 |
|------|------|------|
| Tx定时器超时 | CCFH参数 | TERMINATE/CONTINUE/RETRY_AND_TERM |
| Command层异常码 | `ADD CMDRCACT` | 特定返回码→特定动作 |
| MSCC层异常码 | `ADD MSCCRCACT` | 特定返回码→特定动作 |
| Command层默认 | `SET CMDLEVDFTBEH` | 未配置的异常码走默认 |
| MSCC层默认 | `SET MSCCLEVDFTBEH` | 未配置的异常码走默认 |
| 一键放通 | `SET FHBYPASS` | 紧急场景，优先级最高 |

**CCFH（Credit Control Failure Handling）三种模式**：
| 模式 | 行为 | 说明 |
|------|------|------|
| TERMINATE | 去激活用户 | 最严格 |
| CONTINUE | 重试备OCS→失败转离线 | 推荐，需配failover |
| RETRY_AND_TERM | 重试备OCS→失败去激活 | 中间方案 |

**Failover配置**：
```
ADD DCCTEMPLATE:DCCTMPLTNAME="dcc-test",CCFH=CONTINUE,
  FAILOVERSUP=ENABLE,AUTOFAILBACK=ENABLE;
```
- `FAILOVERSUP=ENABLE`：支持主备切换
- `AUTOFAILBACK=ENABLE`：主OCS恢复后自动回切

**其他关键配置**：
| 命令 | 用途 |
|------|------|
| `SET TXTIMER` | Tx定时器时长（建议10秒） |
| `SET HOLDINGTIME` | 转离线后业务保持时长 |
| `SET OCSDOWNACTION` | 用户激活时OCS不可达的处理 |
| `SET TCTIMER` | TCP重连间隔 |
| `SET TWTIMER` | DWR超时处理 |

**异常返回码处理动作**：
| 动作 | 含义 |
|------|------|
| TERM_WITH_CCRT | 终止并上报CCR |
| BLOCK | 阻塞业务 |
| CONTINUE | 继续业务 |
| REDIRECT | 重定向 |

**Agent 规则**：
- 推荐 CCFH=CONTINUE + FAILOVERSUP=ENABLE，故障时先服务
- `SET FHBYPASS` 优先级最高，需客户书面认可后才能使用
- 转离线后的保持时长通过 HOLDINGTIME 控制

(→ feature/UNC/SMF/WSFD-109001/配置DCC处理动作_95923447.md)

## 21. SMF 侧配置顺序

> 来源: `5G Core 计费解决方案/方案配置_90839325.md`、`WSFD-011206/使用全局CCT模板进行融合计费实例_93029782.md`、`WSFD-011206/配置融合计费费率标识_90679537.md`、`WSFD-109001/配置普通在线计费实例_95923459.md`

SMF 侧计费配置分为 **系统级（一次性部署）** 和 **业务级（每业务配置）** 两层。

### 21.1 系统级前置配置（一次性部署）

以下配置在 SMF 首次部署时执行，不随业务变化：

```
// 1. 功能开关
SET PCCFUNC                        // PCC全局开关

// 2. 计费接口模式
SET CHGMODE                        // 全局计费接口选择（融合/在线/离线）
SET CHGIFMODE                      // 接入类型到计费接口映射

// 3. 计费属性 CC
ADD CHARGECHAR                     // CC实例（0x0100/0x0200/0x0400/0x0800）
SET GLBCHARGECHAR                  // 全局CC（或 SET USRPROFCHARGE / SET APNCHARGECTRL 绑定粒度）

// 4. CHF/OCS 连接（融合计费 N40）
ADD TNFINS + ADD TNFINSIP          // CHF 实例
ADD TNFGRP + ADD TNFBINDGRP        // CHF 组
ADD SELCHFGBYIMSI / ADD SELECTCHFGBYCC / SET GLBDFTCHFGROUP  // CHF 选择方式

// 4'. OCS 连接（在线计费 Gy）
ADD DIAMLOCINFO                    // 本端标识
ADD OCS + ADD DIAMPEERADDR         // OCS 信息
ADD OCSGROUP + ADD OCSBINDING      // OCS 组

// 5. 融合计费模板（N40/CCT）
ADD CCT / MOD CCT                  // CCT 模板
ADD SELECTCCTBYCC                  // 按CC选择CCT
SET CHFINIT                        // CHF交互使能
SET FAILHANDLING                   // 超时处理动作
ADD PDUSCACT / ADD RGRCACT         // 异常返回码动作

// 5'. 在线计费模板（Gy/DCC）
ADD DCCTEMPLATE                    // DCC 模板（配额门限、CCR触发）
SET TXTIMER                        // Tx定时器
ADD MSCCRCACT                      // MSCC异常码动作
SET OCSINIT / SET CTXSTARTRATING   // 初始CCR配置

// 6. Trigger（可选）
ADD PDUTRIGGER                     // Session级触发条件
ADD RGTRIGGER                      // RG级触发条件
```

### 21.2 业务级数据面配置（每业务执行，需与UDG保持一致）

以下配置是**每条业务规则**需要执行的，RG/URRID 等参数必须与 UDG 对应对象一致：

```
// 参考实例：5G Core 计费解决方案/配置融合计费费率标识
// 核心原则：SMF 和 UPF 上 URRID、USAGERPTMODE、METERINGTYPE 保持一致

// 1. URR（每业务每计费方式一个）
ADD URR: URRNAME="urr_online_xxx", URRID={与UDG一致}, USAGERPTMODE=ONLINE,
       ONLCOMPOUNDTYPE=ONLYRG, ONLINERG={rg}, ONLMETERINGTYPE={metering};
ADD URR: URRNAME="urr_offline_xxx", URRID={与UDG一致}, USAGERPTMODE=OFFLINE,
       OFFCOMPOUNDTYPE=ONLYRG, RG={rg}, OFFMETERINGTYPE={metering};
// 融合计费：创建 ONLINE + OFFLINE 两个 URR
// 纯在线：只创建 ONLINE URR
// 纯离线：只创建 OFFLINE URR

// 2. URRGROUP（每业务一个，绑定上下行URR）
ADD URRGROUP: URRGROUPNAME="UrrGp_xxx",
       UPURRNAME1="urr_online_xxx", DOWNURRNAME1="urr_online_xxx",    // 1号位=ONLINE
       UPURRNAME2="urr_offline_xxx", DOWNURRNAME2="urr_offline_xxx"; // 2号位=OFFLINE

// 3. PCC策略组
ADD PCCPOLICYGRP: PCCPOLICYGRPNM="PPG_xxx", URRGROUPNAME="UrrGp_xxx";

// 4. RULE（需与UPF侧 RULENAME、POLICYTYPE、POLICYNAME 一致）
ADD RULE: RULENAME="Rule_xxx", POLICYTYPE=PCC, PRIORITY={pri}, POLICYNAME="PPG_xxx";

// 5. 绑定（USERPROFILE名称在 PCF/SMF/UPF 间一致）
ADD RULEBINDING: USERPROFILENAME="userprofile_xxx", RULENAME="Rule_xxx";
SET URRGRPBINDING: USERPROFILENAME="userprofile_xxx", DFTURRGRPNAME="UrrGp_abnormal";
```

### 21.3 SMF vs UDG 配置职责对比

| 配置项 | SMF（UNC） | UDG（UPF） | 是否需一致 |
|--------|-----------|-----------|----------|
| URRID | ADD URR | ADD URR | **必须一致** |
| USAGERPTMODE | ADD URR | ADD URR | **必须一致** |
| METERINGTYPE | ADD URR | ADD URR | **必须一致** |
| RG / ONLINERG | ADD URR | ADD URR | **必须一致** |
| RULENAME | ADD RULE | ADD RULE | **必须一致** |
| POLICYTYPE | ADD RULE | ADD RULE | **必须一致** |
| POLICYNAME(PCCPOLICYGRPNM) | ADD RULE | ADD RULE | **必须一致** |
| USERPROFILENAME | ADD RULEBINDING | ADD USERPROFILE | **必须一致**（PCF/SMF/UPF三方一致） |
| FILTER/FLOWFILTER | **SMF不配置** | ADD FILTER + FLOWFILTER | SMF侧通过PCF动态下发 |
| REFRESHSRV | **SMF无此命令** | SET REFRESHSRV（必须最后） | — |
| CCT/DCC模板 | SMF独有 | **UDG无** | — |
| CHF/OCS连接 | SMF独有 | **UDG无** | — |

**Agent 规则**：
- **SMF 侧不配置 FILTER/FLOWFILTER/L7FILTER**：流匹配条件由 PCF 动态下发或由 UDG 本地配置
- **SMF 侧不需要 REFRESHSRV**：配置即时生效
- **融合计费 SMF 侧也是双 URR 模式**：UPURRNAME1=ONLINE URR, UPURRNAME2=OFFLINE URR
- 生成配置时，系统级前置配置只需在首次部署脚本中包含；日常业务变更只生成 §21.2 的数据面配置

## 22. 补充隐性规则与约束

> 来源: 计费场景统一知识库 K201-K255

### 22.1 融合计费开关 RGAPPLIED 约束

SET CHARGECTRL 的 RGAPPLIED 参数控制业务申请上报模式：
- **DEFAULT**：URRGROUP 中**不能**同时配 RG 相同的离线和在线 URR
- **ONLINERGONLY** 或 **OFFLINERGONLY**：**建议**同时配离线和在线 URR

### 22.2 跨网元名称一致性汇总

| 配置项 | 一致性要求 | 涉及网元 |
|--------|-----------|----------|
| USERPROFILENAME | 必须一致 | PCF, SMF, UPF |
| RULENAME | 必须一致 | SMF, UPF |
| RULE.POLICYTYPE+POLICYNAME | 必须一致 | SMF, UPF |
| URRID | 必须一致 | SMF, UPF |
| USAGERPTMODE | 必须一致 | SMF, UPF |
| METERINGTYPE | 必须一致 | SMF, UPF |
| 配额名称(Quota) | PCF配额名须与CHF侧一致 | PCF, CHF |
| FQDN | SMF的TNFINS.FQDN须与PCF下发一致 | SMF, PCF |

### 22.3 内容计费双 License

内容计费需**两个**License 同时开启：
- UNC 侧：LKV3W9BCC12
- UDG 侧：LKV3G5BCBC01

两处必须均为 ENABLE 才能正常使用。

### 22.4 URRFAILACTION — 新业务无 URR 容灾

UNC 未配置新业务 URR 时，新业务被阻塞。
通过 `SET URRFAILACTION: RETRYFAILACT=CONTINUE` 可放通。

### 22.5 计费三件套修改命令链

新业务配额下发失败时，需按层级顺序修改：
```
MOD URR → MOD URRGROUP → MOD PCCPOLICYGRP → MOD RULE
```

### 22.6 费率切换配置

粒度优先级：User Profile > APN > CC > 整机

命令链：ADD FESTIVAL(节假日) → ADD WEEKDAY(星期表) → ADD TARIFFGROUP(时间段) → SET N40QUOTACTRL(TCMODE) → 绑定(SET APNCHARGECTRL 或 SET USRPROFCHARGE)

约束：CCVALUE 不能为 0x0200(统一费率)，每 DNN/UP 只能绑一个费率切换组。

### 22.7 计费消息缓存配置

场景：主备 CHF 均故障时 SMF 缓存计费消息。

前提：SET FAILHANDLING 的 FHACTION=CONTINUE。此功能为定制能力，要求对端为华为 NCG。

命令链：SET N40MSGSTG(缓存开关+回放间隔/速率) → SET STGTRIGGER(缓存期间trigger) → SET CNVRGDCHGPARA(CHGDATAREFGEN=SMF) → SET CDRSTORAGECTRL(超期天数) → SET N4CHGMSGCTRL(缓存池扩展)
