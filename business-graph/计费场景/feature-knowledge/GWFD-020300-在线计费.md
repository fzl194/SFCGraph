# GWFD-020300 在线计费 知识文档

## 概述

### 定义

在线计费是 UDG 对移动数据用户使用网络资源进行实时计费的功能。当用户发起数据业务时，OCS 将用户账户金额转换成流量、时间通过 PGW-C 下发给 PGW-U，用户访问业务的过程中，OCS 实时跟踪配额的使用情况，配额耗尽时终止业务。

### 适用网元

- **PGW-U**（UDG 侧）
- **PGW-C**（UNC 侧）
- **OCS**（Online Charging System，在线计费系统）

### 支持版本

| 涉及NF | 支持版本 | 功能说明 |
|--------|----------|----------|
| PGW-C | UNC 20.0.0 及后续版本 | 与 OCS 交互，获取和管理配额信息；与 PGW-U 交互，下发业务级配额、获取 PGW-U 统计的计费信息 |
| PGW-U | UDG 20.0.0 及后续版本 | 统计用户的计费信息（时长、流量等），上报给 PGW-C，并执行信用控制动作 |
| OCS | 无特殊要求 | 进行用户业务配额管理和在线计费控制 |

### License 要求

本特性需要 License 许可：**82209821 LKV3G5OLCH01 在线计费**。

### 遵循标准

| 标准类别 | 标准编号 | 标准名称 |
|----------|----------|----------|
| 3GPP | 23.502 | Procedures for the 5G System; Stage 2 (Release 15) |
| 3GPP | 29.244 | Interface between the Control Plane and the User Plane Nodes; Stage 3 |

### 客户价值

| 受益方 | 受益描述 |
|--------|----------|
| 运营商 | 对用户访问网络资源进行实时计费，降低用户欠费风险；根据用户使用数据业务的信息进行精确收费；为运营商之间或运营商与 ISP 之间的结算提供依据 |
| 用户 | 用户不感知该特性 |

## 核心概念

### 在线计费架构

在线计费涉及三个核心网元的协作：

```
OCS (在线计费系统)
  ↕ Gy接口 / DCC协议
PGW-C (UNC, 控制面)
  ↕ N4接口 / PFCP协议
PGW-U (UDG, 用户面)
```

- **OCS**：负责用户账户管理、配额分配和信用控制决策。通过 Gy 接口与 PGW-C 交互，使用 DCC（Diameter Credit Control）协议。
- **PGW-C**：控制面网元，作为 OCS 和 PGW-U 之间的桥梁。从 OCS 获取配额，通过 N4/PFCP 接口下发给 PGW-U；同时收集 PGW-U 上报的用量信息，转发给 OCS。
- **PGW-U**：用户面网元（UDG），负责实际的用户面流量统计（流量、时长），执行配额管理和信用控制动作（放通/阻塞业务）。

### 关键概念

#### USAGERPTMODE=ONLINE

`USAGERPTMODE` 是 URR（Usage Reporting Rule）的核心参数之一，用于标识该 URR 的使用量上报模式。取值为 `ONLINE` 时，表示该 URR 用于在线计费场景。

**核心约束**：URRID、USAGERPTMODE、ONLMETERINGTYPE 这三个参数在 PGW-C 和 PGW-U 上的配置必须保持一致。

#### ONLMETERINGTYPE（在线计费统计类型）

ONLMETERINGTYPE 定义在线计费的统计维度，支持以下取值：

| 取值 | 含义 |
|------|------|
| VOLUME | 流量计费 |
| DURATION | 时长计费 |
| EVENT | 事件计费 |
| EVENT_VOLUME_TIME | 事件+流量+时长（综合统计） |

PGW-U 根据此参数决定统计哪些使用量信息并上报。

#### 配额（Quota）

配额是 OCS 分配给用户的可使用资源量，可以是流量配额（Volume Quota）或时间配额（Time Quota）。配额由 OCS 通过 PGW-C 下发到 PGW-U，PGW-U 负责监控配额消耗。

#### Credit Pooling

Credit Pooling 是在线计费的增强功能，允许多个 URR 共享一个配额池。该功能仅对 Gy 接口生效，当前仅支持流量计费，不支持不同权重处理。在 20.15.2 版本中引入（特性版本 02）。

### 特性依赖

| 交互类型 | 相关特性 | 控制项名称 | 交互说明 |
|----------|----------|------------|----------|
| 依赖 | GWFD-020301 内容计费基本功能 | 82209822 内容计费基本功能 | 如果需要基于业务粒度对用户业务进行在线计费，需要同时开启内容计费基本功能 |

### 应用限制

- UDG 不支持 enhanced-ECUR、IEC 模式的事件计费功能。
- UDG 不支持紧耦合功能。
- UDG 的 Credit Pooling 功能仅对 Gy 接口生效，当前仅支持流量计费，且不支持不同权重处理。

## 原理与流程

### 在线计费总体工作原理

1. 用户激活时，OCS 下发配额到 PGW-C。
2. PGW-C 通过 N4/PFCP 接口将配额下发到 PGW-U。
3. 用户开始访问业务，PGW-U 统计流量和时长。
4. 当配额耗尽（或达到阈值），PGW-U 通过 N4 上报给 PGW-C。
5. PGW-C 到 OCS 申请新的配额。
6. PGW-C 将新配额下发到 PGW-U。

### 流程一：用户创建承载时下配额

当 PGW-C 在用户创建承载（PFCP Session Establishment）时已经从 OCS 获取到配额，会随 PFCP 会话建立消息一起下发。

```
步骤1: PGW-C → PGW-U
  PFCP Session Establishment Request
  携带 Create URR 信元组（含配额信息）

步骤2: PGW-U → PGW-C
  PFCP Session Establishment Response
  会话建立成功

步骤3: 用户访问业务，配额耗尽时
  PGW-U → PGW-C
  PFCP Session Report Request
  携带 Usage Report 信元组（含使用量统计）

步骤4: PGW-C → PGW-U
  PFCP Session Report Response

步骤5: PGW-C → PGW-U
  PFCP Session Modification Request
  携带 Update URR 信元组（更新配额/下发新配额）

步骤6: PGW-U → PGW-C
  PFCP Session Modification Response
  配额更新完成
```

**步骤1 中 Create URR 信元组包含的关键子信元：**

| 子信元 | 信元解释 |
|--------|----------|
| URR ID | 唯一标识一个 Create URR |
| Measurement Method | 标识计费方式：流量计费、时长计费或事件计费 |
| Reporting Triggers | 计费上报触发条件：Periodic Reporting（定期上报）、Volume Quota（流量配额）、Time Quota（时间配额）、Linked Usage Reporting（关联 URR）等 |
| Volume Quota | 流量配额 |
| Time Quota | 时间配额 |
| Linked URR ID | 上报关联：A link B，当 B 上报时也会上报 A |

**步骤3 中 Usage Report 信元组包含的关键子信元：**

| 子信元 | 信元解释 |
|--------|----------|
| URR ID | 标识上报的 URR |
| UR-SEQN | 唯一标识 URR 的 Usage Report |
| Usage Report Trigger | 标识 URR 的 Trigger |
| Start Time | 起始时间 |
| End Time | 结束时间 |
| Volume Measurement | 流量使用情况：总流量、上行流量和下行流量 |
| Duration Measurement | 时长使用情况 |
| Time of First Packet | 首包时间 |
| Time of Last Packet | 尾包时间 |

### 流程二：用户创建承载时未下配额

当 PGW-C 在用户创建承载时没有配额可下发（或下发的 URR ID 与用户实际访问业务匹配到的本地 URR ID 不一致），PGW-U 需要主动申请配额。

```
步骤1: PGW-C → PGW-U
  PFCP Session Establishment Request
  未携带 Create URR 信元组
  （或携带了但 URR ID 与本地配置不匹配）

步骤2: PGW-U → PGW-C
  PFCP Session Establishment Response
  会话建立成功

步骤3: 用户访问数据业务时，PGW-U 主动申请配额
  PGW-U → PGW-C
  PFCP Session Report Request
  携带 Usage Report 信元组（含 URR ID、UR-SEQN、Usage Report Trigger）

步骤4: PGW-C → PGW-U
  PFCP Session Report Response

步骤5: PGW-C → PGW-U
  PFCP Session Modification Request
  携带 Create URR 信元组（下发配额）

步骤6: PGW-U → PGW-C
  PFCP Session Modification Response
  配额下发完成
```

**注意**：步骤5 中使用的是 **Create URR**（而非 Update URR），因为此时是首次为该 URR 创建配额。配额下发完成后，后续的配额耗尽上报流程与"流程一"中的步骤3~4 相同。

### 默认配额机制（Default Quota）

当配置了 default quota 功能时（通过 SET UPDEFAULTQUOTA 命令使能"新业务默认配额使能开关"），在 PGW-U 主动申请配额期间：

- 使用由 ADD URR 命令的 **DFTQUOTAVOLUME** 和 **DFTQUOTATIME** 参数配置的默认配额。
- 报文放通，无需缓存报文。
- 避免在申请配额等待期间阻塞用户业务。

### 配额耗尽后的超时阻塞机制

当配额耗尽、PGW-U 申请新配额时，启动超时定时器：

- **定时器时长** = T3RESPONSE x N3REQUEST + 4 秒
- **T3RESPONSE** 和 **N3REQUEST** 由 SET UPPFCPPATH 命令配置。
- 定时器超时未收到配额更新消息时，**PGW-U 阻塞业务**。

## 配置规则

### 配置前提

1. 必须已开启在线计费 License：`SET LICENSESWITCH:LICITEM="LKV3G5OLCH01",SWITCH=ENABLE;`
2. 需要阅读 Ga/Gy 接口离线/在线计费原理文档。
3. 如果需要基于业务粒度进行在线计费，需同时开启内容计费基本功能（GWFD-020301）。

### 配置一致性要求

以下参数在 PGW-C 和 PGW-U 上必须保持一致：

| 参数 | 所属命令 | 说明 |
|------|----------|------|
| URRID | ADD URR | URR 标识 |
| USAGERPTMODE | ADD URR | 使用量上报模式（在线计费场景取 ONLINE） |
| ONLMETERINGTYPE | ADD URR | 在线计费统计类型 |
| RULENAME | ADD RULE | 规则名称 |
| RULE 下的 POLICYTYPE | ADD RULE | 策略类型 |
| RULE 下的 POLICYNAME | ADD RULE | 策略名称 |
| USERPROFILE 下绑定的 RULENAME | ADD RULEBINDING | 规则绑定 |
| USERPROFILE 下绑定的 POLICYTYPE | ADD RULEBINDING | 策略类型 |

**注意**：USERPROFILE 的名称需在 PCRF、PGW-C、PGW-U 间保持一致，不一致时 PCF 下发的策略无法生效。PGW-C、PGW-U 上 USERPROFILE 下绑定的 RULENAME 和 POLICYTYPE 不一致时，策略无法按照预期生效。

### 在线计费配置三件套

与离线计费相同，在线计费的核心配置也是三件套结构：

```
ADD URR（定义使用量上报规则）
  → ADD URRGROUP（组合上行/下行 URR）
    → ADD PCCPOLICYGRP（PCC 策略组绑定 URR 组）
```

### URR 关键参数

| 参数 | 含义 | 在线计费场景取值 |
|------|------|-----------------|
| URRNAME | 使用量上报规则名称 | 自定义（如 onlineURR） |
| URRID | URR 标识 | 全网规划（如 1100），PGW-C 和 PGW-U 一致 |
| USAGERPTMODE | 使用量上报模式 | **ONLINE** |
| ONLMETERINGTYPE | 在线计费统计类型 | EVENT / VOLUME / DURATION / 组合（如 EVENT_VOLUME_TIME） |
| DFTQUOTAVOLUME | 默认流量配额 | 配合 default quota 功能使用 |
| DFTQUOTATIME | 默认时间配额 | 配合 default quota 功能使用 |

### 规则优先级

只绑定三四层 filter 的 RULE 的优先级应该**低于**绑定七层 filter 的 RULE 的优先级。数字越小优先级越高。

### 相关 MML 命令清单

| 命令 | 用途 |
|------|------|
| ADD URR | 增加使用量上报规则 |
| ADD URRGROUP | 增加 URR 组 |
| ADD PCCPOLICYGRP | 增加 PCC 策略组 |
| ADD FILTER | 增加过滤器（三四层） |
| ADD FLOWFILTER | 增加流过滤器 |
| ADD FLTBINDFLOWF | 增加流过滤器的过滤器绑定关系 |
| ADD RULE | 增加规则 |
| ADD USERPROFILE | 增加用户模板 |
| ADD RULEBINDING | 增加用户模板和规则的绑定关系 |
| SET URRFAILACTION | 设置 URR 上报失败后的动作处理参数 |
| SET LICENSESWITCH | 开关 License |
| SET UPDEFAULTQUOTA | 设置全局默认配额开关 |
| SET UPPFCPPATH | 设置 PFCP 路径参数（含 T3RESPONSE、N3REQUEST） |
| SET REFRESHSRV | 刷新服务（使配置生效） |

### 相关软参

| 软参 | 功能 |
|------|------|
| BIT1253 | 控制在线计费配额不足重定向 |
| BIT1556 | 控制是否开启聚合 URR 优化功能 |
| BIT1718 | 控制 Gy-OneShot 功能是否启用重定向最大次数限制功能 |
| BYTE57 | 设置内容计费资源耗尽或阻塞后的动作 |

| BIT1711 | 控制是否开启 rule 快速查找算法使用的 hash 节点的检查和修复功能 |

## 配置案例

### 案例：Gy 接口在线计费（URR ID=1100，按事件+流量+时长统计）

**场景描述**：用户进行在线计费，访问的业务的 URR ID 为 1100，统计类型为事件+流量+时长。

**完整配置脚本**：

```mml
// 1. 开启 License
SET LICENSESWITCH:LICITEM="LKV3G5OLCH01",SWITCH=ENABLE;

// 2. 配置在线计费 URR 三件套
ADD URR:URRNAME="onlineURR",URRID=1100,USAGERPTMODE=ONLINE,ONLMETERINGTYPE=EVENT_VOLUME_TIME;
ADD URRGROUP:URRGROUPNAME="urrgroup001",UPURRNAME1="onlineURR",DOWNURRNAME1="onlineURR";
ADD PCCPOLICYGRP:PCCPOLICYGRPNM="cg_red",URRGROUPNAME="urrgroup001";

// 3. 配置三四层过滤条件
ADD FILTER:FILTERNAME="filter_red",L34PROTTYPE=STRING,L34PROTOCOL=TCP,SVRSTARTPORT=80,SVRENDPORT=80;
ADD FLOWFILTER:FLOWFILTERNAME="fg_red";
ADD FLTBINDFLOWF:FLOWFILTERNAME="fg_red",FILTERNAME="filter_red";

// 4. 配置内容计费规则
ADD RULE:RULENAME="Rule002",POLICYTYPE=PCC,FILTERINGMODE=FLOWFILTER,FLOWFILTERNAME="fg_red",POLICYNAME="cg_red",PRIORITY=120;
SET REFRESHSRV:REFRESHTYPE=USERPROFILE;

// 5. 绑定用户模板
ADD USERPROFILE:USERPROFILENAME="up_test";
ADD RULEBINDING:USERPROFILENAME="up_test",RULENAME="Rule002";
```

**配置数据说明**：

| 类别 | 参数 | 取值 | 说明 |
|------|------|------|------|
| URR | URRNAME | onlineURR | 在线计费 URR 名称 |
| URR | URRID | 1100 | 全网规划的 URR ID |
| URR | USAGERPTMODE | ONLINE | 在线计费模式 |
| URR | ONLMETERINGTYPE | EVENT_VOLUME_TIME | 综合统计 |
| URRGROUP | URRGROUPNAME | urrgroup001 | URR 组名称 |
| URRGROUP | UPURRNAME1 / DOWNURRNAME1 | onlineURR | 上下行使用同一 URR |
| PCCPOLICYGRP | PCCPOLICYGRPNM | cg_red | PCC 策略组名称 |
| PCCPOLICYGRP | URRGROUPNAME | urrgroup001 | 绑定的 URR 组 |
| FILTER | FILTERNAME | filter_red | 三四层过滤器 |
| FILTER | L34PROTOCOL | TCP | 匹配 TCP 协议 |
| FILTER | SVRSTARTPORT/SVRENDPORT | 80 | 匹配端口 80 |
| RULE | RULENAME | Rule002 | 规则名称 |
| RULE | POLICYTYPE | PCC | 策略类型 |
| RULE | POLICYNAME | cg_red | 绑定的 PCC 策略组 |
| RULE | PRIORITY | 120 | 规则优先级 |
| USERPROFILE | USERPROFILENAME | up_test | 用户模板名称 |

## 验证与调测

### 告警

本特性无相关告警。

### 测量指标

本特性无相关测量指标。

### 关键检查点

1. **License 状态**：确认 LKV3G5OLCH01 在线计费 License 已使能。
2. **URR 一致性**：确认 PGW-C 和 PGW-U 上的 URRID、USAGERPTMODE、ONLMETERINGTYPE 配置一致。
3. **规则绑定一致性**：确认 PGW-C 和 PGW-U 上 USERPROFILE 下绑定的 RULENAME、POLICYTYPE 一致。
4. **PFCP 会话**：检查 PFCP 会话是否正常建立，配额是否正常下发。
5. **配额超时**：关注 T3RESPONSE 和 N3REQUEST 的配置，超时可能导致业务被阻塞。
