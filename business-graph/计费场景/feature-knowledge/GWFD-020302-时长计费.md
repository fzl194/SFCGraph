# GWFD-020302 基于业务时长的计费 知识文档

## 概述

基于业务时长的计费是在业务识别解析的基础上，按照该业务的使用时间进行计费。UDG 支持多种业务时长统计方式，可同时应用于在线计费、离线计费和融合计费三种计费模式。

| 维度 | 说明 |
|------|------|
| 特性ID | GWFD-020302 |
| 特性名称 | 基于业务时长的计费 |
| 适用NF | SGW-U、PGW-U、UPF（用户面）；SGW-C、PGW-C、SMF（控制面） |
| 版本要求 | UDG 20.0.0 及后续版本（用户面）；UNC 20.0.0 及后续版本（控制面） |
| License | 82209823 LKV3G5TBCS01 基于业务时长的计费 |
| 首次发布 | 20.0.0 |

**时长计费与内容计费的核心差异**：

内容计费（GWFD-020301）的核心是按流量（VOLUME）统计使用量；时长计费是内容计费的增强形式，核心区别在于 URR 命令中将统计类型设置为 `DURATION`（而非 `VOLUME`），UPF 统计的是用户使用业务的时长而非流量。

时长计费是否根据时长向终端用户收费，由 OCS（在线）、计费中心（离线）或 CHF（融合）决定，UDG 只负责统计和上报时长数据。

## 核心概念

### 时长计费的两种方式

基于业务的时长计费在在线和离线模式下均支持两种时长计费方式：

| 方式 | 英文缩写 | 特征 | 精度 |
|------|----------|------|------|
| 连续时长计费 | CTP (Charging Time Period) | QCT=0；连续时长统计 | 较低 |
| QCT 时长计费 | QCT (Quota-Consumption-Time) | QCT!=0；通过 QCT 定时器统计时长 | 较高 |

> 注意：CTP/DTP 和 QCT 是两个不同维度的概念。CTP/DTP 由 Time-Quota-Mechanism 中的 Time-Quota-Type 信元控制，属于时长配额管理方式；QCT 由 Quota-Consumption-Time AVP 控制，属于时长统计的精细度方式。两者可以组合使用。

### 时长统计方法的优先级规则（在线计费）

计费方式的确定有优先级顺序：

1. **TQM 优先**：如果 OCS 同时下发了 QCT 和 TQM（Time-Quota-Mechanism），则 TQM 优先。
2. **OCS QCT 次之**：如果 OCS 只下发了 QCT（未下发 TQM），则按 OCS 下发的 QCT 值决定。
3. **本地配置兜底**：如果没有携带 QCT 和 TQM，则根据 PGW-C/SMF 上配置的 QCT 值决定是采用 QCT>0 模式还是 QCT=0 模式。

> OCS 下发的时长计费方式优先级高于 PGW-C/SMF 本地配置。

### 基于业务 vs 基于用户的时长统计

| 统计粒度 | 开始时间 | 结束时间 | 说明 |
|----------|----------|----------|------|
| 基于业务 | UPF 首次收到该业务数据包 | 业务终止 | 只统计特定业务的使用时长 |
| 基于用户 | 用户激活上线 | 用户下线去激活 | 无论用户是否进行业务，全部时长均作为有效计费时间 |

基于用户的时长统计（即 "duration" 字段记录的从用户上线到下线之间的时长）与 QCT=0、QCT!=0、CTP、DTP 方式无关，是独立的统计维度。

## 原理与流程

### 1. 在线时长计费原理

在线时长计费是指 OCS 在 CCA 消息中下发了时长配额，PGW-U/UPF 统计用户使用业务的时长，通过 PGW-C/SMF 上报给 OCS。

#### 1.1 在线连续时长计费（QCT=0）

| 维度 | 说明 |
|------|------|
| 计费特征 | QCT=0，连续时长统计 |
| 配置来源 | PGW-C/SMF 上 ADD DCCTEMPLATE 命令中参数 QCT 设置为 0；或 OCS 通过 MSCC 中 Quota-Consumption-Time AVP 下发为 0 |
| 优先级 | OCS 下发的计费方式优先级高于 PGW-C/SMF 上配置 |

**时长统计方法**：
- 时长统计从 PGW-U/UPF 首次收到 PGW-C/SMF 发送的 OCS 授权配额成功创建信用控制实例时开始。
- 直到用户去激活时或者该信用控制实例终止结束。
- **无论用户是否进行该信用实例业务，全部时长均作为有效计费时间。**

#### 1.2 在线 QCT 时长计费（QCT!=0）

| 维度 | 说明 |
|------|------|
| 计费特征 | QCT!=0，通过 QCT 定时器统计时长 |
| 配置来源 | PGW-C/SMF 上 ADD DCCTEMPLATE 命令中参数 QCT 设置不为 0；或 OCS 通过 MSCC 中 Quota-Consumption-Time AVP 下发不为 0 |
| 优先级 | OCS 下发的计费方式优先级高于 PGW-C/SMF 上配置 |

**时长统计方法**：
- 从 PGW-U/UPF 首次收到业务数据包时开始计时。
- 当没有业务数据包传输时不立即停止计时，而是启动一个 QCT 定时器。
- **空闲时间 <= QCT**：该段空闲时间被统计为有效计费时间。
- **空闲时间 > QCT**：定时器超时后停止计时，超出 QCT 的业务空闲时间不计入有效计费时间，只有 QCT 时长统计到计费时长中。
- QCT 超时前有数据包转发，则定时器被重置。

#### 1.3 在线时长计费流程

```
1. 用户激活
   - 用户发送会话建立请求
   - PGW-C/SMF 发送 CCR-I 到 OCS
   - OCS 回复 CCA-I 允许激活
   - PGW-C/SMF 与 PGW-U/UPF 建立 PFCP Session
   - PGW-C/SMF 给用户返回会话建立响应

2. 用户访问业务
   - PGW-U/UPF 收到用户访问业务请求
   - PGW-U/UPF 向 PGW-C/SMF 请求配额
   - PGW-C/SMF 向 OCS 发送 CCR-U 申请配额
   - OCS 回复 CCA-U 返回配额（含 CC-Time 和计费方式）
   - PGW-C/SMF 向 PGW-U/UPF 下发配额和计费方式
   - 用户访问业务，PGW-U/UPF 统计业务访问时长

3. 用户去活
   - 用户终止业务，UE 发起终止会话请求
   - PGW-C/SMF 通知 PGW-U/UPF 用户去活
   - PGW-U/UPF 上报业务统计结果
   - PGW-C/SMF 向 OCS 发送 CCR-T 汇报配额使用情况
   - OCS 回复 CCA-T 并扣除费用
```

**OCS CCA-U 消息中的关键信元**：

| 信元 | 说明 |
|------|------|
| CC-Time | 携带用户业务对应的时间配额 |
| Quota-Consumption-Time | 值为 0 表示连续时长计费，不为 0 表示 QCT 计费方式 |
| Time-Quota-Mechanism - Time-Quota-Type | 携带 CTP 或 DTP 计费方式 |
| Time-Quota-Mechanism - Base-Time-Interval | 控制时长配额的基本时间间隔 |

**优先级规则**：如果同时携带 QCT 和 TQM，则 TQM 优先；如果没有携带 QCT 和 TQM，则根据 PGW-C/SMF 上配置的 QCT 值决定。

### 2. 离线时长计费原理

离线时长计费是指 SGW-C/PGW-C/SMF 产生的话单中记录时间信息，并将该话单上报给计费系统 BS。

#### 2.1 离线连续时长计费（QCT=0）

| 维度 | 说明 |
|------|------|
| 计费特征 | QCT=0，连续时长统计 |
| 配置来源 | SGW-C/PGW-C/SMF 上 ADD OFCSRVTEMPLATE 命令中参数 TQM 设置为 QCT 且 QCT 时长为 0 |

**时长统计方法**：
- **基于业务的计费**：从 PGW-U/UPF 首次收到业务数据包时开始统计，直到业务终止时结束（如 RG1 和 RG2 的业务统计时长）。
- **基于用户的计费**：从用户激活上线时开始统计，直到用户下线去激活时结束，无论用户是否进行业务，全部时长均作为有效计费时间（如 User 的业务统计时长）。

> 注意与在线连续时长计费的差异：在线模式下从"收到 OCS 授权配额"开始计时（即使无业务数据也计时），离线模式下从"首次收到业务数据包"开始计时。

#### 2.2 离线 QCT 时长计费（QCT!=0）

| 维度 | 说明 |
|------|------|
| 计费特征 | QCT!=0，通过计算两个数据包间的空闲时间统计时长 |
| 配置来源 | SGW-C/PGW-C/SMF 上 ADD OFCSRVTEMPLATE 命令中参数 TQM 设置为 QCT 且 QCT 时长不为 0 |

**时长统计方法**：同在线 QCT 时长计费方法（QCT 定时器机制一致）。

#### 2.3 离线时长计费流程

```
1. 用户激活
   - 用户发送会话建立请求
   - SGW-C/PGW-C/SMF 向 PGW-U/UPF 发送 PFCP Session Establishment Request
     （携带时间配额和计费方式）
   - PGW-U/UPF 返回 PFCP Session Establishment Response
   - SGW-C/PGW-C/SMF 给用户返回会话建立响应

2. 用户访问业务
   - 用户开始访问业务
   - PGW-U/UPF 统计业务访问时长
   - 时间配额用完时，PGW-U/UPF 向 SGW-C/PGW-C/SMF 上报统计时长
   - SGW-C/PGW-C/SMF 回复 PFCP Session Report Response
   - 满足话单产生条件时，SGW-C/PGW-C/SMF 向 CG 发送话单
   - CG 返回响应

3. 用户去活
   - 用户终止业务，UE 发起终止会话请求
   - SGW-C/PGW-C/SMF 通知 PGW-U/UPF 用户去活
   - PGW-U/UPF 上报业务统计结果
   - SGW-C/PGW-C/SMF 向 CG 发送话单
   - CG 返回成功响应
```

**PFCP Session Establishment Request 中的关键信元**：

| 信元 | 说明 |
|------|------|
| CC-Time | 携带用户业务对应的时间配额 |
| Quota-Consumption-Time | 值为 0 或不为 0，表示连续时长计费或 QCT 计费方式 |
| Time-Quota-Mechanism - Time-Quota-Type | 携带 CTP 或 DTP 计费方式 |
| Time-Quota-Mechanism - Base-Time-Interval | 控制时长配额的基本时间间隔 |

### 3. 融合时长计费原理

融合计费中，SMF 根据 CHF 下发的计费事件（Triggers）与 CHF 进行交互。

**交互信元**：
- CHF 下发时长配额：`Charging Data Request > MultipleUnitInformation > grantedUnit > time`
- SMF 上报时长用量：`Charging Data Request > MultipleUnitUsage > UsedUnitContainer > time`

融合计费使用**连续时长方式**统计时长，带配额管理与不带配额管理的业务时长统计起止时间不同：

| 模式 | 开始时间 | 结束时间 | 说明 |
|------|----------|----------|------|
| 带配额管理 | UPF 首次收到授权配额成功时 | 用户去激活或配额管理实例终止 | 无论用户是否进行该配额业务，全部时长均为有效计费时间 |
| 不带配额管理 | UPF 首次收到业务数据包时 | 业务终止 | 仅统计有业务数据传输的时间段 |

**基于用户的融合计费**：使用时长从用户激活上线时开始统计，直到用户下线去激活时结束，无论用户是否进行业务，全部时长均作为有效计费时间。

**离线 RG 的时长统计方式可修改**：
- UNC 可通过命令的 `OFLRGTIMECALC` 参数控制 N4 接口 Measurement Information 信元中 `ISTM` 的取值。
- 当 ISTM=1 时，统计方式改为与带配额时长统计方式相同（从 UPF 首次收到授权配额成功时开始）。

---

### 三种计费模式的时长统计对比

| 对比维度 | 在线计费 | 离线计费 | 融合计费 |
|----------|----------|----------|----------|
| 连续时长起算点 | UPF 首次收到 OCS 授权配额 | UPF 首次收到业务数据包 | 带配额：收到授权配额；不带配额：收到业务数据包 |
| QCT 配置命令 | ADD DCCTEMPLATE | ADD OFCSRVTEMPLATE | CHF 下发 |
| 配额来源 | OCS（CCA-U） | 本地配置 | CHF（MultipleUnitInformation） |
| 用量上报对象 | OCS（CCR-U/CCR-T） | CG（话单） | CHF（MultipleUnitUsage） |
| QCT 优先级 | OCS > 本地配置 | 仅本地配置 | CHF 下发 |
| 话单/报告 | CCR/CCA (Diameter) | DataRecord Transfer (FTP) | Charging Data Request (N40) |

## 配置规则

### 时长计费的关键 MML 参数

时长计费与流量计费的配置结构完全相同（URR -> URRGROUP -> PCCPOLICYGRP），核心差异在于 URR 的统计类型参数。

#### URR 命令中时长相关的参数

| 参数 | 说明 | 取值 |
|------|------|------|
| USAGERPTMODE | 使用量上报模式 | ONLINE（在线）/ OFFLINE（离线） |
| OFFMETERINGTYPE | 离线计费统计类型 | **DURATION**（按时长）/ VOLUME（按流量）/ FREE（免费） |
| ONLMETERINGTYPE | 在线计费统计类型 | **DURATION**（按时长）/ VOLUME（按流量）/ FREE（免费） |

> 时长计费时，OFFMETERINGTYPE 或 ONLMETERINGTYPE 设置为 DURATION。

#### 时长计费方式相关的 UNC 端参数

| 命令 | 参数 | 说明 |
|------|------|------|
| ADD DCCTEMPLATE | QCT | 在线计费模板中的 QCT 值，0=连续时长，非0=QCT时长 |
| ADD OFCSRVTEMPLATE | TQM | 离线计费模板中的时间配额机制，设置为 QCT 并指定 QCT 时长 |
| SET USRPROFCHARGE | RGAPPLIED | 融合计费场景下 RG 应用方式，DEFAULT 时注意冲突 |
| SET APNCHARGECTRL | RGAPPLIED | 融合计费场景下 RG 应用方式 |
| ADD CHARGEMETHOD | RGAPPLIED | 融合计费场景下 RG 应用方式 |
| UNC 命令 | OFLRGTIMECALC | 控制离线 RG 时长统计方式（ISTM 字段） |

### 配置一致性规则

| 一致性要求 | 说明 |
|------------|------|
| URRID 一致 | SMF 和 UPF 上的 URRID 必须一致 |
| USAGERPTMODE 一致 | SMF 和 UPF 上的使用量上报模式必须一致 |
| OFFMETERINGTYPE 一致 | SMF 和 UPF 上的离线计费统计类型必须一致 |
| ONLMETERINGTYPE 一致 | SMF 和 UPF 上的在线计费统计类型必须一致 |
| RULENAME 一致 | SMF 和 UPF 上的规则名称必须一致 |
| RULE 下 POLICYTYPE/POLICYNAME 一致 | SMF 和 UPF 上规则绑定的策略类型和名称必须一致 |
| USERPROFILENAME 一致 | PCF、SMF、UPF 间用户模板名称必须一致 |

### URRGROUP 配置规则

- URRGROUP 下必须配置在线 URR 和离线 URR，只配置一种计费模式且计费模式和用户的计费模式不匹配时无法计费。
- **融合计费特殊场景**：当 SMF 使用融合计费时，SET USRPROFCHARGE / SET APNCHARGECTRL / ADD CHARGEMETHOD 命令的 RGAPPLIED 配置为 DEFAULT 时，URRGROUP 下**不能同时配置离线和在线 URR**，否则会导致计费冲突。

### 配置操作流程

内容计费/时长计费的配置步骤如下：

```
步骤 1: 配置 URR（ADD URR） — 设置计费方式和统计类型
步骤 2: 配置 URR 组（ADD URRGROUP） — 绑定上下行 URR
步骤 3: 配置 PCC 策略组（ADD PCCPOLICYGRP） — 绑定 URRGROUP
步骤 4: 配置过滤条件（ADD FILTER / ADD FILTERIPV6 / ADD L7FILTER）
步骤 5: 配置流过滤器（ADD FLOWFILTER）
步骤 6: 将过滤器与流过滤器绑定（ADD FLTBINDFLOWF / ADD PROTBINDFLOWF）
步骤 7: 配置规则（ADD RULE） — 绑定 PCC 策略组和流过滤器
步骤 8: 添加用户模板（ADD USERPROFILE）
步骤 9: 绑定规则到用户模板（ADD RULEBINDING）
步骤 10: 配置缺省 URR 组（SET URRGRPBINDING）
步骤 11: 配置全局缺省 URR 组（SET SPECTRAFURRGRP）
步骤 12: 配置防欺诈 URR 列表（ADD SPECURRGRPLIST）
步骤 13: 刷新过滤器（SET REFRESHSRV） — 使配置生效
```

### 相关 MML 命令清单

| 命令 | 用途 |
|------|------|
| ADD URR | 配置使用量上报规则（含统计类型） |
| ADD URRGROUP | 配置使用量上报规则组 |
| ADD PCCPOLICYGRP | 配置 PCC 策略组 |
| ADD FILTER | 配置 IPv4 三四层过滤器 |
| ADD FILTERIPV6 | 配置 IPv6 三四层过滤器 |
| ADD L7FILTER | 配置七层过滤器 |
| ADD FLOWFILTER | 配置流过滤器 |
| ADD FLTBINDFLOWF | 将过滤器与流过滤器绑定 |
| ADD PROTBINDFLOWF | 将七层过滤器、协议与流过滤器绑定 |
| ADD RULE | 配置规则 |
| ADD USERPROFILE | 添加用户模板 |
| ADD RULEBINDING | 将规则与用户模板绑定 |
| SET URRGRPBINDING | 配置用户模板的缺省 URR 组 |
| SET SPECTRAFURRGRP | 配置全局缺省 URR 组 |
| ADD SPECURRGRPLIST | 配置防欺诈 URR 列表 |
| SET REFRESHSRV | 刷新过滤器使配置生效 |

## 配置案例

### 按时长计费的 URR 配置示例

以下展示按时长计费的 URR 配置（与按流量计费的配置结构相同，仅统计类型参数不同）：

```mml
// ===== 时长计费 URR 配置 =====
// 离线时长计费 URR — 注意 OFFMETERINGTYPE=DURATION
ADD URR:URRNAME="urr_offline_duration", URRID=12500, USAGERPTMODE=OFFLINE, OFFMETERINGTYPE=DURATION;

// 在线时长计费 URR — 注意 ONLMETERINGTYPE=DURATION
ADD URR:URRNAME="urr_online_duration", URRID=2300, USAGERPTMODE=ONLINE, ONLMETERINGTYPE=DURATION;

// ===== URRGROUP 配置 =====
ADD URRGROUP:URRGROUPNAME="UrrGp_duration",
  UPURRNAME1="urr_offline_duration", DOWNURRNAME1="urr_offline_duration",
  UPURRNAME2="urr_online_duration", DOWNURRNAME2="urr_online_duration";

// ===== PCC 策略组 =====
ADD PCCPOLICYGRP:PCCPOLICYGRPNM="PPG_duration", URRGROUPNAME="UrrGp_duration";
```

  MSSTARTPORT=0, MSENDPORT=65535, SVRSTARTPORT=0, SVRENDPORT=65535;

  UPURRNAME1="urr_offline_IMS", DOWNURRNAME1="urr_offline_IMS",
  UPURRNAME2="urr_online_IMS", DOWNURRNAME2="urr_online_IMS";

// ===== 场景3: 基础业务(any) =====
