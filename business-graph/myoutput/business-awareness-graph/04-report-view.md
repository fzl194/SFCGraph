# 业务感知业务图谱展示视图

本文件用于后续沉淀展示视角的成稿。

## 一、业务图谱是什么

业务图谱不是命令清单，也不是特性目录，而是把一个业务域中的核心业务对象组织成：

```text
业务场景
-> 参考范式
-> 交付方案
-> 工程任务
-> 运行链
-> 支撑约束
-> 向下桥接到产品能力与配置对象
```

的一种知识结构。

业务图谱重点不是“某条命令怎么配”，而是回答：

1. 这个业务域里到底有哪几类业务问题
2. 每类业务问题通常靠什么机制解决
3. 这些机制如何映射到 UDG/UPF 与 UNC/SMF
4. 这张图为什么比文档目录更接近一线工程和数字员工真正需要的知识组织方式

## 三、为什么用“业务感知”做例子

基于第一批原始语料，`业务感知` 这个域天然适合拿来说明“业务图谱长什么样”，因为它同时具备：

1. 清晰的业务定义
2. 明确的运行过程
3. 多类稳定业务场景
4. 明显的 UDG/UPF 与 UNC/SMF 分工
5. 从业务问题到控制面、用户面的完整桥接链路

原始语料中，业务感知被定义为：

- 在用户会话过程中解析报文
- 区分用户使用的不同业务
- 对不同业务执行不同的策略控制和计费控制

因此它不是单一特性，而是一个完整业务域。

## 四、业务感知业务图谱总览

### 1. 总体结构

```text
BusinessDomain: 业务感知
  -> 一级业务场景
  -> 次一级机制
  -> UDG/UPF 侧落点
  -> UNC/SMF 侧落点
  -> 运行验证与约束
```

### 2. 一级业务场景

当前第一版建议保留 4 类一级场景：

1. `差异化计费与免流`
2. `配额耗尽后的体验控制`
3. `访问限制与阻断`
4. `带宽控制与体验分层`

### 3. 次一级机制

这些不是一级场景，但会横向出现在多个场景中：

1. `默认兜底策略`
2. `重定向`
3. `优先级裁决`
4. `缺省URR组绑定`

## 五、业务感知为什么能形成图谱

基于原始语料，业务感知不是离散知识点，而是一条稳定主链：

```text
PCF下发规则
-> SMF翻译/承接策略
-> UPF解析与识别业务流
-> 规则匹配
-> 执行绑定策略
-> 数据继续转发
-> Usage/计费信息上报
```

这条主链说明：

- 业务感知同时有静态对象和动态运行链
- 业务图谱必须同时表达：
  - “有哪些业务场景”
  - “这些场景如何在网络中真实生效”

## 七、UDG/UPF 与 UNC/SMF 的分工

### UDG/UPF 侧

更偏：

- 识别
- 匹配
- 执行
- 统计

典型落点：

- `SA-Basic`
- `Filter / L7Filter / FlowFilter`
- `Rule`
- `PCC动作`
- `带宽控制执行`
- `Usage上报`

### UNC/SMF 侧

更偏：

- 策略承接
- 规则编排
- 计费组织
- 配额控制
- 动态更新

典型落点：

- `SM策略 / PCC规则`
- `内容计费`
- `融合计费`
- `Trigger`
- `URR -> URRGroup -> PCCPolicyGrp -> Rule -> UserProfile`

## 八、几类一级场景如何理解

### 1. 差异化计费与免流

核心问题：

- 不同业务是否采用不同费率
- 特定业务是否免费

核心机制：

- 业务识别
- Rule 匹配
- PCC/URR 计费链

### 2. 配额耗尽后的体验控制

核心问题：

- 配额耗尽后是阻断、重定向还是继续放通

核心机制：

- 在线计费
- 配额管理
- Trigger 驱动策略更新
- 缺省动作控制

### 3. 访问限制与阻断

核心问题：

- 哪些业务可以访问
- 哪些业务要限制或阻断

核心机制：

- 三四层/七层识别
- Rule 优先级
- 阻断或策略动作执行

### 4. 带宽控制与体验分层

核心问题：

- 哪些业务应限速
- 哪些业务应保障体验

核心机制：

- 业务识别
- 用户/用户组/整机三级作用域
- BWM 策略控制

## 九、为什么这张图比文档目录更有价值

因为产品文档按：

- 专题
- 特性
- 命令
- 配置步骤

组织。

而业务图谱按：

- 业务问题
- 参考范式
- 交付方案
- 工程任务
- 运行链
- 控制面/用户面分工

组织。

这样更容易看到：

- 业务感知能解决哪些类问题
- 这些问题背后有没有稳定方法论
- 后续是否能沉淀成数字员工可用知识底座

## 十、当前版本边界

这版仍然是第一版展示骨架，不是最终知识库。

当前有意控制了展开深度：

- 不下钻每条命令和每个参数
- 不做全量场景覆盖
- 不把所有配置对象都拉进主图

当前重点是先把：

```text
业务图谱是什么
+ 业务感知这个域长什么样
+ 为什么它能作为业务图谱样板
```

讲清楚。

## 当前表达方向

展示版不按：

- CSV 表
- 命令清单
- 参数清单

来讲。

而按下面三层来讲：

1. `业务图谱是什么`
2. `业务感知这个领域长什么样`
3. `业务感知如何从业务场景一路连接到 UDG/UPF 与 UNC/SMF`
