# 业务感知业务图谱 v2 迭代计划

## 1. 目标

本文件不是图谱正文，而是 `v2` 的迭代执行计划。

本轮要完成的事情只有两步：

1. 先把 `v2` 的阅读批次、更新机制、输出方式定清楚。
2. 待审核通过后，再按批次逐个阅读原始语料，并在每一批结束后立即更新图谱。

## 2. 输出文件约定

审核通过后，实际执行时采用以下文件约定：

1. 保留原文件不动：
   - `myoutput/business-awareness-graph/05-business-awareness-business-graph-v1.md`
2. 当前已生成 `v2` 正文：
   - `myoutput/business-awareness-graph/06-business-awareness-business-graph-v2.md`
3. `v2` 的最初版本处理方式：
   - 先拷贝 `v1`
   - 再在拷贝后的 `v2` 上逐批迭代
4. 从下一次迭代开始，改为“逐版拷贝”机制：
   - 每进入一个新批次，不直接在上一版正文上继续改
   - 而是先拷贝上一版，生成一个新的版本文件，再在新版本上修改
5. 版本推进规则固定为：
   - `v1 -> v2 -> v3 -> v4 ...`
   - 每一版都保留，便于回看每轮融合后的差异
6. 因此：
   - `T01` 的结果保留在 `06-business-awareness-business-graph-v2.md`
   - 下一批 `T02` 的结果应输出到新的 `v3` 文件中

## 3. 迭代机制

### 3.1 总原则

1. 不一次性读完所有文档。
2. 必须按 task 顺序执行。
3. 每个 task 阅读文档数 `不超过 10 个`。
4. 每读完一批，必须先更新一次当前版本正文，再做一次整体融合。
5. 没有完成“阅读一批 -> 更新一次图谱 -> 整体融合”之前，不进入下一批。
6. 从 `T02` 开始，每一批更新都必须落到一个新的版本文件中，而不是覆盖上一版。

### 3.2 单个 task 的固定动作

每个 task 固定按下面 5 步执行：

1. 读取本 task 规定的原始文档。
2. 记录本批新增结论：
   - 新增了哪些业务定义
   - 修正了哪些 schema 含义
   - 新增或修正了哪些场景
   - 新增或修正了哪些 `ReferencePattern / DeliverySolution / RuntimeFlow / RiskConstraint / Evidence`
3. 如果本批不是首批，就先拷贝上一版正文，生成新的版本文件。
4. 立即更新当前批次对应的新版本正文。
5. 以本批原始语料为主证据，对当前版本做一次整体融合：
   - 允许删除重复表达
   - 允许合并重叠对象
   - 允许重命名不准确的抽象
   - 允许重排章节与主图，只要能更准确表达图谱
   - 不允许只基于旧版 md 闭门改写，必须回到本批原始文档核对
6. 在当前版本文档中追加一条本批更新记录，说明：
   - 本批读了哪些文档
   - 本批改了图谱的哪些部分
   - 本批完成了哪些整体融合动作
   - 哪些结论仍待后续批次压实

### 3.3 整体融合阶段

整体融合不是“再补几条新增结论”，而是对当前 `v2` 做一次全局审视。

每一批吸收完成后，至少要从以下 4 个角度回看一次：

1. `定义是否更准确`
   - 本批原始文档是否改变了“业务图谱是什么”或“业务感知是什么”的表述重心
2. `抽象是否有重叠`
   - 是否出现两个 schema 实例表达同一层含义
   - 是否出现“为了吸收新知识而堆名词”的情况
3. `主链是否更清楚`
   - `NetworkScenario -> ReferencePattern -> DeliverySolution -> RuntimeFlow`
     是否需要收紧、合并或改名
4. `主图是否需要改写`
   - 完整主图必须反映融合后的结果，而不是只在局部章节里新增内容

### 3.4 每批更新的最小要求

每批更新时，至少检查并按需更新下列内容：

1. `业务图谱定义` 是否需要更精确
2. `schema 定义表` 是否需要补充字段业务含义
3. `schema 关系表` 是否需要修正
4. `业务感知 NetworkScenario` 是否需要增删或重命名
5. `ReferencePattern / DeliverySolution` 是否需要调整
6. `RuntimeFlow / ValidationPlan / DiagnosisRule / RiskConstraint / Evidence` 是否需要补充
7. `UDG/UPF` 与 `UNC/SMF` 的分工是否需要修正

其中第 `6` 项不只是“补充”，还要检查是否需要对已有运行链进行整体收敛。

### 3.5 中断恢复机制

为保证中断后可继续执行，执行时必须遵守以下规则：

1. 每个 task 处理完成后，在本文件对应 task 后标记状态。
2. 状态固定为：
   - `pending`
   - `in_progress`
   - `completed`
3. 每个版本正文中都必须有“迭代记录”区，记录本版承接的上一版来源、已吸收的语料和修改点。
4. 后续恢复时，只从第一个 `pending` task 继续，不重读已完成批次。
5. 恢复执行时，默认从最近一个已完成版本拷贝出下一版，再进入下一批。

## 4. task 拆分原则

task 的拆分遵循以下规则：

1. 按 `myoutput/business-awareness-discovery/02-business-description-candidates.md` 的文件出现顺序来拆。
2. 先处理主链定义与主链场景，再处理方案，再处理风险验证，再处理背景与能力摘要。
3. 同类材料尽量放在同一个 task 中。
4. 单个 task 的文档数严格控制在 `10 个以内`。
