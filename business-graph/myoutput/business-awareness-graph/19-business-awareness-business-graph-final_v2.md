# 业务图谱定义与业务感知业务图谱

## 1. 文档目标

本文只回答两个问题：

1. `业务图谱的定义是什么`
2. `业务感知对应的业务图谱是什么`

因此本文分成两部分：

### Part A

先定义：

- 什么叫业务图谱
- 业务图谱有哪些 schema
- schema 之间是什么关系

### Part B

再实例化：

- 在 `BusinessDomain = 业务感知` 下
- 这些 schema 分别对应什么内容
- 它们如何组成一张完整的业务感知业务图谱

## 2. 原始语料范围

本文结论来自七组主题化原始语料，而不是来自某一轮中间稿：

1. `定义与原理语料`
   - 业务感知定义、过程、相关概念
   - SA 识别、协议识别、协议特征库、协议解析库
   - 策略、规则、过滤条件
2. `场景与样板语料`
   - 业务感知场景举例
   - 计费、带宽控制、访问限制套餐样板
   - 七层/三四层阻塞、重定向、用户 Portal
3. `分流与边缘卸载语料`
   - Why/What/How 分流
   - 基于预定义规则、位置、动态规则、漫游地动态签约的分流策略控制
   - 公网私网业务独立计费
4. `计费与配额主线语料`
   - 内容计费、融合计费、计费解决方案概述
   - 标准计费流程、计费事件触发条件、计费用量搜集、多 UPF 配额管理
   - 一望5G 计费解决方案总览
5. `PCC 与 QoS 编排语料`
   - PCC 基本功能、PCC 静态规则原理
   - E2E QoS 管理机制、QoS 生成/下发/更新
   - PCF 发起和 SMF 发起的策略更新
6. `调测与故障语料`
   - UDG 业务感知调测、七层/三四层阻塞与重定向调测
   - UNC 计费故障案例、欠费重定向、default quota、融合计费/内容计费调测
7. `产品能力与边界语料`
   - UNC/UDG 产品描述、5G Core 解决方案、UDG 典型组网
   - UDG License 资源控制项

这些原始材料共同支撑本文中的定义层、场景层、方案层、运行层、支撑层和边界层结论。

## Part A. 业务图谱的定义

## 3. 什么是业务图谱

业务图谱不是：

- 功能目录
- 特性目录
- 命令目录
- 配置步骤索引

业务图谱是：

```text
把某一业务域中的“业务问题空间、参考范式、方案闭包、工程任务、运行链、支撑约束、
以及它们与下层能力/配置世界的连接关系”
组织成一套结构化对象网络。
```

对 `业务感知` 这个业务域来说，它本质上是“对用户数据报文进行解析，从而区分不同业务，并据此实施策略控制和计费控制”的运行型能力域。

业务感知不仅是“识别业务”，也是对流量进行经营的一套方法。它最终服务的是计费精细化、体验分层、安全防护和灵活运营。

这套定义不是 UDG 单侧的叙述，而是控制面与用户面共享的业务语言。也就是说，业务感知图谱从一开始就应该按 `PCF/SMF 编排` 与 `UPF 执行` 的双侧结构来表达。

因此，业务感知图谱不能只并列列出若干场景，还必须把运行主链作为骨架保留下来。

业务感知图谱至少需要同时保留两条组织轴线：

1. `运行轴`
   - 规则建立
   - 报文到达
   - 解析识别
   - 规则匹配
   - 策略执行
   - 数据转发
   - 使用情况上报
2. `业务问题轴`
   - 计费场景
   - 带宽控制场景
   - 访问限制场景
   - 本地分流场景

如果只保留其中一条，图谱都会失真：

- 只有运行轴，会退化成过程说明
- 只有业务问题轴，会退化成场景目录

原始语料里的一级场景名称已经比较稳定，最直接的来源就是 `套餐1：计费场景`、`套餐2：带宽控制`、`套餐3：访问限制场景` 以及 `Why/What/How分流`、`基于预定义规则的分流策略控制` 这一组本地分流材料。

因此，本文在一级场景层尽量直接采用原始语料中的场景名称，而不再把 `配额耗尽后`、`QoS`、`重定向` 这类动作、机制或结果单独上升为一级场景名。

`计费场景` 不能只理解成“识别出业务后挂一个费率”。它背后还存在一条稳定的 `计费会话创建 -> Trigger驱动更新 -> 用量搜集 -> 配额重授权/会话释放` 闭环；而“配额耗尽后重定向”在原始语料中也主要作为计费闭环的一部分出现。

`带宽控制场景` 也不能只理解成“在UPF上配一个限速动作”。它背后同样存在一条稳定的 `PCF生成SM策略 -> SMF翻译并建立QoS Flow -> UPF/RAN/UE执行映射 -> PCF或SMF按事件更新策略` 的控制面-用户面编排链，因此 `QoS` 更适合作为该场景的关键实现机制。

还需要把另一个边界收紧：QoS 编排链只对 `NS-02` 成立，不能直接外推成所有业务感知方案的统一运行链；本地分流更强调 `策略来源 -> UL CL/辅锚点选择 -> 会话修改 -> 分流生效`，计费主线更强调 `计费会话建立 -> Trigger驱动更新 -> 用量上报 -> 配额/会话处理`，访问限制更强调 `NS-03` 的 UDG 用户面验证与诊断。因此，统一运行链需要收紧到更中性的“规则/会话建立 + 按场景建立执行通路 + 结果上报与按需更新”表达。

计费主线不仅有“怎么生效”，还需要明确“哪里最容易出错、出错后先查哪里、哪些告警和控制点最关键”。稳定的排障骨架至少包含四类入口：

1. `对象一致性`
   - 如 `CP/UP URR配置不一致`
2. `RG与计费方式规划`
   - 如 `用户计费方式/RG计费方式不符合预期`
   - `预申请配额未携带预期RG`
3. `Trigger与会话更新`
   - 如 `Trigger未上报`
   - `default quota`
4. `N40/CHF交互与运行结果`
   - 如 `业务被放通未上报CHF`
   - `N40接口计费流量与实际流量不一致`

因此，最终图谱把一级场景收敛为 `计费场景 / 带宽控制场景 / 访问限制场景 / 本地分流场景`，并将 `配额耗尽后动作`、`QoS编排`、`重定向`、`独立计费` 纳入这些场景下的方案层与支撑层。

业务感知之所以能成为一个稳定业务域，不只是因为它有 `Filter / Rule / PCC / URR` 这条对象链，还因为 5G Core 的产品能力版图为它提供了持续成立的结构条件。其成立依赖同一组稳定前提：

1. `SBA 与控制面/用户面解耦`
   - 策略、会话、计费与执行职责被稳定分布在 `PCF / SMF / UPF`
2. `分布式部署与边缘计算`
   - 用户面既可以位于中心，也可以位于边缘，从而使本地分流、边缘卸载和独立计费成为正式产品能力
3. `计费、QoS、业务感知、智能策略控制的并列能力版图`
   - 业务感知不是孤立特性，而是与计费、QoS、PCC、分布式网络一起构成可组合能力
4. `资源与License边界`
   - 吞吐、会话数、承载数以及特性能力开关会直接约束业务感知方案能否按预期落地

因此，业务感知图谱除了保留 `运行轴` 与 `业务问题轴`，还必须承认一条更上层的 `产品结构支撑轴`：如果没有 `SBA + CP/UP分工 + 分布式用户面 + 资源/License边界`，很多图谱对象都只能停留在概念上，不能稳定成为产品能力。

业务感知业务图谱中的基础“运行链”可以先收紧为：

```text
规则建立
-> 报文到达
-> 解析识别
-> 规则匹配
-> 策略执行
-> 数据转发
-> 使用情况上报
```

也就是说，业务图谱不仅要回答“业务问题如何分类”，还要回答“业务是如何在运行时被建立、识别、命中、执行和回传统计结果的”。

也就是说，业务图谱描述的不是“怎么敲命令”，而是：

1. 这个业务域里有哪些稳定业务问题
2. 这些业务问题通常如何归类
3. 每类问题通常依赖哪些业务机制
4. 这些机制如何收敛为方案与工程任务
5. 它们如何向下连接到控制面、用户面、特性、配置对象

如果一个知识结构无法回答这五个问题，那它更像资料目录，不像业务图谱。

## 4. 业务图谱的 schema

业务图谱的 schema 分三层：

1. `业务层核心 schema`
2. `支撑与约束 schema`
3. `向下桥接 schema`

### 4.1 schema 总表

| schema对象 | 中文对象名 | 所属层 | 定义 | 在图谱中的作用 | 需要重点展开的业务含义 |
| --- | --- | --- | --- | --- | --- |
| `BusinessDomain` | 业务域 | 业务层 | 表示一类长期存在的工程知识空间 | 作为整张图谱的根对象，限定讨论边界 | 不是某篇文档，而是一类稳定业务空间 |
| `NetworkScenario` | 现网场景 | 业务层 | 表示一线遇到的需求类型或问题情境 | 作为图谱的一级业务入口 | 不是命令，不是参数，而是“现场到底在解决什么问题” |
| `ReferencePattern` | 产品参考范式 | 业务层 | 表示产品文档中可复用的跨特性参考样板 | 为场景和方案提供直接依据 | 不是文档标题搬运，而是从文档中抽出的样板知识 |
| `DeliverySolution` | 交付方案 | 业务层 | 表示针对某个场景形成的可落地方案闭包 | 把场景推进到“怎么组织落地” | 要表达一组机制如何一起工作 |
| `EngineeringTask` | 工程任务 | 业务层 | 表示方案中的可执行或可检查任务 | 把方案拆成可执行工作项 | 要说明规划什么、绑定什么、验证什么 |
| `Participant` | 参与方 | 支撑层 | 表示用户、网元、外部系统、计费系统等 | 说明方案涉及哪些参与方 | 谁负责控制、谁负责执行、谁负责计费 |
| `Scope` | 生效范围 | 支撑层 | 表示用户、DNN/APN、切片、区域、套餐等 | 表达作用域和控制粒度 | 用户级、用户组级、APN/DNN、切片等差异 |
| `DecisionPoint` | 决策点 | 支撑层 | 表示方案中需要判断或选择的点 | 表达同一场景下的不同分支 | 动态/本地PCC，阻断/重定向/继续转发 |
| `RuntimeFlow` | 运行流程 | 支撑层 | 表示业务运行时的跨网元流程 | 连接静态配置与动态生效过程 | 规则如何下发、命中、执行和更新 |
| `ValidationPlan` | 验收计划 | 支撑层 | 表示如何确认业务生效 | 提供验收路径 | 看什么现象、查什么链路、判定什么结果 |
| `DiagnosisRule` | 诊断规则 | 支撑层 | 表示失败后如何回查 | 提供定位路径 | 哪个环节出问题应先看哪里 |
| `RiskConstraint` | 风险约束 | 支撑层 | 表示规格、限制、互斥、性能影响 | 表达边界和风险 | 性能下降、配置不一致、能力不支持 |
| `Evidence` | 证据来源 | 支撑层 | 表示文档页、命令说明、流程、告警、话单、抓包等证据来源 | 保证图谱可追溯 | 每个结论最终都能回到原始文档 |
| `DomainSemanticObject` | 领域语义对象 | 桥接层 | 表示某个业务域内部用于表达业务含义的语义对象 | 防止业务图谱直接坠落成命令图 | 先把业务问题翻成识别、计费、配额等语义单元 |
| `Feature` | 特性 | 桥接层 | 表示产品文档中的特性条目 | 承接业务语义背后的能力载体 | 说明哪类正式能力在支撑业务场景 |
| `ConfigObject` | 配置对象 | 桥接层 | 表示 MML 命令创建、修改、引用的对象 | 说明图谱最终会落到哪些对象链 | 不展开所有参数，但要说明对象链长什么样 |

## 5. schema 之间的关系

业务图谱不是一堆对象平铺，而是有固定关系主链。

### 5.1 schema 关系总表

| 关系标识 | 中文关系名 | 起点schema | 终点schema | 业务含义 |
| --- | --- | --- | --- | --- |
| `contains` | 包含 | `BusinessDomain` | `NetworkScenario` | 一个业务域由若干稳定业务场景组成 |
| `has_pattern` | 具有参考方案 | `NetworkScenario` | `ReferencePattern` | 每类场景应有可追溯的样板依据 |
| `instantiated_as` | 实例化为 | `NetworkScenario` | `DeliverySolution` | 一个场景可归纳成一个或多个方案闭包 |
| `decomposes_to` | 分解为 | `DeliverySolution` | `EngineeringTask` | 方案必须拆成可执行任务 |
| `involves` | 涉及 | `DeliverySolution` | `Participant` | 方案总是由多个参与方共同完成 |
| `applies_to` | 作用于 | `DeliverySolution` | `Scope` | 方案总是有明确生效范围 |
| `has_decision` | 包含决策点 | `DeliverySolution` | `DecisionPoint` | 方案总有关键分支要选 |
| `has_runtime_flow` | 具有运行流程 | `DeliverySolution` | `RuntimeFlow` | 方案需要通过运行链真正生效 |
| `validated_by` | 由其验收 | `DeliverySolution` | `ValidationPlan` | 方案需要验收路径 |
| `diagnosed_by` | 由其诊断 | `DeliverySolution` | `DiagnosisRule` | 方案需要异常回查路径 |
| `constrained_by` | 受其约束 | `DeliverySolution` | `RiskConstraint` | 方案总会受边界和风险约束 |
| `uses_semantic_object` | 使用领域语义对象 | `NetworkScenario` | `DomainSemanticObject` | 场景先关联稳定语义对象，再下钻实现 |
| `instantiates_semantic_object` | 实例化领域语义对象 | `DeliverySolution` | `DomainSemanticObject` | 方案把抽象语义落实为可交付的语义组合 |
| `requires_capability` | 需要能力 | `DeliverySolution` | `Feature` | 方案依赖正式能力载体 |
| `realized_by_config` | 由配置对象实现 | `DeliverySolution` | `ConfigObject` | 方案最终要收敛到关键对象链 |
| `supported_by` | 由证据支撑 | `ReferencePattern` / `DeliverySolution` | `Evidence` | 图谱对象和结论都必须能回到原始语料 |

### 5.2 业务主链

```text
BusinessDomain
  -> NetworkScenario
  -> ReferencePattern
  -> DeliverySolution
  -> EngineeringTask
```

含义：

- 先定义业务域
- 再识别这个业务域下有哪些典型场景
- 每类场景有哪些文档样板
- 从样板中归纳出方案
- 再把方案拆成工程任务

### 5.3 支撑关系

```text
DeliverySolution
  -> Participant
  -> Scope
  -> DecisionPoint
  -> RuntimeFlow
  -> ValidationPlan
  -> DiagnosisRule
  -> RiskConstraint
  -> Evidence
```

含义：

- 一个方案不是孤立存在
- 它必须说明由谁参与、对谁生效、如何生效、如何验证、有什么风险

### 5.4 向下桥接关系

```text
NetworkScenario / DeliverySolution
  -> DomainSemanticObject
  -> Feature
  -> ConfigObject
```

含义：

- 场景和方案不会直接跳成命令
- 中间要先经过语义桥，再连接正式特性和配置对象

## 6. 一张抽象的业务图谱

如果不带任何具体业务域，一张抽象业务图谱长这样：

```text
BusinessDomain
└─ 某业务域
   ├─ NetworkScenario
   │  ├─ 场景A
   │  ├─ 场景B
   │  └─ 场景C
   │
   ├─ ReferencePattern
   │  ├─ 样板1
   │  └─ 样板2
   │
   ├─ DeliverySolution
   │  ├─ 方案1
   │  └─ 方案2
   │
   ├─ EngineeringTask
   │  ├─ 任务1
   │  ├─ 任务2
   │  └─ 任务3
   │
   ├─ Participant / Scope / DecisionPoint
   ├─ RuntimeFlow / ValidationPlan / DiagnosisRule
   └─ DomainSemanticObject / Feature / ConfigObject / Evidence
```

这就是“业务图谱的定义”。

## Part B. 业务感知的业务图谱

## 7. `BusinessDomain = 业务感知`

这里的 `BusinessDomain` 对应中文对象名是 `业务域`。

`业务感知` 可直接实例化为一个 `BusinessDomain`。

原因不是“文档里有这个名字”，而是它满足业务域定义：

1. 有明确业务目标
2. 有稳定的“过滤条件 - 规则 - 策略”组织骨架
3. 有稳定运行链
4. 有多类稳定业务场景
5. 有明显的控制面/用户面分工
6. 可向下桥接到特性和配置对象
7. 有稳定的产品结构支撑
   - `PCF / SMF / UPF` 的控制面-用户面分工
   - `UDG/UPF` 可中心部署也可边缘部署
   - `业务感知 / 计费 / QoS / 智能策略控制 / 分布式网络` 可在同一产品能力版图中组合
   - 资源和 License 控制项会对能力开通、会话规模和吞吐边界形成正式约束

因此：

```text
BusinessDomain: 业务感知
```

## 8. 业务感知的 `NetworkScenario`

这里的 `NetworkScenario` 对应中文对象名是 `现网场景`。

业务感知主图收敛为 4 类一级场景。

这 4 类一级场景直接对齐原始语料中的场景名称：

- `计费场景`
- `带宽控制场景`
- `访问限制场景`
- `本地分流场景`

这里的划分原则是：

- 一级场景尽量采用原始语料中的稳定场景名
- `重定向`、`阻塞`、`头增强`、`Portal`、`QoS`、`配额耗尽后动作`、`独立计费` 这些内容保留在场景下的机制层、方案层或结果层

### 8.1 `NetworkScenario` 实例表

| 场景ID | 场景名称 | 业务目标 | 判断基础 | 关键语义 | 典型结果 |
| --- | --- | --- | --- | --- | --- |

## 14. 业务感知的 `DecisionPoint`

这里的 `DecisionPoint` 对应中文对象名是 `决策点`。

最关键决策点为：

- 规则供给方式是 `预定义规则`、`动态规则` 还是 `本地规则`
- 策略类型是 `PCC`、`BWM`、`HEADEN`、`IPREDIR`、`WEBPROXY` 还是 `SMARTREDIRECT`
- 分流触发方式是 `DNN/TAC`、`PRA`、`动态规则+DNAI` 还是 `漫游地 selectedDnn`
- 策略更新由 `PCF发起` 还是 `SMF发起`
- 配额耗尽后是 `BLOCK` / `REDIRECT` / `FORWARD`
- 计费粒度是 `PDU会话计费(普通)` / `业务流计费(内容)` / `QoS Flow计费(漫游)`
- 计费方式是 `在线计费(申请配额)` / `离线计费(不申请配额)`

### 14.1 `DecisionPoint` 实例表

| 决策点 | 可选项 | 触发背景 | 业务影响 |
| --- | --- | --- | --- |
| 规则供给方式选择 | 预定义规则 / 动态规则 / 本地规则 | 规则内容能否预先规划、是否依赖 PCF、网络中是否存在可关联 PCF | 决定规则是在 CP/UP 预先存在后按需启用，还是由 PCF 实时生成下发，或在无 PCF 时走本地兜底 |
| 策略类型选择 | PCC / BWM / HEADEN / IPREDIR / WEBPROXY / SMARTREDIRECT | 业务目标属于计费控制、带宽控制、头增强、IP导向、代理引导还是智能重定向 | 决定规则后续能绑定哪类策略，以及最终落到哪条能力和对象链 |
| 访问导向路径选择 | PCC重定向 / IPREDIR / Portal处理路径 | 访问类场景到底要把用户导向 URL、IP 还是个人门户 | 决定“导向”是一次性重定向，还是带 captive/non-captive 状态切换的持续门户机制 |
| 分流触发方式选择 | 基于DNN/TAC / 基于PRA / 基于动态规则+DNAI / 基于漫游地selectedDnn | 本地分流到底由静态签约、精细位置、外部应用触发还是漫游地即时签约驱动 | 决定本地DN导向是预先规划型方案，还是会话运行中按位置、外部业务或漫游场景动态建立 |
| 策略更新发起方选择 | PCF发起 / SMF发起 | 策略变化来自签约条件变化，还是来自配额、位置、事件上报等运行态变化 | 决定控制面是主动下发新策略，还是由SMF带着事件变化回推PCF重新生成策略 |
| 配额耗尽动作选择 | BLOCK / REDIRECT / FORWARD | 在线 RG 配额耗尽或配额更新异常 | 决定用户体验是中断、跳转还是继续 |
| 计费粒度选择 | PDU会话计费(普通) / 业务流计费(内容) / QoS Flow计费(漫游) | 计费方案需要决定"对谁计费"——是按整个会话统一费率、按业务流差别费率、还是按 QoS Flow 漫游费率 | 决定计费精细度和 RG 划分方式，直接影响 `NS-01` 以及 `DS-05` 独立计费旁支的落地形式 |
| 计费方式选择 | 在线计费(申请配额) / 离线计费(不申请配额) | 计费方案需要决定"是否需要实时信用控制"——在线计费需要申请配额和 Trigger 闭环，离线计费只搜集用量生成话单 | 决定是否需要 CHF 配额管理和 Trigger 驱动的会话更新闭环 |

### 14.2 `DecisionPoint` 字段业务含义

| 字段 | 含义 | 为什么重要 |
| --- | --- | --- |
| `可选项` | 该决策点到底有哪些分支 | 没有分支就不是决策点 |
| `触发背景` | 为什么要做这次选择 | 决定它属于业务规划问题还是运行态问题 |
| `业务影响` | 选不同项会导致什么业务结果 | 决定该决策点在图谱中的重要性 |

## 15. 业务感知的 `RuntimeFlow`

这里的 `RuntimeFlow` 对应中文对象名是 `运行流程`。

业务感知在图谱中必须保留运行链。

本文不把 QoS 编排链直接写成全域统一流程，而是保留一条能够同时覆盖计费、分流、导向和 QoS 的更中性主链：

```text
PCF/SMF按场景生成并下发策略
-> SMF翻译策略并建立规则或会话上下文
-> 按场景建立执行通路（UPF规则安装 / 分流会话修改 / QoS Flow映射）
-> 用户报文到达UPF
-> UPF解析与识别
-> 规则匹配
-> 执行绑定策略
-> 数据转发与业务结果生效
-> Usage/事件上报并按需触发计费会话更新、策略更新或分流会话更新
```

其中：

- `NS-02` 会在“执行通路建立”阶段展开成 `QoS Flow` 建立、`PFS/QFI` 映射和 `PCF/SMF` 发起更新。
- `DS-05` 会在同一阶段展开成 `UL CL/辅锚点UPF` 的选择、插入和会话修改。
- `NS-01` 及其配额动作相关方案则会在最后阶段展开成 `Trigger` 驱动的计费会话更新、配额申请和重授权闭环。

### 15.1 `RuntimeFlow` 实例表

| 流程阶段 | 主要参与方 | 发生的事情 | 业务含义 |
| --- | --- | --- | --- |
| 策略生成与规则建立 | UDM、PCF、SMF、UPF | UDM提供签约基线，PCF按条件生成 SM 策略，SMF 承接并翻译，UPF 建立规则库与执行上下文 | 这一步明确了业务感知不是 UPF 单点能力，而是带签约输入的 CP/UP 协同建立过程 |
| 执行通路建立 | SMF、UPF、AMF、RAN、UE | SMF按场景建立执行通路：普通场景下安装UPF规则；分流场景下选择并插入UL CL/辅锚点UPF、修改PDU会话；QoS场景下建立QoS Flow并下发映射 | 这一步说明不同场景共享同一业务骨架，但在“通路怎么准备”上会展开出不同分支 |
| 报文到达 | 基站、UPF | 用户接入完成并开始会话后，业务报文到达 UPF | 图谱要说明业务动作发生在真实流量到达之后，而不是停留在静态配置层 |
| 解析识别 | UPF | 先做三四层解析，再做协议识别、七层解析和特征提取 | 业务感知首先是“把报文看懂”，然后才谈得上策略 |
| 规则匹配 | UPF | 把识别结果与 Rule 条件比较并按优先级裁决，决定命中哪条业务路径 | 决定最终生效的是计费、配额、导向、分流还是QoS控制 |
| 策略执行 | UPF、RAN、UE | 执行计费、阻断、重定向、带宽控制、头增强、QoS控制、分流转发等动作 | 真正把识别结果翻译成用户可感知的业务处理结果 |
| 数据转发与业务结果生效 | UPF、RAN、业务服务器、本地DN/边缘DN | 命中规则后的数据继续被转发到外部网络、本地DN或业务服务器，并在需要时形成端到端体验保障 | 业务感知不是停在动作选择，而是必须形成真实的转发路径和业务结果 |
| Usage/事件上报与按需更新 | UPF、SMF、PCF、CHF/OCS、CTF、AMF | UPF按 URR/Trigger 上报用量或事件；SMF按场景向CHF更新计费会话、向PCF上报变化事件，或继续推动分流会话/QoS策略更新 | 形成业务识别、计费闭环、策略更新和会话演进的统一回路 |

### 15.2 `RuntimeFlow` 字段业务含义

| 字段 | 含义 | 为什么重要 |
| --- | --- | --- |
| `流程阶段` | 把复杂运行链拆成稳定阶段 | 让图谱能解释业务如何生效，而不是只展示静态对象 |
| `主要参与方` | 每个阶段谁在主导 | 让控制面和用户面分工变清楚 |
| `发生的事情` | 该阶段的关键动作 | 连接业务语言和网络行为 |
| `业务含义` | 这一步为什么对业务重要 | 防止把运行链讲成纯协议过程 |

## 16. 业务感知的 `ValidationPlan`

这里的 `ValidationPlan` 对应中文对象名是 `验收计划`。

业务感知图谱不能只写“怎么配”，还要写“如何判定已经生效”。

同时，需要明确一件事：当前支撑层的依据强度并不均衡。`NS-03` 更强的是 UDG 用户面验证链，`NS-01` 更强的是计费闭环，`NS-02` 更强的是 QoS 更新链，`NS-04` 更强的是分流会话链。因此，下面这些验收项都带 `适用场景`，不按“全域等强覆盖”理解。

### 16.1 `ValidationPlan` 实例表

| 验证主题 | 适用场景 | 观察点 | 原始依据来源 | 验证目标 | 业务含义 |
| --- | --- | --- | --- | --- | --- |
| 差异化计费验证 | `NS-01` | 专项业务流量与其他业务流量是否分开计费 | 内容计费与融合计费调测页 | 确认不同业务流进入不同费率或免费路径 | 证明差异化计费不是概念，而是实际产生了不同计费结果 |
| 免费业务验证 | `NS-01` | 免费业务是否未产生计费 | 融合计费流量计费调测页 | 确认免费业务对应的计费统计类型和开关符合预期 | 证明免流场景真正落地 |
| 配额耗尽动作验证 | `NS-01 / DS-02` | 配额耗尽后是否出现阻断、重定向或继续转发 | 配额动作配置页与场景举例页 | 确认在线计费与动作切换链条生效 | 证明配额动作不是静态配置，而是计费闭环中的运行态切换 |
| Trigger与计费会话验证 | `NS-01 / DS-02` | 会话建立后是否已申请配额、Trigger发生后是否触发 Update/Termination | 标准计费流程与计费事件触发条件页 | 确认计费不是“只统计流量”，而是具备事件驱动的会话闭环 | 证明差异化计费和配额控制背后存在稳定的会话机制 |
| 多UPF配额验证 | `DS-02 / DS-05` | 主/辅锚点UPF是否按各自配额独立运行 | 多UPF配额管理页 | 确认多UPF场景下不是简单共享额度，而是按UPF独享配额 | 证明分流/MEC 场景下的计费边界清楚 |
| 预申请RG验证 | `NS-01 / DS-02 / DS-05中的独立计费旁支` | Charging Data Request [Initial] 中是否携带预期RG，且RG个数与来源符合规划 | 预申请配额未携带预期RG案例 + 带配额管理的融合计费调测页 | 确认用户激活时初始配额申请已覆盖预期RG | 证明计费闭环从会话建立一刻就按规划启动 |
| QoS Flow与策略更新验证 | `NS-02` | QoS Flow是否按预期建立、策略变化后是否完成更新和重映射 | QoS生成/下发/更新页 | 确认体验分层不是纸面参数，而是真实建立了通路和更新链 | 证明带宽控制具备端到端实施能力 |
| 本地分流验证 | `NS-04 / DS-05` | 用户进入指定区域或命中分流条件后，流量是否进入 UL CL/辅锚点路径并访问本地DN | Why/What/How分流与 UL CL 分流特性概述 | 确认本地业务流已从中心路径切换到边缘路径 | 证明本地分流场景已经真实建立边缘导向路径 |
| 带宽控制验证 | `NS-02` | 指定业务或范围对象是否出现预期速率变化 | 带宽控制特性页 | 确认识别条件和带宽动作已经联动 | 证明带宽控制场景已生效 |
| 控制面与用户面一致性验证 | `全域` | Rule、UserProfile、URR 等关键对象是否一致 | UNC/UDG 配置映射页 | 确认 CP/UP 对同一业务理解一致 | 证明业务规则不会因两侧不一致而失效 |
| CP/UP URR一致性告警验证 | `NS-01 / DS-02 / DS-05中的独立计费旁支` | 是否出现 `ALM-81026`、`ALM-81054` 告警以及告警恢复情况 | CP和UP URR配置不一致案例 | 确认计费对象链的一致性已恢复 | 证明计费规则不会因CP/UP URR不一致而失真 |
| 七层阻塞功能验证 | `NS-03` | 终端访问触发URL时，Gi/N6侧是否无HTTP报文，只有三四层报文 | UDG 调测七层阻塞页 | 确认七层识别+阻塞动作链完整生效 | 证明七层访问限制从识别到动作的闭环 |
| 七层重定向功能验证 | `NS-03` | 终端访问触发URL时，是否收到302重定向报文并跳转到目标页面 | UDG 调测七层重定向页 | 确认URL重定向动作替代了阻塞动作 | 证明从阻塞切换到重定向只需修改 PCCActionProp |
| 三四层阻塞功能验证 | `NS-03` | 终端访问目标IP时是否被阻止访问 | UDG 调测三四层阻塞页 | 确认三四层Filter+PCC策略阻塞链生效 | 证明基于IP/协议的三四层访问限制可用 |
| 三四层重定向功能验证 | `NS-03` | 抓包是否看到目的MAC被改为重定向设备MAC（目的IP不变） | UDG 调测三四层重定向页 | 确认IPREDIR策略类型只改MAC不改IP | 证明三四层重定向是MAC层重定向，不是NAT |
| 欠费重定向与重授权验证 | `NS-01 / DS-02` | 配额耗尽后 CCR-Update 的 `Reporting-Reason` 是否为 `QUOTA_EXHAUSTED`，RAR/RAA 与重授权后业务是否恢复 | 调测在线计费欠费重定向功能页 | 确认欠费重定向、充值唤醒和重授权闭环正常 | 证明配额耗尽不是单次阻断，而是可恢复的信用控制流程 |
| Default Quota验证 | `NS-01 / DS-02` | CCR/CCA之间是否已有业务放通，且默认配额流量/时长是否与配置一致 | 调测default quota功能页 | 确认在正式配额返回前，默认配额已按规划生效 | 证明在线计费场景可以避免因等待配额导致新业务被误阻断 |
| 计费可靠性验证 | `NS-01 / DS-02 / DS-05中的独立计费旁支` | CHF主备切换时计费是否不中断，主备均故障时业务是否按预期处理 | 一望5G 计费解决方案总览 | 确认计费可靠性机制实际工作 | 证明计费故障不会导致业务中断或计费丢失 |

### 16.2 `ValidationPlan` 字段业务含义

| 字段 | 含义 | 为什么重要 |
| --- | --- | --- |
| `适用场景` | 这条验收路径主要服务哪些场景 | 避免把某一批调测材料误解成全域通用方法 |
| `验证主题` | 这次验证围绕哪类业务目标展开 | 防止验证动作和场景目标脱节 |
| `观察点` | 需要看什么现象、对象或结果 | 让验证从抽象判断变成可执行检查 |
| `原始依据来源` | 验证方法来自哪类原始材料 | 保证验证路径可追溯 |
| `验证目标` | 这次验证要证明什么 | 决定验证是否闭环 |
| `业务含义` | 验证成功意味着什么 | 把调测动作重新翻译回业务价值 |

## 17. 业务感知的 `DiagnosisRule`

这里的 `DiagnosisRule` 对应中文对象名是 `诊断规则`。

业务感知图谱还要说明“业务没按预期生效时先查哪里”。

和 `ValidationPlan` 一样，这一层也不能按“全域等强覆盖”理解。`NS-03` 更强的是 UDG 侧对象链诊断，计费主线更强的是计费会话、Trigger 和 CHF 选择，`NS-02` 更强的是 QoS 更新链，`NS-04` 更强的是分流会话链。因此，诊断规则也显式挂到对应适用场景上。

### 17.1 `DiagnosisRule` 实例表

| 异常现象 | 适用场景 | 优先检查点 | 原始依据来源 | 诊断逻辑 | 业务含义 |
| --- | --- | --- | --- | --- | --- |
| 业务流未进入专项计费 | `NS-01` | Rule 与 PCCPolicyGrp、URRGroup、URR 绑定链 | 融合计费流量计费调测页 | 先查 Rule 是否绑定策略组，再查策略组是否绑定计费组 | 计费异常常常不是识别失败，而是对象链断裂 |
| 免费业务仍被计费 | `NS-01` | URR 计费统计类型、UserProfile 免费业务开关 | 融合计费流量计费调测页 | 先查 FREE 配置，再查 UserProfile 是否允许免费业务不计费 | 免流异常往往是计费属性没落到模板 |
| 配额耗尽后动作不对 | `NS-01 / DS-02` | QUOTAEXHAUSTACT 配置、Trigger、CHF 交互状态 | 融合计费与配额动作页 | 先查配额动作，再查是否真正收到了额度变化和更新事件 | 动作异常通常与配额闭环有关 |
| Trigger未按预期触发 | `NS-01 / DS-02` | Session/RG Trigger配置、CHF下发Trigger、本地忽略Trigger策略 | Trigger参考页与标准计费流程页 | 先判断是Session级还是RG级触发，再查本地Trigger是否覆盖了CHF下发逻辑 | 计费更新异常往往不是业务没发生，而是触发条件或优先级被配错 |
| 预申请配额未携带预期RG | `NS-01 / DS-02 / DS-05中的独立计费旁支` | `CCT/CHFINIT` 的初始RG个数、RG来源、`CTXSTARTRATING` 与 `USERPROFILE/URRGROUP/URR` 绑定关系 | 预申请配额未携带预期RG案例 | 先查初始RG个数是否足够，再查 RG 来源策略，最后查 CTXSTARTRATING 和对象绑定 | 初始RG错误会让计费闭环从激活时就偏离规划 |
| 用户计费方式或RG计费方式不符合预期 | `NS-01 / DS-02 / DS-05中的独立计费旁支` | `CHARGECTRL / USRPROFCHARGE / APNCHARGECTRL / CHARGEMETHOD / URR.USAGERPTMODE` | 用户计费方式和/或RG计费方式不符合预期案例 | 先看用户级融合计费使能和RGAPPLIED，再看RG自身上报模式是否与用户计费方式一致 | 计费方式错配会让N40会话走错在线/离线路径 |
| 多UPF配额异常 | `DS-02 / DS-05` | 主/辅锚点UPF是否被分别申请配额、UL CL是否被错误纳入配额管理 | 多UPF配额管理页 | 先查是否按 UPF ID 单独向 CHF 申请配额，再查 UL CL 是否被误认为计费点 | 多UPF场景一旦把配额边界弄错，体验控制和计费都会偏离预期 |
| QoS策略更新未生效 | `NS-02` | PCF规则重算、SMF更新请求/响应、QoS Flow重建与映射链 | QoS更新页 | 先判断由PCF还是SMF发起，再查SMF是否真正安装新策略并触发通路更新 | 体验类异常常常不是识别失败，而是控制面更新链断在中途 |
| 本地分流未生效 | `NS-04 / DS-05` | DNN/位置/PRA 条件、UL CL 选择、辅锚点插入、DNAI 或 selectedDnn 传递链 | 分流 How 页与 MEC 相关特性概述 | 先判断分流触发条件是否满足，再查 SMF 是否选择 UL CL/辅锚点并下发分流规则 | 分流异常常常不是识别本身失败，而是触发条件、会话改写或锚点选择链中断 |
| 业务被放通未上报CHF | `NS-01 / DS-02 / DS-05中的独立计费旁支` | `Charging Data Request/Response` 往返、`ALM-100072`、CHF响应码 | 业务被放通未上报CHF案例 | 先查CHF是否无响应，再查N40链路告警，最后查CHF是否返回异常结果码 | 放通但未上报会导致计费缓存甚至漏计费，是高风险闭环故障 |
| 动态规则未生效 | `全域` | PCF 下发、SMF 承接、UPF 执行链 | PCC 基本功能与策略偶联流程页 | 先判断策略是否下发，再判断是否被承接和执行 | 决策链任何一段中断都会导致业务失效 |
| 业务统计不一致 | `全域` | CP/UP 对象命名、RG/URRID 规划、一致性告警 | UNC/UDG 配置映射页 | 先看两侧对象是否一致，再看是否存在告警或缺绑 | 一致性问题会直接破坏业务闭环 |
| CP/UP URR配置不一致 | `NS-01 / DS-02 / DS-05中的独立计费旁支` | `ALM-81026`、`ALM-81054` 及对应不一致配置项 | CP和UP URR配置不一致案例 | 先根据告警直接定位不一致对象，再判断是否为近期配置调整导致 | 这是计费主线里最直接的一致性故障信号 |
| 七层阻塞不生效 | `NS-03` | APN → RuleBinding → Rule(FlowFilter+PCCPolicyGrp) → PCCActionProp(门控) → L7Filter(URL) → Filter(三四层) → 协议特征库加载状态 | UDG 调测七层阻塞页 | 先查终端APN是否正确，再逐层查 RuleBinding→Rule→PCCActionProp→FlowFilter→L7Filter→Filter→特征库，直到找到断裂点 | 七层访问限制是多层对象链，断裂点可能在识别层(Filter/L7Filter)或动作层(PCCActionProp)或底座(特征库) |
| 七层重定向不生效 | `NS-03` | Rule策略组 → PCCActionProp(重定向名+门控) → REDIRECT对象(URL) | UDG 调测七层重定向页 | 先查 Rule 绑定的 PCCPolicyGrp 是否正确，再查 PCCActionProp 的重定向名称和门控状态，再查 REDIRECT 对象的 URL | 重定向不生效常常是 PCCActionProp 未从阻塞改为重定向，或 REDIRECT 对象未配置 |
| 三四层重定向不生效 | `NS-03` | Rule(策略类型=IPREDIR + IPREDIRECTIP) → Filter(协议类型+生效标记) | UDG 调测三四层重定向页 | 先查 Rule 策略类型是否为 IPREDIR，再查 IPREDIRECTIP 是否正确，再查 Filter 生效标记和协议类型 | 重定向异常常常是策略类型未切换或 Filter 未刷新生效 |
| CHF选择异常 | `NS-01 / DS-02 / DS-05中的独立计费旁支` | SMF选择的CHF路径（PCF FQDN / UDM签约CC / NRF发现 / 本地CC） | 一望5G 计费解决方案总览 | 先查 SMF 按哪个优先级选择了 CHF，再查该路径对应的配置是否正确 | CHF选择错误会导致计费会话建立失败或配额管理异常 |
| N40计费流量与实际访问流量不一致 | `NS-01 / DS-02 / DS-05中的独立计费旁支` | 免费业务配置、动态Rule是否指定RG、预定义Rule到RG绑定链、`SPECTRAFURRGRP` | N40接口计费流量与用户实际访问流量不一致案例 | 先排除正常的免费业务和欠费丢弃流量，再查动态/预定义规则是否真正挂到RG | 这类异常常见于“识别了业务，但没把业务流可靠挂到计费链上” |

### 17.2 `DiagnosisRule` 字段业务含义

| 字段 | 含义 | 为什么重要 |
| --- | --- | --- |
| `适用场景` | 该排障路径主要服务哪些场景 | 避免把单个专题的故障链误解成全域统一排障链 |
| `异常现象` | 从现场可观察到的问题出发 | 让图谱先接住真实故障现象 |
| `优先检查点` | 第一批最该查的对象或链路 | 避免排障顺序混乱 |
| `原始依据来源` | 诊断路径来自哪类原始材料 | 保证诊断规则不是拍脑袋 |
| `诊断逻辑` | 为什么先查这些点 | 让规则具备可解释性 |
| `业务含义` | 这个故障本质上破坏了什么 | 把诊断从命令层拉回业务层 |

## 18. 业务感知的 `RiskConstraint`

这里的 `RiskConstraint` 对应中文对象名是 `风险约束`。

最关键约束为：

- 识别与解析会带来性能影响
- 控制面与用户面配置必须一致
- 某些加密业务不能直接做内容计费
- 多规则优先级错误会导致业务偏离预期
- Portal/HTTP重定向能力受协议与浏览器能力约束
- UL CL 分流受融合计费、单辅锚点和位置触发边界约束
- Trigger优先级、NCG组网差异和多UPF独享配额会影响计费闭环表现
- 静态规则、QoS Flow映射和策略更新链会影响体验控制闭环表现
- 资源和License控制项会影响业务感知能力可开通范围、会话规模和吞吐边界

这里同样需要强调适用边界。`重定向缺省行为约束` 主要属于 `NS-03` 的 HTTP 导向链，`QoS编排约束` 主要属于 `NS-02`，而 `多UPF配额约束`、`计费可靠性约束` 则主要作用于计费和分流计费旁支。下面统一显式标出适用场景。

### 18.1 `RiskConstraint` 实例表

| 约束类型 | 适用场景 | 约束内容 | 主要影响对象 | 业务后果 |
| --- | --- | --- | --- | --- |
| 性能约束 | `全域，尤以 NS-02 / NS-03 明显` | 深度解析、七层识别会增加处理开销 | UDG/UPF | 识别越细，吞吐和资源压力越大 |
| 一致性约束 | `全域` | UNC/SMF 与 UDG/UPF 的对象规划必须一致 | Rule、UserProfile、URR、RG 等 | 两侧理解不一致会导致业务不生效或统计偏差 |
| 能力边界 | `NS-01 / NS-03` | 加密流量或不可见内容不能直接按内容粒度识别 | 识别类场景 | 会限制差异化计费和访问控制的精细度 |
| 裁决约束 | `NS-01 / NS-02 / NS-03 / NS-04` | 多条 Rule 同时命中时必须依赖优先级 | 差异化计费、带宽控制、访问限制、本地分流 | 优先级设计错误会导致业务结果偏离预期 |
| 对外交互约束 | `NS-01 / DS-02 / DS-05中的独立计费旁支` | CHF/OCS/N40 交互异常会影响配额和计费闭环 | 配额控制、融合计费 | 可能导致无法实时控制业务或上报失败 |
| Trigger约束 | `NS-01 / DS-02` | Session级和RG级 Trigger 存在优先级，本地可配置忽略部分 CHF 下发 Trigger | 差异化计费、配额控制、会话更新 | Trigger配置不当会导致上报时机、会话更新和配额行为偏离预期 |
| 多UPF配额约束 | `DS-02 / DS-05` | 华为融合计费只支持由 CHF 对每个锚点UPF独享配额管理，UL CL/I-UPF 不参与配额管理 | 多UPF计费、MEC、本地分流 | 会决定多锚点场景下谁可以申请额度、谁只负责转发 |
| 组网约束 | `NS-01 / DS-02 / DS-05中的独立计费旁支` | 是否部署 NCG 会改变计费可靠性、话单生成路径和在线/离线职责分工 | 计费解决方案、可靠性设计 | 同样是融合计费，实际组网不同会改变故障兜底和网络侧数据保留能力 |
| 初始RG约束 | `NS-01 / DS-02 / DS-05中的独立计费旁支` | 初始RG个数、RG来源和 `CTXSTARTRATING` 共同决定激活时是否能为预期RG预申请配额 | 融合计费模板、预申请配额、预定义RG | 激活阶段如果拿不到预期RG，后续计费闭环会从一开始就偏离规划 |
| 计费方式匹配约束 | `NS-01 / DS-02 / DS-05中的独立计费旁支` | 用户级融合计费方式、RGAPPLIED 和 URR 的 `USAGERPTMODE` 必须匹配 | `CHARGECTRL / CHARGEMETHOD / URR` | 用户和RG的在线/离线方式错配时，RG计费方式不会按预期生效 |
| CHF链路与响应约束 | `NS-01 / DS-02 / DS-05中的独立计费旁支` | N40链路故障、CHF无响应或异常返回码会导致业务放通但不上报CHF | `Charging Data Request/Response`、`ALM-100072` | 会导致计费缓存甚至漏计费，是计费可靠性的直接风险来源 |
| 静态规则约束 | `NS-02，兼及需要精细识别的 NS-01` | PCF不具备七层业务识别能力，对特定业务的精细化体验控制必须依赖UPF侧预定义/本地规则，且PCF/SMF/UPF规则标识和URR ID需一致 | 带宽控制场景、计费场景 | 会决定哪些体验控制能由动态规则完成，哪些必须走静态规则编排 |
| QoS编排约束 | `NS-02` | QoS策略需要经历SMF翻译、QoS Flow建立、PFS/QFI映射以及RAN无线侧调度，多环节协同复杂 | 带宽控制场景 | 任何一段映射或更新异常，都会导致体验分层结果与策略意图不一致 |
| 导向约束 | `NS-03` | Portal/HTTP重定向依赖 HTTP/WAP 能力，且相关解析与重定向处理会带来系统吞吐下降 | 访问限制、Portal处理路径、重定向类方案 | 可能导致某些业务无法按预期导向，或在高负荷下影响系统容量 |
| 分流架构约束 | `DS-05` | UL CL 方案依赖融合计费、单用户单辅锚点、位置或外部触发链条，且部分场景对私有实现或会话改写能力有要求 | 本地分流、边缘卸载、漫游地动态签约分流 | 可能导致本地分流不能在全部接入/漫游/多业务并发场景下稳定复用 |
| 计费可靠性约束 | `NS-01 / DS-02 / DS-05中的独立计费旁支` | CHF主备均故障时，需在"业务放通+SMF本地缓存话单"和"去活用户"之间选择；华为SMF+华为CHF且短时间故障时才可安全放通，第三方CHF或长时间故障可能导致计费损失 | 所有在线计费场景 | 可能导致计费损失或用户体验中断 |
| Default Quota约束 | `NS-01 / DS-02` | default quota 的使能开关、新业务/非新业务场景开关以及默认流量/时长配额共同决定新业务在正式配额返回前是否放通 | `UPAPNCHARGE / UPDEFAULTQUOTA / SRVCOMMONPARA / URR` | 配置不当会导致本应临时放通的新业务被阻塞，或放通额度与规划不一致 |
| 重定向缺省行为约束 | `NS-03` | HTTP重定向时，若当前业务报文无法做重定向（如承载条件不满足），UDG缺省行为是直接阻塞该报文；如需改为正常转发，需通过IPFarm的DEFAULTACT=PASS配置 | 访问限制、七层重定向 | 可能导致本应导向的业务被意外阻塞 |
| 资源与License约束 | `全域，尤以 NS-02 / NS-03 / 高吞吐计费场景明显` | 会话数、承载数、吞吐能力以及 SA-Basic、PCC、Portal、业务感知带宽控制、在线计费等资源/特性控制项共同决定方案是否可正式开通和稳定运行 | UDG/UPF、控制面配套能力、全局容量规划 | 设计上可行的方案如果超出资源或License边界，现网仍可能无法上线或在高负荷下失稳 |

### 18.2 `RiskConstraint` 字段业务含义

| 字段 | 含义 | 为什么重要 |
| --- | --- | --- |
| `适用场景` | 该约束主要作用于哪些场景 | 避免把某条局部约束误读成全域限制 |
| `约束类型` | 风险来自性能、一致性还是外部交互 | 有助于分类理解问题边界 |
| `约束内容` | 具体受限的点是什么 | 决定设计方案时必须避开的坑 |
| `主要影响对象` | 这类约束首先影响哪层对象 | 帮助把风险落到图谱中的具体节点 |
| `业务后果` | 如果忽略该约束会出现什么 | 让风险不再是抽象提醒，而是业务后果 |

## 19. 业务感知的 `Evidence`

这里的 `Evidence` 对应中文对象名是 `证据来源`。

业务图谱不是自由总结，所有关键对象都必须能回到原始语料。

### 19.1 `Evidence` 实例表

| 证据ID | 证据类型 | 代表性原始材料 | 支撑的图谱对象或关系 | 业务含义 |
| --- | --- | --- | --- | --- |
| `EV-01` | 定义与过程证据 | 业务感知定义页、过程页 | `BusinessDomain`、`RuntimeFlow` | 证明业务感知是稳定业务域，并且存在完整运行链 |
| `EV-02` | 场景样板证据 | 业务感知场景举例、计费实例页 | `NetworkScenario`、`ReferencePattern` | 证明差异化计费、配额切换等不是虚构场景 |
| `EV-03` | 特性能力证据 | SA-Basic、PCC 基本功能、带宽控制 | `Feature`、`DomainSemanticObject` | 证明识别、策略执行、带宽控制有正式能力承载 |
| `EV-04` | 控制面对象证据 | 内容计费、融合计费、费率标识配置页 | `ConfigObject`、`DeliverySolution` | 证明控制面对象链和配额动作是正式机制 |
| `EV-05` | 映射与调测证据 | UNC/UDG 配置映射页、调测页 | `ValidationPlan`、`DiagnosisRule`、`RiskConstraint` | 证明一致性检查、故障回查和验证路径有原始依据 |
| `EV-06` | 分流与边缘卸载证据 | Why/What/How分流、UL CL 分流/MEC 特性概述 | `ReferencePattern`、`DeliverySolution`、`Participant`、`RiskConstraint` | 证明业务感知的“导向”可以扩展到本地DN/边缘DN，并带来会话编排与计费约束 |
| `EV-07` | 计费会话与Trigger证据 | 计费解决方案概述、融合计费概述、标准计费流程、Trigger/用量/多UPF参考页 | `ReferencePattern`、`DeliverySolution`、`RuntimeFlow`、`ValidationPlan`、`DiagnosisRule`、`RiskConstraint` | 证明计费主线不是静态费率配置，而是会话、Trigger、用量、配额和多UPF共同构成的闭环 |
| `EV-08` | 静态规则与QoS编排证据 | PCC静态规则 Why/What/How、E2E QoS管理机制、QoS生成/下发/更新页 | `ReferencePattern`、`DeliverySolution`、`Participant`、`DecisionPoint`、`RuntimeFlow`、`RiskConstraint` | 证明体验分层既依赖UPF业务识别，也依赖PCF/SMF/RAN/UE共同完成QoS编排和更新 |
| `EV-09` | 计费粒度/方式/Trigger总览证据 | 一望5G 计费解决方案总览页（UDG/UNC镜像） | `ReferencePattern`、`DecisionPoint`、`RuntimeFlow`、`RiskConstraint` | 证明计费粒度、计费方式、Trigger机制和CHF可靠性是正式产品知识，不是零散参数 |
| `EV-10` | UDG 调测验证证据 | UDG 调测七层阻塞、七层重定向、三四层阻塞、三四层重定向页 | `ValidationPlan`、`DiagnosisRule`、`RiskConstraint` | 证明访问限制类功能的验证和诊断路径有完整的UDG调测文档支撑 |
| `EV-11` | UNC 计费故障与调测证据 | UNC 常见故障案例、欠费重定向、default quota、融合计费/内容计费调测页 | `ValidationPlan`、`DiagnosisRule`、`RiskConstraint`、`ConfigObject` | 证明计费主线不仅有原理和对象链，还有稳定的调测、告警和排障骨架 |
| `EV-12` | 产品能力版图与资源边界证据 | UNC/UDG 产品描述、5G Core 解决方案、UDG 典型组网、UDG License 资源控制项 | `BusinessDomain`、`Participant`、`Scope`、`Feature`、`RiskConstraint` | 证明业务感知并非孤立专题，而是建立在 5GC 控制面/用户面分工、分布式用户面部署和资源边界之上的正式产品能力 |

### 19.2 `Evidence` 字段业务含义

| 字段 | 含义 | 为什么重要 |
| --- | --- | --- |
| `证据类型` | 这是定义类、样板类、特性类还是调测类证据 | 决定它更适合支撑哪类图谱对象 |
| `代表性原始材料` | 具体来自哪组原始语料 | 保证图谱可回溯 |
| `支撑的图谱对象或关系` | 它到底证明了什么 | 让证据和图谱不是松散对应 |
| `业务含义` | 这组证据在业务层说明了什么 | 防止把证据退化成文档清单 |

## 20. 业务感知的 `DomainSemanticObject`

这里的 `DomainSemanticObject` 对应中文对象名是 `领域语义对象`。

业务感知域中的语义对象不是“识别+计费+导向”的简单累加，而是进一步收敛为 16 个关键语义对象。

其中最需要注意的是：`BA.TrafficRecognition` 被去掉了。

原因不是识别不重要，而是它在原始语料里已经被拆开成更稳定的三个层次：

1. `报文解析`
2. `协议/应用识别`
3. `识别知识底座`

如果继续保留一个总括性的 `BA.TrafficRecognition`，反而会和 `PacketParsing / ProtocolRecognition / 协议库` 这组对象重叠。

本文保留的关键语义对象为：

- `BA.PacketParsing`
- `BA.ProtocolRecognition`
- `BA.FilterCondition`
- `BA.ProtocolFeatureLibrary`
- `BA.ProtocolParserLibrary`
- `BA.Rule`
- `BA.Policy`
- `BA.Priority`
- `BA.Binding`
- `BA.Charging`
- `BA.Quota`
- `BA.ChargingTrigger`
- `BA.Runtime`
- `BA.ValidationDiagnosis`
- `BA.LocalBreakout`
- `BA.QoSOrchestration`

这些对象的作用不是替代场景，而是把场景和下层实现连接起来。

### 20.1 `DomainSemanticObject` 实例表

| 语义对象 | 定义 | 在业务感知中的业务含义 | 典型落点 |
| --- | --- | --- | --- |
| `BA.PacketParsing` | 如何把原始报文拆解成可读字段 | 表达三四层解析、七层解析和字段提取的过程 | 三四层解析、七层解析、字段内容提取 |
| `BA.ProtocolRecognition` | 如何从报文和连接行为中识别协议/应用 | 表达知名端口、特征字、关联识别、行为识别等识别方式 | 协议识别引擎、端口识别、特征字识别、关联识别、行为识别 |
| `BA.FilterCondition` | 如何把已解析的报文特征组织成可匹配条件 | 表达 3/4 层过滤、7 层过滤、协议过滤以及它们组合成 Flow Filter 的方式 | Filter、FilterGroup、L7Filter、Protocol、ProtocolGroup、FlowFilter、FlowFilterGroup |
| `BA.ProtocolFeatureLibrary` | 协议特征如何被组织和更新 | 表达新业务识别能力来自特征库而非总是来自软件升级 | 协议特征库、规则库更新 |
| `BA.ProtocolParserLibrary` | 协议解析能力的固定基础集合 | 表达广泛协议的基础解析能力是业务识别的前置底座 | 协议解析库、基础协议解析能力 |
| `BA.Rule` | 业务感知的最小组织单元 | 把过滤条件、策略、优先级以及其他参数组织成一个可下发、可匹配、可执行的单元 | Rule、预定义规则、动态规则、本地规则 |
| `BA.Policy` | 规则命中后要执行的策略类型与目标 | 表达 PCC、BWM、HEADEN、IPREDIR、WEBPROXY、SMARTREDIRECT 以及 Portal 处理路径等不同策略目标 | PCC策略组、BWM分类策略、头增强策略、IP重定向、WebProxy、智能重定向、Portal 动作 |
| `BA.Priority` | 多规则同时命中时如何裁决 | 解决专项业务和默认业务冲突 | Rule Priority |
| `BA.Binding` | 规则和范围对象如何绑定 | 表达业务套餐、生效对象、默认策略承载方式 | UserProfile、RuleBinding、默认URR组绑定 |
| `BA.Charging` | 如何按业务收费 | 表达差异化计费、免费业务、默认计费 | URR、URRGroup、RG、PCCPolicyGrp |
| `BA.Quota` | 如何处理配额 | 表达在线配额、额度耗尽、后续动作 | QuotaExhaustAct、CHF交互、RG |
| `BA.ChargingTrigger` | 什么事件会驱动计费会话更新、重授权或释放 | 表达 Session级/RG级 Trigger、立即/延迟上报以及会话创建/更新/释放条件 | PDUTRIGGER、RGTRIGGER、Trigger优先级、Charging Data Request |
| `BA.Runtime` | 场景如何动态生效 | 表达策略建立、更新、释放和 Trigger 驱动切换 | PDR/FAR/Usage/Trigger |
| `BA.ValidationDiagnosis` | 如何验证和回查 | 表达生效校验、一致性核查和异常定位 | CP/UP一致性、运行现象、上报链路 |
| `BA.LocalBreakout` | 如何把本地业务流从中心路径导向本地DN/边缘DN | 表达基于 DNN、位置、PRA、DNAI、selectedDnn 或外部触发建立 UL CL 分流路径的能力 | UL CL UPF、辅锚点UPF、本地DN导向、边缘卸载 |
| `BA.QoSOrchestration` | 如何把业务意图翻译成 QoS Flow、PFS/QFI 映射和端到端体验保障 | 表达静态/动态规则、SM策略生成、QoS Flow建立、RAN映射和PCF/SMF发起更新这条控制面-用户面编排链 | QFI、PFS、QER、QoS Flow、Radio Bearer 映射、QoS策略更新 |

### 20.2 为什么必须有语义对象

如果没有这一层，图谱会变成：

- 场景名
- 特性名
- 命令名

三堆名词之间直接相连。

而加入语义对象后，才能表达：

```text
业务问题
-> 解析语义/识别语义/过滤条件语义/计费语义/配额语义
-> 正式特性
-> 配置对象链
```

这才是“图谱”，而不是“目录汇总”。

这一层至少已经能稳定表达出下面这条桥接链：

```text
业务问题
-> 报文解析
-> 协议/应用识别
-> 过滤条件组织
-> 规则匹配与策略执行
-> 计费/配额/转发结果
-> 正式特性
-> 配置对象链
```

## 21. 业务感知的 `Feature`

## 22. 业务感知的 `ConfigObject`

这里的 `ConfigObject` 对应中文对象名是 `配置对象`。

业务图谱层面不展开所有命令，但必须说明关键对象链。

在计费方案里，`Rule` 不是唯一的控制面骨架，`Trigger` 与 `URR/URRGroup` 一起决定了计费会话何时创建、更新和释放。

### UDG/UPF 侧关键对象

- `Filter`
- `L7Filter`
- `FlowFilter`
- `Rule`
- `PCCActionProp / Redirect / HEADEN / IPFarm`

### UNC/SMF 侧关键对象链

```text
URR
-> URRGroup
-> PCCPolicyGrp
-> Rule
-> UserProfile
-> RuleBinding
-> Default URRGroup Binding
```

### 22.1 `ConfigObject` 实例表

| 对象链位置 | 关键对象 | 所属侧别 | 业务作用 |
| --- | --- | --- | --- |
| 识别层 | `Filter` | UDG/UPF | 表达三四层过滤条件 |
| 识别层 | `L7Filter` | UDG/UPF | 表达七层 URL/Host/Method 等匹配条件 |
| 识别组合层 | `FlowFilter` / `FlowFilterGroup` | UDG/UPF | 把多种识别条件组合成最终匹配入口 |
| 规则层 | `Rule` | UDG/UPF + UNC/SMF | 把条件、动作、优先级绑定成可执行规则 |
| 动作层 | `PCCActionProp` / `Redirect` / `HEADEN` / `IPFarm` | UDG/UPF | 承载阻断、URL导向、头增强、Portal目标池等具体动作 |
| 分流触发层 | `App ID` / `selectedDnn` / `PRA绑定` / `DNAI` | UNC/SMF + AMF/PCF | 表达本地分流何时被触发、导向到哪个本地接入点 |
| 分流执行层 | `PDR` | UNC/SMF + UDG/UPF | 承载 UL CL 分流规则以及主锚点/辅锚点的转发路径控制 |
| QoS编排层 | `QER` / `PFS` / `QFI` | UNC/SMF + UDG/UPF + RAN/UE | 表达业务流如何映射到 QoS Flow，以及端到端体验控制如何被识别和实施 |
| 融合计费模板层 | `CCT` / `CHFINIT` / `CTXSTARTRATING` | UNC/SMF | 表达初始RG个数、RG来源以及激活时优先申请哪些RG配额 |
| 计费方式控制层 | `CHARGECTRL` / `USRPROFCHARGE` / `APNCHARGECTRL` / `CHARGEMETHOD` | UNC/SMF | 表达用户是否启用融合计费、按User Profile/DNN/CC的哪种方式计费，以及RG按在线/离线如何生效 |
| 计费Trigger层 | `PDUTRIGGER` / `RGTRIGGER` / `CNVRGDCHGPARA` | UNC/SMF | 表达会话级、费率组级 Trigger 以及本地对 CHF Trigger 的覆盖策略 |
| 缺省配额控制层 | `UPAPNCHARGE` / `UPDEFAULTQUOTA` / `SRVCOMMONPARA` | UDG/UPF + UNC/SMF | 表达在线计费场景在正式配额返回前是否允许使用 default quota，以及默认流量/时长是多少 |
| 计费对象层 | `URR` | UNC/SMF + UDG/UPF | 表达使用量统计与计费标识 |
| 计费组合层 | `URRGroup` | UNC/SMF + UDG/UPF | 把计费对象组合成上下行或在线/离线计费组 |
| 策略组层 | `PCCPolicyGrp` | UNC/SMF + UDG/UPF | 把 URRGroup 绑定成可被 Rule 调用的策略对象 |
| 特殊流量计费层 | `SPECTRAFURRGRP` / `URRFAILACTION` | UDG/UPF | 表达欠费/特殊流量是否继续统计、丢弃还是继续放通 |
| 绑定层 | `UserProfile` | UNC/SMF + UDG/UPF | 承载一组规则和默认策略，形成业务模板 |
| 绑定层 | `RuleBinding` | UNC/SMF + UDG/UPF | 把具体 Rule 绑定到 UserProfile |
| 默认计费层 | `Default URRGroup Binding` | UNC/SMF + UDG/UPF | 为未命中特殊业务的流量提供默认计费去向 |

### 22.2 `ConfigObject` 字段业务含义

| 字段 | 含义 | 为什么重要 |
| --- | --- | --- |
| `对象链位置` | 该对象位于哪一层 | 帮助理解它在全链路中的职责，而不是只看名字 |
| `所属侧别` | 主要在哪一侧存在或要求一致 | 决定它是控制面编排对象、用户面执行对象，还是两侧一致性对象 |
| `业务作用` | 它对业务目标的贡献 | 把配置对象重新翻译回业务语言 |

## 25. 最终结论

“业务图谱的定义”和“业务感知的业务图谱”不是一回事。

二者关系是：

```text
业务图谱定义
  = 一套通用 schema + schema 关系主链

业务感知的业务图谱
  = 在这套 schema 上，对 BusinessDomain=业务感知 的具体实例化
```

所以：

1. `Part A` 解释了什么叫业务图谱
2. `Part B` 给出了业务感知对应的完整业务图谱

这两部分合在一起，才是完整答案。
