## 直接读出的稳定结论

### 1. UNC/SMF 侧是业务感知的控制面编排中心

从 5G PCC、内容计费、融合计费三组原始语料综合看，UNC/SMF 侧承担的不是单点功能，而是：

- 从 PCF 接收策略
- 管理 SM 策略
- 组织 PCC 规则
- 下发 PDR/PCC/计费相关信息给 UPF
- 处理 Trigger 驱动的策略更新
- 与 CHF 交互进行计费和配额控制

这说明当前图谱表达里：

```text
UNC/SMF 侧应被表达为“控制面策略与计费编排中心”
```

### 2. `WSFD-109101 PCC基本功能（5G）` 定义了控制面策略世界

原始语料明确把 PCC 策略分为：

- `AM策略`
- `UE策略`
- `SM策略`

其中与业务感知最直接相关的是：

- `SM策略`
  - `会话规则`
  - `PCC规则`

并且 `PCC规则` 直接对应：

- 一条或一组 `SDF`
- 相应的 QoS
- 计费参数
- 时间/流量上报参数

这意味着业务图谱中：

```text
UNC 侧的 Rule 不是简单本地对象，
而是“SM策略中的业务粒度控制单元”
```

### 3. `动态PCC / 预定义规则 / 本地PCC` 是业务图谱里必须保留的决策点

原始语料清楚给出三种规则来源：

- 动态规则
- 预定义规则
- 本地 PCC

其中：

- 动态规则适合访问时协商出的业务
- 预定义规则适合运营商预定义、激活前即可确定的业务
- 本地 PCC 以 DNN 粒度生效

这说明：

```text
DecisionPoint
```

在当前展示里不应被隐藏掉，至少要体现：

- 规则来自 PCF 还是本地
- 某个 DNN 使用动态 PCC 还是本地 PCC

### 4. `内容计费基本功能` 确认了“差异化计费”在控制面侧的正式定义

原始语料把内容计费定义为：

- 基于业务粒度的计费
- 对不同业务实施不同费率
- 控制面从 PCF 或本地获取计费规则
- 将业务信息作为 `PDI` 下发给 UDG/UPF
- UPF 根据 SA 解析和规则匹配标记 `URR ID`
- 触发 Trigger 后由 SMF 上报计费信息

这说明：

```text
差异化计费与免流
```

不是 UDG 专题里的案例，而是被 UNC 控制面正式承认和支撑的场景类。

### 5. `支持融合计费` 把“配额耗尽后的体验控制”变成稳定业务场景

原始语料中融合计费明确区分：

- 在线计费：申请配额，额度耗尽时可以实时控制业务
- 离线计费：不上配额，只上报使用量

并明确内容计费可以同时叠加：

- 内容计费 + 在线计费
- 内容计费 + 离线计费

这意味着：

```text
配额耗尽后的体验控制
```

不是业务举例里临时拼出的剧情，而是融合计费体系下自然存在的标准能力面。

### 6. `配置融合计费费率标识` 直接给出了控制面对象链

该页最重要的不是命令本身，而是它把对象链完整摊开了：

```text
URR
-> URRGroup
-> PCCPolicyGrp
-> Rule
-> UserProfile
-> RuleBinding
-> Default URRGroup Binding
```

并补充了：

- 业务条件只在 UPF 上配置
- SMF 侧主要配置费率标识与规则绑定链
- `RG` / `(RG,SID)` 是费率标识核心

这对当前业务图谱展示非常重要，因为它给出了：

- UDG/UPF 侧更偏“条件识别”
- UNC/SMF 侧更偏“费率和规则编排”

### 7. `ADD QUOTAEXHAUSTACT` 所代表的动作是控制面标准动作

在费率标识配置页中，明确把：

- `BLOCK`
- `REDIRECT`
- `FORWARD`

定义为在线 RG 配额耗尽后的默认动作，并说明最终对应到 N4/FAR 的处理方式。

这说明：

```text
重定向
```

在第一版虽然不单列一级场景，但必须在：

- `配额耗尽后的体验控制`

之下被明确展示为一个标准动作分支。

## 当前结论

基于三轮语料，当前业务感知全局业务图谱第一版中，以下一级场景已经比较稳：

1. `差异化计费与免流`
2. `配额耗尽后的体验控制`
3. `访问限制与阻断`
4. `带宽控制与体验分层`

以下对象也应该进入正式展示：

1. `ReferencePattern`
2. `DeliverySolution`
3. `RuntimeFlow`
4. `Scope`
5. `DecisionPoint`
