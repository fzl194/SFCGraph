# Round 1 阅读笔记：UDG 业务感知专题

本轮只回答四个问题：

1. UDG 原始语料如何定义“业务感知”
2. UDG 原始语料如何描述业务感知过程
3. UDG 原始语料直接支撑哪些一级场景
4. 这些场景如何挂到业务图谱对象 schema

## 直接读出的稳定结论

### 1. 业务感知首先是一个业务控制能力域，不是命令集合

原始定义强调：

- 在用户会话过程中解析数据报文
- 区分用户使用的不同业务
- 对不同业务执行不同的策略控制和计费控制

这说明：

```text
BusinessDomain = 业务感知
```

不是“某个专题页名字”，而是一类长期存在的网络运营控制域。

### 2. 业务感知的主链天然适合做业务图谱骨架

`业务感知过程` 直接给出了一条稳定过程链：

```text
PCF下发规则
-> SMF翻译并下发到UPF
-> UPF解析与识别
-> 报文与规则匹配
-> 执行规则绑定的策略
-> 数据继续转发
-> UPF上报使用情况给SMF
```

这条链的价值在于：

- 它天然把 `UNC/SMF` 和 `UDG/UPF` 串起来
- 它天然说明 `RuntimeFlow` 应该是业务图谱里的正式对象
- 它说明业务图谱不能只停在“场景列表”，必须保留运行链

### 3. 一级业务场景应按“业务目标/动作目标”划分

从 `业务感知定义` 和 `场景举例` 直接能读出的业务目标包括：

- 内容计费
- 免费业务
- 带宽控制
- 阻塞
- 重定向
- 头增强
- 安全防护
- 行为识别

其中在当前第一批语料里证据最强的，是：

1. `差异化计费与免流`
2. `配额耗尽后的体验控制`
3. `访问限制与阻断`
4. `带宽控制与体验分层`

`重定向` 在原始语料中已明确出现，但当前更像：

- 一种重要动作形态
- 可附着在配额耗尽、访问控制等场景下

第一版不必急着单列为一级主类。

### 4. Rule 是场景和配置世界之间的关键枢纽

原始语料对 `规则` 的定义非常关键：

- 规则由条件和动作组成
- 规则中绑定过滤条件、策略、优先级和其他参数
- 同一策略类型内按优先级裁决

这意味着在业务图谱里：

```text
Rule 不是普通配置项
而是“业务语义落地的组合枢纽”
```

它连接：

- `TrafficRecognition`
- `PolicyAction`
- `Priority`
- `Binding`

### 5. 过滤条件层支撑“业务感知”这个域成立

原始语料说明过滤条件可以覆盖：

- 3/4 层
- 协议/协议组
- 7 层

并通过：

- `Filter`
- `L7 Filter`
- `Protocol`
- `Protocol Group`
- `Flow Filter`
- `Flow Filter Group`

形成复杂识别逻辑。

这说明业务图谱里的“场景差异”，很多并不是来自不同网元，而是来自：

```text
识别条件差异
+ 规则动作差异
+ 生效范围差异
```

### 6. UDG 原始语料已经直接给出一个完整场景闭包样板

`业务感知场景举例` 的闭包非常完整：

- 用户访问 A 网站视频不计流量
- 访问非 A 网站视频正常计费
- 流量耗尽后重定向到充值页面

它同时包含：

- 识别条件
- 策略对象
- 规则及优先级
- 用户属性与绑定
- 会话过程中的规则启停切换
- 最终用户面动作

这是第一版业务图谱中最重要的 `ReferencePattern` 候选。

## 对业务图谱 schema 的反向修正

### 本轮验证更强的对象

- `BusinessDomain`
- `NetworkScenario`
- `ReferencePattern`
- `RuntimeFlow`
- `Participant`
- `DomainSemanticObject`
- `Evidence`

### 本轮被证明必须保留的桥接语义

至少需要显式保留这些语义桥：

- `TrafficRecognition`
- `Rule`
- `PolicyAction`
- `Priority`
- `Binding`
- `Charging`
- `Runtime`
- `ValidationDiagnosis`

### 本轮暂未直接强化的对象

仅靠本轮材料还不足以稳定展开：

- `DecisionPoint`
- `DiagnosisRule`
- `Capability`
- `Feature`

这些需要在下一轮特性侧原始语料中再补强。

## 当前建议的一级场景草案

基于本轮语料，第一版建议保留这 5 类：

1. `差异化计费与免流`
2. `默认兜底策略`
3. `配额耗尽后的体验控制`
4. `访问限制与阻断`
5. `带宽控制与体验分层`

说明：

- `重定向` 保留为重要动作，不先独立成一级类
- `增强与安全` 在定义页里已出现，但本轮没有足够实例支撑其作为主块展开

## 当前疑问

1. `默认兜底策略` 是独立场景，还是计费类场景内部机制，仍需后续文档验证。
2. `重定向` 是否在后续特性文档中呈现出足够独立性，待下一轮确认。
3. `Participant` 中是否需要显式把 `PCF/SMF/UPF/业务服务器/CHF` 都纳入主图，待侧别阅读后收敛。
