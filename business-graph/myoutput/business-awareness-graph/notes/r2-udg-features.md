## 直接读出的稳定结论

### 1. `SA-Basic` 是业务感知域的识别底座

原始语料直接给出：

- 适用 NF：`SGW-U / PGW-U / UPF`
- 基本能力：三四层解析、协议识别、七层解析
- 业务后续动作：计费、重定向、头增强、Tethering 识别等

这说明：

```text
SA-Basic 不是某个子场景
而是业务感知这个域得以成立的通用底座特性
```

### 2. `SA-Basic` 天然把控制面和用户面连接起来

`SA-Basic` 概述里明确说明：

- `UNC/SMF` 可以动态下发规则给 `UDG`
- 控制面和用户面要合一规划：
  - `Rule`
  - `User Profile`
  - `URR/CbbID` 等参数
- 当控制面和用户面配置重复时，`UPF` 优先使用控制面下发配置

这说明业务图谱里必须显式区分：

- `UDG/UPF 侧的识别与执行`
- `UNC/SMF 侧的规划与下发`

### 3. `PCC基本功能` 是规则动作编排骨架，不是单独某个计费特性

原始语料说明：

- PCC 用于策略和计费控制
- 控制面 `PGW-C/SMF` 负责策略承接、QoS/计费/门控/路由决策
- 用户面 `PGW-U/UPF` 负责业务流检测、使用量统计、流量上报、QoS控制、带宽管理、重定向等执行

这意味着：

```text
PCC 是“业务感知多场景共享的控制框架”
```

它不是某个一级业务场景本身。

### 4. `RuntimeFlow` 在业务图谱里必须占正式位置

`PCC业务流程` 明确给出了：

- PDU 会话建立
- PDU 会话修改
- PDU 会话释放

并说明：

- 策略由 `PCF/PCRF` 选择和下发
- `SMF` 完成 QoS 映射
- `UPF` 基于收到的 SDF/QoS/策略执行流量处理
- 触发器事件会引起策略更新

这比 Round 1 的过程描述更强，说明：

```text
RuntimeFlow 不是补充说明，
而是业务图谱从“场景”走向“可生效运行”的核心对象。
```

### 5. 带宽控制可以稳定作为一级业务场景

`GWFD-110311` 直接给出：

- 定义
- 独立客户价值
- 典型应用场景
- 独立原理
- 独立业务控制域

而且其业务控制不只是“一个动作”，而是有完整结构：

- 业务识别
- 用户/用户组/整机三级作用域
- 用户属性/流量属性约束
- BWM Service / Controller / Rule / User Group / ServiceLevelPolicy

这足以证明：

```text
带宽控制与体验分层
可以稳定作为一级 NetworkScenario 类
```

### 6. 业务感知里的“作用范围域”应进入业务图谱展示

`带宽控制实现原理` 中非常明确地提出：

- 整机级
- 用户组级
- 用户级

并且用户组级可以进一步覆盖：

- `APN`
- `切片`
- `UserProfile`

这对业务图谱很重要，因为它说明：

```text
Scope 不是附属字段，
而是会直接改变策略控制方式的正式对象。
```

### 7. Rule 再次被验证为跨层枢纽对象

在带宽控制实现原理中：

- Rule 为每个匹配条件映射流类别
- 再与协议、协议组、TOS 等一起构成业务匹配条件
- 通过 Rule 与 BWM Rule / Policy 关联

这再次证明：

```text
Rule 既是配置对象，
又是业务语义向控制动作转译的核心桥。
```
