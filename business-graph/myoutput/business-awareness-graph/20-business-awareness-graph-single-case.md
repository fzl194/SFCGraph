这份文档不再平铺整个 `业务感知` 域的所有对象，而是只做一件事：


## 3. 案例本身

原始案例的业务要求可以直接压成一句话：

```text
普通用户访问视频业务时，
访问 A 网站的视频不计入流量，
A 网站以外的视频按 1 元/Gb 计费，
每月最多 100G，
流量耗尽时重定向到充值页面。
```

这个案例的价值在于，它不是只讲“识别业务”，也不是只讲“怎么收费”，而是把下面几件事放在了一条线上：

1. 业务识别
2. 免费业务
3. 差异化计费
4. 规则优先级
5. 在线计费配额管理
6. 配额耗尽后的动作切换
7. PCF/SMF/UPF/CHF 之间的运行闭环

因此，它非常适合用来串起整张业务图谱。

## 5. 用这个案例把图谱串起来

### 5.1 从业务问题开始

这个案例的起点不是 `Rule`，也不是 `URR`，而是一个很具体的业务问题：

```text
同样是视频业务，
A 网站视频免费，
其他视频网站按流量收费，
并且流量包用完后要把用户引导到充值页。
```

这一步在图谱里对应：

- `BusinessDomain = 业务感知`
- `NetworkScenario = 计费场景`

因为它解决的核心问题是：

```text
同一类业务流，
如何按识别结果进入不同计费路径，
并在额度变化后切换处理方式。
```

### 5.2 再看这个业务问题如何被组织成方案

原始案例已经给出了最关键的组织方式：

1. 用 `FlowFilter1` 识别 A 网站视频
2. 用 `FlowFilter2` 识别非 A 网站视频
3. 用 `PCCGroup2` 承接 A 网站免费视频策略
4. 用 `PCCGroup1 + URR1` 承接其他视频按 1 元/Gb 计费
5. 用 `PCCGroup3` 承接流量耗尽后的重定向动作
6. 用 `Rule1 / Rule2 / Rule3` 把识别条件和动作链组织起来
7. 用 `User Profile1`、`User Profile Group1`、`APN1` 把这些规则挂到用户侧

这一步在图谱里对应的是：

- `ReferencePattern = 复合业务套餐样板`
- `DeliverySolution = 差异化计费、免费业务与默认计费组合方案`
- `DeliverySolution = 配额耗尽后体验切换方案`

也就是说，这个案例不是单一方案，而是：

```text
一个计费主方案
+ 一个配额动作切换子方案
```

### 5.3 这一步把哪些 schema 串起来了

| 串接阶段 | 串起来的对象 | 案例里的具体东西 |
| --- | --- | --- |
| 业务问题定义 | `BusinessDomain`、`NetworkScenario` | 业务感知、计费场景 |
| 样板抽取 | `ReferencePattern` | 场景举例、套餐样板、计费闭环样板 |
| 方案组织 | `DeliverySolution` | 差异化计费方案 + 配额动作切换方案 |
| 任务拆解 | `EngineeringTask` | 识别、绑定、计费、动作切换 |

## 6. 再把运行链串起来

这部分是整个案例最关键的地方。因为图谱如果只停在方案层，仍然是静态的。

### 6.1 接入前：把业务意图翻译成规则

原始案例在“用户接入之前”给出的动作，本质上就是：

1. 规划识别条件
2. 规划计费策略
3. 规划规则与优先级
4. 规划用户模板与绑定关系

这一步在图谱里对应：

- `EngineeringTask`
  - `规划识别条件`
  - `规划Rule与优先级`
  - `规划计费对象与费率标识`
  - `规划生效范围`

关键对象链是：

```text
FlowFilter
-> PCCPolicyGrp
-> URR
-> Rule
-> UserProfile
-> RuleBinding
-> APN
```

### 6.2 会话建立时：控制面把规则下发给用户面

原始案例明确写到：

1. 用户发起会话
2. `SMF` 把用户信息发给 `PCF`
3. `PCF` 找到 `APN1` 上绑定的 `Rule1 / Rule2`
4. `PCF` 下发到 `SMF`
5. `SMF` 再下发到 `UPF`

这一步在图谱里对应：

- `Participant`
  - `PCF`
  - `SMF`
  - `UPF`
- `RuntimeFlow`
  - `PCF/SMF按场景生成并下发策略`
  - `SMF翻译策略并建立规则或会话上下文`

这也说明：

```text
业务感知不是“UPF自己识别完自己决定”，
而是控制面先决定规则集，用户面再执行。
```

### 6.3 流量未耗尽时：先免费，再收费

原始案例给出的命中逻辑是：

1. `UPF` 上启用 `Rule1`、`Rule2`
2. 报文先匹配 `Rule1`
3. 命中则访问 A 网站视频，免费
4. 未命中再匹配 `Rule2`
5. 命中则访问其他视频网站，按流量计费

这一步在图谱里对应：

- `DecisionPoint = 规则优先级选择`
- `DomainSemanticObject`
  - `FilterCondition`
  - `Rule`
  - `Charging`
  - `Priority`
- `RuntimeFlow`
  - `UPF解析与识别`
  - `规则匹配`
  - `执行绑定策略`

这一步特别重要，因为它说明：

```text
免费业务不是一个孤立开关，
而是依赖“识别条件 + Rule优先级 + 计费对象链”共同成立。
```

### 6.4 流量耗尽后：从计费链切换到重定向链

原始案例给出的切换逻辑是：

1. `UPF` 定期上报用户用量
2. 通过 `SMF` 到达 `CHF`
3. 流量耗尽后，`CHF` 通知 `SMF`
4. `SMF` 向 `PCF` 请求更新会话
5. `PCF` 下发 `Rule3`，停用 `Rule2`
6. 此后 `UPF` 上启用的是 `Rule1 + Rule3`
7. 非 A 网站视频命中 `Rule3`，重定向到充值页面

这一步在图谱里对应：

- `DeliverySolution = 配额耗尽后体验切换方案`
- `DecisionPoint = 配额耗尽动作选择`
- `DomainSemanticObject`
  - `ChargingTrigger`
  - `Quota`
  - `Rule`
  - `Policy`
- `RuntimeFlow`
  - `Usage/事件上报`
  - `按需触发计费会话更新`

这一步把整个案例真正串活了。因为它证明：

```text
这个案例不是“先配免费、再配收费、再配重定向”三个并列功能，
而是一条会随配额状态变化而切换的运行链。
```

## 7. 这个案例里的验收和排障怎么落

### 7.1 验收怎么做

如果按图谱来验收，这个案例至少要验四件事：

| 验收点 | 在本案例里看什么 |
| --- | --- |
| 差异化计费验证 | 非 A 网站视频是否真的产生计费 |
| 免费业务验证 | A 网站视频是否真的不计费、不计流量 |
| Trigger与计费会话验证 | 用量上报后，是否真的触发了会话更新 |
| 配额耗尽动作验证 | 流量耗尽后，非 A 网站视频是否真的从计费切到重定向 |

### 7.2 如果出问题先查哪里

如果按图谱来排障，这个案例最典型的四个入口是：

| 异常现象 | 优先检查点 |
| --- | --- |
| 非 A 网站视频没有计费 | `Rule2 -> PCCGroup1 -> URR1` 是否闭合 |
| A 网站视频仍被计费 | `Rule1` 是否优先命中，免费路径是否真的不计费 |
| 流量耗尽后没有跳转充值页 | `QUOTA / Trigger / CHF -> SMF -> PCF` 更新链是否成立 |
| 流量耗尽后仍然继续走计费 | `Rule2` 是否被停用，`Rule3` 是否真的下发并生效 |

这一步对应的是：

- `ValidationPlan`
- `DiagnosisRule`

也就是说，图谱不是只回答“怎么设计”，也回答“怎么验”和“怎么查”。

## 8. 为什么这个案例能把图谱串起来

因为它在一个案例里同时具备下面几层：

1. 有明确业务目标
   - 免费
   - 差异化计费
   - 配额耗尽后重定向

2. 有明确对象链
   - `FlowFilter -> PCCPolicyGrp -> URR -> Rule -> UserProfile`

3. 有明确控制面到用户面的协同
   - `PCF -> SMF -> UPF`

4. 有明确运行态更新
   - `UPF -> SMF -> CHF`
   - `CHF -> SMF -> PCF -> SMF -> UPF`

5. 有明确结果切换
   - 流量未耗尽：免费或计费
   - 流量耗尽：重定向

因此，这个案例足以把业务图谱里的核心对象串成一条线：

```text
业务问题
-> 场景
-> 样板
-> 方案
-> 任务
-> 对象链
-> 运行链
-> 验收
-> 排障
```

## 9. 最后把整条线压成一句话

如果只用一句话概括这个案例在业务图谱中的意义，可以写成：

```text
业务感知并不是“识别出视频业务”这么简单，
而是把“识别哪个视频、是否免费、何时收费、额度耗尽后如何切换”
组织成一条从规则规划、控制面下发、用户面命中、
到计费会话更新和业务动作切换的完整业务链。
```

这就是为什么用一个案例，就能把整张业务感知业务图谱串起来。
