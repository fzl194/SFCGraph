# 业务感知候选特性目录盘点

## 1. 说明

本文件只盘点 `候选特性目录`，不展开命令级细节。

判定标准：

1. 直接承载业务感知定义、识别、策略、计费、配额、重定向、带宽控制。
2. 虽不以业务感知命名，但直接支撑业务感知关键场景。
3. 能为后续命令回溯提供稳定入口。

补充口径：

1. 本轮目标是把 6 个目录中的 `特性边界` 判清，不是只挑几个“像相关”的特性。
2. 判断时以 `特性概述/定义/应用场景/与其他特性的交互` 为主，不下钻到命令。
3. 对同目录下的 `单文件特性`、`组合特性簇` 也要显式判定，避免只看目录名造成遗漏。

## 1.1 六个目录的批次判断

你关心的 6 个目录，不能按“目录中所有特性都相关”处理，当前判断如下：

| 网元 | 目录 | 是否整目录纳入 | 当前判断 |
| --- | --- | --- | --- |
| UDG | `业务感知功能` | 否 | 以 `SA-Basic` 和识别/过滤/规则库相关特性为主，不是所有特性都进入主链 |
| UDG | `智能策略控制功能` | 否 | 以 `PCC / 累计流量 / 重定向 / 基于业务感知的带宽控制` 为主，不是所有特性都进入主链 |
| UDG | `计费功能` | 否 | 以 `在线计费 / 融合计费 / 内容计费` 和业务粒度计费相关特性为主，不是所有特性都进入主链 |
| UNC | `智能PCC解决方案` | 否 | 以 `PCC / ADC / 累计流量 / 业务感知带宽控制` 为主，不是所有特性都进入主链 |
| UNC | `计费管理功能` | 否 | 以 `融合计费 / 内容计费 / 在线计费 / 业务粒度计费` 为主，不是所有特性都进入主链 |
| UNC | `MEC解决方案` | 否 | 不能整目录纳入，但其中一部分 `基于预定义规则/动态规则/位置的分流策略控制` 与业务感知对象链高度相似，应作为旁支候选保留 |

## 1.2 UNC 的 `MEC解决方案` 是否相关

当前判断：

- `MEC解决方案` **不是业务感知主目录**
- 但其中部分 `ULCL/分流策略控制` 特性与业务感知**强相关**

原因不是它们在语义上等于业务感知，而是它们会复用和暴露相同的一批关键对象链：

- `FLOWFILTER`
- `RULE`
- `USERPROFILE`
- `RULEBINDING`
- `APNUSRPROFG`
- `ULCLPROP`

这些对象链虽然主要服务于 `ULCL 分流`，但会帮助我们补齐：

1. `UNC 侧规则与模板组织方式`
2. `FLOWFILTER / RULE / USERPROFILE / RULEBINDING` 的命令入口
3. `业务识别条件 -> 规则 -> 模板绑定` 的控制面组织模式

因此，`MEC解决方案` 应该作为：

```text
UNC 业务感知相关的旁支候选目录
```

而不是直接并入业务感知主链。

   - 或明显属于其他专题能力
