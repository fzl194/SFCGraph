# GWFD-110332 基于小区负荷上报的无线资源优化 - 特性知识文档

> **场景归属**: 业务感知 → 带宽控制子场景（辅助特性：小区负荷触发带宽调整）
> **产品**: UDG (UPF网元)
> **场景定位**: 小区负荷上报 → PCRF/SMF决策 → 动态调整带宽策略
> **辅助特性说明**: 小区负荷作为带宽策略的动态触发条件

## 1. 特性概述

### 1.1 基本信息

| 属性 | 值 |

| **适用NF** | PGW-U |
| **产品** | UDG (用户面功能) |
| **UDG版本** | UDG 20.8.2 及后续版本 |
| **UNC版本** | UNC 20.7.0 及后续版本 |

| **License控制项** | 82200DHW LKV3G5WOCR01 基于小区负荷上报的无线资源优化 |

### 1.2 核心能力一句话描述

当小区负荷状态（Cell Load Level）发生变化时，无线侧（RAN）通过GTP-U扩展头实时将小区负荷状态上报给PGW-U，PGW-U通过N4接口向PGW-C上报，PGW-C触发CELL_CONGESTION_CHANGE trigger并上报PCRF，由PCRF根据小区负荷状态动态下发策略控制规则，对部分用户或指定业务进行带宽调整，达到解除/减缓小区拥塞的目的。

### 1.3 在带宽控制场景中的定位

本特性在带宽控制场景中定位为**辅助特性**，其核心价值在于提供了一个动态的、基于无线资源实际状态的带宽策略触发条件：

- **动态触发条件**：小区负荷状态作为带宽策略调整的实时触发器。不同于BWM（基于业务感知的静态带宽控制）和FUP（基于累计流量的周期性策略控制），本特性提供的是基于无线网络实时拥塞状态的动态策略触发能力。
- **闭环带宽控制**：小区负荷高（拥塞/过载）时触发更严格的限速策略；负荷恢复正常时放宽或解除限速策略，形成闭环的动态带宽调整机制。
- **与静态带宽策略的互补**：BWM、Shaping、FUP等特性提供基于业务类型、用户套餐、流量累计的带宽控制能力，本特性在此基础上增加了基于无线资源状态的动态触发维度，使带宽控制更加精细化。

### 1.4 与其他特性的协作关系

| 关联特性 | 协作类型 | 协作说明 |
|----------|----------|----------|
| **GWFD-020351 PCC基本功能** | 依赖 | UDG侧需要开启PCC基本功能以从PCRF获取策略。本特性的策略控制完全依赖PCC架构，PCRF根据小区负荷信息进行策略决策并下发规则给PGW-C。 |
| **GWFD-110311 基于业务感知的带宽控制（BWM）** | 联动 | BWM提供基于业务类型的带宽控制规则，本特性提供动态触发条件。两者结合可实现：小区拥塞时对特定业务（如视频、P2P）触发更严格的BWM限速策略。 |
| **GWFD-020353/GWFD-110312 基于累计流量的策略控制（FUP）** | 联动 | FUP提供基于流量累计的周期性策略控制，本特性提供基于无线状态的实时触发。两者结合可实现：用户已达FUP门限且小区拥塞时叠加限速。 |
| **GWFD-020354/GWFD-110313 Shaping** | 联动 | Shaping提供流量整形能力，本特性的负荷触发可联动Shaping策略在拥塞时启用更激进的整形参数。 |

## 2. 核心概念

### 2.1 小区负荷（Cell Load Level）

小区负荷状态是本特性的核心数据单元。RAN侧（基站）实时检测每个小区的拥塞状态，并将负荷状态分为四个等级：

| 负荷等级 | 数值 | 含义 | 带宽控制策略含义 |
|----------|------|------|------------------|
| 无效值 | 0 | Invalid | 无效数据，不触发策略 |
| 正常 | 1 | Normal | 小区资源充足，可放宽限速或不执行限速 |
| 拥塞 | 2 | Congestion | 小区资源紧张，应触发限速策略，限制低优先级用户/业务带宽 |
| 过载 | 3 | Overload | 小区资源严重不足，应执行最严格的限速策略，保障高价值用户/业务 |

负荷等级是触发策略调整的关键判据。当负荷等级发生变化（如从Normal变为Congestion，或从Congestion恢复为Normal），即触发策略更新流程。

### 2.2 负荷上报机制

本特性采用**带内信令（In-band Signaling）**方式上报小区负荷，即通过用户数据报文中的GTP-U扩展头携带负荷信息，无需额外的信令通道：

- **上行方向**：RAN（eNodeB/gNodeB）在发送给PGW-U的上行用户数据报文中，通过GTP-U扩展头携带小区负荷信息。
- **下行方向**：PGW-U在发送给RAN的下行用户数据报文中，通过GTP-U扩展头携带PGW-U的能力信息。
- **透传机制**：SGW-U负责透传携带GTP-U扩展头的上下行报文，不解析也不修改扩展头内容。

### 2.3 负荷阈值与触发条件

本特性的触发条件不是基于固定的阈值配置，而是基于RAN侧检测到的负荷等级**变化事件**：

- **触发事件类型**：CELL_CONGESTION_CHANGE trigger，即小区拥塞状态变化事件。
- **触发条件**：当RAN检测到用户所在小区的负荷等级发生变化时（如Normal→Congestion、Congestion→Overload、Overload→Congestion、Congestion→Normal等），通过GTP-U扩展头上报新的负荷等级。
- **变化检测**：PGW-U负责检测收到的负荷等级是否与上次上报的不同，只有发生变化时才向PGW-C上报，避免重复上报相同状态。
- **策略决策**：具体的限速阈值和策略内容由PCRF决策，PCRF综合负荷等级、用户信息（如用户优先级、套餐）、业务信息等进行策略计算。

### 2.4 动态带宽策略调整

动态带宽策略调整是本特性在带宽控制场景中的核心价值体现：

- **PCRF策略决策**：PCRF收到PGW-C上报的小区负荷变化信息后，根据预设策略规则计算新的策略，并通过CCA或RAR消息下发新的PCC规则给PGW-C。
- **PGW-C规则下发**：PGW-C收到PCRF的新策略后，安装规则并下发给PGW-U执行。
- **PGW-U执行**：PGW-U根据最新策略，控制低优先级用户或特定业务的带宽。具体执行方式包括QoS调整、限速规则更新、门控开关等。
- **闭环调整**：当负荷恢复Normal时，PCRF下发恢复策略，PGW-U解除限制，形成闭环。

### 2.5 与PCRF/SMF的交互

本特性涉及多网元协同交互，交互链路如下：

```
RAN ──GTP-U扩展头(上行)──> PGW-U ──N4/S5──> PGW-C ──Gx(CCR)──> PCRF
RAN <──GTP-U扩展头(下行)── PGW-U <──N4/S5── PGW-C <──Gx(CCA/RAR)── PCRF
```

- **Gx接口**：PCRF与PGW-C之间的策略接口，遵循3GPP TS 29.212规范。CELL_CONGESTION_CHANGE trigger通过Gx接口下发和触发。
- **S5/N4接口**：PGW-C与PGW-U之间的接口，用于规则下发和事件上报。
- **GTP-U扩展头**：RAN与PGW-U之间的带内信令通道，遵循3GPP TS 29.060和29.274规范。

## 3. 实现原理与流程

### 3.1 小区负荷采集机制

RAN侧负责小区负荷状态的检测和采集：

1. **负荷检测**：RAN（eNodeB/gNodeB）实时监测每个小区的无线资源利用率，包括PRB利用率、调度队列长度、用户数等指标。
2. **等级判定**：RAN根据内部算法将负荷状态映射为四个等级：Normal(1)、Congestion(2)、Overload(3)，以及无效值(0)。
3. **变化检测**：RAN持续检测负荷等级是否发生变化。
4. **上报触发**：当检测到负荷等级变化时，RAN在后续发送的上行GTP-U数据报文中通过扩展头携带新的负荷等级信息。

### 3.2 能力协商过程

PCC用户激活后，PGW-U和RAN侧需要进行能力协商，确认双方是否均支持小区负荷上报功能。能力协商有两种发起方式：

#### 3.2.1 无线侧发起的能力协商

```
RAN                          PGW-U
 |                              |
 |  上行GTP-U报文(携带RAN能力)   |
 |----------------------------->|
 |                              |
 |  下行GTP-U报文(携带PGW-U能力) |
 |<-----------------------------|
 |                              |
 |     能力协商完成               |
```

当PGW-U收到RAN侧发送的携带RAN侧能力的上行GTP-U报文后，判断RAN是否支持小区负荷上报。如果RAN支持，PGW-U通过下行GTP-U报文扩展头将自身能力通知RAN侧。

#### 3.2.2 UDG(PGW-U)发起的能力协商

```
PGW-U                        RAN
 |                              |
 |  下行GTP-U报文(携带PGW-U能力) |
 |----------------------------->|
 |                              |
 |  上行GTP-U报文(携带RAN能力)   |
 |<-----------------------------|
 |                              |
 |     能力协商完成               |
```

PGW-U支持在收到第一个下行数据报文时，主动向RAN侧发起能力协商过程，通过GTP-U扩展头携带PGW-U能力信息。

#### 3.2.3 能力变化检测

能力协商完成后，UDG支持实时检测RAN侧能力的变化情况，便于实时调整UDG对数据报文的处理策略。这意味着即使初始协商时RAN不支持该功能，后续RAN升级支持后UDG也能自动感知。

### 3.3 GTP-U扩展头信元格式

本功能使用华为私有的GTP-U扩展头来携带能力信息和小区负荷信息。扩展头信元格式定义如下：

#### 3.3.1 扩展头总体格式

| 字段 | 说明 |
|------|------|
| **Length** | 该信元的长度（octets） |
| **Next Sub Extension Header Type** | 指示Sub Extension Header Content中包含的首个子信元类型 |
| **Sub Extension Header Content** | 华为扩展头信元所携带的内容，可以包含多个子信元 |
| **Next Extension Header Type** | 下一个扩展头的类型，遵循3GPP标准 |

#### 3.3.2 子信元类型定义

| Sub Extension Header Type (十进制) | 方向 | 信元内容 |
|-------------------------------------|------|----------|
| **0** | 上行/下行 | 指示没有更多的子信元 |
| **1** | 上行 | RAN节点能力信息，包括版本信息（Version） |
| **2** | 上行 | UMTS小区负荷信息，包括小区标识（CGI）、会话标识和小区负荷等级（0、1、2、3） |
| **3** | 上行 | LTE小区负荷信息，包括小区标识（ECGI）、会话标识和小区负荷等级（0、1、2、3） |
| **4** | 上行 | RAN与CN之间的心跳维护消息，只在上行报文中携带 |
| **6 ~ 7** | - | Reserved（保留） |
| **8** | 下行 | GW能力信息，包括版本信息（Version） |
| **9** | 下行 | 业务优先级标识FPI（Flow Priority Indicator） |
| **10 ~ 15** | - | Reserved（保留） |

#### 3.3.3 小区负荷等级值定义

子信元类型2（UMTS）和3（LTE）中携带的小区负荷等级定义：

| 负荷等级值 | 含义 |
|------------|------|
| 0 | 无效值（Invalid） |
| 1 | 正常（Normal） |
| 2 | 拥塞（Congestion） |
| 3 | 过载（Overload） |

### 3.4 负荷上报流程（UPF→PGW-C→PCRF）

完整的负荷上报端到端流程如下：

```
RAN          PGW-U(SGW-U透传)     PGW-C          PCRF
 |               |                  |               |
 |  [步骤1] 用户激活时，PCRF下发CELL_CONGESTION_CHANGE trigger给PGW-C     |
 |               |                  |<--------------|  CCA/RAR: trigger
 |               |                  |               |
 |  [步骤2] PCC用户激活后，PGW-U与RAN进行能力协商                          |
 |<------------->|                  |               |  GTP-U扩展头
 |   能力协商     |                  |               |
 |               |                  |               |
 |  [步骤3] RAN检测到负荷变化，通过GTP-U扩展头上报                         |
 |-------------->|                  |               |  上行GTP-U报文
 | (SGW-U透传)   |                  |               |  扩展头含负荷信息
 |               |                  |               |
 |  [步骤4] PGW-U检测到负荷变化，上报给PGW-C                               |
 |               |----------------->|               |  N4/S5上报
 |               |                  |               |
 |  [步骤5] PGW-C触发CELL_CONGESTION_CHANGE trigger，通过CCR上报PCRF       |
 |               |                  |--------------->|  CCR: 位置+负荷
 |               |                  |               |
 |  [步骤6] PCRF策略决策，下发新策略给PGW-C                                |
 |               |                  |<---------------|  CCA/RAR: 新策略
 |               |                  |               |
 |  [步骤7] PGW-C安装规则，下发给PGW-U执行                                 |
 |               |<-----------------|               |  规则下发
 |               |                  |               |
 |  [步骤8] PGW-U根据新策略控制用户/业务带宽                                |
 |               |                  |               |  执行限速/恢复
```

#### 各步骤详细说明：

**步骤1 - Trigger下发**：PCC用户激活时，PCRF将CELL_CONGESTION_CHANGE trigger下发给PGW-C。PCRF也可以选择在任何的CCA消息或RAR消息中下发该trigger给PGW-C。

**步骤2 - 能力协商**：PCC用户激活后，PGW-U和RAN侧通过GTP-U扩展头进行能力协商。只有彼此都具有小区负荷上报能力的情况下，RAN才会开始上报负荷信息。

**步骤3 - RAN负荷上报**：RAN检测到用户小区负荷状态发生变化时，通过上行数据报文中的GTP-U扩展头将用户位置信息及小区负荷信息上报给PGW-U（经SGW-U透传）。

**步骤4 - PGW-U检测变化**：PGW-U检查到小区负荷状态发生变化（与上次上报值不同），将变化信息上报给PGW-C。

**步骤5 - PGW-C触发trigger并上报PCRF**：PGW-C触发CELL_CONGESTION_CHANGE trigger，并通过CCR消息将用户位置信息及小区负荷状态上报给PCRF。

**步骤6 - PCRF策略决策**：PCRF综合上报的小区负荷信息、用户签约信息、业务信息等进行策略决策，计算新的策略规则。

**步骤7 - 策略下发**：PCRF通过CCA或RAR消息将新策略下发给PGW-C，PGW-C安装规则并下发给PGW-U。

**步骤8 - PGW-U执行**：PGW-U根据最新策略控制低优先级用户或特定业务的带宽。

### 3.5 负荷触发策略调整流程（带宽控制视角）

从带宽控制视角，本特性的策略调整流程可归纳为两种典型场景：

#### 3.5.1 高负荷触发限速场景

```
小区负荷: Normal(1) → Congestion(2) 或 Overload(3)

触发链路:
RAN上报负荷等级2/3 → PGW-U → PGW-C → PCRF
                                    ↓
PCRF策略决策:
  - 选择低优先级用户或特定业务（如P2P、视频流媒体）
  - 下发更严格的QoS规则或限速规则（URR）
  - 降低GBR/MBR速率 或 启用更低的AMBR
                                    ↓
PGW-C → PGW-U执行限速
  - 低优先级用户带宽受限
  - 特定业务流量被限速或整形
  - 高价值用户/业务不受影响或受影响最小
```

#### 3.5.2 低负荷恢复限速场景

```
小区负荷: Congestion(2)/Overload(3) → Normal(1)

触发链路:
RAN上报负荷等级1 → PGW-U → PGW-C → PCRF
                                    ↓
PCRF策略决策:
  - 恢复原有QoS规则或限速规则
  - 提高GBR/MBR速率 或 恢复原AMBR
  - 解除对低优先级用户/特定业务的限速
                                    ↓
PGW-C → PGW-U恢复带宽
  - 用户/业务带宽恢复正常
```

## 4. 配置规则

### 4.1 涉及的MML命令总览

本特性涉及的MML命令共3条：

| 序号 | 命令 | 用途 | 适用NF |
|------|------|------|--------|
| 1 | **SET LICENSESWITCH** | 打开本特性的License配置开关 | PGW-U |
| 2 | **SET APNREPORTATTR** | 开启指定APN的小区负荷上报功能 | SGW-U、PGW-U、UPF |
| 3 | **LST APNREPORTATTR** | 查询APN的上报属性配置（含拥塞控制状态） | SGW-U、PGW-U、UPF |

### 4.2 SET LICENSESWITCH - License开关配置

#### 命令功能

该命令用于设置License项的配置开关。当License文件中含有某功能的许可时，通过此命令可以设置该功能是否开通。

#### 关键参数

| 参数标识 | 参数名称 | 必选/可选 | 取值范围 | 说明 |
|----------|----------|-----------|----------|------|
| LICITEM | License项 | 必选 | 字符串，0~255字符 | 指定需要开通的License项。本特性使用 `LKV3G5WOCR01` |
| SWITCH | 开关 | 必选 | ENABLE / DISABLE | 开启或关闭License控制项 |

#### 注意事项

- 该命令执行后立即生效。
- 当License文件未激活时，无法设置任何License项。
- 激活后只能设置已购买且支持配置开关的License项。

#### 配置示例

```
SET LICENSESWITCH:LICITEM="LKV3G5WOCR01",SWITCH=ENABLE;
```

### 4.3 SET APNREPORTATTR - APN上报属性配置

#### 命令功能

该命令用于配置性能统计使用的APN类型、头增强使用的APN类型、业务报表使用的APN类型以及支持拥塞控制的APN类型。

#### 关键参数

| 参数标识 | 参数名称 | 必选/可选 | 取值范围 | 说明 |
|----------|----------|-----------|----------|------|
| APN | APN名称 | 必选 | 字符串，1~63字符 | 指定配置上报属性的APN。只能由"-"、数字、大小写字母和"."组成 |
| HEADERENRICH | 上报给头增强的APN名 | 可选 | REQUESTED / SERVICE | 使用请求的APN还是真实的APN |
| PERFORMANCE | 上报给话统的APN名 | 可选 | REQUESTED / SERVICE | 使用请求的APN还是真实的APN |
| SERVICEREPORT | 业务报表使用的APN名 | 可选 | REQUESTED / SERVICE | 使用请求的APN还是真实的APN |
| CONGESTIONRPT | 拥塞控制 | 可选 | DISABLE / ENABLE | **本特性核心参数**：表示当前APN下的用户是否使能小区拥塞上报功能 |

#### CONGESTIONRPT参数详解

| 取值 | 含义 | 带宽控制场景影响 |
|------|------|------------------|
| **ENABLE** | 使能小区拥塞上报 | 开启后，该APN下的用户将参与小区负荷上报流程，PCRF可根据负荷动态调整带宽策略 |
| **DISABLE** | 不使能小区拥塞上报 | 关闭后，该APN下的用户不受小区负荷触发影响，维持原有静态带宽策略 |

#### 初始值

| 参数标识 | HEADERENRICH | PERFORMANCE | SERVICEREPORT | CONGESTIONRPT |
|----------|--------------|-------------|---------------|---------------|
| 初始值 | SERVICE | SERVICE | SERVICE | **DISABLE** |

> CONGESTIONRPT默认为DISABLE，使用本特性前必须显式设置为ENABLE。

#### 注意事项

- 该命令执行后立即生效。
- 系统最多支持配置10000条ApnReportAttr。
- 每个APN支持一条配置。
- UPF支持虚拟APN、别名APN、普通APN。
- 修改perf统计APN之前，先去活原来的APN下的所有用户。

#### 配置示例

```

```

## 5. 配置案例

### 5.1 案例1：高负荷时触发P2P业务限速

#### 场景描述

某运营商网络中，小区在晚间高峰时段出现拥塞，P2P下载业务占用大量带宽。运营商希望通过小区负荷上报功能，在小区拥塞时自动对P2P业务触发限速策略，保障高价值用户的业务体验。当小区负荷恢复正常时，自动解除对P2P业务的限速。

#### 前提条件

- RAN侧（V100R017C13及后续版本）已配置小区拥塞状态上报开关和上报周期。
- PCRF侧（V100R017C13及后续版本）已配置CELL_CONGESTION_CHANGE trigger下发规则和拥塞时的PCC策略。
- PGW-C侧（UNC 20.7.0及后续版本）已部署。
- PGW-U侧（UDG 20.8.2及后续版本）已部署。
- 已加载本特性的License。

#### UDG侧配置步骤

**步骤1：打开License开关**

```
SET LICENSESWITCH:LICITEM="LKV3G5WOCR01",SWITCH=ENABLE;
```

**步骤2：开启目标APN的小区负荷上报功能**

```
SET APNREPORTATTR:APN="internet",CONGESTIONRPT=ENABLE;
```

**步骤3：验证配置**

```
LST APNREPORTATTR:APN="internet";
```

预期输出中"拥塞控制"字段应为"使能"。

#### 端到端工作流程

1. **用户激活**：PCC用户在APN "internet" 下激活会话，PCRF下发CELL_CONGESTION_CHANGE trigger给PGW-C。
2. **能力协商**：PGW-U与RAN通过GTP-U扩展头完成能力协商，双方确认支持小区负荷上报。
3. **正常状态**：小区负荷为Normal(1)，用户正常使用所有业务，P2P业务无限制。
4. **拥塞触发**：晚间高峰，RAN检测到小区负荷从Normal变为Congestion(2)，通过上行GTP-U报文扩展头上报。
5. **负荷传递**：PGW-U收到负荷变化，上报PGW-C，PGW-C触发trigger并通过CCR上报PCRF。
6. **策略决策**：PCRF根据Congestion等级和预设策略，下发新的PCC规则，对P2P业务执行限速（如MBR降到1Mbps）。
7. **策略执行**：PGW-C将规则下发给PGW-U，PGW-U对P2P业务流量执行限速。
8. **负荷恢复**：高峰过后，RAN检测到负荷恢复Normal(1)，同样流程上报。
9. **策略恢复**：PCRF下发恢复策略，PGW-U解除P2P业务限速。

#### 关键说明

- UDG侧配置仅需2步（License + APN拥塞控制开关），具体限速策略在PCRF侧配置。
- 本案例中UDG的角色是"透传+上报"：透传GTP-U扩展头给PGW-C，执行PGW-C下发的限速规则。
- PCRF侧策略配置超出UDG MML命令范围，需要通过PCRF策略管理系统配置。

### 5.2 案例2：多APN场景下的选择性负荷上报

#### 场景描述

某运营商在UDG上配置了多个APN，包括互联网APN（internet）、企业专线APN（enterprise）、物联网APN（iot）。运营商希望仅在互联网APN和企业专线APN上开启小区负荷上报功能，物联网APN由于业务特性（小流量、低带宽）暂不开启。

#### 配置步骤

**步骤1：打开License开关（全局生效，只需一次）**

```
SET LICENSESWITCH:LICITEM="LKV3G5WOCR01",SWITCH=ENABLE;
```

**步骤2：开启internet APN的负荷上报**

```
SET APNREPORTATTR:APN="internet",CONGESTIONRPT=ENABLE;
```

**步骤3：开启enterprise APN的负荷上报**

```
SET APNREPORTATTR:APN="enterprise",CONGESTIONRPT=ENABLE;
```

**步骤4：确认iot APN保持默认关闭状态**

```
LST APNREPORTATTR:APN="iot";
```

预期输出中"拥塞控制"字段应为"不使能"（默认值DISABLE）。

**步骤5：批量验证配置**

```
LST APNREPORTATTR;
```

查看所有APN的拥塞控制状态，确认internet和enterprise为"使能"，iot为"不使能"。

#### 关键说明

- CONGESTIONRPT是APN级别的开关，可按APN独立配置。

### 5.3 案例3：与BWM联动的动态带宽控制

#### 场景描述

某运营商已部署GWFD-110311（基于业务感知的带宽控制BWM），对不同业务配置了静态限速规则。现希望叠加小区负荷上报功能，实现：正常时维持原有BWM限速参数，拥塞时对特定业务（如视频流媒体）触发更严格的限速参数。

#### UDG侧配置

**步骤1：确认BWM功能已配置并正常运行**（参考GWFD-110311特性配置）

**步骤2：打开本特性License**

```
SET LICENSESWITCH:LICITEM="LKV3G5WOCR01",SWITCH=ENABLE;
```

**步骤3：开启负荷上报**

```
SET APNREPORTATTR:APN="internet",CONGESTIONRPT=ENABLE;
```

#### 联动机制说明

```
┌──────────────────────────────────────────────────────────┐
│                 动态带宽控制策略矩阵                        │
├──────────┬─────────────────┬─────────────────────────────┤
│ 小区负荷  │ BWM限速(静态)    │ 负荷触发限速(动态叠加)       │
├──────────┼─────────────────┼─────────────────────────────┤
│ Normal   │ 视频限速10Mbps   │ 无叠加，维持BWM原有参数       │
│ (1)      │ P2P限速5Mbps    │                             │
├──────────┼─────────────────┼─────────────────────────────┤
│ Congestion│ 视频限速10Mbps  │ PCRF下发新规则：             │
│ (2)      │ P2P限速5Mbps    │  - 视频叠加到5Mbps           │
│          │                 │  - P2P叠加到1Mbps           │
├──────────┼─────────────────┼─────────────────────────────┤
│ Overload │ 视频限速10Mbps  │ PCRF下发最严格规则：          │
│ (3)      │ P2P限速5Mbps    │  - 视频叠加到2Mbps           │
│          │                 │  - P2P完全阻塞               │
└──────────┴─────────────────┴─────────────────────────────┘
```

> 注：表中"视频限速10Mbps"等为BWM静态配置的示例值，"叠加"参数由PCRF根据负荷等级动态决策并通过Gx接口下发。UDG侧无需额外配置动态限速参数，仅负责执行PCRF下发的规则。

## 6. 验证与调测

### 6.1 负荷上报功能验证

#### 6.1.1 配置验证

执行以下命令验证UDG侧配置是否正确：

```
LST APNREPORTATTR:APN="internet";
```

检查输出结果中：
- "拥塞控制"字段必须为"使能"
- APN名称与规划一致

#### 6.1.2 License验证

验证License开关是否已开启：

```
LST LICENSESWITCH:LICITEM="LKV3G5WOCR01";
```

确认SWITCH状态为ENABLE。

#### 6.1.3 能力协商验证

小区负荷上报功能依赖PGW-U与RAN之间的能力协商。验证要点：
- 确认RAN侧已开启小区拥塞状态上报功能
- 确认RAN侧版本为V100R017C13及后续版本
- 确认用户为PCC本地用户（非拜访用户、非漫游用户）
- 检查UDG日志中是否有能力协商成功记录

### 6.2 策略调整效果验证

#### 6.2.1 负荷变化触发验证

当小区负荷发生变化时，验证策略调整是否生效：

1. **监控GTP-U扩展头**：在UDG上检查是否收到RAN上报的携带负荷信息的GTP-U扩展头。
2. **监控trigger触发**：在PGW-C上检查是否成功触发CELL_CONGESTION_CHANGE trigger。
3. **监控PCRF策略下发**：在PGW-C上检查是否收到PCRF下发的新策略。
4. **监控PGW-U执行**：在UDG上检查是否收到并执行了新的限速规则。

#### 6.2.2 带宽控制效果验证

当拥塞策略生效后：
- 使用流量测试工具验证受影响的用户/业务带宽是否符合PCRF下发的限速值。
- 验证高价值用户/高优先级业务不受影响。
- 当负荷恢复Normal后，验证限速策略被正确解除。

### 6.3 常见问题排查

#### 6.3.1 负荷信息未上报

| 可能原因 | 排查方法 | 解决方案 |
|----------|----------|----------|
| CONGESTIONRPT未开启 | 执行LST APNREPORTATTR检查 | SET APNREPORTATTR开启 |
| License未加载或未开启 | 执行LST LICENSESWITCH检查 | 加载License并开启开关 |
| RAN侧未开启上报 | 检查RAN侧配置 | 在RAN侧开启拥塞状态上报 |
| 能力协商失败 | 检查RAN和PGW-U是否均为华为设备 | 确认双方设备版本和能力 |
| 用户非PCC本地用户 | 检查用户类型 | 本特性不支持拜访用户和漫游用户 |

#### 6.3.2 报文单通导致负荷上报异常

**问题描述**：用户建立缺省承载和专有承载后，无线侧累计发送上行GTP-U报文100次而无下行响应时（即报文单通），无线侧再发起GTP-U报文时不会携带GTP-U扩展头，导致UDG上报的小区负荷状态可能与实际不符。

**排查方法**：
- 检查是否存在单通链路
- 检查UDG与RAN之间的连通性

**解决方案**：
- 修复单通链路问题
- 在单通场景下，UDG上报的负荷状态可能不准确，需结合其他监控手段综合判断

#### 6.3.3 策略未生效

| 可能原因 | 排查方法 | 解决方案 |
|----------|----------|----------|
| PCRF未配置CELL_CONGESTION_CHANGE trigger | 检查PCRF策略配置 | 在PCRF上配置trigger下发 |
| PGW-C未收到PCRF策略 | 检查Gx接口信令 | 排查Gx接口连通性 |
| PGW-U未收到PGW-C规则 | 检查N4/S5接口信令 | 排查PGW-C到PGW-U的规则下发 |
| PCC基本功能未开启 | 检查GWFD-020351 PCC基本功能状态 | 开启PCC基本功能 |

#### 6.3.4 MTU规划问题

**问题描述**：GTP-U扩展头增加了IP报文长度，可能导致超过网络MTU值，引发分片或丢包。

**排查方法**：
- 检查RAN和核心网之间的MTU配置
- 检查是否有IP分片告警

**解决方案**：
- 整网规划MTU值，考虑GTP-U扩展头的额外开销
- 确保传输网络MTU能够容纳扩展头

| **用户** | 有效保障高价值用户的业务体验。用户可以有选择地在非拥塞时段使用高带宽业务。 |
