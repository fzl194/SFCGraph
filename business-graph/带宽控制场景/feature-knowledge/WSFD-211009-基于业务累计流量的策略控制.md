# WSFD-211009 基于业务累计流量的策略控制 - 特性知识文档

> **场景归属**: 业务感知 → 带宽控制子场景（FUP业务级累计流量触发）
> **产品**: UNC (SMF网元)
> **场景定位**: 业务级FUP，按SVC/APP累计流量触发策略
> **对应UDG特性**: GWFD-110312 基于业务累计流量的策略控制
> **与WSFD-109104差异**: 本特性针对业务级（per SVC/per APP），WSFD-109104针对会话级

---

## 1. 特性概述

### 1.1 基本信息

| 项目 | 内容 |
|------|------|
| 特性ID | WSFD-211009 |
| 特性名称 | 基于业务累计流量的策略控制 |
| 适用NF | GGSN、PGW-C、SMF（均为UNC产品） |
| 网元角色 | UNC（GGSN-C/PGW-C/SMF）控制面 + UDG（GGSN-U/PGW-U/UPF）用户面 |
| 首次发布版本 | UNC 20.3.0 |
| License控制项 | 82200BNU LKV2FUPSAT01 基于业务累计流量的策略控制 |

### 1.2 核心能力一句话描述

针对用户访问的特定业务（如P2P、视频、IM）按一段时间内的业务累计流量进行策略控制，当业务累计流量达到预设阈值时，改变计费费率、阻塞用户业务或降低业务速率。

### 1.3 在带宽控制场景中的定位

本特性是 FUP（Fair Usage Policy）的**业务级版本**，与 WSFD-109104（会话级FUP）构成两个粒度互补的累计流量策略控制能力。在带宽控制子场景中：

- **业务级FUP**（本特性）：按业务（per SVC/per APP）累计流量，例如P2P业务一周内最多50MB，视频业务一个月内最多100GB。不同业务独立累计、独立触发策略。
- **会话级FUP**（WSFD-109104）：按整会话累计流量，不区分业务类型，例如用户一个月内总流量100GB后降速。

本特性支持两种接口路径：
- **Gx接口**：4G/5G融合场景，PCRF通过Gx接口下发业务级Usage Monitoring阈值（Usage-Monitoring-Level=PCC_RULE_LEVEL），PCEF(UNC)上报业务级Usage。
- **N7接口**：纯5G场景，PCF通过N7接口下发业务级流量阈值（PccRule.refUmData引用umDecs.umId），SMF(UNC)转换为N4 URR规则，URR只绑定在指定业务流对应的PDR上。

### 1.4 与WSFD-109104（会话级FUP）的差异

| 维度 | WSFD-211009 业务级FUP（本特性） | WSFD-109104 会话级FUP |
|------|------|------|
| 累计粒度 | 按业务（per SVC/per APP） | 按整会话（per Session） |
| Gx接口UM Level | Usage-Monitoring-Level = PCC_RULE_LEVEL(1) | Usage-Monitoring-Level = SESSION_LEVEL(0) |
| N7接口引用方式 | PccRule.refUmData引用umDecs.umId，URR绑定指定业务流的PDRs | umDecs独立下发，URR绑定会话所有PDRs |
| Monitoring-Key作用 | 标识一种业务（per SVC），不同业务不同Key | 标识会话级累计，与规则配置无关 |
| 典型场景 | 视频业务月限100GB后降速、P2P业务周限50MB | 用户月总流量100GB后降速 |
| License | LKV2FUPSAT01 | LKV3W9PCBT12 |
| FUPSESSIONEXC | 可设ENABLE排除Session级累计 | 不涉及该参数 |

### 1.5 与UDG侧 GWFD-110312 的对应关系

| 维度 | WSFD-211009（UNC侧） | GWFD-110312（UDG侧） |
|------|------|------|
| 产品 | UNC（SMF/PGW-C/GGSN-C）控制面 | UDG（UPF/PGW-U/GGSN-U）用户面 |
| 职责 | 接收PCRF/PCF阈值、转换为N4 URR规则、累计业务流量、向PCRF/PCF上报 | 执行业务数据检测、流量累计计数、策略执行（降速/阻塞） |
| 接口 | Gx（PCRF）、N7（PCF）、N4（UPF） | N4（SMF） |

### 1.7 客户价值

| 受益方 | 受益描述 |
|--------|----------|
| 运营商 | 通过用户访问业务的累计流量进行带宽和计费策略控制，为运营商提供灵活的流量控制解决方案，优化网络资源分配。 |
| 用户 | 对恶意抢占网络资源的业务（如P2P）使用该特性，可以保证用户公平使用网络资源，提升用户上网体验。 |

### 1.8 典型应用场景

1. **业务总量限制场景**：限制某类业务在固定周期内总的使用量，比如P2P业务在一周内使用总量最多为50MB。
2. **FUP公平使用场景**：通过限制用户某类业务在某周期内的使用量，在用户业务访问量超出限额时降低其访问速率或提高费率，达到网络公平使用的原则。对于包月用户，随着用户特定业务使用量的增加，逐步降低用户业务带宽，直至阻塞业务。

## 2. 核心概念

### 2.1 通用概念

| 概念 | 定义 |
|------|------|
| **业务级累计流量策略控制** | 针对用户访问的特定业务按一段时间内的业务累计流量进行策略控制，当业务累计流量达到阈值时改变计费和控制策略 |
| **业务级FUP** | 公平使用策略的业务级版本，按业务粒度限制使用量，超限后降低该业务速率或提高费率 |
| **会话级FUP** | 按整会话粒度累计流量（对应WSFD-109104），与本特性构成互补关系 |
| **业务累计流量** | 以PCC规则（per SVC/per APP）为粒度的流量累计，区别于会话级流量 |

### 2.2 Gx接口特有概念（4G/5G融合）

| 概念 | 定义 |
|------|------|
| **Usage Monitoring机制** | 3GPP标准的流量监控机制，PCRF通过Gx接口下发业务级流量阈值，PCEF上报业务级累计流量 |
| **Usage-Monitoring-Information (UMI)** | Gx接口AVP，用于业务级流量阈值下发和流量累计上报的核心信元 |
| **Monitoring-Key** | UMI的子AVP，用于标识一种业务。UNC根据PCRF下发的Monitoring-Key执行业务级累计流量动作。对于PCC动态规则，Monitoring-Key AVP在Charging-Rule-Definition中扩展；对于PCC预定义规则，Monitoring-Key在UNC使用ADD PCCPOLICYGRP命令配置；对于PCC预定义规则组，Monitoring-Key在UNC使用ADD USERPROFILE命令配置 |
| **Granted-Service-Unit** | UMI的子AVP，用于业务级流量阈值下发 |
| **Used-Service-Unit** | UMI的子AVP，用于业务级流量累计上报 |
| **Usage-Monitoring-Level** | UMI的子AVP，业务级累计流量时取值为PCC_RULE_LEVEL(1) |
| **Usage-Monitoring-Report** | UMI的子AVP，指示UNC应该上报的累积流量 |
| **Usage-Monitoring-Support** | UMI的子AVP，指示Monitoring Key值的配额监控去使能 |
| **UMCH（Usage Monitoring Congestion Handling）** | Usage Monitoring拥塞处理机制。PCRF通过Monitoring-Time AVP提前指定配额重置时间点，PCEF分别统计重置时间点前后的流量，通过重置后最近一次流量上报流程一起上报给PCRF/PCF，避免Gx接口信令拥塞。需ADD PCRF命令的FEATURELIST包含"UMCH" |
| **Event-Trigger** | Gx接口触发条件，业务级累计流量上报对应Usage-Report触发条件 |

### 2.3 N7接口特有概念（纯5G）

| 概念 | 定义 |
|------|------|
| **SmPolicyDecision** | PCF下发给SMF的会话管理策略决策，包含业务级流量阈值 |
| **PccRule** | SmPolicyDecision的主信元，包含pccRuleId、flowInfos（业务流描述）、refUmData（引用使用监控数据） |
| **refUmData** | PccRule的子信元，引用umDecs中的使用监控数据决策。业务级累计流量在PccRule中定义，通过umID引用 |
| **umDecs** | DecisionData，包含umId（使用监控策略数据标识）和volumeThreshold（总流量阈值） |
| **umId** | 标识使用监控策略数据在一个PDU会话中 |
| **policyCtrlReqTriggers** | 标识达到流量阈值后上报的触发条件，如US_RE |
| **suppFeat** | 标识特性协商结果，如UMC（Usage Monitoring Control） |
| **authSessAmbr** | Npcf_SMPolicyControl_Update Response中携带的信元组，负责更新QoS规则实现带宽控制 |
| **accuUsageReports** | SMF发起的Npcf_SMPolicyControl_Update Request中携带的信元，上报业务累计流量 |
| **预定义规则（L7层）** | N7接口的L7层业务级累计流量支持预定义规则配置。PGW-C/SMF配置的预定义PCC规则名和PCF保持一致，预定义URR ID和umID的映射关系也需配置一致 |

### 2.4 N4接口概念

| 概念 | 定义 |
|------|------|
| **URR（Usage Reporting Rule）** | N4接口使用量上报规则，SMF根据PCC规则和umDecs生成 |
| **URR ID与UMID映射** | SMF建立URR ID和UMID之间的映射关系，用于流量上报时的关联 |
| **预定义URR ID** | L7层场景使用，URR ID采用最高位bit区分预定义URR和动态URR。高位bit为1代表预定义URR ID |
| **Activate Predefined Rule** | PFCP Session Establishment Request中Create PDR的子信元，下发预定义规则名 |

### 2.5 业务级FUP vs 会话级FUP 核心差异

| 维度 | 业务级FUP（本特性） | 会话级FUP（WSFD-109104） |
|------|------|------|
| 累计对象 | 特定业务（视频/P2P/IM各自累计） | 整会话所有流量 |
| 监控粒度 | per SVC / per APP / per PCC Rule | per Session |
| Gx Usage-Monitoring-Level | PCC_RULE_LEVEL(1) | SESSION_LEVEL(0) |
| N7 PccRule.refUmData | 引用umDecs.umId，URR绑定指定业务流PDRs | umDecs独立，URR绑定会话全部PDRs |
| Monitoring-Key语义 | 标识一种业务，不同业务不同Key | 标识会话级累计 |
| 典型触发动作 | 对特定业务降速/限速/重定向/阻塞 | 对整会话降速/限速 |

## 3. 实现原理与流程

### 3.1 Gx接口业务级UM实现原理

基于业务级的累计流量策略控制采用3GPP标准的Usage Monitoring机制。关键AVP和触发条件如下：

| 实现方式 | 关键AVP | AVP说明 | Event-Trigger |
|----------|---------|---------|---------------|
| 3GPP标准Usage Monitoring | Usage-Monitoring-Information | 用于业务级流量阈值下发和流量累计上报 | Usage-Report |
| 3GPP标准Usage Monitoring | Monitoring-Key | 标识一种业务，UNC根据PCRF下发的Monitoring-Key执行累计流量动作。PCC动态规则在Charging-Rule-Definition中扩展；PCC预定义规则用ADD PCCPOLICYGRP配置；PCC预定义规则组用ADD USERPROFILE配置 | Usage-Report |
| 3GPP标准Usage Monitoring | Granted-Service-Unit | 用于业务级流量阈值下发 | Usage-Report |
| 3GPP标准Usage Monitoring | Used-Service-Unit | 用于业务级流量累计上报 | Usage-Report |
| 3GPP标准Usage Monitoring | Usage-Monitoring-Level | 业务级累计流量取值为PCC_RULE_LEVEL(1) | Usage-Report |
| 3GPP标准Usage Monitoring | Usage-Monitoring-Report | 指示UNC应该上报的累积流量 | Usage-Report |
| 3GPP标准Usage Monitoring | Usage-Monitoring-Support | 指示Monitoring Key值的配额监控去使能 | Usage-Report |

**UMCH（Usage Monitoring Congestion Handling）机制**：
对于包月用户，需要在某个时间点（如月末）进行配额重置。如果PCRF/PCF同时要求所有在线用户上报流量，会增加Gx接口信令负荷造成拥塞；如果分批要求上报，则流量上报有延迟且不准确。UMCH机制解决方案：
- PCRF通过Monitoring-Time AVP提前指定配额重置的时间点
- PCEF分别统计配额重置时间点前后的流量
- 通过重置后最近一次的流量上报流程一起上报给PCRF/PCF
- 开启条件：ADD PCRF命令的FEATURELIST参数包含"UMCH"
- 限制：UMCH仅支持总流量阈值

### 3.2 Gx接口业务级累计流量策略控制流程（2/3/4G用户）

完整流程参见源文档"基于业务累计流量的策略控制流程图"。流程概述：

1. PCRF/PCF对用户业务进行授权，下发业务级流量上报阈值、计费规则和QoS规则给UNC（GGSN-C/PGW-C/SMF）
2. UNC将PCC规则下发给GGSN-U/PGW-U/UPF
3. GGSN-U/PGW-U/UPF检测到用户数据后，执行PCRF/PCF下发的计费动作和QoS动作
4. 达到上报触发条件时，GGSN-U/PGW-U/UPF向UNC上报用户业务的流量
5. UNC累计用户业务的流量，达到上报触发条件时，向PCRF/PCF上报业务级流量（CCR-U携带Usage-Monitoring-Information）
6. PCRF/PCF对用户业务流量进行累计，根据设置条件进行策略决策和策略下发

### 3.3 N7接口L3/L4层业务级累计流量策略控制流程

适用于纯5G SA场景，PCF通过N7接口下发策略。完整流程（14步）：

1. UE发起PDU Session Establishment流程
2. AMF进行PGW-C/SMF选择，向PGW-C/SMF发送Nsmf_PDUSession_CreateSMContext消息
3. PGW-C/SMF收到消息后，发送Npcf_SMPolicyControl_Create Request (SmPolicyContextData)请求会话策略，其中suppFeat字段指示是否支持Usage Monitoring Control功能
4. PCF从UDR获取策略签约数据
5. PCF进行特性协商，同时进行策略决策生成会话管理策略（包括FUP相关策略）。PCF回复Npcf_SMPolicyControl_Create Response（SmPolicyDecision），包括下发业务级流量阈值

**SmPolicyDecision主要信元结构（L3/L4层）**：

| 主信元 | 子信元 | 信元解释 |
|-------|--------|---------|
| PccRule | - | PCCRule |
| PccRule | pccRuleId | PCC规则标识 |
| PccRule | flowInfos | 业务流描述 |
| PccRule | refUmData | 对使用监控数据决策的引用。即业务级累计流量在PCCRule中定义，通过umID引用 |
| umDecs | - | DecisionData |
| umDecs | umId | 标识使用监控策略数据在一个PDU会话 |
| umDecs | volumeThreshold | 总流量阈值 |
| policyCtrlReqTriggers | - | 标识达到流量阈值后上报的触发条件，如US_RE |
| suppFeat | - | 标识特性协商结果，如UMC |

6. PGW-C/SMF根据Response消息中pccRule和umDecs等生成N4接口的URR规则。**关键点**：由于PccRule只对指定业务流生效，该URR**只绑定在指定业务流对应的PDRs上**。同时PGW-C/SMF建立URR ID和UMID之间的映射
7. PGW-C/SMF向PGW-U/UPF发送PFCP Session Establishment Request（Create PDR），SDF Filter中的Flow Description从PccRule中的flowInfos提取
8. PGW-C/SMF收到PFCP Session Establishment Response后进行后续流程处理，PDU会话建立成功
9. 用户访问指定数据业务，PGW-U/UPF检测到该业务流量阈值耗尽，发送PFCP Session Report Request通知PGW-C/SMF，消息中携带Usage Report信元组（记录用户使用的流量等信息）
10. PGW-C/SMF根据URR ID和UMID映射关系，向PCF通知流量阈值耗尽。业务累计流量上报信息从N4接口Usage Report中提取
11. PGW-C/SMF发送Npcf_SMPolicyControl_Update Request (SmPolicyUpdateContextData)通知PCF流量阈值耗尽
12. PCF进行策略决策，不再下发流量阈值，同时更新QoS规则进行带宽控制（如上行带宽1 Mbit/s，下行带宽2 Mbit/s）。Npcf_SMPolicyControl_Update Response中authSessAmbr信元组负责更新QoS规则
13. PGW-C/SMF收到Response后检查是否下发新流量阈值。若无新阈值，通知PGW-U/UPF不再上报流量
14. PGW-C/SMF发送PFCP Session Modification Request，通知PGW-U/UPF更新业务规则

### 3.4 N7接口L7层业务级累计流量策略控制流程

适用于纯5G SA场景下的L7层（应用层）业务识别和累计流量控制。与L3/L4层的核心差异在于使用**预定义规则**机制。完整流程（15步）：

**预定义规则配置前提**（PGW-C/SMF/PCF/PGW-U/UPF需保持一致）：
- PGW-C/SMF配置的预定义PCC规则名和PCF保持一致
- PGW-C/SMF配置预定义URR ID和umID对应关系，umID和PCF保持一致
- PGW-C/SMF配置的预定义规则名和PGW-U/UPF保持一致
- PGW-C/SMF配置的预定义URR ID和PGW-U/UPF保持一致

1. UE发起PDU Session Establishment流程
2. AMF进行PGW-C/SMF选择，发送Nsmf_PDUSession_CreateSMContext消息
3. PGW-C/SMF发送Npcf_SMPolicyControl_Create Request请求会话策略，suppFeat指示是否支持Usage Monitoring Control
4. PCF从UDR获取策略签约数据
5. PCF进行特性协商和策略决策，回复Npcf_SMPolicyControl_Create Response（SmPolicyDecision），包括下发业务级流量阈值
   - 预定义规则在pccRules中定义，只需定义pccRuleId和refCondData（可选）
   - umDecs中定义流量阈值，但umId和pccRule无绑定关系
6. PGW-C/SMF根据预定义规则名读取该预定义规则配置，获得预定义URR ID和UMID的映射关系，同时生成预定义规则PDR和预定义URR
7. PGW-C/SMF向PGW-U/UPF发送PFCP Session Establishment Request（Create PDR），消息中下发预定义规则

**Create PDR主要子信元（L7层预定义规则）**：

| 子信元 | 信元解释 |
|--------|---------|
| Activate Predefined Rule | 预定义规则名 |
| URR ID | 标识上报的URR。URR ID采用最高位bit区分预定义URR和动态URR，高位bit为1代表预定义URR ID |

8. PGW-C/SMF收到PFCP Session Establishment Response后处理后续流程，PDU会话建立成功
9. 用户访问数据业务，PGW-U/UPF检测到指定L7业务流量阈值耗尽，发送PFCP Session Report Request通知PGW-C/SMF，消息中携带Usage Report
10. PGW-C/SMF根据URR ID和UMID映射关系，向PCF通知指定L7业务流量阈值耗尽
11. PGW-C/SMF发送Npcf_SMPolicyControl_Update Request通知PCF流量阈值耗尽
12. PCF进行策略决策，不再下发流量阈值，同时更新QoS规则进行带宽控制（authSessAmbr负责更新QoS规则）
13. PGW-C/SMF收到Response后检查是否有新流量阈值。若无，通知PGW-U/UPF不再上报
14. PGW-C/SMF发送PFCP Session Modification Request，通知PGW-U/UPF更新业务规则

### 3.5 L3/L4层 vs L7层业务级累计流量对比

| 维度 | L3/L4层 | L7层 |
|------|---------|------|
| 业务识别方式 | 五元组/SDF Filter（IP+端口） | 应用层特征（DPI） |
| PCC规则定义 | PCF动态下发PccRule（含flowInfos） | 预定义规则，PGW-C/SMF本地配置 |
| URR生成方式 | SMF根据PccRule和umDecs动态生成 | SMF根据预定义规则配置生成预定义URR |
| URR与PDR绑定 | URR只绑定指定业务流对应的PDRs | 预定义URR，高位bit为1 |
| umId与pccRule关系 | refUmData引用umDecs.umId | umId和pccRule无绑定关系（映射在本地配置） |
| 规则名一致性要求 | PCF动态下发，无需预配置 | PGW-C/SMF/PCF/PGW-U/UPF四方需保持一致 |

### 3.6 业务识别到策略触发完整链路

```
业务识别 → 流量累计 → 阈值匹配 → 策略触发

1. [业务识别]
   Gx路径: PCRF下发PCC规则（含Monitoring-Key标识业务）→ PCEF下发UPF
   N7 L3/L4路径: PCF下发PccRule（含flowInfos+refUmData）→ SMF生成URR绑定指定业务流PDR
   N7 L7路径: 预定义规则 → SMF/UPF本地DPI识别

2. [流量累计]
   UPF按URR规则对指定业务流累计流量
   UM按业务（per SVC/per APP）独立累计

3. [阈值匹配]
   UPF检测业务流量达到volumeThreshold → PFCP Session Report Request
   SMF根据URR ID→UMID映射关联到具体业务

4. [策略触发]
   Gx: SMF上报CCR-U(含Usage-Monitoring-Information) → PCRF决策 → CCA下发新QoS
   N7: SMF上报Npcf_SMPolicyControl_Update(含accuUsageReports) → PCF决策 → Response含authSessAmbr
   触发动作: 降速(MBR变更)/限速/重定向/阻塞
```

| 配置维度 | Gx接口（4G/5G融合） | N7接口（纯5G） |

| UMID映射 | Monitoring-Key与URR ID的映射 | URR ID与UMID的映射 |

### 4.7 软件参数（软参）

| 软参 | 说明 |
|------|------|
| DWORD504 BIT21 | 控制SMF是否主动查询并向PCF上报没有绑定任何动态规则的业务级URR的流量 |
| BIT1538 | 控制Gx接口某些信令交互流程中流量的上报 |
| BIT162 | 控制使用R9标准协议规定的累计流量上报是否采用核减抹平机制 |
| BIT1876 | Gx接口是否在接入侧更新消息触发的CCR请求消息中携带累计流量信息 |
| BIT893 | 控制FUP上报流程中，CCA返回的Session级别FUP的key值发生变化时的业务处理原则 |

### 4.8 约束条件与应用限制

- **UMCH限制**：UMCH（Usage Monitoring Congestion Handling）仅支持总流量阈值
- **UE Category用户阈值下限**：对于UE Category用户，要求PCRF/PCF下发的业务级流量阈值不低于1GB，否则影响该用户业务流量累计的准确性
- **系统性能影响**：启用该功能需要GGSN-C/PGW-C/SMF与PCRF/PCF不断交互，同时动态处理策略下发，对系统性能影响较大，规划时需考虑启用该特性的用户比例
- **License前置**：必须先获得License许可（LKV2FUPSAT01），并打开SET LICENSESWITCH
- **PCC基本功能前置**：必须先完成WSFD-109101 PCC基本功能激活
- **Monitoring-Key一致性**：SET PCCFUNC的MKPARSEFORMAT需与PCRF/PCF保持一致
- **URR ID一致性**：ADD URR的URRID需与PGW-U/UPF上的URR标识保持一致

## 5. 配置案例

### 5.1 案例1：P2P业务累计流量超阈值降速

**场景描述**：运营商对P2P业务实施FUP，用户P2P业务月累计流量达到50MB后，将P2P业务带宽降至上行1Mbit/s、下行2Mbit/s。

**配置命令序列**：

```mml
// 步骤1：打开本特性的License配置开关
SET LICENSESWITCH:LICITEM="LKV2FUPSAT01",SWITCH=ENABLE;

// 步骤2：设置Monitoring-Key的解析方式（与PCRF/PCF协商一致）
SET PCCFUNC:MKPARSEFORMAT=UNSIGNED32;

// 步骤3：配置P2P业务的URR标识与监控属性值绑定
// MONITORINGKEY="2001"标识P2P业务，需与PCRF/PCF一致
ADD URR:URRNAME="urr_p2p_01",URRID=2000,USAGERPTMODE=MONITORINGKEY,MONITORINGKEY="2001";

// 步骤4：配置使用量上报规则组（上行/下行均引用同一URR）
ADD URRGROUP:URRGROUPNAME="urrgrp_p2p_01",UPURRNAME1="urr_p2p_01",DOWNURRNAME1="urr_p2p_01";

// 步骤5：配置PCC策略组，绑定URR组
// FUPSESSIONEXC=ENABLE表示P2P业务流量不计入Session级累计
ADD PCCPOLICYGRP:PCCPOLICYGRPNM="pccpolicygrp_p2p_01",URRGROUPNAME="urrgrp_p2p_01",FUPSESSIONEXC=ENABLE;

// 步骤6：配置P2P业务规则
ADD RULE:RULENAME="rule_p2p_fup",POLICYTYPE=PCC,PRIORITY=20,POLICYNAME="pccpolicygrp_p2p_01";

// 步骤7：创建用户模板
ADD USERPROFILE:USERPROFILENAME="up_p2p_test",UPTYPE=PCC;

// 步骤8：绑定规则到用户模板
ADD RULEBINDING:USERPROFILENAME="up_p2p_test",RULENAME="rule_p2p_fup";
```

**验证说明**：
- PCRF/PCF侧配置P2P业务的Usage-Monitoring-Information，Granted-Service-Unit设置为50MB（实际由PCRF/PCF下发，阈值在PCRF/PCF配置）
- 当用户P2P流量累计达到50MB时，UPF通过PFCP Session Report上报，SMF转发至PCRF/PCF
- PCRF/PCF决策后下发新的QoS规则（MBR: UL=1Mbit/s, DL=2Mbit/s）
- Gx路径：检查CCA消息中QoS-Information AVP的MBR
- N7路径：检查Npcf_SMPolicyControl_Update Response中QoSData的MBR

### 5.2 案例2：视频业务累计流量超阈值限速（多业务独立累计）

**场景描述**：运营商对视频业务和IM业务分别实施FUP。视频业务月累计100GB后降速，IM业务月累计10GB后降速。两种业务独立累计，互不影响。

**配置命令序列**：

```mml
// 步骤1：打开License开关
SET LICENSESWITCH:LICITEM="LKV2FUPSAT01",SWITCH=ENABLE;

// 步骤2：设置Monitoring-Key解析方式
SET PCCFUNC:MKPARSEFORMAT=UNSIGNED32;

// 步骤3：配置视频业务URR（MONITORINGKEY="3001"标识视频业务）
ADD URR:URRNAME="urr_video_01",URRID=3000,USAGERPTMODE=MONITORINGKEY,MONITORINGKEY="3001";

// 步骤4：配置IM业务URR（MONITORINGKEY="3002"标识IM业务）
ADD URR:URRNAME="urr_im_01",URRID=3001,USAGERPTMODE=MONITORINGKEY,MONITORINGKEY="3002";

// 步骤5：配置视频业务URR组
ADD URRGROUP:URRGROUPNAME="urrgrp_video_01",UPURRNAME1="urr_video_01",DOWNURRNAME1="urr_video_01";

// 步骤6：配置IM业务URR组
ADD URRGROUP:URRGROUPNAME="urrgrp_im_01",UPURRNAME1="urr_im_01",DOWNURRNAME1="urr_im_01";

// 步骤7：配置视频业务PCC策略组
ADD PCCPOLICYGRP:PCCPOLICYGRPNM="pccpolicygrp_video_01",URRGROUPNAME="urrgrp_video_01",FUPSESSIONEXC=ENABLE;

// 步骤8：配置IM业务PCC策略组
ADD PCCPOLICYGRP:PCCPOLICYGRPNM="pccpolicygrp_im_01",URRGROUPNAME="urrgrp_im_01",FUPSESSIONEXC=ENABLE;

// 步骤9：配置视频业务规则
ADD RULE:RULENAME="rule_video_fup",POLICYTYPE=PCC,PRIORITY=15,POLICYNAME="pccpolicygrp_video_01";

// 步骤10：配置IM业务规则
ADD RULE:RULENAME="rule_im_fup",POLICYTYPE=PCC,PRIORITY=25,POLICYNAME="pccpolicygrp_im_01";

// 步骤11：创建用户模板
ADD USERPROFILE:USERPROFILENAME="up_multi_svc_test",UPTYPE=PCC;

// 步骤12：绑定视频业务规则到用户模板
ADD RULEBINDING:USERPROFILENAME="up_multi_svc_test",RULENAME="rule_video_fup";

// 步骤13：绑定IM业务规则到用户模板
ADD RULEBINDING:USERPROFILENAME="up_multi_svc_test",RULENAME="rule_im_fup";
```

**关键说明**：
- 每种业务配置独立的URR（不同URRID和MONITORINGKEY），确保流量独立累计
- 每种业务配置独立的PCC策略组，引用各自的URR组
- 通过ADD RULEBINDING将多个规则绑定到同一个用户模板，实现多业务同时FUP
- PRIORITY值不同（视频15 < IM 25），优先级值越小优先级越高
- PCRF/PCF侧为视频业务设置100GB阈值，为IM业务设置10GB阈值

### 5.3 案例3：N7接口L7层业务级FUP（预定义规则场景）

**场景描述**：纯5G SA场景，使用L7层DPI识别特定应用（如某视频APP），基于预定义规则实施业务级FUP。

**前提配置**（需四方一致）：
- PCF配置预定义PCC规则名（如"predef_video_app"）和umId（如"um_video_5001"）
- PGW-C/SMF配置预定义PCC规则名与PCF一致，配置预定义URR ID和umID映射
- PGW-U/UPF配置预定义规则名与PGW-C/SMF一致，配置预定义URR ID与PGW-C/SMF一致

**PGW-C/SMF侧配置命令序列**：

```mml
// 步骤1：打开License开关
SET LICENSESWITCH:LICITEM="LKV2FUPSAT01",SWITCH=ENABLE;

// 步骤2：设置Monitoring-Key解析方式
SET PCCFUNC:MKPARSEFORMAT=UNSIGNED32;

// 步骤3：配置预定义URR（URRID高位bit为1标识预定义URR）
// 假设预定义URR ID为8001（二进制高位bit为1）
ADD URR:URRNAME="urr_predef_video",URRID=8001,USAGERPTMODE=MONITORINGKEY,MONITORINGKEY="5001";

// 步骤4：配置URR组
ADD URRGROUP:URRGROUPNAME="urrgrp_predef_video",UPURRNAME1="urr_predef_video",DOWNURRNAME1="urr_predef_video";

// 步骤5：配置PCC策略组
ADD PCCPOLICYGRP:PCCPOLICYGRPNM="pccpolicygrp_predef_video",URRGROUPNAME="urrgrp_predef_video",FUPSESSIONEXC=ENABLE;

// 步骤6：配置规则（引用预定义PCC策略组）
ADD RULE:RULENAME="rule_predef_video_fup",POLICYTYPE=PCC,PRIORITY=20,POLICYNAME="pccpolicygrp_predef_video";

// 步骤7-8：创建用户模板并绑定规则
ADD USERPROFILE:USERPROFILENAME="up_predef_test",UPTYPE=PCC;
ADD RULEBINDING:USERPROFILENAME="up_predef_test",RULENAME="rule_predef_video_fup";
```

**关键说明**：
- L7层场景下，PCF下发的SmPolicyDecision中预定义规则只需定义pccRuleId和refCondData（可选）
- umDecs中定义流量阈值，但umId和pccRule无绑定关系（映射在本地配置）
- PFCP Session Establishment Request中Create PDR包含Activate Predefined Rule（预定义规则名）和URR ID
- URR ID高位bit为1区分预定义URR和动态URR

- 使用测试终端继续进行P2P业务
- 查看镜像接口的抓包信息
- 通过第三方抓包工具统计功能计算已使用流量是否达到流量阈值的100%

**步骤6：业务级UM上报验证**
- 流量达到阈值100%后，查看UNC发起的上报消息：
  - **Gx路径**：查看CCR-U消息中是否携带`Usage-Monitoring-Information` AVP
  - **N7路径**：查看Npcf_SMPolicyControl_Update Request（POST）消息中是否携带`accuUsageReports`
- 验证上报的流量与下发的阈值的100%是否接近相等

**步骤7：QoS策略执行验证**
- **Gx路径**：查看CCR-U消息中的`QoS-Information` AVP
- **N7路径**：查看Npcf_SMPolicyControl_Update Response（200 OK）消息中的`QoSData`
- 检查MBR是否降为设置的QoS带宽

**步骤8：异常排查**
- 若步骤6上报消息中未携带业务级流量信息，执行：
```mml
LST APNPCCFUNC:;
```
检查指定DNN下是否开启PCC功能。如未开启，参考PCC基本功能激活重新配置。

### 6.4 常见问题排查

| 问题 | 可能原因 | 排查方法 |
|------|----------|----------|
| CCA-I/Response中无业务级UM信息 | PCRF/PCF规则配置错误 | 联系PCRF/PCF技术支持检查规则配置 |
| CCR-U/Update中无业务级流量上报 | APN未开启PCC功能 | 执行LST APNPCCFUNC检查 |
| CCR-U/Update中无业务级流量上报 | URR未正确绑定到业务流PDR | 检查ADD URR/ADD URRGROUP/ADD PCCPOLICYGRP配置链路 |
| 流量上报但不准确 | UE Category用户阈值低于1GB | 确保PCRF/PCF下发的业务级流量阈值不低于1GB |
| 带宽未降速 | PCRF/PCF QoS规则配置错误 | 检查CCA/Response中QoS-Information/QoSData的MBR值 |
| L7层预定义规则不生效 | 四方规则名/URR ID不一致 | 检查PGW-C/SMF/PCF/PGW-U/UPF预定义规则配置一致性 |
| Gx接口信令拥塞（月末配额重置） | 未开启UMCH | ADD PCRF的FEATURELIST添加"UMCH" |

### 6.5 升级排查

若以上步骤均无法解决问题：
1. 在镜像接口或服务器上开启抓包工具，执行调测流程并保存报文
2. 执行EXP MML命令将当前配置数据导出为MML脚本文件并保存
3. 收集并保存上述所有查询信息
4. 收集归纳所有信息并联系华为技术支持

### 7.8 对系统的影响

启用该功能需要GGSN-C/PGW-C/SMF与PCRF/PCF不断进行交互，同时需要动态处理PCRF/PCF下发的策略，对系统性能影响较大，规划时需考虑启用该特性的用户比例。详细性能影响需基于流量模型评估，请联系华为技术支持获取服务。
