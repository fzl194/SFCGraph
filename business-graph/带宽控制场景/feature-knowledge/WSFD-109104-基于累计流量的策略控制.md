# WSFD-109104 基于累计流量的策略控制 - 特性知识文档

> **场景归属**: 业务感知 → 带宽控制子场景（FUP累计流量触发策略）
> **产品**: UNC (SMF网元)
> **场景定位**: FUP（Fair Usage Policy）基础特性，与GWFD-020353（UDG侧）对应

---

## 1. 特性概述

### 1.1 基本信息

| 项目 | 内容 |
|------|------|
| 特性ID | WSFD-109104 |
| 特性名称 | 基于累计流量的策略控制 |
| 适用NF | GGSN、PGW-C、SMF（均为UNC产品） |
| 网元角色 | UNC（GGSN-C/PGW-C/SMF）控制面 + UDG（GGSN-U/PGW-U/UPF）用户面 |
| 首次发布版本 | UNC 20.3.0 |
| License控制项 | 82207980 LKV3W9PCBT12 基于累计流量的策略控制 |

### 1.2 核心能力一句话描述

针对用户在一段时间内的会话级累计流量进行策略控制，当会话级流量达到预先设置的阈值时，改变计费和控制策略（改变费率、阻塞用户业务或降低业务速率）。

### 1.3 在带宽控制场景中的定位

本特性是 FUP（Fair Usage Policy）的核心实现机制，是带宽控制场景中"累计流量触发的带宽变更"这一关键能力的承载者。在带宽控制子场景中，FUP的典型动作是：用户累计流量达到阈值后，PCRF/PCF下发新的QoS规则（如降低AMBR/MBR），由UPF执行带宽限制。

本特性支持两种接口路径：
- **Gx接口**：4G/5G融合场景，PCRF通过Gx接口下发Usage Monitoring阈值，PCEF(UNC)上报Usage。
- **N7接口**：纯5G场景，PCF通过N7接口下发Session级流量阈值，SMF(UNC)转换为N4 URR规则下发UPF。

### 1.4 与其他特性的关系

| 关系类型 | 特性 | 说明 |
|----------|------|------|
| 依赖 | WSFD-109101 PCC基本功能 | PCRF/PCF下发PCC策略给GGSN-C/PGW-C/SMF，开通本功能必须同时开启PCC基本功能（License: 82207979） |
| 对应（UDG侧） | GWFD-020353 | UDG侧对应的累计流量策略控制特性，PGW-U/UPF负责执行策略动作 |
| 配合 | WSFD-109101 PCC基本功能（5G） | N7接口路径需先完成PCC基本功能（5G）激活 |
| 配合 | WSFD-109101 PCC基本功能（2G/3G/4G，基于Gx接口） | Gx接口路径需先完成PCC基本功能（2G/3G/4G）激活 |

### 1.5 客户价值

| 受益方 | 受益描述 |
|--------|----------|
| 运营商 | 通过用户的会话级累计流量进行带宽和计费策略控制，为运营商提供灵活的流量控制解决方案，优化网络资源分配。 |
| 用户 | 对恶意抢占网络资源的用户使用该特性，可以保证用户公平使用策略FUP（Fair Usage Policy），提升用户网络体验。 |

### 1.6 典型应用场景

1. **总量限制场景**：限制某类用户在固定周期内总的使用量，比如普通用户在一周内使用总量最多为50MB。
2. **FUP公平使用场景**：通过限制用户在某周期内的使用量，在用户访问量超出限额时降低其访问速率或提高费率，达到网络公平使用的原则。对于包月用户，随着用户使用量的增加，逐步降低用户带宽，直至阻塞业务。

## 2. 核心概念

### 2.1 通用概念

| 概念 | 定义 |
|------|------|
| **累计流量策略控制** | 针对用户在一段时间内的会话级累计流量进行策略控制，当流量达到阈值时触发策略变更 |
| **FUP（Fair Usage Policy）** | 公平使用策略，通过限制用户在周期内的使用量，超限后降低速率或提高费率，保证不同用户得到同样的网络体验 |
| **会话级累计流量** | 以PDU会话（或IP-CAN会话）为粒度累计的流量，区别于业务级累计流量 |
| **业务级累计流量** | 以PCC规则为粒度的流量累计，Usage-Monitoring-Level取值为PCC_RULE_LEVEL(1) |
| **流量阈值** | PCRF/PCF预设的累计流量门限，达到该门限后触发策略动作 |

### 2.2 Gx接口特有概念（4G/5G融合）

| 概念 | 定义 |
|------|------|
| **Usage Monitoring机制** | 3GPP标准的流量监控机制，PCRF通过Gx接口下发流量阈值，PCEF上报累计流量 |
| **Usage-Monitoring-Information (UMI)** | Gx接口AVP，用于流量阈值下发和流量累计上报的核心信元 |
| **Monitoring-Key** | UMI的子AVP，会话级累计流量时，累计流量实现与规则中是否配置Monitoring-Key及其具体值无关；UNC上报时是否携带及具体值与PCRF下发保持一致 |
| **Granted-Service-Unit** | UMI的子AVP，用于流量阈值下发 |
| **Used-Service-Unit** | UMI的子AVP，用于流量累计上报 |
| **Usage-Monitoring-Level** | UMI的子AVP，业务级累计流量时取值为PCC_RULE_LEVEL(1) |
| **Usage-Monitoring-Report** | UMI的子AVP，指示PCEF应该上报的累积流量 |
| **Usage-Monitoring-Support** | UMI的子AVP，指示Monitoring Key值的配额监控去使能 |
| **UMCH（Usage Monitoring Congestion Handling）** | 流量监控拥塞处理机制，PCRF通过Monitoring-Time AVP提前指定配额重置时间点，PCEF分别统计重置前后的流量，通过重置后最近一次流量上报流程一起上报，避免Gx接口信令拥塞 |
| **Event-Trigger: Usage-Report** | Gx接口事件触发器，指示Usage上报事件 |

### 2.3 N7接口特有概念（纯5G）

| 概念 | 定义 |
|------|------|
| **UMC（Usage Monitoring Control）** | N7接口的Usage监控控制特性，通过suppFeat字段协商 |
| **umDecs（Usage Monitoring Decisions）** | SmPolicyDecision中的信元组，定义使用监控策略数据 |
| **umId** | 标识使用监控策略数据在一个PDU会话内 |
| **refUmData** | SessionRule中对使用监控数据决策的引用（通过umID引用） |
| **volumeThreshold** | 总流量阈值 |
| **volumeThresholdUplink** | 上行流量门限 |
| **volumeThresholdDownlink** | 下行流量门限 |
| **accuUsageReports** | N7接口累计流量上报信元组 |
| **refUmId** | accuUsageReports中的UMID引用 |
| **volUsage** | 累计流量信息，从N4 Usage Report中获得 |
| **authSessAmbr** | 授权的Session AMBR，PCF通过此信元更新QoS规则执行带宽控制 |
| **policyCtrlReqTrigger** | 策略控制请求触发条件，达到流量阈值后上报，如US_RE |

### 2.4 N4接口相关概念

| 概念 | 定义 |
|------|------|
| **URR（Usage Reporting Rule）** | N4接口的流量上报规则，SMF根据N7决策生成，关联到PDR |
| **URR ID** | 唯一标识一个URR |
| **Measurement Method** | 计费方式标识，包括流量计费(VOLUM)、时长计费、事件计费 |
| **Reporting Trigger** | URR的上报触发条件，如Periodic Reporting、Volume Quota、Time Quota、Linked Usage Reporting、VOLTH（流量阈值耗尽）等 |
| **Volume Threshold** | N4接口的总流量阈值 |
| **Volume Measurement** | 流量使用情况，包含总流量、上行流量和下行流量 |
| **Usage Report Trigger** | URR的Trigger标识，VOLTH表示流量阈值耗尽，TERMR表示终止PFCP Session上报流量 |

## 3. 实现原理与流程

### 3.1 总体架构

基于累计流量的策略控制采用"控制面决策 + 用户面执行"的分离架构：

```
PCRF/PCF（策略决策）
    |
    | Gx接口（4G/5G融合）或 N7接口（纯5G）
    v
UNC（GGSN-C/PGW-C/SMF，控制面）
    - 接收PCRF/PCF下发的流量阈值
    - 将策略转换为N4规则下发给UPF
    - 累计用户流量，达到条件时向PCRF/PCF上报
    |
    | N4接口（PFCP）
    v
UDG（GGSN-U/PGW-U/UPF，用户面）
    - 执行计费动作和QoS动作
    - 检测用户数据流量
    - 达到阈值时向SMF上报Usage Report
```

核心交互链路：PCRF/PCF下发流量阈值 → UNC接收并转换为N4 URR规则 → UPF执行流量检测 → 达到阈值后UPF上报Usage Report → UNC向PCRF/PCF上报累计流量 → PCRF/PCF策略决策（下发新阈值或QoS变更） → UNC执行带宽变更。

### 3.2 Gx接口实现原理

#### 3.2.1 Usage Monitoring机制

Gx接口采用3GPP标准的Usage Monitoring机制。关键AVP及其用途如下表：

| AVP | 用途 | Event-Trigger |
|-----|------|---------------|
| Usage-Monitoring-Information | 流量阈值下发和流量累计上报的核心AVP | Usage-Report |
| Monitoring-Key | 会话级累计流量时，累计流量实现与规则中是否配置Monitoring-Key及具体值无关。UNC上报时是否携带及具体值与PCRF下发保持一致 | Usage-Report |
| Granted-Service-Unit | 流量阈值下发 | Usage-Report |
| Used-Service-Unit | 流量累计上报 | Usage-Report |
| Usage-Monitoring-Level | 业务级累计流量时取值为PCC_RULE_LEVEL(1) | Usage-Report |
| Usage-Monitoring-Report | 指示UNC应该上报的累积流量 | Usage-Report |
| Usage-Monitoring-Support | 指示Monitoring Key值的配额监控去使能 | Usage-Report |

#### 3.2.2 UMCH流量监控拥塞处理机制

对于包月用户，需要在某个时间点（如月末）进行配额重置。如果PCRF同时要求所有在线用户上报流量，会造成Gx接口信令拥塞；如果分批要求上报，则流量上报有延迟且不准确。

UMCH机制解决方案：
- PCRF通过Monitoring-Time AVP提前指定配额重置的时间点。
- PCEF（UNC）分别统计配额重置时间点前后的流量。
- 通过重置后最近一次的流量上报流程一起上报给PCRF。
- 从而避免Gx接口的Usage Monitoring信令拥塞。

**UMCH开启条件**：ADD PCRF命令中的"Feature列表（FEATURELIST）"参数需包含"UMCH"。

**UMCH限制**：仅支持总流量阈值（不支持分上下行独立阈值）。

#### 3.2.3 Gx接口累计流量策略控制流程

```
1. 用户上线，建立IP-CAN会话
2. PCRF通过Gx接口CCA-I下发：
   - Usage-Monitoring-Information（含Granted-Service-Unit流量阈值）
   - 计费规则和QoS规则
3. UNC（GGSN-C/PGW-C）将策略传递给GGSN-U/PGW-U
4. 用户访问业务，UPF检测用户数据
5. UPF执行计费和QoS动作，达到上报条件时向UNC上报流量
6. UNC累计用户流量
7. 达到上报触发条件时，UNC通过CCR-U向PCRF上报：
   - Usage-Monitoring-Information（含Used-Service-Unit累计流量）
   - Event-Trigger: Usage-Report
8. PCRF累计流量，进行策略决策：
   - 如下发新阈值：通过CCA-U下发新的Granted-Service-Unit
   - 如执行带宽控制：通过CCA-U下发新的QoS-Information和Default-EPS-Bearer-QoS
9. UNC执行PCRF下发的计费动作和QoS动作
```

### 3.3 N7接口实现原理

#### 3.3.1 会话建立与阈值下发流程

N7接口的会话级累计流量策略控制流程（完整端到端步骤）：

```
步骤1: UE发起PDU Session Establishment流程
步骤2: AMF进行PGW-C/SMF选择，向PGW-C/SMF发送Nsmf_PDUSession_CreateSMContext消息
步骤3: PGW-C/SMF发送Npcf_SMPolicyControl_Create Request (SmPolicyContextData)请求会话策略
       - suppFeat字段指示是否支持Usage Monitoring Control功能
步骤4: PCF从UDR获取策略签约数据
步骤5: PCF进行特性协商，生成会话管理策略（含FUP相关策略）
       PCF回复Npcf_SMPolicyControl_Create Response，携带SmPolicyDecision信元组
```

**SmPolicyDecision关键信元结构**：

| 主信元 | 子信元 | 信元解释 |
|--------|--------|----------|
| sessRules | - | SessionRule，会话级规则 |
| sessRules | refUmData | 对使用监控数据决策的引用。会话级累计流量在SessionRule中定义，通过umID引用 |
| umDecs | - | DecisionData，使用监控决策数据 |
| umDecs | umId | 标识使用监控策略数据在一个PDU会话 |
| umDecs | volumeThreshold | 总流量阈值 |
| umDecs | volumeThresholdUplink | 上行流量门限 |
| umDecs | volumeThresholdDownlink | 下行流量门限 |
| policyCtrlReqTrigger | - | 达到流量阈值后上报的触发条件，如US_RE |
| suppFeat | - | 特性协商结果，如UMC |

```
步骤6: PGW-C/SMF根据sessRule和umDecs信元生成N4接口URR规则
       - 由于sessRule对整个会话有效，URR需关联到所有Create PDRs
       - PGW-C/SMF建立URR ID和UMID之间的映射
步骤7: PGW-C/SMF向PGW-U/UPF发送PFCP Session Establishment Request
       - 携带Create URR信元组，指示流量阈值信息
步骤8: PGW-C/SMF收到PFCP Session Establishment Response，PFCP会话建立成功
```

**Create URR关键子信元**：

| 子信元 | 信元解释 |
|--------|----------|
| URR ID | 唯一标识一个Create URR |
| Measurement Method | 计费方式：流量计费、时长计费或事件计费 |
| Reporting Trigger | 上报触发条件：Periodic Reporting、Volume Quota、Time Quota、Linked Usage Reporting等 |
| Volume Threshold | 总流量阈值 |

#### 3.3.2 FUP触发与带宽控制流程（阈值耗尽-首次）

```
步骤9:  用户访问数据业务，PGW-U/UPF检测到流量阈值耗尽
        PGW-U/UPF发送PFCP Session Report Request通知PGW-C/SMF
        - 携带Usage Report信元组
```

**Usage Report关键子信元（阈值耗尽时）**：

| 子信元 | 信元解释 |
|--------|----------|
| URR ID | 标识上报的URR |
| Usage Report Trigger | URR的Trigger。VOLTH表示流量阈值耗尽 |
| Volume Measurement | 流量使用情况，包含总流量、上行流量和下行流量 |

```
步骤10: PGW-C/SMF根据URR ID和UMID映射关系，提取累计流量信息
步骤11: PGW-C/SMF发送Npcf_SMPolicyControl_Update Request (SmPolicyUpdateContextData)通知PCF流量阈值耗尽
        - 携带accuUsageReports信元组
```

**accuUsageReports关键子信元**：

| 子信元 | 信元解释 |
|--------|----------|
| refUmId | UMID |
| volUsage | 累计流量信息，从N4 Usage Report中获得 |

```
步骤12: PCF进行策略决策，决定下发新的阈值
        PGW-C/SMF响应Npcf_SMPolicyControl_Update Response下发新的阈值
步骤13: PGW-C/SMF检查响应消息，若有新阈值则更新URR规则
步骤14: PGW-C/SMF发送PFCP Session Modification Request
        - 携带Update URR信元组，下发新阈值
```

**Update URR关键子信元**：

| 子信元 | 信元解释 |
|--------|----------|
| URR ID | 标识上报的URR |
| Measurement Method | 计费方式，信元值为VOLUM |
| Reporting Trigger | 上报触发条件，信元值为VOLTH |
| Volume Threshold | 新的流量阈值 |

#### 3.3.3 FUP触发与带宽控制流程（阈值再次耗尽-降速）

```
步骤15: 用户继续访问业务，PGW-U/UPF检测到流量阈值再次耗尽
        PGW-U/UPF发送PFCP Session Report Request通知PGW-C/SMF
步骤16: PGW-C/SMF再次发送Npcf_SMPolicyControl_Update Request通知PCF流量阈值耗尽
步骤17: PCF进行策略决策，不再下发流量阈值，同时更新QoS规则对用户进行带宽控制
        - 示例：上行带宽降为1 Mbit/s，下行带宽降为2 Mbit/s
        - Npcf_SMPolicyControl_Update Response中携带authSessAmbr信元组负责更新QoS规则
步骤18: PGW-C/SMF检查响应消息，若无新流量阈值下发，通知PGW-U/UPF不再上报流量
步骤19: PGW-C/SMF发送PFCP Session Modification Request，通知PGW-U/UPF更新业务规则
```

**关键点**：带宽控制通过authSessAmbr（授权的Session AMBR）实现，PCF通过此信元更新Session AMBR来限制用户的上下行带宽。

#### 3.3.4 会话释放时累计流量上报流程

```
步骤1-7: （与正常会话建立流程相同）PDU会话建立，流量阈值已下发
步骤8:  用户发起PDU会话释放流程
步骤9:  AMF向PGW-C/SMF发送Nsmf_PDUSession_UpdateSMContext消息
步骤10: PGW-C/SMF执行PDU会话释放，向PGW-U/UPF发送PFCP Session Delete Request释放N4会话
步骤11: 由于URR规则处于激活状态，PGW-U/UPF在PFCP Session Delete Response中携带Usage Report
```

**Usage Report关键子信元（会话释放时）**：

| 子信元 | 信元解释 |
|--------|----------|
| URR ID | 标识上报的URR |
| Usage Report Trigger | TERMR表示终止PFCP Session上报流量，且删除会话策略 |
| Volume Measurement | 流量使用情况，实际使用的累计流量（未达到阈值） |

```
步骤12-13: PCF进行策略决策，PCF回复Npcf_SMPolicyControl_Delete Response给PGW-C/SMF
```

### 3.4 Gx与N7接口对比

| 对比维度 | Gx接口 | N7接口 |
|----------|--------|--------|
| 适用场景 | 4G/5G融合 | 纯5G |
| 策略下发 | PCRF通过Gx接口CCA-I/CCA-U | PCF通过N7接口Npcf_SMPolicyControl_Create/Update Response |
| 流量阈值下发 | Usage-Monitoring-Information AVP（Granted-Service-Unit） | umDecs（volumeThreshold/volumeThresholdUplink/volumeThresholdDownlink） |
| 流量上报 | CCR-U（Usage-Monitoring-Information，Used-Service-Unit） | Npcf_SMPolicyControl_Update Request（accuUsageReports，volUsage） |
| 带宽控制 | QoS-Information AVP和Default-EPS-Bearer-QoS AVP | authSessAmbr信元组 |
| UNC配置需求 | 需要本页面配置（SET PCCFUNC、MOD PCRF、MOD PCCPOLICYGRP） | 仅需PCF侧配置，本端PGW-C/SMF不需要专有配置 |
| 拥塞处理 | UMCH机制（Monitoring-Time AVP） | 无特殊拥塞处理机制 |
| N4规则生成 | UNC内部处理 | SMF根据sessRule和umDecs生成URR规则，建立URR ID与UMID映射 |

### 3.5 关键信令交互汇总

| 交互方向 | 接口 | 消息/信元 | 说明 |
|----------|------|-----------|------|
| PCRF/PCF → UNC | Gx | CCA-I（Usage-Monitoring-Information） | 会话建立时下发流量阈值 |
| PCRF/PCF → UNC | N7 | Npcf_SMPolicyControl_Create Response（umDecs） | 会话建立时下发流量阈值 |
| UNC → UPF | N4 | PFCP Session Establishment（Create URR） | 下发URR规则及流量阈值 |
| UPF → UNC | N4 | PFCP Session Report（Usage Report） | 阈值耗尽时上报累计流量 |
| UNC → PCRF/PCF | Gx | CCR-U（Usage-Monitoring-Information） | 上报累计流量 |
| UNC → PCRF/PCF | N7 | Npcf_SMPolicyControl_Update Request（accuUsageReports） | 上报累计流量 |
| PCRF/PCF → UNC | Gx | CCA-U（QoS-Information, Default-EPS-Bearer-QoS） | 下发新QoS规则执行带宽控制 |
| PCRF/PCF → UNC | N7 | Npcf_SMPolicyControl_Update Response（authSessAmbr） | 下发新Session AMBR执行带宽控制 |
| UNC → UPF | N4 | PFCP Session Modification（Update URR） | 更新URR规则或下发新阈值 |

| 参数名 | 参数标识 | 取值样例 | 说明 |

### 4.7 MOD PCCPOLICYGRP / ADD PCCPOLICYGRP - 修改/增加PCC策略组

**用途**：配置将某类业务/应用的流量不统计到会话级流量。

**关键参数**：

| 参数名 | 参数标识 | 取值样例 | 说明 |
|--------|----------|----------|------|
| PCC策略组名称 | PCCPOLICYGRPNM | pccpolicygrp_01 | 使用ADD PCCPOLICYGRP定义的PCC策略组名称 |
| Session级FUP累计标识 | FUPSESSIONEXC | ENABLE | ENABLE表示该PCC策略组对应的业务不计入Session级流量；DISABLE（默认）表示计入Session级流量 |

**配置约束**：
- FUPSESSIONEXC默认值为DISABLE，表示该PCC策略组对应的业务计入Session级流量。

## 5. 配置案例

### 5.1 案例1：Gx接口FUP会话级累计流量策略控制

**业务场景**：
运营商部署基于Gx的会话级累计流量FUP策略。包月用户使用量逐步增加后降低带宽。UMCH功能用于避免月末配额重置时的信令拥塞。某类特定业务（如免流应用）需要排除在Session级流量统计之外。

**配置步骤（命令序列）**：

```mml
// 步骤1：打开本特性的License配置开关
SET LICENSESWITCH:LICITEM="LKV3W9PCBT12",SWITCH=ENABLE;

// 步骤2：设置累计流量的上报方式（Monitoring-Key解析方式与PCRF一致）
SET PCCFUNC:MKPARSEFORMAT=UNSIGNED32;

// 步骤3：使能流量监控拥塞处理（UMCH）功能
MOD PCRF: HOSTNAME="pcrf_01",SUPFEANEGOSW=ENABLE,FEATURELIST=UMCH-1;

// 步骤4：配置将某类业务/应用的流量不统计到会话级流量
ADD PCCPOLICYGRP:PCCPOLICYGRPNM="pccpolicygrp_01", FUPSESSIONEXC=ENABLE;
```

**关键参数取值说明**：

| 参数 | 取值 | 理由 |
|------|------|------|
| LICITEM | LKV3W9PCBT12 | 本特性对应的License控制项标识 |
| SWITCH | ENABLE | 打开License开关 |
| MKPARSEFORMAT | UNSIGNED32 | 与PCRF协商一致的Monitoring-Key解析格式 |
| HOSTNAME | pcrf_01 | 已通过ADD PCRF定义的PCRF主机名 |
| SUPFEANEGOSW | ENABLE | 开启Supported-Features动态协商 |
| FEATURELIST | UMCH-1 | 使能UMCH功能 |
| PCCPOLICYGRPNM | pccpolicygrp_01 | 已通过ADD PCCPOLICYGRP定义的策略组名称 |
| FUPSESSIONEXC | ENABLE | 该策略组对应的业务不计入Session级流量 |

**验证方式**：
1. 执行 `LST LICENSESWITCH` 确认License开关为ENABLE。
2. 在镜像接口抓包检查CCA-I消息中的Usage-Monitoring-Information AVP。
3. 用户使用流量后，检查CCR-U消息中的Used-Service-Unit是否正确上报。
4. 检查UMCH功能：在配额重置时间点前后分别统计流量，验证合并上报。

### 5.2 案例2：N7接口FUP会话级累计流量策略控制

**业务场景**：
纯5G网络，SMF与PCF通过N7接口交互。用户累计流量达到阈值后，PCF通过authSessAmbr将用户带宽降为上行1 Mbit/s、下行2 Mbit/s。

**配置步骤**：

```mml
// N7接口场景，本端PGW-C/SMF不需要专有配置
// 会话级累计流量仅需在PCF上进行配置

// 步骤1：打开本特性的License配置开关
SET LICENSESWITCH:LICITEM="LKV3W9PCBT12",SWITCH=ENABLE;

// 步骤2：确保已完成激活PCC基本功能（5G）
// （参考WSFD-109101 PCC基本功能（5G）激活流程）

// PCF侧配置（非本特性MML范围，仅说明逻辑）：
// - 配置umDecs：设置volumeThreshold、volumeThresholdUplink、volumeThresholdDownlink
// - 配置policyCtrlReqTrigger：设置为US_RE
// - 配置suppFeat：协商UMC特性
// - 配置authSessAmbr：用于阈值耗尽后的带宽控制
```

**验证方式**：
1. 执行 `LST LICENSESWITCH` 确认License开关为ENABLE。
2. 创建用户跟踪任务，消息类型选择N7和N4。
3. 用户上线后，检查Npcf_SMPolicyControl_Create Response(201 Created)消息中的umDecs和sessRules。
4. 用户使用流量达到阈值80%后，检查Npcf_SMPolicyControl_Update Request(POST)消息中的accuUsageReports。
5. 用户使用流量达到阈值100%后，检查Npcf_SMPolicyControl_Update Response(200 OK)消息中的QoSData和authSessAmbr。
6. 验证MBR、AMBR是否降为规划值（上行1 Mbit/s、下行2 Mbit/s）。

### 5.3 配置注意事项

1. **N7接口无需UNC专有配置**：PGW-C/SMF与PCF通过N7接口交互时，会话级累计流量仅需在PCF上配置，本端PGW-C/SMF不需要专有配置。但License仍需打开。

2. **UMCH限制**：UMCH机制仅支持总流量阈值，不支持上下行独立阈值。

3. **UE Category用户约束**：要求PCRF/PCF下发的会话级流量阈值不低于1GB，否则影响该用户流量累计的准确性。

4. **FUPSESSIONEXC语义**：ENABLE表示该PCC策略组对应的业务**不计入**Session级流量（即排除该业务的流量统计），DISABLE（默认）表示**计入**Session级流量。

5. **MKPARSEFORMAT一致性**：SET PCCFUNC的MKPARSEFORMAT必须与PCRF保持一致，否则会导致Monitoring-Key解析不匹配。

## 6. 验证与调测

### 6.1 调测前提条件

| 前提条件 | 说明 |
|----------|------|
| 阅读特性文档 | WSFD-109104 基于累计流量的策略控制 |
| 阅读PCC基本功能 | WSFD-109101 PCC基本功能 |
| 完成激活 | 激活基于累计流量的策略控制 |
| 完成PCC基本功能激活 | Gx路径或N7路径的PCC基本功能激活 |
| 安装抓包工具 | N4/Gx/N7镜像接口已安装第三方抓包工具 |

### 6.2 查询命令

| 命令 | 用途 | 关键检查项 |
|------|------|------------|
| LST LICENSESWITCH | 查询License配置开关状态 | SWITCH是否为ENABLE |
| LST APNPCCFUNC | 查询指定DNN下是否开启PCC功能 | PCC功能是否开启 |
| LST APN | 查询APN配置 | 获取用户激活使用的APN |

### 6.3 调测步骤（FUP触发全流程验证）

#### 步骤1：验证License开关

```mml
LST LICENSESWITCH;
```
- 如果SWITCH为ENABLE，继续下一步。
- 如果SWITCH为DISABLE，执行 `SET LICENSESWITCH:LICITEM="LKV3W9PCBT12",SWITCH=ENABLE;`。

#### 步骤2：创建用户跟踪任务

- 在MML命令行-UNC窗口创建用户跟踪任务。
- 消息类型选择N4、Gx和N7。

#### 步骤3：用户上线并访问业务

- 测试终端使用规划APN（如"apn-test"）接入网络。
- 用户访问业务，进行在线视频或FTP电影下载。

#### 步骤4：验证阈值下发

- 查看用户跟踪中的CCA-I（Gx）或Npcf_SMPolicyControl_Create Response(201 Created)（N7）消息。
- 检查PCRF/PCF是否下发该用户的正确的Session级流量监控信息。
  - Gx：检查Usage-Monitoring-Information AVP。
  - N7：检查UsageMonitoringData信元。
- 如果下发的阈值与规划值一致，表示规则下发正确。
- 如果不一致，联系PCRF/PCF技术支持检查PCRF/PCF上的规则配置。

#### 步骤5：验证80%阈值上报

- 继续进行业务，通过抓包工具统计已使用流量。
- 当已使用流量达到流量阈值的80%时（具体百分比以局点配置为准）：
  - Gx：检查UNC发起的CCR-U消息中是否携带Usage-Monitoring-Information AVP。
  - N7：检查UNC发起的Npcf_SMPolicyControl_Update Request(POST)消息中是否携带accuUsageReports。
  - 上报的流量应与下发的阈值的80%接近相等。
- 如果未上报：
  - 执行 `LST APNPCCFUNC` 检查指定DNN下是否开启PCC功能。
  - 如果未开启，重新配置PCC基本功能。
  - 如果已开启，联系华为技术支持工程师。

#### 步骤6：验证短信提醒（可选）

- 查看测试终端是否收到短信提醒（提醒已经使用了多少G流量）。
- 如果未收到，联系PCRF/PCF技术支持检查PCRF/PCF上的规则配置。

#### 步骤7：验证100%阈值带宽降速

- 继续使用业务，通过抓包工具统计已使用流量。
- 当已使用流量达到流量阈值的100%时：
  - Gx：检查CCR-U消息中的QoS-Information AVP和Default-EPS-Bearer-QoS AVP。
  - N7：检查Npcf_SMPolicyControl_Update Response(200 OK)消息中的QoSData。
  - 验证MBR、AMBR是否降为规划的QoS带宽。
- 如果带宽与规划值不一致，联系PCRF/PCF技术支持检查规则配置。
- 如果PCRF/PCF未下发QoS相关AVP或QoSData，联系华为技术支持工程师。

### 6.4 故障收集命令

当问题无法自行定位时，收集以下信息：

```mml
// 导出当前配置数据为MML脚本文件
EXP MML;
```

其他需要收集的信息：
1. 在镜像接口或服务器上开启抓包工具，执行调测流程并保存报文。
2. 执行EXP MML导出配置数据。
3. 收集并保存所有查询信息。
4. 联系华为技术支持工程师。

### 6.5 常见问题与排查

| 问题现象 | 可能原因 | 排查方法 |
|----------|----------|----------|
| CCA-I/N7 Create Response未下发流量阈值 | PCRF/PCF上未配置FUP策略 | 检查PCRF/PCF上的规则配置 |
| 流量达到阈值后未上报 | DNN下未开启PCC功能 | 执行LST APNPCCFUNC检查 |
| 流量达到80%阈值后未上报 | 抓包统计不准确或阈值百分比配置不同 | 重新统计流量，确认局点配置的实际百分比 |
| 带宽未降速 | PCRF/PCF未下发QoS规则 | 检查QoS-Information AVP或QoSData/authSessAmbr |
| UMCH功能不生效 | ADD PCRF的FEATURELIST未包含UMCH | 检查ADD PCRF/MOD PCRF配置 |
| UE Category用户流量累计不准 | 会话级流量阈值低于1GB | 要求PCRF/PCF下发不低于1GB的阈值 |

### 7.4 对系统的影响

启用该功能需要GGSN-C/PGW-C/SMF与PCRF/PCF不断进行交互，同时需要动态处理PCRF/PCF下发的策略，对系统性能影响比较大，规划时需考虑启用该特性的用户比例。详细性能影响需要基于流量模型进行评估。

### 7.5 应用限制

1. UMCH（Usage Monitoring Congestion Handling）仅支持总流量阈值，不支持上下行独立阈值。
2. 对于UE Category用户，要求PCRF/PCF下发的会话级流量阈值不低于1GB，否则影响该用户流量累计的准确性。
