# GWFD-020305 终端异常下行流量检测 - 特性知识文档

> **场景归属**: 业务感知 → 带宽控制子场景（辅助特性：异常流量检测触发带宽保护）
> **产品**: UDG (UPF网元)
> **场景定位**: 检测异常下行流量 → 触发限速/阻塞/上报 → 带宽保护
> **辅助特性说明**: 异常检测作为带宽控制的触发条件之一

---

## 1. 特性概述

- **特性ID**: GWFD-020305
- **特性名称**: 终端异常下行流量检测
- **所属产品**: UDG 产品文档 CH 20.15.2
- **适用网元（NF）**: PGW-U、UPF
- **首次发布版本**: UDG 20.2.0（后续版本持续支持）

### 核心能力一句话描述

UDG 对用户下行发起的业务流量进行检测，当被检测的业务流量满足异常判定条件时认定为异常业务流量，对检测过程中已放行的下行流量免费处理（不计费），对检测完成后的报文进行阻塞处理，从而抑制下行流量攻击并避免终端用户经济损失。

### 在带宽控制场景中的定位（辅助：异常检测触发带宽保护）

本特性在带宽控制子场景中作为**辅助触发特性**存在，核心价值体现为：

- **异常检测作为带宽保护的第一道触发条件**：当终端出现异常下行流量（DDoS攻击、恶意下载、异常大流量）时，本特性识别异常后直接阻塞后续报文，等价于对该终端的异常流执行硬性带宽限制（即阻塞级别限速）。
- **与BWM（带宽管理）的协作关系**：本特性的阻塞动作可视为最严格的带宽控制手段（直接丢弃），用于处理已判定为异常的流量。对正常流量的限速仍由 BWM（GWFD-110311/WSFD-211005 SA带宽管理）和 Shaping（GWFD-020354/110313 流整形）特性承担。
- **与FUP（基于使用量的策略控制）的边界**：FUP 基于用户配额用量触发限速，本特性基于流量行为异常触发阻塞，两者覆盖不同的异常/超额场景。
- **与SA（业务感知）的依赖关系**：本特性依赖内容计费基本功能（GWFD-020301），异常检测基于业务流（Service Flow）粒度进行，必须先开启内容计费基本功能。

### 异常流量类型覆盖

本特性检测的"终端异常下行流量"主要包含以下场景：

1. **下行流量攻击**：网络侧向终端发送大量异常下行报文，导致终端产生大流量话单。
2. **终端离线后的异常下行流量**：终端用户关闭数据开关，但在 UDG 上会话仍处于激活状态，此时网络侧下发的报文仍被转发并计费，造成用户经济损失。
3. **无对应上行业务的异常下行流量**：在未收到终端上行报文的情况下，连续收到大量下行报文（典型如 DDoS 攻击的下行反射、Smurf 攻击变种）。
4. **异常 RST 报文场景**：在未收到上行其他类型报文的情况下收到上行 RST 报文，判定为异常下行流量。

### 与其他带宽控制特性的协作关系总览

| 协作特性 | 特性ID | 协作方式 |
|----------|--------|----------|
| 内容计费基本功能（依赖） | GWFD-020301 | 本特性的前置依赖，异常检测基于业务流粒度，需先开启内容计费基本功能 |
| SA带宽管理 BWM | GWFD-110311 / WSFD-211005 | 正常流量的限速由 BWM 承担；本特性处理的异常流量等价于阻塞级限速 |

| FUP 基于使用量的策略控制 | GWFD-020353 / 110312 / WSFD-109104 / 211009 | FUP 基于配额触发限速；本特性基于行为异常触发阻塞，互补关系 |

## 2. 核心概念

### 2.1 终端异常下行流量

终端异常下行流量指 UDG 检测到的、非终端用户正常业务产生的下行业务流量。判定为异常后，UDG 对检测过程中已放行的下行流量执行**免费处理**（不进行计费），对判定异常之后的报文执行**阻塞处理**（直接丢弃）。

关键语义要点：

- **检测时已放行的流量免费处理**：由于异常判定需要一个检测窗口（连续收到若干下行报文才能确认异常），窗口内已放行的流量不能向用户收费，避免终端用户经济损失。
- **检测完成后阻塞**：一旦判定为异常，该业务流后续的所有报文将被阻塞，等价于带宽控制场景中的最严格限速动作。
- **基于业务流（Service Flow）粒度**：异常检测不是针对整个终端或整个 APN，而是针对具体的业务流，需要依赖内容计费基本功能提供的业务流识别能力。

### 2.2 异常检测算法（阈值、速率、行为）

本特性提供两种实现方案，对应两种检测算法：

#### 方案一：简单的终端异常下行流量检测

- **检测算法**：UDG 禁止下行报文触发创建流表，直接丢弃该下行报文，达到阻塞终端异常下行流量的目的。
- **判定条件**：任何尝试触发创建流表的下行报文。
- **适用限制**：对于 VoLTE 等特殊业务的 APN **不应**开启此方案，因为开启后可能误识别下行静默帧为异常下行流量，导致通话中断。
- **检测粒度**：报文级，无阈值参数。

#### 方案二：基于流特征的终端异常下行流量检测

- **检测算法**：通过对下行发起的数据报文进行检测，满足以下两个条件之一即认定为异常下行流量。
- **判定条件1（连续下行报文计数阈值）**：在未收到上行报文时，连续收到的下行报文个数超过阈值，则认定为异常下行流量。阈值通过 `SET ABNTRADTTHR` 命令设置。
- **判定条件2（异常RST报文）**：在未收到上行其他类型报文时，收到上行 RST 报文，则认定为异常下行流量。此条件无需配置阈值，由系统自动判定。

两个条件的关系为**逻辑或**（OR），满足任一即触发异常判定。

### 2.3 检测触发动作

检测到异常下行流量后，UDG 执行的动作：

| 动作类型 | 执行时机 | 说明 |
|----------|----------|------|
| 免费处理（不计费） | 检测过程中已放行的下行流量 | 避免终端用户经济损失 |
| 阻塞（丢弃） | 判定为异常后的该业务流后续报文 | 等价于带宽控制场景中的最严格限速 |

注意：源文档未明确提及"上报 PCRF"动作，本特性的处理动作集中在"免费处理 + 阻塞"层面。与 PCRF 的联动需通过 PCC 架构的其他特性（如 PCCPOLICYGRP 规则触发）实现，不属于本特性直接范围。

### 2.4 黑名单/白名单机制

本特性通过 APN 级开关控制启用范围，本质是一种基于 APN 的白名单/黑名单机制：

- **APN 级开关（ADD ABNTRAFFICDT）**：按 APN 粒度开启或关闭异常下行流量检测功能。
- **基于流行为的检测开关（TRAFFICBEHAVIORSWITCH）**：在 APN 级总开关之外，单独控制是否启用基于流特征的检测方案（方案二）。
- **VoLTE 排除建议**：对于 VoLTE 等特殊业务的 APN，不应开启简单的终端异常下行流量检测（方案一），避免误识别下行静默帧导致通话中断。

### 2.5 与带宽控制的关系

| 维度 | 本特性（异常检测） | BWM/Shaping/FUP（常规带宽控制） |
|------|---------------------|--------------------------------|
| 触发条件 | 流量行为异常（无上行+连续下行、异常RST） | 配额用量、SLA等级、时间窗口 |
| 动作类型 | 阻塞（丢弃后续报文）+ 已放行流量免费 | 限速（AMBR/GPI/URR级别） |
| 粒度 | 业务流级 | 终端级/APN级/业务流级 |
| 计费影响 | 已放行流量免费，后续流量不计费（阻塞） | 流量正常计费，按限速后实际速率 |
| 应用场景 | 异常流量防护 | 正常流量管理 |

## 3. 实现原理与流程

### 3.1 异常检测机制详解

#### 方案一：简单的终端异常下行流量检测

**机制描述**：

UDG 配置禁止下行报文触发创建流表。当下行报文到达 UDG 且对应业务流的流表尚未创建时，UDG 直接丢弃该下行报文，从而达到阻塞终端异常下行流量的目的。

**判定逻辑**：

```
下行报文到达 → 查找流表 → 流表不存在？
  → 是：丢弃报文（阻塞）
  → 否：正常转发
```

**风险**：VoLTE 等业务的下行静默帧在流表尚未建立的窗口内可能被误判为异常下行流量，导致通话中断。

#### 方案二：基于流特征的终端异常下行流量检测

**机制描述**：

UDG 对下行发起的数据报文进行检测，在业务流粒度上统计上行/下行报文的行为特征，当满足异常条件时触发阻塞。

**判定逻辑（条件1：连续下行报文计数）**：

```
下行报文到达 → 检查是否已收到对应上行报文？
  → 已收到上行报文：正常转发
  → 未收到上行报文：统计连续下行报文计数
    → 连续下行报文数 > 阈值（SET ABNTRADTTHR配置）？
      → 是：判定为异常 → 阻塞该业务流后续报文，已放行流量免费
      → 否：继续检测
```

**判定逻辑（条件2：异常RST报文）**：

```
收到上行RST报文 → 检查是否已收到上行其他类型报文？
  → 已收到上行其他类型报文：正常处理
  → 未收到上行其他类型报文：判定为异常 → 阻塞该业务流后续报文
```

### 3.2 检测触发流程（检测→判定→动作）

完整的端到端检测触发流程：

```
[1] License 检查
    └─ SET LICENSESWITCH 启用 LKV3G5ADTD01
       └─ 未启用 → 特性不生效

[2] APN 级开关检查
    └─ ADD ABNTRAFFICDT 配置 APN + SWITCH=ENABLE
       └─ 未开启 → 该 APN 下不执行检测

[3] 业务流级检测
    ├─ 方案一（简单检测）：下行报文触发流表创建 → 丢弃
    └─ 方案二（流特征检测）：
        ├─ 条件1：连续下行报文计数 > 阈值 → 判定异常
        └─ 条件2：无上行其他报文 + 收到上行RST → 判定异常

[4] 异常处理动作
    ├─ 检测过程中已放行的下行流量 → 免费处理（不计费）
    └─ 判定异常后的后续报文 → 阻塞（丢弃）

[5] EMS-DROP-DATA 消息上报
    └─ 丢弃报文时上报 EMS-DROP-DATA 消息
       └─ cause 字段："This packet is discarded because it is an abnormal downlink packet."
```

### 3.3 与内容计费基本功能（GWFD-020301）的联动

本特性依赖 GWFD-020301 内容计费基本功能：

- **依赖原因**：终端异常下行流量检测基于业务流（Service Flow）进行，需要内容计费基本功能提供业务流识别和分类能力。
- **前置条件**：必须先开启内容计费基本功能（License 控制项 `82209822 内容计费基本功能`），否则本特性的检测逻辑无法执行。
- **计费联动**：检测过程中已放行的下行流量执行免费处理，这一动作直接作用于计费流程，需要内容计费基本功能的支持。

### 4.2 前置约束配置

#### SET SRVCOMMONPARA - 业务公共参数

**关键约束**：通过 `SET SRVCOMMONPARA` 命令的 `SIGDEFERREDTIME` 参数配置的信令报文关联业务计费的迟滞时间**不能小于 60s**。

**约束原因**：异常下行流量检测需要一定的检测窗口（连续下行报文计数），如果信令报文关联业务计费的迟滞时间过短，可能导致检测尚未完成时计费已触发，无法正确执行"已放行流量免费处理"的逻辑。

**命令路径**：`OM参考/命令/UDG MML命令/用户面服务管理/业务匹配策略/业务匹配公共配置/业务公共参数管理/业务公共参数/设置业务公共参数（SET SRVCOMMONPARA）`

### 4.3 异常检测阈值配置

#### SET ABNTRADTTHR - 设置异常流量检测报文阈值

**功能**：设置基于流特征的终端异常下行流量检测方案（方案二）中"连续下行报文计数"的判定阈值。

**适用条件**：仅在使用方案二（基于流特征的检测）时需要配置。方案一（简单检测）无需配置阈值。

**命令路径**：`OM参考/命令/UDG MML命令/用户面服务管理/业务防欺诈/异常下行流量检测阈值/设置异常流量检测报文阈值（SET ABNTRADTTHR）`

**关键参数**：

| 参数 | 说明 | 配置建议 |
|------|------|----------|
| 报文阈值 | 在未收到上行报文时连续收到的下行报文个数超过此值则判定为异常 | 根据网络流量模型调整，值过小易误判，值过大延迟阻塞 |

## 5. 配置案例

### 案例1：基于流特征的终端异常下行流量检测（推荐方案）

**场景描述**：

运营商网络中存在下行流量攻击，导致终端用户产生大流量话单并引起投诉。需要在指定 APN（apn1.com）下开启基于流特征的终端异常下行流量检测，当终端在未发送上行报文的情况下连续收到超过阈值的下行报文时，自动阻塞该业务流后续报文，已放行流量免费处理。

**前置条件**：

- 已通过 ADD APN 配置 APN `apn1.com`
- 已开启 GWFD-020301 内容计费基本功能（License `82209822`）
- 已完成 License 加载（`LKV3G5ADTD01`）

**完整命令序列**：

```
// 步骤1：确保信令报文关联业务计费的迟滞时间不小于60s
SET SRVCOMMONPARA:SIGDEFERREDTIME=60;

// 步骤2：打开本特性的 License 配置开关
SET LICENSESWITCH:LICITEM="LKV3G5ADTD01",SWITCH=ENABLE;

// 步骤3：设置异常流量检测报文阈值（假设阈值为100个报文）
SET ABNTRADTTHR:THRESHOLD=100;

// 步骤4：配置APN下使能终端异常下行流量检测功能，启用基于流行为的检测
ADD ABNTRAFFICDT:APN="apn1.com",SWITCH=ENABLE,TRAFFICBEHAVIORSWITCH=ENABLE;
```

**效果说明**：

- apn1.com 下所有终端的下行流量将执行基于流特征的异常检测。
- 当终端在未发送上行报文的情况下连续收到超过100个下行报文时，判定为异常。
- 检测过程中已放行的下行流量免费处理（不计费）。
- 判定异常后，该业务流后续报文被阻塞（丢弃）。
- 丢弃报文时上报 EMS-DROP-DATA 消息，cause 字段为 "This packet is discarded because it is an abnormal downlink packet."

### 案例2：简单的终端异常下行流量检测

**场景描述**：

某企业客户 APN（enterprise.apn.com）下终端均处于被动接收状态，存在大量异常下行报文。客户要求直接丢弃所有触发流表创建的下行报文，不使用基于流特征的检测（避免检测窗口内的流量放行）。

**注意事项**：此方案不适用于 VoLTE 等特殊业务的 APN，可能误识别下行静默帧导致通话中断。

**完整命令序列**：

```
// 步骤1：确保信令报文关联业务计费的迟滞时间不小于60s
SET SRVCOMMONPARA:SIGDEFERREDTIME=60;

// 步骤2：打开本特性的 License 配置开关
SET LICENSESWITCH:LICITEM="LKV3G5ADTD01",SWITCH=ENABLE;

// 步骤3：配置APN下使能终端异常下行流量检测功能，不启用基于流行为的检测（仅方案一）
ADD ABNTRAFFICDT:APN="enterprise.apn.com",SWITCH=ENABLE,TRAFFICBEHAVIORSWITCH=DISABLE;
```

**效果说明**：

- enterprise.apn.com 下，UDG 禁止下行报文触发创建流表。
- 所有尝试触发创建流表的下行报文直接被丢弃（阻塞）。
- 无检测窗口，无已放行流量，无需免费处理逻辑。
- 不需要配置 SET ABNTRADTTHR 阈值（方案一不使用阈值）。

### 案例3：多APN差异化配置

**场景描述**：

运营商需要在普通数据 APN 上启用基于流特征的检测（方案二），在专用物联网 APN 上启用简单检测（方案一），在 VoLTE APN 上不开启检测。

**完整命令序列**：

```
// 公共配置：License 和迟滞时间
SET LICENSESWITCH:LICITEM="LKV3G5ADTD01",SWITCH=ENABLE;
SET SRVCOMMONPARA:SIGDEFERREDTIME=60;
SET ABNTRADTTHR:THRESHOLD=100;

// 普通数据APN：方案二（基于流特征检测）
ADD ABNTRAFFICDT:APN="internet.apn.com",SWITCH=ENABLE,TRAFFICBEHAVIORSWITCH=ENABLE;

// 物联网APN：方案一（简单检测，无阈值）
ADD ABNTRAFFICDT:APN="iot.apn.com",SWITCH=ENABLE,TRAFFICBEHAVIORSWITCH=DISABLE;

// VoLTE APN：不开启检测（不执行ADD ABNTRAFFICDT命令）
// 或显式关闭：
// ADD ABNTRAFFICDT:APN="ims.apn.com",SWITCH=DISABLE,TRAFFICBEHAVIORSWITCH=DISABLE;
```

## 6. 验证与调测

### 6.1 调测前置条件

- 已仔细阅读 GWFD-020305 终端异常下行流量检测特性概述。
- 已完成激活终端异常下行流量检测（License 开启 + APN 级开关配置）。

### 6.2 调测所需数据与工具

**数据准备**：

| 类别 | 参数名称 | 取值样例 | 获取方法 |
|------|----------|----------|----------|
| 使能终端异常下行流量检测功能 | APN名称(APN) | apn1.com | 取自激活配置，可用 LST ABNTRAFFICDT 查询 |
| 使能终端异常下行流量检测功能 | 开关(SWITCH) | ENABLE | 取自激活配置，可用 LST ABNTRAFFICDT 查询 |
| 用户信息查询 | SUPI或IMSI | 460000123456789 | 测试终端自带 |

**工具准备**：

- 测试终端（支持 SUPI/IMSI 查询）
- OM Portal 跟踪工具

### 6.3 异常检测效果验证步骤

**步骤1：验证 License 开关**

```
LST LICENSESWITCH:LICITEM="LKV3G5ADTD01";
```

- 若回显中 "Switch" 为 "ENABLE"，继续步骤2。
- 若回显中 "Switch" 为 "DISABLE"，需先开启 License 后再继续调测。

**步骤2：创建 UDG 用户跟踪任务**

在 OM Portal 上创建 UDG 用户跟踪任务：

1. 在"参数配置"栏输入用户 SUPI 或 IMSI（如 `460000123456789`）。
2. 在"消息类型"复选框中选择 "EMS_DROP_DATA" 消息类型。

**步骤3：测试终端接入网络**

使用测试终端接入网络并访问业务：

- 如果测试终端成功接入网络，继续步骤4。
- 如果测试终端无法接入网络，需先调测 UDG 的接入功能，成功接入后继续。

**步骤4：关闭测试终端的数据开关**

关闭测试终端的数据开关，模拟终端离线但 UDG 会话仍激活的场景。

**步骤5：检查 EMS-DROP-DATA 消息**

打开 EMS-DROP-DATA 消息，查看该消息中 "cause" 字段是否为：

```
This packet is discarded because it is an abnormal downlink packet.
```

- 如果是：说明终端异常下行流量检测功能正常，调测任务结束。
- 如果不是：执行步骤6。

### 6.4 故障排查步骤

**步骤6：检查 APN 级检测开关**

```
LST ABNTRAFFICDT
```

检查 UDG 上的终端异常下行流量检测功能开关是否打开：

- 如果是（SWITCH=ENABLE）：执行步骤7（收集信息寻求技术支持）。
- 如果不是（SWITCH=DISABLE）：参考激活终端异常下行流量检测重新激活特性，完成后返回步骤2重新调测。

**步骤7：收集信息并寻求华为技术支持**

1. 重复步骤3的操作并保存报文。
2. 在相应的 MML 窗口下执行配置导出：

```
EXP MML
```

将当前配置数据导出为 MML 脚本文件并保存。

3. 收集并保存上述所有查询信息。
4. 收集归纳所有信息并联系华为技术支持解决。

### 6.5 常见问题排查

| 问题现象 | 可能原因 | 排查方法 |
|----------|----------|----------|
| 异常下行流量未被阻塞 | License 未启用 | LST LICENSESWITCH:LICITEM="LKV3G5ADTD01"; 检查 Switch 是否 ENABLE |
| 异常下行流量未被阻塞 | APN 级开关未开启 | LST ABNTRAFFICDT 检查目标 APN 的 SWITCH 是否 ENABLE |
| 方案二检测未生效 | TRAFFICBEHAVIORSWITCH 未开启 | LST ABNTRAFFICDT 检查 TRAFFICBEHAVIORSWITCH 是否 ENABLE |
| 方案二检测阈值不当 | SET ABNTRADTTHR 阈值过大或过小 | 根据流量模型调整阈值 |
| VoLTE 通话中断 | 在 VoLTE APN 上开启了方案一 | 对 VoLTE APN 关闭检测或仅使用方案二 |
| 已放行流量仍被计费 | SIGDEFERREDTIME 小于 60s | SET SRVCOMMONPARA 调整 SIGDEFERREDTIME >= 60s |
| 内容计费基本功能未开启 | License 82209822 未启用 | 检查 GWFD-020301 内容计费基本功能 License |
| EMS-DROP-DATA 消息中 cause 不匹配 | 检测功能异常或报文非异常下行流量 | 按6.4节步骤7收集信息联系华为技术支持 |

## 7. 参考信息

### 7.1 特性交互

|----------------|------------|------|

### 7.3 应用场景

**场景一：下行流量攻击抑制**

当客户网络存在下行流量攻击，导致终端用户产生大流量话单，引起投诉。开启终端异常下行流量检测可以一定程度上抑制下行流量攻击。

**场景二：终端离线后的异常计费防护**

当终端用户关闭数据开关，但在 UDG 上仍处于激活的状态，此时如果收到下行报文，会正常转发给手机并计费。开启终端异常下行流量检测可以减少终端用户经济损失。

### 7.4 对系统的影响

- **性能影响**：在 UDG 上开启终端异常下行流量检测特性后，由于 UDG 需要对下行报文进行处理，系统处理负荷将增加，报文转发性能和吞吐量将下降。
- **性能评估基准**：性能影响程度基于华为默认 SA 模型评估。
- **精确评估**：详细性能影响需要基于流量模型进行评估，请联系华为技术支持获取服务。

### 7.5 应用限制

- 通过 SET SRVCOMMONPARA 命令的 SIGDEFERREDTIME 参数配置的信令报文关联业务计费的迟滞时间**不能小于 60s**。
- 对于 VoLTE 等特殊业务的 APN **不应**开启简单的终端异常下行流量检测（方案一），开启后可能误识别下行静默帧为异常下行流量，导致通话中断。

### 7.6 特性规格与遵循标准

- 本特性无特殊规格。
- 本特性不涉及标准约束。
- 本特性无相关告警。
- 本特性无相关软参。
- 本特性无相关测量指标。
