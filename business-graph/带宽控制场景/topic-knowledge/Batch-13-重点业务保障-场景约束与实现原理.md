# Batch-13: 重点业务保障 — 场景约束与实现原理

## 1. 概述

本批次覆盖"5G Core 重点业务保障解决方案"业务专题中的场景概述、协议遵循、方案约束与影响、以及实现原理四大板块，共10篇核心文档。这些文档从顶层设计到底层算法，完整刻画了重点业务保障方案的"为什么做、做到什么边界、怎么判断质差、怎么决策保障、怎么识别码率"这一技术主线。

具体涵盖：

1. **场景概述**：重点业务保障方案要解决的核心问题——基于NWDAF实现签约用户的体验实时感知与按需动态保障，避免无条件GBR保障造成的资源浪费。
2. **协议遵循**：方案遵循的11个3GPP标准（TS 23.501/502/503、TS 23.288、TS 29.244/502/508/514/520/552/564），覆盖系统架构、策略计费控制、网络数据分析服务、用户面功能服务等。
3. **方案约束与影响**：分为通用约束（NWDAF容灾、漫游限制、4G/NSA网络约束、智能码率识别约束、海外局点约束）和异厂商PCF场景约束（双N7接口策略、不支持RAN-OAM数据获取）。
4. **实现原理**：小区容量评估和负载预测（UDN周期训练与推理）、质差判断（UPF基于KPI的体验分析及上报机制）、智能码率识别报表（码率库可视化）、保障建议判断条件（小区拥塞、小区容量、会话级阻塞、小区级阻塞四条件）、基于智能码率识别的保障建议（聚类算法驱动的带宽档位匹配）。

这些文档共同回答了"签约用户发生体验质差后，网络如何感知质差、如何评估小区资源、如何决策是否发起保障、如何确定保障带宽"这一完整决策链。

## 2. 核心知识点

### 2.1 重点业务保障的核心目标与设计理念

重点业务保障方案是5G网络智能化的首个业务场景，依托NWDAF（网络数据分析功能）为5G核心网增加智能分析能力。方案面向"直播"、"云游戏"等5G重点业务的签约用户，实现以下核心价值：

- **按需动态保障**：只有当UPF感知到用户发生质差，且NWDAF根据小区负载等信息判断可以为用户提供保障时，才触发GBR保障。避免签约保障套餐用户触发业务时无论体验好坏均立即进行GBR保障而浪费网络资源。
- **无效保障自动识别**：保障后用户体验无提升时识别为无效保障，及时释放GBR，最大化利用无线GBR资源。
- **网络调度公平性**：基于无线历史数据构建模型预测未来两小时以上PRB数据，根据PRB负载预测值决策是否进入过载保护，防止PRB重重度载后仍建立保障专载而引起普通用户公平性问题。
- **小区粒度精准控制**：小区容量精准获取，实现小区粒度的GBR资源控制，避免过量建立保障专载使大众用户基础业务受损。

### 2.2 协议遵循：标准接口保障互通性

方案完全基于3GPP标准接口实现，遵循的核心协议包括：

| 标准编号 | 作用 |
|----------|------|
| TS 23.501 | 5G系统架构 |
| TS 23.502 | 5G系统流程 |
| TS 23.503 | 策略与计费控制框架（PCC） |
| TS 23.288 | 5G系统支持网络数据分析服务的架构增强（NWDAF核心定义） |
| TS 29.244 | 控制面与用户面节点间接口（N4接口） |
| TS 29.502 | 会话管理服务 |
| TS 29.508 | 会话管理事件暴露服务 |
| TS 29.514 | 策略授权服务（N5接口） |
| TS 29.520 | 网络数据分析服务 |
| TS 29.552 | 网络数据分析信令流程 |
| TS 29.564 | 用户面功能服务 |

这一协议集合说明方案的核心交互链路为：UPF（用户面感知质差）-> NWDAF（数据分析与保障决策）-> PCF（策略授权与专载建立），通过N4、N5、Nnwdaf等标准接口串联。

### 2.3 通用约束与影响

通用约束覆盖以下维度：

**NWDAF容灾**：NWDAF故障后，容灾的其他NWDAF无用户保障记录。PCF检测到NWDAF故障后通过设备核查释放故障NWDAF发起的所有保障会话，用户回到缺省承载。PCF向新NWDAF重新发起订阅，若用户仍质差且满足保障条件，保障专载重建时长通常在10秒内。

**漫游限制**：不支持异网漫游和国际漫游。用户在支持/不支持保障区域间移动的四种场景均有约束处理，移动至不支持区域时暂停QOS_ANALYSIS订阅，返回后恢复。

**2G/3G网络不支持**：PCF感知用户切换到不支持保障网络后暂停QOS_ANALYSIS订阅，NWDAF保留N23订阅会话但取消SMF订阅，返回后恢复。

**4G/NSA网络约束**：SMF/UPF/PCF需为4G/5G融合网元；不支持用户跨NWDAF POOL区域移动时的保障；MME上报用户实时位置周期长导致小区资源配额管理不准确（可能造成拥塞小区更拥塞或延时保障）。

**智能码率识别约束**：仅对进行GBR保障的直播应用支持，整机最多支持10个直播应用；必须同时开启体验感知信息采集功能。

**海外局点约束**：不支持从RAN-OAM获取无线资源进行小区容量评估和PRB利用率预测，需在NWDAF上通过MML配置小区容量。

**应用规模**：支持40个应用大类，所有应用大类下的应用个数总和不超过1000个。

### 2.4 异厂商PCF场景特有约束

当大网PCF为异厂商设备时，方案采用双N7接口架构（大网PCF通过第一个N7接口与SMF通信，智能PCF通过第二个N7接口与SMF通信），存在以下特有约束：

- **仅支持N7接入**：包括Gx转N7的用户，不支持Gx接入用户。
- **保障用户识别**：通过PCC Rule划分，SMF通过PCC Rule识别保障用户。
- **单一套餐**：当前方案仅支持配置一个重点业务保障套餐，不支持基于用户粒度差异化保障。
- **双N7接口异常处理**：SMF创建第二个N7会话失败时不重试，需等UE下次重新创建会话时再尝试；不支持智能PCF重选和第二N7会话failover。
- **不支持RAN-OAM数据获取**：NWDAF不能根据预测的小区容量和PRB利用率判断是否发起保障建议，规避措施为NWDAF本地配置小区容量。

### 2.5 小区容量评估和负载预测算法

这是NWDAF准确评估保障建议的关键条件，核心目标是解决"无线话统上报不及时"问题——无线侧每15分钟统计PRB利用率，核心网侧获取时已缺乏实效性。

**整体架构**：UDN作为FTP/SFTP客户端从RAN-OAM获取无线侧性能指标，周期性训练与推理，生成分析结果通过HTTP接口提供给UDC（即NWDAF-FE）。

**训练数据处理**：
- 时间偏差校准：通过"与无线工作站的时间偏移量（分钟）"和"每周期获取时间偏移（分钟）"参数分别设置UDN与RAN-OAM之间、以及RAN-OAM生成单据的时间偏差。
- 数据完整性判断与补齐：
  - **训练数据集**：缺失比例<30%时支持补齐；>=30%时不补齐，不作为训练数据。
  - **推理数据集**：前12个周期数据缺失时支持补齐，依赖历史推理结果；无历史推理结果则无法补齐，当前周期推理失败。

**模型训练与推理**：
- 预置模型：UDN系统发布时已包含预置训练模型。
- 动态训练：当"动态模型配置"设置为"开启模型预测"时，UDN针对每个小区创建独立训练任务，每周运行一次，抽取前两周数据分别进行上行/下行模型学习。
- 容量推理：每周一次，通过历史数据计算每个小区平均容量，得出上行/下行容量信息。
- 负载推理：每15分钟一次，通过前12个周期数据得出当前周期上行/下行PRB信息。
- 存储方式：推理结果持久化存储在PGSQL数据库，供UDC调用。

**PRB利用率准确性评估**：
- 比对模型预测结果与真实数值，连续两次结果偏差>10%则触发反向退回功能。
- 四种预测退化策略（通过"预测退化策略"参数配置）：
  - **强制不退化**：即使偏差>10%仍使用模型推理数据。
  - **废弃预测结果**：不存入数据库，异常方向使用配置值。
  - **就近实报值**：使用最近一次PRB利用率（T-1或历史T平均），不存在则用配置值。
  - **使用配置值**：直接使用配置值作为上报数据。

### 2.6 质差判断机制：UPF基于KPI的体验分析

UPF负责感知保障套餐签约用户的业务质量，按配置的质量基线进行评估。支持保障的UPF需部署智能板，通过UDG的 `SET VVIPBASICFUNC` 命令开启端到端重点业务保障功能。

**质差分析三步流程**：

1. **业务流识别**：UPF通过N4接口接收质差分析订阅事件后，在用户业务流到来时通过业务感知功能识别协议，基于N4订阅消息中的用户信息及appId判断当前流是否为需质量判断的流。

2. **KPI汇聚与质差判断**：
   - 大流过滤：通过 `ADD POLICYCONDITION` 命令配置"上行大流阈值"或"下行大流阈值"，仅保留大于阈值的业务流。
   - KPI指标：支持通过上行/下行平均速率、时延等指标判断业务质量。通过 `ADD SSUPROTCOLGROUP` 与 `ADD POLICYCONDITION` 命令配置不同业务的质量基线参数及检测条件。
   - 例如直播业务通过上行平均速率大小评估质量是否达标。

3. **质差/体验数据上报**：
   - 质差事件上报：当发现上网质量不满足基线时，UPF基于Nupf接口周期性（默认5秒）向NWDAF通知质差事件。
   - 体验单据上报：UPF周期性（默认5分钟）上报用户体验单据给NWDAF。
   - 上报策略通过 `ADD APPPOLICYCTRL` / `SET VVIPBASICFUNC` 命令配置。

### 2.7 保障建议判断条件：四重门控

NWDAF支持发起GBR或Non-GBR两种保障建议（通过UDC的 `ADD APPGROUP` 命令"保障类型"参数配置）。

**GBR保障需满足全部四个条件**：

| 条件 | 说明 | 配置命令 |
|------|------|----------|
| 小区拥塞（可选） | 小区PRB利用率未达到NWDAF配置的拥塞阈值时才可发起保障 | UDC `SET QOSGUARCOND` |
| 小区容量 | 用户所在小区剩余可用GBR带宽达到门限，实现小区粒度的GBR资源控制 | UDC `SET QOSGUARCOND`（上行/下行GBR门限占比） |
| 会话级阻塞 | 会话阻塞定时器未运行，即当前无因保障失败或无效保障导致的临时阻塞 | UDC `SET QOSGUARCOND` + 软参 `DWORD1 BIT2` |
| 小区级阻塞 | 小区级阻塞定时器未运行，即当前小区无因保障请求过多或失败率过高导致的临时阻塞 | UDC `SET QOSGUARCOND` |

**Non-GBR保障需满足三个条件**：小区拥塞、会话级阻塞、小区级阻塞（不含小区容量）。

**小区容量管理细节**：
- 三种容量获取方式优先级：指定号段配置 > 从UDN获取 > 全局配置。
  - 指定号段：UDC的 `ADD NRCELLCAPACITY` / `ADD EPCCELLCAPACITY` 命令配置5G/4G小区号段的上行/下行容量。
  - 从UDN获取：UDC的 `SET UDNCTRL` 命令控制，"小区容量有效时长(分钟)"为0时不从UDN获取。
  - 全局配置：UDC的 `SET QOSGUARCOND` 命令的缺省参数。
- 动态管理：每次保障成功后记录已触发GBR和已使用带宽，撤回保障时回退记录。
- 示例：上行GBR资源占比门限60%，小区上行容量300Mbps，最多分配180Mbps给VVIP用户。每GBR 6Mbps可建30个，每GBR 3Mbps可建60个。

**会话级阻塞两种场景**：
- 场景一（保障失败）：PCF通知保障失败（资源分配失败、专载去激活等），NWDAF释放N5会话并启动阻塞定时器。
- 场景二（无效保障）：保障成功后仍持续收到质差通知。无效保障判断有两种模式：
  - 全局判断：保障成功后1分钟内质差次数超过 `SET QOSGUARCOND` 中"界定无效保障的质差次数"阈值。
  - 应用级判断：`ADD APP` 命令配置APP模式后，在"保障专载建立成功后界定无效保障的检测周期时长"内，质差次数大于等于配置阈值即为无效。

**小区级阻塞两个触发条件**（5分钟统计周期内需同时满足）：
- 条件一：小区级保障请求次数达到 `SET QOSGUARCOND` 中"小区级保障请求次数"阈值（指NWDAF按小区统计收到UPF上报的质差通知次数）。
- 条件二：小区级保障失败次数达到 `SET QOSGUARCOND` 中"小区级保障失败阈值(%)"（含GBR配额不足、小区拥塞、PCF通知保障失败、无效保障）。

### 2.8 基于智能码率识别的保障建议：AI驱动的带宽匹配

智能码率识别是基于聚类算法对保障带宽进行优化的机制，为保障用户选择合理带宽档位（类比直播视频清晰度档位：标清、高清等）。

**三种保障带宽来源及优先级**：

通过UDC的 `SET BANDMATCHPOLICY` 命令设置优先级，三种来源为：

| 带宽来源 | 说明 | 优点 | 约束 |
|----------|------|------|------|
| 智能码率识别 | 基于聚类算法训练，自动识别应用清晰度档位和带宽范围 | 应用新增清晰度或速率变更时快速识别并应用 | 整系统复位后样本数据丢失，需重新累计 |
| 静态配置（带宽档位） | UDC `ADD BANDWIDBNDAPP` 命令配置应用带宽档位 | 智能算法异常时提供可靠后备 | 需依据拨测设置合理值 |
| 静态配置（最高带宽） | UDC `ADD APP` 命令配置应用最高保障带宽 | 兜底配置，确保VVIP权益 | 需依据拨测设置合理值 |

**智能码率识别训练流程**：

1. **样本采集**：保障用户上线后，UPF以分钟级向NWDAF上报质优数据（非质差数据）。NWDAF记录用户平均瞬时速率（码率），筛选满足规则的数据作为训练样本。
2. **聚类训练**：NWDAF根据 `SET CODETRAINRULE` 命令配置的训练规则执行码率聚类训练，生成应用清晰度档位和对应的最高/最低带宽形成档位码率库。
3. **训练结果评估**：当次训练结果不符合 `SET CODETRAINRULE` 阈值范围时，判定为不可信，使用前一次满足要求的训练结果。
4. **保障带宽确定**：用户质差时，NWDAF根据平均瞬时码率在码率库中匹配档位，确定保障带宽值。
5. **持续优化**：保障后UPF持续每5秒检测平均速率（分钟级上报），NWDAF将质优数据处理为应用级样本存储，用于持续聚类训练。
6. **可视化呈现**：UDC将训练结果通过 `Nudn_DataManagement_StorageRequest` 发送给UDN，在UDN侧可视化呈现。

**样本采集的关键规则**：
- 质差数据不作为训练样本。
- 质优数据且距离上次质差单据间隔>5分钟时才作为样本。
- 首次质差时默认使用静态配置的最高带宽发起保障。

**保障建议更新逻辑**（`SET BANDMATCHPOLICY` 命令初始配置）：
- 智能算法训练为第一优先级：收到质优单据后，根据码率在码率库匹配档位确定保障带宽。
- 配置带宽档位为第一优先级：收到质优单据后，根据码率匹配 `ADD BANDWIDBNDAPP` 配置档位确定带宽。
- 默认最高档位为第一优先级：不发起保障带宽更新流程。

### 2.9 智能码率识别报表

智能码率识别报表呈现NWDAF智能码率识别的应用码率库信息，支持以下统计字段：

| 字段 | 说明 |
|------|------|
| 训练开始时间 | 训练开始的时间戳 |
| NWDAF | 当前训练结果所属的NWDAF（UDC） |
| 应用 | 训练的子应用名称 |
| 带宽档位 | 训练推理出的档位信息 |
| 训练时长（小时） | 训练持续的时长 |
| 样本数量 | 训练使用的样本数量 |
| 最大码率 | 对应档位的最大带宽值（单位自适应或固定Mbps） |
| 最小码率 | 对应档位的最小带宽值（单位自适应或固定Mbps） |

报表界面支持显示1000条查询结果，超出可通过下载离线查看，支持定时导出任务和远程通知（邮箱/SFTP服务器）。

**使用前提**：
- 完成与UDC的对接配置。
- 加载包含"智能码率识别可视"控制项的License并打开开关。
- 远程通知需配置邮箱服务器或SFTP服务器。

## 3. 配置调测要点

### 3.1 质差判断关键参数

| 参数 | 配置命令 | 说明 |
|------|----------|------|
| 端到端保障开关 | UDG `SET VVIPBASICFUNC` | 开启UPF智能板上的保障功能 |
| 上行大流阈值(kB/s) | UDG `ADD POLICYCONDITION` | 过滤业务流，仅保留大于阈值的流 |

## 4. 与带宽控制的关系

本批次文档描述的重点业务保障方案与带宽控制场景存在紧密的上下游关系，构成"感知-决策-执行"完整链路：

### 4.1 质差判断是带宽控制触发条件

质差判断是整个带宽控制（保障）流程的入口。UPF基于KPI（上下行平均速率、时延等）判断用户体验是否达标，不达标时上报质差事件给NWDAF。这一机制直接决定了带宽控制（GBR保障）是否需要启动。质差判断的KPI阈值和质量基线配置，实质上就是带宽控制的触发条件。

### 4.2 保障建议判断是带宽调整决策

NWDAF收到质差通知后，通过四重门控（小区拥塞、小区容量、会话级阻塞、小区级阻塞）决策是否发起保障建议。这四个条件本质上是带宽资源分配的准入控制：
- 小区拥塞判断小区整体带宽是否已过载。
- 小区容量配额控制VVIP用户可占用的总GBR带宽上限。
- 会话级/小区级阻塞机制防止无效或失败的带宽分配重复尝试。

### 4.3 智能码率识别是带宽调整精度保障

保障建议中的保障带宽来源（智能码率识别/静态配置档位/静态配置最高带宽）直接决定了GBR专载分配的带宽值。智能码率识别通过AI聚类算法自动学习应用的清晰度档位和带宽范围，使带宽分配从"粗粒度固定值"进化为"精细匹配应用实际需求"。

### 4.4 小区容量评估是小区级带宽资源管理

小区容量评估和负载预测算法为带宽控制提供了小区级的资源视图。通过模型预测PRB利用率（而非依赖15分钟统计的滞后数据），使NWDAF能在保障决策时实时感知小区带宽资源状态，避免在资源不足时建立保障专载。

### 4.5 与带宽管理特性的协同

重点业务保障方案中的GBR保障建议最终通过PCF下发策略，触发SMF建立保障专载，这个过程与带宽管理（BWM）特性中GBR QoS流管理直接对接。保障带宽值即为GBR QoS流的GBR值，由智能码率识别或静态配置决定。

## 5. 关键术语

| 术语 | 简释 |
|------|------|
| NWDAF | Network Data Analytics Function，网络数据分析功能，3GPP定义的5G核心网网元，提供控制面和用户面智能分析服务 |
| UDN | User Data Node，华为UDG产品中的数据分析节点，负责从RAN-OAM获取数据并执行模型训练与推理 |
| UDC | 对应NWDAF-FE（前端），负责保障决策分析和与PCF/UPF/SMF的交互 |
| PRB | Physical Resource Block，物理资源块，无线资源调度的最小单位，PRB利用率过高表示网络进入拥塞 |
| GBR | Guaranteed Bit Rate，保证比特速率，为业务流提供确定性的带宽保障 |
| Non-GBR | 非保证比特速率，不提供确定性带宽保障但可享受优先调度 |
| 质差 | 用户体验质量不达标，即KPI指标低于配置的质量基线 |
| 质优 | 用户体验质量达标，KPI指标满足或超过配置基线 |
| 无效保障 | 保障专载建立成功后用户体验仍无改善（持续质差），需及时释放以节约资源 |
| 会话级阻塞 | 针对单个用户会话的临时保障冻结机制，因保障失败或无效保障触发 |
| 小区级阻塞 | 针对整个小区的临时保障冻结机制，因小区保障请求过多或失败率过高触发 |
| 智能码率识别 | 基于聚类算法自动学习应用清晰度档位和带宽范围的AI能力 |
| 码率档位 | 智能码率识别训练出的带宽区间，类比直播清晰度（标清、高清等） |
| 大流过滤 | 仅保留大于配置阈值的业务流进行KPI分析，过滤低流量干扰 |
| RAN-OAM | Radio Access Network Operation Administration and Maintenance，无线接入网操作维护管理 |
| PCF | Policy Control Function，策略控制功能 |
| SMF | Session Management Function，会话管理功能 |
| UPF | User Plane Function，用户面功能 |
| 双N7接口 | 异厂商PCF场景下，大网PCF通过第一个N7接口与SMF通信，智能PCF通过第二个N7接口与SMF通信 |
| 反向退回 | 模型预测连续两次偏差>10%时触发的预测结果处理策略 |
