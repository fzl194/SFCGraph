## 2. 核心知识点

### 2.1 PCC策略三大分类与E2E QoS管理机制总览

#### 2.1.1 PCC策略定义

PCC（Policy and Charging Control）策略是标准 PCC 架构下 PCF/PCRF 侧生成的策略，用于为无线承载网提供以下功能：

- UE路由策略
- 接入控制策略
- **QoS控制**（带宽控制的核心）
- 计费控制

**QoS控制是PCC策略的子部件**，目的是在资源有限的情况下"按需定制"，为不同用户业务提供差异化服务质量。

#### 2.1.2 5G PCC策略三大类型

5G 相比4G 新增了 AM策略和 UE策略，形成了三大策略类型：

| 策略类型 | 关键QoS参数 | 内容描述 | 策略下发/执行 |
|----------|-------------|----------|---------------|
| **SM策略** | QoS Flow级：5QI, ARP, GFBR/MFBR, RQA, 计费参数；会话级：Session-AMBR | QoS带宽控制、计费控制、短信服务通知、重定向 | PCF下发 → SMF执行 |
| **AM策略**（5G新增） | SAR（服务区限制）、RFSP索引（无线频率选择优先级） | UE允许/禁止接入的TAI列表、无线资源管理 | PCF下发 → AMF/UE/RAN执行 |
| **UE策略**（5G新增） | ANDSP（UE选网策略）、URSP（UE路由选择策略） | 辅助UE选网、切片选择、DNN选择、路由决策 | PCF下发 → UE执行 |

**与带宽控制的关系**：SM策略是带宽控制的核心策略类型。带宽管理（BWM）规则属于SM策略范畴，通过Session-AMBR、GFBR/MFBR等参数实现会话级和业务级的带宽控制。

#### 2.1.3 SM策略QoS管理流程

SM策略中QoS的端到端管理机制，以交通系统类比理解：

| 类比项 | QoS概念 | 交通类比 |
|--------|---------|----------|
| 通道 | 默认QoS Flow（5QI 5~9，不保证bit率） | 汽车出行（兜底方案，必须创建） |
| 通道 | GBR QoS Flow（5QI 1~4, 65~67, 75，保证bit率） | 飞机出行（保证速度，按需创建） |
| 通道 | Non-GBR QoS Flow（5QI 5~9, 69~80，不保证bit率） | 轮船出行（不保证速度，按需创建） |
| 通道属性 | 5QI（QoS通路的综合评估） | 通勤方式综合评价 |
| 通道属性 | ARP（分配保留优先级） | 建路优先级（高速路有优先特权） |
| 通道属性 | MFBR/GFBR | 承诺速度/最高速度 |
| 通道属性 | Session-AMBR | 一趟车上限乘客数 |

**关键规则**：
- GBR QoS Flow：PCF下发必然携带保证带宽参数，无线侧资源支持时一定保障速率
- Non-GBR QoS Flow：速率是否保证不完全取决于5QI；PCF若配置保证带宽参数，无线侧仍可保障速率

### 2.2 业务感知SA（Service Awareness）——定义与价值

#### 2.2.1 何为业务感知（What）

**业务感知（Service Awareness，简称SA）**：在用户会话过程中，对用户的数据报文进行解析，从而区分出用户使用的不同业务，并根据业务策略对业务数据进行转发/控制的过程。

> 注意：此处的SA（Service Awareness）与5G SA（Standalone）组网是不同概念，需区分。

**核心类比**：业务感知类似快递分拣中心——查看包裹信息（数据包特征），根据配送政策（业务策略）将包裹送达不同地区（数据中心）。

**典型场景**：用户同时浏览网页、下载视频、打游戏时，核心网通过SA可以：
- 识别出不同业务
- 给下载业务分配较多带宽
- 给游戏业务配置较低时延
- 预留少量带宽给网页浏览
- 禁止访问不安全IP

#### 2.2.2 为什么使用业务感知（Why）

业务感知本质是对流量的"经营"，三大应用方向：

| 应用方向 | 具体能力 | 价值 |
|----------|----------|------|
| **计费精细化运营** | 基于时间段、位置、用户类别、业务的策略控制；APP免流量；高峰时段限速；特定位置免费体验 | 运营商收益保证与提升 |
| **带宽控制（BWM）** | 对P2P/FTP等协议带宽控制，保证网络资源均衡分配；网络资源充沛时保证签约P2P业务的服务质量 | 防止个别用户挤占带宽，提供差异化带宽服务 |
| **安全防护** | 过滤管理UPF数据面流量；避免恶意攻击和无恶意误操作 | 保障网络性能和用户业务安全 |

**SA三大套餐**（与带宽控制场景直接相关）：
- **套餐1 - 计费场景**：免流量、差异化计费、时段/位置策略
- **套餐2 - 带宽控制场景**（本场景核心）：P2P/FTP限速、用量超额降速、签约业务保障
- **套餐3 - 访问限制场景**：URL过滤、阻塞、重定向、接入控制

### 2.3 业务感知工作原理（How）——端到端流程

#### 2.3.1 业务感知全景七步流程

| 步骤 | 动作 | 执行网元 | 说明 |
|------|------|----------|------|
| Step1 | 规则下发 | PCF → SMF → UPF | 用户接入时，PCF下发规则经SMF"翻译"后到UPF规则库 |
| Step2 | 数据到达UPF | 基站 → UPF | 用户数据报文经基站到达UPF |
| Step3 | 解析与识别 | UPF | 三四层解析 → 协议识别 → 七层解析 |
| Step4 | 规则匹配 | UPF | 报文特征与规则过滤条件匹配 |
| Step5 | 策略执行 | UPF | 计费/阻塞/重定向/带宽控制/头增强 |
| Step6 | 数据转发 | UPF → 网络/服务器 | 处理后的报文继续转发 |
| Step7 | 上报 | UPF → SMF | 定期上报用量、时长、时间等信息 |

#### 2.3.2 规则规划与获取

规则规划是业务感知的前置步骤，需要确定：
- **对什么样的用户**（用户类型）
- **浏览的什么内容**（过滤条件）
- **需要启用何种类型**（策略类型和具体策略）

**示例业务套餐与规则映射**：

| 业务编号 | 过滤条件 | 策略类型 | 策略 | 优先级 |
|----------|----------|----------|------|--------|
| 1 专项业务 | www.huawei.com | PCC | 1元/GB | 中 |
| 2 免费业务 | RTSP协议+IP范围 | PCC | 免费 | 高 |
| 3 阻塞业务 | www.example.com | PCC | 阻塞 | 最高 |
| 4 兜底业务 | 任意数据流 | PCC | 2元/GB | 低 |
| 5 重定向业务 | 任意数据流 | PCC | 免费（话费耗尽时） | 最低 |
| **6 限速业务** | **任意数据流** | **BWM** | **限速1Mbps（>50G时）** | **一般** |
| 7 头增强业务 | www.vmall.com | HEADEN | 插入MSISDN、防欺诈 | 一般 |

> **带宽控制关键点**：业务6是典型的BWM规则——当用户流量超过50G时，PCF下发BWM规则，UPF执行限速至1Mbps。这展示了SA→BWM的完整链路：SA识别流量→累计用量→PCF触发BWM规则→UPF执行限速。

#### 2.3.3 规则下发机制

| 场景 | 规则下发方式 | 触发条件 |
|------|-------------|----------|
| 用户接入时 | PCF下发预定义/动态规则 | PCF根据用户信息判断 |
| 会话过程中 | PCF下发/更新/删除规则 | UPF上报信息经SMF到PCF，PCF判断后操作 |
| PCF不可用 | SMF直接下发本地规则 | 无PCF或PCF故障时 |

**策略变更发起方**：UE（业务变更）、SMF（UDM本地策略修改）、PCF（本端策略修改）、基站（QoS不满足要求）。

#### 2.3.4 解析与识别三层次

| 解析层次 | 获取信息 | 说明 |
|----------|----------|------|
| 三四层解析（浅度报文检测） | 源/目的IP、IP协议类型、源/目的端口号 | 所有业务感知的基础 |
| 协议/协议组识别 | 报文的应用层协议类型（HTTP/RTSP等） | 识别协议后才能进行七层解析 |
| 七层解析（深度报文检测） | URL、Method、响应码等七层关键字段 | 精确业务感知，获取最有价值信息 |

#### 2.3.5 规则匹配三原则

1. **优先级匹配**：所有报文根据规则优先级逐条匹配
2. **全条件匹配**：报文必须符合一条规则的所有过滤条件才算匹配成功
3. **跨类型独立匹配**：不同类型规则同时匹配，每种类型至多匹配一条

**多类型规则匹配示例**（16条4种类型规则）：

| 规则类型 | 匹配过程 | 结果 |
|----------|----------|------|
| PCC规则 | 规则3→2→1失败，规则4成功 | 匹配规则4 |
| BWM规则 | 规则5→6失败，规则7成功 | **匹配规则7** |
| HEADEN规则 | 规则9失败，规则10成功 | 匹配规则10 |
| Web Proxy规则 | 全部失败 | 无匹配 |

> **带宽控制关键点**：BWM规则与PCC规则独立匹配，一条报文可以同时匹配PCC规则和BWM规则。这意味着用户访问某个网站时，既会执行计费策略（PCC），也会执行限速策略（BWM）。

### 2.4 业务感知核心概念——过滤条件/策略/规则

#### 2.4.1 过滤条件（业务感知的"手"）

过滤条件用于将用户流量特征与规则匹配：

| 过滤条件类型 | 颗粒度 | 识别内容 |
|-------------|--------|----------|
| 3、4层过滤 | 网络层/传输层 | 协议、端口号、IP地址 |
| 7层过滤 | 应用层 | URL、User-Agent、Method等七层关键字段 |
| 协议/协议组过滤 | 应用层协议 | HTTP、RTSP等协议或协议组合 |
| 流过滤器 | 组合 | 以上三种过滤条件的集合，实现复合过滤逻辑 |

**关系**：流过滤器是其他过滤条件的集合，一条规则只能绑定一个流过滤器，但流过滤器内可包含多个过滤条件。

#### 2.4.2 策略（业务感知的"腿"）

策略代表业务感知需要实现的内容：

| 策略类型 | 含义 | 与带宽控制关系 |
|----------|------|---------------|
| **PCC** | 计费与策略控制 | 实现计费和策略控制（禁止访问/重定向） |
| **BWM** | **带宽管理** | **核心：分类结果用于带宽管理策略匹配** |
| HEADEN | 头增强 | 向HTTP/RTSP请求报文头域插入字段 |
| WEBPROXY | Web Proxy | 设置WebProxy服务器地址池 |
| SMARTREDIRECT | 智能重定向 | CaptivePortal/DNS重写 |

一条规则只能绑定一条策略。规则可按策略类型分类为PCC规则、BWM规则、HEADEN规则等。

#### 2.4.3 规则（业务感知的"身体"）

规则是完成业务感知的最小单元，绑定过滤条件+策略+优先级+其他参数。

**规则按协议定义分为三类**：

| 规则类型 | 规划位置 | 启用方式 | 特点 |
|----------|----------|----------|------|
| 预定义规则 | UPF和SMF | PCF通知启用 | 基本家具，覆盖大多数需求 |
| 动态规则 | PCF | PCF直接下发启用 | 灵活但功能有限（无7层过滤），节约成本 |
| 本地规则 | UPF和SMF | SMF找不到PCF时下发 | 应急灯/灭火器，无PCF时保障业务 |

### 2.5 QoS基础——数据通路与QoS Flow模型

#### 2.5.1 数据通路概念

**数据通路**是端到端QoS控制的最小粒度：
- 4G：EPS Bearer（由EBI标识）
- 5G：QoS Flow（由QFI标识）

**核心规则**：相同通路上所有数据流获得相同QoS保障（调度策略、缓冲队列管理等）；不同QoS保障需要不同通路。

#### 2.5.2 4G与5G数据通路差异

| 对比项 | 4G | 5G |
|--------|-----|-----|
| 数据通路 | EPS Bearer | QoS Flow |
| 通路标识 | EBI | QFI |
| 通路分类 | 默认承载/专有承载 | 默认QoS Flow/专有QoS Flow |
| 接口实现 | 各段底层承载一一对应 | 无端到端承载概念，共用NG-U Tunnel |
| CN-RAN关系 | 各网元维护承载标识和绑定关系 | CN与RAN功能解耦，RAN自主决策空口承载 |

**5G关键优势**：QoS Flow使CN和RAN实现功能解耦，5G CN可适配各种接入技术（如固网接入），由RAN自主决策空口如何承载QoS Flow。

#### 2.5.3 业务识别与分流（两级映射）

5G采用**两级映射**机制（4G为一级映射）：

```
数据包 → [PFS匹配] → QoS Flow → [空口映射] → Radio Bearer
```

| 映射环节 | UL执行网元 | DL执行网元 | 关联工具 |
|----------|-----------|-----------|----------|
| 数据包→QoS Flow | UE | UPF | QoS Rule(UL) / PDR(DL) |
| QoS Flow→Radio Bearer | UE/AN | AN | 空口映射规则 |

**关联工具关系**（5G为N:1映射）：
- QoS Rule : QoS Flow = N : 1
- PDR : QoS Flow = N : 1
- QFI用于NG-U隧道中区分不同QoS Flow的数据包

### 2.6 QoS参数体系

#### 2.6.1 参数总览（4G/5G对比）

| 维度 | 4G参数 | 5G参数 | 含义 |
|------|--------|--------|------|
| QoS Flow级索引 | QCI | **5QI** | 业务质量索引（资源类型/优先级/可靠性/丢包率集合） |
| 通路优先级 | ARP | **ARP** | 分配保留优先级（优先级/抢占能力/被抢占属性） |
| GBR带宽 | GBR/MBR | **GFBR/MFBR** | 保证带宽/最大带宽 |
| Non-GBR聚合带宽 | APN-AMBR | **Session-AMBR** | 会话级聚合最大比特率 |
| UE级聚合带宽 | UE-AMBR | UE-AMBR | UE级聚合最大比特率 |

#### 2.6.2 5QI详解（5G新增内容）

5QI对应4G的QCI，是业务质量索引。5G在QCI基础上进行了增强：

**新增资源类型**：
- 4G：GBR / Non-GBR
- 5G：GBR / Non-GBR / **Delay-critical GBR**（用于uRLLC业务）

**新增标准5QI值**：69, 79~85（支持车联网、远程控制等新业务）

**新增参数**：

| 新增参数 | 适用对象 | 作用 |
|----------|----------|------|
| Averaging Window | GBR QoS Flow | 计算GFBR/MFBR的时间窗口 |
| Maximum Data Burst Volume | Delay-critical GBR QoS Flow | 空口在时延预算内传输的最大数据包长度 |

**5QI典型值速查**：

| 5QI范围 | 资源类型 | 典型业务 |
|---------|----------|----------|
| 1~4 | GBR | 语音会话、视频会话 |
| 5~9 | Non-GBR | 默认QoS Flow、IMS信令、TCP网页浏览 |
| 65~67 | GBR | 高质量视频会议 |
| 69~79 | Delay-critical GBR / Non-GBR | uRLLC、V2X、直播 |
| 75 | GBR | 高质量VR |
| 80~85 | Non-GBR | 低延迟TCP、视频/游戏 |

#### 2.6.3 5G QoS可选参数

| 可选参数 | 适用对象 | 作用 | 应用场景 |
|----------|----------|------|----------|
| RQA | Non-GBR QoS Flow | 指示支持Reflective QoS机制 | 不需保证带宽的业务（网页浏览） |
| Notification Control | GBR QoS Flow | RAN不能满足GFBR时通知SMF | 可根据QoS变化改变速率的业务（视频直播） |
| Maximum Packet Loss Rate | GBR QoS Flow | 空口侧最大丢包率 | 语音业务 |

### 2.7 QoS控制机制

#### 2.7.1 QoS参数决策层级

PCF拥有QoS参数最高决策权，SMF保存最终决策：

| QoS Flow类型 | 决策方式 |
|-------------|----------|
| 默认QoS Flow（5QI/ARP/Session-AMBR） | SMF从UDM获取签约QoS → 本地配置协商 → 上报PCF → PCF可覆盖 |
| 默认QoS Flow（UE-AMBR） | AMF从UDM获取 → 发送给RAN |
| 网络侧发起专有QoS Flow | PCF直接下发 |
| UE发起专有QoS Flow | UE请求中携带 → 最终以PCF下发为准 |

#### 2.7.2 QoS Flow控制两种机制

| 控制机制 | 适用对象 | 原理 |
|----------|----------|------|
| **信令控制** | 所有QoS Flow | SMF在PDU会话建立/修改时向UPF/RAN/UE发送消息，携带QoS Flow信息 |
| **Reflective QoS** | Non-GBR QoS Flow | UE根据下行数据包自行推演上行QoS Rule，减少信令开销 |

**Reflective QoS工作流程**：
1. UE注册/会话建立时上报Reflective QoS能力
2. SMF确定激活后，发送RQI给UPF，发送RQA给AN
3. UPF在下行数据包隧道包头打上RQI标记
4. AN将RQI和QFI通过空口传给UE
5. UE根据收到的数据生成上行QoS Rule，启动RQ定时器
6. 定时器超时则删除推导出的QoS Rule

## 3. 配置调测要点

### 3.1 QoS参数配置要素

#### 3.1.1 带宽控制相关QoS参数配置

| 参数 | 配置位置 | 带宽控制作用 |
|------|----------|-------------|
| **Session-AMBR** | PCF下发，SMF执行 | 限制PDU会话所有Non-GBR QoS Flow的总速率上限 |
| **GFBR** | PCF下发 | GBR QoS Flow的保证带宽，无线侧资源支持时必须保障 |
| **MFBR** | PCF下发 | GBR QoS Flow的最大带宽限制 |
| **5QI** | PCF/SMF决定 | 决定QoS Flow的资源类型、优先级、时延预算、丢包率 |
| **ARP** | PCF/SMF决定 | 决定QoS Flow建立/修改/抢占优先级 |
| **UE-AMBR** | UDM签约，AMF获取 | 限制UE所有PDU会话的总速率上限 |

#### 3.1.2 默认QoS Flow与专有QoS Flow配置

| 类型 | 创建时机 | 典型5QI | 特点 |
|------|----------|---------|------|
| 默认QoS Flow | PDU会话建立时 | 5~9（Non-GBR） | 必须创建，会话生命周期内存在 |
| 专有QoS Flow | 按需创建 | 1~4, 65~67, 75（GBR）或按需 | 按需创建/删除，用于特定业务QoS保障 |

### 3.2 SA识别配置要素

#### 3.2.1 规则配置关键参数

| 配置项 | 说明 | 示例 |
|--------|------|------|
| 规则类型 | 预定义/动态/本地 | BWM规则通常使用预定义规则 |
| 过滤条件 | 流过滤器（3/4层+协议+7层组合） | URL=www.example.com |
| 策略类型 | PCC/BWM/HEADEN/WEBPROXY/SMARTREDIRECT | BWM（带宽管理） |
| 规则优先级 | 同类型规则间匹配顺序 | 数值越大优先级越高 |
| 规则状态 | 部署未启用/启用 | PCF决定何时启用 |

#### 3.2.2 规则规划最佳实践

1. **确定业务逻辑和目的**：对什么用户→什么内容→启用什么策略
2. **区分策略类型**：计费用PCC、限速用BWM、头增强用HEADEN
3. **设置合理优先级**：避免不同业务过滤条件交集导致误匹配
4. **PCF条件判断独立配置**：如"话费耗尽"、"流量超限"等条件在PCF上配置，不在规则中体现
5. **本地规则作为兜底**：无PCF场景下保障基本业务

#### 3.2.3 BWM规则配置要点（带宽控制核心）

BWM（带宽管理）规则是带宽控制场景的核心规则类型：

| 维度 | 要点 |
|------|------|
| 过滤条件 | 可使用任意数据流（全量限速）或特定协议/IP（定向限速） |
| 策略绑定 | 绑定带宽管理分类策略 |
| 触发方式 | PCF根据用量/条件判断后下发启用 |
| 与PCC关系 | 独立匹配，一条报文可同时命中PCC和BWM规则 |
| 执行结果 | 分类结果用于带宽管理策略匹配 |

## 4. 与带宽控制场景的关系

### 4.1 SA是带宽控制场景的业务识别入口

带宽控制场景的完整链路：

```
用户数据到达UPF
    ↓
SA识别业务（三四层解析→协议识别→七层解析）
    ↓
规则匹配（BWM规则/URR规则等）
    ↓
触发带宽控制策略（限速/降速/保障）
    ↓
QoS执行（Session-AMBR限制/GBR保障/专用QoS Flow建立）
```

**SA在带宽控制中的具体作用**：
- **识别需要限速的业务**：如P2P/FTP协议识别后匹配BWM规则执行限速
- **识别需要保障的业务**：如视频会议业务识别后建立GBR QoS Flow保障带宽
- **累计流量识别**：SA识别特定业务后，URR对该业务流量进行累计，达到门限触发降速

### 4.2 QoS是带宽控制的执行框架

| QoS机制 | 带宽控制作用 | 典型场景 |
|---------|-------------|----------|
| **Session-AMBR** | 会话级带宽上限限制 | 用户套餐总速率限制 |
| **UE-AMBR** | UE级带宽上限限制 | 多PDU会话总速率限制 |
| **GFBR/MFBR** | GBR业务带宽保障与限制 | 视频会议保证带宽 |
| **5QI** | 业务QoS等级分配 | VIP用户分配高优先级5QI |
| **专有QoS Flow** | 按需建立专用通道 | 高价值业务带宽保障 |
| **ARP** | 带宽资源抢占优先级 | 资源不足时高优先级业务抢占 |

### 4.3 E2E QoS管理是带宽控制的全链路视角

带宽控制策略的端到端下发链路：

```
PCF（策略决策）→ SMF（策略翻译/执行控制）→ UPF/RAN/UE（策略执行）
```

**各网元在带宽控制中的职责**：

| 网元 | 在带宽控制中的职责 |
|------|-------------------|
| **PCF** | 决策：根据用户签约、用量上报、业务规则决定何时下发/修改/删除BWM规则和QoS参数 |
| **SMF** | 翻译：将PCF策略翻译为N4接口信令，控制UPF执行；管理QoS Flow建立/修改/删除 |
| **UPF** | 执行：执行SA识别、规则匹配、带宽限速、用量上报 |
| **RAN** | 执行：空口资源调度，根据QoS参数保障或限制无线带宽 |
| **UE** | 执行：上行数据包QoS映射，Reflective QoS推演 |

### 4.4 SA三大套餐与带宽控制场景的对应关系

| SA套餐 | 核心策略类型 | 与带宽控制的关系 |
|--------|-------------|-----------------|
| **套餐1 计费场景** | PCC规则 | 与带宽控制互补：计费策略和带宽策略可并行执行 |
| **套餐2 带宽控制场景**（本场景） | BWM规则 | **核心**：SA识别业务→BWM规则限速/降速 |
| **套餐3 访问限制场景** | PCC规则（阻塞/重定向） | 与带宽控制前置：访问限制决定哪些流量进入带宽控制链路 |

## 5. 关键术语

### 5.1 业务感知相关术语

| 术语 | 全称 | 含义 |
|------|------|------|
| **SA** | Service Awareness | 业务感知，对用户数据报文解析识别不同业务并执行策略的过程 |
| **PCC** | Policy and Charging Control | 策略与计费控制，5G核心网策略框架 |
| **BWM** | Bandwidth Management | 带宽管理，SA五大策略类型之一，带宽控制的核心策略 |
| **HEADEN** | Head Enhancement | 头增强，向HTTP/RTSP请求报文头域插入字段 |
| **PFS** | Packet Filter Set | 包过滤器集，5G中识别业务的最小单位集合 |
| **流过滤器** | Flow Filter | 3/4层过滤+协议过滤+7层过滤的组合 |
| **预定义规则** | Predefined Rule | 在UPF/SMF上规划，PCF通知启用的规则 |
| **动态规则** | Dynamic Rule | 在PCF上规划，PCF直接下发启用的规则（无7层过滤） |
| **本地规则** | Local Rule | SMF在无PCF时直接下发的规则 |

### 5.2 QoS相关术语

| 术语 | 全称 | 含义 |
|------|------|------|
| **QoS** | Quality of Service | 服务质量，在资源有限时为业务提供差异化服务 |
| **QoS Flow** | - | 5G端到端QoS控制的最小粒度，由QFI标识 |
| **QFI** | QoS Flow ID | QoS Flow标识符，PDU会话内唯一 |
| **5QI** | 5G QoS Identifier | 5G QoS索引，代表一组QoS参数取值集合 |
| **ARP** | Allocation and Retention Priority | 分配保留优先级，控制QoS Flow建立/修改/抢占优先级 |
| **GBR** | Guaranteed Bit Rate | 保证比特率，资源支持时一定保障的带宽 |
| **Non-GBR** | Non-Guaranteed Bit Rate | 非保证比特率 |
| **GFBR** | Guaranteed Flow Bit Rate | 保证流比特率（5G名称，对应4G的GBR） |
| **MFBR** | Maximum Flow Bit Rate | 最大流比特率（5G名称，对应4G的MBR） |
| **AMBR** | Aggregate Maximum Bit Rate | 聚合最大比特率，一组Non-GBR通路的总速率上限 |
| **Session-AMBR** | - | 会话级AMBR，限制PDU会话所有Non-GBR QoS Flow总速率 |
| **UE-AMBR** | - | UE级AMBR，限制UE所有PDU会话总速率 |
| **RQA** | Reflective QoS Attribute | 反射QoS属性，指示QoS Flow支持Reflective QoS |
| **DRB** | Data Radio Bearer | 数据无线承载，空口侧承载 |
| **PDU Session** | Protocol Data Unit Session | 协议数据单元会话 |
| **PDR** | Packet Detection Rule | 包检测规则，UPF上用于数据包→QoS Flow映射 |
| **DNN** | Data Network Name | 数据网络名称（对应4G的APN） |

### 5.3 网元相关术语

| 术语 | 全称 | 含义 |
|------|------|------|
| **PCF** | Policy Control Function | 策略控制功能，QoS策略最高决策者 |
| **SMF** | Session Management Function | 会话管理功能，策略翻译和执行控制 |
| **UPF** | User Plane Function | 用户面功能，SA识别和策略执行 |
| **AMF** | Access and Mobility Management Function | 接入和移动性管理功能 |
| **UDM** | Unified Data Management | 统一数据管理，存储用户签约QoS |
| **RAN** | Radio Access Network | 无线接入网 |
