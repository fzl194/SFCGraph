## 1. 维度一：地址分配维度

### 1.1 地址分配来源 × 地址类型 正交矩阵（6 来源 × 3 类型）

| 地址类型 ↓ / 分配来源 → | UPF/PGW-U 本地池 | SMF/PGW-C 本地池 | UDM 静态签约 | Radius 下发 | DHCP / DHCPv6 代理 | LNS（PPP） |
|---|---|---|---|---|---|---|
| **IPv4** | ✅ POOLTYPE=LOCAL [GWFD-010105] | ✅ [WSFD-010502][WSFD-010504] | ✅ Framed-IP [WSFD-010502] | ✅ Framed-Pool [GWFD-010105 子方式4][WSFD-011306] | ✅ DHCP 代理 [WSFD-104413] | ✅ IPCP [GWFD-020412] |
| **IPv6** | ✅ IPv6 SECTION [GWFD-010105][GWFD-020401] | ✅ [WSFD-010502] | ✅ Framed-IPv6-Pool | ✅ [WSFD-011306] | ✅ DHCPv6 代理 [WSFD-104005] | ✅ RA+IPv6CP [GWFD-020412] |
| **IPv4v6 双栈** | ✅ 双池双段 + License LKV3G5VDSA01 [GWFD-020403] | ✅ [WSFD-104002] | ✅ UDM 纠正 PDP Type=IPv4→IPv4v6 [WSFD-104002] | ✅ | ✅ DHCP+DHCPv6 双代理 | ✅ L2TP v02+ [GWFD-020412] |

> **矩阵结论**：6 种来源（UDG/SMF/UDM/Radius/DHCP/LNS）× 3 种类型（IPv4/IPv6/双栈）**理论上 18 格全可填**，但实际组合受特性 License 与互斥约束限制（详见 §1.4）。

### 1.2 地址池共性机制（跨 4 种子方式收敛）

UDG 本地地址分配 4 种子方式（基于 APN/DNN / 基于位置 / 基于 SMF / 基于 Radius 下发地址池名）共享同一套基础设施 [GWFD-010105]：

```
SECTION → POOL(LOCAL) → POOLGROUP → POOLGRPMAP(映射条件) → 规则匹配 → 分配
                                                        ↑
                               全局规则 SET IPALLOCRULE / APN 级规则 SET APNIPALLOCRULE
                               三级优先级规则字符串：APN-{0|1}&LOCATION-{0|1}&SMF-{0|1}
```

**5 项共性增值机制**（4 种子方式通用）[GWFD-010105]：
1. 地址池优先级（仅同类池间比较，段无优先级）
2. 地址延迟释放（避免敏感网元异常）
3. 地址租期续约（用户重接入复用原 IP）
4. 地址强制回收（需先锁定，强收回收会让在线用户下线）
5. 防止地址冲突（CONFLICTIP 标识）

**关键真值表**：UDG 地址分配方式选择由 PFCP `UE IP Address` 信元的 **CHV4/CHV6/V4/V6** 控制 [GWFD-010104][GWFD-010105]：
- CHV4/CHV6=1 & V4/V6=0 → **用户面分配**（UDG 本地）
- V4/V6=1 & CHV4/CHV6=0 → **外部分配**（SMF/Radius/UDM 下发）
- 软参 BIT391=0 标准 / BIT391=1 华为私有（Spare 扩展）

### 1.3 C-U 决策执行分离（UNC 决策 / UDG 执行）

**核心 C-U 分层模式**：地址分配在 C 面决策、U 面执行，通过 PFCP 消息解耦：

| 决策/执行 | UNC（C 面：SMF/PGW-C/GGSN） | UDG（U 面：UPF/PGW-U） |
|---|---|---|
| **地址分配方式决策** | [WSFD-010502] 4 种决策源（UDM/Radius/SMF本地/DHCP）；[WSFD-010504] ALLOCPRECEDENCE 选 SMF 或 UPF 本地 | [GWFD-010105] 执行 C 面指示的分配方式与类型 |
| **能力协商** | 发起 PFCP Association Setup | 响应携带 **UP Function Flag ADUP=1**（恒定） |
| **地址类型指示** | PFCP Session Est Req 携带 CHV4/CHV6/V4/V6 | 按真值表执行分配或接收外部地址 |
| **地址池名下发** | Radius→SMF→UPF（Framed-Pool） | 解析池名按指定池分配 [GWFD-010105 子方式4] |
| **位置信息透传** | PFCP 携带 LAC/TAC | 基于位置匹配池组 [GWFD-020421] |

> **跨产品对应**：GWFD-010104/010105（UDG）↔ WSFD-010502/010504（UNC）；两侧 License 均为"无"，但双栈/位置子方式独立 License。

### 1.4 IPv6 承载依赖链（License 串联）

IPv6 完整承载需**线性 License 依赖链**，是地址分配维度最长的依赖路径：

```
[GWFD-020401 IPv6承载上下文] LKV3G5V6PB01
        │ （承载基础设施，必装）
        ▼
[GWFD-020403 IPv4v6双栈接入] LKV3G5VDSA01
        │ （双栈能力使能层：使能 GWFD-010105 的 IPv4v6 取值）
        ▼
[GWFD-020406 IPv6 Prefix Delegation] LKV3G5P6PD01（V6PREFIXLENGTH<64 时触发）
        │
        ▼ （UNC 侧对称）
[WSFD-104002][WSFD-104004][WSFD-104005] IPv6/双栈/PD/DHCPv6
```

**关键规律**：
- **GWFD-020403 是能力使能层**，不替代 GWFD-010105 的分配机制，而是使能其 IPv4v6 地址类型落地 [GWFD-020403 §7.2]
- **前缀长度 64 是 PD 分水岭**：V6PREFIXLENGTH ∈ [49,64]；<64 切换为 IPv6 Prefix Delegation 模式 [GWFD-020403][GWFD-020406]
- **GWFD-020401 是 IPv6 leg 的承载管理基础**，独立于地址池机制

## 2. 维度二：鉴权维度

### 2.1 AUTHMODE 4 方式与 Radius 功能联动矩阵

UNC 侧 4 种鉴权接入方式（ACCESSMODE）与 Radius 功能（WSFD-011306）的依赖关系 [WSFD-011305]：

| ACCESSMODE | 中文 | 是否鉴权 | Radius 功能依赖 | AAA Server | 账密来源 | 计费 | PPP 用户 | 地址空间 |
|---|---|---|---|---|---|---|---|---|
| **TRANS_NON_AUTH** | 透明接入 | 否 | **否** | 否 | 不适用 | 可选 | ✅ | 运营商 |
| **TRANS_AUTH** | 透明鉴权（匿名） | 是（UNC 代发） | **必须** | **必须** | UNC 公用配置 | 可选 | ✅ | ISP/企业网 |
| **NON_TRANS** | 非透明接入 | 是（UE 透传） | **必须** | **必须** | UE PCO 携带 | 可选 | ✅ | ISP/企业网 |
| **LOC_AUTH** | 本地鉴权 | 是（本地匹配） | **否** | 否 | UE PCO + UNC 本地 | 不涉及 | **❌ 不支持** | 同其他 |

> **联动核心结论**：仅 TRANS_AUTH / NON_TRANS 强依赖 WSFD-011306；TRANS_NON_AUTH / LOC_AUTH 不调用 Radius [WSFD-011305 §3.3]。Radius 鉴权协议支持 PAP/CHAP（UNC 本地匹配仅 LOC_AUTH 不走 Radius）。

### 2.2 两套并列鉴权体系（底层 AKA vs. APN 业务鉴权）

APN 业务域存在**两套完全独立并列**的鉴权体系，文档层面互不引用 [WSFD-010301 §3.7][WSFD-011305 §3.5]：

| 维度 | WSFD-010301 鉴权功能（底层接入鉴权） | WSFD-011305 Radius 鉴权接入（APN 业务鉴权） |
|---|---|---|
| **鉴权层** | 网络接入层 | 应用层 / APN 级 |
| **鉴权对象** | SIM/USIM 卡身份（IMSI/SUPI/SUCI） | 用户名密码 |
| **鉴权协议** | AKA 系列（GSM/UMTS/EPS/5G AKA、EAP-AKA'） | PAP / CHAP via Radius 或本地匹配 |
| **鉴权网元** | UE ↔ SGSN/MME/AMF ↔ HLR/HSS/UDM(ARPF)/AUSF | UE/UNC ↔ AAA Server 或 UNC 本地 |
| **鉴权时机** | 移动性管理流程（注册/附着/RAU/TAU/服务请求） | PDU 会话/承载/PDP 上下文创建时 |
| **代际覆盖** | 2G&3G / 4G / 5G 三代 | 2G&3G / 4G / 5G 三代（GGSN/PGW-C/SMF） |
| **5G 特殊增强** | **归属网络鉴权确认**（AUSF 比对 RES* vs XRES*，防欺诈） | 无 |
| **License** | 无 | 无 |
| **与 AAA Server** | 不涉及 | TRANS_AUTH/NON_TRANS 强依赖 |

> **执行时序**：底层 AKA 先发生（UE 通过接入鉴权进入网络），APN 业务鉴权后发生（创建 PDU 会话时按 ACCESSMODE 触发）。两者隐式协同，无直接信令耦合。

### 2.3 鉴权级联依赖链（Radius 三件套 + 二次鉴权）

企业 DN 场景下形成 **5 特性级联强依赖链**（文档逐级明确声明）[WSFD-108007 §2.2]：

```
[WSFD-011306 Radius 功能]（协议承载：PAP/CHAP、服务器组、Bypass）
        │ 必须先激活
        ▼
[WSFD-011305 Radius 鉴权接入]（AUTHMODE 决策：TRANS_AUTH/NON_TRANS/LOC_AUTH 触发）
        │ 必须先激活
        ▼
[WSFD-108007 终端二次鉴权]（DN-AAA 经 UPF N4 GTP-U 隧道转接，License 82200DSF LKV2SECAA01）
        │ 二次鉴权对象：UE 访问企业/园区 DN 的授权
        ▼
[WSFD-011307 Radius 抄送]（鉴权/计费消息并行抄送给 WAP GW 等第三方）
```

**一次 vs 二次鉴权**[WSFD-108007 §8.2]：
- **一次鉴权（WSFD-011305）**：PDU 会话建立时，对接入侧 AAA（运营商/ISP/企业），决定能否接入网络
- **二次鉴权（WSFD-108007）**：会话建立后访问特定企业 DN 时，对 **DN-AAA（部署在企业/园区内部）**，决定能否访问该 DN
- **物理路径差异**：一次鉴权 SMF↔AAA 可直连；二次鉴权 SMF↔UPF↔DN-AAA **必须经 UPF 转接**（N4 GTP-U 隧道），根因是 UPF 下沉 + DN-AAA 部署企业内网导致 SMF 与 DN-AAA 不可直达
- **协议限制**：二次鉴权 **不支持 EAP、不支持 Diameter**，仅 PAP/CHAP via Radius

## 3. 维度三：接入方式维度

### 3.1 四隧道对比表（GRE / IPSec / MPLS / L2TP）

| 维度 | GRE [IPFD-015002] | IPSec [IPFD-015004/016000] | MPLS VPN [GWFD-020411/WSFD-104411] | L2TP VPN [GWFD-020412/WSFD-104410] |
|---|---|---|---|---|
| **隧道层次** | 三层（网络层封装） | 三层（IP 安全框架） | 三层（L3VPN，标签交换） | **二层**（PPP 隧道） |
| **封装协议** | GRE（IP 协议号 47） | AH(51)/ESP(50) + IKE | MPLS 标签（I-L 内层 + O-L 外层） | L2TP + PPP |
| **加密能力** | ❌ 不加密 | ✅ ESP 加密 + AH/ESP 认证 | ❌ 不加密（依赖骨干网信任） | ❌ 不加密（可叠加 IPSec） |
| **组播/广播支持** | ✅ | ❌（需 GRE over IPSec 解决） | ✅（VPN 路由） | ✅（PPP 协议） |
| **License** | **无** | **无** | **必须**（UDG:待补 / UNC:81203325 LKV2MPVPN01） | **C-U 不对称**（UDG 必须 82200BVC LKV3G5L2TP01；UNC 无） |
| **C-U 模式** | **对称同构**（两侧 ADD GRETUNNEL） | **对称同构**（两侧特性文档 hash 完全相同） | **对称同构**（两侧同名特性，9 子文档一一对应） | **C 决策 U 执行**（UNC 决策下发 LNS 参数，UDG 作 LAC 执行封装） |
| **核心配置对象** | GRETUNNEL + Tunnel 接口 + LoopBack | ACL→Proposal→IKEPeer→Policy→Tunnel 接口（UDG 独有 VNRS+IPsec 微服务双配） | VPN 实例(VRF) + RD + VPN Target + MP-BGP | APNL2TPATTR + L2TPGROUP/L2TPRDSCLIENT + PPPCFG |
| **关键约束** | 源地址与 IPSec 互斥；最多两层嵌套 | DH 组不能为 None；NAT 穿越仅 ESP 隧道模式 | 标签规格 299008；建议 perInstance + perNexthop | 与 GWFD-010108/020482 互斥；不支持 PPP 用户/DHCP 延迟分配/IPv6 PD/NAT |
| **对系统性能** | 无影响（轻量） | 有影响（加解密运算） | 对 PE 性能要求高 | 时延增加 + 封装开销 |
| **典型场景** | 异种网络互通、Radius 带内/带外组网、多租户共享 | 公网安全传输、GRE over IPSec、OSPF over IPSec、主备隧道、国密 | Intranet/Extranet/Hub&Spoke/跨域/MCE | 远程接入企业网（LAC-LNS） |
| **遵循标准** | RFC 1701/1702/2784 | RFC 2401/2402/2406 等 | RFC 4271/4760/4360/4364 等（60+） | RFC 2661/2868/5072/5571/8064 |

### 3.2 C-U 模式分类（对称同构 vs. C 决策 U 执行）

接入方式类 4 隧道按 C-U 协同模式清晰二分：

**模式 A：对称同构型**（GRE / IPSec / MPLS VPN）
- UDG 与 UNC 两产品**均部署同名特性**，两侧配置对象与命令基本对称
- 证据：IPFD-015004 与 IPFD-016000 特性概述 hash 完全相同（61317289）；GWFD-020411 与 WSFD-104411 子文档结构一一对应（基本概念/分标签/Hub&Spoke/跨域/IPv6 VPN/MCE/GR/NSR 共 9 篇）
- 场景：两侧各建隧道，用于 U 面或 C 面穿越异种/不可信网络

**模式 B：C 决策 U 执行型**（L2TP VPN）
- UNC 侧 WSFD-104410 **决策** LNS 参数并经 Sx/N4 PFCP 下发；UDG 侧 GWFD-020412 作 LAC **执行**封装与 PPP 透传 [GWFD-020412 §1.1]
- License **C-U 不对称**：UDG 必须 License 82200BVC，UNC 无 License
- C-U 协同通道：N4 接口 L2TP 私有信元（可用 SET L2TPKEY/L2TPN4KEY 加密）

### 3.3 GRE over IPSec 组合（弥补单方案缺陷）

**互补根因**[IPFD-015002 §1.6][IPFD-015004 §1.6]：
- GRE 弱点：不加密（明文）
- IPSec 弱点：不支持组播/广播（OSPF、路由协议无法直接跑）
- **GRE over IPSec**：先用 GRE 封装组播/广播报文，再引入 IPSec 加密 → 同时获得"组播支持 + 加密"

**硬约束**：GRE 隧道源地址 **不能与** IPSec 隧道源地址相同（双向互斥）。

## 4. 维度四：地址类型维度

### 4.1 IPv4 / IPv6 / 双栈选择决策树

```
UE 发起 PDU 会话，申请 PDU Type (IPv4 / IPv6 / IPv4v6)
        │
        ▼
UDM 签约数据查 PDP/PDU Type 签约
        │
        ├─ IPv4 ──────────────→ 走 IPv4 地址分配（任一 6 来源）
        │                       池配置：IPVERSION=IPV4 + V4STARTIP/V4ENDIP
        │
        ├─ IPv6 ──────────────→ 走 IPv6 地址分配
        │                       池配置：IPVERSION=IPV6 + V6PREFIXSTART/V6PREFIXEND + V6PREFIXLENGTH
        │                       需 License：LKV3G5V6PB01 [GWFD-020401]
        │
        └─ IPv4v6 双栈 ───────→ License 检查 LKV3G5VDSA01 [GWFD-020403]
                                │
                                ├─ License 已加载 → 双栈对称配置：
                                │   APN: HASVPN + HASVPNIPV6 双绑定
                                │   池组: IPv4 池 + IPv6 池双池并存
                                │   POOLGROUP: IPV4ALLOCPRIALG + IPV6ALLOCPRIALG 双使能
                                │   RA 通告：UDG 主动下发 Router Advertisement（★020403 独有，010105 未涉及）
                                │
                                └─ License 未加载 → 拒绝/降级为单栈 [WSFD-104002 纠正场景]
```

### 4.2 双栈 = 能力使能层（GWFD-020403 使能 GWFD-010105 的 IPv4v6 取值）

**关键分层关系**[GWFD-020403 §7.2]：

| 层 | 特性 | 职责 | License |
|---|---|---|---|
| **Layer A：能力使能层** | GWFD-020403 IPv4v6双栈接入 | 控制 UDG 是否允许创建 IPv4v6 上下文；RA 通告机制；双栈对称配置形态 | **必须** LKV3G5VDSA01 |
| **Layer B：分配机制层** | GWFD-010105 用户面地址分配 | 4 种子方式、三级优先级规则、PFCP 类型协商（CHV4=1&CHV6=1）、本地池体系 | **无** |

**关系本质**：GWFD-010105 的 `variant_dimensions` 中包含 `"IP地址类型(IPv4 / IPv6 / IPv4v6双栈)"`，但 IPv4v6 取值的真正落地**必须**叠加 GWFD-020403 的 License 使能。020403 是 010105 的"双栈特化版本"，非父子关系、非包含关系。

### 4.3 前缀长度 64 阈值（PD 分水岭）

IPv6 地址前缀长度 `V6PREFIXLENGTH ∈ [49, 64]`，**<64 即切换为 IPv6 Prefix Delegation 模式** [GWFD-020403 §3.1.3][GWFD-020406]：

| 前缀长度 | 分配模式 | 触发特性 | UNC 对应 |
|---|---|---|---|
| = 64 | 普通 IPv6 地址池分配（NDP/RA 通告接口 ID） | GWFD-020401 IPv6 承载上下文 | WSFD-104001 |
| < 64（49~63） | **IPv6 Prefix Delegation**（向 UE/企业代理整个 /60、/56 等网段） | GWFD-020406 IPv6 Prefix Delegation | WSFD-104004 |

> **PD 模式强依赖链**：GWFD-020406 文档明确列出 7 条特性交互（地址分配域依赖最密集的特性）；UNC 侧 WSFD-104004 必须配合 Radius 鉴权获取 PD 网段。

## 6. Stage 3 cross-analysis 预留问题

> 本维度归纳阶段无法定论，需 Stage 3 横向分析时进一步验证。

1. **C-U 地址分配决策权切分**：WSFD-010504（控制面地址分配方式）的 `ALLOCPRECEDENCE` 参数在何种场景下选择"SMF 本地分配"而非"UPF 本地分配"？两者并存时地址冲突如何避免？
2. **Radius 地址分配的端到端链路**：GWFD-010105 子方式4（UDG 侧基于 Radius 下发地址池名）↔ WSFD-011306（UNC 侧 Radius 功能）↔ WSFD-010502（UNC 侧 Radius 分配决策），三者完整链路信元映射需在 Stage 3 绘制时序图。
3. **IPv6 PD 在 UNC 侧的 3 个强依赖**：WSFD-104004 文档声明 3 条强依赖（详见该特性 §1.6），但具体依赖哪 3 个特性需 Stage 3 读取完整交互章节确认。
4. **终端二次鉴权与接入方式（IPSec/GRE）的场景关联**：WSFD-108007 文档未直接引用 IPFD-016000/IPFD-015002，但归类 MEC 解决方案 + UPF 下沉场景天然关联，Stage 3 需验证企业专网典型组网中二者的协同配置。
5. **多 PDN/PDU 与地址类型组合**：WSFD-010503 支持单用户 11 PDN（EPC）/ 15 PDU（5GC），多会话场景下每个会话的地址类型可独立决策（IPv4/IPv6/双栈混发）？需 Stage 3 验证。
6. **AUTHMODE 与地址空间归属联动**：TRANS_NON_AUTH 属运营商地址空间，NON_TRANS/TRANS_AUTH 属 ISP/企业网地址空间 —— 此联动是否在图谱中建模为 FeatureRule？
7. **C-U 对称同构 vs. C 决策 U 执行的判定准则**：GRE/IPSec/MPLS 对称 vs. L2TP 非对称，是否还存在其他非对称型接入方式（如 IPSec 在某些场景下是否也呈现 C-U 分工）？
