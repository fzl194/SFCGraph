## 1. 概述

### 1.1 特性定义（是什么）

UNC 的会话管理功能是 **SGSN / MME / SMF 的基本功能**，负责建立并管理 MS/UE 与 GGSN/PGW/UPF（对应外部 PDN/DN）之间的连接，是用户与外部数据网络建立连接、进行数据业务的基础。会话管理功能包括 **网络侧发起的会话管理** 和 **UE 发起的会话管理** 两类。

按接入代际不同，会话管理对象和术语有所差异：

| 接入代际 | 控制面 NF（UNC 侧） | 会话管理对象 | 用户面 NF（UDG 侧） |
|---------|--------------------|------------|--------------------|
| 2G&3G | SGSN、GGSN-C | PDP 上下文（PDP Context） | GGSN-U |
| 4G | MME、SGW-C、PGW-C | EPS 承载（默认承载 + 专有承载） | SGW-U、PGW-U |
| 5G | AMF、SMF | PDU 会话（含 QoS Flow） | UPF |

本特性是 **APN 业务域的根底座**：所有 APN 相关子特性（地址分配 WSFD-010502、多PDN/PDU WSFD-010503、别名APN WSFD-106203、用户接入控制 WSFD-106003、UPF 选择 WSFD-107010 等）都建立在本特性提供的会话生命周期管理之上。

> 来源：`WSFD-010501 会话管理特性概述（适用于5G）_48483833.md`（"定义"章节）、`WSFD-010501 会话管理特性概述（适用于4G）_73595351.md`（"定义"章节）、`WSFD-010501 会话管理特性概述（适用于2G&3G）_85436070.md`（"定义"章节）

## 2. 核心概念

### 2.1 关键术语

| 术语 | 全称 | 说明 |
|------|------|------|
| PDP Context | Packet Data Protocol Context（分组数据协议上下文） | 2G&3G 会话管理对象，保存用户面隧道转发的所有信息（RNC/GGSN 的用户面IP、TEID、QoS等） |
| PDN 连接 | PDN Connection | 4G 会话管理对象，EPS 通过 E-UTRAN+EPC 为 UE 提供的 UE 与 PDN 之间的 IP 连接 |
| EPS 承载 | EPS Bearer | 4G QoS 控制粒度，唯一标识 UE 与 P-GW 之间的共同 QoS 业务流；分默认承载与专有承载 |
| 默认承载 | Default Bearer | 4G PDN 连接建立时创建的永久性承载；删除则 PDN 连接释放 |
| 专有承载 | Dedicated Bearer | 4G 除默认承载外按不同 QoS 需求建立的承载（如视频、网页） |
| PDU 会话 | PDU Session | 5G 会话管理对象，UE 与 DNN 标识的数据网络之间的 PDU 连接业务 |
| QoS Flow | QoS Flow | 5G QoS 控制粒度，类似 4G 承载；分默认 QoS Flow 与专有 QoS Flow |
| DNN | Data Network Name | 5G 数据网络名称（对应 4G 的 APN） |
| SSC Mode | Session and Service Continuity Mode | 5G 会话与业务连续性模式（SSC1/SSC2/SSC3） |
| APN NI | APN Network Identifier | APN 网络标识，必选部分，标识外部数据网络类型 |
| APN OI | APN Operator Identifier | APN 运营商标识，可选部分，由 IMSI 导出的 MCC/MNC 组成 |
| 野卡APN | Wildcard APN | 签约时不限制业务类型，签约 APN 为 *，可访问任何 PDP/PDN Address |
| TFT | Traffic Flow Template | 4G 一组数据包过滤器的集合，UE 用于决定业务数据流与无线承载的匹配关系 |
| TEID | Tunnel Endpoint Identifier | GTP 隧道端点标识 |
| PFCP | Packet Forwarding Control Protocol | 4G CUPS/5G 控制面与用户面间的信令交互协议（替代 GTP-C 部分功能） |

> 来源：`会话管理基本概念_78831488.md`、`WSFD-010501 会话管理特性概述（适用于4G）_73595351.md`、`WSFD-010501 会话管理特性概述（适用于2G&3G）_85436070.md`（"基本概念"章节）

### 2.2 对象模型（4G vs 5G 会话管理概念对比）

4G 与 5G 会话管理在对象模型上有显著演进，对比如下（5G 是 SMF 集中化的关键变化）：

```
┌─────────────────────────────────────────────────────────────┐
│ 4G 网络架构（会话管理由 MME + S-GW/P-GW 共同完成）            │
│                                                             │
│   UE ── eNodeB ── MME（移动性+部分SM）                       │
│                  ── S-GW/P-GW（SM + 数据转发）               │
│                                                             │
│   会话管理对象：EPS 承载                                      │
│     ├── 默认承载（PDN连接建立时创建，永久）                    │
│     └── 专有承载（按 QoS 需求建立）                           │
│   QoS 控制粒度：EPS 承载                                     │
└─────────────────────────────────────────────────────────────┘
                          ↓ 演进（CUPS + 5G）
┌─────────────────────────────────────────────────────────────┐
│ 5G 网络架构（SMF 聚合 MME的SM模块 + S-GW/P-GW的C面功能）      │
│                                                             │
│   UE ── gNodeB ── AMF（接入和移动性管理，透传SM NAS消息）     │
│                  └─ SMF（★集中会话管理）                      │
│                       └─ UPF（数据转发）                      │
│                                                             │
│   会话管理对象：PDU 会话                                      │
│     ├── 默认 QoS Flow（按需建立，非强制）                     │
│     └── 专有 QoS Flow（默认不能满足时建立）                   │
│   QoS 控制粒度：QoS Flow                                     │
│                                                             │
│   关键变化：移动性管理与会话管理解耦                           │
│   关键变化：UE 注册时不强制建立"默认承载"（NB-IoT 友好）       │
└─────────────────────────────────────────────────────────────┘
```

> 来源：`会话管理基本概念_78831488.md`（"会话管理的控制网元"、"会话管理4G vs 5G"章节）

### 2.3 SSC Mode（5G 会话与业务连续性，5G 独有概念）

5G 系统支持会话和业务的连续性（SSC）。一个 PDU 会话的 SSC Mode 在该会话生命周期里保持不变。当已有 PDU 会话的 SSC Mode 不满足应用要求时，UE 会为应用建立新的 PDU 会话。

| SSC Mode | 原理 | 应用场景 |
|----------|------|---------|
| **SSC Mode 1**（提供IP连续性） | 网络提供给 UE 的 IP 地址与为 UE 选择的 UPF **保持不变** | IMS 语音等对业务连续性有高要求的应用 |
| **SSC Mode 2**（不提供IP连续性） | 当 SMF 确定服务 UPF 需改变时（如非最优路径），SMF 请求 UE **释放原 PDU 会话，重新建立到相同 DN 的新 PDU 会话**，SMF 为新会话选择新 UPF（**先断后连**） | 缓存类视频等对连续性要求不高、允许短暂中断的应用 |
| **SSC Mode 3**（提供短暂IP连续性） | 当 SMF 决定切换会话路径时，SMF 请求 UE **重新建立新 PDU 会话**，SMF 选新 UPF，**定时器到时或业务流转移后释放原会话**（**先连后断**） | MPTCP 等支持多径传输的应用 |

SSC Mode 选择策略：运营商向 UE 提供 SSC Mode 选择策略；SMF 与 UDM 交互获取 SMF 级别签约信息（允许的 SSC Mode 和缺省 SSC Mode）。

> 来源：`会话管理基本概念_78831488.md`（"SSC Mode"章节）

## 3. 原理与流程

### 3.1 5G 会话管理三大流程（SMF 集中决策，★本特性重点）

5G 核心网支持 PDU 连接业务，即在 UE 和 DNN 所标识的数据网络之间提供 PDU 交换的服务。PDU 连接业务通过 UE 请求建立的 PDU 会话来支持。**PDU 会话的创建、修改、释放通过 UE 和 SMF 之间的 NAS 信令交换实现**。

5G 会话管理的三大业务流程：

| 流程 | 触发场景 | 关键动作 |
|------|---------|---------|
| **PDU 会话建立** | UE 需要与外部网络业务交互、从 4G PDN 连接切换到 5G PDU 会话 | 包括 UE 的 IP 地址分配、UPF 选择、PCC 策略获取、UE 接入外部网络的权限检查 |
| **PDU 会话修改** | UE 和网络之间一个或多个 QoS 参数被修改、UE 能力变更 | 更新 QoS 参数；5G 中 QoS Flow 的新增/修改/删除统一由 PDU 会话修改流程实现 |
| **PDU 会话释放** | UE 不再需要相关业务、PCF/SMF 本地配置释放策略 | 释放该会话相关的资源（释放所有 QoS Flow） |

> 关键区别（5G vs 4G）：5G 新增 **"已有 PDU 会话用户面连接的选择性激活/去激活"** 流程
> - 激活：通过 Service Request 流程激活已有 PDU 会话，**只需建立用户面连接**
> - 去激活：删除已有 PDU 会话对应的用户面，**信令面连接保持不变**
> - 目的：节约无线侧资源的同时便于快速恢复 PDU 会话

> 来源：`WSFD-010501 会话管理特性概述（适用于5G）_48483833.md`（"原理概述"章节）、`会话管理基本概念_78831488.md`（"会话管理的流程"章节）

### 3.2 AMF/SMF/UPF 会话交互流程（PDU 会话建立端到端）

5G PDU 会话建立涉及 AMF（接入与移动性管理）、SMF（会话管理）、UPF（用户面）三者的协同交互：

```
┌─────────────────────────────────────────────────────────────┐
│ 步骤1：UE 发起 PDU 会话建立请求                              │
│   UE → AMF：PDU Session Establishment Request（NAS 消息）    │
│   携带：DNN、PDU Session ID、Request Type、SSC Mode 等       │
└──────────────────────┬──────────────────────────────────────┘
                       │ AMF 透传 SM NAS 消息（AMF 不处理 SM）
                       ▼
┌─────────────────────────────────────────────────────────────┐
│ 步骤2：SMF 选择（AMF 完成）                                  │
│   AMF 根据 DNN/S-NSSAI/NSF 选择 SMF                          │
│   AMF → SMF：调用 SMF 的 Nsmf_PDUSession_CreateSMContext    │
└──────────────────────┬──────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐

└──────────────────────┬──────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
```

### 3.3 PDU 会话修改流程（多触发源）

5G PDU 会话修改可由以下触发源发起（与 4G 承载更新类似但统一为会话级）：

| 触发源 | 修改场景 |
|--------|---------|
| UE 发起 | UE 能力变更、请求 QoS 变更 |
| PCF 发起 | PCF 下发新策略导致 QoS Flow 变更 |
| AN 发起 | (R)AN 通知 QoS 参数变化 |
| UDM 发起 | UDM 签约数据变更 |

5G 将 4G 的"专有承载激活、承载更新、承载去激活"三个流程统一整合为"PDU 会话修改"流程，因为 5G 隧道模型从承载粒度改为**会话粒度**。

> 来源：`会话管理基本概念_78831488.md`（"会话管理的流程"章节）

### 3.4 PDU 会话释放流程（多触发源）

| 触发源 | 释放场景 |
|--------|---------|
| UE 发起 | UE 不再需要相关业务 |
| PCF 发起 | PCF 策略变更触发释放 |
| SMF 发起 | SMF 本地配置释放策略（如超时、非法检测） |
| AMF 发起 | AMF 释放 UE 上下文时连带释放 |

释放动作 = 释放该会话相关的所有资源 = **释放所有 QoS Flow**（对应 4G 的 PDN 去连接 = 释放所有承载）。

> 来源：`WSFD-010501 会话管理特性概述（适用于5G）_48483833.md`（"原理概述"章节）

### 3.5 4G 会话管理流程（EPS 承载视角，对比参考）

4G 会话管理分为网络侧发起和 UE 发起两类：

**网络侧发起的会话管理**：

| 流程 | 目的 | 触发 |
|------|------|------|
| 专有 EPS 承载激活 | 在 UE 和 EPC 之间建立特定 QoS 和 TFT 的承载 | PGW-C 触发；UE 请求的承载资源分配/修改 |
| EPS 承载修改 | 修改 EPS 承载的 QoS 和 TFT | HSS/PGW-C 触发；UE 请求 |
| EPS 承载去激活 | 去激活一个 EPS 承载或去激活到 PDN 的所有 EPS 承载（断开 PDN 连接） | MME/eNodeB/PGW-C 触发；UE 请求 PDN 断连 |

**UE 发起的会话管理**：

| 流程 | 目的 |
|------|------|
| 默认承载上下文激活 | 在 UE 和 EPC 之间建立默认 EPS 承载（也是附着流程的一部分） |
| UE 请求 PDN 连接建立 | 请求建立到一个 PDN 的默认承载（允许多 PDN 接入） |
| UE 请求 PDN 连接断连 | UE 请求从 PDN 断开（释放所有 EPS 承载） |
| UE 请求承载资源分配/修改 | 为新业务数据流请求分配/修改承载资源 |

**4G 会话管理典型触发场景**：

| 流程 | 典型场景 |
|------|---------|
| 默认承载激活 | 4G 用户打开手机后 UE 发起默认承载，建立 PDN 连接（PDN 连接与默认承载同步建立） |
| 专有承载激活 | 部署动态 PCC：PCRF→P-GW 发送 RAR 触发专有承载建立（含 QoS policy）；未部署：P-GW 应用本地 QoS 策略 |
| 承载更新 | HSS 发起修改 QCI/ARP/UE-AMBR/APN-AMBR；P-GW 发起修改 QCI/GBR/MBR/ARP（如 PCRF FUP 触发 IP-CAN modification） |
| 承载去激活 | eNodeB/MME 发起（仅专有承载）；OAM 发起；P-GW 发起（专有或全部）；3GPP↔Non-3GPP 切换；无数据传输超时释放 |

> 来源：`WSFD-010501 会话管理特性概述（适用于4G）_73595351.md`（"原理概述/会话管理功能"章节、表1/表2/表3）

### 3.6 2G&3G 会话管理流程（PDP 上下文视角，对比参考）

2G&3G 会话管理一方面完成核心网络 SGSN 到 GGSN 之间的隧道建立、修改、释放控制；另一方面完成 SGSN 和 RNC/MS 之间无线接入承载（RAB）建立、修改、释放控制。

主要流程：

| 流程 | 说明 | 变体 |
|------|------|------|
| PDP 上下文激活 | 发起一次分组数据业务呼叫；分配空口带宽和 GTP 隧道资源 | MS 发起、网络发起、二次激活 |
| PDP 上下文去激活 | 删除一个会话并释放相关资源 | MS 发起、SGSN 发起、GGSN 发起 |
| PDP 上下文修改 | 修改已激活 PDP 上下文的 QoS 等参数 | MS/SGSN/GGSN 发起、RAB/Iu 释放引起 |
| **PDP 上下文保留** | 长时间无数据传输时暂时释放 RAB 和 Iu 资源，**PDP 上下文内容不受影响保留**；需传输时通过 Service Request 重建 RAB | 3G 独有 |

**MS 发起的 PDP 上下文激活流程**（GnGp SGSN 场景）：

```
1. MS → SGSN：Activate PDP Context Request（NSAPI, TI, PDP Type, PDP Address, APN, QoS Requested）
   （PDP Address 如是动态地址则设为空）
2. （GSM）可能执行安全流程
3. SGSN 根据 PDP Type/PDP Address/APN 签约数据验证请求有效性
4. SGSN 给 PDP Context 分配 TEID；选择 APN；向 GGSN 发送 Create PDP Context Request
5. GGSN 分配动态地址/charging ID/用户面和控制面IP+TEID；协商 QoS
   GGSN → SGSN：Create PDP Context Response
6. （UMTS）执行 RAB 指配过程建立 RAB
   - SGSN → RNC：RAB Assignment Request
   - RNC 建立无线承载
   - RNC → SGSN：RAB Assignment Response
7. SGSN → MS：Activate PDP Context Accept（含 PDP 地址、QoS）
```

**MS 发起的 PDP 上下文去激活流程**（GnGp SGSN 场景）：

```
1. MS → SGSN：Deactivate PDP Context Request（TI, Teardown Ind）
   （Teardown Ind 指示是否去激活共享地址的 PDP Context）
2. SGSN → GGSN：Delete PDP Context Request
3. GGSN → SGSN：Delete PDP Context Response
4. SGSN → MS：Deactivate PDP Context Accept
5. （UMTS）SGSN 调用 RAB 指配过程释放 RAB
```

> 来源：`WSFD-010501 会话管理特性概述（适用于2G&3G）_85436070.md`（"原理概述"章节）、`实现原理_85436071.md`（"具体流程介绍"章节）

### 3.7 APN 选择原理（2G&3G SGSN 与 4G MME 共有，5G 演进为 DNN 解析）

SGSN/MME 根据 UE 请求参数和 HLR/HSS 签约参数进行 APN 选择。APN 由两部分组成：

- **APN NI（Network Identifier）**：必选，标识外部数据网络类型（如 `huawei.com`）
- **APN OI（Operator Identifier）**：可选，标识运营商类型（如 `MNCxxx.MCCyyy.gprs`）

**APN 选择场景**（SGSN/MME 解析请求消息中的 PDN/PDP type、Address、APN 三元组与签约三元组匹配）：

| 用户请求消息包含 | APN 选择逻辑 |
|---------------|------------|
| type + address + APN | 签约含三项 → 选请求的 APN |
| type + address | 签约含对应 address → 选签约 APN；多条 → 选缺省 APN |
| type + APN | 签约含请求 APN → 选请求 APN；无签约仅野卡 → 选 UE 请求 APN |
| 仅 type | 签约仅一条野卡 + MME/SGSN 配置缺省 APN → 选缺省 APN |
| 都不含 | 签约一条记录 → 按签约 address/APN 选；野卡 → 选缺省 APN |

**APN OI 选择模式**（控制漫游用户使用 HPLMN/VPLMN 网关，软参 `DWORD_EX5 BIT17=1` 时生效）：

| 模式 | 含义 | 形成的 APN OI 列表 |
|------|------|------------------|
| H-HPLMN only | 只允许本网用户接入 HPLMN 网关 | 本地配置 APN OI + IMSI 提取 MCC/MNC |
| C-HPLMN only | 只允许外网用户接入 HPLMN 网关 | IMSI 提取 MCC/MNC |
| C-VPLMN only | 只允许外网用户接入 VPLMN 网关 | 本地配置 APN OI + ADD HPLMN 配置 |
| C-VPLMN first | 外网用户先 VPLMN，DNS 解析失败再 HPLMN | C-VPLMN only + C-HPLMN only |

> 来源：`实现原理/APN选择_75597698.md`（"APN选择"章节）、`实现原理_85436071.md`（"APN选择"章节）

| 命令/参数 | 作用 | 关联 |
|---------|------|------|
| ADD GWSELPLCY | 增加 GGSN/PGW 选择策略 | 控制 H-HPLMN/C-VPLMN 网关选择；本地配置 APN OI 来源 |

### 4.4 约束条件

| 约束类型 | 约束内容 | 适用代际 |
|---------|---------|---------|
| 每用户最大会话数 | 11（4G 显式声明，2/3G SGSN 同样支持最多 11 个 PDP 上下文） | 4G/2G&3G |
| 每用户每 DNN 最大 PDU 会话数 | 由 `SET SMPDUCTRL` 配置（防止资源过度消耗） | 5G |
| SSC Mode 生命周期不变性 | 一个 PDU 会话的 SSC Mode 在该会话生命周期里**保持不变**；不满足时需建立新 PDU 会话 | 5G |
| 网络发起 PDP 激活地址限制 | 在网络发起 PDP 上下文激活请求时，**只能使用静态地址**（不能用动态地址） | 2G&3G |
| 二次激活资源重用 | 主 PDP（Primary）与二次 PDP 上下文使用**相同 APN 和相同 IP 地址**，但 QoS 不同 | 2G&3G |
| QoS 协商取最小值 | 会话管理在 MS/RNC/SGSN/GGSN 之间进行 QoS 协商，在签约 QoS、SGSN 能提供的最大 QoS、其它节点满足的 QoS 之间**取最小值** | 2G&3G |
| PDN 连接与默认承载同步 | PDN 连接和默认承载同步建立；默认承载删除则 PDN 连接释放 | 4G |
| APN OI 模式列表顺序解析 | APN 域名列表按顺序依次到 DNS 解析，直至成功 | 4G/2G&3G |

## 5. 配置案例

### 5.1 典型场景一：2G&3G MS 发起 PDP 上下文激活（GnGp SGSN，UMTS）

**场景描述**：MS 通过 GPRS/UMTS 网络接入外部 PDN，发起 PDP 上下文激活请求，建立 MS 与 GGSN 之间的会话连接。

**配置/触发流程（原样保留文档中的流程描述）**：

```
1. MS → SGSN：Activate PDP Context Request
   消息信元：NSAPI, TI, PDP Type, PDP Address, Access Point Name, QoS Requested
   说明：PDP Address 如是动态地址则设为空

2. （GSM系统）可能执行安全流程

3. （GSM系统，激活BSS跟踪时）SGSN → BSS：Invoke Trace

4. SGSN 根据用户和 PDP Context 签约数据中的 PDP Type(optional), PDP Address(optional),
   Access Point Name(optional) 验证 Activate PDP Context Request 的有效性
   - SGSN 给 PDP Context 分配 TEID
   - 如使用动态地址，要求 GGSN 分配动态地址
   - SGSN 根据算法选择 APN
   - SGSN → GGSN：Create PDP Context Request
   - GGSN 分配：动态地址、charging ID、GGSN用户面IP+TEID、GGSN控制面IP+TEID、协商QoS
   - GGSN → SGSN：Create PDP Context Response
   说明：如 MS 要求外部网分配 IP 地址，将 PDP Address 设为 0.0.0.0，后续外部网分配后执行
        GGSN 发起的 PDP Context 修改过程

5. （UMTS系统）执行 RAB 指配过程建立 RAB
   a. SGSN → RNC：RAB Assignment Request（可指定无线优先级、是否允许抢占和排队）
   b. RNC 建立无线承载
   c. RNC → SGSN：RAB Assignment Response（QoS 原因指配失败则降低 QoS 重发）

6. （UMTS系统）SGSN → RAN：Invoke Trace

7. （GSM系统）SGSN 可能执行 BSS Packet Flow Context 过程与 BSS 进行 QoS 流控参数协商

8. 如 SGSN 降低了 QoS（UMTS第4/5步或 GSM 第7步）
   SGSN → GGSN：Update PDP Context Request
   GGSN → SGSN：Update PDP Context Response

9. SGSN → MS：Activate PDP Context Accept
   信元：NSAPI, PDP ADDR, GGSN ADDR, TEID, QoS
```

> 来源：`实现原理_85436071.md`（"具体流程介绍/MS发起的PDP上下文激活流程"章节）

### 5.2 典型场景二：2G&3G S4 SGSN 场景 UE 发起 PDP 上下文激活（UMTS）

**场景描述**：引入 S4 SGSN 网元后，PDP 上下文激活流程有所差异（Gateway 缺省承载创建+更新）。

**关键差异步骤**：

```
Gateway 缺省承载创建：
  - S4 SGSN → S-GW：Create Session Request
    （指示 S-GW 创建缺省承载；开启 Direct Tunnel 时携带 RNC 的用户面地址和 TEID，
     未开启时携带 SGSN 的用户面地址和 TEID）
  - S-GW → P-GW：Create Session Request（通知 P-GW 创建缺省承载）
  - P-GW → S-GW：Create Session Response（指示创建缺省承载）
  - S-GW → S4 SGSN：Create Session Response
    （携带 S-GW 和 P-GW 的用户面 TEID 和 IP 地址，缺省承载建立完成）

Gateway 缺省承载更新（仅 Direct Tunnel 开启时）：
  - S4 SGSN → S-GW：Modify Bearer Request（请求更新承载）
  - S-GW → S4 SGSN：Modify Bearer Response（更新完成）
  说明：告知 RNC S12 口的用户名地址和 TEID
```

> 来源：`实现原理_85436071.md`（"具体流程介绍/S4 SGSN场景"章节）

### 5.3 典型场景三：2G&3G MS 发起 PDP 上下文去激活（GnGp SGSN）

**场景描述**：MS 主动发起 PDP 上下文去激活，释放会话相关资源。

**配置/触发流程（原样保留）**：

```
1. MS → SGSN：Deactivate PDP Context Request
   信元：TI, Teardown Ind
   说明：Teardown Ind 指示是否去激活和指定 TI 共享地址的激活的 PDP Context

2. （GSM系统）可能执行安全流程

3. SGSN → GGSN：Delete PDP Context Request（删除 GGSN PDP Context）
   GGSN → SGSN：Delete PDP Context Response

4. SGSN → MS：Deactivate PDP Context Accept（表示去激活接受）

5. （UMTS系统）SGSN 调用 RAB 指配过程释放 RAB

6. （GSM系统）SGSN 可能执行 BSS Packet Flow Context 过程与 BSS 进行 QoS 流控参数协商
```

> 来源：`实现原理_85436071.md`（"具体流程介绍/MS发起的PDP上下文去激活流程"章节）

### 6.5 常见问题与排查

| 问题现象 | 可能原因 | 排查方法 |
|---------|---------|---------|
| 5G PDU 会话建立失败 | UDM 签约未配置允许的 SSC Mode / DNN | 核对 UDM 签约：DNN、allowed SSC Mode、default SSC Mode |
| 5G PDU 会话建立失败 | UPF 选择失败（无可用 UPF） | 检查 WSFD-107010 UPF 选择配置；NSF/S-NSSAI 配置 |
| 5G PDU 会话建立失败 | 单 DNN PDU 会话数超限 | 检查 `SET SMPDUCTRL` 配置的最大值 |
| 4G 默认承载激活失败 | APN 未在 HSS 签约 | 核对 HSS 签约 APN；野卡 APN 配置 |
| 4G APN 选择错误 | 野卡 APN 与缺省 APN 优先级混淆 | 参考 APN 选择场景表；检查 MME 缺省 APN 配置 |
| 4G 专有承载无法激活 | PCRF 策略未下发 / P-GW 本地 QoS 策略缺失 | 检查动态 PCC 部署；PCRF RAR 消息跟踪 |
| 2/3G PDP 激活失败 | 网络发起激活时使用了动态地址（约束：只能用静态） | 改用静态地址；或改为 UE 发起 |
| 2/3G 二次激活冲突 | 主/二次 PDP 必须相同 APN 和 IP，仅 QoS 不同 | 核对二次激活参数 |
| 2/3G QoS 协商不达预期 | QoS 在签约/SGSN最大/其他节点间取最小值 | 逐节点核对 QoS 签约与配置 |
| APN OI 解析只走本地配置 | DWORD_EX5 BIT17=0（仅本地配置） | 评估是否需设为 1 启用多模式列表解析 |
| 漫游用户无法接入 | ADD GWSELPLCY / SET SDCFG RAH 配置不允许 | 检查网关选择策略与签约数据配置的漫游属性 |
| S-CDR 话单异常（2/3G） | PDP 修改 QoS 改变，SET CHGCHAR 配置决定是否重生成话单 | 核对 SET CHGCHAR 配置 |
