## 1. 概述

### 1.1 特性定义（是什么）

本特性在 UNC（控制面）上为不同代际网络分别提供 **IPv6 承载上下文** 的建立能力。按接入代际定义略有差异，但本质一致——为用户建立 IPv6 类型的承载/会话并分配 IPv6 类型的 IP 地址（前缀）：

| 代际 | 文档原文定义 | 承载对象 |
|------|-------------|---------|
| **2G&3G** | "SGSN、GGSN 支持 **IPv6 PDP 上下文**的激活、去激活和修改。IPv6 PDP 上下文承载于 GPRS 和 UMTS 的 IPv4/IPv6 骨干网" | PDP 上下文 |
| **4G** | "UNC 支持 **IPv6 EPS 承载**的激活、去激活和更新" | EPS Bearer |
| **5G** | "本特性支持在5G网络的 **PDU 会话建立流程**中为UE分配 **IPv6前缀类型的地址**，建立 **IPv6 PDU 会话**的用户上下文，并支持 IPv6 PDU 会话的创建、修改、释放" | PDU 会话 |

三者共性：UNC 能为用户创建 IPv6 类型的承载/会话，为用户分配 IPv6 类型的 IP 地址/前缀，IPv6 类型的数据包与 IPv4 类型的数据包一样在用户面接口上透明传输。

> 来源：`特性概述（SGSN/GGSN）_48043355.md`（"定义"章节）、`特性概述（MME、SGW-C/PGW-C）_48043369.md`（"定义"章节）、`特性概述（AMF/SMF）_48043368.md`（"定义"章节）

## 2. 原理

### 2.1 IPv6 技术基础（2/3G + 4G 共享描述）

随着 Internet 应用及规模的飞速发展，IPv4 协议的可用地址空间越来越少，IPv6 协议被提出来解决 IPv4 协议遇到的问题。IPv6 标准采取了一系列措施，增加了许多新功能。IPv6 中的变化体现在以下五个重要方面：

- **扩展地址**：地址由 IPv4 的 32bit 扩展为 128bit，IP 可以继续增长而无需考虑资源的匮乏，该地址结构对于提高路由效率也有所帮助
- **简化头格式**：包头的简化减少了路由器上所需的处理过程，从而提高了选择路由的效率
- **增强对于扩展和选项的支持**：可以在几乎不影响普通数据包和特殊包选路的前提下适应更多的特殊需求
- **流标记**：为更加高效地处理包流提供了一种机制，对于实时应用尤其有用
- **身份验证和保密**：使得 IPv6 更加适用于要求对敏感信息和资源特别对待的商业应用

> 来源：`特性概述（SGSN/GGSN）_48043355.md`（"原理概述/IPv6中的变化"章节）、`特性概述（MME、SGW-C/PGW-C）_48043369.md`（"原理概述/IPv6中的变化"章节）

### 2.2 系统实现（共性）

UNC（SGSN/GGSN/MME/SGW-C/PGW-C/AMF/SMF）能够为用户创建 IPv6 类型的承载/会话，为用户分配 IPv6 类型的 IP 地址，IPv6 类型的数据包与 IPv4 类型的数据包一样在用户面接口上透明传输：

| 代际 | 系统实现描述 | 透明传输接口 |
|------|-------------|------------|
| 2/3G | UNC 为用户创建 IPv6 类型的 PDP 上下文，分配 IPv6 类型的 IP 地址；**UNC 还能够为同一用户分配不同 IP 地址类型的 PDP 上下文**，支持 IPv4 和 IPv6 的移动终端可以同时运行 IPv4 和 IPv6 的 PDP 上下文 | Gn、Gb 或 Iu 接口 |
| 4G | UNC 为用户创建 IPv6 类型的 PDP 上下文 / EPS bearer，分配 IPv6 类型的 IP 地址 | S5/S11/S1-U 接口 |
| 5G | SMF 在 PDU 会话建立流程中为用户创建 IPv6 的 PDU 会话，分配 IPv6 类型的 IP 地址/前缀 | N3/N9（用户面） |

> ★ 2/3G 系统实现明确声明"UNC 还能够为同一用户分配不同 IP 地址类型的 PDP 上下文"（即支持 IPv4+IPv6 并存于不同 PDP 上下文）——这与 WSFD-104002 双栈（同一上下文含 V4+V6）不同：本特性描述的是**并存式**多 PDP 上下文，WSFD-104002 描述的是**单上下文双栈**。

> 来源：`特性概述（SGSN/GGSN）_48043355.md`（"原理概述/系统实现"章节）、`特性概述（MME、SGW-C/PGW-C）_48043369.md`（"原理概述/系统实现"章节）、`特性概述（AMF/SMF）_48043368.md`（"原理概述"章节）

### 2.3 IPv6 PDP 上下文激活流程（2/3G）

以用户携带**静态 PDP Address**发起激活为例：

```
1. MS → SGSN：Activate PDP Context Request
   携带：PDP Type=IPv6, PDP Address 不为空（静态地址）
2. SGSN → GGSN：Create PDP Context Request
   携带：PDP Type=IPv6, PDP Address 不为空
   ★ GGSN 根据请求消息中携带的静态 PDP Address 进行地址有效性检查
   检查通过 → GGSN → SGSN：Create PDP Context Response
3. 执行 RAB 指配过程建立 RAB
4. SGSN → MS：Activate PDP Context Accept（通知激活成功）
5. ★ IPv6 场景下：GGSN 生成 IPv6 路由广播（前缀）并且经 SGSN 将其发送到 MS
```

> 来源：`特性概述（SGSN/GGSN）_48043355.md`（"原理概述/业务流程"章节）

### 2.4 IPv6 的 EPS Bearer 激活流程（4G，23步完整流程）

```
1.  UE → eNodeB：PDN 连接请求
    携带：PDN Type=IPv6, PDN Address 不为空
2.  eNodeB → MME：转发 PDN 连接请求
    携带：PDN Type=IPv6, PDN Address 不为空
3.  MME → SGW-C：Create Session Request
    携带：控制面 TEID、PDN GW address、APN、默认承载 QoS、计费参数、Dual Address Bearer Flag 等
4.  SGW-C → SGW-U：PFCP Session Establishment Request
    （SGW-C 在承载上下文列表中加入新记录）
5.  SGW-U → SGW-C：PFCP Session Establishment Response
    （SGW-U 缓存所有从 PGW-U 而来的下行数据包）
6.  SGW-C → PGW-C：Create Session Request
    （若从 MME 获得 MSISDN，则消息携带该信息）
7.  [可选] PGW-C 发起 IP-CAN 建立流程（启用动态 PCC）；否则采用网关本地配置策略
8.  PGW-C 在承载上下文列表加入新记录，产生 Charging ID
    ★ PGW-C 分配用户的 PDN Address
    PGW-C → PGW-U：PFCP Session Establishment Request
    （PGW-U 可向 SGW-U 发送用户面分组数据单元，开始计费）
9.  PGW-U → PGW-C：PFCP Session Establishment Response
10. PGW-C → SGW-C：Create Session Response
11. SGW-C → SGW-U：PFCP Session Modification Request
12. SGW-U → SGW-C：PFCP Session Modification Response
13. SGW-C → MME：Create Session Response
14. MME → eNodeB：E-Rab Setup Request
    含 NAS 消息 Activate default EPS bearer context request
15. eNodeB → UE：RRC Connection Reconfiguration
    （含无线承载 ID + 转发 Activate default EPS bearer context request）
16. UE → eNodeB：RRC Connection Reconfiguration Complete
17. eNodeB → MME：E-Rab Setup Response
    （含 eNodeB 的 TEID 和地址，供传输下行数据使用）
18. UE → eNodeB：Activate default EPS bearer context accept
19. eNodeB → MME：转发 Activate default EPS bearer context accept
20. MME → SGW-C：Modify Bearer Request (EPS Bearer Identity)
    （建立 S1-U 缺省承载，建立 S1-U 与 S5/S8-U 缺省承载映射）
21. SGW-C → SGW-U：PFCP Session Modification Request
22. SGW-U → SGW-C：PFCP Session Modification Response
    （SGW-U 收到 eNodeB 的 TEID 和地址，可发送缓存的下行数据包）
23. SGW-C → MME：Modify Bearer Response (EPS Bearer Identity)
```

> ★ 4G 流程文档未单独列出"PGW-C 生成 IPv6 路由广播（前缀）"步骤（与 2/3G 步骤 5、5G 步骤 17 不同）。结合 WSFD-104002 4G 流程明确声明"IPv6 场景下 PGW-C 生成 IPv6 路由广播（前缀）经 MME 发送到 UE"，可推断本特性在 4G 也具备该能力，但本文档未在 IPv6 单栈流程中显式列出（待 Stage 3 核对）。

> 来源：`特性概述（MME、SGW-C/PGW-C）_48043369.md`（"原理概述/业务流程"章节，23步）

### 2.5 IPv6 的 PDU 会话建立流程（5G，17步完整流程）

> 仅描述与 IPv6 地址分配强相关的消息和信元；PDU 会话建立详细流程可参见 3GPP 23.502 协议。

```
1.  UE → AMF：PDU 会话建立请求
    携带：Requested PDU Session Type=IPv6 或 IPv4v6
2.  AMF：根据 S-NSSAI、DNN 信息等为 PDU 会话建立选择 SMF
3.  AMF → SMF：Nsmf_PDUSession_CreateSMContext Request（请求创建 SM 上下文）
    携带：Requested PDU Session Type=IPv6 或 IPv4v6
4.  [可选] SMF → UDM：注册 PDU 会话、获取 UE 签约数据、订阅 UE 签约数据变化通知
5.  SMF → AMF：返回 PDU 会话响应（接受创建 PDU 会话）
6.  [可选] 获取 PCC 规则
7.  ★ SMF 选择 UPF，并根据会话请求消息中携带的 Requested PDU Session Type 类型，
       从 UDM 中存储的静态地址类型分配 IPv6 的 IP 地址/前缀
8.  [可选] SMF 发起 SM Policy Association Modification 流程向 PCF 报告信息
    （例如分配的 IPv6 地址/前缀）；PCF 可向 SMF 提供更新的策略信息
9.  SMF → UPF：N4 Session Establishment 流程（建立与 UPF 的连接）
    提供：数据监测、报告规则、CN 隧道信息等
10. SMF → AMF：Namf_Communication_N1N2MessageTransfer
    携带：分配的 IPv6 地址/前缀（通知 RAN 和 UE 需要建立 PDU 会话）
11. AMF → RAN：N2 PDU Session Request
    含 NAS message 信元，其 N1 SM Container 中包含 PDU Session Establishment Accept 及分配的 IPv6 地址/前缀
12. RAN ↔ UE：信令交互（转发 SMF 需要发送给 UE 的 N1 SM Container 消息）
    请求 UE 建立 PDU 会话，包含 PDU Session Establishment Accept 及分配的 IPv6 地址/前缀
13. [可选] RAN → AMF：N2 PDU Session Response（建立 AN 隧道信息）
14. AMF → SMF：转发从 RAN 接收到的 N2 SM Information
15. SMF → UPF：N4 Session Modification 流程
    （将 AN 隧道信息和相应的转发规则发送给 UPF）
16. SMF → AMF：Nsmf_PDUSession_UpdateSMContext Response
17. ★ IPv6 场景下：SMF 生成 IPv6 路由广播（前缀）并且经由 N4 会话和 UPF 将其发送到 UE
```

> ★ 5G 流程两处关键 IPv6 动作：步骤 7（SMF 分配 IPv6 地址/前缀）+ 步骤 17（SMF 生成 IPv6 路由广播经 N4+UPF 发送到 UE）。这两处即 WSFD-104002 5G 文档声明"依赖 WSFD-104001"的具体能力来源。

> 来源：`特性概述（AMF/SMF）_48043368.md`（"原理概述/业务流程"章节，17步）

### 2.6 IPv6 路由广播（前缀）生成机制（★核心能力，跨代际共性）

本特性的核心输出之一是 **IPv6 路由广播（前缀）的生成与下发**，这是 IPv6 地址分配的"无状态自动配置（SLAAC）"基础：

| 代际 | 生成方 | 下发路径 | 流程步骤 |
|------|--------|---------|---------|
| 2/3G | **GGSN** | GGSN → SGSN → MS | PDP 上下文激活流程步骤 5 |
| 4G | **PGW-C**（推断，文档未在 IPv6 单栈流程显式列出） | PGW-C → SGW-C → MME → UE | 见 §2.4 备注 |
| 5G | **SMF** | SMF → N4 会话 → UPF → UE | PDU 会话建立流程步骤 17 |

> ★ 此能力被 WSFD-104002（IPv4v6双栈接入）5G 文档明确声明为强依赖："对于给用户分配的IPv6地址，需要依赖该特性（WSFD-104001）生成IPv6路由广播并下发给UE"。WSFD-104002 2/3/4G 流程也描述了 GGSN/PGW-C 生成 IPv6 路由广播，但 2/3/4G 概述声明"不涉及交互"——存在声明与流程描述的内部矛盾（见 §8.1）。

> 来源：`特性概述（SGSN/GGSN）_48043355.md`（业务流程步骤5）、`特性概述（AMF/SMF）_48043368.md`（业务流程步骤17）、`WSFD-104002 IPv4v6双栈接入（5G）/特性概述_48043373.md`（依赖声明）

### 2.7 IPv6 地址分配来源（5G，两种决策源）

5G 场景中 SMF 分配 IPv6 地址/前缀有两种来源（与 WSFD-104002 双栈地址分配方式呼应）：

| 决策方式 | 地址来源 | SMF 角色 |
|---------|---------|---------|
| **UDM 静态分配** | UE 在 UDM 注册分组业务时确定的 IPv6 地址/前缀（静态地址） | 步骤 7 中"从 UDM 中存储的静态地址类型分配 IPv6 的 IP 地址/前缀" |
| **SMF 动态分配** | SMF 本地地址池 | SMF 在 PDU 会话建立阶段为该 PDU 会话分配 IPv6 地址/前缀 |

> ★ 本特性定义了"IPv6 单栈"的地址分配，WSFD-104002 定义了"IPv4v6 双栈"的地址分配。二者地址来源决策机制相同（UDM 静态 / SMF 动态），区别在于分配的地址类型（单 IPv6 vs IPv4+IPv6）。

> 来源：`特性概述（AMF/SMF）_48043368.md`（业务流程步骤7）

步骤1：进入"MML命令行-UNC"窗口
