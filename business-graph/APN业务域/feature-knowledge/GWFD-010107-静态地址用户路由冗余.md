## 1. 概述

### 1.1 特性定义（是什么）

**背景**：在网络中，终端地址分配方式分为静态地址分配和动态地址分配。运营商对于行业用户、M2M 业务用户，地址分配方式一般主推 HLR/HSS/UDM 静态签约分配 IP，或者 RADIUS 分配 IP。运营商为提升网络可靠性，部署两台 UDG 时，对于静态地址用户，可以选择主机路由方式或路由冗余方式实现 UDG 容灾部署。随着静态签约用户大量增加，主机路由方式需要通告大量主机路由，会对骨干网造成冲击。

**定义**：在网络部署时，同时部署冗余的主备两台 UDG。当主用 UDG 故障的时候，备用 UDG 可以及时承担业务。**主用 UDG 恢复后，通过主备两台 UDG 间建立的 GRE Tunnel，完成业务报文在主备 UDG 上的传输**，使得在备用 UDG 上激活的静态地址用户能够正常进行业务。

> 来源：`GWFD-010107 静态地址用户路由冗余特性概述_19612410.md`（"定义"章节）

对于静态签约用户，在运营商部署了主备 UDG 场景下，需要开通此特性，在主用 UDG 故障时，备用 UDG 可以及时承担起主用的功能，保证静态地址签约用户正常进行业务。

> 来源：`GWFD-010107 静态地址用户路由冗余特性概述_19612410.md`（"应用场景"章节）

### 1.9 对系统的影响（★ 对照 C 侧 WSFD-107021 全无）

**开启静态地址用户路由冗余特性后，对于 UDG 故障和故障恢复过程中，静态签约用户重新在备用 UDG 上激活，但静态签约用户的下行数据流通过 GRE Tunnel 转发到备份 UDG 上，系统处理负荷增加，报文转发性能和吞吐量将下降**，具体下降程度需要根据话务模型来评估。华为公司提供容量评估专业服务。

> 来源：`GWFD-010107 静态地址用户路由冗余特性概述_19612410.md`（"对系统的影响"章节）

### 1.10 应用限制（★ 对照 C 侧 WSFD-107021 全无）

- **限制1**：当用户上行数据流也通过备份 GRE Tunnel 由备用 UDG 转发到主用 UDG 时，**业务重定向功能失效**
- **限制2**：用户上下行报文必须经过同一组防火墙/外置 SA 设备，否则可能导致 UE/防火墙/外置 SA 设备处理异常

> 来源：`GWFD-010107 静态地址用户路由冗余特性概述_19612410.md`（"应用限制"章节）

### 2.3 在地址分配场景的角色

本特性是**地址分配域的可靠性增强特性**，位于"静态 IP 用户下行路由生成与发布"链路的冗余执行层：

```
静态IP用户/Radius用户/SMF分配用户等6类用户接入
    ↓
[UNC WSFD-107021] 基于静态IP段绑定关系选主备UPF（C面决策）
    ↓ N4 PFCP（会话建立指示）
    │
[GWFD-010104] UDG 地址分配方式（父场景，无冗余：单UDG下行路由发布）
    ↓
    ├── 正常场景：业务报文经主用 UDG 直达 PDN/DN
    │
    ↓ ★ GWFD-010107 冗余执行介入点
    │
[GWFD-010107] UDG 侧通过 GRE 隧道实现实际报文冗余转发（U面执行）
    ├── 主用正常：报文不经 GRE，直经主用 UDG
    ├── 主用故障：备用 UDG 承载业务（不经 GRE，因主用不发布路由）
    └── 主用恢复：备用 UDG 上用户上下行报文经 GRE 隧道转发回主用 UDG
```

具体角色说明：

1. **执行层（U 面）**：UDG 通过 GRE 隧道实现实际报文冗余转发，是冗余保护的执行方
2. **保护对象**：**6 类静态地址用户**（静态签约 + SMF 分配 + Radius + Radius 手机后路由 + Radius IPv6 PD + UL CL 辅锚点）
3. **触发条件**：主用 UDG 故障 / 故障恢复（通过 SMF N4 心跳检测感知 + 路由发布/撤销）
4. **与 GWFD-010104 的协同**：复用 GWFD-010104 的静态地址池配置体系（POOL/SECTION/POOLGROUP/POOLBINDGROUP/POOLGRPMAP 完全一致），新增 GRE 隧道 + 重定向 IP + 冗余开关 + 路由优先级控制

> 来源：综合 `GWFD-010107 静态地址用户路由冗余特性概述_19612410.md`（"原理概述"章节）、`激活静态地址用户路由冗余_19612403.md`

## 3. 原理与流程

### 3.1 实现原理（★ U 面 GRE 转发执行机制，三场景）

本特性的核心是基于 **主备 UDG + GRE Tunnel + 路由优先级** 的报文冗余转发机制：

```
┌─────────────────────────────────────────────────────────────────┐
│ 场景一：主用 UDG-1 正常                                          │
│                                                                 │

│ 场景二：主用 UDG-1 故障                                          │
│                                                                 │

│                                                                 │

│      └── 下行：PDN → PE5/PE6 → PGW-U/UPF1 → PE1/PE2 → 基站 → UE│
│                                                                 │

│      ↓                                                          │

│           下行路由走到 UPF1。但用户在 UPF2 激活，UPF1 无用户   │

│           PDN → PE5/PE6 → PGW-U/UPF1 →GRE隧道→ PGW-U/UPF2      │

```

### 3.2 下行路由发布机制（★ 网段级发布/撤销）

主备 UDG 的下行路由由 UDG 侧负责发布与撤销：

| UDG 状态 | 下行路由发布 | 下行数据流向 |
|---------|-------------|-------------|
| UPF-1 正常 | UPF-1 和 UPF-2 同时向 PDN 侧发布 UE 静态签约 IP 网段（OSPF/BGP） | 选路到 UPF-1（高优先级，低 COST） |
| UPF-1 故障 | **UPF-1 不对外发布下行路由**，UPF-2 继续发布 | 选路到 UPF-2 |
| UPF-1 恢复 | UPF-1 恢复发布；UPF-2 上已有用户的下行报文经 GRE 隧道转发到 UPF-1 | PDN→UPF-1→GRE隧道→UPF-2→UE（已在 UPF-2 的用户） |

**路由优先级控制方式**（3 种，来源：正常场景文档"说明"段落）：
1. UDG 控制：MOD OSPFIMPORTROUTE 的 COST 值（主=100，备=200）或 MOD IMPORTROUTE 的 MED 值
2. DC-GW 控制高低优先级路由
3. PDN/DN 侧服务器端配置高低优先级路由（此时需在主 UDG 配置 ADD SRROUTE/SRROUTE6 下行缺省低优先级静态地址用户网段路由入冗余 GRE 隧道）

> 来源：`主用PGW-U_UPF正常场景_19644451.md`（"说明"段落）、`主用PGW-U_UPF故障场景_19644452.md`（"说明"段落）、`主用PGW-U_UPF故障恢复场景_19644453.md`

### 3.3 静态地址路由冗余叠加 UE 互访场景（★ MASTERFLAG 机制）

随着 5G 2B 业务发展，UE 互访场景逐渐增多，叠加静态地址路由冗余后存在问题：

```
问题场景：UE 互访失败
  UPF1/UPF2 主备组网，使能静态路由冗余 + 开启上行报文隧道转发
  主 UPF 故障期间，UE1 和 UE2 注册到备 UPF
  主 UPF 恢复后，UE1/UE2 依然在备 UPF 激活

  UE2 → UE1 发包：
    UPF2 收到包 → 开启冗余上行隧道转发 → GRE 隧道 → UPF1
    UPF1 判断包从隧道来 → 为防重入隧道 → 丢弃数据包
    结果：UE 无法互通

解决机制：MASTERFLAG=ENABLE（主 UPF 上 ADD POOL 配置）
  UPF1 丢包前判断 MASTERFLAG=ENABLE
    → 将包重入 GRE 隧道 → 发送给 UPF2
    → UPF2 基于 UE1 信息发包到 UE1
    → 完成 UE 互访
```

**关键参数**：MASTERFLAG 仅在主用 UDG 的 ADD POOL 中配置为 ENABLE；使用前提是"使能上行转发功能 + 需具备 UE 互访能力"。

> 来源：`静态地址路由冗余叠加UE互访场景_12681278.md`（全文）

## 4. 配置规则

### 4.1 激活步骤

本特性无独立 License。激活即"配置使能"：在 GWFD-010104 静态地址池配置基础上，增加 GRE 隧道 + 重定向 IP + 冗余开关 + 路由优先级控制。

**主用 UDG 激活步骤骨架（14步）**：

```
步骤1：配置 EXTERNAL 类型地址池（REDUNDFUNC=ENABLE, HASVPN=ENABLE）
  └── ADD POOL

步骤2：配置静态地址池地址段（IPv4 + IPv6）
  ├── ADD SECTION (IPv4: V4STARTIP/V4ENDIP)
  └── ADD SECTION (IPv6: V6PREFIXSTART/V6PREFIXEND/V6PREFIXLENGTH)

步骤3：配置地址池组
  └── ADD POOLGROUP

步骤4：绑定地址池到地址池组
  └── ADD POOLBINDGROUP

步骤5：配置 APN 名称（HASVPN=ENABLE, VPNINSTANCE）
  └── ADD APN

步骤6：配置地址池组与 APN 映射
  └── ADD POOLGRPMAP

步骤7：配置冗余备份重定向 IP（IPv4 + IPv6）
  └── ADD REDUNDRDTIP  → ★ 本特性关键：虚拟 IP 将业务流重定向到 GRE Tunnel

步骤8：配置 LoopBack 接口并绑定 VPN（vpn2）
  ├── ADD INTERFACE
  ├── ADD IPBINDVPN
  └── ADD IFIPV4ADDRESS

步骤9：配置 GRE Tunnel 接口（MTU=1500）
  └── ADD INTERFACE

步骤10：配置备份 GRE Tunnel（REDUNDANCYEN=TRUE, KEEPALVEN=TRUE）
  ├── ADD GRETUNNEL       → ★ 本特性核心
  ├── ADD IPBINDVPN       → Tunnel 绑定 vpn1（与 LoopBack 的 vpn2 不同）
  ├── ADD IFIPV4ADDRESS
  ├── SET IFIPV6ENABLE
  └── ADD IFIPV6ADDRESS

步骤11：配置 IPv4/IPv6 静态路由（目的=REDUNDRDTIP，出接口=Tunnel）
  ├── ADD SRROUTE
  └── ADD SRROUTE6

步骤12（可选）：配置下行缺省低优先级静态地址用户网段路由入冗余 GRE 隧道
  ├── ADD SRROUTE   → PDN/DN 侧配高低优先级路由场景使用
  └── ADD SRROUTE6

步骤13：主备 LoopBack 路由互通 + N6 口路由发布优先级控制
  ├── ADD OSPFINTERFACE    → 打通主备 LB 路由
  └── MOD OSPFIMPORTROUTE  → 主 COST=100（高），备 COST=200（低）

步骤14：使能静态地址用户路由冗余功能（全局开关）
  └── SET REDUNDUSER SWITCH=ENABLE  → ★ 与 ADD POOL REDUNDFUNC 双使能
```

**备用 UDG 激活步骤**：重复主用 UDG 步骤 1-14，**额外增加步骤 15**：

```
步骤15：使能 IPv4/IPv6 上行报文隧道转发功能（仅备用 UDG 配置）
  └── SET APNREDUNDUPSW APN/IPV4SWITCH=ENABLE/IPV6SWITCH=ENABLE
```

> 来源：`激活静态地址用户路由冗余_19612403.md`（"操作流程/操作步骤"章节）

| LST GRETUNNEL | 查询 GRE 隧道（查 REDUNDANCYEN/KEEPALVEN） |
| LST REDUNDRDTIP | 查询冗余备份重定向 IP |
| EXP MML | 导出 MML 配置文件（故障收集） |

### 4.3 关键参数说明

#### 4.3.1 ADD POOL 关键参数（★ 本特性冗余开关 + 主用标记）

| 参数 | 取值 | 说明 |
|------|------|------|
| POOLNAME | 字符串（如 pool1 主用, pool3 备用） | 地址池名称 |
| POOLTYPE | **EXTERNAL** | 静态地址池类型固定 EXTERNAL |
| IPVERSION | IPV4 / IPV6 | IP 地址类型 |
| HASVPN | **ENABLE** | 是否绑定 VPN（本特性要求 ENABLE） |
| VPNINSTANCE | 字符串（如 vpn1） | VPN 实例名，需与 APN/Tunnel 绑定的 VPN 一致 |
| **REDUNDFUNC** | **ENABLE** | **★ 本特性核心：冗余备份功能开关，与全局 SET REDUNDUSER 双使能** |
| **MASTERFLAG** | 主用 UDG: **ENABLE**<br>备用 UDG: DISABLE | **★ 主用标记**：使能上行转发 + 需 UE 互访能力时，主 UPF 配 ENABLE 防数据包重入隧道丢包 |

#### 4.3.2 ADD GRETUNNEL 关键参数（★ 本特性核心 GRE 隧道）

| 参数 | 取值 | 说明 |
|------|------|------|
| TNLNAME | 字符串（主:Tunnel1, 备:Tunnel2） | 隧道名称，Tunnel 接口编号仅本地意义，两端可不同 |
| TNLTYPE | **gre** | 隧道类型固定 gre |
| SRCTYPE | **if_name** | IPv4 源类型为接口 |
| SRCIFNAME | LoopBack1（主）/ LoopBack2（备） | 源接口名，本端 LoopBack |
| DSTIPADDR | 对端 LoopBack IP（主→10.10.0.1, 备→10.11.1.1） | 目的 IPv4 地址，对端 LoopBack IP |
| DSTVPNNAME | vpn2 | 目的 VPN 名称（与 LoopBack 绑定的 vpn2 一致） |
| **REDUNDANCYEN** | **TRUE** | **★ 使能 GRE 隧道备份静态地址用户流量功能** |
| KEEPALVEN | **TRUE** | 使能 Keepalive，源端周期发送，对端不可达时本端 Tunnel 置 Down |

#### 4.3.3 ADD REDUNDRDTIP 关键参数（★ 重定向 IP）

| 参数 | 取值 | 说明 |
|------|------|------|
| IPVERSION | IPv4 / IPv6 | IP 版本 |
| IPV4ADDRESS | 192.168.0.1（主备可相同或不同） | 虚拟重定向 IP，仅内部重定向转发不对外发布，只要不与其他路由冲突 |
| IPV6ADDRESS | fc00:3::1（主备可相同或不同） | IPv6 虚拟重定向 IP |

#### 4.3.4 SET REDUNDUSER / SET APNREDUNDUPSW 关键参数（★ 双开关）

| 命令 | 参数 | 取值 | 说明 |
|------|------|------|------|
| SET REDUNDUSER | SWITCH | **ENABLE** | **★ 全局冗余开关**：与 ADD POOL REDUNDFUNC 双使能才生效 |
| SET APNREDUNDUPSW | APN | 字符串（如 test） | APN 名称（★ 仅备用 UDG 配置） |
| SET APNREDUNDUPSW | IPV4SWITCH | ENABLE | APN 支持 IPv4 上行转发功能 |
| SET APNREDUNDUPSW | IPV6SWITCH | ENABLE | APN 支持 IPv6 上行转发功能 |

#### 4.3.5 路由优先级参数（N6 口主备优先级）

| 命令 | 参数 | 主用 UDG | 备用 UDG | 说明 |
|------|------|---------|---------|------|
| MOD OSPFIMPORTROUTE | COST | **100**（低开销=高优先级） | **200**（高开销=低优先级） | OSPF 方式控制 N6 口路由发布优先级 |
| MOD IMPORTROUTE | MED | 低 MED | 高 MED | BGP 方式控制优先级 |

### 4.4 约束条件

| 约束类型 | 约束内容 |
|---------|---------|
| 双开关使能 | SET REDUNDUSER（全局）与 ADD POOL REDUNDFUNC（地址池）必须同时 ENABLE |
| 静态地址池必配段 | 配置 POOL 后必须配置 SECTION，否则用户激活失败；一个静态地址池最多 64 个地址段 |
| 主备地址段一致 | 主备 UDG 要规划**相同的地址段**，且需包含 SMF 上配置的静态地址段 |
| VPN 三者一致 | 地址池绑定的 VPN、APN 绑定的 VPN、Tunnel 接口绑定的 VPN 三者必须一致 |
| LoopBack/Tunnel VPN 划分 | LoopBack 接口绑定 vpn2，Tunnel 接口绑定 vpn1，**不同 VPN 完成物理与业务划分** |
| 重定向 IP 唯一性 | REDUNDRDTIP 仅内部重定向转发不对外发布，保证不与其他路由冲突；主备可相同或不同 |
| 主备 LoopBack 路由互通 | 必须通过 OSPF/BGP/静态路由打通主备 LoopBack 接口路由（GRE 隧道前提） |
| N6 口路由优先级 | 主 UPF 优先级高于备 UPF（OSPF COST 主=100，备=200；或 BGP MED；或 PDN/DN 侧配） |
| 开关前后无用户 | 特性开启和关闭前都要确保 APN 下无使用静态路由冗余功能的用户 |
| 上行转发仅备UDG | SET APNREDUNDUPSW 仅在备用 UDG 配置 |
| 互斥 NAT | 使能 NAT 功能后，UDG 无需再使用静态路由冗余功能 |
| 互斥 GWFD-010108 | 静态地址路由冗余功能正常时，静态地址用户不能做用户面地址自动探测 |
| 依赖 GRE 特性 | 静态地址路由冗余需要配置 GRE 隧道，依赖 IPFD-015002 GRE |
| C-U 协同 | 本特性与 UNC 配合使用生效（WSFD-107021） |

> 来源：`激活静态地址用户路由冗余_19612403.md`（"必备事项/操作步骤"章节）、`GWFD-010107 静态地址用户路由冗余特性概述_19612410.md`（"与其他特性的交互"章节）

## 5. 配置案例

### 5.1 典型场景：主备 UDG 静态地址池 + GRE 隧道冗余保护

**场景描述**：网络同时部署主用备用两台 UDG。当主用 UDG 故障或主用 UDG 通信链路故障时，通过备份 tunnel 将静态地址用户的下行数据备份到备用 UDG 上。UDG 上规划 IPv4 和 IPv6 两个静态地址池。

**主备 UDG 配置数据规划**：

| 类别 | 主 UDG 取值 | 备 UDG 取值 |
|------|-----------|-----------|
| 地址池名称（IPv4/IPv6） | pool1 / pool2 | pool3 / pool4 |
| 地址段（IPv4） | 10.1.1.0~10.1.1.225 | 10.1.1.0~10.1.1.225（相同） |
| 地址段（IPv6） | fc00:1::1~fc00:1::1111/64 | fc00:1::1~fc00:1::1111/64（相同） |
| 地址池组 | poolgroup1 | poolgroup2 |
| APN | test | test（相同） |
| 重定向 IP（IPv4/IPv6） | 192.168.0.1 / fc00:3::1 | 192.168.0.1 / fc00:3::1（可相同） |
| LoopBack 接口 | LoopBack1 (10.11.1.1) | LoopBack2 (10.10.0.1) |
| GRE Tunnel | Tunnel1 → 对端 10.10.0.1 | Tunnel2 → 对端 10.11.1.1 |
| OSPF COST | 100（高优先级） | 200（低优先级） |
| MASTERFLAG | ENABLE | DISABLE |
| 上行隧道转发 | 不配置 | SET APNREDUNDUPSW ENABLE |

**MML 命令序列（原样保留）**：

**主用 UDG 配置**：

```
// 配置地址池信息
ADD POOL: POOLNAME="pool1", POOLTYPE=EXTERNAL, IPVERSION=IPV4, HASVPN=ENABLE, VPNINSTANCE="vpn1", REDUNDFUNC=ENABLE;
ADD POOL: POOLNAME="pool2", POOLTYPE=EXTERNAL, IPVERSION=IPV6, HASVPN=ENABLE, VPNINSTANCE="vpn1", REDUNDFUNC=ENABLE;
ADD SECTION: POOLNAME="pool1", SECTIONNUM=1, IPVERSION=IPV4, V4STARTIP="10.1.1.0", V4ENDIP="10.1.1.225";
ADD SECTION: POOLNAME="pool2", SECTIONNUM=2, IPVERSION=IPV6, V6PREFIXSTART="fc00:1::1", V6PREFIXEND="fc00:1::1111", V6PREFIXLENGTH=64;
ADD POOLGROUP: POOLGRPNAME="poolgroup1", IPV4ALLOCPRIALG=ENABLE;
ADD POOLBINDGROUP:POOLGROUPNAME="poolgroup1", POOLNAME="pool1";
ADD POOLBINDGROUP:POOLGROUPNAME="poolgroup1", POOLNAME="pool2";
ADD APN: APN="test", HASVPN=ENABLE, VPNINSTANCE="vpn1";
ADD POOLGRPMAP: MAPPINGNAME="mapping1", APN="test", POOLGROUPNAME="poolgroup1";

// 配置冗余备份重定向IP
ADD REDUNDRDTIP: IPVERSION=IPv4, IPV4ADDRESS="192.168.0.1";
ADD REDUNDRDTIP: IPVERSION=IPv6, IPV6ADDRESS="fc00:3::1";

// 配置loopBack接口
ADD INTERFACE: IFNAME="LoopBack1", IFADMINSTATUS=up;
ADD IPBINDVPN: IFNAME="LoopBack1", VRFNAME="vpn2";
ADD IFIPV4ADDRESS: IFNAME="LoopBack1", IFIPADDR="10.11.1.1", SUBNETMASK="255.255.255.255";

// 配置GRE Tunnel
ADD INTERFACE: IFNAME="Tunnel1", IFMTU=1500;
ADD GRETUNNEL: TNLNAME="Tunnel1", SRCTYPE=if_name, DSTIPADDR="10.10.0.1", TNLTYPE=gre, SRCIFNAME="LoopBack1", DSTVPNNAME="vpn2", REDUNDANCYEN=TRUE, KEEPALVEN=TRUE;
ADD IPBINDVPN: IFNAME="Tunnel1", VRFNAME="vpn1";
ADD IFIPV4ADDRESS: IFNAME="Tunnel1", IFIPADDR="10.11.1.3", SUBNETMASK="255.255.255.0";
SET IFIPV6ENABLE: IFNAME="Tunnel1", ENABLEFLAG=TRUE, MTU=1500;
ADD IFIPV6ADDRESS: IFNAME="Tunnel1", IPV6ADDRESS="fc00:2::1", PREFIXLEN=64;

// 配置IPv4/IPv6静态路由
ADD SRROUTE: AFTYPE=ipv4unicast, PREFIX="192.168.0.1", MASKLENGTH=32, VRFNAME="vpn1", IFNAME="Tunnel1";
ADD SRROUTE6: AFTYPE=ipv6unicast, PREFIX="fc00:3::1", MASKLENGTH=128, VRFNAME="vpn1", IFNAME="Tunnel1"; 

// （可选）PGW-U/UPF与PDN/DN之间配置GRE隧道，并且由PDN/DN侧服务器端配置高低优先级路由。此场景下应在主UDG配置下行缺省低优先级静态地址用户网段路由入冗余GRE隧道。
ADD SRROUTE: AFTYPE=ipv4unicast, PREFIX="10.1.1.0", MASKLENGTH=24, VRFNAME="vpn1", IFNAME="Tunnel1", BFDENABLE=FALSE, DHCPENABLE=FALSE;
ADD SRROUTE6: AFTYPE=ipv6unicast, PREFIX="fc00:1::1", MASKLENGTH=64, VRFNAME="vpn1", IFNAME="Tunnel1", BFDENABLE=FALSE, DHCPENABLE=FALSE;

// 主备LoopBack接口的路由打通（OSPF方式）
ADD OSPFINTERFACE: PROCID=6, AREAID="0.0.0.5", IFNAME="LoopBack1";

// N6口路由发布优先级：主UPF优先级高（COST=100）
MOD OSPFIMPORTROUTE: PROCID=600, TOPOID=0, PROTOCOL=wlr, IMPTCOSTCFG=TRUE, COST=100;

// 使能静态地址用户路由冗余功能
SET REDUNDUSER: SWITCH=ENABLE;
```

## 6. 验证与调测

### 6.1 验证方法（主备切换 + GRE 隧道调测）

#### 6.1.1 调测前提与目的

当运营商需要部署静态地址用户路由冗余功能时，需对 UDG 的静态地址用户路由冗余功能进行调测，确保本功能可以正常使用。当主用 UDG 故障时，通过主备两台 UDG 间建立的备份 GRE Tunnel，备用 UDG 及时承担起主用 UDG 的网络功能。

**注意**：本章节需要手工构造主用 UDG 故障场景，因此**建议在空载场景下进行特性调测**，避免影响正常的在线用户业务。

> 适用：PGW-U、UPF

#### 6.1.2 调测数据准备

| 类别 | 参数名称 | 取值样例 | 获取方法 | 说明 |
|------|---------|---------|---------|------|
| 用户跟踪任务 | 用户IMSI号（IMSI） | 460000123456789 | 测试终端自带 | 终端地址是静态地址 |
| 测试终端使用的APN | APN名称（APN） | apn1 | 已配置数据中获取 | - |

工具：测试终端、OM Portal、U2020/MAE

#### 6.1.3 调测执行步骤（★ 含主备切换 + GRE 隧道 + 性能指标验证）

**步骤1**：对于主备 UDG，在 OM Portal 上创建 UDG 用户跟踪任务。

**步骤2**：测试终端使用 "apn1" APN 接入网络，查看**主用 UDG** 的 OM Portal 用户跟踪任务结果。
- 如果测试终端成功接入网络 → 执行步骤3
- 如果测试终端无法接入网络 → 调测 UDG 的接入功能

**步骤3**：测试终端访问 **www.huawei.com** 网页。
- 如果可以正常访问网页 → 执行步骤4
- 如果无法正常访问网页 → 调测 UDG 的 Web 浏览功能

**步骤4**：★ **手工构造主用 UDG 故障**（例如：复位主用 UDG）。测试终端重新接入网络，查看**备用 UDG** 的 OM Portal 用户跟踪任务结果。
- 如果测试终端成功接入网络 → 执行步骤5
- 如果测试终端无法接入网络 → 调测 UDG 的接入功能

**步骤5**：测试终端访问 **www.huawei.com** 网页。
- 如果可以正常访问网页 → 执行步骤6
- 如果无法正常访问网页 → 调测 UDG 的 Web 浏览功能

**步骤6**：★ **主用 UDG 故障恢复后**，在 U2020/MAE 上创建如下性能指标：
- **主 UDG 指标**（4条发送方向）：
  - 1914307630 用户平面静态备份用户发送千字节数
  - 1914307631 用户平面静态备份用户发送包数
  - 1914307632 用户平面静态备份用户发送峰值千字节速率
  - 1914307633 用户平面静态备份用户发送峰值包速率
- **备 UDG 指标**（5条接收方向 + 会话数）：
  - 1914307626 用户平面静态备份用户接收千字节数
  - 1914307627 用户平面静态备份用户接收包数
  - 1914307628 用户平面静态备份用户接收峰值千字节速率
  - 1914307629 用户平面静态备份用户接收峰值包速率
  - 1914307822 用户平面当前在线静态备份会话数
- 如果主备 UDG 上成功收集到静态备份功能相关的性能指标 → 调测结束
- 如果不能成功收集 → 执行步骤7

**步骤7**：执行 **DSP TUNNELINFO** 查看 GRE 隧道状态是否正常。
- 隧道状态 UP → 执行步骤12
- 隧道状态 DOWN → 执行步骤8

**步骤8**：执行 **LST REDUNDUSER** 查看静态地址用户路由冗余功能是否使能。
- 使能 → 执行步骤9
- 不使能 → 参考激活文档重新配置

**步骤9**：执行 **LST POOL** 查看 UDG 上配置的地址池数据是否与规划数据一致（查 REDUNDFUNC 冗余备份功能开关）。
- 一致 → 执行步骤10
- 不一致 → 重新配置

**步骤10**：执行 **LST GRETUNNEL** 查看 UDG 上配置 Tunnel 的隧道数据是否与规划数据一致（查 REDUNDANCYEN=TRUE）。
- 一致 → 执行步骤11
- 不一致 → 重新配置

**步骤11**：执行 **LST REDUNDRDTIP** 查看冗余备份重定向 IP 地址是否与规划数据一致。
- 一致 → 执行步骤12
- 不一致 → 重新配置

**步骤12**：收集信息并寻求技术支持。
- a. 在 OM Portal 上创建用户跟踪任务，执行步骤2和步骤4并保存主备 UDG 对应的报文
- b. 执行 **EXP MML** 命令将当前配置数据导出为 MML 脚本文件并保存

### 6.5 常见问题与排查

| 问题现象 | 可能原因 | 排查方法 |
|---------|---------|---------|
| 主备冗余不生效 | SET REDUNDUSER 全局开关或 ADD POOL REDUNDFUNC 未双使能 | LST REDUNDUSER 查全局开关；LST POOL 查 REDUNDFUNC |
| GRE 隧道 DOWN | 主备 LoopBack 路由未打通 | DSP TUNNELINFO 查隧道状态；检查 ADD OSPFINTERFACE 配置 |
| GRE 隧道 DOWN | Keepalive 检测对端不可达 | 检查对端 UDG 状态；检查 LoopBack 接口 IP/VPN 绑定 |
| 主用故障后用户未切到备用 | 下行路由未正确撤销 | 检查 OSPF/BGP 路由发布；核查主用 UDG 是否故障隔离 |
| 主用恢复后老用户业务异常 | SET APNREDUNDUPSW 未在备用 UDG 配置 | 检查备用 UDG 上行隧道转发开关 |
| UE 互访失败 | 主用 UDG 未配 MASTERFLAG=ENABLE | ADD POOL 主用标记 ENABLE（叠加 UE 互访场景） |
| 业务重定向失效 | 开启上行隧道转发后业务重定向功能失效 | 应用限制已知，无法规避（避免同时使用） |
| 用户激活失败 | POOL 配置但未配 SECTION | 补配 ADD SECTION |
| 用户激活失败 | 主备地址段不一致或未包含 SMF 静态段 | 核对主备 SECTION 一致，且 ⊇ SMF 静态段 |
| VPN 绑定不一致 | POOL/APN/Tunnel 三者 VPN 不一致 | 核对 VPNINSTANCE 三者一致 |
| 上下行报文经不同防火墙丢包 | 跨 DC 部署，未开启上行隧道转发 | 备用 UDG 配 SET APNREDUNDUPSW 使上下行同路径 |
| 与 NAT 冲突 | 同时使能 NAT 与路由冗余 | 二选一（互斥） |
| 与 GWFD-010108 冲突 | 同时使能地址自动检测 | 静态地址用户不做地址自动探测（互斥） |
