## 2. 核心知识点

### KP-01: UDG 双模部署形态与UPF角色分类
- **内容**：UDG 支持 5G SA 独立部署（UDG=UPF）与 2/3/4/5G 互操作融合部署（UDG=UPF+PGW-U+SGW-U，SGW-U 可选）。一个 PDU 会话数据面路径中可插入多个 UPF，UPF 角色按功能分类：

| UPF类型 | 全称 | 定义 | 出现场景 |
|---------|------|------|----------|
| **PSA UPF** | PDU Session Anchor | 通过 N6 接口与 DN 连接的会话锚点UPF | 4G/5G互操作时 PGW-U/SGW-U 仅能与 PSA UPF 共建 |
| **ULCL UPF** | Uplink Classifier | 上行分类器，将本地业务转发到本地 PSA UPF | IPv4/IPv6 PDU 会话 |
| **BP UPF** | Branching Point | 分叉点，IPv6 多宿主 PDU 会话数据转发到不同 PSA UPF | IPv6 multi-homed PDU 会话 |
| **I-UPF** | Intermediate UPF | (R)AN 与 PSA UPF 之间的所有中间 UPF（含 ULCL/BP） | 家泛称 |

- **来源**：逻辑接口介绍_68634155.md（表1）
- **核心度**：★★★★★
- **关联特性**：无（架构基础认知，非特性）
- **关联命令**：无
- **关联配置对象**：UPF 角色（决定后续逻辑接口集）
- **对接面**：CS-5（全局前提，决定 CS-1/CS-2 接口集）

### KP-02: 5G SA 参考点结构 — UPF 控制面/用户面分离
- **内容**：5G SA 中控制面 NF 之间采用**服务化接口（SBI）**；UPF 仍使用**点对点参考点接口**，包含 N3、N4、N6、N9 四个用户面接口。控制面与 UPF 之间为 N4（点对点）。参考点是服务化架构映射到传统网元点对点接口的参考模型，并非真实接口。
  - 控制面NF间：服务化接口参考点（N7、N8 等）
  - 控制面 ↔ UPF / 无线：点对点参考点（N2、N4）
  - UPF 对外：N3（接 gNodeB）、N4（接 SMF）、N6（接 DN）、N9（接 UPF）
- **来源**：熟悉网络架构_75096814.md（图1 服务化结构、图2 参考点结构）
- **核心度**：★★★★★
- **关联特性**：无
- **关联命令**：无
- **关联配置对象**：N3/N4/N6/N9 参考点（与 CS-1控制面、CS-2用户面对应）
- **对接面**：CS-5（CS-1/CS-2 的协议层基础）

### KP-03: 2/3/4/5G 互操作融合部署四网元原生合一
- **内容**：互操作网络需支持四个网元原生融合部署（统一记法）：
  - **UDM+HSS**
  - **PCF+PCRF**
  - **SMF+PGW-C**
  - **UPF+PGW-U**
- 简化拓扑、优化转发路径，**建议 SGW-C 融合到 SMF、SGW-U 融合到 UPF**。
- **关键约束**：用户一个 PDU 会话数据路径中存在多个 UPF 时，**仅 N6 接口连接 DN 的锚点 UPF（PSA）** 才需要与 PGW-U/SGW-U 合设，其余角色 UPF 无此要求。
- **来源**：熟悉网络架构_75096814.md（图3 互操作网络架构）
- **核心度**：★★★★
- **关联特性**：无
- **关联命令**：无
- **关联配置对象**：UPF+PGW-U+SGW-U 共建形态（决定逻辑接口集，见 KP-05 表4）
- **对接面**：CS-5（部署模式决策点）

### KP-04: VNF 与外部网络互通原理（vNIC→CSLB→EOR 三段映射）
- **内容**：VNF 通过**逻辑接口**经 vNIC → 逻辑网络 → 物理网络 → 外部设备 与外部网络互通。以两个逻辑接口为例：
  - 逻辑接口1 使用 vNIC4、vNIC5 与外部互通
  - 逻辑接口2 使用 vNIC6、vNIC7 与外部互通
- 逻辑接口与 vNIC 的绑定关系是组网配置的核心。NFVI 以 FusionSphere 为例说明逻辑网络与物理网络映射。
- **来源**：组网互通原理_75096776.md（图1 配置过程示意、图2 逻辑网络与物理网络映射）
- **核心度**：★★★★
- **关联特性**：无
- **关联命令**：无
- **关联配置对象**：逻辑接口 ↔ vNIC ↔ VPN
- **对接面**：CS-5（CS-4 路由对接的物理基础）

### KP-05: IP路由组网方案（IPv4/IPv6 × 路由协议）
- **内容**：VNF 通过业务逻辑接口对接组网结构：
  - 两台 EOR 配置为 **M-LAG 或 ETrunk**，呈现为一台逻辑设备；EOR 用 4 个 40GE 接口聚合成一个**负荷分担 Trunk 接口**与交换机连接，配置 IP 和允许通过的 VLAN
  - 两个 **CSLB** 上为不同逻辑接口配置外联口并对应到 vNIC 口
  - VNF 上配置逻辑接口并**绑定 VPN**，通过 VPN 与外联口 vNIC 关联
  - PE/DC-GW/EOR 与业务逻辑接口间通过**等价路由 + BFD 检测**保证可靠性

- 常用组网方案（按 IP 版本与路由协议组合）：

| IPv4 | IPv6 |
|------|------|
| OSPF动态路由+BFD | OSPFv3动态路由+IPv6 BFD |
| 静态路由+BFD | IPv6静态路由+BFD |
| BGP over OSPF/静态路由+BFD | BGP over OSPFv3/静态路由+BFD |

- **来源**：组网互通原理_75096776.md（图3 组网结构）
- **核心度**：★★★★
- **关联特性**：无（路由协议能力分散在 IPFD-010001 等）
- **关联命令**：无
- **关联配置对象**：VPN、逻辑接口、外联口、BFD
- **对接面**：CS-4（路由对接）

### KP-06: ★3GPP接口 → UDG抽象接口映射表（CS-1/CS-2对接面基础）
- **内容**：UDG 将 4G 和 5G 相同功能的逻辑接口融合为 5 组抽象接口，减少 IP 地址资源占用。**这是所有对接面的接口识别基础**。

| 抽象接口 | 全称 | 包含的3GPP逻辑接口 | 对接网元 | 支持独立配置 | 强制约束 |
|----------|------|---------------------|----------|--------------|----------|
| **Saif** | S-GW/UPF Access Interface | S1-U、N3、N3mb | SGW-U<->eNodeB / UPF<->gNodeB | 是 | S1-U/N3/N3mb 可独立配置或合一，**二选一**；4G/5G 隔离组网推荐独立配置，融合推荐 Saif 合一 |
| **Scif** | S-GW/UPF Core Interface | S5/S8-S、N9c | SGW-U<->SGW-U/PGW-U / UPF<->PSA/ULCL/BP/I-UPF | 是 | 同上二选一；隔离组网推荐独立配置 |
| **Paif** | P-GW/UPF Access Interface | S5/S8-P、N9a、S2b、Gn/Gp-U | PGW-U<->SGW-U/ePDG/SGSN / UPF<->ULCL/BP/I-UPF | **否** | 4G/5G互操作时 S5/S8-P 与 N9a 地址必须相同，**只能 Pa 合一配置**，不能单独配置 |
| **N4if** | N4 Interface | Sxa、Sxb、N4 | SGW-U<->SGW-C / PGW-U<->PGW-C / UPF<->SMF | **否** | 所有控制面接口统一抽象为 N4if，不能单独配置 Sxa/Sxb |
| **PDN/DN外联口** | - | Gi、SGi、N6 | PGW-U/PSA UPF<->PDN/DN | 否 | 统一使用**外联口物理接口**实现，**无需配置逻辑接口** |

- **3GPP 接口功能速查**（按 NF 角色）：

| NF角色 | 接口 | 对接网元 | 作用 |
|--------|------|----------|------|
| UPF | N3 | gNodeB | 上下行用户面数据 |
| UPF | N4 | SMF | 控制面+数据面消息，规则下发/RADIUS/隧道建立；遵循 3GPP R15 |
| UPF | N9c | 右侧UPF(PSA/ULCL/BP/I-UPF) | 与右侧UPF互通 |
| UPF | N9a | 左侧UPF(ULCL/BP/I-UPF) | 与左侧UPF互通 |
| UPF | N6 | DN | 基于IP/路由与DN通信（外联口物理实现，无需配置逻辑接口） |
| PGW-U | S5/S8-P | SGW-U | PGW侧上下行用户数据，GTP协议（S5=同PLMN，S8=跨PLMN） |
| PGW-U | SGi | PDN | 与PDN通信（外联口物理实现） |
| PGW-U | Sxb | PGW-C | PGW-C与PGW-U规则下发/隧道建立 |
| PGW-U | S2b | ePDG | Untrusted non-3GPP接入用户面 |
| PGW-U | Gn-U / Gp-U | SGSN | Indirect Tunnel模式下用户面（Gn=同PLMN，Gp=跨PLMN） |
| PGW-U | Gi | PDN | 与PDN通信（外联口物理实现） |
| SGW-U | S5/S8-S | SGW-U/PGW-U | SGW侧上下行用户数据，GTP协议 |
| SGW-U | S1-U | eNodeB | 上下行用户面数据 |
| SGW-U | Sxa | SGW-C | SGW-C与SGW-U规则下发 |
| SGW-U | S11-U | MME | NB-IoT 物联网业务数据流 |

- **来源**：逻辑接口介绍_68634155.md（表2、表3）
- **核心度**：★★★★★
- **关联特性**：GWFD-010234 Single IP（逻辑接口命名规则详见此特性）
- **关联命令**：无（接口配置命令见 IPFD-010001 等）
- **关联配置对象**：Saif / Scif / Paif / N4if / PDN-DN外联口
- **对接面**：CS-5 → CS-1(N4if 控制面)、CS-2(Saif/Scif/Paif 用户面)、CS-4(PDN/DN外联口路由)

### KP-07: UDG 接口命名规范与 N4 必备性原则
- **内容**：逻辑接口命名规则为"逻辑接口类型+接口编号"，部署形态决定可配置的独立接口名。

**部署形态 × 接口 × 配置名速查表**（UPF+SGW-U+PGW-U 共建 / UPF+PGW-U 共建 / UPF独立 三种形态）：

| 接口 | 独立接口名 | 抽象接口名 | 备注 |
|------|------------|------------|------|
| S1-U | S1-uif | Saif | - |
| S5/S8-S | S5-sif | Scif | - |
| N3 | N3if | Saif | - |
| N9c | N9cif | Scif | - |
| S11-U | S11-uif | - | 无抽象接口 |
| S5/S8-P、S2b、N9a、Sxa、Sxb、N4、Gn-U、Gp-U | -（不支持独立配置） | Paif / N4if | 仅抽象接口 |
| SGi、N6、Gi | -（外联口物理实现，无需配置） | - | - |

- **★N4 必备性原则**：UPF 与周边网元对接时，**N4 接口是唯一必须的接口**，其余接口按 UPF 功能与位置确认：
  - 接入侧接 (R)AN → 配 N3；接 UPF → 配 N9a
  - 核心侧接 DN → 配 N6；接 UPF → 配 N9c
  - 示例：本地数据分流场景 ULCL UPF + 本地 PSA UPF 共建，需配 N3、N4、N9c、N9a、N6 五接口

- **来源**：逻辑接口介绍_68634155.md（表4、表5、N4必备说明）
- **核心度**：★★★★★
- **关联特性**：GWFD-010234 Single IP、IPFD-010001 接口管理特性概述
- **关联命令**：无
- **关联配置对象**：N3if / N9cif / N4if / Saif / Scif / Paif
- **对接面**：CS-5（接口配置是 CS-1/CS-2 前置）

### KP-10: License 加载操作流程（OM Portal 6步 + U2020/MAE 4步）
- **内容**：

**OM Portal 路径（网元License）**：
1. 路径：应用管理 > 应用License > **License文件管理**
2. 上传：选应用名 → 上传License文件 → 浏览选文件 → 打开 → 确定
3. 查看列表：文件名/ESN/是否激活/产品名/产品版本/上传时间
4. 激活：路径切到 应用管理 > 应用License > **License管理** → 选应用名 → 激活/更新 → 选License文件 → 激活
5. 警告弹窗处理（颜色编码）：
   - **红色**：资源项值变小且<使用值 / 功能项关闭或缺失 / 资源项缺失且需使用 / 已过期 → 必须勾选"已了解并接受上述风险"高危确认
   - **黄色**：控制项值变小但≥使用值 / 控制项未使用但缺失 / 当前使用值为空 / 不在默认配置文件中
   - **白色**：授权截止日期变短但未过期
6. 告警检查：OM Portal 首页 > 监控分析 > 告警管理 > 活动告警，查 License 相关告警

**U2020/MAE 路径**：
1. 路径：许可证 > 网元许可证管理 > Cloud NE（网元）/ Service Fabric
2. 上传：左侧网元树选网元 → 上传到网元 → 选文件上传
3. 激活：点 License "操作"按钮 → 系统执行激活命令；首次激活直接确认，更新激活需核对警告对话框
4. 查验：网元ESN 与 许可证ESN **必须一致**，否则重新申请

- **来源**：上传并激活License文件 (U2020_MAE)_16360948.md、上传License文件_13519616.md、激活License_13519617.md
- **核心度**：★★★★
- **关联特性**：无
- **关联命令**：**DSP RVKLICINFO**（MML，查询失效码；在 应用管理 > MML 执行）
- **关联配置对象**：License 文件、控制项、ESN
- **对接面**：CS-5

### KP-11: License 生命周期扩展操作
- **内容**：

| 操作 | 触发场景 | 关键约束 |
|------|----------|----------|
| **启动紧急** | 火山地震等特殊场景 | 开启产品最大配置；**每个大版本仅3次，每次7天**（大版本如 20.X.X，20.1.0与20.2.0同大版本） |
| **更新** | 当前License无法满足需求 | 上传并激活新文件即可（U2020/MAE无独立更新入口） |
| **失效** | 调整容量/变更ESN或客户名 | 失效后进入宽限状态；通过 DSP RVKLICINFO 获取失效码 → 由技术支持申请新License |
| **比较** | 文件与已激活License对比 | - |
| **导出控制项** | 导出使用中License控制项 | .csv 格式 |
| **同步**（U2020/MAE） | 网元License信息同步到网管 | 可选多网元批量同步 |
| **删除**（U2020/MAE） | 删除License | **必须先去激活** |

## 3. 关键发现

### 跨文档隐性规则

1. **N4 是绝对必备接口**（KP-07）：UPF 对接时 N4 是唯一必配接口，其余按 UPF 角色（PSA/ULCL/BP/I-UPF）和位置动态确认。这是 CS-1 控制面对接的硬约束。

2. **Paif 强制合一、N4if 强制合一**（KP-06）：因 4G/5G互操作时 S5/S8-P 与 N9a 地址必须相同，Paif 抽象接口**不支持独立配置**；Sxa/Sxb/N4 统一抽象为 N4if，也**不支持独立配置**。这与 Saif/Scif "独立或合一二选一"形成对比。

3. **N6/SGi/Gi 不配逻辑接口**（KP-06）：与 DN/PDN 互通统一用**外联口物理接口**实现，无需配置逻辑接口。这影响 CS-4 路由对接的接口规划——N6 不是逻辑接口而是物理外联口。

4. **抽象接口 vs 独立接口的组网偏好**（KP-06）：4G/5G 隔离组网推荐独立配置（S1-uif/N3if、S5-sif/N9cif），融合组网推荐抽象接口合一（Saif/Scif）。Saif 合一时无线侧 NG-U 与 S1-U 接口IP也合一的场景，要求无线侧配在同一用户面 EPGROUP——**需先与无线侧工程师确认部署方案**。

5. **PDU 多 UPF 场景的接口组合规则**（KP-01、KP-07）：同一物理 UDG 因 ULCL + 本地 PSA 共建，需配 N3+N4+N9c+N9a+N6 五接口；拆分部署时 ULCL UPF 配 N3+N4+N9c，PSA UPF 配 N4+N9a+N6。**位置决定接口集**。

6. **Saif 合一的跨域协同要求**（KP-06）：UDG 用 Saif 抽象接口（N3与S1-U IP合一）时，若无线侧 NG-U 与 S1-U IP 也合一，无线侧要求两者配在**同一个用户面 EPGROUP**。这是跨设备协同约束，必须在 CS-2 用户面对接前确认。

7. **License 双路径能力不对称**（KP-09）：Service Fabric License **只能通过 U2020/MAE 加载**，OM Portal 不支持。决策时若涉及 Service Fabric，必须选 U2020/MAE 路径。

8. **License 控制项颜色告警对应业务风险等级**（KP-10）：红色=功能缺失或资源不足（高危确认），黄色=降级但可用，白色=期限缩短。更新License时必须读懂颜色编码判断是否影响业务。

9. **ESN 一致性是License生效硬门槛**（KP-10）：网元ESN 与 许可证ESN 必须一致，不一致需重新申请License。"网元ESN=NULL"表示未配置ESN对象，"未知"表示网管未收到网元返回。

10. **紧急License稀缺资源**（KP-11）：每个大版本仅3次、每次7天，是灾难恢复的最后手段，不可日常使用。
