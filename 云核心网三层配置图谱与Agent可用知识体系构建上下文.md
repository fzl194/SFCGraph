# 云核心网三层配置图谱与 Agent 可用知识体系构建上下文

> **用途说明**  
> 本文档用于给新的 Agent 或协作者快速对齐项目上下文。  
> 重点不是“为了建图谱而建图谱”，而是围绕云核心网配置生成与核查，构建一套 Agent 能够真正使用的垂直领域知识体系。

---

## 0. 一句话总目标

将云核心网领域中复杂、隐性、分散的业务知识，转化为 Agent 可检索、可理解、可推理、可调用、可生成、可核查、可持续演进的运行时知识体系。

三层图谱是该体系的结构化主干，但不是全部。系统还需要结合 RAG 证据、配置范式、规则、Action、Skill 和运行反馈。

---

## 1. 项目背景与核心问题

当前项目围绕“云核心网配置生成”展开，典型目标场景包括：

- 方案理解；
- 业务开通与变更；
- LLD 设计；
- MML 脚本生成；
- 配置语法核查、规范核查、语义核查；
- 回退方案生成；
- 现网变更 MOP 理解与复用。

项目希望解决的问题不是简单问答，而是：

```text
用户模糊业务需求 / 动网MOP / 配置指导书
  ↓
业务意图理解
  ↓
业务作业场景归一
  ↓
具体网元特性分解
  ↓
特性配置范式选择
  ↓
配置块实例化
  ↓
MML命令生成
  ↓
语法核查 / 规范核查 / 语义核查
  ↓
回退方案 / 证据追溯 / 人工修正回流
```

也就是说，目标是把“业务语义”逐层收敛为“可执行配置”。

---

## 2. 对项目目标的重新对齐

### 2.1 不是单纯构建图谱

需要避免把目标理解成：

```text
构建业务图谱、特性图谱、命令图谱
```

更准确的目标是：

```text
以三层图谱为结构化主干，将云核心网复杂业务知识转化为 Agent 可用的运行时知识体系。
```

图谱只是知识组织形态之一。复杂领域知识不能全部强行结构化，必须区分不同知识形态：

| 知识形态 | 作用 |
|---|---|
| 图谱 / 本体 | 表达对象、关系、层级、约束、可推理结构 |
| RAG 证据文本 | 保留产品文档、MOP、指导书中的原文依据 |
| 配置范式 / 模板 | 表达稳定的配置组合和生成路径 |
| 规则 / 核查项 | 表达业务规则、特性规则、命令规则 |
| Action / Tool | 执行查询、生成、核查、回退、比对等动作 |
| Skill | 面向具体业务场景封装图谱、证据、规则、工具和流程 |
| 运行反馈 | 记录人工修正、生成失败、核查失败、新场景发现 |

因此，项目实质上是在构建：

```text
Agent Runtime Knowledge Layer
```

即面向 Agent 执行期的领域知识运行层。

---

## 3. 三层图谱的核心递进关系

三层图谱不是三套孤立知识库，而是从业务意图逐层下钻到可生成、可核查配置结构的路径。

核心递进链路为：

```text
开放业务意图
  → 业务作业场景
  → 具体网元特性
  → 特性配置范式
  → 配置块
  → 配置对象
  → MML命令
```

可以理解为：

| 层级 | 输入 | 输出 |
|---|---|---|
| 业务图谱 | 开放业务意图 | 收敛业务作业场景 + 网元特性分解 |
| 特性图谱 | 具体网元特性 | 配置范式 + 配置块组合 + 特性规则 |
| 命令图谱 | 配置范式 / 配置块 | 配置对象 + 命令 + 核查规则 |

---

## 4. 三层图谱重新定义

### 4.1 业务图谱

推荐定义：

> **业务图谱是业务意图—作业场景—网元特性分解图，用于将开放意图归一为收敛的业务作业场景。**

业务图谱不是业务百科，也不是直接生成配置。它负责把开放、模糊、发散的业务诉求，归一到稳定的业务作业场景，并进一步分解到具体网元特性。

业务图谱负责回答：

```text
用户想做什么？
这属于什么业务作业场景？
业务对象是什么？
作业动作是什么？
涉及哪些网元？
哪些业务条件影响分解路径？
这个业务场景最终分解到哪些具体网元特性？
依据来自哪里？
```

业务图谱的核心链路：

```text
IntentExpression
  → BusinessScenario
  → Operation / BusinessObject / BusinessCondition
  → NFFeatureDecomposition
  → FeatureGraph
```

业务图谱核心字段建议：

```text
业务意图表达
业务作业场景
作业动作
业务对象
业务条件
场景边界
网元特性分解
证据链
```

第一阶段最小字段：

```text
场景名称
场景族
作业动作
业务对象
场景定义
场景边界
业务条件
网元特性分解
证据来源
状态/版本
```

---

### 4.2 特性图谱

推荐定义：

> **特性图谱是网元特性—配置范式—规则约束沉淀图，用于将具体网元特性沉淀为可复用配置范式。**

特性图谱不是产品特性说明书。它不是简单描述“这个特性是什么”，而是沉淀“这个特性如何配置”。

特性图谱负责回答：

```text
某个具体网元特性有哪些主要配置路径？
哪些配置块组合构成这个特性？
不同业务条件下选择哪类配置范式？
配置块之间有哪些 has / bind to / 依赖关系？
该特性下有哪些规则约束？
哪些规则需要核查？
```

当前定义：

```text
说明：单网元内单特性中所有配置路径的总集
数据来源：产品手册、现网脚本
构建方式：基于配置块的频繁子图挖掘技术
```

特性可能包含多种配置范式，每类配置范式包含：

1. **配置块（点）**  
   通用配置块的部分实例化。  
   例如：APN 通用配置块包含几十个配置对象，但在某特性下只需要配置 APN 和 APN 地址族两个配置对象。

2. **配置块关系（边）**  
   配置块之间的 bind to、has、依赖关系。

3. **当前配置范式下的场景约束**  
   当前特性下的规则约束。  
   例如：基于 APN/DNN 地址分配下，命令 `SET IPALLOCRULE` 的 `FIRSTRULE` 参数必须为 `APN-1&LOCATION-0&SMF-0`。

特性图谱的构建流程：

```text
现网脚本、配置范例
  → 脚本图谱实例化
  → 基于配置块的子图拆分 + 随机游走子图生成
  → 图数据库构建
  → 频繁子图挖掘
  → 子图匹配迭代修正
  ↔ 特性图谱配置范式
```

这被称为：

> **基于配置块的频繁子图挖掘技术**

---

### 4.3 命令图谱

推荐定义：

> **命令图谱是配置对象—对象关系—配置块结构底座图，用于将命令体系抽象为可生成、可核查的配置对象网络。**

命令图谱不是 MML 命令字典，也不是命令参数表。它的核心是把命令体系抽象为配置对象、配置对象关系和配置块结构。

命令图谱负责回答：

```text
命令背后创建、修改、删除、查询的配置对象是什么？
配置对象之间有哪些 has / bind to / depends on / conflicts with 关系？
哪些对象组成一个配置块？
配置块之间有哪些交集、依赖、绑定关系？
哪些命令、参数、规则用于语法核查和规范核查？
不同版本之间有哪些差异？
```

当前定义：

```text
说明：配置对象实体和关系
数据来源：MML图谱1.0、产品文档
构建方式：继承 MML 图谱 1.0，同时基于 LLM 辅助提取本体及关系
```

配置块定义：

```text
配置块：基于 has 边连接的连通块组成的配置块。
1. 根节点可以独立配置，子节点必须在父节点存在时才能存在。
2. 满足该约束的所有配置对象组成一棵树。
```

配置块和配置块之间关系：

```text
1. 基于边的 bind to：从一个配置块的某节点 A bind to 另一个配置块的某节点 B。
2. 基于点的 bind to：基于特定命令将某两个/多个节点绑定起来，如 ADD POOLBINDGROUP。
3. 配置块之间可能存在交集，可单独作为一个更小的配置块。
```

命令图谱技术建议命名为：

> **面向配置对象的命令本体建模技术**

其目标是：

```text
以配置对象为核心，统一建模命令、参数、配置对象树、配置块模式、对象绑定关系、核查规则和版本差异。
```

## 5. 三层图谱的 PPT 级短定义

如果放在一张图中，可以使用以下定义：

```text
业务图谱：业务意图—作业场景—网元特性分解图
特性图谱：网元特性—配置范式—规则约束沉淀图
命令图谱：配置对象—对象关系—配置块结构底座图
```

三层递进关系：

```text
开放意图 → 作业场景 → 网元特性 → 配置范式 → 配置对象
```

## 6. 三项关键技术

三项关键技术建议为：

```text
1. 基于意图语义单元的业务图谱挖掘技术
2. 基于配置块的频繁子图挖掘技术
3. 面向配置对象的命令本体建模技术
```

三者分别对应：

```text
业务层：挖掘意图模式
特性层：挖掘配置范式
命令层：建模配置对象
```

## 7. 关键技术的关键能力与成效

### 7.1 基于意图语义单元的业务图谱挖掘技术

关键能力：

```text
意图归一与场景收敛：将开放业务表达归一为标准作业场景。
业务到特性分解：建立业务场景到具体网元特性的结构化路径。
```

成效：

```text
稳定业务入口：降低业务理解与方案拆解成本。
贯通生成链路：为特性选择、配置生成和业务核查提供入口。
```

---

### 7.2 基于配置块的频繁子图挖掘技术

关键能力：

```text
脚本经验结构化：将现网脚本拆解为配置块子图。
配置范式挖掘：挖掘高频配置组合与特性级规则。
```

成效：

```text
沉淀配置范式：形成可复用、可匹配的特性配置资产。
支撑特性核查：为配置块编排、语义核查和版本演进提供依据。
```

---

### 7.3 面向配置对象的命令本体建模技术

关键能力：

```text
配置对象本体建模：建模命令、参数、对象层级及语义关系。
配置块与规则融合：关联配置块模式、核查规则和版本差异。
```

成效：

```text
对象化命令体系：将命令参数表升级为配置对象关系网络。
生成核查共用底座：支撑生成、校验、修改、删除和回退。
```

## 9. 指标设计思路

不要一开始承诺“全量覆盖”或过高准确率。建议采用三档指标：

```text
保底目标：必须交付，保证项目不失控
目标值：正常推进下应该达到
挑战值：体现进取性，但不作为硬承诺
```

指标分三类：

```text
覆盖度：覆盖多少场景、特性、对象
准确性：抽取、归并、匹配是否正确
可用性：能否支撑生成、核查、复用
```

### 9.1 2026 年推荐目标值

业务图谱：

```text
覆盖 3 个高频业务场景
解析 50 条 MOP/需求样本
意图识别准确率达到 85%
业务到网元特性映射准确率达到 85%
```

特性图谱：

```text
覆盖 5 个关键特性
解析 80 条现网脚本/配置范例
沉淀 10 个特性配置范式
子图匹配命中率达到 80%
```

命令图谱：

```text
完成 150 个配置对象建模
构建 30 个配置块模式
配置对象抽取准确率达到 88%
生成命令核查通过率达到 85%
```

### 9.2 2027 年推荐方向

```text
业务图谱：扩展到 8–10 个业务场景
特性图谱：沉淀 30+ 个配置范式
命令图谱：扩展到 500+ 配置对象、100+ 配置块模式
```

## 10. 产品文档处理原则

后续如果拿到一个具体产品文档案例，不要急着判断它属于业务图谱、特性图谱还是命令图谱。

应该先拆解该文档包含哪些知识类型：

```text
定义类知识
概念类知识
约束类知识
配置步骤类知识
配置案例类知识
特性交互类知识
命令参数类知识
注意事项类知识
核查类知识
示例脚本类知识
```

然后分别转化为不同知识资产：

| 文档知识类型 | 转化目标 |
|---|---|
| 概念定义 | 术语库 / 本体概念 / RAG证据 |
| 特性说明 | 特性节点 / 特性边界 |
| 配置步骤 | 配置范式 / 作业流程 |
| 配置案例 | 配置块子图 / 特性配置范式 |
| 特性交互 | 特性关系 / 依赖约束 |
| 命令参数 | 命令图谱 / 配置对象本体 |
| 注意事项 | 规则 / 核查项 / 风险约束 |
| 核查说明 | 核查规则 / 核查 Action |
| 示例脚本 | 脚本图谱实例 / 配置块实例 |

## 11. 哪些知识应该结构化，哪些保留原文

### 11.1 适合结构化的知识

```text
对象
关系
命令参数
依赖关系
配置块
规则条件
作业动作
业务对象
网元特性分解
配置范式
```

### 11.2 适合保留为非结构化证据的知识

```text
复杂说明
背景描述
异常说明
经验性描述
长段产品解释
人类可读的注意事项
```

正确方式是：

> **结构化图谱承载可推理主干，非结构化证据承载上下文依据，Action / Skill 承载可执行能力。**

## 12. Agent 使用方式

Agent 不是直接读图谱然后生成答案，而是应通过 Skill / Tool 组合使用知识。

建议运行时链路：

```text
用户需求
  ↓
业务图谱：场景归一 + 网元特性分解
  ↓
特性图谱：配置范式选择 + 特性规则约束
  ↓
命令图谱：配置块实例化 + 命令对象生成
  ↓
核查工具：语法 / 规范 / 语义核查
  ↓
RAG证据：补充解释依据
  ↓
输出配置方案、MML脚本、核查结果、回退方案
```

Agent 可调用的能力包括：

```text
业务场景匹配
业务条件识别
网元特性分解
特性配置范式查询
配置块查询
配置对象查询
命令生成
规则核查
证据追溯
回退生成
```

## 13. Skill 的角色

Skill 是面向具体场景的知识封装形态。

例如 APN 开通 Skill 应该封装：

```text
业务场景识别说明
需要抽取的业务条件
业务图谱查询方式
相关网元特性列表
特性配置范式选择逻辑
配置块实例化逻辑
MML生成模板
核查规则调用方式
回退方案生成方式
输出格式
边界条件
异常处理
证据追溯方式
```

Skill 不是简单 Prompt，而是：

```text
图谱 + RAG证据 + Action + 规则 + 输出模板 + 边界条件
```

## 14. 运行反馈与持续演进

系统必须支持知识反馈回流，否则图谱无法持续变好。

需要回流的信息包括：

```text
用户补充了哪些业务条件
人工修正了哪些场景匹配结果
哪些特性分解不准确
哪些配置范式无法匹配
哪些命令生成失败
哪些核查规则误报/漏报
哪些场景无法归入已有模板
哪些配置块被高频调用
```

反馈应回流到：

```text
业务图谱：修正场景、业务条件、网元特性分解
特性图谱：修正配置范式、特性规则、子图匹配
命令图谱：修正配置对象、对象关系、配置块、核查规则
Skill：修正流程、提示词、调用逻辑、输出模板
```

## 15. 当前重点问题

当前需要进一步确认和推进的问题：

```text
1. 三层图谱边界是否在具体产品文档案例中依然成立。
2. 产品文档中的不同知识片段应该进入哪类知识资产。
3. 业务图谱字段是否足够支撑业务场景归一和网元特性分解。
4. 特性图谱是否能够从产品文档和现网脚本中稳定抽取配置范式。
5. 命令图谱如何从配置对象角度重新组织 MML 命令体系。
6. 哪些知识必须图谱化，哪些知识应保留为 RAG 证据。
7. 哪些能力应作为 Action / Tool 供 Agent 调用。
8. Skill 如何使用这些图谱、证据、规则和工具。
9. 运行反馈如何回流到图谱、规则和 Skill。
```

## 16. 后续拿到产品文档案例后的处理流程

当提供一个具体产品文档案例时，新的 Agent 应按以下步骤处理：

### 第一步：文档结构拆解

识别文档中的章节和知识类型：

```text
特性概述
适用网元
原理说明
配置流程
配置举例
参数说明
特性交互
注意事项
核查方法
命令说明
```

### 第二步：知识类型判定

逐段判断：

```text
这是概念定义？
这是特性边界？
这是配置范式？
这是配置块关系？
这是命令配置对象？
这是规则约束？
这是核查规则？
这是证据文本？
```

### 第三步：进入对应知识资产

```text
业务图谱：业务场景、业务对象、作业动作、网元特性分解
特性图谱：特性、配置范式、配置块组合、特性规则
命令图谱：配置对象、命令参数、has/bind to 关系、配置块
RAG证据：长段解释、背景说明、注意事项
Action/Tool：查询、生成、核查、回退等可执行能力
Skill：场景化封装
```

### 第四步：保持证据链

每个结构化知识都必须保留：

```text
来源文档
章节
原文片段
抽取置信度
审核状态
版本信息
```

### 第五步：判断是否支持 Agent 使用

最终要判断：

```text
Agent 能不能用这条知识做决策？
能不能查到依据？
能不能调用工具执行？
能不能生成配置？
能不能核查结果？
能不能在失败后回流修正？
```

---

## 17. 重要原则

后续讨论和实现必须遵守：

```text
1. 不为建图谱而建图谱，目标是 Agent 可用知识体系。
2. 图谱承载结构化主干，RAG 保留原文证据。
3. 业务图谱负责意图收敛和网元特性分解，不直接生成命令。
4. 特性图谱负责配置范式和特性规则，不做业务意图识别。
5. 命令图谱负责配置对象和命令规则，不做业务场景判断。
6. 配置块是特性图谱和命令图谱之间的关键桥梁。
7. 证据链必须保留，否则图谱不可维护、不可审核。
8. Action / Tool 是图谱能力落地的执行接口。
9. Skill 是面向具体业务场景的知识封装形态。
10. 运行反馈必须回流，用于修正场景、范式、规则和配置块。
11. 不要强行把所有知识结构化，结构化与非结构化要分工协作。
12. 每一个图谱节点和关系都要考虑 Agent 是否能真正使用。
```

## 18. 对当前方向的判断

三层图谱方向仍然成立，但需要从“图谱建设项目”升级为：

> **垂直领域知识可执行化项目。**

也就是说，当前工作不是单纯构建业务图谱、特性图谱、命令图谱，而是：

```text
以三层图谱为结构化主干，
将云核心网业务知识转化为 Agent 可用的运行时知识体系，
支撑从业务意图到 MML 生成与核查的闭环。
```

后续所有字段设计、文档抽取、图谱构建、Skill 设计、Action 设计，都必须围绕这个目标展开。
