## Slide 2: 为什么现在要调整三层图谱的组织方式

- 旧做法不是没有产出，而是产出方式天然耦合：
  - 每个场景单独构建一套三层图谱
  - 图谱直接放入 SKILL
  - 运行时由 SKILL 自带知识完成生成与核查
- 这种做法在早期适合快速打样，但随着资产增多，三个问题会同时放大：
  - **SKILL 与资产耦合**：SKILL 不只是流程封装，还承载图谱内容，升级、裁剪、保护都困难
  - **构建与维护耦合**：每新增一个场景，往往意味着重新组织一套“业务到命令”的局部闭环，复用弱
  - **静态与动态耦合**：稳定对象、稳定关系、具体配置步骤、具体参数决策混在一个场景包里，边界不清
- 在业务感知域中，这个问题已经非常明显：
  - 同一套材料里同时出现业务域/场景/方案
  - 同时出现 Feature、License、PCC/SA 等正式能力
  - 同时出现步骤级配置方法、命令级原子配置、参数条件和约束
- 结果不是“信息不够”，而是“信息没有被正确分层”：
  - 复用依赖人工记忆
  - 场景扩展要重复组织内容
  - SKILL 难做到轻量化和资产保护
  - 后续服务化接口难统一抽取子场景、子方案知识

**本页结论**

`这次调整的重点，不是重做图谱，而是把资产沉淀、配置表达和 SKILL 消费方式分开。`

**Layout role**

- `context / problem / transition`

**Visual idea**

- 左侧：旧模式闭环图
  - `场景 -> 三层图谱 -> SKILL -> 生成/核查`
  - 用 3 个红色耦合标签标明 `资产耦合 / 维护耦合 / 配置知识耦合`
- 中间：业务感知域真实内容混杂示意
  - 业务对象、特性对象、Task、命令、参数片段混排
- 右侧：转型方向
  - `静态资产独立沉淀`
  - `动态 Task 独立编排`
  - `SKILL 运行时按需取用`

## Slide 3: 三层图谱的核心结构

- 新版本的核心不是“多加一层”，而是把知识分成两类：
  - **静态知识**：系统里稳定存在什么对象、对象之间有哪些长期稳定关系
  - **动态知识**：真正配置时，这些对象应该如何按层级展开成可执行的配置方法
- 因此，图谱主体分成两部分：
  - **静态三层**
    - 业务层：`BusinessDomain / NetworkScenario / ConfigurationSolution`
    - 特性层：`Feature / License`
    - 命令层：`MMLCommand / CommandParameter / ConfigObject`
  - **动态一层**
    - 统一叫 `Task`
    - 但 `Task` 内部再按层级组织为：
      - `feature task`
      - `compound task`
      - `atom task`
- 这里的关键变化不是命名，而是职责边界：
  - 业务层负责表达“业务问题空间”和“方案选择空间”
  - 特性层负责表达“正式产品能力及其授权控制”
  - 命令层负责表达“命令、参数、配置对象的稳定事实”
  - Task 层负责表达“具体怎么配置”
- `Task` 层不是命令列表，也不是把命令重新包装一遍：
  - `feature task` 讲某个特性如何配置
  - `compound task` 讲一个可复用的步骤骨架如何组织
  - `atom task` 讲一条命令在配置语义下怎么配
- 顺序关系也不再混在场景描述里，而是自然内嵌在 Task 分层中：
  - feature task 编排 compound task
  - compound task 组织 atom task
  - atom task 最终落到具体命令与参数
- SKILL 在这套结构里的位置也要明确放出来：
  - SKILL 不承载图谱资产全文
  - SKILL 内部保留模块化执行流程
  - 图谱资产通过 API 按子场景/子方案动态提供

**本页结论**

`三层图谱负责沉淀稳定对象，Task 负责表达具体配置方法，SKILL 负责按流程调用和执行。`

**Layout role**

- `core architecture / concept explanation`

**Visual idea**

  - 上层 `业务层`
  - 中层 `特性层`
  - 下层 `命令层`
  - 右侧单独竖列 `Task层`

  - `feature task`

## Slide 4: 业务感知域里，这套结构已经长成什么样

- 这一页不讲抽象定义，直接讲当前已经落地的真实内容。
- 以 [业务感知域](../assets/business/business-awareness/BusinessDomain@business-awareness.md) 为例，业务层已经形成：
  - 一个业务域：`业务感知`
  - 三个场景：
    - `计费`
    - `带宽控制`
    - `访问限制`
  - 每个场景下面再继续拆成方案簇
- 当前业务层不是“目录树”，而是“业务诉求到方案的路由层”：
  - 计费场景已拆成 7 个方案
    - 融合计费
    - 在线计费
    - 离线计费
    - 内容计费基础
    - 计量形态增强
    - 配额降速与体验切换
    - 兜底默认计费
  - 带宽控制场景已拆成 12 个方案
  - 访问限制场景已拆成 8 个方案
- 方案下面并不是直接写命令，而是开始编排 Task：
  - 例如 `内容计费基础` 方案，会编排 UDG 与 UNC 两侧多个 `feature task`
  - 例如 `在线计费` 方案，会调用在线计费、内容计费、事件计费等 `feature task`
- 再往下，feature task 会复用已有步骤骨架：
  - `计费三件套`
  - `过滤链`
  - `规则与用户模板绑定`
  - `计费收尾`
- 再往下，步骤骨架才继续拆成命令级原子配置：
  - `配置URR`
  - `配置URRGROUP`
  - `配置PCC策略组`
  - `配置RULE`
  - `配置USERPROFILE`
- 这说明现在的真实落地已经具备了完整主链：
  - `业务域 -> 场景 -> 方案 -> feature task -> compound task -> atom task -> 命令`

**本页结论**

`业务感知域已经具备从业务诉求到方案、Task 和命令的完整组织链路。`

- `evidence-backed structure / pre-demo bridge`

  - `业务感知`
  - `计费 / 带宽控制 / 访问限制`
- 中部：选中 `计费` 场景，展开 7 个方案卡片
