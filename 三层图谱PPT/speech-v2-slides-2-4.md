## Slide 2：为什么现在要调整三层图谱的组织方式

这一页先不讲新结构，先讲为什么必须改。  
我们原来的做法，是按场景构建一套三层图谱，再把它放进 SKILL 里使用。这个方式在早期打样阶段是有效的，因为它可以很快形成某个场景的小闭环。  

但随着资产增多，这种方式的问题会越来越明显。  
第一，SKILL 和资产绑在一起，SKILL 不只是流程封装，还承载图谱内容，这样不利于升级、裁剪和资产保护。  
第二，场景和图谱构建绑在一起，每扩一个场景，往往就要重新组织一套局部闭环，复用效率不高。  
第三，静态知识和动态知识绑在一起，业务对象、特性对象、配置步骤、参数条件都混在一个场景包里，边界越来越不清楚。  

所以这次调整的重点，不是把图谱再做细一点，而是把图谱资产怎么沉淀、配置方法怎么表达、SKILL 怎么消费这三件事拆开。  

这一页的核心结论就是：  
我们现在要解决的，不是资料缺不缺，而是图谱资产、配置编排和 SKILL 消费方式之间的耦合问题。

## Slide 3：三层图谱的核心结构

这一页讲新版本的核心结构。  
新版本的关键变化，不是简单多加一个对象，也不是把旧图重画一遍，而是把知识分成两类。  

第一类是静态知识，也就是系统里稳定存在的对象和稳定关系。  
这一部分继续放在三层里：业务层、特性层、命令层。  
业务层负责表达业务域、场景、方案；特性层负责表达正式产品能力和授权控制；命令层负责表达命令、参数和配置对象这些稳定事实。  

第二类是动态知识，也就是实际配置时“具体怎么配”。  
这一部分统一收敛到 Task 层，但 Task 层内部再按粒度分成三层：feature task、compound task、atom task。  

feature task 讲的是某个特性如何配置；  
compound task 讲的是可复用的步骤骨架；  
atom task 讲的是一条命令在配置语义下到底怎么配。  

同时，SKILL 在这个结构里不再承载资产全文，而是保留流程本身，通过服务/API 在运行时按需取用某个子场景、某个方案需要的知识。  

这一页想让领导记住的只有一句：  
三层图谱负责沉淀稳定对象，Task 负责表达配置方法，SKILL 负责按流程调用和执行。

## Slide 4：业务感知域里，这套结构已经长成什么样

前面两页讲的是为什么改、怎么改，这一页直接讲现在已经做成了什么。  
我们不讲抽象 schema，而是直接看业务感知域。  

在业务层，业务感知域下面已经形成了三个场景：计费、带宽控制、访问限制。  
而且每个场景下面不是简单挂几个名词，而是已经进一步拆成方案簇。  
比如计费场景下面，已经拆成融合计费、在线计费、离线计费、内容计费基础、计量形态增强、配额降速与体验切换、兜底默认计费这 7 个方案。  

更重要的是，方案下面并不会直接落到底层命令。  
它会先编排 feature task，再复用 compound task，再下探到 atom task，最后才落到真正执行的命令。  

比如内容计费基础这个方案，往下会走到 UDG 内容计费这个 feature task，再复用计费三件套这个 compound task，再进一步下探到配置 URR 这样的 atom task，最后才关联到 ADD URR 这条命令。  

也就是说，现在的主链已经打通了：  
业务域到场景，到方案，再到 feature task、compound task、atom task，最后进入 SKILL 的执行流程。  

所以这一页的核心不是展示数量，而是说明一件事：  
业务感知域已经具备从业务诉求到方案、Task 和 SKILL 执行的完整组织链路。
