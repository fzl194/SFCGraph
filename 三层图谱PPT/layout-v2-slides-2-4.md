## Slide 2 排版：为什么现在要调整三层图谱的组织方式

### 页面标题

`为什么现在要调整三层图谱的组织方式`

### 页面副标题

`不是资料不够，而是资产沉淀、构建维护和消费方式耦合在一起。`

### 推荐版式

整页用 `三栏结构 + 底部结论条`：

- 左栏：旧模式
- 中栏：当前暴露的问题
- 右栏：新版本转向目标
- 底部：一句总结

### 左栏：旧模式

标题：

`旧模式：每个场景各做一套图谱`

内容建议画成一个竖向闭环：

`场景`
↓
`局部三层图谱`
↓
`放入 SKILL`
↓
`生成 / 核查`

在闭环旁边挂 3 个红色小标签：

- `SKILL 承载资产`
- `场景单独维护`
- `静态/动态知识混写`

### 中栏：当前暴露的问题

标题：

`问题会随着资产增多同步放大`

中间放 4 个卡片，建议上下 2×2 排列：

- `SKILL 与资产耦合`
  - SKILL 不再只是流程
  - 升级、裁剪、资产保护都困难

- `构建与维护耦合`
  - 每扩一个场景
  - 都要重新组织局部闭环

- `静态与动态耦合`
  - 业务对象、特性对象、步骤、参数混在一起
  - 边界越来越不清

- `服务化消费困难`
  - 很难统一抽取
  - “某子场景/某方案”的最小知识包

### 右栏：新版本转向目标

标题：

`新版本的转向不是重做，而是解耦`

右栏用 3 个蓝绿色箭头块：

- `图谱资产独立沉淀`
- `Task 独立表达配置方法`
- `SKILL 运行时按需取用`

在箭头块下面再放一个简洁结果区：

- `复用增强`
- `维护拆开`
- `便于保护资产`
- `便于做 API`

### 底部结论条

整页底部用深蓝结论条：

`这次调整的重点，不是重做图谱，而是把资产沉淀、配置表达和 SKILL 消费方式分开。`

## Slide 3 排版：三层图谱的核心结构

### 页面标题

`三层图谱的核心结构`

### 页面副标题

`静态三层沉淀稳定对象，动态 Task 分级表达“具体怎么配置”。`

### 推荐版式

整页用 `中心主图 + 右侧 SKILL 流程 + 底部结论`：

- 左中区域：主结构图
- 右侧：SKILL 模块流程
- 底部：一句结构性结论

### 左中主图：四块结构

主图建议画成 `三层横排 + Task 侧挂 + SKILL 外置`。

#### 主体三层

从上到下三条横带：

- `业务层`
  - `BusinessDomain`
  - `NetworkScenario`
  - `ConfigurationSolution`

- `特性层`
  - `Feature`
  - `License`

- `命令层`
  - `MMLCommand`
  - `CommandParameter`
  - `ConfigObject`

#### Task 侧挂结构

在右侧单独竖列一个橙色区块：

- `Task层`
  - `feature task`
  - `compound task`
  - `atom task`

三级之间画清晰向下关系：

`feature task -> compound task -> atom task`

并在 atom task 下用细线连到命令层，强调：

`最终落到命令与参数`

#### 右侧：SKILL 模块流程

在最右侧放一个独立灰蓝色流程区：

- `需求输入`
- `业务域 / 子场景识别`
- `图谱与 Task 加载`
- `方案确认`
- `LLD确认`
- `配置生成`
- `AI MML 核查`
- `输出目标配置`

在流程区旁边用两条小注释：

- `SKILL 保留流程，不保存资产全文`
- `图谱和 Task 通过接口按需提供`

### 底部结论条

`三层图谱负责沉淀稳定对象，Task 负责表达配置方法，SKILL 负责按流程调用和执行。`

## 统一视觉要求

- 背景：浅灰白，不要纯白刺眼
- 主色：
  - 业务层蓝色
  - 特性层青绿色
  - Task 层橙色
  - 命令层偏红橙
  - SKILL / API / 服务层用灰蓝色
- 线条：尽量水平/垂直，不做复杂弯折
- 卡片：轻阴影，边框细，不做厚重面板
- 文案：标题短、结论硬、正文不用长段堆砌
- 每页底部必须有结论区，保持与旧版 deck 的管理汇报气质一致

## 可追加页排版：图谱验证与迭代优化框架

### 页面标题

`图谱验证与迭代优化框架`

### 页面副标题

`图谱质量不只靠静态审阅，而要在配置生成任务中被测试、定位和持续修正。`

### 推荐版式

整页用 `中轴主闭环 + 底部人工反馈支路 + 右下结论区`。

### 中轴主闭环

从左到右做 5 个模块卡片：

- `Agent SOP 构建图谱`
- `SKILL 通过 API 获取图谱`
- `Test Agent 跑 input/output 测试集`
- `Debug Agent 定位图谱问题`
- `反向修改三层图谱`

模块间用箭头连接，并从最后一个模块回箭头到第一个模块，形成自动闭环。

### 各模块建议说明

- `Agent SOP 构建图谱`
  - 按既有 SOP 构建业务层、Task 层、命令层资产

- `SKILL 通过 API 获取图谱`
  - 在配置生成任务中消费图谱子集

- `Test Agent 跑 input/output 测试集`
  - 用标准测试集验证配置生成效果

- `Debug Agent 定位图谱问题`
  - 判断问题落在场景、方案、Task 还是命令层

- `反向修改三层图谱`
  - 资产修正后重新进入验证

### 底部人工反馈支路

底部单独一条横向支路：

`人工使用反馈 -> 标记问题位置 -> 图谱定向修订`

这条支路向上接入“反向修改三层图谱”模块。

### 页面视觉重点

- 自动闭环用主色蓝绿
- Debug / 问题定位可用少量橙红强调
- 人工反馈支路用灰蓝色，避免喧宾夺主
- 不做复杂算法感图形，重点是“工程闭环”

### 结论条

`三层图谱不是一次性建完的静态资产，而是在配置生成、测试验证、问题定位和人工修订中持续优化的工程资产。`
