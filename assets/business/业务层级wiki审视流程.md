# 业务层 Wiki 审视流程（R1）

> 目的：对已建业务层 wiki（BD/NS/CS）做**独立批判性审视**，补构建者自检（SOP §7）的盲区。
> 核心关切：**task 覆盖度**、**复用是否抹平差异**、**跨网元协同是否完整**。
> 适用：每完成一族（或一批）CS 后执行；构建者自检 ≠ 审视（自检问"建全了吗"，审视问"复用过度吗/覆盖真实吗/协同完整吗"）。
> 配套：`assets/business/业务层级构建SOP.md`（§3 复用/§4 模板/§6 硬约束/§7 验收）。镜像 `assets/task/特性task_wiki审视流程.md`。

---

## R1 核心审视五维度（必做）

### R1.1 task 覆盖度（防遗漏）—— 方案应编排的 task 都要引用

对每个 CS：
1. 读原始产品文档（方案/业务专题）「操作步骤 / 配置流程」**全部步骤** + 方案涉及的各 feature task 的配置流程。
2. 读 CS「配置与协同」步骤。
3. **逐步骤核对**：方案应编排的 feature/步骤/命令 task 是否都引用了？
   - ✓ 覆盖（直接引用 或 并入某 compound 且该 compound 命令集真包含所需命令）
   - ✗ 遗漏（应有而未引用）
   - ⚠ 语义偏离（引用了但参数/命令/顺序与原始不一致）
4. **方案特有额外步骤**（跨网元协同的、不属于任何特性的命令）是否都建了 atom/compound 进 `task/`（非空写在 CS 叙述里）？
5. 复用 compound 时，**必须打开该 compound 确认其命令集真的包含这一步需要的命令**——不能假设"复用了就覆盖了"。

> 典型盲区：方案原始文档有跨网元协同的额外步骤（如 UNC 侧 CHF 选择 / URRID 跨侧一致性核验命令），CS 默认并入某 compound，但该 compound 命令集未必含 → 静默遗漏。

### R1.2 复用合理性（防一刀切）—— 复用不能抹平差异

对 CS 复用的每个 compound：
1. compound 命令集是否真覆盖 CS 这步所需**全部命令**？（Jaccard ≥ 0.75 且相位同义，SOP §3）
2. CS 的**跨网元协同差异**（参数变种 / 专属命令 / 协同约束）是否进了：
   - CS 的决策点影响表（必全），**且**
   - compound 的「场景差异」小节（若该差异是该步骤的阶段级选择）？
3. 有无**假复用**？命令集重合 < 0.75 或相位不同却强行复用 → 跨网元专属命令丢失、或被塞进不相关 compound。
4. **假通用判定（critical，必查）**：逐 CS 核对——该 CS 有的核心差异维度（跨网元参数变种/专属命令/协同约束/组装方式），compound「场景差异」是否有对应条目？若 **CS 有差异而 compound 场景差异无** → compound 对该 CS 是**假通用**。**复用时必须把差异双向回填到 compound 场景差异，不能只写进 CS**（SOP §3 强制）。

> 典型盲区：多个计费 CS 都复用 1-00010 计费三件套，但融合的"双 URR + RGAPPLIED 约束"、在线/离线的"USAGERPTMODE 差异"——这些差异**只写进各 CS 决策点，1-00010 场景差异无** → 1-00010 对多 CS 是假通用。

### R1.3 跨网元协同完整性（业务层特有 ★）

对每个 CS：
1. CS 是否表达清 **UDG 用户面 + UNC 控制面**的协同（顺序 / 一致性约束 / CP-UP 参数对齐）？
2. 有无**只写一侧漏另一侧**？如计费 CS 只写 URG 侧 URR 三件套，漏 UNC 侧 SMF/CHF/CCT 对接。
3. 跨网元**一致性约束**是否记进 CS 约束段？（如 URRID/RG 跨侧一致、USERPROFILE/RULENAME 跨侧一致）
4. 跨网元**协同顺序**是否在「配置与协同」表达？（如先 UNC 侧 CCT 模板 + CHF 选择，再 UDG 侧 URR 执行，或双侧协同链）

> 典型盲区：CS 引用了 UDG + UNC feature task，但「配置与协同」只线性列 UDG 步骤 + UNC 步骤，未表达两者的协同时序与一致性约束。

### R1.4 前置门（critical）

对每个 CS：
1. CS 涉及的 feature task `2-`（UDG + UNC 两侧）是否真建完（非 `[[占位]]`）？
2. 有无引用了不存在的 feature task（断链）？
3. 前置门是**结构口径**（无占位、内容齐全）；`status: active` 是单独人审步骤，不阻塞 CS 开建，但断链占位是 critical。

### R1.5 证据链（critical，防编造）

CS「配置与协同」的命令/参数必须可追溯到 task 或 evidence：
1. 每个命令：有对应 atom（查 `_numbering.json`）或有原始文档脚本出处。
2. 每个参数：方案原始文档 / 命令 wiki / feature task / 数据规划表有该参数取值证据。
3. **无证据的参数必须移除或改为"UDG/UNC 侧不配 / 由上层决定"**，否则配置生成编造参数。
4. **方案特有额外步骤**若 activation/方案文档未直接演示，**不能默认有证据**——须明确标 evidence 出处（命令 wiki / feature task / 场景推断），脚注注"未直接演示，证据来自 XXX"。

> **★ 审视者警示：无证据结论须穷尽搜索后再判 critical**。审视发现"疑似编造参数/假约束"时，须先 grep 原始文档 / 特性 wiki / **官方差异表** / **软件参数文档（BIT/BYTE 等）** / 命令 wiki 全文，确认真无证据再判 critical（镜像特性审视 2 次反转教训）。**判 critical 前穷尽搜索；修复移除前也先核实**。

## R2 辅助维度（顺带查）

- **R2.1 命令完整性**：CS 涉及命令是否都有 atom（查 `_numbering.json`）？缺失是否标注待补？
- **R2.2 决策点影响表**：每 option 影响全记？CS 含跨网元路径维度（UDG only / UNC only / 双侧协同）？
- **R2.3 双向链接**：CS ↔ task 闭环（per-task-md"被引用于"有本 CS）；无 `[[占位]]` 残留（除明确待建）；无断链。
- **R2.4 模板合规**：BD/NS 无「配置与协同」段（SOP §4.1）；CS 有配置与协同/决策点/约束/关联；叙述式非字段填表；**front-matter 字段匹配类型**（SOP §4.2 必填对照）——BD 缺 `domain:` / NS 缺 `domain:` 或 `scenario:` / CS 缺任一字段即不通过（影响 categories 分组与左树渲染）。

## R3 输出格式

### 每 CS
```
## {方案名}（ConfigurationSolution@charging-xxx）
### R1.1 task 覆盖度：应编排 N 步 → 逐条 ✓/✗/⚠
  - 步1 ✓ | 步2 ✓ | 步3 ✗(遗漏:xxx) | 步4 ⚠(偏离:xxx)
### R1.2 复用：✓合理 / ⚠差异未记全(列哪条) / ✗假复用(列哪个compound)
### R1.3 跨网元协同：✓完整 / ⚠漏一侧(列) / ✗未表达协同
### R1.4 前置门：✓ / ✗(断链:xxx)
### R1.5 证据链：✓ / ✗(编造:xxx)
### 问题清单：[critical|warning|info] 具体问题 + 修复建议
```

### 族级总结
- **共性问题**（多 CS 重复出现的）
- **复用模式是否合理**（compound 是否被正确复用/有无假通用）
- **跨网元协同模式**（共性协同约束是否沉淀）
- **SOP/拆分方法改进建议**（沉淀回 SOP 或本流程）

---

## R4 严重度

- **critical**：前置门未过（断链占位）/ task 覆盖遗漏 / 假复用致命令丢失 / 跨网元一致性约束未记 / 编造参数 → 必须修
- **warning**：差异未进场景差异 / 决策点影响不全 / 协同一侧漏写 / 模板段错位 → 应修
- **info**：措辞/链接/格式 → 可选
