# ADD NRFNFTIMER（增加指定NF在NRF上的时长信息）

## 功能

![](增加指定NF在NRF上的时长信息（ADD NRFNFTIMER）_07649542.assets/notice_3.0-zh-cn_2.png)

服务发现缓存有效时长（DISCVALIDTIMER）参数过小时，将会导致网元频繁触发服务发现。订阅有效时长（SUBVALIDTIMER）参数过小时，将会导致网元频繁触发订阅请求。

**适用NF：NRF**

该命令用于增加指定NF的有效时长信息。当该NF来NRF服务发现后，会将服务发现结果进行缓存，并保存NRF返回的服务发现缓存有效时长，已缓存的NF信息可在该NF本地进行服务发现，等待缓存有效时长到期后，该NF才会重新来NRF服务发现。另外该NF发起订阅到NRF后，会根据NRF返回的订阅有效时长来进行续订阅，有效时长到期会重新来NRF订阅。NRF整系统通过SET NRFTIMER命令设置统一有效时长，当整系统有效时长不满足现网实际需求时，可通过该命令设置该NF服务发现缓存有效时长(秒)以及订阅的有效时长(秒)。

## 注意事项

- 该命令执行后立即生效。

- 本命令中配置的NF实例对应时长信息取本命令和SET NRFTIMER中的配置最小值。

- 最多可输入1024条记录。

## 权限

G_1，管理员级别命令组；G_2，操作员级别命令组

## 参数

| 参数标识 | 参数名称 | 参数说明 |
| --- | --- | --- |
| NFINSTANCEID | NF实例标识 | 可选必选说明：必选参数<br>参数含义：该参数用于表示NF实例标识。<br>数据来源：本端规划<br>取值范围：字符串类型，输入长度范围是1~36。<br>默认值：无<br>配置原则：无 |
| DISCVALIDTIMER | 服务发现缓存有效时长(秒) | 可选必选说明：必选参数<br>参数含义：该参数用于表示系统对服务发现结果的缓存有效时长。在缓存有效时长内，该系统中的NF实例再次服务发现相同NF实例时，可以直接使用缓存中的结果，不需要再去NRF上处理。超过缓存有效时间，服务发现结果缓存数据将被清除。系统中的NF实例再次服务发现相同NF实例时需要重新向NRF发起服务发现请求。<br>数据来源：本端规划<br>取值范围：整数类型，取值范围是0~172800。<br>默认值：无<br>配置原则：<br>- 服务发现缓存有效时长配置过小，系统中的NF实例再次服务发现相同NF实例时，可能每次都向NRF发送服务发现请求，导致系统内信令消息较多。<br>- 服务发现缓存有效时长配置过大，请求方NF不能及时感知服务提供方的NF变更。在缓存有效时长内，如果服务提供方的NF变更，请求方NF直接使用的缓存数据可能导致业务流程失败，造成信令浪费。<br>- 实际配置由运营商统一规划。 |
| SUBVALIDTIMER | 订阅有效时长(秒) | 可选必选说明：必选参数<br>参数含义：该参数用于表示NF服务订阅有效时长。在订阅有效时长内，订阅方NF订阅的内容生效且能正常收到被订阅NF的变化通知。超过此订阅有效时长，NF订阅的信息不再生效，被订阅方NF的变化将不再通知给订阅方NF。<br>数据来源：本端规划<br>取值范围：整数类型，取值范围是0~604800。<br>默认值：无<br>配置原则：<br>- 订阅有效时长配置过小，如果订阅方NF希望实时感知被订阅方NF，就需要频繁发起订阅请求，增加系统内订阅相关信令。<br>- 订阅有效时长配置过大，订阅信息将会较长时间存储在NRF上，对NRF存储订阅信息的内存要求较高。<br>- 实际配置由运营商统一规划。 |

## 操作的配置对象

- [[configobject/UNC/20.15.2/NRFNFTIMER]] · 指定NF在NRF上的时长信息（NRFNFTIMER）

## 使用实例

运营商希望将NF实例标识为"88888888-4444-1234-5678-123456789ABC"的服务发现缓存有效时长和订阅有效时长进行单独控制，分别设置为120秒和120秒，执行如下命令。

```
ADD NRFNFTIMER: NFINSTANCEID="88888888-4444-1234-5678-123456789ABC", DISCVALIDTIMER=120, SUBVALIDTIMER=120;
```
