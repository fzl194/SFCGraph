# ADD FILTER（增加过滤器）

## 功能

**适用NF：PGW-U、UPF**

该命令用于增加过滤器，其中包含三、四层的协议、源端口号、目的端口号、源IP地址和目的IP地址等。数据报文通过时，系统会根据过滤器选择到对应的规则，然后执行选中规则对应的计费策略和动作策略。

## 注意事项

- 该命令执行后需要等待执行SET REFRESHSRV命令（REFRESHTYPE参数设置为USERPROFILE或ALL）刷新后生效。
- Filter和FilterIPv6的总记录数最大为150000。当配置记录数大于规格的85%时，会上报“ALM-135602215 配置数量超阈值”告警。当配置记录数小于等于配置规格85%时，恢复“ALM-135602215 配置数量超阈值”告警。阈值可以通过MOD CFGTHRESHOLD命令修改。
- 当过滤器指定了IP列表名称配置时，系统会自动生成多个过滤器，生成的过滤器会占用系统的过滤器总规格。生成过滤器数目的计算公式为：生成过滤器数目 = 手机IP列表名称的IP数目 * 服务器IP列表名称的IP数目。
- 配置的过滤器用于上行方向报文的规则匹配时，其中的MSIP、MSIPMASK、MSSTARTPORT、MSENDPORT、SVRIP、SVRIPMASK、SVRSTARTPORT、SVRENDPORT分别对应待匹配报文的源地址、源地址反掩码、源端口号范围、目的地址、目的地址反掩码、目的端口号范围。
- 配置的过滤器用于下行方向报文的规则匹配时，其中的MSIP、MSIPMASK、MSSTARTPORT、MSENDPORT、SVRIP、SVRIPMASK、SVRSTARTPORT、SVRENDPORT分别对应待匹配报文的目的地址、目的地址反掩码、目的端口号范围和源地址、源地址反掩码、源端口号范围。
- 当过滤器用于基于APN或AclDefault的上行入口的Acl处理时，其中待匹配的报文是解GTP封装后的用户报文，一般情况下源地址和端口号是手机端的IP地址和端口号，目的地址和端口号是访问的服务器端的IP地址和端口号；当过滤器用于基于APN或AclDefault的上行出口的Acl处理时，其中待匹配的报文是出Gi接口的报文，一般情况下源地址和端口号是手机端的IP地址和端口号，目的地址和端口号是访问的服务器端的IP地址和端口号。对于使用L2TP隧道用户或者使用MIP反向隧道的用户，待匹配的报文是隧道封装后的报文，源地址和端口号是隧道封装后报文的源IP地址和端口号，目的地址和端口号是隧道封装后报文的目的IP地址和端口号。
- 当过滤器用于基于APN或AclDefault的下行入口的Acl处理时，其中待匹配的报文是Gi入口报文，如果存在隧道封装是在解隧道封装后得到的用户报文，一般情况下源地址和端口号是服务器端的IP地址和端口号，目的地址和端口号是手机端的IP地址和端口号；当过滤器用于基于APN或AclDefault的下行出口的Acl处理时，其中待匹配的报文是GTP封装后的报文，一般情况下源地址和端口号是Pa接口的IP地址和端口号，目的地址和端口号是SGSN的IP地址和端口号。
- 当L34PROTOCOL选择为ICMP或GRE时，不允许配置端口。
- Rule、Filter、Flowfilter、Userprofile和URRGrpBinding相关配置在特定组合下需要兜底配置，否则可能会影响计费准确性。
- 请合理配置掩码范围，否则IP匹配范围变大，会影响规则匹配。
- 当SvrIpMode配置为DOMAIN模式的时候，会基于配置的DomainValue去查询Host记录，查询到Host记录，直接关联Host记录到Filter，丢弃配置的Host信息，否则使用当前配置的信息生成一条Host记录插入，该Host将会在命令执行后60s生效。
- 建议过滤器配置参数尽量精确，避免一个业务能匹配中的过滤器数目过多（超过500），从而对性能造成显著的恶化。
- 反掩码举例：网络的默认掩码255.255.255.0，其对应反掩码就是0.0.0.255，在这个例子中，IP地址的最后8位被用作主机地址。

## 使用实例

```
ADD FILTER:FILTERNAME="filter1",L34PROTTYPE=STRING,L34PROTOCOL=ANY,MSIPMODE=IP,MSIP="10.0.0.1",MSIPMASKTYPE=IPTYPE,MSIPMASK="0.0.0.0";
```
