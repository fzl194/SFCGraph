#### [适用NF](#ZH-CN_CONCEPT_0202422959)

SGW-U、PGW-U

#### [定义](#ZH-CN_CONCEPT_0202422959)

SRVCC（Single Radio Voice Call Continuity）为单语音呼叫持续。在LTE网络部署初期，LTE网络未完全覆盖，当VoLTE用户移动到了LTE未覆盖到的区域，语音业务则需要切换到CS网络（无线侧为GSM/UMTS），SRVCC可以保证在网络切换过程中，正在进行的语音业务不中断。

eSRVCC（enhanced SRVCC）是对SRVCC的增强。相对于SRVCC，eSRVCC优化了组网，通过设定信令面和媒体面锚点，使整个切换时延更短，提升了用户体验。因eSRVCC与SRVCC的差异主要在核心网，这里不区分介绍，SRVCC即代表SRVCC/eSRVCC。

#### [客户价值](#ZH-CN_CONCEPT_0202422959)

| 受益方 | 受益描述 |
| --- | --- |
| 运营商 | 实现语音业务从LTE网络到GSM/UMTS网络的无缝切换，有助于实现灵活的组网，降低运营成本。 |
| 用户 | 业务无中断，有良好的用户体验，提升对运营商的满意度。 |

#### [应用场景](#ZH-CN_CONCEPT_0202422959)

SRVCC业务主要应用在混合组网的网络中，在LTE网络部署初期，只在部分地区有网络存在，当用户从LTE网络漫游到只有GSM/UMTS网络区域时，原有的LTE网络语音接入就需要切换到GSM/UMTS网络中。SRVCC保证了切换的过程中用户语音业务不中断，提高运营商服务质量。

#### [应用限制](#ZH-CN_CONCEPT_0202422959)

- 仅支持PS域到CS域的切换，切换后，不支持CS域到PS域的切换。
- 仅支持LTE到GSM/UMTS的切换。
- 仅支持语音业务的切换。
- 对于语音切换场景，建议采用流量计费，因为语音切换到CS域后，用户的承载并没有删除。在语音通话中，用户没有使用数据业务，也会被收费。

#### [原理概述](#ZH-CN_CONCEPT_0202422959)

**实现原理**

当用户从E-UTRAN网络覆盖区域移动到UTRAN/GERAN网络覆盖区域，通过SRVCC特性将正在进行的语音业务切换到UTRAN/GERAN网络保证用户通话不中断，具体实现原理参见 [图2](#ZH-CN_CONCEPT_0202422959__fig145517584310) 。

**图2** E-UTRAN向UTRAN/GERAN CS切换

<br>

![](GWFD-020252 SRVCC特性概述_02422959.assets/zh-cn_image_0202631783.png "点击放大")

SRVCC实现步骤如下：

1. UE在E-UTRAN网络建立IMS语音承载。
2. eNodeB发起E-UTRAN PS域到GERAN/UTRAN CS域的切换流程后，将语音承载从MME切换到MSC。
3. MSC在GERAN/UTRAN网络建立IMS语音承载。

**SRVCC** **解决方案附着流程和普通附着流程的不同之处**

- UE在Attach Request消息中携带MS network capability，该参数指示是否支持SRVCC功能。
- MME通过Update Location Request消息中的UE-SRVCC-Capability信元把UE的SRVCC能力以及MME的“Homogeneous Support of IMS Voice Over PS Sessions”能力传递给HSS。
- HSS给MME回复的Update Location Response消息中携带STN-SR和MSISDN。MME将这两个参数保存在EPS上下文中。
- 如果MME支持SRVCC业务，MME在Initial Context Setup Request携带“SRVCC is possible”的标志，表示UE和MME都支持SRVCC业务。

> **说明**
> 在TAU和service request流程之中，UE也会携带SRVCC标识，以通知MME，UE支持SRVCC。

**业务流程**

SRVCC业务流程如 [图3](#ZH-CN_CONCEPT_0202422959__fig3) 所示。

**图3** SRVCC业务流程

<br>

![](GWFD-020252 SRVCC特性概述_02422959.assets/zh-cn_image_0206018329.png)

1. UE上报无线资源测量报告。
2. 源E-URTAN根据UE发送的无线资源测量报告（eNodeB可以从MS Radio Access capability中得知终端是否支持DTM/DTM HO/PS HO，发现UE或者GERAN不支持DTM），eNodeB判决应发起SRVCC切换到UTRAN/GERAN。
3. 源E-URTAN向MME发送切换请求消息(handover required)（消息中包括用户标识、Target ID，generic Source to Target Transparent Container，SRVCC HO Indication, MS Classmark2, MS Classmark3, Handover Type(例如=LTEtoGERAN)），并标识该切换为SRVCC(SRVCC HO Indication=CS only;Source to Target Transparent Container=old BSS to new BSS information)。
4. MME根据SRVCC HO Indication和QCI=1，将语音承载（QCI＝1）和其他的PS承载进行分离。接着MME执行PS-CS切换程序。MME会存储承载上下文，承载上下文中有QCI值和SRVCC HO Indication。
5. MME根据handover required中的Target ID选择具有SRVCC功能的MSC，发送PS到CS的切换请求等信息。消息中包括：IMSI，Target ID，STN-SR，C MSISDN(也就是IMS里的IMPU)，Source to Target Transparent Container，MM Context。请求中包含STN-SR(SRVCC会话切换号码)、C-MSISDN、MM上下文（包含CS security key）等信息。其中STN-SR，C-MSISDN签约在SAE-HSS。MME与MSC之间详细Sv接口描述可参考3GPP TS 29.280。
6. SRVCC MSC收到PS-CS切换请求后，根据Target ID(这个在2G来说是小区编号Cell ID，对目标是3G来说是RNC ID)来选择目标MSC以执行MSC间切换，然后向目标MSC发送Prepare Handover Request请求。这里SRVCC MSC需要构造(通过下面MSC的位置区小区表的配置)一个源SAI(小区编号:Service Area Identifier)以填充必选信元。
7. 目标MSC向目标BSS发送切换请求(HO Req)，请求分配CS无线承载资源。
8. 目标BSS确认CS无线承载资源分配完毕，目标MSC向SRVCC MSC发送Prepare HO Resp。
9. SRVCC MSC使用切换号码建立与目标MSC CS承载连接（ISUP/BICC/SIP）。在SRVCC MSC和MSC合一的形态中，SRVCC MSC和目标MSC可能是同一个，这时候就是局内切换。
10. SRVCC MSC发起Session Transfer过程，发送IAM/INVITE（被叫号码为STN-SR，用于SRVCC的session transfer number；C-MSISDN作为主叫号码）到IMS。
11. SCC AS收到INVITE消息。
12. SCC AS检查Request-URI为STN-SR号码(SCC AS需在本地配置STN-SR号码)，说明这是一个切换的呼叫，SCC AS执行session transfer流程。SCC AS将本端CS接入腿（新SDP）信息，即MSC控制下的MGW的IP地址，端口通知到对端，并释放本端IMS接入腿。SCC AS向远端呼叫腿发送re-INVITE消息，进行Session transfer过程。
13. SRVCC MSC向源MME发送PS to CS Resp，通知MME CS无线承载资源分配完毕。
14. MME向E-UTRAN发送Handover Command（Target to Source Transparent Container），只包含语音成分。
15. E-UTRAN向UE发送HO command，指示UE开始切换。
16. UE开始向GERAN的CS域下尝试接入。
17. UE切换到GERAN下，UE向目标BSS发送HO detection，目标BSS检测到UE已切换到目标小区。
18. 由于UE不支持DTM，故UE无法在切换后的PS接入上激活业务。UE向目标BSS发送suspend请求，目标BSS向目标SGSN转发suspend请求，目标SGSN向源MME转发suspend Notification，MME响应suspend Acknowledge。
19. 目标BSS通知目标MSC UE已切换成功。
20. 目标MSC发送SES（HO completed）到SRVCC MSC。
21. 目标MSC发送ISUP ANM/SIP 200 OK，完成与SRVCC MSC与目标MSC之间CS承载的建立。
22. SRVCC MSC发送PS to CS Complete通知MME切换完成。
23. MME通知SGW-C/PGW-C挂起Non-GBR承载，删除其他GBR承载。
24. SGW-C/PGW-C给SGW-U/PGW-U发送PFCP Session Modification Request消息，通知SGW-U/PGW-U挂起Non-GBR承载，删除其他GBR承载。
25. SGW-U/PGW-U给SGW-C/PGW-C发送PFCP Session Modification Response消息。
26. 如果多个MSC/MGW都采用同一个LAI（Location Area Identity），MSC/MGW要采用自己的NRI(network resource identifier)向终端发起TMSI relocation过程。
27. 如果此用户在SRVCC MSC中的注册状态为unkown，SRVCC MSC发送Update Location到HLR更新用户位置信息。这样，后续的CS域用户发起呼叫就可以正确地（通过SRVCC MSC）找到被叫。

> **说明**
> CS语音结束并且满足接入LTE的条件，UE会向LTE进行切换：
>
> - MME接到切换请求，会发送Resume Notification到SGW-C，SGW-C透传到PGW-C，PGW-C清除UE的挂起标记，向SGW-C传递消息。
> - SGW-C/PGW-C给SGW-U/PGW-U发送PFCP Session Modification Request消息，通知SGW-U/PGW-U取消挂起，SGW-U/PGW-U给SGW-C/PGW-C发送PFCP Session Modification Response消息。SGW-C回应Resume Acknowledge给MME。
