#### [用户附着流程](#ZH-CN_CONCEPT_0269258753)

本特性Attach流程如 [图1](#ZH-CN_CONCEPT_0269258753__fig243115811463) 所示，相较于传统4G流程做了如下特殊处理：

- 终端和网络侧通过Attach流程协商数据传输方式。终端在Attach Request消息中携带UE network capability信元，指示终端是否支持CP-CIoT数据传输方式。
- 在Create Session Request消息中，MME会把S11-U的TEID以及终端支持的NB-IoT RAT Type通知SGW-C，SGW-C返回Create Session Response消息，将SGW-C侧S11-U接口TEID以及用户面IP发送给MME，用于为终端创建基于S11-U接口的用户面承载。
- 不发送Initial Context Setup Request消息通知eNodeB建立E-RAB承载，而是通过Downlink NAS Transport消息将Attach Accept消息直接传递给终端。

**图1** 用户附着流程

<br>

![](实现原理_69258753.assets/zh-cn_image_0000001456545488.png)

1. UE向MME发送附着请求，其中携带RAT Type（NB-IoT）及UE network capability信元，指示终端是否支持CP-CIoT。若终端不支持CP-CIoT，则流程失败，用户正常Attach，不启用该特性。
2. 网络建立NAS安全连接并且完成和UE之间的相互鉴权。
3. MME从HSS处获取UE的签约信息，如果UE附着的MME与上次不一致，则新MME通过Update Location Request更新UE的位置信息。
4. MME根据签约信息和本地配置判断是否允许NB-IoT终端接入。
5. MME向SGW-C发送会话建立请求，其中携带RAT Type、MME侧S11-U接口的TEID和用户面IP。
6. SGW-C和SGW-U之间建立PFCP SessionEstablishment。
7. SGW-C向PGW-C发送Session请求。
8. PGW-C和PGW-U之间建立PFCP SessionEstablishment。
9. PGW-C向SGW-C返回Session响应。
10. SGW-C和SGW-U之间建立PFCP SessionModification。
11. SGW-C向MME回应会话建立请求，其中携带S11-U接口的TEID以及用户面IP。
12. MME通过NAS消息将Attach Accept传递给UE，并且指示终端协商结果支持CP-CIoT。
13. UE回应MME消息，完成Attach。

#### [用户发起的数据传输流程](#ZH-CN_CONCEPT_0269258753)

本特性用户发起的数据传输流程图如下：

**图2** 用户发起的数据传输流程

<br>

![](实现原理_69258753.assets/zh-cn_image_0000001454403540.png)

1. UE建立RRC连接，在NAS消息中发送已加密和完整性保护的上行数据。
  > **说明**
  > UE在NAS消息中可包含Release Assistance Information。
  >
  > - 如果Release Assistance Information中没有下行数据指示，MME将数据传输给SGW-C后就立即释放连接。
  > - 如果Release Assistance Information中有下行数据指示，且要求收到下行数据后释放连接，MME将数据传输给SGW-C后等待下行数据，当接收到下行数据后，下发给eNodeB，并通知eNodeB在完成空口下行数据传输后释放连接。
2. eNodeB发送S1-AP Initial UE message给MME，同时控制面和用户面之间修改PFCP Session。
3. MME检查NAS消息的完整性，然后解密数据。
4. **可选：**如果MME与SGW-U之间没有建立数据通道（S11-U连接），MME发送Modify Bearer Request消息，提供MME的下行传输地址和TEID给SGW-C，并通过PFCP会话发送给SGW-U。
5. **可选：**如果RAT类型或者用户位置信息有变化，SGW-C会发送Modify Bearer Request消息给PGW-C。PGW-C给SGW-C回复Modify Bearer Response消息。
6. **可选：**SGW-C在响应消息Modify Bearer Response中给MME提供上行传输的SGW-U地址和TEID。
7. MME将上行数据经SGW-U发送给PGW-U。
8. **可选：**如果在[1](#ZH-CN_CONCEPT_0269258753__li1)的Release Assistance Information中没有下行数据指示，MME将上行数据发送给SGW-U/PGW-U后，立即释放连接，执行步骤[13](#ZH-CN_CONCEPT_0269258753__li13)。否则，则进行下行数据传输，如果没有接收到下行数据，则跳过步骤[9](#ZH-CN_CONCEPT_0269258753__li9)-[12](#ZH-CN_CONCEPT_0269258753__li12)进行释放流程。
9. **可选：**MME接收到下行数据后，会进行加密和完整性保护。
10. **可选：**如果有下行数据，MME会在NAS消息中下发给eNodeB。如果上行数据有Release Assistance Information指示，要求在MME在接收到下行数据并转发给eNodeB后进入[13](#ZH-CN_CONCEPT_0269258753__li13)，启动S1连接释放。
11. **可选：**eNodeB将NAS data下发给UE。如果同时收到MME的S1连接释放消息，eNodeB会先发送NAS Data，再释放RRC连接，参见[13](#ZH-CN_CONCEPT_0269258753__li13)。
12. 如果NAS传输有一段时间没活动，eNodeB则进入[13](#ZH-CN_CONCEPT_0269258753__li13)启动S1连接释放。
13. S1释放流程，释放S1-MME连接和S11/S11-U连接。

#### [网络侧发起的数据传输流程](#ZH-CN_CONCEPT_0269258753)

本特性网络侧发起的数据传输流程图如下：

**图3** 网络侧发起的数据传输流程

<br>

![](实现原理_69258753.assets/zh-cn_image_0000001454958644.png)

1. 需要发送给UE的下行数据包经PGW-U传输给SGW-U。
2. SGW-U收到下行数据发送PFCP Session Modification Report Request给SGW-C。
3. 当该UE的用户面连接尚未建立时，即SGW-U上下文数指示不存在下行链路用户面TEID，SGW-U将在本地缓存这部分数据包，并识别该UE对应的MME。（如果UE的用户面连接已建立，则SGW-U通过S11-U接口直接将数据下行数据经MME送达UE。）
4. SGW-C发送Downlink Data Notification消息给MME。
5. MME给SGW-C响应Downlink Data Notification Acknowledge消息。
6. SGW-C收到消息后返回PFCP Session Modification Report Response给SGW-U。
7. MME向eNodeB发送Paging消息发起寻呼流程。
8. eNodeB向UE发起Paging消息。
9. UE向MME发送Control Plane Service Request消息作为寻呼应答。
  > **说明**
  > MME不需要触发建立无线数据承载，MME可通过NAS PDU直接传输下行数据给eNodeB。
10. **可选：**如果S11-U数据通道没有建立，执行此步骤：MME发送Modify Bearer Request给SGW-U，消息包含MME地址以及下行TEID、 RAT类型等，SGW-U可以发送下行数据给UE。
11. SGW-C收到Modify Bearer Request以后发送PFCP Session Modification Request给SGW-U。SGW-U收到消息后返回PFCP Session Modification Response消息给SGW-C。
12. 如果用户位置信息发生变化等，执行此步骤：SGW-C会发送Modify Bearer Request给PGW-C。
13. 如果用户位置信息发生变化等，执行此步骤：PGW-C会发送PFCP Session Modification Request给PGW-U。PGW-U发送PFCP Session Modification Response给PGW-C。
14. PGW-C发送Modify Bearer Response给SGW-C。
15. SGW-C发送Modify Bearer Response给MME。
16. SGW-U发送缓存的下行数据给MME。
17. MME检查下行数据消息的完整性，然后加密数据。
18. MME通过NAS消息将下行数据发送给eNodeB。
19. eNodeB通过NAS消息将下行数据发送给UE。
20. **可选：**如果RRC一直处于连接状态，可通过NAS PDU继续传送上下行数据，无需重新建立用户面数据承载。
21. 如果NAS传输有一段时间没活动，eNodeB则进入[22](#ZH-CN_CONCEPT_0269258753__li17)启动S1释放。
22. S1释放流程，释放S1-MME连接和S11连接。
