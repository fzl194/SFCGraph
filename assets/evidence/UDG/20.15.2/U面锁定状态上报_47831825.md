# U面锁定状态上报

Full Mesh场景下， 当UPF进行升级等操作时通过 **LCK NF** 命令直接锁定UPF，UPF通过偶联消息上报锁定状态，对应的SMF感知UPF锁定后，新用户激活不再选择已经锁定的UPF。UPF支持上报锁定状态信息的场景和方式如 [表1](#ZH-CN_TOPIC_0000001147831825__table1078203113136) 所示。

*表1 UPF锁定状态上报场景和方式*

| 锁定状态上报场景 | 锁定状态上报方式 |
| --- | --- |
| SMF主动发起偶联建立场景 | UPF收到SMF发送的PFCP Association Setup Request消息后，在回复PFCP Association Setup Response消息中给SMF携带UPF当前的锁定状态。无论UPF当前是否处于锁定状态，都要在偶联建立响应中携带UPF的状态信息。 |
| SMF和UPF建链完成后修改锁定状态场景 | - UPF通过**LCK NF**命令配置锁定后，UPF通过PFCP Association Update Request消息上报锁定状态，收到SMF的PFCP Association Update Response消息后确认状态上报成功。<br>- UPF通过**LCK NF**命令解锁定后，通过PFCP Association Update Request消息上报解锁状态，收到SMF的PFCP Association Update Response消息后确认状态上报成功。 |
| N4链路故障恢复场景 | N4链路故障恢复后，通过偶联更新消息上报UPF锁定状态。<br>- 当SMF主动发送PFCP Association Update Request消息时，UPF在PFCP Association Update Response消息中上报UPF的当前锁定状态。<br>- 当UPF主动发送PFCP Association Update Request消息时，UPF在请求消息中主动上报UPF的当前锁定状态。 |
| 周期性上报场景 | 为了避免信令并发，SMF接收消息时序和UPF发送消息时序不一致，导致SMF保存错误的UPF锁定状态。UPF通过周期性发送PFCP Association Update Request消息上报锁定状态，周期时长为5分钟。 |
