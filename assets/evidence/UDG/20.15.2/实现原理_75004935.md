# 实现原理

VoNR语音是NR部署下的语音目标方案，呼叫时延低、语音质量高、数据业务体验好。

*表1 EPS Fallback与VoNR的体现/连续性比对*

| 体验/连续性 | EPS FallBack | VoNR |
| --- | --- | --- |
| 呼叫建立时延 | EPS FallBack存在NR-LTE的回落过程中，呼叫建立时延较大。 | 相比EPS FallBack，省去NR-LTE的回落，单侧呼叫建立时延可节省1s左右。 |
| 语音质量 | 若UE和网络侧都支持EVS，语音质量差异不大。 | VoNR语音，终端大多都支持EVS，实现超清语音, 更低的带宽可获得更高MOS值。 |
| 业务连续性 | 5G UE通过EPS FallBack回落到4G实现语音业务，UE在4G网络的通话可通过SRVCC切换到2/3G网络。 | 建议VoNR部署以4G网络作为基础网络，便于后续可通过4G-5G互操作实现VoNR-VoLTE间的业务连续性。 |
| 数据业务体验 | EPS Fallback是UE从5G回落到4G执行语音的解决方案。因此通话期间，UE在4G实现数据业务。 | UE在VoNR通话中，可享受5G数据业务的大带宽、低时延的高速数据业务体验。 |

*表2 EPS FallBack与VoNR的方案比对*

| 解决方案 | EPS FallBack | VoNR |
| --- | --- | --- |
| 语音方案原理 | 打电话时由NG-RAN触发切换回落到4G，基于VoLTE提供语音。 | 用户在NR覆盖下打电话时直接被NR基站处理，采用VoNR通话。 |
| 部署前提 | - 要求VoLTE作为基础网，4G网络覆盖包含5G网络。<br>- EPC和5GC开N26接口，并启用4G/5G切换。 | - 无需4G网络覆盖包含5G网络。<br>- EPC和5GC开N26接口，并启用4G/5G切换。 |
| 设备改造 | - NR基站无需开通话音功能。<br>- 5GC需开通EPS FallBack。<br>- IMS支持接入5GC。 | - NR基站需开通话音功能。<br>- 5GC需开通VoNR。<br>- IMS支持接入5GC。 |
| 接续体验 | 接续时长（2.7~3.7秒），数据业务也将跟随切换到4G。 | 接续时长（1~2秒），数据与语音业务均在NR完成。 |
| 4G/5G切换时延 | 大于200 ms | 不大于120 ms |
| 视频通话体验 | 4G基站无法承载过多的视频呼叫，话务量大时很容易拥塞。 | NR基站可承载大量的视频呼叫，用户在NR下体验更好。 |
| 音视频通话体验 | LTE低频覆盖，穿透力强，切换等引起中断少。 | NR高频覆盖，穿透力差，切换等引起中断多。 |
| 推荐方案 | 过渡阶段推荐 | 目标网推荐 |

**VoNR注册**

5G用户通过标准的注册流程和PDU Session建立流程执行5G语音注册。流程图如 [图1](#ZH-CN_TOPIC_0000001175004935__fig238144316198) 所示。

**图1** VoNR注册流程

<br>

![](实现原理_75004935.assets/zh-cn_image_0000001504579509.png)

1. UE向5GS发起初始注册请求。
2. AMF发送Nudm_ECM_Registration消息向UDM发起用户注册，并获取和订阅用户签约数据。AMF根据本地配置设置Nudm_ECM_Registration消息中的imsVoPs信元。
3. **可选：**AMF根据本地配置决策是否向NG-RAN/UE获取该UE的Vo5G能力。
4. **可选：**AMF根据本地配置决策是否触发AM Policy耦连建立流程。
5. AMF根据本地配置和UE Radio Capability Check流程的结果，决策是否在Registration Accept消息的信元5GS Network Feature Support中如何设置IMS-VoPS-3GPP。如果设置为IMS voice over PS session supported over 3GPP access，则表明允许UE使用Vo5G。
6. 注册流程成功后，UE会发起IMS语音PDU Session的建立流程，UL NAS Transport消息中会携带SM消息PDU Session Establishment Request, IMS DNN, S-NSSAI, PDU Session ID等消息。
7. AMF根据DNN和位置信息TAI向NRF查询SMF，并将PDU Session Establishment Request消息通过服务化接口转发给选中的SMF。
8. SMF收到PDU Session Establishment Request消息后，向UDM获取用户的会话管理签约数据，并订阅签约数据变化事件。
9. SMF根据DNN，SUPI或者GPSI向NRF查询PCF，并向选中的PCF请求用户的默认语音策略数据。
10. PCF返回用户的默认语音策略后，SMF向UPF请求分配用户面资源，例如UE IP。
  > **说明**
  > 协议既支持SMF分配UE IP，也支持UPF分配UE IP，如果是UPF分配UE IP，则UPF需要在步骤10中返回UE IP给SMF。
11. SMF发送Npcf_SMPolicyControl_Update消息携带UE IP通知PCF。
12. SMF向AMF发送Namf_Communication_N1N2MessageTransfer，携带PDU Session Resource Setup Request Transfer请求NG-RAN建立语音RAB，携带PDU Session Establishment Accept通知UE PDU Session建立成功。其中PDU Session Establishment Accept中会携带SMF为UE选择的P-CSCF地址。
13. AMF转发PDU Session Resource Setup Request携带PDU Session Resource Setup Request Transfer以及PDU Session Establishment Accept给NG-RAN，NG-RAN转发PDU Session Establishment Accept给UE。NG-RAN为IMS PDU Session分配完资源后，发送PDU Session Resource Setup Response携带PDU Session Resource Setup Response Transfer给AMF。
  > **说明**
  > PDU Session Resource Setup Response Transfer携带无线基站为UE分配的语音下行用户面转发地址。
14. AMF通过Nsmf_PDUSession_UpdateSMContext Request消息携带PDU Session Resource Setup Response Transfer给SMF。
15. SMF通过N4接口消息将PDU Session Resource Setup Response Transfer包含的语音下行用户面地址发送UPF。这样，UE在5GC的语音控制面和用户面通道就建立成功。
16. UE向P-CSCF发起IMS注册，携带5G NR接入类型及5G位置区小区信息，IMS后续消息交互流程与VoLTE注册相同。

**VoNR主叫流程**

**图2** VoNR主叫流程

<br>

![](实现原理_75004935.assets/zh-cn_image_0000001504586017.png)

1. UE发起Service Request流程进入ECM_Connected状态。
2. UE发起5G语音主叫。
3. P-CSCF向DRA发送AAR消息触发语音专有QoS Flow建立，DRA收到AAR消息后，根据本地配置的号段决策是否需要向BSF查询UE所在的PCF。
4. 如果需要，则DRA调用服务化接口向BSF发送Nbsf_Management_Discovery消息携带UE IP向BSF查询PCF ID。
5. BSF根据UE IP从本地存储的UE IP和PCF地址的表中获取到PCF地址通过Nbsf_Management_Discovery Response消息返回给DRA。
6. DRA发送AAR消息给PCF。
7. PCF给DRA返回AAR消息，DRA再传递给P-CSCF。
8. PCF发送Npcf_SMPolicyControl_UpdateNotify消息请求SMF触发5QI=1/2的语音/视频专有QoS Flow的建立。
  > **说明**
  > - 语音场景，PCF下发携带5QI为1的PCC Rule。
  > - 视频场景，PCF同时下发携带5QI为1和2的PCC Rules。
9. SMF触发语音或者语音和视频的专有QoS Flow建立，发送Namf_Communication_N1N2MessageTransfer消息传递给AMF。
10. AMF向RAN发送PDU Session Modification Request消息请求建立5QI=1/2的专有QoS Flow。
11. RAN将PDU Session Modification Command消息转发给UE建立5QI=1/2的专有QoS Flow。
12. UE将PDU Session Modification Command ACK消息转发给RAN，确认该QoS Flow创建成功。
13. RAN向AMF发送PDU Session Modification Response消息确认该QoS Flow创建成功。
14. AMF通过Nsmf_PDUSession_UpdateSMContext携带PDU Session Resource Modification Response Transfer信元给SMF。
15. SMF通过PFCP Session Modification Request/Response消息更新流程将RAN分配的下行语音用户面转发地址发送给UPF。
16. UE给RAN回复PDU Session Modification Complete消息。
17. RAN给AMF回复PDU Session Modification Response消息。
18. AMF给SMF发送Nsmf_PDUSession_UpdateSMContext通知专有QoS Flow建立成功。该流程完成后，主叫侧的IMS语音或者视频转发通道就建立成功。
19. SMF通过Npcf_SMPolicyControl_Update消息通知PCF该专有QoS Flow建立成功。
20. PCF通过RAR消息通知P-CSCF该专有QoS Flow建立成功。
21. P-CSCF向PCF回复RAA消息。

**VoNR被叫流程**

**图3** VoNR被叫流程

<br>

![](实现原理_75004935.assets/zh-cn_image_0000001130491272.png)

1. IMS向UPF发送SIP INVITE消息。如果UE处于IDLE态，则5GC启动Paging流程，对被叫UE进行寻呼。
2. UPF向SMF发送Data Notification。
3. SMF向UPF回复Data Notification Ack。
4. SMF给AMF发送Namf_Communication_N1N2MessageTransfer消息，消息中包括PDU会话ID、N2 SM information、ARP、寻呼策略等信息。
5. AMF发送Paging消息给RAN，在UE注册的区域范围内寻呼UE。
6. 若UE收到寻呼请求，并处于IDLE态，发起Service Request流程，建立UE到UPF端到端的信令连接和用户面承载。
7. UPF向UE发送SIP INVITE消息。
8. UE返回183响应，表示呼叫请求正在处理，消息中携带被叫的媒体信息。
9. IMS发送AAR消息给PCF，请求获取UE的接入网类型。
10. PCF给IMS响应AAA消息，其中IP-CAN-Type和RAT-Type AVP携带了UE的接入网类型，触发创建专用QoS Flow的流程。
11. PCF会通知5GC进行媒体专用的QoS Flow建立及用户位置的上报并携带接入类型和位置信息。
12. PCF将接入类型和位置信息通过RAR消息发送给IMS。
13. IMS收到RAR消息后，向PCF返回RAA响应。
