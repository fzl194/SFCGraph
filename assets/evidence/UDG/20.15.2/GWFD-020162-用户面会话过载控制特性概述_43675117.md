#### [适用NF](#ZH-CN_TOPIC_0243675117)

SGW-U、PGW-U、UPF

#### [定义](#ZH-CN_TOPIC_0243675117)

为了避免 UDG 上EPS承载/PDU会话残留导致资源浪费从而出现会话过载的情况，通过 UDG 向SGW-C/PGW-C/SMF发送Report核查消息清除残留EPS承载/PDU会话资源，达到控制用户面会话过载的目的。

#### [客户价值](#ZH-CN_TOPIC_0243675117)

| **受益方** | **受益描述** |
| --- | --- |
| 运营商 | 提高用户接入的可靠性，清除残留EPS承载/PDU会话资源，保证用户接入数量。 |
| 用户 | 用户不感知该特性。 |

#### [应用场景](#ZH-CN_TOPIC_0243675117)

当N4/Sx链路故障或其他原因导致用户去激活，SGW-U/PGW-U/UPF未收到SGW-C/PGW-C/SMF的去激活消息时导致用户的EPS承载/PDU会话资源残留时，该特性可以清除SGW-U/PGW-U/UPF上的残留EPS承载/PDU会话信息。

#### [可获得性](#ZH-CN_TOPIC_0243675117)

**涉及NF**

| **涉及NF** | **支持版本** | **功能说明** |
| --- | --- | --- |
| SGW-C/PGW-C/SMF | UNC 20.3.2及后续版本 | 根据SGW-U/PGW-U/UPF上报的PFCP Session Report Request消息，查询SGW-C/PGW-C/SMF上是否存在相关的EPS承载/PDU会话信息，并在回复PFCP Session Report Response消息时告知SGW-U/PGW-U/UPF该EPS承载/PDU会话资源是否存在。 |
| SGW-U/PGW-U/UPF | UDG 20.3.2及后续版本 | 当满足会话核查上报条件时，触发与SGW-C/PGW-C/SMF之间的会话信息核查。 |

#### [应用限制](#ZH-CN_TOPIC_0243675117)

UDG 上残留用户会话上下文期间又收到会话创建消息场景下，只有消息中携带IMSI，且该IMSI下的EPS承载/PDU会话个数超过配置设置的门限时才会触发Report消息核查。

#### [原理概述](#ZH-CN_TOPIC_0243675117)

当N4/Sx链路故障或其他原因导致用户去激活，SGW-U/PGW-U/UPF未收到SGW-C/PGW-C/SMF的去激活消息时将导致用户的EPS承载/PDU会话资源残留。当用户再次激活时在SGW-U/PGW-U/UPF上创建新的EPS承载/PDU会话，导致EPS承载/PDU会话资源浪费，SGW-U/PGW-U/UPF上的可激活用户数大幅减少。为了避免用户面会话过载，SGW-U/PGW-U/UPF利用会话核查功能删除老的EPS承载/PDU会话资源。

当满足会话核查上报条件时，SGW-U/PGW-U/UPF向SGW-C/PGW-C/SMF发送PFCP Session Report Request消息，请求SGW-C/PGW-C/SMF查询是否存在用户的老的EPS承载/PDU会话，当SGW-C/PGW-C/SMF回复PFCP Session Report Response消息，并告知在SGW-C/PGW-C/SMF上不存在相关的EPS承载/PDU会话资源时，SGW-U/PGW-U/UPF将删除残留的EPS承载/PDU会话资源，如 [图1](#ZH-CN_TOPIC_0243675117__fig1978588810) 所示。

**图1** 会话核查流程

<br>

![](GWFD-020162 用户面会话过载控制特性概述_43675117.assets/zh-cn_image_0243675112.png)

触发SGW-U/PGW-U/UPF向SGW-C/PGW-C/SMF发送Report消息进行会话核查的场景如下：

- SGW-U/PGW-U/UPF收到SGW-C/PGW-C/SMF通过N4/Sx接口下发的PFCP Session Establishment Request消息后，如果消息中携带了用户的IMSI信元，SGW-U/PGW-U/UPF检查本地该IMSI下的EPS承载/PDU会话个数是否超过[**SET SESSCHKFUNC**](../../../../../OM参考/命令/UDG MML命令/用户面服务管理/会话管理/业务恢复管理/业务核查/设置会话核查配置（SET SESSCHKFUNC）_37678016.md)命令的“SESSNUMTHD”参数设置的门限，如果超过配置指定的门限，需要针对老的EPS承载/PDU会话信息触发一次和对应SGW-C/PGW-C/SMF之间的Report消息核查，防止老的EPS承载/PDU会话在SGW-U/PGW-U/UPF上残留。新的EPS承载/PDU会话创建流程则正常处理，继续完成新的EPS承载/PDU会话的创建流程。
- 每天当地凌晨四点固定触发Report消息进行会话信息核查，SGW-U/PGW-U/UPF将本地所有EPS承载/PDU会话向对应SGW-C/PGW-C/SMF发送Report消息进行会话核查。
- N4链路从Down恢复到正常状态后，SGW-U/PGW-U/UPF将本地和该N4链路相关的所有EPS承载/PDU会话向对应SGW-C/PGW-C/SMF发送Report消息进行会话核查。
- Session SG进程复位、POD主备倒换和POD复位后，SGW-U/PGW-U/UPF会触发与SGW-C/PGW-C/SMF之间的会话信息核查。
- POD会话数达到总会话数的90%时，SGW-U/PGW-U/UPF会触发与SGW-C/PGW-C/SMF之间的会话信息核查。
