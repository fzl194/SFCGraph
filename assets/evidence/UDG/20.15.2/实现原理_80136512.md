#### [IPv6内容计费](#ZH-CN_TOPIC_0180136512)

IPv6内容计费，即基于IPv6业务的计费，是指移动网络可通过对IPv6用户的上下行报文进行包过滤和分析，识别区分出用户上下行数据传送的IPv6业务种类，并把相关的信息上报给SMF，并进一步上报给OCS、CG或CHF，针对不同的IPv6业务，实施不同的计费方式或费率计费。IPv6用户内容计费的详细原理请参见 [GWFD-020301 内容计费基本功能](../../计费功能/GWFD-020301 内容计费基本功能_66863836.md) 。在IPv6用户报文到达PGW-U/UPF后，PGW-U/UPF对IPv6用户报文进行三四层\七层的过滤和解析，区分不同IPv6业务，基于IPv6业务进行流量或时长的统计，其中：

- 在线内容计费：基于PGW-C/SMF下发的配额和阈值，触发业务控制和使用量上报PGW-C/SMF，最终汇集到OCS。
- 离线内容计费：基于统计数据生成话单，发送到CG。
- 融合内容计费：参考[GWFD-010173 融合计费](../../计费功能/GWFD-010173 融合计费_70301692.md)，对于带配额管理的计费，基于PGW-C/SMF下发的配额和阈值，触发业务控制和使用量上报PGW-C/SMF，最终汇集到CHF；对于不带配额管理的计费，则在使用过程中及结束后收集计费信息并上报给CHF。

#### [IPv6 URL重定向](#ZH-CN_TOPIC_0180136512)

IPv6 URL重定向是指将IPv6用户重定向到IPv6 Redirect Server去获取URL资源。IPv6 URL重定向有以下两种场景：

- 本地触发IPv6 URL重定向
  URL Redirect动作配置在rule中。
  PGW-U/UPF支持对IPv6用户报文进行三四层或/和七层解析和规则匹配成功后触发URL Redirect动作，并且能够构造IPv6格式的重定向报文。
  如果PGW-U/UPF上未使用 [**SET SRVCOMMONPARA**](../../../../../OM参考/命令/UDG MML命令/用户面服务管理/业务匹配策略/业务匹配公共配置/业务公共参数管理/业务公共参数/设置业务公共参数（SET SRVCOMMONPARA）_82837309.md) 命令开启内嵌IPv4的IPv6报文处理功能，PGW-U/UPF对IPv6用户报文解析和规则匹配成功后获取的Redirect Server地址为IPv4地址时，对于这种地址类型不匹配的情况执行丢包处理。
  如果PGW-U/UPF上使用 [**SET SRVCOMMONPARA**](../../../../../OM参考/命令/UDG MML命令/用户面服务管理/业务匹配策略/业务匹配公共配置/业务公共参数管理/业务公共参数/设置业务公共参数（SET SRVCOMMONPARA）_82837309.md) 命令开启内嵌IPv4的IPv6报文处理功能，PGW-U/UPF对IPv6用户报文解析和规则匹配成功后获取的Redirect Server地址为IPv4地址时，PGW-U/UPF根据IPv4地址对应的URL执行重定向动作。
- PCF下发IPv6 URL重定向
  IPv6用户配额耗尽时，PGW-C/SMF向PCF请求下发规则。PCF根据PGW-C/SMF发来的包含IPv6地址的CCR消息下发IPv6重定向充值页面。
  如果PGW-U/UPF上未使用 [**SET SRVCOMMONPARA**](../../../../../OM参考/命令/UDG MML命令/用户面服务管理/业务匹配策略/业务匹配公共配置/业务公共参数管理/业务公共参数/设置业务公共参数（SET SRVCOMMONPARA）_82837309.md) 命令开启内嵌IPv4的IPv6报文处理功能，PCF给IPv6用户下发IPv4重定向充值页面时，PGW-U/UPF记录异常后正常转发报文，不执行重定向动作。
  如果PGW-U/UPF上使用 [**SET SRVCOMMONPARA**](../../../../../OM参考/命令/UDG MML命令/用户面服务管理/业务匹配策略/业务匹配公共配置/业务公共参数管理/业务公共参数/设置业务公共参数（SET SRVCOMMONPARA）_82837309.md) 命令开启内嵌IPv4的IPv6报文处理功能，PCF给IPv6用户下发IPv4重定向充值页面时，PGW-U/UPF根据IPv4地址对应的URL执行重定向动作。

#### [IPv6 Captive Portal](#ZH-CN_TOPIC_0180136512)

Captive Portal即个人门户网站，可应用于广告或新闻订阅推送。IPv6 Captive Portal是指PGW-U/UPF支持在IPv6用户访问某些网站时，被自动重定向到IPv6 Portal Server，IPv6 Portal Server推送广告或订阅新闻给用户进行浏览，后续IPv6用户可以正常进行业务访问。

IPv6 Captive Portal的详细原理请参见 [GWFD-110281 用户Portal](../../智能策略控制功能/GWFD-110281 用户Portal_66620113.md) ，不同点在于以下几方面要支持IPv6：

- PGW-U/UPF持IPv6 Captive Portal动作配置。
- PGW-U/UPF支持对IPv6用户报文进行七层解析和规则匹配成功后触发Captive Portal动作。当IPv6用户处于captive模式时， PGW-U/UPF会把用户发出的Get/Post请求报文重定向到IPv6 Portal Server；在non-captive模式下，用户可以正常进行业务，PGW-U/UPF不会对用户报文做重定向处理。
- 如果PGW-U/UPF上未使用[**SET SRVCOMMONPARA**](../../../../../OM参考/命令/UDG MML命令/用户面服务管理/业务匹配策略/业务匹配公共配置/业务公共参数管理/业务公共参数/设置业务公共参数（SET SRVCOMMONPARA）_82837309.md)命令开启内嵌IPv4的IPv6报文处理功能，PGW-U/UPF对IPv6用户报文解析和规则匹配成功后获取的Portal Server地址为IPv4地址时，对于这种地址类型不匹配的情况执行丢包处理。
- 如果PGW-U/UPF上使用[**SET SRVCOMMONPARA**](../../../../../OM参考/命令/UDG MML命令/用户面服务管理/业务匹配策略/业务匹配公共配置/业务公共参数管理/业务公共参数/设置业务公共参数（SET SRVCOMMONPARA）_82837309.md)命令开启内嵌IPv4的IPv6报文处理功能，PGW-U/UPF对IPv6用户报文解析和规则匹配成功后获取的Portal Server地址为IPv4地址时，PGW-U/UPF根据IPv4对应的重定向服务器URL进行重定向。

#### [IPv6 P2P流量控制](#ZH-CN_TOPIC_0180136512)

P2P技术的逐步普及，占用了很大的网络带宽资源，并且很多P2P应用是对网络资源进行“恶意的”占用，导致网络不同程度的出现拥塞。IPv6 P2P流量控制是指为了确保用户IPv6业务正常使用，PGW-U/UPF通过带宽管理对IPv6 P2P流量进行控制。PGW-U/UPF通过三四层、七层的协议特征信息，或者流量的TOS值来区分业务。然后以业务为中心，在不同的作用范围内进行IPv6业务的流量控制。

IPv6 P2P流量控制的详细原理请参见 [GWFD-110311 基于业务感知的带宽控制](../../智能策略控制功能/GWFD-110311 基于业务感知的带宽控制_77219468.md) ，不同点在于以下几方面要支持IPv6：

- PGW-U/UPF支持IPv6 P2P流量的带宽管理配置，可以基于用户组、用户的不同作用范围进行层次化控制，也可以基于用户属性(RAT、ROAMING)、流量属性（流量方向、时段）的不同采用不同的带宽管理策略。
- PGW-U/UPF支持基于IPv6 P2P流量三四层、七层协议解析的带宽管理，这与IPv6 SA相关。
- PGW-U/UPF支持基于IPv6 P2P流量TOS值的带宽管理，IPv4中的TOS对应到IPv6的Traffic Class。

#### [内嵌IPv4地址的IPv6业务处理](#ZH-CN_TOPIC_0180136512)

随着网络中IPv4地址的耗尽，运营商需要部署IPv6业务，但在现网中部署的大量OTT SP（OTT业务的服务提供商）仅支持IPv4地址，需要运营商在OTT SP设备前部署NAT64转换设备，进行IPv4地址和IPv6地址的转换，还需部署DNS64设备，将域名解析返回的IPv4地址转换为IPv6地址。

PGW-U/UPF支持使用 [**SET SRVCOMMONPARA**](../../../../../OM参考/命令/UDG MML命令/用户面服务管理/业务匹配策略/业务匹配公共配置/业务公共参数管理/业务公共参数/设置业务公共参数（SET SRVCOMMONPARA）_82837309.md) 命令开启内嵌IPv4地址的IPv6业务处理功能，以实现对IPv6业务的策略控制，同时又节省了系统资源。PGW-U/UPF开启内嵌IPv4地址的IPv6业务处理功能后，各业务功能的处理如 [表1](#ZH-CN_TOPIC_0180136512__table_025EE349) 所示。

*表1 业务处理说明*

| 业务功能 | 处理说明 |
| --- | --- |
| IP重定向 | PGW-U/UPF将内嵌IPv4的IPv6业务报文匹配上IPv4的IP重定向策略后重定向到IPv4地址的服务器。 |
| Captive Portal | PGW-U/UPF根据内嵌IPv4的IPv6业务报文匹配上的IPv4 Captive Portal重定向策略，构造的重定向响应报文中携带IPv4对应的重定向服务器URL。如果匹配得到的重定向地址为IP地址，则对端设备可能无法处理，因此为确保正确触发重定向，建议在IP Farm下配置URL。 |
| URL重定向 | PGW-U/UPF根据内嵌IPv4的IPv6业务报文匹配上的IPv4重定向策略，构造的重定向响应报文中携带IPv4对应的重定向服务器URL。如果重定向策略中的URL地址为IP地址格式，则对端设备可能无法处理，因此为确保正确触发重定向，建议配置URL格式的重定向地址。 |
| 头增强 | PGW-U/UPF支持在HTTP/RTSP头中插入内嵌IPv4的IPv6 UE IP地址，此时要求对端设备支持对MS IPv6地址的处理。 |
| 业务触发专有承载创建 | PGW-U/UPF根据内嵌IPv4的IPv6业务报文匹配上的IPv4业务触发专有承载策略，构造的专有承载创建消息中的TFT，是根据命中的IPv4 filter增加前缀转换出来的。 |
| HTTP计费防欺诈 | PGW-U/UPF在进行计费防欺诈Proxy欺诈检测时，根据内嵌IPv4的IPv6业务报文匹配到的IPv4 Proxy地址来决定是否命中欺诈Proxy。 |
| DNS纠错 | 当IPv6业务报文匹配获取到第三方Platform的IPv4地址时，PGW-U/UPF构造IPv6的DNS响应报文发送给用户。 |
| FUI重定向 | PGW-U/UPF根据内嵌IPv4的IPv6业务报文匹配到的FUI重定向策略进行重定向处理。如果FUI重定向下发的重定向策略为IPv4的IP-farm，则构造的重定向响应中携带IPv4对应的重定向服务器URL。 |
| ACL | PGW-U/UPF根据内嵌IPv4的IPv6业务报文匹配到的ACL策略进行ACL控制。如果命中的IP重定向规则为IPv4规则，将业务报文重定向到IPv4地址的服务器。 |
