#### [业务场景](#ZH-CN_TOPIC_0000001422418374)

PDU会话建立发生的业务场景如下：

- UE需要与外部网络进行业务交互，比如用户开机完成注册后会建立默认会话。
- UE从4G EPS切换到5GS发起PDU会话建立。
- UE从非3GPP接入方式切换到3GPP接入方式发起PDU会话建立。

1. 消息交互：UE发送[PDU session establishment request](../../../../../../../快速入门/协议与流程/5G Core信令分析/消息及信元/非服务化接口协议/N1接口/相关消息解释/PDU session establishment request_76783928.md)给AMF。
  触发条件：UE需要与外部网络进行交互(比如上网、通话等)会建立PDU会话。
  关键信元：
  | S-NSSAI | 指示网络切片标识。 |
  | --- | --- |
  | DNN | 指示数据网络名称（Data Network Name）。 |
  | request type | 请求类型，比如initial request表示初始请求。 |
  | PDU Session ID | 指示PDU会话标识，标识新创建的1个PDU会话。 |
  | Integrity protection maximum data rate | 指示UE支持的上下行用户面完整性保护的最大速率。 |
  | SSC mode | 指示SSC模式。比如SSC模式1提供IP连续性，适合语音等强连接应用，SSC模式2不提供IP连续性，适合网页浏览等不强连接应用。 |
  | PDU session type | 指示PDU会话类型。比如IPv4、IPv6、IPv4IPv6等。 |
  相关机制：UE向AMF发送NAS消息，消息中包括S-NSSAI、DNN、PDU Session ID、PDU session establishment request等信息。
  相关特性：SSC模式相关参见UNC特性：WSFD-107015 支持SSC Mode1和UNC特性：WSFD-107016 支持SSC Mode2。
2. 内部流程：SMF选择流程。
  相关机制：AMF选择SMF的机制参见UNC章节：会话管理流程的网元选择。
3. 消息交互：AMF发送[Nsmf_PDUSession_CreateSMContext Request](../../../../../../../快速入门/协议与流程/5G Core信令分析/消息及信元/服务化接口协议/Nsmf/N11/Nsmf_PDUSession_CreateSMContext/Nsmf_PDUSession_CreateSMContext Request_34935460.md)给SMF。
  关键信元：
  | supi | 指示用户的supi信息。 |
  | --- | --- |
  | pduSessionId | 指示PDU会话标识。 |
  | dnn | 指示数据网络名称（Data Network Name）。 |
  | Guami | AMF的GUAMI。 |
  | sNssai | 指示网络切片标识。 |
  | requestType | 指示PDU会话类型。比如"INITIAL_REQUEST"表示建立新的PDU会话。 |
  | smContextStatusUri | 用于接收SM上下文状态通知消息的回调URI。 |
  | epsInterworkingInd | 指示PDU会话是否可能移动到EPS，以及EPS互操作过程中是否使用N26接口。 |
  相关机制：AMF向SMF发送消息请求创建SM上下文。
4. 内部流程：UDM选择流程。
  相关机制：如果SMF没有为该PDU会话进行注册，则SMF选择UDM并进行注册。SMF选择UDM的机制参见UNC章节：会话管理流程的网元选择。
5. 消息交互：SMF发送[Nudm_UECM_RegistrationSMF Request](../../../../../../../快速入门/协议与流程/5G Core信令分析/消息及信元/服务化接口协议/Nudm/N10/Nudm_UECM_Registration/Nudm_UECM_RegistrationSMF Request_07206934.md)给UDM。
  触发条件：如果SMF没有为该PDU会话进行注册，则SMF将通过该消息向UDM注册PDU会话。
  关键信元：
  | ueId | 携带用户标识，此处为SUPI。 |
  | --- | --- |
  | smfInstanceId | SMF的NF实例标识。 |
  | pduSessionId | 指示PDU会话标识。 |
  | singleNssai | 单个网络切片选择辅助信息。 |
  | plmnid | 服务节点的PLMN标识。 |
  | dnn | 指示数据网络名称（Data Network Name）。 |
6. 消息交互：UDM发送[Nudm_UECM_RegistrationSMF Response](../../../../../../../快速入门/协议与流程/5G Core信令分析/消息及信元/服务化接口协议/Nudm/N10/Nudm_UECM_Registration/Nudm_UECM_RegistrationSMF Response_06767502.md)给SMF。
  相关机制：如果请求成功，UDM发送“201 Created”响应消息，如果请求失败，由于用户有没对应的签约数据，接入禁止，或者漫游限制等原因导致操作无法授权，则返回HTTP状态码“403 Forbidden”。
7. 消息交互：SMF发送[Nudm_SDM_GetUESessionManagementSubscriptionData Request](../../../../../../../快速入门/协议与流程/5G Core信令分析/消息及信元/服务化接口协议/Nudm/N10/Nudm_SDM_Get/Nudm_SDM_GetUESessionManagementSubscriptionData Request_06927250.md)给UDM。
  关键信元：
  | supi | 携带用户标识。 |
  | --- | --- |
  | sm-data | 标识请求信息类型，此处为会话管理签约数据。 |

  | dnn | 指示数据网络名称（Data Network Name）。 |
  相关机制：SMF发送消息获取用户的会话管理签约数据。
8. 消息交互：UDM发送[Nudm_SDM_GetUESessionManagementSubscriptionData Response](../../../../../../../快速入门/协议与流程/5G Core信令分析/消息及信元/服务化接口协议/Nudm/N10/Nudm_SDM_Get/Nudm_SDM_GetUESessionManagementSubscriptionData Response_07167122.md)给SMF。
  关键信元：
  | singleNssai | 单个网络切片选择辅助信息。 |
  | --- | --- |
  | dnnConfigurations | 网络切片的附加DNN配置信息。 |
  | sscModes | 表示默认/允许的SSC模式。 |
  | iwkEpsInd | 表示是否支持EPS互操作。 |
  | 5gQosProfile | 表示数据网络会话关联的5G QoS参数。 |
  | 5qi | 标识一个标准的5G QoS特征组合，5QI值和对应的这组QoS特征一一映射。针对每个会话UDM会签约缺省的5qi。 |
  | arp | 通过ARP来控制QoS Flow的创建或修改的优先级。针对每个会话UDM会签约缺省的arp。 |
  | sessionAmbr | sessionAmbr限制了UE每个PDU会话的所有Non-GBR QoS Flow提供的聚合速率最大值。针对每个会话UDM会签约缺省的sessionAmbr。 |
  | staticIpAddress | 签约的UE的静态IP地址。 |
  相关机制：如果请求成功，UDM发送“200 OK”响应消息，如果请求失败，UDM没有UE的有效签约数据，或者UE签约数据存在，但请求的会话管理签约数据不可用（例如，查询参数中包含不属于UE数据中的single-nssai和/或dnn），则返回HTTP状态码“404 Not Found”。
  相关特性：QOS相关参数参见UNC章节：5G QoS关键参数及概念。
9. 消息交互：SMF发送[Nudm_SDM_SubscribeCreate Request](../../../../../../../快速入门/协议与流程/5G Core信令分析/消息及信元/服务化接口协议/Nudm/N10/Nudm_SDM_SubscribeCreate/Nudm_SDM_SubscribeCreate Request_07206954.md)给UDM。

  | singleNssai | 单个网络切片选择辅助信息。 |
  | dnn | 指示数据网络名称（Data Network Name）。 |
  相关机制：SMF发送该消息请求订阅变更通知。
10. 消息交互：UDM发送[Nudm_SDM_SubscribeCreate Response](../../../../../../../快速入门/协议与流程/5G Core信令分析/消息及信元/服务化接口协议/Nudm/N10/Nudm_SDM_SubscribeCreate/Nudm_SDM_SubscribeCreate Response_56087189.md)给SMF。
  关键信元：
  | Location | 创建成功的订阅信息的URI。后续当订阅者删除订阅信息时，使用此URI通知UDM。 |
  | --- | --- |

11. 消息交互：SMF发送[Nsmf_PDUSession_CreateSMContext Response](../../../../../../../快速入门/协议与流程/5G Core信令分析/消息及信元/服务化接口协议/Nsmf/N11/Nsmf_PDUSession_CreateSMContext/Nsmf_PDUSession_CreateSMContext Response_83735005.md)给AMF。

  | --- | --- |

  相关机制：SMF向AMF返回创建PDU会话结果，如果成功，SMF响应状态码“201 Created”，如果失败或重定向，SMF返回“4xx/5xx”的HTTP状态码。
12. 内部流程：PCF选择流程。
  相关机制：如果部署动态PCC，则SMF选择PCF进行策略控制，SMF选择PCF的机制参见UNC章节：会话管理流程的网元选择。
13. 消息交互：SMF发送[Npcf_SMPolicyControl_Create](../../../../../../../快速入门/协议与流程/5G Core信令分析/消息及信元/服务化接口协议/Npcf/N7/Npcf_SMPolicyControl_Create_67572324.md)给PCF。
  触发条件：如果部署动态PCC，则SMF选择PCF进行策略控制，如果没有部署动态PCC，则SMF采用本地策略获取PCC规则。
  关键信元：
  | supi | 表示用户永久标识。 |
  | --- | --- |
  | pduSessionId | 表示用户的PDU会话标识。 |
  | notificationUri | 表示PCF发送的SM策略更新通知的接收方。 |
  | pduSessionType | 表示用户的PDU会话类型。 |
  | sliceInfo | 标识S-NSSAI。 |
  相关特性：PCC功能参见UNC特性：WSFD-109101 PCC基本功能。
14. 消息交互：PCF发送[Npcf_SMPolicyControl_Create](../../../../../../../快速入门/协议与流程/5G Core信令分析/消息及信元/服务化接口协议/Npcf/N7/Npcf_SMPolicyControl_Create_67572324.md)给SMF。
  关键信元：
  | sessRules | 表示一组会话规则，是会话级QoS规则，用于进行PDU会话粒度的策略控制。 |
  | --- | --- |
  | authSessAmbr | 授权的会话AMBR。 |
  | authDefQos | 表示授权的默认QoS信息，包括5qi、arp等。 |
  | pccRules | 表示一组PCC规则，是业务级QoS规则，用于进行QoS Flow粒度的策略控制。会话可以对应多个pccRule和多个qosflow。 |
  | flowInfos | 表示一组IP数据流的过滤信息。 |
  | policyCtrlReqTriggers | 触发策略更新的事件。 |
  | qosDesc | 业务级QoS的详细内容，包括5qi、arp等。 |
  | chargingInfo | 表示计费信息，包含PDU会话的CHF地址。 |
  | ipv4Index | IPv4地址分配方式。 |
  | ipv6Index | IPv6地址分配方式。 |
  相关机制：如果请求被接受，PCF应响应“201 Created”HTTP状态码，如果失败或重定向时，PCF应返回HTTP状态码。
15. 内部流程：UPF选择流程。
  相关机制：SMF选择UPF的机制参见UNC章节：UPF选择原理。SMF根据UE在UDM中的签约信息为UE分配IP地址 ，也可以由UPF根据本地地址池为UE分配IP地址。IP地址分配机制参见UNC章节：5G Core 用户IP地址管理解决方案。如果SMF分配UE IP地址，则SMF触发到PCF更新策略，如果是UPF分配IP地址，则UPF分配完IP地址后再触发到PCF更新策略。
16. 消息交互：SMF发送[PFCP Session Establishment Request](../../../../../../../快速入门/协议与流程/5G Core信令分析/消息及信元/非服务化接口协议/N4接口/相关消息解释/PFCP Session Establishment Request_32945585.md)给UPF。
  关键信元：
  | CP F-SEID | 指示控制面即SMF侧SEID，是控制面分配的标识会话的唯一标识，并用于后续信令交互。 |
  | --- | --- |
  | Create PDR | 表示Packet Detection Rule ，用于详细指示UPF针对每一条QoS Flow的流处理策略。PDR由流规则与流动作组成。PDR区分上下行，通常有N3接口的PDR，称之为上行PDR，N6接口的PDR，称之为下行PDR。 |
  | PDR ID | 指示PFCP会话配置的PDR标识。详细可参见UNC章节：PDR是什么？都有哪些类型？ |
  | PDI | 指示匹配的报文。 |
  | Source interface | PDR区分上下行，比如取值为access表示上行PDR，即收到access侧的报文进行匹配。也即从N3接口收到的报文进行处理。比如取值为core表示下行PDR，即收到core侧的报文进行匹配。 |
  | Source interface type | 取值为n3 3gpp access，指示接口为N3接口。取值为n6，指示接口为N6接口。 |
  | Local F-TEID | 指示由SMF或UPF分配用户面的隧道端点标识，以及SMF分配的隧道端点标识。此隧道用于N3接口与RAN进行数据传输。详细可参见UNC章节：N4偶联建立要求和UNC章节：TEIDU分配要求。 |
  | SDF Filter | 用于对业务数据进行识别，分类。 |
  | UE IP Address | 指示SMF分配的UE IP地址。详细可参见UNC章节：SMF分配IP地址流程。 |
  | UE IP Address Pool Identity | 指示UPF分配IP地址的地址池名称。详细可参见UNC章节：UPF分配IP地址流程。 |
  | Create FAR | 表示Forwarding Action Rule，用于定义流转发类动作，包括流量转发、丢弃、缓存等。FAR同PDR一样区分上下行。 |
  | FAR ID | 指示PFCP会话配置的FAR标识。 |
  | Apply Action | 指示对数据包适用的动作。 |
  | Forwarding Parameters | 指示Apply Action为请求转发数据包，包含用户面功能需要应用的转发指令。 |
  | Destination interface | 取值为core表示目的接口为core侧，即上行FAR，收到上行的消息后转发至core侧。取值为access表示目的地为接入侧，即下行FAR，收到下行的消息后转发至access侧。 |
  | Destination interface type | 取值为n6，表示core侧接口为N6接口，取值为n3 3gpp access，表示access侧接口为n3接口。 |
  | Create URR | 表示Usage Reporting Rule ，用于定义统计类上报动作，如流量耗尽后上报、离线计费上报等。 |
  | Create QER | 表示QoS Enforcement Rule，用于定义流控类动作。通常SMF下发3类QER，1个Session级别QER，1类QosFlow级别的QER，1类标识QFI的QER。详细可参见UNC章节：N7接口与N4接口的信令映射。 |
  | QER ID | 指示PFCP会话配置的QER标识。 |
  | Gate Status | 指示数据包是否允许在上行和/或下行方向被允许转发或丢弃。 |
  | QoS Flow Identifier | 指示QoS Flow标识。 |
  | Create BAR | 表示Buffering Action Rule，用于定义对报文执行的缓存动作。 |
  相关机制：SMF给UPF发送此消息请求建立新的PFCP会话上下文，提供用于该PDU会话的数据监测，报告规则，CN隧道信息等。
  相关特性：SM策略相关内容参见UNC章节：N7接口与N4接口的信令映射。
17. 消息交互：UPF发送[PFCP Session Establishment Response](../../../../../../../快速入门/协议与流程/5G Core信令分析/消息及信元/非服务化接口协议/N4接口/相关消息解释/PFCP Session Establishment Response_85720872.md)给SMF。
