# 消息交互

**心跳检测**

SMF和UPF之间支持PFCP心跳检测和Echo心跳检测两种方式探测通信路径和对端设备是否工作正常。

PFCP心跳检测采用PFCP Heartbeat Request/PFCP Heartbeat Response消息探测通信路径。

在UPF与SMF建立关联时，UPF会周期性发送PFCP Heartbeat Request消息给SMF，并通过判断SMF返回的PFCP Heartbeat Response，确认通信路径和对端设备工作正常。如果在设定的时间周期内未收到SMF返回的PFCP Heartbeat Response，就判定与SMF间的通信路径故障或者SMF设备故障。UPF与SMF建立关联后，UPF会主动向SMF发送PFCP Heartbeat Request消息。

在SMF与UPF未建立关联时，SMF会向UPF发送PFCP Heartbeat Request消息，UPF向SMF返回PFCP Heartbeat Response消息。即在UPF重启已经丢失关联信息的情况下，SMF可以通过UPF返回的PFCP Heartbeat Response消息中的Recovery值，判断UPF是否已经重启。

> **说明**
> - 若**[SET CPACCESSLISTFUNC](../../../../../../OM参考/命令/UDG MML命令/用户面服务管理/路径管理/PFCP路径管理/CP节点管理/CP白名单开关配置/配置CP白名单开关（SET CPACCESSLISTFUNC）_86530396.md)**命令使能了SMF白名单功能，则UPF仅针对白名单范围内的SMF返回心跳响应。
> - 若UPF与SMF无关联建立，则UPF不主动发送PFCP Heartbeat Request消息。

Echo心跳检测采用Echo Request/Echo Response消息探测通信路径。

在UPF与SMF会话建立之后，UPF会周期性发送Echo Request消息给SMF，并通过判断SMF返回的Echo Response，确认通信路径和对端设备工作正常。如果在设定的时间周期内未收到SMF返回的Echo Response，就判定与SMF间的通信路径故障或者SMF设备故障。在UPF重启已经丢失会话的情况下，SMF可以通过UPF返回的Echo Response消息中的Recovery值，判断UPF是否已经重启。

> **说明**
> 若 [**SET UPGTPPATH**](../../../../../../OM参考/命令/UDG MML命令/用户面服务管理/路径管理/GTP路径管理/GTP协议参数管理/GTP路径全局参数/设置GTP路径相关属性（SET UPGTPPATH）_82837227.md) 命令使能了SMF白名单功能，则UPF仅针对白名单范围内的SMF发送心跳检测请求。

**交互消息**

| 消息类型 | 消息名称 | 方向 | 作用 |
| --- | --- | --- | --- |
| 数据面心跳检测消息 | Echo Request | SMF->UPF<br>UPF->SMF | 用于探测通信路径和对端设备是否工作正常。 |
| 数据面心跳检测消息 | Echo Response | SMF->UPF<br>UPF->SMF | 用于探测通信路径和对端设备是否工作正常。 |
| 信令面心跳检测消息 | PFCP Heartbeat Request | SMF->UPF<br>UPF->SMF | 用于探测通信路径和对端设备是否工作正常。 |
| 信令面心跳检测消息 | PFCP Heartbeat Response | SMF->UPF<br>UPF->SMF | 用于探测通信路径和对端设备是否工作正常。 |
| 节点管理消息 | PFCP Association Setup Request | SMF->UPF | 偶联建立，协商SMF和UPF之间的能力，如两端各自支持的功能。 |
| 节点管理消息 | PFCP Association Setup Response | UPF->SMF | 偶联建立，协商SMF和UPF之间的能力，如两端各自支持的功能。 |
| 节点管理消息 | PFCP Version Not Supported Response | SMF->UPF<br>UPF->SMF | 偶联建立的响应消息，不支持该PFCP版本。当收到对端网元发送的PFCP Association Setup Request消息后，若不支持对端网元的PFCP版本，可发送PFCP Version Not Supported Response给对端网元。 |
| 节点管理消息 | PFCP Association Update Request | SMF->UPF<br>UPF->SMF | 偶联更新，如：SMF支持的功能发生变化时，SMF发起偶联更新流程；UPF支持的功能发生变化、用户面可获得资源发生变化、配置变更、请求释放偶联时，UPF周期性发起偶联更新流程。<br>单偶联多路径场景下，UPF如果未通过PFCP Heartbeat感知到路径故障，仍会通过故障路径发送偶联更新消息，待PFCP Heartbeat感知到路径故障后切换路径进行周期性偶联消息更新。 |
| 节点管理消息 | PFCP Association Update Response | SMF->UPF<br>UPF->SMF | 偶联更新，如：SMF支持的功能发生变化时，SMF发起偶联更新流程；UPF支持的功能发生变化、用户面可获得资源发生变化、配置变更、请求释放偶联时，UPF周期性发起偶联更新流程。<br>单偶联多路径场景下，UPF如果未通过PFCP Heartbeat感知到路径故障，仍会通过故障路径发送偶联更新消息，待PFCP Heartbeat感知到路径故障后切换路径进行周期性偶联消息更新。 |
| 节点管理消息 | PFCP Association Release Request | SMF->UPF | 偶联释放，如：因为操作运维管理原因需要终结偶联。 |
| 节点管理消息 | PFCP Association Release Response | UPF->SMF | 偶联释放，如：因为操作运维管理原因需要终结偶联。 |
| 节点管理消息 | PFCP Node Report Request | UPF->SMF | 节点级信息上报，如：UPF的N3/N9接口链路故障信息上报给SMF。 |
| 节点管理消息 | PFCP Node Report Response | SMF->UPF | 节点级信息上报，如：UPF的N3/N9接口链路故障信息上报给SMF。 |
| 会话管理消息 | PFCP Session Establishment Request | SMF->UPF | 为用户PDU Session建立N4接口会话。 |
| 会话管理消息 | PFCP Session Establishment Response | UPF->SMF | 为用户PDU Session建立N4接口会话。 |
| 会话管理消息 | PFCP Session Modification Request | SMF->UPF | 更新PFCP会话，如：新增一个PDR、更新一个PDR、删除一个PDR，触发PFCP会话更新。 |
| 会话管理消息 | PFCP Session Modification Response | UPF->SMF | 更新PFCP会话，如：新增一个PDR、更新一个PDR、删除一个PDR，触发PFCP会话更新。 |
| 会话管理消息 | PFCP Session Deletion Request | SMF->UPF | 释放PFCP会话，如：PDU Session删除，触发PFCP会话删除。 |
| 会话管理消息 | PFCP Session Deletion Response | UPF->SMF | 释放PFCP会话，如：PDU Session删除，触发PFCP会话删除。 |
| 会话管理消息 | PFCP Session Report Request | UPF->SMF | 会话信息上报，如：用户流量配额用完，触发UPF通过该消息上报用户流量。 |
| 会话管理消息 | PFCP Session Report Response | SMF->UPF | 会话信息上报，如：用户流量配额用完，触发UPF通过该消息上报用户流量。 |

详细的消息及流程介绍请参见3GPP TS 29.244 V15.3.0。

**单偶联多路径处理**

N4接口支持单偶联多路径，单偶联路径最大支持16条。即一个偶联中，本端一个IP，对端最多支持16个IP。同一条偶联收到对端新IP头的节点与会话管理消息时，详细处理方式如 [表1](#ZH-CN_TOPIC_0268733395__table66954442577) 所示。

*表1 单偶联多路径消息处理*

| 消息名称 | 单偶联路径<16时消息处理 | 单偶联路径≥16时消息处理 |
| --- | --- | --- |
| PFCP Association Setup Request | 清除原来所有的IP地址，保留新IP地址。 | 清除原来所有的IP地址，保留新IP地址。 |
| PFCP Association Update Request | 新IP与本端IP在偶联中新建一条路径。 | 回复失败原因值No resources available。 |
| PFCP Association Release Request | 删除偶联。 | 删除偶联。 |
| PFCP Heartbeat Request | 偶联无变化。 | 偶联无变化。 |
| PFCP Session Establishment Request | 新IP与本端IP在偶联中新建一条路径。 | 回复失败原因值No resources available。 |
| PFCP Session Modification Request | 新IP与本端IP在偶联中新建一条路径。<br>CP F-SEID中IP刷新。 | 回复失败原因值No resources available。 |
| PFCP Session Deletion Request | 新IP与本端IP在偶联中新建一条路径。 | 会话去激活成功。 |
