#### [适用NF](#ZH-CN_CONCEPT_0205341699)

SGW-U、PGW-U、UPF

#### [定义](#ZH-CN_CONCEPT_0205341699)

系统过载控制功能又称为流控，此功能通过控制系统接入业务量来保证系统过载情况下业务的正常进行，避免造成整系统瘫痪。

#### [客户价值](#ZH-CN_CONCEPT_0205341699)

| 受益方 | 受益描述 |
| --- | --- |
| 运营商 | 提高大业务量情况下系统的可靠性，提升运营商网络的可靠性，运营商对用户分级别，对重要客户的业务采取重点保障。 |
| 用户 | 在业务高峰时段部分用户的业务仍可正常进行。 |

#### [应用场景](#ZH-CN_CONCEPT_0205341699)

实际运行中，有很多种场景可以造成 UDG 过载，此时需启动流控机制，保证 UDG 在过载情况下能够提供一定的业务处理能力，并且不会宕机，主要应用场景有：

- 激活速率超过UDG的承受能力。
- 短时间内大量用户激活、去激活。
- 某个APN的接入失败，直接导致UDG的整个系统的CPU占有率上升（例如此APN下用户不停的重发各种AAA鉴权或者计费消息等），从而影响其他APN的接入。

#### [对系统的影响](#ZH-CN_CONCEPT_0205341699)

UDG 在超出系统处理能力时，流控机制启动，会丢弃部分消息，导致部分用户激活失败或业务中断。

#### [原理概述](#ZH-CN_CONCEPT_0205341699)

**相关概念**

- 流控简介
  UDG 使用多个CPU，系统过载控制基于每个CPU单独控制，每个CPU上包含不同的线程，每类线程被分配了一定比例的CPU资源，当线程的CPU占用率达到设置的阈值时，系统即开始对该线程进行流控。
- UDG的流控算法
  UDG 的流控算法采用WAL（Windows Access Limit）算法。WAL算法是一种系统根据即时CPU占用率动态调整用户信令消息处理速率的流控算法。通过监控与限制CPU占有率，WAL算法使系统资源的使用达到可控，在系统过载时尽量使系统CPU占有率保持在预期目标区间（83%～85%）之内。WAL算法的流控效果如 [图1](#ZH-CN_CONCEPT_0205341699__fig1) 所示。

  **图1** WAL算法效果图

  <br>

  ![](GWFD-010251 系统过载控制特性概述_05341699.assets/zh-cn_image_0205341771.png)

  - 在系统没有过载时只做最大处理能力保护，即每秒处理信令消息数不允许超过最大处理能力。
    - 在系统进入轻度过载状态后（CPU占有率>70%），每秒处理业务量在上一周期实际业务量的基础上可以缓慢增加，直到CPU占有率>83%，允许业务量不再增加。
    - 当系统进入重度过载状态后（CPU占有率>85%），每秒处理业务量在上一周期实际业务量的基础上开始减少，一直到CPU占有率<90%，目标是使系统CPU占有率稳定在83%～85%之间。
  WAL算法是基于队列的流控算法，把用户信令区分不同的优先级，根据信令消息的优先级决策丢弃消息的比例。基本思想是首先保证在线用户的业务不受影响，其次保证用户尽可能的接入，根据CPU占有率动态调整信令消息的处理速率，同时还有一个基本原则，不做无效的处理，保证处理都是有效的，简单来说，比如处理100个用户的消息，但是最终每个用户的消息流程都不完整，导致最终每个消息的处理都是无效的，这样不如处理一个用户完整流程的消息。

**N4接口流控**

UDG 与SMF通过N4接口互联，需要支持自保流控。 UDG 上实现的对自身的流控保护只针对信令消息，主要是在系统过载的情况下减轻ISU的负担，保证高优先级业务的处理。自保流控包括：阈值流控，消息优先级流控及APN接入速率控制。

- 阈值流量
  将到达ISU的数据消息和信令消息区分开来，对于超出 ISU 处理能力的信令消息被丢弃。
- 消息优先级流控
  按照不同优先级处理信令消息。当系统过载时进行流控，即丢弃无法处理的信令消息，流控现象如 [表1](#ZH-CN_CONCEPT_0205341699__table_03C0DC79) 所示。

  *表1 信令消息处理优先级*

  | 处理优先级 | 消息类 |
  | --- | --- |
  | 高 | Modification类、Deletion类及Report类消息 |
  | 低 | Establishment类消息 |
  UDG 对以下信令消息进行流控：
    - PFCP Session Establishment Request
    - PFCP Session Modification Request
    - PFCP Session Deletion Request
    - PFCP Session Report Request
- APN接入速率控制
  控制APN的用户接入速率，保证APN下的用户平稳接入，不会因为此APN下的接入速率过快冲击 UDG 周边网元或者影响其他APN的接入速率。 通过 [**SET APNACCESSWAL**](../../../../../OM参考/命令/UDG MML命令/用户面服务管理/会话管理/会话信息管理/APN的接入速率属性配置/设置Apn接入速率配置（SET APNACCESSWAL）_06054797.md) 命令配置APN下用户的接入速率。

**N3接口流控**

eNodeB/gNodeB向SGW-U/PGW-U/UPF发送EPS承载/PDU会话的数据报文后，如果SGW-U/PGW-U/UPF上没有用户信息，SGW-U/PGW-U/UPF向eNodeB/gNodeB返回Error Indication消息，触发eNodeB/gNodeB通过MME/AMF通知SGW-C/PGW-C/SMF释放EPS承载/PDU会话，进而触发UE重建EPS承载/PDU会话。如果SGW-U/PGW-U/UPF发送Error Indication速率过快，可能导致产生大量EPS承载/PDU会话重建的流量，引起无线侧过载，故需要SGW-U/PGW-U/UPF针对发送的Error Indication消息 通过配置软参 [DWORD47](../../../../../OM参考/软件参数/UDG软件参数/用户面软件参数/软件参数/接入控制管理/DWORD47 发送Error Indication消息流控阈值_96403669.md) 做速率控制，避免由于发送Error Indication速率过快引起无线侧过载。
