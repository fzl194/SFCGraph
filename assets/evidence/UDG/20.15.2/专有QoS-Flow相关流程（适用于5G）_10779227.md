# 专有QoS Flow相关流程（适用于5G）

**专有QoS Flow建立**

专有QoS Flow建立由业务触发，当用户请求的业务数据流到达时，UPF将匹配到的QoS规则向SMF上报，SMF使用预定义规则中的QoS策略发起专有QoS Flow建立流程，如 [图1](#ZH-CN_TOPIC_0210779227__fig470211246496) 所示。

**图1** 专有QoS Flow建立流程

<br>

![](专有QoS Flow相关流程（适用于5G）_10779227.assets/zh-cn_image_0210791624.png)

其详细流程介绍如下：

1. **可选：**如果部署了动态PCC，PCF向SMF下发预定义规则。如果未部署动态PCC，则SMF在本地配置QoS策略规则。
  SMF通过向UPF发送PFCP Session Modification Request消息下发预定义规则名称，UPF将本地配置的规则内容和规则名称进行匹配之后，向SMF发送PFCP Session Modification Response消息。
  > **说明**
  > 在IPv4v6双栈用户使用专有承载功能时，当配置AnytoAny的Filter规则创建专有承载，UPF向SMF发送专有承载消息需要使用 SET UPGLBPMPARA 命令中FLOWDANYIPFMT配置具体的Flow-Description信息格式，以便SMF区分上报的消息是IPv4还是IPv6。
2. 当有上行数据流到达时，UPF对报文进行规则匹配，当匹配到预定义规则且规则中绑定QoS-URR时，向SMF发送PFCP Session Report Request消息，消息携带Usage Report信元组，包含URR ID、Rule Information。
  SMF收到消息后处理上报事件，基于QoS策略触发QoS Flow的更新操作，并向UPF下发PFCP Session Report Response消息，消息携带Update PDR、Update QER等信元组，包含PDR ID、URR ID、SDF等信元。
3. SMF向UPF发送PFCP Session Modification Request消息，携带QoS Flow Identifier、QER ID、Gate Status等信元，提供用于该专有QoS Flow的数据监测，报告规则，CN隧道信息等，UPF向SMF发送PFCP Session Modification Response响应。
4. SMF向AMF发送Namf_Communication_N1N2MessageTransfer（Namf_N1N2MessageTransfer Request）消息，携带的信息包括发送给(R)AN的N2 SM Information，其中包含QFI、 QoS Profile、CN Tunnel Info等信息，发送给UE的N1 SM Container，其中包含PDU Session Establishment Accept、Allocated IPv4 Address等信息，通知(R)AN和UE需要建立专有QoS Flow。
5. AMF向(R)AN发送PDU Session Resource Setup Request，请求中包含N2 SM Information，包含QFI、QoS Profile、CN Tunnel Info，发送给UE的N1 SM Container，包含PDU Session Establishment Accept、Allocated IPv4 Address。
6. (R)AN与UE发起信令交互，将SMF需要发送给UE的PDU Session ID、N1 SM Container消息转发至UE，请求UE建立专有QoS Flow，消息中包含PDU Session Establishment Accept、Allocated IPv4 Address。
7. (R)AN向AMF发送PDU Session Resource Setup Response，建立AN隧道信息，响应消息中包含AN Tunnel Info、 List of Accepted/Rejected QFI等。
8. AMF向SMF发送Nsmf_PDUSession_UpdateSMContext Request（Nsmf_Update_SMContext Request），将从(R)AN接收到的N2 SM Information转发给SMF。
9. SMF向UPF发送PFCP Session Modification Request，将AN隧道信息和相应的转发规则发送给UPF。UPF发送PFCP Session Modification Response。
10. SMF向AMF发送Nsmf_PDUSession_UpdateSMContext Response（Nsmf_Update_SMContext Response）。

**专有QoS Flow更新**

UPF向SMF上报QoS更新事件从而发起的专有QoS Flow更新流程如 [图2](#ZH-CN_TOPIC_0210779227__fig19507380526) 所示。

**图2** 伴随QoS更新的专有QoS Flow更新流程

<br>

![](专有QoS Flow相关流程（适用于5G）_10779227.assets/zh-cn_image_0210783884.png)

其详细流程介绍如下：

1. **可选：**如果部署了动态PCC，PCF向SMF下发预定义规则。如果未部署动态PCC，则SMF在本地配置QoS策略规则。
  SMF通过向UPF发送PFCP Session Modification Request消息下发预定义规则名称，UPF将本地配置的规则内容和规则名称进行匹配之后，向SMF发送PFCP Session Modification Response消息。
2. 当有上行数据流到达时，UPF对报文进行规则匹配，当匹配到预定义规则且规则中绑定QoS-URR时，向SMF发送PFCP Session Report Request消息，消息携带Usage Report信元组，包含URR ID、Rule Information。
  SMF收到消息后处理上报事件，基于QoS策略触发QoS Flow的更新操作，并向UPF下发PFCP Session Report Response消息，消息携带Update PDR、Update QER等信元组，包含PDR ID、URR ID、SDF等信元。
3. SMF通过Namf_Communication_N1N2MessageTransfer响应AMF，携带的信息包括发送给(R)AN的N2 SM Information，发送给UE的N1 SM Container，通知(R)AN和UE需要更新的QoS信息。
4. AMF向(R)AN发送PDU Session Resource Modify Request，请求中包含从SMF获得的N2 SM Information、N1 SM Container（PDU Session Modification Command）。
5. (R)AN与UE发起信令交互，将SMF需要发送给UE的PDU session ID、N1 SM Container（PDU Session Modification Command）消息转发至UE，请求UE更新QoS信息。
6. (R)AN向AMF发送PDU Session Resource Modify Response，包含N2 SM Information。
7. AMF通过Nsmf_PDUSession_UpdateSMContext Request（Nsmf_UpdateSMContext Request）把从(R)AN获得的N2 SM Information转发给SMF，通知SMF (R)AN已经更新QoS信息。SMF发送Nsmf_PDUSession_UpdateSMContext Response（Nsmf_UpdateSMContext Response）。
8. 当N4会话有修改时，SMF通过PFCP Session Modification Request消息更新N4会话，将更新的QoS信息通知专有QoS Flow修改流程涉及的UPF；UPF发送PFCP Session Modification Response进行响应。
9. UE向(R)AN发送包含PDU session ID、PDU Session Modification Complete的NAS消息，确认接受专有QoS Flow修改。
10. (R)AN将接收的NAS消息转发至AMF。
11. AMF通过Nsmf_PDUSession_UpdateSMContext Request（Nsmf_UpdateSMContext Request）将从(R)AN获得的PDU Session Modification Complete等信息发送至SMF，通知SMF UE已经更新QoS信息。SMF发送Nsmf_PDUSession_UpdateSMContext Response（Nsmf_UpdateSMContext Response）。
12. 当N4会话有修改时，SMF通过PFCP Session Modification Request消息更新N4会话，将更新的QoS信息通知专有QoS Flow更新流程涉及的UPF；UPF发送PFCP Session Modification Response进行响应。

**专有QoS Flow释放**

专有QoS Flow释放流程用于删除一个专有QoS Flow，当业务暂停一段时间，业务流老化时，UPF向SMF上报QoS事件从而发起专有QoS Flow释放流程。专有QoS Flow释放成功后，用户即不能使用该数据业务。专有QoS Flow释放流程如 [图3](#ZH-CN_TOPIC_0210779227__fig14727124553113) 所示。

**图3** 专有QoS Flow释放流程

<br>

![](专有QoS Flow相关流程（适用于5G）_10779227.assets/zh-cn_image_0210783882.png)

其详细流程介绍如下：

1. **可选：**如果部署了动态PCC，PCF向SMF下发预定义规则。如果未部署动态PCC，则SMF在本地配置QoS策略规则。
  SMF通过向UPF发送PFCP Session Modification Request消息下发预定义规则名称，UPF将本地配置的规则内容和规则名称进行匹配之后，向SMF发送PFCP Session Modification Response消息。
2. 当业务暂停一段时间，业务流老化时，或者业务暂停时间超过专有QoS创建时SMF下发给UPF的Deactivation Timer时，UPF向SMF发送PFCP Session Report Request消息，消息携带Usage Report信元组，包含URR ID、Rule Information。
  SMF收到消息后处理上报事件，向UPF下发PFCP Session Report Response消息，通知删除PDR、QER、URR等信息。
  > **说明**
  > 为了减少专有QoS Flow的频繁创建和删除，可以通过 [**SET QOSURRRPTCTRL**](../../../../../../OM参考/命令/UDG MML命令/用户面服务管理/业务控制策略/用户QOS控制/业务QOS控制/QoS类型URR上报控制/设置QoS URR上报的相关参数（SET QOSURRRPTCTRL）_15006423.md) 命令设置迟滞时间，业务流老化时，UPF延迟向SMF发送申请删除专有QoS Flow的PFCP Session Report Request消息。
3. SMF释放该专有QoS Flow分配的IP地址/前缀与相应的用户面资源。
    a. SMF向该专有QoS Flow对应的UPF发送PFCP Session Deletion Request。UPF将丢弃与该专有QoS Flow相关的数据包，并释放与N4会话相关的所有隧道资源和上下文。
      目前PFCP Session Deletion Request消息中不包含信元，通过消息主体中的Session Endpoint Identifier与PFCP Session Establishment Response中由UPF分配的唯一标识会话的UP-FSEID对应，表示释放对应会话相关的资源。
    b. UPF向SMF发送PFCP Session Deletion Response。
4. SMF向AMF发送Namf_Communication_N1N2MessageTransfer，携带的信息包含发送给(R)AN的N2 SM Resource Release Request，发送给UE的N1 SM Container，当专有QoS Flow的UP连接处在激活状态时，还包含发送给(R)AN的N2 Resource Release Request，通知(R)AN和UE释放专有QoS Flow。
5. 当UE处于空闲态并且指示需要发送N1 SM消息时，AMF指示网络侧发起Service Request流程向(R)AN发送PDU Session ID、N1 SM container；当UE处于连接态时，AMF向(R)AN发送N2 Resource Release Request，请求中包含从SMF接收的N2 SM Information、N1 SM Container（PDU Session Release Command）。
6. (R)AN与UE发起信令交互，释放与专有QoS Flow相关的AN资源。
7. (R)AN向AMF发送N2 SM Resource Release Ack。
8. AMF发送Nsmf_PDUSession_UpdateSMContext Request（Nsmf_UpdateSMContext Request），将N2 SM Resource Release Ack发送给SMF，通知SMF (R)AN已经释放相关专有QoS Flow资源。SMF向AMF发送Nsmf_PDUSession_UpdateSMContext Response（Nsmf_UpdateSMContext Response）。
9. UE向(R)AN发送PDU Session Release Ack（PDU Session Release Complete）确认已释放专有QoS Flow。
10. (R)AN向AMF发送N2 NAS Uplink Transport，携带PDU Session Release Ack（PDU Session Release Complete）。
11. AMF发起Nsmf_PDUSession_UpdateSMContext Request（Nsmf_UpdateSMContext Request）将PDU Session Release Ack（PDU Session Release Complete）发送给SMF，通知SMF UE已经释放相关专有QoS Flow资源。SMF向AMF发送Nsmf_PDUSession_UpdateSMContext Response（Nsmf_UpdateSMContext Response）。
12. SMF发起Nsmf_PDUSession_SMContextStatusNotify通知AMF该专有QoS Flow的SM上下文已经释放。AMF释放SMF ID 、PDU ID、 DNN、S-SNSSAI。
13. 如果此会话部署了动态PCC，SMF向PCF发起SM Policy Association Termination流程来删除专有QoS Flow相应的信息。
