# AH和ESP

- [AH](#ZH-CN_TOPIC_0262244157__1.3.2.1)
- [ESP](#ZH-CN_TOPIC_0262244157__1.3.10.1)
- [AH与ESP的差异](#ZH-CN_TOPIC_0262244157__1.3.19.1)

IPsec协议包含两种安全协议：AH（Authentication Header）和ESP（Encapsulating Security Payload）。

#### [AH](#ZH-CN_TOPIC_0262244157)

AH是IP认证头协议（IP协议号为51），提供数据源认证、数据完整性校验和防报文重放功能，它能保护通信免受篡改，但不能防止窃听，适合用于传输非机密数据。

**工作原理**

AH工作原理是在每个数据包上添加一个身份验证报文头（即AH Header），如 [图1](#ZH-CN_TOPIC_0262244157__fig10303123213401) 所示。

**图1** AH报文头

<br>

![](AH和ESP_62244157.assets/zh-cn_image_0270749817.png)

- Next Header（下一个头）：8 bit，表示AH报文头的下一个负载的类型。在隧道模式下，AH所保护的是整个IP包，该值是4（IPv4）、41（IPv6）或者6（TCP），表示IP-in-IP协议。
- Payload Length（负载长度）：8 bit，AH报文头长度。对于IPv6，报文头总长度必须为8位字节单位的倍数。
- Reserved（保留域）：16 bit，用于以后的扩展。发送方必须设置为0，接收方应该忽略该字段。
- SPI（安全参数索引）：32 bit，接收方根据报文中的SPI查找SA数据库，再找到对应的SA。SPI值“0”被保留用来表示“没有SA存在”。
- Sequence Number（序列号）：从1开始的32 bit单增序列号，不允许重复，唯一地标识了每一个发送数据包，为SA提供防重放保护。接收端校验该Sequence Number的数据包是否已经被接收过，如果已经接收过，则拒绝接收该数据包。
- Authentication Data（验证数据）：是一个长度可变的域，包含完整性校验值（Integrity Check Value，以下简称ICV）。该字段必须为32 bit（IPv4）的整数倍或64 bit的整数倍（IPv6）。

认证过程概括如下：

- 发送方：使用协商好的算法和密钥对IP报文（可变字段除外）进行计算得到一个消息摘要（即ICV），将该ICV填充到AH报文头中的“Authentication Data”字段中，随IP报文一起发送给接收方。
- 接收方：使用相同的算法和密钥对接收到的IP报文进行计算，将计算得到的结果与AH报文头中的“Authentication Data”字段进行比较，如果一致，则认为IP报文在网络传送过程中没有被非法篡改，否则会认为已经被篡改并丢弃。

**认证算法**

AH支持如下几种认证算法：

- SHA-1（Secure Hash Algorithm）：通过输入长度小于2<sup>64</sup>比特的消息，产生160比特的消息摘要。
- SHA2-256：通过输入长度小于2<sup>64</sup>比特的消息，产生256比特的消息摘要。
- SHA2-384：通过输入长度小于2<sup>128</sup>比特的消息，产生384比特的消息摘要。
- SHA2-512：通过输入长度小于2<sup>128</sup>比特的消息，产生512比特的消息摘要。
- SM3：通过输入长度小于2<sup>64</sup>比特的消息，产生256比特的消息摘要。
- MD5：通过输入长度小于2<sup>64</sup>比特的消息，产生128比特的消息摘要。

由于SHA2算法的消息摘要长于SHA-1，安全性更强，因此建议使用SHA2算法。

#### [ESP](#ZH-CN_TOPIC_0262244157)

ESP是IP封装安全载荷协议（IP协议号为50），提供数据源认证、数据完整性校验、防重放攻击、数据加密等功能。

**工作原理**

ESP的工作原理是在每一个数据包的标准IP包头后面添加一个ESP报文头，并在数据包后面追加一个ESP尾。与AH协议不同的是，ESP将需要保护的用户数据进行加密后再封装到IP包中，以保证数据的机密性。

ESP报文头包括：

- SPI（安全参数索引）：32 bit，接收方根据报文中的SPI查找SA数据库，再找到对应的SA。SPI值“0”被保留用来表示“没有SA存在”。
- Sequence Number（序列号）：从1开始的32 bit单增序列号，不允许重复，唯一地标识了每一个发送数据包，为SA提供防重放保护。接收端校验该Sequence Number的数据包是否已经被接收过，如果已经接收过，则拒绝接收该数据包。
- IV（Initial Vector，初始化向量）：某些加密算法需要使用IV，这种特殊加密算法必须定义IV的位置，IV是受保护数据的第一个8位组。对IV验证但不进行加密。

EPS报尾包括：

- Padding（填充字段）：0-255个字节。可以使用填充字段填充明文，以达到采用的加密算法要求的长度。不管加密算法要求如何，也可以要求填充字段来确保结果密文是1字节的整数倍。填充字段还可以用于隐藏有效载荷实际长度，支持信息流机密性。
- Padding Length（填充字段长度）：8 bit。接收端根据该字段长度去除数据中的扩展位。
- Next Header（下一个头）：8bit，指明了封装在载荷中的数据类型，例如6表示TCP。隧道模式下使用ESP，这个值就是4，标识IP-in-IP。
- Authentication Data（验证数据）：变长。包含数据完整性校验结果，通常是一个经过密钥处理的散列函数输出值。验证数据是可选字段，只有选择了验证服务时才会有该字段。

ESP认证过程与AH方式相同，但是认证报文的完整性检查部分只包括ESP报文头、传输层协议报头、应用数据和ESP报尾，不包括IP报头。因此，ESP不能保证IP报头不被篡改，其安全性没有AH方式高。

ESP加密部分包括上层传输协议信息、数据和ESP报尾。

当ESP认证和加密都被选择了，由于认证总是在加密后进行的，因此收到数据包后先检查数据的有效性，再解密。

**认证算法和加密算法**

ESP支持如下认证算法和加密算法

- 认证算法
    - SHA-1（Secure Hash Algorithm）：通过输入长度小于2<sup>64</sup>比特的消息，产生160比特的消息摘要。
    - SHA2-256：通过输入长度小于2<sup>64</sup>比特的消息，产生256比特的消息摘要。
    - SHA2-384：通过输入长度小于2<sup>128</sup>比特的消息，产生384比特的消息摘要。
    - SHA2-512：通过输入长度小于2<sup>128</sup>比特的消息，产生512比特的消息摘要。
    - SM3：通过输入长度小于2<sup>64</sup>比特的消息，产生256比特的消息摘要。
    - MD5：通过输入长度小于2<sup>64</sup>比特的消息，产生128比特的消息摘要。
  由于SHA2算法的消息摘要长于SHA-1，安全性更强，因此建议使用SHA2算法。
- 加密算法
    - AES-CBC-128（Advanced Encryption Standard Cipher Block Chaining 128）：使用128比特加密算法对IP报文进行加密。
    - AES-CBC-192（Advanced Encryption Standard Cipher Block Chaining 192）：使用192比特加密算法对IP报文进行加密。
    - AES-CBC-256（Advanced Encryption Standard Cipher Block Chaining 256）：使用256比特加密算法对IP报文进行加密。
    - AES-GCM-128（Advanced Encryption Standard Galois/Counter Mode 128）：使用128比特算法对IP报文进行加密和认证。
    - AES-GCM-256（Advanced Encryption Standard Galois/Counter Mode 256）：使用256比特算法对IP报文进行加密和认证。
    - SM4：使用128比特加密算法对IP报文进行加密。
    - AES-GMAC-128（Advanced Encryption Standard Galois Message Authentication Code 128）：使用128比特算法对IP报文进行认证。
    - DES（Data Encryption Standard）：使用64比特算法对IP报文进行加密。

#### [AH与ESP的差异](#ZH-CN_TOPIC_0262244157)

从功能来看：AH只提供认证，不提供加密方式；ESP可以提供认证和加密。

从认证强度来看：AH认证强于ESP，AH可以对报头和有效载荷进行认证，ESP只对有效载荷进行认证。

AH和ESP可以单独使用，也可结合使用。如果两种协议结合使用，则在发送方先对IP报文进行ESP封装，然后进行AH封装。接收方收到IP报文后，先进行AH解封装，再进行ESP解封装。
