# 实现原理

NB-IoT eDRX模式数据传输流程如 [图1](#ZH-CN_CONCEPT_0267487576__fig17320229125411) 所示，主要有以下四部分：

1. eNodeB支持eDRX，并通过系统消息SI告诉UE。
2. 如果UE支持eDRX，会在Attach/TAU请求消息中通知MME其支持eDRX，MME根据本地决策eDRX的寻呼周期，下发eDRX周期给eNodeB和UE。
3. SGW-U收到下行数据包后，发送PFCP Session Report Request消息给SGW-C，SGW-C通过DDN消息通知MME发起寻呼，MME在DDN ACK消息中携带DL Buffering Duration信元/DL Buffering Suggested Packet Count信元，通知SGW-C下行数据的缓存时长/缓存包数，SGW-C通过响应消息PFCP Session Report Response将下行数据的缓存时长/缓存包数带给SGW-U。
    - 下行数据缓存时长的获取方式：
      SGW-U收到SGW-C的PFCP Session Report Response消息，由SGW-U判断消息中是否携带DL Buffering Duration信元，如果没有携带，不需要刷新下行数据缓存时长，采用本地配置；如果消息中携带了DL Buffering Duration信元，则使用信元中携带的时长作为下行数据包的缓存时长。
    - 下行数据缓存包数的获取方式：
      下行数据缓存包数由命令 **[SET GLBDLLTBUFFER](../../../../../../OM参考/命令/UDG MML命令/用户面服务管理/会话管理/GTP隧道管理/全局下行数据长时间缓存/设置全局下行数据长时间缓存个数（SET GLBDLLTBUFFER）_86530268.md)** / **[SET APNDLLTBUFFER](../../../../../../OM参考/命令/UDG MML命令/用户面服务管理/会话管理/GTP隧道管理/APN下行数据长时间缓存/设置基于APN的下行数据长时间缓存配置（SET APNDLLTBUFFER）_86530338.md)** 的 “MAXBUFFERNUM” （即最大缓存包个数）参数值和N4接口消息下发的DL Buffering Suggested Packet Count的参数值共同控制。下行数据缓存包的取值如下：
          - “MAXBUFFERNUM”存在初始值，如果没有通过**[SET GLBDLLTBUFFER](../../../../../../OM参考/命令/UDG MML命令/用户面服务管理/会话管理/GTP隧道管理/全局下行数据长时间缓存/设置全局下行数据长时间缓存个数（SET GLBDLLTBUFFER）_86530268.md)**/**[SET APNDLLTBUFFER](../../../../../../OM参考/命令/UDG MML命令/用户面服务管理/会话管理/GTP隧道管理/APN下行数据长时间缓存/设置基于APN的下行数据长时间缓存配置（SET APNDLLTBUFFER）_86530338.md)**修改该参数的值，则最大缓存个数一直为初始值（取值为1）。
          - 如果通过**[SET GLBDLLTBUFFER](../../../../../../OM参考/命令/UDG MML命令/用户面服务管理/会话管理/GTP隧道管理/全局下行数据长时间缓存/设置全局下行数据长时间缓存个数（SET GLBDLLTBUFFER）_86530268.md)**/**[SET APNDLLTBUFFER](../../../../../../OM参考/命令/UDG MML命令/用户面服务管理/会话管理/GTP隧道管理/APN下行数据长时间缓存/设置基于APN的下行数据长时间缓存配置（SET APNDLLTBUFFER）_86530338.md)**修改了“MAXBUFFERNUM”：当N4接口没有下发值时，下行数据最大缓存个数为该参数取值；当N4接口有下发值时，取下发值和“MAXBUFFERNUM”两者的最小值作为下行最大缓存个数。
4. MME根据eDRX周期确定寻呼时刻，通过eNodeB向用户发起寻呼，寻呼成功后重新建立数据链路，SGW-U将缓存的下行数据传输给UE。
