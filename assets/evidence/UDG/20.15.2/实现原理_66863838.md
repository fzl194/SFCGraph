#### [内容解析](#ZH-CN_TOPIC_0166863838)

内容解析：对业务报文进行内容解析（SA功能）是实现内容计费的基础。借助SA功能， UDG 对通过的业务报文进行特征分析。三四层信息的解析，可以获得五元组（源IP、目的IP、源端口、目的端口和协议类型），七层信息的解析，可以获取URL等。详细信息可以参见 [GWFD-110101 SA-Basic](../../业务感知功能/GWFD-110101 SA-Basic_24082506.md) 。

#### [计费规则匹配](#ZH-CN_TOPIC_0166863838)

计费规则匹配，即指将解析业务报文得到的关键信息和内容计费规则进行比较，相符合的即匹配成功，并按照匹配成功的计费规则记录流量、时长等信息。

**计费规则匹配**

计费规则由UNC通过N4接口的Create PDR（Packet Detection Rule）信元组下发。相关子信元如 [表1](#ZH-CN_TOPIC_0166863838__table6427132202514) 所示。

*表1 Create PDR包含的主要子信元*

| 子信元 | 信元解释 |
| --- | --- |
| PDR ID | 唯一标识一个Create PDR。 |
| Precedence | 标识PDR优先级。 |
| PDI（Packet Detection Information） | 标识下发的匹配用户报文的动态规则，包含Source Interface、Local F-TEID 、Network Instance、UE IP address等。 |
| Activate Predefined Rules | 标识下发的预定义规则或规则组。用于激活本地配置的User Profile（<br>[**ADD USERPROFILE**](../../../../../OM参考/命令/UDG MML命令/用户面服务管理/业务匹配策略/业务匹配公共配置/业务规则管理/用户模板/增加用户模板（ADD USERPROFILE）_82837279.md)<br>）或Rule（通过<br>[**ADD RULE**](../../../../../OM参考/命令/UDG MML命令/用户面服务管理/业务匹配策略/业务匹配公共配置/业务规则管理/规则/增加规则（ADD RULE）_82837266.md)<br>命令配置）。 |

规则匹配时， UDG 会匹配所有PDR，包括动态PDR（PDR中未携带Activate Predefined Rules子信元）和预定义PDR（PDR中携带了Activate Predefined Rules子信元），其中预定义PDR关联本地配置的User Profile（ [**ADD USERPROFILE**](../../../../../OM参考/命令/UDG MML命令/用户面服务管理/业务匹配策略/业务匹配公共配置/业务规则管理/用户模板/增加用户模板（ADD USERPROFILE）_82837279.md) ）或Rule（通过 [**ADD RULE**](../../../../../OM参考/命令/UDG MML命令/用户面服务管理/业务匹配策略/业务匹配公共配置/业务规则管理/规则/增加规则（ADD RULE）_82837266.md) 命令配置）。匹配成功的规则中，优先级最高的规则被 UDG 使用。

- 若匹配成功的规则是动态PDR，则使用此动态PDR里的URR ID计费，此URR ID会关联到UNC中配置的RG+SID。
- 若匹配成功的规则是预定义PDR，则使用此预定义PDR里的URR ID+匹配中的预定义Rule里的URR ID计费，此URR ID会关联到UNC中配置的RG+SID。

其中User Profile中Rule的规则匹配的流程如 [图1](#ZH-CN_TOPIC_0166863838__fig193351330164714) 所示。

一个User Profile下包含一组Rule，每个Rule有不同的优先级，匹配时，优先使用高优先级的Rule进行匹配，当高优先级匹配失败时，再使用低优先级的Rule进行匹配。

每个Rule可以配置多种过滤条件，L34层Filter、L7层协议和L7层Filter，只有当所有过滤条件均匹配成功时，这个Rule才算匹配成功。各种过滤条件组合如下：

- Filter
- Filter + Protocol
- Filter + Protocol + L7Filter
- Protocol
- Protocol + L7Filter

#### [可选功能](#ZH-CN_TOPIC_0166863838)

**重定向计费**

UDG 上重定向应用场景有多种：IP重定向、用户Portal重定向、URL重定向、在线计费FUI重定向。这些重定向都应用了三四层重定向或七层重定向，重定向计费属于内容计费的范畴。

三四层重定向：IP重定向属于三四层重定向，通过设置MS/UE和目的服务器间的数据流费率RG进行计费。

七层重定向：用户Portal重定向、URL重定向、在线计费FUI重定向都属于七层URL重定向，目前 UDG 仅支持HTTP/WAP业务的URL重定向。七层重定向被触发时， UDG 会构造返回给MS/UE的HTTP 302重定向报文，以及发送给MS/UE和业务服务器的TCP RST拆链报文。之后，会转发到Portal Server的重定向报文。

- 对于HTTP 302报文和两条TCP RST报文，根据 [**SET URRGRPBINDING**](../../../../../OM参考/命令/UDG MML命令/用户面服务管理/业务匹配策略/业务匹配公共配置/业务规则管理/用户模板/设置用户模板的URR组绑定关系（SET URRGRPBINDING）_82837281.md) 命令的“重定向URR组名称”参数设置的URR ID进行计费。如果没有配置URR，则使用对应的业务流的URR进行计费。
- 对于重定向到Portal页面的报文，根据 [**ADD RULE**](../../../../../OM参考/命令/UDG MML命令/用户面服务管理/业务匹配策略/业务匹配公共配置/业务规则管理/规则/增加规则（ADD RULE）_82837266.md) 设置的到Portal页面数据流的URR ID进行计费。

对于在线计费用户余额耗尽FUI重定向，重定向报文免费通过。

> **说明**
> 如果FUI重定向同时配置了在线重定向费率和离线重定向费率，存在在线重定向费率配额不足导致无法对重定向报文计费，只有离线重定向费率计费的情况，造成重定向计费在线和离线不一致。 故推荐重定向费率只配置离线费率。

**关联URL计费**

UDG 可以通过配置 **[**ADD L7FILTER**](../../../../../OM参考/命令/UDG MML命令/用户面服务管理/业务匹配策略/业务过滤器管理/七层规则管理/七层过滤器/增加七层过滤器（ADD L7FILTER）_82837397.md)** 命令的 **Referer关联计费开关** 参数，控制从源页面链接出去的浏览业务按照源页面的计费费率进行内容计费，将源页面访问流量和链接到其他网站的页面访问流量统计到相同的内容计费实例中。

**FTP内容计费**

如果需要对FTP流量进行单独计费，则需要在 **[ADD PROTBINDFLOWF](../../../../../OM参考/命令/UDG MML命令/用户面服务管理/业务匹配策略/业务过滤器管理/流过滤器管理/流过滤器的协议绑定/增加流过滤器协议绑定关系（ADD PROTBINDFLOWF）_82837370.md)** 命令中配置 “协议名称” 参数为 **FTP** 。不能只是根据知名端口进行过滤。因为FTP协议的信令面与数据面分开，数据面使用的端口是随机的。
