# 专有承载相关流程（适用于2/3/4G）

**专有承载激活**

专有承载的激活由业务触发，当用户请求的业务数据流到达时，PGW-U将匹配到的QoS规则向PGW-C上报，PGW-C使用预定义规则中的QoS策略发起专有承载激活流程，如 [图1](#ZH-CN_TOPIC_0181656835__fig66281729134913) 所示。

**图1** 专有承载激活流程

<br>

![](专有承载相关流程（适用于2_3_4G）_81656835.assets/zh-cn_image_0187452372.png)

详细流程如下：

1. **可选：**如果部署了动态PCC，PCRF向PGW-C下发预定义规则。如果未部署动态PCC，则PGW-C在本地配置QoS策略规则。
  PGW-C通过向PGW-U发送PFCP Session Modification Request消息下发预定义规则名称，PGW-U将本地配置的规则内容和规则名称进行匹配之后，向PGW-C发送PFCP Session Modification Response消息。
2. 当有上行数据流到达时，PGW-U对报文进行规则匹配，当匹配到预定义规则且规则中绑定QoS-URR时，向PGW-C发送PFCP Session Report Request消息，消息携带Usage Report信元组，包含URR ID、Rule Information。
  PGW-C收到消息后处理上报事件，基于QoS策略触发专有承载的创建，并向PGW-U下发PFCP Session Report Response消息，消息携带Create PDR、Create QER等信元组，包含PDR ID、URR ID、SDF等信元。
3. PGW-C向SGW-C发送Create Bearer Request消息，消息包含：IMSI、PTI、EPS Bearer QoS、TFT、S5/S8 TEID、Charging Id、LBI（Linked EPS Bearer Identity）等信元。LBI是UE的缺省承载ID。
  SGW-C收到消息后，建立专有承载上下文，根据LBI关联缺省承载和专有承载。同时向SGW-U发送PFCP Session Modification Request消息。
4. SGW-C向MME发送Create Bearer Request消息。
5. MME向eNodeB发送Bearer Setup Request消息，消息包含：EPS Bearer Identity、EPS Bearer QoS、Session Management Request, S1-TEID。其中Session Management Request是MME构造的消息，消息包含：PTI、TFT、EPS Bearer QoS parameters (excluding ARP)、Protocol Configuration Options、EPS Bearer Identity、Linked EPS Bearer Identity (LBI)。
6. eNodeB将EPS Bearer QoS映射为Radio Bearer QoS，然后向UE发送RRC Connection Reconfiguration消息，消息包含：Radio Bearer QoS、Session Management Request、EPS RB Identity。
  UE保存该消息中的信息，并通过LBI关联专有承载和缺省承载。
7. UE向eNodeB响应RRC Connection Reconfiguration Complete消息，确认无线承载激活。
8. eNodeB向MME发送Bearer Setup Response消息，确认空口承载已激活，消息包含：EPS Bearer Identity, S1-TEID。
9. UE向eNodeB发送Direct Transfer消息，消息中携带UE NAS层构建的Session Management Response消息，消息包含：EPS Bearer Identity。
10. eNodeB向MME发送Session Management Response消息。
11. MME在收到[8](#ZH-CN_TOPIC_0181656835__li7)中的Bearer Setup Response消息和[10](#ZH-CN_TOPIC_0181656835__li9)中的Session Management Response消息后，向SGW-C响应Create Bearer Response消息确认承载激活，消息包含：EPS Bearer Identity、S1-TEID。
  SGW-C保存S1-U TEID信息，传递给SGW-U，SGW-U建立和eNodeB间的S1-U专有承载，同时将承载信息传递给SGW-C。
12. SGW-C向PGW-C响应Create Bearer Response消息确认承载激活，消息包含：EPS Bearer Identity、S5/S8-TEID。
  PGW-C收到消息后，保存S5/S8-TEID，建立专有承载。
13. PGW-C将承载信息传递给PGW-U。

**专有承载更新**

PGW-U向PGW-C上报QoS更新事件从而发起的专有承载更新流程如 [图2](#ZH-CN_TOPIC_0181656835__fig166068107506) 所示。

**图2** 伴随QoS更新的专有承载更新流程

<br>

![](专有承载相关流程（适用于2_3_4G）_81656835.assets/zh-cn_image_0210778338.png)

其详细流程介绍如下：

1. **可选：**如果部署了动态PCC，PCRF向PGW-C下发预定义规则。如果未部署动态PCC，则PGW-C在本地配置QoS策略规则。
  PGW-C通过向PGW-U发送PFCP Session Modification Request消息下发预定义规则名称，PGW-U将本地配置的规则内容和规则名称进行匹配之后，向PGW-C发送PFCP Session Modification Response消息。
2. 当有上行数据流到达时，PGW-U对报文进行规则匹配，当匹配到预定义规则且规则中绑定QoS-URR时，向PGW-C发送PFCP Session Report Request消息，消息携带Usage Report信元组，包含URR ID、Rule Information。
  PGW-C收到消息后处理上报事件，基于QoS策略触发专有承载的更新操作，并向PGW-U下发PFCP Session Report Response消息，消息携带Update PDR、Update QER等信元组，包含PDR ID、URR ID、SDF等信元。
3. PGW-C生成TFT并更新EPS Bearer QoS，然后给SGW-C发Update Bearer Request（EPS Bearer Identity，EPS Bearer QoS，APN-AMBR，TFT）。
4. SGW-C收到该消息后，给MME发Update Bearer Request（EPS Bearer Identity，EPS Bearer QoS，APN-AMBR，TFT），请求承载更新。
5. MME给eNodeB发Bearer Modify Request消息，请求更改承载，还包含一条NAS消息即Session Management Request消息。
6. eNodeB将已修改的EPS Bearer QoS映射为Radio Bearer QoS。并给UE发送RRC Connection Reconfiguration消息，同时Modify EPS Bearer Context Request消息也将会转发给UE。
7. UE给eNodeB发RRC Connection Reconfiguration Complete消息以确认无线承载的更改。
8. eNodeB给MME发送Bearer Modify Response（EPS Bearer Identity）消息以确认承载修改，并指示请求的EPS Bearer QoS是否能够被分配。
9. UE向eNodeB发送Direct Transfer消息，消息中携带UE NAS层构建的Session Management Response消息，消息包含：EPS Bearer Identity。
10. eNodeB转发Session Management Response消息给MME。
11. 当MME接收到第[8](#ZH-CN_TOPIC_0181656835__li7)步Bearer Modify Response消息以及第[10](#ZH-CN_TOPIC_0181656835__li9)步Session Management Response消息后，会发送Update Bearer Response（EPS Bearer Identity）消息给SGW-C以确认承载更新。
12. SGW-C通过向SGW-U发送PFCP Session Modification Request消息将承载更新信息传递给SGW-U。
13. SGW-C给PGW-C发Update Bearer Response（EPS Bearer Identity）以确认承载更新。
14. PGW-C通过PFCP Session Modification Request消息将承载更新信息传递给PGW-U。

**专有承载去激活**

承载去激活流程用于删除一个承载上下文，当业务暂停一段时间，业务流老化时，PGW-U向PGW-C上报QoS事件从而发起承载去激活流程。承载去激活成功后，用户即不能使用该数据业务。承载去激活流程如 [图3](#ZH-CN_TOPIC_0181656835__fig751815915112) 所示。

**图3** 承载去激活流程

<br>

![](专有承载相关流程（适用于2_3_4G）_81656835.assets/zh-cn_image_0210778339.png)

其详细流程介绍如下：

1. **可选：**如果部署了动态PCC，PCRF向PGW-C下发预定义规则。如果未部署动态PCC，则PGW-C在本地配置QoS策略规则。
  PGW-C通过向PGW-U发送PFCP Session Modification Request消息下发预定义规则名称，PGW-U将本地配置的规则内容和规则名称进行匹配之后，向PGW-C发送PFCP Session Modification Response消息。
2. 当业务暂停一段时间，业务流老化时，PGW-U向PGW-C发送PFCP Session Report Request消息，消息携带Usage Report信元组，包含URR ID、Rule Information。
  PGW-C收到消息后处理上报事件，向PGW-U下发PFCP Session Report Response消息，通知删除PDR、QER、URR等信息。
3. PGW-C决定发起承载去活过程，给SGW-C发Delete Bearer Request（EPS Bearer Identity）。
4. SGW-C向SGW-U发送PFCP Session Modification Request消息，请求释放承载上下文。
5. SGW-C给MME发Delete Bearer Request（EPS Bearer Identity）。
6. MME给eNodeB发送E-Rab Release Command（EPS Bearer Identity）消息请求去激活承载，还包含一条NAS消息即Deactivate EPS Bearer Context Request消息。
7. eNodeB给UE发Radio Bearer Release Request消息，请求释放无线承载。同时Deactivate EPS Bearer Context Request消息也会转发给UE。
8. UE给eNodeB回Radio Bearer Release Response消息，释放无线承载完成。eNodeB给MME发送E-Rab Release Response（EPS Bearer Identity）消息。
9. UE NAS层建立Deactivate EPS Bearer Context Accept消息，该消息包括EPS承载标识，然后将Deactivate EPS Bearer Context Accept消息发送给eNodeB。eNodeB转发Deactivate EPS Bearer Context Accept（EPS Bearer Identity）消息给MME，确认承载去激活。
10. MME删除相关专有承载上下文，并发送Delete Bearer Response（EPS Bearer Identity）消息给SGW-C，确认承载去激活。
11. SGW-C在收到MME的Delete Bearer Response后，向SGW-U发送PFCP Session Deletion Request，删除与该EPS专有承载相关的上下文。
12. SGW-C给PGW-C发Delete Bearer Response（EPS Bearer Identity）。
13. PGW-C通过向PGW-U发送PFCP Session Deletion Request消息释放承载信息。
