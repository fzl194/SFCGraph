# 实现原理

**UE发起的激活流程**

**图1** 激活流程图

<br>

![](实现原理_70759398.assets/zh-cn_image_0272462068.png)

1. UE完成Untrusted Non-3GPP接入网络的认证授权，获取Local IP地址。
2. UE使用Local IP向ePDG发起IKEv2鉴权和隧道建立流程（UE向ePDG发送CFG_Request消息）。鉴权成功后，ePDG从3GPP AAA/HSS获取用户的签约信息，如S2b接口使用的协议（如GTPv2）、APN-AMBR、Static QoS Profile、Trace Information(Trace Reference、Trace Type、Trigger Id、OMC Identity)等信息。
  > **说明**
  > 如果UE发给ePDG的CFG_Request消息中没有包含Remote IP信息，则说明这是一个初始附着流程，ePDG要执行PGW-C选择流程，UE应该在CFG_Request消息中携带地址类型（IPv4、IPv6或者IPv4v6）。
3. ePDG感知到终端发起PDN连接建立时，向PGW-C发送Create Session Request消息，消息中包含IMSI、APN、RAT type、ePDG TEID for control plane、PDN Address Allocation、EPS Bearer Identity、Default EPS Bearer QoS、ePDG Address for the user plane、ePDG TEID of the user plane、APN-AMBR、Selection Mode、Dual Address Bearer Flag、Charging Characteristics、UE Local IP Adress IE和UE UDP Port等信息。
  > **说明**
  > 其中RAT type指示非3GPP接入类型（如WiFi），PAA中的PDN Type由CFG_Request消息中请求的类型及E-UTRAN网络中签约的地址类型综合决定。
4. **可选：**PGW-C向PCRF发起IP-CAN会话建立流程。根据PCRF下发的PCC规则，PGW-C完成承载绑定，并用PCC规则响应PGW-C。
5. PGW-C向3GPP AAA/HSS发起用户鉴权过程。将PGW-C标识、S2b接口使用的协议和PGW-C IP地址通知给3GPP AAA。
  > **说明**
  > 如果3GPP AAA Server返回了static QoS Profile、Trace Information、APN-AMBR等参数，PGW-C会忽略这些参数，不进行处理。
6. PGW-C在HSS完成注册后，HSS通知MME更新PGW-C的地址，以便后续发生WiFi到LTE网络切换时，终端能找到同一个PGW-C地址进行接入，避免切换失败。
7. MME向HSS返回更新响应。
8. PGW-C向PGW-U发起PFCP Session Establishment Request消息。
9. PGW-U向PGW-C回复PFCP Session Establishment Reponse消息。
10. PGW-C向ePDG返回Create Session Response消息，消息中包含PDN GW Address for the user plane、PDN GW TEID of the user plane、PDN GW TEID of the control plane、PDN Address Allocation、EPS Bearer Identity、 EPS Bearer QoS、APN-AMBR、Charging ID、Cause等信息。
11. ePDG建立与UE间的IPSec隧道和与PGW-C间的GTP-U隧道的对应关系。
12. ePDG将PGW-C分配给UE的IP地址通过IKEv2流程传递给UE。
13. 用户数据在IPSec隧道和GTP隧道中进行转发。

**UE发起的更新流程**

UE的Local IP地址发生更新时，UE会向ePDG发起更新流程，ePDG向PGW-C发送Modify Bearer Request消息更新UE的WLAN地址，消息中包含UE Local IP和UE UDP Port IE等。

**UE/ePDG发起的去活流程**

**图2** 去活流程图

<br>

![](实现原理_70759398.assets/zh-cn_image_0272462067.png)

1. UE/ePDG发起去活，IKEv2隧道释放触发GTP-U隧道释放。
2. ePDG向PGW-C发送Delete Session Request消息，删除对应IKEv2隧道的PDN连接。消息中使用TEIDC标识具体的PDN连接。
3. PGW-C向PGW-U发起PFCP Session Deletion Request消息。
4. PGW-C通知3GPP AAA Server释放PDN连接。
5. **可选：**PGW-C向PCRF发起IP-CAN会话终止流程。
6. PGW-U向PGW-C回复PFCP Session Deletion Reponse消息。
7. PGW-C向ePDG回复Delete Session Response消息。
8. Untrusted Non-3GPP网络进行资源释放流程。

**PGW-C发起的去活流程**

**图3** 去活流程图

<br>

![](实现原理_70759398.assets/zh-cn_image_0272462065.png)

<br>

1. **可选：**在QoS策略变更场景下，PGW-C收到PCRF发起的IP-CAN session Termination流程。
2. PGW-C向ePDG发送Delete Bearer Request消息，消息中携带EBI或LBI以及去活原因等信息。
  > **说明**
  > 当去活整个PDN连接时，携带代表缺省承载的LBI，当去活指定专有承载时，携带该承载的EBI。
3. 如果PGW-C发起的是整个PDN连接的释放，则ePDG向WiFi网络发起IKEv2隧道的释放流程。
4. Untrusted Non-3GPP网络进行资源释放流程。
5. ePDG向PGW-C回复Delete Bearer Response消息。
6. 如果是整个PDN连接被释放，PGW-C通知3GPP AAA Server已经释放PDN连接。
