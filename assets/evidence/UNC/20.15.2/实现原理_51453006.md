#### [I-UPF选择策略](#ZH-CN_CONCEPT_0251453006)

- 移动性注册更新、Service Request、Xn Handover、N2 Handover流程，基于当前I-UPF（Intra-AMF的切换）或者老侧I-UPF（Inter-AMF的切换）是否存在，根据UE的位置信息存在不同的I-UPF选择策略，详细见[表1](#ZH-CN_CONCEPT_0251453006__table145712157349)：

*表1 I-UPF选择策略1*

| 场景 | 当前I-UPF是否存在 | UE所在区域是否在UPF服务范围 | UE所在区域是否在当前I-UPF服务范围 | I-UPF选择策略 |
| --- | --- | --- | --- | --- |
| 1 | 是 | 否 | 是 | 不变 |
| 2 | 是 | 否 | 否 | 变更 |
| 3 | 是 | 是 | 是 | 删除 |
| 4 | 是 | 是 | 否 | 删除 |
| 5 | 否 | 是 | 不涉及 | 不涉及 |
| 6 | 否 | 否 | 不涉及 | 新增 |

- 会话建立、4G到5G Handover流程、4G到5G移动性注册更新流程，均只涉及到I-UPF的新增，如[表2](#ZH-CN_CONCEPT_0251453006__table0467134161920)所示。

*表2 I-UPF选择策略2*

| 场景 | I-UPF是否存在 | UE是否在UPF服务区域 | UE是否在当前I-UPF服务区域 | I-UPF选择策略 |
| --- | --- | --- | --- | --- |
| 1 | 否 | 是 | 不涉及 | 不涉及 |
| 2 | 否 | 否 | 不涉及 | 新增 |

- 5G到4G Handover流程、5G到4G的TAU流程，只涉及到I-UPF的删除。
- 去注册流程、PDU会话修改、PDU会话释放，I-UPF都不变化。
- AN Release可能涉及I-UPF不变或者删除。
- 初始注册流程、周期注册更新、紧急注册都不涉及I-UPF的选择。

#### [异常处理](#ZH-CN_CONCEPT_0251453006)

当SMF/I-SMF在Service Request流程中选择I-UPF失败时，SMF/I-SMF对当前用户的PDU会话有以下两种处理方式，具体如何处理可通过命令 **[SET IUPFSELFAIL](../../../../../OM参考/命令/UNC MML命令/业务服务管理/会话管理/接入管理/UPF选择管理/I-UPF选择失败处理/设置I-UPF选择失败处理方式（SET IUPFSELFAIL）_59000295.md)** 配置。

- 释放会话：SMF/I-SMF先通知AMF拒绝该PDU会话的用户面建立请求，然后发送Namf_Communication_N1N2MessageTransfer Request消息并内置NAS删除消息（即PDU Session Release Command消息）来释放UE会话。其中NAS删除消息中携带的释放原因值可通过本配置命令**[**SET IUPFSELFAIL**](../../../../../OM参考/命令/UNC MML命令/业务服务管理/会话管理/接入管理/UPF选择管理/I-UPF选择失败处理/设置I-UPF选择失败处理方式（SET IUPFSELFAIL）_59000295.md)**中的“RELCAUSE”参数来指定。
- 保留会话：SMF/I-SMF保持当前用户PDU会话，并通知AMF拒绝该PDU会话的用户面连接建立请求。
