#### [适用NF](#ZH-CN_CONCEPT_0277897409)

MME

#### [定义](#ZH-CN_CONCEPT_0277897409)

随着物联网的广泛应用，eMTC终端被广泛应用在多种行业中，例如水、电表的上报、物流、智能停车等。目前运营商对于eMTC终端业务的接入控制较为薄弱，当大量的eMTC终端在很短的时间内同时发起业务（例如指定日期的水表上报），可能会造成系统过载并影响普通终端用户的业务接入。MME通过APN识别eMTC终端，并发送Back-off time信元来减缓eMTC终端业务的接入速度，确保普通终端业务的正常进行，预防系统进入拥塞状态。

#### [客户价值](#ZH-CN_CONCEPT_0277897409)

| 受益方 | 受益描述 |
| --- | --- |
| 运营商 | 在M2M终端业务突发的场景下，提高普通终端用户业务的成功率，提升设备承载的普通终端业务量和网络可靠性。 |
| 用户 | 提高普通终端用户业务的成功率和业务体验。 |

#### [应用场景](#ZH-CN_CONCEPT_0277897409)

触发本特性流控的场景可分为以下2类：

- 在特定的时间段内，大量eMTC终端会同时向网络发起接入请求，此时会导致系统过载。如：家用水表会在每月的特定日期向自来水公司上报用水量。
- 当应用服务器（如水表的数据服务器）等发生故障时，大量eMTC终端频繁的向网络发起接入请求，此时也会导致系统过载。

#### [对系统的影响](#ZH-CN_CONCEPT_0277897409)

本特性会拒绝部分eMTC终端消息，导致相应的业务受损。

#### [应用限制](#ZH-CN_CONCEPT_0277897409)

- 本功能的使用需要MME与SPGW-C同时开启基于延迟定时器的信令拥塞控制的License，两者配合才能实现基于延迟定时器的信令拥塞控制功能。若连接的网关中有一个或多个不支持此功能，则不能达到流控效果。
- eMTC终端非首次接入，且Attach Request消息未做完整性保护或完整性检查失败时，MME下发的Attach Reject不会做完整性保护，这可能导致Back-off timer不生效。

#### [原理概述](#ZH-CN_CONCEPT_0277897409)

**流控机制**

- **低优先级APN流控**：针对低优先级APN（如，eMTC终端业务配置的特定的APN），系统基于当前CPU负荷，自动调节接入消息允许处理速率，将用户接入速率控制在一定范围内。系统实时调节流控效果，将CPU控制在恢复门限65%到过载控制门限70%之间。并且，在发给终端的接入消息（Attach Request、Service Request、Control Plane Service Request）的拒绝响应中，携带Back off timer指示UE在定时器超时后再接入网络。
- **基于PGW-C Back-off timer的APN流控**：针对支持PGW-C Back-off APN流控的APN，当PGW-C或周边网元过载，根据PGW-C的指示，MME会限制eMTC终端业务接入，缓解网络拥塞，保证普通终端业务的成功率。

*表1 拥塞控制机制与流控消息*

| 消息类型 | 低优先级APN流控 | 基于PGW-C Back-off timer的APN级流控 |
| --- | --- | --- |
| Attach Request | √ | √ |
| Service Request | √ | - |
| PDN Connectivity Request | - | √ |

**WAL算法**

低优先级APN流控时，MME根据系统CPU占用率实时调节流控效果，详情请参考如下 **WAL算法** ：

- WAL算法是一种基于实时系统CPU占用率来动态调整系统允许通过的最大和最小业务量的流控算法。
- WAL算法的应用使系统CPU资源的使用可控。
- 在本特性流控时，当系统CPU占用率大于CPU过载控制门限时，系统通过WAL算法来控制业务消息的数量，使系统CPU占用率保持在恢复门限到过载门限之间。
- 由于系统CPU占用率是实时波动的，对于造成系统CPU占用率超过过载门限的消息通过WAL算法被丢弃，而在系统CPU占用率低于恢复门限的部分则说明此时系统流控力度过强、丢弃消息过多，需要通过WAL算法减少实时消息丢弃量使CPU占用率提升到恢复门限，来保证业务稳定性。假设恢复门限为65%，过载门限为70%时，WAL算法如下图所示：

**流控实现原理：**

- **低优先级APN流控** ：通过配置命令 [**SET LOWPRIAPNFC**](../../../../../../OM参考/命令/UNC MML命令/业务服务管理/Pre 5G接入业务管理/控制面管理/操作维护/设备管理/流控管理/业务流控管理/低优先级APN流控管理/设置低优先级APN流控参数(SET LOWPRIAPNFC)_72345757.md) 中的 “FCSWITCH” 参数为 “ON” 来开启 **“APNNI组号”** 对应的APN的流控开关。配置 [**SET LOWPRIAPNFC**](../../../../../../OM参考/命令/UNC MML命令/业务服务管理/Pre 5G接入业务管理/控制面管理/操作维护/设备管理/流控管理/业务流控管理/低优先级APN流控管理/设置低优先级APN流控参数(SET LOWPRIAPNFC)_72345757.md) 中的参数 “MINTHD” 和 “MAXTHD” 来设定该低优先级APN的用户接入速率区间。MME发送给UE的Back-off timer的值为 [**SET LOWPRIAPNFC**](../../../../../../OM参考/命令/UNC MML命令/业务服务管理/Pre 5G接入业务管理/控制面管理/操作维护/设备管理/流控管理/业务流控管理/低优先级APN流控管理/设置低优先级APN流控参数(SET LOWPRIAPNFC)_72345757.md) 中的 “Back off timer最小值” 到 “back of timer最大值” 之间的随机值。系统根据基于当前CPU负荷，自动调节该低优先级APN的接入消息允许处理速率，实时调节流控效果，将CPU控制在恢复门限65%到过载控制门限70%之间。

- **基于PGW Back-off timer的APN级流控**：当本特性的开关在MME和SGW-C/PGW-C上都打开时（命令[**SET PGWBACKOFF**](../../../../../../OM参考/命令/UNC MML命令/业务服务管理/Pre 5G接入业务管理/控制面管理/操作维护/设备管理/流控管理/业务流控管理/P-GW Backoff流控管理/设置P-GW Back-off流控参数(SET PGWBACKOFF)_72345761.md)中的“FCSWITCH”参数为“ON”），PGW会启动控制机制，且下发Create Session Response携带APN congestion原因值和PGW Back-Off timer信元，通知MME限制eMTC终端接入。MME会在0到PGW Back-Off timer之间随机选取UE Back-off timer的值发送给UE，延迟UE的接入。
    - 若流控效果不理想，PGW eMTC终端APN拥塞未解除时，系统每10秒（一个统计周期）内收到PGW发来的APN congestion原因值，会自动下调5%的通过率。
      若通过率下降至0（流控机制依然运行），系统在当前统计周期内不会再收到APN congestion原因值。在下一至两个统计周期后，系统会尝试上升3%的通过率。
    - 若流控效果理想，PGW eMTC终端APN拥塞恢复时，系统一个统计周期内未收到PGW发来的APN congestion原因值，系统会自动上升3%的通过率。
      若通过率上升至100%，且通过率连续6个统计周期都稳定在100%，流控解除。

> **说明**
> 在PGW Back-Off timer超时后，对于新接入的Attach Request 或PDN Connectivity Request 消息：
>
> 1. 系统会按正常的网关选择功能选择相应的网关进行接入。
> 2. 选中的网关必须在命令[**SET PGWBACKOFF**](../../../../../../OM参考/命令/UNC MML命令/业务服务管理/Pre 5G接入业务管理/控制面管理/操作维护/设备管理/流控管理/业务流控管理/P-GW Backoff流控管理/设置P-GW Back-off流控参数(SET PGWBACKOFF)_72345761.md)配置的白名单范围内。
