SGSN

#### [定义](#ZH-CN_CONCEPT_0185628479)

Smartphone控制基础功能包括基于信令行为控制、基于终端类型控制和SmartPaging三个部分：

- 基于信令行为的Smartphone控制功能
  通过分析信令识别Smartphone，系统利用禁止使用Direct Tunnel和禁止启用去激活空闲PDP功能的机制来节省信令资源。
- 基于终端类型的Smartphone控制功能
  通过识别Smartphone终端类型，系统针对不同的智能终端实行差异化策略，利用禁止某一类智能终端使用Direct Tunnel功能的机制来节省信令资源。
- SmartPaging功能
  SGSN解析下行消息中的DSCP(Differentiated Services Code Point)字段，并根据下行数据包的优先级以及设备本身的拥塞情况决定是否下发Paging消息，根据实际情况丢弃低优先级的下行包，确保高优先级的业务正常处理。

#### [客户价值](#ZH-CN_CONCEPT_0185628479)

| 受益方 | 受益描述 |
| --- | --- |
| 运营商 | - 避免网络拥塞和设备过载。<br>- 提升网络可用性和可靠性。 |
| 用户 | 终端设备根据控制策略进行网络服务，减少终端设备电量消耗。 |

#### [应用场景](#ZH-CN_CONCEPT_0185628479)

在UTRAN网络中，Smartphone的应用使网络信令日益增加，利用Smartphone控制基础功能可以避免网络拥塞和设备过载现象的发生。

#### [原理概述](#ZH-CN_CONCEPT_0185628479)

**基于信令行为的Smartphone控制功能**

UNC 通过记录和分析用户的历史信令行为来识别用户是否为智能终端，以推断用户的信令资源需求。对智能终端信令行为的分析主要是对智能手机发起的Service request次数的分析。

基于信令行为的Smartphone控制功能的具体过程如下：

1. 系统通过命令 [**SET SMARTCFG**](../../../../../OM参考/命令/UNC MML命令/业务服务管理/Pre 5G接入业务管理/控制面管理/业务安全管理/Smartphone管理/Smartphone控制基础功能/设置智能用户功能（SET SMARTCFG）_26145750.md) 中的参数 “SMARTSW” 取值来控制是否启用Smartphone用户识别功能。
    - 如果启用识别功能，则执行[2](#ZH-CN_CONCEPT_0185628479__zh-cn_concept_0130429015_p124955156180641)。
    - 如果不启用识别功能，则基于信令行为的Smartphone控制功能不生效。
2. 系统通过命令 [**SET SMARTCFG**](../../../../../OM参考/命令/UNC MML命令/业务服务管理/Pre 5G接入业务管理/控制面管理/业务安全管理/Smartphone管理/Smartphone控制基础功能/设置智能用户功能（SET SMARTCFG）_26145750.md) 中的参数 “SERVREQTHRESHOLD” 设置每小时用户Service Request门限次数来识别用户是否为Smartphone用户。如果用户一小时内发起的Service Request次数大于等于该门限值，则此用户为Smartphone用户。此参数的默认值为10次/小时。
    - 如果识别用户是Smartphone用户，则可以选择性执行[3](#ZH-CN_CONCEPT_0185628479__zh-cn_concept_0130429015_step3)。
    - 如果识别用户不是Smartphone用户，则基于信令行为的Smartphone控制功能不生效。
3. 确定终端用户为Smartphone用户后，系统可启用以下功能：
    - 禁止该用户使用Direct Tunnel功能
      系统通过命令 [**SET SMARTCFG**](../../../../../OM参考/命令/UNC MML命令/业务服务管理/Pre 5G接入业务管理/控制面管理/业务安全管理/Smartphone管理/Smartphone控制基础功能/设置智能用户功能（SET SMARTCFG）_26145750.md) 中的参数 “DTSW” 控制是否禁止Smartphone用户使用Direct Tunnel。
      禁止Smartphone终端使用Direct Tunnel，而使用Two Tunnel，可以减少终端在连接态和空闲态之间频繁切换而产生的大量Update PDP消息（Gn接口消息），从而减少对GGSN信令的冲击。
    - 禁止启用去激活空闲PDP功能
      系统通过命令 [**SET SMARTCFG**](../../../../../OM参考/命令/UNC MML命令/业务服务管理/Pre 5G接入业务管理/控制面管理/业务安全管理/Smartphone管理/Smartphone控制基础功能/设置智能用户功能（SET SMARTCFG）_26145750.md) 中的参数 “DEAPDPSW” 控制是否禁止去激活Smartphone用户的空闲PDP。
      由于Smartphone的永远在线特点，在 UNC 去激活空闲PDP后，Smartphone会重新激活PDP。Smartphone终端不释放其空闲PDP，可以减少此去激活/激活流程的频繁发生，从而减少对 UNC 信令的冲击。
      > **说明**
      > UNC 为保证PDP资源的有效利用，会根据License和数据配置决定是否去激活长时间没有流量的空闲PDP。

**基于终端类型的Smartphone控制功能**

UNC 通过设置与Smartphone基本信息相关联的数据库来识别用户的终端类型，以针对不同的终端类型用户进行差异化控制。

- 识别Smartphone终端类型
  系统通过 [**ADD IMEILIB**](../../../../../OM参考/命令/UNC MML命令/业务服务管理/Pre 5G接入业务管理/控制面管理/业务安全管理/Smartphone管理/终端类型识别/IMEI库管理/增加IMEI库记录（ADD IMEILIB）_26145734.md) 命令建立IMEI（International Mobile Equipment Identity）库或通过 [**ADD APNNILIB**](../../../../../OM参考/命令/UNC MML命令/业务服务管理/Pre 5G接入业务管理/控制面管理/业务安全管理/Smartphone管理/终端类型识别/APNNI库管理/增加APNNI库记录（ADD APNNILIB）_26145736.md) 命令建立APN NI（Access Point Name Network Identifier）库来识别终端类型：
    - IMEI库
      系统内部建立各Smartphone终端类型与用户IMEI的对应关系数据库。移动性管理中，系统在用户首次发起的附着、RAU、Inter/Intra SGSN Relocation流程中获得用户IMEI信息，通过查询配置的IMEI库识别终端类型。如果系统在用户后续发起的附着、RAU、Inter/Intra SGSN Relocation流程中，获取到的IMEI和之前流程中获取的不同，则系统将重新查询配置的IMEI库识别终端类型。另外，系统也会在用户多次完成激活流程后，自动重新查询配置的IMEI库识别终端类型。
      > **说明**
      > IMEI的组成结构为TAC（Type Allocation Code）+SNR（Serial Number）+Spare digit，用于唯一标识一个用户终端。
    - APN NI库
      系统内部建立各Smartphone终端类型与用户APN对应关系的数据库。移动性管理中，系统通过附着、RAU、Handover等流程获得用户签约的APN；会话管理中，系统通过PDP上下文激活流程获得用户请求的APN，并且查询配置的APN NI库识别终端类型。
  > **说明**
  > IMEI库和APN NI库两者可以同时配置，优先级从高到低依次为：签约APN，IMEI，请求APN。
  系统可识别出的终端类型如下：
    - 4种已知类型：Android、Blackberry、iOS、Windows
    - 12种扩展类型：Custom Type 1～Custom Type 12
- 禁止基于终端类型的Direct Tunnel
  [图2](#ZH-CN_CONCEPT_0185628479__zh-cn_concept_0130429015_fig3) 是禁止基于终端类型的Direct Tunnel功能的实现流程。

  禁止基于终端类型的Direct Tunnel功能的具体过程如下：
    1. 本功能实现的前提是系统内部已经建立IMEI库/APN NI库。
    2. 获取用户信息后，系统通过查询IMEI库/APN NI库识别用户终端类型。
          - 如果识别到有效的终端类型，执行[3](#ZH-CN_CONCEPT_0185628479__li158851915194)。
          - 如果没有识别到有效的终端类型，但仍需要禁止终端的Direct Tunnel功能，则可通过[基于信令行为的Smartphone控制功能](#ZH-CN_CONCEPT_0185628479__zh-cn_concept_0130429015_sub1_2)功能实现。
    3. 系统根据命令 [**ADD SMARTDT**](../../../../../OM参考/命令/UNC MML命令/业务服务管理/Pre 5G接入业务管理/控制面管理/业务安全管理/Smartphone管理/基于终端类型的DT限制/增加基于终端类型的DT限制(ADD SMARTDT)_26145738.md) 中的参数 “DTLIMIT” 控制是否禁止Smartphone用户使用Direct Tunnel。
      此功能作用与 [基于信令行为的Smartphone控制功能](#ZH-CN_CONCEPT_0185628479__zh-cn_concept_0130429015_sub1_2) 作用相同，使用Two Tunnel，减少对GGSN信令的冲击。

**SmartPaging功能**

**图3** 业务识别及优先级

<br>

![](特性概述_85628479.assets/zh-cn_image_0193235814_2.png)

<br>

如 [图3](#ZH-CN_CONCEPT_0185628479__zh-cn_concept_0130429015_fig20) 所示为SmartPaging的业务识别和优先级排序。

SmartPaging对下行业务数据包进行如下识别和排序：

1. GGSN利用SA（Service Awareness）来识别业务类型并根据策略确定业务优先级，然后通过GTPU净荷（协议封装前的用户数据内容）里面的IP包头TOS字段传给SGSN。
2. SGSN根据当前的负荷情况按照TOS字段的业务类型决定是否发起寻呼，对需要寻呼的业务划分高优先级和低优先级，并按照一定的调度算法对寻呼次数进行流控:
    - 2G：直接从数据包里获取业务优先级信息
    - 3G：在信令处理模块获取业务优先级信息

使用SmartPaging后会带来如下好处：

- 部署VoIP业务的网络时业务能够得到优先调度。
- SGSN的信令流程触发的寻呼流程可以进行优先级排序。
