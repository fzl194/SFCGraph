#### [适用NF](#ZH-CN_TOPIC_0185005558)

SGSN

#### [定义](#ZH-CN_TOPIC_0185005558)

SGSN支持用户发起的二次PDP上下文激活。二次PDP上下文激活是为了重用已激活的PDP上下文资源，如APN和PDP地址。二次激活的会话与一次会话的不同在于采用了不同的QoS。

#### [客户价值](#ZH-CN_TOPIC_0185005558)

| 受益方 | 受益描述 |
| --- | --- |
| 运营商 | 不同业务使用不同QoS等级的承载，提高了SGSN的资源的利用效率，节省GGSN的IP地址资源。 |
| 用户 | 不同业务使用不同QoS等级的承载，降低了使用费用。 |

#### [应用场景](#ZH-CN_TOPIC_0185005558)

当用户在终端上运行多个应用程序，且使用单APN与IP地址时，因为不同业务使用不同QoS等级的承载，需要使用终端侧二次PDP上下文激活功能。例如：用户需要一边浏览网页，一边下载文件时，主PDP用于浏览网页，二次PDP用于传送媒体流。

#### [原理概述](#ZH-CN_TOPIC_0185005558)

**图1** 主PDP和二次PDP上下文的关系

<br>

![](WSFD-106001 终端侧二次PDP上下文激活特性概述_85005558.assets/zh-cn_image_0193105310_2.png)

<br>

从 [图1](#ZH-CN_TOPIC_0185005558__zh-cn_concept_0130429208_fig1) 中可以知道，主PDP（Primary PDP Context 2）与两个二次PDP上下文使用相同的APN（APN 2）和相同的IP地址（IP address 2），但是QoS不同，分别是QoS 2、QoS 4、QoS 5。

SGSN可以支持多个主PDP及多个二次PDP上下文，每个用户最多能支持11个PDP上下文。

**终端侧二次PDP上下文激活流程**

如 [图2](#ZH-CN_TOPIC_0185005558__zh-cn_concept_0130429208_fig2) 所示为 UNC 作为SGSN时，3G网络的终端侧二次PDP上下文激活流程，2G网络的流程类似。

**图2** 终端侧二次PDP上下文激活流程

<br>

![](WSFD-106001 终端侧二次PDP上下文激活特性概述_85005558.assets/zh-cn_image_0193105309_2.png)

<br>

具体流程如下：

1. MS发起主PDP上下文激活流程，激活成功。
2. MS发起二次PDP上下文激活流程，向SGSN发送“Activate Secondary PDP Context Request”消息，消息中携带了需关联的事务标识TI (Linked-Transaction Identity)以及用户期望的QoS。
3. SGSN根据消息中的关联TI，定位到MS已激活的主PDP上下文，从中获取APN与PDP地址，不需要再对APN与PDP地址进行处理，直接供本次PDP上下文激活使用。SGSN然后结合网络质量的情况，分配空口带宽和GTP隧道资源，并向GGSN发送“Create PDP Context Request”消息。GGSN向SGSN反馈“Create PDP Context Response”消息，消息中携带了GGSN调整后的QoS参数。
  > **说明**
  > 二次PDP上下文激活时的GGSN与主PDP上下文激活时的GGSN相同。
4. SGSN通过Radio Access Bearer Setup过程，与UTRAN协商最终的QoS参数。
5. 若SGSN与UTRAN协商的最终QoS参数与GGSN调整后的QoS参数不一致，则SGSN向GGSN发送“Update PDP Context Request”请求，消息中携带了最终协商的QoS参数。GGSN向SGSN返回“Update PDP Context Response”消息，消息中携带了最终协商的QoS参数。
6. SGSN向MS发送“Activate Secondary PDP Context Accept”消息，携带最终协商的QoS参数。MS的二次PDP上下文激活成功。

> **说明**
> 在去激活二次PDP上下文时，用户或者网络侧发送Deactivate PDP context request消息，消息中携带TI信元。通过判断消息中是否携带Tear down标识，SGSN对二次PDP上下文的释放有不同的处理：
>
> - 若消息中携带了Tear down标识，SGSN去激活所有与TI关联的主PDP上下文及与之共享同一IP地址和APN的二次激活的PDP上下文。
> - 若消息中未携带Tear down标识，SGSN只去激活与此TI关联的一个二次PDP上下文。
>
> 如果主PDP上下文被去激活，则与之共享同一IP地址和APN的第一个二次PDP变成主PDP上下文。

UNC 作为S4 SGSN时，MS发起的二次PDP激活流程和作为SGSN时的有差异。下面以3G网络的终端二次PDP激活为例介绍差异，2G网络的差异和3G网络的类似。

3G网络MS发起的二次PDP激活流程如 [图3](#ZH-CN_TOPIC_0185005558__zh-cn_concept_0130429208_fig_S4) 所示。

**图3** Iu模式MS发起的二次激活流程

<br>

![](WSFD-106001 终端侧二次PDP上下文激活特性概述_85005558.assets/zh-cn_image_0193620576_2.png)

<br>

- Gateway创建专有承载
    - 步骤3：S4 SGSN收到Activate Secondary PDP Context Request消息后，S4 SGSN应该保证和相关联的一次激活PDP上下文共用一个S-GW和P-GW。S4 SGSN发送Bearer Resource Command消息给S-GW，指示S-GW创建专有承载。
    - 步骤4：S-GW发送Bearer Resource Command给P-GW，通知P-GW建立专有承载。
    - 步骤5：请求成功后，P-GW回复Create Bearer Request消息给S-GW。
    - 步骤6：S-GW回复Create Bearer Request消息给S4 SGSN，该消息中包含S-GW用户面的地址和TEID。
- Gateway专有承载创建完成
    - 步骤8：RAB成功创建以后，S4 SGSN把RNC的地址和TEID记录到本地，同时回复Create Bearer Response消息给S-GW。
      > **说明**
      > 若网络中开启了Direct Tunnel功能，Create Bearer Response消息中将包含RNC的地址和TEID，这样RNC和S-GW之间的隧道信息就完成了交换，最终实现Direct Tunnel模式，上下行用户面数据就可以直接在RNC和S-GW之间进行转发。
    - 步骤9：S-GW回复Create Bearer Response消息给P-GW，专有承载创建完成。

#### [计费与话单](#ZH-CN_TOPIC_0185005558)

二次PDP上下文的激活会产生S-CDR话单；二次PDP上下文的去激活会关闭S-CDR话单；二次PDP上下文的修改会导致旧话单的关闭及新话单的产生。
