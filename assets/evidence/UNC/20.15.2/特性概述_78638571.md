#### [适用NF](#ZH-CN_TOPIC_0278638571)

SGW-C

#### [定义](#ZH-CN_TOPIC_0278638571)

为了适应物联网低功耗、时延不敏感的业务，3GPP标准协议在省电方面提出了eDRX（extended Discontinuous Reception）技术，通过扩展现有DRX周期使eDRX周期可以达到分钟级别，延长了终端在空闲状态下的监控寻呼指示信道的周期，从而极大节省终端功耗，延长终端电池使用寿命。

UNC 提供终端用户在eDRX模式下的下行数据缓存机制，在下行数据到达SGW-C时，SGW-C根据MME返回的报文缓存时长，先缓存用户报文，待数据通道建立之后，传递用户上下行报文，和终端进行业务交互。

#### [客户价值](#ZH-CN_TOPIC_0278638571)

| 受益方 | 受益描述 |
| --- | --- |
| 运营商 | 节省eMTC（enhanced Machine Type Communication）终端功耗，延长电池使用寿命，提升IoT业务的竞争能力。 |
| 用户 | 终端将在非数据发送周期内进入深度睡眠工作模式，从而极大节省终端的功耗，延长终端电池的寿命。 |

#### [应用场景](#ZH-CN_TOPIC_0278638571)

eDRX适用于终端既省电又满足一定时延的交互式应用，如手环/跟踪/街灯控制等。根据应用业务的时延容忍，结合运营商的要求，合理设置eDRX周期。根据协议推荐，对时延要求高的应用，如跟踪等业务建议eDRX周期设置为秒级别，对时延要求不高的应用，建议eDRX周期设置为分钟级别。

#### [应用限制](#ZH-CN_TOPIC_0278638571)

- 支持同时用于缓存数据的内存资源为用户总规格的10%。
- 当UNC作为独立SGW-C设备时，可缓存的数据报文长度最大为1600字节。当UNC作为融合网关设备时，对于Non-IP用户，可缓存的数据报文长度最大为1600字节-（IP+ UDP）头长度。

#### [原理概述](#ZH-CN_TOPIC_0278638571)

**相关概念**

- **eMTC**
  由于传统无线网络没有对物联网业务进行过专门设计和优化，因此无线数传功耗较大，成本较高，不适用于大连接、低速率、低功耗、低成本的物联网业务场景。协议3GPP V13.2.0针对此类物联网业务的特点，协议标准上基于LTE进行演进，设计了专门用于物联网的eMTC技术。
  eMTC支持的终端类型RAT Type包括EUTRAN和LTE-M。同时UE Category的不同，引入了新的UE Category M1类型的终端。eMTC应用于中低速率（<1Mbps）的物联网场景，如Pos机，可穿戴设备，智能家居等。
- **eDRX**
  由于传统网络要兼容终端低功耗和业务及时性要求，所以在传统网络中这个DRX周期最大为2.56s。在一个DRX周期，终端只在寻呼时刻监控寻呼指示信道，其他的时刻终端不监控寻呼指示信道，从而降低了终端的功耗。eDRX技术扩展现有的DRX周期，最大约为43.9分钟，进一步降低了eMTC终端功耗，如 [图1](#ZH-CN_TOPIC_0278638571__fig1) 所示。

**系统实现**

eMTC eDRX模式数据传输流程 如 [图2](#ZH-CN_TOPIC_0278638571__fig3) 所示， 主要有以下四部分：

1. eNodeB支持eDRX，并通过系统消息SI告诉UE。
2. 如果UE使能了eDRX，UE在attach/TAU请求消息中通知MME其支持eDRX，MME根据本地决策eDRX的寻呼周期，下发eDRX周期给eNodeB和UE。
3. SGW-U收到下行数据包后，发送PFCP Session Report Request消息给SGW-C，SGW-C向MME发起寻呼，MME在DDN ACK消息中携带DL Buffering Duration信元/DL Buffering Suggested Packet Count信元，通知SGW-C下行数据的缓存时长/缓存包数，SGW-C响应消息PFCP Session Report Response给SGW-U。
    - 下行数据缓存时长的获取方式：
      SGW-C收到DDN ACK消息后，判断消息中是否携带DL Buffering Duration信元，如果没有携带，不需要刷新下行数据缓存时长；如果消息中携带了DL Buffering Duration信元，则使用信元中携带的时长作为下行数据包的缓存时长。
    - 下行数据缓存包数的获取方式：
      SGW-C收到DDN ACK消息后，判断消息中是否携带DL Buffering Suggested Packet Count信元，如果没有携带，不需要刷新下行数据缓存包的个数；如果消息中携带了DL Buffering Suggested Packet Count信元，则使用信元中携带的包数作为下行数据缓存包数。
4. MME根据eDRX周期确定寻呼时刻，通过eNodeB向用户发起寻呼，寻呼成功后重新建立数据链路，SGW-U将缓存的下行数据传输给UE。
