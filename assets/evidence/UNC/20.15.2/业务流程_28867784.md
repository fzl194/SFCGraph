# 业务流程

当终端在附着后主动发起用于传输SIP信令的一次上下文激活/缺省承载，通过该上下文/缺省承载在IMS域里完成用户注册。

用户在IMS域注册成功后，终端可随时作为主叫/被叫发起/接收SIP信令，执行呼叫和媒体协商流程。

IMS域中的AF在处理终端的SIP信令的过程中，根据SDP消息中的参数得出此次会话所采用的媒体编码格式、流信息和所需带宽等，发送给PCRF，PCRF根据这些信息为用户选择相应的PCC规则，最终触发PGW-C发起网络侧发起的二次PDP上下文激活/专有承载创建请求。

- 网络侧发起的二次PDP上下文激活流程如[图1](#ZH-CN_TOPIC_0228867784__zh-cn_topic_0228459037_fig11718712411)所示。
- 网络侧发起的专有承载创建流程如[图2](#ZH-CN_TOPIC_0228867784__zh-cn_topic_0228459037_fig1244615915125)所示。

**图1** 网络侧发起的二次PDP上下文激活流程

<br>

![](业务流程_28867784.assets/zh-cn_image_0000001195121889_2.png)

1. 用户（UE A和UE B）接入到网络中建立默认承载，利用默认承载完成在IMS域中的注册。当主叫UE A拨打被叫UE B的号码时，会通过默认承载向AF发送SIP INVITE消息，消息中携带主被叫的用户ID，同时在SDP的参数中携带主叫UE A在当次呼叫所支持的媒体编码格式/带宽。
2. 主叫UE A的SIP INVITE消息发送到被叫端UE B。
3. 根据主/被叫终端的能力执行媒体流的协商过程，最终确定当次通话所使用的编码格式。
4. 被叫UE B在最终确认了通话采用的编码格式后，以包含最终确定的SDP媒体参数的SIP 200 OK响应报文确认呼叫成功。
5. 被叫UE B的SIP 200 OK响应报文发送到主叫UE A。
6. AF从SDP中过滤媒体类型、所需带宽和编码格式等信息，通过Rx接口的AAR/AAA消息将这些媒体信息下发给PCRF。
7. PCRF将AAA消息关联到用户的会话上，并根据消息中的媒体信息创建授权的QoS和规则。
8. PCRF将创建的规则和QoS通过Gx接口发送给GGSN-C/PGW-C。
9. GGSN-C将消息中的规则绑定到用户的会话上（这一过程称为承载绑定）。
10. GGSN-C向GGSN-U发送PFCP Session Modification Request消息，指示GGSN-U创建专有承载。GGSN-U创建专有承载，给GGSN-C返回PFCP Session Modification Response消息。
11. 当用户不存在能够绑定规则的承载资源（满足授权QoS的承载）时，GGSN-C将授权的QoS参数和流过滤器（TFT）封装到专有承载创建消息Initiate PDP Context Activation Request中，发送给Gn/Gp SGSN。
12. Gn/Gp SGSN收到GGSN-C发起的上述消息后，将该次交互关联到用户的一次上下文，分配交互事务ID，向终端用户发送二次上下文激活请求。
13. 支持UE/NW承载控制模式的终端，网络侧发起的NAS消息二次上下文激活请求会触发终端用户执行二次PDP上下文激活，终端用户向Gn/Gp SGSN发起激活二次PDP上下文请求。
  消息中携带手机分配的NSAPI、APN和QoS参数等信息。
14. Gn/Gp SGSN收到终端发起的二次PDP上下文请求后，向GGSN-C发送Create PDP Context Request消息，请求创建二次PDP上下文。
  消息中携带Gn/Gp SGSN的用户面IP 、TEID、SGSN协商后的QoS参数等信息。
15. GGSN-C向Gn/Gp SGSN返回Create PDP Context Response消息。
16. RAN侧或Gn/Gp SGSN对QoS参数降级时，Gn/Gp SGSN通过Update PDP Context Request消息通知GGSN-C降级后的QoS参数，消息中携带RAN侧用户面IP 、TEID以及False的DTI标记。
  GGSN-C向Gn/Gp SGSN返回Update PDP Context Response消息，PDP Context更新成功。若GGSN-C更新PDP Context失败，则Gn/Gp SGSN发起去激活流程删除所创建的资源。
17. Gn/Gp SGSN向终端用户发送Activate Secondary PDP Accept消息，接受二次PDP上下文请求。
18. 二次激活流程成功后，Gn/Gp SGSN通过网络侧发起的二次激活响应到GGSN-C完成网络侧发起的二次激活过程。
19. GGSN-C通过PFCP Session Modification Request消息将承载更新消息传递给GGSN-U。
20. GGSN-C在完成网络侧发起的专有承载激活流程成功后，通过相应的Gx会话事件向PCRF上报资源安装成功。
21. PCRF通知IMS网络该用户的本次会话资源预留成功。

> **说明**
> 由于网络发起的二次上下文激活是由SIP终端在信令面的SDP中标识的媒体流信息触发，因此UE在执行二次上下文激活时携带的请求QoS以及流过滤器等参数应当和网络侧授权的相应参数相同。

**图2** 网络侧发起的专有承载创建流程

<br>

![](业务流程_28867784.assets/zh-cn_image_0228656217_2.png)

1. 用户（UE A和UE B）接入到网络中建立默认承载，利用默认承载完成在IMS域中的注册。当主叫UE A拨打被叫UE B的号码时，会通过默认承载向AF发送SIP INVITE消息，消息中携带主被叫的用户ID，同时在SDP的参数中携带主叫UE A在当次呼叫所支持的媒体编码格式/带宽。
2. 主叫UE A的SIP INVITE消息发送到被叫端UE B。
3. 根据主/被叫终端的能力执行媒体流的协商过程，最终确定当次通话所使用的编码格式。
4. 被叫UE B在最终确认了通话采用的编码格式后，以包含最终确定的SDP媒体参数的SIP 200 OK响应报文确认呼叫成功。
5. 被叫UE B的SIP 200 OK响应报文发送到主叫UE A。
6. AF从SDP中过滤媒体类型、所需带宽和编码格式等信息，通过Rx接口的AAR/AAA消息将这些媒体信息下发给PCRF。
7. PCRF将AAA消息关联到用户的会话上，并根据消息中的媒体信息创建授权的QoS和规则。
8. PCRF将创建的规则和QoS通过Gx接口发送给PGW-C。
9. PGW-C将消息中的规则绑定到用户的会话上（这一过程称为承载绑定）。
10. PGW-C向PGW-U发送PFCP Session Modification Request消息，指示PGW-U创建专有承载。PGW-U创建专有承载，给PGW-C返回PFCP Session Modification Response消息。
11. 当用户不存在能够绑定规则的承载资源（满足授权QoS的承载）时，PGW-C将授权的QoS参数和流过滤器封装到专有承载创建消息中，发送给MME。
  PGW-C和SGW-C分离部署时，PGW-C通过SGW-C将专有承载创建消息发送给MME。
12. MME向eNodeB发送Bearer Setup Request消息。
  消息包含：EPS Bearer Identity、EPS Bearer QoS、Session Management Request、S1-TEID。其中Session Management Request是MME构造的消息，消息包含：PTI、TFT、EPS Bearer QoS parameters (excluding ARP)、Protocol Configuration Options、EPS Bearer Identity、Linked EPS Bearer Identity (LBI)。
13. eNodeB将EPS Bearer QoS映射为Radio Bearer QoS，然后向UE发送RRC Connection reconfiguration消息。
  消息包含：Radio Bearer QoS、Session Management Request、EPS RB Identity。
  UE保存该消息中的信息，并通过LBI关联专有承载和缺省承载。
14. UE向eNodeB响应RRC Connection Reconfiguration Complete消息，确认无线承载激活。
15. eNodeB向MME返回Bearer Setup Response。
16. UE向eNodeB发送Direct Transfer消息，消息中携带UE NAS层构建的Session Management Response消息，消息包含：EPS Bearer Identity。
17. eNodeB向MME发送Session Management Response消息。
18. MME向S/PGW-C返回Create Bearer Response。
19. S/PGW-C通过PFCP Session Modification Request消息将承载更新消息传递给S/PGW-U。
20. PGW-C在完成网络侧发起的专有承载激活流程成功后，通过相应的Gx会话事件向PCRF上报资源安装成功。
21. PCRF通知IMS网络该用户的本次会话资源预留成功。
