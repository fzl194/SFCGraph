[**协议标准流程的迁移过程**](#ZH-CN_TOPIC_0245935193__sub3)

为降低迁移对业务的影响，需要将用户迁移分为两个阶段，如 [图1](#ZH-CN_TOPIC_0245935193__fig02) 所示。

> **说明**
> 由于 [基于eNodeB进行迁移](特性概述_45935192.md#ZH-CN_TOPIC_0245935192__li914593023180640) 只能迁移eNodeB下面的连接态用户，所以只需要进行如下第二阶段迁移。协议标准流程的迁移类型中除了基于eNodeB的迁移的其余迁移类型，需要进行如下两个阶段迁移。

- EMS控制迁移用户第一阶段：用户主动发起接入时，在接入完成后发起连接释放，进行用户迁移。
- EMS控制迁移用户第二阶段：第一阶段的定时器超时后，MME对仍然在线的用户主动发起连接释放，进行用户迁移。

基于eNodeB的迁移类型，在迁移开始前， UNC 会自动给迁移涉及的eNodeB下发MME的设备能力为 “0” ，并在迁移任务结束后， UNC 自动将MME的设备能力恢复为原来的取值。 协议标准流程的迁移类型中除了基于eNodeB的迁移，其余迁移类型在迁移开始前，首先需要通过EMS配置MME的设备能力为 “0” ，下发给eNodeB，避免由于新的用户再接入而造成迁移进度不准确。在迁移结束后，系统不会自动将MME的设备能力恢复为原来的取值，需要通过EMS手工恢复MME的设备能力。

**图1** 用户迁移过程

<br>

![](实现原理_45935193.assets/zh-cn_image_0000001474959288_2.png "点击放大")

- EMS控制迁移用户第一阶段
  在迁移第一阶段（即第一阶段定时器超时前），MME不会主动对用户进行迁移，而是等待用户主动发起连接后，再进行迁移处理。对处于不同状态的用户，迁移第一阶段过程如下：
    - 连接态用户
      eNodeB主动发起S1 Release流程，MME在接收到来自eNodeB的UE Context Release Request消息后，向eNodeB发送携带原因值为“Load Balancing TAU Required”的UE Context Release Command消息。之后，eNodeB向UE发送原因值为“Load Balancing TAU Required”的RRC释放请求，要求UE发起TAU流程。UE会在TAU流程中重新选择MME。
    - 空闲态用户
      若用户向MME发起TAU，仍然允许用户正常接入。在TAU流程结束后，MME主动发起S1 Release流程，向eNodeB发送携带原因值为“Load Balancing TAU Required”的UE Context Release Command消息。之后，eNodeB向UE发送原因值为“Load Balancing TAU Required”的RRC释放请求，要求UE发起TAU流程。UE会在TAU流程中重新选择MME。
      针对IMS VoPS业务，系统不针对主叫业务和被叫业务引起的Service Request进行特殊处理，允许用户建立用于通话的专有承载。
    - 分离态用户
      若用户发起附着，仍然允许用户正常接入。在附着流程结束后，MME主动发起S1 Release流程，向eNodeB发送携带原因值为“Load Balancing TAU Required”的UE Context Release Command消息。之后，eNodeB向UE发送原因值为“Load Balancing TAU Required”的RRC释放请求，要求UE发起TAU流程。UE会在TAU流程中重新选择MME。
  > **说明**
  > - 第一阶段定时器可以通过 [**SET OFFLOADINF**](../../../../../../../OM参考/命令/UNC MML命令/业务服务管理/Pre 5G接入业务管理/控制面管理/网络管理/迁移控制/设置迁移配置表（SET OFFLOADINF）_72345695.md) 命令中的参数 “第一阶段迁移时间” 进行设置。
  > - 如果在第一阶段定时器超时前，用户主动接入进行迁移的用户数已经等于或大于需要迁移的用户数，则迁移过程结束，不进入迁移二阶段。
- EMS控制迁移用户第二阶段
  第一阶段定时器超时后，如果被迁移的用户数还未达到需要迁移的用户数，则系统进入迁移第二阶段。此时MME不再等待用户主动再接入，立刻强制进行迁移动作。但是为了保证语音通话的连续性，针对正在进行语音通话的用户，不进行迁移处理。
  在迁移第二阶段，MME不会对用户的连接状态进行迁移优先级区分，只按进程上的用户分布进行迁移。对处于不同状态的用户，迁移第二阶段过程如下：
    - 连接态用户
      MME主动发起S1 Release流程，向eNodeB发送携带原因值为“Load Balancing TAU Required”的UE Context Release Command消息。之后，eNodeB向UE发送原因值为“Load Balancing TAU Required”的RRC释放请求，要求UE发起TAU流程。UE会在TAU流程中重新选择MME。
    - 空闲态用户
      MME主动发起Paging流程，待UE发送Service Request消息后，MME再向eNodeB发送携带原因值为“Load Balancing TAU Required”的UE Context Release Command消息。之后，eNodeB向UE发送原因值为“Load Balancing TAU Required”的RRC释放请求，要求UE发起TAU流程。UE会在TAU流程中重新选择MME。
      为避免在第二阶段存在大量空闲态的用户，导致S1接口因大量Paging信令出现负荷过重的场景，需要第一阶段定时器时间足够长。第一阶段的缺省时长应该是用户可达定时器的长度加上隐式分离定时器的长度。
- 迁移状态
  如 [图2](#ZH-CN_TOPIC_0245935193__fig04) 、 [图3](#ZH-CN_TOPIC_0245935193__fig05) 所示了WB-SAE用户迁移过程的MME状态。

> **说明**
> - 将“DWORD_EX23 BIT32”软参设置为1，再执行[**STR OFFLOADBYMME**](../../../../../../../OM参考/命令/UNC MML命令/业务服务管理/Pre 5G接入业务管理/控制面管理/网络管理/迁移控制/启动MME迁移任务（STR OFFLOADBYMME）_72345693.md)时，可同时迁移WB-SAE用户和NB-IoT用户。
> - 启动基于TA迁移用户的命令**[STR OFFLOADBYTA](../../../../../../../OM参考/命令/UNC MML命令/业务服务管理/Pre 5G接入业务管理/控制面管理/网络管理/迁移控制/启动TA迁移任务（STR OFFLOADBYTA）_02884717.md)**时，可同时迁移WB-SAE用户和NB-IoT用户。
> - 启动NB-IoT用户独立迁移的命令为[**STR NBOFFLOADBYMME**](../../../../../../../OM参考/命令/UNC MML命令/业务服务管理/Pre 5G接入业务管理/控制面管理/网络管理/迁移控制/启动NB-IoT用户的MME迁移任务（STR NBOFFLOADBYMME）_26146094.md)。
> - 无论在迁移第一阶段还是迁移第二阶段，系统都支持迁移备份。这样当发生进程或VM倒换时，MME仍然可以正常查询用户信息、上报监控信息、迁移剩余待迁移用户。

[**自定义流程的迁移过程**](#ZH-CN_TOPIC_0245935193__sub3)

为满足运营商拨测场景的需要， UNC 增加了按IMSI迁移单个用户迁移至Pool内指定目标MME的功能，该功能属于非协议标准类功能，迁移过程中不需要通过EMS配置将MME的设备能力置为 “0” ，迁移流程如 [图4](#ZH-CN_TOPIC_0245935193__fig03) 所示。

基于TA且指定目标MME（ **[STR OFFLOADBYTA](../../../../../../../OM参考/命令/UNC MML命令/业务服务管理/Pre 5G接入业务管理/控制面管理/网络管理/迁移控制/启动TA迁移任务（STR OFFLOADBYTA）_02884717.md)** 命令配置 “目标MME编码（MMEC）” 参数）的迁移任务启动时， UNC 会将指定TA的用户迁移至Pool内指定目标MME，迁移流程如 [图4](#ZH-CN_TOPIC_0245935193__fig03) 所示。

> **说明**
> 禁止使用该功能对正在进行业务流程的用户进行迁移。

**图4** 指定IMSI或者指定TA用户向指定目标MME的迁移流程

<br>

![](实现原理_45935193.assets/zh-cn_image_0000001526198705_2.png "点击放大")

- 对于连接态用户，源MME通过GUTI重分配流程给某指定IMSI的UE分配一个包含目标MME Code的GUTI，然后由源MME触发对该UE的分离流程(分离方式为re-attach required)，最后释放S1连接。UE从源MME上分离后，重新发起RRC建链请求，在请求消息中携带目标MME CODE，eNodeB基于此目标MME CODE发起S1连接建立请求，最终实现UE在目标MME上的接入。
- 对于空闲态用户，需要先将UE寻呼为连接态，之后再执行上述连接态的迁移流程来实现UE至指定目标MME上的迁移。
