#### [紧急呼叫定位流程（NI-LR）](#ZH-CN_TOPIC_0000001889609372)

**图1** 发起紧急呼叫触发的NI-LR基本流程图

<br>

![](实现原理_89609372.assets/zh-cn_image_0000001900163320_2.png)

1. UE触发紧急PDU会话。AMF通过消息中携带的信元感知紧急PDU Session建立。
2. 当紧急PDU Session流程完成时，AMF根据NRF查询或本地配置选择一个LMF。AMF向选中LMF发送Nlmf_Location_DetermineLocation Request消息，请求UE当前的位置。消息包含了LCS Correlation ID, Location QoS，External Client Type, AMF Instance ID，UeLcsCapability信元等。
  > **说明**
  > - AMF可以通过[**ADD NGNILRPARA**](../../../../../OM参考/命令/UNC MML命令/业务服务管理/5G接入业务管理/5G定位服务管理/紧急定位服务管理/增加NI-LR功能参数（ADD NGNILRPARA）_44006465.md)命令的在本地配置紧急呼叫场景是否发起定位流程，以及定位策略。
  >   例如， [**ADD NGNILRPARA**](../../../../../OM参考/命令/UNC MML命令/业务服务管理/5G接入业务管理/5G定位服务管理/紧急定位服务管理/增加NI-LR功能参数（ADD NGNILRPARA）_44006465.md) 的 “NILR（是否允许网络触发定位功能）” 为 “YES（是）” ， “NILRPLCY（网络触发的定位策略）” 为 “PROTOCOL_POSITION（协议模式定位）” ，则表示支持紧急呼叫场景发起定位流程，且定位策略为协议模式定位。如果此场景下AMF获取不到LMF，则直接执行无LMF定位，即AMF上报当前NCGI给GMLC，可参考 [无LMF的定位流程（NI-LR）](#ZH-CN_TOPIC_0000001889609372__section1117773812456) 。
  > - Nlmf_Location_DetermineLocation Request消息包含的信元有：
  >     - LCS Correlation ID信元，该信元由AMF分配，用于标识本地定位流程，在后续AMF向LMF转发UE侧LPP或者NRPPa消息时作为本次定位流程的标识。因此，AMF需要保证每个用户的定位流程分配的LCS Correlation ID唯一。
  >     - Location QoS，LCS QoS信元包含“HA（水平精度）”和“RESPTIME（响应时间）”，可由命令[**ADD NGNILRPARA**](../../../../../OM参考/命令/UNC MML命令/业务服务管理/5G接入业务管理/5G定位服务管理/紧急定位服务管理/增加NI-LR功能参数（ADD NGNILRPARA）_44006465.md)进行配置。
  >     - External Client Type信元，NI-LR场景为“Emergency_Service”。
  >     - UeLcsCapability信元，为“LPP Support”。
3. LMF进行定位测量并将定位结果发送给AMF。按照协议标准定位流程方法进行定位测量，详细参见特性[WSFD-106701 网络侧发起的5G位置定位服务](../../位置业务管理/WSFD-106701 网络侧发起的5G位置定位服务_46574614.md)实现原理章节。AMF在此过程中支持3GPP协议的定位方法和相关信元的透传。该定位流程中，AMF负责透传gNodeB和LMF之间的NRPPa定位消息、UE和LMF之间的LPP定位消息。
4. LMF向AMF返回Nlmf_Location_DetermineLocation Response消息，返回UE当前的位置信息。消息包括用户的经纬度等位置信息。
5. AMF根据本地配置或者基于目标NFType等参数到NRF选择GMLC。AMF向选中的GMLC发送Namf_Location_EventNotify消息，通知GMLC发起紧急会话。消息携带紧急呼叫标识“LocationEvent”以及[步骤4](#ZH-CN_TOPIC_0000001889609372__li17362499917)从LMF获取到的位置信息给GMLC，其中紧急呼叫标识“LocationEvent”为“EMERGENCY_CALL_ORIGINATION”。
6. GMLC将位置转发给外部紧急业务客户端，紧急呼叫中心根据紧急呼叫和位置信息快速的采取措施。
7. UE结束紧急呼叫，释放紧急PDU会话。
  > **说明**
  > - 紧急呼叫结束后，用户会发起PDU会话释放流程，AMF需要判断紧急呼叫释放场景是否上报位置，可由命令[**ADD NGNILRPARA**](../../../../../OM参考/命令/UNC MML命令/业务服务管理/5G接入业务管理/5G定位服务管理/紧急定位服务管理/增加NI-LR功能参数（ADD NGNILRPARA）_44006465.md)进行配置。
  >   例如：当参数 “RLSRPT（紧急呼叫释放是否上报位置）” 为 “YES（是）” ，则在紧急呼叫结束后，用户发起的PDU会话释放流程中，AMF向GMLC上报UE的位置信息。
  >
  > - 在紧急呼叫结束后，用户发起PDU会话释放流程场景中，如果AMF判断需要上报位置，则只上报当前NCGI即可（因为在紧急呼叫流程中，AMF已经获取到用户最新的位置信息，则不需要再次获取实时位置信息）。
8. AMF向GMLC发送Namf_Location_EventNotify消息，通知GMLC紧急会话已经释放，GMLC和LRF可以释放与紧急会话相关的所有资源。

#### [无LMF的定位流程（NI-LR）](#ZH-CN_TOPIC_0000001889609372)

在3GPP协议定义的标准LCS组网场景中，引入了网元LMF，用于和UE/gNodeB进行LCS定位协商和交互，并计算最终的定位结果。这种场景下可以获取用户的经纬度信息，定位精度较高。但是，在实际网络中，运营商可能希望在不增加新网元部署的情况下，实现LCS定位功能。

在网络中未部署LMF的场景下， UNC 支持通过GMLC网元接收来自LCS Client的定位请求，AMF获取UE的最新NCGI（NR Cell Global Identifier）信息，并将NCGI信息通过NLg接口上报给GMLC网元。该LCS定位方法的定位精度取决于小区的覆盖范围，精度较低。

> **说明**
> AMF可以通过 [**ADD NGNILRPARA**](../../../../../OM参考/命令/UNC MML命令/业务服务管理/5G接入业务管理/5G定位服务管理/紧急定位服务管理/增加NI-LR功能参数（ADD NGNILRPARA）_44006465.md) 命令的在本地配置紧急呼叫场景是否发起定位流程，以及定位策略。
>
> 例如， [**ADD NGNILRPARA**](../../../../../OM参考/命令/UNC MML命令/业务服务管理/5G接入业务管理/5G定位服务管理/紧急定位服务管理/增加NI-LR功能参数（ADD NGNILRPARA）_44006465.md) 的 “NILR（是否允许网络触发定位功能）” 为 “YES（是）” ， “NILRPLCY（网络触发的定位策略）” 为 “NO_LMF_POSITION（无LMF定位）” ，则表示支持紧急呼叫场景发起定位流程，且定位策略为无LMF定位。

#### [选择LMF和GMLC的策略](#ZH-CN_TOPIC_0000001889609372)

AMF选择LMF和选择GMLC的策略如 [表1](#ZH-CN_TOPIC_0000001889609372__tab5) 和 [表2](#ZH-CN_TOPIC_0000001889609372__tab6) 。

*表1 选择LMF策略*

| LMF的个数 | 选择LMF策略 |
| --- | --- |
| 无LMF | 无法从LMF获取准确的位置信息，AMF直接将NCGI信息通过NLg接口上报给GMLC网元。 |
| 单个LMF | 根据NRF查询或本地配置来选择LMF，该LMF在配置范围内可以选到。 |
| 多个LMF | 根据NRF查询或本地配置来选择LMF，该LMF在配置范围内可以选到。 |

*表2 选择GMLC策略*

| GMLC的个数 | 选择GMLC策略 |
| --- | --- |
| 无GMLC | 无法完成位置信息上报。 |
| 单个GMLC | 根据本地配置或基于目标NFType等参数到NRF选择GMLC，该GMLC在配置范围内可以选到。 |
| 多个GMLC | 根据本地配置或基于目标NFType等参数到NRF选择GMLC，该GMLC在配置范围内可以选到。 |

#### [其它场景位置信息上报](#ZH-CN_TOPIC_0000001889609372)

- 在NI-LR流程中，inter AMF Handover/intra AMF Handover场景，根据协议AMF需要规定由源侧AMF还是由目标AMF上报位置信息，如[表3](#ZH-CN_TOPIC_0000001889609372__tab7)所示。
  *表3 在inter AMF Handover/intra AMF Handover场景下的处理*

  | 上报网元 | inter AMF Handover场景 | intra AMF Handover场景 |
  | --- | --- | --- |
  | 源侧上报 | 源侧AMF在切换成功后，向GMLC发送Namf_location_EventNotify消息上报位置信息。 | intra AMF Handover场景不区分源侧和目标侧，在AMF切换成功后，向GMLC发送Namf_location_EventNotify消息上报位置信息。 |
  | 目标侧上报 | 目标侧AMF在切换成功后，向GMLC发送Namf_location_EventNotify消息上报位置信息。 | intra AMF Handover场景不区分源侧和目标侧，在AMF切换成功后，向GMLC发送Namf_location_EventNotify消息上报位置信息。 |
- 在NI-LR流程中，4G和5G互切换场景，根据协议AMF和MME需要规定由源侧AMF/MME还是由目标侧AMF/MME上报位置信息（通过[**ADD NGNILRPARA**](../../../../../OM参考/命令/UNC MML命令/业务服务管理/5G接入业务管理/5G定位服务管理/紧急定位服务管理/增加NI-LR功能参数（ADD NGNILRPARA）_44006465.md)和[**ADD LCSPARAEX**](../../../../../OM参考/命令/UNC MML命令/业务服务管理/Pre 5G接入业务管理/控制面管理/业务安全管理/LCS/LCS扩展参数/增加LCS扩展参数(ADD LCSPARAEX)_26305624.md)命令的“HORPTTYPE”参数进行配置，该参数在全网AMF和MME中的配置需保持一致。），如[表4](#ZH-CN_TOPIC_0000001889609372__table55609319344)所示。
  *表4 在4G切换到5G/5G切换到4G场景下的处理*

  | 上报网元 | 4G切换到5G场景 | 5G切换到4G场景 |
  | --- | --- | --- |
  | 源侧上报 | - 目标侧AMF向源侧MME发送的Forward Relocation Response消息中携带AMF Identifier信元。<br>- 源侧MME解析AMF Identifier信元，从中获取AMF Instance ID信元。<br>- 源侧MME向GMLC发送Subscriber Location Report消息，该消息中携带Location Event和AMF Instance ID信元。- Location Event：该信元的值为“EMERGENCY_CALL_HANDOVER (3)”，通知GMLC发生4G到5G的切换。<br>- AMF Instance ID：源侧MME上报目标侧AMF的紧急呼叫定位结果给GMLC。 | - 当命令[**ADD MMECHARACT**](../../../../../OM参考/命令/UNC MML命令/业务服务管理/Pre 5G接入业务管理/控制面管理/GTP-C接口管理/GnGp-SGSN_S10_S16_S3接口管理/MME属性/增加MME属性配置信息（ADD MMECHARACT）_26305766.md)的“MMEIDIE”参数为“YES（是）”时，目标侧MME发送给源侧AMF的Forward Relocation Response消息中将包含MME Identifier信元。<br>- 源侧AMF解析MME Identifier信元，从中获取targetMmeName和targetMmeRealm信元。<br>- 源侧AMF向GMLC发送Namf_location_EventNotify消息，该消息中携带Location Event、targetMmeName和targetMmeRealm信元。- Location Event：该信元的值为“EMERGENCY_CALL_HANDOVER”，通知GMLC发生5G到4G的切换。<br>- targetMmeName和targetMmeRealm：源侧AMF上报目标侧MME的紧急呼叫定位结果给GMLC。 |
  | 目标侧上报 | 目标侧AMF向GMLC发送Namf_location_EventNotify消息，该消息中携带Location Event和AMF Identifier信元。<br>- Location Event信元的值为“EMERGENCY_CALL_HANDOVER”，通知GMLC发生4G到5G的切换。<br>- AMF Identifier：目标侧AMF上报目标侧AMF的紧急呼叫定位结果给GMLC。 | 目标侧MME向GMLC发送Subscriber Location Report消息，该消息中携带Location Event和MME Identifier信元。<br>- Location Event信元的值为“EMERGENCY_CALL_HANDOVER (3)”，通知GMLC发生5G到4G的切换。<br>- MME Identifier：目标侧MME上报目标侧MME的紧急呼叫定位结果给GMLC。 |
