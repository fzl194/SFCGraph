# 实现原理（Gx）

基于业务级的累计流量策略控制实现方式如 [表1](#ZH-CN_TOPIC_0229039989__table163583478370) 所示。

*表1 业务级的累计流量策略控制实现方式*

| 实现方式 | 流量阈值下发和流量累计上报关键AVP | AVP说明 | Event-Trigger |
| --- | --- | --- | --- |
| 3GPP标准的Usage Monitoring机制 | Usage-Monitoring-Information | 用于流量阈值下发和流量累计上报。 | Usage-Report |
| 3GPP标准的Usage Monitoring机制 | [ Monitoring-Key ] | 用于标识一种业务，<br>UNC<br>（GGSN-C/PGW-C）根据PCRF下发的Monitoring-key执行累计流量动作。<br>- 对于PCC动态规则，Monitoring-Key AVP在Charging-Rule-Definition中扩展。<br>- 对于PCC预定义规则，Monitoring-Key在UNC（GGSN-C/PGW-C）可以使用[**ADD PCCPOLICYGRP**](../../../../../OM参考/命令/UNC MML命令/业务服务管理/会话管理/计费和策略的业务管理/业务策略/PCC策略组/增加PCC策略组（ADD PCCPOLICYGRP）_09897173.md)命令配置。<br>- 对于PCC预定义规则组，Monitoring-Key在UNC（GGSN-C/PGW-C）可以使用[**ADD USERPROFILE**](../../../../../OM参考/命令/UNC MML命令/业务服务管理/会话管理/计费和策略的业务管理/业务模板/用户模板/增加用户模板（ADD USERPROFILE）_09897206.md)命令配置。 | Usage-Report |
| 3GPP标准的Usage Monitoring机制 | [ Granted-Service-Unit ] | 用于流量阈值下发。 | Usage-Report |
| 3GPP标准的Usage Monitoring机制 | [ Used-Service-Unit ] | 用于流量累计上报。 | Usage-Report |
| 3GPP标准的Usage Monitoring机制 | [ Usage-Monitoring-Level ] | 业务级累计流量时取值为PCC_RULE_LEVEL (1)。 | Usage-Report |
| 3GPP标准的Usage Monitoring机制 | [ Usage-Monitoring-Report ] | 指示<br>UNC<br>（GGSN-C/PGW-C）应该上报的累积流量。 | Usage-Report |
| 3GPP标准的Usage Monitoring机制 | [ Usage-Monitoring-Support ] | 指示Monitoring Key值的配额监控去使能。 | Usage-Report |

采用3GPP标准的Usage Monitoring机制时，对于包月用户，会要求在某个时间点（例如月末）进行配额重置，如PCRF/PCF同时要求所有在线用户上报流量，会增加Gx接口的信令负荷，造成信令拥塞；如果PCRF/PCF分批要求在线用户上报流量，则流量上报有延迟，且上报不准确。为了解决该问题，引入UMCH（Usage Monitoring Congestion Handling）机制，PCRF通过Monitoring-Time AVP提前指定配额重置的时间点，PCEF分别统计配额重置时间点前后的流量，并通过重置后最近一次的流量上报流程一起上报给PCRF/PCF，从而避免Gx接口的Usage Monitoring信令拥塞。若开启该功能，需要 [**ADD PCRF**](../../../../../OM参考/命令/UNC MML命令/业务服务管理/会话管理/PCC管理/PCRF Diameter连接/PCRF/增加PCRF（ADD PCRF）_09897101.md) 命令中的 “Feature列表（FEATURELIST）” 参数包含 “UMCH” 。
