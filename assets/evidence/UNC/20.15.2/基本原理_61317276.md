# 基本原理

**同网段内ARP的实现**

以HostA向HostB发送数据报文为例，结合 [图1](#ZH-CN_CONCEPT_0161317276__f66ced1d6e702465f96b3a61ece0bf8cc) 详细介绍同网段内ARP是如何实现的。

**图1** 同网段HostA和HostB之间地址解析的实现过程

<br>

![](基本原理_61317276.assets/fig_dc_fenix_nls_ip_feature_arp_000202_2.png)

1. HostA查看自己的ARP表，发现没有目的主机HostB的IP地址和MAC地址的映射关系。于是，HostA对外发送ARP请求报文，请求HostB的MAC地址。该ARP请求报文中的源IP地址和源MAC地址为HostA的IP地址和MAC地址，目的IP地址和目的MAC地址分别为HostB的IP地址和全0的MAC地址，以太网源MAC地址和目的MAC地址分别为HostA的MAC地址和广播MAC地址。
2. Device B收到ARP请求报文后，将该报文在本网段内广播。
3. HostB收到ARP请求报文后，更新自己的ARP表，将HostA的MAC地址添加到ARP表，同时向该ARP请求报文的发送方HostA发送ARP应答报文。该ARP应答报文中的源IP地址和源MAC地址为HostB的IP地址和MAC地址，目的IP地址和目的MAC地址为HostA的IP地址和MAC地址，以太网源MAC地址和目的MAC地址分别为HostB的MAC地址和HostA的MAC地址。
  > **说明**
  > 由于ARP请求报文的目的IP地址不是 Device A的IP地址，所以 Device A在收到ARP请求报文后，直接将其丢弃。
4. Device B收到该ARP应答报文后，将该报文转发给目的主机HostA。
5. HostA收到ARP应答报文后，更新自己的ARP表，将HostB的MAC地址添加到ARP表中，同时将数据报文发送给HostB。

**不同网段间ARP的实现**

> **说明**
> ARP报文是二层报文，所以ARP协议只适用于同一物理网段。当不同网段的两台主机需要通信时，发送方主机先将数据报文发给默认网关，再由默认网关完成将数据报文转发给接收方主机的操作。因此，不同网段间的ARP可以分为两个相同网段内的ARP来实现。通过这两个相同网段内ARP的实现，使得不同网段间主机可以互通。

以HostA向HostC发送数据报文为例，结合 [图2](#ZH-CN_CONCEPT_0161317276__fc1f354af07c04fc99888dc893f5a2452) 和 [图3](#ZH-CN_CONCEPT_0161317276__f60df20b3164e419bb5e382afef0c9272) 详细介绍不同网段间ARP是如何实现的。

[图2](#ZH-CN_CONCEPT_0161317276__fc1f354af07c04fc99888dc893f5a2452) 描述了同网段的HostA和 Device A之间ARP的实现过程。通过这个网段内ARP的实现，HostA可以将数据报文发送到 Device A。

**图2** 同网段内HostA和 Device A之间地址解析的实现过程

<br>

![](基本原理_61317276.assets/fig_dc_fenix_nls_ip_feature_arp_000203_2.png)

1. HostA查看自己的ARP表，发现没有到达目的主机HostC的默认网关 Device A的接口Interface1的IP地址和MAC地址的映射关系。于是，HostA对外发送ARP请求报文，请求 Device A的接口Interface1的MAC地址。该ARP请求报文中的源IP地址和源MAC地址为HostA的IP地址和MAC地址，目的IP地址和目的MAC地址分别为 Device A的接口Interface1的IP地址和全0的MAC地址，以太网源MAC地址和目的MAC地址分别为HostA的MAC地址和广播MAC地址。
2. Device B收到ARP请求报文后，将该报文在本网段内广播。
3. 网关 Device A收到ARP请求报文后，更新自己的ARP表，将HostA的MAC地址添加到ARP表，同时向该ARP请求报文的发送方HostA发送ARP应答报文。该ARP应答报文中的源IP地址和源MAC地址为 Device A的接口Interface1的IP地址和MAC地址，目的IP地址和目的MAC地址为HostA的IP地址和MAC地址，以太网源MAC地址和目的MAC地址分别为 Device A的接口Interface1的MAC地址和HostA的MAC地址。
  > **说明**
  > 由于ARP请求报文的目的IP地址不是HostB的IP地址，所以HostB在收到ARP请求报文后，直接将其丢弃。
4. Device B收到该ARP应答报文后，将该报文转发给目的主机HostA。
5. HostA收到ARP应答报文后，更新自己的ARP表，将 Device A的接口Interface1的MAC地址添加到ARP表中，同时将数据报文发送给 Device A。

[图3](#ZH-CN_CONCEPT_0161317276__f60df20b3164e419bb5e382afef0c9272) 描述了同网段的 Device A和HostC之间ARP的实现过程。通过这个网段内ARP的实现， Device A可以将数据报文发送给HostC。

**图3** 同网段内 Device A和HostC之间地址解析的实现过程

<br>

![](基本原理_61317276.assets/fig_dc_fenix_nls_ip_feature_arp_000204_2.png)

首先， Device A通过查询路由表，将数据报文从Interface1发送到Interface2。

1. Device A查看自己的ARP表，发现没有目的主机HostC的IP地址和MAC地址的映射关系。于是， Device A发送ARP请求报文，请求HostC的MAC地址。该ARP请求报文中的源IP地址和源MAC地址为 Device A的接口Interface2的IP地址和MAC地址，目的IP地址和MAC地址分别为HostC的IP地址和全0的MAC地址，以太网源MAC地址和目的MAC地址分别为 Device A的接口Interface2的MAC地址和广播MAC地址。
2. Device C收到ARP请求报文后，将该报文在本网段内广播。
3. HostC收到ARP请求报文后，更新自己的ARP表，将 Device A的接口Interface2的MAC地址添加到ARP表，同时向该ARP请求报文的发送方 Device A发送ARP应答报文。该ARP应答报文中的源IP地址和源MAC地址为HostC的IP地址和MAC地址，目的IP地址和目的MAC地址为 Device A的接口Interface2的IP地址和MAC地址，以太网源MAC地址和目的MAC地址分别为HostC的MAC地址和 Device A的接口Interface2的MAC地址。
  > **说明**
  > 由于ARP请求报文的目的IP地址不是HostD的IP地址，所以HostD在收到ARP请求报文后，直接将其丢弃。
4. Device C收到该ARP应答报文后，将该报文发送给目的设备 Device A。
5. Device A收到ARP应答报文后，更新自己的ARP表，将HostC的MAC地址添加到ARP表中，同时将数据报文发送给HostC。

通过以上两个同网段内ARP的实现，不同网段间的主机HostA和HostC可以进行数据报文的传送。

> **说明**
> 1. ARP请求报文以广播形式发送，而ARP应答报文是以单播形式发送。
> 2. 交换设备 Device B和 Device C在上述的ARP实现过程中，只是起到了转发的作用，不会对经过本设备的以太报文作任何更改。
