#### [PDU会话重建判断](#ZH-CN_CONCEPT_0248998102)

对于非本地SMF/I-SMF的PDU会话连接， UNC 可以基于NF Instance ID进行识别。NF Instance ID命名规则采用通用唯一标识符(UUID)版本4格式：time-low“-”time-mind“-”time-high-and-version“-”clock-seq-and-reserved clock-seq-low“-”node。

*表1 NF Instance ID命名规则*

| 字段 | time-low | time-mind | time-high-and-version | clock-seq-and-reserved<br>clock-seq-low | clock-seq-and-reserved<br>clock-seq-low | clock-seq-and-reserved<br>clock-seq-low | node |
| --- | --- | --- | --- | --- | --- | --- | --- |
| 字段位数（16进制） | 8位 | 4位 | 4位 | 4位 | 4位 | 4位 | 12位 |
| 字段含义 | 时间戳的低位 | 时间戳的中间位 | 时间戳高位部分与版本号 | 随机变量与时钟序列低位部分 | 随机变量与时钟序列低位部分 | 随机变量与时钟序列低位部分 | 节点标识符 |
| 举例 | f81d4fac | 7dec | 41d0 | a765 | a765 | a765 | 00a0c91e6bf6 |

- **AMF侧****识别非本地SMF的PDU会话连接**
  运营商需要在node部分使用固定的字符表示指定的区域 （ AMF通过 **[ADD SMFRESEL](../../../../../OM参考/命令/UNC MML命令/业务服务管理/5G接入业务管理/移动性管理/NAS传输管理/本地SMF的PDU会话重建功能管理/增加本地SMF重选策略（ADD SMFRESEL）_45495622.md)** 命令的 “区域标识起始位置” 和 “区域标识终止位置” 决定 ） 。 UNC 会针对AMF和SMF的NF Instance ID中node部分的指定区域位进行精确匹配，如果不相同，说明用户使用了非本地的SMF，判断需要进行PDU会话重建。
  举例：假设运营商规划node部分的第5、6个字符表示区域信息，如 [表2](#ZH-CN_CONCEPT_0248998102__table100) 所示例， 0A 表示A省， 10 表示B省。UE在移动之前(A省)，所采用的AMF和SMF是同区域的。UE移动到B省后， UNC 根据NF Instance ID识别出AMF和SMF不属于同一省份，为该PDU会话连接设置“重选本地SMF标识”，待UE处于CM-IDLE态之后进行 [PDU会话重建连接](#ZH-CN_CONCEPT_0248998102__section016817119166) ，为UE选择与AMF同一区域的SMF。
  *表2 用户在移动过程中所使用SMF示例*

  | UE移动前 | SMF | 789e1234-e56b-42d3-a789-4200<br>0A<br>000001 |
  | UE移动后PDU会话重建前 | AMF | 567e1234-e89b-49d1-a123-1234<br>10<br>000001 |
  | UE移动后PDU会话重建前 | SMF | 789e1234-e56b-42d3-a789-4200<br>0A<br>000001 |
  | UE移动后PDU会话重建后 | AMF | 567e1234-e89b-49d1-a123-1234<br>10<br>000001 |
  | UE移动后PDU会话重建后 | SMF | 666e4321-e12b-46d1-a123-7890<br>10<br>000001 |
- **AMF侧识别存在I-SMF的PDU会话连接**
  针对已经存在PDU会话的UE，AMF判断如果需要插入、改变I-SMF，说明原SMF/I-SMF已经不能处理当前UE所在的位置区域，AMF会对此会话进行PDU会话重建标记，判断需要进行PDU会话重建。
- **SMF侧****识别非本地SMF的PDU会话连接**
  运营商需要在node部分使用固定的字符表示指定的区域 （ SMF通过命令的 **[ADD SMFPDUREACT](../../../../../OM参考/命令/UNC MML命令/业务服务管理/会话管理/接入管理/跨区域PDU会话管理/跨区域PDU会话重建/增加跨区域PDU会话重建策略（ADD SMFPDUREACT）_64343866.md)** 命令的 “区域标识起始位置” 和 “区域标识终止位置” 决定 ） 。 UNC 会针对当前SMF和插入的I-SMF的NF Instance ID中node部分的指定区域位进行精确匹配，如果不相同，说明用户使用了非本地的I-SMF，判断需要进行PDU会话重建。
  举例：假设运营商规划node部分的第5、6个字符表示区域信息，如 [表3](#ZH-CN_CONCEPT_0248998102__table15597163803019) 所示例， 0A 表示A省， 10 表示B省。UE只存在缺省的QoSFlow，UE移动到B省后， UNC 根据NF Instance ID识别出SMF和I-SMF不属于同一省份，进行 [PDU会话重建连接](实现原理_48998102.md#ZH-CN_CONCEPT_0248998102__section016817119166) ，为UE重新选择新接入区域的SMF。
  *表3 用户在移动过程中所使用SMF示例*

  | UE行为 | 网元名称 | NF Instance ID |
  | --- | --- | --- |
  | UE移动后PDU会话重建前 | SMF | 789e1234-e56b-42d3-a789-4200<br>0A<br>000001 |
  | UE移动后PDU会话重建前 | I-SMF | 666e4321-e12b-46d1-a123-7890<br>10<br>000001 |
  | UE移动后PDU会话重建后 | SMF | 567e1234-e89b-49d1-a123-1234<br>10<br>000001 |

#### [PDU会话重建连接](#ZH-CN_CONCEPT_0248998102)

涉及PDU会话重建的流程为包含如下：

- Inter-AMF的移动注册更新流程
- Inter-AMF的Handover流程
- EPS到5GS的空闲态移动性
- EPS到5GS切换
- Inter-AMF的重定向流程

UNC 针对需要重建的PDU会话进行重建：

- AMF侧：在UE处于CM-IDLE态且满足以下PDU会话重建条件后寻呼UE，释放**“**重选本地SMF标识**”**标识的PDU会话连接，要求UE进行PDU会话重建。
    - 用户处于空闲态的时长达到一定阈值，避免大量用户同时重建。阈值通过 **[ADD SMFRESEL](../../../../../OM参考/命令/UNC MML命令/业务服务管理/5G接入业务管理/移动性管理/NAS传输管理/本地SMF的PDU会话重建功能管理/增加本地SMF重选策略（ADD SMFRESEL）_45495622.md)** 命令中的 “CMIDLETIME” 设置。
    - 到达开启PDU会话重建的时间，如“03:00～04:00”、“00:00～23:59”等时间段。此时间段通过 **[ADD SMFRESEL](../../../../../OM参考/命令/UNC MML命令/业务服务管理/5G接入业务管理/移动性管理/NAS传输管理/本地SMF的PDU会话重建功能管理/增加本地SMF重选策略（ADD SMFRESEL）_45495622.md)** 命令中的 “ST” 和 “ET” 设置。
- SMF侧：对PDU会话内只有缺省QosFlow的UE，释放PDU会话连接，要求UE进行PDU会话重建。

**AMF侧实现的PDU会话重建流程如 [图1](#ZH-CN_CONCEPT_0248998102__fig2624175323710) 所示。**

**图1** AMF发起的PDU会话重建流程

<br>

![](实现原理_48998102.assets/zh-cn_image_0249551729_2.png)

1. AMF识别需要重建的PDU会话，对对应的UE进行寻呼，要求UE进行PDU会话释放。
2. UE发起Service Request流程，UE被寻呼到。
3. AMF向SMF发送Nsmf_PDUSession_ReleaseSMContext Request消息，携带原因值为“PDU_SESSION_STATUS_MISMATCH”释放“重选本地SMF标识”标识的PDU会话连接。
4. SMF返回Nsmf_PDUSession_ReleaseSMContext Response消息，通知该PDU Session已经被释放。
5. AMF向UE返回Service Accept响应消息，携带PDU Session Status信元通知该PDU Session已经被释放。
6. UE发起PDU Session Establishment Request，重建PDU会话。
7. AMF选择与同区域的SMF2，发送Nsmf_PDUSession_CreateSMContext Request，请求创建SM上下文。
8. SMF2向AMF返回Nsmf_PDUSession_CreateSMContext Response确认接受创建PDU会话。

**SMF侧实现的PDU会话重建流程如 [图2](实现原理_48998102.md#ZH-CN_CONCEPT_0248998102__fig1154143083514) 所示。**

**图2** SMF发起的PDU会话重建流程

<br>

![](实现原理_48998102.assets/zh-cn_image_0000002212224557_2.png)

1. SMF1识别需要重建的PDU会话，触发PDU Session Release流程。

2. 步骤2~4，SMF1/I-SMF通知UPF/I-UPF释放IP地址&用户面资源。

3. 步骤5~14，SMF通知UE释放PDU会话连接，在AMF向UE发送的PDU Session Release Command消息中携带原因值为“reactivation requested”，UE收到PDU会话释放消息时，基于原因值“reactivation requested”释放当前流程。

15. UE基于原因值“reactivation requested”，在流程释放完后，重新发起PDU Session Establishment Request，重建PDU会话。

16. AMF选择与同区域的SMF2，发送Nsmf_PDUSession_CreateSMContext Request，请求创建SM上下文。

17. SMF2向AMF返回Nsmf_PDUSession_CreateSMContext Response确认接受创建PDU会话。
