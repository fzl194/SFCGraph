**AMF Region、AMF Set、AMF Pool**

协议定义AMF的标识为GUAMI（Globally Unique AMF Identifier），GUAMI格式如下所示，包括4部分：

![](实现原理_98576516.assets/zh-cn_image_0207939593_2.png)

- MCC/MNC：MCC+MNC即为AMF所属的PLMN。
- AMF Region ID：协议定义，gNodeB需要连接AMF Region中的所有AMF。所以，服务于一组相同gNodeB的AMF被规划为一个AMF Region，服务于不同gNodeB组的AMF被规划为不同的AMF Region。
- AMF Set ID：AMF Region内按照网络切片规划不同的AMF Set，一个AMF Set的AMF对网络切片的支持能力完全相同。AMF Set内的各AMF按照自身的处理能力规划一个权重，权重取值范围是0~255，权重越大被选择到的概率越大。
- AMF Pointer：AMF节点在AMF Set内的标识。

GUAMI中的MCC/MNC + AMF Region ID全球唯一标示了一个AMF Region，AMF Set ID在AMF Region内唯一，AMF Pointer则在AMF Set内唯一。

AMF Region、AMF Set、gNodeB之间的互联关系如下 [图1 AMF组网示意](#ZH-CN_TOPIC_0198576516__fig5796141723817) 所示。

**图1** AMF组网示意

<br>

![](实现原理_98576516.assets/zh-cn_image_0198954885_2.png)

5G网络中引入了网络切片，因此5G网络中AMF Region下又增加一层AMF Set，AMF Set等同于AMF Pool。本文中提到的AMF Pool等同于AMF Set。

**5G-GUTI**

UE首次接入该AMF Region时，gNodeB根据UE请求的网络切片信息选择AMF Set，并在该AMF Set中根据权重选择AMF接入该UE。UE在该AMF上注册成功后，AMF为UE分配5G-GUTI作为UE临时身份标识（用来避免在网络上传输永久身份标识，防止攻击者跟踪用户的位置及活动状况），其中包含GUAMI信息。后续UE在该AMF Region范围内移动时，UE会将从5G-GUTI中获取的GUAMI信息传递给gNodeB，gNodeB根据该GUAMI将消息直接路由给对应的AMF。

5G-GUTI和EPS GUTI对比和映射关系如下图所示。

![](实现原理_98576516.assets/zh-cn_image_0207940238_2.png "点击放大")

当UE在4/5G之间移动时，4G/5G互操作流程中，AMF作为新网元，可以识别对端网元是AMF还是MME，但是MME作为存量网元，尽量不识别对端网元是AMF还是MME，统一按照对端是MME进行处理。

- 4G-->5G注册更新：用户从4G网络到5G网络的注册更新流程中，会把4G-GUTI映射为5G-GUTI在NAS消息中携带给AMF，并通过UE status信元知会AMF该5G-GUTI为4G-GUTI映射而来的。AMF把该5G-GUTI信息还原成4G-GUTI，取出其中的GUMMEI，组装MME FQDN，再通过DNS查询，获取MME的S10接口地址，完成互操作流程。
- 5G-->4G TAU：用户从5G网络到4G网络的TAU流程中，AMF会把5G-GUTI映射为4G-GUTI在NAS消息中携带给MME，MME处理方案如下：
  MME不识别对端是MME还是AMF，按照4G-GUTI进行解析，取出其中的GUMMEI，组装MME FQDN，通过DNS查询，获取MME的S10接口地址或（实际为AMF的N26接口地址），完成互操作流程。这种处理方案要求MME Pool和AMF Pool不能完全独立规划，因为需要把AMF的GUAMI映射为GUMMEI（称为AMF的Mapped GUMMEI），并将其N26接口地址作为S10接口地址，在DNS Server上配置AMF的Mapped MME FQDN对应的S10接口地址。

  > **说明**
  > 为了避免Mapped MME FQDN与现网MME的Native MME FQDN冲突，要求GUAMI不仅在全球的GUAMI内唯一，并且其Mapped GUMMEI与4G网络中的Native GUMMEI不能冲突，MME pool和AMF Pool的规划不完全独立。

**部署方式**

UNC 支持MME与AMF共建：

- UNC作为MME时，在S1 Setup流程中会把自己作为AMF时支持的GUAMI映射为Mapped GUMMEI，和自己作为MME时支持的Native GUMMEI一起发送给eNodeB。
  当UE进行5G->4G TAU流程时，UE会把5G-GUTI中获取的GUAMI信息映射为Mapped GUMMEI，并将其在空口发送给eNodeB，eNodeB根据该信息寻址到与原AMF共建的MME。
- UNC作为AMF时，在NG Setup流程中会把自己作为MME时支持的GUMMEI映射为Mapped GUAMI，和自己作为AMF时支持的Native GUAMI一起发送给gNodeB。
  当UE进行4G->5G注册更新流程时，UE会把4G-GUTI中获取的GUMMEI信息映射为Mapped GUAMI，并将其在空口发送给gNodeB，gNodeB根据该信息寻址到与原MME共建的AMF。

  > **说明**
  > 由于3GPP协议限制，AMF下发给gNodeB的GUAMI，还无法区分哪些是Native GUAMI，哪些是Mapped GUAMI。因此，当前在组网上要求gNodeB连接的AMF Region内的融合MME/AMF节点的Native GUMMEI与Mapped GUMMEI不冲突。否则，可能造成UE从4G迁移到5G网络后，gNodeB未选择与用户原先接入的MME共建的MME/AMF节点。因此，必须满足GUAMI不仅在全球的GUAMI内唯一，并且其Mapped GUMMEI与4G网络中的Native GUMMEI不能冲突的条件。

**负载均衡原理**

在AMF Set的组网中，由于一个gNodeB与AMF Set内的多个AMF连接，所以当TA业务区中的UE发起新的业务时，为了使gNodeB可以根据负载均衡等原则为UE从Set中选择一个服务的AMF，AMF需要将权重下发给gNodeB。

如 [图2](#ZH-CN_TOPIC_0198576516__fig843502274615) 所示，假设AMF1的有效用户容量为4，AMF2的有效用户容量为3，则gNodeB会按4∶3的比例，将UE接入到AMF1和AMF2上，具体分发效果取决于gNodeB的实现方式。

**图2** 负载均衡原理图

<br>

![](实现原理_98576516.assets/zh-cn_image_0198576674_2.png)

AMF通知gNodeB自身的权重（NGAP连接建立时）

在AMF Set中，一个gNodeB与AMF Set中的多个AMF连接，所以当建立专用N2连接时，需要为UE选择一个服务的AMF，这里的选择功能被称为“节点选择功能”。gNodeB是根据各AMF对应的“负荷权重”比例进行选择，AMF的负荷权重值是通过NGAP消息发送给池区中的gNodeB的。AMF的负荷权重值是根据Set中其它AMF节点的可用容量加以设置的。负荷权重值可以通过NG Setup Response消息（ [图3](#ZH-CN_TOPIC_0198576516__fig16636722817) ）或者AMF Configuration Update消息携带给gNodeB（ [图4](#ZH-CN_TOPIC_0198576516__fig55702032101116) ）。

**图3** 在NG设备连接建立时AMF通知gNodeB自己的负荷权重流程图

<br>

![](实现原理_98576516.assets/zh-cn_image_0198602334_2.png)

1. 用户根据AMF Set中各AMF的可用容量的比例关系分别配置各AMF的负荷权重，取值范围为：0～255。
2. AMF保存各自的负荷权重。
3. gNodeB启动后，向AMF发起SCTP偶联建立请求并建立成功。
4. gNodeB向AMF发送NG Setup Request以建立N2接口设备连接。
5. AMF向gNodeB发送NG Setup Response消息将AMF的负荷权重(Relative AMF Capacity)带给gNodeB。Relative AMF Capacity可以通过的[**ADD AMFINFO**](../../../../../OM参考/命令/UNC MML命令/业务服务管理/本局信息管理/AMF/AMF信息管理/增加AMF信息（ADD AMFINFO）_09652298.md)的“相对容量（CAPACITY）”参数配置。
6. gNodeB保存AMF的负荷权重信息。

AMF的负荷权重值可以进行修改，但不会频繁更新。对于一个相对成熟的网络，建议一个月更新一次即可，如在新增了RAN侧或CN侧网元时进行更新。

同时，当AMF容量变更的情况下，也可能需要负荷权重值的变更。如对于一个Set中新增的网元，可以设置一个相对较高的负荷权重值，以便该新增网元尽快承接业务，达到与Set内其它网元的负荷均衡。

**图4** AMF通知gNodeB更新AMF的负荷权重流程图

<br>

![](实现原理_98576516.assets/zh-cn_image_0198602520_2.png)

1. 用户修改AMF的负荷权重，取值范围为：0～255。
2. AMF保存修改后的负荷权重。
3. AMF主动向所有连接的gNodeB发送AMF Configuration Update，其中携带标识该AMF负荷权重的Relative AMF Capacity信元。
4. gNodeB更新AMF的负荷权重，作为后续UE接入时选择AMF的依据。
5. gNodeB若成功更新数据配置，则回复AMF Configuration Update Acknowledge消息给AMF。

**容灾原理**

实际上，负荷分担处理已经使AMF Set组网具备了一定的AMF级的容灾能力。当AMF Set中某AMF发生故障时，gNodeB识别该AMF故障后，gNodeB会将本来由该AMF提供服务的用户发起的新业务转移到Set中其他有效的AMF上；同时对新接入的用户进行AMF选择时不会再考虑该AMF，从而实现AMF Set内AMF间的容灾。

如 [图5](#ZH-CN_TOPIC_0198576516__fig102287528278) 所示，在非Pool组网情况下，某UE通过gNodeB2接到AMF1上，如果AMF1故障将导致gNodeB2到AMF1的链路断开，此时UE的注册更新流程失败。

**图5** 非Pool组网情况下故障示意图

<br>

![](实现原理_98576516.assets/zh-cn_image_0198602665_2.png)

如 [图6](#ZH-CN_TOPIC_0198576516__fig485805519287) 所示，在Pool组网情况下，规划图中的gNodeB分别连接到两个AMF上，UE通过gNodeB2连接到AMF1上，在AMF1故障以致gNodeB2到AMF1的链路断开的情况下，UE在进行移动注册更新的时候流程时可以通过gNodeB2将消息发到AMF2上，由AMF2提供业务。（如果只是gNodeB2到AMF1的链路断开，AMF1设备可用，则AMF2收到注册更新请求消息后，可以进行注册更新流程；如果AMF1设备彻底故障，注册更新请求消息直接被拒绝，UE会通过AMF2重新发起注册流程，这个时候gNodeB2会将注册消息发到AMF2上，即相当于UE通过gNodeB2注册到AMF2，由AMF2提供业务。）

  **图10** Pool区域内初始注册流程的容灾处理消息流程

- 移动注册更新流程的容灾处理
  AMF Pool容灾场景下，用户发的移动注册更新消息中携带了GUAMI，但是gNodeB到GUAMI对应AMF的N2接口故障触发移动注册更新流程。
  gNodeB发现UE注册的AMF故障，将Registration Request消息基于负载均衡的原则发送到Pool区域内其他状态正常的AMF上。若用户原注册的AMF正常，则触发了AMF Pool区域内移动注册更新流程，如 [图10](#ZH-CN_TOPIC_0198576516__fig197312476420) 所示。若用户原来注册的AMF故障，则New AMF回拒绝消息，用户重新发起初始注册到新的AMF上。
- Service Request流程的容灾处理
  **图11** Service Request流程的容灾处理

  <br>

  ![](实现原理_98576516.assets/zh-cn_image_0000002153271541_2.png)
  Pool容灾场景下，用户发的Service请求消息中携带了5G-S-TMSI，但是gNodeB到5G-S-TMSI对应AMF的N2接口故障触发重新发起注册。
    1. Service Request消息
      gNodeB发现UE注册的AMF故障，将Service Request消息基于负载均衡的原则发送到Pool区域内其他状态正常的AMF上。
    2. Service Reject消息
      New AMF没有用户上下文，向UE回Service Reject。
    3. Registration Request消息
      UE发起注册请求，gNodeB发现UE注册的AMF故障，将Registration Request消息基于负载均衡的原则发送到Pool区域内其他状态正常的AMF上。
  其他流程描述与非AMF POOL组网场景相同。
