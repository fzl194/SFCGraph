# EPS中移动性管理与2G/3G中的差别

新增标识：

- GUTI（Globally Unique Temporary UE Identity）：MME分配，全局唯一，包含两个部分：
    - GUMMEI：GUMMEI由MCC、MNC、MME标识组成。
    - M-TMSI：M-TMSI长度为32个bit，在一个MME内唯一的标识一个UE。
      GUTI可以在附着或跟踪区更新过程中隐式分配，也可以通过GUTI重分配过程显式分配。
- TAI（Tracking Area Identity）：TAI= MCC (Mobile Country Code)+MNC (Mobile Network Code) + TAC (Tracking Area Code)
- TA List：如 [图1](#ZH-CN_CONCEPT_0166636629__fig1) 所示，将若干跟踪区TA组合为一个TA List。
    - 将若干跟踪区TA组合为一个TA List，移动终端在此TA List包含的所有TA区中移动时候，不再需要发起TA更新。
    - 网络寻呼用户时，在一个TA List包含的所有TA区中寻呼用户。
    - MME在Attach Accept，TAU Accept或者GUTI Reallocation Command消息中会将TA List下发给UE。

以下通过签约数据机制、安全、状态分析、寻呼以及位置更新来描述EPS与2G/3G的差别。

签约数据机制：

*表1 签约数据机制*

| 签约数据 | 2G/3G | EPS |
| --- | --- | --- |
| 签约数据来源 | HLR | HSS |
| 传递数据协议栈 | MAP+SS7 | Diameter+SCTP |
| 获取签约数据寻址方式 | GT寻址方式，根据IMSI选择HPLMN的HLR。 | 根据IMSI选择HSS |
| 获取依据 | IMSI | IMSI |
| 签约数据获取流程 | 签约数据获取流程与鉴权流程分离，独立完成。 | 签约数据获取流程与鉴权流程分离，独立完成。 |
| 签约数据保留策略 | 签约数据在用户入网后保存在SGSN上，用户分离后签约数据可以继续保留在SGSN上。 | 签约数据在用户入网后保存在MME上，用户分离后签约数据可以继续保留在MME上。 |
| 签约数据更新策略 | 签约数据有修改时，HLR会同步到SGSN上。 | 签约数据有修改时，HSS会同步到MME上。 |

安全：

*表2 安全*

| 安全管理 | 2G | 3G | EPS |
| --- | --- | --- | --- |
| 鉴权方式 | 单向鉴权 | 双向鉴权 | 双向鉴权 |
| 安全向量 | 三元组（Rand，Sres，Kc） | 五元组（Rand，Xres，CK，IK，AUTN） | 全新EPS安全向量(RAND， AUTN， XRES， K<br>ASME<br>)。实现UMTS鉴权向量与EPS鉴权向量的隔离。 |
| 是否完整性保护 | 否 | 是 | 是 |
| 是否兼容SIM | 是 | 兼容 | 不兼容 |
| 层次化安全架构 | 否 | 否 | 对NAS层和AS层分别进行保护，同时对NAS层密钥和AS层密钥也进行分级管理。NAS层与AS层也分别进行算法协商。 |

> **说明**
> 安全部分详细介绍，请参见安全管理特性描述。

状态分析：

*表3 非注册态区别*

| 分离/退网态 | 2G/3G | EPS |
| --- | --- | --- |
| 不同点 | 2G/3G Detach后不保存安全上下文。 | 网络侧保存用户的签约数据/鉴权集等安全上下文。 |
| 相同点 | 空口资源释放，基站侧不保存用户任何信息。CN侧无MS的动态信息。 | 空口资源释放，基站侧不保存用户任何信息。CN侧无MS的动态信息。 |

*表4 Active态区别*

| Active态 | 2G/3G | EPS |
| --- | --- | --- |
| 位置准确程度 | SGSN准确到路由区和服务RNC的范围。RNC准确到小区。 | MME准确到Tracking Area和小区。 |
| 可否位置更新 | 是，路由区变化时更新。 | UE接入TA LIST变化时TA会更新。 |

*表5 IDLE态分析*

| IDLE态 | 2G/3G | EPS |
| --- | --- | --- |
| 位置准确程度 | 可知，准确到路由区。 | 可知，准确到TA List。 |
| 可寻呼到MS | 可以 | 可以 |
| 可否位置更新 | 是，路由区/TA LIST变化时更新。 | 是，路由区/TA LIST变化时更新。 |
| 控制上下文 | 核心网侧保留用户上下文和PDP上下文，仅删除接入侧数据承载信息。SGSN与GGSN之间的GTPU隧道保留。 | MME保留用户上下文和承载上下文信息，删除接入侧数据承载信息。S-GW保留CN侧数据承载信息。 |
| 安全上下文 | 核心网侧保留。 | 核心网侧保留。 |
| 接入侧资源 | 接入侧已释放MS的资源，不保留上下文。 | 接入侧已释放MS的资源，不保留上下文。 |
| MS侧行为 | 该状态下MS均不会进行频繁切换。 | 该状态下MS均不会进行频繁切换。 |

寻呼：

*表6 寻呼区别*

| 寻呼 | 3G | EPS |
| --- | --- | --- |
| 寻呼范围 | 路由区、小区 | TA List、TA |
| 寻呼过程 | 下行数据终结在SGSN或者GGSN，由SGSN发起寻呼。 | 下行数据终结在S-GW上，由S-GW通知MME发起寻呼。 |
| 寻呼结果 | MS发起业务请求流程，重新建立信令连接以及数据承载，进入连接态。 | MS发起业务请求流程，重建数据承载，状态会修改。 |
| 数据缓存 | SGSN的用户面 | S-GW |
| 发起寻呼 | SGSN的控制面 | MME |

位置管理：

*表7 位置管理区别*

| 位置管理 | 3G | EPS |
| --- | --- | --- |
| 切换时业务连续策略 | RNC缓存转发策略 | eNodeB缓存转发策略 |
| 位置更新机制 | 通知SGSN当前所在的路由区信息等。当发生SGSN间路由区更新时，控制上下文、安全上下文等信息发生迁移，原OLD SGSN上资源全部删除。 | 通知MME MS当前所在的跟踪区信息。当发生MME间跟踪区更新时，控制上下文、安全上下文等信息发生迁移，原OLD MME上资源全部删除。 |
| 位置更新是否通知用户数据管理中心 | 发生SGSN间的位置更新流程时，需要通知HLR。 | 发生MME间的TAU流程时，需要通知HSS。 |
