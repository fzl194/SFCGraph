#### [注册流程](#ZH-CN_TOPIC_0000002147111957)

如果非紧急注册流程并且不存在活动的紧急PDU会话，UE可以在注册过程中指示其支持寻呼分组的能力，流程如下：

**图1** 注册流程

<br>

![](实现原理_47111957.assets/zh-cn_image_0000002149313217_2.png)

1. UE发起注册流程，发送Registration Request消息给NG-RAN。当注册流程非紧急注册并且不存在活动的紧急PDU会话时，如果UE支持分组寻呼能力时，会在Registration Request消息中携带以下关键信元：
  | **信元** | **说明** |
  | --- | --- |
  | 5GMM capability: RCMAP | RCMAP（Registration complete message for acknowledging negotiated PEIPS assistance information）：表示当注册接受消息中携带协商的PEIPS辅助信息信元时，发送注册完成消息的能力。<br>- 0：不支持发送协商PEIPS辅助信息的Registration Complete消息<br>- 1：支持发送协商PEIPS辅助信息的Registration Complete消息 |
  | 5GMM capability: NR-PSSI | NR-PSSI（NR paging subgroup support indication）：支持NR寻呼分组的能力。<br>- 0：不支持NR寻呼分组<br>- 1：支持NR寻呼分组 |
  | Requested PEIPS assistance information | 请求的PEIPS辅助信息，可能包含寻呼分组信息或者UE寻呼概率信息。 |
2. NG-RAN发送Registration Request消息给AMF。
3. AMF保存UE的分组寻呼能力以及寻呼概率，根据自身配置决策出寻呼分组，进行注册流程的其它处理。
  UNC 支持通过 **[ADD NGPEIPSPLCY](../../../../../OM参考/命令/UNC MML命令/业务服务管理/接口管理/N2接口管理/NGAP接口寻呼管理/智能寻呼策略/增加基于分组的智能寻呼策略（ADD NGPEIPSPLCY）_35319045.md)** 命令，根据UE的IMSI、归属网络、DNN、切片类型、寻呼概率等配置寻呼分组信息，同时也支持使用UE注册请求中携带的寻呼分组。
4. AMF在注册成功后，向NG-RAN发送INITIAL CONTEXT SETUP REQUEST，消息中携带Core Network Assistance Information for RRC INACTIVE信元和Registration Accept消息。其中，Core Network Assistance Information for RRC INACTIVE信元中包含PEIPS assistance information，用于向NG-RAN指示寻呼分组信息。
  如果UE在注册请求消息的5GMM能力信元中设置RCMP为“支持PEIPS协商参数的注册完成消息发送”，且注册接受消息中包含PEIPS辅助信息信元。AMF启动定时器T3550，并进入状态5GMM-COMMON-PROCEDURE-INITIATED状态。
  > **说明**
  > - INITIAL CONTEXT SETUP REQUEST消息是否携带Core Network Assistance Information for RRC INACTIVE信元受**[SET NGMMFUNC](../../../../../OM参考/命令/UNC MML命令/业务服务管理/5G接入业务管理/移动性管理/MM协议参数管理/5G移动性管理/设置5G移动性管理功能（SET NGMMFUNC）_09653748.md)**命令的“RRC Inactive功能（RRCINACTIVE）”参数和软参[DWORD60 BIT3](../../../../../OM参考/软件参数/UNC软件参数/业务软件参数/软件参数（AMF）/移动性管理/DWORD60 BIT3 控制AMF下发Core Network Assistance Informa_ba1b8364_07524608.md)控制。
  > - 如果**[ADD NGPEIPSPLCY](../../../../../OM参考/命令/UNC MML命令/业务服务管理/接口管理/N2接口管理/NGAP接口寻呼管理/智能寻呼策略/增加基于分组的智能寻呼策略（ADD NGPEIPSPLCY）_35319045.md)**命令配置为使用配置的寻呼分组信息，但用户未匹配上任何配置，则Core Network Assistance Information for RRC INACTIVE信元和Registration Accept消息中不携带PEIPS assistance information。
5. NG-RAN向UE透传Registration Accept消息，消息中携带Negotiated PEIPS assistance information，用于向UE指示寻呼分组信息。
6. NG-RAN向AMF返回INITIAL CONTEXT SETUP RESPONSE消息。
7. 如果UE在Registration Request消息携带了RCMAP指示，则UE在收到Registration Accept消息后需要向AMF回复Registration Complete消息以确认收到了Negotiated PEIPS assistance information。NG-RAN将Registration Complete消息透传给AMF。
8. AMF接收到Registration Complete消息时，AMF停止定时器T3550并改变到状态5GMM-REGISTERED。
  如果在从UE接收到注册完成消息之前发生底层故障，并且定时器T3550正在运行，则：
    - AMF应中止该过程，如果是初始注册流程，则进入5GMM-REGISTERED状态；如果是移动性注册流程和周期性注册更新流程，进入5GMM-IDLE状态。
    - 如果Registration Accept消息携带协商的PEIPS辅助信息信元包含新的寻呼分组ID，并且UE先前被分配了不同的寻呼分组ID，则AMF应认为新旧寻呼分组ID都是有效的。
    - 如果Registration Accept消息携带的协商的PEIPS辅助信息信元不包含寻呼分组ID或没有携带协商的PEIPS辅助信息信元，则AMF应删除存储在UE的5GMM上下文中的所有旧寻呼分组ID。
    - 如果在Registration Accept消息中向UE分配了新的5G-GUTI和新的寻呼分组ID，且后续UE使用新的5G-GUTI，则AMF应认为新的寻呼分组ID是有效的，如果有旧的寻呼分组ID，则认为旧的寻呼分组ID是无效的。
    - 如果在Registration Accept消息中向UE分配了新的5G-GUTI和新的寻呼分组ID，且后续UE使用旧的5G-GUTI，则AMF应该发起配置更新流程，消息中包含PEIPS辅助信息。
    - 如果在Registration Accept消息中向UE分配了新的5G-GUTI和没有分配寻呼分组ID，且后续UE使用旧的5G-GUTI，则AMF应该发起配置更新流程，消息中不包含PEIPS辅助信息。
    - 如果在Registration Accept消息中向UE分配了新的5G-GUTI和新的寻呼分组ID，且后续UE使用旧的5G-GUTI，如果要发起寻呼：
          - 在Registration Accept消息之前没有为UE分配过寻呼分组，则使用新的寻呼分组ID对UE进行寻呼。
          - 在Registration Accept消息之前为UE分配过寻呼分组，且旧的寻呼分组ID和新的寻呼分组ID相同，则使用新的寻呼分组ID对UE进行寻呼。
          - 在Registration Accept消息之前为UE分配过寻呼分组，且旧的寻呼分组ID和新的寻呼分组ID不同，则先使用旧的寻呼分组ID对UE进行寻呼，再使用新的寻呼分组ID对UE进行寻呼。

如果支持NR寻呼分组的UE在最后一次注册过程中由于3GPP接入的紧急PDU会话激活而没有指示支持NR寻呼分组的能力，当紧急PDU会话通过3GPP接入释放后，UE应发起移动性注册流程和周期性注册更新流程，以指示其支持NR寻呼子分组的能力。

#### [切换流程](#ZH-CN_TOPIC_0000002147111957)

**基于Xn接口切换**

UE从一个源NG-RAN节点切换到一个目标NG-RAN节点，切换过程中使用Xn接口时，流程如 [图2](#ZH-CN_TOPIC_0000002147111957__fig377101312418) 所示。

**图2** 基于Xn接口切换流程

<br>

![](实现原理_47111957.assets/zh-cn_image_0000002140488364_2.png)

1. 目标NG-RAN向AMF发送PATH SWITCH REQUEST消息，通知AMF用户已经移动到新的区域。
2. AMF与SMF交互完成后，根据当前配置决策寻呼分组信息，向目标NG-RAN回复PATH SWITCH REQUEST ACKNOWLEDGE消息，消息中携带Core Network Assistance Information for RRC INACTIVE信元，PEIPS Assistance Information信元包含在Core Network Assistance Information for RRC INACTIVE信元中用于指标寻呼分组。
3. **可选：**如果切换后伴随移动性注册流程或者周期性注册更新流程，则通过移动性注册流程或者周期性注册更新流程将寻呼分组信息通知给UE和NG-RAN。
4. **可选：**如果切换后未伴随移动性注册流程或者周期性注册更新流程，且决策的分组信息有变化，则AMF会主动触发CONFIGURATION UPDATE COMMAND流程，并在消息中携带Updated PEIPS assistance information信元向终端指示新的寻呼分组信息。
  如果在收到CONFIGURATION UPDATE COMPLETE消息之前检测到底层故障，CONFIGURATION UPDATE COMMAND消息中的更新的PEIPS辅助信息信元包括新的寻呼分组ID，并且UE先前被分配了不同的寻呼分组ID，则AMF应将旧寻呼分组ID和新寻呼分组ID都视为有效。

**N2切换**

UE从一个源NG-RAN节点切换到一个目标NG-RAN节点，当NG-RAN之间不存在Xn接口时或者基于Xn接口的切换流程失败（比如，目标NG-RAN与源UPF之间无IP连接）时，需要通过N2接口进行切换

**图3** N2接口切换流程

<br>

![](实现原理_47111957.assets/zh-cn_image_0000002141286876_2.png)

1. 源NG-RAN向源AMF发送HANDOVER REQUIRED消息，通知源AMF有用户要切换。
2. 如果源AMF不能再服务UE，则源AMF根据HANDOVER REQUIRED消息中的targetID选择一个可提供服务的目标AMF。
3. 源AMF向目标AMF发送Namf_Communication_CreateUEContext Request消息创建UE上下文。
4. 目标AMF与SMF交互，建立目标AMF与SMF的关联。
5. 目标AMF根据配置决策寻呼分组信息，向目标NG-RAN发送HANDOVER REQUEST消息，请求建立无线侧网络资源，并在消息中携带Core Network Assistance Information for RRC INACTIVE信元，PEIPS Assistance Information信元包含在Core Network Assistance Information for RRC INACTIVE信元中用于指标寻呼分组。
6. 目标NG-RAN向目标AMF回复HANDOVER REQUEST ACKNOWLEDGE消息。
7. 目标AMF与SMF交互，告知SMF目标NG-RAN的N3隧道信息和N3转发隧道信息。
8. 目标AMF发送Namf_Communication_CreateUEContext Response给源AMF。
9. 源AMF发送HANDOVER COMMAND给源NG-RAN，通知切换准备完成。
10. 源NG-RAN发送HANDOVER COMMAND给UE，UE将释放被目标NG-RAN所拒绝的PDU会话资源。
11. UE发送HANDOVER CONFIRM消息给目标NG-RAN，确认UE切换成功。
12. 目标NG-RAN发送HANDOVER NOTIFY给目标AMF，通知目标AMF，UE已经位于目标小区，目标NG-RAN上切换成功。
13. 会话更新处理。
14. **可选：**如果后续没有伴随移动性注册更新流程，[5](#ZH-CN_TOPIC_0000002147111957__li209241244115113)中决策的寻呼分组信息有变化，则目标AMF会主动触发CONFIGURATION UPDATE COMMAND流程，并在消息中携带Updated PEIPS assistance information信元向终端指示新的寻呼分组信息。
  如果在收到CONFIGURATION UPDATE COMPLETE消息之前检测到底层故障，CONFIGURATION UPDATE COMMAND消息中的更新的PEIPS辅助信息信元包括新的寻呼分组ID，并且UE先前被分配了不同的寻呼分组ID，则目标AMF应将旧寻呼分组ID和新寻呼分组ID都视为有效。
15. 如果伴随了移动性注册更新流程，在移动性注册更新流程中重新决策寻呼分组信息，通过Registration Accept消息知会给终端新的寻呼分组信息。
16. 如果决策的寻呼分组信息有变化，则AMF主动触发UE CONTEXT MODIFICATION REQUEST消息，消息中携带Core Network Assistance Information for RRC INACTIVE信元，PEIPS Assistance Information信元包含在Core Network Assistance Information for RRC INACTIVE信元中用于指标寻呼分组。

#### [寻呼流程](#ZH-CN_TOPIC_0000002147111957)

当AMF寻呼UE时，向NG-RAN发送的PAGING消息中携带PEIPS Assistance Information信元指示寻呼分组信息：

- 如果UE上下文中仅有一个寻呼分组ID，则PAGING消息中携带该寻呼分组ID进行寻呼。
- 如果UE上下文中有新旧两个寻呼分组ID，且两个寻呼分组ID相同，则PAGING消息中携带该寻呼分组ID进行寻呼。
- 如果UE上下文有新旧两个5G-GUTI和新旧两个寻呼分组ID，且两个寻呼分组ID不同，则先使用旧的5G-GUTI、TA List、寻呼分组ID进行寻呼，如果寻呼成功，则AMF刷新上下文中的寻呼分组ID；如果寻呼失败，则使用新的5G-GUTI、TA List、寻呼分组ID进行寻呼。

#### [AMF POOL迁移流程](#ZH-CN_TOPIC_0000002147111957)

对于POOL用户迁移场景需要通过 **[SET AMFSBICMPT](../../../../../OM参考/命令/UNC MML命令/业务服务管理/5G接入业务管理/移动性管理/MM协议参数管理/AMF服务化接口兼容性参数管理/设置AMF服务化接口兼容性参数（SET AMFSBICMPT）_98011756.md)** 命令的 “N14接口是否携带PEIPS信元以及寻呼分组标识（SNDPEIPSINFO）” 参数配置是否向目标AMF传递PEIPS信元以及寻呼分组标识。在 “N14接口是否携带PEIPS信元以及寻呼分组标识（SNDPEIPSINFO）” 参数配置为 “YES” 时：

- 对于连接态用户，待迁移AMF通过调用目标AMF的Namf_Communication_N1MessageNotify服务操作，把UE Context以及Global RAN Node ID、RAN UE NGAP ID等N2接口信息一并发送给目标AMF，其中UE Context中包含PEIPS信元和分组寻呼标识。
- 对于空闲态用户，待迁移AMF主动发起Paging流程，UE响应Paging进入连接态后，待迁移AMF按照连接态用户的迁移方式处理，将UE迁移到AMF Set内的其它AMF上。
