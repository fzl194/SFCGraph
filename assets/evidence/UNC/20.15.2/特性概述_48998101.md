#### [适用NF](#ZH-CN_CONCEPT_0248998101)

AMF、SMF

#### [定义](#ZH-CN_CONCEPT_0248998101)

UE在移动过程中，可能会出现AMF与SMF不在同一区域（例如，不同省份）需要插入I-SMF的情况，导致I-SMF和SMF管理下UPF之间的迂回流量增加，业务时延也会增加。UNC中的AMF支持识别AMF与SMF不在同一区域的PDU会话连接，SMF支持识别I-SMF和SMF不在同一区域的PDU会话连接，AMF和SMF都可以实现让UE重新选择SMF进行PDU会话重建。

#### [客户价值](#ZH-CN_CONCEPT_0248998101)

| 受益方 | 受益描述 |
| --- | --- |
| 运营商 | 减少UE跨区域移动后不同区域AMF和SMF之间的迂回流量，节约移动承载网络资源。 |
| 用户 | 减少跨区域移动后产生的业务时延，提升业务体验。 |

#### [应用场景](#ZH-CN_CONCEPT_0248998101)

大型运营商网络，UE移动可能会涉及跨区域，不同区域AMF和SMF之间的迂回流量较多，占用了原本就不丰富的移动承载网络资源，此场景建议使用本特性。

#### [对系统的影响](#ZH-CN_CONCEPT_0248998101)

- 使用AMF的实现完成此特性：用户新接入的AMF要向需进行本地SMF的PDU会话重建的用户发起寻呼，并要求用户重新建立PDU会话连接。因此，网络中的寻呼次数、PDU会话释放/建立次数会增加。具体增加次数的比例，取决于跨区域移动用户的比例，以及这些用户所具有的PDU会话连接个数。
- 使用SMF的实现完成此特性：SMF删除需要进行PDU会话重建的会话，并要求用户重新建立PDU会话连接。因此，网络中的PDU会话释放/建立次数会增加。具体增加次数的比例，取决于跨区域移动用户的比例，以及这些用户所具有的PDU会话连接个数。

#### [应用限制](#ZH-CN_CONCEPT_0248998101)

- 基于NF Instance ID进行PDU会话重建判断时，运营商需要按照本特性要求的NF Instance ID格式进行规划。
- 本特性仅针对特性开启后的UE跨区域（跨省）移动且AMF改变时生效。
- 本特性开启后，AMF侧实现时是在UE处于CM-IDLE态时才会进行PDU会话重建，避免影响正在进行的业务；SMF侧实现时是对于PDU会话内只有缺省QosFlow的UE才会进行PDU会话重建，避免影响正在进行的业务。
- 本特性只适用于LBO（Local Breakout）组网场景。

#### [原理概述](#ZH-CN_CONCEPT_0248998101)

本特性中，AMF和SMF都具备识别需要重建的PDU会话并要求UE进行PDU会话重建的能力，两者完成的功能相同， **使用此特性时选择一种实现即可** ，下面进行分别介绍。

在5G网络中，AMF根据签约信息等为UE选择合适的归属地SMF进行业务。如 [图1](#ZH-CN_CONCEPT_0248998101__fig109491110124018) 所示，Area1的AMF1会优先为UE选择同区域的SMF1。

当UE从Area1移动到Area2（例如，从A省移动到B省）后，AMF2会为其重新选择一个本地的I-SMF保持业务连续性，SMF1作为数据锚点，AMF2最终选择SMF1进行服务，跨区域之间的I-SMF和SMF1迂回流量增加，造成承载网负荷增加。另外，业务传输由于路径变长而时延增加，对于IMS语音业务等实时类业务会造成用户体验的下降。

- **AMF侧本地PDU会话重建**
  AMF侧本地PDU会话重建实现有两种方式 ，通过 **[ADD SMFRESEL](../../../../../OM参考/命令/UNC MML命令/业务服务管理/5G接入业务管理/移动性管理/NAS传输管理/本地SMF的PDU会话重建功能管理/增加本地SMF重选策略（ADD SMFRESEL）_45495622.md)** 命令中的 “RESELCOND” 选择使用其中一种方式 ：
  当UE从Area1移动到Area2（例如，从A省移动到B省）后：

  - **方式一，根据 NF Instance ID判断，****此种方式需要运营商按照要求在NF Instance ID的node部分规划AMF和SMF的特定区域**：
          - 针对已经存在PDU会话的UE，AMF2能够比较AMF和SMF的NF Instance ID，识别SMF1非本地，并在不影响用户正常业务的情况下（用户变为CM-IDLE态之后）且满足PDU会话重建条件时，通过释放SMF1承载的PDU会话，要求UE发起PDU会话重建的方式，AMF2为UE重新选择同区域Area2的SMF2，减少流量迂回以及业务时延。
          - 针对不存在PDU会话的UE，后续新建的PDU会话按照已有功能的AMF选择SMF策略直接选择Area2的SMF2。
    - **方式二，根据是否需要插入、改变I-SMF判断，****此种方式不需要运营商严格规划NF Instance ID的node部分；同时，SMF服务覆盖范围粒度更小（小于NF Instance ID的可规划粒度）时适用**：
          - 针对已经存在PDU会话的UE，AMF2判断是否需要插入、改变I-SMF，如果需要，不影响用户正常业务的情况下（用户变为CM-IDLE态之后）且满足PDU会话重建条件时，通过释放SMF1承载的PDU会话，要求UE发起PDU会话重建的方式，AMF2为UE重新选择同区域Area2的SMF2，减少流量迂回以及业务时延。
          - 针对不存在PDU会话的UE，后续新建的PDU会话按照已有功能的AMF选择SMF策略直接选择Area2的SMF2。
- **SMF****侧本地PDU会话重建**
  当UE从Area1移动到Area2（例如，从A省移动到B省）后：
    - 针对PDU会话内只有缺省QosFlow的UE，SMF1能够比较I-SMF和SMF的NF Instance ID，识别I-SMF非本地，直接通过释放SMF1承载的PDU会话，要求UE发起PDU会话重建的方式，重新选择同区域Area2的SMF2，减少流量迂回以及业务时延。对于存在专有的QosFlow的UE，SMF不进行PDU会话释放，会等到专有QosFlow释放后（只存在缺省QosFlow），再进行PDU会话释放和要求重建。
    - 针对不存在PDU会话的UE，后续新建的PDU会话按照已有功能的AMF选择SMF策略直接选择Area2的SMF2。
