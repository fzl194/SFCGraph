#### [RFSP ID传递](#ZH-CN_CONCEPT_0168358238)

UNC 在移动管理过程对RFSP ID的传递，如 [表1](#ZH-CN_CONCEPT_0168358238__tbl_example) 所示。

*表1 移动管理流程中对RFSP ID的处理*

| 流程 | 携带RFSPID消息 | 说明 |
| --- | --- | --- |
| 签约数据传递业务流程（2G/3G） | IDR（Insert Subscriber Data Request）<br>ULA（Update Location Answer） | SGSN从HSS收到IDR/ULA消息，解析出RFSP ID，保存为用户上下文中的RFSP ID，若签约数据变更：<br>- 若UE为IDLE态，SGSN则不主动通知RAN，只有在用户移动管理过程中通过Attach，RAU、业务建立通过Service Request（3G）或者激活流程中的CREATE-BSS-PFC（2G）随下行信令更新到RAN侧。<br>- 若UE为CONNECT态，可通过将参数“CONNSUBSPID”配置为“YES”([**SET MMFUNC**](../../../../../../OM参考/命令/UNC MML命令/业务服务管理/Pre 5G接入业务管理/控制面管理/移动性管理/MM扩展功能管理/设置移动性管理扩展功能(SET MMFUNC)_26145512.md))使SGSN通过Common ID消息向RNC下发新RFSP ID。 |
| 签约数据传递业务流程（4G） | IDR（Insert Subscriber Data Request）<br>ULA（Update Location Answer） | MME从HSS收到IDR/ULA消息，解析出RFSP ID，保存为用户上下文的RFSP ID。若签约数据变更：<br>- 若UE为IDLE态，不会通知无线侧，当UE从IDLE态转变成CONNECT态时，MME下发RFSP ID给无线侧。<br>- 若UE为CONNECT态，UNC会重新计算使用的RFSP ID并和系统中实际使用的RFSP ID进行比较，如果不一致，则会通过UE Context Modification Request消息下发重新计算出的RFSP ID。 |
| Attach/RAU/PDP上下文激活流程（2G） | DL-UNITDATA（Attach Accept/RAU Accept）<br>CREATE-BSS-PFC(PDP上下文激活流程) | - 未启用Gb接口动态自动配置情况下:当[**ADD NSE**](../../../../../../OM参考/命令/UNC MML命令/业务服务管理/Pre 5G接入业务管理/控制面管理/Gb接口管理/信令实体管理/增加信令实体(ADD NSE)_26146028.md)中的“SPID”取值为“YES”，DL-UNITDATA/CREATE-BSS-PFC消息会携带SPID。<br>- 启用Gb接口动态自动配置情况下:当[**ADD GBNSECFGPARA**](../../../../../../OM参考/命令/UNC MML命令/业务服务管理/Pre 5G接入业务管理/控制面管理/Gb接口管理/Gb自动配置管理/NSE属性模板管理/增加NSE属性模板(ADD GBNSECFGPARA)_26146004.md)中的“SPID”取值为“YES”，DL-UNITDATA/CREATE-BSS-PFC消息会携带SPID。 |
| Attach/RAU/Detach/Service Request（3G） | COMMON ID<br>Direct Transfer（Attach Accept/RAU Accept） | 当<br>[**ADD RNC**](../../../../../../OM参考/命令/UNC MML命令/业务服务管理/Pre 5G接入业务管理/控制面管理/Iu接口管理/Iu接口RNC信息/增加Iu接口RNC信息(ADD RNC)_26146040.md)<br>中的“SPID”取值为“YES”时，COMMON ID/Direct Transfer消息会携带SPID。 |
| Attach/Service Request（4G） | Initial Context Setup Request | Attach/Service Request时，会通过这条消息携带RFSP ID。 |
| TAU | Initial Context Setup Request | 在TAU request消息中"Active flag"信元值为"Bearer establishment requested"的场景下。 |
| TAU | Downlink NAS Transport（TAU Accept） | 在TAU request消息中"Active flag"信元值为"No bearer establishment requested"的场景下，且<br>[**SET S1CMPT**](../../../../../../OM参考/命令/UNC MML命令/业务服务管理/Pre 5G接入业务管理/控制面管理/S1接口管理/S1接口兼容性/设置S1接口兼容性(SET S1CMPT)_72345837.md)<br>中的“SPID”取值为“YES”时。 |
| Inter<br>UNC<br>Handover后TAU | UE Context Modification Request | UNC<br>会重新计算使用的RFSP ID，并和旧侧比较是否一致。如果不一致，就会通过该消息将使用的RFSP ID传递给无线侧。 |
| Inter<br>UNC<br>Handover后TAU | Downlink NAS Transport（TAU Accept） | 当<br>[**SET S1CMPT**](../../../../../../OM参考/命令/UNC MML命令/业务服务管理/Pre 5G接入业务管理/控制面管理/S1接口管理/S1接口兼容性/设置S1接口兼容性(SET S1CMPT)_72345837.md)<br>中的“SPID”取值为“YES”时，Downlink NAS Transport消息会携带SPID。 |

#### [典型部署场景](#ZH-CN_CONCEPT_0168358238)

UNC 针对RFSP ID有以下三种典型应用场景：

- **场景一：**运营商规划完全信赖用户在HSS的签约信息中包含的RFSP ID。UNC不做任何额外的RFSP控制，直接使用用户签约信息中的RFSP ID。即：SGSN/MME从HSS收到IDR/ULA消息，解析出RFSP ID。
- **场景二：**运营商不信赖HSS中的用户签约信息（例如对于漫游用户），或在HSS签约未规划或HSS不支持该功能，为了实现自定义的驻留优先级控制，根据IMSI前缀和本网/外网属性等灵活配置用户的RFSP ID。
- **场景三：**运营商需要某号段的用户优先使用签约的RFSP ID对应的频点优先级进行驻留控制（例如：4G用户回落3G后，优先使用签约RFSP ID对应的频点优先级进行驻留控制），如果没有签约RFSP ID，按照配置对应的频点优先级进行驻留控制。
