# 基于N26接口的5GS到EPS切换

- [业务场景](#ZH-CN_CONCEPT_0281112295__1.3.1.1)
- [信令流程](#ZH-CN_CONCEPT_0281112295__1.3.2.1)

#### [业务场景](#ZH-CN_CONCEPT_0281112295)

基于N26接口的系统间切换，可用于为用户提供连续性无缝切换的业务，例如EPS Fallback语音业务。当AMF和MME之间存在N26接口时，可使用基于N26接口的5G到4G的互操作流程实现无缝切换，切换过程就是将数据从5G的传输通道（红线）切换到EPS传输通道（蓝线）的过程，如 [图1](#ZH-CN_CONCEPT_0281112295__zh-cn_concept_0276784454_fig12965433131513) 所示。

为支持5GS和EPS间基于N26的互操作，在PDU会话建立/GBR QoS Flow建立时，5G核心网做如下相关操作：

- QoS映射及TFT分配：如果部署了PCC，SMF+PGW-C将从PCF获取的5G QoS参数映射成EPS QoS映射，并根据获取的PCC规则分配TFT；否则，SMF+PGW-C在本地映射EPS QoS和分配TFT。SMF+PGW-C忽略不适合EPC的5G QoS参数（例如，QoS Notification control）。
- 分配EPS Bearer ID： SMF需要根据QoS Profile和运营商策略确定哪些QoS Flow需要分配EPS Bearer ID。当SMF判断需要为QoS流分配EPS Bearer ID（基于Dnninfo中的iwkEpsInd标志位，即UE在此DNN中是否签约了Interworking）时，由SMF请求服务AMF分配EPS Bearer ID。针对每个PDU会话，EPS承载ID分配给缺省EPS承载和专用EPS承载。
- UE、RAN、SMF+PGW-C都存储QoS flow和EBI、EPS QoS参数的联系。如果QoS flow删除了，UE、RAN、SMF+PGW-C也会删除与这个QoSflow关联的EPS QoS参数。

- 支持5GS和EPS间切换，UE的首次5G注册流程相比与纯5G的注册流程新增如下变更点：
    1. UE在[Registration Request](../../../../../../快速入门/协议与流程/5G Core信令分析/消息及信元/非服务化接口协议/N1接口/相关消息解释/Registration request_76783894.md)消息中[5GMM capability](../../../../../../快速入门/协议与流程/5G Core信令分析/消息及信元/非服务化接口协议/N1接口/相关信元解释/5GMM capability_76785392.md)信元的S1 mode比特置1，AMF根据此信息判断UE具备4G接入能力。
    2. AMF通过Nudm_UECM_Registration Request和Nudm_UECM_Registration Response消息与UDM交互，不携带双注册标识或携带的双注册标识为0，告知UDM保留AMF或者MME的单注册。单注册模式下，UE只有一个激活的MM状态，即5GC的RM状态或EPC的EMM状态。对于3GPP接入，网络只保留AMF上的或MME上的UE的一个有效的MM状态。
    3. AMF在[Registration Accept](../../../../../../快速入门/协议与流程/5G Core信令分析/消息及信元/非服务化接口协议/N1接口/相关消息解释/Registration accept_76785445.md)消息中携带[5GS network feature support IE](../../../../../../快速入门/协议与流程/5G Core信令分析/消息及信元/非服务化接口协议/N1接口/相关信元解释/5GS network feature support_76785319.md)，其中的“IWK N26”比特置为0或者不携带，标识网络侧中部署了N26接口。
    4. UE的订阅可能包括对核心网类型 (EPC) 的限制和对E-UTRAN的限制。如果有，则UDM+HSS向AMF提供这些限制。AMF将RAT和核心网类型限制包含在切换限制列表中给(R)AN，AMF和(R)AN使用这些限制条件来确定UE在EPS或E-UTRAN下的移动性。

#### [信令流程](#ZH-CN_CONCEPT_0281112295)

UE注册到5G网络后，切换到EPS的过程，就是将数据从5G的传输通道切换到EPS传输通道的过程。具体的切换流程如 [图2](#ZH-CN_CONCEPT_0281112295__zh-cn_concept_0276784454_fig775443716452) 所示。

- [图2](#ZH-CN_CONCEPT_0281112295__zh-cn_concept_0276784454_fig775443716452)以AMF与MME之间的N26接口独立部署为例，5G到4G的切换流程，AMF通过TAI构造域名查询对应MME设备地址是否为本端IP，如果DNS返回地址和MME地址一样，也支持走内部4/5G互操作流程。
- 如果是在MME的S10接口和AMF的N26接口IP地址合一的场景下，5G到4G的切换流程，AMF查询4G寻呼表判断目标基站是否在本设备服务范围内，如果在，则支持走UAM内部互操作流程；AMF也可以通过DNS查询MME地址是否为本端IP，如果DNS返回地址和MME地址一样，也支持走内部4/5G互操作流程。

2. AMF向SMF+PGW-C发送[Nsmf_PDUSession_RetrieveSmContext Request](../../../../../../快速入门/协议与流程/5G Core信令分析/消息及信元/服务化接口协议/Nsmf/N11/Nsmf_PDUSession_RetrieveSmContext/Nsmf_PDUSession_RetrieveSmContext Request_83854845.md)请求SM上下文。
  ```
  domain: request, role: client
  rid=228128355, scheme=http, stream id=11889
  POST /nsmf-pdusession/v1/sm-contexts/2188574823/retrieve HTTP/2.0
  user-agent: AMF
  content-type: application/json
  host: bbsmf001bhw.bb.aa.node.5gc.mnc000.mcc123.3gppnetwork.org
  content-length: 40
  {
  "targetMmeCap":{"nonIpSupported":true}
  }
  //AMF会告知SMF目标MME的能力（是否支持NON-IP）
  ```
  SMF+PGW-C感知分配了EBI的QoSFlow，对应QoSFlow切换到4G，SMF+PGW-C发送N4 Session modification给UPF+PGW-U，UPF+PGW-U为该EPS承载建立CN隧道，分配S5/S8资源，并且提供EPS承载上下文给AMF。
3. AMF给目标MME发送[Forward Relocation Request](../../../../../../快速入门/协议与流程/5G Core信令分析/消息及信元/非服务化接口协议/N26接口/相关消息解释/Forward Relocation Request_76784876.md)，AMF将PDN连接上下文、安全上下文、MM上下文通过N26发送给目标MME。消息中包含从步骤[1](#ZH-CN_CONCEPT_0281112295__zh-cn_concept_0276784454_li156628382009)中收到Direct Forwarding Path Availability信元，通知目标MME NG-RAN是否可以直接传输数据到目标E-UTRAN。
  如果N26接口不支持在Forward relocation request消息中携带终端能力srvcc-to-geran-utran-capability，会导致无法在Handover request消息中携带SRVCC Operation Possible，进而影响用户继续向23G网络回落，可以设置 **BYTE_EX_B330 BIT2** 为1，使DOWNLINK NAS TRANSPORT消息中携带SRVCC能力。

4. MME选择S-GW并向S-GW发送Create Session Request消息，通知S-GW建立承载。
5. S-GW分配本地资源，并向MME返回Create Session Response消息。
6. MME发送 [Handover Request](../../../../../../快速入门/协议与流程/5G Core信令分析/消息及信元/非服务化接口协议/N2接口/相关消息解释/HANDOVER REQUEST_76783992.md) 消息给E-UTRAN，请求建立无线网络资源，包含承载信息，安全上下文。
7. E-UTRAN向MME发送[Handover Request Acknowledge](../../../../../../快速入门/协议与流程/5G Core信令分析/消息及信元/非服务化接口协议/N2接口/相关消息解释/HANDOVER REQUEST ACKNOWLEDGE_76785121.md)消息。此时E-UTRAN做好在已经建立的E-RAB上接收GTP分组数据单元的准备。
8. 如果采用Indirect Forwarding的数据转发方式(即NG-RAN缓存的下行数据要通过核心网网元转发给E-UTRAN)，MME向S-GW发送Create Indirect Data Forwarding Tunnel Request消息，包含目标E-UTRAN的转发通道的地址和TEID。S-GW向MME发送Create Indirect Data Forwarding Tunnel Response响应消息。
9.
  MME向AMF发送 [Forward Relocation Response](../../../../../../快速入门/协议与流程/5G Core信令分析/消息及信元/非服务化接口协议/N26接口/相关消息解释/Forward Relocation Response_76784284.md) 消息。对于间接转发方式，消息中还包含S-GW的间接转发通道的地址和TEID。

  *表3 Forward Relocation Response消息的关键信元*

  | 关键信元 | 信元解释 |
  | --- | --- |
  | ![](基于N26接口的5GS到EPS切换_81112295.assets/zh-cn_image_0276785464_2.png) | cause正常为Request accepted，异常为Relocation failure。 |
  | ![](基于N26接口的5GS到EPS切换_81112295.assets/zh-cn_image_0276783886_2.png "点击放大") | bearer-context里面包含可切换至4G的承载，消息中可包含多个bearer-context。<br>bearer-context中List of Set-up Bearers信元中包含eps-bearer-id、sgw-dl-data-forward-teid（间接转发隧道信息）。 |
  | ![](基于N26接口的5GS到EPS切换_81112295.assets/zh-cn_image_0276785203_2.png) | E-UTRAN Transparent Container携带的是目标无线给源无线的信息，即E-UTRAN给NG-RAN的信息。 |
10. 如果使用Indirect Forwarding数据转发的方式，则AMF向SMF+PGW-C发送SGW间接转发通道的地址和TEID，用于建立间接转发隧道，SMF+PGW-C可能选择一个中间的UPF+PGW-U进行数据转发。
    a. AMF发送[Nsmf_PDUSession_CreateSMContext Request](../../../../../../快速入门/协议与流程/5G Core信令分析/消息及信元/服务化接口协议/Nsmf/N11/Nsmf_PDUSession_CreateSMContext/Nsmf_PDUSession_CreateSMContext Request_34935460.md)给SMF+PGW-C，将S-GW的地址和TEID发送给SMF+PGW-C。
    b. 基于QoSFlow ID与EPS bearer ID之间的映射关系，SMF+PGW-C需要根据QoS Profile和运营商策略确定哪些QoS Flow需要分配EPSBearer ID。SMF+PGW-C将QFI、S-GW地址和TEID发送给UPF+PGW-U。SMF+PGW-C或者UPF+PGW-U分配数据转发的CN隧道信息。
    c. SMF+PGW-C返回[Nsmf_PDUSession_UpdateSmContext Response](../../../../../../快速入门/协议与流程/5G Core信令分析/消息及信元/服务化接口协议/Nsmf/N11/Nsmf_PDUSession_UpdateSmContext/Nsmf_PDUSession_UpdateSmContext Response_34935436.md)，携带数据转发的CN隧道信息、QoS flow。
11. AMF发送[Handover Command](#ZH-CN_CONCEPT_0281112295__zh-cn_concept_0276784454_table9311193481814)给NG-RAN，通知切换准备完成。NG-RAN把[Handover Command](#ZH-CN_CONCEPT_0281112295__zh-cn_concept_0276784454_table9311193481814)消息发送到UE，通知UE切换到目的接入网络。此时下行数据可以通过UPF+PGW-U和SGW，从NG-RAN发送到目的E-UTRAN。

  | 关键信元 | 信元解释 |
  | --- | --- |
  | ![](基于N26接口的5GS到EPS切换_81112295.assets/zh-cn_image_0276784184_2.png) | 切换类型为five egs-to-eps，即5G至4G的切换。 |
  | ![](基于N26接口的5GS到EPS切换_81112295.assets/zh-cn_image_0276785173_2.png) | PDU Session Resource Handover List，包括<br>- PDU session ID<br>- Handover Command Transfer，该信元中携带DL Forwarding UP TNL Information（如果是间接转发场景），QoS Flow to be Forwarded List（可以进行切换的QoS Flow） |
  | ![](基于N26接口的5GS到EPS切换_81112295.assets/zh-cn_image_0276785070_2.png) | Target to Source Transparent Container携带的是目标无线给源无线的信息，即E-UTRAN给NG-RAN的信息。 |
12. UE成功切换到目标小区后，上下行数据通过E-UTRAN进行发送。
    a. UE成功同步到目标小区后，发送Handover confirm消息给E-UTRAN，此时E-UTRAN将缓存的下行数据下发到UE，上行数据则经由UE->E-UTRAN->S-GW->UPF+PGW-U。
    b. E-UTRAN发送 [Handover Notify](../../../../../../快速入门/协议与流程/5G Core信令分析/消息及信元/非服务化接口协议/N2接口/相关消息解释/HANDOVER NOTIFY_76783953.md) 消息到MME，通知目标MME UE已经位于目标小区。
    c. MME发送 [Forward Relocation Complete Notification](../../../../../../快速入门/协议与流程/5G Core信令分析/消息及信元/非服务化接口协议/N26接口/相关消息解释/Forward Relocation Complete Notification_76785406.md) 消息到AMF。
      ![](基于N26接口的5GS到EPS切换_81112295.assets/zh-cn_image_0276784554_2.png)
    d. AMF发送 [Forward Relocation Complete Acknowledge](../../../../../../快速入门/协议与流程/5G Core信令分析/消息及信元/非服务化接口协议/N26接口/相关消息解释/Forward Relocation Complete Acknowledge_76785047.md) 消息响应MME。此时AMF会启用一个定时器，用于超时后释放源侧资源。
      ![](基于N26接口的5GS到EPS切换_81112295.assets/zh-cn_image_0276783954_2.png)
13. MME发送Modify Bearer Request消息到S-GW，告知E-UTRAN的用户面地址和TEID。
14. S-GW发送Modify Bearer Request到SMF+PGW-C，S-GW分配与UPF+PGW-U间下行数据传输隧道的地址和TEID，并将其与E-UTRAN的用户面地址和TEID一起告知SMF+PGW-C。
15. SMF+PGW-C向UPF+PGW-U发起N4 Session Modification流程，更新用户面路径。
16. SMF+PGW-C回复Modify Bearer Response给S-GW。此时下行数据包就可以通过新建立的下行数据通道通过S-GW转发给目标E-UTRAN。这一步UE到UPF+PGW-U间端到端的缺省承载和专有承载的用户面数据传输通道都建立完成。
17. S-GW发送Modify Bearer Response消息响应MME。
18. UE发起TAU流程。
19. 在切换完成后，SMF+PGW-C可决定将映射到缺省承载中的部分数据流映射为专有承载，SMF+PGW-C可发起专用承载建立。
20. 步骤20和21，如果使用了间接转发，源侧和目标侧的“删除间接转发隧道定时器”超时后，AMF和MME发起间接转发隧道删除。
