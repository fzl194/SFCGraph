#### [业务场景](#ZH-CN_TOPIC_0000001419673516)

在注册流程中，(R)AN侧优先根据5G GUAMI查找目标AMF，其次根据Requested NSSAI查找目标AMF，如果(R)AN无法根据UE在AN消息中携带的5G GUAMI或Requested NSSAI查找到目标AMF，会选择一个缺省的AMF（也称为初始AMF）进行注册流程。因此当初始AMF接收到注册请求时，初始AMF可能需要将注册请求重路由到另一个AMF（因为初始AMF不是为UE服务的合适的AMF），因此要执行AMF重分配流程的注册流程，将UE的NAS消息重路由到目标AMF，由目标AMF继续为UE提供注册服务。

#### [信令流程](#ZH-CN_TOPIC_0000001419673516)

![](AMF重分配注册_19673516.assets/zh-cn_image_0000001447594017_2.png)

1. 消息交互：(R)AN发送[Registration request](../../../../../../../../快速入门/协议与流程/5G Core信令分析/消息及信元/非服务化接口协议/N1接口/相关消息解释/Registration request_76783894.md)给初始AMF。
  触发条件：如果AN消息中未携带5G-S-TMSI，或者5G-S-TMSI不能指示一个合法的AMF时，(R)AN根据RAT（radio access technology）和请求的网络切片标识（Requested NSSAI）选择AMF。如果(R)AN不能选择合适的AMF，则将注册请求转发给(R)AN中已配置的default AMF即初始AMF进行注册。
  关键信元：
  | 5GS registration type | 指示5GS注册类型，初始注册时5GS registration type为initial registration。 |
  | --- | --- |
  | 5GS mobile identity | 携带用户标识5G-GUTI或SUCI。优先选择携带5G-GUTI作为UE身份标识，若无法获取5G-GUTI，UE会携带自身的SUCI作为UE身份标识。<br>如果用户是首次在网络注册，比如首次开机注册则携带SUCI，如果是已经在网络注册成功过，比如关机后再开机或者飞行模式后再关闭飞行模式发起初始注册，此时用户携带之前注册成功时AMF分配的5G-GUTI。<br><5G-GUTI> := <GUAMI（Globally Unique AMF Identifier）> <5G-TMSI><br><GUAMI> := <MCC> <MNC> <AMF Region ID> <AMF Set ID> <AMF Pointer><br>5G-TMSI：（Temporary Mobile Subscriber Identity）临时用户标识，网络侧AMF分配的临时用户标识。<br>SUPI：（Subscription Permanent Identifier）用户永久标识，5G网络中标识一个用户。<br>SUCI：（Subscription Concealed Identifier）用户隐藏标识，UE通过归属网络的公共密钥加密SUPI生成SUCI。 |
  | Requested NSSAI | 标识请求的网络切片。 |
  | 5G-S-TMSI | 5G-S-TMSI是5G-GUTI的部分字段，使用5G-S-TMSI可以缩短空口消息长度，提升空口效率。<br><5G-S-TMSI> := <AMF Set ID> <AMF Pointer><5G-TMSI> |
2. 引用流程：初始AMF执行[初始注册](初始注册_19356600.md#ZH-CN_TOPIC_0000001419356600)的步骤4~步骤9。
  触发条件：初始AMF从老侧AMF获取用户上下文以及鉴权加密。
3. 内部流程：UDM选择流程。
  触发条件：如果未能从老侧获取到UE的切片签约数据，则初始AMF会从UDM获取用户切片签约数据。
  相关机制：AMF选择UDM的机制参见UNC章节：注册管理流程的网元选择。
4. 消息交互：初始AMF发送[Nudm_SDM_GetUESubscribedNSSAI Request](../../../../../../../../快速入门/协议与流程/5G Core信令分析/消息及信元/服务化接口协议/Nudm/N8/Nudm_SDM_Get/Nudm_SDM_GetUESubscribedNSSAI Request_06927266.md)给UDM。
  触发条件：如果未能从老侧获取到UE的切片签约数据，则初始AMF会从UDM获取用户切片签约数据。
  关键信元：
  | ueId | 此处携带用户的SUPI。 |
  | --- | --- |
  | nssai | 请求获取用户的切片签约数据，包括S-NSSAI列表以及默认S-NSSAI列表。 |
5. 消息交互：UDM发送[Nudm_SDM_GetUESubscribedNSSAI Response](../../../../../../../../快速入门/协议与流程/5G Core信令分析/消息及信元/服务化接口协议/Nudm/N8/Nudm_SDM_Get/Nudm_SDM_GetUESubscribedNSSAI Response_56206893.md)给初始AMF。
  关键信元：
  | defaultSingleNssais | 表示默认S-NSSAI列表。 |
  | --- | --- |
  | singleNssais | 表示S-NSSAI列表。 |
  相关机制：如果请求成功，UDM发送“200 OK”响应消息，如果请求失败，UDM没有UE的有效签约数据，则返回HTTP状态码“404 Not Found”。
  相关配置：通过UDM命令：ADD SNSSAI配置用户签约的切片信息。
6. 内部流程：NSSF选择流程。
  触发条件：AMF通过本地配置判断是否由AMF决策切片，如果否，则AMF选择NSSF进行切片决策。如果是，AMF进一步判断是否支持UE请求的Requested NSSAI和签约S-NSSAI的交集切片或所有被标记为default的签约S-NSSAI，如果不支持则AMF向NSSF发起切片选择。
  相关机制：AMF通过配置判断是否通过NRF选择NSSF，或通过本地配置选择NSSF。
  相关配置：

  - 通过UNC命令：SET NSSELPARA配置是否由AMF决策切片。
    - 通过UNC命令：SET NFDISCPLCY配置选择NSSF的优先方式。
    - 通过UNC命令：ADD PNFPROFILE配置NSSF的对接信息，用于本地配置查询NSSF。
  相关特性：切片选择参见UNC特性：WSFD-110001 支持基于NSSAI选择切片。
7. 消息交互：初始AMF发送[Nnssf_NSSelection_Get Request](../../../../../../../../快速入门/协议与流程/5G Core信令分析/消息及信元/服务化接口协议/Nnssf/N22/Nnssf_NSSelection_Get/Nnssf_NSSelection_Get Request_84640229.md)给NSSF。
  触发条件：AMF通过本地配置判断是否由AMF决策切片，如果否，则AMF选择NSSF进行切片决策。如果是，AMF进一步判断是否支持UE请求的Requested NSSAI和签约S-NSSAI的交集切片或所有被标记为default的签约S-NSSAI，如果不支持则AMF向NSSF发起切片选择。
  关键信元：
  | nf-type | 表示NF服务消费者的NF类型，比如AMF。 |
  | --- | --- |
  | nf-id | 表示NF服务消费者的NF id。 |
  | slice-info-request-for-registration | 表示在注册过程中向NSSF请求网络切片信息。 |
  | subscribedNssai | 该IE应包含签约的S-NSSAI列表和每个S-NSSAI的指示（如果是默认S-NSSAI）。 |
  | allowedNssaiCurrentAccess | 当前接入类型允许的S-NSSAI列表。 |
  | allowedNssaiOtherAccess | 其他接入类型允许的S-NSSAI列表。 |
  | defaultConfiguredSnssaiInd | 默认配置的NSSAI是否由UE指示。 |
8. 消息交互：NSSF发送[Nnssf_NSSelection_Get Response](../../../../../../../../快速入门/协议与流程/5G Core信令分析/消息及信元/服务化接口协议/Nnssf/N22/Nnssf_NSSelection_Get/Nnssf_NSSelection_Get Response_36360658.md)给初始AMF。
  关键信元：
  | allowedNssaiList | NSSF在服务PLMN中授权的允许的S-NSSAI。 |
  | --- | --- |
  | configuredNssai | NSSF在服务PLMN中授权的配置的S-NSSAI。 |
  | targetAmfSet | 目标AMF集合。 |
  | candidateAmfList | 候选AMF(s)的列表。 |
  相关机制：如果请求成功，NSSF发送“200 OK”响应消息，如果请求失败，找不到用于请求的切片选择信息的切片实例，NSSF应返回“403 Forbidden”的HTTP状态码。
  相关配置：

  - NSSF通过UNC命令：ADD AMFSETID配置AMF集合与切片的关系。
    - NSSF通过UNC命令：ADD CANDIDATEAMF配置AMF实例与AMF集合的关系。
9. 消息交互：初始AMF发送[Namf_Communication_RegistrationStatusUpdate Request](../../../../../../../../快速入门/协议与流程/5G Core信令分析/消息及信元/服务化接口协议/Namf/N14/Namf_Communication_Registration_4cc19f0e/Namf_Communication_RegistrationStatusUpdate Request_84640657.md)给老侧AMF。
  触发条件：初始AMF发送一个拒绝指示到老侧AMF，告知当前UE的注册流程未完成。老侧AMF收到该指示后会继续保存UE上下文以便目标AMF获取。
  关键信元：
  | transferStatus | 携带NOT_TRANSFERRED，指示UE的注册请求被重定向到另一个AMF。 |
  | --- | --- |
10. 消息交互：老侧AMF发送[Namf_Communication_RegistrationStatusUpdate Response](../../../../../../../../快速入门/协议与流程/5G Core信令分析/消息及信元/服务化接口协议/Namf/N14/Namf_Communication_Registration_4cc19f0e/Namf_Communication_RegistrationStatusUpdate Response_84800889.md)给初始AMF。
  关键信元：
  | regStatusTransferComplete | 携带false，指示未成功完成上下文传输。 |
  | --- | --- |
  相关机制：如果请求被接受，源AMF应响应“200 OK”HTTP状态码，如果失败或重定向时，源AMF应返回“403 Forbidden/404 Not Found”等HTTP状态码。
11. 内部流程：NRF选择流程。
  触发条件：AMF通过本地配置判断如何选择目标AMF，如果通过NRF方式，则先进行NRF选择。如果是本地配置方式则查询本地配置选择目标AMF。
  相关机制：AMF选择目标AMF的机制参见UNC章节：注册管理流程的网元选择。
  相关配置：

  - AMF通过UNC命令：SET NFDISCPLCY配置选择AMF的优先方式。
    - AMF通过UNC命令：ADD PNFAMFINFO、ADD PNFPROFILE配置AMF的对接信息，用于本地配置查询AMF。
    - AMF通过UNC命令：ADD NRF、ADD NRFADDR配置NRF的对接信息，用于本地配置查询NRF。
12. 消息交互：初始AMF发送[Nnrf_NFDiscovery](../../../../../../../../快速入门/协议与流程/5G Core信令分析/消息及信元/服务化接口协议/Nnrf/Nnrf接口服务操作/Nnrf_NFDiscovery_76668377.md)给NRF。
  触发条件：AMF通过本地配置判断如何选择目标AMF，如果通过NRF方式，则发送消息给NRF查询目标AMF。
  关键信元：
  | target-nf-type | 目标NF类型。 |
  | --- | --- |
  | requester-nf-type | 请求方NF类型。 |
  | amf-set-id | AMF集标识。 |
  | target-nf-instance-id | 目标NF实例ID。 |
13. 消息交互：NRF发送[Nnrf_NFDiscovery](../../../../../../../../快速入门/协议与流程/5G Core信令分析/消息及信元/服务化接口协议/Nnrf/Nnrf接口服务操作/Nnrf_NFDiscovery_76668377.md)给初始AMF。
  关键信元：
  | nfInstances | NF实例的相关信息包括IP地址等。 |
  | --- | --- |
  相关机制：如果发现请求成功，NRF返回200（OK），如果发现请求失败，NRF应返回“400 Bad Request/403 Forbidden/413 Payload Too Large/429 Too Many Requests/500 Internal Server Error/503 Service Unavailable”的HTTP状态码。
14. 消息交互：初始AMF发送[Namf_Communication_N1MessageNotify Request](../../../../../../../../快速入门/协议与流程/5G Core信令分析/消息及信元/服务化接口协议/Namf/N14/Namf_Communication_N1MessageNotify/Namf_Communication_N1MessageNotify Request_36201626.md)给目标AMF。
  触发条件：AMF通过本地配置判断重分配是采用重定向方式或重路由方式。如果配置为重定向方式则执行本消息交互。
  关键信元：
  | n1MessageContainer | 指示N1消息内容。 |
  | --- | --- |
  | registrationCtxtContainer | 指示注册上下文信息。 |
  相关配置：AMF通过UNC命令：SET NGMMFUNC配置AMF重分配方式。
15. 消息交互：目标AMF发送[Namf_Communication_N1MessageNotify Response](../../../../../../../../快速入门/协议与流程/5G Core信令分析/消息及信元/服务化接口协议/Namf/N14/Namf_Communication_N1MessageNotify/Namf_Communication_N1MessageNotify Response_36361118.md)给初始AMF。
  触发条件：AMF通过本地配置判断重分配是采用重定向方式或重路由方式。如果配置为重定向方式则执行本消息交互。
  相关机制：如果通知成功，目标AMF应返回响应状态码“204 No Content”，如果失败，目标AMF应返回“403 Forbidden”等HTTP状态码。
16. 消息交互：初始AMF发送[REROUTE NAS REQUEST](../../../../../../../../快速入门/协议与流程/5G Core信令分析/消息及信元/非服务化接口协议/N2接口/相关消息解释/REROUTE NAS REQUEST_76783852.md)给(R)AN。
  触发条件：AMF通过本地配置判断重分配是采用重定向方式或重路由方式。如果配置为重路由方式则执行本消息交互。
  关键信元：
  | NGAP Message | 包含N1消息类和N1消息内容。 |
  | --- | --- |
  | AMF Set ID | 该IE用于在AMF区域内唯一标识一个AMF集合。 |
  | Allowed NSSAI | 该IE包含允许的NSSAI。 |
  相关配置：AMF通过UNC命令：SET NGMMFUNC配置AMF重分配方式。
17. 消息交互：(R)AN发送[Registration request](../../../../../../../../快速入门/协议与流程/5G Core信令分析/消息及信元/非服务化接口协议/N1接口/相关消息解释/Registration request_76783894.md)给目标AMF。
  触发条件：AMF通过本地配置判断重分配是采用重定向方式或重路由方式。如果配置为重路由方式则执行本消息交互。
18. 引用流程：目标AMF执行[初始注册](初始注册_19356600.md#ZH-CN_TOPIC_0000001419356600)的步骤4~步骤34。
19. 消息交互：目标AMF发送[Registration accept](../../../../../../../../快速入门/协议与流程/5G Core信令分析/消息及信元/非服务化接口协议/N1接口/相关消息解释/Registration accept_76785445.md)给(R)AN。
