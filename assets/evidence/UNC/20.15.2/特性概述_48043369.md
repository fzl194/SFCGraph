#### [适用NF](#ZH-CN_CONCEPT_0248043369)

MME、SGW-C、PGW-C

#### [定义](#ZH-CN_CONCEPT_0248043369)

UNC 支持IPv6 EPS承载的激活、去激活和更新。

#### [客户价值](#ZH-CN_CONCEPT_0248043369)

| 受益方 | 受益描述 |
| --- | --- |
| 运营商 | 增加可用的IP地址空间，使网络同时附着更多的用户数，并以此开展更丰富的业务。 |
| 用户 | 可以通过接入IPv6网络，使用更丰富的业务。 |

#### [应用场景](#ZH-CN_CONCEPT_0248043369)

当用户终端需要访问IPv6业务时，可开通本特性。

#### [可获得性](#ZH-CN_CONCEPT_0248043369)

**涉及NF**

| 涉及NF | 支持版本 | 功能说明 |
| --- | --- | --- |
| UE | 无特殊要求 | 将自身IP能力指示给网络侧。UE必须支持IPv6。 |
| MME | UNC 20.0.0及后续版本 | 为UE创建IPv6类型PDN连接。 |
| SGW-C/PGW-C | UNC 20.0.0及后续版本 | 为UE创建IPv6类型PDN连接。支持为用户分配IPv6类型的IP地址。 |
| SGW-U/PGW-U | UNC 20.0.0及后续版本 | 传递上行、下行用户数据流。 |

#### [原理概述](#ZH-CN_CONCEPT_0248043369)

随着Internet应用及规模的飞速发展，IPv4协议的可用地址空间越来越少，已经很难满足日益增长的需求，为此，IPv6协议被提出来解决IPv4协议遇到的问题。针对IPv4的不足，IPv6标准采取了一系列措施，增加了许多新功能，包括足够大的地址空间、更高的安全性、对移动性和QoS更好的支持等。IPv6为IP网的可持续发展奠定了良好的基础。

IPv6中的变化体现在以下五个重要方面：

- 扩展地址：地址由IPv4的32bit扩展为128bit，意味着IP可以继续增长而无需考虑资源的匮乏，该地址结构对于提高路由效率也有所帮助。
- 简化头格式：包头的简化减少了路由器上所需的处理过程，从而提高了选择路由的效率。
- 增强对于扩展和选项的支持：意味着可以在几乎不影响普通数据包和特殊包选路的前提下适应更多的特殊需求。
- 流标记：流标记方法为更加高效地处理包流提供了一种机制，这种方法对于实时应用尤其有用。
- 身份验证和保密：身份验证和保密方面的改进使得IPv6更加适用于那些要求对敏感信息和资源特别对待的商业应用。

**系统实现**

UNC 能够为用户创建IPv6类型的PDP上下文/EPS bearer，为用户分配IPv6类型的IP地址，IPv6类型的数据包跟IPv4类型的数据包一样，在S5/S11/S1-U接口上透明传输。

**业务流程**

**图1** IPv6的EPS bearer激活流程

<br>

![](特性概述_48043369.assets/zh-cn_image_0248043349_2.png)

1. UE发起PDN连接请求消息给eNodeB，消息中携带PDN Type为IPv6，PDN Address不为空。
2. eNodeB根据该UE已有的连接选择MME，消息中携带PDN Type为IPv6，PDN Address不为空。
3. MME根据APN选择一个PGW-C，建立默认承载，并且为默认承载分配承载ID，向SGW-C发送Create Session Request（控制面TEID，PDN GW address， APN，默认承载QoS，计费参数，Dual Address Bearer Flag等）消息。
4. SGW-C在承载上下文列表中加入新记录，然后SGW-C向SGW-U发送PFCP Session Establishment Request消息。
5. SGW-U向SGW-C发送PFCP Session Establishment Response消息。SGW-U将缓存所有从PGW-U而来的下行数据包。
6. SGW-C向PGW-C发送Create Session Request消息。如果从MME获得了MSISDN，则消息中也会携带该信息。
7. 如果启用动态的PCC（Policy and Charging Control），则PGW-C发起IP-CAN建立流程；如果没有部署动态PCC，采用网关本地配置策略。
8. PGW-C在承载上下文列表中加入新的记录，并且产生一个Charging ID。同时PGW-C分配用户的PDN Address，然后PGW-C向PGW-U发送PFCP Session Establishment Request消息。PGW-U可以向SGW-U发送用户面分组数据单元，并开始计费。
9. PGW-U向PGW-C发送PFCP Session Establishment Response消息。
10. PGW-C向SGW-C发送Create Session Response消息。
11. SGW-C向SGW-U发送PFCP Session Modification Request消息。
12. SGW-U向SGW-C发送PFCP Session Modification Response消息。
13. SGW-C返回给MME一个Create Session Response消息。
14. MME向eNodeB发送E-Rab Setup Request消息。该消息还包含一条NAS消息即Activate default EPS bearer context request消息。
15. eNodeB给UE发送RRC Connection Reconfiguration消息，这个消息包括无线承载的ID，同时Activate default EPS bearer context request消息也将会转发给UE。
16. UE向eNodeB发送RRC Connection Reconfiguration Complete消息。
17. eNodeB向MME发送E-Rab Setup Response消息。这个控制信息还包括eNodeB的TEID和地址（供传输下行数据使用）。
18. UE向eNodeB发送Activate default EPS bearer context accept消息。
19. eNodeB向MME转发Activate default EPS bearer context accept消息。
20. SGW-C收到MME的Modify Bearer Request （EPS Bearer Identity）消息，建立S1-U缺省承载，并建立S1-U和S5/S8-U缺省承载的映射关系。
21. SGW-C向SGW-U发送PFCP Session Modification Request消息。
22. SGW-U向SGW-C发送PFCP Session Modification Response消息，表明它已经收到eNodeB的TEID和地址，这样SGW-U就能发送缓存的下行数据包了。
23. SGW-C向MME发送Modify Bearer Response （EPS Bearer Identity）。
