UNC 支持对通过GPRS/UMTS网络接入、LTE网络接入和WLAN网络接入的用户进行PCC策略控制，并支持PCC用户在EPC网络和GPRS/UMTS网络之间进行切换。

对于GnGp-SGSN接入情况下， UNC 需要按照QoS映射机制将Pre-R8 QoS映射为EPC QoS，并且使用PCC标准定义的EPC Gx接口的触发器和AVP与PCRF进行交互，UE请求的信息封装在PCC标准定义的信元中上报给PCRF。

#### [系统实现](#ZH-CN_TOPIC_0229056158)

策略是运营商根据自身运营需要，有目的有选择地对用户、网络、业务施加一些控制手段。例如，在忙时限制用户对P2P业务的使用，以减轻或消除分组网络资源的过载。策略控制和计费控制一起，构成了PCC的两大基本功能。

用户激活/更新过程中， UNC 根据用户信息从PCRF或本地配置获得该用户的策略和计费控制相关信息，并传递给GGSN-U/PGW-U。用户使用数据业务时，GGSN-U/PGW-U使用SA功能对用户报文进行解析，将解析结果与策略中的条件进行匹配，获得并执行对应的计费控制、QoS控制。

PCC策略来源有如下两种：

- PCRF下发
  在PCC标准架构中，策略由PCRF定义，由PCEF执行。用户激活过程中 UNC 与PCRF交互，通过Gx接口获取策略和计费控制相关的授权信息，并将其处理后提供给GGSN-U/PGW-U中的各个业务模块使用，以实现策略和计费控制。PCRF可以下发动态规则、预定义规则或预定义规则组。
- 本地策略（Local Policy）
  网络中未部署PCRF，则GGSN-U/PGW-U使用 UNC 本地配置和传递的规则进行策略匹配和执行。

PCRF下发的策略的优先级高于本地策略的优先级，即 UNC 优先选择PCRF下发的策略。

动态规则、预定义规则和预定义规则组的定义如 [表1](#ZH-CN_TOPIC_0229056158__table15811192511515) 所示。

#### [业务流程](#ZH-CN_TOPIC_0229056158)

PCC业务流程包括如下几种情况：

- IP-CAN会话建立
  在IP-CAN会话建立过程中， UNC 发送CCR消息给PCRF，消息中携带CC-Request-Type AVP（值为INITIAL_REQUEST）、Subscription-Id AVP（标识用户信息）及其他属性信息 ，用于PCRF策略决策。
  PCRF收到CCR消息后，发送CCA消息响应请求。 UNC 收到响应后，与GGSN-U/PGW-U进行交互，传递PCRF下发的策略并且建立会话。
  PCRF同时下发缺省承载的rule和专有承载的rule时，PGW-C/GGSN创建缺省承载后，支持创建专有承载。PGW-C/GGSN成功创建该类专有承载，可提升专有承载创建成功率。
- IP-CAN会话更新
  会话更新过程可以由UE、SGSN、MME、GGSN-C/PGW-C、GGSN-U/PGW-U或PCRF发起。由 UNC 发起策略更新请求（CCR-U），或者由PCRF下发RAR消息来更新业务所使用的策略，更新的策略通过N4接口由 UNC 传递给GGSN-U/PGW-U，并更新会话。
  触发器定义了引起 UNC 发起策略更新请求的事件，决定PCRF策略的决策时机，可以由PCRF在CCA或RAR消息中下发给 UNC 。当触发器定义的事件发生时， UNC 在CCR消息中包含该触发器指示策略更新请求的原因，PCRF根据事件消息类型判定将来的动作或下发对应的策略。
  各触发器的定义请参见 [Event Trigger](#ZH-CN_TOPIC_0229056158__section385583279) 。
- IP-CAN会话终结
  会话终结过程可以由UE、网络侧发起。 UNC 与PCRF间通过CCR-T/CCA-T结束该IP-CAN会话，并通过N4接口通知到GGSN-U/PGW-U，最终释放相应资源。
- 专有承载激活
  UNC 支持UE和网络侧发起的专有承载激活，如果PCRF发送的CCA/RAR消息中携带的规则中定义的QCI/ARP与已有承载的QCI/ARP不同，则触发建立专有承载。如果动态规则中没有包含QoS信息，则不会建立专有承载，该规则将安装在缺省承载上。 UNC 上创建的专有承载的规则会通过N4接口传递到GGSN-U/PGW-U上供业务匹配使用。
- 专有承载更新
  当PCRF发送RAR或者CCA-U消息给 UNC ，消息中携带更新后的动态PCC规则，则发起专有承载的更新流程。 UNC 上更新的专有承载的规则会通过N4接口传递到GGSN-U/PGW-U上供业务匹配使用。
  专有承载的更新主要有如下几种：

  - UE/SGSN发起的QoS更新
      UNC 收到SGSN发送的Update PDP Context Request消息，消息中包含UE请求的TFT和QoS信息，如果该消息中包含与原专有承载中不同的QoS信息，则发起QoS更新流程， UNC 发送CCR-U消息给PCRF请求更新。
    - UE/MME发起的QoS更新
      UNC 收到MME发送的Bearer Resource Command消息，消息中包含UE请求的TFT和QoS信息，如果该消息中包含与原专有承载中不同的QoS信息，则发起QoS更新流程， UNC 发送CCR-U消息给PCRF请求更新。
    - HSS发起的QoS更新
      HSS上的签约QoS发生变化时，会发送Insert Subscriber Data消息给MME，MME再用Modify Bearer Command消息将变化的QoS信息传递到 UNC 上， UNC 将变化了的QoS信息（主要包括APN-AMBR和缺省承载的QCI与ARP），在Gx接口上通过QoS-Information AVP和Default-EPS-Bearer-QoS AVP通知给PCRF。
    - 网络侧发起的TFT修改
      当PCRF通过RAR消息发送的PCC规则包含更新后的流信息（filter），且该PCC规则已安装在原专有承载中，则会发起TFT修改流程。原专有承载中的TFT filter将根据更新后的filter进行修改。
    - 增加动态规则到已有承载
      当 UNC 收到新的动态PCC规则，且该规则中包含的QCI和ARP与已激活的专有承载中的相同，则发起专有承载修改流程。新收到的QoS中的带宽值将会累加在对应的已激活专有承载的带宽中，此时基于新的QoS发起专有承载的更新，并安装更新后的TFT filter到专有承载中。
    - 删除已有承载中的动态规则
      当动态PCC规则需要从已激活的专有承载中删除时，会发起专有承载的修改流程。专有承载的带宽将减去需要删除的PCC规则的带宽值，并删除需要删除的PCC规则中对应的TFT filter。
- 专有承载终结
  专有承载的终结流程可以由SGSN、MME、 UNC 、GGSN-U/PGW-U或PCRF发起。 UNC 与PCRF之间通过CCR-U/CCA-U或RAR/RAA消息去活某专有承载，并删除该专有承载上的所有PCC规则。 UNC 上终结的专有承载信息会通过N4接口传递到GGSN-U/PGW-U，以删除GGSN-U/PGW-U上所使用的PCC规则。

#### [组网可靠性-PCRF主备/负荷分担](#ZH-CN_TOPIC_0229056158)

UNC 可以基于整机、DNN（APN）或IMSI/MSISDN号段来选择PCRF组，PCRF组内的PCRF之间有如下三种分担方式：

- 主备模式
  PCRF组内的PCRF之间为主备模式，其中一个PCRF为主，其他PCRF为备用。当主用PCRF故障时， UNC 选择其他某个可用的PCRF来处理Gx接口交互信息。
- 基于轮选方式进行负荷分担
  PCRF组内的PCRF之间为轮选方式的负荷分担模式，此时无需配置各PCRF的负荷分担百分比，各PCRF间默认等比例方式负荷分担Gx接口交互信息。当某个PCRF故障时， UNC 在选择PCRF时会跳过该PCRF，选择下一个可用的PCRF， UNC 与PCRF之间的交互信息在其他几个可用的PCRF之间等比例负荷分担。
- 基于配置的百分比进行负荷分担
  PCRF组内的PCRF之间为按配置比例方式的负荷分担模式，即PCRF间按照配置比例负荷分担Gx接口交互信息。当某个PCRF故障时， UNC 在选择PCRF时会跳过该PCRF，选择下一个可用的PCRF。
  例如，有3台PCRF，分别为A、B、C，负荷分担百分比设置为30%:30%:40%。当有10个PCC用户依次激活时，PCRF的分配是ABC、ABC、ABC、C，三台PCRF承担的用户数分别是3:3:4。一个PcrfGroup中所有PCRF负荷的百分比例总和不超过100%，否则提示配置错误，但是允许比例总和低于100%。
  如果PcrfGroup中只有部分PCRF配置了百分比，其他未配置百分比的PCRF平均分担剩余的百分比例。
  UNC 支持PCRF组内负荷分担功能，可以充分利用现网PCRF的性能，但在网络规划时，需要考虑PCRF的容灾能力，避免PCRF组内的PCRF均接近满负荷，导致某个PCRF故障时无法将流量分担到其他可用的PCRF上。

#### [链路可靠性-多Diameter链路](#ZH-CN_TOPIC_0229056158)

UNC 支持与一个PCRF的多个IP或多个SCTP端点建立多条直连Diameter链路，构成一个链路组，链路组内的多条链路间支持主备或负荷分担模式（链路选择模式通过 [**ADD DIAMCONNGRP**](../../../../../../OM参考/命令/UNC MML命令/业务服务管理/接口管理/计费和策略接口管理/Diameter管理/Diameter连接/Diameter链路组/增加Diameter链路组（ADD DIAMCONNGRP）_09897261.md) 命令中的 “SELECTMODE” 参数来设置）。当链路组内的一条链路故障时， UNC 选择链路组内的其他可用链路进行消息交互，增强信令组网的可靠性。

UNC 支持通过TCP或SCTP协议来承载Diameter协议，可以分别构建多Diameter链路：

- 采用TCP传输时，UNC与一个PCRF的多个IP建立多条TCP链路，各链路间为主备或负荷分担模式。
- 采用SCTP传输时，UNC与一个PCRF的多个SCTP端点间建立多条SCTP偶联，各偶联间为主备或负荷分担模式。SCTP功能的详细介绍请参见[WSFD-104508 SCTP](../../../组网功能/WSFD-104508 SCTP_27680892.md)。

#### [迟滞时间](#ZH-CN_TOPIC_0229056158)

PGW-C支持PCRF下发的规则携带激活时间/去激活时间（灵活时间）。相关AVP为 **Rule-Activation-Time** 和 **Rule-Deactivation-Time** ，可同时携带两个AVP或只携带一个AVP。PGW-C根据下发的激活时间/去激活时间，指示PGW-U安装或删除对应的规则。

PCRF向同一个用户下发多个rule且携带激活时间/去激活时间，若两次切换时间间隔小于迟滞时间，则PGW-C会延迟到迟滞时间后再做状态切换。如PCRF下发的rule1的激活时间（ **Rule-Activation-Time** ）是08:30，rule2激活时间（ **Rule-Activation-Time** ）是08:35，rule2和rule1两次灵活切换时间间隔小于10分钟（迟滞时间），则PGW-C激活rule1后，将在08:40（08:30+迟滞时间）时激活rule2。

迟滞时间默认为10分钟，不支持修改。

#### [一次重定向](#ZH-CN_TOPIC_0229056158)

一次重定向（One-Shot Redirection）只对用户业务重定向一次，后续用户可以正常上网。如用户激活后正常上网时，向用户推送一次运营商定制的Portal页面；用户在机场上网时，弹出机场的主页，向用户推送业务。

一次重定向中PCRF通过Gx接口下发的策略中需要指定重定向规则，如果相关字段为enable，则表示为one-shot重定向。另外，PCRF通过Gx接口还可以下发REDIRECTION（100）Event-Trigger，用于指示 UNC 在重定向成功后，向PCRF发送CCR-U消息，通知PCRF重定向成功。

#### [Supported Features动态协商](#ZH-CN_TOPIC_0229056158)

为了可以更加灵活的与支持不同协议版本和不同能力的PCRF进行对接， UNC 支持Supported Features动态协商功能，即 UNC 与PCRF在IP-CAN Session建立过程中进行协商，决定 UNC 和PCRF之间的会话按照哪个协议版本进行交互，或指示 UNC 和PCRF是否具有相应的feature能力。

UNC 支持的feature列表，请参见 [**ADD PCRF**](../../../../../../OM参考/命令/UNC MML命令/业务服务管理/会话管理/PCC管理/PCRF Diameter连接/PCRF/增加PCRF（ADD PCRF）_09897101.md) 中的 “FEATURELIST” 参数的取值范围。

UNC 的Supported Features动态协商功能使能，则 UNC 与PCRF之间Feature能力的协商过程分如下两种：

- UNC本地配置的PCRF Feature能力与对端PCRF实际支持的Feature能力不一致，协商过程如[图1](#ZH-CN_TOPIC_0229056158__fig108761642112613)所示的首个用户上线过程和下个用户上线过程。
- UNC本地配置的PCRF Feature能力与对端PCRF实际支持的Feature能力一致，首个用户上线的协商过程如[图1](#ZH-CN_TOPIC_0229056158__fig108761642112613)中下个用户上线过程类似，均只需要一次协商过程，唯一差别在于CCR-I消息中携带的Feature能力的获取来源不同。
    - 首个用户上线：来源于本地[**ADD PCRF**](../../../../../../OM参考/命令/UNC MML命令/业务服务管理/会话管理/PCC管理/PCRF Diameter连接/PCRF/增加PCRF（ADD PCRF）_09897101.md)命令配置的feature能力。
    - 下个用户上线：来源于本地存储的第一个IP-CAN会话的Supported Features动态协商结果。

<br>

由于二次协商流程会导致Gx接口的信令交互增加，故 UNC 会在第一个会话建立过程中学习并存储对端PCRF的实际能力，即实际的协商结果，后续会话建立时，直接根据对端的实际能力发起协商，从而可以一次协商成功，提高协商成功率。在网络中实际部署时建议将本地配置的PCRF feature能力与PCRF实际支持的feature能力一致，减少二次协商过程。

考虑到PCRF的升级场景， UNC 需定时清除本地缓存的对端PCRF能力，以保证新接入的用户可以发起全新的协商流程，从而学习到对端PCRF新的真实的支持能力。在 UNC 上可以使用 [**SET PCCTIMER**](../../../../../../OM参考/命令/UNC MML命令/业务服务管理/会话管理/PCC管理/信令控制/定时器/设置PCC定时器（SET PCCTIMER）_09897082.md) 命令配置本地缓存的定时器时长，或使用 [**RST PCCTIMER**](../../../../../../OM参考/命令/UNC MML命令/业务服务管理/会话管理/PCC管理/信令控制/复位PCC定时器/复位PCC定时器（RST PCCTIMER）_09897075.md) 命令手动重启定时器清除本地缓存的对端能力信息。

如果 UNC 动态协商功能使能，而PCRF不支持Supported-Features AVP，则PCRF会返回值为DIAMETER_AVP_UNSUPPORTED的Result-Code AVP，并且携带Failed-AVP AVP指示无法识别Supported-Features AVP，最终IP-CAN会话创建失败。

如果 UNC 动态协商功能不使能，则两端根据规划的一致的最高协议版本进行交互。如 UNC 本地配置PCRF支持Rel8、Rel9协议版本，对端PCRF自身支持Rel8、Rel9协议版本，则 UNC 与PCRF按照Rel9协议版本进行交互。

如需了解更多动态协商的介绍，请参考3GPP TS 29.229和3GPP TS 29.212。

#### [QoS映射机制](#ZH-CN_TOPIC_0229056158)

UNC 收到GnGp-SGSN发送的承载创建或修改请求或收到PCRF下发的授权QoS信息后， UNC 需要按照如下原则进行QoS参数映射。

- GPRS/UMTS QoS参数与EPC Gx接口AVP的映射关系
  *表3 GPRS/UMTS QoS参数与Gx接口AVP的映射关系*

  | 承载类型 | GPRS/UMTS PDP中的QoS参数 | EPC Gx接口AVP |
  | --- | --- | --- |
  | Non-GBR承载 | MBR | APN-Aggregate-Max-Bitrate-UL AVP<br>APN-Aggregate-Max-Bitrate-DL AVP |
  | Non-GBR承载 | 一次PDP中的QoS参数（Traffic class、Traffic Handling Priority、Signalling Indication、Source Statistics Descriptor） | Default-EPS-Bearer-QoS AVP |

  > 当 UNC 收到PCRF修改的APN-Aggregate-Max-Bitrate-UL AVP和APN-Aggregate-Max-Bitrate-DL AVP取值时，需要针对所有的Non-GBR承载，发起承载MBR参数的更新流程。
  >
  > 当GPRS/UMTS的QoS参数映射为QCI的2或3时，GPRS/UMTS中的Transfer Delay与其他QoS参数共同参与映射，而SDU Error Ratio参数将被直接忽略。
- QCI和GPRS/UMTS QoS参数的映射
  QCI和GPRS/UMTS QoS参数的映射关系请参见 [WSFD-010701 QoS与流量管理](../../../QoS功能/WSFD-010701 QoS与流量管理_65076395.md) 。
- ARP映射
  GPRS/UMTS ARP与EPC ARP的映射关系请参见 [WSFD-010701 QoS与流量管理](../../../QoS功能/WSFD-010701 QoS与流量管理_65076395.md) 。
  如果GnGp-SGSN发送的请求消息中包含eARP，则 UNC 优先使用eARP参数与PCRF进行协商，并将协商后的eARP通过响应消息发送给GnGp-SGSN。同时协商后的eARP需要按照映射原则映射为ARP带给GnGp-SGSN。

**QoS能力开放**

UNC 支持QoS能力开放功能，通过私有Restif接口提供IP地址给SBC，当SBC需要查询用户信息时， UNC 查询到用户的相关信息并返回结果给SBC。 UNC 通过服务化API接口开放PCC会话查询信息，包括用户的PCRF ID、PCEF ID等信息。SBC网元可以向 UNC 请求查询用户所在的PCRF ID信息，SBC基于PCRF ID直接在Rx接口寻址用户所在PCRF的网元，完成动态QoS策略申请。

在地址复用的场景下，用户IP地址不唯一，当需要根据用户IP地址查询用户时，SBC发送给 UNC 的信息包含IP地址+IPDomain，可以根据IPDomain和用户IP地址查询确定唯一用户。
