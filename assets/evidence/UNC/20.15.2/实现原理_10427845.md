# 实现原理

本迁移方案，待迁移AMF只处理连接态用户。针对空闲态的用户，待迁移AMF主动发起Paging流程，UE响应Paging进入连接态后，如连接态用户一样被迁移。

迁移段过程简介：

- 连接态用户
  待迁移AMF通过调用目标AMF的Namf_Communication_N1MessageNotify服务操作，把UE Context以及Global RAN Node ID、RAN UE NGAP ID等N2接口信息一并发送给目标AMF，目标AMF接收到相关信息后：
    - 通过向NG-RAN发起UE Context Modification流程，更新UE-TNLA-binding关系到目标AMF上（NG-RAN将UE绑定到该AMF的一个TNLA（Transport Network Layer Association，传输层连接）上，生成一个“UE-TNLA-binding”关系，后续该UE的上行N2消息都在该TNLA上发送）。该流程中目标AMF向NG-RAN发送给的UE CONTEXT MODIFICATION REQUEST消息中，包含目标AMF为该UE分配的AMF UE NGAP ID信息。
    - 向UE重分配5G-GUTI，该5G-GUTI中包含目标AMF自己的GUAMI信息。
    - 向周边NF，如SMF、UDM、PCF等更新目标AMF的信息。
- 空闲态用户
  针对空闲态UE，待迁移AMF主动发起Paging流程，UE响应Paging进入连接态后，待迁移AMF按照连接态用户的迁移方式处理，将UE迁移到AMF Set内的其它AMF上。

为满足运营商拨测或测试场景的需要， UNC 增加了按IMSI迁移单个用户迁移至Pool内指定目标AMF的功能。

> **说明**
> 禁止使用该功能对正在进行业务流程的用户进行迁移。
