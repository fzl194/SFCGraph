# 实现原理

**业务流程**

基于SGs接口实现短消息业务特性涉及的主要业务包括：

- [主叫短消息](#ZH-CN_CONCEPT_0166292237__li2)
- [被叫短消息](#ZH-CN_CONCEPT_0166292237__li3)

基于SGs接口实现短消息业务特性的移动性管理主要是通过SGs接口实现的，用户在附着/去附着网络，或者发生位置更新时，MME通过SGs接口跟MSC Server进行信息交互。

- 主叫短消息流程
    - 空闲态的主叫短消息流程
      **图1** 空闲态的主叫短消息流程图

      <br>

      ![](实现原理_66292237.assets/zh-cn_image_0166667456_2.png)

      <br>
      具体流程描述如下：
      1.UE在发起主叫短消息之前，已经完成了EPS/IMSI的联合附着。
      2.如果UE处于IDLE态，则UE在发送短消息前需要先进行业务请求流程，伴随着空口RRC连接和S1信令连接的建立。
      3.UE发送Uplink NAS Transport消息给MME，相关的短消息信息CP-DATA/RP-DATA/TPDU/SMS-SUBMIT作为参数NAS message container打包在上行NAS消息里。
      4.MME查询UE的SGs关联信息，确定目标MSC，发送SGsAP Uplink Unitdata消息给MSC，其中含有参数NAS message container。MME另外给MSC提供参数IMEISV，the local time zone，the Mobile Station Classmark 2，TAI和E CGI，用于产生话单。
      4a.MSC发送SGsAP Downlink Unitdata消息给MME，确认收到短消息，MME将其中的NAS消息通过Downlink NAS Transport消息传给UE。
      5–8.主叫短消息在CS核心网传递的过程，短消息中心给MSC/VLR回Delivery Report。
      9.MSC将接收报告Delivery Report传递给MME。
      10.MME将接收报告通过下行NAS消息发给UE。
      11–12.UE反馈接收到接收报告。
      13.MSC发送SGsAP Release Request消息指示MME，关于该UE，MSC/VLR没有剩余的NAS消息需要交互。
    - 连接态的主叫短消息流程
      连接态的主叫短消息流程与空闲态的主叫短消息流程基本类似，除了以下区别：UE无需发起业务请求流程，直接在已有的信令连接上与MME交互。
    - 多条主叫短消息流程
      当UE第一条主叫短消息还没有发送完毕，又发送另一条短消息，UE先不发送 [图1](#ZH-CN_CONCEPT_0166292237__fig_02) 中的第11步确认消息，先发送Uplink NAS Transport消息给MME。这样避免了释放空口和S1资源后，又重新建立。
- 被叫短消息流程
    - 空闲态的被叫短消息流程
      **图2** 空闲态的被叫短消息流程图

      <br>

      ![](实现原理_66292237.assets/zh-cn_image_0166667453_2.png)
      具体流程描述如下：
      1.UE完成了EPS/IMSI的联合附着。
      2–4.被叫短消息在CS核心网传递的过程。
      5.MSC/VLR查询UE的SGs关联信息，发送SGsAP Paging Request消息给MME，其中参数service indicator指示这是一个短消息业务。
      6.MME发送Paging（S-TMSI，CN Domain Indicator）消息给eNodeB。在该场景下，MME设置CN Domain Indicator为“PS”。
      7.LTE空口的寻呼流程。
      8.UE响应寻呼，发起业务请求流程，伴随着空口RRC信令连接和S1信令连接的建立。
      8a.MME发送SGsAP Service Request消息给MSC/VLR，指示UE与MME之间已经可以传递NAS消息。MME另外给MSC提供参数IMEISV，the local time zone，the Mobile Station Classmark 2，TAI和E CGI，用于产生话单。MSC给MME发送SGsAP Alert Request消息给MME，要求当UE和MME联系后通知MSC。当UE与MME有交互后，MME发送SGsAP UE Activity Indication消息给MSC。MSC发送MAP Ready For SM消息给HLR通知UE状态可及。
      9a-9d.MSC/VLR发送SGsAP Downlink Unitdata消息给MME，其中短消息作为参数打包在NAS message container里，MME将其中的短消息通过Downlink NAS Transport消息传给UE，UE给MSC/VLR回应，确认收到短消息。
      10.UE将接收报告发送给MME。
      11.MME将接收报告传递给MSC。
      12–13.MSC/VLR将Delivery Report发送给SC。
      14–15.与12步，13步并行，MSC/VLR通过MME给UE回收到的Delivery Report确认消息。
      16.MSC发送SGsAP Release Request消息指示MME，关于该UE，MSC/VLR没有剩余的NAS消息需要交互。
    - 连接态的被叫短消息流程
      连接态的被叫短消息流程与空闲态的被叫短消息流程基本类似，除了以下区别：
          - MME在[图2](#ZH-CN_CONCEPT_0166292237__fig_03)中第5步收到SGsAP Paging Request消息后，查询UE上下文，知道当前UE处于连接态，MME不会去寻呼UE，也即跳过[图2](#ZH-CN_CONCEPT_0166292237__fig_03)中的第6步到第8步，直接执行8a步，MME发送SGsAP Service Request消息给MSC/VLR，指示UE与MME之间已经可以传递NAS消息。
    - 多条被叫短消息流程
      当MSC有多条短消息需要发送给UE时，MSC不发送 [图2](#ZH-CN_CONCEPT_0166292237__fig_03) 中的第16步的SGsAP Release Request消息，先通过发送SGsAP Downlink Unitdata消息将新的短消息传给MME。这样避免了释放空口和S1资源后，再次寻呼UE。
