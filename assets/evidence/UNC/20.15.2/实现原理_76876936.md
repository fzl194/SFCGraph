[**系统概述**](#ZH-CN_TOPIC_0276876936__sub3)

UP-CIoT的组网与传统4G组网相同，是基于用户面数据传输方式上的优化，如 [图1](#ZH-CN_TOPIC_0276876936__fig01) 所示。

**图1** UP-CIoT组网

<br>

![](实现原理_76876936.assets/zh-cn_image_0276876997_2.png "点击放大")

<br>

系统实现的功能如下：

- NB-IoT场景下，MME根据终端、eNodeB、MME的CIoT能力以及本特性的License判断是否采用UP-CIoT数传方式。
- 当终端进入Idle态时，终端可通过挂起流程（Connection Suspend）由连接态转向空闲态挂起状态，挂起流程中MME保留S1-AP相关联接信息、UE上下文和承载上下文。当有数据要传输时，UP-CIoT终端可发起连接恢复流程（Connection Resume），只需恢复上游SGW承载即可恢复数传。无需频繁S1 Release/Service Request流程释放和建立E-RAB承载，节省信令开销，提升数传效率。
- UP-CIoT可以实现较大数据的传输，当CP-CIoT不能满足数据传输要求时，终端可在Control Plane Service Request消息中携带active flag信元，MME将建立S1–U用户面，实现CP-CIoT到UP-CIoT之间的切换，满足终端需要传输较大数据的要求。

与传统4G流程相比，本特性做了如下特殊处理：

- MME依据Attach Request/TAU Request消息中携带UE network capability信元指示的CIoT能力、本地配置的eNodeB的CIoT能力、本地配置的MME的CIoT能力以及本特性的License判断出MME支持UP-CIoT。eNodeB的CIoT能力可通过 [**ADD TAICIOT**](../../../../../OM参考/命令/UNC MML命令/业务服务管理/Pre 5G接入业务管理/控制面管理/业务安全管理/M2M管理/CIoT能力配置/增加CIoT能力配置(ADD TAICIOT)_26305590.md) 命令配置。MME的CIoT能力由 [**ADD M2MPLCY**](../../../../../OM参考/命令/UNC MML命令/业务服务管理/Pre 5G接入业务管理/控制面管理/业务安全管理/M2M管理/M2M策略参数配置/增加M2M策略参数(ADD M2MPLCY)_72225443.md) 命令中的参数 “S1-U能力” 控制。
- 如果判断结果两种数传方式（UP-CIoT/CP-CIoT）均支持，MME将会根据Attach Request/TAU Request消息中携带的Additional update type信元进一步确定数传方式。该信元中携带了用于标识终端偏好的标志位。
- Attach流程中，MME在Initial Context Setup Request消息中携带UE User Plane CIoT Support Indicator信元，指示eNodeB MME支持UP-CIoT，后续eNodeB可以发起Connection Suspend和Connection Resume流程。

[**UP-CIoT承载管理**](#ZH-CN_TOPIC_0276876936__sub3_3)

终端在进入Idle态时，通过Connection Suspend流程将承载挂起 [图4](#ZH-CN_TOPIC_0276876936__fig04) ，终端、eNodeB和MME均保留用于恢复连接的S1-AP关联信息、用户上下文和承载上下文信息。处于Idle挂起状态的UP-CIoT终端有数据需要传输时，触发Connection Resume流程快速恢复承载。如 [图5](#ZH-CN_TOPIC_0276876936__fig05) 。

**图4** Connection Suspend流程

<br>

![](实现原理_76876936.assets/zh-cn_image_0276876998_2.png "点击放大")

<br>

1. eNodeB向MME发送UE Context Suspend Request消息，标识UE RRC连接挂起。eNodeB、MME和UE都保留用于恢复连接的S1-AP联接、UE上下文信息、承载上下文信息。
2. MME发送Release Access Bearers Request通知SGW释放S1–U承载上下文。
3. S-GW释放所有UE相关的eNodeB信息（地址和下行TEID）并且回复Release Access Bearers Response消息给MME。
4. MME收到SGW回复的Release Access Bearers Response消息后，向eNodeB发送UE Context Suspend Response消息。
5. eNodeB给UE发送RRC Connection Suspend消息，流程结束。UE进入Idle挂起状态。

**图5** Connection Resume流程

<br>

![](实现原理_76876936.assets/zh-cn_image_0276876999_2.png "点击放大")

<br>

1. 当UE有上行数据待发送或欲发起上行NAS信令消息时，UE向eNodeB触发Random Access流程。
2. UE触发RRC Connection Resume流程，流程中包含eNodeB所需的信息来接入UE存储的AS上下文。
3. eNodeB向MME发送UE Context Resume Request。
4. MME恢复承载成功，MME向eNodeB回复UE Context Resume Response。如果MME没有恢复所有的承载，MME将恢复失败的E-RAB承载列表携带在消息中。
5. 如果MME在第四步中携带了恢复失败的E-RAB承载列表，eNodeB需要重新配置无线承载。
6. 此时UE的上行数据包可以被eNodeB转发给S-GW。S-GW再将上行包转发给P-GW。
7. MME给S-GW发送Modify Bearer Request消息恢复UP-CIoT承载。
8. MME收到S-GW的Modify Bearer Response响应，流程结束，开始下行数传。

[**CP-CIoT到UP-CIoT的数传方式切换**](#ZH-CN_TOPIC_0276876936__sub3_3)

终端采用CP-CIoT数传方式（对应的License控制项为“82207029 LKV2DNAS01 基于信令面的数据传输 ”），当业务需要切换数传方式时，终端在发给MME的Control Plane Service Request消息时携带active flag。如果MME支持UP-CIoT能力以及本特性License开启，则会通过Initial Context Setup Request/Response消息建立S1–U用户面，实现CP-CIoT到UP-CIoT承载的切换。如果MME由于license或者配置原因不支持UP-CIoT，则直接拒绝Control Plane Service Request。
