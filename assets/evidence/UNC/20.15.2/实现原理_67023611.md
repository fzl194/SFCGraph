# 实现原理

VoLTE业务流程请参考 [VoLTE业务流程](../VoLTE业务流程_73710793.md) ，这里仅详细介绍VoLTE寻呼优化。

VoLTE寻呼优化指对VoLTE语音业务设置单独的寻呼间隔和重发次数。VoLTE寻呼如果按照普通用户的寻呼参数设置，在寻呼重发的场景下，寻呼间隔时间较长，会导致VoLTE接通时延较长甚至接续失败，影响用户业务体验。

MME支持VoLTE寻呼参数独立配置。寻呼间隔和重发次数可通过命令 [**SET EMM**](../../../../../OM参考/命令/UNC MML命令/业务服务管理/Pre 5G接入业务管理/控制面管理/移动性管理/MM协议参数管理/S1模式MM协议参数/设置S1模式MM协议参数(SET EMM)_72225207.md) 中参数 “高优先级业务的T3413(s)” 、 “高优先级业务的N3413(times)” 和 “高优先级业务的重寻呼间隔递增值(s) ” 配置，当MME识别出VoLTE语音业务时（识别方法见 [表1](#ZH-CN_CONCEPT_0167023611__table_035FF3E9) ），这些参数的优先级高于 [**SET EMM**](../../../../../OM参考/命令/UNC MML命令/业务服务管理/Pre 5G接入业务管理/控制面管理/移动性管理/MM协议参数管理/S1模式MM协议参数/设置S1模式MM协议参数(SET EMM)_72225207.md) 中已配置的普通寻呼参数 “T3413(s)” “N3413(times)” 和 “重寻呼间隔递增值(s) ” 。

通过命令 [**ADD QCIPAGINGINFO**](../../../../../OM参考/命令/UNC MML命令/业务服务管理/Pre 5G接入业务管理/控制面管理/移动性管理/QCI寻呼策略参数配置/增加QCI寻呼策略参数配置(ADD QCIPAGINGINFO)_72225211.md) 对特定QCI的寻呼配置了寻呼间隔和重发次数的情况下，如果在S11接口根据CBR/UBR（Creat Bearer Request/Update Bearer Request）消息或DDN（Downlink Data Notification）消息获取到的QCI信息匹配到 [**ADD QCIPAGINGINFO**](../../../../../OM参考/命令/UNC MML命令/业务服务管理/Pre 5G接入业务管理/控制面管理/移动性管理/QCI寻呼策略参数配置/增加QCI寻呼策略参数配置(ADD QCIPAGINGINFO)_72225211.md) 的标准QCI值，那么该寻呼将采用 [**ADD QCIPAGINGINFO**](../../../../../OM参考/命令/UNC MML命令/业务服务管理/Pre 5G接入业务管理/控制面管理/移动性管理/QCI寻呼策略参数配置/增加QCI寻呼策略参数配置(ADD QCIPAGINGINFO)_72225211.md) 配置的寻呼策略而不再受到 [**SET EMM**](../../../../../OM参考/命令/UNC MML命令/业务服务管理/Pre 5G接入业务管理/控制面管理/移动性管理/MM协议参数管理/S1模式MM协议参数/设置S1模式MM协议参数(SET EMM)_72225207.md) 配置的寻呼间隔和重发次数的影响。

> **说明**
> CE Mode B终端（一种eMTC终端）的寻呼时长要长于普通终端，因此 [**ADD QCIPAGINGINFO**](../../../../../OM参考/命令/UNC MML命令/业务服务管理/Pre 5G接入业务管理/控制面管理/移动性管理/QCI寻呼策略参数配置/增加QCI寻呼策略参数配置(ADD QCIPAGINGINFO)_72225211.md) 配置的寻呼间隔和重发次数不适用于该终端。对于CE Mode B终端用户，可使用 [**SET EMTCBEMM**](../../../../../OM参考/命令/UNC MML命令/业务服务管理/Pre 5G接入业务管理/控制面管理/业务安全管理/M2M管理/eMTC-MM协议参数管理/设置S1模式eMTC MM协议参数（SET EMTCBEMM）_26145778.md) 配置寻呼间隔和重发次数。

*表1 MME识别VoLTE语音业务的方法*

| 触发寻呼业务 | 识别方法 | 识别的业务类型 |
| --- | --- | --- |
| VoLTE普通语音被叫 | MME根据Downlink Data Notification、Update Bearer Request消息中的Bearer ID，或Create Bearer Request消息中的LBI（Linked EPS Bearer Identity），查询该承载对应的APN，若APN为IMS APN（IMS APN可以是“IMS”或者<br>[**ADD VOICEDEPLOY**](../../../../../OM参考/命令/UNC MML命令/业务服务管理/Pre 5G接入业务管理/控制面管理/业务安全管理/语音业务管理/增加语音部署配置(ADD VOICEDEPLOY)_72345361.md)<br>配置的APN），则判断为VoLTE语音被叫业务。 | VoLTE语音被叫 |
| VoLTE紧急呼叫 | MME根据Downlink Data Notification、Update Bearer Request消息中的Bearer ID，或Create Bearer Request消息中的LBI（Linked EPS Bearer Identity），查询该承载对应的APN，若APN为紧急APN（紧急APN由<br>[**ADD EMGCFG**](../../../../../OM参考/命令/UNC MML命令/业务服务管理/Pre 5G接入业务管理/控制面管理/移动性管理/紧急呼叫配置/紧急呼叫功能配置/增加运营商紧急呼叫功能配置（ADD EMGCFG）_26305316.md)<br>命令中的<br>“紧急APN”<br>参数配置），则判断为VoLTE紧急呼叫业务。 | VoLTE紧急呼叫 |
| 基于HSS的状态/位置查询 | 根据HSS向MME发送的Insert Subscriber Data Request消息中IDR-Flags信元识别，满足如下任一条件时，判断为VoLTE语音业务。<br>- 信元中的Bit2为1，即EPS User State Request为1。<br>- 信元中的Bit3和Bit4都为1，即EPS Location Information Request和Current Location Request都为1。 | VoLTE语音被叫 |
| 基于PCC的位置查询 | 根据Update Bearer Request消息中Bearer ID识别该承载对应的APN。<br>- 若APN为IMS APN，则判断为VoLTE语音被叫业务。<br>- 若APN为紧急APN，则判断为VoLTE紧急呼叫业务。 | VoLTE语音被叫或VoLTE紧急呼叫 |
| 基于HSS的P-CSCF故障恢复 | VoLTE呼叫业务中，S-CSCF检测到主用P-CSCF故障时，发送INVITE消息给备用P-CSCF。HSS发送Insert Subscriber Data Request消息给MME通知P-CSCF地址变更，消息中携带信元IDR-Flags，且信元中的bit8为1，即表示P-CSCF发起的容灾请求，MME识别为VoLTE语音。 | VoLTE语音 |

根据3GPP 29.274定义，Downlink Data Notification消息中的Bearer ID为可选信元。为确保MME能够通过Downlink Data Notification消息中的Bearer ID识别VoLTE语音被叫，需要在 UNC 上通过 [**SET DDNATTR**](../../../../../OM参考/命令/UNC MML命令/业务服务管理/会话管理/接入管理/接入管理运维/DDN消息携带的属性/设置DDN消息参数以及Delay信元处理开关（SET DDNATTR）_09651485.md) 命令将 “携带EBI” 设置为 “Enable(使能)” ，以支持在此消息中携带Bearer ID。
