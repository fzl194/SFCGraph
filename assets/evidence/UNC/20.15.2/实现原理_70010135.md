#### [语音寻呼优先业务](#ZH-CN_TOPIC_0170010135)

VoLTE/CSFB用户做被叫接续时，MME发起寻呼，在下发给eNodeB的寻呼消息中包含有“CN Domain”和“Paging Priority”信元。

- CN Domain：用于区分寻呼CS域还是PS域。寻呼CS域则进行CSFB语音业务，寻呼PS域则进行VoLTE语音业务。
- Paging Priority：用于指示差异化寻呼用户。部署语音寻呼优先业务后，寻呼消息中会携带Paging Priority，携带Paging Priority的寻呼消息的优先级比不携带的寻呼消息优先级高，携带的Paging Priority取值越小，优先级越高。

**VoLTE被叫寻呼优先处理**

**图1** VoLTE被叫寻呼优先流程

<br>

![](实现原理_70010135.assets/zh-cn_image_0000001268122842_2.png)

1～3. SGW-C收到发送给UE的下行数据包，并传递给MME。

4. MME识别为VoLTE语音业务，则进行高优先级寻呼，在发给eNodeB的Paging消息中携带Paging Priority。

在MME上需要将 [**SET MMFUNC**](../../../../../OM参考/命令/UNC MML命令/业务服务管理/Pre 5G接入业务管理/控制面管理/移动性管理/MM扩展功能管理/设置移动性管理扩展功能(SET MMFUNC)_26145512.md) 命令中 “VOICEPAGINGPRIORITY” 参数设置为“YES”，开启语音寻呼优先功能。

*表2 MME识别VoLTE语音业务的方法*

| 触发寻呼业务 | 识别方法 | 识别的业务类型 |
| --- | --- | --- |
| VoLTE普通语音被叫 | MME根据Downlink Data Notification、Update Bearer Request消息中的Bearer ID，或Create Bearer Request消息中的LBI（Linked EPS Bearer Identity），查询该承载对应的APN，若APN为IMS APN（IMS APN可以是“IMS”或者<br>[**ADD VOICEDEPLOY**](../../../../../OM参考/命令/UNC MML命令/业务服务管理/Pre 5G接入业务管理/控制面管理/业务安全管理/语音业务管理/增加语音部署配置(ADD VOICEDEPLOY)_72345361.md)<br>配置的APN），则判断为VoLTE语音被叫业务。 | VoLTE语音被叫 |
| VoLTE紧急呼叫 | MME根据Downlink Data Notification、Update Bearer Request消息中的Bearer ID，或Create Bearer Request消息中的LBI（Linked EPS Bearer Identity），查询该承载对应的APN，若APN为紧急APN（紧急APN由<br>[**ADD EMGCFG**](../../../../../OM参考/命令/UNC MML命令/业务服务管理/Pre 5G接入业务管理/控制面管理/移动性管理/紧急呼叫配置/紧急呼叫功能配置/增加运营商紧急呼叫功能配置（ADD EMGCFG）_26305316.md)<br>命令中的<br>“紧急APN”<br>参数配置），则判断为VoLTE紧急呼叫业务。 | VoLTE紧急呼叫 |
| 基于HSS的状态/位置查询 | 根据HSS向MME发送的Insert Subscriber Data Request消息中IDR-Flags信元识别，满足如下任一条件时，判断为VoLTE语音业务。<br>- 信元中的Bit2为1，即EPS User State Request为1。<br>- 信元中的Bit3和Bit4都为1，即EPS Location Information Request和Current Location Request都为1。 | VoLTE语音被叫 |
| 基于PCC的位置查询 | 根据Update Bearer Request消息中Bearer ID识别该承载对应的APN。<br>- 若APN为IMS APN，则判断为VoLTE语音被叫业务。<br>- 若APN为紧急APN，则判断为VoLTE紧急呼叫业务。 | VoLTE语音被叫或VoLTE紧急呼叫 |
| 基于HSS的P-CSCF故障恢复 | VoLTE呼叫业务中，S-CSCF检测到主用P-CSCF故障时，发送INVITE消息给备用P-CSCF。HSS发送Insert Subscriber Data Request消息给MME通知P-CSCF地址变更，消息中携带信元IDR-Flags，且信元中的bit8为1，即表示P-CSCF发起的容灾请求，MME识别为VoLTE语音。 | VoLTE语音 |

5. eNodeB根据MME下发的Paging Priority信息进行高优先级寻呼。

**CSFB被叫寻呼优先处理**

**图2** CSFB被叫寻呼优先流程

<br>

![](实现原理_70010135.assets/zh-cn_image_0000001317082793_2.png)

1. MSC发起语音寻呼流程。

2. MME识别为CSFB语音业务，则进行高优先级寻呼，在发给eNodeB的Paging消息中携带Paging Priority。

在MME上需要将 [**SET MMFUNC**](../../../../../OM参考/命令/UNC MML命令/业务服务管理/Pre 5G接入业务管理/控制面管理/移动性管理/MM扩展功能管理/设置移动性管理扩展功能(SET MMFUNC)_26145512.md) 命令中 “VOICEPAGINGPRIORITY” 参数设置为“YES”，开启语音寻呼优先功能。

本特性中不包含CSFB优先语音业务（eMLPP），eMLPP业务详情见 [WSFD-102502 基于CSFB的优先语音服务](../WSFD-102502 基于CSFB的优先语音服务_64009915.md) 。eMLPP业务等同于本特性的MPS业务。

*表3 MME识别CSFB语音业务的方法*

| 触发寻呼业务 | 识别方法 | 识别的业务类型 |
| --- | --- | --- |
| CSFB普通语音被叫 | MME根据SGsAP Paging消息中service indicator来判断是否为CSFB语音被叫。若service indicator为“CS call indicator”，则为CSFB语音被叫。普通被叫和紧急呼叫不做区分，统一按照CSFB语音被叫处理。（根据3GPP 23.272定义，SGsAP Paging消息中无法区分普通被叫和紧急呼叫，故统一识别为CSFB语音被叫业务。） | CSFB语音被叫 |
| CSFB紧急呼叫 | MME根据SGsAP Paging消息中service indicator来判断是否为CSFB语音被叫。若service indicator为“CS call indicator”，则为CSFB语音被叫。普通被叫和紧急呼叫不做区分，统一按照CSFB语音被叫处理。（根据3GPP 23.272定义，SGsAP Paging消息中无法区分普通被叫和紧急呼叫，故统一识别为CSFB语音被叫业务。） | CSFB语音被叫 |

> **说明**
> 对于1xCSFB语音业务，MME收到IWS的1xRTT CS Paging请求时，判断寻呼请求消息中是否携带了寻呼优先级。如果携带，则识别为eMLPP业务，否则识别为1xCSFB语音被叫，优先级同CSFB语音被叫。

3. eNodeB根据MME下发的Paging Priority信息进行高优先寻呼。

#### [MPS业务](#ZH-CN_TOPIC_0170010135)

部署MPS业务功能后，用户在网络接入、承载资源处理等所有业务时都会进行高优先级处理。如下仅以用户在网络附着和被叫寻呼场景为例进行介绍。

**网络附着的优先处理**

**图3** 网络附着的优先处理

<br>

![](实现原理_70010135.assets/zh-cn_image_0000001268282706_2.png)

1. UE发起附着请求时，如果用户已签约MPS优先业务，则请求建立优先承载的HighPriorityAccess。

2. eNodeB根据UE携带的高优先级指示在initial UE message的RRC establishment cause中设定HighPriorityAccess。

3～4. 用户在HSS中签约了基于VoLTE的优先语音服务业务，HSS将用户的优先级签约标识（MPS EPS Priority）下发给MME。

5. MME根据HighPriorityAccess和HSS下发的优先级签约标识，判断该UE符合高优先级接入要求，则允许该UE进行高优先接入。

6. SGW-C/PGW-C向PCRF发起CCR请求。

7～8. PCRF从SPR获取用户签约的优先级。在PCRF中设置用户优先级时，使用SPR中用户基本信息的category信元来设置用户优先级。在PCRF的策略中，根据category取值不同设置不同的QoS策略。

9. PCRF将用户的优先级信息转换成承载的ARP信息，并传递SGW-C/PGW-C。

10. SGW-C/PGW-C根据PCRF下发的ARP和QCI创建承载。

11～18. 后续处理流程与普通的用户的网络附着一致，此处不再赘述。

**被叫寻呼优先处理**

**图4** 寻呼流程图

<br>

![](实现原理_70010135.assets/zh-cn_image_0000001317322645_2.png)

1～2. SGW-C收到发送给UE的下行数据包，并传递给MME。SGW-C在Downlink Data Notification消息中携带ARP信息（根据用户的签约数据决定）给MME。

[图4](#ZH-CN_TOPIC_0170010135__fig13858164017816) 仅以Downlink Data Notification消息为例，其它消息Create Bearer Request、Update Bearer Request中也会携带ARP信息给MME。

4. MME将已配置的ARP优先级阈值与SGW-C下发的ARP中的pl值（即用户签约的优先级数据）比较，决定是否进行高优先级寻呼。值越小，优先级越高。

MME上需要通过 [**SET MPSARP**](../../../../../OM参考/命令/UNC MML命令/业务服务管理/Pre 5G接入业务管理/控制面管理/移动性管理/MPS配置/MPS ARP信息配置/设置MPS ARP配置(SET MPSARP)_26305312.md) 命令中的 “PRILVL(优先级)” 参数配置ARP优先级阈值。

- SGW-C下发的ARP（pl）值＞MME配置的ARP优先级阈值，即用户签约的优先级小于MME配置的阈值，MME按照普通业务处理，在发给eNodeB的Paging Request消息中不携带Paging Priority。
- SGW-C下发的ARP（pl）值≤MME配置的ARP优先级阈值，即用户签约的优先级高于MME配置的阈值，MME根据SGW-C下发的ARP信息进行高优先级寻呼，在发给eNodeB的Paging消息中增加Paging Priority。

Paging Priority的值与SGW-C下发的ARP（pl）值的对应关系如 [表4](#ZH-CN_TOPIC_0170010135__table148601040481) 所示。当ARP（pl）小于等于8，Paging Priority的值等于ARP（pl）减1。当ARP（pl）大于8，Paging Priority的值等于7。

*表4 Paging Priority的取值与SGW-C下发的ARP（pl）值的对应关系*

| SGW-C下发的ARP中的pl值 | Paging Priority的取值 |
| --- | --- |
| 1 | 0 |
| 2 | 1 |
| 3 | 2 |
| 4 | 3 |
| 5 | 4 |
| 6 | 5 |
| 7 | 6 |
| 8 | 7 |
| 9~15 | 7 |

#### [MME对各类优先级业务的处理原则](#ZH-CN_TOPIC_0170010135)

MME根据识别的业务类型，设置下发给eNodeB的寻呼消息的“Paging Priority”。eNodeB根据“Paging Priority”差异化处理寻呼消息。“Paging Priority”数值越小，优先级越高。

MME对各种业务处理优先级的处理逻辑从高到低顺序如下：

- VoLTE紧急呼叫，“Paging Priority”固定为0。
- MPS/eMLPP业务，“Paging Priority”为6~0。
- CSFB/VoLTE语音被叫业务，“Paging Priority”固定为7。
