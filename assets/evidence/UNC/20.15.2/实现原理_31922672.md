# 实现原理

- [原理概述](#ZH-CN_CONCEPT_0231922672__1.3.1.1)

#### [原理概述](#ZH-CN_CONCEPT_0231922672)

**RADIUS功能特性的基本功能：**

- APN/AAA Server映射功能
  UNC 支持根据不同APN分别配置AAA Server，每个APN可以绑定一个Radius-server Group。
- AAA Server工作模式可配置
  UNC 支持通过 **[ADD RDSSVRGRP](../../../../../OM参考/命令/UNC MML命令/业务服务管理/会话管理/RADIUS管理/连接管理/Radius服务器组/增加Radius服务器组（ADD RDSSVRGRP）_09896730.md)** 命令配置Radius-server Group下的AAA Server的工作模式。不论哪种工作模式，每个Radius-server Group最多可配置16个主用和16个备用RADIUS鉴权服务器，每个Radius-server Group最多可配置16个主用和16个备用RADIUS计费服务器。
    - 主备模式：
          - 1+1主备：即一主一备，仅支持两个鉴权/计费服务器工作在主备模式。当配置1个以上主用服务器以及1个以上备用服务器时，仅一主一备服务器参与工作，其他服务器不参与。主AAA Server状态正常的情况下，所有RADIUS鉴权和计费消息都发送给主服务器，否则都发送给备服务器。
          - 按优先级顺序选择主服务器：通过配置多个主鉴权/计费服务器的优先级参数，实现按优先级顺序选择鉴权/计费主服务器。当配置1个以上主服务器时，仅设置优先级的服务器参与工作，备服务器以及无优先级的主服务器均不参与。发送RADIUS鉴权和计费报文时，优先发送给优先级高的主服务器，优先级高的主服务器故障时才发送给优先级低的主服务器。
    - 负荷分担模式：RADIUS鉴权和计费报文按照轮流选择的方式发送给多台AAA Server。该模式下的主备服务器都支持参与轮选，且按照一主一备的顺序进行轮选，无备用服务器时，则在主用服务器间进行轮选。详细内容请参见[WSFD-011308 AAA负荷分担](../WSFD-011308 AAA负荷分担/特性概述_14758811.md)。

UNC 支持如下功能：

- 与AAA Server之间通讯使用的端口号可配置
  支持AAA Server目的端口可配置， UNC 在鉴权和计费服务器的配置中，目的端口号可选，默认是1812、1813，可选范围是1～65535。这主要是为了兼容现网中AAA端口非1812、1813时的情况。
- 3GPP扩展属性支持
  支持3GPP 29.061协议中定义的扩展属性，可以和AAA Server交互这些属性值。
- 用户名域名剥离
  支持根据配置选择发往AAA的鉴权或计费消息中的用户名中是否带域名。用户鉴权或发送计费消息时，判断所在APN是否配置了剥离。如果所在APN配置为剥离，则将用户名中的域名剥离后发往AAA鉴权或计费服务器，即AAA收到的用户名中不带域名。如果配置为不剥离，则不做域名剥离操作。如果APN下没有配置，默认为不剥离。
- 计费和鉴权消息重发次数和超时的时间可配置
  支持RADIUS计费和鉴权消息重发次数和超时的时间可配置，可以让用户根据网络情况灵活决定。

**RADIUS鉴权包含如下功能点：**

- RADIUS方式分配MS IP地址
  移动用户所分配的IP地址是在PDP上下文激活阶段获得的地址（动态地址）。
  华为 UNC 支持通过RADIUS认证过程从AAA Server获取用户IP地址。
- Message-Authenticator验证
  为了确保鉴权消息的正确性和完整性、以及客户端的合法性，无论是RADIUS Client还是RADIUS Server，如果收到的交互消息中包含了Message-Authenticator属性，则需要对Message-Authenticator属性进行验证，即按协议规定重新计算Message-Authenticator的值、并和交互消息中的Message-Authenticator进行比较，看是否相同。如果数值不相同即验证不通过，则丢弃该消息。
- MSISDN等用户信息转发功能
  UNC 支持通过RADIUS消息向WAP GW等外部网元转发用户信息，包括MSISDN、IMSI、APN、用户IP地址等，从而可以让外部网元获取用户的标识信息。
- 多种鉴权方式
  UNC 支持PAP（Password Authentication Protocol）、CHAP（Challenge Handshake Authentication Protocol）、MS-CHAP（Microsoft Challenge Handshake Authentication Protocol）三种鉴权方式。如 [表1](#ZH-CN_CONCEPT_0231922672__table15328863496) 所示，在UE入网时，根据UE携带信元种类， UNC 支持对以下类型的用户进行鉴权：
  *表1 支持鉴权类型*

  | 信元 | 鉴权方式 | 支持鉴权的用户类型 |
  | --- | --- | --- |
  | PCO | PAP | 普通IP用户、PPP用户、L2TP用户 |
  | PCO | CHAP | 普通IP用户、PPP用户、L2TP用户 |
  | PCO | MS-CHAP（仅限MS-CHAPv1和MS-CHAPv2） | 普通IP用户 |
  | ePCO | PAP | L2TP用户 |
  | ePCO | CHAP | L2TP用户 |
- AAA Bypass
  针对RADIUS鉴权超时无响应和RADIUS计费服务器无响应， UNC 支持通过命令配置控制是否去活用户，见 [表2](#ZH-CN_CONCEPT_0231922672__tab1) 。
  *表2 Bypass处理策略*

  | 场景 | 相关命令和参数 | 动作类型 |
  | --- | --- | --- |
  | RADIUS鉴权超时无响应 | **[SET APNAUTHATTR](../../../../../OM参考/命令/UNC MML命令/业务服务管理/会话管理/接入管理/APN管理/APN鉴权属性/设置APN鉴权属性配置（SET APNAUTHATTR）_28567656.md)**<br>AAANORSPCTRL=AAA鉴权服务器无响应处理; | - **DEACTIVE**<br>：立即去活。<br>- “HOLDINGTIME”<br>为0：放通用户。<br>- “HOLDINGTIME”<br>为非0：在一段时间后去活。 |
  | RADIUS计费服务器无响应 | **[SET APNRDSACCTCTRL](../../../../../OM参考/命令/UNC MML命令/业务服务管理/会话管理/RADIUS管理/RADIUS计费管理/APN计费控制/设置APN RADIUS计费控制参数（SET APNRDSACCTCTRL）_09896770.md)**<br>DEACTIVE=计费服务器请求超时是否去活用户; | - **CONTINUE**<br>：放通用户。<br>- **DEACTIVE**<br>：立即去活。<br>- “SUPPORTACCTRSP”<br>为<br>**ENABLE**<br>：在PDP上下文激活的过程中，需要等待计费开始响应消息才能回应激活成功应答消息，如果等待超时则PDP上下文激活失败。<br>- “SUPPORTACCTRSP”<br>为<br>**DISABLE**<br>：在激活PDP上下文的过程中，不需要等待计费开始响应消息，直接回应激活成功应答，如果等待响应超时，则根据Deactive判断是否去活上下文。 |
  | RADIUS服务器超时无响应紧急处理<br>说明：采用放通用户的动作类型时，用户地址分配方式不能是RADIUS分配。 | **[SET FHBYPASS](../../../../../OM参考/命令/UNC MML命令/业务服务管理/接口管理/计费和策略接口管理/失败旁路处理/设置失败旁路处理（SET FHBYPASS）_09896714.md)**<br>说明：- 本命令仅用于故障场景下的紧急处理，优先级高于**[SET APNAUTHATTR](../../../../../OM参考/命令/UNC MML命令/业务服务管理/会话管理/接入管理/APN管理/APN鉴权属性/设置APN鉴权属性配置（SET APNAUTHATTR）_28567656.md)**:AAANORSPCTRL和**[SET APNRDSACCTCTRL](../../../../../OM参考/命令/UNC MML命令/业务服务管理/会话管理/RADIUS管理/RADIUS计费管理/APN计费控制/设置APN RADIUS计费控制参数（SET APNRDSACCTCTRL）_09896770.md)**:DEACTIVE的Bypass配置。<br>- 由于该命令的配置将影响用户的激活状态，只有在获得了客户的书面认可后方可使用。 | - “HOLDINGTIME”<br>为0：放通用户。<br>- “HOLDINGTIME”<br>为非0：在一段时间后去活。 |
  动作类型说明：

  - 立即去活。
      当本地配置命令参数设置为 **DEACTIVE** 时，则不允许用户继续激活，网关立即去活用户。
    - 放通用户（不被去活，直到用户主动下线）。
      当本地配置命令参数 **CONTINUE** 或 “HOLDINGTIME” 设置为0时（ “HOLDINGTIME” 的取值范围为0~1440），则用户仍然可以激活成功并保持在线，直到用户主动下线。
    - 在一段时间后去活。
      当本地配置命令参数 “HOLDINGTIME” 设置为非0时（ “HOLDINGTIME” 的取值范围为0~1440），则用户仍然可以激活成功并保持在线，直到 “HOLDINGTIME” 时长超时后网关主动发起用户去活。
      为防止AAA故障后经过 “HOLDINGTIME” 时长后，瞬时出现大批量用户集中去活，而引发的信令激增导致网络设备负荷较高的情况，可以通过配置 **ADJUSTRANGE** 参数，使得用户在 **ADJUSTRANGE** 时长范围内随机延迟用户去活时间，使得用户可以离散的去活，避免了集中去活而引发的信令激增问题。
- RADIUS鉴权响应消息端口默认检查
  UNC 支持默认检查AAA Server返回的Access Accept、Access Reject消息的源端口，可以通过 **[SET RDSRSPADDRCHK](../../../../../OM参考/命令/UNC MML命令/业务服务管理/会话管理/RADIUS管理/连接管理/RADIUS响应消息地址检查/设置全局RADIUS响应消息源IP_端口检查配置（SET RDSRSPADDRCHK）_09896744.md)** 命令进行配置。

**RADIUS计费包含如下功能点：**

- RADIUS非实时计费
  UNC 可以和外部数据网的RADIUS服务器交互完成RADIUS计费流程。通过这个功能，可以实现由移动运营商和Internet运营商分别收取费用。RADIUS计费功能是可以根据不同APN分别配置。
- RADIUS准实时计费
  UNC 支持RADIUS准实时计费，根据用户的上网时间和流量门限触发，通过Accounting Request Interim-Update消息发送到AAA Server。
- 计费响应开关
  UNC 支持计费请求消息响应开关。通过该开关可以控制没有收到Accounting Start Response消息时是否去激活PDP，此功能在一些上层应用实体利用计费消息获得用户信息时非常有用。例如：WAP Gateway通过Accounting Start消息获得用户信息，而又不回应Accounting Start Response消息。
  当计费响应功能关闭，并且计费服务器没有响应时，先激活用户，后去激活用户。在用户接入高峰时，会频繁出现激活、去激活用户的操作。为了避免这种情况， UNC 提供计费消息流控功能，通过设定计费消息阈值，周期内未超过阈值先激活用户，后去激活用户，超过阈值后直接去激活用户。
- Account ON/OFF消息
  UNC 支持基于APN发送Account ON/OFF消息，用于 UNC 整机重启场景下，通知AAA快速批量清除残留用户。
- 利用RADIUS鉴权服务器计费
  目前，在一些运营商中，AAA鉴权服务器分配地址需要计费消息accounting确认，且只有收到对应用户的accounting stop消息才会释放地址。利用RADIUS鉴权服务器计费正是为这一应用而产生的。
  利用鉴权服务器计费特性，目的是通过将RADIUS计费消息发送给AAA鉴权服务器以协助AAA鉴权服务器分配、回收地址。
  在需要AAA鉴权的前提下，当 UNC 收到AAA鉴权服务器的鉴权响应时， UNC 针对用户记录该鉴权服务器；当用户激活、更新、去激活等原因诱发 UNC 发送计费消息时， UNC 根据记录将计费消息发往用户的鉴权服务器。
  该特性还支持以下具体场景应用：

  - 当用户激活二次上下文时，不进行鉴权，但是当二次上下文激活、更新、去激活时，需要向用户的鉴权响应服务器发送计费消息。
    - 在部署计费抄送服务器时，UNC将计费消息发送给鉴权服务器同时，并抄送给计费抄送服务器。
    - 支持UNC的PBU主备倒换，特性功能不受影响。
    - UNC给鉴权服务器发送计费消息时，不关心鉴权服务器的状态。当用户在线期间，鉴权服务器发生倒换，新激活的用户会将计费消息发送给倒换后的鉴权服务器处理。
    - 用户接入需要鉴权的前提下，才可以使用该特性；如果不需要鉴权，则不能使用该特性。
- RADIUS计费响应消息端口默认检查
  UNC 支持默认检查AAA Server返回的Accounting Response消息的源端口，可以通过 **[SET RDSRSPADDRCHK](../../../../../OM参考/命令/UNC MML命令/业务服务管理/会话管理/RADIUS管理/连接管理/RADIUS响应消息地址检查/设置全局RADIUS响应消息源IP_端口检查配置（SET RDSRSPADDRCHK）_09896744.md)** 命令进行配置。

- 基于PGW-U/UPF主送RADIUS计费/鉴权消息
  UNC 支持根据用户所锚定的PGW-U/UPF给特定的主送服务器发送AAA消息。可以通过 [**ADD UPLIST4RDS**](../../../../../OM参考/命令/UNC MML命令/业务服务管理/会话管理/RADIUS管理/连接管理/UP列表/向RADIUS服务器使用的UP列表中增加UP（ADD UPLIST4RDS）_52749060.md) 配置PGW-U/UPF List，通过 **[ADD RDSSVR](../../../../../OM参考/命令/UNC MML命令/业务服务管理/会话管理/RADIUS管理/连接管理/Radius服务器/增加RADIUS服务器（ADD RDSSVR）_09896756.md)** 配置主送服务器绑定PGW-U/UPF List，实现将多个PGW-U/UPF host绑定到主送服务器上。 UNC 根据用户当前所在PGW-U/UPF host匹配主送服务器，然后将计费/鉴权消息发送给匹配的主送服务器。当PGW-U/UPF绑定的AAA服务器全部故障时，通过在用户面控制是否将PGW-U/UPF状态置为异常，从而使新用户激活不会选择到该PGW-U/UPF上。
  UNC 也支持RADIUS计费多UPF场景的处理。在多UPF场景下，只处理主锚点UPF的上报处理事件，辅锚点的UPF上报只做流量累加处理，不做计费事件处理。

**RADIUS策略控制包含如下功能点：**

- 从AAA获得用户和业务属性
  用户属性是指用户的预付费及后付费属性；业务属性指用户的签约业务，当用户进行业务时， UNC 识别并限制用户只能访问业务列表中的业务。用户在签约时需要签约一个业务列表，只能访问业务列表中的业务。
  在用户激活时，AAA将用户属性和业务策略下发给 UNC ，并由 UNC 传递给PGW-U/UPF，PGW-U/UPF根据外部实体下发的信息对业务进行控制和处理。外部实体主要是AAA Server。当AAA Server失效时，可以使用本地的业务策略。业务策略优先级顺序从高到低依次是：AAA服务器/本地APN/本地GLOBAL。
- AAA下发策略
  目前，对于一些运营商来说，部署PCRF实现PCC功能成本较高，且需要更改网络部署新增网元相对较复杂，因此一些运营商需要通过扩展AAA鉴权响应消息来实现策略下发功能。
  UNC 支持以下场景下采用AAA下发策略规则进行计费策略控制：

  - 无PCC应用
    - 某PDP的PCC功能通过AAA鉴权响应消息指示不使能
    - PCRF状态异常
  如果网络已部署PCRF且PCC功能正常时， UNC 则将AAA下发的策略转发给PCRF做参考，由PCRF决策后，向 UNC 发送新的策略。
  UNC 本地可以配置User-profile、Rule。AAA下发的Charging-rule-base-name，对应本地配置的User-profile策略名称；AAA下发的charging-rule-name对应本地配置的rule策略名称。 UNC 根据下发的Charging-rule-base-name和charging-rule-name激活本地配置的预定义规则。
  AAA下发的策略目前只支持预定义规则，不支持动态规则。
  如果对于同一个会话，同时存在AAA下发的策略与本地配置的策略，则以AAA下发的策略为准。如果AAA没有下发策略，则以本地配置为准。
  如果同时下发多个User-profile，使用第一个User-profile下绑定的策略。
- RAT触发属性更新
  RAT可以用于标识用户接入网络。通过在 UNC 上的配置，可以控制RAT改变时是否触发Accouting Interim Update Message，用于通知其他网元用户接入网络的变化。
- 基于RADIUS POD（Packet of Disconnect）消息去激活上下文
  POD（Packet of Disconnect）消息是3GPP协议定义的，由AAA Server主动向设备发起的RADIUS消息，用于去激活用户。
  UNC 支持根据POD消息去激活上下文。支持对POD消息作合法性检查，只接收配置Server IP的POD消息。
  可以根据Disconnect-request消息中携带的Acct-Session-Id和3GPP-Teardown-Indicator去激活用户；为支持不兼容3GPP扩展属性的AAA Server下发POD消息，可以根据 UNC 的接入侧接口IP地址和Charging ID去激活用户。
- 支持业务触发RADIUS消息
  UNC 支持在用户激活过程中或者进行特定业务时给AAA Server发送RADIUS消息，以便使得AAA Server可以通过RADIUS消息获取到用户信息。（通过 **[SET APNRDSACCTCTRL](../../../../../OM参考/命令/UNC MML命令/业务服务管理/会话管理/RADIUS管理/RADIUS计费管理/APN计费控制/设置APN RADIUS计费控制参数（SET APNRDSACCTCTRL）_09896770.md)** 命令配置此功能。）
    - 用户激活过程中发送RADIUS消息
      UNC 在用户激活过程中给AAA Server发送RADIUS Accounting-Request Start消息，消息发送后并不等待AAA Server返回应答消息，只是将用户信息通过RADIUS Accounting-Request Start消息携带给AAA Server。并且此RADIUS Accounting-Request Start消息不对用户激活过程产生任何影响。
    - 访问特定业务时发送RADIUS消息
      UNC 在激活过程中不发送RADIUS Accounting-Request Start消息，只有当用户访问了特定的业务时， UNC 会根据L3/4层的业务匹配规则判断出需要将访问此业务的用户信息发送给AAA Server。此时 UNC 发送RADIUS Accounting-Request Start消息，携带用户信息发送给AAA Server。并且支持通过 **[SET APNRDSACCTCTRL](../../../../../OM参考/命令/UNC MML命令/业务服务管理/会话管理/RADIUS管理/RADIUS计费管理/APN计费控制/设置APN RADIUS计费控制参数（SET APNRDSACCTCTRL）_09896770.md)** 命令的 “BUFFERTIMEVALUE” 参数配置等待RADIUS响应消息的时长。
