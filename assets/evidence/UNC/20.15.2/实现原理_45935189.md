**相关概念**

- GUTI（Globally Unique Temporary UE Identity ）：MME分配，全局唯一，包含两个部分。
    - GUMMEI：GUMMEI由MCC、MNC、MME标识组成（为便于互操作时区分接入制式，将MMEGI的高比特位规划为1）。
    - M-TMSI：M-TMSI长度为32个bit，在一个MME内唯一标识一个UE。
      GUTI可以在附着或跟踪区更新过程中隐式分配，也可以通过GUTI重分配过程显式分配。
- TAI（Tracking Area Identity）：TAI=MCC(Mobile Country Code)+MNC(Mobile Network Code)+TAC(Tracking Area Code)。
- TA List：如[图1](#ZH-CN_TOPIC_0245935189__fig2)所示，将若干跟踪区TA组合为一个TA List。
    - 将若干跟踪区TA组合为一个TA List，移动终端在此TA List包含的所有TA区中移动时，不再需要发起TA更新。
    - 网络寻呼用户时，在整个TA List包含的所有TA区中寻呼用户。
    - MME在Attach Accept，TAU Accept或者GUTI Reallocation Command消息中会将TA List下发给UE。

**负载均衡原理**

在MME Pool的组网中，由于一个eNodeB与MME Pool内的多个MME连接，所以当TA业务区中的UE发起新的业务时，为了使eNodeB可以根据负载均衡等原则为UE从Pool中选择一个服务的MME，MME需要将负荷权重下发给eNodeB。

如 [图2](#ZH-CN_TOPIC_0245935189__fig3) 所示，假设MME1的有效用户容量为4，MME2的有效用户容量为3，则eNodeB会按4∶3的比例，将UE接入到MME1和MME2上，具体分发效果取决于eNodeB的实现方式。

**图2** 负载均衡原理图

<br>

![](实现原理_45935189.assets/zh-cn_image_0245935219_2.png)

- MME通知eNodeB自身的负荷权重（S1AP连接建立时）
  在MME Pool中，一个eNodeB与MME Pool中的多个MME连接，所以当建立专用S1连接时，需要为UE选择一个服务的MME，这里的选择功能被称为“节点选择功能”。eNodeB是根据各MME对应的“负荷权重”比例进行选择，MME的负荷权重值是通过S1AP消息发送给池区中的eNodeB的。MME的负荷权重值是根据Pool中其它MME节点的可用容量加以设置的。负荷权重值可以通过S1 Setup Response消息（ [图3](#ZH-CN_TOPIC_0245935189__fig4) ）或者MME Configuration Update消息携带给eNodeB（ [图4](#ZH-CN_TOPIC_0245935189__fig5) ）。
  **图3** 在S1设备连接建立时MME通知eNodeB自己的负荷权重流程图

  <br>

  ![](实现原理_45935189.assets/zh-cn_image_0245935224_2.png)
    1. 用户根据MME Pool中各MME的可用容量的比例关系分别配置各MME的负荷权重，取值范围为：0～255。
    2. MME保存各自的负荷权重。
    3. eNodeB向MME发起SCTP偶联建立请求并建立成功。
    4. eNodeB向MME发送S1 Setup Req以建立S1接口设备连接。
    5. MME向eNodeB发送S1 Setup Rsp消息将MME的负荷权重(Relative MME Capacity)带给eNodeB。
    6. eNodeB保存MME的负荷权重信息。
- MME通知eNodeB自身的负荷权重（S1AP连接建立后用户修改配置）
  MME的负荷权重值可以进行修改，但不会频繁更新。对于一个相对成熟的网络，建议一个月更新一次即可，如在新增了RAN侧或CN侧网元时进行更新。
  同时，当MME容量变更的情况下，也可能需要负荷权重值的变更。如对于一个Pool中新增的网元，可以设置一个相对较高的负荷权重值，以便该新增网元尽快承接业务，达到与Pool内其它网元的负荷均衡。
  **图4** 在S1设备连接建立后MME通知eNodeB自己的负荷权重流程图

  <br>

  ![](实现原理_45935189.assets/zh-cn_image_0245935226_2.png)
    1. 用户修改MME的负荷权重，取值范围为：0～255。
    2. MME保存修改后的负荷权重。
    3. MME主动向所有连接的eNodeB发送MME Configuration Update，其中携带标识该MME负荷权重的Relative MME Capacity信元。
    4. eNodeB更新MME的负荷权重，作为后续UE接入时选择MME的依据。
    5. eNodeB若成功更新数据配置，则回复MME Configuration Update Ack消息给MME。

**Pool区内TAU原理**

由于MME Pool Area内的任意一个TA均被MME Pool内所有的MME所服务，所以UE业务过程中在MME Pool Area内移动时，只进行INTRA TAU，不进行INTER TAU（与传统组网相比，这可以减少INTER TAU，从而减少S6a和S10接口消息，减少丢包，从而提高服务质量）。

**容灾原理**

实际上，负荷分担处理已经使MME Pool组网具备了一定的MME级的容灾能力。当MME Pool中某MME发生故障时，eNodeB识别该MME故障后，eNodeB会将本来由该MME提供服务的用户发起的新业务转移到Pool中其他有效的MME上；同时对新接入的用户进行MME选择时不会再考虑该MME，从而实现MME Pool内MME间的容灾。

如 [图5](#ZH-CN_TOPIC_0245935189__fig6) 所示，在非Pool组网情况下，某UE通过eNodeB2接到MME1上，如果MME1故障将导致eNodeB2到MME1的链路断开，此时UE的TAU流程失败。

**图5** 非Pool组网情况下故障示意图

<br>

![](实现原理_45935189.assets/zh-cn_image_0245935230_2.png)

如 [图6](#ZH-CN_TOPIC_0245935189__fig7) 所示，在Pool组网情况下，规划图中的eNodeB分别连接到两个MME上，UE通过eNodeB2连接到MME1上，在MME1故障以致eNodeB2到MME1的链路断开的情况下，UE在进行TAU的时候流程时可以通过eNodeB2将消息发到MME2上，由MME2提供业务。（如果只是eNodeB2到MME1的链路断开，MME1设备可用，则MME2收到TAU消息后，可以进行INTER TAU流程；如果MME1设备彻底故障，TAU消息直接被拒绝，UE会通过MME2重新发起附着流程，这个时候eNodeB2会将附着消息发到MME2上，即相当于UE通过eNodeB2附着到MME2，由MME2提供业务。）

> **说明**
> 当MME宕机后，如果有下行数据或信令，在一个TAU周期内，（如果UE没有主动向网络侧发起注册）则可能会无法对用户成功进行寻呼。如果部署了语音业务（包括VoLTE、CSFB），则可能在一个TAU周期内无法成功进行被叫。

**业务流程**

- Attach流程的容灾处理
  **图7** Pool区域内Attach流程的容灾处理消息流程

  <br>

  ![](实现原理_45935189.assets/zh-cn_image_0245935242_2.png)
  流程说明
  MME Pool区域容灾场景下，对于MME Pool区域内关机然后再开机的用户，eNodeB通过用户所带GUMMEI找对应的MME，但当GUMMEI对应MME的S1接口故障触发Attach流程。
  eNodeB基于负载均衡原则，在MME Pool区域内选择一个S1接口正常的MME，而这个MME不是用户上次附着的MME，且用户原附着的MME正常，则触发了Pool区域内Inter Attach流程，如 [图7](#ZH-CN_TOPIC_0245935189__fig12419163119451) 所示，但不包含图中向UE发Identity Request的过程。若用户原附着的MME故障，则Attach流程如 [图7](#ZH-CN_TOPIC_0245935189__fig12419163119451) 所示，但不包含图中向Old MME发Identification Request的过程。
- TAU流程的容灾处理
  **图8** Pool区域内TAU流程的容灾处理消息流程

  <br>

  ![](实现原理_45935189.assets/zh-cn_image_0245935244_2.png)
  流程说明
  MME Pool容灾场景下，用户发的TAU请求消息中携带了GUMMEI，但是eNodeB到GUMMEI对应MME的S1接口故障触发TAU流程。
  eNodeB发现UE附着的MME故障，将TAU Request消息基于负载均衡的原则发送到Pool区域内其他状态正常的MME上。若用户原附着的MME正常，则触发了MME Pool区域内Inter TAU流程，如 [图8](#ZH-CN_TOPIC_0245935189__fig1239872054812) 所示。若用户原来附着的MME故障，则New MME回TAU Reject，用户重新发起附着接入新的MME。
- Service Request流程的容灾处理
  **图9** Pool区域内Service Request流程的容灾处理消息流程

  <br>

  ![](实现原理_45935189.assets/zh-cn_image_0245935249_2.png)
  流程说明
  Pool容灾场景下，用户发的Service请求消息中携带了GUMMEI，但是eNodeB到GUMMEI对应MME的S1接口故障触发重新发起附着。
    1. Service Request消息
      eNodeB发现UE附着的MME故障，将Service Request消息基于负载均衡的原则发送到Pool区域内其他状态正常的MME上。
    2. Service Reject消息
      New MME没有用户上下文，向UE回Service Reject。
    3. Attach Request消息
      UE发起附着请求，eNodeB发现UE附着的MME故障，将Attach Request消息基于负载均衡的原则发送到Pool区域内其他状态正常的MME上。
  其他流程描述与非MME Pool组网场景相同，请参考3GPP 23.401。
