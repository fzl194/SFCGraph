# 移动性接入限制信息更新

如果在上述流程中，UDM签约数据发生变更，处理策略如 [表1](#ZH-CN_TOPIC_0248748811__table1848318480571) 。

*表1 UDM签约数据变更场景处理策略*

| **变更字段** | **变更方式** | **处理**<br>策略 |
| --- | --- | --- |
| **RAT限制（RAT restriction ）** | NR接入：不限制修改为限制NR | 对于连接态用户立即去注册。<br>对于空闲态用户，如果用户不可达，则AMF通过隐式分离方式删除用户；如果用户可达，则AMF先寻呼用户，再通过显式分离方式删除用户。 |
| **RAT限制（RAT restriction ）** | NR接入：限制NR修改为不限制 | AMF无相应处理；<br>依赖UE通过开关机、插拔USIM卡等触发重新注册到5GC。 |
| **RAT限制（RAT restriction ）** | NR接入：限制NR修改为限制NR_REDCAP | AMF无相应处理；<br>限制NR已经包括了NR_REDCAP。 |
| **RAT限制（RAT restriction ）** | NR接入：限制NR_REDCAP修改为限制NR | 对于连接态普通NR用户立即去注册。<br>对于空闲态普通NR用户，如果用户不可达，则AMF通过隐式分离方式删除用户；如果用户可达，则AMF先寻呼用户，再通过显式分离方式删除用户。 |
| **RAT限制（RAT restriction ）** | NR_REDCAP接入：限制NR_REDCAP用户修改为不限制 | AMF无特殊处理。<br>依赖UE通过开关机、插拔USIM卡等触发重新注册到5GC。 |
| **RAT限制（RAT restriction ）** | NR_REDCAP接入：不限制修改为限制NR_REDCAP接入 | 对于连接态RedCap用户立即去注册。<br>对于空闲态RedCap用户，如果用户不可达，则AMF通过隐式分离方式删除用户；如果用户可达，则AMF先寻呼用户，再通过显式分离方式删除用户。 |
| **RAT限制（RAT restriction ）** | NR_REDCAP用户接入：不限制到限制NR | 对于连接态RedCap用户立即去注册。<br>对于空闲态RedCap用户，如果用户不可达，则AMF通过隐式分离方式删除用户；如果用户可达，则AMF先寻呼RedCap用户，再通过显式分离方式删除用户。 |
| **RAT限制（RAT restriction ）** | NR_REDCAP用户接入：限制NR_REDCAP用户变为限制NR | AMF无特殊处理。<br>依赖UE通过开关机、插拔USIM卡等触发重新注册到5GC。 |
| **核心网限制（Core Network type restriction）** | 5GC：不限制修改为限制 | 对于连接态用户立即去注册。<br>对于空闲态用户，如果用户不可达，则AMF通过隐式分离方式删除用户；如果用户可达，则AMF先寻呼用户，再通过显示分离方式删除用户。 |
| **核心网限制（Core Network type restriction）** | 5GC：限制修改为不限制 | AMF无相应处理；<br>依赖UE通过开关机、插拔USIM卡等触发重新注册到5GC。 |

如果在上述流程中，AMF的本地配置的移动性接入限制信息发生变化，会在下一次移动性注册流程生效。
