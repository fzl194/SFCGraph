#### [网络切片选择概述](#ZH-CN_TOPIC_0191042355)

注册流程中，NG-RAN首先为UE选择初始AMF。当初始AMF的配置为本地决策切片时，则在如下两种场景下，初始AMF可以继续处理注册流程，为UE决策Allowed NSSAI、Configured NSSAI：

- UE请求消息中携带Requested NSSAI，初始AMF支持UE请求的Requested NSSAI和签约S-NSSAI的交集切片。
- UE未携带Requested NSSAI，但初始AMF支持所有被标记为default的签约S-NSSAI。

如果初始AMF的配置希望通过NSSF来决策或配置为本地决策但无法支持UE需要的网络切片时，初始AMF向NSSF发起切片查询，通过NSSF重选目标AMF，由目标AMF继续为UE处理注册流程。若AMF不能成功查找到NSSF，且初始AMF不支持全部或部分UE需要的切片时，会携带为UE分配的Allowed NSSAI、当前TAI请求目标AMF，并触发AMF重选。

NSSF和AMF在切片选择流程中的作用分别如下：

- AMF：当NG-RAN为UE选择的初始AMF无法为UE需要的网络切片服务时，初始AMF向NSSF发起切片查询，通过NSSF重选目标AMF。若需要由AMF自行决策切片，AMF支持配置TA级限制切片。AMF支持配置在拒绝注册时下发Rejected S-NSSAI给UE的能力，若NSSF或初始AMF查询不到可用AMF且初始AMF不能为UE提供切片服务时，会拒绝UE的注册请求，并在Registration Reject消息中携带Rejected S-NSSAI信元 。
- NSSF：预先配置所管理的AMF支持的切片列表，以及本网PLMN支持的切片配置信息。切片选择查询时返回可以提供服务的AMF SET或Candidate AMFs，以及决策下发给UE的切片配置信息。

**图2** 初始注册流程

<br>

![](实现原理_91042355.assets/zh-cn_image_0000001579286228_2.png)

1. 用户发起注册流程，请求消息携带用户的标识(SUCI、5G-GUTI or PEI)、Requested NSSAI等信元，(R)AN根据UE携带的AN信息(Requested NSSAI、5G-S-TMSI)结合相关策略选择初始AMF，并将注册请求发送给初始AMF。
2. 初始AMF获取用户的签约信息，其中包括用户的签约S-NSSAI。AMF首先根据配置决定是否由本AMF决策切片
    a. 若配置为自行决策或没有相关配置：则AMF判断是否可以同时支持UE请求的Requested NSSAI和签约S-NSSAIs交集的切片集合，若UE未携带Requested NSSAI，则AMF判断是否支持所有标记为default的签约S-NSSAI。如果初始AMF可以全部支持，则继续后续注册流程的处理。如果无法全部支持且本地没有可用NSSF的缓存信息时则AMF根据本地配置的NRF地址，通过服务化接口向NRF发送NF Discovery请求消息查询获取NSSF信息。若获取NSSF信息失败，则AMF会判断能否部分支持UE请求的Requested NSSAI和签约S-NSSAIs交集的切片集合，如果可以则由本AMF继续提共后续注册流程的处理，否则将由初始AMF向NRF查询目标AMF。若由AMF决策切片，则AMF根据TA级区域限制的相关配置进行切片决策并下发给UE。
    b. 若没有配置为自行决策：则AMF会直接通过服务化接口向NRF发送NF Discovery请求消息查询获取NSSF信息，并由NSSF来进行网络切片的分配，后续流程同a。
3. AMF根据NRF返回的可用NSSF信息以及AMF和NSSF的通信状态，选择一个可用的NSSF，并向选择的NSSF发送切片选择请求消息Nnssf_NSSelection_Get，消息携带NF Type、NF Id、Requested NSSAI（该信元为可选信元，如果UE请求没有携带Requested NSSAI则不需要携带）、签约S-NSSAIs、TAI等信元。
4. NSSF根据AMF请求消息携带的信息以及NSSF配置的切片信息，生成该用户允许接入的Allowed S-NSSAIs列表，然后根据本地的切片配置信息，查询可以为UE提供切片服务的AMF信息，可以是AMF SET或Candidate AMF List。
5. NSSF返回切片选择响应消息Nnssf_NSSelection_Get Response，消息携带NSSF分配的Allowed NSSAI和Configured NSSAI、Target AMF SET或Candidate AMF List等信息。如果AMF采用通过RAN间接重路由的方式则NSSF必须返回AMF SET。
6. 初始AMF收到NSSF返回的切片查询消息后进行AMF重路由选择，可以通过RAN间接重路由或者AMF间直接重路由方式，需要根据RAN的支持能力选择重路由方式。如果选择间接重路由则需要NSSF必须返回Target AMF SET信元；如果选择直接重路由方式，AMF需要根据Target AMF SET或Candidate AMF List查询NRF获取目标AMF的IP地址。
7. 重路由到目标AMF后，目标AMF继续完成Registration注册流程。若NSSF或初始AMF查询不到可用AMF且初始AMF不能为UE提供切片服务时，会拒绝UE的注册请求，并在Registration Reject消息中携带Rejected S-NSSAI信元 。

**图3** 签约数据更新流程

<br>

![](实现原理_91042355.assets/zh-cn_image_0000001579446048_2.png)

1. UE切片签约数据发生变更时，包括UE新增签约新的S-NSSAI以及删除已签约的S-NSSAI，UDM需要将更新的切片签约数据通知给UE注册的AMF。
2. 切片签约数据变更后，网络侧需要为UE重新生成Allowed NSSAI和Configured NSSAI切片信息。
3. 根据运营商策略/配置，如果AMF无法决策生成切片信息，则发送Nnssf_NSSelection_Get查询NSSF获取Allowed NSSAI和Configured NSSAI切片信息。消息至少需要包含AMF保存的UE最后接入时的Requested NSSAI、更新后的签约S-NSSAIs信息，以及用户的SUCI PLMN ID。
4. NSSF根据配置的PLMN的切片数据以及切片查询请求信息，重新生成用户的Allowed NSSAI、Configured NSSAI。
5. NSSF将新生成的Allowed NSSAI、Configured NSSAI返回给AMF。因为签约数据的变更，可能当前AMF不一定能继续提供切片服务，但在签约数据变更场景，AMF无需进行Reroute重选流程。AMF将重新生成的Allowed NSSAI和Configured NSSAI，通过UE Configuration Update Command流程通知给UE，并指示UE进行注册流程。如果AMF可以继续提供切片服务，则无需指示UE进行注册流程。

6. AMF根据收到的切片配置信息，保存并判断如果切片相关信息发生改变则通知相关UE。
7. AMF通知UE更新切片相关配置信息，对于连接态用户，AMF直接发送UE Configuration Update Command通知UE；对于空闲态用户，AMF根据实现策略决定触发网络侧Paging后通知UE，或者等UE变成连接态以后再通知UE。
8. UE收到消息后，更新本地保存的信息，回复UE Configuration Update Complete响应消息确认完成更新。

#### [Configured NSSAI分配](#ZH-CN_TOPIC_0191042355)

**Configured NSSAI**

Configured NSSAI是当前为UE服务的运营商（Serving PLMN）为其配置的网络切片，这里Serving PLMN既可以是归属运营商（Home PLMN），也可以是拜访运营商（Visited PLMN）。UE会在本地保存Configured NSSAI并在发送Requested NSSAI信元时从中挑选切片，与Allowed NSSAI使用方法类似，但Configured NSSAI比Allowed NSSAI范围更广，没有接入类型和区域的属性，表示整PLMN的切片能力。

**Default Configured NSSAI**

Default Configured NSSAI是Configured NSSAI的一种缺省配置，区别于普通Configured NSSAI，它可以应用于任何没有提供Configured NSSAI和Allowed NSSAI的PLMN，可以理解为一个最基本的NSSAI，且不区分PLMN。Default Configured NSSAI是由UE归属运营商的UDM通过UE Parameters Update流程配置给UE的Configured NSSAI，可以理解为一个可以在各运营商网络间通用的缺省切片集。所以Default Configured NSSAI一般是协议定义的标准S-NSSAI，否则在多个运营商之间比较难协调（因为涉及不同运营商间的网络切片映射）。

**Configured NSSAI的分配**

当前5G核心网为UE分配或者重新分配Configured NSSAI的场景如下：

- 如果Requested NSSAI为空，则表示UE本地无当前Serving PLMN的Configured NSSAI，5GC需要为UE分配Configured NSSAI。
- 如果Requested NSSAI非空，但是Requested NSSAI是从Default Configured NSSAI中摘录的（可通过注册请求中的Network slicing indication参数识别），则也能证明UE本地无当前Serving PLMN的Configured NSSAI，5GC需要为UE分配Configured NSSAI。
- 如果Requested NSSAI非空，但是Requested NSSAI中存在某个或某几个网络切片超过了当前Serving PLMN规划的或者UE签约的网络切片范围，则表示UE本地维护的该Serving PLMN下的网络切片与网侧不一致，那么5GC也需要通过重分配Configured NSSAI刷新UE本地配置。
- Inter AMF场景和4G网络到5G网络的互操作场景，网络切片需要在切换运营商/地区导致PLMN变化或切换网络制式时重新分配。

针对上述第四种场景，存在区域部署切片无法在当前PLMN的全网AMF、NSSF进行相同配置的问题。举例来说，此问题会导致已签约S-NSSAI 1、2的用户从区域A（下发Configured NSSAI为S-NSSAI 1）漫游到专网区域B（下发Configured NSSAI为S-NSSAI 1、2）时，无法正常使用切片S-NSSAI 2。我们可以通过 **[SET NSSFFUNCSWITCH](../../../../../OM参考/命令/UNC MML命令/业务服务管理/NSSF业务及策略管理/NSSF功能开关配置/设置NSSF功能开关（SET NSSFFUNCSWITCH）_96243184.md)** 和 ****[SET NGMMFUNC](../../../../../OM参考/命令/UNC MML命令/业务服务管理/5G接入业务管理/移动性管理/MM协议参数管理/5G移动性管理/设置5G移动性管理功能（SET NGMMFUNC）_09653748.md)**** 去配置网络侧按签约分配Configured切片，保证区域A的网络侧依旧可以分配到S-NSSAI 2。
