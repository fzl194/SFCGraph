# 计费会话创建流程

- [业务场景](#ZH-CN_TOPIC_0278888683__1.3.1.1)
- [信令流程](#ZH-CN_TOPIC_0278888683__1.3.2.1)

#### [业务场景](#ZH-CN_TOPIC_0278888683)

PDU会话建立过程中建立计费会话，SMF发送消息到CHF，并为指定的RG申请配额。计费会话创建流程在以下两种情况下存在不同：

- UPF分配用户IP地址时的融合计费会话创建流程。
- SMF分配用户IP地址时的融合计费会话创建流程。

#### [信令流程](#ZH-CN_TOPIC_0278888683)

融合计费会话创建流程有如下几种：

- [UPF分配用户IP地址时的融合计费会话创建流程](#ZH-CN_TOPIC_0278888683__p101939573216)
- [SMF分配用户IP地址时的融合计费会话创建流程](#ZH-CN_TOPIC_0278888683__p9891113612513)

**UPF分配用户IP地址时，融合计费会话创建流程** 如 [图1](#ZH-CN_TOPIC_0278888683__fig1010920514215) 所示。

![](计费会话创建流程_01_10001.assets/zh-cn_image_0278888687_2.png)

1. UE发起PDU会话建立请求流程，携带PDU Session ID、PDU Type、SSC mode、用户位置信息和接入类型信息等。
2. SMF判断需要向PCF获取PCC规则和计费策略，向PCF发送[Npcf_SMPolicyControl_Create](../../../../../../快速入门/协议与流程/5G Core信令分析/消息及信元/服务化接口协议/Npcf/N7/Npcf_SMPolicyControl_Create_67572324.md)Request消息，请求创建SM策略关联，消息中携带pduSessionId、pduSessionType、dnn、notificationUri和Snssai等信息。若网络中未部署PCF时，SMF使用本地配置的PCC规则和计费策略。
3. PCF根据[Npcf_SMPolicyControl_Create](../../../../../../快速入门/协议与流程/5G Core信令分析/消息及信元/服务化接口协议/Npcf/N7/Npcf_SMPolicyControl_Create_67572324.md)Request请求消息中携带的用户标识，在本地的用户签约信息中查询到该用户签约的业务关联了Sy配额（通过在UPCF Web UI界面上将配额参数“配额管理归属”配置为“Sy”），会向CHF发送Nchf_SpendingLimitControl_Subscribe-Initial消息请求建立N28会话，获取该用户的用量计数信息。其中，PCF通过从NRF查询来选择CHF。
4. CHF创建N28会话，根据用户标识supi查询用户的用量计算信息，并向PCF返回Nchf_SpendingLimitControl_Subscribe-Initial响应消息。其中，Nchf_SpendingLimitControl_Subscribe-Initial响应消息中通过如[表1](#ZH-CN_TOPIC_0278888683__table22451554162611)所示的信元携带信息。
  *表1 Nchf_SpendingLimitControl_Subscribe-Initial响应消息中关键信元解释*

  | 关键信元 | 信元解释 |
  | --- | --- |
  | supi | 用户标识。 |
  | statusInfos | 用户配额信息，PCF支持通过配置配额状态映射表将statusInfos携带的配额状态转换为PCF内部的Quota Status。<br>- policyCounterId：用户配额名称<br>- currentStatus：用户配额的状态<br>- penPolCounterStatuses：指示UPCF在某个时间点切换用户的配额状态- policyCounterStatus：切换后的用户配额状态<br>- activationTime：切换用户配额状态的时间点 |
5. PCF将从CHF获取的用户配额状态保存到对应的用户信息中，并根据用户的签约信息及CHF查询到用户的用量计算信息，下发对应的PCC规则和计费策略**，**通过[Npcf_SMPolicyControl_Create](../../../../../../快速入门/协议与流程/5G Core信令分析/消息及信元/服务化接口协议/Npcf/N7/Npcf_SMPolicyControl_Create_67572324.md)Response消息发送给SMF。
  其中，计费策略中的 **计费方式** 划分为离线计费和在线计费：
    - 在线计费：计费会话创建时，SMF需要到CHF申请业务的配额和相关计费参数。
    - 离线计费：计费会话创建时，SMF只需要到CHF申请相关计费参数，但无需申请业务的配额。
6. SMF向UPF发送[PFCP Session Establishment Request](../../../../../../快速入门/协议与流程/5G Core信令分析/消息及信元/非服务化接口协议/N4接口/相关消息解释/PFCP Session Establishment Request_32945585.md)消息，请求建立PFCP（PFCP Session Establishment Request）会话，消息中包含Create PDR、Create FAR等。SMF通过会话建立请求消息中的UE IP Address信元指示UPF具体的地址分配方式。
  *表2 PFCP Session Establishment Request消息中的关键信元*

  | 关键信元 | 信元解释 |
  | --- | --- |
  | ![](计费会话创建流程_01_10001.assets/zh-cn_image_0285047149_2.png "点击放大") | - PDR（Packet Detection Rule ） ID：PFCP会话配置的所有PDR中唯一标识。<br>- Precedence: 指示PDR在PFCP会话的所有PDRs中应用的优先级。<br>- PDI（Packet Detection Information）: 用于匹配进来的报文。 |
  | ![](计费会话创建流程_01_10001.assets/zh-cn_image_0285047151_2.png "点击放大") | - FAR（Forwarding Action Rule ） ID：该IE必须唯一地标识为该PFCP会话配置的所有FAR中的一个。<br>- Apply Action：该信元应指示对数据包适用的动作。<br>- Forwarding Parameters：当Apply Action请求转发数据包时，该IE应该出现。当Apply Action请求转发的报文时，该IE应包含用户面功能需要应用的转发指令。 |
  | ![](计费会话创建流程_01_10001.assets/zh-cn_image_0284974908_2.png) | UE IP Address信元用于指示地址分配方式。其中，CHV6和CHV4表示U面分配的地址，V4和V6表示C面分配的地址。<br>当CHV4为1，CHV6为0，V4为0，V6为0时，表示由UPF根据本地地址池为UE分配IPv4地址。 |
7. UPF根据UE IP Address信元中的CHV4、CHV6、V4、V6 标志位判断地址分配方式，将分配好的IP地址在[PFCP Session Establishment Response](../../../../../../快速入门/协议与流程/5G Core信令分析/消息及信元/非服务化接口协议/N4接口/相关消息解释/PFCP Session Establishment Response_85720872.md)消息中发给SMF，消息中携带的关键信元如下。

  | --- | --- |
  | ![](计费会话创建流程_01_10001.assets/zh-cn_image_0285047161_2.png "点击放大") | 包含在响应消息中，在响应消息中，原因值表示接受或拒绝相应的请求消息。 |

8. SMF**选择CHF**的方法按优先级从高到低排列如下所示。其中，CC是3GPP协议定义的一种用户签约付费属性，可以由UDM下发，也可以在SMF上本地配置。
    a. 拨测场景，SMF根据本地配置**ADD SELCHFGBYIMSI**命令选择CHF，即基于IMSI选择CHF。
    b. 配置基于IMSI号段选择CHF，针对现网存在多个CHF主备组，为了避免CHF之间业务不均衡，通过将不同IMSI号段范围和CHF主备组的绑定将用户分流到不同CHF。
    c. SMF从PCF下发的chargingInfo属性获取CHF的信息。
          - chargingInfo属性值为CHF的域名信息时，需要根据域名信息选择本地配置的CHF。
          - chargingInfo属性值为CHF的IP地址信息时，直接使用该IP地址信息。
    d. SMF根据从UDM获取的签约CC，选择本地配置的CHF。
      UDM下发的CC有两种：Session级的CC和基于APN/DNN的CC，DNN/APN级CC优先级更高。
    e. SMF通过NRF发现CHF。该方式下不支持3GPP 29510协议中规定的通过主备方式选择CHF，仅支持通过优先级和权重选择CHF。
    f. SMF根据本地配置**ADD SELECTCHFGBYCC**命令选择CHF，即基于本地配置的CC选择CHF。

10. CHF检查该用户账户余额是否能满足配额预留，如果该用户账户有足够余额，CHF执行相应预留动作并开启CDR话单。CHF会生成一张CDR原始话单，同时转换生成CDR最终话单，存储到最终话单临时文件中。当最终话单临时文件达到转正阈值（默认1M大小或半小时），则转正为最终话单文件。
11. 12. SMF本地保存配额信息，向UPF发送[PFCP Session Modification Request](../../../../../../快速入门/协议与流程/5G Core信令分析/消息及信元/非服务化接口协议/N4接口/相关消息解释/PFCP Session Modification Request_32686661.md)消息，请求修改PFCP会话，消息中包含Create PDR、Create FAR、Create URR，以及它们之间的关联关系，URR中包含计费事件和计费参数，FAR中包含欠费处理策略。其中，用于指示UPF如何上报的用量上报规则Create URR包含的主要信元如[表6](#ZH-CN_TOPIC_0278888683__table5217133513420)所示。
  *表6 Create URR包含的主要信元*

  | 子信元 | 信元解释 |
  | --- | --- |
  | URR ID | 唯一标识一个Create URR。 |
  | Measurement Method | 标识计费方式，包括流量计费、时长计费或事件计费。 |
  | Reporting Triggers | 标识计费上报的触发条件，包含Periodic Reporting（定期上报）、Volume Quota（流量配额）、Time Quota（时间配额）、Volume Threshold（流量阈值）、Time Threshold（时间阈值）、Linked Usage Reporting（关联URR）等。 |
  | Volume Quota | 流量配额。 |
  | Time Quota | 时间配额。 |
  | Volume Threshold | 标识流量阈值，包含总流量、上行流量和下行流量。 |
  | Time Threshold | 标识时间阈值。 |
  | Linked URR ID | 上报某个URR时，也会上报其关联的URR，例如A link B，当B上报时，也会上报A。 |
  | Measurement Information | 标识测量信息，包含了多个标志位，如ISTM、RADI等。<br>当ISTM为1，指示UPF的时长统计从首次收到授权的配额成功时开始，直到用户去激活时或者该配额管理实例终止结束。SMF通过<br>**SET CNVRGDCHGPARA**<br>命令的<br>“OFLRGTIMECALC”<br>参数可以控制离线RG业务对应的ISTM值。 |
  > **说明**
  > SMF上维护RG与URR ID之间的映射关系，所有的RG都有对应的URR ID，但由于URR并不都是用作计费，所以有些非计费的URR ID没有对应的RG。
  >
  > - Nchf接口上，SMF按照RG粒度向CHF申请配额，并按RG或RG+SID的粒度上报配额；CHF按照RG粒度向SMF下发配额。
  > - N4接口上，SMF向UPF下发URR，URR中包含计费事件和计费参数，当UPF上满足计费条件上报时，UPF按URR向SMF上报配额用量。
13. UPF本地保存配额，向SMF返回[PFCP Session Modification Response](../../../../../../快速入门/协议与流程/5G Core信令分析/消息及信元/非服务化接口协议/N4接口/相关消息解释/PFCP Session Modification Response_86008994.md)消息，PFCP会话建立成功。
14. PDU会话建立成功。用户访问数据业务，UPF按命中的PDR对应的URR统计流量、时长等信息，当满足计费事件上报时，通过N4接口上报到SMF。

> **说明**
> 融合计费场景，当用户采用离线计费即无配额管理时，与在线计费即有配额管理的消息中携带的信元差异如下：
>
> - Nchf_ConvergedCharging_Create Request消息中不包含**requestedUnit**等配额管理相关信元。
> - Nchf_ConvergedCharging_Create Response消息中不包含**grantedUnit**、**timeQuotaThreshold**、**volumeQuotaThreshold**、**unitQuotaThreshold**等配额管理相关信元，**triggerType**不包含配额管理相关Trigger（比如QUOTA_THRESHOLD和QUOTA_EXHAUSTED）。
> - PFCP Session Modification Request消息中的Create URR不包含**Volume Quota、Time Quota、Event Quota**等配额管理相关信元。

> **说明**
> SMF分配用户IP地址时的融合计费会话创建流程和UPF分配用户IP地址时的融合计费会话创建流程差异如下：
>
> 当SMF分配用户IP地址时，SMF将直接携带用户IP地址向选择的CHF发送 [Nchf_ConvergedCharging_ChargingDataCreate Request](../../../../../../快速入门/协议与流程/5G Core信令分析/消息及信元/服务化接口协议/Nchf/N40/Nchf_ConvergedCharging_Charging_30f76786/Nchf_ConvergedCharging_ChargingDataCreate Request_23690012.md) 消息，申请授权业务使用以及申请配额。当CHF将配额信息返回给SMF后，SMF向UPF发送 [PFCP Session Establishment Request](../../../../../../快速入门/协议与流程/5G Core信令分析/消息及信元/非服务化接口协议/N4接口/相关消息解释/PFCP Session Establishment Request_32945585.md) 消息，请求建立PFCP会话，消息中包含Create PDR、Create FAR、Create URR，以及它们之间的关联关系，URR中包含计费事件和计费参数，FAR中包含欠费处理策略。其他流程与 [图1](#ZH-CN_TOPIC_0278888683__fig1010920514215) 相同。
