# 信令流程解读

本章节将对AF触发的UL CL动态分流相关的一些典型的业务流程进行描述，流程简介如 [表1](#ZH-CN_TOPIC_0000001147597513__table815864316529) 所示。 [表1 信令流程场景及介绍](#ZH-CN_TOPIC_0000001147597513__table815864316529) 以及以下流程图中PSA0 UPF表示位于老区域中的锚点UPF，PSA1 UPF、PSA2 UPF表示位域新区域中的辅锚点UPF。

*表1 信令流程场景及介绍*

| 业务流程场景 | 概述 |
| --- | --- |
| UE已经建立了到主锚点PSA0 UPF的PDU会话，AF触发插入UL CL UPF和辅锚点UPF | UE已经建立了到主锚点PSA0 UPF的会话，用户位置移动进入到某区域范围内，AF感知到用户位置满足UL CL分流条件后触发插入UL CL UPF和辅锚点UPF。详细流程如<br>[图1](#ZH-CN_TOPIC_0000001147597513__fig7370143342219)<br>所示。 |
| UE已经建立到主、辅锚点UPF的会话，AF触发更新分流策略 | UE前面已经经历了AF触发的方式插入了UL CL UPF和辅锚点UPF，建立了到主、辅锚点UPF的会话，这时如果AF侧发现该用户的分流策略有更新会触发策略更新流程，下发更新后的策略。详细流程如<br>[图2](#ZH-CN_TOPIC_0000001147597513__fig552514333518)<br>所示。 |
| UE已经建立到主、辅锚点UPF的会话，AF触发更新DNAI | UE前面已经经历了AF触发的方式插入了UL CL UPF和辅锚点UPF，建立了到主、辅锚点UPF的会话，这时如果AF侧发现对该用户的分流DNAI有更新会触发策略更新流程，下发更新后的DNAI。伴随DNAI的更新流程，用户会话会经历删除老的辅锚点UPF并插入新的辅锚点UPF的过程。详细流程如<br>[图3](#ZH-CN_TOPIC_0000001147597513__fig1812710484017)<br>所示。 |
| UE已经建立到主、辅锚点UPF的会话，这两个锚点UPF变更事件触发更新UL CL分流策略或DNAI | 这种场景下如果AF前期通过NEF-PCF-SMF订阅了锚点UPF变更事件，携带订阅的事件类型为dnaiChgType，和事件触发后上报事件的目的URL地址（callbackURL），当用户移出当前辅锚点UPF的覆盖范围，SMF会根据订阅指示向NEF返回事件订阅结果，用于指示网络侧的进一步的行为（如触发网络侧更新UL CL策略等）。详细流程如<br>[图4](#ZH-CN_TOPIC_0000001147597513__fig1896917310193)<br>所示。 |
| UE已经建立到主、辅锚点UPF的会话，AF触发删除UL CL UPF和辅锚点UPF | UE前面已经经历了AF触发的方式插入了UL CL UPF和辅锚点UPF，建立了到主、辅锚点UPF的会话，这时如果AF侧发现用户位置变化，最新的位置已经不满足UL CL的分流条件，则触发删除UL CL UPF和辅锚点UPF。详细流程如<br>[图5](#ZH-CN_TOPIC_0000001147597513__fig4743118144111)<br>所示。 |
| UE已经建立到主、辅锚点UPF的会话，UE发起去激活流程 | UE前面已经经历了AF触发的方式插入了UL CL UPF和辅锚点UPF，建立了到主、辅锚点UPF的会话，这时UE发起去活流程，SMF删除用户会话关联的PCC策略后，需要PCF通知NEF终止会话，NEF收到通知后会向PCF发起请求来删除N5会话。详细流程如<br>[图6](#ZH-CN_TOPIC_0000001147597513__fig196719147423)<br>所示。 |

**图1** UE已有PDU会话AF触发插入UL CL流程

<br>

![](信令流程解读_47597513.assets/zh-cn_image_0000001151357757_2.png)

1. UE激活后已创建到主锚点PSA0 UPF的会话。
2. AF发送Nnef_TrafficInfluence_Create Request消息给NEF，携带分流策略和DNAI，具体包括UE IP或GPSI、Aftransid（AF会话ID）、flowinfo（分流策略）以及分流的目标UPF的DNAI信息。
3. NEF基于UE IP地址信息或GPSI向BSF查询UE当前会话所在的PCF。
4. 找到用户会话所在的PCF后，NEF向PCF发送Npcf_PolicyAuthorization_Create Request消息，携带medComponents(flowinfo)和routetoloca(dnai)。
5. PCF通过Npcf_SMPolicyControl_UpdateNotify流程下发动态分流策略给用户会话所在的SMF，消息中包含参数FlowInformation和RouteToLocation(dnai)。
6. PCF回应NEF的请求。
7. NEF回应AF的请求。
8. SMF基于用户当前所处的位置以及RouteToLocation(dnai)选择UL CL UPF和辅锚点UPF。
9. SMF通过PFCP Session Establishment流程与UL CL UPF和辅锚点UPF建立N4会话，并向UL CL UPF下发分流的IP五元组策略。
10. SMF更新主锚点UPF的下行隧道至UL CL UPF。
11. 11~13更新RAN侧的上行隧道至UL CL UPF。
12. 14a~14b如果RAN侧资源有变化，在收到RAN侧回应后，SMF还需要通过PFCP Session Modification流程更新UL CL UPF的下行隧道信息至新侧RAN。

**图2** UE已建立UL CL会话AF触发更新分流策略流程

<br>

![](信令流程解读_47597513.assets/zh-cn_image_0000001151901629_2.png)

1. UE已建立到主、辅锚点UPF的会话。
2. AF侧发现用户的分流策略有更新，向NEF发送Nnef_TrafficInfluence_Update Request消息，携带更新后的flowinfo。
3. NEF向BSF查询UE会话关联的PCF。
4. NEF向PCF发送Npcf_PolicyAuthorization_Update消息，携带参数medComponents(flowinfo)更新分流规则。
5. PCF通过Npcf_SMPolicyControl_UpdateNotify Request消息下发动态分流策略给用户会话所在的SMF，消息中包含参数flowInfo。
6. SMF通过PFCP Session Modification流程将新的sdf filter到UL CL UPF。
7. 7~9 SMF、PCF逐个给上游网元回复Response消息。

**图3** UE已建立UL CL会话AF触发更新DNAI流程

<br>

![](信令流程解读_47597513.assets/zh-cn_image_0000001105301810_2.png)

1. UE已建立到主、辅锚点UPF的会话。
2. AF侧感知到用户位置变化，触发更新用于选择辅锚点UPF的DNAI。向NEF发送Nnef_TrafficInfluence_Update Request消息，携带更新后的DNAI和flowinfo。
3. NEF向BSF查询UE会话关联的PCF。
4. NEF向PCF发送Npcf_PolicyAuthorization_Update Request消息，携带参数medComponents(flowinfo)更新分流规则，携带参数routetoloca(dnai)更新DNAI。
5. PCF通过Npcf_SMPolicyControl_UpdateNotify Request消息下发动态分流策略给用户会话所在的SMF，消息中包含参数flowInfo和dnai。
6. 5b~7 SMF、PCF逐个给上游网元回复Response消息。
7. 8，SMF基于UE当前的位置以及从PCF侧获取到的DNAI选择UL CL UPF和辅锚点PSA2 UPF。
8. 9a~9b，SMF向UL CL UPF和PSA2 UPF发起PFCP Session Establishment流程，伴随该流程下发分策略。
9. 10a~10b，SMF向PSA0 UPF发起PFCP Session Modification流程更新PSA0 UPF下行隧道到新的UL CL UPF。
10. 11~13 更新RAN侧上行隧道信息到新的UL CL UPF。
11. 14a~14b 如果RAN侧资源有变，更新UL CL UPF的下行隧道至新RAN侧。
12. 15a~15b SMF删除到老侧UL CL UPF和PSA1 UPF的N4会话。

**图4** 锚点UPF变更事件触发更新分流策略或DNAI流程

<br>

![](信令流程解读_47597513.assets/zh-cn_image_0000001105309832_2.png)

1. UE已建立到主锚点PSA0 UPF的会话。
2. AF向NEF发送Nnef_TrafficInfluence_Create Request消息订阅用户面事件，携带订阅的事件类型为dnaiChgType，携带事件上报的目的URL地址callbackURL。
3. NEF向BSF获取用户会话当前关联的PCF。
4. NEF向PCF发送Npcf_PolicyAuthorization_Create Request消息订阅用户面事件，携带订阅的事件类型为dnaiChgType，携带事件上报的目的URL地址callbackURL。
5. PCF向SMF发送Npcf_SMPolicyControl_UpdateNotify Request消息订阅用户面管理事件，事件类型分为early或late或early_late。
6. 5b~7 SMF、PCF逐个给上游网元回复Response消息。
7. UE移动移出了当前辅锚点PSA1 UPF的覆盖范围，触发了用户面锚点变更事件。
8. 9a~9b SMF根据前面订阅的事件类型分别作出如下响应。
    - 如果订阅的是early类型，SMF在辅锚点PSA2 UPF重选之前通过Nsmf_EventExposure_Notify上报用户面管理事件到NEF，此时可能会触发AF下发新的路由策略，并携带新的DNAI指示UPF的重选，详细流程参见[图3](#ZH-CN_TOPIC_0000001147597513__fig1812710484017)。
    - 如果订阅的是late类型，SMF在辅锚点PSA2 UPF重选之后通过Nsmf_EventExposure_Notify上报用户面管理事件到NEF。
    - 如果订阅的是early_late类型，SMF在辅锚点PSA2 UPF重选之前和之后都通过Nsmf_EventExposure_Notify上报用户面管理事件到NEF。
9. NEF给AF发送Nnef_Trafficinfluence_Notify EventNotification消息给AF，将用户面锚点变更事件发送给AF。

**图5** AF触发删除UL CL UPF和辅锚点UPF流程

<br>

![](信令流程解读_47597513.assets/zh-cn_image_0000001151901627_2.png)

1. UE已建立到主、辅锚点UPF的会话。
2. AF侧感知到用户位置移出了UL CL UPF的覆盖范围，向NEF发送Nnef_TrafficInfluence_Delete Request消息删除路由策略。
3. NEF向BSF查询UE会话关联的PCF。
4. NEF向PCF发送Npcf_PolicyAuthorization_Delete Request消息，通知PCF删除用户会话的动态分流策略。
5. PCF给SMF发送Npcf_SMPolicyControl_UpdateNotify Request消息通知SMF删除用户会话的动态分流策略。
6. 5b~7 SMF、PCF逐个给上游网元回复Response消息。
7. SMF删除用户会话的动态分流策略。
8. SMF向PSA0 UPF发起PFCP Session Modification流程更新PSA0 UPF下行隧道到RAN侧。
9. 9~11 更新RAN侧上行隧道信息到PSA0 UPF。
10. 12a~12b SMF释放到UL CL UPF和辅锚点UPF的N4会话。

**图6** UE去活会话终止UL CL分流流程

<br>

![](信令流程解读_47597513.assets/zh-cn_image_0000001151621745_2.png)

1. UE发起PDU会话去活流程。
2. SMF向PCF发起Npcf_SMPolicyControl_Delete流程，请求终止会话，删除和用户会话关联的PCC策略。
3. PCF向NEF发起Npcf_PolicyAuthorization_Notify流程，通知NEF终止会话。
4. NEF响应SMF的会话终止通知后，向PCF发起N5会话删除请求，PCF删除N5会话后向NEF返回删除成功响应。
5. 继续完成用户的PDU会话释放流程（删除用户面通路等）。
