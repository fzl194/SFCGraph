## [操作场景](#ZH-CN_OPI_0231422956)

UNC与PCRF和UDG完成互通数据调测后，需要调测基本的PCC功能。

> **说明**
> 适用于 GGSN、 PGW-C。

## [必备事项](#ZH-CN_OPI_0231422956)

前提条件

- 请仔细阅读 [PCC基本功能（2G/3G/4G，基于Gx接口）](../../PCC基本功能（2G_3G_4G，基于Gx接口）_29056156.md) 。
- 完成[激活PCC基本功能](../激活PCC基本功能_29056159.md)。

## [操作流程](#ZH-CN_OPI_0231422956)

1. 激活用户。
2. PCRF下发计费和策略控制规则。
3. 观察用户能否正常激活，并按照下发策略进行计费控制和策略控制。
4. 去激活用户。

## [操作步骤](#ZH-CN_OPI_0231422956)

1. 进入 “MML命令行-UNC” 窗口。 执行 [**LST LICENSESWITCH**](../../../../../../../OM参考/命令/UNC MML命令/平台服务管理/操作维护/License管理/查询License配置项开关（LST LICENSESWITCH）_09651570.md) 命令查询GGSN/PGW-C上 PCC基本功能 对应的License配置开关是否打开。
  ```
  LST LICENSESWITCH:LICITEM="LKV3W9SPCC11";
  ```
    - 如果“SWITCH”为“ENABLE”，请执行[步骤 2](#ZH-CN_OPI_0231422956__step1618175015812)。
    - 如果“SWITCH”为“DISABLE”，则执行[**SET LICENSESWITCH**](../../../../../../../OM参考/命令/UNC MML命令/平台服务管理/操作维护/License管理/设置License项的开关（SET LICENSESWITCH）_09654190.md)命令打开本特性对应的License配置开关。
2. 在OM Portal上建立用户跟踪消息。
3. 激活用户，并跟踪该用户消息。
4. 查看用户消息跟踪，检查消息跟踪中是否存在Gx接口的Credit Control Request-Initial、Credit Control Answer-Initial消息，以及激活响应消息Create Session Response中的返回码是否为request accepted (16)。
    - 如果Gx接口消息存在，Create Session Response中的返回码为request accepted (16)，表明PCC用户激活成功，请继续执行[步骤 5](#ZH-CN_OPI_0231422956__step5)。
    - 如果Gx接口消息不存在，Create Session Response中的返回码为request accepted (16)，请执行[步骤 7](#ZH-CN_OPI_0231422956__stp5)，确认该用户激活使用的APN是否采用PCC功能。
    - 如果Gx接口消息不存在，Create Session Response中的返回码不为request accepted (16)，请执行[步骤 11](#ZH-CN_OPI_0231422956__stp9)。
    - 如果Gx接口消息存在，但是Create Session Response中的返回码不为request accepted (16)，请执行[步骤 12](#ZH-CN_OPI_0231422956__stp10)。
5. 进入 “MML命令行-UNC” 窗口。 执行 [**DSP PCCSESSINFO**](../../../../../../../OM参考/命令/UNC MML命令/业务服务管理/会话管理/PCC管理/PCC维护/PCC会话查询/显示PCC会话信息（DSP PCCSESSINFO）_09897070.md) 命令，查看指定用户的所有规则是否安装成功。
    - 如果规则安装成功，根据[调测离线计费](../../../../计费管理功能/WSFD-011201 支持离线计费/调测离线计费_25768970.md)验证PCRF下发的规则执行正确，则本任务调测结束。
    - 如果规则安装失败，请执行[步骤 6](#ZH-CN_OPI_0231422956__stp4)。
6. 请查看GGSN/PGW-C上的消息跟踪，查看PCRF下发策略后，GGSN/PGW-C上报的Credit Control Request-Update消息中的rule-failure-code，该信元上报表示在PCC用户激活过程中，PCRF下发的所有规则中的某一些规则在GGSN/PGW-C上安装失败。该rule-failure-code的含义与解决办法请参考 [表1](#ZH-CN_OPI_0231422956__tab1) 中描述。
7. 进入 “MML命令行-UNC” 窗口。 执行 [**DSP PCRFGRPSTATUS**](../../../../../../../OM参考/命令/UNC MML命令/业务服务管理/会话管理/PCC管理/PCRF Diameter连接/PCRF组状态/显示PCRF组状态（DSP PCRFGRPSTATUS）_09897121.md) 命令，查看使用的PCRF组的状态是否正常。
  ```
  DSP PCRFGRPSTATUS:PCRFGRPNAME="pcrf_group_2";
  ```

    - 如果状态正常，请执行[步骤 8](#ZH-CN_OPI_0231422956__stp6)。
    - 如果状态异常，请参考[调测到PCRF的数据](调测到PCRF的数据_31422954.md)进行调测，再执行[步骤 3](#ZH-CN_OPI_0231422956__stp2)。

          a. 检查Credit Control Answer-Initial消息中，是否存在charging-rule-install信元，如 [图1](#ZH-CN_OPI_0231422956__fig14) 所示。
            如果不存在该信元，表示PCRF未给用户下发任何策略。这种情况可参见 [配置异常场景数据](../激活PCC基本功能/配置异常场景数据_31422947.md) 来配置是否仍将用户激活，如果无法激活，请在PCRF网元侧修改配置。修改完成后重复执行 [步骤 4](#ZH-CN_OPI_0231422956__stp3) 。
            如果存在该信元，表示PCRF已经给用户下发策略，请执行 [12.b](#ZH-CN_OPI_0231422956__stpb) 。

            如果Credit Control Request-Terminal中消息上报charging-rule-report信元，则可以根据该信元提供的信息定位PCRF下发的错误内容。例如下图指出PCRF下发的rule名称为b2df6的rule安装失败，失败的原因是rule-failure-code中的描述“rating-group-error”。
            **图2** 消息字段

            ![](调测PCC业务_31422956.assets/zh-cn_image_0231594890_2.png)

            <br>
            [表1](#ZH-CN_OPI_0231422956__tab1) 描述了rule-failure-code的含义和解决办法。除了GW/PCEF_MALFUNCTION (4)，其他所有错误码都和PCRF下发的内容错误有直接关系，需要PCRF修改配置或者定位。PCRF修改完成后，请重新执行 [步骤 4](#ZH-CN_OPI_0231422956__stp3) 。
            *表1 信元Rule-Failure-Code的含义*

            | Rule-Failure-Code | 说明及对策 |
            | --- | --- |
            | UNKNOWN_RULE_NAME (1) | 针对PCRF下发的静态rule或者user-profile，表示下发的rule名称在GGSN/PGW-C上不存在。 |
            | UNKNOWN_RULE_NAME (1) | 请在PCRF上重设rule名称，或者在GGSN/PGW-C上增加匹配的rule。 |
            | RATING_GROUP_ERROR (2) | PCRF下发动态rule时，没有下发rating-group信元。 |
            | RATING_GROUP_ERROR (2) | 请在PCRF上设置该动态rule的RG。 |
            | SERVICE_IDENTIFIER_ERROR (3) | PCRF下发动态rule时，当reporting-level信元为sid时，该动态rule没有同时下发service－identifier信元。 |
            | SERVICE_IDENTIFIER_ERROR (3) | 请在PCRF上设置该动态rule的sid。 |
            | GW/PCEF_MALFUNCTION (4) | GGSN/PGW-C内部处理错误，返回该错误码时，可能是GGSN/PGW-C内部处理出现异常。 |
            | GW/PCEF_MALFUNCTION (4) | 请直接联系华为技术支持工程师。 |
            | RESOURCES_LIMITATION (5) | PCRF下发的rule或者flow-description需要占用资源的信元太多，导致GGSN/PGW-C内部无法处理。 |
            | RESOURCES_LIMITATION (5) | 请在PCRF设置的service上减少rule或者flow-description。 |
            | MAX_NR_BEARERS_REACHED (6) | GGSN/PGW-C暂时不能发出该错误码。 |
            | UNKNOWN_BEARER_ID (7) | PCRF下发的charging-rule-install或者charging-rule-remove信元中，下发了错误的bearer id。 |
            | UNKNOWN_BEARER_ID (7) | 一旦发生这种错误，说明PCRF内部处理有问题。 |
            | MISSING_BEARER_ID (8) | PCRF下发的charging-rule-install或者charging-rule-remove信元中，没有指明bearer id。 |
            | MISSING_BEARER_ID (8) | 一旦发生这种错误，说明PCRF内部处理有问题。 |
            | MISSING_FLOW_INFORMATION (9) | PCRF下发的动态rule中没有<br>flow-information<br>。 |
            | MISSING_FLOW_INFORMATION (9) | 请在PCRF上设置该动态rule的<br>flow-information<br>。 |
            | DUPLICATE_RULE_NAME (1001) | PCRF下发的动态rule和GGSN/PGW-C上本地配置的静态rule重名。 |
