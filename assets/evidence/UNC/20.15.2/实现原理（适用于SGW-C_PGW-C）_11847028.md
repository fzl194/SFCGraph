# 实现原理（适用于SGW-C/PGW-C）

本节主要描述VoLTE承载故障快速恢复在不同故障场景的实现原理，包括如下内容：

> **说明**
> - PGW-C故障/重启场景主叫与被叫的恢复流程相同。
> - 故障发生后，需要等待Echo间隔+T3N3时长后，才开始启动被叫恢复流程。MME、SGW-C/PGW-C和SGW-U/PGW-U上的Echo间隔必须配置一致，但是MME上的T3N3要大于SGW-C/PGW-C上的T3N3。Echo间隔通过[**SET GTPCFUNCPARA**](../../../../../../OM参考/命令/UNC MML命令/业务服务管理/接口管理/GTP-C接口配置管理/GTP-C功能参数/设置GTP-C功能参数（SET GTPCFUNCPARA）_09653666.md)的参数“V2EI”配置，T3N3通过参数“T3”和“N3”配置。
> - MME故障/重启场景中，SGW-C支持在DDN消息中携带TAI，需要开启位置上报控制功能，SGW-C才能在DDN消息中正确携带TAI给MME，以便MME根据TAI进行区域性的寻呼。
> - 故障和重启的判断区别在于：
>     - 重启是通过Echo Response消息中的Recovery counter值是否变更判断。
>     - 故障是通过Echo Response消息是否超出重发次数*重发间隔和防闪断定时器时长的总和判断。防闪断定时器是为了防止在网络状态不好的情况下，频繁发生设备故障的手段。
>     - SGW-U/PGW-U重启是通过PFCP Heartbeat Response消息中的Recovery Time Stamp值是否增加进行判断的。

  **MME故障/重启场景**
  **图1** MME故障/重启场景被叫恢复的业务流程

  <br>

  ![](实现原理（适用于SGW-C_PGW-C）_11847028.assets/zh-cn_image_0000001739557026_2.png)
  > **说明**
  > MME1为故障MME，MME2为正常MME。
    1. SGW-C向MME1发送Echo Request消息，携带Sending Node Features信元，进行NTSR功能协商。
      > **说明**
      > NTSR功能是指在MME故障/重启后，SGW-C根据配置识别保留对应的承载，等待接收网络侧的数据/信令，在SGW-C触发语音业务恢复流程，NTSR功能开启受 [**SET GTPCFUNCPARA**](../../../../../../OM参考/命令/UNC MML命令/业务服务管理/接口管理/GTP-C接口配置管理/GTP-C功能参数/设置GTP-C功能参数（SET GTPCFUNCPARA）_09653666.md) 命令的 “REQPEERNTSR” 参数控制，具体判断逻辑如下：
      >
      > - “REQPEERNTSR”设置为“ON”时，表示以Echo Request消息中携带的NTSR标识判断对端是否支持NTSR功能。
      > - “REQPEERNTSR”设置为“OFF”时，表示业务判断时默认对端支持NTSR功能。
    2. MME1回应Echo Response消息。
    3. MME1向SGW-C发送Echo Request消息，携带Sending Node Features信元，进行NTSR功能协商。
    4. SGW-C回应Echo Response消息。
      > **说明**
      > MME Pool的信息由 [**ADD MMEPOOL**](../../../../../../OM参考/命令/UNC MML命令/业务服务管理/接口管理/网络管理/业务快速恢复/MME Pool/增加MME POOL（ADD MMEPOOL）_31453510.md) 和 [**ADD MMEIPTOMMEPOOL**](../../../../../../OM参考/命令/UNC MML命令/业务服务管理/接口管理/网络管理/业务快速恢复/MME IP/增加MME IP（ADD MMEIPTOMMEPOOL）_31453389.md) 两条命令配置。
    5. SGW-C发送Echo Request消息探测到对应MME1故障或者重启时，SGW-C判定MME1设备故障。
      > **说明**
      > 在MME1故障/重启后。
      >
      > - 如果SGW-C与MME1的NTSR协商已经完成，SGW-C会保留MME1路径上的承载上下文。承载上下文的保留时间由 [**SET FASTRECOVERY**](../../../../../../OM参考/命令/UNC MML命令/业务服务管理/接口管理/网络管理/业务快速恢复/全局参数/设置全局业务快速恢复配置（SET FASTRECOVERY）_31453525.md) 命令配置，默认保留时间为59分钟。
      > - 如果SGW-C上有QCI值等于“1”或“2”的专有承载，保留S1-U接口的资源。对于QCI值不等于“1”或“2”的专有承载，释放S1-U接口的资源。
    6. SGW-U向SGW-C发送 PFCP Session Report Request消息。
    7. MME2收到Downlink Data Notification消息后，回应Downlink Data Notification Acknowledge消息。
    8. MME2根据Downlink Data Notification消息中的IMSI号，给用户下发寻呼请求。
    9. SGW-C向SGW-U回复 PFCP Session Report Response消息。
    10. UE收到寻呼后，便发起Service Request流程。
  **SGW-C故障/重启场景**
  **图2** SGW-C故障/重启场景主叫恢复的业务流程

  <br>

  ![](实现原理（适用于SGW-C_PGW-C）_11847028.assets/zh-cn_image_0000001786716229_2.png)
  > **说明**
  > SGW-C1为故障SGW-C，SGW-C2为正常SGW-C。
    1. MME周期性发送Echo Request消息判断SGW-C1是否正常。
    2. MME发送Echo Request消息探测到对应SGW-C1故障或者重启时，MME对语音用户进行SGW-C重选操作。
    3. MME选择一个状态正常的SGW-C2，发送Create Session Request消息，重新创建语音承载。
    4. SGW-C2向SGW-U发送PFCP Session Establishment Request消息，然后SGW-U向SGW-C2回应PFCP Session Establishment Response消息。
    5. SGW-C2发送Modify Bearer Request消息到PGW-C，创建修改语音承载。
    6. PGW-C向PGW-U发送PFCP Session Modification Request消息，然后PGW-U向PGW-C回应PFCP Session Modification Response消息。
    7. PGW-C向SGW-C2回复Modify Bearer Response消息。
    8. SGW-U向SGW-C2发送PFCP Session Modification Request消息，然后SGW-C2向SGW-U回应PFCP Session Modification Response消息。
    9. SGW-C2向MME回应Create Session Response消息。
    10. 后续无线资源处理流程同标准MM流程对应的处理流程。
  **图3** SGW-C故障/重启场景下行信令触发被叫恢复的业务流程

  <br>

  ![](实现原理（适用于SGW-C_PGW-C）_11847028.assets/zh-cn_image_0000001786636549_2.png)
  > **说明**
  > SGW-C1为故障SGW-C，SGW-C2为正常SGW-C。
    1. MME在S11接口的Create Session Request或者Modify Bearer Request消息中携带MME ID给SGW-C1，SGW-C1将MME ID透传带给PGW-C，PGW-C将新收到的MME ID保存或者覆盖原来上下文中的MME ID。
    2. PGW-C回复Create Session Response或者Modify Bearer Response消息。
    3. PGW-C发送Echo Request消息探测到对应SGW-C1故障或者重启时，将SGW-C1相关承载置为保留承载。
    4. PGW-C向PGW-U发送PFCP Session Modification Request消息，通知PGW-U保留承载，然后PGW-U向PGW-C回应PFCP Session Modification Response消息。
    5. PGW-C收到保留承载的下行信令。
    6. PGW-U向PGW-C发送PFCP Session ReportRequest消息，PGW-C保留承载上下文收到了下行数据报文，给PGW-U返回PFCP Session ReportResponse响应消息。
    7. PGW-C在收到IMS APN对应PDN连接的数据包后，根据UE的注册MME IP，PGW-C向和此MME关联的SGW-C列表中随机选择可用的一个SGW-C2发送PDTN消息。PGW-C为完成该功能需要配置维护MME IP、SGW IP以及其关联关系。
    8. SGW-C2收到该消息后向MME发送对应的PDTN消息。
    9. MME收到PDTN消息，若支持该功能，并找到对应UE上下文，则MME触发网络侧发起的Service Request流程，为UE重选SGW-C。
    10. SGW-C2收到MME返回的PGW Downlink Triggering Ack响应消息后回复PGW Downlink Triggering Ack给PGW-C。SGW-C2为完成此功能同样需要配置维护MME IP。
      > **说明**
      > 如果SGW-C2向MME发送PGW Downlink Triggering Notification消息，收到Context not found的失败响应，并且携带了IMSI和MME ID，则SGW-C2需要向MME Pool内的所有MME广播PDTN消息。
    11. SGW-C2向PGW-C发送Modify Bearer Request消息，然后PGW-C向SGW-C2回复Modify Bearer Response消息。
    12. PGW-C收到Modify Bearer Request消息恢复保留承载，并发送PFCP Session Modification Request消息给PGW-U。
  **图4** SGW-C故障/重启场景下行数据触发被叫恢复的业务流程

  <br>

  ![](实现原理（适用于SGW-C_PGW-C）_11847028.assets/zh-cn_image_0000001739557478_2.png)
  > **说明**
  > SGW-C1为故障SGW-C，SGW-C2为正常SGW-C。
    1. 在用户激活和SGW-C改变的更新流程中，PGW-C需要通过私有扩展信元将SGW-C的IP地址携带给PGW-U。
    2. SGW-C1发送GTP信令消息给PGW-C，消息中Recovery counter值发生变化或者超出重发次数*重发间隔和防闪断定时器时长的总和。
    3. PGW-C向PGW-U发送PFCP Session Modification Request消息，通知PGW-U保留承载，然后PGW-U向PGW-C回应PFCP Session Modification Response消息。
    4. PGW-U收到保留承载的下行数据。
    5. PGW-U向PGW-C发送PFCP Session Report Request消息，PGW-C保留承载上下文收到了下行信令报文，给PGW-U返回PFCP Session Report Response响应消息。
    6. PGW-C在收到IMS APN对应PDN连接的数据包后，根据UE的注册MME IP，PGW-C向和此MME关联的SGW-C列表中随机选择可用的一个SGW-C2发送PDTN消息。PGW-C为完成该功能需要配置维护MME IP、SGW IP以及其关联关系。
    7. SGW-C2收到该消息后向MME发送对应的PDTN消息。
    8. MME收到PDTN消息，若支持该功能，并找到对应UE上下文，则MME触发网络侧发起的Service Request流程，为UE重选SGW-C。
    9. SGW-C2收到MME返回的PGW Downlink Triggering Ack响应消息后回复PGW Downlink Triggering Ack给PGW-C。SGW-C2为完成此功能同样需要配置维护MME IP。
      > **说明**
      > 如果SGW-C2向MME发送PGW Downlink Triggering Notification消息，收到context not found的失败响应，并且携带了IMSI和MME ID，则SGW-C2需要向MME Pool内的所有MME广播PDTN消息。
    10. MME重选新的SGW-C2重建会话。
    11. SGW-C向PGW-C发送Modify Bearer Request消息，然后PGW-C向SGW-C回复Modify Bearer Response消息。
    12. PGW-C收到Modify Bearer Request消息恢复保留承载，并发送PFCP Session Modification Request消息给PGW-U。
  **SGW-U故障/重启场景**
  **图5** SGW-U故障/重启场景主叫/被叫的业务流程

  <br>

  ![](实现原理（适用于SGW-C_PGW-C）_11847028.assets/zh-cn_image_0000001717982036_2.png)
    1. SGW-C发送PFCP Heartbeat Request消息探测到对应SGW-U1故障或者重启时，SGW-C判定SGW-U1设备故障。
    2. SGW-C发现SGW-U1故障或者重启后，本地删除对应SGW-U1的相应的PDN连接资源（SGW-U重启时不等待防闪断定时器超时）。
    3. 后续SGW-C收到信令都回复Context not found的失败响应，触发PGW-C保留承载。
    4. 用户重新激活IMS PDN时，MME还有可能选到该SGW-C，SGW-C选择一个可用的SGW-U2。
    5. （可选）如果SGW-C下通过[**ADD UPBINDS11**](../../../../../../OM参考/命令/UNC MML命令/业务服务管理/会话管理/接入管理/UPF选择管理/UPF绑定S11接口/增加SGW-U与SGW-C侧S11接口的绑定关系（ADD UPBINDS11）_09653045.md)绑定同一S11接口的SGW-U都故障，会触发[WSFD-213004 GW-U故障隔离](../../../容灾功能/WSFD-213004 GW-U故障隔离_74590850.md)功能，则MME会通过[重选SGW-C](#ZH-CN_CONCEPT_0000001111847028__fig4)进行故障恢复或重启。
    6. （可选）如果SGW-C下通过[**ADD UPBINDS11**](../../../../../../OM参考/命令/UNC MML命令/业务服务管理/会话管理/接入管理/UPF选择管理/UPF绑定S11接口/增加SGW-U与SGW-C侧S11接口的绑定关系（ADD UPBINDS11）_09653045.md)绑定同一S11接口存在未故障的SGW-U，但是该SGW-U不支持该TAI区域，请收集告警信息、日志信息和配置信息，并联系华为公司技术支持处理。
  **PGW-C故障/重启场景**
  **图6** PGW-C故障/重启场景主叫/被叫的业务流程

  <br>

  ![](实现原理（适用于SGW-C_PGW-C）_11847028.assets/zh-cn_image_0000001786636765_2.png)
  > **说明**
  > PGW–C1为故障PGW-C，PGW-C2为正常PGW-C。
    1. SGW-C发送Echo Request消息探测到对应PGW-C1故障或者重启时，SGW-C判定PGW-C1设备故障。
    2.
      SGW-C发现PGW-C1故障或者重启后，本地删除对应PGW-C1的相应的PDN连接资源。

      > **说明**
      > 删除PGW-C1相应的PDN连接资源，对PGW-C1进行去活，需要开启 [**SET GTPCFUNCPARA**](../../../../../../OM参考/命令/UNC MML命令/业务服务管理/接口管理/GTP-C接口配置管理/GTP-C功能参数/设置GTP-C功能参数（SET GTPCFUNCPARA）_09653666.md) 命令的 “对端Recovery处理开关” 参数。
    3. SGW-C删除资源的同时，向各个PDN连接对应的MME发送PGW Restart Notification，MME通过此消息获知对应PGW-C1故障或者重启。
    4. MME回复PGW Restart notification Ack给SGW-C。
    5. MME按照 **[SET GWOFFLOADPARA](../../../../../../OM参考/命令/UNC MML命令/业务服务管理/Pre 5G接入业务管理/控制面管理/网络管理/网关故障管理/设置网关迁移参数（SET GWOFFLOADPARA）_72345681.md)** 配置的速率去活PDN的流程中携带“reactivation requested”给UE，UE便重新发起该PDN连接资源重建。
    6. MME遍历上下文通知UE重建过程需要较长时间，期间会有尚未被通知到UE发起了Service Request消息。
    7. MME判断该用户对应的PDN连接都在故障或者重启PGW-C，MME拒绝和此PGW-C相关的消息请求，触发UE重新附着。
  **PGW-U故障/重启场景**
  **图7** PGW-U故障/重启场景主叫/被叫的业务流程

  <br>

  ![](实现原理（适用于SGW-C_PGW-C）_11847028.assets/zh-cn_image_0000001765621717_2.png)
    1. PGW-C发送PFCP Heartbeat Request消息探测到对应PGW-U1故障或者重启时，PGW-C判定PGW-U1设备故障。
    2. 防闪断定时器超时后，PGW-C按照**[SET DEACTIVERATE](../../../../../../OM参考/命令/UNC MML命令/业务服务管理/会话管理/接入管理/接入管理运维/去活PDP速率/设置去激活用户承载的速率（SET DEACTIVERATE）_09652156.md)**配置的速率去活PDN的流程中携带“reactivation requested”给UE，UE便重新发起该PDN连接资源重建，PGW-C会重选PGW-U2进行激活。
