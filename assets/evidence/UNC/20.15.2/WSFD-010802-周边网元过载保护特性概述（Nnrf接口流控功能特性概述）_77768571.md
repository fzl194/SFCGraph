#### [适用NF](#ZH-CN_CONCEPT_0277768571)

NRF

#### [定义](#ZH-CN_CONCEPT_0277768571)

Nnrf接口流控功能是指在Nnrf接口存在拥塞风险时， UNC 可以基于以下配置减少服务化接口拥塞，避免对端网元过载：

- 配置服务化接口流控速率阈值，控制接口上发送消息的速率。
- 配置NRF平滑通知参数，对频繁变更的NF进行一定时间阈值范围内的变更事件合并发送，减少变更通知的发送数量。

#### [客户价值](#ZH-CN_CONCEPT_0277768571)

| 受益方 | 受益描述 |
| --- | --- |
| 运营商 | 提高整网接入成功率，减少服务化接口拥塞的发生，从而提高用户满意度，增加收益。 |
| 用户 | 减少了因服务化接口拥塞而引起的用户不能使用网络的情况，使用户享受更稳定的网络服务。 |

#### [应用场景](#ZH-CN_CONCEPT_0277768571)

由于5G网络部署越来越广泛，用户量剧增，服务化接口拥塞的可能性也越来越大，为了保证用户接通率，改善用户体验，引入Nnrf流控功能，默认开启本功能。应用场景包括：

- 升级过程中系统重启
- 系统掉电重启、误重启
- 重大节日、大型活动等特殊时间段

#### [可获得性](#ZH-CN_CONCEPT_0277768571)

**涉及NF**

| **涉及NF** | 支持版本 | 功能说明 |
| --- | --- | --- |
| AMF/SMF/NSSF | UNC 20.1.0及后续版本 | 依据配置的固定流控速率控制服务化接口上发送消息速率。 |
| NRF | UNC 20.3.0及后续版本 | 依据配置的固定速率控制每秒钟发送的通知消息个数，减少对周边网元的冲击。 |
| NRF | UNC 20.5.0及后续版本 | 依据配置的NRF平滑通知参数，抑制合并频繁变更NF的变更通知，减少NRF发送变更通知的数量，减少对周边网元的冲击。 |

#### [对系统的影响](#ZH-CN_CONCEPT_0277768571)

UNC 对Nnrf接口上接收和发送的消息流控，流控速率配置值小于周边NF处理能力时，用户接入会过慢，用户接入时失败比例较高。

UNC 对Nnrf接口上做NF变更通知合并发送功能，订阅方NF接收的部分变更通知可能会有短暂的延迟 （最大延迟时间：NF更新事件缓冲最长时长(秒)） 。

#### [应用限制](#ZH-CN_CONCEPT_0277768571)

本特性无应用限制。

#### [原理概述](#ZH-CN_CONCEPT_0277768571)

NFPROFILE的变更事件包括NF注册事件、NFPROFILE变更事件和NF去注册事件。

- Nnrf接口发送消息速率控制
  通过 **[SET NRFNOTIFYFCPARA](../../../../../../OM参考/命令/UNC MML命令/业务服务管理/NRF业务及策略管理/NRF业务参数/NRF流控参数/设置通知流控参数（SET NRFNOTIFYFCPARA）_09653165.md)** 配置指定NRF每秒处理NFPROFILE变更事件的数量，每个NFPROFILE的变更事件会触发NRF向所有订阅请求方NF发送通知，NRF通过命令的配置控制每秒并发处理NFPROFILE变更事件的数量，减少对周边网元的冲击。
  *表1 不同业务类型处理机制*

  | 业务类型 | 处理机制 |
  | --- | --- |
  | NF/NFS服务发现 | 不控制 |
  | NF注册 | 不控制 |
  | NF去注册 | 不控制 |
  | NF更新 | 不控制 |
  | NF状态订阅 | 不控制 |
  | NF状态通知 | 速率控制 |
- Nnrf接口变更事件抑制合并发送控制
    - 通过**[SET NRFNOTIFYFCPARA](../../../../../../OM参考/命令/UNC MML命令/业务服务管理/NRF业务及策略管理/NRF业务参数/NRF流控参数/设置通知流控参数（SET NRFNOTIFYFCPARA）_09653165.md)**配置“NF更新事件缓冲最长时长(秒)”和**“NF更新事件最短缓冲时长(秒)”**，
    - 当NF发生首次变更后进入通知缓存队列，NRF会实时判断后续NF的数据变更情况，对符合条件的NF数据变更进行合并发送，减少NRF发送变更通知的数量，减少对周边网元的冲击（其中，数据变更内容包含：事件包括NF注册、NFPROFILE变更和NF去注册）：
          - 如果在**“NF更新事件最短缓冲时长(秒)”**范围内，NF没有再次发生数据变更，NRF直接将通知发送出去。
          - 如果在**“NF更新事件最短缓冲时长(秒)”**范围内，NF再次发生了数据变更，NRF会将数据变更进行合并同时保留在缓存队列，再次启动实时判断：
                  - **“NF更新事件缓冲最长时长(秒)”**时长到达前，如果发生了**“NF更新事件最短缓冲时长(秒)”**范围内NF没有再次发生数据变更，此时NRF将缓存队列的数据变更合并发送出去。
                  - 如果每次判断**“NF更新事件最短缓冲时长(秒)”**范围内NF都发生了数据变更，直到达到了**“NF更新事件缓冲最长时长(秒)”**时长，则NRF将缓存队列的数据变更合并发送出去。
      > **说明**
      > 对于数据类的变更，由于NRF上述对周边网元过载保护功能的平滑通知功能，NRF对NF侧数据变更的通知发送会存在 **“** **NF更新事件最短缓冲时长(秒)”** ~“ **NF** **更新事件缓冲最长时长(秒)** ”范围的延迟，NRF不能将NF数据变更通知及时发送出去，可能导致订阅方NF不能及时感知目标NF的变化。例如号段类NF的号段从一个NF迁移到其他NF上，NF数据变化通知发送之前的平滑通知功能判断时间内，业务还是选择到原来的NF上，但此NF可能不再支持此号段用户的业务。
    - 通过**[SET NRFNOTIFYFCPARA](../../../../../../OM参考/命令/UNC MML命令/业务服务管理/NRF业务及策略管理/NRF业务参数/NRF流控参数/设置通知流控参数（SET NRFNOTIFYFCPARA）_09653165.md)**配置**“NF状态变更事件最短缓冲时长(秒)”**和**“NF状态变更事件最长缓冲时长(秒)”**，当NF发生首次状态变更后进入通知缓存队列，NRF会实时判断后续NF的状态变更情况，对符合条件的NF状态变更进行合并发送，减少NRF发送变更通知的数量，减少对周边网元的冲击（其中，状态变更内容包含：心跳中断、恢复（仅status或status+load变更）；人工禁止、解禁NF；recoverytime或status+recoverytime变更）：
          - 如果在**“****NF状态变更事件最短缓冲时长(秒)”**范围内，NF没有再次发生状态变更，NRF直接将通知发送出去。
          - 如果在**“****NF状态变更事件最短缓冲时长(秒)”**范围内，NF再次发生了状态变更，NRF会将状态变更进行合并同时保留在缓存队列，再次启动实时判断：
                  - **“NF状态变更事件最长缓冲时长(秒)”**时长到达前，如果发生了**“NF状态变更事件最短缓冲时长(秒)”**范围内NF没有再次发生状态变更，此时NRF将缓存队列的状态变更合并发送出去。
                  - 如果每次判断**“NF状态变更事件最短缓冲时长(秒)”**范围内NF都发生了状态变更，直到达到了**“NF状态变更事件最长缓冲时长(秒)”**时长，则NRF将缓存队列的状态变更合并发送出去。
      > **说明**
      > 对于状态更新的变更，由于NRF对周边网元过载保护功能的平滑通知功能，NRF对NF侧状态类变更的通知发送会存在 **“NF状态变更事件最短缓冲时长(秒)”** ~ **“** **NF** **状态变更事件最长缓冲时长(秒)** **”** 范围的延迟，NRF不能将NF状态、recoverytime类变更通知及时发送出去，可能导致订阅方NF不能及时感知目标NF的状态。例如NF心跳状态由正常变更到故障，NF状态通知发送之前的平滑通知功能判断时间内，业务仍然会选择到原来的NF上，但此NF故障了。
