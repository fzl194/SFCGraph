  具体流程说明如下：

  1. MS向SGSN发送激活请求Activate PDP Context Request消息（NSAPI，TI， PDP Type，PDP Address，Access Point Name，QoS Requested）。PDP Address是指PDP地址。如是动态地址，则设为空。
    2. 如果是GSM系统，可能执行安全流程。
    3. 如果是GSM系统，且激活了BSS跟踪，SGSN向BSS发送Invoke Trace消息。
    4. SGSN根据用户和PDP Context签约数据中的PDP Type（optional），PDP Address（optional）和Access Point Name（optional）来验证Activate PDP Context Request的有效性。
      SGSN给PDP Context分配TEID，如果使用动态地址，则要求GGSN分配一个动态地址。SGSN根据一定的算法选择一个APN，SGSN向GGSN发送Create PDP Context Request消息。
      GGSN为PDP Context分配动态地址，charging ID，GGSN的用户面IP地址和TEID，GGSN的控制面IP地址和TEID，协商QoS。GGSN向SGSN发送Create PDP Context Response消息。如果MS要求外部网分配IP地址，则将PDP Address设为0.0.0.0，在以后外部网分配地址后，执行GGSN发起的PDP Context修改过程。
    5. 如果是UMTS系统，执行RAB指配过程建立RAB。
      RAB指配流程说明：

      a. SGSN向RNC发送RAB Assignment Request消息，请求建立RAB(s)，在指配参数中可指定RAB的无线优先级，是否允许抢占和排队。
          b. RNC建立无线承载。
          c. RNC向SGSN发送RAB Assignment Response，如果因为QoS的原因指配失败，则要发送降低QoS重发指配的请求。

      > **说明**
      > 如果RAB重建时发生QoS改变，则执行SGSN发起的PDP Context修改流程，将QoS通知MS和GGSN。
    6. 如果是UMTS系统，SGSN向RAN发送Invoke Trace消息。
    7. 如果是GSM系统，SGSN可能执行BSS Packet Flow Context过程与BSS进行QoS流控参数协商。
    8. 如果在UMTS系统的第4步/第5步或在GSM系统的第7步SGSN降低了QoS，SGSN会向GGSN发送Update PDP Context Request消息。GGSN发送Update PDP Context Response消息给SGSN确认PDP Context已经更新。
    9. SGSN收到GGSN的Update PDP Context Response消息（NSAPI，PDP ADDR，GGSN ADDR，TEID，QoS），SGSN将PDP地址、QoS等信息通过Activate PDP Context Accept消息发送给MS。
  上述流程是在GnGp场景下的PDP上下文激活，在S4 SGSN场景下，PDP上下文激活流程有所差异。此处以S4 SGSN场景下UE发起的PDP上下文激活流程（UMTS）为例进行描述。S4 SGSN场景下UE发起的PDP上下文激活流程(GSM)与UMTS的类似，此处不再描述。

#### [APN选择](#ZH-CN_CONCEPT_0185436071)

SGSN根据UE请求参数和HLR签约参数进行APN选择，用于激活PDP上下文。APN由两部分组成（3GPP TS 23.003）：

- APN NI（Network Identifier）
  APN NI是APN中的必选部分，用于标识需要接入的外部数据网络的类型。APN NI需要在HLR中进行了签约，激活时才允许使用该APNNI。
  APN NI是由一个标记或用点分隔的多个标记组成，例如“company.com”或 “service.company.com”。
  部分保留的APN用于指示GGSN支持附加的业务，例如"dhcp"和"MIPv4FA"，分别表示GGSN支持使用动态主机配置协议DHCP（Dynamic Host Configuration Protocol）分配外部PDN地址以及支持MIP代理功能。
- APN OI（Operator Identifier）
  APN OI是APN的可选部分，用于标识运营商类型。APN OI的格式完全符合DNS命令规范，由三个标记组成。APN OI使用".gprs"结尾，对于每个运营商，存在一个缺省值的APN OI，此APN OI由IUEI导出，形式为“MNCxxx.MCCyyy.gprs”。

SGSN收到UE的PDP激活请求消息后，解析出激活请求消息中的PDP类型、PDP地址和APN信息三元组，与签约三元组进行匹配。根据用户请求消息中三种参数的不同组合，SGSN进行APN选择分以下几种场景：

- 用户请求消息中包含PDP type，PDP address，APN
  如果签约数据中包含请求的PDP type、PDP address和APN，SGSN选择请求的APN。
- 用户请求消息中包含PDP type，PDP address
  如果签约数据中包含请求的PDP type对应的PDP address，SGSN选择签约数据中的APN。
- 用户请求消息中包含PDP type，APN
  签约数据中包含请求的PDP type对应的请求的APN，SGSN选择请求的APN。如果没有签约数据记录，只有野卡APN记录，SGSN选择UE请求的APN。

  > **说明**
  > 野卡APN：如果HPLMN运营商允许用户接入给定PDP类型的任何网络，则可以给该用户增加签约野卡APN。
- 用户请求消息中包含PDP type
  如果签约数据中只包含一条带有野卡APN记录的PDP上下文，只需要SGSN上已经配置了缺省APN，SGSN会选择使用这个缺省APN。
- 用户请求消息中三个参数都不包含
  如果签约数据中只包含一条PDP上下文记录，且签约的PDP上下文记录中包含PDP address（或还有APN），SGSN会选择签约的APN。
  如果签约数据中只包含一条PDP上下文记录，且签约的PDP上下文记录中没有PDP address只有野卡APN，只需要SGSN上已经配置了缺省APN，SGSN会选择使用这个缺省APN。
  如果签约数据中只包含一条PDP上下文记录，且签约的PDP上下文记录中没有PDP address但有APN，SGSN会选择签约的APN。

**APN NI的选择**

APN NI部分的选择获取方式如下：

如果按照3GPP 23.060 附录A，APN选择模式为“ChosenBySGSN”时，需要使用SGSN配置的APN NI，则按照以下步骤进行：

1. 查询IUEI对应的缺省APN NI，如果存在，使用该APN NI。
2. 查询PDP TYPE对应的缺省APN NI，如果存在，使用该APN NI。

**APN OI的选择**

APN OI按照特定的规则进行选择。

运营商往往希望控制用户使用拜访地或归属地的GGSN。 UNC 支持使用本地策略决定是否允许外网用户使用VPLMN分配的地址，从而确定APN OI的选择模式并判断用户的漫游属性，形成APN域名列表，用户可查询此列表获得相应的APN到DNS解析。APN OI选择模式的流程如 [图7](#ZH-CN_CONCEPT_0185436071__zh-cn_concept_0130427330_fig04) 所示。

1. 判断签约数据匹配得到的APN NI是否由 UNC 提供。

2. 根据用户IMSI的MCC/MNC是否在ADD HPLMN配置列表中，判断用户是否为本网用户。
    - 是 => APN OI模式为H-HPLMN only，漫游属性为Home
    - 否 => [3](#ZH-CN_CONCEPT_0185436071__zh-cn_concept_0130427330_step3_3)

3. 根据 [**ADD GWSELPLCY**](../../../../../../OM参考/命令/UNC MML命令/业务服务管理/Pre 5G接入业务管理/控制面管理/GTP-C接口管理/GnGp-GGSN_S5_S8接口管理/GGSN_P-GW选择/增加GGSN_P-GW选择策略（ADD GWSELPLCY）_26145944.md) 命令配置结果判断是否允许外网用户使用VPLMN分配的地址。

    a. 查看用户是否签约允许使用VPLMN地址。
          - 是 => [3.b](#ZH-CN_CONCEPT_0185436071__zh-cn_concept_0130427330_step3_3_b)
          - 否 => [5](#ZH-CN_CONCEPT_0185436071__zh-cn_concept_0130427330_step3_5)
