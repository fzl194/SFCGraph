# 实现原理

GWCN网络共享模式下的主要业务流程包括：

- Non-supporting UE的注册流程
- Supporting UE的注册流程
- Relocation流程

**GWCN共享模式下，Non-supporting UE的注册流程**

GWCN共享模式下，Non-supporting UE的附着流程如 [图1](#ZH-CN_CONCEPT_0185660613__zh-cn_concept_0130429077_fig2) 所示。

**图1** 网络共享（GWCN）模式下，Non-supporting UE的附着流程示意图

<br>

![](实现原理_85660613.assets/zh-cn_image_0221726305_2.png)

1. UE获取共享RAN的广播系统信息。
2. UE无法解析广播系统信息中的共享网络信息。
3. UE在Common PLMN和其他非共享区的PLMN中进行网络选择。
4. UE发送Attach Request消息。
5. 被共享的SGSN决定是否允许UE接入。
6. 被共享的SGSN发送合适的Accept/Reject消息给UE。

**GWCN共享模式下，Supporting UE的注册流程**

GWCN共享模式下，Supporting UE的附着流程如 [图2](#ZH-CN_CONCEPT_0185660613__zh-cn_concept_0130429077_fig3) 所示。

**图2** 网络共享（GWCN）模式下，Supporting UE的附着流程示意图

<br>

![](实现原理_85660613.assets/zh-cn_image_0221726447_2.png)

1. UE获取共享RAN的广播系统信息。
2. UE可以解析广播系统信息中的共享网络信息，其中的可用核心网运营商PLMN ID参与PLMN选择过程。Common PLMN不参与网络选择过程。
3. UE在可用的PLMN中进行网络选择。
4. UE发送Attach Request消息，该消息指示了选择的核心网运营商。
5. 被共享的SGSN决定是否允许UE接入。
6. 被共享的SGSN发送合适的Accept/Reject消息给UE。

**GWCN共享模式下，Relocation流程**

GWCN共享模式中，Relocation流程的分为Intra和Inter两种情况。对于Intra SGSN Relocation的情况，SGSN只要在Relocation Request消息中将PLMN ID通知到 目标RNC 即可。Inter SGSN Relocation业务流程如 [图3](#ZH-CN_CONCEPT_0185660613__zh-cn_concept_0130429077_fig4) 所示。

**图3** 网络共享（GWCN）模式下，Inter SGSN Relocation流程示意图

<br>

![](实现原理_85660613.assets/zh-cn_image_0221752270_2.png)

这里只描述与非网络共享Relocation流程有变动的相关步骤。如 [图3](#ZH-CN_CONCEPT_0185660613__zh-cn_concept_0130429077_fig4) 红色标记内所示。

Old SGSN：Old SGSN会在Gn口发往New SGSN的Forward Relocation Request消息中携带Select PLMN信元，该信元的信息来自于Source RNC在Relocation Required消息中为UE选择的Target PLMN。

New SGSN：收到Forward Relocation Request消息后，首先判断消息中是否包含Selected PLMN信元。

- 如果Forward Relocation Request消息中包含Selected PLMN信元，New SGSN判断该PLMN是否合法。
    - 如果合法，则正常执行Relocation流程操作，使用该Selected PLMN作为Serving PLMN，并把该Serving PLMN下发给Target RNC。
    - 如果不合法，则判断Target ID信元中RNC ID的PLMN是否合法。如果合法，使用该PLMN作为Serving PLMN，并把该Serving PLMN下发给Target RNC；否则拒绝Relocation流程。
- 如果Forward Relocation Request消息中不包含Selected PLMN信元，则判断Target ID信元中RNC ID的PLMN是否合法。如果合法，使用该PLMN作为Serving PLMN，并把该Serving PLMN下发给Target RNC；否则拒绝Relocation流程。

> **说明**
> GWCN共享模式下，在Relocation流程中，SGSN在获得合法Serving PLMN后，将Target RNC ID对应的PLMN和系统中对应RNC的PLMN配置信息进行匹配，如果匹配，则将Serving PLMN下发到Target RNC，否则Relocation流程失败。因此，当RNC支持多HPLMN功能时，SGSN上应该配置每个PLMN对应的RNC信息（ [**ADD RNC**](../../../../../OM参考/命令/UNC MML命令/业务服务管理/Pre 5G接入业务管理/控制面管理/Iu接口管理/Iu接口RNC信息/增加Iu接口RNC信息(ADD RNC)_26146040.md) ）和3G寻呼数据（ [**ADD IUPAGING**](../../../../../OM参考/命令/UNC MML命令/业务服务管理/Pre 5G接入业务管理/控制面管理/Iu接口管理/Iu接口寻呼数据/增加Iu接口寻呼数据(ADD IUPAGING)_26305844.md) ）。例如RNC的LAC和RAC分别是0x2301和0x21，支持46100和46101两个PLMN，则SGSN需要完成如下配置：
>
> ```
> ADD RNC: RNCX=1, RNCN="RNC_SHARE", RNCMCC="461", RNCMNC="00", RNCID=0, NI=NAT, SPC="0xb01", RNCVER=R6;
> ADD RNC: RNCX=2, RNCN="RNC_SHARE", RNCMCC="461", RNCMNC="01", RNCID=0, NI=NAT, SPC="0xb01", RNCVER=R6;
> ADD IUPAGING: LAI="461002301", RAC="21", RNCINDEX=1;
> ADD IUPAGING: LAI="461012301", RAC="21", RNCINDEX=2;
> ```
