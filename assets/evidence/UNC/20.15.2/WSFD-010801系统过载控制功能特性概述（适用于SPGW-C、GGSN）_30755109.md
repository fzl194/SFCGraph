#### [适用NF](#ZH-CN_CONCEPT_0230755109)

GGSN、 SGW-C、PGW-C

#### [定义](#ZH-CN_CONCEPT_0230755109)

在系统过载时， UNC 通过控制接入系统的业务量来防止拥塞进一步恶化而造成整系统的瘫痪，从而保证了系统的稳定性和高优先级业务的成功率。

#### [客户价值](#ZH-CN_CONCEPT_0230755109)

| 受益方 | 受益描述 |
| --- | --- |
| 运营商 | 在网络设备高倍过载场景下，提高用户业务的接通率，提升设备承载的有效业务量和网络可靠性。 |
| 用户 | 在网络设备高倍过载场景下，提高用户业务尤其是高优先级业务（如语音业务）的成功率，提升用户业务体验和满意度。 |

#### [应用场景](#ZH-CN_CONCEPT_0230755109)

造成系统过载的主要原因可分为以下4类，在这些情况下，系统会进行过载流控。

- 整系统或周边网元的供电中断、升级。
- 传输供电中断、自然灾害或战争导致传输线路物理中断。
- 网络运维，如割接、网络调整等。
  以上3种场景在恢复后，都会有大量业务重新连接、大量链路重新建立等，此时会对系统造成冲击，造成过载。
- 大型活动、节假日、促销推送活动等
  短时间内活跃用户的增多，使数据业务、语音、视频增多，增加系统的负荷。

#### [对系统的影响](#ZH-CN_CONCEPT_0230755109)

流控会丢弃部分消息，可能导致部分用户无法建立PDN连接、已有业务中断。

#### [原理概述](#ZH-CN_CONCEPT_0230755109)

- **基于系统CPU负载，业务流程首消息流控**
  基于系统CPU负载，业务流程首消息流控
  UNC 根据系统CPU负载，评估系统过载等级，然后根据过载等级，按照业务优先级对业务首消息进行流控。当系统过载等级为中度或重度时，启动流控。流控算法采用WAL算法。
  **默认系统过载等级分为：**
    - 轻度过载：CPU占用率大于等于75%，小于80%
    - 中度过载：CPU占用率大于等于80%，小于85%
    - 重度过载：CPU占用率大于等于85%，小于100%
  **WAL算法**
  WAL算法应用在系统过载控制中，是一种基于即时系统CPU占用率来动态调整系统允许通过的最大和最小业务量的流控算法。
  WAL算法的应用使系统CPU资源的使用可控，例如在系统中度过载（CPU占用率大于80%）时，系统通过WAL算法来控制业务消息的数量，使系统CPU占用率保持在75%到80%之间；由于系统CPU占用率是实时波动的，当系统CPU占用率超过80%时，部分消息通过WAL算法会被丢弃，而在系统CPU占用率低于75%的部分则说明此时系统流控力度过猛、丢弃消息过多，需要通过WAL算法减少即时消息丢弃量使CPU占用率提升到75%，来保证业务稳定性。如下图所示：

- **队列控制**
  WAL算法是基于队列的流控算法，把用户信令区分不同的优先级，根据信令消息的优先级决策丢弃消息的比例。基本思想是首先保证在线用户的业务不受影响，其次保证用户尽可能的接入，根据CPU占有率动态调整信令消息的处理速率，同时还有一个基本原则，不做无效的处理，保证处理都是有效的，简单来说，比如处理100个用户的消息，但是最终每个用户的消息流程都不完整，导致最终每个消息的处理都是无效的，这样不如处理一个用户完整流程的消息。
  *表1 信令消息处理优先级及流控现象*

  | 处理优先级 | 消息类 | 过载时的流控现象 |
  | --- | --- | --- |
  | 高 | Delete类消息 | 用户去激活失败，SPGW-C/GGSN上用户信息残留 |
  | 中 | Update类消息 | 承载更新失败，可能导致用户去激活 |
  | 低 | Create类消息 | 用户激活失败 |
  SPGW-C/GGSN对以下信令消息进行流控：
    - Create Session Request/Create Session Response
    - Modify Bearer Request/Modify Bearer Response
    - Delete Session Request/Delete Session Response
    - Create Bearer Request/Create Bearer Response
    - Update Bearer Request/Update Bearer Response
    - Delete Bearer Request/Delete Bearer Response
    - Bearer Resource Command
    - Bearer Resource Failure Indication
    - Downlink Data Notification Acknowledge
    - Downlink Data Notification Failure Indication
    - Delete Indirect Data Forwarding Tunnel Request/Delete Indirect Data Forwarding Tunnel Response
    - Modify Bearer Command
    - Modify Bearer Failure Indication
    - Delete Bearer Command
    - Delete Bearer Failure Indication
    - Create Indirect Data Forwarding Tunnel Request/Create Indirect Data Forwarding Tunnel Response
    - Release Access Bearers Request/Release Access Bearers Response
    - Delete PDN Connection Set Request/Delete PDN Connection Set Response
    - Update PDN Connection Set Request/Update PDN Connection Set Response
    - Change Notification Request/Change Notification Response
    - Report Event Request
    - Create PDP Context Request/Create PDP Context Response
    - Update PDP Context Request/Update PDP Context Response
    - Delete PDP Context Request/Delete PDP Context Response
- **信令抑制**
  PGW-C/GGSN收到用户的激活请求，与AAA Server、OCS、PCRF进行消息交互。如果PGW-C/GGSN上开启了信令抑制功能（ **[SET OVERLOADCTRL](../../../../../../OM参考/命令/UNC MML命令/业务服务管理/会话管理/接入管理/接入管理运维/流控管理/信令抑制/设置信令抑制功能开关以及老化定时器时长（SET OVERLOADCTRL）_82122531.md)** 命令的参数“ **CTRLENABLE** ”为“ENABLE”），当满足如下条件时，PGW-C/GGSN将用户的IMSI加入到信令抑制列表中。在 **[SET OVERLOADCTRL](../../../../../../OM参考/命令/UNC MML命令/业务服务管理/会话管理/接入管理/接入管理运维/流控管理/信令抑制/设置信令抑制功能开关以及老化定时器时长（SET OVERLOADCTRL）_82122531.md)** 命令的“ **AGINGTIME** ”参数配置的时间内，PGW-C/GGSN再次收到该用户的激活请求则直接回复激活失败。这样可以避免用户反复尝试激活给PGW-C/GGSN、AAA Server、OCS、PCRF带来的资源空耗和负担。信令抑制列表可以通过执行 **[CLR OVERLOAD](../../../../../../OM参考/命令/UNC MML命令/业务服务管理/会话管理/接入管理/接入管理运维/过载控制信息维护/清除过载控制记录信息（CLR OVERLOAD）_82242443.md)** 命令手动删除。
    - PGW-C/GGSN已过载（CPU占用率高于80%）。
    - AAA Server、OCS、PCRF的WAL值用尽导致用户激活失败。
  > **说明**
  > - VOLTE、紧急呼叫用户不加入信令抑制列表。
  > - AAA Server、OCS、PCRF无响应的情况不加入信令抑制列表。
  如果用户是先激活，即激活时未与OCS、计费AAA Server交互，激活后业务触发向OCS发送CCR-I消息、向计费AAA Server发送Accounting Start消息，但是收到失败响应，PGW-C/GGSN根据 **[SET OVERLOADCTRL](../../../../../../OM参考/命令/UNC MML命令/业务服务管理/会话管理/接入管理/接入管理运维/流控管理/信令抑制/设置信令抑制功能开关以及老化定时器时长（SET OVERLOADCTRL）_82122531.md)** 命令配置决定是否将用户的IMSI添加到信令抑制列表。
- **APN接入速率控制**
  控制APN的用户接入速率，保证APN下的用户平稳接入，不会因为此APN下的接入速率过快冲击PGW-C/GGSN周边网元或者影响其他APN的接入速率。
  通过 [**SET APNACCESSWAL**](../../../../../../OM参考/命令/UNC MML命令/业务服务管理/会话管理/接入管理/APN管理/APN接入速率/设置APN接入速率（SET APNACCESSWAL）_09652087.md) 命令配置APN下用户的接入速率。
  > **说明**
  > PGW-C/GGSN支持别名APN和虚拟APN的系统过载控制，在对别名APN和虚拟APN进行系统过载控制时，均先通过PGW-C/GGSN内部映射表，找到真实APN，然后遵照以上流程，进行流控。
- **GTPC接口固定速率流控**
  通过 **[SET GTPCFIXEDFC](../../../../../../OM参考/命令/UNC MML命令/业务服务管理/接口管理/GTP-C接口配置管理/GTP-C固定速率流控管理/设置指定消息类型固定速率流控信息（SET GTPCFIXEDFC）_35636465.md)** 命令中“ **MSGTYPE** ”参数选择GTPC接口流控的消息类型，然后通过“ **FCSWITCH** ”参数为“ **ON** ”来开启该消息类型的固定速率流控开关，配置参数“ **THRESHOLD** ”来设定该消息类型流控速率门限。当系统因MME向SGW-C发送的消息数或SGW-C向PGW-C发送的消息数超出命令 **[SET GTPCFIXEDFC](../../../../../../OM参考/命令/UNC MML命令/业务服务管理/接口管理/GTP-C接口配置管理/GTP-C固定速率流控管理/设置指定消息类型固定速率流控信息（SET GTPCFIXEDFC）_35636465.md)** 设置的“固定流控速率”阈值而导致被丢弃时，上报告警 **[ALM-100327 GTPC固定速率流控](../../../../../../网络运维/故障处理/UNC告警处理/业务告警/SMF&GW-C&GGSN/ALM-100327 GTP-C固定速率流控_14542095.md)** 。
