## [操作场景](#ZH-CN_OPI_0295923534)

当网络新增 UNC 设备或者 UNC 主机软件升级后，在完成数据对接调测和基本业务调测后，进行离线计费业务调测，如果出现CG（Charging gateway）故障导致话单不能正确发送到CG，需要调测恢复CG功能。

> **说明**
> 适用于SGW-C、PGW-C 、GGSN 。

## [必备事项](#ZH-CN_OPI_0295923534)

前提条件

- 请仔细阅读[Ga/Gy接口离线/在线计费](../../../../../业务专题/5G Core 计费解决方案/计费解决方案概述/计费原理/Ga_Gy接口离线_在线计费_87165686.md)。

## [操作步骤](#ZH-CN_OPI_0295923534)

1. 进入 “MML命令行-UNC” 窗口。 执行 **DSP CGSTATUS** 命令，查看CG状态是否正常。
    - 如果CG状态为normal，表明通信正常，则调测任务结束。
    - 如果CG状态为abnormal，请继续执行[步骤 2](#ZH-CN_OPI_0295923534__stp20)。
2. 执行命令 **PING** ，查看GGSN/SGW-C/PGW-C到CG的连接是否正常。
    - 如果收到对端网元的响应，表明连接正常，请执行[步骤 3](#ZH-CN_OPI_0295923534__stp4)。
    - 如果连接出现TimeOut，表明链路不通，请执行[步骤 4](#ZH-CN_OPI_0295923534__stp2)。
3. 执行 **LST CG** 命令，查看CG的配置与对端CG的配置是否一致。
    - 如果和对端CG配置一致，继续执行[步骤 6](#ZH-CN_OPI_0295923534__stp5)。
    - 如果和对端CG配置不一致，执行**RMV CG**命令删除不正确的CG配置，再执行**ADD CG**命令重新配置CG数据，配置完成后需要等待，等待时间为GGSN/SGW-C/PGW-C向CG发送Echo Request消息且收到CG的响应消息的时长（最长为1分钟），再执行[步骤 1](#ZH-CN_OPI_0295923534__stp1)，查看CG状态。
4. 检查GGSN/SGW-C/PGW-C到CG的互通是否正常。
    a. 根据使用的路由方式，调测GGSN/SGW-C/PGW-C到CG的路由。
    b. 根据使用的组网方式，调测GGSN/SGW-C/PGW-C到CG的组网。
    c. 再次执行 [步骤 1](#ZH-CN_OPI_0295923534__stp1) ，检查GGSN/SGW-C/PGW-C到CG的数据通信。
          - 如果正常，表明连接正常，调测结束。
          - 如果不正常，请执行[步骤 5](#ZH-CN_OPI_0295923534__stp50)。
5. 查看是否存在告警 **ALM-81021 CG无响应** 。
    - 如果产生告警，请参考**ALM-81021 CG无响应**的处理步骤解决。
    - 如果没有产生告警，请执行[步骤 6](#ZH-CN_OPI_0295923534__stp5)。
6. 创建Ga接口跟踪任务，准备抓取 测试终端 的出入报文。
  如 [图1](#ZH-CN_OPI_0295923534__fig158425853418) 所示，如消息中有GGSN/SGW-C/PGW-C Ga接口到CG的Echo Request消息，且CG到GGSN/SGW-C/PGW-C的Echo Response消息正确，则请查看抓包文件中是否有多个IP分片报文。
  **图1** 消息抓包

  <br>

  ![](调测到CG的数据_95923534.assets/zh-cn_image_0295923519_2.png "点击放大")
    - 如有多个IP分片报文，可能会造成CG响应不及时。此时GGSN/SGW-C/PGW-C发送 Data Record Transfer Request 消息后会再重发（重发次数默认为3，请参考 **SET CDRTRANSFER** 命令），如果超过重发次数，则导致CG链路abnormal。在Echo消息正常交互后，CG链路normal，这样便出现CG闪断。
      此时请确认CG服务器性能是否满足，或者GGSN/SGW-C/PGW-C上设置的数据包字节数是否合适，请执行 [步骤 7](#ZH-CN_OPI_0295923534__stp60) 。
    - 如果没有IP分片报文，请执行 [步骤 8](#ZH-CN_OPI_0295923534__step1829245425110) 。
7. 执行 **LST CDRTRANSFER** 命令，查看GTP’消息包含的最大字节数信息。
    - 如果最大字节数不等于1500，请修改最大字节数为1500，然后执行[步骤 1](#ZH-CN_OPI_0295923534__stp1)。
    - 如果最大字节数为1500，CG状态还不正常，请执行[步骤 8](#ZH-CN_OPI_0295923534__step1829245425110)。
8. 收集信息并寻求技术支持。
    a. 在OM Portal上创建用户跟踪任务，执行 [步骤 6](#ZH-CN_OPI_0295923534__stp5) 并保存报文。
    b. 在相应UNC窗口下执行 **EXP MML** 命令将当前配置数据导出为MML脚本文件并保存。
    c. 收集并保存上述所有查询信息。
    d. 收集归纳所有信息并联系华为技术支持解决。

## [附加信息](#ZH-CN_OPI_0295923534)

相关任务

**CG状态检测的机制**

1. GGSN/SGW-C/PGW-C在配置完CG数据后，会主动向CG发送 Node Alive Request 消息，如果没有收到CG的 Node Alive Response 响应消息，则重发，直到收到响应为止。
2. GGSN/SGW-C/PGW-C在配置完CG数据后，每隔一分钟会主动向CG发送Echo Request消息，如果收到CG的Echo Response消息，则CG状态正常，如果超过重发次数（参考**SET CDRTRANSFER**命令，其中RETRANSTIMES代表重发次数，默认为3）后没有收到Echo Response消息，则CG状态不正常。
3. GGSN/SGW-C/PGW-C产生话单后，如果CG状态正常，会将话单发送到CG；如果GGSN/SGW-C/PGW-C没有收到话单响应消息，则根据重发时间的间隔和重发次数发送话单（参考**SET CDRTRANSFER**命令），超过重发次数后，会产生CG状态不正常。
