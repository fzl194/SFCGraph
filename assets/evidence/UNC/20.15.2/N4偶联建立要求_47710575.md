# N4偶联建立要求

SMF和UPF能够为用户提供业务前，需要互相感知对方，并与对方建立关联（PFCP Association），在非Full Mesh组网下，UPF只与一个SMF建立N4连接，在Full Mesh组网下，要求UPF与所有互联的SMF建立N4连接，UPF最大可连接32个SMF。

**图1** N4连接差异对比

<br>

![](N4偶联建立要求_47710575.assets/zh-cn_image_0000001147590555_2.png "点击放大")

N4偶联建立/更新请求可以由SMF发起，也可以由UPF发起，通常由SMF来发现UPF，并向UPF创建关联。

**SMF** **发起的偶联建立/** **更新请求（默认）**

SMF通过本地配置发现UPF，发现方式与SMF/UPF是否为Full Mesh组网无关，属于通用流程。

如 [图2 SMF发起的开工流程](#ZH-CN_TOPIC_0000001147710575__fig15265171814912) 所示，以SMF通过本地配置发现UPF为例，每个SMF向所有互联的UPF发起偶联建立/更新请求，每个UPF支持接受来自多个SMF的Association Setup请求，记录多个SMF的信息，并将UPF的能力在Association Setup响应中携带给SMF，SMF记录所有UPF发送的UPF能力。

**图2** SMF发起的开工流程

<br>

![](N4偶联建立要求_47710575.assets/zh-cn_image_0000001101110782_2.png "点击放大")

1. SMF上配置U面的连接信息，以及本端的相关信息。
2. UPF上配置UE地址分配（可选）和F-TEIDU分配能力。
    - 本地已配置N4逻辑接口地址和数据面逻辑接口（Sa/Sc/Pa）地址。
    - 本地已配置对端信息（**ADD CPASSOCIATION**）。
    - UDG本地支持配置是否支持本地分配F-TEIDU（**SET CPTEIDUALLOC**），该配置不参与偶联建立条件判断。
3. SMF和UPF之间建立偶联。
    a. SMF1向UPF1发起PFCP Association Setup Request，请求获取UPF的UE地址分配和F-TEIDU分配能力，以便决策对应资源在SMF还是UPF分配。
    b. UPF1向SMF1回复PFCP Association Setup Response，告知SMF1 UPF是否支持UE地址分配和F-TEIDU分配能力。消息中携带UP Function Features信元，其FTUP标识位置位表示UPF具备F-TEIDU分配能力。
      如果UPF支持UE地址分配和F-TEIDU分配，则SMF根据配置决定是由SMF还是由UPF分配地址。后续UPF的F-TEIDU分配能力变化（例如本地关闭F-TEIDU分配能力），则会触发偶联更新消息（PFCP Association Update Request）通知SMF这一变化。修改F-TEIDU分配能力必须在无用户在线且锁定 UDG 的情况下进行。
4. SMF2与UPF1之间同步骤3.a-3.b完成PFCP Association关联。
5. SMF2与UPF2之间同步骤3.a-3.b完成PFCP Association关联。
6. SMF1与UPF2之间同步骤3.a-3.b完成PFCP Association关联。

UPF与SMF偶联建立成功后，UPF需要维护多条N4链路，针对每条链路维护对应的状态，并根据维护的链路信息周期性发送Heart beat消息，探测通信路径和对端设备是否工作正常。

**F-TEIDU地址分配流程**

F-TEIDU地址是用户激活时在PDU会话建立过程中被分配的，在整个PDU会话过程中固定唯一。 [图3](#ZH-CN_TOPIC_0000001147710575__fig1898820180423) 展示了PDU会话建立过程中用户面分配F-TEIDU的过程。

**图3** PDU会话建立过程中用户面分配F-TEIDU流程

<br>

![](N4偶联建立要求_47710575.assets/zh-cn_image_0000001151525641_2.png)

1. 用户激活向SMF发起会话建立请求。
2. SMF决策由SMF或UPF为UE分配F-TEIDU（若UPF上报支持F-TEIDU分配，则SMF必须指示由UPF分配F-TEIDU）。
3. SMF发起PFCP会话建立请求（PFCP Session Establishment Request），消息中携带Create PDR中的PDI信元或者Create Traffic Endpoint信元两种方式下发的Local F-TEID子信元，通过其CH BIT位（如[表1](#ZH-CN_TOPIC_0000001147710575__table12248151812381)和[图1](TEIDU分配要求_47590545.md#ZH-CN_TOPIC_0000001147590545__fig737372392913)所示）标识是否需要本地分配F-TEIDU，详细规则如[表2](#ZH-CN_TOPIC_0000001147710575__table1059943417228)所示。
  *表1 PFCP Session Establishment Request消息中携带Create PDR中的PDI信元或者Create Traffic Endpoint信元两种方式下发Local F-TEID子信元*

  | 关键信元 | 信元解释 |
  | --- | --- |
  | Create PDR<br>---PDI<br>------Local F-TEID | PFCP会话建立请求（PFCP Session Establishment Request）消息中携带的Local F-TEID子信元结构如左侧所示。3GPP协议定义该信元的CH BIT位置位为1标识指示U面分配F-TEIDU；如果为0则标识C面分配F-TEIDU。详细规则如<br>[表2](#ZH-CN_TOPIC_0000001147710575__table1059943417228)<br>所示。 |
  | Create Traffic Endpoint<br>---Local F-TEID | PFCP会话建立请求（PFCP Session Establishment Request）消息中携带的Local F-TEID子信元结构如左侧所示。3GPP协议定义该信元的CH BIT位置位为1标识指示U面分配F-TEIDU；如果为0则标识C面分配F-TEIDU。详细规则如<br>[表2](#ZH-CN_TOPIC_0000001147710575__table1059943417228)<br>所示。 |
4. UPF根据SMF的指示分配F-TEIDU，其分配F-TEIDU的逻辑如[表2](#ZH-CN_TOPIC_0000001147710575__table1059943417228)所示。
  *表2 UDG 分配F-TEIDU*

  | CH BIT位 | CHID BIT位 | V4 /V6 BIT位 | TEID字段 | IPv4/v6 address字段 | F-TEIDU分配方式 |
  | --- | --- | --- | --- | --- | --- |
  | 1 | 1 | 0 | 未携带 | 未携带 | UPF分配F-TEIDU，包含TEID字段和IPv4/IPv6字段。 |
  | 1 | 其余情况 | 其余情况 | 其余情况 | 其余情况 | 返回激活失败。 |
  | 0 | - | 1 | 携带 | 携带 | SMF分配F-TEIDU，包含TEID字段和IPv4/IPv6字段。 |
  | 0 | 其余情况 | 其余情况 | 其余情况 | 其余情况 | 返回激活失败。 |
5. UPF返回PFCP会话建立响应消息（PFCP Session Establishment Response），将分配的F-TEIDU通过消息中携带的Created PDR或Created Traffic Endpoint信元中的Local F-TEID子信元，返回给SMF。如果UDG分配F-TEIDU失败，则在响应消息中携带原因值指示SMF，如[表3](#ZH-CN_TOPIC_0000001147710575__table2886661163)所示。
  *表3 UDG 分配F-TEIDU失败原因值*

  | 异常场景 | 失败原因值 |
  | --- | --- |
  | SMF下发的会话建立请求中的Local F-TEID分配方式（CH BIT位）与<br>UDG<br>本地配置的F-TEID分配方式（<br>**SET CPTEIDUALLOC**<br>）不匹配。 | UDG<br>返回会话建立请求失败响应，消息中携带原因值Invalid F-TEID allocation option。 |
  | 如果<br>UDG<br>本地配置分配F-TEIDU，则不允许SMF下发F-TEIDU（CH BIT位为0），如果不符合描述则异常。 | UDG<br>返回会话建立请求失败响应，消息中携带原因值Invalid F-TEID allocation option。 |
  | SMF下发的会话建立请求中携带的Local F-TEID信元中的CH BIT位置位时，CHID BIT位应该同时置位，如果不符合描述则异常。 | UDG<br>返回会话建立请求失败响应，消息中携带原因值Rule Creation/Modification Failure。 |
  | SMF下发的会话建立请求中携带的Local F-TEID信元中的CH BIT位置位时，不应该携带TEID和IPv4/IPv6地址的字段，如果不符合描述则异常。 | UDG<br>返回会话建立请求失败响应，消息中携带原因值Rule Creation/Modification Failure。 |
6. SMF获取用户F-TEIDU后，给左侧网元返回用户激活响应消息。如果返回的是激活成功响应消息，则SMF将分配的用户面隧道信息F-TEIDU传递给无线侧，建立用户面隧道。
