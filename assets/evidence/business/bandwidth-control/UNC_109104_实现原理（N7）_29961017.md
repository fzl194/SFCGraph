# 实现原理（N7）

基于N7的会话级累计流量的策略控制流程如 [图1](#ZH-CN_TOPIC_0229961017__fig1025425943718) 所示。

**图1** 会话级累计流量的策略控制流程图

<br>

![](实现原理（N7）_29961017.assets/zh-cn_image_0230027733_2.png)

1. UE发起PDU Session Establishment流程。
2. AMF进行PGW-C/SMF选择，然后向PGW-C/SMF发送Nsmf_PDUSession_CreateSMContext消息。
3. PGW-C/SMF收到Nsmf_PDUSession_CreateSMContext消息后，再发送Npcf_SMPolicyControl_Create Request (SmPolicyContextData)请求会话策略，其中suppFeat字段指示是否支持Usage Monitoring Control功能。
4. PCF从UDR获取策略签约数据。
5. PCF进行特性协商，同时生成会话管理策略（包括FUP相关策略）。PCF给PGW-C/SMF回复Npcf_SMPolicyControl_Create Response消息，消息中携带SmPolicyDecision信元组，其中包含会话级流量阈值，会话级累计流量等内容。SmPolicyDecision包含的主要信元如[表1](#ZH-CN_TOPIC_0229961017__table12902144718429)所示：
  *表1 SmPolicyDecision包含的主要信元*

  | 主信元 | 子信元 | 信元解释 |
  | --- | --- | --- |
  | sessRules | - | SessionRule。 |
  | sessRules | refUmData | 对使用监控数据决策的引用。即会话级累计流量在SessionRule中定义，通过umID引用。 |
  | umDecs | - | DecisionData。 |
  | umDecs | umId | 标识使用监控策略数据在一个PDU会话。 |
  | umDecs | volumeThreshold | 总流量阈值。 |
  | umDecs | volumeThresholdUplink | 上行流量门限。 |
  | umDecs | volumeThresholdDownlink | 下行流量门限。 |
  | policyCtrlReqTrigger | - | 标识达到流量阈值后上报的触发条件，如US_RE。 |
  | suppFeat | - | 标识特性协商结果，如UMC。 |
6. PGW-C/SMF根据Npcf_SMPolicyControl_Create Response消息中sessRule和umDecs等信元信息生成N4接口的URR规则，由于sessRule对整个会话有效，该URR需要关联到所有Create PDRs。同时PGW-C/SMF建立URR ID和UMID之间的映射。
7. PGW-C/SMF向PGW-U/UPF发送PFCP Session Establishment Request消息，请求建立PFCP会话。消息中携带Create URR信元组，指示流量阈值相关信息。Create URR包含的主要子信元如[表2](#ZH-CN_TOPIC_0229961017__table8448632155511)所示：
  *表2 Create URR包含的主要子信元*

  | 子信元 | 信元解释 |
  | --- | --- |
  | URR ID | 唯一标识一个Create URR。 |
  | Measurement Method | 标识计费方式，包括流量计费、时长计费或事件计费。 |
  | Reporting Trigger | 标识计费上报的触发条件，包含Periodic Reporting（定期上报）、Volume Quota（流量配额）、Time Quota（时间配额）、Linked Usage Reporting（关联URR）等。 |
  | Volume Threshold | 总流量阈值 |
8. PGW-C/SMF收到PFCP Session Establishment Response消息，PFCP会话建立成功。
9. 用户访问数据业务时，PGW-U/UPF检测到用户的流量阈值耗尽，发送PFCP Session Report Request消息通知PGW-C/SMF，消息中携带Usage Report信元组，记录用户使用的流量、时间等信息。Usage Report包含的主要子信元如[表3](#ZH-CN_TOPIC_0229961017__table54623294578)所示：
  *表3 Usage Report包含的主要子信元*

  | 子信元 | 信元解释 |
  | --- | --- |
  | URR ID | 标识上报的URR。 |
  | Usage Report Trigger | 标识URR的Trigger。当信元是VOLTH时，标识流量阈值耗尽 。 |
  | Volume Measurement | 标识流量使用情况，包含总流量、上行流量和下行流量。 |
10. PGW-C/SMF根据URR ID和UMID之间的映射，向PCF通知流量阈值耗尽，业务累计流量上报信息从N4接口中Usage Report中提取。
11. PGW-C/SMF发送Npcf_SMPolicyControl_Update Request (SmPolicyUpdateContextData)消息通知PCF流量阈值耗尽。消息中携带accuUsageReports信元组，accuUsageReports包含的主要子信元如[表4](#ZH-CN_TOPIC_0229961017__table219314202592)所示：
  *表4 accuUsageReports包含的主要子信元*

  | 子信元 | 信元解释 |
  | --- | --- |
  | refUmId | UMID。 |
  | volUsage | 累计流量信息是从N4 Usage Report中获得。 |
12. PCF进行策略决策，决定下发新的阈值，然后PGW-C/SMF响应Npcf_SMPolicyControl_Update Response消息下发新的阈值。
13. PGW-C/SMF收到Npcf_SMPolicyControl_Update Response消息后，检查是否下发新的流量阈值。如果有则更新相应的URR规则。
14. PGW-C/SMF发送PFCP Session Modification Request消息，消息中携带Update URR信元组，向PGW-U/UPF下发新的阈值。Update URR包含的主要子信元如[表5](#ZH-CN_TOPIC_0229961017__table8410155514014)所示：
  *表5 Update URR包含的主要子信元*

  | 子信元 | 信元解释 |
  | --- | --- |
  | URR ID | 标识上报的URR。 |
  | Measurement Method | 标识计费方式，信元值是VOLUM。 |
  | Reporting Trigger | 标识计费上报的触发条件，信元值是VOLTH 。 |
  | Volume Threshold | 新的流量阈值。 |
15. 用户继续访问数据业务，PGW-U/UPF检测到用户的流量阈值再次耗尽时，发送PFCP Session Report Request消息通知PGW-C/SMF。
16. PGW-C/SMF再次发送Npcf_SMPolicyControl_Update Request消息通知PCF流量阈值耗尽。
17. PCF进行策略决策，不再下发流量阈值，同时更新QoS规则，对用户进行带宽控制，上行带宽为1 Mbit/s，下行带宽为2 Mbit/s。Npcf_SMPolicyControl_Update Response消息中携带的authSessAmbr信元组负责更新QoS规则。
18. PGW-C/SMF收到Npcf_SMPolicyControl_Update Response消息后，检查是否下发新的流量阈值。如果没有下发新的流量阈值，则通知PGW-U/UPF不再上报流量。
19. PGW-C/SMF发送PFCP Session Modification Request消息，通知PGW-U/UPF更新业务规则。

**图2** 会话释放时累计流量上报流程图

<br>

![](实现原理（N7）_29961017.assets/zh-cn_image_0230028078_2.png)

1. UE发起PDU Session Establishment流程。
2. AMF进行PGW-C/SMF选择，然后向PGW-C/SMF发送Nsmf_PDUSession_CreateSMContext消息。
3. PGW-C/SMF收到Nsmf_PDUSession_CreateSMContext消息后，进行会话管理相关处理，然后PGW-C/SMF发送Npcf_SMPolicyControl_Create Request请求会话策略。
4. PCF从UDR获取策略签约数据。
5. PCF进行特性协商，同时进行策略决策生成会话管理策略，包括FUP相关策略；然后PCF回复Npcf_SMPolicyControl_Create Response消息，消息中下发会话级流量阈值。
6. PGW-C/SMF向PGW-U/UPF发送PFCP Session Establishment Request消息，下发会话级流量阈值。
7. PGW-C/SMF收到PFCP Session Establishment Response消息后进行后续相关流程处理，PDU会话建立成功。
8. 用户发起PDU会话释放流程。
9. AMF向PGW-C/SMF发送Nsmf_PDUSession_UpdateSMContext消息。
10. PGW-C/SMF执行PDU会话释放流程，向PGW-U/UPF发送PFCP Session Delete Request消息释放N4会话。
11. 由于URR规则处于激活状态，PGW-U/UPF在PFCP Session Delete Response消息中携带Usage Report信元组，信元组记录了上报累计使用流量。Usage Report包含的主要子信元如[表6](#ZH-CN_TOPIC_0229961017__table11774561489)所示：
  *表6 Usage Report包含的主要子信元*

  | 子信元 | 信元解释 |
  | --- | --- |
  | URR ID | 标识上报的URR。 |
  | Usage Report Trigger | 标识URR的Trigger。当值是TERMR时，终止PFCP Session上报流量 ，且删除会话策略。 |
  | Volume Measurement | 标识流量使用情况，实际使用的累计流量，且未达到阈值。 |
12. PCF进行策略决策，决定下发新的阈值，然后PGW-C/SMF响应Npcf_SMPolicyControl_Update Response消息下发新的阈值。
13. PCF回复响应消息Npcf_SMPolicyControl_Delete Response给PGW-C/SMF。
