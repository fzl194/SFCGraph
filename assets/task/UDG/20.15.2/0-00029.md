# 配置业务属性（ADD SERVICEPROP）

> 命令静态知识见 [ADD SERVICEPROP](command/UDG/20.15.2/ADD-SERVICEPROP.md)。本页只讲配置生成实例化时这条命令**怎么配**——按配置维度组织成"字典"，**不**逐特性罗列取值。

## 配置方法

ADD SERVICEPROP 是 PCC 业务属性链的"业务标识定义层"：把"业务属性名 + 业务标识 ID + 优先级"三元组用一条记录描述，产出对象 SRVPROPNAME 供 ADD SRVPBINDPCCPG(0-00030) 引用。命令核心配置空间仅 1 个维度：**三元组完整性**（SRVPROPNAME + SERVICEID + PRIORITY 三者均必选，且三者各自全局唯一）。

### 配置维度 1：业务属性三元组（SRVPROPNAME + SERVICEID + PRIORITY，均必选 + 各自全局唯一）

| 参数 | 取值范围 | 数据来源 | 唯一性 | 备注 |
|---|---|---|---|---|
| **SRVPROPNAME**（业务属性名称） | 字符串 1~63，不支持空格，不区分大小写 | 全网规划 | 全局唯一 | 引用键 |
| **SERVICEID**（业务标识） | 整数 1~4294967295 | 全网规划 | 全局唯一 | 网络侧业务识别号 |
| **PRIORITY**（优先级） | 整数 1~4294967295 | 全网规划 | 全局唯一；**数值越小优先级越高** | 多个 SRVP 时的匹配优先级 |

**典型脚本（020352 IPv6 SA 阻塞业务）**：
```
ADD SERVICEPROP:SRVPROPNAME="sp_test",SERVICEID=1,PRIORITY=1;
```

> 命令 notes：最大记录数 1000；适用 NF：PGW-U、UPF；执行后立即生效；CFGDOMAINNAME 冷门不配。

**步骤位置**：在所属 feature 编排中通常位于"步骤 N：业务属性定义"——前置无；产出 SRVPROPNAME 被 ADD SRVPBINDPCCPG(0-00030) 的 SRVPROPNAME 引用。

## 决策点

（无。本命令三元组语义固定，唯一性约束是配置生成自检的核心，详见约束段。）

## 约束

- **三元各自全局唯一**（critical，rule-0-00329）：SRVPROPNAME / SERVICEID / PRIORITY 三者均各自全局唯一（不同业务属性之间互不重复） — 后果：任一重复冲突 ADD 失败
- **PRIORITY 数值语义**（warning，rule-0-00329）：PRIORITY 数值越小优先级越高（与 SPRINT 等常见"越大越优先"语义相反） — 后果：误判优先级方向导致 PCC 匹配错乱
- **须先于 SRVPBINDPCCPG 配置**（critical，rule-0-00329）：SRVPROPNAME 必须先于 ADD SRVPBINDPCCPG(0-00030) 建成 — 后果：未建引用失败
- **最大记录数 1000**（warning，rule-0-00329）：单板最大 1000 条 — 后果：超规格 ADD 失败
- **适用 NF 限定**（warning，隐含）：PGW-U、UPF — 后果：SGW-U 等不适用
