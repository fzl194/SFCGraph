# 业务触发的QoS保证（GWFD-020358）

> 特性静态知识见 [GWFD-020358](feature/UDG/20.15.2/GWFD-020358.md)。本页讲配置生成实例化时这个特性**怎么配**。本特性是 **QoS 族基准**（业务触发专有承载），复用 [1-00017](task/UDG/20.15.2/1-00017.md) QoS专有承载链；[2-00030](task/UDG/20.15.2/2-00030.md) 视频承载信令控制复用本骨架（差异在 QOSPROP.DECOUPLINGSW）。

## 配置概览

业务触发的QoS保证：用户请求需要 QoS 保障的业务时，UDG 基于匹配的预定义规则向 PGW-C/SMF 上报 QoS 事件，PGW-C/SMF 发起专有承载激活/专有 QoS Flow 建立。核心对象链：`URR(QOS) + URRGROUP + PCCPOLICYGRP(URRGROUPNAME+QOSPROPNAME) + QOSPROP(GBR/MBR→QOSURRNAME)` → 过滤链 → `SADEDICBEARER(PROTOCOL, DOWNLINK_ONLY)` → `RULE(PCC) + USERPROFILE + RULEBINDING` → `SET UPGLBPMPARA(FLOWDANYFMT)`。

两种触发方式（两份 activation）：
- **七层触发**（[配置七层业务触发的QOS保证](evidence/UDG/20.15.2/配置七层业务触发的QOS保证（PGW-U、UPF）_80361858.md)）：按 SA 识别出的协议触发专有承载——过滤链含 L7FILTER+PROTBINDFLOWF + WELLKNOWNPORT + SADEDICBEARER（PROTOCOLLEVEL=PROTOCOL）。11 步。
- **三四层触发**（[配置三四层业务触发的QoS保证](evidence/UDG/20.15.2/配置三四层业务触发的QoS保证（PGW-U、UPF）_80318438.md)）：按三四层报文匹配触发专有承载——过滤链仅 FILTER+FLTBINDFLOWF，**无 L7FILTER/PROTBINDFLOWF/WELLKNOWNPORT/SADEDICBEARER**。8 步（含可选 BIT1104）。

> 两方式差异不在脚本结构（前段过滤链 + QoS 属性 + PCCPOLICYGRP + 规则绑定收尾一致），而在**过滤匹配层 + 是否需 SA 触发专有承载命令**。

## 配置流程（步骤编排，每步显示含命令）

> **下表按七层触发 11 步编排**（命令最全）。三四层触发走法见决策点 DP-1。

1. **License**（单命令）：`SET LICENSESWITCH`（`LICITEM=LKV3G5STQE01,SWITCH=ENABLE`）→ [0-00019](task/UDG/20.15.2/0-00019.md)
2. **过滤链**（业务匹配）：`ADD FILTER`（三四层 any）+ `ADD L7FILTER`（七层 URL）+ `ADD FLOWFILTER` + `ADD FLTBINDFLOWF`（三四层绑定）+ `ADD PROTBINDFLOWF`（七层绑定，PROTOCOLNAME=http）→ [1-00009](task/UDG/20.15.2/1-00009.md)
   - 三四层触发变体：省去 L7FILTER + PROTBINDFLOWF（仅 FILTER+FLOWFILTER+FLTBINDFLOWF）
3. **刷新生效**（单命令，承接过滤链置生效）：`SET REFRESHSRV`（`REFRESHTYPE=USERPROFILE`）→ [0-00015](task/UDG/20.15.2/0-00015.md)
4. **QoS 属性**（绑 QoS URR + GBR/MBR）：`ADD URR`（`USAGERPTMODE=QOS`）+ `ADD QOSPROP`（`QOSPROPNAME + QOSURRNAME + GBRUPLKVALUE/GBRDNLKVALUE + MBRUPLKVALUE/MBRDNLKVALUE`）→ [1-00017](task/UDG/20.15.2/1-00017.md)（QOSPROP 段）
5. **PCC 策略组**（计费三件套 QoS 变体，绑 URRGROUP+QOSPROPNAME）：`ADD URRGROUP` + `ADD PCCPOLICYGRP`（`PCCPOLICYGRPNM + URRGROUPNAME + QOSPROPNAME`）→ [1-00010](task/UDG/20.15.2/1-00010.md)
6. **知名端口**（七层触发专属，加速协议识别）：`ADD WELLKNOWNPORT`（`IDENPROTNAME + PROTOCOLNAME + PORTOP=EQUAL + STARTPORT + PRIORITY`）→ [1-00016](task/UDG/20.15.2/1-00016.md)（WELLKNOWNPORT 段）
   - 三四层触发变体：**无此步**（三四层不识别七层协议）
7. **SA 触发专有承载**（七层触发专属，TRIGGERMODE）：`ADD SADEDICBEARER`（`PROTOCOLLEVEL=PROTOCOL + PROTOCOLNAME + TRIGGERMODE=DOWNLINK_ONLY`）→ [1-00017](task/UDG/20.15.2/1-00017.md)（SADEDICBEARER 段）
   - 三四层触发变体：**无此步**（三四层按报文匹配触发，不经 SA 专有承载命令）
8. **规则与用户模板绑定**：`ADD RULE`（`RULENAME + POLICYTYPE=PCC + FILTERINGMODE=FLOWFILTER + FLOWFILTERNAME + PRIORITY + POLICYNAME=PCCPOLICYGRPNM`）+ `ADD USERPROFILE` + `ADD RULEBINDING` → [1-00011](task/UDG/20.15.2/1-00011.md)
9. **双栈专有承载收尾**（单命令）：`SET UPGLBPMPARA`（`FLOWDANYFMT=AnyFMT`，双栈地址+AnyToAny 规则场景需 IPMASK）→ [0-00297](task/UDG/20.15.2/0-00297.md)
10. **三四层触发可选软参**（三四层专属）：`BIT1104`（控制 L34 业务触发 QoS Flow 创建时 PFCP Session Report Response 行为，三四层 activation §步骤8 标注可选）→ 仅三四层触发场景考虑，七层无此步

> **参数证据链**（R1.5，配置流程脚注）：以上命令+参数均来自两份 activation §数据规划表 + §任务示例脚本（逐条对应）。
> - SET LICENSESWITCH（LKV3G5STQE01,SWITCH=ENABLE）：两份 activation §任务示例脚本（原样一致）
> - ADD FILTER（filter_test / L34PROTTYPE=STRING / L34PROTOCOL=ANY）：两份 activation §数据规划表 ADD FILTER 段 + §任务示例脚本（原样）
> - ADD L7FILTER（l7_test / SUBL7FLTNAME=subl7_test / URL=www.example.com*/*）：仅七层 activation §数据规划表 + §脚本（三四层无）
> - ADD FLOWFILTER（flowfilter_test1）：两份 activation §数据规划表 + §脚本（三四层脚本多 TETHERDETFLAG=DETECT_ONLY，七层无此参数）
> - ADD FLTBINDFLOWF（flowfilter_test1 + filter_test）：两份 activation §数据规划表 + §脚本（原样）
> - ADD PROTBINDFLOWF（flowfilter_test1 + PROTOCOLNAME=http + L7FILTERNAME=l7_test）：仅七层 activation §数据规划表 + §脚本（三四层无）
> - SET REFRESHSRV（REFRESHTYPE=USERPROFILE）：两份 activation §脚本（原样一致，非 ALL——过滤链配完即刷新 USERPROFILE）
> - ADD URR（urr_1 / URRID=1001 / USAGERPTMODE=QOS）：两份 activation §数据规划表 ADD URR 段 + §脚本（原样，USAGERPTMODE=QOS 固定取值）
> - ADD QOSPROP（qos-property1 / QOSURRNAME=urr_1 / GBRUPLKVALUE=110 / GBRDNLKVALUE=110 / MBRUPLKVALUE=220 / MBRDNLKVALUE=220）：两份 activation §数据规划表 ADD QOSPROP 段 + §脚本（原样一致）
> - ADD URRGROUP（urrgroup1 / UPURRNAME1=urr_2 / DOWNURRNAME1=urr_2）：两份 activation §数据规划表 ADD URRGROUP 段（注：URRGROUP 绑的 UP/DOWNURRNAME1=urr_2，与 QoS URR urr_1 不同名——见约束"URRGROUP 绑的 URR 名"）
> - ADD PCCPOLICYGRP（ppg1 / URRGROUPNAME=urrgroup1 / QOSPROPNAME=qos-property1）：两份 activation §数据规划表 + §脚本（原样一致）
> - ADD WELLKNOWNPORT（IDENPROTNAME=http / PROTOCOLNAME=http / PORTOP=EQUAL / STARTPORT=80 / PRIORITY=5）：仅七层 activation §数据规划表 + §脚本（三四层无）
> - ADD SADEDICBEARER（PROTOCOLLEVEL=PROTOCOL / PROTOCOLNAME=http / TRIGGERMODE=DOWNLINK_ONLY）：仅七层 activation §数据规划表 + §脚本（三四层无）

## 决策点

**DP-1（特性级）：触发方式（七层 vs 三四层）+ SADEDICBEARER.TRIGGERMODE**（影响过滤链走法 + 是否需 WELLKNOWNPORT/SADEDICBEARER + 收尾软参）。维度按两份 activation §操作场景 + §操作步骤全列。

| 场景 | 过滤匹配 | WELLKNOWNPORT | SADEDICBEARER | 收尾软参 | 走法 |
|---|---|---|---|---|---|
| **七层触发（activation 演示 HTTP 下行）** | 三四层 any（FILTER）+ 七层（L7FILTER + PROTBINDFLOWF，PROTOCOLNAME=http）| **有**（IDENPROTNAME=http/STARTPORT=80，加速识别）| **有**（PROTOCOLLEVEL=PROTOCOL/PROTOCOLNAME=http/TRIGGERMODE=DOWNLINK_ONLY）| 无 BIT1104 | 骨架 + [1-00009](task/UDG/20.15.2/1-00009.md)（含 L7 段）+ [1-00016](task/UDG/20.15.2/1-00016.md)（WELLKNOWNPORT）+ [1-00017](task/UDG/20.15.2/1-00017.md)（SADEDICBEARER）|
| 三四层触发（activation 演示三四层专有承载创建）| 仅三四层 any（FILTER，无 L7FILTER/PROTBINDFLOWF）| **无** | **无**（三四层不经 SA 触发，按报文匹配触发）| **可选 BIT1104**（L34 触发 QoS Flow 创建时 PFCP Session Report Response 行为）| 骨架 + [1-00009](task/UDG/20.15.2/1-00009.md)（仅 L34 段）+ [1-00017](task/UDG/20.15.2/1-00017.md)（仅 QOSPROP 段，无 SADEDICBEARER）|

**SADEDICBEARER.TRIGGERMODE 子选项**（联动七层触发场景）：

| TRIGGERMODE | 含义 | 数据来源 |
|---|---|---|
| DOWNLINK_ONLY（activation 演示）| 下行报文触发专有承载创建（UDG 创建 rule-filter-type=l7-downlink 模板，所有用户下行报文汇聚到该承载，上行仍走原承载）| 七层 activation §任务示例脚本 |
| UPLINK_ONLY | 上行报文触发专有承载创建 | 命令 wiki（activation 未演示，零证据差异，配置生成按 DOWNLINK_ONLY 产出）|
| BOTH | 上下行均触发 | 命令 wiki（同上，未演示）|
| NOT_TRIGGER（默认）| 不触发（需显式配为 DOWNLINK_ONLY/UPLINK_ONLY/BOTH 才生效）| 命令 wiki |

**联动**：
- 触发方式 → [1-00009](task/UDG/20.15.2/1-00009.md) 过滤链：七层加 L7FILTER+PROTBINDFLOWF；三四层仅 FILTER。
- 触发方式 → [1-00016](task/UDG/20.15.2/1-00016.md) WELLKNOWNPORT：仅七层需（加速协议识别）。
- 触发方式 → [1-00017](task/UDG/20.15.2/1-00017.md) QoS专有承载链：七层含 SADEDICBEARER（TRIGGERMODE）；三四层仅 QOSPROP 段（无 SADEDICBEARER）。
- SADEDICBEARER.TRIGGERMODE → 专有承载触发方向（DOWNLINK_ONLY/UPLINK_ONLY/BOTH），命令 wiki 提供三选项，activation 仅演示 DOWNLINK_ONLY。

## 约束（特性级 rule）

- **License 前置**（critical）：SET LICENSESWITCH 开启 `LKV3G5STQE01` — 未开则 QoS 保证功能不生效。
- **依赖 SA-Basic + PCC**（critical）：[GWFD-110101](feature/UDG/20.15.2/GWFD-110101.md) SA-Basic（业务识别，七层触发必需）+ [GWFD-020351](feature/UDG/20.15.2/GWFD-020351.md) PCC基本功能（从 PCRF/PCF 获取规则）— 缺一则 QoS 保证不工作。
- **QoS URR 上报模式固定 QOS**（critical）：ADD URR 须 `USAGERPTMODE=QOS`（固定取值，两份 activation §数据规划表明示）— 非 QOS 则 QoS 事件上报失效。
- **QOSPROP.QOSURRNAME 引用一致**（critical）：QOSURRNAME 须引用已配的 QoS URR（USAGERPTMODE=QOS）— 不一致则 QoS 属性引用断链（见 [1-00017](task/UDG/20.15.2/1-00017.md)）。
- **PCCPOLICYGRP 绑 QOSPROPNAME**（critical）：PCCPOLICYGRP 须同时绑 URRGROUPNAME + QOSPROPNAME — 漏绑则 QoS 保证不生效（见 [1-00017](task/UDG/20.15.2/1-00017.md)）。
- **URRGROUP 绑的 URR 名**（warning）：URRGROUP.UPURRNAME1/DOWNURRNAME1 在 activation 数据规划表取值 urr_2（与 QoS URR urr_1 不同名）—— activation 数据规划表标注"已配置数据中获取"，实际部署时 urr_2 须为已配 URR。配置生成按"URRGROUP 绑一个 URR（可同 urr_1 或另配 urr_2）"产出，以规划为准。
- **GBR ≤ MBR**（warning）：保证比特率须 ≤ 最大比特率（activation 演示 GBR=110/MBR=220）— 否则 QoS 无法保证。
- **七层触发依赖协议识别**（critical）：SADEDICBEARER.PROTOCOLLEVEL=PROTOCOL 时需 SA 识别协议（[1-00016](task/UDG/20.15.2/1-00016.md) WELLKNOWNPORT + 过滤链 PROTBINDFLOWF）— 未识别则专有承载不触发。
- **并发链路流 ≤ 8**（info）：为避免信令风暴，只有并发链路流小于 8 个协议或协议组才适合配置为基于流触发专有承载（特性 wiki §应用限制）。
- **仅支持 HTTP1.x**（info）：本特性不支持 HTTP2.0 协议、不支持加密场景基于 host 的计费和控制（特性 wiki §应用限制）。
- **双栈需 IPMASK**（warning）：会话激活使用双栈地址 + AnyToAny 规则触发专有承载时，SET UPGLBPMPARA.FLOWDANYFMT 须配 IPMASK — activation 演示 AnyFMT（单栈场景），双栈待补（见 [0-00297](task/UDG/20.15.2/0-00297.md)）。
- **互斥特性**（info）：与 [GWFD-020501](feature/UDG/20.15.2/GWFD-020501.md) ULCL分流、[GWFD-020531](feature/UDG/20.15.2/GWFD-020531.md) 专网业务触发专载不能同时开启。
