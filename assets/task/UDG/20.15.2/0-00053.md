# 配置PFCP路径相关属性（SET UPPFCPPATH）

> 命令静态知识见 [SET UPPFCPPATH](command/UDG/20.15.2/SET-UPPFCPPATH.md)。本页只讲配置生成实例化时这条命令**怎么配**——按配置维度组织成"字典"，**不**逐特性罗列取值。**仅作用于 PFCP 协议路径**（N4/Sxa/Sxb GTP 路径由 [0-00051](task/UDG/20.15.2/0-00051.md) SET UPGTPPATH、N4 接口由 [0-00052](task/UDG/20.15.2/0-00052.md) SET UPN4UPATH 配置）。

## 配置方法（配置方法字典）

SET UPPFCPPATH 是**全局唯一**（最大记录数 1）的"PFCP 协议路径探测参数配置"命令。核心配置空间有 3 个维度：**心跳间隔**（HEARTBEATINTERVAL）、**重发参数**（T3RESPONSE + N3REQUEST）、**去活策略**（DEACTIVEFLAG 门控 + USERTYPE + HEARTBEATTIME 三参联动）。1 个特性命中（GWFD-010102 路径管理）覆盖全量脚本 + 仅改 T3 脚本。

### 配置维度 1：心跳间隔（参数 HEARTBEATINTERVAL，可选）

| 参数 | 取值范围 | 作用 |
|---|---|---|
| **HEARTBEATINTERVAL** | 60~3600 秒 | 发送 PFCP 心跳请求的间隔时间 |

> 系统初始值 60 秒；**设置后只对新激活用户生效**（已有用户不生效）。基站数目较多时建议配较大 HEARTBEATINTERVAL（防单位时间内 HEARTBEAT 过多致 CPU 上升）。

### 配置维度 2：重发参数（参数 T3RESPONSE/N3REQUEST，可选）

| 参数 | 取值范围 | 作用 |
|---|---|---|
| **T3RESPONSE** | 1~20 秒 | PFCP 请求消息的重发时间间隔 |
| **N3REQUEST** | 1~6 次 | PFCP 请求消息的最大尝试发送次数 |

> 系统初始值 T3RESPONSE=3、N3REQUEST=5。**注意**：对接 SMF 上 `SET T1N1PARA` 的 T1 + 5 秒的时长之和**不能**是 SET UPPFCPPATH 的 T3RESPONSE 时长的整数倍——否则 MME 故障场景下系统重传 PFCP Session Report Request 消息可能导致 SMF 向 MME 发送冗余的 DDN 消息。

### 配置维度 3：去活策略（参数 DEACTIVEFLAG + USERTYPE + HEARTBEATTIME）

DEACTIVEFLAG 是门控开关，决定是否启用"路径断告警 → 去激活"的链路：

| 参数 | 必选性 | 取值/范围 | 作用 |
|---|---|---|---|
| **DEACTIVEFLAG** | 可选 | `ENABLE` / `DISABLE` | 路径断告警产生后是否去激活该路径上已激活的上下文 |
| **USERTYPE** | DEACTIVEFLAG=ENABLE 时条件可选 | `NORMAL`（普通用户）/ `MBS`（MBS 广播会话）/ `MBMS`（MBMS 组播会话） | 配置去活的用户类型 |
| **HEARTBEATTIME** | DEACTIVEFLAG=ENABLE 时**条件必选** | 0~60 次 | 路径断告警产生后，执行去激活操作之前重复发送心跳消息的次数；**0 表示路径断告警后不再发心跳但仍去活**（当 DEACTIVEFLAG=ENABLE） |

> **DEACTIVEFLAG=ENABLE 时 HEARTBEATTIME 必须配置**（命令拒收）；USERTYPE 可不配（按需细化去活范围）。系统初始值 DEACTIVEFLAG=ENABLE、HBTIME=15、USERTYPE=MBS&NORMAL。

**典型脚本（010102 全量配置，HEARTBEATINTERVAL=60 + T3=3/N3=5 + 去活策略全开）**：
```
SET UPPFCPPATH:HEARTBEATINTERVAL=60,T3RESPONSE=3,N3REQUEST=5,DEACTIVEFLAG=ENABLE,HEARTBEATTIME=30;
```

**典型脚本（仅改重发参数，T3RESPONSE=10）**：
```
SET UPPFCPPATH: T3RESPONSE=10;
```

> 命令 notes：适用 NF：SGW-U、PGW-U、UPF；本命令属高危命令（配置不合理可能导致告警或用户去活）；**执行后只对新激活用户生效**；**全局唯一最大记录数 1**；与 SMF 的 `SET T1N1PARA` 的 T1+5 秒时长之和不能是本命令 T3RESPONSE 的整数倍；**仅作用于 PFCP 接口**（N4 GTP 路径由 [0-00051](task/UDG/20.15.2/0-00051.md) SET UPGTPPATH、N4 接口由 [0-00052](task/UDG/20.15.2/0-00052.md) SET UPN4UPATH 配置）。

**步骤位置**：在所属 feature 编排中通常位于"步骤 1：PFCP 协议路径探测参数"——前置无；与 [0-00051](task/UDG/20.15.2/0-00051.md)（GTP 路径）、[0-00052](task/UDG/20.15.2/0-00052.md)（N4 接口路径）按协议类型分别配置。

## 决策点：去活策略开关（DP 0-00053）

问：SET UPPFCPPATH 选哪个去活策略组合？

| 选项 | 联动参数 | 适用场景 |
|---|---|---|
| 启用去活（全参数） | DEACTIVEFLAG=ENABLE + HEARTBEATTIME=30 + 可选 USERTYPE | PFCP 接口路径管理（010102 范式：路径断告警后去活） |
| 仅改重发参数 | 仅 T3RESPONSE/N3REQUEST | 修改 PFCP 请求重发参数（010102 修改范式） |
| 关闭去活 | DEACTIVEFLAG=DISABLE | 不启用去活（命令支持，010102 未演示） |

> 1 个特性命中（010102）覆盖全量+仅改重发 2 个脚本；HEARTBEATTIME 在不同 md 实例值不同（010102 配置 md=30/初始值=15），按规划值取。

## 约束

- **全局唯一最大记录数 1**（critical，rule-0-00453）：本命令全局唯一 — 后果：重复配置命令拒收
- **高危命令**（critical，rule-0-00453）：配置不合理可能导致告警或用户去活，建议根据现网规划设置 — 后果：高危配置错误致告警/用户去活
- **仅对新激活用户生效**（critical，rule-0-00453）：命令执行后只对新激活用户生效，已有用户不生效 — 后果：误以为已生效但存量用户不变
- **DEACTIVEFLAG=ENABLE 必带 HEARTBEATTIME**（critical，rule-0-00453）：DEACTIVEFLAG=ENABLE 时 HEARTBEATTIME 为条件必选参数 — 后果：缺 HEARTBEATTIME 命令拒收
- **T1+5 与 T3RESPONSE 整数倍禁忌**（critical，rule-0-00453）：对接 SMF 上 SET T1N1PARA 的 T1 + 5 秒时长之和不能是 SET UPPFCPPATH 的 T3RESPONSE 时长的整数倍 — 后果：MME 故障下系统重传 PFCP Session Report Request 致 SMF 向 MME 发冗余 DDN
- **HEARTBEATTIME=0 语义**（warning，rule-0-00453）：0 表示路径断告警后不再发心跳但 DEACTIVEFLAG=ENABLE 仍会去活 — 后果：HEARTBEATTIME=0 配合 ENABLE 仍触发去活
- **基站多时建议大 HEARTBEATINTERVAL**（warning，rule-0-00453）：基站数目较多时建议配较大 HEARTBEATINTERVAL（防单位时间内 HEARTBEAT 过多致 CPU 上升） — 后果：HEARTBEATINTERVAL 过小致 CPU 上升
- **适用 NF 限定**（warning，隐含）：SGW-U、PGW-U、UPF — 后果：其他 NF 不适用
- **仅作用于 PFCP 接口**（warning，隐含）：非 PFCP 协议路径由 [0-00051](task/UDG/20.15.2/0-00051.md)（SET UPGTPPATH，GTP 路径）、[0-00052](task/UDG/20.15.2/0-00052.md)（SET UPN4UPATH，N4 接口）配置 — 后果：误用本命令配 GTP/N4 路径不生效
