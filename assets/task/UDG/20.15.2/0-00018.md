# 设置URR上报失败动作（SET URRFAILACTION）

> 命令静态知识见 [SET URRFAILACTION](command/UDG/20.15.2/SET-URRFAILACTION.md)。本页只讲配置生成实例化时这条命令**怎么配**——按配置维度组织成"**配置方法字典**"，**不**逐特性罗列取值。

## 配置方法

SET URRFAILACTION 是计费三件套的"URR 上报失败容灾动作"——复用于在线计费（Gy）与离线计费（Ga）。最大记录数 1（全局唯一）。典型场景：业务部署调整导致 UPF 的 [ADD URR](0-00001.md) 在 SMF / 控制面不存在时，SMF 无法处理 UPF 的用量上报——本命令决定 UPF 重发失败后怎么办。本命令的核心配置空间有 3 个维度：**重发策略**（重发次数 + 重发间隔，决定何时停止重试）、**失败动作**（CONTINUE / BLOCK / TERMINATE 三选一，决定业务和计费行为）、**保持时长**（仅 CONTINUE / BLOCK 时配置）。

### 配置维度 1：重发策略（参数 RETRYTIMES + RETRYINTERVAL）

| 参数 | 范围 | 含义 |
|---|---|---|
| `RETRYINTERVAL` | 10~120 秒 | 重发间隔（重发失败后等多久再发下一次） |
| `RETRYTIMES` | 0~10 | 重发次数（重发全部失败后才走下一步 RETRYFAILACT） |

> 0 次 + 不间断 = 不重试直接进入失败动作；典型配法：RETRYINTERVAL=30 + RETRYTIMES=3，先重发 3 次每次间隔 30 秒。

### 配置维度 2：失败动作（参数 RETRYFAILACT，核心策略决策）

| 取值 | 业务行为 | 计费影响 | HOLDINGTIME |
|---|---|---|---|
| `CONTINUE` | **放通业务** | **URR 对应业务不计费（漏计费风险）**——用量上报恢复后恢复计费 | **required** |
| `BLOCK` | **阻塞业务** | 业务不可用 | **required** |
| `TERMINATE` | **终止**业务 | 业务终止 | 不配 |

**典型用法（从容灾放通角度）**：当计费 URR 上报失败业务被阻塞时，若要放通业务执行 `SET URRFAILACTION: RETRYINTERVAL=30, RETRYTIMES=3, RETRYFAILACT=CONTINUE, HOLDINGTIME=30;`——URR 对应业务不计费但业务可通行，待用量上报恢复后恢复计费。

### 配置维度 3：保持时长（参数 HOLDINGTIME，仅 `CONTINUE`/`BLOCK` 时必配）

| 参数 | 范围 | 含义 | 联动 |
|---|---|---|---|
| `HOLDINGTIME` | 10~60 分钟 | 业务阻塞 / 放通后的保持时长 | 必须配 `RETRYFAILACT=CONTINUE` 或 `BLOCK` |

> 保持时长到期后——系统恢复尝试重新上报用量；上报成功则恢复计费。

**生效行为**：仅对执行后**新激活用户或发生承载更新的用户**生效——存量用户不回溯。

**步骤位置**：在所属 feature 编排中属于"计费容灾策略"步骤——通常与在线 / 离线计费特性一起配。

## 决策点：重发失败后的动作取值（DP 0-00051）

问：RETRYFAILACT 取哪个值？

| 选项 | HOLDINGTIME | 业务影响 | 计费影响 | 适用 |
|---|---|---|---|---|
| `continue_business` | 升为 required | 放通业务 | **漏计费风险**（上报恢复后恢复计费）| 容灾优先——业务优先于计费精确性 |
| `block_business` | 升为 required | 阻塞业务 | 计费精确 | 计费精确优先——业务可阻塞 |
| `terminate_business` | 排除（不配） | 终止业务 | 计费精确 | 极少用——业务终止 |

## 约束

- **最大记录数与高危**（warning，rule-0-00119）：最大记录数 1（全局动作）；高危命令——`RETRYFAILACT=CONTINUE` 导致漏计费（放通不计费），`BLOCK` / `TERMINATE` 阻塞业务 — 后果：误配 `CONTINUE` 致漏计费；阻塞动作致业务不可用
- **HOLDINGTIME 必配约束**（critical，隐含）：`RETRYFAILACT=CONTINUE` 或 `BLOCK` 时 `HOLDINGTIME` **必传**，范围 10~60 — 后果：缺配致命令失败
- **适用范围**（info，隐含）：仅适用于业务部署调整致 [ADD URR](0-00001.md) 在 SMF 不存在的容灾场景——非容灾场景不适用 — 后果：非容灾场景误配致正常场景下业务行为异常
- **生效范围**（warning，隐含）：仅对新激活用户 / 承载更新用户生效，存量不回溯 — 后果：误以为全局生效致存量业务不受保护
