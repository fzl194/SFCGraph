# 设置URR上报失败动作（SET URRFAILACTION）

> 命令静态知识见 [SET URRFAILACTION](command/UDG/20.15.2/SET-URRFAILACTION.md)。本页只讲配置生成实例化时这条命令**怎么配**。

## 配置方法

本命令是**计费 URR 上报失败的容灾动作**（最大记录数 1，全局唯一），复用于在线计费（Gy）与离线计费（Ga）。应对场景：业务部署调整导致 UPF 的 ADD URR 在 SMF/控制面不存在时，SMF 无法处理 UPF 的用量上报——本命令决定 UPF 重发失败后怎么办。

**配置范式（三参数一组，按容灾策略选动作）**

| 参数 | 取值约定 | 说明 |
|---|---|---|
| RETRYINTERVAL | 10~120 秒 | 重发间隔。全局规划 |
| RETRYTIMES | 0~10 | 重发次数。全局规划 |
| RETRYFAILACT | CONTINUE / BLOCK / TERMINATE | 重发失败后动作（核心，见下表） |
| HOLDINGTIME | 10~60 分钟 | 保持时长。前提 RETRYFAILACT=BLOCK 或 CONTINUE 时必配 |

**RETRYFAILACT 三选一（核心策略决策）**

| 取值 | 业务行为 | 计费影响 | HOLDINGTIME |
|---|---|---|---|
| CONTINUE | 放通业务 | **URR 对应业务不计费（漏计费风险）**，用量上报恢复后恢复计费 | required |
| BLOCK | 阻塞业务 | 业务不可用 | required |
| TERMINATE | 终止 | 业务终止 | 不配 |

**典型用法（从容灾放通角度）**：当计费 URR 上报失败业务被阻塞时，若要放通业务执行 `SET URRFAILACTION` 配 `RETRYFAILACT=CONTINUE`——URR 对应业务不计费但业务可通行，待用量上报恢复后恢复计费。

## 决策点：重发失败后的动作取值（DP 0-00051）

问：RETRYFAILACT 取哪个值？

| 选项 | HOLDINGTIME | 业务/计费影响 |
|---|---|---|
| 放通不计费 CONTINUE | 升为 required | 业务放通，URR 不计费（漏计费风险），上报恢复后恢复计费 |
| 阻塞 BLOCK | 升为 required | 业务被阻塞 |
| 终止 TERMINATE | 排除（不配） | 业务终止 |

## 约束

- **rule-0-00119**（warning）最大记录数 1（全局动作）；高危命令——RETRYFAILACT=CONTINUE 导致漏计费（放通不计费），BLOCK/TERMINATE 阻塞业务；适用于业务部署调整致 UPF ADD URR 在 SMF 不存在的容灾场景 — 后果：误配 CONTINUE 致漏计费；阻塞动作致业务不可用
