# IPFarm重定向链

> **重定向族（用户Portal/WebProxy）通用 compound**：IP Farm 是重定向 Server 的负荷分担集合 + ICMP 心跳检测基础设施。本 compound 定义整机 IP Farm 全局参数、心跳检测逻辑接口、重定向 IP Farm 及其下 Server。被引用于 [2-00014](task/UDG/20.15.2/2-00014.md)（用户Portal）/[2-00015](task/UDG/20.15.2/2-00015.md)（WebProxy）。

## 配置方法

四步链：整机 IP Farm 全局参数（SERVERTYPE=REDIRECT + 心跳阈值 + 负荷分担方式）→ 心跳检测逻辑接口（Phif 接口，源 IP 取自地址池预留）→ 创建重定向 IP Farm（绑心跳接口）→ IP Farm 下 Server（IP + URL + DEFAULTACT）。

| 步骤 | 命令 | 关键参数 |
|---|---|---|
| 整机 IP Farm 全局参数 | `SET IPFARMGLOBAL` | SERVERTYPE=REDIRECT；TIMETHRESHOLD（心跳时间阈值秒）；HEALTHSUCCLIMIT（连续成功置 up）；HEALTHFAILLIMIT（连续失败置 down）；LBMETHOD（负荷分担：ROUND_ROBIN/LEAST_RECENTLY_USED/LEAST_LOAD）|
| 心跳检测逻辑接口 | `ADD LOGICINF` | NAME（Phif 接口名，如 phif1/0/0）；IPVERSION；IPV4ADDRESS1/IPV4MASK1（或 IPv6，心跳本端 IP，取自地址池预留）|
| 重定向 IP Farm | `ADD IPFARM` | IPFARMNAME；IPVERSION；HEALTHCHECKFLAG=ENABLE；INTERFACENAME（绑心跳检测接口名）|
| IP Farm Server | `ADD IPFARMSERVER` | IPFARMNAME；IPVERSION；SERVERIPV4（或 IPv6）；URL（重定向目标，Portal 场景必配）；DEFAULTACT（BLOCK 缺省/PASS 放行，无法重定向时的缺省动作）|

**典型脚本（用户Portal 标准：LEAST_LOAD + IPv4 + 2 Portal Server，activation 原样）**：
```
SET IPFARMGLOBAL:SERVERTYPE=REDIRECT,TIMETHRESHOLD=10,HEALTHSUCCLIMIT=2,HEALTHFAILLIMIT=4,LBMETHOD=LEAST_LOAD;
ADD LOGICINF:NAME="phif1/0/0",IPVERSION=IPV4,IPV4ADDRESS1="10.10.10.10",IPV4MASK1="255.255.255.255";
ADD IPFARM:IPFARMNAME="farm_test",IPVERSION=IPV4,HEALTHCHECKFLAG=ENABLE,INTERFACENAME="phif1/0/0";
ADD IPFARMSERVER:IPFARMNAME="farm_test",IPVERSION=IPV4,SERVERIPV4="192.168.10.123",URL="www.example.com";
ADD IPFARMSERVER:IPFARMNAME="farm_test",IPVERSION=IPV4,SERVERIPV4="192.168.10.124",URL="www.vmall.com";
```

**步骤位置**：承接 License 前置（SET LICENSESWITCH）；产出 IPFARMNAME 供下游 [1-00011](task/UDG/20.15.2/1-00011.md) 规则绑定中 RULE 经 `POLICYTYPE=SMARTREDIRECT`（Portal）或 `POLICYTYPE=WEBPROXY`（WebProxy）+ `POLICYNAME=<IPFARMNAME>` 直接引用（**关键差异：不经 PCCPOLICYGRP，见场景差异 §1**）。

## 场景差异

> 重定向族（用户Portal/WebProxy）共用本 compound，差异集中在触发 POLICYTYPE、过滤匹配层（L7 URL vs L34）、IPFARMSERVER.URL 配置、captive 模式四类。IP Farm 的负荷分担（LBMETHOD 三选一）与 IP 版本（v4/v6）由 DP 驱动，两个特性都支持全集。

### 1. 触发 POLICYTYPE 与下游规则差异（关键）

| 维度 | 用户Portal ([2-00014](task/UDG/20.15.2/2-00014.md)) | WebProxy ([2-00015](task/UDG/20.15.2/2-00015.md)) |
|---|---|---|
| 触发 POLICYTYPE | **SMARTREDIRECT**（Captive Portal 动作）| **WEBPROXY**（Web 代理）|
| RULE.POLICYNAME 指向 | IPFARMNAME（本 compound 产出）| IPFARMNAME（本 compound 产出）|
| 是否经 PCCPOLICYGRP | 否（直接引用 IPFARMNAME）| 否（直接引用 IPFARMNAME）|
| REFRESHTYPE | USERPROFILE | ALL |

> 关键：SMARTREDIRECT 与 WEBPROXY 都是**独立于 PCC/HEADEN 的 POLICYTYPE**，RULE 经 POLICYNAME 直接引用 IPFARMNAME，不经 PCCPOLICYGRP。SMARTREDIRECT 规则可与 PCC 计费规则**绑定同一 USERPROFILE + 共用同一 FLOWFILTER**（activation §表 演示双 RULE：rule_test1=SMARTREDIRECT + rule_test2=PCC 共用 ff_test1）。

### 2. 过滤匹配层差异

| 维度 | 用户Portal ([2-00014](task/UDG/20.15.2/2-00014.md)) | WebProxy ([2-00015](task/UDG/20.15.2/2-00015.md)) |
|---|---|---|
| 过滤匹配 | **L7 URL 匹配**（ADD L7FILTER URL + ADD PROTBINDFLOWF）+ L34（ADD FILTER TCP + ADD FLTBINDFLOWF）| 仅 L34（ADD FILTER + SVRIP，无 L7）|
| 过滤链走法 | [1-00009](task/UDG/20.15.2/1-00009.md) L34 段 + L7 段（两段都用）| [1-00009](task/UDG/20.15.2/1-00009.md) 仅 L34 段 |

> 用户Portal 重定向需精确匹配到触发 URL（如 `www.huawei.com/*`），故走 L7；WebProxy 主要基于目的 IP 做代理转发，无 URL 触发语义，仅 L34。

### 3. IPFARMSERVER.URL 配置差异

| 维度 | 用户Portal ([2-00014](task/UDG/20.15.2/2-00014.md)) | WebProxy ([2-00015](task/UDG/20.15.2/2-00015.md)) |
|---|---|---|
| IPFARMSERVER.URL | **必配**（Portal Server URL，重定向目标地址）| 可选（WebProxy 主要用 SERVERIPV4 做代理转发，URL 非必须）|

### 4. captive 模式（用户Portal 独有）

| 维度 | 用户Portal ([2-00014](task/UDG/20.15.2/2-00014.md)) | WebProxy ([2-00015](task/UDG/20.15.2/2-00015.md)) |
|---|---|---|
| captive 模式 | **USERPROFILE.CAPMODETHRES** 控制 captive/non-captive 交替（定时器分钟）| 无 captive 语义（CAPMODETHRES 无意义）|

> 用户Portal 双模式：用户上线后首次访问触发 captive 重定向到 Portal；CAPMODETHRES 定时器到期前为 non-captive 状态（不再强制重定向），实现 captive/non-captive 交替。WebProxy 无此语义。

### 5. 负荷分担方式（LBMETHOD，三选一，两特性共享）

| LBMETHOD | 含义 | 适用场景 |
|---|---|---|
| ROUND_ROBIN | 轮询方式 | Server 性能相近，简单轮流 |
| LEAST_RECENTLY_USED | 选择最久未被使用的 server | 避免单 server 过载 |
| **LEAST_LOAD**（activation 演示）| 选择最小负荷 server | 推荐生产场景，server 性能不均时最优 |

### 6. IP 版本（IPVERSION，全链路一致，两特性共享）

| IPVERSION | 含义 | 联动 |
|---|---|---|
| **IPV4**（activation 演示）| IPv4 全链路 | IPFARM/SERVER/LOGICINF/Filter/接口同 v4 |
| IPV6 | IPv6 全链路 | IPFARM/SERVER/LOGICINF/Filter/接口同 v6 |

## 决策点

| 选项 | 影响（参数/命令/联动） |
|---|---|
| 负荷分担 LBMETHOD=ROUND_ROBIN（联动特性级）| 轮询选 Server |
| 负荷分担 LBMETHOD=LEAST_RECENTLY_USED（联动特性级）| 最久未用选 Server |
| 负荷分担 LBMETHOD=LEAST_LOAD（联动特性级，activation 演示）| 最小负荷选 Server |
| IP 版本 IPVERSION=IPV4（联动特性级，activation 演示）| 全链路 IPv4：ADD LOGICINF IPV4ADDRESS1/IPV4MASK1；ADD IPFARM IPV4；ADD IPFARMSERVER SERVERIPV4 |
| IP 版本 IPVERSION=IPV6（联动特性级）| 全链路 IPv6：ADD LOGICINF/IPFARM/IPFARMSERVER 对应 IPv6 参数 |
| 缺省动作 DEFAULTACT=BLOCK（默认，activation 演示）| 无法重定向时阻塞业务报文 |
| 缺省动作 DEFAULTACT=PASS（联动特性级，activation 主脚本未演示）| 无法重定向时正常转发；ADD/MOD IPFARMSERVER 配置；证据 activation §操作步骤4 说明段 |

## 约束

- **IP 版本全链路一致**（critical）：IPFARM/SERVER/LOGICINF/Filter/接口须同 v4 或同 v6 — 不一致则重定向失效；用 `DSP RULECHECK` 检查规则的所有 IP 地址取值的 IP 版本一致性。
- **不同 IP Farm 需配不同心跳检测接口**（critical）：心跳检测接口不共享 — 复用则心跳冲突。
- **一个 IP Farm 所有 Server 共用一个心跳接口**（critical）：ADD IPFARM.INTERFACENAME 绑定一个心跳接口，该 Farm 下所有 Server 共用 — 换接口需重建 Farm。
- **心跳源 IP 建议用 Phif 接口地址**（warning）：从终端地址池中预留一个 IP 用于 IP Farm 心跳检测（ADD LOGICINF 的 IPV4ADDRESS1）— 未预留则地址池冲突。
- **至少一个 Server 状态 up**（warning）：配置完成后 `LST IPFARMSERVER` 查询，须保证至少一个 server 状态 up — 否则重定向无可用目标。
- **UDG 不支持 HTTPS 重定向**（warning）：UDG 不支持 HTTPS 协议报文的重定向处理；若重定向 URL 采用 HTTPS 协议，需配 `ADD BLACKLISTRULE` 放行（[0-00293](task/UDG/20.15.2/0-00293.md)）— 否则重定向后业务无法处理。
- **内嵌 IPv4 的 IPv6 业务处理需配 URL**（warning）：当 `SET SRVCOMMONPARA` 开启内嵌 IPv4 地址的 IPv6 业务处理功能时，为确保正确触发重定向，建议在 IP Farm 下配 URL。
- **整机规格**（info）：整机 ≤64 IP Farm；每个 IP Farm ≤512 Server。
