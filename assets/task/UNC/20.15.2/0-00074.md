# 增加服务化接口本端实体（ADD SBIAPLE）

> 命令静态知识见 [ADD SBIAPLE](command/UNC/20.15.2/ADD-SBIAPLE.md)。本页只讲配置生成实例化时这条命令**怎么配**——按配置维度组织成"字典"，**不**逐特性罗列取值。

## 配置方法

SBIAPLE 是本端 NF 在服务化接口上的"接入点身份证"：声明本端是谁（NFTYPE）、用哪组 HTTP 本端实体（HTTPLEGRPIDX）收发。每条命令对应一个 (INDEX, NFTYPE) 实体，可选携带对端 NF 信息用于客户端模式定向访问。

| 维度 | 参数 | 取值 | 默认 | 作用 | 适用场景 |
|---|---|---|---|---|---|
| 本端实体索引 | INDEX | 本端规划，整数 1~128，本端唯一 | 无（必选） | 指定 SBIAPLE 自身索引 | 全部场景 |
| HTTP 本端实体组索引 | HTTPLEGRPIDX | 引用已配 HTTPLEGRP（1~64），组内必须含 1 个 Server 端 + 1 个 Client 端 HTTP 本端实体 | 无（必选） | 指定服务化接口引用的 HTTP 实体组 | 全部场景 |
| 本端 NF 类型 | NFTYPE | 全网规划（NFTypeAMF/NFTypeNRF/NFTypeBSF/NFTypeSMF/NFTypeCHF 等，INVALID 为无效值） | 无（必选） | 声明本端服务化接口服务的 NF 角色 | 全部场景 |
| 目的 NF 类型 | PEERNFTYPE | 全网规划，仅支持 Nchf | 无（条件可选，NFTYPE=SMF/SMSF 时） | 本端客户端模式定向访问 CHF | 融合计费/CHF 对接 |
| CHF 的目的 NF 类型 | CHFPEERNFTYPE | 本端规划，仅支持 Nnrf/Nocs | 无（条件可选，NFTYPE=CHF 时） | 本端 CHF 访问 NRF 或 OCS 服务 | 本端为 CHF |
| 目的 NF 服务 | PEERSERVICE | 全网规划，仅支持 Nchf_ConvCharg | 无（条件可选，PEERNFTYPE=Nchf 时） | 给 CHF 计费服务独立规划接口 | SMF/SMSF 独立规划计费服务接口 |
| CHF 的目的 NF 实例 ID | CHFPEERNFINSTID | 本端规划，0~255 字符 | 无（条件可选，CHFPEERNFTYPE=Nocs 时） | 按对端实例独立规划接口 | 本端 CHF 多实例分流 |
| 描述 | DESCRIPTION | 全网规划，0~255 字符 | 无（可选） | 配置描述信息 | 通用 |

典型脚本（SBI 加密场景，NRF 本端）：
```
ADD SBIAPLE:INDEX=1,HTTPLEGRPIDX=1,NFTYPE=NFTypeNRF,DESCRIPTION="nrf";
```
典型脚本（融合计费场景，SMF 本端定向访问 CHF 计费服务）：
```
ADD SBIAPLE:INDEX=3,HTTPLEGRPIDX=2,NFTYPE=NFTypeSMF,PEERNFTYPE=Nchf,PEERSERVICE=Nchf_ConvCharg;
```

步骤位置：建 HTTP 本端实体（`ADD HTTPLE`）→ 建 HTTP 本端实体组（`ADD HTTPLEGRP`，含 Server 端 + Client 端）→ 本任务 → 关联 NRF 注册（`ADD NRF`）/ TLS 场景（`ADD TLSSCENE`）

## 决策点

### 决策点：本端 NF 类型声明（DP 0-02127）

| 选项 | 联动参数 | 适用场景 |
|---|---|---|
| 声明 AMF/NRF/BSF 等本端 NF（不配对端 NF） | NFTYPE=对应 NFType，不填 PEERNFTYPE/PEERSERVICE | SBI 加密、QoS 能力开放等纯本端接入点声明 |
| 声明 SMF/SMSF 定向访问 CHF | NFTYPE=NFTypeSMF/NFTypeSMSF，PEERNFTYPE=Nchf（可选 PEERSERVICE=Nchf_ConvCharg） | 融合计费场景，SMF/SMSF 客户端模式定位 CHF |
| 声明 CHF 访问 NRF/OCS | NFTYPE=NFTypeCHF，CHFPEERNFTYPE=Nnrf 或 Nocs（可选 CHFPEERNFINSTID） | 本端为 CHF，需访问 NRF/OCS 服务 |

### 决策点：HTTPLEGRPIDX 引用对象与对端规划粒度（DP 0-02128）

| 选项 | 联动参数 | 适用场景 |
|---|---|---|
| 多 NF 共用一个 HTTPLEGRP | 同一 HTTPLEGRPIDX 被多条 SBIAPLE 引用 | 多 NF 角色共享底层 HTTP 实体组 |
| 每 NF/服务独立 HTTPLEGRP | 每条 SBIAPLE 各引用一个 HTTPLEGRP，INDEX 与 HTTPLEGRPIDX 常同值 | 独立规划每 NF 的服务化接口 |
| CHF 共用 NRF/OCS 实体组 | CHFPEERNFTYPE=Nnrf 与 Nocs 两条 SBIAPLE 指定同一 HTTPLEGRPIDX | NRF 与 OCS 共用一个 HTTP 本端实体组 |

## 约束

· rule-0-02316 · severity=critical · HTTPLEGRPIDX 指向的 HTTP 本端实体组内必须存在一个 Server 端 + 一个 Client 端 HTTP 本端实体，否则服务化连接无法建立
· rule-0-02317 · severity=high · 对端 NF 配置互斥：不配对端 NF 信息时只能配一个服务化接口实体；配对端 NF 时针对同一对端 NF 只能配一个服务化接口实体
· rule-0-02318 · severity=high · 对端 NF 取值收敛：PEERNFTYPE 仅支持 Nchf；CHFPEERNFTYPE 仅支持 Nnrf/Nocs；PEERSERVICE 仅支持 Nchf_ConvCharg
· rule-0-02319 · severity=warning · INDEX 取值 1~128 本端唯一；HTTPLEGRPIDX 取值 1~64（语义不同：INDEX 是 SBIAPLE 自身编号，HTTPLEGRPIDX 指向 HTTP 实体组）
· rule-0-02320 · severity=info · 立即生效：命令执行后立即生效，无需重启；最多 128 条记录
