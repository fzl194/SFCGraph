# MOD RULE 原子任务
> 命令静态知识见 [MOD RULE](command/UNC/20.15.2/MOD-RULE.md)。本页只讲调测/运维实例化时这条命令**怎么用**。

适用 NF：PGW-C、SMF。本命令修改已存在的 RULE（含策略信息与全局优先级），只有 ADD RULE 成功的规则才能 MOD。常用于流量计费调测中"为用户绑定计费 RG"（修改规则使其匹配业务流并指向 URR）、以及语音 PCF/PCRF 故障 Bypass 等动态规则策略场景。

## 配置方法
按规则名 + 策略类型定位，按规划修改优先级/策略名/流过滤器等绑定字段。

| 参数 | 配置方法 | 取值样例 |
|---|---|---|
| RULENAME | 规则名称（必选，1~63，不支持空格，不区分大小写） | 按规划规则名 |
| POLICYTYPE | 策略类型（必选，枚举） | 按规划 |
| PRIORITY | 全局优先级（可选，1~4294967295，越小优先级越高） | 按规划 |
| POLICYNAME | 策略名称（conditional，1~63，不支持空格，不区分大小写） | URR/策略组名称 |
| FLOWFILTERNAME | 流过滤器名称（conditional，1~63，不支持空格） | 按规划 |
| RULERANGE | 规则生效范围（conditional，枚举） | 按规划 |
| NWDAFEVENTS | NWDAF 数据分析事件（可选，枚举） | 按规划 |
| IPERULEDELYSW | 智能规则传递开关（conditional，枚举） | 按规划 |

流量计费调测（WSFD-011206）：当 URR/URRGROUP/PCCPOLICYGRP 已建、用户未绑定计费 RG 时，与 MOD URR/MOD URRGROUP/MOD PCCPOLICYGRP 一起执行 MOD RULE，按规划值为用户绑定 RG，重新激活用户后观察 N40 计费流量与实际访问流量是否一致。

## 决策点

### 策略绑定主体（POLICYNAME vs FLOWFILTERNAME）
RULE 的策略信息含 POLICYNAME（指向具体策略，如计费 URR、QoS 策略组）与 FLOWFILTERNAME（指向流过滤器）。两者同属 conditional，按场景择一/组合配置：流量计费用 POLICYNAME 指向 URR；分流/动态规则用 FLOWFILTERNAME 匹配业务流。绑哪个策略、是否带流过滤器随规划分叉。

### 智能规则传递开关（IPERULEDELYSW）
是否启用智能规则传递（IPE Rule Delayed Switch）随策略同步策略分叉，影响规则下发时序，按规划取枚举值。

### 生效范围（RULERANGE）
规则生效范围（枚举）随部署范围（全局/特定 NF 等）分叉。

## 约束
- **立即生效**：修改后立即生效，影响规则匹配结果。
- **RULENAME 存在性前置**：仅能修改已 ADD RULE 成功的规则；未添加则 MOD 失败，可用 LST RULE 校验。
- **RULENAME/POLICYNAME/FLOWFILTERNAME 不支持空格**：长度 1~63，不区分大小写。
- **PRIORITY 语义**：值越小优先级越高（与 TNFBINDGRP/PCCPOLICYGRP 等同方向，注意与 CAPACITY 等反向字段区分）。
- **POLICYTYPE 取值需与绑定策略类型一致**：策略类型决定后续 POLICYNAME 指向对象，跨类型混用会导致绑定失败。
