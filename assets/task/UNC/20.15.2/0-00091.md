# ADD UPFGBINDLOCG 原子任务
> 命令静态知识见 [ADD UPFGBINDLOCG](command/UNC/20.15.2/ADD-UPFGBINDLOCG.md)。本页只讲配置生成实例化时这条命令**怎么配**。

## 配置方法
绑定关系在 [ADD UPFGLOCGBNDGRP](task/UNC/20.15.2/0-00092.md) 建好的绑定关系组（`UPFGLOCGBNDGNAME`）内逐条登记：每条指定一个 Gx UPF 组（`UPFGRPNAME`）对应一个 Diameter 本端主机组（`DIAMLOCGRPNAME`），并用 `PRIORITY` 排定选择顺序（值越小优先级越高）。三参（UPF 组名 / 本端主机组名 / 优先级）均取自已配置数据规划，按"用户 IP 重叠时不同 PGW-U 必须绑不同本端主机"的规则组织成对。

| 场景 | 必备参数 | 典型脚本 |
|---|---|---|
| 多 PGW-U 用户 IP 重叠（Gx 本端主机区分） | `UPFGLOCGBNDGNAME` + `UPFGRPNAME` + `DIAMLOCGRPNAME` + `PRIORITY` | `ADD UPFGBINDLOCG: UPFGLOCGBNDGNAME="test", UPFGRPNAME="upfgrp_01", DIAMLOCGRPNAME="pgwc_host_group_01", PRIORITY=10;` |

## 决策点

### 优先级编排
问：同一绑定关系组内多条绑定的优先级如何排？
- 值越小优先级越高（取值 1~127）。
- 按"最常匹配的 UPF 组"给小值；证据中两个并列 UPF 组用 10 / 15 区分。

### 本端主机选择模式联动
问：本端主机组如何被实际选中？
- 本命令只登记绑定关系；真正的选择模式由后续"设置 PCC 本端主机名选择模式（基于 APN）"步骤驱动——本原子任务输出绑定池，选择策略在上层。

## 约束
- 单个绑定关系组内最多 64 组绑定关系。`severity=medium` 后果：超出无法新增。
- 同一绑定关系组内 `UPFGRPNAME` 不允许重复。`severity=critical` 后果：UPF 组重复导致本端主机选择歧义。
- 用户 IP 重叠的 PGW-U 对应 UPF 组必须绑定**不同**的 Diameter 本端主机组（证据：upf1/upf2 重叠 → 各绑 pgwc_host_group_01/02）。`severity=critical`
- 最大记录数 3000。`severity=medium`
- 执行后立即生效。`severity=low`
