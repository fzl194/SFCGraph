# SET PCCTIMER 原子任务
> 命令静态知识见 [SET PCCTIMER](command/UNC/20.15.2/SET-PCCTIMER.md)。本页只讲配置生成实例化时这条命令**怎么配**。

本命令为全局单例（最大记录数 1），覆盖 Gx/N7 接口重传、PCC 用户回滚在线保持、重授权/重定向迟滞、Supported-features 协商、动态 PCRF 老化、Failover All-sessions 等 PCC 信令控制定时器。典型使用场景：Gx Failover（WSFD-109101）、语音 PCF/PCRF 故障 Bypass 的全局 Holding Time（WSFD-201207）、Gx over DRA 动态 PCRF 老化（WSFD-011132）。

## 配置方法

| 参数 | 配置方法 | 取值样例 |
|---|---|---|
| APPRETRYTIMEOUT | Gx Tx 定时器：本端按对端 PCRF 接收能力规划重传间隔 | `10`（秒） |
| APPRETRYTIMES | 应用层重传次数（本端规划） | `0~3` |
| HOLDINGTIME | 全局 Holding Time：用户回滚后在线保持时长 | `30`（分钟） |
| ADJUSTRANGE | Holding Time 随机延迟范围（避免雪崩） | `10`（分钟） |
| FASTHOLDINGTIME | 快速回滚（Fast Rollback）在线保持时长 | `10`（分钟） |
| FASTADJUSTRANGE | 快速回滚随机延迟范围 | `5`（分钟） |
| REVALIDHYSTMR | Revalidation 重授权迟滞时间 | `600~3600`（秒） |
| REDIRECTHYSTMR | 重定向迟滞时间 | `1~3600`（秒） |
| SUPFEANEGOTMR | Supported-features 协商定时器（清除缓存 PCRF 能力） | `1~43200`（分钟） |
| DYNPCRFAGETMR | 动态 PCRF 主机列表表项老化时长（Gx over DRA） | `0~43200`（分钟） |
| FAILOVERALLTMR | Failover All-sessions 持续时长（主 PCRF 恢复判定） | `1~60`（分钟） |
| N7TXTIMERPARA | N7 接口 Tx 定时器时长系数 | `1~32` |
| REQDATATIMER | N7 请求信息定时器（对端协商） | `5~60`（秒） |
| PENDRETRYTIMES / PENDRETRYTIMER | 事务等待重传次数/间隔（对端协商） | `1~3` / `1~3`（秒） |

典型脚本（Gx Failover 场景，仅配 Tx）：
```
SET PCCTIMER:APPRETRYTIMEOUT=10;
```
典型脚本（语音 PCF/PCRF Bypass 全局 Holding Time，Gx 接口）：
```
SET PCCTIMER:HOLDINGTIME=30, ADJUSTRANGE=10, FASTHOLDINGTIME=10, FASTADJUSTRANGE=5;
```
典型脚本（语音 PCF/PCRF Bypass 全局 Holding Time，N7 接口，仅普通回滚）：
```
SET PCCTIMER:HOLDINGTIME=30, ADJUSTRANGE=10;
```

## 决策点

### 普通 Holding Time 与 Fast Holding Time 的适用
- 问什么：PCF/PCRF 故障回滚后，用户在线保持时长走普通还是快速回滚路径。
- 语音会话场景下，当存在语音专载且配置 ACTWITDEDBEARER=FASTROLLBACK 时，走 FASTHOLDINGTIME/FASTADJUSTRANGE；其余普通回滚走 HOLDINGTIME/ADJUSTRANGE。
- N7 接口场景通常只配普通 HOLDINGTIME/ADJUSTRANGE；Gx 接口场景两者都配。
- 不配则使用默认值；具体是否启动 Holding Time 还受软参 DWORD519 Bit9 控制。

### 动态协商/动态 PCRF 老化是否使能
- 问什么：是否需要定时清除本地缓存的 PCRF 能力或动态 PCRF 主机表项。
- Supported-features 二次协商场景 → 配 SUPFEANEGOTMR，定时清缓存以学习 PCRF 升级后的真实能力。
- Gx over DRA 场景 → 配 DYNPCRFAGETMR 控制动态学习到的 PCRF 主机表项老化。

### Tx 定时器时长规划（高危）
- 问什么：APPRETRYTIMEOUT（Tx）设多大。
- 强约束：本端 Tx（APPRETRYTIMEOUT）必须小于 ADD GTPCT3N3 的 T3N3，且小于 SET SMCOMMTIMER 的 TPOLICY，否则右侧网元无响应会导致激活响应总时长过长、用户激活失败。
- 建议结合对端 PCRF 接收能力（WAL 限流）一起规划。

## 约束
- 高危命令：超时时长配置不合理可能导致激活总时长过长、用户激活失败。`severity=high`
- 立即生效（命令执行后立即生效，非仅对新激活用户）。`severity=medium`
- 单例规格：最大记录数为 1，全局唯一。`severity=medium`
- Tx 链路规划：等待右侧网元最大时长 < GTPC T3N3（ADD GTPCT3N3）且 < 左侧 MME/SGSN T3N3 且 < SMCOMMTIMER TPOLICY。`severity=high`
- Holding Time 受软参 DWORD519 Bit9 控制是否在激活流程回滚时生效。`severity=medium`
- 取值范围：各参数均有数值上下限（见命令 wiki），ADJUSTRANGE/FASTADJUSTRANGE 为 0~60 分钟，HOLDINGTIME 为 0~1440 分钟。`severity=low`
