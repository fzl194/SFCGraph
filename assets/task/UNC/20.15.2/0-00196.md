# 配置PCC定时器（SET PCCTIMER）

> 命令静态知识见 [SET PCCTIMER](command/UNC/20.15.2/SET-PCCTIMER.md)。本页只讲配置生成实例化时这条命令**怎么配**——按配置维度组织成"字典"，**不**逐特性罗列取值。

## 配置方法

SET PCCTIMER 为全局单例（最大记录数 1），覆盖 Gx/N7 接口重传、PCC 用户回滚在线保持、重授权/重定向迟滞、Supported-features 协商、动态 PCRF 老化、Failover All-sessions 等 PCC 信令控制定时器。典型场景：Gx Failover（WSFD-109101）、语音 PCF/PCRF 故障 Bypass 的全局 Holding Time（WSFD-201207）、Gx over DRA 动态 PCRF 老化（WSFD-011132）。本命令有 3 个配置维度：**Tx/重传定时器**、**Holding Time 回滚保持**、**协商/老化/Failover 定时器**。

### 配置维度 1：Tx/重传定时器（参数 APPRETRYTIMEOUT/APPRETRYTIMES/PENDRETRYTIMES/PENDRETRYTIMER/N7TXTIMERPARA/REQDATATIMER）

| 维度 | 参数 | 取值 | 默认 | 作用 | 适用场景 |
|---|---|---|---|---|---|
| Tx 定时器/重传间隔 | `APPRETRYTIMEOUT` | 0~10（秒） | 5 | APPRETRYTIMES=0 时等价 Tx timer（等响应超时）；=1~3 时为重传间隔 | Gx 接口 CCR 等待响应/重传 |
| 应用层重传次数 | `APPRETRYTIMES` | 0~3 | 0 | 0=不支持重传（APPRETRYTIMEOUT 作 Tx timer）；1~3=按次数重传 | Gx 接口向同一 Peer 重传 |
| 事务等待重传次数 | `PENDRETRYTIMES` | 1~3 | 1 | PendingTransaction 功能重试次数（PCRF 返回 4144 时） | PCRF 返回 DIAMETER_PENDING_TRANSACTION |
| 事务等待重传间隔 | `PENDRETRYTIMER` | 1~3（秒） | 1 | PendingTransaction 功能重试间隔 | PCRF 返回 DIAMETER_PENDING_TRANSACTION |
| N7 Tx 时长系数 | `N7TXTIMERPARA` | 1~32 | 1 | N7 接口 TxTimer = ClientRcvRspTmt × n + 0.5s（ClientRcvRspTmt 由 SET HTTPCONF 配置） | N7 接口 HTTP 消息超时 |
| N7 请求信息定时器 | `REQDATATIMER` | 5~60（秒） | 10 | N7 接口 SMF 等待左侧 AMF/MME 返回信息时长 | N7 接口请求信息 |

> 数据来源：本端规划（APPRETRYTIMEOUT/APPRETRYTIMES/N7TXTIMERPARA）、对端协商（PENDRETRYTIMES/PENDRETRYTIMER/REQDATATIMER）。应用层重传与链路层重传功能重叠可能导致消息重发次数大于预期，APPRETRYTIMES 建议不开启（取 0）。

### 配置维度 2：Holding Time 回滚保持（参数 HOLDINGTIME/ADJUSTRANGE/FASTHOLDINGTIME/FASTADJUSTRANGE）

| 维度 | 参数 | 取值 | 默认 | 作用 | 适用场景 |
|---|---|---|---|---|---|
| 普通 Holding Time | `HOLDINGTIME` | 0~1440（分钟） | 0 | 用户回滚后在线保持时长；0=保持在线直到主动下线 | PCF/PCRF 故障普通回滚 |
| 普通随机延迟范围 | `ADJUSTRANGE` | 0~60（分钟） | 0 | HOLDINGTIME 超时后增加随机延迟（避免雪崩）；HOLDINGTIME=0 时不生效 | 大批量回滚场景 |
| 快速回滚 Holding Time | `FASTHOLDINGTIME` | 0~1440（分钟） | 0 | 用户快速回滚后在线保持时长；0=保持在线直到主动下线 | 语音专载 FASTROLLBACK 场景 |
| 快速回滚随机延迟 | `FASTADJUSTRANGE` | 0~60（分钟） | 0 | FASTHOLDINGTIME 超时后增加随机延迟；FASTHOLDINGTIME=0 时不生效 | 大批量快速回滚场景 |

> 数据来源：全网规划。语音会话场景下，当存在语音专载且配置 ACTWITDEDBEARER=FASTROLLBACK 时走 FASTHOLDINGTIME/FASTADJUSTRANGE；其余普通回滚走 HOLDINGTIME/ADJUSTRANGE。N7 接口场景通常只配普通 HOLDINGTIME/ADJUSTRANGE；Gx 接口场景两者都配。不配则使用默认值；具体是否启动 Holding Time 还受软参 DWORD519 Bit9 控制。

### 配置维度 3：协商/老化/Failover 定时器（参数 REVALIDHYSTMR/REDIRECTHYSTMR/SUPFEANEGOTMR/DYNPCRFAGETMR/FAILOVERALLTMR）

| 维度 | 参数 | 取值 | 默认 | 作用 | 适用场景 |
|---|---|---|---|---|---|
| Revalidation 迟滞 | `REVALIDHYSTMR` | 600~3600（秒） | 600 | 相邻两次重新发起授权请求的最小间隔 | 重授权频繁控制 |
| 重定向迟滞 | `REDIRECTHYSTMR` | 1~3600（秒） | 600 | 相邻两次重定向上报的最小间隔 | 已弃用 |
| Supported-features 协商 | `SUPFEANEGOTMR` | 1~43200（分钟） | 60 | 定时清除缓存 PCRF 能力协商结果 | Supported-features 二次协商 |
| 动态 PCRF 老化 | `DYNPCRFAGETMR` | 0~43200（分钟） | 60 | 动态 PCRF 主机列表表项老化时长；0=关闭老化 | Gx over DRA 动态学习 PCRF |
| Failover All-sessions | `FAILOVERALLTMR` | 1~60（分钟） | 3 | Failover All-sessions 定时器超时，删除 PCRF/PCF Failover 状态 | 主 PCRF 恢复判定 |

> 数据来源：本端规划。Supported-features 二次协商场景配 SUPFEANEGOTMR，定时清缓存以学习 PCRF 升级后的真实能力；Gx over DRA 场景配 DYNPCRFAGETMR 控制动态学习到的 PCRF 主机表项老化。REDIRECTHYSTMR 已弃用。

**典型脚本**

```
# Gx Failover 场景（仅配 Tx）
SET PCCTIMER: APPRETRYTIMEOUT=10;

# 语音 PCF/PCRF Bypass 全局 Holding Time（Gx 接口，普通+快速）
SET PCCTIMER: HOLDINGTIME=30,ADJUSTRANGE=10,FASTHOLDINGTIME=10,FASTADJUSTRANGE=5;

# 语音 PCF/PCRF Bypass 全局 Holding Time（N7 接口，仅普通）
SET PCCTIMER: HOLDINGTIME=30,ADJUSTRANGE=10;

# 动态 PCRF 老化时长 30 分钟
SET PCCTIMER: DYNPCRFAGETMR=30;
```

**步骤位置**：PCRF 对接链配置（1-00012）的可选配套步骤（动态协商定时器）；语音 PCF/PCRF Bypass（WSFD-201207）全局 Holding Time 步骤；Gx over DRA（WSFD-011132）动态 PCRF 老化步骤。

## 决策点

### 决策点：Tx 定时器时长规划（DP 0-03131）

问：APPRETRYTIMEOUT（Tx）设多大，是否开启应用层重传？

| 选项 | 联动参数 | 适用场景 |
|---|---|---|
| APPRETRYTIMES=0（不重传） | `APPRETRYTIMEOUT` 作 Tx timer | Gx 接口发送 CCR 后等响应超时（建议，避免与链路层重传重叠） |
| APPRETRYTIMES=1~3（重传） | `APPRETRYTIMEOUT` 作重传间隔 | 需要应用层重传增强可靠性 |
| N7 接口 | `N7TXTIMERPARA` 时长系数 | 5G N7 接口 HTTP 消息超时 |

> **高危强约束**：本端 Tx（APPRETRYTIMEOUT）必须小于 ADD GTPCT3N3 的 T3N3，且小于 SET SMCOMMTIMER 的 TPOLICY，否则右侧网元无响应会导致激活响应总时长过长、用户激活失败。PENDRETRYTIMES/PENDRETRYTIMER 的最大时长与 APPRETRYTIMEOUT 之和也须小于 T3N3。建议结合对端 PCRF 接收能力（WAL 限流）一起规划。

### 决策点：普通 Holding Time 与 Fast Holding Time 的适用（DP 0-03132）

问：PCF/PCRF 故障回滚后，用户在线保持时长走普通还是快速回滚路径？

| 选项 | 联动参数 | 适用场景 |
|---|---|---|
| 仅普通回滚 | `HOLDINGTIME`/`ADJUSTRANGE` | N7 接口场景、无非语音专载回滚 |
| 普通+快速回滚 | `HOLDINGTIME`/`ADJUSTRANGE` + `FASTHOLDINGTIME`/`FASTADJUSTRANGE` | Gx 接口场景，语音专载配 ACTWITDEDBEARER=FASTROLLBACK |
| 不配（用默认） | 沿用初始值（均为 0） | 无 Holding Time 需求或受软参 DWORD519 Bit9 控制不启动 |

> N7 接口场景通常只配普通 HOLDINGTIME/ADJUSTRANGE；Gx 接口场景两者都配。不配则使用默认值；具体是否启动 Holding Time 还受软参 DWORD519 Bit9 控制。配置为 0 表示回滚后保持在线直到用户主动下线。

## 约束

- rule-0-03326 · severity=critical · Tx 链路规划（高危）：APPRETRYTIMEOUT（Tx）必须小于 ADD GTPCT3N3 的 T3N3，且小于 SET SMCOMMTIMER 的 TPOLICY，且小于左侧 MME/SGSN 的 T3N3；PENDRETRYTIMES/PENDRETRYTIMER 的最大时长与 APPRETRYTIMEOUT 之和也须小于 T3N3 — 后果：超时时长配置不合理导致激活响应总时长过长、用户激活失败
- rule-0-03327 · severity=critical · 高危命令：本命令为高危命令，超时时长配置不合理可能导致激活总时长过长、用户激活失败，前期规划须严格按 Tx 链路约束 — 后果：配置不当导致大面积激活失败
- rule-0-03328 · severity=high · 应用层重传建议不开启：APPRETRYTIMES 建议取 0（不重传），应用层重传与链路层重传功能重叠可能导致消息重发次数大于预期 — 后果：重发次数大于预期致 PCRF 负荷异常
- rule-0-03329 · severity=warning · Holding Time 受软参控制：HOLDINGTIME/FASTHOLDINGTIME 是否在激活流程回滚时生效还受软参 DWORD519 Bit9 控制，不启动则定时器配置不生效 — 后果：缺少软参前置导致 Holding Time 不生效
- rule-0-03330 · severity=info · 单例与立即生效：最大记录数为 1（全局唯一），命令执行后立即生效（非仅对新激活用户），REDIRECTHYSTMR 已弃用 — 后果：全局唯一配置，配置即时生效影响所有用户
