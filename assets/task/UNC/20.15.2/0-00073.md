# ADD RULEBINDING 原子任务
> 命令静态知识见 [ADD RULEBINDING](command/UNC/20.15.2/ADD-RULEBINDING.md)。本页只讲配置生成实例化时这条命令**怎么配**。

## 配置方法
Rule 本身不能直接下发 UPF，必须先挂到 UserProfile 上，SMF 下发 UserProfile 即激活其下全部 Rule。当用户报文匹配多条规则时，把多条规则逐条绑到同一 UserProfile 即可。本命令建立 (USERPROFILENAME, RULENAME[, POLICYTYPE]) 三元组绑定关系。

| 业务场景 | USERPROFILENAME 来源 | RULENAME 来源 | POLICYTYPE | 典型脚本 |
|---|---|---|---|---|
| PCC（计费/QoS/IPv6 SA/语音 Bypass） | ADD USERPROFILE 配的 USERPROFILENAME | ADD RULE 配的 RULENAME（POLICYTYPE=PCC） | 可选，建议显式 PCC | `ADD RULEBINDING:USERPROFILENAME="up_test",RULENAME="rule_test";` |
| 业务触发 QoS 保证 | ADD USERPROFILE | ADD RULE（POLICYTYPE=QOS） | 显式 QOS | `ADD RULEBINDING:USERPROFILENAME="up_test",RULENAME="rule_test",POLICYTYPE=QOS;` |
| 基于业务感知的带宽控制 | ADD USERPROFILE | ADD RULE（POLICYTYPE=BWM） | 可选 BWM | `ADD RULEBINDING:USERPROFILENAME="up_test",RULENAME="rule_test";` |
| ADC（应用检测与控制） | ADD USERPROFILE | ADD RULE（POLICYTYPE=ADC） | 省略 | `ADD RULEBINDING:USERPROFILENAME="up_test",RULENAME="rule_test";` |
| UL CL 分流 | ADD USERPROFILE（UPTYPE=ULCL） | ADD RULE（POLICYTYPE=ULCL） | 显式 ULCL | `ADD RULEBINDING:USERPROFILENAME="testuserprofilename",RULENAME="testrule",POLICYTYPE=ULCL;` |
| 业务触发 RADIUS | ADD USERPROFILE | ADD RULE（POLICYTYPE=SRV_TRIGGER） | 省略 | `ADD RULEBINDING:USERPROFILENAME="up_test",RULENAME="rule_test1";` |
| 融合计费（多规则挂一模板） | 同一 USERPROFILE | 多个 RULENAME 逐条绑 | 省略 | `ADD RULEBINDING:USERPROFILENAME="userprofile001",RULENAME="Rule001_dns";` ×N |

**POLICYTYPE 省略语义**：不指定时，把 RULENAME 相同的所有 POLICYTYPE 的规则都绑到模板。建议同名跨类型时显式指定以避免误绑。

**前置依赖**：必须先执行 [ADD USERPROFILE](task/UNC/20.15.2/0-00094.md)（或其未建编号）和 [ADD RULE](task/UNC/20.15.2/0-00071.md)，顺序不能反。

## 决策点

### POLICYTYPE 是否显式指定
问：同一 RULENAME 是否存在多个 POLICYTYPE 的规则？
- **存在**：必须显式指定 POLICYTYPE，否则会把所有同名规则都绑上模板。
- **不存在（唯一）**：可省略，自动绑定该 RULENAME 对应的唯一规则。证据中 ULCL/QOS/BWM/PCC 场景既有显式也有省略，取决于命名是否跨类型复用。

### USERPROFILENAME 与 RULENAME 取值来源
问：两个名称取已配数据还是本端新规划？
- **已配置数据中获取**（主流，证据中 29 处）：引用 ADD USERPROFILE 和 ADD RULE 已定义的名称。可用 LST USERPROFILE / LST RULE 查询确认。
- 个别场景标"全网规划"（ULCL 分流的 USERPROFILENAME）或"本端规划"（RULENAME）——本质都是引用已配对象，获取口径因规划粒度而异。

### 单模板挂多少规则
问：一个 UserProfile 绑几条 Rule？
- **多规则场景**（融合计费/语音 Bypass）：同一 USERPROFILENAME 连续执行多条 ADD RULEBINDING，每条绑一个 RULENAME。受单模板上限约束。
- **单规则场景**（ADC/QoS 保证）：一条即可。

## 约束
- USERPROFILENAME、RULENAME 长度均 1~63，不支持空格，不区分大小写。`severity=medium`
- 前置：必须先执行 ADD USERPROFILE 和 ADD RULE，否则本命令失败。`severity=critical`
- 命令最大记录数 200000。`severity=medium`
- 同一 ULCL 类型 UserProfile 下最多绑 512 个规则。`severity=medium`
- UPSPECTYPE=SPECIFICATION_LIMITED 时单模板最多 100 个规则；DEFAULT（或不指定）时上限较低（见命令 wiki）。`severity=medium`
- 跨网元一致性：SMF、UPF 上 USERPROFILE 下绑定的 RULENAME、POLICYTYPE 必须一致。`severity=high`
- PCF 通过预定义规则组下发 APP_STA/APP_STO 时，UPF/SMF/PCF 上的规则（组）标识必须一致。`severity=high`
- 执行后立即生效。`severity=low`
- 调测建议：配错时先 RMV RULEBINDING 再重新 ADD，不要直接覆盖。`severity=low`
