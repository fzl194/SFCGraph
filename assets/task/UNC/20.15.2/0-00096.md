# ADD USRPROFGROUP 原子任务
> 命令静态知识见 [ADD USRPROFGROUP](command/UNC/20.15.2/ADD-USRPROFGROUP.md)。本页只讲配置生成实例化时这条命令**怎么配**。

本命令是"UserProfile 三段式组装"的中间环：`ADD USERPROFILE`（建模板，含 UPTYPE/计费属性）→ **本命令**（建组容器）→ `ADD UPBINDUPG`（模板入组）→ `ADD APNUSRPROFG`（组绑 APN/DNN）。组本身只有一个名称参数，真正的策略语义靠后续绑定承载；本命令的配置生成价值在于"起名"。

## 配置方法
单参数命令，按"组名"配置，每条建一个组。

| 参数 | 配置方法 | 取值样例 |
|---|---|---|
| USERPROFGNAME | 本端规划命名（按用途语义命名） | `upg_test`、`upg_ims`、`upg-test`、`usrg`、`ulcl_test` |

典型脚本（建一个通用 PCC 组）：
```
ADD USRPROFGROUP:USERPROFGNAME="upg_test";
```

组名因场景而异，但同一条命令行只填该名：计费场景常 `upg-test`/`upg_test`，UL CL 分流常 `ulcl_test`，IMS bypass 常 `upg_ims`。

## 决策点

### 组名命名来源
- 问什么：USERPROFGNAME 取自哪里。
- 全为"本端规划"（少数 bypass 场景标"全网规划"，实为同套规划口径），由配置生成按全网一致性产出。命名无系统约束之外的强制规则，但需与后续 `ADD UPBINDUPG`、`ADD APNUSRPROFG` 引用名严格一致。
- 跨场景复用同名组（如 `upg_test` 在 PCC/QoS/计费多处出现）时，应识别为同一对象，不重复建组。

### 组是否需要新建
- 问什么：本场景的组是新建还是引用既有组。
- 多数激活文档把建组标为"可选 / 如已配置请跳过"（如 WSFD-109107、WSFD-211005、WSFD-201207 均注明"如已配置则跳过本步骤"）。配置生成需先查 `LST USRPROFGROUP`，命中即复用，未命中才新建。

## 约束

- **立即生效** · severity=LOW。
- **最大记录数 10000**：全网组容量上限 · severity=MEDIUM。
- **USERPROFGNAME 格式**：1~63 字符，不支持空格，不区分大小写 · severity=LOW。
- **唯一性**：组名全网唯一，重复 ADD 报错；应先查后建 · severity=MEDIUM。
- **空组无效**：本命令只建容器，不绑定 UserProfile 的空组不产生策略效果，必须配 `ADD UPBINDUPG` 才生效 · severity=HIGH。
- **下游引用**：组名须被 `ADD UPBINDUPG`（USERPROFGNAME）和 `ADD APNUSRPROFG`（USERPROFGNAME）正确引用，三处必须一致 · severity=HIGH。
