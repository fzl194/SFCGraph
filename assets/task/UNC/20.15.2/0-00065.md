# 增加QoS属性（ADD QOSPROP）

> 命令静态知识见 [ADD QOSPROP](command/UNC/20.15.2/ADD-QOSPROP.md)。本页只讲配置生成实例化时这条命令**怎么配**——按配置维度组织成"字典"，**不**逐特性罗列取值。

## 配置方法

ADD QOSPROP 配置 PCC 预定义规则的 QoS 参数集，是业务触发 QoS 保证、本地 PCC、LTE 一键通、语音 Bypass 等场景的**必备**命令。配置空间以"接入代际 × 业务模型"为主轴：`QOSTYPE` 是总开关，决定走 5G QoS Flow（配 `FQI`）还是 2/3/4G 承诺（配 `QCIVALUE`），两者必备 QoS 标识不同。`QOSPROPNAME` 全局唯一，通过 ADD PCCPOLICYGRP 的 `QOSPROPNAME` 或 ADD RULE 的 `POLICYTYPE=QOS` 两种路径绑定到规则；ADD L2RULE 也可绑定一个 QOSPROP（L7 规则 QoS 路径）。配置空间有 5 个维度：**QoS 属性名称**（必选，全局唯一）、**QoS 属性类型**（必选总开关）、**QoS 等级标识**（条件必选，由类型决定）、**分配保留优先级**（条件必选）、**速率与抢占**（可选，GBR 业务必备速率）。

### 配置维度 1：QoS 属性名称（参数 QOSPROPNAME，必选）

| 维度 | 参数 | 取值 | 默认 | 作用 | 适用场景 |
|---|---|---|---|---|---|
| QoS 属性名称 | `QOSPROPNAME` | 本端规划；字符串 1~31，不支持空格，不区分大小写 | 无 | QoS 属性集全局唯一标识，供 ADD PCCPOLICYGRP/ADD RULE/ADD L2RULE 绑定 | 所有 QoS 保证场景 |

### 配置维度 2：QoS 属性类型（参数 QOSTYPE，必选总开关）

| 维度 | 参数 | 取值 | 默认 | 作用 | 适用场景 |
|---|---|---|---|---|---|
| QoS 属性类型 | `QOSTYPE` | 本端规划；`QOS_BEARER_PARA`（承诺级）/ `QOS_FLOW_PARA`（流级）/ `QOS_L2_PARA`（层二） | 无 | 总开关：决定必备 QoS 标识（BEARER 配 QCIVALUE，FLOW/L2 配 FQI） | 5G 用户走 FLOW；2/3/4G 用户走 BEARER；L7 规则走 L2 |

### 配置维度 3：QoS 等级标识（参数 QCIVALUE 或 FQI，条件必选）

| 维度 | 参数 | 取值 | 默认 | 作用 | 适用场景 |
|---|---|---|---|---|---|
| 2/3/4G 等级标识 | `QCIVALUE` | 本端规划；整数 1~255（标准 QCI 见 3GPP） | 0（无效） | QOSTYPE=QOS_BEARER_PARA 时配置；4/5G 有效，3G 可配不生效 | 2/3/4G 承诺级 QoS |
| 5G QoS 标识 | `FQI` | 本端规划；整数 1~255（标准 5QI） | 无 | QOSTYPE=QOS_FLOW_PARA 或 QOS_L2_PARA 时必配 | 5G QoS Flow 级 QoS |
| 反射 QoS 指示 | `RQI` | 本端规划；`ENABLE`/`DISABLE` | DISABLE | QOSTYPE=QOS_FLOW_PARA 时可选，配置反射 QoS 标识 | 5G 反射 QoS 场景 |

> QoS 等级标识（QCIVALUE 或 FQI）与分配保留优先级 ARPVALUE 必须同时配置；不配时 QCIVALUE 默认 0、ARPVALUE 默认 0 均为无效值，可能影响后续业务功能。

### 配置维度 4：分配保留优先级与抢占（参数 ARPVALUE + EMPCAP + EMPVUL，条件必选）

| 维度 | 参数 | 取值 | 默认 | 作用 | 适用场景 |
|---|---|---|---|---|---|
| 分配保留优先级 | `ARPVALUE` | 本端规划；整数 1~15，值越小优先级越高 | 0（无效） | 资源限制时决定是否允许二次激活；QOS_FLOW_PARA/QOS_L2_PARA 必配 | 所有 QoS 场景 |
| 抢占能力 | `EMPCAP` | 本端规划；`ENABLE`/`DISABLE` | DISABLE | 是否可抢占已分配给低优先级业务的资源 | 高优先级业务（语音、一键通）常 ENABLE |
| 被抢占设置 | `EMPVUL` | 本端规划；`ENABLE`/`DISABLE` | ENABLE | 是否可被高优先级业务抢占 | 高优先级业务常 DISABLE（不被抢占） |

### 配置维度 5：速率与 QoS 使用量上报（参数 GBR/MBR + QOSURRNAME，可选）

| 维度 | 参数 | 取值 | 默认 | 作用 | 适用场景 |
|---|---|---|---|---|---|
| 保证速率 | `GBRUPLKVALUE`/`GBRDNLKVALUE` | 本端规划；整数 0~20000000 kbit/s | 复杂默认规则 | GBR 业务的上下行保证带宽 | GBR 业务必备；NonGBR 不生效 |
| 最大速率 | `MBRUPLKVALUE`/`MBRDNLKVALUE` | 本端规划；整数 0~20000000 kbit/s | 复杂默认规则 | 上下行最大带宽限制 | GBR 业务必备；NonGBR 不生效 |
| QoS 使用量上报规则 | `QOSURRNAME` | 本端规划；字符串 1~31，引用 ADD URR 已建的 URRNAME | 无 | 业务触发承载创建的 QoS 用量统计上报；不设则无法感知承载创建 | 业务触发 QoS 保证、语音 Bypass 必备；本地 PCC 占位可省 |

> GBR 业务需配全 4 项速率；NonGBR 类型承载的 MBR/GBR 值在业务处理中不生效，所有 NonGBR 共用会话级带宽。业务触发承载创建场景下，承载带宽按 ADD QOSPROP 粒度累加（同一 QOSPROP 只累加一次）。

**典型脚本（2/3/4G 承诺级，QCI=66，GBR 业务）**
```
ADD QOSPROP: QOSPROPNAME="qosprop_401", GBRUPLKVALUE=600, GBRDNLKVALUE=6000, MBRUPLKVALUE=1000, MBRDNLKVALUE=10000, QOSTYPE=QOS_BEARER_PARA, QCIVALUE=66, ARPVALUE=5, QOSURRNAME="URR_qos_01";
```

**典型脚本（5G QoS Flow 级，5QI=2，GBR 业务）**
```
ADD QOSPROP: QOSPROPNAME="qosprop_501", GBRUPLKVALUE=600, GBRDNLKVALUE=6000, MBRUPLKVALUE=1000, MBRDNLKVALUE=10000, QOSTYPE=QOS_FLOW_PARA, ARPVALUE=5, FQI=2, QOSURRNAME="URR_qos_01";
```

**典型脚本（最简，本地 PCC 占位）**
```
ADD QOSPROP: QOSPROPNAME="qos_property1", QOSTYPE=QOS_BEARER_PARA;
```

**步骤位置**：在 QoS 保证特性编排中通常位于"配置 QoS 属性"步骤——业务策略/业务质量属性阶段，紧随 [ADD URR](task/UNC/20.15.2/0-00001.md)（QOSURRNAME 引用源）之后，先于 [ADD PCCPOLICYGRP](task/UNC/20.15.2/0-00052.md)（PCC 策略组绑定）与 [ADD RULE](task/UNC/20.15.2/0-00071.md)（规则绑定）。

## 决策点

### 决策点：QoS 属性类型总开关（DP 0-02109）

问：本业务用户走 5G QoS Flow 还是 2/3/4G 承诺？

| 选项 | 联动参数 | 适用场景 |
|---|---|---|
| 5G QoS Flow | `QOSTYPE=QOS_FLOW_PARA` + `FQI`(+可选 `RQI`) + `ARPVALUE` | 5G 用户（证据 WSFD-109107 业务触发 QoS 保证、WSFD-201207 N7 接口 5G 场景） |
| 2/3/4G 承诺 | `QOSTYPE=QOS_BEARER_PARA` + `QCIVALUE` + `ARPVALUE` | 2/3/4G 用户（证据 WSFD-109101 本地 PCC、WSFD-102602 LTE 一键通、WSFD-201207 Gx 接口 2/3/4G 场景） |
| 层二（L7 规则） | `QOSTYPE=QOS_L2_PARA` + `FQI` + `ARPVALUE` | L7 规则 QoS 路径（由 ADD L2RULE 绑定） |
| 跨代业务 | 建 5G 一条 FLOW+FQI、2/3/4G 一条 BEARER+QCIVALUE | 证据 WSFD-109107 显式给出双轨示例 |

### 决策点：绑定路径与 QoS 用量上报（DP 0-02110）

问：QoS 属性如何挂到规则？是否引用 URR 做 QoS 用量统计？

| 选项 | 联动参数 | 适用场景 |
|---|---|---|
| 经 PCC 策略组绑定 | ADD RULE 的 `POLICYTYPE=PCC` + `POLICYNAME=<PCCPOLICYGRPNM>`，QoS 属性通过 ADD PCCPOLICYGRP 的 `QOSPROPNAME` 绑定 | PCC 预定义规则标准路径 |
| 直接绑定规则 | ADD RULE 的 `POLICYTYPE=QOS` + `POLICYNAME=<QOSPROPNAME>` | 纯 QoS 规则路径；后果：PCC 与 QOS 类型的 RULENAME 不能相同 |
| 需 QoS 用量上报 | 配 `QOSURRNAME=<已建 URR 的 URRNAME>`（已配置数据中获取） | 业务触发 QoS 保证、语音 Bypass（必备，否则无法感知承载创建） |
| 不需 QoS 用量上报 | 省略 `QOSURRNAME` | 本地 PCC 占位、纯 QoS 描述 |

## 约束

- **QOSPROPNAME 长度 1~31**（critical，rule-0-02271）：`QOSPROPNAME` 长度 1~31，不支持空格，不区分大小写；存在多条记录时不可重复；QoS 等级标识（QCIVALUE 或 FQI）与分配保留优先级 ARPVALUE 必须同时配置，否则规则失效
- **QOSTYPE=QOS_BEARER_PARA 时需配**（critical，rule-0-02272）：`QOSTYPE=QOS_BEARER_PARA` 时需配 `QCIVALUE`（1~255）；`=QOS_FLOW_PARA` 或 `=QOS_L2_PARA` 时需配 `FQI`（1~255，+可选 `RQI`）；QoS 标识缺失导致规则失效
- **一个 ADD PCCPOLICYGRP 可绑定一个**（warning，rule-0-02273）：一个 ADD PCCPOLICYGRP 可绑定一个 QOSPROP；一个 ADD L2RULE 可绑定一个 QOSPROP；预定义规则的 QOSPROP 仅当数据流匹配到预定义规则才执行；PCC 与 QOS 类型的 RULENAME 不能相同
- **ARPVALUE 取值 1~15**（warning，rule-0-02274）：`ARPVALUE` 取值 1~15，值越小优先级越高；`EMPCAP` 默认 DISABLE、`EMPVUL` 默认 ENABLE；`GBR/MBR` 取值 0~20000000 kbit/s，GBR 业务需配全 4 项速率，NonGBR 类型承载的 MBR/GBR 值业务处理中不生效
- **与 BWM Rule 同时绑定时**（info，rule-0-02275）：与 BWM Rule 同时绑定时，数据流同时匹配则基于 5G PDU level 的 BWM 不执行但 user-group level 的 BWM 执行，此时用配了 QOSPROPNAME 的 PCC Rule 作 5G PDU level 带宽控制；最大记录数 500；执行后立即生效；适用 NF：PGW-C、SMF
