# IPv4v6 双栈接入（WSFD-104002）

> 特性静态知识见 [WSFD-104002](feature/UNC/20.15.2/WSFD-104002.md)。本页讲配置生成实例化时这个特性**怎么配**——按 5G/4G/3G/2G 制式 + 静态/动态地址编排哪些步骤、双栈开关与签约数据纠正如何选。

## 配置概览

核心对象链：**(5G) AMF/SMF 选择 PDU 会话类型** → SMF 与 UDM 交互取静态 IP 签约 → SMF 选 UPF → 经 N4 传双栈地址；**(4G/2-3G) SGSN/MME + GGSN/PGW-C** → Create PDP Context / Create Session Request 携带双栈标识 (Dual Address Bearer Flag=1) + PDN Type=IPv4v6 → GGSN/PGW-C 同时下发 IPv4 和 IPv6 地址。

**核心配置空间**（按 5G 与 4G/2-3G 拆解）：

| 制式 | License 前置 | 核心开关 | 关键联动 |
|---|---|---|---|
| 5G AMF/SMF | `SET LICENSESWITCH LICITEM=LKV2IPDSAM01`（AMF）+ `LKV2IPDSSM01`（SMF）| 无独立 SMFUNC 开关；由 SMF/UDM 签约驱动 | UDM 签约 PDU 会话类型允许 IPv4v6；APN 上 SUPPORTIPV4+IPV6=ENABLE（共享 [UNC 1-00024](task/UNC/20.15.2/1-00024.md) 接入基础设施）|
| 4G SGSN/MME | `SET LICENSESWITCH LICITEM=LKV2DUSA02`（SGSN、MME）| `SET SMFUNC DUALFLAG=YES`（会话管理扩展功能）| 可选 `ADD SMSUBDATA` 签约数据纠正（Gr/S6a 不支持 IPv4v6 时）|
| 4G/5G PGW-C/SMF | `SET LICENSESWITCH LICITEM=LKV2IPDSSM01` | 无独立 SMFUNC 开关 | APN 上 SUPPORTIPV4+IPV6=ENABLE + 双栈地址池（[UNC 1-00023](task/UNC/20.15.2/1-00023.md)）|

> **走法分叉**：本特性的 4G/2-3G 侧除 License 外**还要打开 SMFUNC 双栈全局开关**（`SET SMFUNC DUALFLAG=YES`），5G 侧**不**使用此命令——5G 由 SMF 直接从 UDM 签约驱动。两种制式的"骨架"共享 [UNC 1-00024](task/UNC/20.15.2/1-00024.md)（APN 接入域基础设施），差异在 License + SMFUNC + 签约纠正。

> **不建 4 套流程**：4G/2-3G 下的 APN 接入基础设施复用共享骨架 compound；5G 下的 APN 接入基础设施同样复用同一骨架，差异在 License 名称（AMF/SMF 两套）。

## 配置流程（步骤编排，每步显示含命令）

1. **License 前置**（5G 或 4G 选一）：
   - 5G AMF/SMF：`SET LICENSESWITCH` → [0-00180](task/UNC/20.15.2/0-00180.md)（LICITEM=LKV2IPDSAM01,SWITCH=ENABLE；LICITEM=LKV2IPDSSM01,SWITCH=ENABLE）
   - 4G SGSN/MME：`SET LICENSESWITCH` → [0-00180](task/UNC/20.15.2/0-00180.md)（LICITEM=LKV2DUSA02,SWITCH=ENABLE）
   - 4G/5G PGW-C/SMF：`SET LICENSESWITCH` → [0-00180](task/UNC/20.15.2/0-00180.md)（LICITEM=LKV2IPDSSM01,SWITCH=ENABLE）
2. **APN 接入域基础设施**（VPN/APN 实例 + 使能双栈地址分配）：`ADD L3VPNINST` + `ADD VPNINSTAF`（AFTYPE=ipv4uni+ipv6uni 双族）+ `ADD VPNINST` + `ADD APN`（含 HASVPNIPV6=ENABLE,VPNINSTANCEIPV6=）+ `SET APNADDRESSATTR`（SUPPORTIPV4+SUPPORTIPV6=ENABLE）→ [UNC 1-00024](task/UNC/20.15.2/1-00024.md)
3. **双栈地址池**（UNC ADDRPOOL 系 IPv4+IPv6 对称）：`ADD ADDRPOOL`（IPVERSION=IPv4v6 / 双池对称，POOLTYPE=Local）+ `ADD SECTION`（IPv4/IPv6 段对称）→ [UNC 1-00023](task/UNC/20.15.2/1-00023.md)
4. **接入域路由发布**（OSPFv2 + OSPFv3 双栈发布用户路由）：
   - **IPv4 OSPFv2 部分**：`ADD OSPF`（v2 进程，IPv4 router-id）+ `ADD OSPFAREA` + `ADD OSPFINTERFACE`（IPv4 接口，**UNC 真实命令名为 OSPFINTERFACE**——v1.0 写的 `OSPFNETWORK` 是错的，UNC 无此命令）+ `ADD OSPFIMPORTROUTE`（PROTOCOL=wlr）→ [UNC 1-00029](task/UNC/20.15.2/1-00029.md)
   - **IPv6 OSPFv3 部分**（双栈叠加）：`ADD OSPFV3`（**UNC 真实命令族，全大写**——非小驼峰 `OSPFv3`）+ `ADD OSPFV3AREA` + `ADD OSPFV3INTERFACE`（无 NETWORK 命令族）+ `ADD OSPFV3IMPORTROUTE`（PROTOCOL=wlr） → [1-00032](task/UNC/20.15.2/1-00032.md) UNC OSPFv3 路由发布
   - 关键参数：PROCID（v2/v3 各 1 进程）+ AREAID + IFNAME（绑接口）+ ROUTERID 全网唯一 + PROTOCOL=wlr（用户本地路由）
   - **★ 命令名修正（R1.5 必加脚注，2026-07-12）**：v1.0 写的 `OSPFv3`（小驼峰）+ `OSPFNETWORK` 全部错。UNC 真实命令族：
     - OSPFv2：`ADD OSPF` / `ADD OSPFAREA` / `ADD OSPFINTERFACE` / `ADD OSPFIMPORTROUTE`（无 NETWORK）
     - OSPFv3：`ADD OSPFV3` / `ADD OSPFV3AREA` / `ADD OSPFV3INTERFACE` / `ADD OSPFV3IMPORTROUTE`（路径：`VNRS功能管理/IP服务/路由管理/OSPFv3管理/`）
5. **4G/2-3G 专属：开启 IPv4v6 双栈全局开关**（仅 4G/2-3G SGSN/MME 执行）：`SET SMFUNC DUALFLAG=YES`（SMF 扩展功能全局使能双栈） → [1-00033](task/UNC/20.15.2/1-00033.md) UNC 4G/2-3G 双栈全局开关（命令真实存在 UNC 命令文档：`业务服务管理/Pre 5G接入业务管理/控制面管理/业务安全管理/会话管理/SM扩展功能管理/设置会话管理扩展功能(SET SMFUNC)_26145684.md`，但 UNC pre-build 阶段 atom 未建）
6. **4G/2-3G 可选：签约数据纠正**（仅当 Gr/S6a 连接的 HLR/HSS 不支持 IPv4v6 双栈）：`ADD SMSUBDATA`（PDPTYPE=IPV4、NEWPDPTYPE=IPV4V6、IPV4PDPADDRTYPE/NEWIPV4PDPADDRTYPE/NEWIPV6PDPADDRTYPE=DYNAMIC） → [1-00033](task/UNC/20.15.2/1-00033.md) UNC 4G/2-3G 双栈全局开关（同步骤 5，命令真实存在 UNC 文档：`业务服务管理/Pre 5G接入业务管理/控制面管理/业务安全管理/会话管理/签约数据纠正/增加签约数据纠正参数(ADD SMSUBDATA)_26305486.md`，atom 未建）

> **步骤 2-4 编排共享骨架**：本特性作为地址分配族一员，与 [WSFD-010502](feature/UNC/20.15.2/WSFD-010502.md)（地址分配方式）/[WSFD-104001](feature/UNC/20.15.2/WSFD-104001.md)（IPv6 承载）/[WSFD-104004](feature/UNC/20.15.2/WSFD-104004.md)（IPv6 前缀代理）共享 [UNC 1-00024](task/UNC/20.15.2/1-00024.md) + [UNC 1-00023](task/UNC/20.15.2/1-00023.md) + [UNC 1-00029](task/UNC/20.15.2/1-00029.md) 三骨架，本特性**不**新增骨架 compound。
>
> **步骤 5-6 是本特性专有**（4G/2-3G 侧）：`SET SMFUNC DUALFLAG` 与 `ADD SMSUBDATA` 是双栈特有的两个单命令步骤，**不走骨架 compound**（§3.5 单命令直接 atom）。**两个 atom UNC _numbering.json 当前均未收录**——属待补资源，配置生成时需先补 atom 才能引用。

## 决策点（特性级：制式 × 静态/动态 × 4G/5G 子开关）

| 接入制式 × 地址类型 | 走法 | 关键差异 | 注意事项 |
|---|---|---|---|
| 5G × 静态 IPv4v6 | 步骤 1（5G License）+ 步骤 2-4（共享骨架）+ UDM 签约驱动 | 无 SET SMFUNC；静态 IP 由 UDM 签约下发 SMF | UDM 必须签约 PDU Session Type=IPv4v6 + 静态 IPv4/IPv6 地址 |
| 5G × 动态 IPv4v6 | 步骤 1（5G License）+ 步骤 2-4（共享骨架，地址池 Local 动态）| 无 SET SMFUNC；动态 IP 由本地 ADDRPOOL 分配 | 双栈地址池需 IPv4/IPv6 双 SECTION |
| 4G SGSN/MME × 静态 | 步骤 1（SGSN/MME License）+ 步骤 2-4 + 步骤 5（SMFUNC DUALFLAG）+ 步骤 6（SMSUBDATA，可选）| 多 SET SMFUNC；可选签约纠正 | 若 HSS 不支持 IPv4v6 → 必选步骤 6 |
| 4G SGSN/MME × 动态 | 步骤 1 + 步骤 2-4 + 步骤 5；**无**步骤 6 | 多 SET SMFUNC | 动态场景下 HSS 签约数据正确则无需纠正 |
| 4G/5G PGW-C/SMF × 静态/动态 | 步骤 1（PGW-C/SMF License）+ 步骤 2-4 | 仅共享骨架 | PGW-C/SMF 独立 License，无 SMFUNC |
| 2-3G × 静态/动态 | 步骤 1（SGSN License）+ 步骤 5（SMFUNC DUALFLAG）+ 步骤 6（SMSUBDATA，若 HLR 不支持）+ 步骤 2-4 | 同 4G SGSN/MME | GGSN 需 R9 协议；RNC 需 R9 协议 |

> **走法列**：所有制式**前 3 步走共享骨架 compound**（[UNC 1-00024](task/UNC/20.15.2/1-00024.md) → [UNC 1-00023](task/UNC/20.15.2/1-00023.md) → [UNC 1-00029](task/UNC/20.15.2/1-00029.md)），4G/2-3G 多走步骤 5（SMFUNC）+ 步骤 6（SMSUBDATA，可选）。

## 约束

- **双 License 控制**（critical）：5G 必须同时打开 AMF（`LKV2IPDSAM01`）+ SMF（`LKV2IPDSSM01`）两套 License；4G SGSN/MME 仅需 `LKV2DUSA02`，PGW-C/SMF 需 `LKV2IPDSSM01` — 后果：License 未开 → 特性不可用。
- **5G 与 4G/2-3G 不共存 SMFUNC 开关**（critical）：`SET SMFUNC DUALFLAG=YES` 仅适用 4G/2-3G SGSN/MME/GnGp SGSN；5G AMF/SMF 不执行此命令 — 后果：5G 配置 SET SMFUNC 是误配，特性无效果。
- **GCDR/EGCDR 话单限制**（warning）：双栈用户在离线计费中 GCDR 和 EGCDR 话单**只支持携带一个 IPv4 或 IPv6 字段** — 后果：双栈话单合并记录，话单采集需注意字段缺失。
- **签约数据纠正的 APNNI 范围**（critical）：`ADD SMSUBDATA` 的 APNNIRANGE 决定纠正生效范围；全网规划时取 `APNNI_ALL`（默认值）或精确匹配 — 后果：APNNI 错配致纠正未生效。
- **签约数据纠正参数一致性**（warning）：`PDPTYPE=IPV4` + `NEWPDPTYPE=IPV4V6` 是 IPv4 单栈改双栈的标准组合；若原签约是 IPv6 则 `PDPTYPE=IPV6` + `NEWPDPTYPE=IPV4V6` — 后果：源 PDP 类型不匹配致纠正失败。
- **GGSN/SGSN 协议版本下限**（critical）：互联 GGSN 与 GnGp SGSN 必须已升级到支持 IPv4v6 类型的 **3GPP TS 29.060 R9 2010Q3** 及以后协议；RNC 的 Iu 口必须已升级到支持 IPv4v6 类型的 **3GPP TS 25.413 R9 2010Q2** 及以后协议 — 后果：协议版本过低致双栈 PDP 失败。
- **PGW-C IPv4v6 拒绝原因值受 BIT716 控制**（info）：激活请求消息中 PDN Type 为 IPv4v6 且 Dual Address Bearer Flag=0 时，PGW-C 仅允许 IPv4 或 IPv6 单地址时返回原因值由软参 BIT716 控制 — 后果：BIT716 设置不当致返回原因值不符合预期。
- **特性无应用限制**（info）：5G 侧特性概述明确无应用限制；4G/2-3G 仅协议版本下限限制。
- **依赖 WSFD-104001 IPv6 承载上下文**（critical）：双栈特性依赖 IPv6 承载上下文特性提供 IPv6 路由广播并下发 UE；特性概述明确"对于给用户分配的 IPv6 地址，需要依赖该特性生成 IPv6 路由广播并下发给 UE" — 后果：IPv6 承载上下文未开 → 双栈特性 IPv6 部分不可用。
- **依赖 UDM/HLR/HSS 签约数据**（critical）：双栈 IPv4v6 必须 UDM 签约 PDU Session Type 允许 IPv4v6 + Default PDU Session Type 一致；4G/2-3G 还需 HSS/HLR 签约 PDN/PDP Type 支持 IPv4v6 — 后果：签约不匹配 → SMF 拒绝建立会话（`Nsmf_PDUSession_CreateSMContext Response` 拒绝）。
- **UDG/UPF 侧 L2TP 与 IPv6 PD 互斥**（warning）：IPv6 Prefix Delegation（WSFD-104004）特性概述指出"如果 APN 使能了 L2TP 功能，则无需开启 IPv6 Prefix Delegation 功能" — 后果：同 APN 下同时配 L2TP + IPv6 PD 导致资源浪费。
