# 业务触发的QoS保证（WSFD-109107）

> 特性静态知识见 [WSFD-109107](feature/UNC/20.15.2/WSFD-109107.md)。本页讲配置生成实例化时这个特性**怎么配**。
>
> **特性关系**：带宽族第 3 批。本特性是**GBR 保证路径的 UNC 侧（控制面）实现**——与 UDG 侧 [GWFD-020358](feature/UDG/20.15.2/GWFD-020358.md)（UPF 业务识别+执行 QoS 策略）对应。与 BWM（[WSFD-211005](feature/UNC/20.15.2/WSFD-211005.md) 限速整形上限控制）正交：本特性做 GBR 带宽下限保证（专有承载/专有 QoS Flow，专用资源）。依赖 [PCC基本功能 WSFD-109101](task/UNC/20.15.2/2-00005.md)（从 PCRF/PCF 获取预定义规则时）。本特性新建 [1-00017](task/UNC/20.15.2/1-00017.md) QoS 属性与规则绑定链（URR 的 USAGERPTMODE=QOS 是 QoS 专属变种，命令集与计费费率标识链 [1-00009](task/UNC/20.15.2/1-00009.md) 重合但语义不同——计费绑 URRGROUP+费率，QoS 绑 QOSPROP+GBR），不整体复用 1-00009；RULE/USERPROFILE/RULEBINDING 命令跨 PCC 族共享但 QoS 场景参数集不同（见 [1-00017](task/UNC/20.15.2/1-00017.md) 场景差异）。

## 配置概览

核心对象链：**License 前置** → **QoS URR**（USAGERPTMODE=QOS，QoS 事件上报）→ **QOSPROP**（GBR/MBR/5QI/QCI/ARP 抢占）→ **PCCPOLICYGRP**（绑 QOSPROP，方式一）/ 直接绑 → **RULE**（POLICYTYPE=PCC 或 QOS）→ **USERPROFILE + RULEBINDING** → （本地 PCC 时）**USRPROFGROUP + UPBINDUPG + APNUSRPROFG**（APN 挂接）→ （可选，5G）**SET APNIDLETIME**（专有 QoS Flow 空闲定时器）/ **ADD APNDEACTQFPLCY**（UE 无线丢失延迟释放）。

两条代际路径差异：
- **5G（QOS_FLOW_PARA + FQI/5QI）**：触发专有 QoS Flow 建立；可选配 DEDQFIDLETIMER（专有 QoS Flow 空闲定时器）+ ADD APNDEACTQFPLCY（UE 无线丢失延迟释放 GBR QoS Flow）。
- **2/3/4G（QOS_BEARER_PARA + QCIVALUE/QCI）**：触发专有承载建立；无独立空闲定时器参数（2/3/4G 承载老化），无延迟释放策略。

两种规则绑定方式：方式一 POLICYTYPE=PCC 经 PCCPOLICYGRP 间接绑 QOSPROP（多规则共享 QoS 属性）；方式二 POLICYTYPE=QOS 直接绑 QOSPROP（单规则独占，跳过 PCCPOLICYGRP）。

关键差异维度：① 代际（5G QOS_FLOW_PARA vs 2/3/4G QOS_BEARER_PARA）② 规则绑定方式（PCC 策略组 vs 直接绑 QoS 属性）③ PCC 模式（动态 PCC 从 PCRF/PCF 获规则名 vs 本地 PCC 自配静态规则+APN 绑定）④ 是否配专有 QoS Flow 空闲定时器/延迟释放（仅 5G）。

## 配置流程（步骤编排，每步显示含命令）

1. **激活 QoS 保证 License**（前置）：`SET LICENSESWITCH` LICITEM="LKV3W9STQE11" → [0-00180](task/UNC/20.15.2/0-00180.md)
   - 关键参数：LICITEM="LKV3W9STQE11"（业务触发的QoS保证，控制项 82208819），SWITCH=ENABLE
   - **前置依赖**：从 PCRF/PCF 获取预定义规则时须先完成 [PCC基本功能 WSFD-109101](task/UNC/20.15.2/2-00005.md) 激活（License LKV2PCCBF01+LKV3W9SPCC11 + PCRF 对接 + PCC 开关）；本地 PCC 场景仅 PCC 基本功能 License 前置即可
2. **配置 QoS 属性与规则绑定链**（核心）：`ADD URR`(USAGERPTMODE=QOS) + `ADD QOSPROP`(GBR/MBR/5QI或QCI/ARP) + `ADD PCCPOLICYGRP`(绑QOSPROP，方式一) + `ADD RULE`(POLICYTYPE=PCC或QOS) + `ADD USERPROFILE` + `ADD RULEBINDING` → [1-00017](task/UNC/20.15.2/1-00017.md)
   - 5G 用 QOSTYPE=QOS_FLOW_PARA + FQI（5QI，如 2=语音）；2/3/4G 用 QOSTYPE=QOS_BEARER_PARA + QCIVALUE（QCI，如 66）
   - 方式一 POLICYTYPE=PCC 经 PCCPOLICYGRP 间接绑；方式二 POLICYTYPE=QOS 直接绑 QOSPROPNAME（跳过 ADD PCCPOLICYGRP）
3. **配置 UserProfile 组与 APN 绑定**（本地 PCC 必配；动态 PCC 时 PCRF/PCF 下发规则名则可跳过）：`ADD USRPROFGROUP` + `ADD UPBINDUPG` + `ADD APNUSRPROFG` → [0-00096](task/UNC/20.15.2/0-00096.md) · [0-00089](task/UNC/20.15.2/0-00089.md) · [0-00008](task/UNC/20.15.2/0-00008.md)
   - 关键参数：USERPROFGNAME + USERPROFILENAME（引用步骤 2 的 USERPROFILE）+ APN/DNN
   - activation 原文"无可用 PCRF/PCF 或预定义规则组时使用"——本地 PCC 静态规则挂接到 APN
4. **（可选，仅 5G）配置专有 QoS Flow 空闲定时器**：`SET APNIDLETIME` → [0-00150](task/UNC/20.15.2/0-00150.md)
   - 关键参数：APN；INHERIT=NO；HSMFTIMERLEVEL=QOSFLOW（固定）；DEDQFIDLETIMER（专有 QoS Flow 空闲定时器时长 min，如 600）
   - APN 粒度优先于全局 `SET DFTIDLETIME`（[0-00170](task/UNC/20.15.2/0-00170.md)）；修改后对新激活用户生效，仅对专有 QoS Flow 生效
   - 设置过大长时间占用上下文资源，过小强制拆话（删除后需重建）
5. **（可选，仅 5G）配置去活专有 QoS Flow 延迟释放策略**：`ADD APNDEACTQFPLCY` → [0-00006](task/UNC/20.15.2/0-00006.md)
   - 关键参数：APN；NGAPCAUSEGROUP=RADIO_NETWORK；NGAPCAUSEVALUE=21；QFPLCY=DELAY_RELEASE；DELAYTIMER（延迟释放时长 秒，如 60）
   - UE 无线连接丢失（radio-connection-with-ue-lost）场景，延迟释放专有 GBR 类型 QoS Flow，避免语音等业务中断
   - APN 粒度优先于全局 `ADD DEACTQFPLCY`（[0-00025](task/UNC/20.15.2/0-00025.md)）；2/3/4G 无延迟释放策略

> **activation 演示范围脚注（R1.4）**：WSFD-109107 仅 1 份 activation（`激活业务触发的QoS保证_85397058.md`），9 步操作步骤 + 完整数据规划表 + 任务示例脚本。脚本同时演示 5G（QOS_FLOW_PARA+FQI=2）与 2/3/4G（QOS_BEARER_PARA+QCIVALUE=66）两代际、方式一（PCC 策略组）与方式二（直接绑 QoS）两绑定方式、可选步骤（SET APNIDLETIME/ADD APNDEACTQFPLCY）。**无零证据差异参数**——所有步骤+参数均有 activation 数据规划表 + 脚本双重证据。
>
> **★ activation 脚本分支标注（R1.4 子规则，warning）**：activation 步骤 3-5 把 5G/2-3-4G 两代际、方式一/方式二两绑定方式**并列列出**（脚本注释"（5G）配置…（2/3/4G）配置…（方式一）…（方式二）…"），并非单一脚本一次跑全。配置生成须按 DP1（代际）+ DP2（绑定方式）选定分支产出，不可把 5G 与 2/3/4G 的 QOSPROP 同时产出（同一业务按全网规划选一代际）。命令 wiki [ADD QOSPROP](command/UNC/20.15.2/ADD-QOSPROP.md) 参数表 QOSTYPE 互斥（QOS_FLOW_PARA 配 FQI，QOS_BEARER_PARA 配 QCIVALUE），证据来自 activation 脚本分支注释。

## 决策点（特性级）

### DP1：代际（决定 QoS 标识与建立结果）

| 代际 | 走法 | QOSPROP 参数 | 建立结果 | 可选定时器/去活 |
|---|---|---|---|---|
| 5G | 步骤 1-2（QOS_FLOW_PARA）+3+4+5 | QOSTYPE=QOS_FLOW_PARA；FQI（5QI，如 2=对话型） | 专有 QoS Flow（QFI 由 FQI 映射） | 可配 SET APNIDLETIME:DEDQFIDLETIMER + ADD APNDEACTQFPLCY |
| 2/3/4G | 步骤 1-2（QOS_BEARER_PARA）+3 | QOSTYPE=QOS_BEARER_PARA；QCIVALUE（QCI，如 66） | 专有承载 | 无独立空闲定时器参数；无延迟释放策略 |

### DP2：规则绑定方式（PCC 策略组 vs 直接绑 QoS 属性）

| 绑定方式 | 走法 | 关键参数 | 适用 |
|---|---|---|---|
| 方式一：PCC 策略组 | 步骤 2 含 ADD PCCPOLICYGRP | ADD RULE POLICYTYPE=PCC，POLICYNAME=PCCPOLICYGRPNM | 多规则共享一组 QoS 属性 |
| 方式二：直接绑 QoS | 步骤 2 跳过 ADD PCCPOLICYGRP | ADD RULE POLICYTYPE=QOS，POLICYNAME=QOSPROPNAME | 单规则独占 QoS 属性 |

> PCC 类型的 RULENAME 值与 QOS 类型的 RULENAME 值不能相同（critical）。

### DP3：PCC 模式（决定是否需 UserProfile 组+APN 绑定）

| PCC 模式 | 走法 | 步骤 3 | 前置 |
|---|---|---|---|
| 动态 PCC（有 PCRF/PCF） | 步骤 1-2 + 可选 4-5；PCRF/PCF 下发预定义规则名 | 可跳过（PCRF/PCF 下发规则名时） | [2-00005 PCC基本功能](task/UNC/20.15.2/2-00005.md) 动态 PCC 路径已激活 |
| 本地 PCC（无 PCRF/PCF） | 步骤 1-3 + 可选 4-5；本地静态规则挂 APN | 必配 USRPROFGROUP+UPBINDUPG+APNUSRPROFG | PCC 基本功能 License 前置（本地 PCC 开关） |

### DP4：是否配专有 QoS Flow 空闲定时器/延迟释放（仅 5G）

| 选项 | 影响 | 适用 |
|---|---|---|
| 配 DEDQFIDLETIMER（步骤 4） | 专有 QoS Flow 空闲超时后 SMF 强制释放 | 需控制专有 QoS Flow 生命周期（默认值不满足时） |
| 不配（继承全局 SET DFTIDLETIME） | 用全局默认空闲定时器 | 默认值满足规划 |
| 配 ADD APNDEACTQFPLCY:DELAY_RELEASE（步骤 5） | UE 无线丢失时延迟释放 GBR QoS Flow（如语音弱信号） | 语音等需保持的 GBR 业务 |
| 不配延迟释放 | UE 无线丢失立即释放 | 无延迟保持需求 |

## 约束（特性级 rule）

- **License 前置**（critical）：须激活 LKV3W9STQE11（控制项 82208819） — 否则 QoS 保证功能不生效。
- **依赖 PCC 基本功能**（critical）：从 PCRF/PCF 获取预定义规则时须先激活 [WSFD-109101 PCC基本功能](task/UNC/20.15.2/2-00005.md)（License + PCRF 对接 + PCC 开关） — 否则无 PCC 通道承载规则名。
- **URRID/RULENAME 跨网元一致**（critical）：SMF（UNC）与 UPF（UDG）的 URRID/USAGERPTMODE/RULENAME/POLICYTYPE/POLICYNAME 须一致；存在 PCRF/PCF 时三网元 RULENAME 须一致 — 否则 QoS 事件无法匹配、专有承载/QoS Flow 不建立。
- **USAGERPTMODE=QOS 固定**（critical）：QoS 保证的 URR 必须 USAGERPTMODE=QOS（区别于计费 ONLINE/OFFLINE） — 否则 UPF 不上报 QoS 事件。
- **PCC 类型与 QOS 类型 RULENAME 不可相同**（critical）：PCC 类型的 RULENAME 与 QOS 类型的 RULENAME 不能相同 — 否则规则冲突。
- **GBR/MBR 须满足 GBR≤MBR**（warning）：GBRUPLKVALUE/GBRDNLKVALUE 须不大于对应 MBRUPLKVALUE/MBRDNLKVALUE — 否则 QoS 保证无法兑现。
- **DEDQFIDLETIMER 语义**（warning，仅 5G）：设置过大长时间占用上下文资源；过小强制拆话，删除后需重建；修改仅对新激活用户生效，仅对专有 QoS Flow 生效。
- **APN 粒度优先于全局**（info）：SET APNIDLETIME 优先于 SET DFTIDLETIME；ADD APNDEACTQFPLCY 优先于 ADD DEACTQFPLCY。
- **应用限制**（warning）：园区 ULCL 分流和专用 DNN+ULCL 分流/超级漫游（正向分流）场景下，仅支持公网业务触发的专有承载创建，不支持园区业务触发的专有承载创建。
- **影响上下文回收管理**（info）：[WSFD-205105 上下文回收管理](feature/UNC/20.15.2/WSFD-205105.md) 的"专有 QoS Flow 空闲定时器时长"参数影响本特性创建的专有 QoS Flow 生命周期；超时 SMF 强制释放。
- **加密协议不可 QoS 保证**（info）：对 SSL/FTPS 等加密协议报文业务识别受限（UPF 侧限制），影响触发准确性。
