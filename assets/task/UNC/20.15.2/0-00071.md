# ADD RULE 原子任务
> 命令静态知识见 [ADD RULE](command/UNC/20.15.2/ADD-RULE.md)。本页只讲配置生成实例化时这条命令**怎么配**。

## 配置方法
Rule 是 PCC 体系的枢纽：SMF 收到 PCRF/PCF 下发的预定义规则名后，按本地 Rule 匹配并下发 UPF。POLICYTYPE 决定 Rule 挂哪类策略对象，POLICYNAME 即对应对象名。按业务场景选型：

| 业务场景 | POLICYTYPE | POLICYNAME 来源 | 必备/常见参数 | 典型脚本 |
|---|---|---|---|---|
| 内容计费（离线/在线/融合/事件/IPv6 在线） | `PCC` | ADD PCCPOLICYGRP 的 PCCPOLICYGRPNM | RULENAME+POLICYTYPE+POLICYNAME+PRIORITY | `ADD RULE:RULENAME="Rule001",POLICYTYPE=PCC,PRIORITY=110,POLICYNAME="cg-test";` |
| 业务触发 QoS 保证（经 PCC 策略组） | `PCC` | ADD PCCPOLICYGRP（内含 QOSPROPNAME） | RULENAME+POLICYTYPE+POLICYNAME | `ADD RULE:RULENAME="rule_test",POLICYTYPE=PCC,POLICYNAME="pccpolicy_group";` |
| 业务触发 QoS 保证（直接绑 QoS 属性） | `QOS` | ADD QOSPROP 的 QOSPROPNAME | RULENAME+POLICYTYPE+POLICYNAME | `ADD RULE:RULENAME="rule_test",POLICYTYPE=QOS,POLICYNAME="qosprop_501";` |
| 基于业务感知的带宽控制（BWM） | `BWM` | 无 POLICYNAME | RULENAME+POLICYTYPE+PRIORITY | `ADD RULE:RULENAME="rule_test",POLICYTYPE=BWM,PRIORITY=10;` |
| ADC（应用检测与控制） | `ADC` | 无 POLICYNAME | RULENAME+POLICYTYPE | `ADD RULE:RULENAME="rule_test",POLICYTYPE=ADC;` |
| UL CL 分流 | `ULCL` | ADD ULCLPROP 的属性名 + FLOWFILTERNAME | RULENAME+POLICYTYPE+POLICYNAME+FLOWFILTERNAME | `ADD RULE:RULENAME="testrule",POLICYTYPE=ULCL,POLICYNAME="testulclpropname",FLOWFILTERNAME="ulcl";` |
| 业务触发 RADIUS | `SRV_TRIGGER` | 无 POLICYNAME | RULENAME+POLICYTYPE+PRIORITY | `ADD RULE:RULENAME="rule_test1",POLICYTYPE=SRV_TRIGGER,PRIORITY=50;` |
| 语音 PCF/PCRF 故障 Bypass | `PCC` | ADD PCCPOLICYGRP（信令组/语音组各一） | RULENAME+POLICYTYPE+PRIORITY+POLICYNAME | `ADD RULE:RULENAME="rule_ims_signaling",POLICYTYPE=PCC,PRIORITY=65535,POLICYNAME="pccpolicygrp_signaling";` |

**关键规则**：
- **RULENAME 跨网元一致**：预定义规则场景下，SMF/UPF/PCF 上的 RULENAME 必须一致（PCF 下发规则名，SMF 本地匹配）。
- **PRIORITY 语义**：值越小优先级越高；同值时动态规则优先级高于预定义规则。
- **POLICYTYPE↔POLICYNAME 绑定**：`PCC`→PCCPOLICYGRPNM，`QOS`→QOSPROPNAME，`ULCL`→ULCL 属性名；`BWM`/`ADC`/`SRV_TRIGGER` 不配 POLICYNAME。
- **RULERANGE**：4G/5G 主辅锚点 UPF 分发控制——`ALL`=主+辅，`CENTRAL`=仅主，`LOCAL`=仅辅。仅 LOCAL 才能再绑 DNAI。

## 决策点

### POLICYTYPE 选型
问：本 Rule 服务于哪类业务控制？
- **PCC**（最通用）：计费、QoS 经 PCC 策略组聚合，POLICYNAME 引用 PCCPOLICYGRPNM。证据中内容计费/QoS 保证/语音 Bypass 普遍采用。
- **QOS**：规则直接绑 QoS 属性（不经 PCC 策略组），POLICYNAME 引用 QOSPROPNAME。用于业务触发 QoS 保证的直绑场景。
- **BWM**：基于业务感知的带宽控制，无 POLICYNAME。
- **ADC**：应用检测与控制，无 POLICYNAME。
- **ULCL**：UL CL 分流，POLICYNAME 引用 ULCL 属性，必带 FLOWFILTERNAME。
- **SRV_TRIGGER**：业务触发 RADIUS，无 POLICYNAME。
- 影响参数集：POLICYTYPE 决定 POLICYNAME 是否必填及引用对象类型。

### POLICYNAME 取值来源
问：POLICYNAME 取已配数据还是本端新规划？
- **已配置数据中获取**（主流）：引用 [ADD PCCPOLICYGRP](task/UNC/20.15.2/0-00052.md) 或 [ADD QOSPROP](task/UNC/20.15.2/0-00065.md) 已定义的对象名。证据中绝大多数场景如此。
- 先建策略组/QoS 属性，再建 Rule 引用，顺序不能反。

### RULERANGE 取值
问：规则下发给哪些锚点 UPF？
- `ALL`（默认）：主+辅锚点都下发（4G/5G PCC 基本功能常见）。
- `CENTRAL`：仅主锚点 PGW-U/UPF。
- `LOCAL`：仅辅锚点，且必须配 LOCAL 才能用 [ADD RULEBINDDNAI](task/UNC/20.15.2/0-00072.md) 绑定 DNAI。

### PRIORITY 取值
问：PRIORITY 取本端规划还是与对端协商？
- 多数场景**本端规划**（值越小优先级越高）。
- 语音 Bypass 场景标注**与对端协商**（如 65535），需保证：PCF 下发的语音专有承载规则优先级 > 本地语音专有承载规则 > PCF 下发的 any-to-any 动态规则 > 本地 any-to-any 预定义规则。

## 约束
- RULENAME 长度 1~63，不支持空格，不区分大小写。`severity=medium`
- 多条记录时 RULENAME+POLICYTYPE 组合不能完全相同（唯一性）。`severity=high`
- RULENAME 需与 UPF/PCF 上的预定义规则名一致（跨网元一致性）。`severity=critical`
- 最大记录数：RULESPECTYPE=DEFAULT（或不指定）时 8000；SPECIFICATION_LIMITED 时 100000。配置达 98% 上报 ALM-135602215 告警，≤95% 恢复。`severity=medium`
- PRIORITY 取值 1~4294967295，默认 4294967295（最低优先级），值越小优先级越高。`severity=low`
- 同一 SDF 满足多规则时取高优先级；同值时动态规则 > 预定义规则。`severity=medium`
- POLICYTYPE=PCC 时 POLICYNAME 必须是已配 PCCPOLICYGRPNM；POLICYTYPE=QOS 时必须是已配 QOSPROPNAME。`severity=high`
- 执行后立即生效。`severity=low`
