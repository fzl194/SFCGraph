# 配置HTTP本端实体组（ADD HTTPLEGRP）

> 命令静态知识见 [ADD HTTPLEGRP](command/UNC/20.15.2/ADD-HTTPLEGRP.md)。本页只讲配置生成实例化时这条命令**怎么配**——按配置维度组织成"字典"，**不**逐特性罗列取值。

## 配置方法

HTTP 本端实体组是 HTTPLE 的承载容器：本命令极简，每激活一个 NF（AMF/NRF/BSF）的服务化角色，建一个组，组号随后被 [ADD HTTPLE](task/UNC/20.15.2/0-00037.md) 的 `HTTPLEGRPIDX` 引用。本命令的核心配置空间仅 2 个维度：**组索引**（本端规划，按 NF 分配）和**描述**（可选，用 NF 名标识组归属）。本组须先于其下属 HTTPLE 建立。

### 配置维度 1：组索引（参数 INDEX，必选）

| 取值 | 作用 | 说明 |
|---|---|---|
| 整数 1~64（本端规划） | 本端唯一标识一个 HTTP 本端实体组，被 HTTPLE 的 HTTPLEGRPIDX 引用 | 按 NF 分配典型取值：AMF=1、NRF=2、BSF=3；本端规划，无取值分叉 |

### 配置维度 2：描述（参数 DESC，可选）

| 取值 | 作用 | 说明 |
|---|---|---|
| 字符串 0~255 字符 | 标识该组归属的 NF 名（建议配） | 全网规划；典型取值 `"AMF"` / `"NRF"` / `"BSF"` |

> 同一组下最多 32 个 HTTPLE 成员（SERVER+CLIENT 成对），超规格需新建组。

典型脚本：
```
ADD HTTPLEGRP:INDEX=1,DESC="AMF";
ADD HTTPLEGRP:INDEX=2,DESC="NRF";
ADD HTTPLEGRP:INDEX=3,DESC="BSF";
```

**步骤位置**：在所属 feature 编排中通常位于"步骤 1：配置 HTTP 本端实体组"——是服务化接口配置的起点，先于所有 [ADD HTTPLE](task/UNC/20.15.2/0-00037.md) 成员配置。

## 决策点：组的 NF 归属划分（DP 0-02055）

问：每激活一个 NF 的服务化角色，是否为它单独建一个组？

| 选项 | 联动参数 | 适用场景 |
|---|---|---|
| 每 NF 一个组 | `INDEX=<NF序号>` + `DESC="<NF名>"` | 主流做法：AMF/NRF/BSF 各一组，SERVER+CLIENT 成员配在同组 |
| 多 NF 共用一个组 | 多对 SERVER+CLIENT 共置一组 | NF 数少、成员总数 ≤32 时可合并 |

### 决策点：DESC 是否填写（DP 0-02056）

问：DESC 字段是否需要填写 NF 名标识？

| 选项 | 联动参数 | 适用场景 |
|---|---|---|
| 填 NF 名 | `DESC="AMF"` / `"NRF"` / `"BSF"` | 建议：便于运维识别组归属 |
| 留空 | 不配 DESC | DESC 为可选，留空不影响功能但不利于维护 |

## 约束

- rule-0-02136 · severity=critical · 本组须先于 ADD HTTPLE 建立，否则 HTTPLE 的 HTTPLEGRPIDX 引用失败 — 后果：HTTPLE 无法关联到组，服务化链路建链失败
- rule-0-02137 · severity=high · 同一个 HTTP 本端实体组下最多只能配置 32 个 HTTPLE 成员 — 后果：超规格新增成员失败
- rule-0-02138 · severity=high · INDEX 取值范围 1~64，取值本端唯一 — 后果：组索引冲突，引用歧义
- rule-0-02139 · severity=warning · 单板最多可输入 64 条 HTTPLEGRP 记录 — 后果：超规格新增失败
- rule-0-02140 · severity=info · 执行后立即生效 — 后果：无延迟，可立即被下游 HTTPLE 引用
