# 改进后三层图谱定义

> 定位：在现有业务图谱、特性图谱、任务层、命令图谱资产基础上，统一定义“静态三层 + 动态任务层”的完整图谱模型。
>
> 核心变化：三层图谱不再按“业务 / 特性 / 命令”单线向下硬拆，而是拆成两列：
>
> - 静态图谱：系统中稳定存在什么对象、对象之间有哪些稳定事实。
> - 动态图谱：一次配置、开通、验证或方案落地时，如何把静态对象组织成可执行任务链。

---

## 1. 一句话定义

改进后的三层图谱是：

```text
静态三层：
  业务图谱：BusinessDomain / NetworkScenario / ConfigurationSolution
  特性图谱：Feature / SubFeature / License
  命令图谱：MMLCommand / CommandParameter / ConfigObject

动态层：
  Task / TaskRule / DecisionPoint / task_relations
```

静态层回答：

```text
产品、业务和命令世界里稳定存在什么。
```

动态层回答：

```text
为了完成某次配置目标，应该选择哪些静态对象、按什么顺序执行、受哪些规则约束、在哪些地方分叉。
```

---

## 2. 总体结构

```text
                 静态业务层
  BusinessDomain -> NetworkScenario -> ConfigurationSolution
                                      |
                                      | uses_task
                                      v
                 动态任务层
       Solution Task / Feature Task / Compound Task / Atom Task
              |           |              |              |
              |           |              |              v
              |           |              |        MMLCommand
              |           |              |
              |           |              v
              |           |        task_relations
              |           |
              v           v
           Feature / SubFeature / License

                 静态命令层
       MMLCommand -> CommandParameter
            |
            v
       ConfigObject
```

端到端执行链可以理解为：

```text
ConfigurationSolution(static)
  -> Solution Task(dynamic)
  -> Feature Task(dynamic)
  -> Compound Task(dynamic)
  -> Atom Task(dynamic)
  -> MMLCommand(static)
  -> CommandParameter / ConfigObject(static)
```

## 3. 静态图谱定义

### 3.1 静态业务图谱

业务图谱表达稳定业务问题空间。

保留对象：

| 对象 | 中文名 | 定位 |
| --- | --- | --- |
| `BusinessDomain` | 业务域 | 长期存在的工程知识空间，如接入与会话管理、业务感知 |
| `NetworkScenario` | 场景 | 一线反复出现的稳定需求或问题情境 |
| `ConfigurationSolution` | 配置方案 | 针对场景形成的稳定方案模板或机制组合 |
| `SemanticObject` | 语义对象 | 业务意图到能力和配置对象之间的桥 |
| `EvidenceSource` | 证据 | 原始 md / excel / pdf / txt 来源 |

业务图谱保留的稳定关系：

| 起点 | 关系 | 终点 | 说明 |
| --- | --- | --- | --- |
| `BusinessDomain` | `contains` | `NetworkScenario` | 业务域包含场景 |
| `NetworkScenario` | `instantiated_as` | `ConfigurationSolution` | 场景实例化为方案 |
| `ConfigurationSolution` | `uses_feature` | `Feature / SubFeature` | 方案需要哪些能力 |
| `ConfigurationSolution` | `uses_semantic_object` | `SemanticObject` | 方案涉及哪些业务语义 |
| `ConfigurationSolution` | `uses_task` | `Task` | 方案声明使用哪些任务模板，非执行顺序 |

业务图谱不直接引用：

```text
MMLCommand
CommandParameter
底层 ConfigObject 的参数细节
```

业务方案可以声明自己需要某些特性或任务，但一次具体执行时，应由动态层选择或实例化 `Solution Task`。

### 3.2 静态特性图谱

特性图谱表达产品能力世界。

保留对象：

| 对象 | 中文名 | 定位 |
| --- | --- | --- |
| `Feature` | 特性 | 产品正式能力，有正式特性编号 |
| `SubFeature` | 子特性 | 单个 Feature 内部无正式编号但可独立引用的稳定细分能力 |
| `License` | License 控制项 | 能力授权控制项 |
| `EvidenceSource` | 证据 | 特性概述、License 文档、清单等来源 |

特性图谱保留的稳定关系：

| 起点 | 关系 | 终点 | 说明 |
| --- | --- | --- | --- |
| `Feature` | `has_subfeature` | `SubFeature` | 正式特性内部的稳定细分 |
| `Feature / SubFeature` | `depends_on` | `Feature / SubFeature` | 能力强前置 |
| `Feature / SubFeature` | `conflicts_with` | `Feature / SubFeature` | 能力互斥 |
| `Feature / SubFeature` | `cooperates_with` | `Feature / SubFeature` | 能力协同 |
| `Feature / SubFeature` | `requires_license` | `License` | 能力受 License 门控 |
| `Feature / SubFeature` | `uses_semantic_object` | `SemanticObject` | 能力承接业务语义 |

注意：

```text
Feature depends_on Feature
Feature requires_license License
```

这类是静态能力事实，应保留在特性图谱。

当这些事实影响配置执行时，再投影成动态层 `TaskRule` 或 `task_relations`，不要把静态事实直接删除。

### 3.3 静态命令图谱

命令图谱表达 MML 命令、参数、配置对象的稳定事实。

保留对象：

| 对象 | 中文名 | 定位 |
| --- | --- | --- |
| `MMLCommand` | MML 命令 | 一条正式命令 |
| `CommandParameter` | 命令参数 | 命令参数、取值范围、枚举、条件必选 |
| `ConfigObject` | 配置对象 | 命令创建、修改、引用或查询的稳定对象 |
| `EvidenceSource` | 证据 | 命令手册、内网参数表、参数引用规则 |

命令图谱保留的稳定关系：

| 起点 | 关系 | 终点 | 说明 |
| --- | --- | --- | --- |
| `MMLCommand` | `has_parameter` | `CommandParameter` | 命令包含参数 |
| `MMLCommand` | `creates / modifies / sets / deletes / queries / operates_on` | `ConfigObject` | 命令操作对象 |
| `CommandParameter` | `conditional_required` | `CommandParameter` | 同命令内参数条件必选 |
| `CommandParameter` | `references` | `CommandParameter` | 跨命令参数值引用 |
| `ConfigObject` | `refers_to / contains / depends_on / conflicts_with` | `ConfigObject` | 对象间稳定关系 |

例如：

```text
ADD URRGROUP.UPURRNAME1 references ADD URR.URRNAME
PCCPOLICYGRP refers_to URRGROUP
RULE refers_to FLOWFILTER
RULE refers_to PCCPOLICYGRP
```

这些是命令层静态事实。动态层使用这些事实，定义：

```text
task_relations.propagated_context
TaskRule.binding_rule
TaskRule.consistency_rule
DecisionPoint 参数影响
```

## 4. 动态任务层标准 Schema

动态层是本次改进的核心。

动态层不是案例集合，而是一套标准对象模型：

```text
Task
TaskRule
DecisionPoint
TaskRelation
TaskParameterBinding
```

本节中的字段名、枚举值、对象关系是标准 schema。

后文出现的“配置 URR”“计费三件套”“内容计费”等只允许作为：

```text
Task 实例
TaskRule 实例
DecisionPoint 实例
TaskRelation 实例
定义示例
```

不得反向作为新的标准对象类型，也不得把具体案例名固化进 schema 枚举。

其中：

| 对象 | 定位 |
| --- | --- |
| `Task` | 可复用的动态任务模板 |
| `TaskParameterBinding` | Task 内部对命令参数的取值、引用或决策控制 |
| `TaskRule` | Task 执行时必须满足的规则 |
| `DecisionPoint` | Task 执行中的分叉点 |
| `TaskRelation` | Task 与 Task 之间的编排关系 |

### 4.1 Task 标准定义

`Task` 不是产品官方对象，也不是命令本身。

`Task` 是：

```text
某个配置、开通、验证、规划或方案落地动作的可编排描述。
```

最小粒度不是“一条命令”，而是：

```text
一条命令在某个配置意图下的一种配法。
```

#### 4.1.1 Task 字段

| 字段 | 类型 | 必选 | 说明 |
| --- | --- | --- | --- |
| `task_id` | string | 是 | 全局唯一 ID，建议包含 NF、版本和对象类型 |
| `task_logical_name` | string | 是 | 任务逻辑名，不要求来自产品官方名词 |
| `task_layer` | enum | 是 | `atom / compound / feature / solution / generalized` |
| `task_intent` | string | 是 | 该任务要完成的配置、开通、验证或方案动作 |
| `task_category` | enum | 是 | `plan / configure / verify / activate / rollback / delete / query` |
| `nf` | string | 是 | 网元或产品族 |
| `version` | string | 是 | 产品版本 |
| `status` | enum | 是 | `active / deprecated / draft / inferred` |
| `ref` | string | 否 | 当前 task 直接关联的静态对象 ID。ID 中已有对象类型，因此不拆 `command_ref / feature_ref / solution_ref` |
| `parameter_bindings` | `TaskParameterBinding[]` | 否 | 该任务内的参数绑定 |
| `source_evidence_ids` | string[] | 是 | 证据来源 |
| `confidence` | number | 否 | 证据置信度，范围 0 到 1 |
| `notes` | string | 否 | 人工补充说明 |

Task 不定义独立的输入契约或输出契约字段。

原因是 Task 已经通过命令和参数绑定表达所需配置值：

```text
需要什么值 -> parameter_bindings
值来自哪里 -> binding_type / variable_source / source_ref / decision_ref
后续谁引用 -> task_relations.propagated_context + 命令图谱 references
```

#### 4.1.2 task_layer 枚举

| 枚举值 | 标准含义 | 静态引用 |
| --- | --- | --- |
| `atom` | 单个静态命令在一个配置意图下的任务模板 | `ref` 指向 `MMLCommand` |
| `compound` | 多个 Task 组合成的可复用任务模板 | 不直接引用静态对象，由 `task_relations` 编排子 task |
| `feature` | 一个 Feature 或 SubFeature 的配置、验证或开通任务模板 | `ref` 指向 `Feature / SubFeature` |
| `solution` | 一个 ConfigurationSolution 的执行模板 | `ref` 指向 `ConfigurationSolution` |
| `generalized` | 跨特性、跨方案归纳出的泛化任务模板 | 通常不直接引用静态对象，通过下游 task 展开 |

#### 4.1.3 Task Schema

```yaml
task_id: string
task_logical_name: string
task_layer: atom | compound | feature | solution | generalized
task_intent: string
task_category: plan | configure | verify | activate | rollback | delete | query
nf: string
version: string
status: active | deprecated | draft | inferred

ref: string

parameter_bindings:
  - parameter_ref: string
    binding_type: fixed | variable | reference | decision_driven | derived | optional
    value: string
    source_ref: string
    decision_ref: string

source_evidence_ids:
  - string

confidence: number
notes: string
```

Task 只保留一个明确归口的静态引用字段：

```yaml
ref: UDG@20.15.2@MMLCommand@ADD URR
ref: UDG@20.15.2@Feature@GWFD-020301
ref: UDG@20.15.2@ConfigurationSolution@...
```

不再引入通用锚点对象，原因是静态关系已有各自归口：

```text
command / feature / solution -> Task.ref
parameter -> parameter_bindings.parameter_ref
license -> FeatureGraph: Feature requires_license License
config_object -> CommandGraph: MMLCommand operates_on ConfigObject
semantic_object -> BusinessGraph / ConfigurationSolution
```

### 4.2 TaskParameterBinding 标准定义

`TaskParameterBinding` 是 Task 内部的参数规划行。

它表达：

```text
在这个 Task 语境下，某个命令参数如何取值。
```

它不是 Task 和 CommandParameter 之间的“锚定关系”，也不是 task 之间的关系。

出现条件应改为：

```text
当一个 Task 展开后包含需要规划、固定、引用或由决策控制的命令参数时，可以出现 TaskParameterBinding。
```

常见落点：

| task_layer | 是否通常出现 | 说明 |
| --- | --- | --- |
| `atom` | 是 | 直接描述该命令在本 task 下的参数规划 |
| `compound` | 可选 | 仅当组合 task 需要声明跨子 task 的统一参数口径时出现；更常见做法是让子 atom task 自己持有参数绑定 |
| `feature` / `solution` / `generalized` | 通常不直接出现 | 通过 `task_relations` 展开到下游 task，再由下游 atom task 承载具体参数绑定 |

跨 task 的选择、拆分和参数影响不由 `TaskParameterBinding` 承担，而由 `DecisionPoint.options[].impacts[]` 承担：

```text
影响启用哪个子 task -> target_type: task
影响下游哪个决策点 -> target_type: decision_point
影响某个命令参数 -> target_type: parameter
```

因此，父 task 的 DecisionPoint 可以影响子 task 下不同命令的参数，但这种影响应记录在 DecisionPoint impact 中；子 task 的参数表只记录最终被影响的参数绑定或参数规划。

| 字段 | 类型 | 必选 | 说明 |
| --- | --- | --- | --- |
| `binding_id` | string | 是 | 参数绑定 ID |
| `task_ref` | string | 是 | 所属 Task |
| `parameter_ref` | string | 是 | 静态命令参数 ID |
| `binding_type` | enum | 是 | `fixed / variable / reference / decision_driven / derived / optional` |
| `variable_source` | enum | 否 | `variable` 类绑定的取值来源：`global_planned（全网规划）/ local_planned（本端规划）/ peer_planned（与对端协商）/ reference（已配置数据中获取）/ decision_driven（决策驱动）`。来自命令参数说明的"数据来源"。`fixed` 用 `value`，`reference` 用 `source_ref` |
| `value` | string | 否 | 固定值或取值模板 |
| `source_ref` | string | 否 | 引用来源，如前序 Task 传播的上下文 |
| `decision_ref` | string | 否 | 受哪个 DecisionPoint 控制 |
| `requiredness` | enum | 是 | `required / optional / conditional` |
| `source_evidence_ids` | string[] | 是 | 证据来源 |

```yaml
binding_id: string
task_ref: string
parameter_ref: string
binding_type: fixed | variable | reference | decision_driven | derived | optional
variable_source: global_planned | local_planned | peer_planned | reference | decision_driven
value: string
source_ref: string
decision_ref: string
requiredness: required | optional | conditional
source_evidence_ids:
  - string
```

### 4.3 基于真实产品 md 的 Task 字段审视

本节用真实产品文档反推 Task 实例，验证字段是否足够。

#### 4.3.1 内容计费：多 task 关系链

证据：

```text
output/.../计费功能/GWFD-020301 内容计费基本功能/部署UPF_28493406.md
```

该文档的操作流程明确分成：

```text
配置业务费率 -> 配置过滤规则 -> 配置业务策略规则 -> 配置USERPROFILE -> 配置缺省费率 -> 刷新生效
```

对应 Task 示例：

```yaml
task_id: UDG@20.15.2@Task@00001
task_logical_name: 配置URR
task_layer: atom
ref: UDG@20.15.2@MMLCommand@ADD URR
parameter_bindings:
  - parameter_ref: UDG@20.15.2@CommandParameter@ADD URR:URRNAME
    binding_type: variable
    variable_source: decision_driven
  - parameter_ref: UDG@20.15.2@CommandParameter@ADD URR:USAGERPTMODE
    binding_type: variable
    variable_source: decision_driven
  - parameter_ref: UDG@20.15.2@CommandParameter@ADD URR:OFFMETERINGTYPE
    binding_type: decision_driven
  - parameter_ref: UDG@20.15.2@CommandParameter@ADD URR:ONLMETERINGTYPE
    binding_type: decision_driven
```

```yaml
task_id: UDG@20.15.2@Task@00002
task_logical_name: 配置URR组
task_layer: atom
ref: UDG@20.15.2@MMLCommand@ADD URRGROUP
parameter_bindings:
  - parameter_ref: UDG@20.15.2@CommandParameter@ADD URRGROUP:URRGROUPNAME
    binding_type: variable
  - parameter_ref: UDG@20.15.2@CommandParameter@ADD URRGROUP:UPURRNAME1
    binding_type: reference
    source_ref: UDG@20.15.2@Task@00001.URRNAME
  - parameter_ref: UDG@20.15.2@CommandParameter@ADD URRGROUP:UPURRNAME2
    binding_type: reference
    source_ref: UDG@20.15.2@Task@00001.URRNAME
```

```yaml
task_id: UDG@20.15.2@Task@00101
task_logical_name: 计费三件套
task_layer: compound
ref: null
task_relations:
  - from_task_ref: UDG@20.15.2@Task@00001
    to_task_ref: UDG@20.15.2@Task@00002
    relation_type: precedes
    propagated_context: URRNAME
  - from_task_ref: UDG@20.15.2@Task@00002
    to_task_ref: UDG@20.15.2@Task@00003
    relation_type: precedes
    propagated_context: URRGROUPNAME
```

```yaml
task_id: UDG@20.15.2@Task@00201
task_logical_name: 内容计费基本功能
task_layer: feature
ref: UDG@20.15.2@Feature@GWFD-020301
task_relations:
  - from_task_ref: UDG@20.15.2@Task@00101
    to_task_ref: UDG@20.15.2@Task@00102
    relation_type: precedes
  - from_task_ref: UDG@20.15.2@Task@00102
    to_task_ref: UDG@20.15.2@Task@00103
    relation_type: precedes
```

字段审视结论：

| 字段 | 结论 |
| --- | --- |
| `ref` | 足够。atom 指向 MMLCommand，feature 指向 Feature，compound 为空。无需 `command_ref / feature_ref` 分列 |
| `parameter_bindings` | 必要。数据规划表中的“全网规划 / 本端规划 / 已配置数据中获取 / 决策驱动”都落在这里 |
| `task_relations` | 必要。URRNAME、URRGROUPNAME 这种跨命令引用不是 Task 输入输出字段，而是 relation 上的传播上下文 |
| `DecisionPoint` | 必要。计费方式、过滤模式会同时影响 task 选择和参数出现 |

#### 4.3.2 IPSec：同一骨架，多场景由 DecisionPoint 改命令和参数

证据：

```text
output/.../IPFD-015004 IPSec功能/特性配置/激活IPsec功能（普通IPv4 IPsec隧道）_01_10002.md
output/.../IPFD-015004 IPSec功能/特性配置/激活IPsec功能（普通IPv6 IPsec隧道）_01_10003.md
output/.../IPFD-015004 IPSec功能/特性配置/激活IPsec功能（IPv4 IPsec主备隧道）_01_10004.md
output/.../IPFD-015004 IPSec功能/特性配置/激活IPsec功能（GRE over IPsec）_01_10006.md
```

这些文档共享同一类骨架：

```text
VNRS侧VPN/接口/路由 -> IPsec侧VPN/接口 -> ACL保护流 -> IPsec提议 -> IKE提议 -> IKE对等体 -> IPsec策略 -> 应用策略 -> DPD可选
```

对应 Task 示例：

```yaml

task_logical_name: 建立IPsec隧道
task_layer: feature
ref: UDG@20.15.2@Feature@IPFD-015004
task_relations:
  - from_task_ref: UDG@20.15.2@Task@IPSEC-01001
    to_task_ref: UDG@20.15.2@Task@IPSEC-01002
    relation_type: precedes
  - from_task_ref: UDG@20.15.2@Task@IPSEC-01002
    to_task_ref: UDG@20.15.2@Task@IPSEC-01003
    relation_type: precedes
```

```yaml
task_id: UDG@20.15.2@Task@IPSEC-01001
task_logical_name: 配置VNRS侧隧道基础
task_layer: compound
ref: null
task_relations:
  - from_task_ref: UDG@20.15.2@Task@IPSEC-00011
    to_task_ref: UDG@20.15.2@Task@IPSEC-00012
    relation_type: precedes
    propagated_context: VRFNAME
```

```yaml
task_id: UDG@20.15.2@Task@IPSEC-00011
task_logical_name: 创建VNRS侧VPN实例
task_layer: atom
ref: UDG@20.15.2@MMLCommand@ADD L3VPNINST
parameter_bindings:
  - parameter_ref: UDG@20.15.2@CommandParameter@ADD L3VPNINST:VRFNAME
    binding_type: variable
    variable_source: local_planned
```

IPSec 的关键不是建 9 个 feature task，而是在一个骨架上挂决策点：

```yaml
decision_id: UDG@20.15.2@DecisionPoint@IPSEC-00001
owner_task_ref: UDG@20.15.2@Task@IPSEC-00001
decision_name: IPsec隧道场景选择
options:
  - option_name: 普通IPv4隧道
    impacts: []
  - option_name: IPv6隧道
    impacts:
      - target_type: command
        target_ref: UDG@20.15.2@MMLCommand@ADD IFIPV6ADDRESS
        effect_type: changes_command_set
        effect_detail: 替换IPv4地址配置命令
      - target_type: command
        target_ref: UDG@20.15.2@MMLCommand@ADD ACLRULEADV6IPSEC
        effect_type: changes_command_set
        effect_detail: 使用IPv6 ACL规则
  - option_name: IPv4主备
    impacts:
      - target_type: parameter
        target_ref: UDG@20.15.2@CommandParameter@ADD IPSECPOLICY:WORKMODE
        effect_type: sets_value_pattern
        effect_detail: Master_standby
      - target_type: command
        target_ref: UDG@20.15.2@MMLCommand@ADD ATTACHIKEPEER
        effect_type: adds
        effect_detail: 出现两次，分别绑定主备IKE对等体
  - option_name: GRE over IPsec
    impacts:
      - target_type: command
        target_ref: UDG@20.15.2@MMLCommand@ADD GRETUNNEL
        effect_type: adds
        effect_detail: 额外建立GRE隧道
```

字段审视结论：

| 字段 | 结论 |
| --- | --- |
| `ref` | 足够。Feature task 指向 `Feature@IPFD-015004`，atom task 指向具体命令 |
| `task_layer` | 必要。IPSec 同时需要 feature、compound、atom 三种粒度 |
| `task_relations` | 必要。9 步骨架是 task 间关系，不应塞进一个超长 command list |
| `DecisionPoint.impacts` | 必要且必须支持 `task / command / parameter / decision_point`。IPSec 场景会改命令集合，也会改参数 |
| `TaskParameterBinding` | 只负责命令参数规划，不负责表达场景选择。场景选择由 DecisionPoint 负责 |

#### 4.3.3 字段修正结论

从内容计费和 IPSec 两类真实 md 看，Task 核心字段应收敛为：

```yaml
task_id: string
task_logical_name: string
task_layer: atom | compound | feature | solution | generalized
ref: string | null
nf: string
version: string
task_intent: string
task_category: configure | verify | activate | rollback | delete | query
parameter_bindings: []
task_relations: []
source_evidence_ids: []
status: active | deprecated | draft | inferred
```

不需要的字段或设计：

| 设计 | 处理 |
| --- | --- |
| `command_ref / feature_ref / solution_ref` 三字段 | 合并为 `ref` |
| 独立输入输出契约 | 删除，由 `parameter_bindings` 和 `task_relations.propagated_context` 表达 |
| 通用锚点对象 | 删除，各静态关系回各自图谱 |
| 场景变体字段 | 不加，由 `DecisionPoint.options[].impacts[]` 表达 |

### 4.4 基于真实产品 md 的动态层协同审视

动态层不能只看 `Task` 单对象。真实产品 md 里的配置活动，本质是一张由五类对象协同工作的动态图：

| 对象 | 在动态层中的职责 | 不应该承担的职责 |
| --- | --- | --- |
| `Task` | 表达一个可执行或可展开的配置动作节点 | 不表达跨节点顺序、分支裁剪和规则校验 |
| `TaskRelation` | 表达拆分、包含、顺序、依赖、可选边和上下文传播 | 不表达命令静态参数引用本身 |
| `DecisionPoint` | 表达用户、场景或条件选择，并把选择传播到 task、decision、command、parameter | 不承载参数取值明细表 |
| `TaskParameterBinding` | 表达命令承载 task 内的参数规划、引用、固定值、决策驱动值 | 不表达 task 拆分和任务选择 |
| `TaskRule` | 表达执行时必须满足的约束，校验 task、relation、decision、parameter 的组合结果 | 不替代静态图谱中的稳定事实 |

因此，动态层的运行模型应是：

```text
选择根 Task
-> 按 TaskRelation 展开下级 Task
-> 触发当前 Task 的 DecisionPoint
-> DecisionPoint impacts 裁剪 task、锁定子决策点、改变命令集合或参数模式
-> 按 TaskRelation 传播上下文
-> 在命令承载 task 上落 TaskParameterBinding
-> 用 TaskRule 校验组合后的任务链
-> 形成最终命令序列和参数规划
```

#### 4.4.1 内容计费：关系链、决策传播、规则校验同时存在

证据：

```text
output/.../计费功能/GWFD-020301 内容计费基本功能/部署UPF_28493406.md
ConfigTask/样板-计费功能-统一task架构.md
ConfigTask/计费场景-task资产/tasks/task-00001.yaml
ConfigTask/计费场景-task资产/tasks/task-00101.yaml
ConfigTask/计费场景-task资产/decision_points/dp-00001.yaml
ConfigTask/计费场景-task资产/decision_points/dp-00002.yaml
```

内容计费文档不是简单的“若干条命令列表”，而是：

```text
配置业务费率 -> 配置过滤规则 -> 配置业务策略规则 -> 配置USERPROFILE -> 配置缺省费率 -> 刷新生效
```

它反推出的动态层协同是：

| 协同点 | 应落对象 | 说明 |
| --- | --- | --- |
| URR、URRGROUP、PCCPOLICYGRP、RULE、USERPROFILE、REFRESHSRV 的执行链 | `TaskRelation` | 表达 `precedes / depends_on / must_be_last` |
| `URRNAME` 被 URRGROUP 引用，`URRGROUPNAME` 被策略组或规则引用 | `TaskRelation.propagated_context` + 命令图谱 `references` | 静态层说明“能引用”，动态层说明“这次配置链中从谁传给谁” |
| 在线、离线、融合计费选择 | `DecisionPoint` | 父 task 选择影响启用哪些子 task，也可以锁定子 task 的参数决策 |
| `USAGERPTMODE / OFFMETERINGTYPE / ONLMETERINGTYPE` 等值 | `TaskParameterBinding` | 只在 `ADD URR` 这类命令承载 task 上落参数规划 |
| URRGROUP 必须包含在线和离线 URR、RULE 优先级约束、刷新必须最后执行 | `TaskRule` | 校验 relation、parameter 和 command instance 的组合结果 |

这里最重要的结论是：父 task 的决策不是直接改子 task 的字段，而是通过 `DecisionPoint.options[].impacts[]` 传播。

```yaml
decision_id: UDG@20.15.2@DecisionPoint@DP-A
owner_task_ref: UDG@20.15.2@Task@00301
decision_name: 计费方式选择
options:
  - option_name: 在线计费
    impacts:
      - target_type: task
        target_ref: UDG@20.15.2@Task@00202
        effect_type: adds
      - target_type: decision_point
        target_ref: UDG@20.15.2@DecisionPoint@DP-B
        effect_type: selects_option
        effect_detail: 在线
```

子 task 仍然可以有自己的决策点：

```yaml
decision_id: UDG@20.15.2@DecisionPoint@DP-B
owner_task_ref: UDG@20.15.2@Task@00001
decision_name: URR计费模式参数选择
options:
  - option_name: 在线
    impacts:
      - target_type: parameter
        target_ref: UDG@20.15.2@CommandParameter@ADD URR:USAGERPTMODE
        effect_type: sets_value
        effect_detail: ONLINE
```

这说明动态层需要支持“决策点影响另一个决策点”，否则只能把在线、离线、融合复制成多套 task，泛化能力会很差。

#### 4.4.2 IPSec：同一骨架由决策点切换场景变体

证据：

```text
output/.../IPFD-015004 IPSec功能/特性配置/激活IPsec功能（普通IPv4 IPsec隧道）_01_10002.md
output/.../IPFD-015004 IPSec功能/特性配置/激活IPsec功能（普通IPv6 IPsec隧道）_01_10003.md
output/.../IPFD-015004 IPSec功能/特性配置/激活IPsec功能（IPv4 IPsec主备隧道）_01_10004.md
output/.../IPFD-015004 IPSec功能/特性配置/激活IPsec功能（GRE over IPsec）_01_10006.md
```

这些 md 共享配置骨架：

```text
VNRS侧VPN/接口/路由 -> IPsec侧VPN/接口 -> ACL保护流 -> IPsec提议 -> IKE提议 -> IKE对等体 -> IPsec策略 -> 应用策略 -> DPD可选
```

协同结论：

| 真实 md 现象 | 动态层表达 |
| --- | --- |
| IPv4、IPv6、主备、GRE over IPsec 的骨架相似 | 建一个 feature task 或 generalized task 作为根 |
| 具体场景替换地址命令、ACL 命令、策略参数或增加 GRE task | 用 `DecisionPoint.impacts` 指向 `command / parameter / task` |
| 9 步流程有稳定顺序 | 用 `TaskRelation.precedes / depends_on`，不要塞成一个超长 task |
| DPD 这类可选配置 | 用 `TaskRelation.requiredness: optional` 或受 `condition_ref` 控制 |
| 双侧配置、PSK 一致性、保护流匹配 | 用 `TaskRule.consistency_rule / binding_rule` |

IPSec 证明：动态层表达的是“骨架 + 变体”，不是把每个 md 文件都固化成独立 schema。

#### 4.4.3 L2TP：不是 1 step = 1 task，而是按业务阶段聚合

证据：

```text
output/.../GWFD-020412 L2TP VPN/激活L2TP VPN/本地配置方式激活L2TP VPN_40342129.md
ConfigTask/data/UDG/20.15.2/split_balance_conclusion.md
```

L2TP 文档包含 License、VPN 实例、L2TP 组、全局 L2TP、APN 绑定、软参、逻辑接口、客户端地址、PPP 可选、N4 加密可选等内容。

如果机械按步骤拆，会得到过碎 task；如果只建一个大 task，又会丢失引用和可选分支。更合理的协同拆法是：

| 业务阶段 | 可能的 Task 粒度 | 关键协同对象 |
| --- | --- | --- |
| 授权门控 | 激活 L2TP License | `TaskRule.dependency_rule` 或特性图谱 license 投影 |
| 基础对象 | 创建 VPN 实例、创建 L2TP 组 | `TaskRelation.depends_on` + `propagated_context: VPNINSTANCE` |
| 全局能力 | 设置全局 L2TP | `TaskRelation.precedes` |
| APN 绑定 | 创建或选择 APN、设置 APN L2TP 属性 | `TaskParameterBinding.reference` + `TaskRule.binding_rule` |
| 承载配置 | 软参、逻辑接口、客户端地址 | `TaskRelation.contains / precedes` |
| 可选增强 | PPP 认证、N4 加密 key | `DecisionPoint` + `TaskRelation.condition_ref` |
| 跨端一致性 | 对端 SMF 必须配置相同 key | `TaskRule.cross_nf_sync_rule` |

这也解释了 `TaskRelation` 需要同时表达分解关系和执行关系：

```yaml
relation_type: contains
from_task_ref: UDG@20.15.2@Task@L2TP-FEATURE
to_task_ref: UDG@20.15.2@Task@L2TP-PPP
requiredness: conditional
condition_ref: UDG@20.15.2@DecisionPoint@L2TP-AUTH.option:enable_ppp
```

#### 4.4.4 eDRX：配置范围选择影响 task 分支和规则优先级

证据：

```text
output/.../GWFD-110661 5G eDRX模式/激活5G eDRX模式_58609938.md
```

eDRX 文档显示 APN 级缓存配置和全局级缓存配置可以同时存在，但优先级是：

```text
APN 级配置 > 全局配置
```

动态层表达：

| 真实 md 现象 | 动态层表达 |
| --- | --- |
| 只配 APN 级、只配全局级、两者都配 | `DecisionPoint`：配置范围选择 |
| APN 级涉及 `ADD APN` 后再设置 APN 级缓存参数 | `TaskRelation.depends_on` + `TaskParameterBinding.reference` |
| 全局级是另一组可选 task | `TaskRelation.requiredness: optional / conditional` |
| APN 级优先于全局级 | `TaskRule.selection_rule` 或 `TaskRule.consistency_rule` |

该例子说明 `TaskRule` 不只是校验命令参数，也要校验多条 task 分支合并后的优先级语义。

#### 4.4.5 动态层字段结论

基于上述真实 md，动态层的字段收敛应遵守：

| 结论 | 说明 |
| --- | --- |
| 保留 `Task / TaskRelation / DecisionPoint / TaskParameterBinding / TaskRule` 五类对象 | 少任何一个，都会把分解、分支、参数、校验混到一起 |
| `Task.ref` 保持单字段 | 静态对象 ID 自带类型，三字段拆分没有必要 |
| `TaskParameterBinding` 只在需要表达参数规划时出现 | 它不负责 task 选择，也不负责跨 task 编排 |
| `DecisionPoint.impacts` 必须支持 `task / decision_point / command / parameter` | 计费和 IPSec 都需要跨层影响 |
| `TaskRelation` 需要支持 `condition_ref` | L2TP 可选 PPP、eDRX APN/全局范围、IPSec DPD 都需要条件边 |
| `TaskRule.rule_scope` 应覆盖 `task / relation / parameter / feature / solution` | 真实规则经常约束的是组合结果，而不是单条命令 |

### 4.5 大 Task、子 Task 与复用 Task 的定义语义

动态层允许 Task 递归组合。一个上层 Task 可以拆分为多个下层 Task；下层 Task 也可以继续拆分，直到命令级 atom Task。

标准表达如下：

```text
Feature Task
  contains / expands_to
Step Task
  contains / expands_to
Atom Task
  ref -> MMLCommand
```

这种拆分不通过 `Task.children` 字段表达，而通过 `TaskRelation` 表达：

```yaml
relation_type: contains
from_task_ref: UDG@20.15.2@Task@FeatureA
to_task_ref: UDG@20.15.2@Task@Step3
```

执行顺序通过另一类 relation 表达：

```yaml
relation_type: precedes
from_task_ref: UDG@20.15.2@Task@Step2
to_task_ref: UDG@20.15.2@Task@Step3
```

因此，同一对 Task 可以同时存在不同语义的 relation：

| relation_type | 定义语义 |
| --- | --- |
| `contains` | 父 Task 结构上包含子 Task |
| `expands_to` | 父 Task 运行时展开为子 Task |
| `precedes` | 一个 Task 在另一个 Task 之前执行 |
| `depends_on` | 一个 Task 依赖另一个 Task 提供的上下文 |

#### 4.5.1 父 Task 固定子 Task 的配置方法

一个子 Task 可以保留多个配置方法；当它被某个父 Task 使用时，父 Task 可以约束它采用其中一种固定方法。

当前字段支持两种定义语义。

第一种：父 Task 的决策点固定子 Task 的决策点。

```yaml
decision_id: UDG@20.15.2@DecisionPoint@PARENT-DP
owner_task_ref: UDG@20.15.2@Task@FeatureA
decision_name: 特性A配置路径选择
options:
  - option_name: 标准路径
    impacts:
      - target_type: decision_point
        target_ref: UDG@20.15.2@DecisionPoint@STEP3-DP
        effect_type: selects_option
        effect_detail: option:method_A
```

第二种：父 Task 的规则约束子 Task 的决策结果。

```yaml
rule_id: UDG@20.15.2@TaskRule@FEATUREA-STEP3-METHOD
owner_task_ref: UDG@20.15.2@Task@FeatureA
rule_scope: relation
condition:
  expression_type: task_selected
  expression: Task@Step3 selected
constraint:
  constraint_type: must_select_one
  target_refs:
    - UDG@20.15.2@DecisionPoint@STEP3-DP
  expression: selected_option == method_A
```

两者语义不同：

| 表达方式 | 适用定义 |
| --- | --- |
| `DecisionPoint.impacts -> decision_point` | 父 Task 的某个选择会锁定子 Task 的配置方法 |
| `TaskRule.rule_scope: relation` | 父 Task 天然要求子 Task 只能采用某种方法 |

这个定义支持“步骤 3 是下面某个 Task 的固定配置方法”。子 Task 可以是 compound Task，也可以是 atom Task；如果子 Task 是 atom Task，则父约束可以直接影响其参数：

```yaml
impacts:
  - target_type: parameter
    target_ref: UDG@20.15.2@CommandParameter@ADD URR:USAGERPTMODE
    effect_type: sets_value
    effect_detail: ONLINE
```

#### 4.5.2 Task 复用与合并后的定义语义

随着资产增多，两个原本独立的 Task 可以归并为同一个可复用 Task。定义层面不需要新增对象类型。

合并后的 Task 保留一个稳定身份：

```yaml
task_id: UDG@20.15.2@Task@CREATE-APN
task_layer: atom
task_intent: 创建或规划APN配置对象
ref: UDG@20.15.2@MMLCommand@ADD APN
source_evidence_ids:
  - L2TP_LOCAL_CONFIG_MD
  - EDRX_ACTIVATE_MD
```

不同父 Task 通过 `TaskRelation` 复用它：

```yaml
relation_type: contains
from_task_ref: UDG@20.15.2@Task@L2TP-FEATURE
to_task_ref: UDG@20.15.2@Task@CREATE-APN
```

```yaml
relation_type: contains
from_task_ref: UDG@20.15.2@Task@EDRX-FEATURE
to_task_ref: UDG@20.15.2@Task@CREATE-APN
```

如果两个父 Task 使用同一个子 Task，但参数值不同，差异不应拆回两个重复 Task，而应归入以下对象：

| 差异类型 | 定义归口 |
| --- | --- |
| 子 Task 自身有多个配置方法 | 子 Task 的 `DecisionPoint` |
| 父 Task 固定子 Task 的某种方法 | 父 Task 的 `DecisionPoint.impacts` 或 `TaskRule` |
| 命令参数值不同 | `TaskParameterBinding` 或 `DecisionPoint.impacts -> parameter` |
| 上下文来源不同 | `TaskRelation.propagated_context` |
| 可选步骤不同 | `TaskRelation.requiredness / condition_ref` |

真实案例中，L2TP 与 eDRX 都会用到 APN 相关步骤；内容计费中的 `ADD URR` 也会被在线、离线、融合计费多种路径复用。当前字段允许它们合并成一个稳定 Task，再由父 Task 的 relation、rule、decision 保留个性化语义。

## 5. TaskRule 标准 Schema

### 5.1 TaskRule 的归口

`TaskRule` 表达动态执行时需要遵守的规则。

它不是静态事实本身，而是静态事实在某个 task 语境下的执行约束。

### 5.2 TaskRule 字段

| 字段 | 类型 | 必选 | 说明 |
| --- | --- | --- | --- |
| `rule_id` | string | 是 | 全局唯一 ID |
| `owner_task_ref` | string | 是 | 规则所属 Task |
| `rule_name` | string | 是 | 规则名 |
| `rule_type` | enum | 是 | 规则类型 |
| `rule_scope` | enum | 是 | `task / relation / parameter / feature / solution` |
| `condition` | object | 否 | 触发条件 |
| `constraint` | object | 是 | 约束内容 |
| `violation_effect` | string | 否 | 违反后影响 |
| `severity` | enum | 是 | `critical / high / medium / low / info` |
| `rule_expression_mode` | enum | 是 | `explicit / inferred / projected` |
| `rule_source_kind` | enum | 是 | `command / feature / business / config / projected` |
| `source_static_refs` | string[] | 否 | 投影来源的静态对象或静态边 |
| `source_evidence_ids` | string[] | 是 | 证据来源 |
| `status` | enum | 是 | `active / deprecated / draft / inferred` |

```yaml
rule_id: string
owner_task_ref: string
rule_name: string
rule_type: binding_rule | consistency_rule | dependency_rule | conditional_rule | ordering_rule | selection_rule | scope_rule | cardinality_rule | uniqueness_rule | delete_constraint_rule | cross_nf_sync_rule | performance_rule
rule_scope: task | relation | parameter | feature | solution

condition:
  expression_type: none | parameter_value | decision_option | task_selected | feature_selected | object_exists
  expression: string

constraint:
  constraint_type: must_exist | must_precede | must_reference | must_match | must_not_match | max_count | min_count | must_be_last | must_select_one | must_not_coexist
  target_refs:
    - string
  expression: string

violation_effect: string
severity: critical | high | medium | low | info
rule_expression_mode: explicit
rule_source_kind: config
source_static_refs:
  - string
source_evidence_ids:
  - string
status: active
```

### 5.3 推荐 rule_type

| rule_type | 管什么 | 常见来源 |
| --- | --- | --- |
| `binding_rule` | 跨命令、跨对象引用绑定 | 参数引用规则、部署数据表 |
| `consistency_rule` | 跨对象、跨网元、跨任务一致性 | 部署文档、业务专题 |
| `dependency_rule` | 前置依赖 | 特性依赖、License、命令引用 |
| `conditional_rule` | 条件触发参数、任务或分支 | 参数条件、操作说明 |
| `ordering_rule` | 执行顺序、最后执行 | 操作流程、注意事项 |
| `selection_rule` | 选哪类 task、特性或参数模式 | 方案文档、决策维度 |
| `scope_rule` | 生效范围、适用边界 | 特性概述、业务方案 |
| `cardinality_rule` | 数量上限、规格约束 | 命令 notes、规格章节 |
| `uniqueness_rule` | 唯一性 | 命令 notes、内网重复规则 |
| `delete_constraint_rule` | 删除保护 | 参数引用、删除约束 |
| `cross_nf_sync_rule` | 跨 NF 同步 | C/U 协同文档 |
| `performance_rule` | 性能影响 | 命令 notes、特性限制 |

## 6. DecisionPoint 标准 Schema

### 6.1 DecisionPoint 的归口

`DecisionPoint` 表达动态执行中的分叉。

它应挂在 Task 上：

```text
Task --has_decision--> DecisionPoint
```

业务、特性、命令文档中出现的选择点，最终都应投影到某个 Task 语境下。

### 6.2 DecisionPoint 字段

| 字段 | 类型 | 必选 | 说明 |
| --- | --- | --- | --- |
| `decision_id` | string | 是 | 全局唯一 ID |
| `owner_task_ref` | string | 是 | 所属 Task |
| `decision_name` | string | 是 | 决策点名称 |
| `decision_question` | string | 是 | 要回答的问题 |
| `decision_type` | enum | 是 | `single_choice / multi_choice / boolean / value_selection / object_selection` |
| `trigger_condition` | object | 否 | 何时触发 |
| `options` | `DecisionOption[]` | 是 | 可选项 |
| `default_option` | string | 否 | 默认选项 |
| `requiredness` | enum | 是 | `required / optional / conditional` |
| `source_evidence_ids` | string[] | 是 | 证据来源 |
| `status` | enum | 是 | `active / deprecated / draft / inferred` |

```yaml
decision_id: string
owner_task_ref: string
decision_name: string
decision_question: string
decision_type: single_choice | multi_choice | boolean | value_selection | object_selection

trigger_condition:
  expression_type: always | task_selected | feature_selected | parameter_value | previous_decision
  expression: string

options:
  - option_id: string
    option_name: string
    option_value: string
    impacts:
      - target_type: task | command | parameter | feature | solution | decision_point | semantic_object | config_object
        target_ref: string
        effect_type: adds | skips | excludes | requires | sets_value_pattern | changes_command_set | selects_option | selects_solution | selects_feature | changes_scope
        effect_detail: string

default_option: string
requiredness: required | optional | conditional
status: active
source_evidence_ids:
  - string
```

### 6.3 impacts.target_type

真实资产中已经存在“一个决策影响另一个决策”的情况，因此 `target_type` 应至少支持：

| target_type | 说明 |
| --- | --- |
| `task` | 启用、跳过、排斥或新增某个 task |
| `command` | 影响命令集合 |
| `parameter` | 设置参数取值模式 |
| `feature` | 启用或排除某个特性 |
| `solution` | 选择某个方案模板 |
| `decision_point` | 选择或触发另一个决策点 |
| `semantic_object` | 选择某类业务语义 |
| `config_object` | 影响配置对象族 |

DecisionPoint 的核心作用不是只改当前 task 的字段，而是做动态扇出：

```text
父 task 决策 -> 选择 / 排除 / 增加子 task
父 task 决策 -> 锁定子 task 上的另一个 DecisionPoint
任意 task 决策 -> 影响某个命令参数的取值模式或是否出现
```

例如计费场景中：

```text
Task@00301 计费配置 的“计费方式选择”
  在线 -> adds Task@00202 在线计费
       -> selects_option Task@00001 上的“计费方式选择=在线”

Task@00001 配置URR 的“计费方式选择=在线”
  -> ADD URR.USAGERPTMODE = ONLINE
  -> ADD URR.ONLMETERINGTYPE 条件出现
```

这种父子决策传播应记录在父 DecisionPoint 的 impact 中，子 DecisionPoint 不需要反向知道父决策。

推荐 `effect_type`：

```text
adds
skips_task
excludes
requires
sets_value_pattern
changes_command_set
selects_option
selects_solution
selects_feature
changes_scope
```

## 7. TaskRelation 标准 Schema

`TaskRelation` 表达 task 之间的动态编排。

| 字段 | 类型 | 必选 | 说明 |
| --- | --- | --- | --- |
| `relation_id` | string | 是 | 全局唯一 ID |
| `from_task_ref` | string | 是 | 起点 Task |
| `to_task_ref` | string | 是 | 终点 Task |
| `relation_type` | enum | 是 | 编排关系类型 |
| `requiredness` | enum | 是 | `required / optional / conditional` |
| `condition` | object | 否 | 条件关系的触发条件 |
| `condition_ref` | string | 否 | 触发该关系的决策点、决策选项或上游条件引用 |
| `propagated_context` | string[] | 否 | 从前序 task 传给后序 task 的上下文 |
| `source_static_refs` | string[] | 否 | 来源静态边或静态对象 |
| `source_evidence_ids` | string[] | 是 | 证据来源 |
| `confidence` | number | 否 | 证据置信度 |

```yaml
relation_id: string
from_task_ref: string
to_task_ref: string
relation_type: precedes | depends_on | must_be_last | excludes | fallback_to | parallel_with | expands_to | contains
requiredness: required | optional | conditional
condition:
  expression_type: none | decision_option | feature_selected | parameter_value | task_selected
  expression: string
condition_ref: string
propagated_context:
  - context_name: string
    source_ref: string
    target_ref: string
source_static_refs:
  - string
source_evidence_ids:
  - string
confidence: number
```

关系类型：

| relation_type | 说明 |
| --- | --- |
| `precedes` | A 通常先于 B 执行 |
| `depends_on` | B 依赖 A 提供或传播的配置上下文 |
| `must_be_last` | 目标 task 必须最后执行 |
| `excludes` | 两个 task 不能同时存在 |
| `fallback_to` | A 失败或不适用时转向 B |
| `parallel_with` | 可并行执行 |
| `expands_to` | 上层 Task 展开为下层 Task |
| `contains` | 组合 Task 包含子 Task |

`task_relations` 不替代静态命令图谱的参数引用关系。

它表达的是：

```text
在当前配置任务链里，哪个 task 提供或传播的配置上下文会被哪个 task 使用。
```

例如：

静态命令事实：

```text
ADD URRGROUP.UPURRNAME1 references ADD URR.URRNAME
```

动态任务关系：

```text
配置URR precedes 配置URRGROUP
propagated_context: URRNAME
```

## 8. 静态事实到动态规则的投影

改进后的核心机制是“静态事实投影”。

不是把所有规则都从静态层搬走，而是分两步：

1. 静态图谱保留稳定事实。
2. 动态层根据任务语境，把相关事实投影成 TaskRule / DecisionPoint / task_relations。

### 8.1 投影例子：命令参数条件

静态命令事实：

```text
ADD URR.ONLMETERINGTYPE conditional_required by USAGERPTMODE=ONLINE
ADD URR.OFFMETERINGTYPE conditional_required by USAGERPTMODE=OFFLINE
```

动态投影：

```text
DecisionPoint: 计费方式选择
  在线 -> USAGERPTMODE=ONLINE, 启用 ONLMETERINGTYPE
  离线 -> USAGERPTMODE=OFFLINE, 启用 OFFMETERINGTYPE
```

### 8.2 投影例子：命令间引用

静态命令事实：

```text
ADD URRGROUP.UPURRNAME1 references ADD URR.URRNAME
ADD PCCPOLICYGRP.URRGROUPNAME references ADD URRGROUP.URRGROUPNAME
```

动态投影：

```text
Task: 计费三件套
  配置URR -> 配置URRGROUP -> 配置PCCPOLICYGRP

TaskRule:
  URRGROUP 必须引用已创建 URR
  PCCPOLICYGRP 必须引用已创建 URRGROUP
```

### 8.3 投影例子：特性 License

静态特性事实：

```text
GWFD-020301 requires_license LKV3G5BCBC01
```

动态投影：

```text
Task: 内容计费特性配置
  depends_on License开启Task

TaskRule:
  执行内容计费配置前，必须开启 LKV3G5BCBC01
```

### 8.4 投影例子：业务方案

静态业务事实：

```text
CS-CH-02 在线计费方案 uses_feature GWFD-020300
CS-CH-02 在线计费方案 uses_feature GWFD-020301
```

动态投影：

```text
Task: 在线计费方案配置
  -> 内容计费基础 Feature Task
  -> 在线计费 Feature Task
  -> 配额耗尽动作 Task（可选）
```

## 9. 建模边界

### 9.1 禁止关系

| 禁止关系 | 原因 | 替代 |
| --- | --- | --- |
| `ConfigurationSolution -> MMLCommand` | 业务方案直接坠落命令层 | `ConfigurationSolution -> Solution Task -> Atom Task -> MMLCommand` |
| `Feature -> MMLCommand` | 跳过动态任务层 | `Feature Task -> Atom Task -> MMLCommand` |
| `Feature -> CommandParameter` | 特性过度下沉 | `Feature Task -> DecisionPoint -> parameter impact` |
| `BusinessRule -> CommandParameter` 直接固化参数值 | 业务规则不应拥有命令参数 | `DecisionPoint / TaskRule` |
| `ConfigObject` 承载完整参数定义 | 参数定义属于 CommandParameter | `MMLCommand -> CommandParameter` |

### 9.2 标准关系

```text
ConfigurationSolution -> Solution Task
Feature -> Feature Task
Atom Task -> ref -> MMLCommand
Feature Task -> ref -> Feature
Solution Task -> ref -> ConfigurationSolution
Task -> TaskRule
Task -> DecisionPoint
Task -> task_relations
Atom Task -> MMLCommand
```

## 10. 典型例子：内容计费

### 10.1 静态特性事实

```text
Feature: GWFD-020301 内容计费基本功能
requires_license: LKV3G5BCBC01
depends_on: GWFD-110101 SA-Basic
depends_on: GWFD-020300 在线计费（当使用在线计费系统）
depends_on: GWFD-010171 离线计费（当使用离线计费系统）
```

### 10.2 静态命令事实

```text
ADD URR creates URR
ADD URRGROUP creates URRGROUP
ADD PCCPOLICYGRP creates PCCPOLICYGRP
URRGROUP refers_to URR
PCCPOLICYGRP refers_to URRGROUP
ADD URR.USAGERPTMODE controls ONLMETERINGTYPE / OFFMETERINGTYPE
```

### 10.3 动态 Task 展开

```text
Task: 内容计费特性配置
  -> License 开启
  -> 配置 URR
  -> 配置 URRGROUP
  -> 配置 PCCPOLICYGRP
  -> 配置过滤条件
  -> 配置 FLOWFILTER 与绑定
  -> 配置 RULE
  -> 配置 USERPROFILE
  -> 配置 RULEBINDING
  -> 配置缺省费率
  -> SET REFRESHSRV must_be_last
```

### 10.4 动态规则

```text
URRGROUP 必须引用已配置 URR
PCCPOLICYGRP 必须引用已配置 URRGROUP
FLOWFILTER 必须绑定过滤条件，否则 RULE 无效
只绑定三四层 filter 的 RULE 优先级低于绑定七层 filter 的 RULE
any 兜底 RULE 优先级最低
SET REFRESHSRV 必须最后执行
SMF/UPF 的 RULENAME / POLICYTYPE / POLICYNAME 需要一致
```

### 10.5 动态决策

```text
计费方式选择：
  在线 -> ADD URR.USAGERPTMODE=ONLINE，启用在线计费 Feature Task
  离线 -> ADD URR.USAGERPTMODE=OFFLINE，启用离线计费 Feature Task
  融合 -> 同时配置在线/离线 URR，并叠加融合计费 Feature Task

匹配层次选择：
  L34 -> FILTER / FILTERIPV6 + FLTBINDFLOWF
  L7 -> L7FILTER + PROTBINDFLOWF
  混合 -> 两者都启用
```

## 11. 当前结论

改进后的完整三层图谱不是“业务 -> 特性 -> 命令”的单向静态链，而是：

```text
静态三层提供事实：
  业务事实
  能力事实
  命令事实

动态任务层定义执行语义：
  选择
  编排
  参数绑定
  规则校验
  决策分叉
```

最终目标是让每一层都能单独维护、单独演进：

```text
BusinessGraph 维护业务域 / 场景 / 方案
FeatureGraph 维护特性 / License / 能力关系
CommandGraph 维护命令 / 参数 / 配置对象
TaskGraph 维护 task / rule / decision / 编排关系
```

而配置执行或智能体执行时，只消费动态层展开后的 TaskGraph，并通过 `ref` 回查静态图谱，保证每一步都有能力、命令、参数和证据支撑。
