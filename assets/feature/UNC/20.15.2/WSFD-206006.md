# Smartphone异常信令节省

`WSFD-206006` · enhanced · required

## 适用NF（原文）

SGSN

## 定义

异常信令节省是指当由于未签约APN、网络故障等原因导致UE激活失败时，UE反复进行激活，从而产生大量异常信令时。 UNC 可以根据事先配置的抑制策略对UE进行抑制此现象发生，消除由于UE的异常重复激活行为对网络侧的影响。

## 客户价值

| 受益方 | 受益描述 |
| --- | --- |
| 运营商 | - 减少网络中的异常信令。<br>- 提高网络激活成功率。<br>- 避免网络拥塞和设备过载。<br>- 提升网络可用性和可靠性。 |
| 用户 | 用户不感知该特性。 |

## 应用场景

GERAN和UTRAN网络中，在Smartphone终端反复激活，产生大量异常信令的情况下，利用异常信令控制功能可以减少网络的信令负荷。

## 对系统的影响

本特性对系统无影响。

## 应用限制

只有对端GGSN支持时，才能使用Parking APN假激活功能。

## 原理概述

当未签约APN、网络故障、业务故障等原因导致Smartphone激活被拒绝或是激活成功后随即发生去激活流程时，Smartphone会由于其永远在线的特点而反复进行激活，从而产生大量异常信令，容易造成网络拥塞和设备过载。异常信令控制功能可抑制此现象发生，消除Smartphone的异常重复激活行为。

> **说明**
> - 本特性仅对导致异常激活的APN进行抑制，不影响其他正常APN的业务。
> - 被抑制激活的APN将一直无法激活成功，直到被激活唤醒。

异常信令控制功能的具体过程如下：

1. UNC 每5分钟判定一次用户激活次数是否达到异常激活门限次数，该门限次数由 [**SET SMARTACTPARA**](../../../../../OM参考/命令/UNC MML命令/业务服务管理/Pre 5G接入业务管理/控制面管理/业务安全管理/Smartphone管理/异常信令节省/激活抑制参数管理/设置激活抑制参数（SET SMARTACTPARA）_26305550.md) 命令中的参数 “ABNORACTTHRESH” 设置。如果1小时内用户激活次数未达到异常激活门限次数，则用户激活次数会被清零。
    - 如果用户激活次数超过阈值，则系统判断用户激活行为异常，执行[2](#ZH-CN_CONCEPT_0186930572__zh-cn_concept_0130429029_zh-cn_concept_0130429015_type)。
    - 如果用户激活次数没有超过阈值，则系统判断用户激活行为正常，用户进行正常的激活流程。
2. 识别Smartphone终端类型
  系统通过 [**ADD IMEILIB**](../../../../../OM参考/命令/UNC MML命令/业务服务管理/Pre 5G接入业务管理/控制面管理/业务安全管理/Smartphone管理/终端类型识别/IMEI库管理/增加IMEI库记录（ADD IMEILIB）_26145734.md) 命令建立IMEI（International Mobile Equipment Identity）库或通过 [**ADD APNNILIB**](../../../../../OM参考/命令/UNC MML命令/业务服务管理/Pre 5G接入业务管理/控制面管理/业务安全管理/Smartphone管理/终端类型识别/APNNI库管理/增加APNNI库记录（ADD APNNILIB）_26145736.md) 命令建立APN NI（Access Point Name Network Identifier）库来识别终端类型：
    - IMEI库
      系统内部建立各Smartphone终端类型与用户IMEI的对应关系数据库。移动性管理中，系统在用户首次发起的附着、RAU、Inter/Intra SGSN Relocation流程中获得用户IMEI信息，通过查询配置的IMEI库识别终端类型。如果系统在用户后续发起的附着、RAU、Inter/Intra SGSN Relocation流程中，获取到的IMEI和之前流程中获取的不同，则系统将重新查询配置的IMEI库识别终端类型。另外，系统也会在用户多次完成激活流程后，自动重新查询配置的IMEI库识别终端类型。
      > **说明**
      > IMEI的组成结构为TAC（Type Allocation Code）+SNR（Serial Number）+Spare digit，用于唯一标识一个用户终端。
    - APN NI库
      系统内部建立各Smartphone终端类型与用户APN对应关系的数据库。移动性管理中，系统通过附着、RAU、Handover等流程获得用户签约的APN；会话管理中，系统通过PDP上下文激活流程获得用户请求的APN，并且查询配置的APN NI库识别终端类型。
  > **说明**
  > IMEI库和APN NI库两者可以同时配置，优先级从高到低依次为：签约APN，IMEI，请求APN。
  系统可识别出的终端类型如下：
    - 4种已知类型：Android、Blackberry、iOS、Windows
    - 12种扩展类型：Custom Type 1～Custom Type 12
3. 系统针对不同的终端类型进行激活抑制，具体流程请参考 [激活抑制流程](实现原理_86930573.md#ZH-CN_CONCEPT_0186930573__section141029327322)。
4. 系统通过在唤醒定时器时间内，查看用户的激活情况来判断是否针对抑制规则触发抑制唤醒流程，具体请参考 [抑制唤醒流程](实现原理_86930573.md#ZH-CN_CONCEPT_0186930573__section1760755313328)。
