# UPF故障隔离

`WSFD-213005` · enhanced · required

## 所属目录

- [WSFD-213000](feature/UNC/20.15.2/WSFD-213000.md)

## 适用NF（原文）

SMF

## 定义

UPF故障隔离指：

- 当网络中存在基站与UPF之间的接口（S1-U接口或N3接口）故障时，SMF能够在选择UPF时弃选存在故障的UPF，从而保证用户业务正常进行。
- 当UPF与SMF之间的N4接口故障时，SMF能够将故障UPF上的用户去激活，在用户重新激活时选择正常的UPF，从而保证用户业务正常进行。

## 客户价值

| 受益方 | 受益描述 |
| --- | --- |
| 运营商 | 网络中存在基站与UPF之间的接口故障或者UPF与SMF之间的N4接口故障时，快速恢复用户业务。 |
| 用户 | 网络中存在基站与UPF之间的接口故障或者UPF与SMF之间的N4接口故障时，缩短业务中断时长，提升用户体验。 |

## 应用场景

已知网络中存在基站与UPF之间的接口故障时，可以开启本特性以避免SMF反复选择到存在接口故障的UPF。

为避免N4接口故障（或者UPF故障）导致用户业务持续受影响，可以开启本特性缩短业务中断时长 。

## 应用限制

- 本特性依赖AMF在Nsmf_PDUSession_CreateSMContext Request、Nsmf_PDUSession_UpdateSMContext Request消息中携带UeLocation信元，以使SMF能够获取最新的gNodeB ID信息。如果上述消息信元中未携带gNodeB ID，则将导致本特性无法生效。
- 本特性依赖MME在Create Session Request、Modify Bearer Request消息中携带User Location Information for SGW或User Location Information信元，以使SMF（GW-C融合部署）能够获取最新的eNodeB ID信息。SMF优先选择User Location Information for SGW信元中的eNodeB ID，当不存User Location Information for SGW信元时，选择User Location Information信元中的eNodeB ID。如果上述消息信元中未携带eNodeB ID，则将导致本特性无法生效。
- 本特性不对用户已有流程生效。

## 原理概述

**基站与UPF之间的接口故障场景**

当网络中存在基站与UPF之间的接口（S1-U接口或N3接口）故障时，UE会不断尝试发起会话激活流程。但是，在重新激活会话过程中，SMF依然会选择到接口故障的UPF，导致用户业务无法恢复。

为解决上述问题，运维人员可以在SMF侧配置存在接口故障的基站与UPF，使用户业务快速恢复。以 [图1](#ZH-CN_CONCEPT_0000001206900390__fig1153505141318) 为例，已知gNodeB 1、gNodeB 2与UPF 1之间的链路发生故障，运维人员在SMF侧配置存在接口故障的UPF组（包含UPF 1）和RAN组（包含gNodeB 1和gNodeB 2）。当UE通过gNodeB 1或gNodeB 2接入网络时，SMF在选择UPF时弃选UPF 1，选择UPF 2进行业务流程。

**N4接口故障场景**

N4接口故障可通过PFCP心跳消息和PFCP会话消息探测：

- 通过PFCP心跳消息探测N4接口故障机制参考[WSFD-010600 路径管理](../../可靠性/WSFD-010600 路径管理_68416219.md)特性。
- 当PFCP心跳消息还未探测到N4接口故障时，SMF可通过PFCP会话消息感知到N4接口故障。当SMF发送PFCP会话请求消息给UPF后，请求消息发送次数达到发送次数阈值（由**[SET T1N1PARA](../../../../../OM参考/命令/UNC MML命令/业务服务管理/接口管理/PFCP接口管理/PFCP消息可靠性管理/设置PFCP T1N1参数配置（SET T1N1PARA）_09651332.md)**的参数“发送次数阈值”配置）仍超时（由**[SET T1N1PARA](../../../../../OM参考/命令/UNC MML命令/业务服务管理/接口管理/PFCP接口管理/PFCP消息可靠性管理/设置PFCP T1N1参数配置（SET T1N1PARA）_09651332.md)**的参数“超时间隔”配置）未收到响应，SMF会判定会话流程失败，会去激活该UPF上的用户。在用户重新激活时，SMF会为用户选择其他可用的UPF。对于PFCP Modification Response消息超时场景，SMF支持在发送次数阈值的基础上多发一次请求消息（由**[SET PFCPPARA](../../../../../OM参考/命令/UNC MML命令/业务服务管理/接口管理/PFCP接口管理/PFCP路径管理/PFCP路径参数管理/设置PFCP参数（SET PFCPPARA）_09652597.md)**命令的“PFCP修改请求消息重发优化开关”参数控制），等待时间为200ms。
