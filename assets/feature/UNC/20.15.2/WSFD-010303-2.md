# NAS信令加密与完整性保护-5G

`WSFD-010303-2` · enhanced · required

## 所属目录

- [WSFD-010303](feature/UNC/20.15.2/WSFD-010303.md)

## 适用性

- 适用NF：AMF
- 代际支持矩阵：`SGSN(2.5&3G)=-; GGSN-C(2.5&3G)=-; MME(4G)=M; S/PGW-C(4G)=-; NB-IoT(4G)=M; 5G MME(5G NSA)=M; 5G S/PGW-C(5G NSA)=-; AMF(5G SA)=M; SMF(5G SA)=-; NRF(5G SA)=-`

## 适用NF（原文）

AMF

## 定义

AMF与UE使用协商一致的加密和完整性保护算法对NAS信令进行保护，提高网络的安全性。

## 客户价值

| 受益方 | 受益描述 |
| --- | --- |
| 运营商 | 保证NAS信令的安全性和可靠性。 |
| 用户 | 用户不感知该特性。 |

## 应用场景

用户和网络交互过程中，启用本特性保护NAS信令的安全性和可靠性。

## 可获得性

**涉及NF**

| 涉及NF | 支持版本 | 功能说明 |
| --- | --- | --- |
| AMF | UNC 20.0.0及后续版本 | - 计算NAS信令密钥。<br>- 向UE发送加密算法和密钥。<br>- 对NAS信令进行完整性保护、加密和解密。 |
| UE | 无特殊要求 | - 和网络侧协商加密算法。<br>- 对NAS信令进行完整性保护、加密和解密。 |

## 原理概述

5G网络的信令加密和完整性保护如 [图1](#ZH-CN_TOPIC_0175704848__fig_1) 所示。

- NAS信令加密和完整性保护：AMF与UE之间协商完成。
- RRC信令加密和完整性保护：gNodeB与UE之间协商完成。

**图1** 5G网络信令加密和完整性保护

<br>

![](特性概述_75704848.assets/zh-cn_image_0249770451_2.png)

AMF支持的NAS信令加密算法如 [表1](#ZH-CN_TOPIC_0175704848__table1423828121914) 所示。

*表1 NAS信令加密算法*

| 加密算法 | 详细描述 |
| --- | --- |
| NEA0 (NEA: Encryption Algorithm for 5G) | 空加密算法。使用空加密算法时，加密前后的明文和密文是完全相同的。 |
| 128-NEA1 | SNOW 3G加密算法，密钥长度为128位。 |
| 128-NEA2 | AES加密算法，密钥长度为128位。 |
| 128-NEA3 | ZUC加密算法，密钥长度为128位。 |

AMF支持的NAS信令完整性保护算法如 [表2](#ZH-CN_TOPIC_0175704848__table26461347233) 所示。

*表2 NAS信令完整性保护算法*

| 完整性保护算法 | 详细描述 |
| --- | --- |
| NIA0 (NIA: Integrity Algorithm for 5G) | 空完整性算法。NIA0仅用于紧急业务，不支持配置。 |
| 128-NIA1 | SNOW 3G完整性保护算法，密钥长度为128位。 |
| 128-NIA2 | AES完整性保护算法，密钥长度为128位。 |
| 128-NIA3 | ZUC完整性保护算法，密钥长度为128位。 |

AMF根据 MML命令 [**ADD NGALGPRIORITY**](../../../../../../OM参考/命令/UNC MML命令/业务服务管理/5G接入业务管理/业务安全管理/安全算法优先级管理/增加5G算法优先级属性（ADD NGALGPRIORITY）_09652959.md) 配置的算法优先级和UE进行协商，在UE和AMF同时支持的前提下，选择优先级最高的算法对NAS信令进行加密和完整性保护。如果UE和AMF支持的算法没有交集，则会协商失败。

> **说明**
> 网络中，至少需要开启一种加密和完整性保护算法，以保证网络中的消息都是经过加密和完整性保护的。
