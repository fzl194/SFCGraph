`WSFD-011131` · enhanced · required

## 适用NF（原文）

MME

## 定义

S6a over DRA应用在MME与HSS之间，是指MME与HSS不直接进行通信，而是通过DRA(Diameter Routing Agent)设备进行转发。

## 客户价值

| 受益方 | 受益描述 |
| --- | --- |
| 运营商 | - 避免MME与在其它域的HSS服务器的通信。<br>- 减少MME在HSS增加、变化或删除时的配置负担。 |
| 用户 | 用户不感知该特性。 |

## 应用场景

在MME无法和HSS进行直接通信时（如国际漫游场景），使用DRA设备转发MME和HSS的会话消息，完成MME与HSS的通信。

## 原理概述

**DRA基本概念**

除了客户端和服务器，Diameter协议还引入了Diameter代理DRA功能。Diameter代理负责将Diameter消息在发送设备和接收设备之间转发。

**Diameter请求消息的路由过程**

1. 任何一个Diameter Peer接收到请求消息后，首先判断是否为CER（能力交换请求）\DWR（握手请求）\DPR（断连请求）消息，如果是则按照Peer基本状态机进行处理。
2. 如果不是，则再判断消息之中是否有Destination-Host AVP和Destination-Realm AVP字段。如果有Destination-Host AVP和Destination-Realm AVP字段，则交给本端上层应用处理。
3. 如果存在Destination-Host AVP字段：
    - 如果其值等于本端标识，则交给本端上层应用处理。
    - 如果其值等于一个与本端直接相连的Peer标识，则直接转发给该Peer处理。
4. 当（3）的条件都不成立，则判断Destination-Realm AVP字段。
  如果该字段值等于本端可以支持的域范围，并且本端可以处理该应用的请求，则交给本端上层应用处理。
  否则查询路由表，寻找一个特定的路由将该消息转发给一个与本端直接相连的Peer。
  > **说明**
  > 本端Diameter Peer在转发该请求消息之前会：
  >
  > a. 给该消息分配一个新的Hop-by-Hop值替换该消息之中的Hop-by-Hop字段值，同时将该消息原来的Hop-by-Hop字段值，上一跳信息记录在本端的待响应请求队列之中。
  > b. 在该消息之中添加一个Router-Record字段值，具体信息为该消息的上一跳Peer标识。
  > c. 做完上述步骤之后再转发本消息。

Diameter路由请求消息描述：

- Relay Agent的路由表配置：
  Realm =‘epc.mnc015.mcc123.3gppnetwork.org’, Server Identifier =‘server1.epc.mnc015.mcc123.3gppnetwork.org’;
  Realm =‘epc.mnc016.mcc123.3gppnetwork.org’, Server Identifier =‘relay2.epc.mnc016.mcc123.3gppnetwork.org’;
- Relay Agent的Peer表配置：
  Host Identifier ="Relay2.epc.mnc016.mcc234.3gppnetwork.org";
  Host Identifier =‘MME1.MNC015.MCC123’;
  Host Identifier =‘MME2.MNC015.MCC123’;
  Host Identifier =‘Server1.epc.mnc015.mcc123.3gppnetwork.org’;

- MME1.MNC015.MCC123发出请求消息Request1：
  Destination-Host =‘Server1.epc.mnc015.mcc123.3gppnetwork.org’, Destination-Realm =‘epc.mnc015.mcc123.3gppnetwork.org’, Hop-by-Hop = 10;
  Relay Agent接收到该请求消息之后，根据本端配置的Peer表信息和Request1的Destination-Host字段判断出转发的下一跳。
  给该消息分配一个新的Hop-by-Hop值，例如6，这个值需要保证Relay Agent分配的全局唯一。同时记录本消息的前一跳MME1.MNC015.MCC123，前一跳Hop-by-Hop值10，当前Hop-by-Hop值6到待响应请求队列；在该消息之中添加一个Route-Record字段，值为‘MME1.MNC015.MCC123’后转发该消息。
- MME2.MNC015.MCC123发出请求消息Request2：
  Destination-Host =‘Server2.epc.mnc016.mcc123.3gppnetwork.org’, Destination-Realm =‘epc.mnc016.mcc123.3gppnetwork.org’, Hop-by-Hop = 31;
  Relay Agent接收到该请求消息之后，根据本端配置的路由表信息和Request2的Destination-Realm字段判断出转发的下一跳。
  给该消息分配一个新的Hop-by-Hop值，例如8，同时记录本消息的前一跳MME2.MNC015.MCC123，前一跳Hop-by-Hop值31，当前Hop-by-Hop值8到待响应请求队列；在该消息之中添加一个Route-Record字段，值为‘MME2.MNC015.MCC123’后转发该消息。

**Diameter响应消息的路由过程**

[图2](#ZH-CN_CONCEPT_0162964103__fig21) 为Diameter路由响应消息的示例图。

**图2** Diameter路由响应示例

<br>

![](WSFD-011131 S6a over DRA_62964103.assets/zh-cn_image_0000002553841147_2.png "点击放大")

- Diameter协议对响应消息的路由完全不同于对请求消息的路由，这个过程只基于消息之中的Hop-by-Hop值，并且响应消息一定是按照对应请求消息的原路径返回的。
- 对响应消息的路由过程：
  （1） 任何一个Diameter Peer接收到响应消息后，首先判断是否为CEA（能力交换响应）\DWA（握手响应）\DPA（断连响应）消息，如果是则按照Peer基本状态机进行处理。
  （2） 如果不是，则取该响应消息的Hop-by-Hop字段值在待响应请求队列之中查找。
  如果能够找到，则将该消息的Hop-by-Hop字段值填写为本端待响应请求队列之中记录的前一跳Hop-by-Hop值，然后将响应消息发送给本端待响应请求队列之中记录的前一跳Peer，并删除该待响应请求节点。
  如果无法找到，则直接扔弃该请求消息。
- Server1.epc.mnc015.mcc123.3gppnetwork.org处理Request1后生成对应的响应消息Answer1
  响应消息中，Hop-by-Hop的值应该填写收到的Request1的Hop-by-Hop值6。
  Relay Agent接收到该消息后，在本端待响应请求队列之中，根据Hop-by-Hop值找到Answer1消息对应的Request1消息的上一跳Hop-by-Hop值10和上一跳Peer‘MME1.MNC015.MCC123’。
  Relay Agent将该响应消息的Hop-by-Hop值填写为10，然后转发该消息到上一跳Peer MME1.MNC015.MCC123。
- Server2.epc.mnc016.mcc123.3gppnetwork.org处理Request2后生成对应的响应消息Answer2
  响应消息中，Hop-by-Hop的值应该填写收到的Request2的Hop-by-Hop值8。
  Relay Agent接收到该消息之后，在本端待响应请求队列之中根据Hop-by-Hop值找到Answer2消息对应的Request2消息的上一跳Hop-by-Hop值31和上一跳Peer‘MME2.MNC015.MCC123’。
  Relay Agent将该响应消息的Hop-by-Hop值填写为31，然后转发该消息到上一跳Peer MME2.MNC015.MCC123。
