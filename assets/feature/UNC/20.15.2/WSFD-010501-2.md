## 适用NF（原文）

MME、SGW-C、PGW-C

## 定义

会话管理功能是MME的基本功能，是用户与外部PDN建立连接，进行数据业务的基础。MME的会话管理功能包括网络侧发起的会话管理和UE发起的会话管理。

## 客户价值

| 受益方 | 受益描述 |
| --- | --- |
| 运营商 | 会话管理功能使运营商得以向授权的用户提供从移动终端到分组数据网络间的数据传输服务，从而丰富其网络业务。 |
| 用户 | 通过会话管理功能，用户可以通过移动通信网络，在移动终端和分组数据网络间上传或下载数据。 |

## 应用场景

当用户需要通过LTE网络接入外部PDN时，需要支持本功能。

## 原理概述

**基本概念**

EPC会话管理的对象为EPS承载。每个PDN连接都由EPS承载（EPS Bearer）和IP连接组成。

**PDN连接** ：EPS通过E-UTRAN网络和EPC网络，为UE提供了一条UE与PDN之间的IP连接，称为PDN连接。

**EPS承载** ：用于唯一标识基于GTP协议的UE和P-GW之间的共同QoS业务流，或唯一标识基于PMIP协议（Proxy Mobile IP Protocol ）的UE和S-GW之间的共同QoS业务流。

**缺省承载** ：当UE建立一个到某个PDN的PDN连接时，会建立一个EPS承载，这个承载在PDN连接的过程中不会被释放以提供到该PDN的永久性连接，这个承载被称为缺省承载。缺省承载被删除，PDN连接也将被释放。

**专有承载** ：除缺省承载之外的其他连接到相同的PDN的承载称为专有承载。根据不同的QoS需求建立不同的专有承载，比如观看视频、浏览网页等。

PDN连接、缺省承载、专有承载之间的关系请参照 [图1](#ZH-CN_CONCEPT_0173595351__fig1888153314814) 。

**图1** PDN连接、缺省承载、专有承载之间的关系

<br>

![](WSFD-010501 会话管理特性概述（适用于4G）_73595351.assets/zh-cn_image_0192309308_2.png)

- APN与PDN是一一对应。一个PDN是由一个APN进行代表的，不同的APN，可以连接到不同的PDN。一个终端可以通过多个APN接入到不同的PDN中，PDN中可以有多个服务器（百度、优酷等）。
- 同一个APN下可建立多个PDN连接。同一个PDN连接必须建立一个缺省承载且只能建立一个缺省承载，但可以建立多个专有承载。由于无线侧的限制，一个终端最多只能建立11个承载。

**会话管理网络架构演进**

4G网络架构包括非CUPS网络架构和CUPS网络架构。为了向5G的高带宽、多连接、低时延的演进，产生了CUPS网络架构，将S-GW和P-GW的控制面和用户面拆分，用户面下沉到边缘，可降低传输迂回，降低时延，提升用户体验。

**承载状态管理**

承载的状态分为BEARER CONTEXT INACTIVE和BEARER CONTEXT ACTIVE两种状态，MME中两者之间的转换关系如 [图3](#ZH-CN_CONCEPT_0173595351__fig2979124719323) 所示：

**图3** MME中的承载状态

<br>

![](WSFD-010501 会话管理特性概述（适用于4G）_73595351.assets/zh-cn_image_0192309309_2.png "点击放大")

- 当承载上下文被激活时，MME中承载的状态将由BEARER CONTEXT INACTIVE转化为BEARER CONTEXT ACTIVE。
- 当承载上下文去激活时，MME中承载的状态将由BEARER CONTEXT ACTIVE转化为BEARER CONTEXT INACTIVE。
- 当承载上下文更新时，MME中的承载状态保持为BEARER CONTEXT ACTIVE。

**会话管理功能**

会话管理功能包括网络侧发起的会话管理和UE发起的会话管理。由于用户的不同业务的需求，UE侧可以请求专有EPS承载激活流程、EPS承载修改流程或者EPS承载去激活流程，如果网络侧接收UE的请求，网络侧会发起EPS承载激活流程、EPS承载修改流程或者EPS承载去激活流程，网络侧是最终流程的发起端。

*表1 网络侧发起的会话管理*

| 网络侧发起的会话过程 | 目的 | 触发 |
| --- | --- | --- |
| 专有EPS承载激活 | 目的是在UE和EPC之间建立特定QoS和TFT的承载。 | - 网络侧PGW-C触发。<br>- UE请求：作为UE请求的承载资源分配流程或UE请求的承载资源修改流程的一部分而触发。说明：TFT（Traffic Flow Template）：一组数据包过滤器的集合，UE使用TFT来决定业务数据流Service Data Flows和无线承载Radio Bearer之间的匹配关系。 |
| EPS承载修改 | 目的是修改EPS承载的QoS和TFT。 | - 可以由HSS或PGW-C触发。<br>- UE请求：作为UE请求的承载资源分配流程和UE请求的承载资源修改流程而触发。 |
| EPS承载去激活 | 目的是去激活一个EPS承载或者去激活到PDN的所有EPS承载以断开某个PDN连接。 | - 由MME/eNodeB或PGW-C触发<br>- UE请求：可以作为UE请求PDN断连或者UE请求的承载资源修改流程而触发。 |

*表2 UE发起的会话管理*

| UE发起的会话管理过程 | 目的 | 触发 | 说明 |
| --- | --- | --- | --- |
| 缺省承载上下文激活 | 在UE和EPC之间建立一个缺省的EPS承载上下文。 | 由UE发起。 | 缺省承载上下文激活过程也是附着流程的一部分，如果附着失败，UE认为缺省上下文激活过程也是失败的。 |
| UE请求PDN连接建立 | 请求建立到一个PDN的缺省承载。 | - | 此过程或者被包含在附着过程建立第一个缺省承载或者用来建立后续到其他PDN的缺省承载中，以允许UE同时接入多个PDN。 |
| UE请求PDN连接断连 | UE请求从PDN断开。 | - | 在此过程中，与此PDN相关的所有EPS承载上下文，包括缺省承载上下文和专有承载上下文，全部被释放掉。 |
| UE请求承载资源分配和修改 | - UE请求承载资源分配用于UE为新的业务数据流请求分配承载资源。<br>- UE请求承载资源修改用于UE为已有的业务数据流请求修改承载资源。 | - | UE可以向网络侧请求分配或修改一个特定的QoS，也可以向网络侧请求新的GBR承载或修改已有的GBR承载。<br>说明：关于GBR的详细信息请参见EPS QoS特性。 |

> **说明**
> 本特性只介绍简单的会话管理流程，具体的流程参见 《EPC信令分析》 。

**会话管理流程典型触发场景**

*表3 4G用户会话管理流程典型触发场景*

| 涉及流程 | 触发场景 |
| --- | --- |
| 缺省承载激活 | 4G用户打开手机后，UE侧发起缺省承载，建立PDN连接，此时该用户可以正常上网。PDN连接和缺省承载是同步建立的。 |
| 专有承载激活 | 专有承载的激活流程均由网络侧发起。<br>- 如果部署了动态PCC，PCRF向P-GW发送RAR消息触发专有承载建立，消息包含QoS policy。<br>- 如果未部署动态PCC，则P-GW应用本地QoS策略发起专有承载建立流程。 |
| 承载更新 | - HSS发起的承载更新流程用于修改EPS承载Qos参数，如QCI、ARP或者用于修改UE-AMBR以及APN-AMBR。<br>- P-GW发起的承载更新流程。- 由P-GW发起伴随QoS更新的专有承载更新流程用于修改EPS承载QoS参数，如QCI、GBR、MBR或ARP，或用于修改APN-AMBR。例如，PCRF配置FUP服务，为在线用户发起IP-CAN modification(QCI/GBR/MBR/ARP change)流程。<br>- P-GW可以发起无QoS更新的专有承载更新流程，其与P-GW发起的伴随QoS更新专有承载更新流程没有大的区别，主要的区别是：QoS更新是指EPS Bearer QoS中的QCI，ARP等的参数更新，QoS不更新是指TFT进行更新，而EPS Bearer QoS中的QoS参数没有更新。 |
| 承载去激活 | - eNodeB/MME发起承载去激活，只去激活专有承载。- 当eNodeB向MME发送Bearer Release Request消息，携带EPS Bearer Identity信元。<br>- MME通过OAM（Operation, Administration and Maintenance）发起承载去激活流程。<br>- P-GW发起承载去激活，这种场景去激活一个专有承载或者去激活某个PDN链接的全部承载。- 发生从3GPP到Non-3GPP的切换。<br>- 如果在承载生命期内，没有任何数据传输，承载将会被释放。<br>说明：PGW-C发起承载去激活分为UE的最后一个PDN连接去激活和不是UE的最后一个PDN连接去激活两种情况：<br>主要的差异点：<br>如果网络侧删除最后一条PDN连接时，MME向UE发送Detach Request消息，发起显式分离流程。 |
