# P-CSCF故障时IMS业务恢复

`WSFD-102202` · protocol · required

## 所属目录

- [WSFD-102200](feature/UNC/20.15.2/WSFD-102200.md)

## 适用性

- 适用NF：PGW-C、SMF
- 代际支持矩阵：`SGSN(2.5&3G)=-; GGSN-C(2.5&3G)=-; MME(4G)=-; S/PGW-C(4G)=M; NB-IoT(4G)=-; 5G MME(5G NSA)=-; 5G S/PGW-C(5G NSA)=M; AMF(5G SA)=-; SMF(5G SA)=M; NRF(5G SA)=-`

## 适用NF（原文）

PGW-C、SMF

## 定义

PGW-U/UPF采用ICMP（Internet Control Message Protocol）的方式，周期性的检测P-CSCF状态，当检测到PGW-U/UPF与P-CSCF链接状态异常时，上报给PGW-C/SMF。PGW-C/SMF将一对状态正常的P-CSCF地址推送给UE，UE从中选择一个P-CSCF地址，重新发起IMS业务，从而保证IMS业务自动恢复。

## 客户价值

| 受益方 | 受益描述 |
| --- | --- |
| 运营商 | 及时发现核心网与P-CSCF之间的链接故障，并且<br>UNC<br>主动采取故障处理措施，减少业务中断时间，减少用户投诉。 |
| 用户 | 及时恢复IMS业务，提升用户体验。 |

## 应用场景

部署IMS网络场景下，为了保证用户的IMS业务能够及时恢复，建议开启此特性。

## 对系统的影响

性能影响程度：低

IMS业务恢复特性只对P-CSCF进行状态扫描，根据扫描结果判断是否给MS/UE发送P-CSCF地址列表更新消息，对系统影响可忽略。

## 应用限制

PGW-U/UPF检测P-CSCF的状态仅能检测到传输层链路状态，无法检测到上层应用层故障。如果PGW-U/UPF与P-CSCF之间部署有防火墙，要求防火墙允许ICMP echo报文通过。

## 原理概述

**相关概念**

IP Farm：PGW-C/SMF定义的一个服务器池，PGW-C/SMF可以对配置在服务器池中的多台P-CSCF设备进行状态检测动作。

**探测机制**

UPF和PGW-U对IP Farm Server的探测机制完全相同，如下以PGW-U为例进行介绍。

- PGW-U定时向配置的P-CSCF发送ICMP报文，P-CSCF收到报文后会对PGW-U做出应答。
- 如果PGW-U在一段时间内能收到P-CSCF的应答，则认为这个P-CSCF的状态为正常。
- 如果PGW-U连续几次发送的ICMP报文都收不到响应，则认为P-CSCF处于异常状态，并且会上报ALM-81035 IPFarm服务器无响应。
- 如果IP Farm里面的所有P-CSCF都处于状态异常，会上报ALM-81034 IPFarm无可用server。
- 对于异常的P-CSCF，如果PGW-U连续几次发送的ICMP报文都能收到响应，则认为P-CSCF处于恢复正常状态，并且恢复ALM-81035 IPFarm服务器无响应和ALM-81034 IPFarm无可用server。

**业务流程**

业务流程分为三部分：

[P-CSCF地址获取](#ZH-CN_TOPIC_0226216210__p2311136152612)

[IMS注册流程](#ZH-CN_TOPIC_0226216210__p8945181212712)

[故障恢复流程](#ZH-CN_TOPIC_0226216210__p5305162882715)

**P-CSCF地址获取**

- **VoLTE终端获取P-CSCF地址**

VoLTE终端获取P-CSCF地址，可以在EPS的Attach流程或PDN Connectivity建立流程中，通过PCO获取。VoLTE终端一般使用两个APN，分别用于IMS业务和数据业务(如Web、FTP等)。如果终端在EPS附着过程中激活的是数据业务APN，则需要再通过PDN connectivity Request流程激活IMS业务APN；如果在EPS附着过程中激活的是IMS业务APN，则需要通过PDN connectivity Request流程激活数据APN。无论是哪个方式，在激活IMS业务APN时，VoLTE终端都会通过PCO信元来获取P-CSCF的值。如下以Attach流程为例进行介绍：

**图1** Volte终端获取P-CSCF地址的流程

<br>

![](特性概述_26216210.assets/zh-cn_image_0229015070_2.png "点击放大")

1. UE发起用户附着请求，该Attach Request消息中PDNConnectivityRequest信元中携带PCO，PCO中包含000CH(P-CSCF IPv4 Address Request)或0001H(P-CSCF IPv6 Address Request)参数，表明UE需要接入IMS域。
2. MME发送给SPGW-C的Create Session Request消息，在消息中将UE的PCO内容透传给SPGW-C。
3. SPGW-C将配置的P-CSCF地址列表封装到Create Session Response消息的PCO信元中，从而回复给MME。
4. MME在Attach Accept消息中携带PCO，将此信元透传给UE，从而将P-CSCF的地址带给手机。

- **VoNR终端获取P-CSCF地址**

VoNR终端获取P-CSCF地址可以在IMS PDU会话建立流程中获取。

**图2** VoNR终端获取P-CSCF地址的流程

<br>

![](特性概述_26216210.assets/zh-cn_image_0229018243_2.png "点击放大")

1. UE向AMF发送PDU Session Establishment Request消息，PDU session establishment request中携带PCO（Extended protocol configuration options）。PCO中包含P-CSCF地址请求信息（支持IPV4和IPV6格式），例如p-cscf-ipv4-address-request。
2. AMF发送Nsmf_PDUSession_CreateSMContext消息给SMF，携带DNN为IMS，在Extended protocol configuration options参数中，UE请求网络侧分配P-CSCF地址。
3. SMF向AMF发送Namf_Communication_N1N2Message_transfer消息，消息中携带了P-CSCF的IP地址信息。
4. AMF回复PDU Session Establishment Accept消息给UE，消息中PCO包含P-CSCF的信息。

**IMS注册流程**

如下以VoLTE终端到IMS网络的注册流程为例进行介绍，VoNR到IMS网络的流程类似。

**图3** IMS注册流程

<br>

![](特性概述_26216210.assets/zh-cn_image_0230801060_2.png)

1. UE从PGW-C返回的P-CSCF地址列表中选择一个P-CSCF，发起SIP注册请求。
2. P-CSCF接收到UE的注册请求，提取UE地址和P-CSCF地址封装到AAR消息中发送给PCRF。
3. PCRF接收到AAR消息后，向P-CSCF回复应答消息。
4. PCRF从AAR消息中提取UE地址和P-CSCF地址，将这两个信息封装到RAR消息中（RAR消息中Charging-Rule-Definition信元中的Flow-Information参数，Flow-Information中的源/目的IP对应的是P-CSCF/UE地址），发送给PGW-C。
5. PGW-C接收到RAR消息后，从消息中提取出P-CSCF和UE地址，保存在用户的承载上下文中，并向PCRF回复RAA响应消息。
6. PGW-C进行P-CSCF服务器的状态探测。
7. P-CSCF完成UE的SIP注册流程，向UE回复成功响应。
8. PGW-U根据配置启动和P-CSCF之间的路径探测。

- PGW-C检测到P-CSCF故障后的恢复流程如下所示：

**图4** 故障恢复流程（PGW-C）

<br>

![](特性概述_26216210.assets/zh-cn_image_0000001169937345_2.png "点击放大")

1. PGW-U检测到P-CSCF地址列表中的某个P-CSCF状态不正常，通知PGW-C。
2. PGW-C采用的处理方法由软参[DWORD1017 BIT30](../../../../../OM参考/软件参数/UNC软件参数/业务软件参数/软件参数（SMF_GGSN_SGW-C_PGW-C）/业务控制管理/DWORD1017 BIT30 该软参用于控制GGSN-C、PGW-C和S_PGW-C形态基于UPF和_e038ee2b_78142228.md)控制。当该软参取值为“0”时，执行2a；当该软参取值为“1”时，根据UE对P-CSCF Restoration支持能力不同，PGW-C采用的处理方法不同：如果，UE支持P-CSCF Restoration（UE在PDN Connectivity Request消息中携带p-cscf-reselection-support进行指示），则执行2a，否则执行2b。

      > PGW-C根据故障的P-CSCF地址，查询所有的承载上下文信息，判断当前接入的UE是否使用故障P-CSCF设备进行业务，如果是，PGW-C会对所有的使用故障P-CSCF的UE发送更新请求。
      >
      > 当前大部分VoLTE终端还不支持更新P-CSCF，并且打开更新消息以后，会对周边网元造成一定的信令冲击，可以根据局点情况评估是否开启发送更新消息功能，配置命令为 [**SET PCSCFCTRL**](../../../../../OM参考/命令/UNC MML命令/业务服务管理/会话管理/IMS管理/P-CSCF管理/P-CSCF故障恢复控制/设置P-CSCF控制配置（SET PCSCFCTRL）_09653206.md) 。
    b. PGW-C发送Delete Bearer Request消息给UE，通知UE重新建立到IMS APN的PDN连接。UE重新建立到IMS PDN的连接，并在此流程中重新获取状态正常的P-CSCF地址列表。
3. UE从P-CSCF地址列表中选择一个P-CSCF，重新发起SIP注册请求。

**P-CSCF可靠性**

PGW-C/SMF本地分配P-CSCF地址时，P-CSCF可靠性包括：主备P-CSCF组 、负荷分担P-CSCF组 和P-CSCF组内的权重分担，如下用PGW-C为例介绍3种机制。

- 主备P-CSCF组
  在配置全局P-CSCF组或者APN P-CSCF组时，可以通过 [**ADD PCSCFIMSISDNSEG**](../../../../../OM参考/命令/UNC MML命令/业务服务管理/会话管理/IMS管理/P-CSCF管理/P-CSCF选择的IMSI和MSISDN号段/增加IMSI和MSISDN号段（ADD PCSCFIMSISDNSEG）_09652488.md) 或 [**ADD PCSCFGRPBNDAPN**](../../../../../OM参考/命令/UNC MML命令/业务服务管理/会话管理/IMS管理/P-CSCF管理/P-CSCF组与APN关联关系/增加APN和P-CSCF组关联关系（ADD PCSCFGRPBNDAPN）_09653091.md) 命令绑定主备两个P-CSCF组，并且P-CSCF组内的每个P-CSCF地址都可以通过 [**ADD PCSCFIP**](../../../../../OM参考/命令/UNC MML命令/业务服务管理/会话管理/IMS管理/P-CSCF管理/P-CSCF的IP地址/增加P-CSCF地址配置（ADD PCSCFIP）_09651572.md) 命令配置其对应的Pair ID（结对ID），主备P-CSCF组内Pair ID相同的两个P-CSCF IP互为容灾P-CSCF地址。

- P-CSCF组内的权重分担
  PGW-C在组内选择P-CSCF时，根据 **[ADD PCSCFIP](../../../../../OM参考/命令/UNC MML命令/业务服务管理/会话管理/IMS管理/P-CSCF管理/P-CSCF的IP地址/增加P-CSCF地址配置（ADD PCSCFIP）_09651572.md)** 命令 配置的“weight”值来确定。在用户数达到一定规模后，各P-CSCF被选中的比例符合配置的“weight”比例。
  当超过特定用户数时，P-CSCF组内的P-CSCF注册的用户数比例才会与“weight”比例一致，特定用户的计算公式如下：
  > **说明**
  > 特定用户数=sum(weight)*X
  >
  > X的值与SPU的类型和数量相关。
