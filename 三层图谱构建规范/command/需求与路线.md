# 命令层需求与路线（梳理记录）

> **忠实记录用户对命令层构建的明确要求与待决问题，避免遗忘。**
> 本文是需求梳理，非最终规范。带 ✅=已定、❓=待决。

---

## 1. 总路线转变（最重要）❗

**不再走以前 command 那套"全量结构化抽取"**（即不走 CommandGraph 的 `md→jsonl→重构 wiki、27 字段全抽` 路线）。

**新路线**：**复用原始 md + 加 YAML frontmatter（仅基本字段）**。
- 输出对象是 **命令 / 配置对象** 两类（**参数在命令 md 原文内，不单独建**），但**不全量抽取结构化信息**。
- 每个命令 md **复用原始内容**，只在头部加 YAML（最基本的标识/路由信息）。
- 图谱命令层展示 = 直接展示每个命令的 md，可复用原始 md，但**所有引用必须合理**。

> ⚠️ 这意味着之前 `调研-输入输出与存储格式.md` 里"27 字段全量抽取 / jsonl 中间态 / compile 投影"那套**路线作废**；其中"输入格式 / NF·版本存储 / 通用性"部分仍有效。

---

## 2. 输入（明确指定）✅

| 输入项 | 说明 |
|---|---|
| 产品文档原始文件 | **或**解压后的目录（二选一作输入起点） |
| 网元 NF | 如 UDG / UNC |
| 版本 | 如 20.15.2 |
| 存储位置路径 | 默认：新建一个"三层图谱资产"文件夹（**资产根**，构建产物落地处） |
| MML 命令所在目录 | 产品文档里 MML 命令 md 的目录 |

---

## 3. 输出 ✅

- 该产品文档的**所有 MML 命令构建出的 md**。
- 默认解压位置：**存储位置路径下的 `output/` 目录**。

---

## 4. evidence 改变（重要）✅

**资产中不再有独立 evidence 文件夹。**
- 原始 md 作为证据，**直接引用原始 md 位置**（解压后的 `output/` 里）。
- 即：不拷贝证据进资产，引用 `output/` 里的原始 md。

---

## 5. 资产根与运行时布局（梳理出的结构）✅

```
{存储位置路径}（默认 三层图谱资产/）        ← 资产根（所有引用的相对根）
├── output/                                ← 解压后产品文档（原始 md，兼作 evidence 源）
└── {命令资产 md}                           ← 构建出的命令 md（引用 output/ 里原始 md）
```

> 注意区分：**框架**（`三层图谱构建规范/`=怎么构建）≠ **资产根**（`三层图谱资产/`=构建产物落地处）。
> output/ 在资产根下 → 引用 `output/...` 是合法的"资产根相对路径"。✓ 与第 6 条一致。

---

## 6. 引用规范（硬约束）✅

**引用 = `{网元}@{Type}@{local}`（带网元、不带版本），版本动态解析。**
- 引用**带网元**（命令在网元下），**不带版本**（不强制关联具体版本，业务层跨版本引用下层）。
- 解析：`UDG@MMLCommand@ADD URR` → 存储位置 `Command/UDG/{版本}/MMLCommand@ADD URR.md`，**版本文件夹解析时自动忽略/动态选择**。
- 存储位置不变（仍 `网元/版本/` 两级目录），只是解析时版本动态生成。

---

## 7. 产出文件组织 ✅（已定）

### 7.1 文件布局（`网元/版本/` 两级目录）
```
{资产根 = 三层图谱资产/}
├── output/                                                解压后产品文档（原始md，evidence源）
├── Command/{网元}/{版本}/{网元}@{Type}@{local}.md           如 Command/UDG/20.15.2/UDG@MMLCommand@ADD URR.md
├── ConfigObject/{网元}/{版本}/{网元}@{Type}@{local}.md      如 ConfigObject/UDG/20.15.2/UDG@ConfigObject@URR.md
└── Parameter/{网元}/{版本}/{网元}@{Type}@{local}.md         如 Parameter/UDG/20.15.2/UDG@CommandParameter@ADD URR:URRNAME.md
```

### 7.2 文件名 = 逻辑ID `{网元}@{Type}@{local}`（网元在最前，版本无关，空格保留）
- 文件名 = 逻辑ID `{网元}@{Type}@{local}`（如 `UDG@MMLCommand@ADD URR.md`）：**网元在最前面**，**完全保留命令原始名称（含空格）**。
- 逻辑ID **与网元有关、与版本无关**：网元是身份的一部分（必带、在最前），版本不带（动态解析）。
- 网元同时出现在目录（组织用）和文件名（身份用）——文件名自带网元，脱离目录也能独立识别。
- **id 字段、引用、文件名三者统一**为 `{网元}@{Type}@{local}`（见 §6 / §8）。

## 8. YAML frontmatter 字段 ✅（已定，可扩展）

目的：预留字段供未来快速检索；**预留扩展接口**，未来加字段直接追加。

```yaml
---
id: UDG@MMLCommand@ADD URR              # 逻辑ID（网元@类型@本地名，版本无关；= 引用键）
type: MMLCommand
name: ADD URR                            # 英文命令名（原始，含空格）
name_zh: 增加URR                         # 中文命令名
nf: UDG
version: 20.15.2
category_path: [用户面服务管理, 业务控制策略, 计费控制, 使用率上报规则]  # 目录链
source: output/.../增加URR（ADD URR）_82837629.md   # 原始md位置（evidence）
status: active
# === 检索预留（轻解析，可空） ===
applicable_nf: [PGW-U, UPF]
effect_mode: 立即生效
is_dangerous: false
# === 扩展接口：未来额外字段直接加在这下面 ===
---
```

12 字段（已删 `verb` / `object_keyword` / `command_category`——可从 name 解析或不必要）。正文（功能/注意事项/参数/实例/输出/参考）仍是原始 md，不进 YAML。

## 9. 命令 md 的形态 ✅（已定）

```
---
(YAML frontmatter)              ← 顶部：标识 + 检索字段（§8）
---
(原始 md 正文，原样保留)         ← 中间：复用原始内容，不改
...
## 边                            ← 底部：新增章节，列引用边
- <关系类型>: [[UDG@ConfigObject@URR]]
- ...
```

- 顶部加 YAML，底部加"边"章节，中间是原始 md **原样保留**。
- **边 = 逻辑ID 引用** `[[{网元}@{Type}@{local}]]`（见 §6），解析时动态指向版本目录。
- **图谱使用完全依赖底部"边"章节**（所有关系都在这里表达）。

---

## 10. 边的构建（模块化、可组装）✅

**边 = 可组装模块**：每个边类型/来源是一个独立模块，各模块独立产出边，合并进命令的"边"章节。可通过不同方法构建不同类型的边，未来按需加模块。

| 模块 | 边类型 | 来源 | 方法 |
|---|---|---|---|
| `edge_configobject` | 命令 → 配置对象（操作配置对象） | 命令名 object 部分 | **代码推导**（`ADD URR` → `[[UDG@ConfigObject@URR]]`）|
| `edge_cmdref_body` | 命令 → 命令（参见/参考） | 原始 md 正文 | 正则扫"参见 / 通过 MOD XXX / 参考"等，**结合正文构建初版** |
| `edge_cmdref_intranet` | 命令 → 命令（参数引用） | 内网命令级关联图谱 | 读内网数据源（某命令参数必须引用另命令参数），**可调整 / 可添加** |

- **命令 ↔ 配置对象**：**代码构建**（确定型，从命令名推）。
- **命令 ↔ 命令**：**正文构建初版 + 内网关联图谱补充**，允许人工/Agent 调整、添加后定稿。
- 模块注册表**可扩展**（未来新边类型 → 加新模块）。
- 每个命令 md 的"边"章节 = 各模块产出边的并集（去重）。
