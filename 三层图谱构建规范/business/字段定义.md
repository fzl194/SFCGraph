# Business 层 YAML 字段定义

> business 层对象（业务域 BD / 场景 NS / 方案 CS）的 YAML 字段。每个对象 = YAML + 叙述正文 + `## 边`。
> 体裁：Procedure（agent 读产品文档业务章节 + 已建 task 资产，理解后自己写 md）。命令/特性层是 Spec，task/business 层是 Procedure。
> **跨 NF 类**：业务层横跨 UDG+UNC，**不带 nf/version**，ID 两段 `{Type}@{slug}`（区别于 NF 隔离类的 3 段）。

## 一、BusinessDomain (BD) — 业务域·定义层

> 业务域根对象。ID 两段 `BusinessDomain@{domain-slug}`。最抽象，不涉及具体配置。

| 字段 | 类型 | 必填 | 来源 / 规则 |
|---|---|---|---|
| `id` | string | ✅ | `BusinessDomain@{domain-slug}`，如 `BusinessDomain@business-awareness`（= 文件名；slug 小写 kebab）|
| `type` | string | ✅ | 固定 `BusinessDomain` |
| `name` | string | ✅ | 业务域中文名（如 `业务感知`）|
| `name_zh` | string | ✅ | 同 name（业务域名，中文；保留以与其他层字段集对齐）|
| `domain` | string | ✅ | 自身域 slug（= id 第二段，作路径段 + 前端分组）|
| `status` | string | ✅ | `draft`（默认）/ `active` / `stale` |

> BD 无 scenario（场景属于 NS 层级）。
> 无 nf/version（跨 NF）；无 ref（BD 是根）。

## 二、NetworkScenario (NS) — 场景·问题/边界层

> 场景对象，属某 BD。ID 两段 `NetworkScenario@{scenario-slug}`。讲"解决什么业务问题 + 边界"，不涉及具体配置。

| 字段 | 类型 | 必填 | 来源 / 规则 |
|---|---|---|---|
| `id` | string | ✅ | `NetworkScenario@{scenario-slug}`，如 `NetworkScenario@charging`（= 文件名）|
| `type` | string | ✅ | 固定 `NetworkScenario` |
| `name` | string | ✅ | 场景中文名（如 `计费`）|
| `name_zh` | string | ✅ | 同 name |
| `domain` | string | ✅ | 所属 BD slug（如 `business-awareness`）|
| `scenario` | string | ✅ | 自身场景 slug（= id 第二段，作路径段）|
| `status` | string | ✅ | `draft`/`active`/`stale` |

> 无 nf/version；无 ref（归属关系进 `## 边`：`上游域 → [[BusinessDomain@...]]`）。

## 三、ConfigurationSolution (CS) — 方案·配置层 ★

> 方案对象，属某 NS，向下编排 task。ID 两段 `ConfigurationSolution@{scenario}-{solution-slug}`。承载"怎么配"（特性级变种 + 跨特性协同 + 配置级 DP/约束）。

| 字段 | 类型 | 必填 | 来源 / 规则 |
|---|---|---|---|
| `id` | string | ✅ | `ConfigurationSolution@{scenario}-{solution-slug}`，如 `ConfigurationSolution@charging-converged`（= 文件名；slug 小写 kebab）|
| `type` | string | ✅ | 固定 `ConfigurationSolution` |
| `name` | string | ✅ | 方案中文名（如 `融合计费`）|
| `name_zh` | string | ✅ | 同 name |
| `domain` | string | ✅ | 所属 BD slug |
| `scenario` | string | ✅ | 所属 NS slug |
| `status` | string | ✅ | `draft`/`active`/`stale` |

> 无 nf/version；无 ref（CS 非 1:1 指向单一下层对象——它编排多个 FeatureTask/CompoundTask/AtomTask，关系全进 `## 边`）。
> 编排关系（编排特性/复用步骤/复用命令）只在 `## 边`，**不进 YAML**（同 task 层"关系只在 ## 边"）。

### 不进 YAML（在正文 / 不存）

- 特性关系矩阵、配置级 DP、约束、跨网元协同 → 正文（叙述式 + 表格）
- **DP/约束不编号、无独立锚点**：引用粒度 = md 级（`[[ConfigurationSolution@charging-converged]]` 指整个 CS md），无 md 内部章节级引用（同 task 层）
- 命令/特性的静态知识 → 不抄进 CS（CS 讲方案级编排与变种，使用者按 FeatureTask 查标准配置）
- **无证据**：不写 `source` / `source_evidence_ids`、不设 `## 证据` 段（同 task 层）。所有引用统一裸 `[[逻辑ID]]`，关系全进 `## 边`。

### md 结构（统一，所有对象）

```
--- YAML frontmatter ---     ← 最顶部
# 标题 + > 引子
## 内容段（概览 / 配置与协同 / 决策点 / 约束 ...）   ← 中间
## 边                         ← 最底部（typed edges，承载与上下层对象的双向关系）
```

### 已删（不进 YAML）

- 旧三层图谱写死字段：`scopes`/`participants`/`uses_semantic_object`/`constrained_by`/`has_decision` 表格（融入叙述，不复活字段）
- 旧业务编号：`BD-BSA-01`/`NS-CH-01`/`CS-CH-03`/`DP-CH-xx`（改用两段式 ID；DP 内嵌不编号）

---

## 扩展接口

未来加字段直接在 YAML 末尾追加；脚本/agent 原样保留未知字段（同命令/特性/task 层）。

## 类型-字段必填对照（防漏填）

| 类型 | `id` | `type` | `name` | `name_zh` | `domain` | `scenario` | `status` |
|---|---|---|---|---|---|---|---|
| **BD** | ✅ | ✅ | ✅ | ✅ | ✅（自身域） | — | ✅ |
| **NS** | ✅ | ✅ | ✅ | ✅ | ✅（所属 BD） | ✅（自身场景） | ✅ |
| **CS** | ✅ | ✅ | ✅ | ✅ | ✅（所属 BD） | ✅（所属 NS） | ✅ |

> `domain`/`scenario` slug 必须与 id 第二段（及 CS 的 scenario 前缀）一致——漏填或前后不一致会阻塞平台 classify（path_fields 缺失报错）与前端左树分组。
