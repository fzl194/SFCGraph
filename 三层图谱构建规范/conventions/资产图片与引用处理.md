# 资产图片与引用处理（跨层约定）

> 源产品文档 md → 资产 md 搬迁时，**图片**与**文档引用**怎么处理。所有层共用。
> v0.13.0 起**特性层 / 命令层 / ConfigObject 三层**落地；引用统一为逻辑 `[[ID]]`（前端按 `@` 三段式 ID 查名渲染 + 跳转，与 `## 边` 同源）。

---

## 为什么要这个约定

源端每个 md 由阶段 0 导出器（`scripts/product_doc_md_exporter_optimized.py`）生成，伴随：

1. **图片**：md 旁有一个 `{md名}.assets/` 目录装 PNG，正文用 `![]({md名}.assets/x.png)` 引用（导出器 `_copy_asset` 已把图复制到此处）。
2. **文档引用**：正文里 `[文字](相对路径.md)` 指向源 output 目录树里的其它文档（命令引用、特性引用等）。

资产构建把 md 搬到 `三层图谱资产/{Type}/{nf}/{version}/` 下并**改名/聚合**，如果只搬 md 文本：

- 图片目录没跟过来 + 目标 md 改名 → `![](...)` 全成死链（**图片全丢**）。
- 相对路径指向源 output 树 → 搬迁后全部失效（**引用全断**）。

本约定规定这两类一律**就地修复**，使每个资产自包含、引用可跳。

---

## 规则 1：图片

### 存放位置（按对象自然粒度）
| 对象类型 | 图片目录 | 说明 |
|---|---|---|
| Feature（按 code 聚合的多文档文件夹） | `{特性文件夹}/assets/` | 该特性下**所有**源 md 的 `.assets/` 合并进同一个 `assets/` |
| 单文档对象（命令、license 等） | `{资产 md}.assets/` 或同目录 `assets/` | 每文件旁一个 |

> 统一用 **`assets/`** 子目录命名；特性层因聚合模型放特性文件夹下，单文档对象放文件旁。

### 处理
1. 定位源图：源图绝对路径 = `(源 md 所在目录 / 引用 URL).resolve()`（URL 相对于源 md）。
2. 拷贝到目标 `assets/`，**改写**引用为本地相对路径：`![]({旧}.assets/x.png)` → `![](assets/x.png)`（相对目标 md）。
3. **冲突处理**：
   - 同名 + 同内容（按文件 hash）→ 去重，复用已拷贝的文件名（典型的共享图标，如 `notice_3.0-zh-cn.png`）。
   - 同名 + 不同内容 → 按源文档 slug 前缀消歧（`{slug}_x.png`）。
4. 源图缺失 / 外链（http、data:）/ 锚点 → 原样保留引用，不处理。

> 不在 YAML 加图片字段；图片纯正文资源。

---

## 规则 2：文档引用 `[文字](目标)`

逐条分类解析。**图片语法 `![](...)` 不在此规则范围**（由规则 1 处理）。

### 分类与解析（输出裸逻辑引用 `[[ID]]`）
| 类别 | 识别信号 | 解析 | 命中后输出 |
|---|---|---|---|
| **命令引用** | 叶子名/链接文字含全角括号 `（CMD）`，或标签本身即命令名（如 `[LST ME]`），且该命令名在命令资产索引中 | 抽命令名 | `[[{nf}@MMLCommand@{CMD}]]` |
| **特性引用** | **源文件名**命中「源文件名→目标文档ID」映射（概述+子文档都能命中；子文档文件名无特性码，必须靠此）；否则叶子名含特性码 `[A-Z]+FD-\d{6}` 且在特性索引中 | 概述=`{nf}@Feature@{code}`；子文档=`{nf}@Feature@{code}-{slug}` | `[[{nf}@Feature@{code}]]` 或 `[[{nf}@Feature@{code}-{slug}]]`（精确到具体子文档） |
| **其它/不可解析** | 以上都不匹配，或目标资产不存在；含 PDF / 外链 | — | **剥 URL、保留链接文字**：`[文字](死链)` → `文字` |

> 命令、配置对象、license 是单文档对象，引用天然精确到具体 md；**特性是多文档文件夹**，靠「源文件名→目标ID」映射精确到被引用的那个文档（概述或某子文档），而非笼统指特性文件夹。

### 不动的引用
- 外链（`http://` / `https://` / `//` / `mailto:` / `tel:` / `data:`）
- 页内锚点（`#xxx`）
- 图片（`![](...)`，规则 1 处理）

### 为什么用 `[[ID]]` 而非相对路径
逻辑引用按 ID 不按路径——目录树怎么搬都不断；正文与 `## 边` 同一种格式；前端一套正则识别 `[[...]]` → 按 `@` 切三段 ID → 查目标 `name` 渲染可读名 + 点击跳转。**裸 ID**（不带别名），显示名由前端查目标资产 `name` 字段得到（单一来源、不漂移）。

## 构建依赖与降级

- **命令引用解析依赖命令资产已存在**：构建特性层前，命令层应已建（命令资产索引来自 `Command/{nf}/{ver}/` 或命令构建第一趟的内存命令名集）。若命令资产尚未建，命令引用全部降级为「剥 URL 留文字」，不报错；命令层建后重跑特性层即可恢复跳转。
- **特性子文档精度依赖「源文件名→目标ID」映射**：特性层构建前先预算该映射（概述/子文档 ID 同一套 overview+slug 逻辑），特性引用按源文件名精确命中子文档；命不中才退回特性概述。
- **死链清理是单向的**：剥 URL 后不保留原路径；如需追溯，看源 md。

---

## 各层落地差异（实施备忘）

- **特性层**：`feature/scripts/_common.py` 实现 `rewrite_images` / `rewrite_doc_refs`；`build_features.py` 预算 `src_to_id`（源文件名→目标文档ID）、每特性文件夹建 `assets/`、按文档循环改写。图片索引（hash→name）按**特性文件夹**共享。
- **命令层**：`command/scripts/_common.py` 同样实现两 helper；`build_commands.py` 在 `clean_md` 后接线，命令索引用第一趟内存 `command_names`，图片拷到 `Command/{nf}/{ver}/assets/`（全版本共享）。
- **ConfigObject**：`build_configobjects.py` 的 `desc` 继承自 ADD 命令资产 md（引用已是 `[[ID]]`），仅对 `desc` 补一次 `rewrite_images`（从命令 assets/ 拷图到 ConfigObject assets/）；`[[ID]]` 透传。
- **每层独立实现**（不跨层 import），遵循「层与层互不依赖」原则；逻辑一致、各自维护副本。

---

## 核查要求（对应各层 `check.md`）

- **图片闭环**：`![](assets/x.png)` 引用的 png 实存于同文件夹；无残留 `{旧名}.assets/` 旧路径。
- **引用闭环**：命令/特性引用指向真实存在的资产；死链已剥 URL 留文字；无残留指向 output/ 的相对路径。
- **核查独立性**：引用分类的核查须用**独立抽样/独立正则**复核，不复用构建同一条正则（防盲区一致）。
