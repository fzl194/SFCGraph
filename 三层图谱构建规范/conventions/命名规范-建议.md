# 命名规范-建议

> 基于 `assets/` 现有资产的命名观察 + 统一建议。
> **本文是建议（advisory），非绑定规则。** 各层细化构建时由你定夺；某条建议被采纳为规则后，升格进对应层 SOP 并触发版本升级。

---

## 1. 头号分界：NF 隔离两类资产

现有资产天然分两类，这是所有命名差异的根源：

| 类别 | 是否带 nf@version | ID 段数 | 目录 | 现有对象 |
|---|---|---|---|---|
| **NF 隔离类** | 是 | 4 段 `{nf}@{version}@{Type}@{semantic}` | `{layer}/{nf}/{version}/` | 命令、配置对象、特性、license、task |
| **跨 NF 类** | 否 | 2 段 `{Type}@{semantic}` | `{layer}/{domain}/{scenario}/` | 业务域、场景、方案 |

实样：
- NF 隔离：`UDG@20.15.2@MMLCommand@ADD URR`、`UDG@20.15.2@Feature@GWFD-020301`、`UDG@20.15.2@Task@0-00001`
- 跨 NF：`NetworkScenario@charging`、`ConfigurationSolution@charging-converged`

**为什么业务层不带 nf@version**：一个方案（CS）常同时引用 UDG 和 UNC 的 task（如融合计费同时用两侧命令）。强行加 nf@version 会让跨网元方案无法表达。所以业务层"不带"是设计，不是疏漏。

**建议**：保留这两类划分；每层 SOP 开头声明本层属于哪类。这一条由你在细化各层时确认。

---

## 2. ID 分隔符与段序

观察：统一用 `@` 分段，段序固定（见 §1）。

**建议**：`@` 作 ID 分隔符不变，与文件名分隔符 `-` 区分开，不混用。

---

## 3. 文件名 ↔ ID 的派生（有一处不一致）

| 对象 | ID 语义段 | 文件名 | 一致？ |
|---|---|---|---|
| 命令 | `ADD URR`（带空格） | `ADD-URR.md`（空格→短横） | ✗ |
| 配置对象 | `URR` | `URR.md` | ✓ |
| 特性 | `GWFD-020301` | `GWFD-020301.md` | ✓ |
| license | `LKV3G5BCBC01` | `LKV3G5BCBC01.md` | ✓ |
| task | `0-00001` | `0-00001.md` | ✓ |
| 业务 | `NetworkScenario@charging` | `NetworkScenario@charging.md` | ✓ |

**唯一问题**：命令语义段带空格（`ADD URR`），文件名用短横（`ADD-URR`），导致 wiki 链接和 ID 对不上。

**建议**：文件名必须能从 ID 语义段**确定性派生**。命令的 `ADD URR` 建议统一为无空格——要么 ID 里就写 `ADD-URR`，要么定义"空格→短横"派生规则并写进命令层 SOP。

---

## 4. 语义段的大小写/风格（不统一）

观察：
- 命令/配置对象/特性/license：**大写**代码或 token（`ADD URR`、`URR`、`GWFD-020301`、`LKV3G5BCBC01`）
- task：`{层}-{序号}`（`0-00001`、`1-00002`、`2-00004`）
- 业务：**小写 kebab**（`charging`、`charging-converged`、`bandwidth-adc-trigger`）

**建议**：按"语义来源"定风格，而非全局统一一种：
- **产品原生代码/缩写**（特性码、license 码、命令 token、对象缩写）→ 保留产品原大小写（通常大写）
- **人造描述性 slug**（业务场景/方案名）→ 小写 kebab-case
- **编号类**（task）→ `{层标识}-{序号}`

---

## 5. 元文件与索引

观察：task 目录有 `_atom-refactor-tracker.md`、`_numbering.json`；license 有 `index.md`。

**建议**：
- `_` 前缀 = 非资产元文件（追踪、编号映射等），构建时不当作对象处理
- `index.md` = 目录索引页

---

## 6. evidence 命名（较松）

观察：`{主题}_{数字id}.md`，主题含中文（如 `2_3_4_5G-PCC功能差异_47013471.md`、`3GPP-QoS简介_42967127.md`）。

**建议**：evidence 命名留到"证据约定"时一起约束。暂只建议：命名稳定可复现（同一源多次导出，文件名一致），便于 wiki 链接不失效。

---

## 汇总：需你在各层细化时拍板的点

1. 每层属于 **NF 隔离类** 还是 **跨 NF 类**（建议见 §1 表）
2. 命令语义段空格怎么收（ID 改无空格 / 定义派生规则）
3. 大小写风格是否采纳"按语义来源分"（§4）
4. evidence 命名规则（随证据约定一起定）
