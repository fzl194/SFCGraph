# 命名规范-建议

> 基于 `三层图谱资产/` 与 `assets/business/` 现有资产的命名观察 + 统一建议。
> 命令/特性/task 三层的 3 段 ID 方案**已落地为规则**（见各层 `字段定义.md`），本文记录该共识并给业务层提供参考；业务层的 2 段方案待业务层 SOP 定稿后升格。
> **本文仍为建议（advisory）**：某条建议被采纳为规则后，升格进对应层 `字段定义.md` 并触发版本升级。
>
> **历史**：本文早期版本曾建议 "4 段 `{nf}@{version}@{Type}@{semantic}`"，但 `version` 后移出 ID（只进 YAML `version:` 字段 + 存储路径），实际落地为 **3 段**。本版已对齐落地现实。

---

## 1. 头号分界：NF 隔离两类资产

现有资产天然分两类，这是所有命名差异的根源：

| 类别 | nf/version 是否进 ID | ID 段数 | 存储路径 | 现有对象 |
|---|---|---|---|---|
| **NF 隔离类** | 否（version 只在 YAML + 路径） | 3 段 `{nf}@{Type}@{local}` | `{Type}/{nf}/{version}/` | 命令、配置对象、特性、license、atom/compound/feature_task |
| **跨 NF 类** | 否 | 2 段 `{Type}@{slug}` | `assets/business/{domain}/[{scenario}/]` | 业务域(BD)、场景(NS)、方案(CS) |

实样（已落地）：
- NF 隔离 3 段：`UDG@MMLCommand@ADD URR`、`UDG@Feature@GWFD-020301`、`UDG@ConfigObject@URR`、`UDG@License@LKV3G5BCBC01`、`UDG@AtomTask@ADD URR`、`UDG@CompoundTask@charging-core-trio`、`UDG@FeatureTask@GWFD-010101`
- 跨 NF 2 段：`BusinessDomain@business-awareness`、`NetworkScenario@charging`、`ConfigurationSolution@charging-converged`

**为什么 version 不进 ID**：同一逻辑对象跨版本是同一对象（`ADD URR` 在 20.15.2 与后续版本是同一条命令的逻辑身份）；版本是实例维度，进 YAML `version:` 字段 + 存储路径 `{Type}/{nf}/{version}/` 即可。把 version 塞进 ID 会让"跨版本引用"无法表达。

**为什么业务层不带 nf**：一个方案（CS）常同时引用 UDG 和 UNC 的 task（如融合计费同时用两侧命令）。强行加 nf 会让跨网元方案无法表达。所以业务层"不带"是设计，不是疏漏。

---

## 2. ID 分隔符与段序

观察：统一用 `@` 分段，段序固定（见 §1）；version 不进 ID（见 §1）。

**建议（已落地）**：`@` 作 ID 分隔符不变，与文件名内的 `-`/空格 区分开，不混用。

---

## 3. 文件名 ↔ ID（文件名 = 完整 ID）

| 对象 | ID | 文件名 | 一致？ |
|---|---|---|---|
| 命令 | `UDG@MMLCommand@ADD URR`（带空格） | `UDG@MMLCommand@ADD URR.md` | ✓ 文件名=完整ID，保留空格 |
| 配置对象 | `UDG@ConfigObject@URR` | `UDG@ConfigObject@URR.md` | ✓ |
| 特性（概述） | `UDG@Feature@GWFD-020301` | 目录 `UDG@Feature@GWFD-020301/概述.md` | ✓（概述用固定名 `概述.md`）|
| 特性（子文档） | `UDG@Feature@GWFD-020301-{slug}` | `UDG@Feature@GWFD-020301/{slug}.md` | ✓（slug=源文件名净化，保留中文/空格）|
| license | `UDG@License@LKV3G5BCBC01` | `UDG@License@LKV3G5BCBC01.md` | ✓ |
| atom | `UDG@AtomTask@ADD URR` | `UDG@AtomTask@ADD URR.md` | ✓ |
| compound | `UDG@CompoundTask@charging-core-trio` | `UDG@CompoundTask@charging-core-trio.md` | ✓ |
| feature_task | `UDG@FeatureTask@GWFD-010101` | `UDG@FeatureTask@GWFD-010101.md` | ✓ |
| 业务 | `NetworkScenario@charging` | `NetworkScenario@charging.md` | ✓ |

**规则（已落地）**：除特性子文档外，**文件名 = 完整逻辑 ID**（含 `@`、含命令名空格），不做任何字符替换。特性子文档因归在同一特性目录下，文件名只用 `{slug}` 段（概述固定为 `概述.md`）。

**历史不一致已消除**：早期版本命令文件名曾用 `ADD-URR.md`（空格→短横），导致 wiki 链接与 ID 对不上。现行规则统一为"文件名=完整ID、保留空格"，该问题已不复存在。

## 4. 语义段（local）的大小写/风格

观察：
- 命令/配置对象/特性/license：**保留产品原生大小写**（`ADD URR`、`URR`、`GWFD-020301`、`LKV3G5BCBC01`）——大写 token 直接用
- atom：local = 命令名（保留空格），1:1 锚定命令
- compound：local = **小写 kebab-case 业务语义名**（`charging-core-trio`、`addr-alloc-rule`），Agent 命名
- feature_task：local = feature_code（1:1 锚定特性）
- 业务（BD/NS/CS）：local = **小写 kebab-case 语义 slug**（`business-awareness`、`charging`、`charging-converged`）

**建议（已落地）**：按"语义来源"定风格，而非全局统一一种：
- **产品原生代码/缩写/命令 token**（命令名、对象名、特性码、license 码）→ 保留产品原大小写
- **人造描述性 slug**（compound 业务名、业务场景/方案名）→ 小写 kebab-case

**历史**：task 曾用 `{层}-{序号}`（`0-00001`/`1-00002`/`2-00004`）编号体系，**已废弃**——atom 改用命令名锚、compound 改用 kebab 业务名、feature_task 改用 feature_code 锚。编号不再进 ID。

---

## 5. 元文件与索引

观察：
- NF 隔离类每个 `{Type}/{nf}/{version}/` 下有一个 `_build_manifest.json`（构建产物清单：`sop_version`/`built_at`/count/ID 列表）
- `_` 前缀 = 非资产元文件（manifest、追踪、编号映射、审计报告等），构建时不当作对象处理
- `index.md` = 目录索引页（业务层顶层 `assets/business/index.md` 即 registry）

**建议**：保留 `_` 前缀约定。task 层（Procedure 体裁）目前无 manifest——因非脚本流水线产物，后续是否补一份构建清单由 task 层 SOP 决定。

**历史**：task 曾有 `_numbering.json`/`_atom-refactor-tracker.md` 等编号映射元文件，随编号体系废弃已移除。

---

## 6. evidence 命名（较松）

观察：`{主题}_{数字id}.md`，主题含中文（如 `2_3_4_5G-PCC功能差异_47013471.md`、`3GPP-QoS简介_42967127.md`）。

**建议**：evidence 命名留到"证据约定"时一起约束。暂只建议：命名稳定可复现（同一源多次导出，文件名一致），便于 wiki 链接不失效。

---

## 汇总：各层已落地 / 待拍板的点

已落地（命令/特性/task 三层 `字段定义.md` 权威）：
1. ✅ 3 段逻辑 ID `{nf}@{Type}@{local}`，version 不进 ID
2. ✅ `@` 作 ID 分隔符
3. ✅ 文件名 = 完整 ID（保留空格），特性子文档例外
4. ✅ 大小写按语义来源分（原生代码保留 / 人造 slug 小写 kebab）
5. ✅ task 弃编号，atom/compound/feature_task 各用锚（命令名/kebab/feature_code）
6. ✅ `_` 前缀元文件、`_build_manifest.json` 清单

待业务层 SOP 定稿后升格：
1. 业务层 2 段 ID `{Type}@{slug}`（已观察到，待写进业务层 `字段定义.md`）
