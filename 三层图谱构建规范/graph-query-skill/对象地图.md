# 对象地图

> 四层 10 类型对象的**职责、字段、语义、何时下钻**。选层入口与下钻方向的权威参考。
> 配合 [SKILL.md](SKILL.md) §3（四层职责速查）使用——§3 给选层依据，本文给类型级细节。

---

## 1. 层级总览

四层自上而下，但**是 DAG 不是严格树**——一个对象可被多个上层引用，层间沿 `[[ID]]` 走，不假设路径唯一。

```
业务层（定位 + 诉求路由）  BusinessDomain → NetworkScenario → ConfigurationSolution
                            业务域            场景              方案 ★编排核心
                              │ 方案下挂三类 task
                              ▼
任务层（配置编排）          FeatureTask（特性级）── CompoundTask（步骤级）── AtomTask（命令级）
                              │ ref→Feature              │ ref→MMLCommand
                              ▼                          ▼
特性层（静态产品知识）      Feature / License      命令层（命令+参数）  MMLCommand / ConfigObject
```

- **业务层** = 业务诉求定位 + 方案路由（NS 决策点表）
- **任务层** = 配置编排（怎么配、怎么组装）
- **特性层 / 命令层** = 静态知识源（原理/参数），全量完备

---

## 2. 业务层（跨 NF，2 段 ID `{Type}@{slug}`，无 nf/version）

### BusinessDomain（BD）— 业务域·根
- **干啥**：最抽象的业务归属层。"这需求属于哪块业务"。
- **ID**：`BusinessDomain@{domain-slug}`，如 `BusinessDomain@business-awareness`
- **关键字段**：`name`（中文名）、`domain`
- **何时下钻**：拿不准业务归属时，从 `/domains` 取全部域 md 通读定位。
- **边**：`## 边` = 下游场景 `[[NetworkScenario@*]]`

### NetworkScenario（NS）— 场景·问题/边界 ★入口资产
- **干啥**：讲"解决什么业务问题 + 边界"。**正文含「决策点路由表」**（业务诉求 → 推荐方案）——这是最强的业务语义入口。
- **ID**：`NetworkScenario@{scenario-slug}`，如 `NetworkScenario@charging`
- **关键字段**：`domain`、`scenario`
- **何时下钻**：锁定域后，读 NS 的决策点路由表选方案。
- **边**：`## 边` = 上游域 + 下游方案

### ConfigurationSolution（CS）— 方案·编排核心 ★
- **干啥**：方案落地为 task 编排。讲"这个业务诉求由哪些 task 完成 + 特性级变种 + 跨网元协同 + 配置级 DP/约束"。
- **ID**：`ConfigurationSolution@{scenario}-{solution-slug}`，如 `ConfigurationSolution@charging-converged`
- **关键字段**：`domain`、`scenario`
- **何时下钻**：锁定方案后，沿 CS 的 `## 边` 取它编排的 task。
- **边**：`## 边` = 编排的 FeatureTask/CompoundTask/AtomTask

> 业务层三类型的关系（DP/约束/特性矩阵）都在**正文叙述 + 表格**，不进 YAML。引用粒度 = md 级。

---

## 3. 任务层（NF 隔离，3 段 ID `{nf}@{Type}@{name}`，带 version）

### AtomTask — 命令级·怎么配（1:1 命令）
- **干啥**：讲"这条命令在这个场景里**怎么配**"——配置方法 + **引用约束**（哪个参数只能引用哪种对象）+ 决策点。
- **ID**：`{nf}@AtomTask@{命令名}`，如 `UDG@AtomTask@ADD URR`（命令名保留空格）
- **关键字段**：`ref`（→ MMLCommand）、`name_zh`（配置动作中文名）
- **独特价值**：**引用约束**——MMLCommand 参数表只说"参数含义"，AtomTask 才说"该参数只能引用 ADD X 产出的对象"。查参数引用关系必看 AtomTask。
- **何时下钻**：理解/生成配置时，从 CompoundTask/FeatureTask 的 `## 边` 或正文 `[[ID]]` 取。

### CompoundTask — 步骤级·多命令可复用模块
- **干啥**：高频多命令组合抽成的经典模块（如"计费三件套" URR→URRGROUP→PCCPOLICYGRP），跨方案/跨特性复用。
- **ID**：`{nf}@CompoundTask@{英文名}`，如 `UDG@CompoundTask@charging-core-trio`
- **关键字段**：`command_set`（命令名列表，复用判定 + 快速查询）、`ref=null`
- **何时下钻**：理解"一组命令怎么组装成步骤"时取。

### FeatureTask — 特性级·配置编排
- **干啥**：一个特性分几步配、每步哪些命令。
- **ID**：`{nf}@FeatureTask@{feature_code}`，如 `UDG@FeatureTask@GWFD-020301`
- **关键字段**：`ref`（→ Feature）
- **何时下钻**：从 CS 的编排取，或按 feature_code 定位。

> 任务层三类型互相引用，形成编排图（非层级必经，沿 `[[ID]]` 走）。

---

## 4. 特性层（NF 隔离，全量完备）

### Feature — 特性静态产品文档
- **干啥**：特性的**产品文档原文**（原理 / 流程 / 与其他特性交互 / 配置约束）。一特性多子文档（概述/激活/原理/…），按需取相关子文档。
- **ID**：`{nf}@Feature@{feature_code}`（概述）/ `{nf}@Feature@{feature_code}-{slug}`（子文档）
- **关键字段**：`feature_code`、`doc_type`（概述/激活/原理/…）
- **何时下钻**：需要特性原理/约束（非配置方法）时；FeatureTask.ref 追溯到此。

### License — 特性前置门控
- **干啥**：License 控制项（功能/资源）。特性/方案提及 License 前置时追溯。
- **ID**：`{nf}@License@{license_code}`，如 `UDG@License@LKV3G5V6LI01`
- **关键字段**：`license_code`、`control_item_id`（两格式：纯数字 / 字母数字混合）、`control_item_type`（功能/资源）

---

## 5. 命令层（NF 隔离，全量完备）

### MMLCommand — 命令 + 参数
- **干啥**：命令原文 + **参数定义**（参数说明表在正文，含枚举/类型/约束）。
- **ID**：`{nf}@MMLCommand@{name}`，如 `UDG@MMLCommand@ADD URR`
- **关键字段**：`applicable_nf`、`effect_mode`（立即/对新流/延迟）、`is_dangerous`（高危）
- **参数不单独建**：查参数读对应命令 md 正文。
- **何时下钻**：AtomTask.ref 追溯到此；任何"命令怎么用"的问题直接进此层。

### ConfigObject — 聚合配置对象
- **干啥**：配置对象（如 URR/URRGROUP/RULE），被多条命令操作（ADD/MOD/DEL/RMV/SET/LST/DSP）。差异分析/对象归类用。
- **ID**：`{nf}@ConfigObject@{object}`，如 `UDG@ConfigObject@URR`
- **关键字段**：`object_kind`（entity/global_setting）
- **无原始 md**（聚合而来），正文仅 `## 说明`；`## 边` = 同 object 的所有命令。

## 6. 下钻链（典型路径）

```
GET /stats                          → 摸版图（四层规模 + 哪些域存在）
按问题性质选层 ↓

[业务诉求型]  /domains → BD → NS 决策点表 → CS → ##边 取 task → Atom/Compound
[特性知识型]  /objects?layer=特性层&q= → Feature → 子文档 / License
[命令用法型]  /objects?layer=命令层&q= → MMLCommand（参数在正文）/ ConfigObject
[配置方法型]  /objects?layer=任务层&q= → Compound/Atom → ref → MMLCommand
```

每跳都：`POST /md` 取原文 → 正则扫全文 `[[ID]]`（正文 + 边段）→ 相关则取、无关跳过 → 循环。

## 7. 完备度现状（静态，以 /stats 实时为准）

| 层 | 完备度 |
|---|---|
| 命令层 | ✅ UDG + UNC 全量 |
| 特性层 | ✅ UDG + UNC 全量 |
| 任务层 | ❌ 仅 business-awareness（charging / bandwidth-control / access-control）+ apn-domain（apn-access） |
| 业务层 | ❌ 同上两域 |

> 本表会随建域而过时。**每次查询先 `GET /stats` 看 `per_domain` / `per_domain_scenario` 确认当前到底有哪些域/场景。**
