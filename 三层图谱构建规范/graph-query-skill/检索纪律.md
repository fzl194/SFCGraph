# 检索纪律

> fetch+expand 循环的完备性规则。图谱的"完备遍历"靠这套，不是靠边索引。

---

## 1. 边索引是 curated-only ★最易踩的坑

平台边索引**逐 md 判断**：

| 对象情况 | 进边索引的边 |
|---|---|
| 有 `## 边` 段 | **仅** `## 边` 那几条（正文 `[[ID]]` **不进**） |
| 无 `## 边` 段 | frontmatter `ref` + 正文全部 `[[ID]]`（关系标"参见"） |

**后果**：`/neighbors`、`/subgraph`、反链 `in_edges` 都**只反映 curated 边**。一个对象若被另一个"有 `## 边` 段"的对象在**正文**里 `[[ID]]` 引用，这条引用**不在边索引里**——`/neighbors` 看不到、反链查不到。

**实测**（每个对象"仅正文有、边段没有"的引用数）：方案 CS 3 条、FeatureTask 2 条、CompoundTask 3 条、AtomTask 2 条。

**→ 完备遍历的唯一可靠方式：`POST /md` 取原文 + 正则扫全文 `[[ID]]`（正文 + `## 边` 段都扫）。** 只看 `## 边` 或只调 `/neighbors` 会漏引用。

---

## 2. fetch+expand 四步循环

```
1. POST /md 取回一批 md
2. 正则 \[\[([^\]]+)\]\] 对【每个 md 全文】（正文 + ## 边段）提取全部 [[ID]]
3. 按当前意图判断每个引用相关性：相关 → 加入下一批 ids；无关 → 跳过（防雪崩）
4. POST /md 取下一批，循环至当前问题所需知识齐备
```

> 关键：第 2 步**扫全文**，不是只看 `## 边`。见 §1。

---

## 3. 三条纪律

1. **读必提**：读完每个 md，提取其**全部** `[[ID]]`（正文 + 边段都扫）。
2. **提必判**（按意图）：每个引用按**当前问题**判断相关——相关才取，无关跳过。
3. **按需完备**：只有**与当前问题相关**的引用需取全（漏了答错）；无关的按意图取舍。不强取全部。

---

## 4. 累积去重

每轮 `POST /md` 后记录已取 id 清单。下一批发起前，先检查目标 id 是否已在清单——已在 = 重复，跳过。避免对同一对象反复 fetch。

```
【已取清单】{本轮止已 POST 取过的全部 id}
发起下一批前：候选 ids − 已取清单 = 实际需取
```

---

## 5. 何时停

当**当前问题所需的知识对象已取齐**（能回答问题 / 能支撑下游任务）即停。不必下钻到底——按问题深度决定（见 [SKILL.md](SKILL.md) §3 层职责，需要哪层知识就下钻到哪层）。

---

## 6. 反链怎么办（影响分析/排障类任务）

若任务需要"谁指向了 X"（反链）：

- **快速但不全**：`GET /objects/{X}/neighbors` 的 `in` 字段给 curated 反链。
- **完备**：当前平台**没有**正文引用的反链索引。要完备反查，只能对候选范围的对象逐个 fetch + 扫全文看是否含 `[[X]]`（成本高）。
- **务实建议**：先用 `/neighbors` 的 `in` 拿 curated 反链；若明显不全且任务必需，再限定范围（某层/某域）批量 fetch 扫描。
