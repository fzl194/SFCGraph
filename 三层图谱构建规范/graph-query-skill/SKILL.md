# 图谱查询 SOP

> 本 SKILL 是**通用**图谱查询能力包：教 Agent "怎么把相关知识从三层图谱里取出来"。
> 任务是 Agent 自己的（排障/问答/审计/…），本 SKILL 不规定任务流程，只规定**查询姿势**。

---

## 1. 定位与边界

- **通用查询**：不绑定具体任务。任意核心网任务需要图谱知识时，都走本 SKILL 的查询姿势。
- **与配置生成并行独立**：配置生成 SKILL（`../skill/`）已调好、自成闭环，本 SKILL 不嵌套、不依赖、不修改它。两者是同级消费者。
- **只读**：本 SKILL 只查不改。导入/导出/编辑不在范围内。

---

## 2. 设计原则（四条，必须吃透）

1. **四层平等，皆可入口**。不存在"必须从业务域进"。哪个层都能当起点——由问题性质决定。
2. **先建模，后查询**。Agent 必须先把整张图谱（四层职责 + 完备度）在脑中建好，再下钻。盲目调接口 = 大概率进错层、漏知识。
3. **不用向量**。图谱自带结构化索引：层/类型/name 是分类目录，`[[ID]]` 是显式边。入口靠"读小目录 + Agent 自己推理"（语义判断在 Agent 脑子里，作用在有穷目录上），扩展靠"fetch md + 扫边"。向量是给"搜索空间大到读不完"的场景用的；这里的入口目录小到能整张读完。
4. **完备度感知**。业务层+任务层**不全**（仅业务感知+APN 两域），特性层+命令层**全量**（UDG+UNC）。选层必须判断"目标层对该主题是否完备"，否则会落空。

---

## 3. 四层职责速查（选层依据）★核心

> 这是选层的唯一依据。完整 10 类型字段/语义见 [对象地图.md](对象地图.md)。

| 层 | 回答什么问题（进去找啥） | 进入端点 | 完备度 |
|---|---|---|---|
| **业务层** | "我要做X业务"→ 属于哪个域/场景/方案；业务诉求路由（NS 决策点表） | `/domains`（BD）或 `/objects?type=NetworkScenario/ConfigurationSolution` | ❌ 仅**业务感知 + APN** 两域 |
| **任务层** | "X怎么配"（配置方法 + 引用约束）；多命令怎么组装成步骤 | `/objects?layer=任务层&q=` | ❌ 同上两域 |
| **特性层** | "X特性是啥"（原理/流程/约束/产品知识）；需要/受控于啥 License | `/objects?layer=特性层&q=` | ✅ UDG+UNC **全量** |
| **命令层** | "X命令怎么用"（参数/枚举/用法/是否高危）；URR 这类对象被哪些命令操作 | `/objects?layer=命令层&q=`（+`nf`/`version`） | ✅ UDG+UNC **全量** |

**层间关系（DAG，非严格树）**：

```
业务层  BusinessDomain → NetworkScenario → ConfigurationSolution
            │ 方案下挂 task（编排）
            ▼
任务层  FeatureTask ── CompoundTask ── AtomTask（1:1 命令）
            │ ref 追溯                  │ ref 追溯
            ▼                           ▼
特性层  Feature / License        命令层  MMLCommand / ConfigObject
```

---

## 4. 选层规则

**两步**：① 问题性质 → 候选层；② 候选层对该主题不完备 → 回退到完备层。

| 问题形态 | 候选层 | 回退（候选层不全时） |
|---|---|---|
| "X 怎么配 / X 的配置步骤 / 这参数引用什么对象" | 任务层（Atom/Compound/FeatureTask） | 退命令层查命令用法 + 退特性层查原理 |
| "X 特性的原理/流程/它和谁交互/是干嘛的" | 特性层（Feature） | —（全量） |
| "X 需要/受控于什么 License" | 特性层（License） | —（全量） |
| "X 命令的参数/枚举/用法/高危吗" | 命令层（MMLCommand） | —（全量） |
| "URR/URRGROUP 这类对象被哪些命令操作" | 命令层（ConfigObject） | —（全量） |
| "这需求属于什么业务/哪个方案" | 业务层（NS 决策点表） | 该域没建 → 退特性层按特性名定位 |
| 拿不准 | 先 `/stats` 看版图，按问题里最具体的名词进对应层 | — |

> **关键**：业务层/任务层只在"主题落在已建两域（业务感知/APN）内"时可用。问题若涉及未建域，**直接进特性层或命令层**——它们对全量 UDG+UNC 完备。

## 5. 核心循环

```
① 理解全景（强制第一步，轻）：
   GET /stats → 看 per_layer（四层规模）+ per_domain / per_domain_scenario（哪些域/场景存在）
   ＝ 确认当前版图的活真相（尤其"现在到底有几个域"——本 SKILL 静态写的完备度会过时）
   读 §3 四层职责表 → 在脑中建好图谱模型
② 选层：按 §4
③ 层内定位锚点：/objects?layer=&type=&q=（业务层可 /domains）→ 1~几个候选 ID
④ fetch+expand：POST /md 取原文 → 正则扫全文 [[ID]]（正文 + 边段都扫）→ 按意图剪枝 → 循环
   （循环细节、完备性规则见 检索纪律.md）
⑤ 返回：相关知识对象清单 + 引用链 + 标注哪些层不完备带来的置信度限制
```

> **①是强制的**：未做全景理解就调 fetch = 违反原则 2。`/stats` 很轻，先调它。

---

## 6. 两条硬约束（caveat，踩了就错）

1. **边索引是 curated-only，不能依赖其完备**。`/objects/{id}/neighbors` 只返回对象 `## 边` 段的边（以及"无 `## 边` 段"对象的 ref+正文 `[[ID]]`）。**有 `## 边` 段的对象，其正文里的 `[[ID]]` 不进边索引**。要完备必须 `POST /md` 拿原文 + 正则扫全文。详见 [检索纪律.md](检索纪律.md) §1。
2. **完备度不对称**。见 §3：别在没建的域里找任务/业务知识，会空。

## 8. 与配置生成的关系（不改它）

配置生成 SKILL（`../skill/SKILL.md`）有一套**自带的、面向生成**的图谱用法（业务域根入口 + 7-Phase）。本 SKILL 不复用、不修改那套——配置生成已调好。

本 SKILL 服务**配置生成之外**的任务：那些任务没有"业务域根"这个天然入口，需要 Agent 自己按问题性质选层进入。两套并行，互不干扰。
