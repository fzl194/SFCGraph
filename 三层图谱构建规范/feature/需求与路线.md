# 特性层需求与路线

> 特性层 = **Feature + License**。统一模型：每个文档 = `YAML + 原始文档 + ## 边`，每个可按 ID 引用（同命令层）。**图片不纳入**。

---

## 1. 输入（产品文档内）
- **特性源**：`特性指南/UDG特性指南/{分类}/{feature_code} {名}/`（每特性一个**子文件夹**，含多个 md：特性概述/激活/参考信息…）
- **License 源**：`特性指南/UDG License描述/{UDG|SFIP|VAS} License控制项/{功能|资源}控制项_{id}.md`（每个 md 含**多个 license 段落**）

---

## 2. Feature —— feature_code 聚合模型（v0.10.0，取代文件夹模型 ✅）

> **为何改**：v0.9.0 的「父文件夹名含 code = 一个特性、取其直接子 md」模型在全量 UDG 上严重失效——产品文档目录树是「文件归在分类目录 + 多层嵌套」：某特性 md 常是上级分类目录里的**文件**（非自己文件夹）、或 `实现原理/` 子目录里的文件。结果漏 60% 特性 / 50% 文档，且同 code 多文件夹互相覆盖。
>
> **新模型**：遍历 feature-dir 下**全部 md**，按其路径里**最深的 feature_code** 归组（最深=最具体的特性，跳过上级分类码 IPFD-010000 之类）。同 code 的所有 md → 一个特性文件夹。

```
Feature/{网元}/{版本}/{网元}@Feature@{feature_code}/
├── 概述.md              ID = {网元}@Feature@{feature_code}              ← 特性本体，默认引用指向它
├── OSPF基础.md          ID = {网元}@Feature@{feature_code}-OSPF基础     ← slug=源文件名(去_id后缀)
├── BFD for OSPF.md      ID = {网元}@Feature@{feature_code}-BFD for OSPF
└── ...
```
- **ID 用 slug 不用 doc_type**：doc_type 是多对一分类（一个特性下 N 个「原理」文件），拿它当 ID 后缀会撞名覆盖。故 doc_type **只进 YAML 字段**，ID 区分位用源文件名净化出的 slug。同特性 slug 撞名 → 前补一层父目录消歧。
- **概述判定**：文件名含「特性概述」优先；否则文件名=纯特性名（`{code} {名}`）。无概述的特性进 manifest `features_without_overview`。
- 每个 md 都是 `YAML + 原文 + 边`（统一）。
- **概述**：边 = 所需License / 依赖特性 / 包含子文档（→ 各子文档 slug-id）。
- **子文档**：边 = `属于特性 → 概述`（反向闭环）。

## 3. License —— 段落模型（待确认 ❓）

源（控制项 md）里每个 license 是一段：
```
#### [81203214 LKV3G5V6LI01 逻辑接口支持IPv6](#anchor)
| 归属NF | SGW-U、PGW-U、UPF |
| 功能描述 | ... |
| 实现描述 | ... |
| 取值范围 | 0~1 |
| 默认值 | 1 |
| 对应特性 | GWFD-020405 逻辑接口支持IPv6 |
| 相关控制项 | 无 |
| 应用场景 | ... |
```
→ 每个 license 段建一个资产：
```
License/{网元}/{版本}/{网元}@License@{license_code}.md
```
- **YAML**：license_code / control_item_id / name / applicable_nf（归属NF）/ control_item_type（功能·资源）
- **原文**：该 license 的表内容（功能描述/实现描述/取值范围/默认值/对应特性/应用场景）
- **边**：`对应特性 → Feature`（从"对应特性"字段取 feature_code），与 Feature 的"所需License"反向闭环

---

## 4. 边（从源文档推导，双向闭环；不依赖 FeatureGraph jsonl）
| 边 | 出现位置 | 来源 |
|---|---|---|
| Feature(概述) → License（所需License） | 概述 md | **只扫「可获得性」章节** license_code（不扫全文，避免互斥表里别人 license 误挂）|
| License → Feature（对应特性） | license md | license 表"对应特性"字段（feature_code）|
| Feature → Feature（依赖/冲突） | 概述 md | **只扫「与其他特性的交互」章节** feature_code |
| Feature(概述) ↔ 子文档（包含/属于） | 概述 + 子文档 | 文件夹归属 |

---

## 5. 统一约定（同命令层）
- 逻辑ID `{网元}@{Type}@{local}`（网元在最前，版本动态解析）。
- 存储 `{type}/{网元}/{版本}/...`。
- 无 `source` 字段（原文内联）；无图片。
