# CR-20260716-001-特性文件夹模型不匹配UDG实际目录树

> **状态：✅ 已采纳 / 已关闭** —— 在 v0.10.0 落地（聚合模型 + 文件名 ID）。详见 CHANGELOG [0.10.0]。下文为原始 CR 内容（保留追溯）。

## 场景
全量构建 UDG 特性层（`--nf UDG --version 20.15.2`，feature-dir=`output/UDG_Product_Documentation_CH_20.15.2/特性部署/特性指南/UDG特性指南`）。
自测子集（IPv6 功能，每特性=独立文件夹、md 全是直接子文件）构建正确；但**全量 UDG 的目录树结构与子集不一致**，导致按当前 SKILL 跑出来的 Feature 资产明显不对。

## 规范现状
- `feature/SKILL.md`「Feature：文件夹模型」+ `feature/scripts/build_features.py`：
  - 找特性文件夹：`feature_dir.rglob("*")` 取**目录名含 feature_code** 的文件夹（`_common.parse_feature_code`）
  - 取该特性的 md：`sf.glob("*.md")` —— **只取直接子文件，不递归子目录**
  - 一个 feature_code → 一个输出文件夹 `Feature/{nf}/{version}/{nf}@Feature@{feature_code}/`，直接子 md 各建一资产
- `feature/字段定义.md` / `check.md`：默认引用指向「概述」md（`{nf}@Feature@{feature_code}`）

## 按当前规范跑出来的问题（全量 UDG，源 888 个特性 md）
1. **嵌套 md 漏掉约 49%**：`glob("*.md")` 非递归，**直接子 md 421 个收到 / 嵌套子目录 md 411 个漏掉**。
   - 例：`IPFD-010001 接口管理/` 源有 3 个 md（特性概述 + `实现原理/控制接口震荡特性` + `实现原理/最大传输单元MTU`），构建只产出 `概述.md`，2 个实现原理 md 丢失。
   - `实现原理/`、`特性配置/`、`调测XXX/` 等子目录下的 md 全部不入资产。违背用户要求「特性所有文件都得纳入进来」。
2. **同 feature_code 多文件夹 → 后写覆盖先写（数据丢失/不确定）**：**99 个 feature_code 各自出现在 2~3 个源文件夹**（如 `GWFD-010101` 出现 3 次）。脚本按 `sorted(subfolders)` 顺序逐个写同一输出文件夹，后者覆盖前者。manifest 记 `feature_count=129`，磁盘实际只留 127 个文件夹。
3. **15/127 特性文件夹缺 `概述.md`（默认引用目标缺失）**：原因有二——
   - 概述 md 落在**上级分类文件夹里作为文件**（如 `NPFD-010014 支持NTP功能_xxx.md` 在 `NPFD-010000 操作维护功能/` 下，而 `NPFD-010014 支持NTP功能/` 文件夹内只有 激活/调测）→ 概述归不到自己的特性文件夹。
   - 概述 md 在嵌套子目录里被漏。
4. **分类文件夹被当成特性建**：`IPFD-010000 接口与链路`、`IPFD-014000 路由功能`、`IPFD-015000`、`NPFD-010000 操作维护功能`、`NPFD-010004 设备管理` 等**分类/分组文件夹**目录名也含 feature_code，其直接子文件其实是各子特性的概述 md，被误当成一个「特性」的 doc。
5. **doc_type 误判**：`调测IPv6 SA`、`调测IPv4v6双栈接入` 等「调测」类 md 不命中 `_common._DOC_TYPE_KEYWORDS` → 退化成 `sanitize_doc_type` 产出怪异 doc_type；文件名=纯特性名（如 `IPFD-014003 静态路由`，实为概述）也判错。

> License 侧无此问题：`build_licenses.py` 按段切分、`rglob` 全收控制项 md，98 个 license 全部正确（YAML 7 字段 / 原文表 / 对应特性边 / 概念类 md 无段标题自动跳过）。

## 当时怎么处理的
**严格按当前 SKILL 执行了 `build_all.py`，未擅改脚本**（遵守「Agent 不擅改 SOP 只提 change-request」纪律）。产出已写入 `三层图谱资产/Feature/UDG/20.15.2/`（129 特性声明 / 127 文件夹 / 421 doc）与 `三层图谱资产/License/UDG/20.15.2/`（98 license）。Feature 侧产物**不达标**（漏 49% md、15 特性无概述、99 特性覆盖丢失、分类文件夹误建），等待本 CR 采纳后重建。

## 建议的规范补丁
根因是**「按文件夹归属」的模型不匹配 UDG「按 feature_code 归属 + 多层嵌套」的实际树**。建议改为**按 feature_code 聚合 md**，而非按直接父文件夹：

1. **收集单位改为 md，按 feature_code 分组**：遍历 feature-dir 下所有 md（`rglob("*.md")`），用文件**全路径**里的 feature_code 归组（`_common.parse_feature_code(相对路径)`）。同一 feature_code 的所有 md → 一个特性资产文件夹。天然解决「嵌套漏掉」「同 code 多文件夹覆盖」「概述落在上级文件夹」三类问题。
2. **doc_type 识别增强**：
   - 文件名含「特性概述」→ `概述`（特性本体，默认引用目标）。
   - 补关键词：`调测`→`调测`、`原理/实现原理`→`原理`、`去激活`→`去激活`、`配置`→`配置`。
   - 兜底：含 feature_code 但无 doc 关键词、且路径在 `实现原理` 下的 → `原理`；仍无法判定 → `sanitize_doc_type`（保留现状）。
3. **分类文件夹**：`NPFD-010000`/`IPFD-010000` 这类「分组码」若自身无「特性概述」md，不应单独成特性；其下子特性的概述 md 按规则 1 归到各自 feature_code。
4. **概述缺失保护**：若某 feature_code 聚合后仍无 `概述` doc，`check.md` 标 WARN（而非静默），并在 manifest 记「无概述特性」清单，供人工复核。
5. 同步更新 `SKILL.md`（Feature 文件夹模型 → feature_code 聚合模型）、`字段定义.md`（doc_type 全集补 调测）、`check.md`（概述缺失检查项）。

## 影响范围
- 对象类型：**Feature**（License 不受影响）。
- 数量：UDG 特性源 md 888 个，当前仅入资产 421 个（漏 411 + 覆盖丢失）；采纳后预计覆盖全部 ~300 个特性 / ~888 doc。
- 已建资产影响：**Feature 侧需重建**（License 侧无需重建）。
