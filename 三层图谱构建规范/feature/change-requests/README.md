# change-requests

> 构建中发现规范没覆盖或写错的地方，写在这里。**Agent 写，SOP 维护方审**。

---

## 何时写一条

- 规范没说怎么处理某种情况
- 规范之间互相矛盾
- 按规范做出来明显不对
- 规范可执行性有问题（Agent 读不懂 / 无法执行 / 检查点过不了）

## 怎么写

每条一个 md 文件，命名 `CR-YYYYMMDD-序号-简述.md`。

例：`CR-20260715-001-命令证据无源.md`

文件内容字段：

| 字段 | 说明 |
|---|---|
| **场景** | 什么对象、什么情况下遇到的（给足够上下文，让维护方能复现） |
| **规范现状** | 当前规范怎么说（引用具体文件 + 位置），或明确写"未覆盖" |
| **当时怎么处理的** | Agent 临时采取的处理（要能追溯，便于以后统一修正） |
| **建议的规范补丁** | 你认为规范该怎么改/补 |
| **影响范围** | 估计影响哪些对象类型 / 多少对象 |

## 纪律

- **写 change-request 不阻断当前构建**——Agent 用当前规范的最佳理解继续把活干完。
- 一条 change-request 只讲一件事。
- **不要在这里改规范文件本身**。规范的修改走"升版本 + 写 CHANGELOG"，不在这个文件里直接改。

## 处理流程

```
1. Agent 构建时写 CR
2. SOP 维护方审视：采纳 / 驳回 / 待定（也可在构建环境直接处理）
3. 采纳的 → 改 SOP → 升版本 → CHANGELOG 记录 → 关闭 CR
```

---

## 模板（新建 CR 时复制）

```markdown
# CR-YYYYMMDD-序号-简述

## 场景
（什么对象、什么情况）

## 规范现状
（引用规范文件 + 位置，或"未覆盖"）

## 当时怎么处理的
（Agent 的临时处理）

## 建议的规范补丁
（你认为该怎么改）

## 影响范围
（对象类型 / 数量估计）
```
