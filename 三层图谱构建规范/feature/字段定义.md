# 特性层 YAML 字段定义

> 特性层两类文档（**Feature 文档 / License 文档**）的 YAML 字段。每个文档都是 `YAML + 原始文档 + ## 边`。

## 一、Feature 文档 — 7 字段

> 每特性（按 feature_code 聚合）一个文件夹，里面每个 md（概述/激活/原理…）都用这套字段。`feature_code` 匹配 `[A-Z]+FD-\d{6}`。

| 字段 | 类型 | 必填 | 来源 / 规则 |
|---|---|---|---|
| `id` | string | ✅ | 概述 = `{nf}@Feature@{feature_code}`；子文档 = `{nf}@Feature@{feature_code}-{slug}`（slug=源文件名净化，**不用 doc_type**，因 doc_type 多对一会撞名）|
| `type` | string | ✅ | 固定 `Feature` |
| `name` | string | ✅ | 文档标题（概述 = 特性名；子文档 = slug）|
| `nf` | string | ✅ | `--nf` |
| `version` | string | ✅ | `--version` |
| `feature_code` | string | ✅ | 特性编码（如 `GWFD-020352`），从源 md 路径里最深的 code 解析 |
| `doc_type` | string | ✅ | 分类字段（**不进 ID**）：`概述`/`激活`/`去激活`/`调测`/`参考信息`/`原理`/`配置`/… 文件名关键词优先，否则按所在子目录（实现原理→原理），再退回「其它」 |

> 特性元数据（适用 NF、特性分类、配置相关性等）保留在**概述原文**里，不进 YAML（最小抽取，同命令层）。
> **图片**存于特性文件夹 `assets/`；**文档引用**就地改写/清理；二者均不进 YAML（正文资源，规则见 [conventions/资产图片与引用处理](../conventions/资产图片与引用处理.md)）。

## 二、License 文档 — 7 字段

> 控制项 md 里每个 license 段（`#### [{control_id} {license_code} {名}]` + 表）建一个。

| 字段 | 类型 | 必填 | 来源 / 规则 |
|---|---|---|---|
| `id` | string | ✅ | `{nf}@License@{license_code}` |
| `type` | string | ✅ | 固定 `License` |
| `name` | string | ✅ | license 名（段标题）|
| `nf` | string | ✅ | `--nf` |
| `version` | string | ✅ | `--version` |
| `license_code` | string | ✅ | 如 `LKV3G5V6LI01`（段标题第二段）|
| `control_item_id` | string | ✅ | 控制项 ID（段标题第一段）。**两种格式**：纯数字 `81203214` / 字母数字混合 `82200CKP` |
| `control_item_type` | string | ✅ | `功能` / `资源`（源文件名：功能控制项/资源控制项）|
| `applicable_nf` | list | - | 表"归属NF"行 |

> license 详情（功能描述/实现描述/取值范围/默认值/对应特性/应用场景）保留在**原文（表格）**里，不进 YAML。

## 扩展接口
未来加字段直接在 YAML 末尾追加；脚本原样保留未知字段。
