# 演进机制

> 规范本身如何安全演进。
> 本目录给 Agent 用，**稳定性是第一要求**：今天构建和下个月构建，必须能产出一致的结果。

---

## 核心纪律（三条）

1. **任何修改都升版本 + 写 CHANGELOG**。规范的每次改动必须升版本号并写 CHANGELOG——这是可追溯、可复现的根本保证。
2. **构建时不擅自改规范**。Agent 遇到规范没覆盖的情况，优先提**本层** change-request（`{layer}/change-requests/`），按当前规范的最佳理解继续，不擅自发挥。
3. **修复在维护方或构建环境均可进行**。change-request 由**该层维护方**处理，也可在构建环境直接修规范；无论哪种，处理后都升版本 + 写 CHANGELOG。

这三条保证所有构建基于同一基线，产出可复现、可追溯。

---

## 4 个组件

### 1. 版本号

- **包版本**存在 `VERSION` 文件，格式 `MAJOR.MINOR.PATCH`。
- **每个规范文件**（SOP / 字段定义 / 模板）在 frontmatter 带 `sop_version`，标明它属于哪个包版本。
- 语义：

| 段 | 含义 | 对已建资产 |
|---|---|---|
| **MAJOR** | 不兼容变更（删字段、重命名字段、流程改到产出物结构变了） | 需重建 |
| **MINOR** | 向后兼容的新增（加字段、加可选规则） | 不受影响 |
| **PATCH** | 措辞澄清、笔误，无行为变化 | 不受影响 |

### 2. CHANGELOG

- 位置：`CHANGELOG.md`
- **append-only**，最新在上
- 每条必填：版本 / 日期 / 变了什么 / 为什么 / 影响哪些文件 / 对已建资产的影响
- 格式见 CHANGELOG.md 顶部

### 3. 构建清单（build manifest）

每次构建，Agent **必须**产出一份 manifest，和构建产物一起落盘。manifest 至少记录：

- 用的包版本（读 `VERSION`）
- 构建时间
- 构建范围（对象类型 + 数量 + nf/version 等范围标识）
- 用到的各 SOP 的 `sop_version`

**作用**：任何一份资产都能追溯到"它是按哪个版本的规范建出来的"。

> manifest 的具体格式，在各对象类型的 SOP 里定义（待建）。本机制只强制"必须有、必须含上述字段"。

### 4. 变更请求回路（change-request loop）

构建中发现规范有缺口或错误时：

```
构建时：Agent 遇到缺口 → 写 change-request（场景 + 临时处理 + 建议补丁）
   ↓
SOP 维护方（或在构建环境直接）：审视 → 改 SOP → 升版本 → 写 CHANGELOG
   ↓
后续构建基于新版本
```

这是"构建发现 → 修规范 → 升版本"的演进闭环。change-request **每层一份**（`{layer}/change-requests/`），格式见 [command/change-requests/README.md](command/change-requests/README.md)。

## 升版本时的旧资产处理

**MAJOR 变更时**，CHANGELOG 必须明确声明旧资产怎么办：

| 处理 | 含义 |
|---|---|
| **保留** | 旧资产不动，新构建按新规范 |
| **迁移** | 写迁移步骤，Agent 按步迁移旧资产 |
| **重建** | 旧资产作废，按新规范重建 |

Agent 不自行判断、不自动迁移，**严格按 CHANGELOG 声明执行**。
