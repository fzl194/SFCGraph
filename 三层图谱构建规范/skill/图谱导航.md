# 图谱导航

> 三层图谱的层级架构与检索指南。教 Agent **"图谱长什么样、每层是什么、从一层怎么找到下一层"**。
> 配合 `图谱接口.md` 使用：本文讲**调什么、为什么这么调**（语义），`图谱接口.md` 讲**怎么调**（机制）。

---

## 1. 层级总览

四层自上而下，检索方向**由根到叶**：

```
业务层（定位树）   BusinessDomain → NetworkScenario → ConfigurationSolution
                     业务域            场景              方案 ★核心节点
                         │ 方案下挂三类 task
                         ▼
任务层（编排 DAG）  FeatureTask（特性级）──┐
                    CompoundTask（步骤级）┼─→ 编排配置生成
                    AtomTask（命令级）  ──┘
                         │ 按需追溯
                         ▼
特性层（按需）      Feature（产品文档原文） / License（前置门控）
命令层（按需）      MMLCommand（命令+参数） / ConfigObject（配置对象）
```

- **业务层** = 必经的定位树（域→场景→方案）
- **任务层** = 配置生成的编排核心（方案落地为 task）
- **特性层 / 命令层** = 按需追溯的静态知识源（原理/参数/对象）

---

## 2. 业务层：定位树（非互斥）

**结构**：`BusinessDomain → NetworkScenario → ConfigurationSolution`，三步锁定方案。

- **入口**：`GET /domains` 一次拿全部业务域 md。
- **下钻**：读域 md → 提 `[[NetworkScenario@*]]` → 读场景 md（含**决策点路由表**：业务诉求 → 推荐方案）→ 提 `[[ConfigurationSolution@*]]` → 读方案 md。

**非互斥（多归属）**：图谱是 **DAG 不是严格树**——一个下层对象可被多个上层引用。例如 `CompoundTask@charging-core-trio`（计费三件套）被在线/离线/融合计费三个方案复用；同一特性可能跨场景。**检索时沿引用走即可，不要假设路径唯一**。

---

## 3. 方案 = 核心节点

**方案（ConfigurationSolution）是配置生成的核心**——它声明"这个业务诉求由哪些 task 编排完成"。方案 md 同时关联任务层**三类对象**：

| 任务类型 | 角色 | 方案里的典型形态 |
|---|---|---|
| `FeatureTask` | 特性级编排（一个特性怎么配） | "本方案编排 3 个 UDG 特性" |
| `CompoundTask` | 步骤级复用模块（多命令经典组合） | "计费三件套""过滤链""规则绑定" |
| `AtomTask` | 命令级原子（单条命令怎么配） | 方案特化点提到的单命令 |

> 锁定方案后，沿方案 md 引用**按用户意图动态取相关 task**（不是全取）——仅配置相关的需完备，可选 / 无关的跳过。见 §6.4。

方案 md 还含：**决策点答案**、**特性关系矩阵**（哪些必配/可选/包含/正交/二选一）、**跨网元协同**（UDG/UNC 各侧分工）、**变种/排除项**（本方案对各 task 标准配置方法的特化）。

---

## 4. 任务层：编排 DAG

三类 task 互相引用，形成编排图（非层级必经，沿引用走）：

### 4.1 FeatureTask（特性级）
- **下挂**：`CompoundTask`（步骤）+ `AtomTask`（命令）——讲"这个特性分几步、每步哪些命令"。
- **追溯**：`Feature`（特性层）——需要特性产品文档原文（原理/流程/约束）时取。
- 正文含**配置流程（步骤编排）**，是 Phase 5 生成该特性配置的直接依据。

### 4.2 CompoundTask（步骤级，可复用模块）
- **下挂**：`AtomTask`（命令）+ 可嵌套 `CompoundTask`（步骤组合步骤）。
- 高频多命令组合抽成的经典模块，**跨方案/跨特性复用**（这是它存在的意义）。

### 4.3 AtomTask（命令级）
- **追溯**：`MMLCommand`（命令层）——命令的静态原文/参数。
- 讲"这条命令在这个 task 里**怎么配**"（动态配置方法）。AtomTask 之间也会互引（配套命令的关联）。

> **若某命令没有对应 AtomTask**（任务层未建），直接追溯命令层 `MMLCommand` 对象取参数与原文。

---

## 5. 按需追溯层（特性层 / 命令层）

非必经，**仅在需要静态知识时下钻**：

### 特性层
- `Feature`：特性的**产品文档原文**（原理 / 流程 / 与其他特性交互 / 配置约束）。Feature 可能含**多个子文档**（按主题，如"在线计费流程""Gy 接口配置"），按需取相关子文档即可。
- `License`：特性前置门控（License 控制项），方案/特性提及 License 前置时追溯。

### 命令层
- `MMLCommand`：命令原文 + **参数定义**（CommandParameter，含枚举值/类型/约束）。**参数枚举必须从此取，禁凭记忆**。
- `ConfigObject`：配置对象（如 URR/URRGROUP/RULE），被多条命令操作（ADD/MOD/RMV/LST）。差异分析时用于归类现网对象。

**追溯时机速查**：

| 需求 | 追溯到 |
|---|---|
| 写配置需知命令参数细节/枚举 | `MMLCommand` |
| 理解特性原理/约束（非配置方法） | `Feature` |
| 现网对象归类 / 差异分析 | `ConfigObject` |
| License 前置检查 | `License` |
| 某命令无 AtomTask | 直接 `MMLCommand` |

---

## 6. 检索纪律（★最重要）

### 6.1 边段不完备——必须扫全文
对象 md 末尾的 `## 边` 段**不包含全部引用**。很多 `[[ID]]` 只在正文出现。**实测**（每个对象"仅正文有、边段没有"的引用数）：

| 对象 | 仅正文有的引用数 |
|---|---|
| 方案 ConfigurationSolution | 3 |
| FeatureTask | 2 |
| CompoundTask | 3 |
| AtomTask | 2 |

**→ 提取引用必须扫整个 md（正文 + 边段），只看 `## 边` 会漏。** 正则 `\[\[([^\]]+)\]\]` 对全文跑一遍。

### 6.2 沿引用下钻，不假设路径
图谱是 DAG（多归属、非互斥）。沿 `[[ID]]` 走，相关则取、无关跳过。不要假设"方案只能到 FeatureTask"——方案直接挂三类 task；FeatureTask 也可直接到 AtomTask；CompoundTask 可嵌套 CompoundTask。

### 6.3 产出物驱动深度
不必每次下钻到底。按当前 Phase 产出物需要决定深度：

| Phase | 下钻深度 |
|---|---|
| 0 | 业务层三步锁方案（不下任务层） |
| 1 | 方案 → 任务层三类全取 + 现网匹配；特性/命令层按差异分析需要 |
| 3 | 下钻 `MMLCommand` 取参数枚举 |
| 5 | 按 task 编排 + 命令 md 生成；特性层按需 |
| 任意 | 需要原理/约束时才下钻 `Feature` |

### 6.4 配置完备性（动态，非全量）
**只有配置相关的引用需要确保完备**（不漏，漏了配置出错）；非配置相关的（用户没要的可选特性、维度增强等）按意图跳过。完备范围随用户意图实时调整——方案引用 6 个特性 task、用户只做其中 2 个，就只取那 2 个及其配置链，**不强取全部**。

> 判断口径：该引用是否进入当前要生成的配置链？是 → 取且取全；否 → 跳过。任务层往特性 / 命令层同理，按需不全拉。
