# 图谱接口

> Agent 访问三层图谱的唯一通道。**两个常用端点（POST /domains 入口 + POST /md 主力），命令行 curl 直调**，不传 version（每个 id 各取最新现存版本）。
> BASE：`http://127.0.0.1:8000/api/v1`
>
> **鉴权（必带）**：所有请求须带 header `X-API-Key: $GRAPH_API_KEY`。KEY 由平台管理员在「用户」菜单创建 SKILL 账号时生成，配在 Agent 平台环境变量 `GRAPH_API_KEY`（**默认从环境变量取，无需用户提供**）。
>
> **使用者工号（必带）**：另带 header `X-User-Id: $_AGENT_USERNAME`（当前使用者的工号，配在环境变量 `_AGENT_USERNAME`，默认从环境变量取）。**仅用于平台追溯"谁在用"与取用统计**，与鉴权无关，不存敏感信息。
>
> 缺/错 KEY → 401；账号无 SKILL 权限 → 403。

---

## 端点

| 端点 | 用途 |
|---|---|
| `POST /domains` | **Phase 0 入口**：一次性返回全部业务域 md（业务域特殊化，免两步） |
| `POST /md` | 批量取对象 md（**单个也走批量，全程主力**） |
| `GET /objects?type={Type}` | 列其他类型对象（备用，日常下钻不必用） |

---

## 1. 业务域直取 POST /domains（Phase 0 入口）

业务域是用户业务归属顶层，**特殊化直取**——一次返回全部域 md，免去"先取 id 再 POST /md"两步：

```bash
curl -s -X POST "http://127.0.0.1:8000/api/v1/domains" -H "X-API-Key: $GRAPH_API_KEY" -H "X-User-Id: $_AGENT_USERNAME"
```

返回 `[{id, name, md}, ...]`，每个 md 含 frontmatter + 正文 + `## 边`（下游指向 NetworkScenario）。Agent 拿到后直接读 md 提 `[[NetworkScenario@*]]` 下钻。

> 仅业务域有此特化。其他所有层级（场景/方案/task/特性/命令）都用 POST /md 沿引用取。

---

## 2. 批量取 md POST /md（主力）

**Windows cmd.exe**（echo 管道喂 stdin，避开 JSON body 引号地狱——cmd 不认 `\"` 转义，单行 `-d "{...}"` 的内层双引号会被 cmd 破坏）：

```cmd
echo {"ids":["NetworkScenario@charging","ConfigurationSolution@charging-online"]} | curl -s -X POST http://127.0.0.1:8000/api/v1/md -H "X-API-Key: %GRAPH_API_KEY%" -H "X-User-Id: %_AGENT_USERNAME%" -H "Content-Type: application/json" --data-binary @-
```

**Git Bash / Linux**（单引号直接传 body）：

```bash
curl -s -X POST http://127.0.0.1:8000/api/v1/md -H "X-API-Key: $GRAPH_API_KEY" -H "X-User-Id: $_AGENT_USERNAME" -H "Content-Type: application/json" -d '{"ids":["NetworkScenario@charging"]}'
```

返回 `{id: {version, md}}`，md 含 frontmatter + 正文 + `## 边` 段。某 id 不存在 → 该条 `{error, available_versions}`，**其余 id 照常返回**（部分失败容错）。

可选参数 `version`（全局版本）——**MVP 不传**，让每个 id 各取最新。

> cmd 要点：`echo` 是 cmd 内置命令，**保留双引号**原样输出 JSON；`--data-binary @-` 让 curl 从 stdin 读 body。两者一管道，一条命令搞定，无需临时文件、无需 PowerShell。

---

## 3. 下钻纪律：读 md → 提引用 → 循环

每个 md 的**正文**和 **`## 边` 段**里都散布 `[[ID]]` 引用，指向关联对象。关联不一定全在 `## 边` 段，很多在正文——**必须解析整个 md**。

Agent 下钻四步循环：
1. 取回一批 md（业务域用 POST /domains，其余用 POST /md）
2. 从每个 md 提取所有 `[[ID]]`（正则 `\[\[([^\]]+)\]\]`）
3. 判断哪些与当前产出物相关 → 相关的加入下一批 ids，无关的跳过（防雪崩）
4. POST /md 取下一批，循环直到当前 Phase 产出物所需知识齐备

典型下钻链：

```
POST /domains                       → 全部业务域 md，锁定域
读域 md → [[NetworkScenario@*]]     → 读场景 md（决策点路由表）→ [[ConfigurationSolution@*]]
读方案 md → [[*@FeatureTask@*]] / [[*@AtomTask@*]] / [[*@CompoundTask@*]]
读 task md → [[*@Feature@*]]（静态知识）/ [[*@MMLCommand@*]]（命令原文）
```

**完整性底线**：方案 md 的 `## 边` 段 + "配置与协同"正文会声明该方案编排的全部 task——这些必须取全，作为 Phase 1/5 的知识底盘。

---

## 4. ID 格式

- **业务层**（跨 NF，无版本）：`{Type}@{slug}`，2 段。如 `NetworkScenario@charging`。
- **NF 类**（带版本）：`{nf}@{Type}@{name}`，3 段。如 `UDG@FeatureTask@GWFD-020300`、`UDG@AtomTask@SET UPDEFAULTQUOTA`（含空格）。
- `@` 与空格在 JSON body 里无需转义。
