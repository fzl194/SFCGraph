# 配置生成 SOP

> **v2.0 重大变化**：图谱与 SKILL 彻底解耦。图谱是后端服务化资产，SKILL 只承载 SOP 流程，不再打包任何场景知识。
> ★**设计原则**：SKILL.md 只承载 SOP，不绑定任何业务域/场景/方案。所有场景知识从后端图谱按需取（§3）。
> ★**图谱加载**：Agent 沿对象 md 的 `[[ID]]` 引用逐层 POST 调后端，读一个 md → 提引用 → 取下一批，循环至产出物所需知识齐备。不一次性全量加载。

---

## 1. 定位

Agent 根据用户的需求描述和现网配置脚本，经过**业务域→场景→方案定位、需求理解、图谱匹配、参数收集、配置生成与全量核查**，输出可直接执行的 MML 配置脚本。

本 SOP 是**业务域无关**的通用配置生成工作流：
- 业务域/场景/方案由后端图谱动态定义，本 SOP 不写死任何域/场景
- 新增业务域/场景/方案 = 后端图谱加对象，**本 SOP 不改**

---

## 2. 知识源

### 2.1 通用知识（SKILL 自带，本地读）

| 知识源 | 路径 | 用途 |
|---|---|---|
| 图谱导航 | `图谱导航.md` | 三层图谱层级架构 + 每层语义 + 检索指南（**调什么、为什么**） |
| 图谱接口 | `图谱接口.md` | Agent 访问后端图谱的唯一通道（两端点 + 下钻纪律，**怎么调**） |
| AI MML 核查流程 | `common/AI-MML核查流程.md` | Phase 6 核查接口流程（coremaster configcheck） |

> 跨域通用配置约束（执行时序 / 命名一致 / License / 优先级 / 动网保守）**不再单独成文**：通用部分见 §14 执行约束；场景特定部分从图谱对象的 Rule 字段（方案/任务/命令 md 内的规则段）取。

### 2.2 三层图谱（后端资产，按需取，不打包）

后端服务化的核心资产，SKILL 不内置。Agent 按 §3 图谱交互范式沿引用逐层取：

| 层 | 对象类型 | 用途 |
|---|---|---|
| 业务层 | `BusinessDomain` / `NetworkScenario` / `ConfigurationSolution` | 域→场景→方案定位、决策点路由、业务规则 |
| 任务层 | `FeatureTask` / `CompoundTask` / `AtomTask` | 配置编排、步骤组合、命令绑定 |
| 特性层 | `Feature` / `License` | 特性静态知识、License 前置（按需） |
| 命令层 | `MMLCommand` / `ConfigObject` | 命令原文、参数定义、命令规则（按需） |

### 2.3 知识优先级

**场景规则（方案/任务 md 内的 Rule 段）> 图谱对象 Rule 字段 > §14 通用执行约束**

## 3. 图谱交互范式 ★新版基石

> 每个 Phase 取知识都遵循本节。不掌握本节无法跑通任何 Phase。
> **先读 `图谱导航.md`** 搞清四层（业务/任务/特性/命令）各是什么、层间怎么走、每层何时下钻，再回来看本节的调用机制。

### 3.1 两个端点（curl 直调，全程不传 version）

**Phase 0 入口**——取全部业务域 md（无 body，cmd / bash 通用）：

```
curl -s http://127.0.0.1:8000/api/v1/domains
```

**全程主力**——批量取对象 md（单个也走批量）：

```
# Windows cmd.exe（echo 管道喂 stdin —— cmd 不认 \" 转义，单行 -d "{...}" 内层双引号会被破坏，必须管道）
echo {"ids":["NetworkScenario@charging","ConfigurationSolution@charging-online"]} | curl -s -X POST http://127.0.0.1:8000/api/v1/md -H "Content-Type: application/json" --data-binary @-

# Git Bash / Linux（单引号直接传 body）
curl -s -X POST http://127.0.0.1:8000/api/v1/md -H "Content-Type: application/json" -d '{"ids":["NetworkScenario@charging"]}'
```

- 业务域用 `GET /domains` 直取（返回 `[{id,name,md}]`，含正文+边段）；其余所有对象用 `POST /md` 沿引用取。
- 不传 `version` → 每个 id 各取**最新现存版本**，Agent 不操心版本。
- 部分失败容错：某 id 不存在只该条计错（`{error, available_versions}`），其余照常返回。
- 端点细节见 `图谱接口.md`。

### 3.2 下钻四步循环

```
1. POST /md 取回一批 md
2. 用正则 \[\[([^\]]+)\]\] 提取每个 md 正文 + ## 边段的全部 [[ID]] 引用
3. **按当前用户意图动态判断**每个引用的相关性：相关 → 加入下一批 ids；无关 → 跳过（防雪崩，非配置相关不强取）
4. POST /md 取下一批，循环至当前 Phase 产出物所需知识齐备
```

> **★边段不完备——必须扫全文**。实测：方案 md 有 3 条引用仅正文有（边段没有）、FeatureTask 2 条、CompoundTask 3 条、AtomTask 2 条。`## 边` 段只是部分引用，很多 `[[ID]]` 只在正文（如方案 md "配置与协同"段提各 task 的变种）。**提取引用必须扫整个 md（正文 + 边段），只看 `## 边` 会漏**。详见 `图谱导航.md` §6.1。

### 3.3 检索三纪律（动态，非全量）

1. **读必提**：读完每个 md 提取其全部 `[[ID]]`（正文 + 边段都扫，防漏引用）。
2. **提必判**（按用户意图）：每个引用按**当前用户意图**判断相关性——相关才取，无关跳过。**不是方案声明多少就取多少**。
3. **配置完备**：**只有配置相关的引用**需确保完备（不漏，漏了配置出错）；非配置相关的（用户没要的可选特性 / 维度增强等）按意图取舍。完备范围随用户意图实时调整，**不写死、非全量索引**。

### 3.4 典型下钻链

```
GET /domains                              → 全部业务域 md，锁定域
读域 md            → [[NetworkScenario@*]]          → 锁定场景
POST /md {场景id}   读场景 md → 决策点路由表 + [[ConfigurationSolution@*]] → 锁定方案
POST /md {方案id}   读方案 md → [[*@FeatureTask@*]]/[[@AtomTask@*]]/[[@CompoundTask@*]] → 任务层全貌
POST /md {task ids} 读任务 md → [[*@Feature@*]]/[[@MMLCommand@*]] → 特性/命令层（按需）
```

ID 格式：业务层 2 段 `{Type}@{slug}`（如 `NetworkScenario@charging`）；NF 类 3 段 `{nf}@{Type}@{name}`（如 `UDG@AtomTask@SET UPDEFAULTQUOTA`，含空格）。

## 4. 整体流程

```
Phase 0: 业务域 → 场景 → 方案（沿图谱下钻锁定方案）
  ↓
Phase 1: 任务层下钻 + 现网匹配（沿方案 md 取 task + grep 现网 + 差异分析）
  ↓
Phase 2: 方案确认【GATE-1 STOP】→ 产出 xxx方案.md
  ↓
Phase 3: 参数收集 → 产出 xxxLLD.md（下钻命令层取 CommandParameter）
  ↓
Phase 4: 规划数据确认【GATE-2 STOP】
  ↓
Phase 5: 配置生成（按 task 编排 + 图谱 Rule 内置约束）→ 每网元 MML 文件
  ↓
Phase 6: AI MML 核查（调 coremaster configcheck）
  ↓
Phase 6.5: 核查确认【GATE-3 STOP】
  ↓
Phase 7: 输出交付
```

### 4.1 用户交互规则

专业平实、表格展示、追问说明原因与影响；3 个 GATE 必须 STOP 等用户确认。用户中途可"暂停/重新开始/换业务域/换场景/到此为止"。

### 4.2 强制执行契约

以下规则高于所有 Phase 细节，任何一条未满足都视为**不得进入下一 Phase**：

1. **先取齐图谱知识，再执行业务逻辑**：每个 Phase 所需的图谱对象（按 §3 下钻纪律取）未取齐前，不得开展分析、参数推断、配置生成、核查。
2. **先生成交付文档，再过 Gate**：Phase 2 必须先产出 `xxx方案.md`；Phase 3 必须先产出 `xxxLLD.md`；未产出 = 该 Phase 未完成 = 禁止进入下一 Phase。
3. **先展示文档，再等待用户确认**：GATE-1/2/3 必须把对应文档或结果展示给用户后立即 STOP。
4. **禁止隐式继续执行**：发出 Gate 提示后，不得在同一轮消息继续做下一 Phase 的任何动作。
5. **禁止响应"直接出配置/跳过确认"类诉求**：即使用户催促，必须明确说明当前卡在哪个 Gate、缺哪份文档。
6. **文档未落地时不得声称已完成**：不得只在对话中摘要展示而不生成 `xxx方案.md` / `xxxLLD.md` 文件。
7. **LLD 必须可补数**：`xxxLLD.md` 必须按参数粒度区分固定参数和变化参数。
8. **方案确认与 LLD 产出必须分轮次**：产出 `xxx方案.md` 并进入 GATE-1 后该轮必须结束；只有用户在**下一条消息**明确确认后才能开始 Phase 3。

### 4.3 阶段进入条件总表

| 阶段 | 进入前必须满足 | 本阶段完成标志 | 未满足时动作 |
|---|---|---|---|
| Phase 0 | 后端图谱服务可访问 | 锁定 {域}/{场景}/{方案} | STOP，要求补需求关键词 |
| Phase 1 | Phase 0 已锁方案，已沿方案 md 取齐任务层 | 输出匹配结果摘要 | STOP，要求补现网脚本/决策点 |
| Phase 2 | Phase 1 已完成 | **已生成 `xxx方案.md`** 并展示 | STOP，禁止进入 Phase 3 |
| Phase 3 | **用户已确认 GATE-1** | **已生成 `xxxLLD.md`** | STOP，禁止进入 Phase 4/5 |
| Phase 4 | Phase 3 文档已生成且参数已校验 | 用户已确认 GATE-2 | STOP，禁止进入 Phase 5 |
| Phase 5 | **`xxx方案.md` + `xxxLLD.md` 均已存在且已确认** | 生成每网元 MML 文件 | STOP，禁止直接交付 |
| Phase 6.5 | Phase 6 核查已完成 | 用户已确认 GATE-3 | STOP，禁止进入 Phase 7 |

## 5. Phase 0: 业务域 → 场景 → 方案

> 沿图谱业务层下钻三步，锁定方案。全程 GET 一次 + 多轮 POST /md。

### 5.1 步骤

1. **取全部业务域 md**：
   ```bash
   curl -s "$BASE/domains"
   ```
   一次拿到全部业务域 md（含正文 + `## 边`）。按用户需求关键词匹配业务域。多域命中按主诉求判断；不确定向用户确认；未命中 → 提示补充属于哪个业务域。锁定 `{业务域}`。

2. **读域 md → 取场景清单**：
   从上一步拿到的域 md 提 `[[NetworkScenario@*]]`（域 md 已在手，不必再 POST）。按子场景识别关键词匹配锁定 `{场景}`。

3. **读场景 md → 决策点路由表 + 方案候选**：
   ```bash
   curl -s -X POST "$BASE/md" -d '{"ids":["<场景id>"]}'
   ```
   场景 md 含"决策点"路由表（业务诉求 → 推荐方案）+ `[[ConfigurationSolution@*]]` 下游方案。

4. **读方案 md → 锁定方案**：
   按决策点表 + 用户业务诉求匹配方案，POST /md 取候选方案 md，读概览确认。锁定 `{方案}`。

### 5.2 输出

```
已识别：业务域={域名}，场景={场景名}，方案={方案名}
将进入需求分析与现网匹配（Phase 1）。请提供现网配置脚本。
```

## 6. Phase 1: 任务层下钻 + 现网匹配

⚠ 完成后 STOP 等 Phase 2 确认。禁止在本阶段生成任何 MML 命令。

### 6.0 取相关任务层（按用户意图）

沿方案 md 的正文 + 边段引用，**按当前用户意图筛选相关 task**，POST /md 批量取：
- `FeatureTask`（特性级编排）
- `CompoundTask`（步骤级复用模块）
- `AtomTask`（命令级原子）

> 不是方案声明多少就取多少——按用户意图动态定范围（§3.3 纪律 3）。**仅配置相关的需完备**（漏了配置出错），可选 / 无关的跳过。再沿 task md 引用按需下钻特性层 / 命令层（现网解析与差异分析所需）。

### 6.1 现网脚本前置检查

- **【GATE-A】现网脚本路径**：未提供 → STOP 要求提供相关网元侧配置导出文件。
- **【GATE-B】脚本存在性**：用 **Glob 或 Bash `ls`** 验证每个文件存在且可读，**严禁对现网脚本用 Read 工具**（数万至数十万行，Read 必溢出），否则 STOP。

### 6.2 现网解析

**必须用 Bash 工具调 `grep`** 按命令类型分批搜索现网脚本：
- 每次只搜一个命令类型，严禁合并。
- 命令清单来自 Phase 1 取齐的 task md（AtomTask 的命令名）。
- 示例：`grep -E '^(ADD|SET|MOD|RMV) +URR' 现网脚本.txt`
- 分网元侧提取对象。

> ⚠ 用 Bash grep，不用 Grep 工具：现网脚本是用户提供的大配置文件，Grep 工具基于 ripgrep 为工作区代码设计，不适合现网大配置文件。

### 6.3 差异分析

对每个 ConfigObject 判断 复用 / 修改 / 新建 / 删除。

### 6.4 输出

```markdown
## 匹配结果
**业务域/场景/方案**: {域名}/{场景名}/{方案名}
**决策点**: {该方案 DP 答案}
**现网分析**: {现有对象清单 + 可复用对象}
**任务编排**: {按方案 md + task md 的 FeatureTask/CompoundTask/AtomTask 编排}
```

## 7. Phase 2: 方案确认【GATE-1】

**【GATE-1】方案确认审核点。**

展示 Phase 1 匹配结果前，**必须先产出 `xxx方案.md`**，至少含：

```markdown
# {项目/需求名}方案
## 1. 需求摘要
## 2. 识别结果（业务域/场景/方案）
## 3. 决策点及答案
## 4. 现网分析（对象清单、复用/修改/新建/删除）
## 5. 任务编排（引用 FeatureTask/CompoundTask/AtomTask ID）
## 6. 风险与待确认项
```

**只有在 `xxx方案.md` 已生成后**才进入 GATE-1 展示。展示后 STOP：

```
请确认以上匹配结果（已生成 xxx方案.md）。确认后进入参数收集。
```

- 用户确认 → Phase 3；要求修正 → 回 Phase 1；换方案 → 读其他 ConfigurationSolution md。
- **确认前不得执行 Phase 3，不得生成任何配置命令。**
- **本轮展示方案.md + 发出 GATE-1 后必须结束，不得同轮产出 LLD。**

---

## 8. Phase 3: 参数收集 → LLD

⚠ 必须 GATE-1 通过。完成后 STOP 等 Phase 4。禁止生成 MML。

### 8.1 步骤

1. **下钻命令层取参数**：按 LLD 所需，POST /md 取相关 `MMLCommand` md。**枚举值必须从命令 md 的 CommandParameter 取，禁凭记忆**。
2. **参数分两类**：
   - **固定参数**：不随本次规划变化，从现网脚本、图谱规则、命令默认值、命名规律回填。
   - **变化参数**：随用户意图/规划数据变化，或需用户指定/确认。
3. **产出 `xxxLLD.md`**（Phase 3 唯一主交付物，LLD = markdown 形式的规划数据表）：

```markdown
# {项目/需求名}LLD
## 1. 方案与范围
## 2. 现网依赖与命名继承
## 3. 固定参数
## 4. 变化参数（待用户补充/确认）
## 5. 优先级分析
## 6. 多网元一致性约束
## 7. 待用户确认项
```

固定/变化参数用 markdown 表格承载：

```markdown
## LLD规划数据表 — {方案名}
### 网元: {网元类型1}
| 命令 | 参数 | 参数类型 | 值 | 来源 | 状态 | 说明 |
### 网元: {网元类型2}
| ... |
### 全局参数（多网元共用，必须一致）
| 参数 | 参数类型 | 网元1侧 | 网元2侧 | 来源 | 状态 | 说明 |
```

4. **固定参数**：优先从现网/图谱/默认值回填；无法可靠确定则降级为变化参数，禁强猜。
5. **变化参数**：明确标出对应的用户意图或规划来源，让用户逐项补充。
6. **优先级分析**（涉及有序规则时必须执行）：从 task/命令 md 的 Rule 段确认优先级机制；用 Bash grep 提取现网优先级值排序；**用户确认后才能生成带优先级的命令**。优先级数值语义（越小越高/越大越高）由图谱规则定义，禁凭记忆。

### 8.2 输出

展示 `xxxLLD.md` 的固定参数、变化参数、优先级分析，标注"待提供/待确认"。

---

## 9. Phase 4: 规划数据确认【GATE-2】

**【GATE-2】规划数据确认审核点。**

用户填写/修正后做合理性校验，校验通过展示完整 `xxxLLD.md`。展示后 STOP：

```
参数校验已通过。请检查 xxxLLD.md，确认无误后回复"确认"。
```

- 确认 → Phase 5；要求修改 → 修正后重新校验。
- **确认前不得执行 Phase 5，不得生成 MML。**

---

## 10. Phase 5: 配置生成

⚠ 必须 GATE-2 通过，且 `xxx方案.md` 与 `xxxLLD.md` 均已确认。无例外。

### 10.1 步骤

1. **按 LLD 生成命令**：每网元一个 MML 文件，参数值只能从已确认的 `xxxLLD.md` 取。
2. **按 task 编排排列**：顺序由任务层定义——`FeatureTask` 的配置流程 + `CompoundTask` 的步骤组合 + `AtomTask` 的命令。通用原则：底层对象先于上层、被引用对象先于引用者、绑定/刷新等收尾动作最后；**具体链路由任务层图谱决定，不由本 SOP 写死**。
3. **内置图谱 Rule 约束**（生成时即遵守）：方案 md 的 BusinessRule + 命令 md 的 CommandRule + 任务 md 的 TaskRule + §14 通用执行约束。

**硬性禁止**：
- 禁止在缺少 `xxx方案.md` 或 `xxxLLD.md` 时直接生成 MML。
- 禁止用"聊天中已确认"替代文档存在性检查。
- 禁止跳过 GATE-2。

### 10.2 输出

每网元 MML 文件 + 询问是否核查：
```
配置文件已生成。是否调用 AI MML 核查能力验证脚本正确性？
```

## 11. Phase 6: AI MML 核查

按 `common/AI-MML核查流程.md` 执行（信息确认 → Token → 构建 scriptInfo → 调 general-check API → 解析结果修正）。

- **核查直接调 coremaster configcheck 接口**；图谱 Rule（BR/CR/TR）已在 Phase 5 生成时作为内置约束遵守，核查阶段不再做静态降级。
- 自动修正可确定的问题并告知；无法自动修正 → 暂停要求用户决策；修正后重新核查直到通过。

输出核查报告（核查方式 / 结果 / 各维度检查项 / 修正日志）。

## 12. Phase 6.5: 核查确认【GATE-3】

**【GATE-3】配置确认审核点。**

核查通过后将核查报告 + 配置脚本一并向用户展示。展示后 STOP：

```
以上配置脚本已通过核查。请确认无误后回复"确认"，我将输出最终交付物。
```

- 确认 → Phase 7；要求修改 → 回对应 Phase 修正后重新核查。
- **确认前不得输出最终交付物。**

## 14. Agent 执行约束

1. **禁止跳过任何 Phase**：Phase 0-7 严格顺序，禁止跳过 GATE-1/2/3。
2. **GATE 必须 STOP**：Phase 2/4/6.5 展示后立即停止，等用户下一条消息。
3. **方案/LLD 为强制交付物**：Phase 2 必须生成 `xxx方案.md`；Phase 3 必须生成 `xxxLLD.md`。缺一不得进入 Phase 5。
4. **图谱取齐先于业务逻辑**：每 Phase 所需图谱对象（按 §3 下钻纪律）未取齐前不得开展业务逻辑。**禁止凭记忆填写业务事实**——必须从图谱 md 取。
5. **检索三纪律**（§3.3）：读必提、提必判（按用户意图动态判断）、配置完备（仅配置相关引用需完备，范围随意图动态调整，非全量索引）。
6. **现网脚本为必须**：未验证现网脚本存在前不得进入 Phase 1 业务逻辑。
7. **现网脚本三约束**：① **先图谱后现网**——Phase 1 必须先取齐任务层再处理现网脚本；② **严禁 Read，用 Bash grep**——现网脚本数万至数十万行，必须用 Bash grep 分批搜（不用 Grep 工具、不用 Read），存在性验证只用 Glob/ls；③ **分批**——每次 grep 只查一个命令类型。
8. **严格依赖顺序**：按任务层图谱（FeatureTask/CompoundTask/AtomTask）的编排排列命令。
9. **追问而非猜测**：缺失决策信息或必填参数必须追问。
10. **动网保守**：现网对象修改必须有用户确认。
11. **命名遵循现网**：新对象命名沿用现网规律。
12. **跨网元一致**：多网元共用参数必须一致。
13. **优先级必须用户确认**：涉及有序规则时，提取现网优先级值分析后 STOP 确认；语义由图谱规则定义，禁凭记忆。
14. **参数枚举禁凭记忆**：必须从命令 md 的 CommandParameter 获取。
15. **通用配置约束从图谱取**：执行时序、License 前置、对象依赖等约束从相关 md 的 Rule 段取，不依赖独立通用规则文档。
16. **禁止越级响应**：用户要求"先出配置再补文档""不用方案直接生成"必须拒绝，返回所缺 Gate/文档。
17. **LLD 必须区分固定/变化参数**：固定尽量自动回填，变化映射到用户意图留给用户补确认。
18. **方案与 LLD 禁止同轮输出**：同一条消息最多完成一个 Gate 交付物。

## 15. 扩展与演进（本 SOP 永不改）

- **新增业务域/场景/方案**：后端图谱加对象（BusinessDomain/NetworkScenario/ConfigurationSolution + 关联 task/feature/command），**本 SOP 不改**。
- **图谱加载方式演进**：若后端新增端点（如 MCP 化），只改 `图谱接口.md`，本 SOP 不改。
- **核查能力**：Phase 6 流程见 `common/AI-MML核查流程.md`；reference 资源由部署环境挂载。
