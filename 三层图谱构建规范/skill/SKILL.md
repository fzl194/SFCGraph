# 配置生成 SOP

> **v3.0 变化**（针对 v2.1 执行审计暴露的6大问题，强化图谱驱动和搜索纪律）：
> - ① §3.2 新增"下钻完备性声明"——每轮图谱下钻后必须输出已取/提取/跳过的 ID 清单，禁止不声明就继续
> - ② §4.2 新增3条契约（§4.2-9/10/11）：图谱调用先于现网搜索、图谱调用必须记录日志、现网搜索必须记录日志、搜索必须按计划执行
> - ③ §5.2 新增"Phase 0 完备性检查清单"——4步每步必须声明结果，未声明禁止进入 Phase 1
> - ④ §6.0 新增"图谱知识完备性检查"——进入现网搜索前必须声明图谱已取齐，否则继续下钻
> - ⑤ §6.1a 从"搜索计划"升级为"强制搜索计划"——未输出计划就发起调用=违反契约，计划外搜索必须说明理由
> - ⑥ §6.2a 新增"搜索日志机制"——每次 mml_extract.py 调用必须记录日志+累计已获取对象清单，发起前查清单去重；§6.2 trace 结果必须输出"trace 消费确认"

★**设计原则**：SKILL.md 只承载 SOP，不绑定任何业务域/场景/方案。所有场景知识从后端图谱按需取（§3）。

## 1. 定位

Agent 根据用户的需求描述和现网配置脚本，经过**业务域→场景→方案定位、需求理解、图谱匹配、参数收集、配置生成与全量核查**，输出可直接执行的 MML 配置脚本。

本 SOP 是**业务域无关**的通用配置生成工作流：
- 业务域/场景/方案由后端图谱动态定义，本 SOP 不写死任何域/场景
- 新增业务域/场景/方案 = 后端图谱加对象，**本 SOP 不改**

## 2. 知识源

### 2.1 通用知识（SKILL 自带，本地读）

| 知识源 | 路径 | 用途 |
|---|---|---|
| 图谱导航 | `图谱导航.md` | 三层图谱层级架构 + 每层语义 + 检索指南（**调什么、为什么**） |
| 图谱接口 | `图谱接口.md` | Agent 访问后端图谱的唯一通道（两端点 + 下钻纪律，**怎么调**） |
| MML 提取工具 | `scripts/mml_extract.py` | 现网/动网 MML 配置文件结构化提取（命令类型筛选 + 参数正则筛选 + 引用追踪 + 递归追踪 + 统计），替代逐条 grep/Select-String |

> Phase 6 AI MML 核查由独立 SKILL `aimml-check` 承载，本 SKILL 不再内置核查流程文档和 reference 资源。详见 §11。

### 2.2 三层图谱（后端资产，按需取，不打包）

后端服务化的核心资产，SKILL 不内置。Agent 按 §3 图谱交互范式沿引用逐层取：

| 层 | 对象类型 | 用途 |
|---|---|---|
| 业务层 | `BusinessDomain` / `NetworkScenario` / `ConfigurationSolution` | 域→场景→方案定位、决策点路由、业务规则 |
| 任务层 | `FeatureTask` / `CompoundTask` / `AtomTask` | 配置编排、步骤组合、命令绑定 |
| 特性层 | `Feature` / `License` | 特性静态知识、License 前置（按需） |
| 命令层 | `MMLCommand` / `ConfigObject` | 命令原文、参数定义、命令规则（按需） |

### 2.3 知识优先级

**场景规则（方案/任务 md 内的 Rule 段）> 图谱对象 Rule 字段 > §14 通用执行约束**

## 3. 图谱交互范式 ★新版基石

> 每个 Phase 取知识都遵循本节。不掌握本节无法跑通任何 Phase。
> **先读 `图谱导航.md`** 搞清四层（业务/任务/特性/命令）各是什么、层间怎么走、每层何时下钻，再回来看本节的调用机制。

### 3.1 两个端点（curl 直调，全程不传 version）

**Phase 0 入口**——取全部业务域 md（无 body，cmd / bash 通用）：

```
curl -s -X POST http://127.0.0.1:8000/api/v1/domains -H "X-API-Key: $GRAPH_API_KEY" -H "X-User-Id: $_AGENT_USERNAME"
```

**全程主力**——批量取对象 md（单个也走批量）：

```
# Windows cmd.exe
echo {"ids":["NetworkScenario@charging","ConfigurationSolution@charging-online"]} | curl -s -X POST http://127.0.0.1:8000/api/v1/md -H "X-API-Key: %GRAPH_API_KEY%" -H "X-User-Id: %_AGENT_USERNAME%" -H "Content-Type: application/json" --data-binary @-

# Git Bash / Linux
curl -s -X POST http://127.0.0.1:8000/api/v1/md -H "X-API-Key: $GRAPH_API_KEY" -H "X-User-Id: $_AGENT_USERNAME" -H "Content-Type: application/json" -d '{"ids":["NetworkScenario@charging"]}'
```

### 3.2 下钻四步循环 ★图谱调用核心纪律

```
1. POST /md 取回一批 md
2. 用正则 \[\[([^\]]+)\]\] 提取每个 md 正文 + ## 边段的全部 [[ID]] 引用
3. **按当前用户意图动态判断**每个引用的相关性：相关 → 加入下一批 ids；无关 → 跳过
4. POST /md 取下一批，循环至当前 Phase 产出物所需知识齐备
```

> **★边段不完备——必须扫全文**。实测：方案 md 有 3 条引用仅正文有（边段没有）、FeatureTask 2 条、CompoundTask 3 条、AtomTask 2 条。**提取引用必须扫整个 md（正文 + 边段），只看 `## 边` 会漏**。详见 `图谱导航.md` §6.1。

**★下钻完备性声明（每轮下钻后必须输出）**：

每轮四步循环完成后，Agent **必须在对话中输出**下钻完备性声明，格式如下：

```
【下钻声明】本轮取: [已POST的ID列表] | 提取引用: [从md正文中提取的全部[[ID]]] | 相关→下轮取: [判断为相关的ID] | 跳过: [判断为不相关的ID及理由]
```

**禁止**：不输出声明就进入下一轮下钻或跳到现网搜索。声明的目的：
1. 迫使 Agent 在每次下钻后**主动检查**是否遗漏引用
2. 让"图谱知识是否取齐"有**可审计的轨迹**
3. 防止跳过图谱直接进入现网 grep

### 3.3 检索三纪律（动态，非全量）

1. **读必提**：读完每个 md 提取其全部 `[[ID]]`（正文 + 边段都扫，防漏引用）。
2. **提必判**（按用户意图）：每个引用按**当前用户意图**判断相关性——相关才取，无关跳过。
3. **配置完备**：**只有配置相关的引用**需确保完备（不漏，漏了配置出错）；非配置相关的按意图取舍。完备范围随用户意图实时调整，**不写死、非全量索引**。

### 3.4 典型下钻链

```
POST /domains → 全部业务域 md → 锁定域
读域 md → [[NetworkScenario@*]] → 锁定场景
POST /md {场景id} → 决策点路由表 + [[ConfigurationSolution@*]] → 锁定方案
POST /md {方案id} → [[*@FeatureTask@*]]/[[@AtomTask@*]]/[[@CompoundTask@*]] → 任务层
POST /md {task ids} → [[*@Feature@*]]/[[@MMLCommand@*]] → 特性/命令层（按需）
```

ID 格式：业务层 2 段 `{Type}@{slug}`（如 `NetworkScenario@charging`）；NF 类 3 段 `{nf}@{Type}@{name}`（如 `UDG@AtomTask@SET UPDEFAULTQUOTA`，含空格）。

## 4. 整体流程

```
Phase 0: 业务域 → 场景 → 方案
  ↓
Phase 1: 任务层下钻 + 现网匹配
  ↓
Phase 2: 方案确认【GATE-1 STOP】→ 产出 xxx方案.md
  ↓
Phase 3: 参数收集 → 产出 xxxLLD.md
  ↓
Phase 4: 规划数据确认【GATE-2 STOP】
  ↓
Phase 5: 配置生成 → 每网元 .txt 配置文件
  ↓
Phase 6: AI MML 核查
  ↓
Phase 6.5: 核查确认【GATE-3 STOP】
  ↓
Phase 7: 输出交付
```

### 4.1 用户交互规则

专业平实、表格展示、追问说明原因与影响；3 个 GATE 必须 STOP 等用户确认。用户中途可"暂停/重新开始/换业务域/换场景/到此为止"。

**★决策点展示纪律**：GATE 展示时**必须逐条列给用户**（每条说明：是什么、为什么需要用户定、有哪些选项、不定的后果），**严禁只笼统问"是否继续"**。

### 4.2 强制执行契约

以下规则高于所有 Phase 细节，任何一条未满足都视为**不得进入下一 Phase**：

1. **先取齐图谱知识，再执行业务逻辑**：每个 Phase 所需的图谱对象未取齐前，不得开展分析、参数推断、配置生成、核查。**图谱 POST 调用必须先于现网 mml_extract.py 调用**——Phase 0/1 的图谱下钻未完成前，禁止发起任何现网搜索。
2. **先生成交付文档，再过 Gate**：Phase 2 必须先产出 `xxx方案.md`；Phase 3 必须先产出 `xxxLLD.md`；未产出 = 禁止进入下一 Phase。
3. **先展示文档（含决策点），再等待用户确认**：GATE 必须把决策点 / 待确认项逐条列出让用户逐项决策，然后立即 STOP。
4. **禁止隐式继续执行**：发出 Gate 提示后，不得在同一轮消息继续做下一 Phase。
5. **禁止响应"直接出配置/跳过确认"类诉求**：必须明确说明当前卡在哪个 Gate、缺哪份文档。
6. **文档未落地时不得声称已完成**：不得只在对话中摘要展示而不生成文件。
7. **LLD 必须可补数**：必须按参数粒度区分固定参数和变化参数。
8. **方案确认与 LLD 产出必须分轮次**：GATE-1 后该轮必须结束，用户下一条消息确认后才能开始 Phase 3。
9. **图谱调用必须记录日志**：每次 POST /domains 或 POST /md 调用后，必须输出【图谱调用日志】（见 §3.2 下钻完备性声明）。零日志 = 图谱知识未取齐 = 禁止进入下一阶段。
10. **现网搜索必须记录日志**：每次 mml_extract.py 调用后，必须输出【现网搜索日志】（见 §6.2a 搜索日志机制）。无日志 = 禁止继续搜索。
11. **搜索必须按计划执行**：未列出现网搜索计划（§6.1a）就发起 mml_extract.py 调用 = 禁止。计划外搜索必须在日志中说明理由。
12. **AtomTask 先于 MMLCommand**：Phase 3 参数收集时，必须先取齐 AtomTask（配置方法+引用约束），再取 MMLCommand（枚举/类型）。跳过 AtomTask 直接取 MMLCommand = 参数引用约束缺失 = LLD 错误风险。

### 4.3 阶段进入条件总表

| 阶段 | 进入前必须满足 | 本阶段完成标志 | 未满足时动作 |
|---|---|---|---|
| Phase 0 | 图谱服务可访问 | 锁定 {域}/{场景}/{方案} | STOP，要求补需求关键词 |
| Phase 1 | Phase 0 已锁方案 + 已取齐任务层 | 输出匹配结果摘要 + AtomTask完备检查通过 | STOP，要求补现网脚本/决策点/补取AtomTask |
| Phase 2 | Phase 1 已完成 | **已生成 `xxx方案.md`** 并展示 | STOP，禁止进入 Phase 3 |
| Phase 3 | **用户已确认 GATE-1** | **已生成 `xxxLLD.md`** | STOP，禁止进入 Phase 4/5 |
| Phase 4 | Phase 3 文档已生成且参数已校验 | 用户已确认 GATE-2 | STOP，禁止进入 Phase 5 |
| Phase 5 | **`xxx方案.md` + `xxxLLD.md` 均已确认** | 生成每网元 .txt 配置文件 | STOP，禁止直接交付 |
| Phase 6.5 | Phase 6 核查已完成 | 用户已确认 GATE-3 | STOP，禁止进入 Phase 7 |

## 5. Phase 0: 业务域 → 场景 → 方案

### 5.1 步骤

1. **取全部业务域 md**：
   ```
   curl -s -X POST http://127.0.0.1:8000/api/v1/domains -H "X-API-Key: $GRAPH_API_KEY" -H "X-User-Id: $_AGENT_USERNAME"
   ```
   一次拿到全部业务域 md（含正文 + `## 边`）。按用户需求关键词匹配业务域。多域命中按主诉求判断；不确定向用户确认；未命中 → 提示补充属于哪个业务域。锁定 `{业务域}`。

2. **读域 md → 取场景清单**：
   从上一步拿到的域 md 提 `[[NetworkScenario@*]]`（域 md 已在手，不必再 POST）。按子场景识别关键词匹配锁定 `{场景}`。

3. **读场景 md → 决策点路由表 + 方案候选**：
   ```
   # Windows cmd.exe
   echo {"ids":["<场景id>"]} | curl -s -X POST http://127.0.0.1:8000/api/v1/md -H "X-API-Key: %GRAPH_API_KEY%" -H "X-User-Id: %_AGENT_USERNAME%" -H "Content-Type: application/json" --data-binary @-
   ```
   场景 md 含"决策点"路由表（业务诉求 → 推荐方案）+ `[[ConfigurationSolution@*]]` 下游方案。

4. **读方案 md → 锁定方案**：
   按决策点表 + 用户业务诉求匹配方案，POST /md 取候选方案 md，读概览确认。锁定 `{方案}`。

### 5.2 Phase 0 完备性检查清单 ★进入 Phase 1 前必须逐项声明

| # | 检查项 | 必须输出 | 未满足时 |
|---|---|---|---|
| C0-1 | 已调用 POST /domains 并读取全部域 md | 列出命中的业务域 ID | STOP，无法开始 |
| C0-2 | 已从域 md 提取 `[[NetworkScenario@*]]` 并锁定场景 | 输出场景 ID + 匹配关键词 | STOP，要求补需求关键词 |
| C0-3 | 已 POST 场景 md，读取决策点路由表 | 列出决策点 + 匹配的方案 ID | STOP，需用户选方案 |
| C0-4 | 已 POST 方案 md，确认方案概览 | 输出方案 ID + 核心编排内容 | STOP，方案未确认 |

**禁止**：跳过任何检查项直接进入 Phase 1；未输出 C0-1~C0-4 声明就发起现网搜索。

### 5.3 输出

```
已识别：业务域={域名}，场景={场景名}，方案={方案名}
将进入需求分析与现网匹配（Phase 1）。请提供现网配置脚本。
```

## 6. Phase 1: 任务层下钻 + 现网匹配

⚠ 完成后 STOP 等 Phase 2 确认。禁止在本阶段生成任何 MML 命令。

### 6.0 取相关任务层 + 取必读纪律 + 图谱完备性检查

**★进入本节前，必须完成 Phase 0 完备性检查（§5.2），并输出方案 md 已读取的声明。**

沿方案 md 引用，**按用户意图筛选相关 task**，POST /md 批量取 FeatureTask / CompoundTask / AtomTask。按需下钻特性层/命令层。

**★AtomTask必取约束**：CompoundTask/FeatureTask 引用的 AtomTask 是"命令怎么配"的核心知识层，包含：
- ① 参数引用约束（哪个参数只能引用哪种对象）——这是 MMLCommand 参数表不会以同样粒度说明的
- ② 命令间的配置方法（步骤顺序、联动参数、典型脚本）
- ③ 命令级约束（critical 约束，如"FLOWFILTERNAME 只能引用 ADD FLOWFILTER 产出的对象"）
跳过 AtomTask 直接读 MMLCommand = 丢失引用约束 = 参数引用错误。Phase 1 图谱完备性声明中，已取 CompoundTask 引用的 AtomTask 必须列出；未取的 = 图谱不完备 = 禁止进入现网搜索。

**★图谱知识完备性检查（进入现网搜索前必须声明）**：

在发起任何现网 mml_extract.py 调用之前，必须输出：

```
【图谱完备性声明】
已取方案: {方案ID}
已取任务层: {列出所有已POST取得的FeatureTask/CompoundTask/AtomTask ID}
已取命令层: {列出所有已POST取得的MMLCommand ID，或"尚未下钻命令层"}
未取但配置相关: {列出尚未取得但配置必需的引用，或"无"}
★AtomTask完备检查: {逐个检查已取CompoundTask的引用，是否所有被引用的AtomTask都已取。若存在未取的AtomTask = 不完备}
图谱知识是否齐备: {是/否} — 若"否"，继续下钻，禁止进入现网搜索
```

**禁止**：图谱完备性声明为"否"时发起现网搜索。AtomTask完备检查存在未取项时，视为"否"。

**★取必读纪律**：取回的所有 md **必须通读**以下三段，禁止只看概览：
1. **配置方法**（步骤表 + 命令清单）——漏一步 = 配置缺命令
2. **约束/Rule**（critical 项必须记录到 LLD）——漏约束 = 配置出错
3. **决策点**（影响本次配置的必须确认）——漏决策点 = 参数错误

尤其注意 **tail/收尾类 CompoundTask**，其命令常被遗漏但约束常标注 critical。

### 6.1 现网脚本前置检查

- **【GATE-A】现网脚本路径**：未提供 → STOP 要求提供。
- **【GATE-B】脚本存在性**：用 Glob 或 `ls` 验证存在且可读，**严禁 Read 工具**（数万行必溢出），否则 STOP。

### 6.1a 强制搜索计划 ★现网搜索前置门控

**【GATE-C】搜索计划门控**：进入现网搜索前，**必须先输出搜索计划并等待自身确认**。未输出搜索计划就发起 mml_extract.py 调用 = 违反契约 §4.2-11。

搜索计划分三步，**每步必须完整列出后才能执行**：

**Step 1 批量广拉**：用多命令类型 + 宽正则一次拉出所有相关对象，而非逐类型逐次查询。

| # | 目标 | 命令类型 (-c) | 参数筛选 (-p) | 说明 |
|---|---|---|---|---|

### 6.2 现网解析

**使用 `mml_extract.py` 脚本**查询现网，**禁止逐条 PowerShell Select-String / findstr / grep**。

脚本用法：

```bash
# 按命令类型筛选
python scripts/mml_extract.py -f 现网脚本.txt -c "ADD RULE"

# 按命令类型 + 参数筛选（参数名和值均支持正则）
python scripts/mml_extract.py -f 现网脚本.txt -c "ADD URR" -p "RG=133"
python scripts/mml_extract.py -f 现网脚本.txt -c "ADD RULE" -p "RULENAME=l_tk.*"

# ★多命令类型（逗号分隔）——优先使用，一把拉出多类型对象
python scripts/mml_extract.py -f 现网脚本.txt -c "ADD URR,ADD URRGROUP,ADD PCCPOLICYGRP"

# 多参数筛选（AND 关系）——同一命令必须同时满足所有参数条件
python scripts/mml_extract.py -f 现网脚本.txt -c "ADD URR" -p "RG=121" -p "USAGERPTMODE=OFFLINE"

# 统计模式
python scripts/mml_extract.py -f 现网脚本.txt --stats

# 引用追踪——单对象
python scripts/mml_extract.py -f 现网脚本.txt --trace "ffg_tik_ytb"

# ★引用追踪——多对象批量（一次追踪多个对象，避免逐个调用）
python scripts/mml_extract.py -f 现网脚本.txt --trace "ffg_tik_ytb,ffg_rs,cg_tiktok_shaping"

# ★递归追踪——沿 references 方向自动展开多层（避免手动逐层 trace）
python scripts/mml_extract.py -f 现网脚本.txt --trace "cg_tiktok_shaping" --depth 2
# depth=1 = 仅直接引用（等同于旧版 --trace）
# depth=2 = 自动展开到下一层（如 PCCPOLICYGRP→URRGROUP）
# depth=3 = 继续展开（如 URRGROUP→URR）

# JSON 格式输出
python scripts/mml_extract.py -f 现网脚本.txt -c "ADD RULE" --format json
```

> 脚本自动缓存（按文件修改时间失效），后续查询秒返回。跨平台 Python 3.7+，零依赖。

**★参数筛选语义关键约束 — `-p` 是全局 AND，跨所有命令类型生效**：

`-p "RG=121"` 的含义是"**每条**返回的命令都必须包含名为 RG 且值为 121 的参数"。这与 `-c` 指定的命令类型是**交集**关系：

```
筛选逻辑: 先按 -c 过滤命令类型 → 再对每条命令逐个检查 -p 条件 → 不满足 -p 的命令被丢弃
```

**⚠ 严重陷阱**：不同命令类型的参数集不同！`ADD URR` 有 `RG` 参数，但 `ADD URRGROUP`、`ADD PCCPOLICYGRP`、`ADD RULE` 等命令**没有 `RG` 参数**。如果写：

```bash
# ❌ 错误示例——ADD URRGROUP/PCCPOLICYGRP 没有 RG 参数，会被全部丢弃，实际只返回 ADD URR
python scripts/mml_extract.py -f 现网脚本.txt -c "ADD URR,ADD URRGROUP,ADD PCCPOLICYGRP" -p "RG=121"
```

**正确做法**：多命令类型 + 参数筛选时，**只对共享该参数的命令类型加 `-p`**，其余类型单独查询或不用 `-p`：

```bash
# ✅ 正确——不共用参数的多类型，不加 -p，全量拉出后再人工/脚本过滤
python scripts/mml_extract.py -f 现网脚本.txt -c "ADD URR,ADD URRGROUP,ADD PCCPOLICYGRP"

# ✅ 正确——需要参数筛选时，拆分为只包含该参数的命令类型
python scripts/mml_extract.py -f 现网脚本.txt -c "ADD URR" -p "RG=121"

# ✅ 正确——用 trace/depth 替代多类型+参数筛选，trace 自动沿引用链展开
python scripts/mml_extract.py -f 现网脚本.txt --trace "cg_rs_shap_s8m" --depth 3
```

> **搜索计划（§6.1a）制定时必须考虑此约束**：Step 1 批量广拉的 `-c` 和 `-p` 组合，必须确保每个 `-c` 类型都包含 `-p` 指定的参数名。不确定时，宁可不加 `-p`（全量拉出后过滤），或拆成多次查询。

**★现网搜索三条纪律**：

1. **批量拉取优先**：能用多命令类型 `-c` 一次拉出多类对象，就不要拆成多次逐类型查询。能用宽正则匹配一批相关对象，就不要逐个精准查询后再加下一个。**一次调用能拿到的数据绝不分两次**。
2. **trace 结果必须完整消费**：`--trace` / `--depth` 返回的每条 MML 命令行已包含完整参数格式和引用关系。**禁止对 trace/depth 已返回的对象再发起单查**。如果 trace 结果不够详细，用 `--depth` 加深，不要回到逐类型单查。每次 trace/depth 调用后必须输出**trace 消费确认**：
   ```
   【trace消费确认】对象: {trace的对象名} | 返回对象: {列出返回的所有对象名} | 已消费: {列出已从返回结果中提取信息的对象} | 无需再查: {列出trace已覆盖、禁止再单查的对象}
   ```
3. **搜索结果即入 LLD**：每次现网搜索的结果立即记入 LLD 草稿（参数格式、命名规律、对象清单），**禁止重复搜索同一对象**。Phase 3 直接复用 Phase 1 已收集的现网数据，不再重复查现网。

**展开纪律**：当现网存在 A 引用 B 的关系时，必须完整提取 A 的所有成员引用，不得只取 A 名字就跳过。可使用 `--trace` + `--depth` 追踪。

### 6.2a 搜索日志机制 ★每次 mml_extract.py 调用必须记录

每次执行 mml_extract.py 后，**必须输出搜索日志**，格式如下：


示例：
```

【现网搜索日志 #2】步骤: S2-1 | 命令: --trace "cg_tiktok_shaping" --depth 3 | 目标: 递归追踪TikTok计费链 | 结果摘要: 完整链PCCPOLICYGRP→URRGROUP→URR | 已获取对象: cp_tiktok_shap_11, cbb_tiktok_shap_offline(URRID=1311,RG=131), cbb_tiktok_shap_online(URRID=1310,RG=131)
累计已获取: {l_tk_nu_1265(PCC/BWM), cg_tiktok_shaping, cp_tiktok_shap_11, cbb_tiktok_shap_offline, cbb_tiktok_shap_online}
```

> **日志作用**：① 防止重复搜索（查累计清单即知）② 审计搜索效率（日志数/总调用数比）③ Phase 3 直接从累计清单复用数据，禁止重复查现网。

### 8.1 步骤

## 14. 独立执行约束

> 以下约束仅在 §4.2 和各 Phase 节中**未出现过**，属于增量独立约束。§4.2 已覆盖的（禁止跳 Phase/GATE 必须 STOP/文档必须先落地/图谱取齐先于逻辑/检索三纪律/方案LLD禁止同轮等）此处不再重述。

1. **严格依赖顺序**：按任务层图谱（FeatureTask/CompoundTask/AtomTask）的编排排列命令。
2. **追问而非猜测**：缺失决策信息或必填参数必须追问。
3. **动网保守**：现网对象修改必须有用户确认。
4. **命名遵循现网**：新对象命名沿用现网规律。
5. **跨网元一致**：多网元共用参数必须一致。
6. **优先级必须用户确认**：涉及有序规则时，提取现网优先级值分析后 STOP 确认；数值语义由图谱规则定义，禁凭记忆。
7. **参数定义禁凭记忆**：必须从 AtomTask + MMLCommand 取。不仅枚举值，"配置原则""参数含义""引用约束"也必须从图谱取，禁止凭记忆填写参数引用的对象类型。★参数引用约束优先从 AtomTask 取——AtomTask 明确声明"参数 X 引用 ADD Y 产出的对象"，MMLCommand 参数表通常不提供此粒度。
8. **通用配置约束从图谱取**：执行时序、License 前置、对象依赖等约束从相关 md 的 Rule 段取。
9. **图谱调用先于现网搜索**：Phase 0/1 的图谱下钻未输出完备性声明前，禁止发起 mml_extract.py 调用。违反 = 直接回 Phase 0 补齐图谱调用。
10. **零图谱调用 = 禁止生成配置**：若最终生成配置时，图谱调用日志为空（没有 POST /md 记录），则所有参数均来自"凭记忆/现网推断"，配置不可信，禁止交付。
11. **现网搜索累计清单去重**：每次 mml_extract.py 调用前检查累计已获取清单，目标对象已在清单中则禁止发起重复搜索。

- **新增业务域/场景/方案**：后端图谱加对象，**本 SOP 不改**。
